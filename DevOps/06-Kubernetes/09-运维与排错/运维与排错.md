## 1. kubectl 命令速查？

**回答：**

```bash
# ===== 集群信息 =====
kubectl cluster-info
kubectl version --short
kubectl get nodes -o wide
kubectl top nodes                      # 需要 Metrics Server
kubectl api-resources                  # 所有 API 资源
kubectl api-versions                   # 所有 API 版本

# ===== 上下文管理 =====
kubectl config get-contexts
kubectl config current-context
kubectl config use-context production
kubectl config set-context --current --namespace=production

# ===== 资源查看 =====
kubectl get pods -n prod -o wide
kubectl get pods -A                    # 所有 Namespace
kubectl get pods -l app=myapp          # 按 Label 过滤
kubectl get pods -o yaml               # YAML 输出
kubectl get pods -o json               # JSON 输出
kubectl get pods -o jsonpath='{.items[*].status.podIP}'
kubectl get pods --field-selector=status.phase=Running
kubectl get pods --sort-by='.metadata.creationTimestamp'
kubectl get all -n production          # 所有资源

# ===== 详细信息 =====
kubectl describe pod myapp-xxx
kubectl describe node node1
kubectl describe svc myapp-svc

# ===== 创建/更新/删除 =====
kubectl apply -f manifest.yaml         # 声明式 (推荐)
kubectl create -f manifest.yaml        # 命令式
kubectl delete -f manifest.yaml
kubectl delete pod myapp-xxx
kubectl delete pods -l app=myapp       # 按 Label 删除
kubectl edit deployment myapp          # 在线编辑

# ===== 扩缩容 =====
kubectl scale deployment myapp --replicas=5
kubectl autoscale deployment myapp --min=2 --max=10 --cpu-percent=70

# ===== 日志 =====
kubectl logs myapp-xxx
kubectl logs myapp-xxx -f              # 持续跟踪
kubectl logs myapp-xxx --tail=100      # 最后 100 行
kubectl logs myapp-xxx -c sidecar      # 指定容器
kubectl logs myapp-xxx --previous      # 上一个容器的日志
kubectl logs -l app=myapp --all-containers

# ===== 执行与调试 =====
kubectl exec -it myapp-xxx -- bash
kubectl exec myapp-xxx -- cat /etc/config/app.conf
kubectl port-forward myapp-xxx 8080:80
kubectl port-forward svc/myapp-svc 8080:80
kubectl cp myapp-xxx:/app/data.log ./data.log
kubectl debug -it myapp-xxx --image=busybox

# ===== 资源使用 =====
kubectl top pods
kubectl top pods --sort-by=cpu
kubectl top pods --sort-by=memory
kubectl top nodes

# ===== 事件 =====
kubectl get events --sort-by='.lastTimestamp'
kubectl get events -n production --field-selector type=Warning
```

## 2. Pod 启动失败排查？

**回答：**

```bash
# 排查流程:
# 1. 查看 Pod 状态
kubectl get pod myapp -o wide
# 关注: STATUS, RESTARTS, NODE

# 2. 查看详细事件
kubectl describe pod myapp
# 重点看 Events 部分

# 3. 查看容器日志
kubectl logs myapp
kubectl logs myapp --previous          # CrashLoopBackOff 时看上次日志
kubectl logs myapp -c init-container   # Init Container 日志

# 4. 进入容器调试
kubectl exec -it myapp -- sh
kubectl debug -it myapp --image=busybox
```

### 常见问题对照表

```
状态                  排查思路
────────────────────  ──────────────────────────────────────

Pending (一直)
  → 调度失败:
    kubectl describe pod → Events
    "Insufficient cpu"     → Node 资源不足, 增加 Node 或减少 requests
    "no nodes available"   → 检查 nodeSelector, Taint/Toleration
    "Unschedulable"        → Node 被 cordon, kubectl uncordon
    "volume not found"     → PVC Pending, 检查 StorageClass

ImagePullBackOff / ErrImagePull
  → 镜像问题:
    检查镜像名是否拼写正确
    检查镜像 tag 是否存在
    检查 imagePullSecrets (私有 Registry)
    Node 能否访问 Registry (网络)

CrashLoopBackOff
  → 容器不断崩溃重启:
    kubectl logs myapp --previous   # 查看崩溃前日志
    检查 CMD/ENTRYPOINT 是否正确
    检查配置文件是否正确 (ConfigMap/Secret)
    检查依赖服务是否可用
    检查 OOM (dmesg | grep oom)

OOMKilled
  → 内存不足:
    增加 memory limits
    检查应用内存泄漏

CreateContainerConfigError
  → 配置错误:
    ConfigMap/Secret 不存在
    key 不存在
    检查 env valueFrom 引用

RunContainerError
  → 启动错误:
    命令不存在 (二进制路径错误)
    权限问题 (readOnlyRootFilesystem + 写操作)
    securityContext 限制

Evicted
  → Node 资源不足被驱逐:
    检查 Node 资源
    清理旧 Pod: kubectl get pods --field-selector=status.phase=Failed -o name | xargs kubectl delete
```

## 3. Node 问题排查？

**回答：**

```bash
# 查看 Node 状态
kubectl get nodes
kubectl describe node node1

# Node Conditions
kubectl get node node1 -o jsonpath='{.status.conditions[*].type}'
# Ready, MemoryPressure, DiskPressure, PIDPressure, NetworkUnavailable

# 关注:
#   Ready = False/Unknown → Node 异常
#   MemoryPressure = True → 内存不足, 开始驱逐 Pod
#   DiskPressure = True   → 磁盘不足
#   PIDPressure = True    → PID 不足

# 查看 Node 资源使用
kubectl top node node1
kubectl describe node node1 | grep -A 10 "Allocated resources"
# 查看 requests/limits 已分配量

# Node NotReady 排查
# 1. SSH 到 Node
# 2. 检查 kubelet 状态
systemctl status kubelet
journalctl -u kubelet -f

# 3. 检查容器运行时
systemctl status containerd
crictl ps
crictl pods

# 4. 检查系统资源
top
free -h
df -h
dmesg | tail -50

# 5. 常见原因:
#   kubelet 崩溃 → 重启 kubelet
#   containerd 崩溃 → 重启 containerd
#   磁盘满 → 清理镜像/日志
#   内核 panic → 查看 dmesg
#   网络中断 → 检查网络连接和防火墙
```

## 4. Service 与网络排查？

**回答：**

```bash
# ===== Service 排查 =====

# 1. 检查 Service 定义
kubectl get svc myapp-svc -o yaml

# 2. 检查 Endpoints
kubectl get endpoints myapp-svc
# ENDPOINTS 为空 → Service selector 没匹配到 Pod
# 解决: 检查 Pod labels 与 Service selector 是否一致

# 常见原因:
#   Pod labels 不匹配
kubectl get pods --show-labels

#   Pod 不是 Ready 状态 → readinessProbe 未通过
kubectl get pods -o wide
kubectl describe pod myapp

# 3. 测试 Service 连通性
kubectl run tmp --rm -it --image=busybox -- wget -qO- http://myapp-svc:80

# ===== DNS 排查 =====
kubectl run tmp --rm -it --image=busybox -- nslookup myapp-svc
# 失败 → 检查 CoreDNS
kubectl get pods -n kube-system -l k8s-app=kube-dns
kubectl logs -n kube-system -l k8s-app=kube-dns

# ===== Ingress 排查 =====
kubectl describe ingress myapp-ingress
kubectl logs -n ingress-nginx -l app.kubernetes.io/name=ingress-nginx

# ===== 网络排查工具 =====
kubectl run netshoot --rm -it --image=nicolaka/netshoot -- bash
# curl, dig, nslookup, tcpdump, iperf, ss, ip, traceroute, nmap

# 抓包
kubectl exec netshoot -- tcpdump -i eth0 -nn port 80 -c 20
```

## 5. 存储排查？

**回答：**

```bash
# PVC Pending
kubectl describe pvc myapp-data
# 原因:
#   StorageClass 不存在
#   StorageClass provisioner 异常 (CSI 驱动)
#   没有匹配的 PV (静态)
#   存储配额超限

# Volume 挂载失败
kubectl describe pod myapp
# FailedAttachVolume:
#   EBS 已 attach 到其他 Node (RWO 限制)
#   CSI 驱动异常
# FailedMount:
#   NFS 服务器不可达
#   文件系统损坏
#   权限问题

# 检查 CSI 驱动
kubectl get pods -n kube-system | grep csi
kubectl logs -n kube-system csi-driver-xxx

# 检查 VolumeAttachment
kubectl get volumeattachments

# 强制解除 attach (危险, 确保旧 Pod 已停止)
kubectl delete volumeattachment <name>

# 容器内 I/O 问题
kubectl exec myapp -- df -h /data
kubectl exec myapp -- ls -la /data
```

## 7. 日志与监控方案？

**回答：**

```
K8s 日志架构:

方案 1: Node 级日志收集 (推荐)
  Pod stdout/stderr → /var/log/containers/ → DaemonSet (Fluent Bit) → 后端

方案 2: Sidecar 日志收集
  Pod (应用日志文件) → Sidecar (Fluent Bit) → 后端

方案 3: 应用直接推送
  Pod → 直接发送到日志后端 (不推荐, 耦合)
```

### 日志栈

```yaml
# Fluent Bit DaemonSet (Node 级收集)
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: fluent-bit
  namespace: logging
spec:
  template:
    spec:
      tolerations:
        - operator: Exists
      containers:
        - name: fluent-bit
          image: fluent/fluent-bit:latest
          volumeMounts:
            - name: varlog
              mountPath: /var/log
              readOnly: true
            - name: containers
              mountPath: /var/log/containers
              readOnly: true
      volumes:
        - name: varlog
          hostPath:
            path: /var/log
        - name: containers
          hostPath:
            path: /var/log/containers
```

### 监控栈

```
常用监控方案:

Prometheus + Grafana (最流行):
  Prometheus      → 指标采集和存储
  Grafana         → 可视化
  AlertManager    → 告警
  Node Exporter   → 节点指标
  kube-state-metrics → K8s 对象状态指标

安装 (kube-prometheus-stack):
  helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
  helm install prometheus prometheus-community/kube-prometheus-stack

关键指标:
  节点:  CPU/Memory/Disk 使用率, Node Ready
  Pod:   CPU/Memory 使用率, Restart Count, OOMKilled
  容器:  资源使用 vs 限制
  API:   请求延迟, 错误率
  etcd:  延迟, Pod 数量, Leader 选举
```

## 8. 集群升级？

**回答：**

```bash
# kubeadm 集群升级流程

# 1. 查看当前版本
kubectl version --short
kubeadm version

# 2. 查看可用版本
apt-cache madison kubeadm | head

# ===== 升级 Control Plane =====

# 3. 升级 kubeadm
apt-get update
apt-get install kubeadm=1.29.0-*
apt-mark hold kubeadm

# 4. 验证升级计划
kubeadm upgrade plan

# 5. 执行升级
kubeadm upgrade apply v1.29.0

# 6. 排空 Master Node
kubectl drain master1 --ignore-daemonsets

# 7. 升级 kubelet 和 kubectl
apt-get install kubelet=1.29.0-* kubectl=1.29.0-*
apt-mark hold kubelet kubectl
systemctl daemon-reload
systemctl restart kubelet

# 8. 恢复调度
kubectl uncordon master1

# ===== 升级 Worker Node (逐个) =====

# 9. 排空 Worker
kubectl drain worker1 --ignore-daemonsets --delete-emptydir-data

# 10. SSH 到 Worker, 升级
apt-get install kubeadm=1.29.0-*
kubeadm upgrade node
apt-get install kubelet=1.29.0-* kubectl=1.29.0-*
systemctl daemon-reload
systemctl restart kubelet

# 11. 恢复
kubectl uncordon worker1

# 注意:
#   - 只能升级到下一个小版本 (1.28 → 1.29, 不能跳 1.28 → 1.30)
#   - 先升级 Control Plane, 再升级 Worker
#   - 逐个 Worker 升级, 确保服务可用
#   - 升级前备份 etcd
```

## 9. etcd 备份与恢复？

**回答：**

```bash
# ===== 备份 =====
ETCDCTL_API=3 etcdctl snapshot save backup.db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key

# 验证备份
ETCDCTL_API=3 etcdctl snapshot status backup.db --write-out=table

# ===== 恢复 =====
# 1. 停止 kube-apiserver 和 etcd
mv /etc/kubernetes/manifests/kube-apiserver.yaml /tmp/
mv /etc/kubernetes/manifests/etcd.yaml /tmp/

# 2. 恢复数据
ETCDCTL_API=3 etcdctl snapshot restore backup.db \
  --data-dir=/var/lib/etcd-restored \
  --name=master1 \
  --initial-cluster=master1=https://master1:2380 \
  --initial-advertise-peer-urls=https://master1:2380

# 3. 修改 etcd 数据目录
# 编辑 /tmp/etcd.yaml 中的 --data-dir 指向 /var/lib/etcd-restored

# 4. 恢复 Static Pod
mv /tmp/etcd.yaml /etc/kubernetes/manifests/
mv /tmp/kube-apiserver.yaml /etc/kubernetes/manifests/

# 5. 验证
kubectl get nodes
kubectl get pods -A

# 自动化备份 (CronJob)
# 使用 etcd 备份容器定时执行 snapshot save
```

## 10. 常用排查工具？

**回答：**

```bash
# ===== kubectl debug =====
# 临时调试容器
kubectl debug -it myapp --image=busybox -- sh

# 复制 Pod 调试 (不影响原 Pod)
kubectl debug myapp -it --copy-to=myapp-debug --container=app -- sh

# 调试 Node
kubectl debug node/node1 -it --image=ubuntu
# 宿主机挂载在 /host, chroot /host 进入宿主机

# ===== crictl (容器运行时调试) =====
crictl ps
crictl pods
crictl logs <container-id>
crictl inspect <container-id>
crictl stats

# ===== stern (多 Pod 日志聚合) =====
# 比 kubectl logs 更方便
stern myapp                           # 所有名字包含 myapp 的 Pod
stern -l app=myapp                    # 按 Label
stern myapp -n production --since 1h

# ===== k9s (终端 UI) =====
k9s                                   # 交互式集群管理

# ===== kubectx / kubens =====
kubectx                              # 切换集群
kubens                               # 切换 Namespace

# ===== kubectl-neat =====
kubectl get pod myapp -o yaml | kubectl neat  # 清理 YAML

# ===== kube-bench =====
# CIS Kubernetes Benchmark 安全检查
kube-bench run --targets master
kube-bench run --targets node

# ===== 常用排查 one-liner =====
# 所有非 Running 的 Pod
kubectl get pods -A --field-selector=status.phase!=Running

# 最近的 Warning 事件
kubectl get events -A --field-selector type=Warning --sort-by='.lastTimestamp' | tail -20

# 重启次数最多的 Pod
kubectl get pods -A --sort-by='.status.containerStatuses[0].restartCount' | tail -10

# 资源使用 Top
kubectl top pods -A --sort-by=cpu | head -20
```
