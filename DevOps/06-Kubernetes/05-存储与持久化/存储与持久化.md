# 存储与持久化

---

## 1. K8s 存储体系概览？

**回答：**

```
K8s 存储架构:

  ┌──────────────────────────────────────────────────────────┐
  │  Pod                                                     │
  │   └─ volumeMounts: /data                                 │
  └────────────────────────┬─────────────────────────────────┘
                           │
  ┌────────────────────────▼─────────────────────────────────┐
  │  Volume (Pod 级别)                                       │
  │  emptyDir / hostPath / configMap / secret / downwardAPI   │
  └──────────────────────────────────────────────────────────┘

  ┌──────────────────────────────────────────────────────────┐
  │  PersistentVolumeClaim (PVC)  — 用户申请存储             │
  └────────────────────────┬─────────────────────────────────┘
                           │ 绑定
  ┌────────────────────────▼─────────────────────────────────┐
  │  PersistentVolume (PV) — 存储资源                        │
  │  手动创建 (静态) 或 StorageClass 自动创建 (动态)         │
  └────────────────────────┬─────────────────────────────────┘
                           │ CSI 驱动
  ┌────────────────────────▼─────────────────────────────────┐
  │  底层存储: AWS EBS / GCE PD / NFS / Ceph / Local SSD    │
  └──────────────────────────────────────────────────────────┘
```

## 2. 临时存储 Volume 类型？

**回答：**

### emptyDir

```yaml
# emptyDir: Pod 级别临时存储
# Pod 被删除时数据丢失
# 同一 Pod 内的容器共享
spec:
  containers:
    - name: app
      volumeMounts:
        - name: cache
          mountPath: /cache
        - name: shared-logs
          mountPath: /var/log/app
    - name: log-agent
      volumeMounts:
        - name: shared-logs
          mountPath: /var/log/app
          readOnly: true
  volumes:
    - name: cache
      emptyDir:
        sizeLimit: 500Mi          # 可选, 限制大小
    - name: shared-logs
      emptyDir: {}
    - name: tmpfs-vol
      emptyDir:
        medium: Memory            # 使用内存 (tmpfs), 更快但占用内存配额

# 用途:
#   - 容器间共享文件 (日志、Socket)
#   - 缓存
#   - 临时计算空间
```

### hostPath

```yaml
# hostPath: 挂载 Node 上的路径到 Pod
# 注意：不同 Node 上的数据不同，Pod 迁移后数据不跟随
spec:
  volumes:
    - name: host-data
      hostPath:
        path: /data/myapp
        type: DirectoryOrCreate   # 不存在则创建

# type 值:
#   ""                  → 不检查
#   DirectoryOrCreate   → 不存在则创建目录
#   Directory           → 必须存在
#   FileOrCreate        → 不存在则创建文件
#   File                → 必须存在
#   Socket              → Unix Socket
#   CharDevice          → 字符设备
#   BlockDevice         → 块设备

# ⚠️ 安全风险: 可以访问 Node 文件系统
# 仅用于 DaemonSet (如日志收集、监控) 和 Static Pod
```

### configMap / secret / downwardAPI

```yaml
spec:
  containers:
    - name: app
      volumeMounts:
        - name: config
          mountPath: /etc/config
          readOnly: true
        - name: secrets
          mountPath: /etc/secrets
          readOnly: true
        - name: podinfo
          mountPath: /etc/podinfo
  volumes:
    - name: config
      configMap:
        name: myapp-config
        items:                     # 可选, 选择特定 key
          - key: app.conf
            path: application.conf
        defaultMode: 0644

    - name: secrets
      secret:
        secretName: myapp-secret
        defaultMode: 0400

    - name: podinfo
      downwardAPI:                 # 注入 Pod 元数据
        items:
          - path: labels
            fieldRef:
              fieldPath: metadata.labels
          - path: cpu-limit
            resourceFieldRef:
              containerName: app
              resource: limits.cpu
```

## 3. PV 与 PVC 详解？

**回答：**

```
PV (PersistentVolume):
  集群级资源, 由管理员预创建或 StorageClass 动态创建
  独立于 Pod 的生命周期

PVC (PersistentVolumeClaim):
  用户对存储的请求, 命名空间级资源
  PVC 绑定到满足条件的 PV
```

### PV

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: pv-nfs-01
  labels:
    type: nfs
spec:
  capacity:
    storage: 100Gi
  accessModes:
    - ReadWriteMany
  persistentVolumeReclaimPolicy: Retain
  storageClassName: nfs
  mountOptions:
    - hard
    - nfsvers=4.1
  nfs:
    server: nfs-server.example.com
    path: /data/vol01

# Access Modes:
#   ReadWriteOnce (RWO) → 单节点读写
#   ReadOnlyMany  (ROX) → 多节点只读
#   ReadWriteMany (RWX) → 多节点读写
#   ReadWriteOncePod (RWOP) → 单 Pod 读写 (K8s 1.27+)
#
# 注意: 不同存储后端支持的模式不同
#   AWS EBS  → RWO 只
#   NFS      → RWO, ROX, RWX
#   CephFS   → RWO, ROX, RWX

# Reclaim Policy:
#   Retain  → PVC 删除后保留 PV 和数据 (手动回收)
#   Delete  → PVC 删除后自动删除 PV 和底层存储
#   Recycle → 已废弃 (rm -rf /volume/*)
```

### PVC

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: myapp-data
  namespace: production
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: fast-ssd       # 指定 StorageClass
  resources:
    requests:
      storage: 20Gi
  selector:                        # 可选, 选择特定 PV
    matchLabels:
      type: ssd

# Pod 使用 PVC
spec:
  containers:
    - name: app
      volumeMounts:
        - name: data
          mountPath: /data
  volumes:
    - name: data
      persistentVolumeClaim:
        claimName: myapp-data
```

### PV 绑定过程

```
PVC 创建 → PV Controller 查找满足条件的 PV:
  1. accessModes 匹配
  2. storageClassName 匹配
  3. capacity >= requests.storage
  4. selector 匹配 (如有)
  5. volumeMode 匹配

绑定结果:
  PVC.status.phase = Bound    → 成功绑定
  PVC.status.phase = Pending  → 未找到匹配 PV (等待动态供给或手动创建)

PV 状态:
  Available → 可用, 未绑定
  Bound     → 已绑定到 PVC
  Released  → PVC 已删除, PV 等待回收
  Failed    → 回收失败
```

## 4. StorageClass 与动态供给？

**回答：**

```
StorageClass 定义存储类型, 支持动态创建 PV

流程:
  PVC 指定 storageClassName
    → StorageClass 的 provisioner 自动创建 PV
    → PV 绑定到 PVC
    → Pod 使用 PVC
```

```yaml
# AWS EBS StorageClass
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"  # 设为默认
provisioner: ebs.csi.aws.com           # CSI 驱动
parameters:
  type: gp3
  iops: "4000"
  throughput: "200"
  encrypted: "true"
reclaimPolicy: Delete                  # Retain / Delete
allowVolumeExpansion: true             # 允许扩容
volumeBindingMode: WaitForFirstConsumer  # Immediate / WaitForFirstConsumer
mountOptions:
  - debug

# GCE PD StorageClass
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: ssd
provisioner: pd.csi.storage.gke.io
parameters:
  type: pd-ssd
  replication-type: regional-pd        # 区域级复制

# NFS StorageClass (使用 NFS Subdir External Provisioner)
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: nfs
provisioner: nfs-subdir-external-provisioner
parameters:
  archiveOnDelete: "true"
```

```
volumeBindingMode:
  Immediate             → PVC 创建即绑定 PV(可能在不同 Zone)
  WaitForFirstConsumer  → 等 Pod 调度后再绑定(确保 PV 和 Pod 在同一 Zone)
                          推荐, 避免跨 Zone 存储问题
```

## 5. 存储扩容？

**回答：**

```yaml
# 前提: StorageClass 必须设置 allowVolumeExpansion: true
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: expandable-ssd
provisioner: ebs.csi.aws.com
allowVolumeExpansion: true
```

```bash
# 扩容 PVC (只能扩大, 不能缩小)
kubectl patch pvc myapp-data -p '{"spec":{"resources":{"requests":{"storage":"50Gi"}}}}'

# 或编辑
kubectl edit pvc myapp-data
# 修改 spec.resources.requests.storage

# 扩容过程:
#   1. PVC 更新 → CSI 驱动扩展底层存储
#   2. 文件系统扩展:
#      - 某些 CSI 支持在线扩展 (不用重启 Pod)
#      - 某些需要重新启动 Pod 才能扩展文件系统

# 查看扩容状态
kubectl describe pvc myapp-data
# Conditions:
#   Type: FileSystemResizePending
#   → 文件系统还未扩展, 需要 Pod 重建
```

## 6. CSI (Container Storage Interface)？

**回答：**

```
CSI 是 K8s 存储的标准接口
存储厂商通过 CSI 驱动将存储接入 K8s

架构:
  kubelet ──→ CSI Driver (Node Plugin) ──→ 底层存储
  Controller ──→ CSI Driver (Controller Plugin) ──→ 底层存储

CSI 组件:
  Controller Plugin:
    - CreateVolume / DeleteVolume
    - ControllerPublish / Unpublish (Attach/Detach)
    - CreateSnapshot / DeleteSnapshot

  Node Plugin:
    - NodeStage / NodeUnstage (格式化/挂载到全局路径)
    - NodePublish / NodeUnpublish (挂载到 Pod 路径)

常见 CSI 驱动:
  驱动                  存储
  ────────────────────  ────────────
  ebs.csi.aws.com       AWS EBS
  pd.csi.storage.gke.io GCE PD
  disk.csi.azure.com    Azure Disk
  csi.vsphere.vmware.com VMware vSphere
  rook-ceph.cephfs      Ceph (via Rook)
  nfs.csi.k8s.io        NFS
```

```bash
# 查看集群中的 CSI 驱动
kubectl get csidrivers

# 查看 CSI 节点信息
kubectl get csinodes
```

## 7. VolumeSnapshot 快照？

**回答：**

```yaml
# VolumeSnapshotClass
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshotClass
metadata:
  name: ebs-snapshot
driver: ebs.csi.aws.com
deletionPolicy: Retain            # Delete / Retain

# 创建快照
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshot
metadata:
  name: myapp-data-snapshot
spec:
  volumeSnapshotClassName: ebs-snapshot
  source:
    persistentVolumeClaimName: myapp-data

# 从快照恢复
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: myapp-data-restored
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: fast-ssd
  resources:
    requests:
      storage: 20Gi
  dataSource:
    name: myapp-data-snapshot
    kind: VolumeSnapshot
    apiGroup: snapshot.storage.k8s.io
```

## 8. 数据库在 K8s 中的存储最佳实践？

**回答：**

```yaml
# PostgreSQL StatefulSet 存储示例
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
spec:
  serviceName: postgres-headless
  replicas: 3
  template:
    spec:
      containers:
        - name: postgres
          image: postgres:16
          ports:
            - containerPort: 5432
          env:
            - name: PGDATA
              value: /var/lib/postgresql/data/pgdata
          volumeMounts:
            - name: data
              mountPath: /var/lib/postgresql/data
            - name: wal
              mountPath: /var/lib/postgresql/wal
  volumeClaimTemplates:
    - metadata:
        name: data
      spec:
        accessModes: ["ReadWriteOnce"]
        storageClassName: fast-ssd
        resources:
          requests:
            storage: 100Gi
    - metadata:
        name: wal
      spec:
        accessModes: ["ReadWriteOnce"]
        storageClassName: high-iops     # WAL 需要高 IOPS
        resources:
          requests:
            storage: 20Gi
```

```
数据库存储最佳实践:

  1. 使用 StatefulSet + volumeClaimTemplates
     每个 Pod 独立 PVC, 数据不丢失

  2. 使用高性能 StorageClass
     数据库 → SSD (gp3/io2)
     WAL → 高 IOPS SSD

  3. Reclaim Policy 设为 Retain
     防止误删 PVC 导致数据丢失

  4. 数据目录和 WAL 分离
     不同磁盘, 避免 IO 竞争

  5. 定期快照备份
     VolumeSnapshot + CronJob

  6. 考虑 Operator
     推荐: CloudNativePG (PostgreSQL)
           Percona Operator (MySQL)
           Strimzi (Kafka)

  7. 是否应该在 K8s 中运行数据库?
     ✅ 适合: 开发测试, 中小规模, 有自动化运维
     ❌ 不适合: 超大规模, 性能极致要求
     替代: 使用云托管数据库 (RDS/Cloud SQL) + ExternalName Service
```

## 9. Projected Volume 与 Ephemeral Volume？

**回答：**

### Projected Volume

```yaml
# 将多个 Volume 源投射到同一个目录
spec:
  volumes:
    - name: all-in-one
      projected:
        sources:
          - configMap:
              name: myapp-config
              items:
                - key: app.conf
                  path: config/app.conf
          - secret:
              name: myapp-secret
              items:
                - key: password
                  path: secrets/password
          - downwardAPI:
              items:
                - path: labels
                  fieldRef:
                    fieldPath: metadata.labels
          - serviceAccountToken:
              path: token
              expirationSeconds: 3600
              audience: api
  containers:
    - name: app
      volumeMounts:
        - name: all-in-one
          mountPath: /etc/myapp
```

### Generic Ephemeral Volume (K8s 1.23+)

```yaml
# 使用 CSI 驱动创建临时卷, 随 Pod 生命周期
spec:
  volumes:
    - name: scratch
      ephemeral:
        volumeClaimTemplate:
          spec:
            accessModes: ["ReadWriteOnce"]
            storageClassName: fast-ssd
            resources:
              requests:
                storage: 10Gi
  containers:
    - name: app
      volumeMounts:
        - name: scratch
          mountPath: /scratch

# Pod 删除时 PVC 和 PV 自动清理
# 比 emptyDir 好: 可以使用 CSI 特性 (加密、高 IOPS)
```

## 10. 存储排查与运维？

**回答：**

```bash
# ===== PVC 状态排查 =====
kubectl get pvc
# NAME        STATUS    VOLUME    CAPACITY   ACCESS   STORAGECLASS   AGE
# myapp-data  Pending   -         -          -        fast-ssd       5m

# Pending 原因:
kubectl describe pvc myapp-data
# Events:
#   waiting for a volume to be created...
#   → StorageClass provisioner 无法创建 PV
#   → 检查 CSI 驱动是否正常
#   → 检查 StorageClass 参数
#   → 检查配额和存储容量

# ===== PV 排查 =====
kubectl get pv
kubectl describe pv pv-name

# ===== Pod Volume 挂载失败 =====
kubectl describe pod myapp
# Events:
#   FailedAttachVolume: ...
#   → EBS 可能已 attached 到其他 Node
#   → 检查 RWO Volume 是否被其他 Pod 使用
#
#   FailedMount: ...
#   → 文件系统损坏
#   → NFS 服务器不可达
#   → 权限问题

# ===== 存储空间 =====
# 查看 PV 使用率
kubectl exec myapp -- df -h /data

# 查看 StorageClass
kubectl get sc
kubectl describe sc fast-ssd

# ===== 常见问题 =====
# 1. PVC Pending
#    → 无匹配 PV (静态) 或 provisioner 异常 (动态)
#    → 检查 StorageClass 是否存在
#    → WaitForFirstConsumer 模式需要 Pod 调度后才绑定
#
# 2. Multi-Attach 错误
#    → RWO Volume 已被其他 Node attach
#    → 等旧 Pod 完全终止, 或强制删除 VolumeAttachment
#
# 3. 文件权限问题
#    → 使用 fsGroup 设置文件组
#    → initContainer 修改权限
#
# 4. PV Released 无法复用
#    → 手动删除 PV 上的 claimRef
#    kubectl patch pv pv-name -p '{"spec":{"claimRef": null}}'
```
