# 配置管理与密钥

---

## 1. ConfigMap 详解？

**回答：**

```
ConfigMap 存储非敏感配置数据（键值对或文件内容）
与应用代码分离, 支持不重建镜像修改配置
```

### 创建方式

```bash
# 1. 命令行创建
kubectl create configmap myapp-config \
  --from-literal=db_host=db.example.com \
  --from-literal=db_port=5432

# 2. 从文件创建
kubectl create configmap nginx-config \
  --from-file=nginx.conf \
  --from-file=mime.types

# 3. 从目录创建 (目录中每个文件成为一个 key)
kubectl create configmap app-config --from-file=config-dir/

# 4. 从 env 文件创建
kubectl create configmap env-config --from-env-file=app.env
```

```yaml
# 5. YAML 声明式
apiVersion: v1
kind: ConfigMap
metadata:
  name: myapp-config
  namespace: production
data:
  # 简单键值对
  db_host: "db.example.com"
  db_port: "5432"
  log_level: "INFO"

  # 多行配置文件
  app.properties: |
    server.port=8080
    spring.profiles.active=production
    logging.level.root=INFO

  nginx.conf: |
    server {
      listen 80;
      location / {
        proxy_pass http://localhost:8080;
      }
    }

binaryData:                      # 二进制数据 (Base64)
  logo.png: <base64-encoded>

immutable: true                  # 不可变 ConfigMap (K8s 1.21+)
                                 # 设置后不能修改, 提升性能
```

### 使用方式

```yaml
spec:
  containers:
    - name: app
      # 方式 1: 环境变量 — 单个 key
      env:
        - name: DB_HOST
          valueFrom:
            configMapKeyRef:
              name: myapp-config
              key: db_host

      # 方式 2: 环境变量 — 全部 key
      envFrom:
        - configMapRef:
            name: myapp-config
          prefix: APP_              # 可选前缀: APP_db_host

      # 方式 3: 文件挂载 — 全部
      volumeMounts:
        - name: config
          mountPath: /etc/config

      # 方式 4: 文件挂载 — 特定 key
      volumeMounts:
        - name: nginx-config
          mountPath: /etc/nginx/nginx.conf
          subPath: nginx.conf       # subPath 不会覆盖目录

  volumes:
    - name: config
      configMap:
        name: myapp-config
        defaultMode: 0644
    - name: nginx-config
      configMap:
        name: myapp-config
        items:
          - key: nginx.conf
            path: nginx.conf
```

### 更新行为

```
使用方式        ConfigMap 更新后    延迟
──────────────  ──────────────────  ──────
环境变量        Pod 不会更新        需要重启 Pod
Volume 挂载     自动更新文件        ~60s (kubelet sync)
subPath 挂载    不会更新            需要重启 Pod

自动重启 Pod 方案:
  1. Reloader (stakater/Reloader)
     annotations:
       reloader.stakater.com/auto: "true"
  2. 修改 Pod 模板的 annotation 触发 rollout
     kubectl patch deployment myapp -p \
       '{"spec":{"template":{"metadata":{"annotations":{"configHash":"v2"}}}}}'
```

## 2. Secret 详解？

**回答：**

```
Secret 存储敏感数据（密码、Token、证书等）
Base64 编码存储在 etcd 中（非加密, 需要额外启用加密）
```

### Secret 类型

```
类型                              用途
──────────────────────────────    ────────────────────
Opaque (默认)                    通用密钥数据
kubernetes.io/tls                 TLS 证书 (tls.crt + tls.key)
kubernetes.io/dockerconfigjson    镜像拉取凭证
kubernetes.io/basic-auth          基本认证 (username + password)
kubernetes.io/ssh-auth            SSH 认证 (ssh-privatekey)
kubernetes.io/service-account-token  SA Token (自动创建)
```

### 创建方式

```bash
# Opaque
kubectl create secret generic myapp-secret \
  --from-literal=db_password=S3cureP@ss \
  --from-literal=api_key=abc123

# TLS
kubectl create secret tls tls-secret \
  --cert=tls.crt \
  --key=tls.key

# Docker Registry
kubectl create secret docker-registry registry-secret \
  --docker-server=registry.example.com \
  --docker-username=admin \
  --docker-password=password
```

```yaml
# YAML 声明式
apiVersion: v1
kind: Secret
metadata:
  name: myapp-secret
type: Opaque
data:
  db_password: UzNjdXJlUEBzcw==     # echo -n 'S3cureP@ss' | base64
  api_key: YWJjMTIz
stringData:                           # 明文, 自动 base64 编码
  config.yaml: |
    database:
      password: S3cureP@ss

---
# TLS Secret
apiVersion: v1
kind: Secret
metadata:
  name: tls-secret
type: kubernetes.io/tls
data:
  tls.crt: <base64-cert>
  tls.key: <base64-key>
```

### 使用方式

```yaml
spec:
  # 镜像拉取凭证
  imagePullSecrets:
    - name: registry-secret

  containers:
    - name: app
      # 环境变量
      env:
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: myapp-secret
              key: db_password

      # 文件挂载
      volumeMounts:
        - name: secrets
          mountPath: /etc/secrets
          readOnly: true

  volumes:
    - name: secrets
      secret:
        secretName: myapp-secret
        defaultMode: 0400           # 权限 400 更安全
```

## 3. Secret 安全最佳实践？

**回答：**

```
Secret 安全问题:
  1. Base64 不是加密, 任何人 decode 就能看
  2. etcd 默认明文存储
  3. 有 RBAC 权限的人可以读取
  4. YAML 文件中可能泄露

安全措施:

1. 启用 etcd 静态加密
   # /etc/kubernetes/enc/encryption-config.yaml
   apiVersion: apiserver.config.k8s.io/v1
   kind: EncryptionConfiguration
   resources:
     - resources: ["secrets"]
       providers:
         - aescbc:
             keys:
               - name: key1
                 secret: <base64-encoded-32-byte-key>
         - identity: {}
   
   # kube-apiserver 参数
   --encryption-provider-config=/etc/kubernetes/enc/encryption-config.yaml

2. RBAC 限制 Secret 访问
   只授权必要的 ServiceAccount 读取特定 Secret

3. 不在 Git 中存储 Secret YAML
   使用: Sealed Secrets / SOPS / Vault

4. 使用外部密钥管理
   → External Secrets Operator
   → HashiCorp Vault + Vault Agent
   → AWS Secrets Manager / GCP Secret Manager
```

## 4. External Secrets Operator？

**回答：**

```
External Secrets Operator (ESO) 从外部密钥管理系统同步到 K8s Secret

架构:
  外部密钥管理 (Vault/AWS/GCP/Azure)
        ↓ 定期同步
  ExternalSecret (CR)
        ↓ 创建/更新
  K8s Secret
        ↓
  Pod (env / volume)
```

```yaml
# 1. SecretStore — 配置外部系统连接
apiVersion: external-secrets.io/v1beta1
kind: SecretStore
metadata:
  name: aws-secret-store
spec:
  provider:
    aws:
      service: SecretsManager
      region: us-east-1
      auth:
        jwt:
          serviceAccountRef:
            name: external-secrets-sa

# 2. ExternalSecret — 定义同步规则
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: myapp-secret
spec:
  refreshInterval: 1h              # 同步间隔
  secretStoreRef:
    name: aws-secret-store
    kind: SecretStore
  target:
    name: myapp-secret              # 生成的 K8s Secret 名称
    creationPolicy: Owner
  data:
    - secretKey: db_password        # K8s Secret 中的 key
      remoteRef:
        key: prod/myapp/db          # 外部密钥路径
        property: password          # 外部密钥中的属性
    - secretKey: api_key
      remoteRef:
        key: prod/myapp/api
        property: key
```

## 5. Sealed Secrets (加密 Secret 存 Git)？

**回答：**

```
Sealed Secrets 由 Bitnami 开发
将 Secret 加密为 SealedSecret, 可以安全存入 Git

流程:
  1. kubeseal CLI 用集群公钥加密 Secret → SealedSecret
  2. SealedSecret 存入 Git
  3. 集群中 Sealed Secrets Controller 用私钥解密 → K8s Secret
```

```bash
# 安装 Controller
kubectl apply -f https://github.com/bitnami-labs/sealed-secrets/releases/download/v0.25.0/controller.yaml

# 安装 kubeseal CLI
# 创建普通 Secret
kubectl create secret generic myapp-secret \
  --from-literal=password=S3cure --dry-run=client -o yaml > secret.yaml

# 加密为 SealedSecret
kubeseal --format yaml < secret.yaml > sealed-secret.yaml

# sealed-secret.yaml 可以安全提交到 Git
cat sealed-secret.yaml
```

```yaml
# SealedSecret (加密后, 安全)
apiVersion: bitnami.com/v1alpha1
kind: SealedSecret
metadata:
  name: myapp-secret
  namespace: production
spec:
  encryptedData:
    password: AgBxxxxxxxxxxxxxxxxxxxxxxxxxxxx...
  template:
    metadata:
      name: myapp-secret
    type: Opaque
```

## 6. ConfigMap 与 Secret 的使用对比？

**回答：**

```
对比:
  特性          ConfigMap           Secret
  ──────────    ──────────────────  ──────────────────
  用途          非敏感配置          敏感数据
  存储          明文                Base64 (可加密)
  大小限制      1 MB                1 MB
  使用方式      env / volume        env / volume
  RBAC          通常较宽松          应严格限制
  etcd 加密     不需要              建议开启
  挂载权限      0644                建议 0400

最佳实践:
  数据库主机/端口     → ConfigMap
  数据库密码           → Secret
  应用配置文件         → ConfigMap
  TLS 证书             → Secret (kubernetes.io/tls)
  镜像拉取凭证         → Secret (dockerconfigjson)
  API 密钥              → Secret + External Secrets
  非敏感环境变量       → ConfigMap
  功能开关             → ConfigMap
```

## 7. Kustomize 配置管理？

**回答：**

```
Kustomize 是 kubectl 内置的配置管理工具
无需模板语言, 通过 overlay 方式自定义 K8s 资源

目录结构:
  myapp/
  ├── base/
  │   ├── kustomization.yaml
  │   ├── deployment.yaml
  │   └── service.yaml
  ├── overlays/
  │   ├── dev/
  │   │   ├── kustomization.yaml
  │   │   └── patch-replicas.yaml
  │   └── prod/
  │       ├── kustomization.yaml
  │       ├── patch-replicas.yaml
  │       └── patch-resources.yaml
```

```yaml
# base/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - deployment.yaml
  - service.yaml
commonLabels:
  app: myapp

# overlays/prod/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - ../../base
namespace: production
namePrefix: prod-
commonLabels:
  env: production
patches:
  - path: patch-replicas.yaml
  - path: patch-resources.yaml
configMapGenerator:
  - name: myapp-config
    literals:
      - LOG_LEVEL=WARN
secretGenerator:
  - name: myapp-secret
    literals:
      - DB_PASSWORD=prod-password
images:
  - name: myapp
    newTag: v1.2.3

# overlays/prod/patch-replicas.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 5
```

```bash
# 预览
kubectl kustomize overlays/prod/
# 或
kustomize build overlays/prod/

# 部署
kubectl apply -k overlays/prod/
```

## 8. Helm 与配置管理？

**回答：**

```
Helm 是 K8s 的包管理器, 使用 Go 模板生成 K8s YAML

核心概念:
  Chart     — 应用包 (模板 + 默认值)
  Release   — Chart 的一次安装实例
  Values    — 配置值 (覆盖默认值)
  Repository — Chart 仓库
```

```yaml
# values.yaml (默认配置)
replicaCount: 3
image:
  repository: myapp
  tag: "1.0"
  pullPolicy: IfNotPresent
service:
  type: ClusterIP
  port: 80
resources:
  requests:
    cpu: 250m
    memory: 256Mi
  limits:
    cpu: 500m
    memory: 512Mi
env:
  LOG_LEVEL: INFO

# templates/deployment.yaml (模板)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ include "myapp.fullname" . }}
spec:
  replicas: {{ .Values.replicaCount }}
  template:
    spec:
      containers:
        - name: {{ .Chart.Name }}
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
          resources:
            {{- toYaml .Values.resources | nindent 12 }}
```

```bash
# 安装
helm install myapp ./mychart -f prod-values.yaml

# 升级
helm upgrade myapp ./mychart -f prod-values.yaml

# 回滚
helm rollback myapp 1

# 查看渲染结果
helm template myapp ./mychart -f prod-values.yaml
```

### Kustomize vs Helm

```
特性          Kustomize          Helm
──────────    ──────────────     ──────────────
方式          Overlay 覆盖       模板引擎
模板语言      无 (纯 YAML)       Go Template
学习曲线      低                 中
复杂度        简单定制           复杂应用打包
社区 Charts   无                 丰富
回滚          kubectl            helm rollback
推荐场景      内部应用定制       应用分发/复杂部署
```

## 9. 环境变量注入的各种方式？

**回答：**

```yaml
spec:
  containers:
    - name: app
      # 1. 直接设置
      env:
        - name: APP_NAME
          value: "myapp"

        # 2. 从 ConfigMap
        - name: DB_HOST
          valueFrom:
            configMapKeyRef:
              name: myapp-config
              key: db_host

        # 3. 从 Secret
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: myapp-secret
              key: db_password

        # 4. 从 Pod 字段 (Downward API)
        - name: POD_NAME
          valueFrom:
            fieldRef:
              fieldPath: metadata.name
        - name: POD_NAMESPACE
          valueFrom:
            fieldRef:
              fieldPath: metadata.namespace
        - name: POD_IP
          valueFrom:
            fieldRef:
              fieldPath: status.podIP
        - name: NODE_NAME
          valueFrom:
            fieldRef:
              fieldPath: spec.nodeName

        # 5. 从容器资源
        - name: CPU_LIMIT
          valueFrom:
            resourceFieldRef:
              containerName: app
              resource: limits.cpu
        - name: MEMORY_REQUEST
          valueFrom:
            resourceFieldRef:
              containerName: app
              resource: requests.memory

      # 6. 批量注入 ConfigMap
      envFrom:
        - configMapRef:
            name: myapp-config
          prefix: CFG_              # 可选前缀

      # 7. 批量注入 Secret
      envFrom:
        - secretRef:
            name: myapp-secret
          prefix: SEC_
```

## 10. 配置管理最佳实践？

**回答：**

```
1. 配置与代码分离
   不要在镜像中硬编码配置
   使用 ConfigMap + Secret 注入

2. 敏感数据用 Secret
   非敏感 → ConfigMap
   敏感   → Secret + 外部密钥管理

3. 不在 Git 中存储明文 Secret
   使用 Sealed Secrets / SOPS / External Secrets

4. 使用 immutable ConfigMap/Secret
   不会变的配置设 immutable: true
   减少 API Server Watch 压力

5. 配置变更自动重启
   使用 Reloader 或 configmap hash annotation

6. 多环境管理
   Kustomize overlays (简单)
   Helm values (复杂)
   
7. 版本化
   ConfigMap 名称加 hash: myapp-config-abc123
   Kustomize/Helm 自动处理

8. 最小权限
   RBAC 限制 Secret 访问
   只授权需要的 ServiceAccount

9. 关注大小限制
   ConfigMap/Secret 最大 1MB
   大文件考虑: Volume 挂载外部存储

10. 配置校验
    CI 中 lint YAML
    使用 Admission Webhook 校验配置格式
```
