# 工作负载控制器

---

## 1. Deployment 详解？

**回答：**

```
Deployment 是最常用的工作负载控制器, 管理无状态应用

层级关系:
  Deployment → 管理 ReplicaSet → 管理 Pod

  Deployment (myapp)
    ├─ ReplicaSet (myapp-abc123)  ← 当前版本
    │    ├─ Pod (myapp-abc123-x1)
    │    ├─ Pod (myapp-abc123-x2)
    │    └─ Pod (myapp-abc123-x3)
    └─ ReplicaSet (myapp-def456)  ← 旧版本 (replicas=0, 用于回滚)
```

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
  namespace: production
  labels:
    app: myapp
spec:
  replicas: 3
  revisionHistoryLimit: 10         # 保留的 ReplicaSet 历史版本数
  selector:
    matchLabels:
      app: myapp
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1                  # 更新时最多多出几个 Pod
      maxUnavailable: 0            # 更新时最多不可用几个 Pod
  template:
    metadata:
      labels:
        app: myapp
        version: "1.0"
    spec:
      containers:
        - name: myapp
          image: myapp:1.0
          ports:
            - containerPort: 8080
          resources:
            requests:
              cpu: "250m"
              memory: "256Mi"
            limits:
              cpu: "500m"
              memory: "512Mi"
          livenessProbe:
            httpGet:
              path: /health
              port: 8080
          readinessProbe:
            httpGet:
              path: /ready
              port: 8080
```

### 更新策略

```
RollingUpdate (默认):
  逐步替换旧 Pod, 零停机

  maxSurge + maxUnavailable 控制更新速度:

  maxSurge    maxUnavailable  效果
  ──────────  ──────────────  ────────────────────────
  1           0               最保守: 先创建新 Pod, 再删旧 Pod
  0           1               先删旧 Pod, 再创建新 Pod
  25%         25%             默认, 按比例
  2           2               较快更新

Recreate:
  先删除所有旧 Pod, 再创建新 Pod
  会有停机时间, 适用于不能同时运行新旧版本的场景

strategy:
  type: Recreate
```

## 2. 滚动更新与回滚？

**回答：**

```bash
# ===== 触发更新 =====
# 方式 1: 修改镜像
kubectl set image deployment/myapp myapp=myapp:2.0

# 方式 2: 编辑
kubectl edit deployment myapp

# 方式 3: apply 新 YAML
kubectl apply -f deployment-v2.yaml

# 方式 4: patch
kubectl patch deployment myapp -p '{"spec":{"template":{"spec":{"containers":[{"name":"myapp","image":"myapp:2.0"}]}}}}'

# ===== 查看更新状态 =====
kubectl rollout status deployment/myapp
# Waiting for rollout to finish: 1 out of 3 new replicas have been updated...
# deployment "myapp" successfully rolled out

# ===== 更新历史 =====
kubectl rollout history deployment/myapp
# REVISION  CHANGE-CAUSE
# 1         <none>
# 2         kubectl set image deployment/myapp myapp=myapp:2.0

# 查看某个版本详情
kubectl rollout history deployment/myapp --revision=2

# ===== 回滚 =====
kubectl rollout undo deployment/myapp               # 回滚到上一版本
kubectl rollout undo deployment/myapp --to-revision=1 # 回滚到指定版本

# ===== 暂停/恢复（多次修改合并一次更新） =====
kubectl rollout pause deployment/myapp
kubectl set image deployment/myapp myapp=myapp:2.0
kubectl set resources deployment/myapp -c=myapp --limits=cpu=500m
kubectl rollout resume deployment/myapp

# ===== 记录 CHANGE-CAUSE =====
kubectl annotate deployment/myapp kubernetes.io/change-cause="Update to v2.0"
```

### 滚动更新过程

```
初始状态: RS-v1 (3个Pod)

Step 1: 创建新 RS-v2, 启动 1 个 Pod
  RS-v1: 3 Pods (Running)
  RS-v2: 1 Pod  (Starting)

Step 2: v2 Pod Ready 后, 缩减 v1
  RS-v1: 2 Pods (Running)
  RS-v2: 1 Pod  (Running)

Step 3: 继续...
  RS-v1: 1 Pod  (Running)
  RS-v2: 2 Pods (Running)

Step 4: 完成
  RS-v1: 0 Pods (保留用于回滚)
  RS-v2: 3 Pods (Running)
```

## 3. ReplicaSet 详解？

**回答：**

```
ReplicaSet 确保在任何时刻运行指定数量的 Pod 副本

通常不直接使用, 由 Deployment 管理

ReplicaSet vs ReplicationController:
  ReplicaSet → 集合选择器 (matchExpressions)
  ReplicationController → 等式选择器 (已废弃)
```

```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: myapp-rs
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
    matchExpressions:
      - key: env
        operator: In
        values: ["production", "staging"]
  template:
    metadata:
      labels:
        app: myapp
        env: production
    spec:
      containers:
        - name: myapp
          image: myapp:1.0
```

```bash
# 扩缩容
kubectl scale replicaset myapp-rs --replicas=5

# 查看
kubectl get rs
kubectl describe rs myapp-rs

# Pod 被 ReplicaSet 管理的标识:
kubectl get pod myapp-rs-xxxxx -o yaml | grep ownerReferences
# ownerReferences:
# - apiVersion: apps/v1
#   kind: ReplicaSet
#   name: myapp-rs
```

## 4. StatefulSet 详解？

**回答：**

```
StatefulSet 管理有状态应用, 提供:

1. 稳定的网络标识
   Pod 名称固定: <statefulset-name>-<ordinal>
   例: mysql-0, mysql-1, mysql-2

2. 稳定的持久存储
   每个 Pod 绑定自己的 PVC, 不会因 Pod 重建而丢失

3. 有序部署和终止
   创建: mysql-0 → mysql-1 → mysql-2 (顺序)
   删除: mysql-2 → mysql-1 → mysql-0 (逆序)

4. 有序滚动更新
   逆序更新: mysql-2 → mysql-1 → mysql-0

StatefulSet vs Deployment:
  特性          Deployment        StatefulSet
  ──────────    ──────────────    ──────────────
  Pod 名称      随机后缀          固定序号
  网络标识      不稳定            稳定(Headless Service)
  持久存储      共享 PVC          每个 Pod 独立 PVC
  部署顺序      并行              有序
  适用场景      无状态应用        数据库、消息队列
```

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql
spec:
  serviceName: mysql-headless      # 必须关联 Headless Service
  replicas: 3
  podManagementPolicy: OrderedReady  # OrderedReady(默认) / Parallel
  updateStrategy:
    type: RollingUpdate
    rollingUpdate:
      partition: 0                 # 只更新序号 >= partition 的 Pod
  selector:
    matchLabels:
      app: mysql
  template:
    metadata:
      labels:
        app: mysql
    spec:
      containers:
        - name: mysql
          image: mysql:8.0
          ports:
            - containerPort: 3306
          env:
            - name: MYSQL_ROOT_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: mysql-secret
                  key: root-password
          volumeMounts:
            - name: data
              mountPath: /var/lib/mysql
  volumeClaimTemplates:            # 每个 Pod 自动创建 PVC
    - metadata:
        name: data
      spec:
        accessModes: ["ReadWriteOnce"]
        storageClassName: fast-ssd
        resources:
          requests:
            storage: 20Gi

---
# 必须配合 Headless Service
apiVersion: v1
kind: Service
metadata:
  name: mysql-headless
spec:
  clusterIP: None                  # Headless Service
  selector:
    app: mysql
  ports:
    - port: 3306

# DNS 记录:
#   mysql-0.mysql-headless.default.svc.cluster.local
#   mysql-1.mysql-headless.default.svc.cluster.local
#   mysql-2.mysql-headless.default.svc.cluster.local
```

## 5. DaemonSet 详解？

**回答：**

```
DaemonSet 确保在每个(或指定) Node 上运行一个 Pod 副本

典型用途:
  - 日志收集: Fluentd, Fluent Bit, Filebeat
  - 监控代理: Node Exporter, Datadog Agent
  - 网络插件: Calico, Cilium, Flannel (kube-proxy)
  - 存储守护: Ceph, GlusterFS

特点:
  - 新 Node 加入集群自动创建 Pod
  - Node 被删除时 Pod 自动回收
  - 删除 DaemonSet 时清理所有 Pod
```

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: node-exporter
  namespace: monitoring
spec:
  selector:
    matchLabels:
      app: node-exporter
  updateStrategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1           # 更新时最多几个不可用
      maxSurge: 0
  template:
    metadata:
      labels:
        app: node-exporter
    spec:
      hostNetwork: true           # 使用宿主机网络
      hostPID: true
      tolerations:
        - operator: Exists        # 容忍所有污点(包括 Master)
      containers:
        - name: node-exporter
          image: prom/node-exporter:latest
          ports:
            - containerPort: 9100
              hostPort: 9100
          resources:
            requests:
              cpu: 100m
              memory: 128Mi
            limits:
              cpu: 200m
              memory: 256Mi
          volumeMounts:
            - name: proc
              mountPath: /host/proc
              readOnly: true
            - name: sys
              mountPath: /host/sys
              readOnly: true
      volumes:
        - name: proc
          hostPath:
            path: /proc
        - name: sys
          hostPath:
            path: /sys
```

```bash
# 只在特定 Node 运行 DaemonSet
# 方式 1: nodeSelector
spec:
  template:
    spec:
      nodeSelector:
        node-type: worker

# 方式 2: nodeAffinity
spec:
  template:
    spec:
      affinity:
        nodeAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
            nodeSelectorTerms:
              - matchExpressions:
                  - key: node-type
                    operator: In
                    values: ["worker"]
```

## 6. Job 与 CronJob 详解？

**回答：**

### Job

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: data-migration
spec:
  completions: 1                   # 需要成功完成的 Pod 数
  parallelism: 1                   # 并行运行的 Pod 数
  backoffLimit: 3                  # 失败重试次数
  activeDeadlineSeconds: 3600      # 超时时间(秒)
  ttlSecondsAfterFinished: 300     # 完成后自动清理时间
  template:
    spec:
      restartPolicy: OnFailure    # Job 只能是 OnFailure 或 Never
      containers:
        - name: migration
          image: myapp:1.0
          command: ["python", "migrate.py"]
          env:
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: db-secret
                  key: url
```

```
Job 模式:
  completions  parallelism  描述
  ───────────  ───────────  ──────────────────────
  1            1            单个任务（默认）
  N            1            串行执行 N 次
  N            M            并行执行, 共完成 N 次
  -            N            工作队列模式(Indexed)

# Indexed Job (K8s 1.21+)
spec:
  completions: 5
  parallelism: 3
  completionMode: Indexed    # 每个 Pod 通过 JOB_COMPLETION_INDEX 获取索引
```

### CronJob

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: daily-backup
spec:
  schedule: "0 2 * * *"           # Cron 表达式 (每天凌晨2点)
  timeZone: "Asia/Shanghai"       # 时区 (K8s 1.27+)
  concurrencyPolicy: Forbid       # Allow / Forbid / Replace
  successfulJobsHistoryLimit: 3   # 保留成功 Job 数
  failedJobsHistoryLimit: 3       # 保留失败 Job 数
  startingDeadlineSeconds: 200    # 错过调度的截止时间
  suspend: false                  # 暂停
  jobTemplate:
    spec:
      template:
        spec:
          restartPolicy: OnFailure
          containers:
            - name: backup
              image: backup-tool:1.0
              command: ["./backup.sh"]

# Cron 表达式:
#   ┌──── 分钟 (0-59)
#   │ ┌── 小时 (0-23)
#   │ │ ┌ 日 (1-31)
#   │ │ │ ┌ 月 (1-12)
#   │ │ │ │ ┌ 星期 (0-6, 0=周日)
#   * * * * *
#
#   */5 * * * *     → 每5分钟
#   0 */2 * * *     → 每2小时
#   0 0 * * 0       → 每周日午夜
#   0 2 1 * *       → 每月1号凌晨2点

# concurrencyPolicy:
#   Allow   → 允许并发运行 (默认)
#   Forbid  → 如果上次未完成, 跳过本次
#   Replace → 如果上次未完成, 取消上次启动新的
```

## 7. Horizontal Pod Autoscaler (HPA) 详解？

**回答：**

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 2
  maxReplicas: 20
  metrics:
    # CPU 指标
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70     # CPU 使用率 > 70% 扩容

    # 内存指标
    - type: Resource
      resource:
        name: memory
        target:
          type: AverageValue
          averageValue: 500Mi

    # 自定义指标 (需要 Prometheus Adapter)
    - type: Pods
      pods:
        metric:
          name: http_requests_per_second
        target:
          type: AverageValue
          averageValue: 1000

    # 外部指标 (如消息队列长度)
    - type: External
      external:
        metric:
          name: queue_messages_ready
          selector:
            matchLabels:
              queue: worker-tasks
        target:
          type: AverageValue
          averageValue: 30

  behavior:
    scaleUp:
      stabilizationWindowSeconds: 0    # 扩容不等待
      policies:
        - type: Percent
          value: 100                   # 最多扩容 100%
          periodSeconds: 60
        - type: Pods
          value: 4                     # 最多一次扩 4 个
          periodSeconds: 60
      selectPolicy: Max                # 取最大策略
    scaleDown:
      stabilizationWindowSeconds: 300  # 缩容等 5 分钟
      policies:
        - type: Percent
          value: 10                    # 最多缩 10%
          periodSeconds: 60
```

```bash
# 前提: 安装 Metrics Server
kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml

# 命令行创建 HPA
kubectl autoscale deployment myapp --cpu-percent=70 --min=2 --max=10

# 查看 HPA 状态
kubectl get hpa
# NAME       REFERENCE         TARGETS   MINPODS   MAXPODS   REPLICAS
# myapp-hpa  Deployment/myapp  45%/70%   2         20        3

kubectl describe hpa myapp-hpa
```

## 8. Vertical Pod Autoscaler (VPA)？

**回答：**

```
VPA 自动调整 Pod 的 CPU/Memory requests 和 limits

HPA vs VPA:
  HPA → 水平扩展, 增加 Pod 数量
  VPA → 垂直扩展, 调整单个 Pod 资源

注意: HPA 和 VPA 不要同时基于 CPU/Memory
      可以 HPA 用自定义指标 + VPA 调 resources
```

```yaml
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: myapp-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  updatePolicy:
    updateMode: "Auto"            # Off / Initial / Recreate / Auto
  resourcePolicy:
    containerPolicies:
      - containerName: myapp
        minAllowed:
          cpu: 100m
          memory: 128Mi
        maxAllowed:
          cpu: 4
          memory: 8Gi
        controlledResources: ["cpu", "memory"]

# updateMode:
#   Off       → 只推荐, 不自动调整
#   Initial   → 只在 Pod 创建时设置
#   Auto      → 自动重建 Pod 并调整资源
```

## 9. Pod Disruption Budget (PDB)？

**回答：**

```
PDB 限制自愿中断时同时不可用的 Pod 数量

自愿中断:
  - Node 维护 (kubectl drain)
  - 集群升级
  - 自动伸缩缩容
  - Deployment 滚动更新

非自愿中断 (PDB 不管):
  - 硬件故障
  - Node 内核 panic
  - Pod OOMKilled
```

```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: myapp-pdb
spec:
  # 二选一:
  minAvailable: 2                 # 最少可用 Pod 数
  # maxUnavailable: 1             # 最多不可用 Pod 数 (百分比也行: "25%")
  selector:
    matchLabels:
      app: myapp
  unhealthyPodEvictionPolicy: IfHealthy  # 可选, K8s 1.27+
```

```bash
# 查看 PDB
kubectl get pdb
# NAME        MIN AVAILABLE   MAX UNAVAILABLE   ALLOWED DISRUPTIONS   AGE
# myapp-pdb   2               N/A               1                     1d

# drain Node 时如果违反 PDB, 操作会阻塞
kubectl drain node1 --ignore-daemonsets --delete-emptydir-data
# error: Cannot evict pod as it would violate the pod's disruption budget.
```

## 10. 工作负载控制器选型？

**回答：**

```
应用类型          推荐控制器       说明
────────────────  ──────────────   ─────────────────────────
Web API 服务      Deployment       无状态, 随意扩缩, 滚动更新
前端应用          Deployment       无状态, 配合 Ingress
数据库            StatefulSet      稳定网络标识 + 持久存储
消息队列          StatefulSet      有序部署, 独立存储
(Kafka/RabbitMQ)
日志收集          DaemonSet        每个节点运行一份
监控 Agent        DaemonSet        每个节点运行一份
CNI 网络插件      DaemonSet        每个节点运行一份
数据迁移          Job              一次性执行
ETL 任务          Job              批处理
定时备份          CronJob          周期性执行
定时清理          CronJob          周期性执行

扩缩容选型:
  水平扩展(加 Pod) → HPA (基于 CPU/自定义指标)
  垂直扩展(加资源) → VPA (调整 requests/limits)
  集群扩展(加 Node) → Cluster Autoscaler

高可用:
  PDB → 保证维护时最少可用 Pod 数
  podAntiAffinity → 分散到不同 Node/Zone
  topologySpreadConstraints → 均匀分布

更新策略:
  RollingUpdate (默认) → 零停机
  Recreate → 不能共存新旧版本
  蓝绿/金丝雀 → Argo Rollouts / Flagger
```
