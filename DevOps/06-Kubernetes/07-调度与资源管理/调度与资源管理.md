# 调度与资源管理

---

## 1. K8s 调度器工作原理？

**回答：**

```
调度流程:

  未调度 Pod (spec.nodeName 为空)
       │
       ▼
  ① 过滤 (Filtering) — 排除不满足条件的 Node
       │   NodeResourcesFit    — 资源是否充足
       │   NodePorts           — HostPort 是否冲突
       │   PodToleratesNodeTaints — Taint/Toleration
       │   NodeAffinity        — 节点亲和性
       │   PodTopologySpread   — 拓扑约束
       │   VolumeBinding       — 存储是否可用
       ▼
  ② 打分 (Scoring) — 为满足条件的 Node 评分
       │   LeastAllocated       — 资源剩余越多分越高
       │   MostAllocated        — 资源使用越满分越高(装箱)
       │   BalancedAllocation   — CPU/Memory 利用率越均衡分越高
       │   ImageLocality        — 镜像已存在加分
       │   InterPodAffinity     — Pod 亲和性打分
       │   TaintToleration      — 容忍的污点越少分越高
       ▼
  ③ 绑定 (Binding) — 选择最高分 Node
       │   更新 Pod.spec.nodeName
       │   通知 kubelet
       ▼
  Node 上的 kubelet 创建 Pod
```

### 调度框架 (Scheduling Framework)

```
K8s 1.19+ 使用插件化调度框架, 每个阶段可插入自定义逻辑

扩展点:
  PreFilter  → Filter → PostFilter → PreScore → Score → Reserve →
  Permit → PreBind → Bind → PostBind

自定义调度器:
  可以运行多个调度器, Pod 通过 schedulerName 指定

spec:
  schedulerName: my-custom-scheduler
```

## 2. Taint 与 Toleration 详解？

**回答：**

```
Taint 标记在 Node 上, 排斥 Pod
Toleration 标记在 Pod 上, 容忍特定 Taint

Effect 类型:
  NoSchedule         → 不调度新 Pod (已运行的不影响)
  PreferNoSchedule   → 尽量不调度 (软约束)
  NoExecute          → 不调度 + 驱逐已运行的 Pod
```

```bash
# 添加 Taint
kubectl taint nodes node1 dedicated=gpu:NoSchedule
kubectl taint nodes node1 maintenance=true:NoExecute

# 查看 Node 的 Taint
kubectl describe node node1 | grep Taints

# 删除 Taint (末尾加 -)
kubectl taint nodes node1 dedicated=gpu:NoSchedule-
kubectl taint nodes node1 dedicated-           # 删除 key 的所有 Taint
```

```yaml
# Pod 添加 Toleration
spec:
  tolerations:
    # Equal 操作符: key + value + effect 完全匹配
    - key: "dedicated"
      operator: "Equal"
      value: "gpu"
      effect: "NoSchedule"

    # Exists 操作符: 只匹配 key (忽略 value)
    - key: "maintenance"
      operator: "Exists"
      effect: "NoExecute"
      tolerationSeconds: 3600     # NoExecute 时, 容忍多久后被驱逐

    # 容忍所有 Taint (DaemonSet 常用)
    - operator: "Exists"
```

```
内置 Taint (K8s 自动添加):
  node.kubernetes.io/not-ready         → Node NotReady
  node.kubernetes.io/unreachable       → Node 不可达
  node.kubernetes.io/memory-pressure   → 内存压力
  node.kubernetes.io/disk-pressure     → 磁盘压力
  node.kubernetes.io/pid-pressure      → PID 压力
  node.kubernetes.io/unschedulable     → kubectl cordon
  node.kubernetes.io/network-unavailable → 网络不可用

  默认 Pod 容忍 not-ready 和 unreachable 300s
  (tolerationSeconds: 300)
```

## 3. Node Affinity 详解？

**回答：**

```yaml
spec:
  affinity:
    nodeAffinity:
      # 硬要求: 必须满足 (类似 nodeSelector 增强版)
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
          - matchExpressions:            # 多个条件 AND
              - key: topology.kubernetes.io/zone
                operator: In
                values: ["us-east-1a", "us-east-1b"]
              - key: node.kubernetes.io/instance-type
                operator: In
                values: ["m5.xlarge", "m5.2xlarge"]
          # 多个 nodeSelectorTerms 是 OR 关系

      # 软偏好: 尽量满足
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 80                     # 权重 1-100
          preference:
            matchExpressions:
              - key: disk-type
                operator: In
                values: ["ssd"]
        - weight: 20
          preference:
            matchExpressions:
              - key: gpu
                operator: Exists

# Operator 选项:
#   In            → label value 在列表中
#   NotIn         → label value 不在列表中
#   Exists        → label key 存在
#   DoesNotExist  → label key 不存在
#   Gt            → label value > 指定值 (数值比较)
#   Lt            → label value < 指定值 (数值比较)
```

```
nodeSelector vs nodeAffinity:
  nodeSelector:
    简单, 只支持 key=value 完全匹配
    spec:
      nodeSelector:
        disk-type: ssd

  nodeAffinity:
    强大, 支持 In/NotIn/Exists/Gt/Lt
    支持硬要求 + 软偏好
    支持权重
```

## 4. Pod Affinity 与 Anti-Affinity 详解？

**回答：**

```yaml
spec:
  affinity:
    # Pod Affinity: 与特定 Pod 调度到一起
    podAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        - labelSelector:
            matchExpressions:
              - key: app
                operator: In
                values: ["cache"]
          topologyKey: kubernetes.io/hostname  # 同一 Node
          # topologyKey: topology.kubernetes.io/zone  # 同一 Zone
          namespaces: ["default"]               # 可选, 指定命名空间

    # Pod Anti-Affinity: 避开特定 Pod (分散部署)
    podAntiAffinity:
      # 硬要求: 同一应用的 Pod 不在同一 Node
      requiredDuringSchedulingIgnoredDuringExecution:
        - labelSelector:
            matchExpressions:
              - key: app
                operator: In
                values: ["myapp"]
          topologyKey: kubernetes.io/hostname

      # 软偏好: 尽量不在同一 Zone
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 100
          podAffinityTerm:
            labelSelector:
              matchExpressions:
                - key: app
                  operator: In
                  values: ["myapp"]
            topologyKey: topology.kubernetes.io/zone
```

```
topologyKey 常用值:
  kubernetes.io/hostname              → 同一 Node
  topology.kubernetes.io/zone         → 同一可用区
  topology.kubernetes.io/region       → 同一区域

典型场景:
  1. Web + Cache 同一 Node (podAffinity + hostname)
  2. 同一应用分散到不同 Node (podAntiAffinity + hostname)
  3. 同一应用跨 Zone 部署 (podAntiAffinity + zone)
```

## 5. Topology Spread Constraints 拓扑分布约束？

**回答：**

```yaml
# 控制 Pod 在拓扑域中的分布, 比 Anti-Affinity 更灵活
spec:
  topologySpreadConstraints:
    # 跨 Zone 均匀分布
    - maxSkew: 1                           # 最大偏差
      topologyKey: topology.kubernetes.io/zone
      whenUnsatisfied: DoNotSchedule       # DoNotSchedule / ScheduleAnyway
      labelSelector:
        matchLabels:
          app: myapp
      matchLabelKeys:
        - pod-template-hash               # 滚动更新时只看新版本
      minDomains: 3                        # 最小拓扑域数 (K8s 1.25+)

    # 跨 Node 均匀分布
    - maxSkew: 1
      topologyKey: kubernetes.io/hostname
      whenUnsatisfied: ScheduleAnyway      # 尽量均匀, 不强制
      labelSelector:
        matchLabels:
          app: myapp
```

```
示例: 3 个 Zone, 6 个 Pod, maxSkew: 1

均匀分布:
  zone-a: 2 Pods
  zone-b: 2 Pods
  zone-c: 2 Pods
  最大偏差 = 2-2 = 0 ✓ (≤ maxSkew 1)

不均匀:
  zone-a: 3 Pods
  zone-b: 2 Pods
  zone-c: 1 Pod
  最大偏差 = 3-1 = 2 ✗ (> maxSkew 1)

vs podAntiAffinity:
  Anti-Affinity: "不要和某些 Pod 在一起" (二元)
  TopologySpread: "均匀分散到各域" (量化控制)
```

## 6. 资源管理 requests 与 limits？

**回答：**

```yaml
containers:
  - name: app
    resources:
      requests:              # 调度和保证的资源量
        cpu: "250m"          # 250 毫核 = 0.25 核
        memory: "256Mi"      # 256 MiB
      limits:                # 最大使用量
        cpu: "1000m"         # 1 核
        memory: "1Gi"

# CPU 单位:
#   1 = 1核 = 1000m (毫核)
#   100m = 0.1 核
#   可以超过 limits (被 throttle, 不会被杀)

# Memory 单位:
#   Ki, Mi, Gi, Ti (二进制, 1Ki=1024)
#   K, M, G, T (十进制, 1K=1000)
#   超过 limits → OOMKilled
```

```
requests vs limits:

  requests:
    - 调度依据: Scheduler 用 requests 判断 Node 是否有足够资源
    - 保证资源: kubelet 确保 requests 的资源可用
    - QoS 分级: 决定 Pod 的 QoS 等级

  limits:
    - CPU: 超过 limits 被 throttle (CFS Quota)
    - Memory: 超过 limits 被 OOMKilled
    - 不影响调度

建议:
  requests 设为实际平均使用量
  limits 设为峰值使用量
  不设 limits 的风险: 资源争抢, 影响其他 Pod
  
  limits = requests → Guaranteed QoS (关键服务)
  limits > requests → Burstable QoS (一般服务)
  不设 resources     → BestEffort QoS (避免)
```

## 7. ResourceQuota 与 LimitRange？

**回答：**

### ResourceQuota — 限制 Namespace 资源总量

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: compute-quota
  namespace: production
spec:
  hard:
    # 计算资源
    requests.cpu: "20"
    requests.memory: 40Gi
    limits.cpu: "40"
    limits.memory: 80Gi

    # 对象数量
    pods: "100"
    services: "20"
    services.loadbalancers: "2"
    services.nodeports: "5"
    persistentvolumeclaims: "20"
    configmaps: "50"
    secrets: "50"
    replicationcontrollers: "20"

    # 存储
    requests.storage: 500Gi
    fast-ssd.storageclass.storage.k8s.io/requests.storage: 100Gi

# 多个 ResourceQuota 可以组合
# 有 ResourceQuota 时, Pod 必须设置 resources
```

### LimitRange — 限制单个 Pod/Container 资源

```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: resource-limits
  namespace: production
spec:
  limits:
    # Container 级别
    - type: Container
      default:                 # 默认 limits (未设置时)
        cpu: "500m"
        memory: "512Mi"
      defaultRequest:          # 默认 requests (未设置时)
        cpu: "100m"
        memory: "128Mi"
      max:                     # 最大值
        cpu: "4"
        memory: "8Gi"
      min:                     # 最小值
        cpu: "50m"
        memory: "64Mi"
      maxLimitRequestRatio:    # limits/requests 最大比值
        cpu: "10"
        memory: "4"

    # Pod 级别
    - type: Pod
      max:
        cpu: "8"
        memory: "16Gi"

    # PVC 级别
    - type: PersistentVolumeClaim
      max:
        storage: 100Gi
      min:
        storage: 1Gi
```

## 8. Priority 与 Preemption 优先级与抢占？

**回答：**

```
高优先级 Pod 调度不上时, 可以抢占(驱逐)低优先级 Pod

流程:
  1. 高优先级 Pod 无法调度
  2. Scheduler 寻找可以被抢占的低优先级 Pod
  3. 选择影响最小的 Node
  4. 驱逐低优先级 Pod
  5. 调度高优先级 Pod
```

```yaml
# 定义 PriorityClass
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: high-priority
value: 1000000                         # 优先级数值, 越大越高
globalDefault: false                   # 是否为默认
preemptionPolicy: PreemptLowerPriority # PreemptLowerPriority / Never
description: "高优先级, 可抢占"

---
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: low-priority
value: 1000
preemptionPolicy: Never                # 不会被用于抢占
description: "低优先级, 不可抢占"

# Pod 使用 PriorityClass
spec:
  priorityClassName: high-priority

# 内置 PriorityClass:
#   system-cluster-critical  (2000000000) — 集群关键组件
#   system-node-critical     (2000001000) — Node 关键组件
```

## 9. Node 管理 (cordon/drain/uncordon)？

**回答：**

```bash
# ===== cordon — 标记 Node 不可调度 =====
kubectl cordon node1
# Node 状态变为 SchedulingDisabled
# 已运行的 Pod 不受影响, 新 Pod 不会调度到此 Node

# ===== drain — 排空 Node =====
kubectl drain node1 \
  --ignore-daemonsets \          # 忽略 DaemonSet Pod
  --delete-emptydir-data \       # 删除使用 emptyDir 的 Pod
  --grace-period=60 \            # 优雅终止等待时间
  --timeout=300s \               # drain 超时
  --force                        # 强制(删除非控制器管理的 Pod)

# drain 做了什么:
#   1. cordon Node (标记不可调度)
#   2. 逐个驱逐 Pod (尊重 PDB)
#   3. 等待 Pod 在其他 Node 重建

# drain 失败常见原因:
#   PDB 阻止 — Pod 不能被驱逐
#   DaemonSet — 使用 --ignore-daemonsets
#   本地存储 — 使用 --delete-emptydir-data
#   非控制器管理的 Pod — 使用 --force

# ===== uncordon — 恢复可调度 =====
kubectl uncordon node1
```

```
常见运维场景:
  Node 维护 (内核升级、硬件更换):
    kubectl drain node1 --ignore-daemonsets --delete-emptydir-data
    # 执行维护操作
    kubectl uncordon node1

  Node 下线:
    kubectl drain node1 --ignore-daemonsets --force --delete-emptydir-data
    kubectl delete node node1
```

## 10. Cluster Autoscaler？

**回答：**

```
Cluster Autoscaler 自动增减集群中的 Node 数量

扩容条件:
  Pod 因资源不足处于 Pending 状态
  → CA 计算需要多少 Node
  → 调用云 API 创建新 Node

缩容条件:
  Node 利用率低于阈值 (默认 50%)
  Node 上的 Pod 都可以调度到其他 Node
  持续一段时间 (默认 10 分钟)
  → 驱逐 Pod → 删除 Node

不会被缩容的 Node:
  - 有非控制器管理的 Pod
  - 有 PDB 阻止驱逐的 Pod
  - 有 annotation: cluster-autoscaler.kubernetes.io/safe-to-evict=false
  - 有 local storage 的 Pod
  - kube-system 的 Pod (无 PDB)
```

```yaml
# AWS EKS 的 Cluster Autoscaler 配置
apiVersion: apps/v1
kind: Deployment
metadata:
  name: cluster-autoscaler
  namespace: kube-system
spec:
  template:
    spec:
      containers:
        - name: cluster-autoscaler
          image: registry.k8s.io/autoscaling/cluster-autoscaler:v1.29.0
          command:
            - ./cluster-autoscaler
            - --v=4
            - --cloud-provider=aws
            - --skip-nodes-with-local-storage=false
            - --expander=least-waste       # 扩容策略
            - --node-group-auto-discovery=asg:tag=k8s.io/cluster-autoscaler/enabled
            - --balance-similar-node-groups
            - --scale-down-utilization-threshold=0.5
            - --scale-down-delay-after-add=10m
            - --scale-down-unneeded-time=10m

# 替代方案: Karpenter (AWS)
# 比 Cluster Autoscaler 更快, 更灵活
# 直接按 Pod 需求创建最优 Node
```

```
扩缩容全景:

层级        工具                    扩展什么
──────────  ──────────────────────  ──────────
Pod 水平    HPA                     Pod 副本数
Pod 垂直    VPA                     Pod 资源配额
Node 层     Cluster Autoscaler      Node 数量
            Karpenter (AWS)

触发链:
  请求增加 → HPA 扩 Pod → Node 资源不足
  → Pod Pending → Cluster Autoscaler 扩 Node
  → Node Ready → Pod 调度成功
```
