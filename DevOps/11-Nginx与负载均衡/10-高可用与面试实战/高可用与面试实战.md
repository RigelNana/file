## 1. Nginx 高可用架构？

**回答：**

```
高可用方案:

方案 1: Keepalived + VIP (传统)
  ┌──────────────────────────────────┐
  │          VIP: 10.0.0.100          │
  │    ┌──────────┬──────────┐       │
  │    │ Nginx A  │ Nginx B  │       │
  │    │ (Master) │ (Backup) │       │
  │    │ VRRP     │ VRRP     │       │
  │    └──────────┴──────────┘       │
  │         ↓  心跳检测  ↓           │
  │    Master 挂 → VIP 漂移到 Backup  │
  └──────────────────────────────────┘

方案 2: 云厂商 LB + 多 Nginx
  Client → Cloud LB (ELB/ALB/SLB)
              ├→ Nginx 1
              ├→ Nginx 2
              └→ Nginx 3
  云 LB 自带健康检查和故障转移

方案 3: K8s Deployment + Service
  Client → Cloud LB → Service (ClusterIP)
                        ├→ Nginx Pod 1
                        ├→ Nginx Pod 2
                        └→ Nginx Pod 3
  K8s 自动重启失败 Pod, HPA 自动扩缩
```

## 2. Keepalived + Nginx 配置？

**回答：**

```bash
# ============ Keepalived 安装 ============
# Ubuntu
apt install keepalived

# CentOS
yum install keepalived
```

```
# ============ Master 节点配置 ============
# /etc/keepalived/keepalived.conf

vrrp_script check_nginx {
    script "/etc/keepalived/check_nginx.sh"
    interval 2          # 每 2 秒检查一次
    weight -20          # 检查失败时降低优先级 20
    fall 3              # 连续失败 3 次判定为失败
    rise 2              # 连续成功 2 次判定为恢复
}

vrrp_instance VI_1 {
    state MASTER         # Master 节点
    interface eth0       # 网卡
    virtual_router_id 51 # VRRP 组 ID (Master/Backup 必须一致)
    priority 100         # 优先级 (Master > Backup)
    advert_int 1         # VRRP 广播间隔 (秒)

    authentication {
        auth_type PASS
        auth_pass keepalived_secret
    }

    virtual_ipaddress {
        10.0.0.100/24    # VIP
    }

    track_script {
        check_nginx
    }
}
```

```bash
# ============ Nginx 健康检查脚本 ============
# /etc/keepalived/check_nginx.sh
#!/bin/bash
# 检查 Nginx 进程是否存在
if ! pgrep -x nginx > /dev/null; then
    # 尝试重启 Nginx
    systemctl restart nginx
    sleep 2
    if ! pgrep -x nginx > /dev/null; then
        exit 1   # 重启失败, 触发 VIP 切换
    fi
fi
exit 0

# chmod +x /etc/keepalived/check_nginx.sh
```

```
# ============ Backup 节点配置 ============
# 与 Master 相同, 只改两处:
#   state BACKUP
#   priority 90
```

## 3. Nginx 日志分析与排错？

**回答：**

```bash
# ============ 常用日志分析命令 ============

# 统计状态码分布
awk '{print $9}' /var/log/nginx/access.log | sort | uniq -c | sort -rn
#  52341 200
#   1234 301
#    456 404
#     23 502

# 统计最慢的 10 个请求
awk '{print $NF, $7}' /var/log/nginx/access.log | sort -rn | head -10

# 统计 QPS (每秒请求数)
awk '{print $4}' /var/log/nginx/access.log | cut -d: -f1-3 | uniq -c | sort -rn | head -10

# 统计来访 IP TOP 10
awk '{print $1}' /var/log/nginx/access.log | sort | uniq -c | sort -rn | head -10

# 查找 5xx 错误
grep ' 50[0-9] ' /var/log/nginx/access.log | tail -20

# 查找 502 错误 (后端不可用)
grep ' 502 ' /var/log/nginx/access.log | awk '{print $1, $7}' | sort | uniq -c | sort -rn

# 查找慢请求 (> 5s)
awk '$NF > 5 {print $0}' /var/log/nginx/access.log

# ============ error.log 关键错误 ============

# 连接后端失败
# upstream connect() failed (111: Connection refused)
# → 后端服务未启动或端口错误

# 后端超时
# upstream timed out (110: Connection timed out)
# → proxy_read_timeout 太短或后端处理太慢

# 打开文件过多
# socket() failed (24: Too many open files)
# → 增大 worker_rlimit_nofile

# 上传太大
# client intended to send too large body
# → 增大 client_max_body_size
```

## 4. 面试题：请求 URL 到页面展示全过程？

**回答：**

```
用户输入 https://www.example.com/api/users 到看到结果:

1. DNS 解析
   浏览器缓存 → OS 缓存 → 路由器缓存 → ISP DNS → 递归查询
   → 返回 IP 地址 (如 203.0.113.10)

2. TCP 三次握手
   Client → SYN → Server
   Client ← SYN+ACK ← Server
   Client → ACK → Server

3. TLS 握手 (HTTPS)
   ClientHello → ServerHello → Certificate → KeyExchange
   → 协商密钥 → 加密通道建立
   (TLS 1.3: 1-RTT, TLS 1.2: 2-RTT)

4. 到达 CDN / 负载均衡器
   DNS 通常解析到 CDN 或云 LB

5. Nginx 处理
   a. 接收请求 (epoll 事件通知)
   b. 解析 HTTP 请求头
   c. 匹配 server_name → 找到虚拟主机
   d. 匹配 location → 确定处理方式
   e. 执行各阶段: 限流→认证→重写→代理

6. 反向代理到后端
   a. 从 upstream 选择后端 (负载均衡算法)
   b. 与后端建立连接 (复用 keepalive)
   c. 转发请求 (修改 Header)
   d. 等待后端响应

7. 后端处理
   API 服务器处理业务逻辑, 查询数据库等

8. Nginx 返回响应
   a. 接收后端响应
   b. 缓冲/直传给客户端
   c. 添加安全头
   d. Gzip 压缩
   e. 记录访问日志

9. 浏览器渲染
   解析 HTML → 加载 CSS/JS → 构建 DOM → 渲染页面
```

## 5. 面试题：如何排查 502 Bad Gateway？

**回答：**

```
502 = Nginx 无法从后端获取有效响应

排查步骤:

Step 1: 确认范围
  □ 所有请求 502 还是部分?
  □ 所有后端还是特定后端?
  □ 何时开始的? 是否有变更?

Step 2: 检查 Nginx error.log
  tail -f /var/log/nginx/error.log

  常见错误:
  "upstream connect() failed (111: Connection refused)"
    → 后端未运行或端口错误
  "upstream timed out"
    → 后端处理超时
  "no live upstreams"
    → 所有后端都被标记为不可用

Step 3: 检查后端服务
  # 后端进程是否运行
  ps aux | grep <backend>
  
  # 后端端口是否监听
  ss -tlnp | grep :8080
  
  # 直接请求后端 (绕过 Nginx)
  curl -v http://127.0.0.1:8080/api/health

Step 4: 检查网络
  # Nginx 到后端是否通
  telnet 10.0.1.101 8080
  curl -v http://10.0.1.101:8080/

Step 5: 检查资源
  # 后端内存是否 OOM
  dmesg | grep -i oom
  
  # 文件描述符是否耗尽
  ls /proc/<pid>/fd | wc -l
  
  # 端口是否耗尽
  ss -s | grep TIME-WAIT

Step 6: 检查 Nginx 配置
  # upstream 配置是否正确
  nginx -T | grep -A 20 'upstream'
  
  # 超时是否太短
  proxy_connect_timeout
  proxy_read_timeout

常见原因总结:
  1. 后端服务挂了 (最常见)
  2. 后端端口错误
  3. 后端 OOM 被杀
  4. 后端处理超时
  5. 上游连接数耗尽
  6. DNS 解析失败 (upstream 用域名时)
```

## 6. 面试题：Nginx 和 LVS 的区别？

**回答：**

```
LVS (Linux Virtual Server):
  ┌──────────────────┬────────────────┬────────────────┐
  │ 维度              │ LVS            │ Nginx          │
  ├──────────────────┼────────────────┼────────────────┤
  │ 工作层            │ L4 (传输层)    │ L7 (应用层)    │
  │ 性能              │ 极高 (内核级)  │ 高 (用户态)    │
  │ 吞吐量            │ 百万级 QPS     │ 十万级 QPS     │
  │ 功能              │ 简单转发       │ 丰富 (路由,    │
  │                  │               │  缓存, SSL 等)  │
  │ 健康检查          │ 需要 keepalived│ 内置被动检查   │
  │ URL 路由          │ ✗             │ ✓              │
  │ SSL 终止          │ ✗             │ ✓              │
  │ 缓存              │ ✗             │ ✓              │
  │ 会话保持          │ 基于 IP       │ Cookie/Header  │
  │ 部署方式          │ 内核模块       │ 用户态进程     │
  └──────────────────┴────────────────┴────────────────┘

LVS 转发模式:
  NAT:  修改目标 IP (回包经过 LVS, 小规模)
  DR:   修改 MAC (回包直连客户端, 高性能)
  TUN:  IP 隧道 (跨网段)

常见架构:
  Client → LVS (L4, 高性能入口)
             └→ Nginx 1 (L7, URL 路由 + SSL)
             └→ Nginx 2
                  └→ Backend Services

  LVS 做第一层, 分发到多个 Nginx
  Nginx 做第二层, 做 URL 路由和应用层处理
```

## 7. 面试题：如何做到不停机部署？

**回答：**

```
不停机部署策略:

1. Nginx 配置热重载
   nginx -t && nginx -s reload
   零请求丢失: 新 Worker 处理新请求, 旧 Worker 处理完当前请求后退出

2. 后端滚动更新 (K8s)
   strategy:
     type: RollingUpdate
     rollingUpdate:
       maxUnavailable: 0
       maxSurge: 1
   
   配合:
     readinessProbe → 新 Pod 就绪后才接流量
     preStop hook → 优雅关闭 (处理完当前请求)

3. 蓝绿部署
   Blue (当前版本) ← Nginx 指向
   Green (新版本)  ← 部署并验证
   切换: 修改 upstream 指向 Green → reload
   回滚: 指回 Blue → reload

4. 金丝雀发布 (Canary)
   upstream stable { server 10.0.1.1:8080; }
   upstream canary { server 10.0.2.1:8080; }
   
   split_clients $request_id $variant {
       5%  canary;
       *   stable;
   }
   
   逐步增加到 100%

5. 优雅关闭后端
   # 后端收到 SIGTERM:
   # 1. 停止接受新连接
   # 2. 等待当前请求处理完
   # 3. 超时后强制退出
   
   # K8s preStop:
   lifecycle:
     preStop:
       exec:
         command: ["sleep", "10"]
   # 给 Nginx upstream 足够时间感知 Pod 下线
```

## 8. 面试题：如何调试 Nginx 配置问题？

**回答：**

```bash
# ============ 调试工具箱 ============

# 1. 测试配置语法
nginx -t
nginx -T    # 输出完整配置 (含 include)

# 2. curl 测试请求
curl -v https://example.com/api/users
# -v: 显示请求/响应头
# -H "Host: example.com": 指定 Host
# -H "X-Forwarded-For: 1.2.3.4": 模拟真实 IP

# 3. 检查 DNS 解析
nslookup example.com
dig example.com

# 4. 检查端口
ss -tlnp | grep nginx    # Nginx 监听的端口
ss -tlnp | grep :8080    # 后端端口
netstat -an | grep :443   # HTTPS 端口

# 5. 查看连接状态
curl http://localhost:8080/nginx_status

# 6. 查看 error.log
tail -f /var/log/nginx/error.log

# 7. 测试后端连通性
curl -v http://backend-server:8080/health

# 8. 检查证书
openssl s_client -connect example.com:443 -servername example.com
echo | openssl s_client -connect example.com:443 2>/dev/null | openssl x509 -noout -dates

# 9. 检查返回的 Header
curl -I https://example.com
# 查看 X-Cache-Status, Server, Security Headers 等

# 10. strace 追踪系统调用 (高级)
strace -p $(pgrep -f 'nginx: worker') -e trace=network
```

## 9. 面试题：Nginx 常见错误码原因？

**回答：**

```
Nginx 相关 HTTP 状态码:

  ┌──────┬────────────────────────────────────────┐
  │ 状态码│ 原因及排查                              │
  ├──────┼────────────────────────────────────────┤
  │ 400  │ 请求语法错误                             │
  │      │ • 请求头过大 → large_client_header_buffers│
  │      │ • URL 过长 → client_header_buffer_size   │
  ├──────┼────────────────────────────────────────┤
  │ 403  │ 访问被拒                                │
  │      │ • deny 规则                              │
  │      │ • 目录无 index 且 autoindex off          │
  │      │ • 文件权限不对 (Nginx 用户无读权限)       │
  ├──────┼────────────────────────────────────────┤
  │ 404  │ 文件/路径不存在                          │
  │      │ • root/alias 路径错误                    │
  │      │ • try_files 没匹配到                     │
  │      │ • location 没匹配到                      │
  ├──────┼────────────────────────────────────────┤
  │ 413  │ 请求体太大                               │
  │      │ • client_max_body_size 设置太小          │
  ├──────┼────────────────────────────────────────┤
  │ 444  │ Nginx 特有: 直接关闭连接                 │
  │      │ • return 444 (不返回任何内容)             │
  ├──────┼────────────────────────────────────────┤
  │ 499  │ 客户端主动断开                           │
  │      │ • 后端处理太慢, 客户端超时关闭            │
  ├──────┼────────────────────────────────────────┤
  │ 500  │ Nginx 内部错误                           │
  │      │ • 配置错误 (rewrite 死循环)              │
  │      │ • Lua 脚本错误 (OpenResty)               │
  ├──────┼────────────────────────────────────────┤
  │ 502  │ 后端无法响应                              │
  │      │ • 后端未启动                              │
  │      │ • 端口错误                                │
  │      │ • 后端 OOM                                │
  ├──────┼────────────────────────────────────────┤
  │ 503  │ 服务不可用                               │
  │      │ • 所有后端 down                           │
  │      │ • 限流触发 (limit_req)                   │
  ├──────┼────────────────────────────────────────┤
  │ 504  │ 后端超时                                 │
  │      │ • proxy_read_timeout 太短                │
  │      │ • 后端处理时间超过超时设置                │
  └──────┴────────────────────────────────────────┘
```

## 10. 面试高频问题速答？

**回答：**

```
Q: Nginx 为什么高性能?
A: 事件驱动 + 异步非阻塞 + epoll + 单线程事件循环
   少量 Worker 处理大量连接, 避免上下文切换

Q: Worker 数应该设多少?
A: auto (= CPU 核心数), CPU 密集可适当减少

Q: location 匹配优先级?
A: = > ^~ > ~/~* (按顺序) > 最长前缀

Q: proxy_pass 结尾带 / 和不带 / 的区别?
A: 带 /: 替换 location 匹配的路径
   不带: 保持原始 URI

Q: reload 和 restart 区别?
A: reload: 不停止服务, 新 Worker 用新配置, 旧 Worker 处理完退出
   restart: 停止再启动, 有短暂中断

Q: 什么是 sendfile?
A: 零拷贝, 内核直接发送文件到网卡, 不经过用户空间

Q: 如何获取客户端真实 IP?
A: X-Forwarded-For + realip 模块
   set_real_ip_from 设置可信代理, real_ip_recursive on

Q: upstream 中 backup 参数?
A: 备用服务器, 只在所有主服务器不可用时才启用

Q: 如何理解 limit_req 的 burst?
A: 桶容量, 超过 rate 但在 burst 范围内的请求可排队/立即处理
   nodelay: 突发请求立即处理, 不排队

Q: Nginx 能否做 L4 负载均衡?
A: 可以, 使用 stream 模块, 支持 TCP/UDP 代理

Q: 如何实现灰度发布?
A: split_clients / map + Header / Cookie / IP 判断 + 多个 upstream
   K8s: Canary Ingress Annotation

Q: Nginx vs Envoy 选哪个?
A: Web/静态文件: Nginx
   服务网格/云原生: Envoy
   Envoy 支持 xDS 动态配置, gRPC 原生, 可观测性更强

Q: 如何解决惊群效应?
A: accept_mutex on (Worker 排队接受连接)
   REUSEPORT (内核分发, 更高效)
   高并发建议: accept_mutex off + listen reuseport
```
