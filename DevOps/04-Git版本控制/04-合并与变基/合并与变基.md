# 合并与变基

---

## 1. git merge 的工作原理？

**回答：**

### 合并类型

```
Fast-Forward 合并（快进合并）：
  当前分支没有新的提交，只需移动指针

  合并前：
  A ← B ← C (main)
            ↑
            D ← E (feature)
  
  git merge feature (在 main 上)
  
  合并后：
  A ← B ← C ← D ← E (main, feature)
  → 没有创建合并提交，只是移动了 main 指针

三路合并（Three-Way Merge）：
  两个分支都有新提交，需要找公共祖先

  合并前：
        D ← E (feature)
       /
  A ← B ← C ← F (main)
       ↑
    公共祖先 (merge base)
  
  git merge feature (在 main 上)
  
  合并后：
        D ← E (feature)
       /     \
  A ← B ← C ← F ← M (main)  ← M 是合并提交
  
  三路 = 公共祖先 + main端 + feature端
  Git 对比这三个版本来自动合并
```

### 合并命令

```bash
# 普通合并
git merge feature

# 禁止快进合并（总是创建合并提交）
git merge --no-ff feature
# 团队常用：保留分支历史，便于追踪功能开发过程

# 只允许快进合并（有分叉则失败）
git merge --ff-only feature

# 合并但不提交（检查结果）
git merge --no-commit feature

# 合并时 squash（压缩为一个变更但不提交）
git merge --squash feature
git commit -m "feat: add feature X"

# 取消正在进行的合并
git merge --abort

# 查看合并基点
git merge-base main feature
```

## 2. git rebase 的工作原理？

**回答：**

```
变基 = 将一系列提交"重放"到新的基础上
本质: 逐个应用 commit 的 diff 到目标分支上

  变基前：
        D ← E (feature)
       /
  A ← B ← C ← F (main)
  
  git rebase main (在 feature 上)
  
  变基后：
                    D' ← E' (feature)
                   /
  A ← B ← C ← F (main)
  
  D' 和 E' 是新的提交（hash 不同），内容相同
  原来的 D 和 E 变成了不可达提交（最终被 GC 清理）

变基过程：
  1. 找到公共祖先 B
  2. 保存 feature 分支的 commit (D, E)
  3. 将 feature 指针移到 main 的最新位置 (F)
  4. 逐个重新应用 D → D', E → E'
```

### rebase 命令

```bash
# 基本变基
git switch feature
git rebase main

# 变基到特定提交
git rebase --onto main feature~3 feature
# 将 feature 最近 3 个提交变基到 main

# 冲突处理
# 变基过程中每个 commit 都可能产生冲突
# 解决冲突后：
git add resolved_file.txt
git rebase --continue

# 跳过当前提交
git rebase --skip

# 放弃变基
git rebase --abort
```

## 3. merge vs rebase 怎么选？

**回答：**

| 特性 | merge | rebase |
|------|-------|--------|
| 历史 | 保留真实合并历史（非线性） | 线性历史，更清晰 |
| 安全性 | 不修改已有提交 | 重写历史（都是新hash） |
| 冲突 | 一次性解决 | 逐个提交解决（可能多次） |
| 合适场景 | 公共分支合并 | 个人分支整理 |
| 信息量 | 可追溯分支与合并点 | 看不出曾经有分支 |

### 推荐工作流

```bash
# 1. 在自己的功能分支上，用 rebase 保持与 main 同步
git switch feature
git fetch origin
git rebase origin/main        # 将自己的提交放在最新 main 之上

# 2. 合并到 main 时，用 merge 或 squash merge
git switch main
git merge --no-ff feature     # 保留功能分支历史
# 或
git merge --squash feature    # 压缩为一个提交

# 黄金法则：
#   ⚠️ 永远不要 rebase 已经推送到公共仓库的提交！
#   → rebase 会改变 commit hash
#   → 其他人基于旧的 hash 工作，会导致混乱
#   → 只 rebase 自己本地的、未推送的提交
```

## 4. 交互式变基（git rebase -i）？

**回答：**

```bash
# 整理最近 N 个提交
git rebase -i HEAD~5

# 变基到指定提交
git rebase -i abc1234

# 编辑器中显示（从旧到新）：
pick abc1234 feat: add login page
pick def5678 fix: typo in login
pick ghi9012 feat: add validation
pick jkl3456 fix: validation edge case
pick mno7890 feat: add tests

# 可用命令：
# p, pick   = 保留该提交
# r, reword = 保留但修改提交信息
# e, edit   = 暂停，允许修改（amend）
# s, squash = 合并到前一个提交（保留 message）
# f, fixup  = 合并到前一个提交（丢弃 message）
# d, drop   = 删除该提交
# x, exec   = 执行 shell 命令
# b, break  = 在此处暂停（可用 --continue 恢复）
```

### 常见操作示例

```bash
# 1. 合并多个提交为一个
pick abc1234 feat: add login page
fixup def5678 fix: typo in login            # 合并到上一个
fixup ghi9012 fix: validation edge case     # 合并到上一个
→ 结果: 一个干净的 "feat: add login page" 提交

# 2. 修改提交信息
reword abc1234 feat: add login page         # 保存后弹出编辑器修改信息

# 3. 拆分一个提交
edit abc1234 feat: add login and validation
# Git 暂停在该提交
git reset HEAD~1                             # 回退但保留修改
git add login.py && git commit -m "feat: add login"
git add validation.py && git commit -m "feat: add validation"
git rebase --continue

# 4. 重新排序提交
pick ghi9012 feat: add validation            # 移到前面
pick abc1234 feat: add login page
pick def5678 fix: typo

# 5. 每个提交后运行测试
git rebase -i HEAD~5 --exec "npm test"
# 或在编辑器中：
pick abc1234 feat: add feature
exec npm test
pick def5678 fix: bug
exec npm test
```

### autosquash

```bash
# 自动标记 fixup/squash 的提交
git commit --fixup abc1234     # 创建一个 fixup! 前缀的提交
git commit --squash abc1234    # 创建一个 squash! 前缀的提交

# 然后自动排序
git rebase -i --autosquash HEAD~5
# fixup! 提交自动移到对应提交下方并标记为 fixup

# 全局启用 autosquash
git config --global rebase.autoSquash true
```

## 5. 如何解决合并冲突？

**回答：**

### 冲突标记

```
<<<<<<< HEAD
当前分支（main）的内容
=======
被合并分支（feature）的内容
>>>>>>> feature

使用 diff3 风格（推荐，显示公共祖先）：
git config --global merge.conflictstyle diff3

<<<<<<< HEAD
当前分支的内容
||||||| merged common ancestors
公共祖先的原始内容
=======
被合并分支的内容
>>>>>>> feature
→ 有了原始内容，更容易判断应该保留什么
```

### 解决冲突流程

```bash
# 1. 查看冲突文件
git status
git diff --name-only --diff-filter=U    # 只列出冲突文件

# 2. 手动编辑解决冲突
#    删除 <<<<<<< ======= >>>>>>> 标记
#    保留正确的代码

# 3. 使用工具解决
git mergetool                           # 使用配置的合并工具

# 4. 标记为已解决
git add resolved_file.txt

# 5. 完成合并
git commit                              # merge 时
git rebase --continue                   # rebase 时

# 放弃
git merge --abort
git rebase --abort

# 选择保留某一方的版本
git checkout --ours file.txt            # 保留当前分支版本
git checkout --theirs file.txt          # 保留被合并分支版本
```

### 预防冲突

```bash
# 1. 频繁同步 main 到功能分支
git fetch origin
git rebase origin/main

# 2. 小而频繁的提交
# 3. 避免长期存在的功能分支
# 4. 团队沟通：避免多人修改同一文件

# rerere：记住冲突解决方式，自动应用
git config --global rerere.enabled true
# 下次遇到相同冲突，Git 自动解决
```

## 6. Git 的合并策略有哪些？

**回答：**

```bash
# Git 内置合并策略
git merge -s <strategy> feature

# ort (默认, Git 2.33+)
#   递归三路合并的改进版，更快更准确
#   替代了之前的 recursive 策略

# recursive (旧默认)
#   处理多个公共祖先时递归合并
#   支持选项：-X ours, -X theirs, -X patience

# resolve
#   三路合并，只使用一个公共祖先

# octopus
#   合并多个分支（一次合并3个以上分支）
#   git merge feature1 feature2 feature3

# ours
#   保留当前分支内容，丢弃被合并分支
#   只创建合并提交记录
#   git merge -s ours old-branch

# subtree
#   子树合并，将一个仓库作为子目录合并
```

### 合并策略选项

```bash
# 冲突时偏向哪一方
git merge -X ours feature       # 冲突时选择当前分支
git merge -X theirs feature     # 冲突时选择被合并分支

# 注意区别：
git merge -s ours feature       # 完全丢弃 feature 的修改
git merge -X ours feature       # 只在冲突时选择当前分支，非冲突正常合并

# patience 算法（对付复杂的diff）
git merge -X patience feature

# 检测重命名
git merge -X rename-threshold=50 feature
```

## 7. 什么是 Octopus Merge？

**回答：**

```
Octopus Merge = 一次合并多个分支

  合并前：
        C ← D (feature-1)
       /
  A ← B (main)
       \
        E ← F (feature-2)
       \
        G    (feature-3)

  git merge feature-1 feature-2 feature-3
  
  合并后：
        C ← D (feature-1)
       /     \
  A ← B ──────── M (main, octopus merge)
       \     /  /
        E ← F ┘
       \   /
        G ┘

  M 有 4 个父提交: B, D, F, G

限制：
  - 如果有冲突，octopus merge 会失败
  - 适用于多个互不冲突的功能分支

场景：
  - Linux 内核开发中常用
  - 集成多个独立子系统的更新
```

## 8. git rebase --onto 的用法？

**回答：**

```bash
# 语法: git rebase --onto <新基础> <旧基础> <分支>

# 场景 1: 将分支的部分提交移到另一个分支
#   feature 从 develop 分出，现在想移到 main
#
#           E ← F (feature)
#          /
#   A ← B ← C (develop)
#        \
#         D (main)
#
git rebase --onto main develop feature
#
#   A ← B ← C (develop)
#        \
#         D (main) ← E' ← F' (feature)

# 场景 2: 删除中间的几个提交
#   A ← B ← C ← D ← E (main)
#   想删除 C 和 D
#
git rebase --onto B D main
# 结果: A ← B ← E' (main)

# 场景 3: 从 feature 分出的子分支，feature 已合并
#   sub-feature 基于 feature，feature 被删了
#
#   A ← B ← C (main)
#        \
#         D ← E (feature, 已删)
#              \
#               F ← G (sub-feature)
#
git rebase --onto main E sub-feature
# 结果: A ← B ← C (main) ← F' ← G' (sub-feature)
```

## 9. 合并冲突的高级处理技巧？

**回答：**

### rerere（Reuse Recorded Resolution）

```bash
# 启用 rerere
git config --global rerere.enabled true

# 工作原理：
#   1. 第一次遇到冲突，手动解决
#   2. Git 记录冲突和解决方式
#   3. 下次遇到相同冲突，自动应用解决方式

# 场景：
#   频繁 rebase 同一个长期分支
#   每次 rebase 都遇到相同冲突
#   rerere 自动解决已知冲突

# 查看 rerere 记录
git rerere status
git rerere diff

# 清除 rerere 缓存
git rerere forget <path>
git rerere gc
```

### 其他技巧

```bash
# 1. 使用 diff3 格式（看到公共祖先）
git config merge.conflictstyle diff3

# 2. 查看合并冲突的各个版本
git show :1:file.txt    # 公共祖先版本（merge base）
git show :2:file.txt    # 当前分支版本（ours）
git show :3:file.txt    # 被合并分支版本（theirs）

# 3. 合并大型分支前的检查
git merge --no-commit --no-ff feature
# 检查结果后
git merge --abort       # 不满意就取消
git commit              # 满意就提交

# 4. 只看冲突文件的差异
git diff --diff-filter=U

# 5. 标记文件为二进制（避免 merge 冲突）
# .gitattributes
*.lock binary
*.png binary
```

## 10. git merge --squash vs rebase squash？

**回答：**

```bash
# merge --squash：将功能分支的所有修改压缩为一个变更
git switch main
git merge --squash feature
git commit -m "feat: add complete feature X"

# 结果: main 上多了一个普通 commit（不是 merge commit）
# feature 的提交历史不会出现在 main 上
# 注意: feature 分支在 --merged 中不会显示为已合并！

# rebase -i squash：在变基时压缩提交
git switch feature
git rebase -i main
# 将除第一个外标记为 squash 或 fixup
git switch main
git merge feature      # 此时 fast-forward

# 区别：
#   merge --squash: 不创建合并关系，feature显示为未合并
#   rebase squash:  先压缩再合并，可以做fast-forward

# 哪个用？
#   PR/MR: 平台上选 "Squash and merge" 按钮（本质是 merge --squash）
#   本地:  git rebase -i 整理后再合并
```
