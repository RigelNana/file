# 撤销与恢复

---

## 1. git reset 的三种模式详解？

**回答：**

```
git reset 移动 HEAD 和分支指针到指定的 commit

三种模式影响范围：
  --soft:  只移动 HEAD/分支指针
  --mixed: 移动 HEAD + 重置暂存区（默认）
  --hard:  移动 HEAD + 重置暂存区 + 重置工作区

               HEAD     暂存区    工作区
  --soft       ✅ 移动   ❌ 不变   ❌ 不变
  --mixed      ✅ 移动   ✅ 重置   ❌ 不变
  --hard       ✅ 移动   ✅ 重置   ✅ 重置  ← 危险！
```

### 详细场景

```bash
# 假设提交历史: A ← B ← C (HEAD/main)

# --soft: 撤回提交但保留所有修改在暂存区
git reset --soft HEAD~1
# HEAD → B, 暂存区包含 C 的修改
# 用途: 修改提交信息、将多个提交合并

# --mixed（默认）: 撤回提交和暂存
git reset HEAD~1
git reset --mixed HEAD~1
# HEAD → B, C 的修改在工作区（未暂存状态）
# 用途: 撤回提交，重新选择要提交的文件

# --hard: 完全丢弃（危险！）
git reset --hard HEAD~1
# HEAD → B, C 的修改完全丢弃
# 用途: 彻底放弃最近的提交

# reset 也可以只针对文件（暂存区操作）
git reset HEAD file.txt          # 取消暂存（保留工作区修改）
# 等同于: git restore --staged file.txt

# reset 到任意提交
git reset --hard abc1234
git reset --hard origin/main     # 强制同步到远程状态
```

## 2. git revert 和 git reset 的区别？

**回答：**

```
reset:  移动指针回退，重写历史
revert: 创建新提交反向撤销，不改变历史

提交历史: A ← B ← C ← D (HEAD)

# reset 方式（重写历史）
git reset --hard HEAD~2
# 结果: A ← B (HEAD)
# C 和 D 从历史中消失（还在 reflog 中）

# revert 方式（安全，添加新提交）
git revert HEAD~1..HEAD          # 撤销 C 和 D
# 结果: A ← B ← C ← D ← D' ← C' (HEAD)
# D' 是反向 D 的提交, C' 是反向 C 的提交
```

### revert 使用

```bash
# 撤销单个提交
git revert abc1234

# 撤销但不自动提交
git revert --no-commit abc1234

# 撤销多个连续提交
git revert HEAD~3..HEAD          # 撤销最近 3 个

# 撤销合并提交
git revert -m 1 <merge-commit-hash>
# -m 1: 保留第一个父提交（main 分支的历史）
# -m 2: 保留第二个父提交（被合并分支的历史）

# 放弃正在进行的 revert
git revert --abort

# 关键区别：
#   reset: 适合本地未推送的提交
#   revert: 适合已推送的提交（安全，不影响他人）
```

## 3. git restore 使用详解？

**回答：**

```bash
# git restore = 从 Git 2.23 提取的文件恢复命令
# 替代了 git checkout -- <file> 的文件恢复功能

# 撤销工作区修改（恢复到暂存区的版本）
git restore file.txt
git restore .                    # 撤销所有修改
git restore src/                 # 撤销指定目录

# 取消暂存（从暂存区移回工作区）
git restore --staged file.txt
git restore --staged .           # 取消所有暂存

# 同时撤销暂存和工作区修改
git restore --staged --worktree file.txt

# 恢复到指定提交的版本
git restore --source=HEAD~1 file.txt     # 恢复到上一个提交的版本
git restore --source=abc1234 file.txt    # 恢复到指定提交
git restore --source=main file.txt       # 恢复到 main 分支版本

# 交互式恢复（选择性恢复部分修改）
git restore -p file.txt
git restore --patch file.txt

# 注意：
#   git restore 不会自动提交
#   对未追踪文件没有作用（用 git clean）
#   --worktree 是默认操作
```

## 4. git reflog 详解？如何利用它恢复？

**回答：**

```bash
# reflog = 引用日志（reference log）
# 记录 HEAD 和分支引用的每次变更

git reflog
# abc1234 HEAD@{0}: commit: latest commit
# def5678 HEAD@{1}: merge feature: Merge branch 'feature'
# ghi9012 HEAD@{2}: checkout: moving from feature to main
# jkl3456 HEAD@{3}: commit: feature work
# mno7890 HEAD@{4}: checkout: moving from main to feature
# pqr1234 HEAD@{5}: reset: moving to HEAD~2

# 查看特定分支的 reflog
git reflog show feature

# 带时间的 reflog
git reflog --date=iso

# reflog 是 Git 的"安全网"
# 即使 reset --hard 也能恢复！
```

### 恢复场景

```bash
# 场景 1: 误执行 reset --hard
git reset --hard HEAD~3          # 误删了 3 个提交
git reflog                        # 找到之前的 HEAD
# abc1234 HEAD@{1}: commit: important work
git reset --hard abc1234          # 恢复！

# 场景 2: 误删分支
git branch -D feature             # 误删分支
git reflog                        # 找到 feature 的最后一个提交
# def5678 HEAD@{5}: commit: feature work (on feature)
git branch feature def5678        # 从 reflog 恢复分支

# 场景 3: 错误的 rebase
git rebase main                   # rebase 出问题了
git reflog                        # 找到 rebase 前的 HEAD
# ghi9012 HEAD@{7}: checkout: moving to feature (rebase 前)
git reset --hard ghi9012          # 恢复到 rebase 前

# 场景 4: 恢复已被覆盖的 amend
git commit --amend                # 修改了提交，想恢复原来的
git reflog
# jkl3456 HEAD@{1}: commit (amend): new message
# mno7890 HEAD@{2}: commit: original message
git reset --hard mno7890

# reflog 保留时间
# 默认: 可达提交 90 天, 不可达提交 30 天
# 配置: git config gc.reflogExpire "180 days"
```

## 5. git cherry-pick 精选提交？

**回答：**

```bash
# 将指定提交应用到当前分支
git cherry-pick abc1234

# 场景：从 feature 挑选一个 bugfix 到 main
#
#   A ← B ← C ← D ← E (feature)
#               ↑ 这个是 bugfix
#   F ← G ← H (main)
#
git switch main
git cherry-pick C
# 结果: F ← G ← H ← C' (main)

# 多个提交
git cherry-pick abc1234 def5678

# 范围（不包含起始）
git cherry-pick abc1234..def5678     # 不包含 abc1234
git cherry-pick abc1234^..def5678    # 包含 abc1234

# 只暂存不提交
git cherry-pick --no-commit abc1234
git cherry-pick -n abc1234

# 保留原作者信息
git cherry-pick -x abc1234          # 在 message 中追加原 commit hash

# 冲突处理
git cherry-pick abc1234
# 冲突
git add resolved_file.txt
git cherry-pick --continue
# 或放弃
git cherry-pick --abort

# cherry-pick 合并提交
git cherry-pick -m 1 <merge-commit>  # 需要指定主线（-m 1 或 -m 2）
```

### 注意事项

```
cherry-pick 创建新的 commit（不同的 hash）
→ 同一修改出现在两个分支，可能导致：
  1. 之后合并时出现冲突（大多数情况 Git 能正确处理）
  2. git log 中同样的修改出现两次

建议：
  - 优先使用 merge 而非 cherry-pick
  - cherry-pick 适合紧急 hotfix 的场景
  - 少量使用，避免滥用
```

## 6. 如何修改已经提交的信息？

**回答：**

```bash
# 修改最近一次提交
git commit --amend -m "new message"
git commit --amend --no-edit            # 追加文件但不改信息

# 修改更早的提交（交互式变基）
git rebase -i HEAD~5
# 将目标提交标记为 reword
# reword abc1234 old message
# 保存后弹出编辑器修改信息

# 修改作者信息
git commit --amend --author="New Name <new@email.com>"

# 修改历史中所有匹配的作者
git filter-branch --env-filter '
if [ "$GIT_AUTHOR_EMAIL" = "old@email.com" ]; then
    export GIT_AUTHOR_EMAIL="new@email.com"
    export GIT_AUTHOR_NAME="New Name"
fi' -- --all
# 推荐使用 git filter-repo 替代 filter-branch

# 注意：修改已推送的提交需要 force push
git push --force-with-lease
```

## 7. 找回丢失的提交和文件？

**回答：**

```bash
# 方法 1: reflog（最常用）
git reflog
git reset --hard <hash>
git branch recovered <hash>

# 方法 2: git fsck（找孤立对象）
git fsck --lost-found
# 孤立的 commit/blob 会列在 .git/lost-found/ 中
git show <dangling-commit-hash>

# 方法 3: 从特定提交恢复文件
git restore --source=abc1234 -- path/to/file
git show abc1234:path/to/file > recovered_file

# 方法 4: 在所有提交中搜索
git log --all --full-history -- path/to/deleted/file
# 找到文件最后存在的提交
git restore --source=<commit> -- path/to/deleted/file

# 方法 5: stash 恢复
git stash list
git stash apply stash@{2}

# 已被 GC 清理的对象无法恢复
# 定期备份重要仓库！
```

## 8. 如何安全地清理 Git 历史中的敏感信息？

**回答：**

```bash
# 场景：误提交了密码、密钥、Token 等敏感信息

# ⚠️ 仅仅删除文件并提交是不够的！
# 历史提交中仍然包含敏感信息

# 方法 1: git filter-repo（推荐，替代 filter-branch）
# 安装: pip install git-filter-repo

# 删除特定文件（从所有历史中）
git filter-repo --invert-paths --path secrets.yml

# 替换文件中的敏感内容
git filter-repo --blob-callback '
  blob.data = blob.data.replace(b"sk-mySecretKey123", b"REDACTED")
'

# 方法 2: BFG Repo-Cleaner（第三方工具，更快）
# 删除文件
bfg --delete-files secrets.yml

# 替换文本
bfg --replace-text passwords.txt

# 清理后：
git reflog expire --expire=now --all
git gc --prune=now --aggressive

# 然后强制推送
git push --force --all
git push --force --tags

# ⚠️ 重要后续步骤：
#   1. 立即轮换泄露的凭据（改密码、废弃 Token）
#   2. 通知团队成员重新克隆
#   3. GitHub: Settings → Danger Zone → 清除缓存
#   4. 添加 .gitignore 防止再次提交
```

## 9. ORIG_HEAD 和其他特殊引用？

**回答：**

```bash
# ORIG_HEAD
# 在 merge/rebase/reset 等危险操作前自动保存旧 HEAD
git merge feature
git reset --hard ORIG_HEAD     # 撤销 merge

git rebase main
git reset --hard ORIG_HEAD     # 撤销 rebase

# MERGE_HEAD
# 合并过程中指向被合并分支的 HEAD
# 合并完成后消失

# FETCH_HEAD
# 最近 fetch 的远程分支
git fetch origin main
git merge FETCH_HEAD           # 等同于 merge origin/main

# CHERRY_PICK_HEAD
# cherry-pick 冲突时指向被 pick 的提交

# REVERT_HEAD
# revert 冲突时指向被 revert 的提交

# 使用场景：
#   做了危险操作 → git diff ORIG_HEAD 查看变化
#   不满意 → git reset --hard ORIG_HEAD 撤销
```

## 10. 常见误操作的恢复速查表？

**回答：**

| 误操作 | 恢复方法 |
|--------|---------|
| 误修改文件 | `git restore file.txt` |
| 误 git add | `git restore --staged file.txt` |
| 误 commit（未push） | `git reset --soft HEAD~1` |
| 误 commit --amend | `git reset --hard HEAD@{1}`（reflog） |
| 误 commit（已push） | `git revert HEAD` |
| 误 reset --hard | `git reflog` → `git reset --hard HEAD@{n}` |
| 误删分支 | `git reflog` → `git branch name <hash>` |
| 误 merge | `git reset --hard ORIG_HEAD` |
| 误 rebase | `git reset --hard ORIG_HEAD` |
| 误删文件 | `git restore file.txt` |
| 提交了敏感信息 | `git filter-repo` + 轮换凭据 |
| push 了错误代码 | `git revert` + `git push` |
| 误 push --force | 让队友 push 他们的版本，或联系 Git 管理员 |

```bash
# 万能恢复法则：
# 1. 不要慌
# 2. git reflog 查看操作历史
# 3. 找到正确的 commit hash
# 4. git reset/branch 恢复

# 预防措施：
# - 重要操作前 git stash 或创建临时分支
# - 启用分支保护规则
# - 使用 --force-with-lease 替代 --force
# - rerere 记住冲突解决方式
```
