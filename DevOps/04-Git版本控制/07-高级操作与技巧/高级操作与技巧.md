## 1. git bisect 二分查找 Bug？

**回答：**

```bash
# 用二分法在提交历史中定位引入 bug 的 commit

# 基本流程
git bisect start
git bisect bad                   # 标记当前有 bug
git bisect good v1.0             # 标记已知正常的版本

# Git 自动切换到中间提交
# 你测试后标记:
git bisect good                  # 这个版本没 bug
# 或
git bisect bad                   # 这个版本有 bug

# Git 继续二分，直到找到引入 bug 的 commit
# abc1234 is the first bad commit

# 结束
git bisect reset

# 100 个提交只需 ~7 步（log2(100) ≈ 7）

# 自动化（提供测试脚本）
git bisect start HEAD v1.0
git bisect run ./test.sh
# test.sh 返回 0 = good, 非0 = bad
# Git 自动完成所有二分步骤

# 自动化示例
git bisect run make test
git bisect run python -m pytest tests/test_login.py
git bisect run bash -c 'grep -q "bug_pattern" output.log && exit 1 || exit 0'

# 跳过无法测试的提交
git bisect skip

# 查看二分过程日志
git bisect log
# 重放日志
git bisect log > bisect.log
git bisect replay bisect.log
```

## 2. git submodule 子模块管理？

**回答：**

```bash
# 添加子模块
git submodule add git@github.com:lib/utils.git libs/utils
git commit -m "chore: add utils submodule"

# 克隆包含子模块的仓库
git clone --recurse-submodules git@github.com:user/repo.git
# 如果克隆时忘了 --recurse-submodules
git submodule update --init --recursive

# 更新子模块到最新
cd libs/utils
git pull origin main
cd ../..
git add libs/utils
git commit -m "chore: update utils submodule"

# 或一键更新所有子模块
git submodule update --remote

# 在所有子模块中执行命令
git submodule foreach git pull origin main
git submodule foreach 'git checkout main && git pull'

# 删除子模块
git submodule deinit libs/utils
git rm libs/utils
rm -rf .git/modules/libs/utils
git commit -m "chore: remove utils submodule"
```

### .gitmodules 文件

```
[submodule "libs/utils"]
    path = libs/utils
    url = git@github.com:lib/utils.git
    branch = main
```

### 常见陷阱

```
1. 忘记 --recurse-submodules
   → 子模块目录为空
   → 解决: git submodule update --init --recursive

2. 子模块在 detached HEAD 状态
   → 在子模块中修改前先 checkout 分支
   → cd libs/utils && git switch main

3. 子模块更新后忘记提交主仓库
   → git diff 显示子模块 hash 变化
   → 要在主仓库 commit 新的子模块引用

4. CI 中需要特殊处理
   → git clone --recurse-submodules
   → 或 actions/checkout@v4 加 submodules: recursive
```

## 3. git subtree 子树管理？

**回答：**

```bash
# subtree vs submodule：subtree 将代码直接合并到主仓库

# 添加子树
git subtree add --prefix=libs/utils git@github.com:lib/utils.git main --squash

# 拉取子树更新
git subtree pull --prefix=libs/utils git@github.com:lib/utils.git main --squash

# 推送修改回子树仓库
git subtree push --prefix=libs/utils git@github.com:lib/utils.git main

# --squash: 压缩子树的历史为一个提交
```

### submodule vs subtree 对比

| 特性 | submodule | subtree |
|------|-----------|---------|
| 代码位置 | 独立仓库，主仓库只存引用 | 代码合并到主仓库 |
| 克隆 | 需要 `--recurse-submodules` | 直接克隆即可 |
| 更新 | `git submodule update` | `git subtree pull` |
| 推送 | 在子模块目录内操作 | `git subtree push` |
| 复杂度 | 较复杂，易出错 | 相对简单 |
| Git 历史 | 独立历史 | 合并到主仓库历史 |
| 适用 | 经常独立开发的组件 | 不频繁更新的共享代码 |

## 4. git worktree 多工作区？

**回答：**

```bash
# 一个仓库同时 checkout 多个分支到不同目录
# 不需要多次 clone，共享 .git 目录

# 创建新的工作区
git worktree add ../hotfix hotfix/urgent
# 在 ../hotfix 目录 checkout hotfix/urgent 分支

# 创建新分支并 checkout
git worktree add -b fix/bug-123 ../bugfix main
# 基于 main 创建 fix/bug-123 分支，在 ../bugfix 目录

# 查看所有工作区
git worktree list
# /home/user/project          abc1234 [main]
# /home/user/hotfix           def5678 [hotfix/urgent]
# /home/user/bugfix           ghi9012 [fix/bug-123]

# 删除工作区
git worktree remove ../hotfix

# 清理不存在的工作区引用
git worktree prune

# 使用场景：
#   1. 正在开发功能，需要紧急修 bug
#      → 不用 stash，直接新建工作区修复
#   2. 同时查看代码的多个版本
#   3. 在一个工作区运行长时间测试，另一个继续开发
#   4. Code Review 时查看某个分支
```

## 5. git sparse-checkout 稀疏检出？

**回答：**

```bash
# 只检出仓库的部分目录（适合 monorepo）

# 启用稀疏检出
git sparse-checkout init

# 设置需要的目录
git sparse-checkout set src/ docs/

# 添加更多目录
git sparse-checkout add tests/

# 查看当前稀疏检出配置
git sparse-checkout list

# 使用 cone 模式（推荐，更快）
git sparse-checkout init --cone
git sparse-checkout set src/frontend src/shared

# 关闭稀疏检出（恢复完整检出）
git sparse-checkout disable

# 克隆时直接使用稀疏检出
git clone --filter=blob:none --sparse git@github.com:org/monorepo.git
cd monorepo
git sparse-checkout set services/my-service shared/

# 场景：
#   大型 monorepo（谷歌、微软级别的仓库）
#   只需要仓库中的某个微服务/模块
#   减少磁盘占用和 checkout 时间
```

## 6. git notes 给提交添加注释？

**回答：**

```bash
# 给提交添加注释（不修改提交本身）
git notes add -m "这个提交修复了生产事故" abc1234

# 查看注释
git log --show-notes
git notes show abc1234

# 修改注释
git notes edit abc1234

# 删除注释
git notes remove abc1234

# 推送/拉取 notes
git push origin refs/notes/commits
git fetch origin refs/notes/commits:refs/notes/commits

# 使用场景：
#   - Code Review 后添加审查记录
#   - 记录部署信息
#   - 给历史提交添加额外说明
#   - 不想改变 commit hash
```

## 7. git archive 打包导出？

**回答：**

```bash
# 导出为 tar/zip（不含 .git 目录）
git archive --format=tar HEAD | gzip > project.tar.gz
git archive --format=zip HEAD > project.zip

# 导出特定目录
git archive HEAD src/ > src.tar

# 导出特定标签版本
git archive --prefix=project-v1.0/ v1.0 | gzip > project-v1.0.tar.gz
# --prefix: 添加顶层目录

# 导出特定分支
git archive --format=zip release/1.0 > release-1.0.zip

# 场景：
#   - 给客户发布源码包（不含 Git 历史）
#   - 创建发布版本的归档
#   - CI/CD 中打包制品
```

## 8. git shortlog 统计和总结？

**回答：**

```bash
# 按作者分组统计提交
git shortlog -sn
#  150  Alice
#  120  Bob
#   80  Charlie

# -s: 只显示数量
# -n: 按数量排序

# 指定范围
git shortlog -sn v1.0..v2.0

# 按邮箱统计
git shortlog -sne

# 生成 changelog
git shortlog v1.0..v2.0
# Alice (3):
#   feat: add login page
#   fix: validation error
#   docs: update README
#
# Bob (2):
#   feat: add dashboard
#   fix: memory leak

# 统计代码行数变更
git log --author="Alice" --oneline --shortstat
git log --author="Alice" --format="" --numstat | awk '{add+=$1; del+=$2} END {print "Added:", add, "Deleted:", del}'

# 日期范围统计
git shortlog -sn --since="2024-01-01" --until="2024-12-31"
```

## 9. git bundle 离线传输仓库？

**回答：**

```bash
# 将仓库打包为单个文件（离线传输）

# 打包整个仓库
git bundle create repo.bundle --all

# 打包特定分支
git bundle create main.bundle main

# 打包增量（只打包某些新提交）
git bundle create update.bundle v1.0..main

# 从 bundle 克隆
git clone repo.bundle my-repo

# 从 bundle 拉取更新
git pull update.bundle main

# 验证 bundle
git bundle verify repo.bundle

# 场景：
#   - 无网络环境的代码传输
#   - 通过 USB/邮件传输仓库
#   - 备份仓库到离线介质
```

## 10. Git 常用别名和效率技巧？

**回答：**

```bash
# 推荐别名配置
git config --global alias.st status
git config --global alias.co checkout
git config --global alias.sw switch
git config --global alias.br branch
git config --global alias.ci commit
git config --global alias.unstage 'restore --staged'
git config --global alias.last 'log -1 HEAD'
git config --global alias.lg "log --oneline --graph --all --decorate"
git config --global alias.ll "log --pretty=format:'%C(yellow)%h%Creset %s %C(green)(%cr)%Creset %C(blue)<%an>%Creset' --abbrev-commit"
git config --global alias.d 'diff --stat'
git config --global alias.who 'shortlog -sn'
git config --global alias.undo 'reset --soft HEAD~1'
git config --global alias.amend 'commit --amend --no-edit'
git config --global alias.wip 'commit -am "WIP"'

# 在 ~/.gitconfig 中直接编辑
[alias]
    st = status
    lg = log --oneline --graph --all --decorate
    cleanup = "!git branch --merged main | grep -v main | xargs -n 1 git branch -d"

# 效率技巧
# 1. 使用 - 引用上一个分支
git switch -            # 切换到上一个分支

# 2. 使用 @{-N} 引用之前的分支
git merge @{-1}         # 合并上一个分支

# 3. 使用 --fixup 和 autosquash
git commit --fixup HEAD~2
git rebase -i --autosquash HEAD~5

# 4. 使用 git maintenance 后台优化
git maintenance start   # 后台自动执行 gc、prefetch 等
```
