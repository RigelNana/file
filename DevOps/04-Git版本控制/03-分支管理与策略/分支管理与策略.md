# 分支管理与策略

---

## 1. Git 分支的本质是什么？为什么这么轻量？

**回答：**

```
Git 分支 = 一个指向 commit 对象的可移动指针

本质上就是一个文件：
  .git/refs/heads/main → 包含 40 字符的 commit hash

创建分支 = 创建一个 41 字节的文件（40字符hash + 换行符）
切换分支 = 修改 HEAD 指针 + 更新工作区
→ 几乎瞬间完成

对比 SVN：
  SVN 创建分支 = 复制整个目录树
  Git 创建分支 = 写入一个 41 字节的文件

     HEAD → refs/heads/main → commit D
     
     A ← B ← C ← D (main)
                ↑
           refs/heads/feature → commit C
```

## 2. 分支的基本操作？

**回答：**

```bash
# 查看分支
git branch                       # 本地分支
git branch -r                    # 远程跟踪分支
git branch -a                    # 所有分支（本地 + 远程）
git branch -v                    # 显示最后一次提交
git branch -vv                   # 显示追踪的远程分支

# 创建分支
git branch feature               # 创建分支（不切换）
git checkout -b feature          # 创建并切换（老方式）
git switch -c feature            # 创建并切换（推荐，Git 2.23+）
git branch feature abc1234       # 从指定 commit 创建

# 切换分支
git checkout feature             # 老方式
git switch feature               # 推荐方式（更安全，不会意外操作文件）
git switch -                     # 切换到上一个分支

# 重命名分支
git branch -m old-name new-name
git branch -M old-name new-name  # 强制重命名（即使目标已存在）

# 删除分支
git branch -d feature            # 删除已合并的分支
git branch -D feature            # 强制删除（即使未合并）

# 删除远程分支
git push origin --delete feature
git push origin :feature         # 另一种方式

# 清理已删除的远程分支的本地跟踪
git fetch --prune
git remote prune origin

# 查看已合并/未合并的分支
git branch --merged main         # 已合并到 main 的分支（可安全删除）
git branch --no-merged main      # 未合并到 main 的分支
```

## 3. git switch 和 git checkout 的区别？

**回答：**

```
Git 2.23 引入了 git switch 和 git restore，
拆分了 git checkout 的多重职责

git checkout 的问题：它做了太多事情
  git checkout <branch>     → 切换分支
  git checkout <file>       → 恢复文件（危险：会丢弃修改！）
  git checkout <commit>     → 进入 detached HEAD
  git checkout -b <branch>  → 创建并切换分支

拆分后：
  git switch   → 专门用于切换分支
  git restore  → 专门用于恢复文件
```

| 操作 | checkout(老) | switch/restore(新) |
|------|-------------|-------------------|
| 切换分支 | `git checkout feature` | `git switch feature` |
| 创建并切换 | `git checkout -b feature` | `git switch -c feature` |
| Detached HEAD | `git checkout abc123` | `git switch --detach abc123` |
| 恢复文件 | `git checkout -- file` | `git restore file` |
| 取消暂存 | `git checkout HEAD -- file` | `git restore --staged file` |

## 4. Git Flow 工作流详解？

**回答：**

```
Git Flow 定义了严格的分支模型，适合有明确版本发布周期的项目

长期分支：
  main     → 生产代码（每个 commit 都是一个发布版本）
  develop  → 开发主线（集成所有功能分支）

短期分支：
  feature/* → 新功能开发
  release/* → 准备发布
  hotfix/*  → 紧急修复

流程图：
  main:    ──●──────────────────●───────────●──
              ↑                  ↑            ↑
  hotfix:     │                  │     ●──●──┘
              │                  ↑      
  release:    │          ●──●──●┘
              ↑          ↑
  develop: ──●───●───●───●───●───●───●───●──
              ↑   ↑       ↑       ↑
  feature:    │   ●──●──●┘       ●──●──●┘
              │
              └── v1.0

各分支规则：
  feature/xxx:
    从 develop 创建
    开发完成合并回 develop
    
  release/x.x:
    从 develop 创建
    只做 bug 修复和文档
    完成后合并到 main + develop
    在 main 上打版本标签
    
  hotfix/xxx:
    从 main 创建
    修复完成合并到 main + develop
    在 main 上打版本标签
```

### Git Flow 工具

```bash
# 安装 git-flow
apt install git-flow

# 初始化
git flow init

# 功能分支
git flow feature start add-login
git flow feature finish add-login   # 合并到 develop

# 发布分支
git flow release start 1.0
git flow release finish 1.0         # 合并到 main + develop + 打标签

# 热修复
git flow hotfix start fix-crash
git flow hotfix finish fix-crash    # 合并到 main + develop
```

## 5. GitHub Flow 和 GitLab Flow？

**回答：**

### GitHub Flow

```
极简的分支策略，适合持续部署

规则：
  1. main 分支始终可部署
  2. 从 main 创建功能分支
  3. 经常提交并推送到远程
  4. 创建 Pull Request
  5. Code Review 通过 + CI 通过
  6. 合并到 main 并部署

  main: ──●───●───●───●───●───●──
            ↑       ↑       ↑
            │   feature-1   │
            │   ●──●──●────┘
            │
            └── feature-2
                ●──●──●

优点：简单，适合 Web 项目持续部署
缺点：不适合多版本并行维护
```

### GitLab Flow

```
在 GitHub Flow 基础上增加环境分支

方式一：环境分支
  main → pre-production → production
  
  main: 开发主线
  pre-production: 预发布环境
  production: 生产环境
  
  功能合并到 main → cherry-pick/merge 到 pre-production → 再到 production

方式二：发布分支
  main → release/1.0 → release/1.1
  
  main: 最新开发代码
  release/*: 稳定发布版本
  bug 修复先合并到 main，再 cherry-pick 到需要的 release 分支

优点：比 Git Flow 简单，比 GitHub Flow 灵活
```

## 6. Trunk-Based Development（TBD）？

**回答：**

```
核心思想：所有开发者都向 main（trunk）频繁集成

规则：
  - 功能分支存活时间 ≤ 1-2 天
  - 频繁的小提交到 main
  - 使用 Feature Flag 控制未完成的功能
  - 依赖强大的 CI/CD 和自动化测试

  main: ──●──●──●──●──●──●──●──●──●──
            ↑   ↑       ↑
            │  short    short
            │  branch   branch
            │  ●──●┘    ●┘
            │
            └── 也可以直接提交到 main

Feature Flag 示例：
  if feature_flags.is_enabled("new_checkout"):
      new_checkout_flow()
  else:
      old_checkout_flow()

优点：
  - 避免长期分支带来的合并地狱
  - 持续集成，快速反馈
  - Google, Facebook 采用的方式

缺点：
  - 需要成熟的 CI/CD 和测试基础设施
  - 需要 Feature Flag 系统
  - 团队纪律要求高
```

## 7. 分支命名规范？

**回答：**

```
常见命名约定：

功能:     feature/JIRA-123-add-user-login
修复:     fix/JIRA-456-login-crash
热修复:   hotfix/JIRA-789-security-patch
发布:     release/1.2.0
重构:     refactor/extract-auth-service
文档:     docs/update-api-guide
测试:     test/add-payment-tests
性能:     perf/optimize-query

规则：
  ✅ 使用 / 分级（Git 视为目录结构）
  ✅ 使用 - 分隔单词（kebab-case）
  ✅ 包含 issue/ticket 编号
  ✅ 简短描述性名称
  
  ❌ 使用空格或特殊字符
  ❌ 过长的名称
  ❌ 纯数字名称
  ❌ 与标签同名
```

## 8. 如何保护重要分支？

**回答：**

### GitHub 分支保护规则

```
Settings → Branches → Branch protection rules

常用保护规则：
  ✅ Require pull request before merging
     → 不能直接 push 到 main
     → 必须通过 PR 合并
  
  ✅ Require approvals (1-2人)
     → Code Review 至少 1-2 人通过
  
  ✅ Require status checks to pass
     → CI/CD 必须通过
  
  ✅ Require branches to be up to date
     → 分支必须是最新（防止合并过时代码）
  
  ✅ Require signed commits
     → 必须 GPG 签名

  ✅ Require linear history
     → 只允许 rebase 或 squash 合并（线性历史）

  ✅ Include administrators
     → 管理员也受限
```

### GitLab 分支保护

```
Settings → Repository → Protected Branches

  - 设置允许推送的角色
  - 设置允许合并的角色
  - 配合 Merge Request 权限

推荐配置：
  main:    只允许 Maintainers 合并, 无人可直接推送
  develop: 允许 Developers 合并
```

## 9. Detached HEAD 是什么？如何处理？

**回答：**

```
Detached HEAD = HEAD 直接指向 commit，而不是分支

正常状态：
  HEAD → main → commit D
  
Detached HEAD：
  HEAD → commit C（没有分支指向它）
  
触发场景：
  git checkout abc1234     # checkout 某个 commit
  git checkout v1.0        # checkout 某个标签
  git checkout origin/main # checkout 远程分支

风险：
  在 detached HEAD 状态下的新 commit 不属于任何分支
  切换分支后，这些 commit 变成"孤儿"，最终被 GC 清理

  HEAD → E (新commit，无分支指向)
         ↑
  A ← B ← C ← D (main)
```

### 处理方法

```bash
# 1. 回到正常状态
git switch main

# 2. 如果在 detached HEAD 做了有用的提交
#    创建新分支保存
git switch -c save-my-work

# 3. 如果已经切走了，用 reflog 找回
git reflog
# abc1234 HEAD@{2}: commit: my detached work
git branch save-my-work abc1234

# 4. 安全地查看历史版本（推荐）
git switch --detach v1.0   # 明确告知进入 detached HEAD
# 查看完毕后
git switch main            # 回到正常
```

## 10. 分支的合并策略选择？（PR/MR 合并按钮）

**回答：**

### GitHub PR 三种合并方式

```
1. Create a merge commit（合并提交）
   → 保留所有提交 + 创建一个合并 commit
   → 历史完整但不够清晰
   
   main: A → B → M (merge commit)
                ↗
   feature: C → D

2. Squash and merge（压缩合并）
   → 将所有提交压缩为一个 commit 合并
   → 历史整洁, 但丢失 feature 的详细提交
   
   main: A → B → S (squashed single commit)

3. Rebase and merge（变基合并）
   → 将提交变基到 main 上，线性历史
   → 历史清晰
   
   main: A → B → C' → D' (rebased commits)
```

### 选择建议

| 场景 | 推荐方式 |
|------|---------|
| 保留完整历史 | Merge commit |
| 小功能/bug修复 | Squash and merge |
| 追求线性历史 | Rebase and merge |
| 大功能（多个有意义的提交） | Rebase and merge |
| 外部贡献者的 PR | Squash and merge |

```
团队统一策略很重要！
推荐：
  - 默认 Squash and merge（小 PR）
  - 大功能用 Rebase and merge
  - 在仓库设置中限制可用的合并方式
```
