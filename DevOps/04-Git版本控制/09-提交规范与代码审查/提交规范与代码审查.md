## 1. Conventional Commits 规范？

**回答：**

```
格式:
  <type>(<scope>): <subject>
  <空行>
  <body>
  <空行>
  <footer>

各字段说明:
  type   → 提交类型（必填）
  scope  → 影响范围（可选, 如模块名）
  subject → 简要描述（必填, 不超过 72 字符）
  body   → 详细描述（可选, 说明 why 和 what）
  footer → 关联信息（可选, BREAKING CHANGE / issue 引用）
```

### type 类型

| type | 说明 | 示例 |
|------|------|------|
| feat | 新功能 | feat(auth): add SSO login |
| fix | Bug 修复 | fix(api): resolve null pointer on empty response |
| docs | 文档 | docs(readme): add deployment section |
| style | 格式（不影响逻辑） | style: fix indentation in config |
| refactor | 重构（非 feat/fix） | refactor(db): extract connection pool |
| perf | 性能优化 | perf(query): add index for user lookup |
| test | 测试 | test(auth): add login failure tests |
| chore | 构建/工具/依赖 | chore: upgrade webpack to v5 |
| ci | CI/CD 配置 | ci: add deploy stage to pipeline |
| build | 构建系统 | build: switch from webpack to vite |
| revert | 回退 | revert: revert "feat(auth): add SSO login" |

### 完整示例

```
feat(user): add email verification flow

Implement email verification for new user registration.
Users now receive a verification link upon signup.

- Add verification token generation
- Add email sending service integration
- Add verification endpoint

Closes #234
BREAKING CHANGE: /api/register now requires email field
```

## 2. .gitignore 规则详解？

**回答：**

```gitignore
# .gitignore 语法

# 1. 注释
# 以 # 开头的行

# 2. 通配符
*        # 匹配任意字符（不含 /）
**       # 匹配任意路径（含多级目录）
?        # 匹配单个字符
[abc]    # 匹配 a/b/c 中的一个
[0-9]    # 匹配数字

# 3. 目录 vs 文件
logs/    # 只忽略目录
logs     # 忽略目录和文件

# 4. 取反
*.log        # 忽略所有 .log
!important.log # 但保留 important.log

# 5. 路径
/build   # 只忽略根目录下的 build
build/   # 忽略所有目录中的 build/

# 6. 多级目录
**/logs       # 任意深度的 logs 目录
**/logs/*.log # 任意深度 logs 目录下的 .log 文件
```

## 3. .gitattributes 配置？

**回答：**

```gitattributes
# .gitattributes — 路径级别的 Git 属性配置

# ===== 行尾处理 =====
# 自动检测文本文件
* text=auto

# 强制行尾
*.sh    text eol=lf
*.bat   text eol=crlf
*.ps1   text eol=crlf

# 明确指定文本/二进制
*.md    text
*.txt   text
*.png   binary
*.jpg   binary
*.pdf   binary

# ===== Git LFS =====
*.psd   filter=lfs diff=lfs merge=lfs -text
*.zip   filter=lfs diff=lfs merge=lfs -text
*.mp4   filter=lfs diff=lfs merge=lfs -text
*.model filter=lfs diff=lfs merge=lfs -text

# ===== 自定义 diff =====
*.lock  linguist-generated=true
*.min.js linguist-generated=true
# GitHub 统计不计这些自动生成文件

# ===== 合并策略 =====
# 合并冲突时总保留当前版本
package-lock.json merge=ours
yarn.lock merge=ours

# ===== 导出忽略 =====
# git archive 时排除
.gitignore export-ignore
.gitattributes export-ignore
.github/ export-ignore
tests/ export-ignore
```

### 行尾问题

```
跨平台行尾问题:
  Windows: CRLF (\r\n)
  Linux/macOS: LF (\n)

解决方案:
  1. .gitattributes 统一规则（推荐）
  2. core.autocrlf 配置（备选）

# core.autocrlf 设置:
  true   → 提交时 CRLF→LF, checkout 时 LF→CRLF (Windows)
  input  → 提交时 CRLF→LF, checkout 不转换 (Linux/Mac)
  false  → 不转换
```

## 4. Code Review 最佳实践？

**回答：**

### Review 原则

```
1. 及时性
   → 收到 Review 请求后 24 小时内回复
   → PR 不应超过 400 行变更

2. 发现问题的优先级
   ├── P0: 安全漏洞、数据丢失风险
   ├── P1: Bug、逻辑错误
   ├── P2: 设计问题、可维护性
   ├── P3: 性能优化建议
   └── P4: 代码风格、命名建议（nit）

3. 评论规范
   → 区分"必须修改"和"建议"
   → 前缀标记:
     "nit:"       → 可选的小建议
     "question:"  → 需要解释
     "suggestion:" → 改进建议
     "blocker:"   → 必须修改才能合并
```

## 5. Code Review Checklist（审查清单）？

**回答：**

```
Code Review 检查清单:

✅ 功能正确性
  □ 逻辑是否正确？
  □ 边界条件是否处理？
  □ 错误处理是否完善？
  □ 是否符合需求描述？

✅ 安全性
  □ 是否有注入风险（SQL/XSS/SSRF）？
  □ 是否有敏感信息泄露？
  □ 输入是否经过验证？
  □ 权限检查是否正确？

✅ 性能
  □ 是否有 N+1 查询？
  □ 是否有不必要的循环/递归？
  □ 是否使用了合适的数据结构？
  □ 是否考虑了大数据量场景？

✅ 可维护性
  □ 命名是否清晰（变量/函数/类）？
  □ 函数是否职责单一？
  □ 是否有重复代码？
  □ 复杂逻辑是否有注释？

✅ 测试
  □ 是否有单元测试？
  □ 测试覆盖了正常/异常路径？
  □ 测试是否可维护？

✅ 架构
  □ 变更是否符合现有架构？
  □ 依赖关系是否合理？
  □ 是否引入了不必要的耦合？

✅ 运维
  □ 是否有日志输出？
  □ 是否有监控指标？
  □ 是否考虑了向后兼容？
  □ 数据库变更是否有回滚方案？
```

## 6. Signed Commits（签名提交）？

**回答：**

```bash
# 签名提交 = 用 GPG/SSH 密钥签名，证明提交者身份

# ===== GPG 签名 =====
# 生成 GPG 密钥
gpg --full-generate-key

# 查看密钥
gpg --list-secret-keys --keyid-format=long

# 配置 Git 使用 GPG
git config --global user.signingkey <KEY_ID>
git config --global commit.gpgsign true    # 默认签名所有提交
git config --global tag.gpgSign true       # 默认签名所有标签

# 签名提交
git commit -S -m "feat: signed commit"

# 签名标签
git tag -s v1.0.0 -m "Release v1.0.0"

# 验证签名
git log --show-signature
git verify-commit HEAD
git verify-tag v1.0.0

# ===== SSH 签名（Git 2.34+）=====
# 更简单，无需 GPG
git config --global gpg.format ssh
git config --global user.signingkey ~/.ssh/id_ed25519.pub
git config --global commit.gpgsign true

# GitHub/GitLab 配置:
#   将公钥添加到账户设置 → SSH and GPG keys
#   提交会显示 "Verified" 标记

# 导出 GPG 公钥（添加到 GitHub）
gpg --armor --export <KEY_ID>
```

## 7. 语义化版本（Semantic Versioning）？

**回答：**

```
语义化版本格式: MAJOR.MINOR.PATCH[-prerelease][+build]

示例: 2.1.3-beta.1+build.123

  MAJOR (主版本) → 不兼容的 API 变更 (BREAKING CHANGE)
  MINOR (次版本) → 向后兼容的新功能 (feat)
  PATCH (补丁版本) → 向后兼容的 Bug 修复 (fix)

版本号递增规则:
  1.0.0 → 首个稳定版
  1.0.1 → 修复 Bug
  1.1.0 → 新增功能
  2.0.0 → 不兼容变更

预发布版本:
  1.0.0-alpha.1    → 内部测试
  1.0.0-beta.1     → 公开测试
  1.0.0-rc.1       → 发布候选

Git Tag 与版本:
  git tag v1.0.0
  git tag -a v1.0.0 -m "Release version 1.0.0"
  git push origin v1.0.0
  git push origin --tags
```

### 自动化版本管理

```bash
# standard-version（基于 Conventional Commits 自动判断版本）
npx standard-version          # 自动决定升级类型
npx standard-version --first-release
npx standard-version --release-as minor

# semantic-release（完全自动化）
# 根据 commit 类型:
#   fix: → PATCH
#   feat: → MINOR
#   BREAKING CHANGE: → MAJOR
```

## 8. CODEOWNERS 文件？

**回答：**

```
# .github/CODEOWNERS (GitHub)
# .gitlab/CODEOWNERS (GitLab)
# 
# 定义代码所有者 → 修改对应文件时自动请求 Review

# 格式: <pattern>  <owner1> <owner2> ...

# 默认所有文件的 owner
*                    @team-leads

# 特定目录
/docs/               @docs-team
/src/api/            @backend-team
/src/frontend/       @frontend-team
/infrastructure/     @devops-team
/.github/            @devops-team

# 特定文件
Dockerfile           @devops-team
docker-compose.yml   @devops-team
*.tf                 @infra-team

# 特定文件类型
*.js                 @frontend-team
*.go                 @backend-team

# 敏感文件 — 需要安全团队审查
/security/           @security-team
*.pem                @security-team
```

```
CODEOWNERS 工作流:
  1. 开发者创建 PR
  2. GitHub 自动根据变更文件匹配 CODEOWNERS
  3. 自动将对应 owner 添加为 Reviewer
  4. 配合 Branch Protection:
     → Require review from Code Owners
     → 必须 Code Owner 批准才能合并
```

## 9. Git Blame 与问题追溯？

**回答：**

```bash
# git blame — 查看每一行的最后修改者

# 基本用法
git blame <file>

# 指定行范围
git blame -L 10,20 <file>
git blame -L '/func main/,/^}/' <file>    # 正则范围

# 忽略空格变更
git blame -w <file>

# 检测行移动/复制
git blame -M <file>      # 同文件内移动
git blame -C <file>      # 跨文件复制
git blame -CCC <file>    # 任意提交中查找

# 忽略特定提交（如格式化提交）
git blame --ignore-rev <hash> <file>

# 批量忽略（创建忽略列表）
echo "<format-commit-hash>" > .git-blame-ignore-revs
git config blame.ignoreRevsFile .git-blame-ignore-revs

# GitHub 也支持 .git-blame-ignore-revs 文件
# 在 blame 视图中自动忽略这些提交
```

### 问题追溯流程

```
发现 Bug → 追溯流程:
  1. git blame 找到引入问题的提交
  2. git show <commit> 查看完整变更
  3. git log --follow <file> 查看文件历史
  4. git bisect 二分法定位（处理复杂情况）

组合使用:
  git log -p -S "buggy_function" -- <file>
  → 搜索添加/删除了 "buggy_function" 的提交
  
  git log --all --oneline --graph -- <file>
  → 可视化文件在所有分支上的变更历史
```

## 10. PR/MR 合并策略与分支保护？

**回答：**

### 合并策略比较

```
三种 PR/MR 合并方式:

1. Merge Commit (Create a merge commit)
   main: A---B-------M
              \     /
   feature:    C---D
   
   特点: 保留完整分支历史
   适用: 需要追溯 feature 分支的完整上下文

2. Squash and Merge
   main: A---B---S
   (S 包含 C+D 的所有变更)
   
   特点: 一个 feature 一个提交, 历史简洁
   适用: feature 分支有很多 WIP 提交

3. Rebase and Merge
   main: A---B---C'---D'
   
   特点: 线性历史, 每个提交保留
   适用: 每个提交都有意义且独立
```

### 分支保护规则

```yaml
# GitHub Branch Protection（main 分支推荐配置）
Branch protection rules:
  ✅ Require a pull request before merging
    ✅ Required approving reviews: 2
    ✅ Dismiss stale reviews on new pushes
    ✅ Require review from Code Owners
  ✅ Require status checks to pass
    ✅ Require branches to be up to date
    Status checks: [ci/test, ci/lint, security/scan]
  ✅ Require conversation resolution
  ✅ Require signed commits
  ✅ Require linear history  # 禁止 merge commits
  ✅ Include administrators   # 管理员也受限
  ✅ Restrict force pushes
  ✅ Restrict deletions

# GitLab Protected Branches
Protected branches:
  Branch: main
  Allowed to merge: Maintainers
  Allowed to push: No one
  Allowed to force push: No
  Code owner approval: Required
```
