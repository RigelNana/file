# 远程仓库与协作

---

## 1. git remote 远程仓库管理？

**回答：**

```bash
# 查看远程仓库
git remote                       # 列出远程名称
git remote -v                    # 显示 URL
# origin  git@github.com:user/repo.git (fetch)
# origin  git@github.com:user/repo.git (push)

# 添加远程仓库
git remote add origin git@github.com:user/repo.git
git remote add upstream git@github.com:org/repo.git  # 上游仓库

# 修改远程 URL
git remote set-url origin git@github.com:user/new-repo.git

# 重命名远程
git remote rename origin old-origin

# 删除远程
git remote remove upstream

# 查看远程详细信息
git remote show origin
# → 显示远程分支、追踪关系、推送/拉取配置

# 清理已删除的远程分支的本地追踪
git remote prune origin
```

## 2. git fetch、git pull、git push 详解？

**回答：**

### fetch

```bash
# 获取远程更新（不修改工作区，最安全）
git fetch origin                 # 获取 origin 所有分支
git fetch origin main            # 只获取 main 分支
git fetch --all                  # 获取所有远程仓库
git fetch --prune                # 获取并清理已删除的远程分支

# fetch 后远程跟踪分支更新了
# 但本地分支和工作区没有变化
# 可以对比后再决定如何合并
git log main..origin/main        # 查看远程有哪些新提交
git diff main origin/main        # 查看差异
```

### pull

```bash
# fetch + merge（默认）
git pull origin main

# fetch + rebase（推荐，保持线性历史）
git pull --rebase origin main

# 设置默认 pull 策略
git config --global pull.rebase true      # 默认 rebase
git config --global pull.ff only          # 只允许 fast-forward

# 带自动 stash（有未提交修改时）
git pull --autostash

# pull 策略配置（避免每次被提示）
git config --global pull.rebase true
# 或
git config --global pull.rebase merges    # 保留本地 merge commits
```

### push

```bash
# 基本推送
git push origin main

# 推送并设置上游追踪
git push -u origin main
git push --set-upstream origin feature

# 推送所有分支
git push --all origin

# 推送标签
git push origin v1.0
git push origin --tags

# 强制推送（危险！）
git push --force origin feature        # 覆盖远程
git push --force-with-lease origin feature  # 安全的强制推送（推荐）
# --force-with-lease 只有在远程没有别人新推送时才会成功

# 删除远程分支
git push origin --delete feature

# 推送到不同名称
git push origin local-branch:remote-branch
```

## 3. 上游追踪分支（Tracking Branch）？

**回答：**

```bash
# 追踪关系：本地分支 ↔ 远程跟踪分支
# main → origin/main

# 查看追踪关系
git branch -vv
# * main     abc1234 [origin/main] latest commit
#   feature  def5678 [origin/feature: ahead 2, behind 1] wip

# ahead 2:  本地比远程多 2 个提交（需要 push）
# behind 1: 远程比本地多 1 个提交（需要 pull）

# 设置追踪
git branch --set-upstream-to=origin/main main
git branch -u origin/main

# 取消追踪
git branch --unset-upstream

# 创建分支时自动追踪
git switch -c feature origin/feature
# 或
git checkout --track origin/feature

# 推送时设置追踪
git push -u origin feature
# 之后可以直接 git push / git pull 而不需要指定远程和分支
```

## 4. Fork 和 Pull Request 工作流？

**回答：**

```
开源项目协作的标准流程：

1. Fork（在 GitHub/GitLab 上复制仓库到自己账户）
   org/project → your-user/project

2. Clone 自己的 Fork
   git clone git@github.com:your-user/project.git

3. 添加上游仓库
   git remote add upstream git@github.com:org/project.git

4. 创建功能分支
   git switch -c fix/bug-123

5. 开发、提交
   git commit -m "fix: resolve bug #123"

6. 推送到自己的 Fork
   git push origin fix/bug-123

7. 在 GitHub 上创建 Pull Request
   your-user/project:fix/bug-123 → org/project:main

8. Code Review、CI 检查通过后合并

9. 同步上游更新
   git fetch upstream
   git switch main
   git merge upstream/main
   git push origin main
```

### 保持 Fork 同步

```bash
# 定期同步上游
git fetch upstream
git switch main
git rebase upstream/main        # 或 merge
git push origin main

# 更新功能分支
git switch feature
git rebase main
```

## 5. git push --force 和 --force-with-lease？

**回答：**

```bash
# --force：强制覆盖远程分支（危险！）
git push --force origin feature
# 会丢失远程上其他人的提交！

# --force-with-lease：安全的强制推送（推荐）
git push --force-with-lease origin feature
# 只有远程分支的状态和你上次 fetch 时一致才会成功
# 如果有人在你之后推送了新提交，会拒绝

# 需要强制推送的场景：
#   1. rebase 后推送（commit hash 变了）
#   2. amend 后推送
#   3. 修改了提交历史

# 使用建议：
#   ✅ 个人分支随意（但用 --force-with-lease 更安全）
#   ❌ 公共分支永远不要 --force
#   ❌ main/develop 分支应该设置保护规则禁止 force push
```

## 6. 多人协作的冲突场景？

**回答：**

### 推送被拒绝

```bash
# 场景：你和同事同时修改了 main
git push origin main
# ! [rejected]        main -> main (non-fast-forward)
# error: failed to push some refs

# 原因：远程有你没有的新提交

# 解决方案 1：先 pull 再 push
git pull --rebase origin main
# 解决可能的冲突
git push origin main

# 解决方案 2：先 fetch 查看再合并
git fetch origin
git log main..origin/main        # 看看别人改了什么
git rebase origin/main           # 或 merge
git push origin main
```

### 推送大文件被拒绝

```bash
# GitHub 限制单文件 100MB
# 如果误提交了大文件：

# 1. 如果还没 push
git reset HEAD~1                 # 撤回提交
# 添加到 .gitignore
git commit

# 2. 如果已经在历史中
# 使用 git filter-repo（推荐）或 BFG Repo-Cleaner
git filter-repo --strip-blobs-bigger-than 100M

# 正确做法：使用 Git LFS
git lfs track "*.bin"
git add .gitattributes
```

## 7. git clone 的各种方式和优化？

**回答：**

```bash
# 完整克隆
git clone git@github.com:user/repo.git

# 浅克隆（只最近 N 个提交）
git clone --depth 1 git@github.com:user/repo.git
# CI/CD 中常用，减少克隆时间和空间

# 单分支克隆
git clone --single-branch --branch main git@github.com:user/repo.git

# 裸克隆（没有工作区，用于镜像）
git clone --bare git@github.com:user/repo.git

# 镜像克隆（完整镜像包含所有引用）
git clone --mirror git@github.com:user/repo.git

# 稀疏克隆（只检出部分目录，Git 2.25+）
git clone --filter=blob:none --sparse git@github.com:user/repo.git
cd repo
git sparse-checkout set src/ docs/

# 部分克隆（不下载 blob，按需获取）
git clone --filter=blob:none git@github.com:user/repo.git
# Blob 在 checkout 时按需下载

# 浅克隆后转为完整克隆
git fetch --unshallow

# 克隆到指定目录
git clone git@github.com:user/repo.git my-project
```

## 8. Git 凭据管理？

**回答：**

```bash
# 凭据存储方式
git config --global credential.helper store     # 明文存储在 ~/.git-credentials（不推荐）
git config --global credential.helper cache     # 内存缓存（默认15分钟）
git config --global credential.helper 'cache --timeout=3600'  # 缓存1小时

# macOS Keychain
git config --global credential.helper osxkeychain

# Windows Credential Manager
git config --global credential.helper manager

# Linux libsecret
git config --global credential.helper /usr/lib/git-core/git-credential-libsecret

# SSH 认证（推荐）
# 生成密钥
ssh-keygen -t ed25519 -C "you@example.com"
# 添加到 SSH Agent
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519
# 将公钥添加到 GitHub/GitLab

# GitHub 个人访问令牌（PAT）
# HTTPS 方式时用 PAT 替代密码
# Token 权限最小化

# GitHub CLI
gh auth login               # 交互式登录
gh auth status               # 查看状态
```

## 9. 多远程仓库管理？

**回答：**

```bash
# 场景：同时使用 GitHub 和 GitLab

# 添加多个远程
git remote add github git@github.com:user/repo.git
git remote add gitlab git@gitlab.com:user/repo.git

# 推送到指定远程
git push github main
git push gitlab main

# 同时推送到多个远程（配置 push URL）
git remote set-url --add --push origin git@github.com:user/repo.git
git remote set-url --add --push origin git@gitlab.com:user/repo.git
git push origin main    # 同时推送到两个

# 查看配置
git remote -v
# origin  git@github.com:user/repo.git (fetch)
# origin  git@github.com:user/repo.git (push)
# origin  git@gitlab.com:user/repo.git (push)

# 场景：Fork 协作
git remote add upstream git@github.com:org/repo.git
# origin:   自己的 Fork
# upstream: 上游仓库

# 从上游同步
git fetch upstream
git rebase upstream/main
```

## 10. Refspec 引用规范是什么？

**回答：**

```bash
# Refspec 格式: +<src>:<dst>
# + 表示非快进也更新（force）
# src: 源引用
# dst: 目标引用

# 默认 fetch refspec（在 .git/config 中）
[remote "origin"]
    url = git@github.com:user/repo.git
    fetch = +refs/heads/*:refs/remotes/origin/*
    # 将远程所有分支映射为 origin/xxx

# 手动指定 refspec
git fetch origin main:refs/remotes/origin/main
git fetch origin +refs/tags/*:refs/tags/*

# push refspec
git push origin main:main                     # 推送 main 到远程 main
git push origin feature:refs/heads/feature    # 完整写法
git push origin :feature                      # 删除远程 feature 分支

# 高级用法：只获取特定分支
git config remote.origin.fetch "+refs/heads/main:refs/remotes/origin/main"
git config --add remote.origin.fetch "+refs/heads/develop:refs/remotes/origin/develop"
# 只跟踪 main 和 develop 分支

# Pull Request 引用
# GitHub 的 PR 可以通过 refspec 获取
git fetch origin +refs/pull/123/head:refs/pull/123
git checkout refs/pull/123
```
