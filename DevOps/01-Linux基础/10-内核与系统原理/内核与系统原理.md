## 一、Linux 内核概述

### 1. Linux 内核的基本架构？

**答：**

```
Linux 内核架构 (宏内核 Monolithic Kernel):

┌─────────────────────────────────────────────────────────┐
│                    用户空间 (User Space)                  │
│  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐     │
│  │Shell  │  │ App  │  │ libc │  │ GUI  │  │ 守护进程│   │
│  └──┬───┘  └──┬───┘  └──┬───┘  └──┬───┘  └──┬───┘     │
└─────┼────────┼────────┼────────┼────────┼──────────────┘
══════╪════════╪════════╪════════╪════════╪══════════════
 系统调用接口 (System Call Interface) ~ 400+ 个系统调用
══════╪════════╪════════╪════════╪════════╪══════════════
┌─────┴────────┴────────┴────────┴────────┴──────────────┐
│                    内核空间 (Kernel Space)                │
│  ┌─────────────────────────────────────────────────┐    │
│  │              进程调度器 (Scheduler)               │    │
│  │        CFS / RT / Deadline / EEVDF               │    │
│  └─────────────────────────────────────────────────┘    │
│  ┌────────────┐  ┌────────────┐  ┌────────────────┐    │
│  │ 内存管理    │  │  VFS       │  │   网络子系统    │    │
│  │ (MM)       │  │ (虚拟文件   │  │  TCP/IP 协议栈  │    │
│  │ 虚拟内存    │  │  系统)      │  │  Netfilter     │    │
│  │ 页面分配    │  │            │  │  Socket        │    │
│  └─────┬──────┘  └─────┬──────┘  └──────┬─────────┘    │
│  ┌─────┴──────┐  ┌─────┴──────┐  ┌──────┴─────────┐    │
│  │ 块设备层    │  │ 文件系统    │  │   网络驱动      │    │
│  │ IO 调度器   │  │ ext4/XFS.. │  │                │    │
│  └─────┬──────┘  └────────────┘  └──────┬─────────┘    │
│  ┌─────┴───────────────────────────────┴──────────┐    │
│  │              设备驱动框架                         │    │
│  │        字符设备 / 块设备 / 网络设备                │    │
│  └─────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘
═══════════════════════════════════════════════════════════
                    硬件 (Hardware)
         CPU  |  内存  |  磁盘  |  网卡  |  ...
```

**宏内核 vs 微内核：**
- 宏内核（Linux）：所有核心功能在一个大的内核空间运行，效率高
- 微内核（Minix, QNX）：只有最基本功能在内核，其他在用户空间，稳定但性能低
- Linux 通过内核模块 (LKM) 实现灵活性

### 2. 用户空间和内核空间的区别？

**答：**

```
地址空间划分 (64位系统):
┌──────────────────────────┐  0xFFFFFFFFFFFFFFFF
│     内核空间              │
│  (所有进程共享相同映射)    │  128TB
│  直接访问硬件              │
├──────────────────────────┤  0xFFFF800000000000
│     非规范地址区域          │  (未使用)
├──────────────────────────┤  0x00007FFFFFFFFFFF
│     用户空间              │
│  (每个进程独立地址空间)    │  128TB
│  需要通过系统调用访问内核   │
└──────────────────────────┘  0x0000000000000000

关键区别:
┌──────────┬────────────────────┬────────────────────┐
│          │ 用户空间            │ 内核空间            │
├──────────┼────────────────────┼────────────────────┤
│ 权限      │ Ring 3 (最低)      │ Ring 0 (最高)      │
│ 地址空间  │ 进程独立            │ 所有进程共享        │
│ 访问方式  │ 直接执行            │ 通过系统调用        │
│ 崩溃影响  │ 只影响该进程        │ 整个系统崩溃       │
│ 能做什么  │ 普通计算           │ 硬件访问、中断处理  │
└──────────┴────────────────────┴────────────────────┘

内核态/用户态切换触发条件:
1. 系统调用 (syscall)       — 主动请求内核服务
2. 异常 (exception)         — 缺页、除零
3. 中断 (interrupt)         — 硬件中断
```

### 3. 系统调用的工作过程？

**答：**

```
系统调用过程 (以 x86_64 为例):

用户空间                          内核空间
┌──────────────┐
│ printf("hi") │
│   ↓          │
│ write(1,     │
│  "hi", 2)    │     libc 包装
│   ↓          │
│ syscall 指令  │─────────────────→ ┌──────────────────┐
│ rax = 1 (号) │   切换到 Ring 0    │ 系统调用入口       │
│ rdi = fd     │   保存用户寄存器   │ sys_call_table[1] │
│ rsi = buf    │                   │     ↓              │
│ rdx = count  │                   │ sys_write()        │
│              │                   │   → VFS            │
│              │                   │   → 文件系统        │
│              │                   │   → 设备驱动        │
│              │   切换回 Ring 3    │     ↓              │
│ 返回值 ←─────│←──────────────────│ 返回结果           │
│ (写入字节数)  │   恢复用户寄存器   └──────────────────┘
└──────────────┘

开销:
- 每次系统调用有上下文切换开销（~100-1000ns）
- 优化方法: vDSO（某些调用如 gettimeofday 直接在用户空间完成）
```

```bash
# 查看系统调用
ausyscall --dump                  # 所有系统调用编号
strace -c ls                     # 统计 ls 的系统调用
```

## 二、进程调度

### 4. Linux 进程调度算法？

**答：**

```
Linux 调度类 (Scheduling Classes) 优先级从高到低:

1. Stop 调度类   — 最高优先级，用于 CPU 热迁移等
2. Deadline 调度 — SCHED_DEADLINE，硬实时
3. RT 调度       — SCHED_FIFO / SCHED_RR（实时进程）
4. CFS 调度      — SCHED_NORMAL / SCHED_BATCH（普通进程）
5. Idle 调度     — SCHED_IDLE（最低优先级）

CFS (Completely Fair Scheduler):
- Linux 2.6.23 引入，默认调度器
- 核心 = 虚拟运行时间 (vruntime)
- 每个进程维护一个 vruntime
- 调度器选择 vruntime 最小的进程运行
- nice 值越高 → vruntime 增长越快 → 获得的 CPU 时间越少
- 使用红黑树存储可运行进程，O(log n) 查找

EEVDF (Earliest Eligible Virtual Deadline First):
- Linux 6.6 引入，替代 CFS
- 更好的延迟保证
```

```bash
# 查看进程调度策略
chrt -p <PID>               # 查看进程调度策略和优先级

# 设置调度策略
chrt -f -p 50 <PID>         # SCHED_FIFO，优先级 50
chrt -r -p 50 <PID>         # SCHED_RR，优先级 50

# nice 值 (-20 到 +19)
nice -n 10 command           # 以 nice=10 启动
renice -n -5 -p <PID>       # 修改运行中进程

# 查看调度信息
cat /proc/<PID>/sched        # 调度统计
cat /proc/<PID>/stat         # 包含优先级、nice 等
```

### 5. 中断和软中断？

**答：**

```
中断处理机制:

硬件中断 (Hardware Interrupt):
┌────────┐    IRQ     ┌────────┐    中断向量    ┌────────────┐
│ 硬件    │ ─────────→│ 中断   │ ─────────────→│ 中断处理    │
│ (网卡)  │           │ 控制器  │               │ 程序        │
└────────┘           └────────┘               └────────────┘
                                               top half (硬中断)
                                               - 快速处理
                                               - 关中断执行
                                               - 不可睡眠
                                                     │
                                                     ▼
                                               bottom half (软中断)
                                               - 延迟处理
                                               - 开中断执行
                                               - 可被打断

软中断类型 (softirq):
HI_SOFTIRQ          高优先级
TIMER_SOFTIRQ        定时器
NET_TX_SOFTIRQ       网络发送
NET_RX_SOFTIRQ       网络接收
BLOCK_SOFTIRQ        块设备
IRQ_POLL_SOFTIRQ     IRQ 轮询
TASKLET_SOFTIRQ      Tasklet
SCHED_SOFTIRQ        调度
HRTIMER_SOFTIRQ      高精度定时器
RCU_SOFTIRQ          RCU
```

```bash
# 查看中断
cat /proc/interrupts          # 硬件中断统计
cat /proc/softirqs            # 软中断统计

# 查看 CPU 软中断负载
mpstat -P ALL 1               # si 列

# 中断亲和性（绑定中断到特定 CPU）
cat /proc/irq/<IRQ>/smp_affinity        # 当前设置
echo 2 > /proc/irq/<IRQ>/smp_affinity   # 绑定到 CPU1

# 软中断过高的排查
# 1. 网络 RX 软中断过高 → 多队列网卡 + 中断亲和性
# 2. 定时器软中断 → 检查高精度定时器使用
```

## 三、内存管理深入

### 6. Linux 虚拟内存管理？

**答：**

```
虚拟内存地址空间 (进程视角):
┌─────────────────────┐ 高地址
│     内核空间          │
├─────────────────────┤ 
│     栈 (Stack)       │ ↓ 向下增长
│     ↓               │
│                     │
│     ↑               │
│     堆 (Heap)       │ ↑ 向上增长 (malloc/new)
├─────────────────────┤
│     BSS             │ 未初始化全局变量
├─────────────────────┤
│     Data            │ 已初始化全局变量
├─────────────────────┤
│     Text (Code)     │ 代码段 (只读)
└─────────────────────┘ 低地址 (0x0)

页表 (Page Table):
- 将虚拟地址映射到物理地址
- x86_64 使用 4 级页表 (PGD→PUD→PMD→PTE)
- 每个页 4KB (也支持 2MB/1GB 大页)
- TLB 缓存最近的地址转换

缺页中断 (Page Fault):
- 访问虚拟地址 → MMU 查页表 → 未映射 → 缺页中断
- 内核处理: 分配物理页 → 更新页表 → 返回
- 类型:
  - Minor Fault: 页在内存中(其他进程的或缓存)
  - Major Fault: 页在磁盘(swap)，需要磁盘 IO
```

```bash
# 查看进程内存映射
cat /proc/<PID>/maps            # 地址映射
cat /proc/<PID>/smaps           # 详细映射（包含 RSS/PSS/Swap）
pmap -x <PID>                  # 可读格式
cat /proc/<PID>/status | grep -E 'Vm|Rss|Swap'

# 缺页统计
ps -o min_flt,maj_flt,cmd -p <PID>
# min_flt: Minor Fault
# maj_flt: Major Fault（> 0 说明有 swap 活动）

# 查看系统内存详情
cat /proc/meminfo
# MemTotal:     总内存
# MemFree:      完全空闲
# MemAvailable: 可用（包括可回收缓存）
# Buffers:      块设备缓冲区
# Cached:       页缓存
# SwapTotal:    Swap 总量
# SwapFree:     Swap 空闲
# Slab:         内核 slab 分配器
# SReclaimable: 可回收 slab
# Dirty:        脏页（待写回）
# Writeback:    正在写回的页
# AnonPages:    匿名页（非文件映射）
# Mapped:       映射的文件页
# Shmem:        共享内存
# KernelStack:  内核栈
# PageTables:   页表
# HugePages_*:  大页
```

### 7. 大页 (Huge Pages) 和透明大页 (THP)？

**答：**

```bash
# 普通页: 4KB
# 大页 (HugePage): 2MB 或 1GB
# 优势: 减少 TLB miss → 提升性能（数据库、JVM 等大内存应用）

# ===== 静态大页 (HugePages) =====
# 查看大页信息
cat /proc/meminfo | grep Huge
# HugePages_Total: 0
# HugePages_Free:  0
# Hugepagesize:    2048 kB

# 配置大页（例如分配 1024 个 2MB 大页 = 2GB）
echo 1024 > /proc/sys/vm/nr_hugepages          # 临时
# /etc/sysctl.conf: vm.nr_hugepages = 1024      # 永久

# 挂载 hugetlbfs
mount -t hugetlbfs nodev /mnt/hugepages

# ===== 透明大页 (THP) =====
# THP 由内核自动管理，应用不需要修改
cat /sys/kernel/mm/transparent_hugepage/enabled
# [always] madvise never

# 某些数据库建议禁用 THP（如 MongoDB, Redis）
echo never > /sys/kernel/mm/transparent_hugepage/enabled
echo never > /sys/kernel/mm/transparent_hugepage/defrag

# 永久禁用: 添加内核参数
# transparent_hugepage=never
```

## 四、内核模块

### 8. 内核模块的管理？

**答：**

```bash
# 列出已加载模块
lsmod
lsmod | grep <module>

# 模块信息
modinfo <module>              # 模块详细信息
modinfo ext4                  # ext4 模块信息

# 加载模块
modprobe <module>              # 加载（自动处理依赖）
modprobe bonding               # 加载网卡绑定模块
insmod /path/to/module.ko      # 直接加载（不处理依赖）

# 卸载模块
modprobe -r <module>           # 卸载
rmmod <module>                 # 直接卸载

# 模块参数
modprobe bonding mode=1 miimon=100
# 或
echo "options bonding mode=1 miimon=100" > /etc/modprobe.d/bonding.conf

# 开机自动加载
# RHEL
echo "bonding" > /etc/modules-load.d/bonding.conf
# Debian
echo "bonding" >> /etc/modules

# 黑名单（禁止加载）
echo "blacklist nouveau" > /etc/modprobe.d/blacklist-nouveau.conf
echo "install nouveau /bin/false" >> /etc/modprobe.d/blacklist-nouveau.conf

# 模块目录
ls /lib/modules/$(uname -r)/kernel/
```

## 六、Namespace 和 Cgroup

### 11. Linux Namespace 有哪些？（容器基础）

**答：**

Namespace 是 Linux 内核实现资源隔离的机制，是容器技术的基础。

| Namespace | 隔离内容 | 系统调用标志 | 说明 |
|-----------|---------|-------------|------|
| Mount (mnt) | 文件系统挂载点 | CLONE_NEWNS | 独立挂载视图 |
| UTS | 主机名和域名 | CLONE_NEWUTS | 独立 hostname |
| IPC | 进程间通信 | CLONE_NEWIPC | 独立 IPC |
| PID | 进程 ID | CLONE_NEWPID | 独立 PID 树 |
| Network (net) | 网络栈 | CLONE_NEWNET | 独立网络 |
| User | 用户和组 | CLONE_NEWUSER | UID/GID 映射 |
| Cgroup | cgroup 根目录 | CLONE_NEWCGROUP | 独立 cgroup 视图 |
| Time | 系统时钟 | CLONE_NEWTIME | 独立时钟（Linux 5.6+） |

```bash
# 查看进程的命名空间
ls -la /proc/<PID>/ns/
lsns                         # 列出所有命名空间

# 创建和进入命名空间
unshare --pid --mount --fork --mount-proc /bin/bash
# --pid:   新 PID namespace
# --mount: 新 Mount namespace
# --fork:  fork 新进程
# --mount-proc: 重新挂载 /proc

# 进入已有命名空间
nsenter -t <PID> --pid --net --mount bash
# 进入容器的命名空间（等价于 docker exec）
```

### 12. Cgroup v1 vs v2？

**答：**

```
Cgroup (Control Groups): 限制和隔离进程组的资源使用

Cgroup v1 vs v2:
┌──────────────┬──────────────────┬──────────────────┐
│              │ Cgroup v1        │ Cgroup v2         │
├──────────────┼──────────────────┼──────────────────┤
│ 层级结构      │ 多层级(每个控制器│ 单一统一层级      │
│              │ 独立挂载)        │                   │
│ 控制器绑定    │ 控制器可在不同   │ 控制器绑定到      │
│              │ 层级             │ 同一层级           │
│ 线程支持      │ 有限             │ 线程级控制        │
│ 压力监控      │ 无               │ PSI（Pressure     │
│              │                  │  Stall Info）     │
│ 挂载点       │ /sys/fs/cgroup/  │ /sys/fs/cgroup/   │
│              │ ├─ cpu/          │ (统一目录)         │
│              │ ├─ memory/       │                   │
│              │ └─ ...           │                   │
│ 使用现状      │ 传统             │ Linux 5.x+ 推荐  │
└──────────────┴──────────────────┴──────────────────┘
```

```bash
# 检查版本
stat -fc %T /sys/fs/cgroup/
# cgroup2fs → v2
# tmpfs     → v1

mount | grep cgroup

# ===== Cgroup v2 操作示例 =====
# 创建 cgroup
mkdir /sys/fs/cgroup/mygroup

# 添加控制器
echo "+cpu +memory +io" > /sys/fs/cgroup/cgroup.subtree_control

# 设置 CPU 限制（100ms 周期内最多用 50ms = 50% CPU）
echo "50000 100000" > /sys/fs/cgroup/mygroup/cpu.max

# 设置内存限制 (256MB)
echo 268435456 > /sys/fs/cgroup/mygroup/memory.max

# 设置 IO 限制
echo "8:0 rbps=10485760 wbps=10485760" > /sys/fs/cgroup/mygroup/io.max

# 将进程加入 cgroup
echo <PID> > /sys/fs/cgroup/mygroup/cgroup.procs

# 查看当前使用
cat /sys/fs/cgroup/mygroup/cpu.stat
cat /sys/fs/cgroup/mygroup/memory.current
cat /sys/fs/cgroup/mygroup/io.stat

# 查看 PSI（压力）
cat /sys/fs/cgroup/mygroup/cpu.pressure
cat /sys/fs/cgroup/mygroup/memory.pressure
cat /sys/fs/cgroup/mygroup/io.pressure
# some avg10=0.00 avg60=0.00 avg300=0.00 total=0
```

### 13. 如何编译 Linux 内核？

```bash
# 1. 安装依赖

    libssl-dev libelf-dev bc dwarves
```

### 14. eBPF 是什么？

**答：**

```
eBPF (extended Berkeley Packet Filter):
- 在内核中安全运行沙盒程序，无需修改内核源码或加载模块
- 验证器保证安全：不会崩溃内核、不会无限循环
- 使用场景：网络、安全、性能分析、可观测性

eBPF 架构:
┌─────────────────────────────────────────────────┐
│                    用户空间                       │
│  ┌─────────┐  ┌──────────┐  ┌──────────────┐    │
│  │bpftrace │  │  bcc     │  │ libbpf/cilium│    │
│  │ (一行)   │  │ (Python) │  │   (CO-RE)    │    │
│  └────┬────┘  └────┬─────┘  └──────┬───────┘    │
└───────┼────────────┼───────────────┼─────────────┘
════════╪════════════╪═══════════════╪═════════════
┌───────▼────────────▼───────────────▼─────────────┐
│  eBPF 虚拟机                                      │
│  验证器 → JIT 编译 → 运行在各种 hook 点            │
│  ├── kprobes   (内核函数)                          │
│  ├── tracepoints (静态追踪点)                      │
│  ├── XDP       (网络包处理)                        │
│  ├── TC        (流量控制)                          │
│  └── cgroup    (资源控制)                          │
│                                                   │
│  Maps (数据结构): 在内核和用户空间共享数据           │
└───────────────────────────────────────────────────┘

常用 eBPF 工具 (bcc-tools):
```

```bash
# 安装 bcc-tools
apt install bpfcc-tools          # Debian
```

### 15. 系统启动过程中内核做了什么？

**答：**

```
Linux 启动详细过程:

1. BIOS/UEFI
   └→ 硬件自检 (POST) → 找到启动设备

2. Bootloader (GRUB2)
   └→ 加载内核镜像 (vmlinuz) 和 initramfs 到内存

3. 内核初始化
   ├→ 解压自身
   ├→ 设置初始页表 (内存管理引导)
   ├→ 初始化 CPU、中断控制器
   ├→ 初始化内存管理 (伙伴系统、slab)
   ├→ 初始化调度器
   ├→ 初始化 VFS、设备模型
   ├→ 挂载 initramfs 为临时 rootfs
   └→ 执行 /init (initramfs 中)

4. initramfs
   ├→ 加载存储驱动 (SCSI、RAID、LVM)
   ├→ 解密磁盘 (LUKS)
   ├→ 找到真正的 root 分区
   ├→ switch_root 切换到真正的 rootfs
   └→ 执行 /sbin/init (systemd)

5. systemd (PID 1)
   ├→ 加载 /etc/systemd/system/ 配置
   ├→ 启动 default.target 依赖链
   │   ├→ sysinit.target (文件系统、swap、内核模块)
   │   ├→ basic.target (日志、socket、timer)
   │   ├→ multi-user.target (网络、服务)
   │   └→ graphical.target (显示管理器，如果有)
   └→ 系统就绪
```

```bash
# 查看启动过程
dmesg                           # 内核启动消息
journalctl -b                   # 当次启动所有日志
systemd-analyze                 # 启动时间
systemd-analyze blame           # 各服务耗时
systemd-analyze critical-chain  # 关键路径

# 重建 initramfs
# Debian
update-initramfs -u
# RHEL
dracut --force

# 查看 initramfs 内容
lsinitrd                        # RHEL
lsinitramfs /boot/initrd.img    # Debian
```
