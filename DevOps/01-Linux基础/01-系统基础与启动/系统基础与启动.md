## 一、Linux 概述

### 1. 什么是 Linux？Linux 和 Unix 的关系？

**答：** Linux 是由 Linus Torvalds 于 1991 年开发的开源类 Unix 操作系统内核。严格来说，Linux 只是内核，完整的操作系统称为 GNU/Linux。

| 特性 | Unix | Linux |
|------|------|-------|
| 开源 | 大多为商业闭源 | 开源（GPL 协议） |
| 代表 | Solaris, AIX, HP-UX, macOS | Ubuntu, CentOS, Debian, RHEL |
| 内核 | 各厂商自有内核 | Linux 内核 |
| 硬件支持 | 特定硬件平台 | 广泛支持 x86, ARM 等 |
| 使用成本 | 高（商业授权） | 免费（企业支持另计） |

### 2. 常见的 Linux 发行版有哪些？如何选择？

**答：**

| 发行版 | 包管理 | 特点 | 适用场景 |
|--------|--------|------|---------|
| **Ubuntu** | apt/deb | 社区活跃，文档丰富 | 开发环境、云服务器 |
| **CentOS/Rocky/Alma** | yum/rpm | RHEL 兼容，稳定 | 企业生产环境 |
| **RHEL** | yum/rpm | 商业支持，长期维护 | 金融、电信等企业 |
| **Debian** | apt/deb | 极度稳定，自由软件 | 服务器、嵌入式 |
| **Alpine** | apk | 极小体积（~5MB） | Docker 容器基础镜像 |
| **Amazon Linux** | yum/rpm | AWS 定制优化 | AWS 云环境 |
| **Arch Linux** | pacman | 滚动更新，最新软件 | 高级用户、开发 |
| **SUSE/openSUSE** | zypper/rpm | 企业级，YaST 管理工具 | 欧洲企业 |

**DevOps 场景选择建议：**
- 云服务器：Ubuntu Server LTS / Amazon Linux 2
- 容器基础镜像：Alpine / Distroless
- 传统企业：RHEL / Rocky Linux

### 3. 什么是 Linux 内核？用户空间和内核空间的区别？

**答：**

```
┌─────────────────────────────────────────────┐
│              用户空间 (User Space)            │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐       │
│  │   App1   │ │   App2   │ │  Shell  │       │
│  └─────────┘ └─────────┘ └─────────┘       │
│  ┌─────────────────────────────────────┐     │
│  │           C 标准库 (glibc)           │     │
│  └─────────────────────────────────────┘     │
├─────────── 系统调用接口 (System Call) ────────┤
│              内核空间 (Kernel Space)          │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐       │
│  │进程管理   │ │内存管理   │ │文件系统   │       │
│  └─────────┘ └─────────┘ └─────────┘       │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐       │
│  │网络协议栈 │ │设备驱动   │ │ 安全模块  │       │
│  └─────────┘ └─────────┘ └─────────┘       │
├─────────────────────────────────────────────┤
│                 硬件 (Hardware)               │
└─────────────────────────────────────────────┘
```

| 特性   | 用户空间        | 内核空间                     |
| ---- | ----------- | ------------------------ |
| 权限   | Ring 3（受限）  | Ring 0（特权）               |
| 内存访问 | 只能访问自己的虚拟地址 | 可访问所有物理内存                |
| 硬件访问 | 不能直接操作硬件    | 可直接操控硬件                  |
| 崩溃影响 | 进程崩溃不影响系统   | 内核崩溃导致系统宕机（Kernel Panic） |
| 切换   | 通过系统调用进入内核  | 系统调用返回到用户空间              |

**系统调用 (System Call)：** 用户态程序请求内核服务的唯一方式。常见系统调用：
- 文件：`open()`, `read()`, `write()`, `close()`
- 进程：`fork()`, `exec()`, `wait()`, `exit()`
- 网络：`socket()`, `bind()`, `listen()`, `accept()`
- 内存：`mmap()`, `brk()`

## 二、Linux 启动过程

### 4. Linux 的完整启动过程？（详细版）

**答：**

```
 电源开启
    │
    ▼
┌──────────┐
│ BIOS/UEFI │  1. 上电自检 (POST)
│           │  2. 检测 CPU、内存、外设
│           │  3. 查找启动设备 (硬盘/USB/网络)
│           │  4. 加载 MBR/GPT 中的 Bootloader
└────┬─────┘
     ▼
┌──────────┐
│ GRUB2     │  5. 显示启动菜单
│           │  6. 加载内核镜像 (vmlinuz) 到内存
│           │  7. 加载初始 RAM 磁盘 (initramfs/initrd)
│           │  8. 传递内核参数 (root=, quiet, splash 等)
└────┬─────┘
     ▼
┌──────────┐
│ Kernel    │  9. 解压内核，初始化内存管理
│           │  10. 初始化硬件驱动
│           │  11. 挂载 initramfs 为临时根文件系统
│           │  12. 执行 init 脚本，加载必要驱动
│           │  13. 挂载真实的根文件系统 (pivot_root)
│           │  14. 启动 PID 1 进程（systemd）
└────┬─────┘
     ▼
┌──────────┐
│ systemd   │  15. 读取 default.target
│ (PID 1)   │  16. 按依赖关系并行启动各个 Unit
│           │  17. 挂载文件系统 (/etc/fstab)
│           │  18. 启动网络、日志等基础服务
│           │  19. 启动 getty/sshd 等登录服务
└────┬─────┘
     ▼
  用户登录
```

### 5. BIOS 和 UEFI 的区别？MBR 和 GPT 的区别？

**答：**

**BIOS vs UEFI：**

| 特性   | BIOS                      | UEFI                                  |
| ---- | ------------------------- | ------------------------------------- |
| 全称   | Basic Input/Output System | Unified Extensible Firmware Interface |
| 启动模式 | 16位实模式                    | 32/64位保护模式                            |
| 启动速度 | 慢                         | 快（跳过部分自检）                             |
| 磁盘支持 | MBR（最大2TB）                | GPT（最大9.4ZB）                          |
| 安全启动 | 无                         | Secure Boot（验证启动签名）                   |
| 界面   | 文本                        | 图形化可鼠标操作                              |

**MBR vs GPT：**

| 特性 | MBR | GPT |
|------|-----|-----|
| 全称 | Master Boot Record | GUID Partition Table |
| 最大磁盘 | 2 TB | 9.4 ZB (9.4×10²¹ bytes) |
| 最大分区数 | 4个主分区（或3主+1扩展） | 128个分区 |
| 数据冗余 | 无，分区表损坏不可恢复 | 头尾各一份，可恢复 |
| 兼容性 | BIOS/UEFI | 仅 UEFI |

### 6. GRUB2 引导程序如何配置和修复？

**答：**

```bash
# GRUB2 配置文件
/etc/default/grub              # 主配置（修改这个）
/boot/grub/grub.cfg            # 生成的配置（不要直接改）

# 常修改的参数
GRUB_TIMEOUT=5                 # 启动菜单等待时间
GRUB_DEFAULT=0                 # 默认启动项
GRUB_CMDLINE_LINUX_DEFAULT="quiet splash"  # 默认内核参数
GRUB_CMDLINE_LINUX=""          # 所有内核的参数

# 更新 GRUB 配置
update-grub                    # Debian/Ubuntu
grub2-mkconfig -o /boot/grub2/grub.cfg  # RHEL/CentOS

# 修复 GRUB（系统无法启动时）
# 1. 用 Live USB 启动
# 2. 挂载系统分区
mount /dev/sda2 /mnt
mount /dev/sda1 /mnt/boot/efi  # UEFI 系统
mount --bind /dev /mnt/dev
mount --bind /proc /mnt/proc
mount --bind /sys /mnt/sys
# 3. chroot 进入系统
chroot /mnt
# 4. 重新安装 GRUB
grub-install /dev/sda          # BIOS
grub-install --target=x86_64-efi --efi-directory=/boot/efi  # UEFI
update-grub
```

### 7. 内核参数 (Kernel Parameters) 有哪些常用选项？

**答：**

```bash
# 查看当前内核参数
cat /proc/cmdline

# 常用内核启动参数
root=/dev/sda2              # 指定根文件系统
ro / rw                     # 只读/读写挂载根
quiet                       # 减少启动输出
splash                      # 显示启动画面
init=/bin/bash              # 指定 init 程序（救援用）
single / 1                  # 单用户模式
nomodeset                   # 禁用内核模式设置（显卡问题排查）
acpi=off                    # 禁用 ACPI（硬件兼容性）
mem=4G                      # 限制可用内存
console=ttyS0,115200        # 串口控制台（服务器/虚拟机）
net.ifnames=0               # 使用传统网卡命名 (eth0)

# 运行时修改内核参数
sysctl -a                   # 查看所有参数
sysctl -w vm.swappiness=10  # 临时修改
echo "vm.swappiness=10" >> /etc/sysctl.d/99-custom.conf  # 永久
sysctl --system             # 重新加载所有配置
```

## 三、systemd 深入

### 8. systemd 的核心概念和架构？

**答：**

```
systemd 架构:
┌──────────────────────────────────────┐
│                systemd                │
│  ┌────────┐ ┌─────────┐ ┌─────────┐ │
│  │systemctl│ │journalctl│ │loginctl │ │  ← 管理工具
│  └────────┘ └─────────┘ └─────────┘ │
│                                      │
│  ┌────────┐ ┌─────────┐ ┌─────────┐ │
│  │ service │ │  timer   │ │  mount  │ │
│  │  units  │ │  units   │ │  units  │ │  ← Unit 类型
│  └────────┘ └─────────┘ └─────────┘ │
│  ┌────────┐ ┌─────────┐ ┌─────────┐ │
│  │ socket │ │  target  │ │  path   │ │
│  │  units │ │  units   │ │  units  │ │
│  └────────┘ └─────────┘ └─────────┘ │
│                                      │
│  ┌────────────────────────────────┐  │
│  │        cgroups (资源控制)       │  │
│  └────────────────────────────────┘  │
└──────────────────────────────────────┘
```

**Unit 类型：**

| 类型      | 后缀       | 说明                  |
| ------- | -------- | ------------------- |
| Service | .service | 后台服务进程              |
| Socket  | .socket  | 套接字激活（按需启动服务）       |
| Target  | .target  | 一组 Unit 的集合（类似运行级别） |
| Timer   | .timer   | 定时器（替代 cron）        |
| Mount   | .mount   | 文件系统挂载点             |
| Path    | .path    | 文件路径监控              |
| Slice   | .slice   | 资源控制组 (cgroup)      |
| Device  | .device  | 内核设备                |
| Scope   | .scope   | 外部创建的进程组            |
| Swap    | .swap    | 交换空间                |

### 9. systemd 的 Unit 文件详解？

**答：**

```ini
# /etc/systemd/system/myapp.service
# 完整的 service unit 文件示例

[Unit]
Description=My Production Application
Documentation=https://docs.example.com
After=network-online.target postgresql.service redis.service
Wants=network-online.target
Requires=postgresql.service            # 强依赖，postgresql 停则 myapp 也停
BindsTo=                               # 更强的依赖
Conflicts=nginx-legacy.service         # 冲突，不能同时运行
ConditionPathExists=/opt/myapp/bin/app  # 路径存在才启动

[Service]
Type=notify                     # 服务类型
# Type 类型说明:
#   simple  - 默认，主进程即服务进程
#   forking - 传统守护进程（fork 后主进程退出）
#   oneshot - 一次性任务（配合 RemainAfterExit=yes）
#   notify  - 就绪后通过 sd_notify 通知 systemd
#   dbus    - 通过 D-Bus 信号通知就绪

User=appuser
Group=appgroup
WorkingDirectory=/opt/myapp

# 启动/停止/重载命令
ExecStartPre=/opt/myapp/bin/check-config.sh   # 启动前执行
ExecStart=/opt/myapp/bin/app --config /etc/myapp/config.yaml
ExecStartPost=/opt/myapp/bin/post-start.sh    # 启动后执行
ExecReload=/bin/kill -HUP $MAINPID
ExecStop=/bin/kill -SIGTERM $MAINPID
ExecStopPost=/opt/myapp/bin/cleanup.sh

# 重启策略
Restart=on-failure              # 重启条件
#   no        - 不自动重启（默认）
#   always    - 总是重启
#   on-failure - 异常退出时重启（退出码≠0）
#   on-abnormal - 信号/超时/看门狗时重启
RestartSec=5                    # 重启间隔
StartLimitBurst=5               # 启动次数限制（StartLimitInterval 内）
StartLimitIntervalSec=60        # 限制窗口（60秒内最多启动5次）

# 超时
TimeoutStartSec=30              # 启动超时
TimeoutStopSec=30               # 停止超时
WatchdogSec=30                  # 看门狗超时

# 环境变量
Environment=NODE_ENV=production
EnvironmentFile=/etc/myapp/env  # 从文件加载环境变量

# 资源限制
LimitNOFILE=65535               # 文件描述符限制
LimitNPROC=4096                 # 进程数限制
MemoryMax=2G                    # 内存限制 (cgroup v2)
CPUQuota=200%                   # CPU 限制（200% = 2核）

# 安全加固
NoNewPrivileges=true            # 禁止提权
ProtectSystem=strict            # 只读挂载 /usr /boot /etc
ProtectHome=true                # 不可访问 /home
PrivateTmp=true                 # 独立的 /tmp
ReadWritePaths=/var/lib/myapp   # 允许写入的路径

# 日志
StandardOutput=journal          # 标准输出写入 journald
StandardError=journal           # 标准错误写入 journald
SyslogIdentifier=myapp          # 日志标识

[Install]
WantedBy=multi-user.target      # 在哪个 target 下启用
```

### 10. systemd Target 与传统运行级别的对应关系？

**答：**

| 运行级别 | systemd Target | 说明 |
|----------|----------------|------|
| 0 | poweroff.target | 关机 |
| 1 | rescue.target | 单用户/救援模式（有 root shell） |
| 2, 3, 4 | multi-user.target | 多用户命令行（无 GUI） |
| 5 | graphical.target | 图形界面 |
| 6 | reboot.target | 重启 |
| - | emergency.target | 紧急模式（最小系统，只有根分区） |

```bash
# 查看/切换 target
systemctl get-default                    # 查看默认 target
systemctl set-default multi-user.target  # 设置默认
systemctl isolate rescue.target          # 切换到救援模式

# 查看 target 包含的服务
systemctl list-dependencies multi-user.target

# Target 之间的依赖
graphical.target → multi-user.target → basic.target → sysinit.target
```

### 11. systemd Timer 定时器 vs cron？

**答：**

| 特性   | cron   | systemd Timer          |
| ---- | ------ | ---------------------- |
| 日志   | 需要自行处理 | 自动记录到 journald         |
| 依赖管理 | 无      | 可声明依赖                  |
| 资源控制 | 无      | 支持 cgroup 限制           |
| 错过任务 | 不执行    | Persistent=true 可补执行   |
| 精度   | 分钟级    | 微秒级                    |
| 随机延迟 | 无      | RandomizedDelaySec 防惊群 |

```ini
# /etc/systemd/system/backup.timer
[Unit]
Description=Daily backup timer

[Timer]
OnCalendar=*-*-* 02:00:00      # 每天凌晨2点
# 更多时间表达式:
# OnCalendar=Mon..Fri *-*-* 09:00:00   # 工作日9点
# OnCalendar=*-*-01 00:00:00            # 每月1号
# OnCalendar=hourly                     # 每小时
# OnBootSec=5min                        # 开机5分钟后
# OnUnitActiveSec=1h                    # 上次执行后1小时

Persistent=true                 # 错过的执行在下次启动时补上
RandomizedDelaySec=300          # 随机延迟0-300秒
AccuracySec=1s                  # 计时精度

[Install]
WantedBy=timers.target

# /etc/systemd/system/backup.service
[Unit]
Description=Daily backup job

[Service]
Type=oneshot
ExecStart=/opt/scripts/backup.sh
```

```bash
# Timer 管理
systemctl enable --now backup.timer    # 启用定时器
systemctl list-timers --all            # 列出所有定时器
systemctl status backup.timer          # 查看定时器状态
journalctl -u backup.service          # 查看执行日志
```

```bash
    --with-http_ssl_module \
    --with-http_v2_module \
    --with-http_gzip_static_module

```

## 五、环境变量与 Shell 配置

### 15. Shell 启动时读取哪些配置文件？顺序是什么？

**答：**

```
Login Shell (SSH 登录、su - 切换):
┌──────────────────┐
│ /etc/profile      │  ← 全局，所有用户
│   └── /etc/profile.d/*.sh  ← 全局片段
├──────────────────┤
│ ~/.bash_profile   │  ← 用户级（按顺序找第一个存在的）
│ 或 ~/.bash_login  │
│ 或 ~/.profile     │
├──────────────────┤
│ ~/.bashrc         │  ← 被 ~/.bash_profile 通常 source 引入
└──────────────────┘

Non-Login Shell (打开新终端、执行 bash):
┌──────────────────┐
│ /etc/bash.bashrc  │  ← 全局 (Debian/Ubuntu)
│ 或 /etc/bashrc    │  ← 全局 (RHEL/CentOS)
├──────────────────┤
│ ~/.bashrc         │  ← 用户级
└──────────────────┘

退出时:
~/.bash_logout

Zsh 的顺序:
/etc/zshenv → ~/.zshenv → /etc/zprofile → ~/.zprofile
→ /etc/zshrc → ~/.zshrc → /etc/zlogin → ~/.zlogin
```

### 16. 环境变量的设置和查看？

**答：**

```bash
# 查看
env                     # 查看所有环境变量
echo $PATH              # 查看特定变量
printenv HOME           # 另一种查看方式

# 设置（当前会话）
export MY_VAR="value"   # 设置环境变量（子进程可继承）
MY_VAR="value"          # Shell 变量（子进程不继承）

# 永久设置
echo 'export MY_VAR="value"' >> ~/.bashrc        # 当前用户
echo 'export MY_VAR="value"' >> /etc/profile.d/custom.sh  # 所有用户

# 常见环境变量
PATH                   # 可执行文件搜索路径
HOME                   # 用户家目录
USER / LOGNAME         # 当前用户名
SHELL                  # 当前使用的 Shell
LANG / LC_ALL          # 语言区域设置
EDITOR                 # 默认编辑器
HISTSIZE               # 历史记录条数
PS1                    # 命令提示符格式

# PATH 操作
export PATH="$PATH:/opt/myapp/bin"   # 追加到 PATH 末尾
export PATH="/opt/myapp/bin:$PATH"   # 追加到 PATH 开头（优先）
```

## 六、救援与故障恢复

### 17. 系统无法启动时如何排查和修复？

**答：**

```
故障定位流程:

1. 是否进入 GRUB？
   ├── 否 → BIOS/UEFI 问题，检查启动顺序
   └── 是 → 继续

2. 内核是否加载？
   ├── 否 → GRUB 配置问题，在 GRUB 菜单按 e 编辑
   └── 是 → 继续

3. 是否进入 initramfs shell？
   ├── 是 → 根文件系统挂载失败
   │       - 检查 /etc/fstab UUID 是否正确
   │       - 运行 fsck 修复文件系统
   └── 否 → 继续

4. systemd 是否启动？
   ├── 否 → init 问题，用 init=/bin/bash 进入
   └── 是 → 继续

5. 能否进入命令行？
   ├── 否 → 服务启动问题，进入救援模式排查
   └── 是 → 特定服务问题，查 journalctl
```

**常用修复手段：**

```bash
# 1. 进入单用户/救援模式
#    在 GRUB 菜单按 e，在 linux 行末尾添加:
#    single 或 systemd.unit=rescue.target

# 2. 使用 Live CD/USB 修复
mount /dev/sda2 /mnt           # 挂载根分区
mount /dev/sda1 /mnt/boot      # 挂载 boot
for dir in dev proc sys run; do
    mount --bind /$dir /mnt/$dir
done
chroot /mnt                    # 进入系统环境

# 3. 常见修复
fsck /dev/sda2                 # 修复文件系统
grub-install /dev/sda          # 修复 GRUB
update-grub                    # 重建 GRUB 配置
mkinitramfs -o /boot/initrd.img-$(uname -r)  # 重建 initramfs

# 4. 密码重置
# 在 GRUB 中添加 init=/bin/bash
mount -o remount,rw /          # 重新挂载根为读写
passwd root                    # 重置密码
sync
exec /sbin/init                # 恢复正常启动
```

### 18. 文件系统损坏（fsck）如何处理？

**答：**

```bash
# 检查文件系统（必须在未挂载状态下执行）
umount /dev/sda2
fsck /dev/sda2                 # 自动检测文件系统类型
fsck.ext4 /dev/sda2            # 指定 ext4
fsck -y /dev/sda2              # 自动回答 yes

# XFS 文件系统
xfs_repair /dev/sda2
xfs_repair -L /dev/sda2        # 清除日志（最后手段）

# 如果根分区损坏
# 1. 开机在 GRUB 中传入: fsck.mode=force fsck.repair=yes
# 2. 或从 Live USB 启动后修复

# 预防措施
tune2fs -c 30 /dev/sda2        # 每30次挂载自动检查
tune2fs -i 7d /dev/sda2        # 每7天自动检查
dumpe2fs /dev/sda2 | grep -i check  # 查看检查计划
```
