## 一、Linux 日志体系

### 1. Linux 日志系统架构？

**答：**

```
Linux 日志体系:
┌─────────────────────────────────────────────────┐
│                    应用程序                       │
│    (Nginx, MySQL, App...)                        │
│    自己写日志到指定文件                             │
└────────────────┬────────────────────────────────┘
                 │ syslog API / stdout/stderr
┌────────────────▼────────────────────────────────┐
│              日志守护进程                          │
│  ┌──────────────┐  ┌──────────────────────────┐  │
│  │  rsyslog     │  │  systemd-journald        │  │
│  │  (传统)      │  │  (systemd 原生)           │  │
│  │  /var/log/*  │  │  二进制存储               │  │
│  └──────────────┘  └──────────────────────────┘  │
└─────────────────────────────────────────────────┘
                 │
┌────────────────▼────────────────────────────────┐
│              远程日志                             │
│  ELK Stack / Graylog / Loki / Splunk            │
└─────────────────────────────────────────────────┘

现代 Linux (systemd):
- journald 收集所有日志（内核、服务、stdout/stderr）
- rsyslog 从 journald 接收日志，写入传统文件
- 两者并存互补
```

### 2. /var/log 下的重要日志文件？

**答：**

## 二、journald 详解

### 3. journalctl 命令详解？

**答：**

```bash
# ===== 基本查看 =====
journalctl                          # 所有日志
journalctl -e                       # 跳到末尾
journalctl -f                       # 实时跟踪（类似 tail -f）
journalctl --no-pager               # 不分页
journalctl -o json-pretty           # JSON 格式

# ===== 按服务过滤 =====
journalctl -u nginx.service         # 指定服务
journalctl -u nginx -u php-fpm     # 多个服务
journalctl -u nginx --since today   # 今天的 nginx 日志

# ===== 按时间过滤 =====
journalctl --since "2024-01-01"
journalctl --since "2024-01-01 10:00:00"
journalctl --since "1 hour ago"
journalctl --since "30 min ago"
journalctl --since today
journalctl --since yesterday --until today
journalctl --since "2024-01-01" --until "2024-01-02"

# ===== 按优先级过滤 =====
journalctl -p err                   # 错误及以上
journalctl -p warning               # 警告及以上
journalctl -p crit                  # 严重及以上
# 优先级: emerg(0) alert(1) crit(2) err(3) warning(4) notice(5) info(6) debug(7)

# ===== 按进程/用户过滤 =====
journalctl _PID=1234               # 指定 PID
journalctl _UID=1000               # 指定 UID
journalctl _COMM=sshd              # 指定程序名

# ===== 内核日志 =====
journalctl -k                       # 等价于 dmesg
journalctl -k --since "5 min ago"

# ===== 启动相关 =====
journalctl -b                       # 当次启动的日志
journalctl -b -1                    # 上次启动的日志
journalctl --list-boots             # 列出所有启动记录

# ===== 输出格式 =====
journalctl -o short                 # 默认简短格式
journalctl -o short-precise         # 精确时间
journalctl -o verbose               # 详细（所有字段）
journalctl -o json                  # JSON
journalctl -o json-pretty           # JSON 美化
journalctl -o cat                   # 只有消息（无时间戳）

# ===== 磁盘管理 =====
journalctl --disk-usage             # 日志占用空间
journalctl --vacuum-size=500M       # 限制总大小
journalctl --vacuum-time=30d        # 清理 30 天前的日志
journalctl --vacuum-files=5         # 只保留 5 个归档
```

### 4. journald 配置？

**答：**

```ini
# /etc/systemd/journald.conf
[Journal]
Storage=persistent              # auto/volatile/persistent/none
# persistent: 存储到 /var/log/journal/
# volatile: 只在内存 /run/log/journal/
# auto: 如果 /var/log/journal/ 存在则 persistent

Compress=yes                    # 压缩存储
SystemMaxUse=500M               # 日志最大占用空间
SystemKeepFree=1G               # 至少保留1G可用空间
SystemMaxFileSize=50M           # 单个文件最大
SystemMaxFiles=100              # 最大文件数
MaxRetentionSec=1month          # 最大保留时间
RateLimitIntervalSec=30s        # 速率限制窗口
RateLimitBurst=10000            # 窗口内最大条数
ForwardToSyslog=yes             # 转发到 rsyslog
MaxLevelStore=debug             # 存储的最大级别
MaxLevelSyslog=info             # 转发到 syslog 的级别
```

```bash
# 应用配置
systemctl restart systemd-journald

# 验证持久化
mkdir -p /var/log/journal
systemd-tmpfiles --create --prefix /var/log/journal
systemctl restart systemd-journald

# 查看当前配置
systemd-analyze cat-config systemd/journald.conf
```

## 三、rsyslog 配置

### 5. rsyslog 的配置和使用？

**答：**

```bash
# rsyslog 配置文件
/etc/rsyslog.conf               # 主配置
/etc/rsyslog.d/*.conf           # 自定义配置

# syslog 设施 (facility)
# kern     内核
# user     用户进程
# mail     邮件
# daemon   守护进程
# auth     认证
# syslog   syslog 内部
# lpr      打印
# cron     定时任务
# local0-7 自定义

# 优先级 (severity)
# emerg(0) > alert(1) > crit(2) > err(3) > warning(4) > notice(5) > info(6) > debug(7)
```

```bash
# /etc/rsyslog.conf 规则格式
# facility.severity    action

# 示例规则
*.info;mail.none;authpriv.none;cron.none    /var/log/messages
authpriv.*                                   /var/log/secure
mail.*                                       -/var/log/maillog  # 减号表示异步写
cron.*                                       /var/log/cron
*.emerg                                      :omusrmsg:*  # 发给所有在线用户
local7.*                                     /var/log/boot.log

# 自定义日志（应用使用 local0）
local0.*                                     /var/log/myapp.log

# 远程日志
# 发送端
*.* @192.168.1.100:514          # UDP
*.* @@192.168.1.100:514         # TCP

# 接收端 (/etc/rsyslog.conf)
module(load="imudp")
input(type="imudp" port="514")
module(load="imtcp")
input(type="imtcp" port="514")

# 按来源主机分文件
$template RemoteLogs,"/var/log/remote/%HOSTNAME%/%PROGRAMNAME%.log"
*.* ?RemoteLogs
```

## 四、日志轮转 (logrotate)

### 6. logrotate 配置详解？

**答：**

```bash
# logrotate 配置
/etc/logrotate.conf                 # 全局配置
/etc/logrotate.d/                   # 各服务配置

# logrotate 由 cron 触发
# /etc/cron.daily/logrotate
```

```bash
# /etc/logrotate.d/nginx 示例
/var/log/nginx/*.log {
    daily                   # 每天轮转（weekly/monthly/yearly）
    missingok               # 文件不存在不报错
    rotate 30               # 保留 30 个归档
    compress                # gzip 压缩
    delaycompress           # 延迟一轮再压缩（便于分析最近日志）
    notifempty              # 空文件不轮转
    create 0640 nginx adm   # 创建新文件的权限
    sharedscripts           # 多个匹配文件只执行一次脚本
    
    postrotate              # 轮转后执行
        [ -f /var/run/nginx.pid ] && kill -USR1 $(cat /var/run/nginx.pid)
    endscript
}
```

```bash
# 自定义应用日志轮转
/var/log/myapp/*.log {
    daily
    rotate 14
    compress
    delaycompress
    missingok
    notifempty
    create 0644 myapp myapp
    dateext                 # 用日期作后缀 (myapp.log-20240101)
    dateformat -%Y%m%d      # 日期格式
    maxsize 100M            # 超过100M也轮转（不等日期）
    minsize 1M              # 小于1M不轮转
    
    postrotate
        systemctl reload myapp > /dev/null 2>&1 || true
    endscript
}

# 手动测试
logrotate -d /etc/logrotate.d/nginx    # dry-run（不实际执行）
logrotate -f /etc/logrotate.d/nginx    # 强制执行
logrotate -v /etc/logrotate.conf       # 详细输出

# logrotate 状态文件
cat /var/lib/logrotate/status
# 或
cat /var/lib/logrotate.status
```

### 7. 日志轮转常见问题排查？

**答：**

```bash
# 问题 1: 日志未轮转
# 检查 cron 是否运行
systemctl status crond

# 检查 logrotate 配置语法
logrotate -d /etc/logrotate.d/myapp

# 检查状态文件中的时间戳
cat /var/lib/logrotate/status | grep myapp

# 问题 2: 磁盘满了紧急处理
# 清空正在使用的日志文件（不删除文件描述符）
> /var/log/large.log          # 清空内容但保留文件
# 或
truncate -s 0 /var/log/large.log

# ⚠️ 不要 rm 正在被写入的文件，进程仍持有 fd，空间不会释放
# 如果已经 rm 了：
lsof | grep deleted           # 查看被删除但仍占用空间的文件
# 找到 PID 后重启对应进程

# 问题 3: copytruncate vs create
# copytruncate: 复制后截断原文件（不需要通知进程）
# create: 重命名旧文件，创建新文件（需要进程重新打开文件）
# 推荐: 优先用 create + postrotate kill -USR1
```

## 五、审计系统 (auditd)

### 8. Linux 审计系统 auditd 的配置？

**答：**

```bash
# auditd = Linux 审计守护进程
# 记录系统调用、文件访问、用户操作等

# 安装
yum install audit        # RHEL
apt install auditd       # Debian

systemctl enable --now auditd

# 配置文件
/etc/audit/auditd.conf              # 主配置
/etc/audit/audit.rules              # 规则文件
/etc/audit/rules.d/*.rules          # 规则目录
```

```bash
# /etc/audit/auditd.conf 关键参数
log_file = /var/log/audit/audit.log
log_format = ENRICHED              # RAW / ENRICHED
max_log_file = 50                  # MB
num_logs = 10                      # 保留数量
max_log_file_action = ROTATE       # IGNORE/SYSLOG/SUSPEND/ROTATE/KEEP_LOGS
space_left = 75                    # 剩余空间告警（MB）
space_left_action = SYSLOG         # 空间不足时动作
admin_space_left_action = SUSPEND  # 管理员空间不足时
disk_full_action = SUSPEND         # 磁盘满时
```

### 9. auditd 审计规则编写？

**答：**

```bash
# 规则类型:
# 1. 控制规则     设置审计系统行为
# 2. 文件系统规则  监控文件/目录
# 3. 系统调用规则  监控系统调用

# ===== 文件系统规则 =====
# -w <path> -p <permission> -k <key>
# permission: r=读 w=写 x=执行 a=属性变更

# 监控关键配置文件
auditctl -w /etc/passwd -p wa -k identity
auditctl -w /etc/shadow -p wa -k identity
auditctl -w /etc/sudoers -p wa -k sudoers_change
auditctl -w /etc/ssh/sshd_config -p wa -k sshd_config
auditctl -w /etc/hosts -p wa -k hosts_change

# 监控关键目录
auditctl -w /etc/cron.d/ -p wa -k cron_change
auditctl -w /var/log/audit/ -p wa -k audit_log

# ===== 系统调用规则 =====
# -a <list,action> -S <syscall> -F <field=value> -k <key>

# 监控用户/组变更
auditctl -a always,exit -F arch=b64 -S adjtimex -S settimeofday -k time_change
auditctl -a always,exit -F arch=b64 -S execve -k exec_commands

# 监控 root 命令执行
auditctl -a always,exit -F arch=b64 -S execve -F euid=0 -k root_commands

# 监控文件删除
auditctl -a always,exit -F arch=b64 -S unlink -S unlinkat -S rename -S renameat -k file_deletion

# 监控 mount/umount
auditctl -a always,exit -F arch=b64 -S mount -S umount2 -k mounts

# ===== 查看和管理规则 =====
auditctl -l                     # 列出当前规则
auditctl -D                     # 删除所有规则
auditctl -s                     # 审计系统状态
```

```bash
# 永久规则 /etc/audit/rules.d/audit.rules
-D                                                    # 清除已有规则
-b 8192                                               # 缓冲区大小
-f 1                                                  # 失败模式 (1=printk)
-w /etc/passwd -p wa -k identity
-w /etc/shadow -p wa -k identity
-w /etc/sudoers -p wa -k sudoers_change
-a always,exit -F arch=b64 -S execve -F euid=0 -k root_commands
-e 2                                                  # 锁定规则（不可更改）

# 加载规则
augenrules --load
# 或
service auditd restart
```

## 六、安全日志分析

### 11. 如何分析登录安全日志？

**答：**

```bash
# ===== 登录历史 =====
last                                 # 成功登录记录
last -n 20                           # 最近 20 条
last root                            # root 登录记录
last -i                              # 显示 IP
lastb                                # 失败登录记录（需 root）
lastlog                              # 每个用户最后登录

# ===== SSH 暴力破解检测 =====
# Debian/Ubuntu
grep "Failed password" /var/log/auth.log | tail -20
grep "Failed password" /var/log/auth.log | \
  awk '{print $(NF-3)}' | sort | uniq -c | sort -rn | head -10

# RHEL/CentOS
grep "Failed password" /var/log/secure | \
  awk '{print $(NF-3)}' | sort | uniq -c | sort -rn | head -10

# journalctl 方式
journalctl -u sshd --since "1 hour ago" | grep "Failed"

# ===== 成功登录统计 =====
grep "Accepted" /var/log/auth.log | \
  awk '{print $1, $2, $3, $9, $11}' | tail -20

# ===== sudo 使用记录 =====
grep "sudo:" /var/log/auth.log | tail -20
journalctl _COMM=sudo --since today

# ===== 可疑活动检测 =====
# 非工作时间登录
last | awk '$6 ~ /^(0[0-5])/ {print}'    # 凌晨登录

# 来自异常 IP 的登录
last -i | awk '{print $3}' | sort | uniq -c | sort -rn

# 用户切换记录
grep "su:" /var/log/auth.log
grep "session opened" /var/log/auth.log
```

### 12. fail2ban 防暴力破解？

**答：**

```bash
# 安装
yum install fail2ban              # RHEL
apt install fail2ban              # Debian

# 配置文件
/etc/fail2ban/jail.conf           # 默认配置（不要修改）
/etc/fail2ban/jail.local          # 自定义（覆盖 jail.conf）
/etc/fail2ban/jail.d/*.conf       # 自定义配置目录
```

```ini
# /etc/fail2ban/jail.local
[DEFAULT]
bantime = 3600                    # 封禁时间（秒）
findtime = 600                    # 检测时间窗口
maxretry = 5                      # 最大失败次数
banaction = iptables-multiport    # 封禁方式

[sshd]
enabled = true
port = ssh
logpath = /var/log/auth.log       # Debian
# logpath = /var/log/secure       # RHEL
maxretry = 3
bantime = 7200

[nginx-http-auth]
enabled = true
port = http,https
logpath = /var/log/nginx/error.log
maxretry = 5
```

```bash
# 管理命令
systemctl enable --now fail2ban

fail2ban-client status                    # 总状态
fail2ban-client status sshd               # 特定 jail 状态
fail2ban-client set sshd unbanip 1.2.3.4  # 解封 IP
fail2ban-client set sshd banip 1.2.3.4    # 手动封禁
fail2ban-client get sshd banned           # 查看已封禁

# 查看 fail2ban 日志
tail -f /var/log/fail2ban.log
```

## 七、集中化日志

### 13. ELK Stack 日志架构？

**答：**

```
ELK (Elastic Stack) 架构:

┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│ 应用服务器 │────→│ Filebeat  │────→│ Logstash │────→│Elastic-  │
│           │     │ (采集)    │     │ (处理)    │     │search    │
│ 日志文件   │     │          │     │ 解析/过滤 │     │ (存储)    │
└──────────┘     └──────────┘     └──────────┘     └────┬─────┘
                                                        │
                                                   ┌────▼─────┐
                                                   │  Kibana   │
                                                   │ (可视化)   │
                                                   └──────────┘

轻量架构（无 Logstash）:
App → Filebeat → Elasticsearch → Kibana

Filebeat 配置示例:
# /etc/filebeat/filebeat.yml
filebeat.inputs:
- type: log
  enabled: true
  paths:
    - /var/log/nginx/access.log
  fields:
    service: nginx
  multiline:
    pattern: '^\['
    negate: true
    match: after

output.elasticsearch:
  hosts: ["elasticsearch:9200"]
  index: "nginx-%{+yyyy.MM.dd}"

# 或输出到 Logstash
output.logstash:
  hosts: ["logstash:5044"]
```

### 14. 日志等级和结构化日志的最佳实践？

**答：**

```
日志等级规范:
┌──────────┬───────────────────────────────────────────┐
│ 等级      │ 使用场景                                  │
├──────────┼───────────────────────────────────────────┤
│ FATAL    │ 系统无法继续运行，需要立即处理               │
│ ERROR    │ 运行时错误，需要关注但系统仍可运行            │
│ WARN     │ 潜在问题，如重试、降级                      │
│ INFO     │ 重要业务流程信息（启动、配置、关键操作）      │
│ DEBUG    │ 调试信息，线上通常关闭                      │
│ TRACE    │ 最详细的跟踪信息                           │
└──────────┴───────────────────────────────────────────┘

结构化日志最佳实践:
1. 使用 JSON 格式
   {"timestamp":"2024-01-01T10:00:00Z","level":"ERROR","service":"api",
    "message":"DB connection failed","error":"timeout","host":"web-01"}

2. 包含关键字段
   - timestamp (统一时区 UTC)
   - level
   - service / component
   - trace_id / request_id（链路追踪）
   - message
   - error（错误详情）
   - host

3. 不要记录敏感信息
   - 密码、Token、密钥
   - 个人身份信息 (PII)

4. 合理设置日志量
   - 生产环境：INFO 级别
   - 调试时：临时开 DEBUG
   - 监控关键指标：独立 metrics

5. 统一日志格式
   - 团队内统一日志库和格式
   - 便于 ELK 等工具解析
```

### 15. 如何用命令行快速分析日志定位问题？

**答：**

```bash
# ===== 实时监控 =====
# 实时跟踪多个日志
tail -f /var/log/syslog /var/log/auth.log
multitail /var/log/syslog /var/log/auth.log  # 分屏显示

# 实时过滤
journalctl -f -p err                        # 实时查看错误
tail -f app.log | grep --line-buffered 'ERROR'

# ===== 时间段分析 =====
# 提取特定时间段（假设日志格式: 2024-01-01 10:00:00）
awk '$0 >= "2024-01-01 10:00" && $0 <= "2024-01-01 11:00"' app.log

# 按分钟统计错误数（发现错误高峰）
grep 'ERROR' app.log | awk '{print substr($0,1,16)}' | uniq -c

# ===== 错误分类 =====
# 统计各类错误出现次数
grep 'ERROR' app.log | sed 's/.*ERROR //' | sort | uniq -c | sort -rn | head -20

# 统计 Exception 类型
grep -oP '\w+Exception' app.log | sort | uniq -c | sort -rn

# ===== 关联分析 =====
# 通过 request_id 追踪请求
grep 'req-12345' app.log

# 查看错误前后上下文
grep -B5 -A10 'OutOfMemoryError' app.log

# ===== 性能分析 =====
# 慢查询（假设日志包含耗时）
grep -oP 'took \K\d+' app.log | sort -rn | head -10

# 请求量趋势
awk '{print substr($0,1,13)}' access.log | uniq -c  # 按小时
```
