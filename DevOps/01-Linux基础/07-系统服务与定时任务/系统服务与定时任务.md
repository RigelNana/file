## 一、systemd 核心概念

### 1. systemd 是什么？它的架构？

**答：**

```
systemd 是现代 Linux 的 init 系统和服务管理器（PID 1）

特点：
- 并行启动服务（比 SysV init 快）
- 按需加载（socket/bus/path 激活）
- 依赖关系管理
- 进程分组（cgroup 追踪）
- 日志管理（journald）
- 定时任务（timer）

架构:
┌─────────────────────────────────────────────┐
│                  systemd (PID 1)            │
├─────────┬──────────┬──────────┬─────────────┤
│systemd- │systemd-  │systemd-  │ systemd-    │
│journald │logind    │networkd  │ resolved    │
│(日志)    │(登录)    │(网络)     │ (DNS)       │
├─────────┴──────────┴──────────┴─────────────┤
│              Unit Files (单元文件)            │
│    .service  .socket  .timer  .mount  ...    │
└─────────────────────────────────────────────┘
```

### 2. systemd Unit 的类型有哪些？

**答：**

| Unit 类型 | 后缀 | 用途 |
|-----------|------|------|
| Service | `.service` | 服务进程（最常用） |
| Socket | `.socket` | IPC/网络 Socket（按需激活服务） |
| Timer | `.timer` | 定时器（替代 cron） |
| Mount | `.mount` | 文件系统挂载 |
| Automount | `.automount` | 自动挂载 |
| Target | `.target` | 一组 Unit 的逻辑分组（类似运行级别） |
| Path | `.path` | 文件路径监控（文件变化时触发） |
| Device | `.device` | 内核设备 |
| Swap | `.swap` | Swap 分区/文件 |
| Scope | `.scope` | 外部创建的进程组 |
| Slice | `.slice` | cgroup 层级 |
| Snapshot | `.snapshot` | 系统快照 |

### 3. Unit 文件存放位置和优先级？

**答：**

```bash
# 优先级从低到高：
/usr/lib/systemd/system/     # 软件包安装的（RPM/DEB）
/run/systemd/system/         # 运行时生成的（临时）
/etc/systemd/system/         # 系统管理员自定义（最高优先级）

# 用户级服务
~/.config/systemd/user/      # 用户自定义
/usr/lib/systemd/user/       # 软件包提供

# 查看 Unit 文件位置
systemctl show nginx.service -p FragmentPath
# 或
systemctl cat nginx.service
```

## 二、systemctl 命令详解

### 4. systemctl 常用命令大全？

**答：**

```bash
# ===== 服务控制 =====
systemctl start nginx           # 启动
systemctl stop nginx            # 停止
systemctl restart nginx         # 重启
systemctl reload nginx          # 重新加载配置（不中断）
systemctl try-restart nginx     # 仅在运行中时重启
systemctl reload-or-restart nginx  # 优先 reload，不支持则 restart

# ===== 开机自启 =====
systemctl enable nginx          # 开机启动
systemctl disable nginx         # 取消开机启动
systemctl enable --now nginx    # 启用并立即启动
systemctl is-enabled nginx      # 是否已启用

# ===== 查看状态 =====
systemctl status nginx          # 详细状态
systemctl is-active nginx       # 是否在运行
systemctl is-failed nginx       # 是否失败

# ===== 列表查看 =====
systemctl list-units                    # 已加载的 Unit
systemctl list-units --type=service     # 所有服务
systemctl list-units --state=failed     # 失败的 Unit
systemctl list-unit-files               # 所有 Unit 文件
systemctl list-unit-files --type=service --state=enabled  # 已启用服务
systemctl list-timers                   # 所有定时器
systemctl list-sockets                  # 所有 Socket

# ===== 依赖关系 =====
systemctl list-dependencies nginx       # 正向依赖
systemctl list-dependencies --reverse nginx  # 反向依赖

# ===== 系统管理 =====
systemctl daemon-reload         # 重新加载所有 Unit 文件
systemctl daemon-reexec         # 重新执行 systemd

# ===== 目标 (Target / 运行级别) =====
systemctl get-default                    # 当前默认目标
systemctl set-default multi-user.target  # 设置默认（命令行）
systemctl set-default graphical.target   # 设置默认（图形界面）
systemctl isolate rescue.target          # 切换到救援模式

# Target 和 SysV 运行级别对照
# runlevel 0 → poweroff.target
# runlevel 1 → rescue.target
# runlevel 2 → multi-user.target
# runlevel 3 → multi-user.target
# runlevel 5 → graphical.target
# runlevel 6 → reboot.target

# ===== 掩码（完全禁用） =====
systemctl mask nginx            # 完全禁用（无法手动启动）
systemctl unmask nginx          # 取消掩码
```

## 三、自定义 Service Unit 文件

### 6. 如何编写自定义 Service 文件？

**答：**

```ini
# /etc/systemd/system/myapp.service
[Unit]
Description=My Application Server
Documentation=https://docs.myapp.com
After=network.target postgresql.service     # 在这些之后启动
Requires=postgresql.service                 # 强依赖（依赖失败则本服务不启动）
Wants=redis.service                         # 弱依赖（依赖失败仍启动）

[Service]
Type=simple                                 # 进程类型（见下方说明）
User=myapp                                  # 运行用户
Group=myapp                                 # 运行组
WorkingDirectory=/opt/myapp                 # 工作目录
EnvironmentFile=/etc/myapp/env              # 环境变量文件
Environment="NODE_ENV=production"           # 直接定义环境变量

ExecStartPre=/opt/myapp/pre-start.sh        # 启动前执行
ExecStart=/opt/myapp/bin/server --port 8080  # 启动命令
ExecStartPost=/opt/myapp/post-start.sh      # 启动后执行
ExecReload=/bin/kill -HUP $MAINPID          # reload 命令
ExecStop=/bin/kill -TERM $MAINPID           # 停止命令

Restart=on-failure                          # 重启策略
RestartSec=5                                # 重启间隔
StartLimitBurst=3                           # 启动次数限制
StartLimitIntervalSec=60                    # 限制时间窗口

# 资源限制
LimitNOFILE=65535                           # 文件描述符限制
LimitNPROC=4096                             # 进程数限制
MemoryMax=512M                              # 最大内存（cgroup v2）
CPUQuota=200%                               # CPU 配额（2核）

# 安全加固
NoNewPrivileges=true                        # 禁止提权
ProtectSystem=strict                        # 只读文件系统
ProtectHome=true                            # 禁止访问 /home
PrivateTmp=true                             # 独立 /tmp
ReadWritePaths=/var/lib/myapp               # 允许写入的路径

[Install]
WantedBy=multi-user.target                  # 安装到哪个 target
```

### 7. Service Type 有哪些？

**答：**

| Type | 描述 | 适用场景 |
|------|------|---------|
| `simple` | 默认。ExecStart 的进程就是主进程 | 大多数守护进程 |
| `forking` | 进程 fork 后父进程退出，子进程成为主进程 | 传统守护进程（如 nginx 不加 -g "daemon off"） |
| `oneshot` | 进程执行完退出，等执行完才算启动成功 | 一次性任务（如初始化脚本） |
| `notify` | 进程启动后通过 sd_notify() 通知 systemd | 支持 sd_notify 的程序 |
| `dbus` | 进程在 D-Bus 上注册名字后算启动成功 | D-Bus 服务 |
| `idle` | 等所有其他任务完成后再启动 | 不紧急的后台任务 |

```bash
# Restart 策略
# no           不自动重启（默认）
# on-success   正常退出时重启（exit code 0）
# on-failure   异常退出时重启（非0退出、信号杀死、超时等）
# on-abnormal  被信号杀死或超时
# on-watchdog  watchdog 超时
# on-abort     被未捕获信号杀死
# always       总是重启

# 部署后的操作流程
systemctl daemon-reload          # 加载新文件
systemctl enable myapp           # 设置开机启动
systemctl start myapp            # 启动服务
systemctl status myapp           # 检查状态
journalctl -u myapp -f           # 查看日志
```

### 8. systemd 服务排障流程？

**答：**

```bash
# 1. 查看服务状态
systemctl status myapp.service

# 2. 查看详细日志
journalctl -u myapp.service -e --no-pager   # 最近日志
journalctl -u myapp.service --since "5 min ago"
journalctl -u myapp.service -f              # 实时日志

# 3. 验证 Unit 文件语法
systemd-analyze verify /etc/systemd/system/myapp.service

# 4. 查看启动耗时
systemd-analyze                              # 总启动时间
systemd-analyze blame                        # 各服务启动时间
systemd-analyze critical-chain myapp.service  # 关键路径

# 5. 查看属性
systemctl show myapp.service                 # 所有属性
systemctl show myapp.service -p MainPID,ExecStart

# 6. 常见问题排查
# 权限问题：确保 User/Group 存在，目录权限正确
# 路径问题：ExecStart 必须使用绝对路径
# 依赖问题：检查 After/Requires 的服务是否正常
# 环境变量：确认 EnvironmentFile 存在且格式正确
# SELinux：检查 SELinux 上下文
#    ausearch -m avc -ts recent
#    sealert -a /var/log/audit/audit.log
```

## 四、cron 定时任务

### 9. crontab 的格式和用法？

**答：**

```
crontab 格式:
┌───────────── 分 (0-59)
│ ┌───────────── 时 (0-23)
│ │ ┌───────────── 日 (1-31)
│ │ │ ┌───────────── 月 (1-12)
│ │ │ │ ┌───────────── 星期 (0-7, 0和7都是周日)
│ │ │ │ │
* * * * * command

特殊字符:
*        任意值
,        列表 (1,3,5)
-        范围 (1-5)
/        步长 (*/5 = 每5分钟)
```

```bash
# 常见时间表达式
* * * * *          # 每分钟
*/5 * * * *        # 每5分钟
0 * * * *          # 每小时整点
0 */2 * * *        # 每2小时
0 0 * * *          # 每天凌晨
0 0 * * 0          # 每周日凌晨
0 0 1 * *          # 每月1号凌晨
0 0 1 1 *          # 每年1月1号
30 3 * * 1-5       # 工作日3:30
0 9,18 * * *       # 每天9点和18点
*/10 8-18 * * 1-5  # 工作日8-18点每10分钟

# 预定义时间
@reboot            # 系统启动时
@yearly            # 每年 (= 0 0 1 1 *)
@monthly           # 每月 (= 0 0 1 * *)
@weekly            # 每周 (= 0 0 * * 0)
@daily             # 每天 (= 0 0 * * *)
@hourly            # 每小时 (= 0 * * * *)

# crontab 管理
crontab -l                          # 查看当前用户的 crontab
crontab -e                          # 编辑
crontab -r                          # 删除所有定时任务
crontab -l -u username              # 查看指定用户
cat /etc/crontab                    # 系统级 crontab
ls /etc/cron.d/                     # 系统级定时任务目录
ls /etc/cron.{daily,weekly,monthly}/  # 周期性脚本目录
```

### 10. cron 任务编写最佳实践？

**答：**

```bash
# 1. 使用绝对路径
# ✗ 错误
* * * * * python script.py
# ✓ 正确
* * * * * /usr/bin/python3 /opt/scripts/script.py

# 2. 重定向输出（避免邮件轰炸）
# 丢弃输出
* * * * * /opt/scripts/job.sh > /dev/null 2>&1
# 保留日志
* * * * * /opt/scripts/job.sh >> /var/log/job.log 2>&1
# 分离标准输出和错误
* * * * * /opt/scripts/job.sh >> /var/log/job.log 2>> /var/log/job.err

# 3. 设置环境变量
SHELL=/bin/bash
PATH=/usr/local/sbin:/usr/local/bin:/sbin:/bin:/usr/sbin:/usr/bin
MAILTO=admin@example.com

# 4. 加锁防止重复执行
* * * * * /usr/bin/flock -n /tmp/job.lock /opt/scripts/job.sh
# 或使用 flock 超时
* * * * * /usr/bin/flock -w 10 /tmp/job.lock /opt/scripts/job.sh

# 5. 添加超时控制
*/5 * * * * timeout 240 /opt/scripts/long_job.sh > /dev/null 2>&1

# 6. cron 调试
# 检查 cron 服务是否运行
systemctl status crond   # RHEL
systemctl status cron    # Debian

# 查看 cron 日志
grep CRON /var/log/syslog        # Debian
grep CRON /var/log/cron          # RHEL
journalctl -u cron --since "1 hour ago"

# 手动测试（模拟 cron 环境）
env -i SHELL=/bin/bash PATH=/usr/bin:/bin HOME=$HOME /opt/scripts/job.sh
```

## 五、systemd Timer（替代 cron）

### 11. systemd Timer 的优势和用法？

**答：**

systemd Timer 相比 cron 的优势：
- 可以绑定到任何 service unit
- 支持依赖关系
- 有日志（journalctl）
- 可以设置随机延迟（避免惊群）
- 支持单调时钟（系统启动后的时间）
- 错过的任务可以自动补执行

```ini
# /etc/systemd/system/backup.timer
[Unit]
Description=Daily Backup Timer

[Timer]
# ===== 日历时间 =====
OnCalendar=daily                    # 每天 00:00
OnCalendar=*-*-* 03:00:00           # 每天 03:00
OnCalendar=Mon *-*-* 09:00:00       # 每周一 09:00
OnCalendar=*-*-01 00:00:00          # 每月1号
OnCalendar=*:0/15                   # 每15分钟

# ===== 启动后 / 上次运行后 =====
OnBootSec=5min                      # 开机 5 分钟后
OnUnitActiveSec=1h                  # 上次运行后 1 小时
OnUnitInactiveSec=30min             # 上次结束后 30 分钟

# ===== 其他选项 =====
RandomizedDelaySec=30min            # 随机延迟 0-30 分钟
Persistent=true                      # 错过的执行在下次启动后补上
AccuracySec=1s                       # 精度（默认 1min）
Unit=backup.service                  # 关联的 service（默认同名）

[Install]
WantedBy=timers.target
```

```ini
# /etc/systemd/system/backup.service
[Unit]
Description=Daily Backup

[Service]
Type=oneshot
ExecStart=/opt/scripts/backup.sh
User=backup
```

```bash
# 管理 Timer
systemctl daemon-reload
systemctl enable --now backup.timer    # 启用并启动
systemctl list-timers --all            # 查看所有 timer
systemctl status backup.timer          # timer 状态
systemctl status backup.service        # 关联 service 状态
journalctl -u backup.service           # 查看执行日志

# systemd-analyze calendar 验证时间表达式
systemd-analyze calendar "Mon *-*-* 09:00:00"
systemd-analyze calendar "*:0/15"
```

### 12. cron 迁移到 systemd Timer 的对照？

**答：**

| cron 表达式 | systemd OnCalendar |
|------------|-------------------|
| `* * * * *` | `*:*:00` 或 `minutely` |
| `*/5 * * * *` | `*:0/5` |
| `0 * * * *` | `hourly` |
| `0 0 * * *` | `daily` |
| `0 0 * * 0` | `weekly` 或 `Sun *-*-* 00:00:00` |
| `0 0 1 * *` | `monthly` |
| `30 3 * * 1-5` | `Mon..Fri *-*-* 03:30:00` |
| `0 9,18 * * *` | `*-*-* 09,18:00:00` |

## 六、at 和 anacron

### 13. at 和 anacron 的用法？

**答：**

```bash
# ===== at - 一次性定时任务 =====
# 安装
yum install at       # RHEL
apt install at       # Debian

# 启动服务
systemctl enable --now atd

# 使用
at 15:30                    # 今天 15:30 执行
at now + 5 minutes          # 5 分钟后
at now + 2 hours            # 2 小时后
at 9:00 AM tomorrow         # 明天上午 9 点
at 3:00 PM Dec 25           # 12月25日下午3点

# 交互输入命令
$ at now + 1 hour
at> /opt/scripts/task.sh
at> <Ctrl+D>

# 或管道方式
echo "/opt/scripts/task.sh" | at now + 1 hour

# 管理
atq                         # 查看队列
atrm <job_id>               # 删除任务
batch                       # 系统空闲时执行

# 访问控制
/etc/at.allow               # 允许列表
/etc/at.deny                # 拒绝列表

# ===== anacron - 非持续运行系统的周期任务 =====
# anacron 确保任务在系统关机后能在下次开机时补执行
# 适用于笔记本等非 24×7 运行的系统

# /etc/anacrontab
# period  delay  job-id         command
1          5     daily-backup    /opt/scripts/daily.sh
7          10    weekly-report   /opt/scripts/weekly.sh
30         15    monthly-clean   /opt/scripts/monthly.sh
# period: 周期（天）
# delay:  延迟执行时间（分钟）

# 时间戳目录
ls /var/spool/anacron/       # 记录上次执行时间
```

## 七、服务编排与启动优化

### 14. 如何分析和优化 systemd 启动时间？

**答：**

```bash
# 整体启动时间
systemd-analyze

# 各服务启动时间排名
systemd-analyze blame

# 关键路径分析
systemd-analyze critical-chain

# 生成启动时序图（SVG）
systemd-analyze plot > boot.svg

# 查看指定服务的关键路径
systemd-analyze critical-chain nginx.service

# 优化方法：
# 1. 禁用不需要的服务
systemctl disable bluetooth.service
systemctl disable cups.service

# 2. 使用 socket 激活（按需启动）
# 服务只在有连接时才启动

# 3. 将非关键服务设为 Type=idle
# 延迟到其他任务完成后

# 4. 减少依赖链
# 避免不必要的 After/Requires

# 5. 并行化
# 减少串行依赖，利用 systemd 并行能力
```

### 15. 如何使用 systemd 管理用户级服务？

**答：**

```bash
# 用户级服务不需要 root 权限
# 存放位置: ~/.config/systemd/user/

# 创建用户服务
mkdir -p ~/.config/systemd/user/

cat > ~/.config/systemd/user/myapp.service << 'EOF'
[Unit]
Description=My User Application

[Service]
Type=simple
ExecStart=/home/user/bin/myapp
Restart=on-failure
Environment="HOME=%h"

[Install]
WantedBy=default.target
EOF

# 管理用户服务
systemctl --user daemon-reload
systemctl --user enable myapp.service
systemctl --user start myapp.service
systemctl --user status myapp.service
journalctl --user -u myapp.service

# 开机自启（用户不登录也能运行）
loginctl enable-linger username
# 这样即使用户没有登录，用户级服务也会在开机时启动

# 查看是否启用了 linger
loginctl show-user username -p Linger
ls /var/lib/systemd/linger/
```

### 16. 实战：部署一个 Go/Python 应用为 systemd 服务？

**答：**

```ini
# ===== Go 应用示例 =====
# /etc/systemd/system/goapp.service
[Unit]
Description=Go API Server
After=network.target

[Service]
Type=simple
User=goapp
Group=goapp
WorkingDirectory=/opt/goapp
ExecStart=/opt/goapp/server
Restart=on-failure
RestartSec=5
LimitNOFILE=65535
Environment="GIN_MODE=release"
Environment="PORT=8080"

# 安全加固
NoNewPrivileges=true
ProtectSystem=strict
ProtectHome=true
ReadWritePaths=/var/lib/goapp /var/log/goapp
PrivateTmp=true

# 优雅关闭
TimeoutStopSec=30
KillMode=mixed
KillSignal=SIGTERM

[Install]
WantedBy=multi-user.target
```

```ini
# ===== Python (Gunicorn) 应用示例 =====
# /etc/systemd/system/flask-app.service
[Unit]
Description=Flask App with Gunicorn
After=network.target

[Service]
Type=notify
User=www
Group=www
WorkingDirectory=/opt/flask-app
Environment="PATH=/opt/flask-app/venv/bin"
ExecStart=/opt/flask-app/venv/bin/gunicorn \
    --workers 4 \
    --bind unix:/run/flask-app/gunicorn.sock \
    --access-logfile /var/log/flask-app/access.log \
    --error-logfile /var/log/flask-app/error.log \
    wsgi:app
ExecReload=/bin/kill -s HUP $MAINPID
Restart=on-failure
RestartSec=5
RuntimeDirectory=flask-app

[Install]
WantedBy=multi-user.target
```

```bash
# 部署步骤通用流程
# 1. 创建运行用户
useradd -r -s /sbin/nologin goapp

# 2. 准备目录和权限
mkdir -p /opt/goapp /var/lib/goapp /var/log/goapp
chown -R goapp:goapp /opt/goapp /var/lib/goapp /var/log/goapp

# 3. 部署程序文件
cp server /opt/goapp/

# 4. 创建 service 文件
cp goapp.service /etc/systemd/system/

# 5. 加载并启动
systemctl daemon-reload
systemctl enable --now goapp
systemctl status goapp

# 6. 验证
curl http://localhost:8080/health
journalctl -u goapp -f
```
