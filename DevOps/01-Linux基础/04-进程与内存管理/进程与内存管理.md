## 一、进程基础

### 1. 进程和线程的区别？（深入版）

**答：**

| 特性 | 进程 (Process) | 线程 (Thread) |
|------|---------------|---------------|
| 定义 | 资源分配的基本单位 | CPU 调度的基本单位 |
| 内存空间 | 独立的虚拟地址空间 | 共享所属进程的地址空间 |
| 创建开销 | 大（复制页表、文件描述符等） | 小（共享大部分资源） |
| 上下文切换 | 慢（需要切换页表、TLB刷新） | 快（共享页表） |
| 通信方式 | IPC（管道、消息队列、信号、共享内存、Socket） | 直接读写共享数据（需要同步） |
| 崩溃影响 | 独立，一个进程挂不影响其他 | 一个线程崩溃，整个进程终止 |
| CPU 利用 | 多进程可利用多核 | 多线程共用进程资源，利用多核 |

```
进程内存布局:
┌──────────────────────┐ 高地址
│      内核空间          │ (3GB-4GB / 上半部分)
├──────────────────────┤
│       栈 (Stack)      │ ↓ 向下增长
│  局部变量、函数调用帧   │
├──────────────────────┤
│         ↕             │ ← 栈和堆之间的空闲区域
├──────────────────────┤
│       堆 (Heap)       │ ↑ 向上增长
│  malloc/new 动态分配   │
├──────────────────────┤
│    BSS 段 (未初始化)   │ 未初始化的全局/静态变量
├──────────────────────┤
│    数据段 (已初始化)   │ 已初始化的全局/静态变量
├──────────────────────┤
│    代码段 (Text)      │ 可执行指令（只读）
└──────────────────────┘ 低地址
```

### 2. 进程的生命周期和状态？

**答：**

```
fork()                   调度运行              I/O 完成
  │                        │                    │
  ▼                        ▼                    ▼
┌──────┐  被调度   ┌──────────┐  时间片用完  ┌──────────┐
│Created│ ───────→ │ Running  │ ──────────→ │  Ready   │
└──────┘          │  (R)     │ ←───────────│  (R)     │
                  └────┬─────┘  被调度      └──────────┘
                       │
            ┌──────────┼──────────┐
            │          │          │
       等待 I/O    收到信号    调用 exit()
            ▼          ▼          ▼
      ┌──────────┐ ┌────────┐ ┌──────────┐
      │ Sleeping │ │Stopped │ │ Zombie   │ ← 父进程未 wait()
      │ (S/D)    │ │ (T)    │ │ (Z)      │
      └──────────┘ └────────┘ └──────────┘
```

| 状态 | 符号 | ps 标记 | 说明 |
|------|------|---------|------|
| Running | R | R | 正在 CPU 执行或在就绪队列中 |
| Interruptible Sleep | S | S | 可中断睡眠，等待事件（如 I/O）,可被信号唤醒 |
| Uninterruptible Sleep | D | D | 不可中断睡眠，等待 I/O 完成，不响应信号（`kill -9` 也杀不掉） |
| Stopped | T | T | 收到 SIGSTOP/SIGTSTP（Ctrl+Z）暂停 |
| Zombie | Z | Z | 已退出但父进程未回收（wait），仍占 PID |
| Dead | X | X | 完全退出（极短暂状态） |
| Idle | I | I | 空闲内核线程（Linux 4.14+） |

```bash
# ps 状态附加标记
# s  会话领导者 (session leader)
# +  前台进程组
# l  多线程
# <  高优先级
# N  低优先级

# 示例
ps aux | head
# USER  PID %CPU %MEM  VSZ   RSS TTY  STAT START TIME COMMAND
# root    1  0.0  0.1 16784 10240 ?   Ss   Mar09 0:03 /sbin/init
# root    2  0.0  0.0     0     0 ?   S    Mar09 0:00 [kthreadd]
```

### 3. 进程间通信 (IPC) 方式有哪些？

**答：**

| IPC 方式 | 特点 | 适用场景 |
|---------|------|---------|
| **管道 (Pipe)** | 单向，父子进程间 | 命令管道 `ls | grep` |
| **命名管道 (FIFO)** | 有名字，无亲缘限制 | 独立进程间通信 |
| **信号 (Signal)** | 异步通知，携带信息少 | 进程控制（kill、Ctrl+C） |
| **消息队列** | 有格式的消息，内核管理 | 多进程有序通信 |
| **共享内存** | 最快的 IPC，需要同步 | 大数据量高性能场景 |
| **信号量** | 用于同步和互斥 | 控制共享资源访问 |
| **Socket** | 跨网络通信 | 网络服务、Docker API |
| **Unix 域套接字** | 本地 Socket，比 TCP 快 | 本地进程通信（MySQL, Docker） |

```bash
# 管道
ls -la | grep ".log" | wc -l

# 命名管道
mkfifo /tmp/mypipe
echo "hello" > /tmp/mypipe &    # 写端
cat /tmp/mypipe                  # 读端

# 查看 IPC 资源
ipcs                             # 列出消息队列、共享内存、信号量
ipcrm -m <shmid>                 # 删除共享内存段

# Unix 域套接字示例
ls -l /var/run/docker.sock
# srw-rw---- 1 root docker ... /var/run/docker.sock  ← s 类型
```

### 4. 常见信号详解？

**答：**

| 信号 | 编号 | 默认动作 | 可捕获 | 说明 |
|------|------|---------|--------|------|
| SIGHUP | 1 | 终止 | ✅ | 终端挂起 / 重载配置 |
| SIGINT | 2 | 终止 | ✅ | 键盘中断 (Ctrl+C) |
| SIGQUIT | 3 | core dump | ✅ | 键盘退出 (Ctrl+\) |
| SIGILL | 4 | core dump | ✅ | 非法指令 |
| SIGABRT | 6 | core dump | ✅ | abort() 调用 |
| SIGFPE | 8 | core dump | ✅ | 浮点异常（如除零） |
| SIGKILL | 9 | 终止 | ❌ | **强制杀死（不可捕获/忽略）** |
| SIGSEGV | 11 | core dump | ✅ | 段错误（非法内存访问） |
| SIGPIPE | 13 | 终止 | ✅ | 管道破裂（写入无读者的管道） |
| SIGALRM | 14 | 终止 | ✅ | alarm() 定时器到期 |
| SIGTERM | 15 | 终止 | ✅ | **优雅终止（默认信号）** |
| SIGCHLD | 17 | 忽略 | ✅ | 子进程状态变化 |
| SIGCONT | 18 | 继续 | ✅ | 继续被暂停的进程 |
| SIGSTOP | 19 | 暂停 | ❌ | **暂停进程（不可捕获/忽略）** |
| SIGTSTP | 20 | 暂停 | ✅ | 终端暂停 (Ctrl+Z) |
| SIGUSR1 | 10 | 终止 | ✅ | 用户自定义信号1 |
| SIGUSR2 | 12 | 终止 | ✅ | 用户自定义信号2 |

```bash
# 发送信号
kill PID                # 默认发送 SIGTERM (15)
kill -9 PID             # SIGKILL 强制杀死
kill -HUP PID           # SIGHUP 重载配置
kill -USR1 PID          # 用户信号（如 Nginx 日志切割）
kill -STOP PID          # 暂停进程
kill -CONT PID          # 恢复进程

# 按名称杀进程
killall nginx           # 杀死所有名为 nginx 的进程
pkill -f "python app"   # 按命令行模式匹配
pkill -u zhangsan       # 杀死用户的所有进程

# 优雅关闭 vs 强制关闭
kill PID                # 先尝试 SIGTERM（给进程清理机会）
sleep 5                 # 等待
kill -0 PID && kill -9 PID  # 还活着就强杀
```

### 5. ps 命令详解？

**答：**

```bash

ps aux
# a = 显示所有用户进程
# u = 用户格式输出
# x = 包含无终端的进程

# System V 风格（有 -）
ps -ef
# -e = 所有进程
# -f = 完整格式

# 自定义输出列
ps -eo pid,ppid,user,%cpu,%mem,vsz,rss,stat,start,time,comm --sort=-%mem | head
# pid    进程 ID
# ppid   父进程 ID
# user   所有者
# %cpu   CPU 使用率
# %mem   内存使用率
# vsz    虚拟内存大小 (KB)
# rss    常驻内存大小 (KB)
# stat   进程状态
# start  启动时间
# time   累计 CPU 时间
# comm   命令名
# args   完整命令行

# 按 CPU 排序
ps aux --sort=-%cpu | head

# 按内存排序
ps aux --sort=-%mem | head

# 查看进程树
ps axjf                # 以树形显示
pstree -p              # 更清晰的树
pstree -p <PID>        # 某进程的子树

# 查看线程
ps -eLf                # 显示线程
ps -T -p <PID>         # 某进程的所有线程

# 实用组合
ps -eo pid,ppid,cmd --forest    # 树形显示命令
ps -eo user --no-headers | sort | uniq -c | sort -rn  # 按用户统计进程数
```

### 7. 僵尸进程和孤儿进程？

**答：**

**僵尸进程 (Zombie)：**
- 子进程已终止，但父进程未调用 `wait()/waitpid()` 回收
- 进程表中保留一个条目（占用 PID），但不占 CPU/内存
- 大量僵尸进程会耗尽 PID 资源

**孤儿进程 (Orphan)：**
- 父进程已终止，子进程仍在运行
- 被 PID 1 (init/systemd) 收养，不是问题
- 后台守护进程就是利用这个原理

```bash
# 查找僵尸进程
ps aux | awk '$8 == "Z" {print}'
ps -eo pid,ppid,stat,cmd | grep "Z"

# 查看僵尸数量
ps aux | grep -c "Z"

# 处理僵尸进程
# 方法1: 杀死父进程（僵尸被 init 接管并回收）
ps -o ppid= -p <zombie_pid>    # 找到父进程
kill <parent_pid>               # 先尝试 SIGTERM
kill -9 <parent_pid>            # 强制

# 方法2: 给父进程发 SIGCHLD（通知它回收子进程）
kill -SIGCHLD <parent_pid>

# 批量清理
ps -eo pid,ppid,stat | awk '$3 == "Z" {print $2}' | sort -u | xargs kill -SIGCHLD
```

## 三、进程优先级

### 8. nice 和 renice 如何调整进程优先级？

**答：**

```
Linux 进程优先级:
  nice 值范围: -20 (最高优先级) 到 +19 (最低优先级)
  默认 nice: 0
  只有 root 可以设置负 nice 值（提高优先级）
  
  实际优先级 (PR) = 20 + nice
  PR 范围: 0 (最高) 到 39 (最低)
  
  实时优先级: 0-99 (比普通进程优先级更高)
```

```bash
# 以指定优先级启动进程
nice -n 10 ./backup.sh           # nice=10（低优先级）
nice -n -5 ./critical_task.sh    # nice=-5（高优先级，需 root）

# 修改运行中进程的优先级
renice 10 -p <PID>               # 调低优先级
renice -5 -p <PID>               # 调高优先级（需 root）
renice 15 -u zhangsan             # 调整用户所有进程

# 在 top 中按 r 可交互式修改
# 在 systemd 中设置
# [Service]
# Nice=10

# ionice - I/O 调度优先级
ionice -c 2 -n 7 ./backup.sh     # Best Effort, 低 I/O 优先级
# -c 1: Realtime   (实时，需 root)
# -c 2: Best Effort (默认)
# -c 3: Idle        (空闲时才执行 I/O)
```

## 五、内存管理

### 10. Linux 内存管理机制？

**答：**

```
物理内存分配:
┌────────────────────────────────────────────┐
│                  物理内存                    │
├────────────┬──────────┬────────────────────┤
│  内核空间   │  用户空间  │   Page Cache/Buffer │
│  (kernel)  │  (apps)   │   (文件系统缓存)     │
├────────────┴──────────┴────────────────────┤
│                    Swap                     │
└────────────────────────────────────────────┘
```

**关键概念：**

| 概念 | 说明 |
|------|------|
| **虚拟内存** | 每个进程有独立的虚拟地址空间（32位4GB, 64位128TB+） |
| **页表** | 虚拟地址到物理地址的映射 |
| **Page (页)** | 内存分配的最小单位，通常 4KB |
| **Page Cache** | 文件系统缓存，读过的文件数据缓存在内存 |
| **Buffer Cache** | 块设备缓冲区 |
| **Swap** | 磁盘上的虚拟内存，用于内存不足时交换 |
| **OOM Killer** | 内存耗尽时杀死进程的内核机制 |

### 11. free 命令输出如何解读？

**答：**

```bash
free -h
#               total    used    free    shared  buff/cache   available
# Mem:           16Gi    4.5Gi   1.2Gi    256Mi      10.3Gi      11.0Gi
# Swap:          4.0Gi    96Mi    3.9Gi

# 关键理解:
# total     = 物理内存总量
# used      = 已被进程使用的内存
# free      = 完全空闲的内存（没被任何人用）
# shared    = tmpfs 使用的内存
# buff/cache = 文件系统缓存（可以被回收给应用使用）
# available = 应用实际可用的内存 ≈ free + 可回收的 buff/cache

# ⚠️ 常见误区:
# free 很小不代表内存不足！
# 要看 available，这才是真正可用的内存
# Linux 会尽量用空闲内存做缓存（buff/cache），提高性能
# 应用需要时，缓存会自动释放
```

### 12. 内存相关排查命令？

**答：**

```bash
# 内存使用详情
cat /proc/meminfo
# MemTotal:       16384000 kB
# MemFree:         1200000 kB
# MemAvailable:   11200000 kB
# Buffers:          200000 kB
# Cached:         10000000 kB
# SwapTotal:       4096000 kB
# SwapFree:        3900000 kB
# Dirty:             12000 kB   ← 未写入磁盘的脏页
# Slab:             800000 kB   ← 内核 slab 缓存

# 按进程查看内存
ps aux --sort=-%mem | head -20           # 按内存排序
ps -eo pid,user,rss,vsz,comm --sort=-rss | head  # 按 RSS 排序

# RSS vs VSZ
# VSZ (Virtual Size):  进程使用的虚拟内存总量（包括未加载的共享库）
# RSS (Resident Set Size): 实际占用的物理内存（不含 Swap 中的部分）

# 查看单个进程内存详情
cat /proc/<PID>/status | grep -i "vm\|mem\|rss"
# VmPeak: 峰值虚拟内存
# VmRSS:  常驻内存
# VmSwap: 在 Swap 中的部分

pmap -x <PID>                # 进程内存映射
smem -rs rss | head          # 更精确的内存统计（含 PSS）

# 清理缓存（谨慎使用，通常不需要）
sync
echo 1 > /proc/sys/vm/drop_caches   # 清除 Page Cache
echo 2 > /proc/sys/vm/drop_caches   # 清除 Slab 缓存
echo 3 > /proc/sys/vm/drop_caches   # 清除所有
```

### 13. OOM Killer 是什么？如何处理？

**答：**

OOM (Out of Memory) Killer 是 Linux 内核在内存耗尽时启动的保护机制，它会选择并杀死进程以释放内存。

```bash
# 查看 OOM 日志
dmesg | grep -i "oom"
journalctl -k | grep -i "oom"
grep -i "oom" /var/log/messages

# OOM 评分机制
# 每个进程有一个 oom_score，分数越高越可能被杀
cat /proc/<PID>/oom_score          # 查看当前 OOM 分数
cat /proc/<PID>/oom_score_adj      # 查看 OOM 调整值

# 调整 OOM 优先级
# oom_score_adj 范围: -1000 到 1000
# -1000 = 绝不被 OOM 杀死
# 0     = 默认
# 1000  = 最优先被杀

# 保护关键进程（如数据库）
echo -1000 > /proc/<PID>/oom_score_adj

# 在 systemd 中设置
# [Service]
# OOMScoreAdjust=-900

# 预防 OOM
# 1. 合理设置 overcommit
sysctl vm.overcommit_memory
# 0 = 启发式（默认）
# 1 = 总是允许（危险）
# 2 = 不允许超过 swap + ratio% 物理内存

sysctl vm.overcommit_ratio    # 默认 50（当 overcommit_memory=2 时）

# 2. 设置 cgroup 内存限制
# 3. 合理配置 Swap
# 4. 监控内存使用，提前告警
```

## 六、资源限制

### 14. ulimit 和 /etc/security/limits.conf 详解？

**答：**

```bash
# 查看当前限制
ulimit -a                        # 全部限制
ulimit -n                        # 文件描述符限制（open files）
ulimit -u                        # 最大进程数（max user processes）
ulimit -s                        # 栈大小（stack size）

# 临时修改
ulimit -n 65535                  # 增大文件描述符限制

# 永久修改: /etc/security/limits.conf
# 格式: <domain>  <type>  <item>  <value>
# domain: 用户名、@组名、* (所有用户)
# type: soft (软限制，可调整到 hard) / hard (硬限制，root可改)

*               soft    nofile          65535
*               hard    nofile          65535
*               soft    nproc           4096
*               hard    nproc           65535
*               soft    core            unlimited    # core dump 大小
*               hard    core            unlimited
root            soft    nofile          65535
root            hard    nofile          65535
@devops         soft    nofile          131072

# 或使用 /etc/security/limits.d/ 目录下的文件（优先级更高）
# /etc/security/limits.d/99-custom.conf

# 系统级文件描述符限制
cat /proc/sys/fs/file-max              # 系统最大
sysctl -w fs.file-max=2097152          # 系统级修改

# 查看系统当前使用的文件描述符数
cat /proc/sys/fs/file-nr
# 已分配  未使用  最大值

# 查看进程的文件描述符

```

## 七、cgroup 资源控制

### 15. cgroup 是什么？v1 和 v2 的区别？

**答：**

cgroup (Control Groups) 是 Linux 内核提供的资源控制机制，用于限制、隔离、监控进程组的资源。Docker 和 Kubernetes 底层都依赖 cgroup。

| 特性 | cgroup v1 | cgroup v2 |
|------|----------|----------|
| 层级结构 | 每个控制器独立层级 | 统一层级 |
| 控制器绑定 | 一个控制器只能绑定到一个层级 | 统一管理 |
| 权限委派 | 复杂 | 支持非 root 委派 |
| Pressure Stall Information (PSI) | 不支持 | 支持 |
| 线程粒度控制 | 不支持 | 支持 |

```bash
# 查看当前使用的版本
mount | grep cgroup
stat -fc %T /sys/fs/cgroup/    # cgroup2fs = v2, tmpfs = v1

# cgroup v2 资源控制器
# cpu       CPU 使用控制
# memory    内存控制
# io        块设备 I/O 控制
# pids      进程数限制
# cpuset    CPU 和内存节点绑定

# 查看进程的 cgroup
cat /proc/<PID>/cgroup

# systemd 使用 cgroup 进行资源控制
systemctl set-property nginx.service CPUQuota=200%       # 限制2核
systemctl set-property nginx.service MemoryMax=2G        # 限制2GB内存
systemctl set-property nginx.service TasksMax=500        # 限制进程数

# 查看 cgroup 统计
systemd-cgtop                   # 类似 top 的 cgroup 监控
systemctl show nginx.service -p MemoryCurrent  # 查看当前内存使用
```

### 16. 如何使用 cgroup 限制资源？（实操）

**答：**

```bash
# ===== cgroup v2 手动操作 =====
# 创建 cgroup
mkdir /sys/fs/cgroup/mygroup

# CPU 限制（50% 的单核 = 50000/100000）
echo "50000 100000" > /sys/fs/cgroup/mygroup/cpu.max

# 内存限制（512MB）
echo $((512*1024*1024)) > /sys/fs/cgroup/mygroup/memory.max

# 进程数限制
echo 100 > /sys/fs/cgroup/mygroup/pids.max

# 将进程加入 cgroup
echo <PID> > /sys/fs/cgroup/mygroup/cgroup.procs

# 查看 cgroup 状态
cat /sys/fs/cgroup/mygroup/cpu.stat
cat /sys/fs/cgroup/mygroup/memory.current
cat /sys/fs/cgroup/mygroup/memory.stat
cat /sys/fs/cgroup/mygroup/pids.current

# ===== systemd service 中使用 =====
# [Service]
# CPUQuota=200%              # 200% = 最多2个核心
# MemoryMax=2G               # 内存上限
# MemoryHigh=1.5G            # 内存高水位（超过会被回收）
# TasksMax=500               # 最大进程/线程数
# IOWeight=100               # I/O 权重 (1-10000)
# IOReadBandwidthMax=/dev/sda 100M  # 读带宽限制
# CPUWeight=100              # CPU 权重 (1-10000)
```
