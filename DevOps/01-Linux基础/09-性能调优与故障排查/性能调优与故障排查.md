## 一、性能分析方法论

### 1. Linux 性能分析的方法和工具全景？

**答：**

```
USE 方法 (Utilization, Saturation, Errors):
对每种资源检查：
- 利用率 (Utilization): 资源忙碌程度
- 饱和度 (Saturation): 排队/等待程度
- 错误数 (Errors): 错误事件数量

┌──────────┬────────────────┬────────────────┬────────────────┐
│ 资源      │ 利用率(U)      │ 饱和度(S)       │ 错误(E)        │
├──────────┼────────────────┼────────────────┼────────────────┤
│ CPU      │ mpstat,top,%us │ vmstat(r列)     │ dmesg          │
│ 内存      │ free,vmstat    │ vmstat(si/so)   │ dmesg(oom)     │
│ 磁盘 I/O  │ iostat(%util)  │ iostat(avgqu)   │ smartctl       │
│ 网络      │ sar,iftop      │ netstat(recv-Q) │ ifconfig(err)  │
│ 文件描述符 │ lsof           │ ulimit          │ dmesg          │
└──────────┴────────────────┴────────────────┴────────────────┘
```

### 2. 性能排查的标准流程（60秒快速诊断）？

**答：**

```bash
# Netflix 60 秒性能诊断法

# 1. 系统负载概览
uptime
# 看 load average（1分钟, 5分钟, 15分钟）
# load > CPU核数 = 系统过载

# 2. 内核错误
dmesg | tail
dmesg -T | tail -20     # 带时间戳
# 看 OOM、硬件错误、文件系统错误

# 3. 系统整体
vmstat 1 5
# r: 运行队列（>CPU数=CPU饱和）
# b: 阻塞进程
# si/so: swap in/out（>0 = 内存不足）
# us/sy: 用户/系统CPU
# wa: IO等待
# st: 虚拟机偷取时间

# 4. CPU 使用
mpstat -P ALL 1 3
# 看各 CPU 是否均衡，是否有某个 CPU 100%

# 5. 进程级 CPU/IO
pidstat 1 3
# 看哪个进程消耗最多

# 6. 磁盘 IO
iostat -xz 1 3
# %util: 磁盘利用率（>80% 需要关注）
# await: 平均 IO 等待时间
# r/s, w/s: 读写 IOPS

# 7. 内存
free -h
# available 才是真正可用内存（不是 free）

# 8. 网络
sar -n DEV 1 3
# 看各网卡收发速率
sar -n TCP,ETCP 1 3
# 看 TCP 连接速率、重传率

# 9. 进程概览
top
# 按 1: 查看各CPU  M: 按内存排序  P: 按CPU排序

# 10. 追踪可疑进程
strace -p <PID> -c       # 系统调用统计
```

## 二、CPU 性能分析

### 3. CPU 性能分析详解？

**答：**

```bash
# ===== top / htop 深入 =====
# top 输出解读
top - 10:00:00 up 30 days, load average: 2.50, 2.30, 2.10
Tasks: 256 total,   3 running, 253 sleeping
%Cpu(s):  45.0 us,  10.0 sy,  0.0 ni, 40.0 id,  3.0 wa,  0.0 hi,  2.0 si,  0.0 st

# us: 用户空间（应用程序）       ← 高: 应用计算密集
# sy: 内核空间（系统调用）       ← 高: 内核/驱动问题或大量系统调用
# ni: nice 值调整的用户进程
# id: 空闲                      ← 低: CPU 忙
# wa: IO 等待                   ← 高: 磁盘/网络 IO 瓶颈
# hi: 硬件中断
# si: 软件中断                  ← 高: 网络包处理多
# st: 虚拟机偷取                ← 高: 宿主机 CPU 超卖

# top 交互快捷键
# 1    显示每个 CPU
# H    显示线程
# c    显示完整命令
# M    按内存排序
# P    按 CPU 排序
# k    kill 进程
# f    选择显示字段
# V    树状视图

# ===== mpstat - 多核 CPU 分析 =====
mpstat -P ALL 1
# 如果某个 CPU 100% 而其他空闲 → 单线程瓶颈
# 所有 CPU 均高 → 整体负载高

# ===== perf - CPU 性能分析利器 =====
# 安装
yum install perf       # RHEL
apt install linux-tools-common linux-tools-$(uname -r)

# CPU 采样分析
perf top                           # 实时查看热点函数
perf record -g -p <PID> sleep 30   # 录制 30 秒
perf report                        # 分析报告

# 生成火焰图
perf record -g -a sleep 30
perf script | stackcollapse-perf.pl | flamegraph.pl > cpu.svg

# ===== 上下文切换分析 =====
vmstat 1 | awk '{print $12, $13}'  # cs: 上下文切换
pidstat -w 1                        # 按进程查看上下文切换
# cswch/s:  自愿切换（IO等待）
# nvcswch/s: 非自愿切换（时间片用完）

# 高上下文切换原因:
# - 线程数远超 CPU 核数
# - 大量锁竞争
# - 频繁 IO
```

### 4. CPU 负载高如何排查和处理？

**答：**

```bash
# 排查流程
# Step 1: 确认 CPU 高的类型
top    # 看 us/sy/wa/si 哪个高

# 情况 A: us 高（用户空间 CPU 高）
# → 应用程序计算密集
top -Hp <PID>                    # 找到高 CPU 线程
printf '%x\n' <TID>             # 转16进制（Java jstack 用）
perf top -p <PID>               # 查看热点函数
strace -c -p <PID>              # 系统调用统计

# 情况 B: sy 高（内核 CPU 高）
# → 大量系统调用、锁竞争、中断
perf record -g -p <PID> sleep 30
perf report
# 查看是什么内核函数导致

# 情况 C: wa 高（IO 等待高）
# → 磁盘 IO 瓶颈（去看磁盘部分）
iostat -xz 1
iotop -oP

# 情况 D: si 高（软中断高）
# → 网络包处理多
cat /proc/softirqs
# 可能需要调整网卡中断亲和性 (IRQ affinity)
# 或启用 RPS/RFS

# 临时处理
# 限制进程 CPU
cpulimit -p <PID> -l 50         # 限制到 50% CPU
# 降低优先级
renice +10 -p <PID>
# 绑定 CPU
taskset -cp 0-3 <PID>           # 绑定到 CPU 0-3
```

## 三、内存性能分析

### 5. 内存问题如何排查？

**答：**

```bash
# ===== free 输出详解 =====
$ free -h
              total    used    free    shared  buff/cache  available
Mem:           16G     8.0G    1.0G    500M      7.0G       7.2G
Swap:          4.0G    500M    3.5G

# 关键: 看 available 而不是 free!
# available = free + 可回收的 buff/cache
# buff/cache 是内核文件缓存，可按需释放

# ===== 内存使用排行 =====
# 按 RSS（实际物理内存）排序
ps aux --sort=-rss | head -20

# 更精确（查看进程级内存）
smem -rs pss                     # PSS = 比例共享内存
pmap -x <PID>                    # 进程内存映射详情

# /proc/<PID>/status 关键字段
grep -E 'VmSize|VmRSS|VmSwap' /proc/<PID>/status
# VmSize: 虚拟内存（不重要）
# VmRSS:  物理内存（关键）
# VmSwap: 使用的 swap

# ===== 内存泄漏排查 =====
# 1. 持续监控进程内存
while true; do
    ps -p <PID> -o pid,rss,vsz,comm
    sleep 60
done >> /tmp/mem_monitor.log

# 2. 使用 valgrind（开发环境）
valgrind --leak-check=full ./program

# ===== Swap 分析 =====
# 哪些进程在使用 swap
for pid in $(ls /proc/[0-9]*/status 2>/dev/null | cut -d/ -f3); do
    swap=$(grep VmSwap /proc/$pid/status 2>/dev/null | awk '{print $2}')
    if [ "${swap:-0}" -gt 0 ]; then
        name=$(cat /proc/$pid/comm 2>/dev/null)
        echo "$swap kB  PID=$pid  $name"
    fi
done | sort -rn | head -10

# ===== 缓存相关 =====
# 查看缓存详情
cat /proc/meminfo | grep -E 'Cached|Buffers|Slab|SReclaimable'

# 手动清理缓存（紧急情况）
sync && echo 3 > /proc/sys/vm/drop_caches
# 1 = PageCache, 2 = dentries+inodes, 3 = 全部
# 生产环境慎用！正常情况让内核自动管理
```

## 四、磁盘 IO 性能分析

### 6. 磁盘 IO 性能如何分析？

**答：**

```bash
# ===== iostat - 磁盘 IO 统计 =====
iostat -xz 1 5

# 关键指标:
# r/s, w/s     读/写 IOPS
# rkB/s, wkB/s 读/写吞吐量
# await        平均 IO 等待时间 (ms) ← > 10ms 需要关注
# r_await      读等待
# w_await      写等待
# avgqu-sz     平均 IO 队列长度 ← > 1 说明有排队
# %util        磁盘利用率 ← > 80% 可能是瓶颈

# ===== iotop - 按进程查看 IO =====
iotop -oP                   # 只显示有IO的进程
iotop -aoP                  # 累计 IO

# ===== 找出 IO 最重的进程 =====
pidstat -d 1 5              # 按进程统计 IO
# kB_rd/s: 读速率
# kB_wr/s: 写速率

# ===== 查看当前正在做 IO 的进程 =====
cat /proc/<PID>/io
# read_bytes / write_bytes

# ===== 磁盘延迟分析 =====
ioping -c 10 /dev/sda        # 磁盘延迟测试（类似 ping）
dd if=/dev/zero of=/tmp/test bs=1M count=1024 oflag=direct   # 写速度
dd if=/tmp/test of=/dev/null bs=1M iflag=direct               # 读速度

# ===== 文件系统层面 =====
# 查看文件系统碎片
e4defrag -c /dev/sda1         # ext4 碎片率

# 查看 inode 使用
df -i                        # inode 耗尽也会导致无法创建文件

# 查看大文件/大目录
du -ah / --max-depth=2 | sort -rh | head -20
find / -type f -size +1G 2>/dev/null
```

### 7. 磁盘空间满了如何紧急处理？

**答：**

```bash
# Step 1: 确认磁盘使用
df -h
df -i                          # inode 是否满

# Step 2: 快速定位大文件/目录
du -sh /* 2>/dev/null | sort -rh | head -10
du -sh /var/* | sort -rh | head -10

# Step 3: 查找大文件
find / -type f -size +100M -exec ls -lh {} + 2>/dev/null | sort -k5 -rh

# Step 4: 清理
# 清理日志
> /var/log/large.log                # 清空（不删除，保留 fd）
journalctl --vacuum-size=100M       # 清理 journald

# 清理包缓存
yum clean all                       # RHEL
apt clean                           # Debian

# 清理旧内核
# RHEL
package-cleanup --oldkernels --count=2
# Ubuntu
apt autoremove --purge

# 清理 Docker
docker system prune -af             # 清理所有未使用的镜像/容器/网络
docker volume prune                 # 清理无用卷

# 清理 /tmp
find /tmp -type f -mtime +7 -delete

# Step 5: 查找已删除但未释放的文件
lsof +L1 | grep deleted
# 找到后重启对应进程即可释放

# Step 6: 如果 inode 满
# 查找小文件最多的目录
find / -xdev -type f | cut -d "/" -f 2-3 | sort | uniq -c | sort -rn | head
```

## 五、网络性能分析

### 8. 网络性能问题如何排查？

**答：**

```bash
# ===== 带宽检查 =====
iperf3 -s                  # 服务端
iperf3 -c <server-ip>      # 客户端测带宽
iperf3 -c <server-ip> -R   # 反向测试

# ===== 连接数分析 =====
ss -s                       # 连接摘要
ss -tn | awk '{print $4}' | cut -d: -f1 | sort | uniq -c | sort -rn   # 按目标 IP
ss -tn state time-wait | wc -l      # TIME_WAIT 数量
ss -tn state established | wc -l    # 已建立连接数

# ===== 丢包和重传 =====
# TCP 重传
netstat -s | grep -i retrans
ss -ti | grep -i retrans

# 网卡丢包
ip -s link show eth0
# RX errors / TX errors / dropped

ethtool -S eth0 | grep -i drop
ethtool -S eth0 | grep -i error

# ===== 延迟分析 =====
ping -c 100 <target>          # 平均延迟和丢包率
mtr -r -c 100 <target>       # 路由逐跳延迟

# ===== 连接超时排查 =====
# 1. DNS 解析慢？
time dig example.com

# 2. TCP 握手慢？
curl -o /dev/null -s -w "DNS: %{time_namelookup}s\nConnect: %{time_connect}s\nTLS: %{time_appconnect}s\nTotal: %{time_total}s\n" https://example.com

# 3. 防火墙问题？
iptables -L -n -v | grep DROP
conntrack -S                   # 连接跟踪表统计
conntrack -C                   # 连接跟踪表条目数
sysctl net.netfilter.nf_conntrack_max  # 最大值

# 4. 网卡队列溢出？
netstat -s | grep -i overflow
cat /proc/net/softnet_stat     # 软中断统计
```

## 六、综合故障排查

### 9. 系统卡顿/响应慢的排查流程？

**答：**

```bash
# 系统卡顿排查决策树
#
# 系统响应慢
# ├── uptime 看 load average
# │   ├── load 高
# │   │   ├── CPU 高？ → top → us高(应用) / sy高(内核) / wa高(IO)
# │   │   ├── IO 高？ → iostat → iotop 找进程
# │   │   └── 进程多？ → ps aux → 检查僵尸/sleep进程
# │   └── load 正常
# │       ├── 特定进程慢？ → strace -p PID
# │       ├── 网络慢？ → ss/ping/traceroute
# │       └── 内存不足？ → free → swap 使用情况
# └── 无法 SSH 连入
#     ├── ping 通？ → sshd 是否运行？端口是否开放？
#     ├── ping 不通？ → 网络/防火墙问题
#     └── 连接慢？ → DNS 反向解析/UseDNS问题

# === 实操步骤 ===
# 1. 快速概览
w                    # 负载 + 谁在登录
uptime               # 负载趋势

# 2. 分类判断
vmstat 1 5           # r列(CPU)  b列(IO阻塞)  si/so(swap)
# r 高 → CPU 瓶颈
# b 高 → IO 瓶颈
# si/so > 0 → 内存不足

# 3. 定位进程
top -bn1 | head -20   # 非交互模式
ps -eo pid,ppid,cmd,%mem,%cpu --sort=-%cpu | head

# 4. 深入分析
strace -c -p <PID>    # 系统调用统计
strace -T -p <PID>    # 带时间的系统调用跟踪
```

### 10. OOM Killer 触发后如何分析？

**答：**

```bash
# 查看 OOM 事件
dmesg | grep -i "out of memory"
dmesg | grep -i "oom"
journalctl -k | grep -i oom

# OOM 日志分析
# 典型 OOM 日志包含:
# - 触发 OOM 的进程
# - 各进程的内存使用（RSS、页表等）
# - 被杀掉的进程和其 oom_score
# - 系统总内存状态

grep -A 20 "Out of memory" /var/log/kern.log

# OOM Score 查看
cat /proc/<PID>/oom_score        # OOM 评分（越高越容易被杀）
cat /proc/<PID>/oom_score_adj    # 调整值 (-1000 到 1000)

# 保护重要进程
echo -1000 > /proc/<PID>/oom_score_adj   # 最高保护
echo -17 > /proc/<PID>/oom_adj           # 旧内核

# systemd 服务保护
# 在 Service 段添加:
# OOMScoreAdjust=-900

# 预防措施
# 1. 监控内存使用趋势
# 2. 合理配置 swap
# 3. 设置内存限制 (cgroup/ulimit)
# 4. 应用层内存泄漏检测
# 5. 配置 vm.overcommit
sysctl vm.overcommit_memory      # 0=启发式 1=总是允许 2=不超过比例
sysctl vm.overcommit_ratio       # 比例（默认50%）
```

### 11. 如何使用 strace 排查问题？

**答：**

```bash
# strace = 跟踪进程的系统调用

# 基本用法
strace command                    # 跟踪命令
strace -p <PID>                   # 附加到进程
strace -f -p <PID>                # 跟踪子进程/线程
strace -c -p <PID>                # 系统调用统计（最常用）

# 过滤
strace -e trace=open,read,write command    # 只看指定系统调用
strace -e trace=network command             # 只看网络
strace -e trace=file command                # 只看文件
strace -e trace=process command             # 只看进程
strace -e trace=signal command              # 只看信号

# 时间分析
strace -T command                 # 显示每个系统调用的耗时
strace -r command                 # 显示相对时间戳
strace -t command                 # 显示绝对时间

# 输出到文件
strace -o /tmp/trace.log -p <PID>

# 实用场景
# 1. 程序启动失败，找不到文件
strace -e trace=open,openat command 2>&1 | grep -i "no such file"

# 2. 程序卡住不动
strace -p <PID>
# 看最后一个系统调用是什么（通常是某种 read/poll/futex）

# 3. 程序 CPU 高
strace -c -p <PID>
# 看哪个系统调用被大量调用

# 4. 程序慢
strace -T -p <PID> 2>&1 | awk '$NF > 0.1'  # 耗时 > 0.1 秒的调用

# 5. 排查网络问题
strace -e trace=connect,sendto,recvfrom -p <PID>
```

## 七、性能调优

### 12. Linux 内核参数调优清单？

**答：**

```bash
# /etc/sysctl.d/99-performance.conf

# ===== 文件系统 =====
fs.file-max = 2097152                # 系统最大文件描述符
fs.nr_open = 2097152                 # 进程最大文件描述符
fs.inotify.max_user_watches = 524288 # inotify 监控数

# ===== 虚拟内存 =====
vm.swappiness = 10                   # 降低 swap 倾向 (默认60)
vm.dirty_ratio = 20                  # 脏页占内存比例触发同步写
vm.dirty_background_ratio = 5        # 脏页占比触发后台写
vm.vfs_cache_pressure = 50           # 降低 inode/dentry 缓存回收压力
vm.overcommit_memory = 0             # 启发式内存分配

# ===== 网络 =====
net.core.somaxconn = 65535           # listen 队列大小
net.core.netdev_max_backlog = 65535  # 网卡接收队列
net.ipv4.tcp_max_syn_backlog = 65535 # SYN 队列大小
net.ipv4.tcp_tw_reuse = 1           # TIME_WAIT 复用
net.ipv4.tcp_fin_timeout = 15        # FIN 超时
net.ipv4.tcp_keepalive_time = 600    # keepalive 间隔
net.ipv4.ip_local_port_range = 1024 65535  # 端口范围

# 缓冲区
net.core.rmem_max = 16777216
net.core.wmem_max = 16777216
net.ipv4.tcp_rmem = 4096 87380 16777216
net.ipv4.tcp_wmem = 4096 65536 16777216

# 安全
net.ipv4.tcp_syncookies = 1          # SYN 防洪

# ===== 不同场景预设 =====
# Web 服务器: 重点优化网络连接和文件描述符
# 数据库服务器: 重点优化 vm 参数和 IO 调度器
# 缓存服务器: 重点优化内存和网络
```

### 13. 文件描述符限制如何调整？

**答：**

```bash
# 查看当前限制
ulimit -n                        # 当前 shell（soft limit）
ulimit -Hn                      # hard limit
cat /proc/sys/fs/file-max       # 系统级限制
cat /proc/sys/fs/file-nr        # 已使用 / 0 / 最大

# 查看进程文件描述符
ls -la /proc/<PID>/fd | wc -l
cat /proc/<PID>/limits | grep "open files"

# 临时修改
ulimit -n 65535

# 永久修改
# /etc/security/limits.conf

root soft    nofile    65535
root hard    nofile    65535

# /etc/security/limits.d/99-nofile.conf
```

### 14. IO 调度器选择和调优？

**答：**

```bash
# 查看当前调度器
cat /sys/block/sda/queue/scheduler
# [mq-deadline] kyber bfq none

# 调度器比较
# mq-deadline: 默认，适合大多数场景，保证延迟
# kyber:       低延迟，适合 NVMe/SSD
# bfq:         公平排队，适合桌面/交互
# none/noop:   最简单，适合 NVMe（硬件自己调度）

# 临时修改
echo "mq-deadline" > /sys/block/sda/queue/scheduler

# 永久修改 (内核参数)
# /etc/default/grub
GRUB_CMDLINE_LINUX="elevator=mq-deadline"
# grub2-mkconfig -o /boot/grub2/grub.cfg

# 或 udev 规则
# /etc/udev/rules.d/60-scheduler.rules
ACTION=="add|change", KERNEL=="sd*", ATTR{queue/scheduler}="mq-deadline"
ACTION=="add|change", KERNEL=="nvme*", ATTR{queue/scheduler}="none"

# SSD/NVMe 优化
# 确认是否 SSD
cat /sys/block/sda/queue/rotational   # 0=SSD, 1=HDD
# 启用 TRIM
fstrim -av                            # 手动
systemctl enable fstrim.timer         # 自动（每周）
```
