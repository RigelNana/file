# 容器生命周期与管理

---

## 1. 容器的生命周期？

**回答：**

```
容器状态转换:

  docker create         docker start        docker pause
  ─────────→ Created ──────────→ Running ──────────→ Paused
                          ↑         │                   │
                          │         │ docker stop/kill   │ docker unpause
                          │         ↓                   │
                          │      Stopped ←──────────────┘
                          │         │
                          │         │ docker rm
  docker restart ─────────┘         ↓
                                Removed

状态说明:
  Created  → 容器已创建但未启动
  Running  → 容器正在运行
  Paused   → 容器进程被冻结（SIGSTOP）
  Stopped  → 容器进程已退出
  Removed  → 容器已删除

docker run = docker create + docker start
```

## 2. docker run 详解？

**回答：**

```bash
# 基本用法
docker run [OPTIONS] IMAGE [COMMAND] [ARG...]

# ===== 运行模式 =====
docker run -d nginx                  # 后台运行（detach）
docker run -it ubuntu bash           # 交互式终端
docker run --rm alpine echo hello    # 运行后自动删除
docker run -d --name web nginx       # 指定容器名

# ===== 端口映射 =====
-p 8080:80                # 宿主机8080 → 容器80
-p 127.0.0.1:8080:80      # 只绑定 localhost
-p 8080:80/tcp             # 指定协议
-p 8080-8090:80-90         # 端口范围
-P                         # 随机映射所有 EXPOSE 端口

# ===== 环境变量 =====
-e "NODE_ENV=production"              # 单个变量
-e "DB_HOST=db" -e "DB_PORT=5432"     # 多个变量
--env-file .env                       # 从文件读取

# ===== 挂载 =====
-v mydata:/data                 # 命名卷
-v /host/path:/container/path   # Bind Mount
-v /host/conf:/conf:ro          # 只读挂载
--tmpfs /tmp                    # 内存临时文件系统
--mount type=bind,source=/h,target=/c,readonly  # mount 语法

# ===== 资源限制 =====
--memory=512m           # 内存限制
--memory-swap=1g        # 内存+交换区限制
--cpus=1.5              # CPU 限制（1.5 核）
--cpu-shares=512        # CPU 权重（相对值）
--cpuset-cpus="0,1"     # 绑定 CPU 核心
--pids-limit=100        # 进程数限制
--ulimit nofile=65535   # ulimit 设置

# ===== 重启策略 =====
--restart=no              # 不自动重启（默认）
--restart=on-failure:5    # 异常退出时重启（最多5次）
--restart=always          # 总是重启
--restart=unless-stopped  # 总是重启，除非手动停止

# ===== 网络 =====
--network=mynet           # 使用自定义网络
--network=host            # 使用宿主机网络
--dns=8.8.8.8             # 自定义 DNS
--hostname=myhost         # 容器主机名
--add-host=db:10.0.0.5    # 添加 hosts 解析

# ===== 安全 =====
--user=1000:1000          # 指定运行用户
--read-only               # 只读文件系统
--cap-drop=ALL            # 去除所有 capabilities
--cap-add=NET_BIND_SERVICE # 添加特定 capability
--security-opt=no-new-privileges  # 禁止提权
```

## 3. 容器的进入与调试？

**回答：**

```bash
# ===== docker exec（推荐）=====
# 在运行中的容器内执行命令
docker exec -it myapp bash           # 交互式 shell
docker exec -it myapp sh             # 没有 bash 时
docker exec myapp cat /etc/hosts     # 执行单次命令
docker exec -u root myapp bash       # 以 root 用户进入
docker exec -w /app myapp ls         # 指定工作目录
docker exec -e "DEBUG=1" myapp env   # 带额外环境变量

# ===== docker attach =====
# 附加到容器的主进程（stdin/stdout/stderr）
docker attach myapp
# Ctrl+P Ctrl+Q → 分离（不停止容器）
# Ctrl+C         → 发送 SIGINT（可能停止容器）

# exec vs attach:
#   exec → 创建新进程, 不影响容器主进程
#   attach → 连接到容器的 PID 1

# ===== 调试命令 =====
# 查看日志
docker logs myapp
docker logs -f myapp                 # 实时跟踪
docker logs --tail 100 myapp         # 最后 100 行
docker logs --since 30m myapp        # 最近 30 分钟
docker logs -t myapp                 # 显示时间戳

# 查看进程
docker top myapp

# 实时资源监控
docker stats                         # 所有容器
docker stats myapp                   # 指定容器
docker stats --format "table {{.Name}}\t{{.CPUPerc}}\t{{.MemUsage}}"

# 查看容器详情
docker inspect myapp
docker inspect -f '{{.State.Status}}' myapp
docker inspect -f '{{.NetworkSettings.IPAddress}}' myapp
docker inspect -f '{{json .Mounts}}' myapp | jq

# 查看变更的文件
docker diff myapp
# A = 添加, C = 修改, D = 删除
```

## 4. 容器的停止与清理？

**回答：**

```bash
# ===== 停止容器 =====
docker stop myapp              # 优雅停止: SIGTERM → 等10s → SIGKILL
docker stop -t 30 myapp        # 等 30s 再 SIGKILL
docker kill myapp              # 立即 SIGKILL
docker kill -s SIGUSR1 myapp   # 发送指定信号

# 停止所有容器
docker stop $(docker ps -q)

# ===== 删除容器 =====
docker rm myapp                # 删除已停止的容器
docker rm -f myapp             # 强制删除运行中的容器（先 kill）
docker rm -v myapp             # 删除容器及其匿名卷

# 删除所有已停止的容器
docker container prune
docker rm $(docker ps -aq --filter status=exited)

# ===== 批量清理 =====
docker system prune            # 清理: 停止的容器 + 悬空镜像 + 未用网络
docker system prune -a         # 加上: 所有未使用的镜像
docker system prune -a --volumes  # 加上: 未使用的卷

# 查看磁盘使用
docker system df
docker system df -v            # 详细
```

### 优雅停止原理

```
docker stop 流程:
  1. 发送 SIGTERM 给容器 PID 1
  2. 等待 grace period (默认 10s)
  3. 如果进程还在，发送 SIGKILL 强制终止

关键: 应用必须能处理 SIGTERM 信号!

常见问题:
  Shell 形式的 CMD (CMD node app.js)
  → PID 1 是 /bin/sh, 不转发信号给 node
  → 总是等到超时后被 SIGKILL

解决:
  使用 Exec 形式: CMD ["node", "app.py"]
  或使用 tini 作为 init 进程: docker run --init
  或 ENTRYPOINT ["tini", "--", "node", "app.js"]
```

## 5. 容器的资源限制？

**回答：**

### 内存限制

```bash
docker run -d \
  --memory=512m \                # 内存上限
  --memory-swap=1g \             # 内存+swap 上限（-1=无限）
  --memory-reservation=256m \    # 软限制（低优先级回收）
  --oom-kill-disable \           # 禁用 OOM Kill（谨慎使用）
  myapp

# OOM (Out of Memory):
#   容器超过内存限制 → 内核 OOM Killer 杀掉容器进程
#   docker inspect -f '{{.State.OOMKilled}}' myapp

# 内存限制 vs 内存预留:
#   --memory    → 硬限制，超过就 OOM Kill
#   --memory-reservation → 软限制，内存紧张时会被回收到这个值
```

### CPU 限制

```bash
docker run -d \
  --cpus=1.5 \                   # 限制 1.5 个核心
  --cpu-shares=512 \             # 相对权重（默认1024）
  --cpuset-cpus="0,2" \          # 绑定到 CPU 0 和 2
  --cpu-period=100000 \          # CFS 调度周期
  --cpu-quota=150000 \           # CFS 调度配额
  myapp

# --cpus=1.5 等价于 --cpu-period=100000 --cpu-quota=150000
# 含义: 每 100ms 可以使用 150ms CPU 时间 = 1.5 核

# --cpu-shares 是相对权重:
#   容器A shares=1024, 容器B shares=512
#   竞争时 A 获得 2/3 CPU, B 获得 1/3
#   不竞争时各自可使用所有 CPU
```

### IO 限制

```bash
docker run -d \
  --device-read-bps=/dev/sda:10mb \    # 读速率限制
  --device-write-bps=/dev/sda:10mb \   # 写速率限制
  --device-read-iops=/dev/sda:100 \    # 读 IOPS 限制
  --device-write-iops=/dev/sda:100 \   # 写 IOPS 限制
  --blkio-weight=300 \                 # IO 权重 (10-1000)
  myapp
```

## 6. 容器的重启策略详解？

**回答：**

| 策略 | 容器退出时 | Docker 重启时 | 手动停止后 |
|------|----------|-------------|----------|
| no | 不重启 | 不启动 | - |
| on-failure[:N] | 非0退出码重启(最多N次) | 启动 | - |
| always | 总是重启 | 启动 | 也会重启 |
| unless-stopped | 总是重启 | 不启动(手动停的) | 不重启 |

```bash
# 设置重启策略
docker run -d --restart=unless-stopped --name web nginx

# 修改已有容器的重启策略
docker update --restart=always web

# 查看重启次数
docker inspect -f '{{.RestartCount}}' web

# on-failure 示例:
docker run -d --restart=on-failure:5 --name worker myworker
# 异常退出(非0)时重启, 最多重启5次
# 正常退出(exit 0)不重启
```

```
选择建议:
  Web 服务 / 数据库  → always 或 unless-stopped
  一次性任务         → no
  Worker             → on-failure:5
  开发环境           → no

always vs unless-stopped:
  场景: 你手动 docker stop 了容器
  always      → Docker 重启后，容器又会启动
  unless-stopped → Docker 重启后，容器保持停止
```

## 7. 容器文件操作？

**回答：**

```bash
# ===== docker cp — 宿主机和容器之间复制文件 =====
# 从容器复制到宿主机
docker cp myapp:/app/config.yml ./config.yml
docker cp myapp:/var/log/ ./logs/

# 从宿主机复制到容器
docker cp ./config.yml myapp:/app/config.yml
docker cp ./data/ myapp:/app/data/

# ===== docker diff — 查看容器文件系统变更 =====
docker diff myapp
# A /app/new-file.txt       A = 添加
# C /app/config.yml         C = 修改
# D /tmp/old-file.txt       D = 删除

# ===== docker commit — 从容器创建新镜像 =====
# （不推荐在生产中使用，应使用 Dockerfile）
docker commit myapp myapp:modified
docker commit -m "Added config" -a "Author" myapp myapp:v2

# ===== docker export/import =====
# 导出容器文件系统为 tar
docker export myapp > container-fs.tar

# 从 tar 创建镜像（扁平化，只有一层）
docker import container-fs.tar myapp:imported
```

## 8. 容器日志管理？

**回答：**

```bash
# 查看日志
docker logs myapp                    # 全部日志
docker logs -f myapp                 # 实时跟踪 (follow)
docker logs --tail 200 myapp         # 最后 200 行
docker logs --since "2024-01-01" myapp  # 指定时间之后
docker logs --until "2024-01-02" myapp  # 指定时间之前
docker logs --since 1h myapp         # 最近 1 小时
docker logs -t myapp                 # 显示时间戳
docker logs -f --tail 0 myapp        # 只看新日志
```

### 日志驱动

```json
// /etc/docker/daemon.json
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3",
    "compress": "true"
  }
}
```

| 日志驱动 | 说明 | docker logs |
|---------|------|-------------|
| json-file | 默认，JSON 文件 | ✅ 支持 |
| local | 优化的本地文件 | ✅ 支持 |
| journald | systemd journal | ✅ 支持 |
| syslog | syslog 协议 | ❌ 不支持 |
| fluentd | 发送到 Fluentd | ❌ 不支持 |
| awslogs | AWS CloudWatch | ❌ 不支持 |
| gcplogs | Google Cloud Logging | ❌ 不支持 |
| gelf | Graylog GELF | ❌ 不支持 |

```bash
# 容器级别指定日志驱动
docker run -d \
  --log-driver=json-file \
  --log-opt max-size=10m \
  --log-opt max-file=3 \
  myapp

# 日志文件位置
/var/lib/docker/containers/<container-id>/<container-id>-json.log

# 注意: 不配置 max-size 会导致磁盘被撑满!
```

## 9. docker update 动态修改容器配置？

**回答：**

```bash
# docker update 可以修改运行中容器的资源限制

# 修改内存限制
docker update --memory=1g --memory-swap=2g myapp

# 修改 CPU 限制
docker update --cpus=2 myapp
docker update --cpu-shares=1024 myapp
docker update --cpuset-cpus="0-3" myapp

# 修改重启策略
docker update --restart=always myapp

# 修改 pids 限制
docker update --pids-limit=200 myapp

# 批量修改
docker update --memory=512m --cpus=1 container1 container2

# 注意: 不是所有配置都能热更新
# 能更新: 资源限制、重启策略
# 不能更新: 端口映射、卷挂载、网络、环境变量
```

## 10. 容器 Init 进程与信号处理？

**回答：**

```
问题: 容器中的 PID 1 需要承担 init 的职责
  1. 回收僵尸子进程
  2. 正确处理和转发信号

常见问题:
  应用不处理 SIGTERM → docker stop 总是超时
  Shell 形式 CMD → /bin/sh 是 PID 1，不转发信号
  多进程应用 → 产生僵尸进程
```

### 解决方案

```dockerfile
# 方案 1: 使用 --init（推荐）
# docker run --init myapp
# Docker 注入 tini 作为 PID 1，负责信号转发和僵尸回收

# 方案 2: 使用 tini
FROM python:3.11-slim
RUN apt-get update && apt-get install -y tini && rm -rf /var/lib/apt/lists/*
ENTRYPOINT ["tini", "--"]
CMD ["python", "app.py"]

# 方案 3: 使用 Exec 形式（简单应用）
# ❌ Shell 形式: CMD python app.py
#    → PID 1 = /bin/sh -c "python app.py"
#    → /bin/sh 不转发 SIGTERM 给 python

# ✅ Exec 形式: CMD ["python", "app.py"]
#    → PID 1 = python app.py
#    → python 直接收到 SIGTERM

# 方案 4: 应用内处理信号
import signal
import sys

def handle_sigterm(sig, frame):
    print("Received SIGTERM, shutting down...")
    # 清理资源
    sys.exit(0)

signal.signal(signal.SIGTERM, handle_sigterm)
```

### dumb-init vs tini

```
两者都是轻量级 init 进程:

tini:
  - 更小 (~30KB)
  - Docker 官方推荐 (--init 内置)
  - 只做信号转发和僵尸回收

dumb-init (Yelp):
  - 稍大 (~50KB)
  - 支持信号重写
  - 进程组信号管理更灵活

推荐: 优先使用 docker run --init 或 tini
```
