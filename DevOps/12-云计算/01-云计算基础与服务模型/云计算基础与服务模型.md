## 1. 什么是云计算？核心特征？

**回答：**

```
NIST 定义的云计算五大核心特征:

  ┌────────────────────┬──────────────────────────────────────┐
  │ 特征                │ 说明                                │
  ├────────────────────┼──────────────────────────────────────┤
  │ 按需自助服务        │ 按需获取资源, 无需人工干预               │
  │ 广泛的网络访问      │ 通过标准机制 (HTTP/API) 跨设备访问       │
  │ 资源池化            │ 多租户共享物理资源, 动态分配              │
  │ 快速弹性伸缩        │ 按需扩缩, 对用户来说似乎无限             │
  │ 可计量的服务        │ 按使用量计费, 透明监控                   │
  └────────────────────┴──────────────────────────────────────┘

关键价值:
  ✓ 降低前期投入 (CapEx → OpEx)
  ✓ 弹性伸缩 (应对流量波动)
  ✓ 全球化部署 (多 Region)
  ✓ 托管运维 (减轻运维负担)
  ✓ 高可用 (多可用区冗余)
```

## 2. IaaS / PaaS / SaaS / FaaS 区别？

**回答：**

```
四种服务模型管理边界:

                用户管理 ←            → 云商管理
  On-Premise: [应用|数据|运行时|中间件|OS|虚拟化|服务器|存储|网络]
  IaaS:       [应用|数据|运行时|中间件|OS] | [虚拟化|服务器|存储|网络]
  PaaS:       [应用|数据]                  | [运行时|中间件|OS|…]
  FaaS:       [函数代码]                    | [应用框架|运行时|…]
  SaaS:                                     | [全部]

对比表:
  ┌─────────┬──────────────────┬────────────────────┬───────────────┐
  │ 模型     │ 用户管理          │ 典型服务            │ 典型场景       │
  ├─────────┼──────────────────┼────────────────────┼───────────────┤
  │ IaaS    │ OS + 应用 + 数据  │ EC2, Azure VM,     │ 自定义环境     │
  │         │                  │ GCE                │ 完全控制       │
  ├─────────┼──────────────────┼────────────────────┼───────────────┤
  │ PaaS    │ 应用 + 数据       │ Heroku, App Engine,│ 快速开发部署   │
  │         │                  │ Azure App Service  │ 不想管基础设施  │
  ├─────────┼──────────────────┼────────────────────┼───────────────┤
  │ FaaS    │ 函数代码          │ Lambda,            │ 事件驱动       │
  │         │                  │ Cloud Functions    │ 短时任务       │
  ├─────────┼──────────────────┼────────────────────┼───────────────┤
  │ SaaS    │ 无 (直接使用)     │ Gmail, Salesforce, │ 终端用户产品   │
  │         │                  │ Office 365         │               │
  └─────────┴──────────────────┴────────────────────┴───────────────┘

还有延伸模型:
  CaaS (Container as a Service): ECS, AKS, GKE — 容器编排托管
  DBaaS (Database as a Service): RDS, Cloud SQL — 数据库托管
  BaaS (Backend as a Service): Firebase, Supabase — 移动/Web 后端
```

## 3. 公有云 / 私有云 / 混合云 / 多云？

**回答：**

```
部署模型:
  ┌─────────┬────────────┬──────────────┬──────────────────┐
  │ 类型     │ 基础设施    │ 优势          │ 适用场景          │
  ├─────────┼────────────┼──────────────┼──────────────────┤
  │ 公有云   │ 云商拥有    │ 弹性,按需付费  │ 初创,互联网企业    │
  │ 私有云   │ 企业自有    │ 安全,合规可控  │ 金融,政府,医疗     │
  │ 混合云   │ 公+私混合   │ 灵活,兼顾安全  │ 大型企业          │
  │ 多云     │ 多家公有云  │ 避免锁定,容灾  │ 全球化企业         │
  └─────────┴────────────┴──────────────┴──────────────────┘

混合云典型架构:
  ┌────────────────────┐    ┌────────────────────┐
  │ 私有云 (数据中心)    │    │ 公有云 (AWS/Azure)  │
  │ ● 敏感数据          │←→  │ ● Web 前端          │
  │ ● 核心业务系统       │VPN │ ● 弹性计算           │
  │ ● 合规数据存储       │    │ ● AI/ML 训练        │
  └────────────────────┘    └────────────────────┘

  连接方式:
    VPN: 加密隧道, 成本低, 带宽有限
    专线 (Direct Connect/ExpressRoute): 高带宽, 低延迟, 成本高
    SD-WAN: 软件定义广域网, 灵活

多云策略:
  优势: 避免供应商锁定, 利用各家优势, 合规 (数据主权)
  挑战: 运维复杂度高, 需要统一管理平台, 网络互通
  工具: Terraform (多云 IaC), K8s (多云容器编排)
```

## 4. 云原生 (Cloud Native) 是什么？

**回答：**

```
CNCF 定义的云原生技术体系:

  云原生 = 微服务 + 容器化 + DevOps + CI/CD + 声明式 API

  ┌────────────────────────────────────────┐
  │          Cloud Native 技术栈            │
  ├────────────────────────────────────────┤
  │ 微服务架构   → 服务拆分, 独立部署        │
  │ 容器化       → Docker, 标准化打包        │
  │ 容器编排     → Kubernetes, 自动化管理     │
  │ 服务网格     → Istio, Linkerd, 流量治理   │
  │ CI/CD       → GitLab CI, ArgoCD, 持续交付│
  │ 声明式配置   → YAML, Helm, Kustomize     │
  │ 可观测性     → Prometheus, Grafana, Jaeger│
  │ 不可变基础设施 → 镜像部署, 不修改运行实例  │
  │ GitOps      → Git 作为唯一事实来源        │
  └────────────────────────────────────────┘

云原生 vs 传统应用:
  ┌──────────┬──────────────────┬──────────────────┐
  │ 维度      │ 传统应用          │ 云原生应用        │
  ├──────────┼──────────────────┼──────────────────┤
  │ 架构      │ 单体              │ 微服务            │
  │ 部署      │ 物理机/虚拟机      │ 容器/K8s         │
  │ 伸缩      │ 垂直扩展          │ 水平扩展          │
  │ 更新      │ 停机发布          │ 滚动/金丝雀       │
  │ 状态      │ 有状态            │ 无状态 (优先)     │
  │ 配置      │ 配置文件          │ 外部化/ConfigMap  │
  │ 恢复      │ 人工处理          │ 自愈 (K8s)       │
  │ DevOps   │ 开发运维分离       │ DevOps 一体化     │
  └──────────┴──────────────────┴──────────────────┘

12-Factor App (云原生应用 12 要素):
  1. Codebase: 一份代码, 多份部署
  2. Dependencies: 显式声明依赖
  3. Config: 配置存储在环境变量
  4. Backing services: 后端服务作为附加资源
  5. Build/Release/Run: 严格分离构建和运行
  6. Processes: 无状态进程
  7. Port binding: 通过端口绑定暴露服务
  8. Concurrency: 通过进程模型扩展
  9. Disposability: 快速启动和优雅终止
  10. Dev/prod parity: 开发/生产等价
  11. Logs: 日志作为事件流
  12. Admin processes: 管理任务作为一次性进程
```

## 5. 云计算中的 Region 和 Availability Zone？

**回答：**

```
Region (区域):
  地理上独立的数据中心集群
  每个 Region 之间完全隔离
  数据不会自动跨 Region 复制

  AWS:  30+ Regions (us-east-1, ap-southeast-1, cn-north-1...)
  Azure: 60+ Regions
  GCP:   35+ Regions

AZ (可用区, Availability Zone):
  Region 内的独立数据中心
  每个 AZ 有独立的电力、冷却、网络
  同 Region 内 AZ 之间低延迟 (< 2ms)

  us-east-1:
    ├── us-east-1a (AZ-a)
    ├── us-east-1b (AZ-b)
    ├── us-east-1c (AZ-c)
    ├── us-east-1d (AZ-d)
    ├── us-east-1e (AZ-e)
    └── us-east-1f (AZ-f)

Edge Location (边缘节点):
  CDN 缓存节点, 全球 400+
  CloudFront, Route 53 使用

Local Zone / Wavelength (延伸):
  超低延迟场景: 游戏, AR/VR, 5G

选择 Region 考虑因素:
  1. 合规要求: 数据主权 (中国 → cn-north-1)
  2. 延迟: 靠近用户
  3. 服务可用性: 新服务先在主要 Region 上线
  4. 成本: 不同 Region 价格不同 (us-east-1 通常最便宜)

高可用部署原则:
  最低: 跨 2 个 AZ 部署
  推荐: 跨 3 个 AZ 部署
  灾备: 跨 Region 部署 (Active-Active 或 Active-Passive)
```

## 6. 云服务的 SLA 和可用性？

**回答：**

```
SLA (Service Level Agreement) 服务级别协议:

  云商对服务可用性的承诺:
  ┌──────────────────┬──────────┬──────────────────┐
  │ 服务              │ SLA      │ 年停机时间        │
  ├──────────────────┼──────────┼──────────────────┤
  │ EC2 (单实例)      │ 99.5%   │ ~43.8 小时        │
  │ EC2 (多 AZ)      │ 99.99%  │ ~52.6 分钟        │
  │ S3              │ 99.99%   │ ~52.6 分钟        │
  │ S3 持久性        │ 99.999999999% │ 11 个 9        │
  │ RDS Multi-AZ    │ 99.95%   │ ~4.38 小时        │
  │ Lambda          │ 99.95%   │ ~4.38 小时        │
  │ EKS             │ 99.95%   │ ~4.38 小时        │
  └──────────────────┴──────────┴──────────────────┘

可用性计算:
  可用性 = 正常运行时间 / 总时间 × 100%

  "几个 9":
    99%     → 3.65 天/年
    99.9%   → 8.76 小时/年
    99.95%  → 4.38 小时/年
    99.99%  → 52.6 分钟/年
    99.999% → 5.26 分钟/年

  串联系统可用性 = A₁ × A₂ × A₃ × ...
    例: Web(99.99%) × DB(99.95%) = 99.94%

  并联 (冗余) 可用性 = 1 - (1-A)ⁿ
    例: 双 AZ 各 99.9% → 1-(0.001)² = 99.9999%

SLA 赔偿:
  未达到 SLA → 服务积分补偿 (Service Credits)
  通常 10%-100% 月费, 需主动申请
  不赔偿业务损失
```

## 7. 云计算安全责任共担模型？

**回答：**

```
AWS 共享责任模型 (Shared Responsibility Model):

  ┌──────────────────────────────────────────┐
  │           客户的责任                       │
  │      "Security IN the Cloud"             │
  │                                          │
  │  ● 客户数据加密                            │
  │  ● IAM 权限管理                           │
  │  ● 操作系统补丁 (IaaS)                    │
  │  ● 网络安全组配置                          │
  │  ● 应用层安全                              │
  │  ● 数据分类和保护                          │
  ├──────────────────────────────────────────┤
  │           AWS 的责任                       │
  │      "Security OF the Cloud"             │
  │                                          │
  │  ● 物理安全 (数据中心)                     │
  │  ● 硬件维护                               │
  │  ● 网络基础设施                            │
  │  ● 虚拟化层                               │
  │  ● 托管服务安全 (RDS, S3 基础设施)          │
  └──────────────────────────────────────────┘

不同服务模型的责任边界:
  IaaS (EC2):
    客户: OS 补丁, 防火墙规则, 应用安全, 数据加密
    AWS:  虚拟化层以下

  PaaS (RDS):
    客户: 数据加密, 访问控制, 网络安全组
    AWS:  OS 补丁, 数据库引擎更新

  SaaS (S3):
    客户: 桶策略, ACL, 数据加密选项
    AWS:  全部基础设施
```

## 8. 云厂商账号与组织管理？

**回答：**

```
AWS Organizations (多账号管理):

  Management Account (管理账号)
    ├── OU: Security
    │   └── Security Account (安全审计)
    ├── OU: Infrastructure
    │   ├── Shared Services Account (共享服务)
    │   └── Network Account (网络)
    ├── OU: Production
    │   ├── Prod App A Account
    │   └── Prod App B Account
    ├── OU: Development
    │   ├── Dev Account
    │   └── Staging Account
    └── OU: Sandbox
        └── Sandbox Account (实验)

SCP (Service Control Policy):
  在组织级别限制账号可使用的 AWS 服务
  作为权限护栏 (Guardrails)

  例: 禁止关闭 CloudTrail
  {
    "Effect": "Deny",
    "Action": ["cloudtrail:StopLogging"],
    "Resource": "*"
  }

AWS Control Tower:
  自动化多账号治理
  Landing Zone: 预配置的安全多账号环境
  Guardrails: 预设的安全和合规规则

跨账号访问:
  IAM Role + AssumeRole: 临时跨账号访问
  Resource-based Policy: S3 桶策略允许其他账号
  AWS RAM: 共享 VPC 子网、Transit Gateway 等资源
```

## 9. 云计算计费模型？

**回答：**

```
计费模式:
  ┌──────────────────┬──────────────────────────────────┐
  │ 模式              │ 说明                             │
  ├──────────────────┼──────────────────────────────────┤
  │ 按需 (On-Demand)  │ 按秒/小时计费, 最灵活, 最贵       │
  │ 预留 (Reserved)   │ 1-3 年承诺, 节省 30-72%          │
  │ Spot/抢占式       │ 竞价闲置资源, 节省 60-90%         │
  │ Savings Plans    │ 承诺使用金额, 灵活折扣            │
  │ 免费层 (Free Tier)│ 新用户 12 个月或永久免费额度      │
  └──────────────────┴──────────────────────────────────┘

各资源计费维度:
  计算: 实例类型 × 运行时间 + 数据传输
  存储: 容量 (GB/月) + 请求数 + 数据传输
  网络: 同 AZ 免费, 跨 AZ 收费, 出公网收费
  数据库: 实例运行时间 + 存储 + I/O + 备份

成本管理工具:
  AWS:   Cost Explorer, Budgets, Trusted Advisor
  Azure: Cost Management + Billing, Advisor
  GCP:   Billing Reports, Recommender

  第三方: Kubecost (K8s), Infracost (Terraform 预估)

隐藏成本陷阱:
  ✗ 跨 AZ 数据传输 ($0.01/GB)
  ✗ NAT Gateway 数据处理费
  ✗ EBS 快照存储费
  ✗ 闲置 EIP 费用
  ✗ CloudWatch 日志存储费
  ✗ API Gateway 请求费
```
