## 一、云计算基础概念

### 1. 什么是云计算？有哪些服务模型？

**答：**

云计算是通过互联网按需提供 IT 资源（计算、存储、网络等）的技术模式，按使用付费。

**三种服务模型：**

| 模型 | 全称 | 用户管理 | 云服务商管理 | 示例 |
|------|------|---------|------------|------|
| **IaaS** | Infrastructure as a Service | OS、中间件、应用、数据 | 虚拟化、服务器、存储、网络 | AWS EC2, Azure VM |
| **PaaS** | Platform as a Service | 应用、数据 | OS、运行时、中间件 | Heroku, Google App Engine |
| **SaaS** | Software as a Service | 无（直接使用） | 全部 | Gmail, Salesforce |

```
                  用户管理 ←                    → 云商管理
  On-Premise:  [应用|数据|运行时|OS|虚拟化|服务器|存储|网络]
  IaaS:        [应用|数据|运行时|OS] | [虚拟化|服务器|存储|网络]
  PaaS:        [应用|数据]           | [运行时|OS|虚拟化|···]
  SaaS:                              | [全部]
```

### 2. 公有云、私有云、混合云的区别？

**答：**

| 类型 | 所有权 | 优势 | 劣势 | 适用场景 |
|------|-------|------|------|---------|
| **公有云** | 云服务商 | 弹性伸缩、按需付费 | 安全合规受限 | 初创企业、弹性业务 |
| **私有云** | 企业自有 | 安全可控、合规 | 成本高、弹性差 | 金融、政府 |
| **混合云** | 混合 | 灵活、兼顾安全与弹性 | 架构复杂 | 大型企业 |

### 4. EC2 实例类型和选型？

**答：**

| 类型 | 前缀 | 特点 | 适用场景 |
|------|------|------|---------|
| 通用型 | t3, m5 | 均衡的计算/内存/网络 | Web 服务器、微服务 |
| 计算优化 | c5 | 高性能CPU | 批处理、科学计算 |
| 内存优化 | r5, x1 | 大内存 | 数据库、缓存 |
| 存储优化 | i3, d2 | 高IOPS、大存储 | 数据仓库、日志处理 |
| GPU | p3, g4 | GPU 加速 | 机器学习、图形渲染 |

**购买方式：**
- **按需 (On-Demand)**：按秒/小时计费，灵活但最贵
- **预留实例 (Reserved)**：1~3年预付，最高节省72%
- **Spot 实例**：竞价模式，最高节省90%，但可能被回收
- **Savings Plans**：承诺使用量，灵活折扣

## 三、VPC 网络

### 5. VPC 是什么？如何设计 VPC 网络？

**答：**

VPC (Virtual Private Cloud) 是云中的逻辑隔离网络环境。

```
VPC: 10.0.0.0/16
├── 可用区 A (AZ-a)
│   ├── 公有子网: 10.0.1.0/24  (Web 层, 有 IGW)
│   ├── 私有子网: 10.0.2.0/24  (App 层)
│   └── 数据子网: 10.0.3.0/24  (DB 层)
├── 可用区 B (AZ-b)
│   ├── 公有子网: 10.0.11.0/24
│   ├── 私有子网: 10.0.12.0/24
│   └── 数据子网: 10.0.13.0/24
├── Internet Gateway (IGW)     ← 公网出入口
├── NAT Gateway                ← 私有子网访问公网
└── Route Tables               ← 路由规则
```

**核心概念：**
- **子网 (Subnet)**：划分可用区的网段（公有/私有）
- **路由表 (Route Table)**：决定流量走向
- **Internet Gateway (IGW)**：VPC 连接公网的网关
- **NAT Gateway**：私有子网通过 NAT 访问外网
- **Security Group**：实例级有状态防火墙
- **NACL**：子网级无状态防火墙

### 6. Security Group 和 NACL 的区别？

**答：**

| 特性 | Security Group | NACL |
|------|---------------|------|
| 级别 | 实例级 | 子网级 |
| 状态 | 有状态（返回流量自动放行） | 无状态（需分别配置入站出站） |
| 规则 | 只允许 Allow 规则 | 支持 Allow 和 Deny |
| 评估 | 所有规则都评估 | 按编号顺序评估 |
| 默认 | 拒绝所有入站 | 允许所有 |

## 四、IAM 身份与权限管理

### 7. IAM 的核心概念？

**答：**

```
IAM 体系
├── User (用户)          → 个人身份
├── Group (组)           → 用户逻辑分组
├── Role (角色)          → 临时身份（EC2、Lambda 使用）
└── Policy (策略)        → JSON 权限文档
```

**IAM Policy 示例：**
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": "arn:aws:s3:::my-bucket/*"
    },
    {
      "Effect": "Deny",
      "Action": "s3:DeleteObject",
      "Resource": "*"
    }
  ]
}
```

**最佳实践：**
- 遵循最小权限原则
- 使用 IAM Role 替代长期密钥
- 为 Root 账户启用 MFA
- 使用 IAM Policy Simulator 验证权限
- 定期审计和轮换密钥

## 五、存储服务

### 8. S3 存储类别和使用场景？

**答：**

| 存储类别 | 可用性 | 适用场景 | 成本 |
|---------|--------|---------|------|
| S3 Standard | 99.99% | 频繁访问数据 | 高 |
| S3 Intelligent-Tiering | 99.9% | 访问模式不确定 | 自动分层 |
| S3 Standard-IA | 99.9% | 不频繁访问但需快速获取 | 中 |
| S3 Glacier | 99.99% | 归档（分钟~小时取回） | 低 |
| S3 Glacier Deep Archive | 99.99% | 长期归档（12小时取回） | 最低 |

**S3 关键特性：**
- 版本控制（误删恢复）
- 生命周期策略（自动迁移存储类别）
- 跨区域复制 (CRR)
- 服务器端加密（SSE-S3, SSE-KMS, SSE-C）
- 桶策略 + ACL 权限控制
- 事件通知 → Lambda/SQS/SNS

## 六、高可用架构

### 9. 如何设计高可用架构？

**答：**

```
                    Route 53 (DNS)
                        │
                    CloudFront (CDN)
                        │
                    ALB (负载均衡)
                   ╱           ╲
              AZ-a                AZ-b
        ┌──────────────┐   ┌──────────────┐
        │  Auto Scaling │   │  Auto Scaling │
        │  EC2 × N      │   │  EC2 × N      │
        └──────────────┘   └──────────────┘
              │                    │
        ┌──────────────┐   ┌──────────────┐
        │  ElastiCache  │   │  ElastiCache  │
        │  (Redis)      │   │  (Replica)    │
        └──────────────┘   └──────────────┘
              │                    │
        ┌──────────────────────────────┐
        │    RDS Multi-AZ (主从)        │
        │    Primary  ←→  Standby      │
        └──────────────────────────────┘
```

**关键设计原则：**
- **多可用区部署**（AZ 冗余）
- **Auto Scaling**：自动扩缩容
- **负载均衡**：分散流量
- **数据库多副本**：RDS Multi-AZ、Read Replica
- **缓存层**：ElastiCache 减少数据库压力
- **CDN**：CloudFront 加速静态资源
- **备份策略**：自动快照、跨区域备份
- **故障转移**：健康检查 + 自动切换

### 10. Auto Scaling 的工作原理？

**答：**

**组成要素：**
- **启动模板 (Launch Template)**：定义实例配置
- **Auto Scaling 组 (ASG)**：管理实例集群
- **扩缩策略**：定义何时扩缩容

**扩缩策略类型：**
| 策略 | 说明 | 示例 |
|------|------|------|
| 目标追踪 | 维持指标在目标值 | CPU 利用率保持 60% |
| 阶梯扩展 | 不同阈值采取不同动作 | CPU>70% +1台, >90% +3台 |
| 定时扩展 | 按计划执行 | 每天 9:00 扩容 |
| 预测性扩展 | 基于历史模式预测 | ML 预测流量模式 |

## 七、Serverless 与容器服务

### 11. Lambda 函数的特点和限制？

**答：**

**特点：**
- 无需管理服务器
- 按调用次数和执行时间付费
- 自动扩展（并发数）
- 支持多种运行时（Python、Node.js、Go、Java 等）

**触发方式：**
- API Gateway → Lambda (HTTP API)
- S3 事件 → Lambda (文件处理)
- SQS/SNS → Lambda (消息处理)
- CloudWatch Events → Lambda (定时任务)
- DynamoDB Streams → Lambda

**限制：**
| 限制项 | 数值 |
|--------|------|
| 执行超时 | 最大 15 分钟 |
| 内存 | 128 MB ~ 10 GB |
| 部署包大小 | 50 MB (直接) / 250 MB (S3) |
| 并发数 | 默认 1000/区域 |
| /tmp 临时空间 | 512 MB ~ 10 GB |

### 12. ECS 和 EKS 的区别？

**答：**

| 特性 | ECS | EKS |
|------|-----|-----|
| 编排引擎 | AWS 自研 | Kubernetes |
| 学习曲线 | 低 | 高 |
| 可移植性 | AWS 绑定 | 多云/混合云 |
| 生态系统 | AWS 生态 | K8s 生态 |
| 启动类型 | EC2 / Fargate | EC2 / Fargate |
| 定价 | 控制平面免费 | 控制平面 $0.10/h |

**选择建议：**
- 纯 AWS 环境、简单场景 → ECS
- 多云/混合云、K8s 生态需求 → EKS
- 不想管节点 → Fargate (两者都支持)
