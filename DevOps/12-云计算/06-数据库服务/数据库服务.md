## 1. RDS 关系型数据库？

**回答：**

```
RDS (Relational Database Service):
  托管关系型数据库, AWS 管理底层基础设施

  支持引擎:
    MySQL, PostgreSQL, MariaDB, Oracle, SQL Server

  AWS 管理:
    ✓ 硬件/OS 维护
    ✓ 自动备份 (PITR)
    ✓ 引擎补丁
    ✓ 监控 (CloudWatch)
    ✓ 多 AZ 故障转移

  用户管理:
    数据库创建/Schema/索引
    查询优化
    用户权限
    应用连接

Multi-AZ 部署:
  ┌────── AZ-a ───────┐    ┌────── AZ-b ───────┐
  │ Primary (读写)     │←→ │ Standby (同步复制)  │
  └──────────────────┘    └──────────────────┘
  
  同步复制, 自动故障转移
  Standby 不接受读请求
  故障转移时间: ~60 秒 (DNS 切换)

Read Replica (只读副本):
  ┌──────────┐    异步复制    ┌──────────┐
  │ Primary  │ ──────────→  │ Replica 1 │ ← 读请求
  │ (读+写)  │ ──────────→  │ Replica 2 │ ← 读请求
  └──────────┘              │ Replica 3 │ ← 读请求
                            └──────────┘
  
  异步复制, 有延迟
  可跨 Region (灾备 + 就近读)
  可提升为独立实例

RDS Proxy:
  连接池管理, 减少数据库连接数
  适用: Lambda + RDS (避免连接风暴)
  IAM 认证, 自动故障转移
```

## 2. Aurora 详解？

**回答：**

```
Aurora: AWS 自研云原生数据库
  兼容 MySQL 和 PostgreSQL
  性能: 5x MySQL, 3x PostgreSQL

架构:
  ┌─────────────────────────────────────────────┐
  │ Aurora Cluster                               │
  │                                              │
  │  Writer Instance    Reader Instance ×N       │
  │  (Primary)          (Auto Scaling)           │
  │       ↕                   ↕                  │
  │  ┌──────────────────────────────────────┐    │
  │  │ Aurora Storage (共享存储层)            │    │
  │  │ 6 副本跨 3 个 AZ                     │    │
  │  │ 10GB 自动增长, 最大 128 TB           │    │
  │  └──────────────────────────────────────┘    │
  └─────────────────────────────────────────────┘

Aurora vs RDS:
  ┌──────────────┬──────────────┬──────────────┐
  │ 特性          │ RDS          │ Aurora       │
  ├──────────────┼──────────────┼──────────────┤
  │ 性能          │ 标准         │ 3-5x        │
  │ 高可用        │ Multi-AZ    │ 6 副本/3 AZ  │
  │ 存储          │ EBS (手动)   │ 自动增长     │
  │ 故障转移      │ ~60 秒      │ ~30 秒       │
  │ 副本数        │ 最多 5 个    │ 最多 15 个   │
  │ 复制延迟      │ 秒级        │ 毫秒级       │
  │ 价格          │ 便宜        │ ~20% 贵      │
  │ Serverless   │ ✗           │ ✓            │
  └──────────────┴──────────────┴──────────────┘

Aurora Serverless v2:
  按需自动扩缩 (0.5 - 128 ACU)
  适合: 不可预测负载, 开发环境
  几秒内完成扩缩

Aurora Global Database:
  跨 Region 复制 (<1 秒延迟)
  灾备: RPO < 1秒, RTO < 1分钟
  最多 5 个 Secondary Region

Aurora Machine Learning:
  直接在 SQL 中调用 SageMaker/Comprehend
  SELECT sentiment FROM ml_predict(...)
```

## 3. DynamoDB NoSQL 数据库？

**回答：**

```
DynamoDB: 完全托管的 NoSQL 数据库
  键值 + 文档模型, 毫秒级延迟

核心概念:
  Table:        表
  Item:         行 (类似 JSON 文档)
  Attribute:    列/字段
  Partition Key: 哈希键 (必须, 决定分区)
  Sort Key:      范围键 (可选, 同分区内排序)

  主键类型:
    1. Partition Key only: 单键 (UserID)
    2. Partition Key + Sort Key: 复合键 (UserID + Timestamp)

容量模式:
  On-Demand:    按请求计费, 自动扩缩
  Provisioned:  预配置 RCU/WCU, 可配 Auto Scaling

  RCU (Read Capacity Unit):
    1 RCU = 1 强一致性读/秒 (4 KB)
    1 RCU = 2 最终一致性读/秒 (4 KB)
  
  WCU (Write Capacity Unit):
    1 WCU = 1 写/秒 (1 KB)

二级索引:
  GSI (Global Secondary Index):
    不同 Partition Key, 独立容量
    最多 20 个
  
  LSI (Local Secondary Index):
    相同 Partition Key, 不同 Sort Key
    创建表时定义, 最多 5 个

DynamoDB Streams:
  捕获表的变更 (INSERT/MODIFY/DELETE)
  触发 Lambda 处理
  用途: 事件驱动, 复制, 数据聚合

DAX (DynamoDB Accelerator):
  内存缓存, 微秒级读取延迟
  兼容 DynamoDB API
  适用: 读密集型场景
```

## 4. ElastiCache 缓存？

**回答：**

```
ElastiCache: 托管内存缓存

  两种引擎:
  ┌──────────┬──────────────────┬──────────────────┐
  │ 特性      │ Redis            │ Memcached        │
  ├──────────┼──────────────────┼──────────────────┤
  │ 数据结构  │ String/List/Set/ │ 简单 Key-Value   │
  │          │ Hash/Sorted Set  │                  │
  │ 持久化    │ ✓ (RDB/AOF)     │ ✗               │
  │ 复制      │ ✓ (读副本)      │ ✗               │
  │ 集群      │ ✓ (Redis Cluster)│ ✓ (多节点)      │
  │ 事务      │ ✓               │ ✗               │
  │ Pub/Sub  │ ✓               │ ✗               │
  │ Lua 脚本  │ ✓               │ ✗               │
  │ 多线程    │ 主 I/O 单线程   │ 多线程           │
  └──────────┴──────────────────┴──────────────────┘

  选择: 几乎总是选 Redis (功能更全)

Redis 部署模式:
  Standalone:  单节点 (开发)
  Cluster Mode Disabled: 1 主 + N 副本 (同 AZ/跨 AZ)
  Cluster Mode Enabled:  多分片, 每分片 1 主 + N 副本

缓存策略:
  Cache-Aside (懒加载):
    1. 读: 先查缓存, 未命中→查 DB→写缓存
    2. 写: 写 DB→删缓存 (Cache Invalidation)
    优势: 只缓存被请求的数据

  Write-Through:
    写: 同时写缓存和 DB
    优势: 缓存始终最新
    缺点: 写延迟增加, 可能缓存不常用数据

  TTL (Time To Live):
    设置过期时间, 平衡一致性和性能
    热 Key: 长 TTL + 异步更新
    普通 Key: 短 TTL

常见用途:
  Session 存储, 排行榜, 计数器, 限流
  实时分析, 消息队列 (简单场景)
```

## 5. 数据库迁移 DMS？

**回答：**

```
DMS (Database Migration Service):
  数据库迁移和持续复制

  迁移模式:
    Full Load:       全量迁移
    CDC:             持续复制 (Change Data Capture)
    Full Load + CDC: 全量 + 持续增量

  支持场景:
    同构: MySQL → MySQL (RDS)
    异构: Oracle → Aurora PostgreSQL (需 SCT)

  SCT (Schema Conversion Tool):
    异构迁移时转换 Schema
    Oracle PL/SQL → PostgreSQL PgPL/SQL
    SQL Server → Aurora MySQL
    评估报告: 自动/手动转换比例

迁移架构:
  ┌──────────┐     ┌──────────┐     ┌──────────┐
  │ Source DB │ ──→ │ DMS Task │ ──→ │ Target DB│
  │ (On-Prem) │     │ (复制实例)│     │ (RDS)    │
  └──────────┘     └──────────┘     └──────────┘
  
  支持: VPN/DX 连接本地, 跨 Region, 跨账号

常见迁移路径:
  Oracle → Aurora PostgreSQL (节省 License)
  SQL Server → Aurora MySQL
  MySQL → Aurora MySQL (性能提升)
  MongoDB → DynamoDB
  Cassandra → DynamoDB

迁移最佳实践:
  1. 评估: SCT 评估兼容性
  2. 测试: 先迁移测试环境
  3. 全量 + CDC: 减少停机时间
  4. 验证: 数据校验 (行数, 校验和)
  5. 切换: DNS 指向新数据库
```

## 6. 数据库安全？

**回答：**

```
数据库安全层次:

  网络安全:
    ✓ 部署在 Private Subnet
    ✓ Security Group 只允许应用层访问
    ✓ 不启用公网访问 (Publicly Accessible = No)
    ✓ 使用 VPC Endpoint (如需要)

  认证:
    ✓ IAM Database Authentication (MySQL/PostgreSQL)
      用 IAM Token 替代密码
    ✓ Secrets Manager 管理密码 (自动轮换)
    ✓ 强密码策略

  加密:
    静态加密:
      ✓ RDS/Aurora 启用加密 (KMS)
      ✓ 创建时必须启用, 后续不能更改
      ✓ 副本继承加密
    传输加密:
      ✓ 强制 SSL/TLS 连接
      ✓ rds.force_ssl = 1 (PostgreSQL)
      ✓ require_secure_transport = ON (MySQL)

  审计:
    ✓ 数据库审计日志 → CloudWatch Logs
    ✓ RDS 活动流 → Kinesis → 分析
    ✓ CloudTrail 记录 RDS API 调用

  备份安全:
    ✓ 快照加密
    ✓ 跨 Region 加密复制
    ✓ 快照共享限制 (KMS 密钥访问)
```

## 7. 数据库性能优化？

**回答：**

```
RDS/Aurora 性能优化:

  1. 实例选型
     CPU 密集: c5/c6g 系列
     内存密集: r5/r6g 系列 (推荐)
     通用: m5/m6g 系列

  2. 存储优化
     gp3: 通用 (可独立配置 IOPS)
     io1/io2: 高 IOPS (数据库推荐)
     Aurora: 自动管理, 无需选择

  3. 参数调优
     Performance Insights: 可视化性能分析
     → 识别等待事件, 慢查询, 资源瓶颈

  4. 读写分离
     Read Replica → 读请求分流
     应用层: 写→Primary, 读→Replica

  5. 连接管理
     RDS Proxy: 连接池 (Lambda 必用)
     避免连接频繁创建销毁

  6. 缓存
     ElastiCache Redis: 缓存热点查询
     减少数据库压力

  7. 索引优化
     Performance Insights 发现慢查询
     EXPLAIN 分析执行计划
     合理创建索引

监控指标:
  CPUUtilization:      > 80% 考虑升级
  FreeableMemory:      < 25% 考虑升级
  ReadIOPS/WriteIOPS:  评估存储性能
  DatabaseConnections: 监控连接数
  ReplicaLag:          副本延迟
```

## 8. 数据库选型指南？

**回答：**

```
选型决策:

  关系型数据 (SQL)?
    是 → 需要高性能?
         是 → Aurora (MySQL/PostgreSQL 兼容)
         否 → RDS (MySQL/PostgreSQL)
    否 ↓

  Key-Value / 文档 (NoSQL)?
    是 → 需要毫秒延迟 + 无限扩展?
         是 → DynamoDB
    否 ↓

  内存缓存?
    是 → ElastiCache Redis
    否 ↓

  图数据库?
    是 → Neptune
    否 ↓

  时序数据?
    是 → Timestream
    否 ↓

  搜索引擎?
    是 → OpenSearch (Elasticsearch)

服务对照:
  ┌──────────────────┬──────────────┬──────────────────┐
  │ 类型              │ AWS 服务      │ 典型场景          │
  ├──────────────────┼──────────────┼──────────────────┤
  │ 关系型            │ RDS / Aurora │ OLTP, 业务系统    │
  │ NoSQL 键值        │ DynamoDB     │ 用户画像, Session │
  │ NoSQL 文档        │ DocumentDB   │ 内容管理          │
  │ 内存缓存          │ ElastiCache  │ 缓存, 排行榜      │
  │ 图数据库          │ Neptune      │ 社交网络, 推荐    │
  │ 时序数据库        │ Timestream   │ IoT, 监控指标     │
  │ 搜索引擎          │ OpenSearch   │ 日志分析, 全文搜索│
  │ 数据仓库          │ Redshift     │ OLAP, 分析报表    │
  │ 宽列数据库        │ Keyspaces    │ Cassandra 兼容    │
  │ 账本数据库        │ QLDB         │ 不可变审计日志    │
  └──────────────────┴──────────────┴──────────────────┘
```

## 9. 数据库高可用和灾备？

**回答：**

```
高可用方案:

  RDS Multi-AZ:
    同步复制 → 自动故障转移 (~60s)
    Standby 不接受读请求

  Aurora:
    6 副本跨 3 AZ, 自动故障转移 (~30s)
    任意副本可提升为 Writer

  DynamoDB:
    自动跨 3 AZ, 无需配置
    Global Tables: 跨 Region 多主复制

灾备 (DR) 方案:
  ┌──────────────────┬────────┬────────┬──────────────┐
  │ 方案              │ RPO    │ RTO    │ 成本          │
  ├──────────────────┼────────┼────────┼──────────────┤
  │ 跨 Region 快照    │ 小时级 │ 小时级 │ 低            │
  │ RDS 跨 Region    │ 秒-分钟│ 分钟级 │ 中            │
  │  Read Replica    │        │        │              │
  │ Aurora Global DB │ <1 秒  │ <1 分钟│ 中高          │
  │ DynamoDB Global  │ 秒级   │ 秒级   │ 中            │
  │  Tables          │        │        │              │
  └──────────────────┴────────┴────────┴──────────────┘

Aurora 灾备最佳实践:
  1. Aurora Global Database (跨 Region)
  2. Primary Region 故障 → 提升 Secondary
  3. 应用通过 Route 53 故障转移 DNS
  4. RPO < 1s, RTO < 1min

RDS 灾备:
  1. 跨 Region Read Replica
  2. 定期跨 Region 快照复制
  3. 故障时提升 Replica 为独立实例
  4. 更新应用连接字符串或 DNS
```

## 10. 数据库面试题？

**回答：**

```
Q: RDS 和 Aurora 怎么选?
A: 标准需求 + 预算有限 → RDS
   高性能 + 高可用 + 自动扩展 → Aurora
   Aurora 贵约 20%, 但性能 3-5 倍, 存储自动管理

Q: DynamoDB 什么时候用?
A: 毫秒级延迟 + 无限水平扩展的 Key-Value 场景
   适合: 用户画像, Session, IoT, 游戏排行榜
   不适合: 复杂 JOIN, 事务密集的 OLTP

Q: 如何设计 DynamoDB 的 Partition Key?
A: 高基数 (Cardinality): 避免热分区
   均匀分布: 不要用时间戳做 PK (热点)
   好的 PK: UserID, OrderID
   坏的 PK: Date, Status (值少, 分布不均)

Q: Read Replica 有什么注意事项?
A: 异步复制→有延迟 (秒级)
   不能写入
   应用需要容忍最终一致性
   读写分离需要应用层实现

Q: Lambda + RDS 的问题?
A: Lambda 高并发 → 大量数据库连接 → 连接耗尽
   解决: RDS Proxy (连接池)
   或使用 DynamoDB (无连接限制)

Q: 数据库加密后忘了怎么办?
A: RDS 加密创建后不能更改
   需要: 创建加密快照 → 从快照恢复新实例
   或: 创建新加密实例 → DMS 迁移数据

Q: 如何实现零停机数据库迁移?
A: 1. DMS Full Load + CDC
   2. 源库持续同步到目标库
   3. 验证数据一致性
   4. 低峰期切换 DNS
   5. 保留源库一段时间 (回滚)
```
