# 高可用与容灾架构

---

## 1. 高可用架构设计原则？

**回答：**

```
高可用 (HA) 核心原则:

  1. 消除单点故障 (SPOF)
     每个关键组件至少有 2 个冗余实例
     跨 AZ 部署

  2. 多可用区 (Multi-AZ)
     至少 2 个 AZ, 推荐 3 个
     计算/存储/网络全部跨 AZ

  3. 自动故障检测和恢复
     健康检查 + 自动替换
     Auto Scaling + ELB

  4. 松耦合架构
     异步通信 (SQS, EventBridge)
     避免同步依赖导致级联故障

  5. 优雅降级
     核心功能可用, 非核心功能降级
     Circuit Breaker 模式

三层高可用架构:
  ┌─────────────────────────────────────────┐
  │              Route 53 (DNS 故障转移)     │
  │                    │                    │
  │            CloudFront (CDN)              │
  │                    │                    │
  │         ALB (跨 AZ 负载均衡)             │
  │          ╱                    ╲          │
  │   ┌── AZ-a ──┐          ┌── AZ-b ──┐   │
  │   │ ASG EC2  │          │ ASG EC2  │   │
  │   │ ×2-10    │          │ ×2-10    │   │
  │   └──────────┘          └──────────┘   │
  │          │                    │          │
  │   ┌────────────────────────────────┐    │
  │   │ ElastiCache Redis (Multi-AZ)  │    │
  │   └────────────────────────────────┘    │
  │          │                    │          │
  │   ┌────────────────────────────────┐    │
  │   │ Aurora Multi-AZ (Writer+Reader)│    │
  │   └────────────────────────────────┘    │
  └─────────────────────────────────────────┘
```

## 2. 灾备 (DR) 策略？

**回答：**

```
灾备四种策略 (成本从低到高):

1. Backup & Restore (备份恢复)
   ┌──────────┐              ┌──────────┐
   │ Primary  │ ──备份──→    │ Backup   │
   │ Region   │              │ (S3)     │
   └──────────┘              └──────────┘
   RPO: 小时级 | RTO: 小时级 | 成本: 最低
   方式: 自动快照, 跨 Region 复制

2. Pilot Light (最小灯)
   Primary Region               Secondary Region
   [完整基础设施]                [最小核心: DB 副本 running]
                                [其他资源停止/没有]
   RPO: 分钟级 | RTO: 10分钟-1小时 | 成本: 低
   切换: 启动 EC2, 扩展 ASG, DNS 切换

3. Warm Standby (温备)
   Primary Region               Secondary Region
   [完整基础设施]                [缩小版同样架构 running]
                                [最小规格, 少量实例]
   RPO: 秒-分钟 | RTO: 分钟级 | 成本: 中
   切换: Auto Scaling 扩展, DNS 切换

4. Active-Active (双活)
   Region-1 (Active)            Region-2 (Active)
   [完整基础设施]  ←双向同步→   [完整基础设施]
   Route 53 Latency/Weighted 路由
   RPO: 接近 0 | RTO: 接近 0 | 成本: 最高
   两个 Region 同时服务流量

选择:
  ┌──────────────────┬──────────┬──────────┬─────────┐
  │ 策略              │ RPO      │ RTO      │ 月成本   │
  ├──────────────────┼──────────┼──────────┼─────────┤
  │ Backup & Restore │ 1-24h    │ 1-24h    │ $       │
  │ Pilot Light      │ 分钟     │ 10min-1h │ $$      │
  │ Warm Standby     │ 秒-分钟  │ 分钟     │ $$$     │
  │ Active-Active    │ ~0       │ ~0       │ $$$$    │
  └──────────────────┴──────────┴──────────┴─────────┘
```

## 3. Auto Scaling 与弹性？

**回答：**

```
弹性设计模式:

  水平扩展 (Scale Out):
    增加实例数量 (推荐)
    无状态应用 + Auto Scaling Group
    ECS Service Auto Scaling
    K8s HPA

  垂直扩展 (Scale Up):
    增加实例规格 (有上限)
    需要停机 (EC2 修改实例类型)
    适用: 数据库等难以水平扩展的组件

Auto Scaling 策略组合:

  基线: Scheduled Scaling
    工作日 8:00 → min=5, desired=5
    非工作日 → min=2, desired=2

  弹性: Target Tracking
    CPU 维持 60%
    自定义指标: SQS 队列深度

  保护: 
    Cooldown: 扩缩操作间隔 (避免抖动)
    Instance Warm-up: 新实例初始化时间
    Scale-in Protection: 保护正在处理的实例

Auto Scaling 依赖:
  ✓ 无状态应用 (Session 存 Redis/DynamoDB)
  ✓ 快速启动 (AMI 预装 + 轻量 User Data)
  ✓ 健康检查 (ELB Health Check)
  ✓ 优雅终止 (Lifecycle Hook + 信号处理)

事件驱动扩缩:
  SQS 队列深度 → Auto Scaling
  CloudWatch 自定义指标 → Auto Scaling
  Application Auto Scaling: ECS, DynamoDB, Lambda
```

## 4. 负载均衡最佳实践？

**回答：**

```
ALB 最佳架构:

  Internet-facing ALB (公网):
    部署在 Public Subnet
    SSL 终止 (ACM 证书)
    WAF 集成
    Access Log → S3

  Internal ALB (内部):
    部署在 Private Subnet
    服务间通信
    API 网关 → 微服务

多层 LB 架构:
  ┌─────────────┐
  │ NLB (L4)    │ ← 固定 IP, 极高性能
  │  ↓          │
  │ ALB (L7)    │ ← Path/Host 路由
  │  ↓          │
  │ Target Group│ ← EC2/ECS/Lambda
  └─────────────┘

Cross-Zone Load Balancing:
  开启: 流量均匀分配到所有 AZ 的实例
  ALB: 默认开启
  NLB: 默认关闭 (每 AZ 独立)

Connection Draining (Deregistration Delay):
  实例从 Target Group 移除时
  等待现有连接完成 (默认 300 秒)
  设置合理时间: 短请求 30s, 长连接 3600s

Sticky Sessions:
  ALB: Application Cookie / Duration Cookie
  注意: 影响负载均衡效果
  替代: Session 存 Redis/DynamoDB

健康检查优化:
  路径: /health (专用端点, 检查依赖状态)
  间隔: 10-30s
  阈值: Healthy=2, Unhealthy=3
  超时: 5s (短于间隔)
```

## 5. 消息队列与解耦？

**回答：**

```
AWS 消息服务:

  SQS (Simple Queue Service):
    标准队列: 至少一次, 无序, 几乎无限吞吐
    FIFO 队列: 精确一次, 有序, 3000 TPS
    消息保留: 1 分钟 ~ 14 天 (默认 4 天)
    最大消息: 256 KB (大消息 → S3 + 引用)
    
    架构: 生产者 → SQS → Lambda/消费者
    解耦: 前端和后端独立扩缩

  SNS (Simple Notification Service):
    发布/订阅 (Pub/Sub)
    一对多: 1 消息 → N 订阅者
    订阅者: SQS, Lambda, Email, HTTP, SMS
    
    Fan-out 模式:
      SNS → SQS Queue 1 (处理 A)
          → SQS Queue 2 (处理 B)
          → Lambda (处理 C)

  EventBridge:
    事件总线 (Event Bus)
    规则匹配 → 路由到目标
    支持: AWS 服务事件, 自定义事件, SaaS 事件
    
    比 SNS 更灵活: 内容过滤, 转换, 归档

  Kinesis:
    实时流数据处理
    Data Streams: 自管理分片
    Data Firehose: 无服务器, 直接到 S3/Redshift/ES

解耦架构:
  同步 → 异步:
    API → Lambda → SQS → Worker
    削峰: 大量请求先入队, Worker 按能力处理
    重试: 死信队列 (DLQ) 处理失败消息

  事件驱动:
    Service A → EventBridge → Service B/C/D
    松耦合, 独立扩展
```

## 6. 全球化架构？

**回答：**

```
全球化部署方案:

1. 单 Region + CDN
   最简单, 适合大部分场景
   CloudFront CDN 加速全球访问
   动态内容: Regional Edge Cache

2. 多 Region Active-Passive
   Primary Region: 处理所有请求
   Secondary Region: 热备/温备
   Route 53 Failover: 故障自动切换

3. 多 Region Active-Active
   ┌────────────────┐    ┌────────────────┐
   │ Region 1 (US)  │    │ Region 2 (EU)  │
   │ Full Stack     │←→  │ Full Stack     │
   │ Aurora Writer  │    │ Aurora Reader  │
   └────────────────┘    └────────────────┘
         ↑                      ↑
   Route 53 Latency-based Routing
   
   用户就近访问最近 Region
   数据同步: Aurora Global DB / DynamoDB Global Tables

数据同步挑战:
  数据一致性: 跨 Region 有延迟
  冲突解决: DynamoDB Global Tables 用 Last Writer Wins
  读写分离: 写入特定 Region, 就近读
  事件同步: EventBridge Cross-Region

全球加速:
  Global Accelerator:
    Anycast IP → AWS 全球网络 → 就近 Region
    vs CloudFront: GA 用于动态内容, CF 用于静态缓存
    好处: 固定 IP, 低延迟, 快速故障转移
```

## 7. 监控和告警架构？

**回答：**

```
AWS 监控全景:

  ┌─────────────────────────────────────────┐
  │              CloudWatch                  │
  │  Metrics → Dashboards → Alarms          │
  │  Logs → Log Insights → Metric Filters   │
  │  Events → EventBridge → Lambda/SNS      │
  └─────────────────────────────────────────┘

关键监控指标:
  计算:
    CPUUtilization, MemoryUtilization*
    NetworkIn/Out
    StatusCheckFailed

  ALB:
    RequestCount, TargetResponseTime
    HTTPCode_Target_5XX_Count
    HealthyHostCount, UnhealthyHostCount

  RDS:
    CPUUtilization, FreeableMemory
    DatabaseConnections
    ReadIOPS, WriteIOPS, ReplicaLag

  Lambda:
    Invocations, Errors, Throttles
    Duration, ConcurrentExecutions

  SQS:
    ApproximateNumberOfMessagesVisible (队列深度)
    ApproximateAgeOfOldestMessage

  * EC2 内存需要 CloudWatch Agent

告警层级:
  Warning (预警): CPU>70%, 磁盘>75%, 队列积压
  Critical (严重): CPU>90%, 5XX>5%, DB 连接>80%
  Emergency (紧急): 服务不可用, 全部实例 Unhealthy

告警通知:
  CloudWatch Alarm → SNS Topic
    → Email (运维)
    → Lambda → Slack/钉钉/微信
    → PagerDuty (On-Call 轮转)
    → Lambda → 自动修复

Composite Alarm:
  组合多个 Alarm 的逻辑
  CPU > 80% AND Memory > 80% → 真正需要扩容
  避免: 单指标误报
```

## 8. 故障演练 (Chaos Engineering)？

**回答：**

```
AWS Fault Injection Simulator (FIS):
  AWS 托管的混沌工程服务

  支持注入的故障:
    EC2: 实例停止, CPU/Memory 压力, 网络中断
    ECS: 任务停止
    EKS: Pod 删除, 节点驱逐
    RDS: 故障转移
    网络: 子网流量中断

  实验流程:
    1. 定义实验模板 (Actions + Targets + Stop Conditions)
    2. 选择目标资源 (标签过滤)
    3. 设置停止条件 (CloudWatch Alarm)
    4. 执行实验
    5. 观察系统行为
    6. 分析和改进

  安全保障:
    IAM 角色限制操作范围
    停止条件: 指标超阈值自动停止
    只影响有特定标签的资源

混沌工程原则:
  1. 建立稳态假设 (正常指标基线)
  2. 在生产环境验证 (从非生产开始)
  3. 最小爆炸半径 (限制影响范围)
  4. 自动化实验 (可重复)

常见实验:
  ✓ 杀掉一个 AZ 的所有实例
  ✓ RDS 主库故障转移
  ✓ 网络延迟注入 (100ms)
  ✓ 依赖服务不可用
  ✓ DNS 故障
```

## 10. 高可用面试题？

**回答：**

```
Q: 如何设计一个 99.99% 可用的系统?
A: 1. 多 AZ 部署 (至少 3 个)
   2. 每层冗余: ALB + ASG + Aurora Multi-AZ
   3. Auto Scaling 自动恢复
   4. Route 53 健康检查
   5. 跨 Region 灾备 (Warm Standby)
   6. 消除单点: 无状态应用, Session 外部化
   7. 监控告警自动化

Q: RPO 和 RTO 的区别?
A: RPO (Recovery Point Objective): 可容忍的数据丢失量
   RTO (Recovery Time Objective): 可容忍的恢复时间
   RPO=0 → 同步复制 (Multi-AZ/Global DB)
   RTO=0 → Active-Active (双活)

Q: 灾备策略怎么选?
A: 看 RPO/RTO 要求和预算
   一般业务: Pilot Light (RPO 分钟, RTO 小时)
   重要业务: Warm Standby (RPO 秒, RTO 分钟)
   关键业务: Active-Active (接近零停机)

Q: 如何做到零停机部署?
A: 1. ALB + ASG Rolling Update
   2. Blue/Green: 两组 Target Group 切换
   3. CodeDeploy: 自动化 B/G 或 Canary
   4. K8s: Rolling + Readiness Probe

Q: 数据库故障转移会丢数据吗?
A: RDS Multi-AZ: 同步复制, 不丢 (RPO=0)
   Read Replica: 异步, 可能丢少量 (RPO 秒级)
   Aurora: 共享存储, 不丢
   DynamoDB: 全托管, 不丢

Q: 如何处理 AZ 故障?
A: 资源跨 AZ 部署 → 自动切换
   ALB 自动检测 unhealthy → 停止发送流量
   ASG 在健康 AZ 启动新实例
   RDS Multi-AZ 自动故障转移
```
