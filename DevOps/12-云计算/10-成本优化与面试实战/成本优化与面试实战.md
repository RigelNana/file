## 2. Right Sizing (合理选型)？

**回答：**

```
Right Sizing: 选择最合适的实例类型和规格

发现过度配置:
  AWS Compute Optimizer:
    分析 CPU/内存使用率 → 推荐实例类型
    基于 14 天 CloudWatch 数据

  CloudWatch 指标:
    CPU 常态 < 20% → 过度配置
    CPU 常态 > 80% → 配置不足
    内存 (需 CW Agent) < 30% → 过度配置

优化步骤:
  1. 收集 2-4 周指标
  2. 识别过度配置实例
  3. 评估推荐的实例类型
  4. 测试新类型 (staging 先试)
  5. 迁移 (修改实例类型或新建)

常见优化:
  m5.xlarge (4C/16G) → m5.large (2C/8G)
  → 节省 50%!

  x86 → Graviton (ARM):
  m5.large → m6g.large
  → 节省 ~20%, 性能相当或更好

  gp2 → gp3:
  → gp3 基础 3000 IOPS 免费
  → 大多数 gp2 卷可直接换 gp3 省 20%

自动化:
  定期审查 (每月/每季度)
  Auto Scaling: 按需调整实例数
  Serverless: Lambda/Fargate 按使用付费
```

## 3. 预留实例和 Savings Plans？

**回答：**

```
购买优化决策:

  稳定 24/7 运行的基线负载?
    是 → 占比多少?
         70%+ → Savings Plans (灵活)
              → 或 RI (折扣更大但不灵活)
    否 → On-Demand + Auto Scaling

Savings Plans:
  Compute SP: 最灵活
    承诺 $/hour, 可用于 EC2/Fargate/Lambda
    可换实例类型/Region/OS
    折扣: ~66%

  EC2 Instance SP:
    限定实例家族 + Region
    折扣: ~72% (比 Compute SP 高)
    例: m5 家族, us-east-1

  Savings Plans 优先于 RI (更灵活)

购买建议:
  1. Cost Explorer → Savings Plans 推荐
  2. 分析 12 个月用量趋势
  3. 只覆盖稳定基线 (不要超买)
  4. 选 3 年 No Upfront 或 1 年试水
  5. 每月审查利用率

Spot 策略:
  适合: CI/CD, 数据处理, 无状态 Web, 测试环境
  
  Spot Fleet:
    多实例类型 + 多 AZ → 降低中断风险
    容量优化策略: 选择中断概率最低的
  
  ASG 混合:
    On-Demand base: 2 台 (稳定)
    Spot: 0-8 台 (弹性)
    比例: 70% Spot + 30% On-Demand

  Spot 中断处理:
    2 分钟通知 (metadata endpoint)
    ECS/K8s: 优雅终止 + 自动重调度
```

## 4. 标签策略和成本分配？

**回答：**

```
标签 (Tags): 成本管理的基础

  必需标签:
    Environment:  dev / staging / prod
    Project:      项目名称
    Team:         团队/部门
    Owner:        负责人
    CostCenter:   成本中心
    ManagedBy:    terraform / manual

  Cost Allocation Tags:
    AWS Cost Explorer → 按标签维度分析成本
    需要在 Billing 控制台激活标签

标签策略实施:
  1. 定义标准 (标签键和值的命名规范)
  2. AWS Organizations: Tag Policies 强制
  3. IaC: Terraform default_tags 全局标签
  4. AWS Config: required-tags 规则检测
  5. IAM: 条件键强制带标签创建资源
```

```hcl
# Terraform 全局标签
provider "aws" {
  region = "us-east-1"
  default_tags {
    tags = {
      Environment = var.environment
      Project     = var.project
      Team        = "platform"
      ManagedBy   = "terraform"
    }
  }
}
```

```
成本分配报告:
  AWS Cost Explorer → 按标签分组
  → Project: A 花了 $5000, B 花了 $3000
  → Team: Platform $8000, Dev $2000
  → Environment: prod $7000, dev $1000

  FinOps (Financial Operations):
    可视化: 谁花了多少钱
```

## 6. 多云管理策略？

**回答：**

```
多云原因:
  避免供应商锁定
  利用各家优势 (GKE + AWS S3)
  合规/数据主权 (不同地区不同云)
  灾备 (跨云 DR)

多云挑战:
  运维复杂度 × N
  团队技能要求高
  网络互通困难
  统一认证和权限
  成本管理分散

统一管理工具:
  IaC:          Terraform (多 Provider)
  K8s:          多云 K8s (EKS + GKE + AKS)
  Service Mesh: Istio (跨集群)
  监控:         Grafana + Prometheus (跨云)
  认证:         Keycloak / Okta (统一 IdP)
  成本:         CloudHealth / Spot.io

多云最佳实践:
  1. 避免无意义的多云 (大多数不需要)
  2. 主云 + 辅云 (90/10 而非 50/50)
  3. 用 K8s 抽象底层差异
  4. 统一 IaC (Terraform)
  5. 数据层注意锁定风险
  6. 培训团队多云技能

何时真正需要多云:
  ✓ 合规要求 (政府/金融/数据主权)
  ✓ 收购公司用不同云
  ✓ 特定服务在特定云更好
  ✗ "避免锁定" 通常不值得额外复杂度
```

## 8. 隐藏成本陷阱？

**回答：**

```
常见隐藏成本:

1. NAT Gateway
   $0.045/h + $0.045/GB 数据处理
   → 一个 NAT 月费 $32 + 数据费
   优化: VPC Endpoint (S3/DynamoDB 免费 Gateway)
         减少不必要的出站流量

2. 数据传输
   同 AZ: 免费
   跨 AZ: $0.01/GB (双向)
   出公网: $0.09/GB
   → 10TB 出公网 = $900/月!
   优化: CloudFront 缓存, 同 AZ 部署

3. EBS 快照
   增量存储, 但旧快照可能很大
   优化: DLM 自动清理, 定期审计

4. 未使用的 EIP
   挂载到运行实例: 免费
   未使用: $0.005/h ($3.6/月)
   → 清理未挂载的 EIP

5. CloudWatch Logs
   $0.50/GB 采集, $0.03/GB 存储
   → 日志量大时成本可观
   优化: 设置过期策略, 减少不必要日志

6. API Gateway
   $3.50/百万请求 (REST)
   → 高 QPS 时成本高
   优化: 用 HTTP API ($1/百万), ALB 替代

7. S3 请求费
   GET $0.0004/1000 次, PUT $0.005/1000 次
   → 大量小文件操作成本可观
   优化: 合并小文件, 减少 LIST 操作

8. RDS Multi-AZ
   比 Single-AZ 贵一倍!
   → 开发环境不需要 Multi-AZ

排查方法:
  Cost Explorer → 按服务/使用类型 分组
  → 识别异常增长的服务
  → 钻取到具体资源
```

```
  │           ElastiCache Redis (Session + 热点数据)  │
  │     └── 搜索:                                   │

  S3:          ~$30
  NAT:         ~$100
  其他:        ~$120
  总计:        ~$1500/月
```

## 10. 云计算面试高频速答？

**回答：**

```
Q: IaaS / PaaS / SaaS 区别?
A: IaaS 管 OS 以上 (EC2), PaaS 管应用和数据 (Heroku),
   SaaS 啥都不管 (Gmail)

Q: Region 和 AZ 区别?
A: Region 是地理区域, AZ 是 Region 内的独立数据中心
   同 Region 内 AZ 间低延迟 (<2ms)

Q: Security Group 和 NACL?
A: SG 实例级有状态只有 Allow; NACL 子网级无状态有 Allow+Deny

Q: S3 有几个 9 的持久性?
A: 11 个 9 (99.999999999%), 几乎不会丢数据

Q: RDS Multi-AZ 和 Read Replica?
A: Multi-AZ: 同步复制, 自动故障转移, 不可读
   Read Replica: 异步复制, 读分流, 可跨 Region

Q: Lambda 冷启动?
A: 首次调用需要初始化运行时, 用 Provisioned Concurrency 解决

Q: CloudFormation 和 Terraform 选哪个?
A: 多云用 Terraform, 纯 AWS 两个都行
   Terraform 更灵活但需管理 State

Q: VPC Endpoint 用在哪?
A: 私有子网访问 AWS 服务时替代 NAT
   S3/DynamoDB 用 Gateway (免费)
   其他服务用 Interface (收费)

Q: Spot 实例会不会突然没了?
A: 会, 2 分钟前通知, 适合容错型工作负载
   用 Spot Fleet 多类型降低中断风险

Q: 如何实现高可用?
A: 跨 3 AZ + ALB + Auto Scaling + 数据库 Multi-AZ
   消除单点故障, 自动检测和恢复

Q: RI 和 Savings Plans 选哪个?
A: Savings Plans 更灵活 (可换实例类型/Region)
   RI 折扣更大但固定
   新用户推荐 Compute Savings Plans

Q: 如何保证云安全?
A: 共享责任模型 + IAM 最小权限 + 加密 (KMS)
   + 网络隔离 (VPC/SG) + 审计 (CloudTrail)
   + 威胁检测 (GuardDuty)
```
