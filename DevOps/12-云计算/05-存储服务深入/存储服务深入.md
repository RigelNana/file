# 存储服务深入

---

## 1. S3 核心概念和存储类别？

**回答：**

```
S3 (Simple Storage Service):
  对象存储, 无限容量, 11 个 9 持久性

  基本概念:
    Bucket:  存储桶 (全局唯一名称)
    Object:  对象 (文件 + 元数据)
    Key:     对象键 (路径名, 如 images/photo.jpg)
    Region:  桶所在区域

存储类别:
  ┌─────────────────────┬──────────┬───────────────┬────────────┐
  │ 类别                 │ 可用性    │ 取回时间       │ 成本       │
  ├─────────────────────┼──────────┼───────────────┼────────────┤
  │ S3 Standard          │ 99.99%  │ 毫秒          │ $0.023/GB  │
  │ S3 Standard-IA       │ 99.9%   │ 毫秒          │ $0.0125/GB │
  │ S3 One Zone-IA       │ 99.5%   │ 毫秒          │ $0.01/GB   │
  │ S3 Intelligent-Tier. │ 99.9%   │ 毫秒-分钟     │ 自动分层    │
  │ S3 Glacier IR        │ 99.9%   │ 毫秒          │ $0.004/GB  │
  │ S3 Glacier Flexible  │ 99.99%  │ 1-12小时      │ $0.0036/GB │
  │ S3 Glacier Deep Arch.│ 99.99%  │ 12-48小时     │ $0.00099/GB│
  └─────────────────────┴──────────┴───────────────┴────────────┘

  IA (Infrequent Access): 不频繁访问, 取回收费
  One Zone: 单 AZ, 便宜但无跨 AZ 冗余
  Intelligent-Tiering: 自动根据访问模式迁移

选择建议:
  频繁访问:      Standard
  不确定模式:    Intelligent-Tiering
  不常访问:      Standard-IA
  非关键存档:    One Zone-IA
  合规归档:      Glacier Flexible
  超长期归档:    Glacier Deep Archive
```

## 2. S3 生命周期和版本控制？

**回答：**

```
生命周期策略 (Lifecycle Policy):
  自动迁移存储类别或过期删除

  常见策略:
    日志文件:
      0-30 天:    Standard
      30-90 天:   Standard-IA
      90-365 天:  Glacier Flexible
      365+ 天:    Glacier Deep Archive 或删除

    用户上传文件:
      0-90 天:    Standard
      90-180 天:  Standard-IA
      180+ 天:    Glacier IR

  限制:
    Standard → Standard-IA: 至少 30 天
    Standard-IA → Glacier: 至少 30 天
    最小对象大小: 128 KB (IA/Glacier 迁移)

版本控制 (Versioning):
  启用后每次覆盖/删除都保留历史版本
  
  操作:
    PUT: 创建新版本, 旧版本保留
    DELETE: 添加 Delete Marker (逻辑删除)
    真正删除: 指定版本 ID 删除

  与生命周期配合:
    NoncurrentVersionExpiration: 自动删除旧版本
    例: 保留最近 3 个版本, 超过 30 天的旧版本删除

  MFA Delete:
    删除版本或关闭版本控制需要 MFA
    防止误删和恶意删除
    只能 Root 账号配置
```

## 3. S3 安全和访问控制？

**回答：**

```
访问控制层次:

  1. Block Public Access (账号/桶级别)
     最高优先, 直接阻止所有公开访问
     默认启用! (新桶默认全部阻止)

  2. Bucket Policy (桶策略)
     Resource-based Policy, JSON 格式
     控制谁可以对桶执行什么操作

  3. IAM Policy (身份策略)
     控制 IAM 用户/角色可以访问哪些桶

  4. ACL (访问控制列表) → 不推荐, 遗留功能

  优先级: Block Public Access > Deny > Allow

Bucket Policy 示例:
```

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCloudFrontOAC",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudfront.amazonaws.com"
      },
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-website-bucket/*",
      "Condition": {
        "StringEquals": {
          "AWS:SourceArn": "arn:aws:cloudfront::123456:distribution/EXXX"
        }
      }
    },
    {
      "Sid": "DenyUnencryptedTransport",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:*",
      "Resource": "arn:aws:s3:::my-bucket/*",
      "Condition": {
        "Bool": {"aws:SecureTransport": "false"}
      }
    }
  ]
}
```

```
加密:
  SSE-S3:  S3 管理密钥 (默认, 推荐)
  SSE-KMS: KMS 管理密钥 (审计/权限控制)
  SSE-C:   客户提供密钥 (自管理)
  CSE:     客户端加密 (上传前加密)

  所有新桶默认启用 SSE-S3 加密

预签名 URL (Presigned URL):
  临时授权访问 (上传/下载)
  有过期时间 (默认 1 小时)
  不需要修改桶策略或权限
```

## 4. S3 高级功能？

**回答：**

```
跨区域复制 (CRR - Cross Region Replication):
  源桶 (us-east-1) → 目标桶 (eu-west-1)
  用途: 合规, 降低延迟, 灾备
  需要启用版本控制

同区域复制 (SRR - Same Region Replication):
  同 Region 不同桶
  用途: 日志聚合, 测试环境复制

S3 事件通知:
  触发源: PUT, DELETE, COPY, 复制完成等
  目标:
    Lambda → 图片缩略图生成
    SQS    → 异步处理队列
    SNS    → 通知
    EventBridge → 灵活路由

S3 Select / Athena:
  S3 Select: 直接在 S3 上查询 CSV/JSON/Parquet
  Athena:    SQL 查询 S3 数据 (无服务器)
  用途: 日志分析, 数据湖查询, 成本分析

Transfer Acceleration:
  利用 CloudFront Edge 加速上传
  适用: 跨国大文件上传

Multipart Upload:
  大文件分片上传 (建议 > 100 MB)
  支持并行上传, 断点续传

S3 Object Lock:
  WORM (Write Once Read Many)
  合规模式: 任何人不能删除/覆盖
  治理模式: 特权用户可以解除
  用途: 合规归档, 日志保留

S3 Batch Operations:
  批量操作: 复制, 加密, 标签, Lambda 处理
  适用: 大规模数据迁移/处理
```

## 5. EBS vs EFS vs FSx？

**回答：**

```
AWS 文件/块存储对比:

  ┌──────────┬──────────────┬──────────────┬──────────────┐
  │ 特性      │ EBS          │ EFS          │ FSx          │
  ├──────────┼──────────────┼──────────────┼──────────────┤
  │ 类型      │ 块存储       │ NFS 文件系统  │ 多种文件系统  │
  │ 协议      │ Block (挂载) │ NFS v4.1     │ SMB/Lustre/  │
  │          │              │              │ NetApp/ONTAP │
  │ 多实例    │ 单实例 (io2  │ 多实例共享    │ 多实例共享    │
  │          │ Multi-Attach)│              │              │
  │ 跨 AZ    │ 单 AZ        │ 多 AZ        │ 取决于类型   │
  │ 性能      │ 极高 IOPS    │ 中等         │ 取决于类型   │
  │ 适用      │ 数据库, OS   │ 共享文件,    │ Windows 共享,│
  │          │ 磁盘        │ CMS, K8s PV  │ HPC         │
  │ 自动扩展  │ ✗ (手动调整) │ ✓           │ ✓           │
  └──────────┴──────────────┴──────────────┴──────────────┘

EFS 使用场景:
  ✓ 多 EC2 共享文件 (NFS)
  ✓ K8s PersistentVolume (EFS CSI Driver)
  ✓ Lambda 挂载 (共享代码/模型)
  ✓ CMS (WordPress 多实例)

  性能模式:
    General Purpose: 通用 (延迟敏感)
    Max I/O: 高吞吐 (大规模并发)
  
  吞吐模式:
    Bursting: 突发
    Provisioned: 预配置
    Elastic: 弹性 (自动, 推荐)

  存储类别:
    Standard + IA → 生命周期自动分层
    One Zone + IA → 单 AZ, 便宜

FSx 类型:
  FSx for Windows: SMB, 与 AD 集成
  FSx for Lustre:  高性能计算 (HPC), ML 训练
  FSx for NetApp ONTAP: 多协议, 企业功能
  FSx for OpenZFS: Linux 文件系统
```

## 6. 数据迁移服务？

**回答：**

```
数据迁移方案:

  ┌──────────────────┬──────────────┬──────────────────────┐
  │ 方案              │ 数据量       │ 说明                 │
  ├──────────────────┼──────────────┼──────────────────────┤
  │ AWS CLI / SDK    │ < 1 TB       │ 网络传输, 简单直接    │
  │ S3 Transfer Accel│ > 1 GB 跨国  │ CDN Edge 加速上传    │
  │ DataSync         │ 任意         │ 自动持续同步         │
  │ Snowball Edge    │ 10-80 TB     │ 物理设备, 离线传输    │
  │ Snowmobile       │ PB 级        │ 集装箱卡车           │
  │ Storage Gateway  │ 持续         │ 混合云存储网关        │
  └──────────────────┴──────────────┴──────────────────────┘

AWS DataSync:
  本地 NFS/SMB → S3/EFS/FSx
  自动验证, 加密传输, 带宽控制
  定时任务: 增量同步

Snowball Edge:
  物理设备寄送, 离线复制数据
  Compute 版本: 可在边缘运行 EC2/Lambda
  大量数据 + 带宽不足时使用

  10 TB 数据迁移时间对比:
    100 Mbps 网络: ~9 天
    1 Gbps 网络:   ~1 天
    Snowball:      3-5 天 (含邮寄)

Storage Gateway:
  在本地部署网关, 连接 AWS 存储
  File Gateway:   NFS/SMB → S3
  Volume Gateway: iSCSI → S3 (EBS 快照)
  Tape Gateway:   虚拟磁带库 → S3 Glacier
```

## 7. S3 性能优化？

**回答：**

```
S3 性能:
  单前缀: 5500 GET/s + 3500 PUT/s
  多前缀并行: 线性扩展

优化策略:
  1. 前缀分散
     ✗ s3://bucket/2024-01-01/file1
     ✓ s3://bucket/hash1/2024-01-01/file1
     ✓ s3://bucket/hash2/2024-01-01/file2
     (现在 S3 会自动优化, 不再需要随机前缀)

  2. Multipart Upload
     大文件 (>100 MB) 分片并行上传
     断点续传: 失败的分片可重传

  3. S3 Transfer Acceleration
     利用 CloudFront Edge 加速传输
     跨国上传效果显著

  4. Byte-Range Fetches
     Range GET: 并行下载文件的不同部分
     加速大文件下载

  5. 请求速率
     逐步增加请求速率 (避免 503 SlowDown)
     使用指数退避重试

  6. ListObjects 优化
     使用 S3 Inventory 替代 ListObjects (海量对象)
     使用分页 (--max-items, --page-size)

  7. KMS 限制
     SSE-KMS 受 KMS API 配额限制 (5500-30000/s)
     高吞吐场景考虑 SSE-S3 或 Bucket Key
```

## 8. 数据湖架构？

**回答：**

```
AWS 数据湖架构:

  数据源                    数据湖 (S3)               分析服务
  ┌──────────┐           ┌──────────────┐         ┌─────────────┐
  │ 数据库    │ ─Glue─→  │ Raw Zone     │         │ Athena      │
  │ API      │ ─Kinesis→ │ (原始数据)    │─Glue─→ │ (SQL 查询)  │
  │ IoT      │ ─Firehose→│              │ ETL    │             │
  │ 日志     │ ─DataSync→│ Curated Zone │         │ Redshift    │
  │ SaaS     │           │ (清洗后数据)  │         │ (数据仓库)  │
  └──────────┘           │              │         │             │
                         │ Analytics Zone│         │ QuickSight  │
                         │ (分析结果)    │         │ (BI 可视化) │
                         └──────────────┘         └─────────────┘

核心组件:
  S3:        数据湖存储 (核心)
  AWS Glue:  ETL + 数据目录 (Schema 管理)
  Athena:    Serverless SQL 查询 (直接查 S3)
  Lake Formation: 数据湖治理 (权限, 审计)

S3 数据湖最佳实践:
  分层: raw/ → processed/ → curated/ → analytics/
  格式: Parquet / ORC (列式, 压缩率高, 查询快)
  分区: year=2024/month=01/day=15/ (减少扫描)
  加密: SSE-S3 或 SSE-KMS
  访问: Lake Formation 细粒度权限

Athena 查询 S3:
  无需加载数据, 直接 SQL 查询 S3
  按扫描数据量计费 ($5/TB)
  用 Parquet + 分区可大幅减少成本
```

## 9. 备份与灾备存储策略？

**回答：**

```
AWS Backup:
  集中式备份管理, 支持多种服务:
    EC2 (AMI), EBS (快照), RDS, DynamoDB,
    EFS, FSx, S3, Storage Gateway

  Backup Plan:
    备份频率: 每日/每周/每月
    保留策略: 保留 N 天/月
    跨 Region: 自动复制到另一个 Region
    Vault Lock: WORM 合规模式

RDS 备份:
  自动备份: 每日自动快照 + 事务日志
  手动快照: 手动创建, 不自动过期
  PITR: 恢复到最近 5 分钟任意时间点
  跨 Region: 自动复制到其他 Region

EBS 快照:
  增量快照: 只保存变化部分
  跨 AZ/Region 复制
  自动快照: DLM (Data Lifecycle Manager) 定时

  DLM 策略:
    每日快照, 保留 7 天
    每周快照, 保留 4 周
    跨 Region 复制

灾备存储策略:
  RPO (Recovery Point Objective):
    可接受的数据丢失时间
    RPO = 1h → 至少每小时备份

  RTO (Recovery Time Objective):
    可接受的恢复时间
    RTO = 15min → 需要热备/温备

  策略:
    Backup & Restore:  RTO 小时级, 成本最低
    Pilot Light:       RTO 分钟-小时, 最小核心运行
    Warm Standby:      RTO 分钟级, 缩小版环境运行
    Active-Active:     RTO 接近 0, 成本最高
```

## 10. 存储服务面试题？

**回答：**

```
Q: S3 和 EBS 的区别?
A: S3: 对象存储, HTTP API, 无限容量, 适合文件/备份/静态资源
   EBS: 块存储, 挂载到 EC2, 有限容量, 适合数据库/OS 磁盘
   S3 不能挂载为磁盘, EBS 不能直接通过 URL 访问

Q: S3 如何保证 11 个 9 持久性?
A: 数据自动跨 3 个 AZ 冗余存储
   使用校验和验证数据完整性
   自动检测和修复损坏

Q: 如何设计 S3 桶策略防止数据泄露?
A: 1. Block Public Access (全部启用)
   2. Bucket Policy 最小权限
   3. 强制 HTTPS (DenyUnencryptedTransport)
   4. 启用加密 (SSE-KMS 可审计)
   5. 启用访问日志和 CloudTrail S3 数据事件
   6. Macie 自动发现敏感数据

Q: 什么时候用 EFS 而不是 EBS?
A: 需要多实例共享文件 → EFS (NFS)
   需要高 IOPS 磁盘 → EBS (io2/gp3)
   EFS 自动扩展, 多 AZ; EBS 固定大小, 单 AZ

Q: 如何迁移 50 TB 数据到 S3?
A: 网络够用 (>1Gbps): DataSync (网络传输)
   网络不够/时间紧: Snowball Edge (物理设备)
   持续同步: DataSync 定时任务
   跨国: S3 Transfer Acceleration

Q: S3 Glacier 和 Glacier Deep Archive 怎么选?
A: Glacier Flexible: 需要偶尔访问, 分钟-小时取回, 合规归档
   Glacier Deep Archive: 极少访问, 12-48 小时取回, 最低成本
   磁带替代方案 → Deep Archive
```
