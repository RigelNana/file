## 一、网络分层模型

### 1. OSI 七层模型和 TCP/IP 四层模型分别是什么？

**答：**

| OSI 七层 | TCP/IP 四层 | 功能 | 协议/设备示例 |
|---------|------------|------|-------------|
| 应用层 | 应用层 | 提供网络服务给应用程序 | HTTP, HTTPS, DNS, FTP, SSH, SMTP |
| 表示层 | 应用层 | 数据编码、加密、压缩 | SSL/TLS, JPEG, ASCII |
| 会话层 | 应用层 | 建立、管理和终止会话 | RPC, NetBIOS |
| 传输层 | 传输层 | 端到端通信、流量控制 | TCP, UDP |
| 网络层 | 网络层 | 路由选择、IP寻址 | IP, ICMP, ARP; 路由器 |
| 数据链路层 | 网络接口层 | 帧传输、MAC寻址 | Ethernet, PPP; 交换机 |
| 物理层 | 网络接口层 | 比特流传输 | 网线、光纤、集线器 |

### 2. 数据在各层的封装过程是怎样的？

**答：**

```
发送方（自上而下封装）：
应用层：    [数据]
传输层：    [TCP头|数据]                → 段(Segment)
网络层：    [IP头|TCP头|数据]           → 包(Packet)
数据链路层：[帧头|IP头|TCP头|数据|帧尾]  → 帧(Frame)
物理层：    01010110...                → 比特流(Bits)

接收方（自下而上解封装）：反向过程
```

## 二、TCP 协议

### 3. TCP 和 UDP 有什么区别？

**答：**

| 特性 | TCP | UDP |
|------|-----|-----|
| 连接方式 | 面向连接（三次握手） | 无连接 |
| 可靠性 | 可靠传输（确认、重传、排序） | 不可靠，尽最大努力交付 |
| 传输方式 | 字节流 | 数据报 |
| 速度 | 较慢（有握手和确认开销） | 较快 |
| 拥塞控制 | 有 | 无 |
| 头部大小 | 20-60 字节 | 8 字节 |
| 应用场景 | HTTP, SSH, FTP, SMTP | DNS, DHCP, 视频流, VoIP |

### 4. TCP 三次握手的详细过程？为什么是三次？

**答：**

```
客户端(CLOSED)              服务器(LISTEN)
    |                           |
    |--- SYN (seq=x) --------->|   1. 客户端发SYN，进入SYN_SENT
    |                           |      服务器收到，进入SYN_RCVD
    |<-- SYN+ACK (seq=y,       |   2. 服务器回复SYN+ACK
    |    ack=x+1) -------------|
    |                           |   3. 客户端发ACK，进入ESTABLISHED
    |--- ACK (ack=y+1) ------->|      服务器收到，进入ESTABLISHED
    |                           |
```

**为什么是三次而不是两次？**
- 防止已失效的连接请求报文突然又传到服务器，导致服务器误开连接浪费资源
- 确保双方的发送和接收能力都正常
- 两次握手无法确认客户端的接收能力

**为什么不是四次？**
- 三次已经足够确认双方的收发能力，第四次是多余的

### 5. TCP 四次挥手的详细过程？为什么是四次？

**答：**

```
客户端(ESTABLISHED)          服务器(ESTABLISHED)
    |                           |
    |--- FIN (seq=u) --------->|   1. 客户端发FIN，进入FIN_WAIT_1
    |                           |
    |<-- ACK (ack=u+1) --------|   2. 服务器发ACK，进入CLOSE_WAIT
    |                           |      客户端收到，进入FIN_WAIT_2
    |   (服务器可能还有数据要发送)  |
    |<-- FIN (seq=w) ----------|   3. 服务器发FIN，进入LAST_ACK
    |                           |
    |--- ACK (ack=w+1) ------->|   4. 客户端发ACK，进入TIME_WAIT
    |                           |      服务器收到，进入CLOSED
    | (等待 2MSL)                |
    |      CLOSED               |
```

**为什么是四次？**
- TCP 是全双工通信，每个方向需要单独关闭
- 服务器收到 FIN 后可能还有数据要发送，所以 ACK 和 FIN 分开发送

**TIME_WAIT 状态为什么等待 2MSL？**
- MSL (Maximum Segment Lifetime)：报文最大生存时间
- 确保最后一个 ACK 能到达服务器（如果丢失，服务器会重发 FIN）
- 确保本次连接的所有报文从网络中消失，不影响下一次连接

### 6. TCP 的拥塞控制算法有哪些？

**答：**

1. **慢启动（Slow Start）**：
   - 初始窗口为1个MSS，每收到一个ACK窗口翻倍（指数增长）
   - 达到慢启动阈值（ssthresh）后进入拥塞避免

2. **拥塞避免（Congestion Avoidance）**：
   - 窗口线性增长（每个RTT增加1个MSS）

3. **快重传（Fast Retransmit）**：
   - 收到3个重复ACK立即重传，不等超时

4. **快恢复（Fast Recovery）**：
   - 快重传后，ssthresh 和 cwnd 减半，进入拥塞避免（而非慢启动）

### 7. 什么是 TCP 粘包问题？如何解决？

**答：** TCP 是字节流协议，没有消息边界的概念。发送方发送的多个小数据包可能被合并成一个大包发送（Nagle算法），或者接收方一次读取了多个数据包。

**解决方案：**
1. **固定长度**：每个消息固定长度，不足补齐
2. **分隔符**：用特殊字符分隔消息（如 `\r\n`）
3. **消息头+消息体**：头部包含消息长度（最常用）
4. **使用应用层协议**：HTTP、gRPC 等已经处理了这个问题

### 9. HTTP/1.0、HTTP/1.1、HTTP/2、HTTP/3 的区别？

**答：**

| 特性 | HTTP/1.0 | HTTP/1.1 | HTTP/2 | HTTP/3 |
|------|----------|----------|--------|--------|
| 连接方式 | 短连接 | 默认长连接 (Keep-Alive) | 多路复用 | 基于QUIC(UDP) |
| 请求模式 | 每次请求新建连接 | 管道化（但有队头阻塞） | 二进制帧，多流并发 | 解决了TCP队头阻塞 |
| 头部 | 文本格式 | 文本格式 | HPACK压缩 | QPACK压缩 |
| 服务器推送 | 不支持 | 不支持 | 支持 Server Push | 支持 |
| 加密 | 可选 | 可选 | 事实上要求TLS | 内置TLS 1.3 |

### 10. HTTPS 的工作原理？TLS 握手过程？

**答：** HTTPS = HTTP + TLS/SSL，在 HTTP 下层加入了加密层。

**TLS 1.2 握手过程（简化）：**

```
客户端                              服务器
  |--- ClientHello ----------------->|  支持的加密套件、随机数A
  |<-- ServerHello ------------------|  选择的加密套件、随机数B
  |<-- Certificate ------------------|  服务器证书（含公钥）
  |<-- ServerHelloDone ---------------|
  |                                   |
  |--- ClientKeyExchange ----------->|  用公钥加密的预主密钥
  |--- ChangeCipherSpec ------------>|  切换到加密通信
  |--- Finished -------------------->|  加密的验证数据
  |                                   |
  |<-- ChangeCipherSpec --------------|
  |<-- Finished ----------------------|
  |                                   |
  |====== 加密通信开始 ===============|
```

**关键概念：**
- **非对称加密**：用于密钥交换（RSA/ECDHE），安全但慢
- **对称加密**：用于数据传输（AES），快速
- **证书**：由CA签发，验证服务器身份
- **TLS 1.3**：简化握手为1个RTT（0-RTT重连），更安全

### 11. GET 和 POST 的区别？

**答：**

| 特性 | GET | POST |
|------|-----|------|
| 参数位置 | URL 中（Query String） | 请求体中 |
| 缓存 | 可以被缓存 | 默认不缓存 |
| 书签 | 可以做书签 | 不能 |
| 长度限制 | URL长度受浏览器限制（约2KB-8KB） | 无限制 |
| 幂等性 | 幂等（多次请求结果相同） | 非幂等 |
| 安全性 | 参数暴露在URL中 | 参数在请求体中（但仍需HTTPS） |
| 语义 | 获取资源 | 提交数据/创建资源 |

## 四、DNS

### 12. DNS 解析过程是怎样的？

**答：**

```
用户输入 www.example.com
    ↓
1. 浏览器缓存 → 查找是否有缓存
    ↓ 未命中
2. 系统缓存 → /etc/hosts、系统DNS缓存
    ↓ 未命中
3. 本地DNS服务器（递归解析器）→ 运营商DNS或配置的DNS
    ↓ 未命中
4. 根域名服务器（.）→ 返回 .com 顶级域服务器地址
    ↓
5. 顶级域名服务器（.com）→ 返回 example.com 权威DNS服务器地址
    ↓
6. 权威DNS服务器（example.com）→ 返回 www.example.com 的 IP 地址
    ↓
7. 本地DNS服务器缓存结果并返回给客户端
    ↓
8. 浏览器得到 IP 地址，建立 TCP 连接
```

### 13. 常见的 DNS 记录类型有哪些？

**答：**

| 记录类型 | 说明 | 示例 |
|---------|------|------|
| A | 域名映射到 IPv4 地址 | `example.com → 1.2.3.4` |
| AAAA | 域名映射到 IPv6 地址 | `example.com → 2001:db8::1` |
| CNAME | 域名别名，指向另一个域名 | `www.example.com → example.com` |
| MX | 邮件交换记录 | `example.com MX → mail.example.com` |
| NS | 域名服务器记录 | `example.com NS → ns1.example.com` |
| TXT | 文本记录（常用于验证） | SPF、DKIM 等 |
| SRV | 服务定位记录 | `_http._tcp.example.com` |
| PTR | 反向DNS，IP映射到域名 | `4.3.2.1 → example.com` |
| SOA | 起始授权记录 | 域名的主DNS服务器信息 |

### 14. DNS 常用排障命令？

**答：**

```bash
# 查询DNS记录
dig example.com                    # 默认查询A记录
dig example.com MX                 # 查询MX记录
dig +short example.com             # 只输出结果
dig +trace example.com             # 跟踪完整解析过程
dig @8.8.8.8 example.com           # 指定DNS服务器

# nslookup
nslookup example.com
nslookup -type=mx example.com

# host
host example.com
host -t CNAME www.example.com

# 查看/修改DNS配置
cat /etc/resolv.conf               # DNS服务器配置
systemd-resolve --status           # systemd-resolved 状态
resolvectl query example.com       # systemd-resolved 查询
```

## 五、网络基础概念

### 15. IP 地址分类和子网划分？

**答：**

**传统分类：**

| 类别 | 范围 | 私有地址范围 | 默认子网掩码 |
|------|------|------------|-------------|
| A类 | 1.0.0.0 – 126.255.255.255 | 10.0.0.0/8 | 255.0.0.0 (/8) |
| B类 | 128.0.0.0 – 191.255.255.255 | 172.16.0.0/12 | 255.255.0.0 (/16) |
| C类 | 192.0.0.0 – 223.255.255.255 | 192.168.0.0/16 | 255.255.255.0 (/24) |

**CIDR 子网划分示例：**
```
192.168.1.0/24
网络地址：192.168.1.0
广播地址：192.168.1.255
可用主机：192.168.1.1 ~ 192.168.1.254（254个）
子网掩码：255.255.255.0

192.168.1.0/26
子网掩码：255.255.255.192
每个子网主机数：62个
```

### 16. VLAN 是什么？有什么作用？

**答：** VLAN（Virtual LAN，虚拟局域网）通过交换机的逻辑配置将一个物理网络划分为多个逻辑广播域。

**作用：**
- 隔离广播域，减少广播流量
- 提高安全性，不同 VLAN 间需要路由器才能通信
- 灵活管理，同一交换机上的端口可以划分到不同 VLAN

**VLAN 间通信：** 需要三层交换机或路由器（路由器的子接口 Router-on-a-Stick）

### 17. NAT 是什么？有哪些类型？

**答：** NAT（Network Address Translation，网络地址转换）将私有 IP 地址转换为公有 IP 地址。

| NAT 类型 | 说明 |
|---------|------|
| 静态 NAT | 一对一映射，一个私有IP对应一个公有IP |
| 动态 NAT | 从公有IP池中动态分配 |
| PAT/NAPT | 端口地址转换，多个私有IP共享一个公有IP（最常用） |
| DNAT | 目的地址转换，用于端口转发/负载均衡 |
| SNAT | 源地址转换，用于出站流量 |

```bash
# Linux iptables NAT 示例
# SNAT：源地址转换（出站）
iptables -t nat -A POSTROUTING -s 192.168.1.0/24 -o eth0 -j MASQUERADE

# DNAT：目的地址转换（端口转发）
iptables -t nat -A PREROUTING -p tcp --dport 80 -j DNAT --to-destination 192.168.1.100:8080
```

## 六、网络排障

### 18. 浏览器输入 URL 到页面显示的全过程？

**答：** 这是经典面试题，涉及多层网络知识：

1. **URL 解析**：浏览器解析 URL 的协议、域名、端口、路径
2. **DNS 解析**：域名 → IP 地址（浏览器缓存 → 系统缓存 → DNS服务器）
3. **TCP 连接**：三次握手建立 TCP 连接
4. **TLS 握手**（如果是 HTTPS）：协商加密参数，建立安全通道
5. **发送 HTTP 请求**：构建请求报文（方法、头部、体）
6. **服务器处理**：Web服务器（Nginx）→ 应用服务器 → 数据库
7. **返回 HTTP 响应**：状态码、响应头、响应体（HTML）
8. **浏览器渲染**：解析 HTML → 构建 DOM 树 → 加载 CSS/JS → 渲染页面
9. **TCP 四次挥手**：关闭连接（或保持 Keep-Alive）

### 19. 网络故障排查的一般思路？

**答：** 按照 OSI 模型**自下而上**排查：

```bash
# 1. 物理层/链路层：检查网络连接
ip link show                    # 网卡状态（UP/DOWN?）
ethtool eth0                    # 网卡信息和连接状态

# 2. 网络层：检查IP和路由
ip addr show                    # IP 地址是否正确
ip route show                   # 路由表是否正确
ping 127.0.0.1                  # 本地回环是否正常
ping <gateway>                  # 网关是否可达
ping 8.8.8.8                   # 外网是否可达

# 3. DNS：检查域名解析
nslookup example.com            # DNS 解析是否正常
cat /etc/resolv.conf            # DNS 配置

# 4. 传输层：检查端口
ss -tulnp                       # 服务是否在监听
telnet host port                # 端口是否可达
nc -zv host port                # 测试端口连通性

# 5. 应用层：检查服务
curl -v https://example.com     # HTTP 请求测试
systemctl status nginx          # 服务状态

# 6. 防火墙：检查是否被拦截
iptables -L -n                  # 查看iptables规则
firewall-cmd --list-all         # 查看firewalld规则
```

### 20. 如何排查网络延迟和丢包问题？

**答：**

```bash
# 1. ping 测试基本连通性和延迟
ping -c 100 target              # 发100个包看统计

# 2. mtr 综合分析（ping + traceroute）
mtr -r -c 100 target            # 报告模式，发100个包

# 3. traceroute 查看路由路径
traceroute target               # ICMP 模式
traceroute -T -p 80 target      # TCP 模式

# 4. 抓包分析
tcpdump -i eth0 host target -w capture.pcap   # 抓包
# 用 Wireshark 分析 pcap 文件

# 5. 带宽测试
iperf3 -s                       # 服务端
iperf3 -c server_ip             # 客户端

# 6. 查看网络统计
ss -s                           # 连接统计
netstat -s                      # 协议统计（重传、丢包等）
cat /proc/net/snmp              # 内核网络统计
```

## 七、常见网络服务

### 21. DHCP 的工作过程？

**答：** DHCP（动态主机配置协议）自动分配 IP 地址，使用 UDP 端口 67（服务器）和 68（客户端）。

```
客户端                          DHCP服务器
  |--- DHCP Discover -------->|   广播寻找DHCP服务器
  |<-- DHCP Offer ------------|   提供IP地址和配置
  |--- DHCP Request --------->|   客户端选择并请求
  |<-- DHCP Ack --------------|   服务器确认分配
```

**DORA 过程：** Discover → Offer → Request → Acknowledge

### 22. SSH 的工作原理？

**答：** SSH（Secure Shell）提供加密的远程登录和命令执行。

**认证方式：**

| 方式 | 安全性 | 说明 |
|------|-------|------|
| 密码认证 | 一般 | 输入用户密码 |
| 密钥认证 | 高 | 公钥放在服务器，私钥在本地（推荐） |

**密钥认证流程：**

```bash
# 1. 生成密钥对
ssh-keygen -t ed25519 -C "email@example.com"

# 2. 将公钥复制到服务器
ssh-copy-id user@server

# 3. 登录
ssh user@server

# SSH 配置 ~/.ssh/config
Host myserver
    HostName 192.168.1.100
    User devops
    Port 22
    IdentityFile ~/.ssh/id_ed25519
```

**SSH 安全加固：**
```bash
# /etc/ssh/sshd_config
Port 2222                          # 修改默认端口
PermitRootLogin no                 # 禁止root登录
PasswordAuthentication no          # 禁止密码登录
MaxAuthTries 3                     # 最大认证尝试次数
AllowUsers devops                  # 只允许特定用户
```

## 八、网络安全基础

### 23. 常见的网络攻击方式有哪些？

**答：**

| 攻击类型 | 说明 | 防御措施 |
|---------|------|---------|
| DDoS | 分布式拒绝服务攻击 | CDN、流量清洗、限流 |
| SYN Flood | 大量半连接耗尽资源 | SYN Cookie、限制半连接数 |
| ARP 欺骗 | 伪造ARP响应 | 静态ARP绑定、DAI |
| DNS 劫持 | 篡改DNS响应 | DNSSEC、可信DNS |
| 中间人攻击 | 截获和篡改通信 | HTTPS、证书固定 |
| 端口扫描 | 探测开放端口 | 防火墙、端口敲门 |

### 24. 什么是 CDN？它是如何工作的？

**答：** CDN（Content Delivery Network，内容分发网络）通过在全球部署边缘节点缓存内容，使用户从最近的节点获取数据。

**工作流程：**
1. 用户请求 `www.example.com`
2. DNS 解析返回最近的 CDN 边缘节点 IP（通过 CNAME 和智能 DNS）
3. 如果边缘节点有缓存，直接返回
4. 如果没有缓存（Cache Miss），从源站拉取，缓存后返回

**优势：** 降低延迟、减少源站带宽压力、提高可用性、DDoS 防护

### 25. tcpdump 常用命令？

**答：**

```bash
# 基本用法
tcpdump -i eth0                        # 监听 eth0
tcpdump -i any                         # 监听所有接口
tcpdump -i eth0 -c 100                 # 抓100个包

# 过滤条件
tcpdump host 192.168.1.1               # 指定主机
tcpdump src host 192.168.1.1           # 指定源主机
tcpdump dst host 192.168.1.1           # 指定目标主机
tcpdump port 80                        # 指定端口
tcpdump -i eth0 tcp                    # 只抓TCP包
tcpdump 'tcp[tcpflags] & tcp-syn != 0' # 抓SYN包

# 组合过滤
tcpdump 'host 192.168.1.1 and port 80'
tcpdump 'src 192.168.1.0/24 and (dst port 80 or dst port 443)'

# 输出选项
tcpdump -w capture.pcap                # 保存到文件
tcpdump -r capture.pcap                # 读取文件
tcpdump -A                             # ASCII显示包内容
tcpdump -X                             # 十六进制+ASCII显示
tcpdump -nn                            # 不解析主机名和端口名
tcpdump -v / -vv / -vvv               # 增加详细程度
```

### 26. 如何使用 Wireshark 分析网络问题？

**答：**

常用过滤器：
```
# 显示过滤器
ip.addr == 192.168.1.1               # IP 地址
tcp.port == 80                       # TCP 端口
http.request.method == "GET"         # HTTP GET 请求
tcp.analysis.retransmission          # TCP 重传
tcp.analysis.zero_window             # 零窗口（接收方缓冲区满）
dns.qry.name == "example.com"       # DNS 查询
tcp.flags.syn == 1                   # SYN 包

# 常见分析场景
# 1. TCP 重传过多 → 网络质量差或丢包
# 2. RST 包 → 连接被拒绝或异常断开
# 3. 零窗口 → 接收方处理跟不上
# 4. 三次握手时间长 → 网络延迟高
```
