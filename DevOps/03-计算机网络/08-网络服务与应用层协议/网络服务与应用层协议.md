## 1. DHCP 的工作过程？

**回答：**

### DHCP 四步流程（DORA）

```
Client                        DHCP Server
  │                               │
  │── DISCOVER (广播) ───────────→│  1. 客户端广播"谁能给我IP？"
  │                               │
  │←──── OFFER ───────────────────│  2. 服务器回应"我可以给你192.168.1.100"
  │                               │
  │── REQUEST (广播) ────────────→│  3. 客户端广播"我要192.168.1.100"
  │                               │     (可能有多个DHCP服务器，告知选择结果)
  │←──── ACK ─────────────────────│  4. 服务器确认"好的，租约生效"
  │                               │

关键点：
  - DISCOVER 和 REQUEST 是广播（因为客户端还没有 IP）
  - 租约制：有过期时间（通常 24h），到 50% 时续约
  - DHCP 中继：路由器可以转发 DHCP 请求到其他子网的 DHCP 服务器
```

### DHCP 能分配的信息

```
- IP 地址
- 子网掩码
- 默认网关
- DNS 服务器
- 域名
- NTP 服务器
- 租约时间
- 其他自定义选项
```

### DHCP 常见问题排查

```bash
# 查看 DHCP 租约
cat /var/lib/dhcp/dhclient.leases
# 或
journalctl -u dhclient

# 手动续约
dhclient -r eth0    # 释放
dhclient eth0       # 重新获取

# 查看 DHCP 服务器日志
journalctl -u isc-dhcp-server

# 常见问题:
#   - IP 地址池耗尽 → 地址冲突
#   - 多个 DHCP 服务器 → 地址混乱
#   - DHCP 租约过短 → 频繁中断
```

## 2. SSH 工作原理？密钥认证？安全加固？

**回答：**

### SSH 连接过程

```
1. TCP 三次握手（端口 22）
2. 协议版本协商（SSH-2.0）
3. 密钥交换（Key Exchange）
   - 使用 Diffie-Hellman / ECDH 生成会话密钥
   - 客户端验证服务器身份（Host Key）
4. 用户认证
   - 密码认证：加密传输密码
   - 密钥认证：客户端用私钥签名，服务器用公钥验证
5. 建立加密会话通道
```

### 密钥认证

```bash
# 生成密钥对
ssh-keygen -t ed25519 -C "user@example.com"
# 或 RSA（至少 4096 位）
ssh-keygen -t rsa -b 4096

# 将公钥复制到服务器
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@server

# 手动操作
cat ~/.ssh/id_ed25519.pub >> ~/.ssh/authorized_keys
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys

# 原理：
#   服务器发送随机数 → 客户端用私钥签名 → 服务器用公钥验证
#   私钥永远不传输
```

### SSH 安全加固

```bash
# /etc/ssh/sshd_config

# 禁用密码登录（只允许密钥）
PasswordAuthentication no

# 禁止 root 直接登录
PermitRootLogin no

# 修改默认端口
Port 2222

# 限制用户
AllowUsers deploy admin
AllowGroups sshusers

# 限制登录尝试
MaxAuthTries 3
LoginGraceTime 60

# 禁用空密码
PermitEmptyPasswords no

# 使用 Protocol 2
Protocol 2

# 空闲超时
ClientAliveInterval 300
ClientAliveCountMax 2

# 重启生效
systemctl restart sshd

# Fail2Ban 防暴力破解
apt install fail2ban
# /etc/fail2ban/jail.local
[sshd]
enabled = true
maxretry = 5
bantime = 3600
findtime = 600
```

### SSH 高级用法

```bash
# 端口转发（本地转发）
# 本机 8080 → 跳板机 → 内网 192.168.1.100:3306
ssh -L 8080:192.168.1.100:3306 user@jumphost

# 远程转发
ssh -R 9090:localhost:80 user@remote

# 动态代理（SOCKS5）
ssh -D 1080 user@server

# SSH 配置文件 ~/.ssh/config
Host jump
    HostName jump.example.com
    User admin
    Port 2222
    IdentityFile ~/.ssh/id_ed25519

Host internal
    HostName 192.168.1.100
    User deploy
    ProxyJump jump              # 通过跳板机连接

# SSH Agent
eval $(ssh-agent -s)
ssh-add ~/.ssh/id_ed25519
```

## 3. FTP、SFTP 和 SCP 的区别？

**回答：**

| 特性 | FTP | SFTP | SCP |
|------|-----|------|-----|
| 协议 | FTP (TCP 20/21) | SSH (TCP 22) | SSH (TCP 22) |
| 加密 | 明文（FTPS 可加密） | 全程 SSH 加密 | 全程 SSH 加密 |
| 防火墙 | 被动模式需额外端口 | 只需 22 端口 | 只需 22 端口 |
| 功能 | 完整文件管理 | 完整文件管理 | 只能复制文件 |
| 断点续传 | 支持 | 支持 | 不支持 |
| 性能 | 快 | 较慢（加密开销） | 较快 |

### 常用命令

```bash
# SCP
scp file.tar.gz user@server:/tmp/
scp -r dir/ user@server:/opt/        # 递归复制目录
scp -P 2222 file user@server:/tmp/   # 指定端口

# SFTP
sftp user@server
sftp> put localfile remotepath
sftp> get remotefile localpath
sftp> ls / cd / mkdir / rm

# rsync（增量同步，推荐替代 SCP）
rsync -avz --progress src/ user@server:/dest/
rsync -avz --delete src/ user@server:/dest/  # 同步删除
rsync -avz -e "ssh -p 2222" src/ user@server:/dest/
```

## 4. 邮件协议：SMTP、POP3、IMAP？

**回答：**

| 协议 | 端口 | 用途 | 加密端口 |
|------|------|------|---------|
| SMTP | 25 | 发送邮件 | 465(SSL) / 587(STARTTLS) |
| POP3 | 110 | 接收邮件（下载后删除） | 995 |
| IMAP | 143 | 接收邮件（服务端保留） | 993 |

```
邮件发送流程：
  发件人 MUA(Outlook)
    → SMTP → 发件 MTA(smtp.gmail.com)
    → SMTP → 收件 MTA(mx.example.com)
    → 存入邮箱 MDA
    → POP3/IMAP → 收件人 MUA

MUA: Mail User Agent（邮件客户端）
MTA: Mail Transfer Agent（邮件服务器，如 Postfix）
MDA: Mail Delivery Agent（邮件投递代理）

DevOps 场景：
  - 告警邮件发送（SMTP）
  - 需要配置 SPF/DKIM/DMARC 记录防止邮件被标记为垃圾
  - 端口 25 很多云厂商默认封锁
```

### DevOps 中的邮件配置

```bash
# 使用 telnet 测试 SMTP
telnet smtp.example.com 25
HELO test
MAIL FROM:<test@example.com>
RCPT TO:<admin@example.com>
DATA
Subject: Test
Test message
.
QUIT

# 命令行发送邮件
echo "Alert message" | mail -s "Alert" admin@example.com

# SPF 记录（DNS TXT 记录）
# 告诉接收方哪些 IP 可以代发邮件
# v=spf1 ip4:1.2.3.4 include:_spf.google.com -all
```

## 5. NTP 时间同步？为什么时间同步很重要？

**回答：**

### 时间同步的重要性

```
时间不同步会导致：
  - TLS 证书验证失败
  - Kerberos 认证失败（默认允许 5 分钟偏差）
  - 分布式系统数据不一致
  - 日志时间混乱，无法关联排查
  - 定时任务执行异常
  - 数据库主从同步异常
```

### NTP vs Chrony

```bash
# NTP（较老）
ntpdate pool.ntp.org           # 一次性同步
ntpq -p                        # 查看同步状态

# Chrony（推荐，现代替代）
# /etc/chrony.conf
server ntp.aliyun.com iburst
server cn.pool.ntp.org iburst

# 查看状态
chronyc tracking               # 查看同步详情
chronyc sources -v              # 查看时间源

# timedatectl（systemd）
timedatectl status             # 查看时间信息
timedatectl set-ntp true       # 启用 NTP 同步
timedatectl set-timezone Asia/Shanghai

# 重要配置
# 云环境建议使用云厂商提供的 NTP 服务器
# AWS: 169.254.169.123
# 阿里云: ntp.aliyun.com
```

## 6. WebSocket 协议？运维中的应用场景？

**回答：**

### WebSocket vs HTTP

```
HTTP：请求-响应模式，单向，短连接或 Keep-Alive
WebSocket：全双工，双向实时通信，持久连接

WebSocket 握手（基于 HTTP 升级）：
  Client → Server:
    GET /ws HTTP/1.1
    Upgrade: websocket
    Connection: Upgrade
    Sec-WebSocket-Key: x3JJHMbDL1EzLkh9GBhXDw==
    Sec-WebSocket-Version: 13

  Server → Client:
    HTTP/1.1 101 Switching Protocols
    Upgrade: websocket
    Connection: Upgrade
    Sec-WebSocket-Accept: HSmrc0sMlYUkAGmm5OPpG2HaGWk=

  → 之后双方通过帧（Frame）通信，不再是 HTTP
```

### Nginx 反向代理 WebSocket

```nginx
# WebSocket 代理配置
upstream ws_backend {
    server 127.0.0.1:8080;
}

server {
    listen 80;
    server_name ws.example.com;

    location /ws {
        proxy_pass http://ws_backend;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_read_timeout 3600s;    # 防止空闲断开
    }
}
```

### 应用场景

```
DevOps 中的 WebSocket 应用：
  - 实时日志流（Kibana, Grafana Live）
  - 终端 Web 化（terminal-in-browser, ttyd）
  - CI/CD 构建日志实时输出
  - 监控告警实时推送
  - ChatOps 机器人
```

## 7. gRPC 和 REST 的区别？

**回答：**

| 特性 | REST | gRPC |
|------|------|------|
| 协议 | HTTP/1.1 (JSON) | HTTP/2 (Protobuf) |
| 序列化 | JSON (文本) | Protocol Buffers (二进制) |
| 性能 | 较慢 | 快 10 倍+ |
| 流式传输 | 需要 SSE/WebSocket | 原生支持（双向流） |
| 代码生成 | 可选（OpenAPI） | 必须（.proto 定义） |
| 浏览器支持 | 完美 | 需要 gRPC-Web |
| 调试 | 直观（curl/Postman） | 需要专用工具 |
| 适用场景 | 外部 API、Web | 微服务内部通信 |

### gRPC 流模式

```
四种通信模式：
  1. Unary:            客户端 → 一个请求 → 服务端 → 一个响应
  2. Server Streaming: 客户端 → 一个请求 → 服务端 → 多个响应
  3. Client Streaming: 客户端 → 多个请求 → 服务端 → 一个响应
  4. Bidirectional:    客户端 ⇄ 双向流 ⇄ 服务端

DevOps 场景：
  - K8s 组件间通信（kubelet ↔ apiserver）
  - 微服务间高性能 RPC
  - etcd 的 Watch 机制（Server Streaming）
  - 日志/指标收集（Client Streaming）

调试工具:
  grpcurl -plaintext localhost:50051 list           # 列出服务
  grpcurl -plaintext localhost:50051 describe       # 描述服务
  grpcurl -d '{"name":"test"}' localhost:50051 pkg.Service/Method
```

## 8. 什么是 MQTT？IoT 和消息推送协议？

**回答：**

```
MQTT（Message Queuing Telemetry Transport）：
  - 轻量级发布/订阅消息协议
  - 基于 TCP，极低开销
  - 为带宽受限、不稳定网络设计

架构：
  Publisher ──→ Broker ──→ Subscriber
  (传感器)     (服务器)    (监控系统)

核心概念：
  - Topic（主题）：消息分类路径（支持通配符 + / #）
    sensors/temp/room1
    sensors/+/room1      # + 匹配一层
    sensors/#             # # 匹配多层
  
  - QoS（服务质量等级）
    QoS 0: 至多一次（发送即忘）
    QoS 1: 至少一次（确认，可能重复）
    QoS 2: 恰好一次（四次握手，最慢）

  - Retained Message: 保留最后一条消息给新订阅者
  - Last Will: 客户端异常断开时自动发布遗嘱消息

DevOps 场景：
  - IoT 设备监控
  - 轻量级微服务消息传递
  - Broker 软件: Mosquitto, EMQX, VerneMQ
```

## 9. HTTP/2 和 HTTP/3 的核心改进？

**回答：**

### HTTP/2

```
基于 SPDY 协议，2015 年标准化

核心改进：
  1. 二进制分帧：不再是文本协议，解析更高效
  2. 多路复用：一个 TCP 连接上并发多个请求/响应
     → 消除了 HTTP/1.1 的队头阻塞（应用层）
  3. Header 压缩（HPACK）
     → 静态/动态表 + Huffman 编码
  4. 服务器推送（Server Push）
     → 服务器主动推送资源（实际使用不多）
  5. 流优先级：可以设置请求的优先级

Nginx 开启 HTTP/2：
  listen 443 ssl http2;

注意：
  HTTP/2 的多路复用在单个 TCP 连接上
  → 如果 TCP 层丢包，所有流都会被阻塞（TCP 队头阻塞）
  → 这是 HTTP/3 要解决的问题
```

### HTTP/3

```
基于 QUIC 协议（UDP），2022 年标准化

核心改进：
  1. 基于 UDP（QUIC）
     → 解决 TCP 队头阻塞（流之间互不影响）
  2. 0-RTT / 1-RTT 连接建立
     → 传输层 + 加密层握手合并
  3. 内置 TLS 1.3
  4. 连接迁移
     → 用 Connection ID 标识连接
     → WiFi 切换到 4G 不会断开

        HTTP/1.1        HTTP/2          HTTP/3
    ┌──────────┐   ┌──────────┐   ┌──────────┐
    │  HTTP    │   │  HTTP/2  │   │  HTTP/3  │
    ├──────────┤   ├──────────┤   ├──────────┤
    │  TCP     │   │  TCP     │   │  QUIC    │
    ├──────────┤   ├──────────┤   ├──────────┤
    │  IP      │   │  IP      │   │  UDP     │
    └──────────┘   └──────────┘   ├──────────┤
                                  │  IP      │
                                  └──────────┘

Nginx 开启 HTTP/3：
  listen 443 quic reuseport;
  listen 443 ssl http2;
  add_header Alt-Svc 'h3=":443"; ma=86400';
```

## 10. SNMP 网络管理协议？

**回答：**

```
SNMP（Simple Network Management Protocol）
  用途：网络设备监控和管理

组件：
  Manager (NMS): 监控端（Zabbix, Nagios）
  Agent: 被监控设备上的代理程序
  MIB: 管理信息库（OID 树形结构定义监控项）

版本：
  SNMPv1: 明文社区字符串，安全性差
  SNMPv2c: 增加批量查询，仍然明文
  SNMPv3: 认证 + 加密，推荐使用

操作：
  GET:     查询指定 OID 的值
  GETNEXT: 查询下一个 OID
  GETBULK: 批量查询（v2c+）
  SET:     设置 OID 值
  TRAP:    设备主动上报事件
  INFORM:  确认型 TRAP

常用命令：
  snmpwalk -v2c -c public 192.168.1.1 .1.3.6.1.2.1
  snmpget -v2c -c public 192.168.1.1 sysDescr.0
  
DevOps 场景：
  - Zabbix 通过 SNMP 采集网络设备指标
  - 交换机/路由器/UPS 监控
  - 已逐渐被 Prometheus + Exporter 模式替代
```

## 11. 代理协议：SOCKS5 和 HTTP 代理？

**回答：**

### 代理类型

| 类型 | 层级 | 支持协议 | 特点 |
|------|------|---------|------|
| HTTP 代理 | L7 | HTTP/HTTPS | 可以看到 URL，可以缓存 |
| HTTPS 代理 (CONNECT) | L4-L7 | 任意 TCP (通过CONNECT隧道) | 不解密 |
| SOCKS4 | L4 | TCP | 不支持认证，不支持 UDP |
| SOCKS5 | L4 | TCP + UDP | 支持认证，支持 DNS 解析 |

### 使用场景

```bash
# 设置 HTTP 代理环境变量
export http_proxy=http://proxy:8080
export https_proxy=http://proxy:8080
export no_proxy=localhost,127.0.0.1,.internal.com

# curl 使用代理
curl -x http://proxy:8080 https://example.com
curl --socks5 socks5://proxy:1080 https://example.com

# Git 使用代理
git config --global http.proxy http://proxy:8080
git config --global https.proxy http://proxy:8080

# Docker 使用代理
# /etc/systemd/system/docker.service.d/http-proxy.conf
[Service]
Environment="HTTP_PROXY=http://proxy:8080"
Environment="HTTPS_PROXY=http://proxy:8080"
Environment="NO_PROXY=localhost,127.0.0.1"

# 正向代理 vs 反向代理
正向代理：代理客户端 → 客户端通过代理访问互联网
反向代理：代理服务端 → 客户端访问代理，代理转发到后端（Nginx）
```

## 12. LDAP 和统一认证？

**回答：**

```
LDAP（Lightweight Directory Access Protocol）
  端口：389（明文），636（LDAPS）
  用途：集中式用户身份认证和目录服务

常见实现：
  - OpenLDAP（开源）
  - Microsoft Active Directory
  - FreeIPA

LDAP 目录结构（树形）：
  dc=example,dc=com                    # 域
  ├── ou=people                         # 组织单元(用户)
  │   ├── uid=admin                     # 用户
  │   └── uid=deploy
  └── ou=groups                         # 组织单元(组)
      ├── cn=devops
      └── cn=developers

常用属性：
  dn:  Distinguished Name（唯一标识）
  cn:  Common Name（通用名称）
  uid: User ID
  ou:  Organizational Unit
  dc:  Domain Component

DevOps 中的 LDAP：
  - SSH 登录统一认证（sssd / nslcd）
  - Jenkins / GitLab / Harbor 对接 LDAP 登录
  - VPN 认证
  - 堡垒机认证
  - Kubernetes OIDC → 对接企业 LDAP/AD
  
  # ldapsearch 查询示例
  ldapsearch -x -H ldap://ldap.example.com \
    -b "dc=example,dc=com" \
    -D "cn=admin,dc=example,dc=com" \
    -W "(uid=deploy)"
```
