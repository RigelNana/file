## 1. 常见的网络攻击类型？

**回答：**

### 攻击分类总览

| 层级 | 攻击类型 | 描述 |
|------|---------|------|
| L2 | ARP 欺骗 | 伪造 ARP 回复，劫持流量 |
| L3 | IP 欺骗 | 伪造源 IP 地址 |
| L3/L4 | DDoS | 海量流量淹没目标 |
| L4 | SYN Flood | 耗尽目标半连接队列 |
| L4 | 端口扫描 | 探测开放端口和服务 |
| L7 | SQL 注入 | 利用输入注入 SQL |
| L7 | XSS | 注入恶意脚本到网页 |
| L7 | CSRF | 利用用户身份发起请求 |
| L7 | CC 攻击 | 大量 HTTP 请求耗尽资源 |
| 中间人 | MITM | 拦截和篡改通信 |
| DNS | DNS 劫持/投毒 | 篡改 DNS 解析结果 |

## 2. DDoS 攻击和防御？

**回答：**

### DDoS 类型

```
1. 流量型（Volumetric）：
   - UDP Flood：大量 UDP 包淹没带宽
   - DNS 放大：利用开放 DNS 递归解析器放大流量
   - NTP 放大：利用 monlist 命令放大流量
   - 放大倍数：DNS ~50倍, NTP ~500倍, Memcached ~50000倍

2. 协议型（Protocol）：
   - SYN Flood：耗尽半连接队列
   - ACK Flood：消耗状态表资源
   - ICMP Flood：大量 ping 包

3. 应用层（Application Layer）：
   - HTTP Flood / CC 攻击：大量合法 HTTP 请求
   - Slowloris：保持大量慢速 HTTP 连接
   - DNS 查询攻击：大量 DNS 查询耗尽 DNS 服务器资源
```

### 防御方案

```
基础设施层：
  - 云厂商 DDoS 防护（阿里云DDoS高防、AWS Shield）
  - CDN 分散流量（Cloudflare、Akamai）
  - 运营商黑洞路由（极端情况）

网络层：
  - SYN Cookie（无状态处理 SYN）
    sysctl -w net.ipv4.tcp_syncookies=1
  - 限制 ICMP
    iptables -A INPUT -p icmp --icmp-type echo-request -m limit --limit 1/s -j ACCEPT
  - 关闭不必要的服务和端口
  - 启用 RPF（反向路径过滤）防止 IP 欺骗
    sysctl -w net.ipv4.conf.all.rp_filter=1

应用层：
  - 速率限制（Nginx limit_req）
  - WAF（Web 应用防火墙）
  - 验证码、人机验证
  - IP 黑白名单
```

## 3. SYN Flood 攻击原理和防御？

**回答：**

```
正常三次握手：
  Client → SYN → Server     (Server 分配资源，进入 SYN_RECV)
  Client ← SYN+ACK ← Server
  Client → ACK → Server     (完成握手，进入 ESTABLISHED)

SYN Flood 攻击：
  Attacker → SYN (伪造源IP) → Server   (分配资源)
  Attacker → SYN (伪造源IP) → Server   (继续分配)
  Attacker → SYN (伪造源IP) → Server   (继续分配)
  ...
  Server 的半连接队列（SYN Queue）被填满
  → 无法接受新的正常连接

防御：

  # 1. SYN Cookie（最有效）
  sysctl -w net.ipv4.tcp_syncookies=1
  # 不在半连接队列存储状态，用加密Cookie验证

  # 2. 增大半连接队列
  sysctl -w net.ipv4.tcp_max_syn_backlog=65535

  # 3. 减少 SYN+ACK 重试次数
  sysctl -w net.ipv4.tcp_synack_retries=2   # 默认5

  # 4. iptables 限制 SYN 速率
  iptables -A INPUT -p tcp --syn -m limit --limit 100/s --limit-burst 200 -j ACCEPT
  iptables -A INPUT -p tcp --syn -j DROP

  # 5. 使用 SYN Proxy（防火墙代理握手）
  # 硬件防火墙或云DDoS防护的常见做法
```

## 4. ARP 欺骗原理和防御？

**回答：**

```
ARP 工作原理：
  主机 A 要和主机 B (192.168.1.100) 通信
  → 广播 ARP Request: "谁是 192.168.1.100?"
  → 主机 B 回复: "我是, MAC=aa:bb:cc:dd:ee:ff"
  → A 缓存 ARP 映射

ARP 欺骗：
  攻击者不断发送伪造 ARP Reply:
  "192.168.1.1(网关) 的 MAC 是 xx:xx:xx:xx:xx:xx(攻击者MAC)"
  → 受害者的流量都发给攻击者 → 中间人攻击

检测：
  arp -a                     # 查看 ARP 表
  # 如果发现网关 MAC 异常变化 → 可能被欺骗
  arping -D 192.168.1.1      # 检测 IP 冲突

防御：
  # 1. 静态 ARP 绑定
  arp -s 192.168.1.1 aa:bb:cc:dd:ee:ff
  
  # 2. 交换机 Dynamic ARP Inspection (DAI)
  # 在交换机上启用，配合 DHCP Snooping
  
  # 3. 使用 arpwatch 监控 ARP 变化
  apt install arpwatch
  arpwatch -i eth0
  
  # 4. 加密通信（即使 ARP 欺骗也无法解密）
  # → HTTPS, SSH, VPN
```

## 5. iptables/firewalld 防火墙最佳实践？

**回答：**

### 防火墙配置原则

```
1. 默认拒绝（Default Deny）：
   → 只开放必要的端口

2. 最小权限：
   → 能限制源 IP 的就限制
   → 能限制协议的就限制

3. 先规划再实施：
   → 列出需要开放的端口和源
   → 避免锁死自己（特别是 SSH）
```

### 服务器防火墙模板

```bash
#!/bin/bash
# 清空规则
iptables -F
iptables -X

# 默认策略
iptables -P INPUT DROP
iptables -P FORWARD DROP
iptables -P OUTPUT ACCEPT

# 允许已建立的连接
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# 允许回环
iptables -A INPUT -i lo -j ACCEPT

# 允许 ICMP (ping)
iptables -A INPUT -p icmp --icmp-type echo-request -m limit --limit 5/s -j ACCEPT

# 允许 SSH (限制源IP)
iptables -A INPUT -p tcp --dport 22 -s 10.0.0.0/8 -j ACCEPT

# 允许 HTTP/HTTPS
iptables -A INPUT -p tcp --dport 80 -j ACCEPT
iptables -A INPUT -p tcp --dport 443 -j ACCEPT

# 允许监控 (Prometheus Node Exporter)
iptables -A INPUT -p tcp --dport 9100 -s 10.0.0.0/8 -j ACCEPT

# 防止 SYN Flood
iptables -A INPUT -p tcp --syn -m limit --limit 50/s --limit-burst 100 -j ACCEPT
iptables -A INPUT -p tcp --syn -j DROP

# 记录被拒绝的包（调试用）
iptables -A INPUT -j LOG --log-prefix "iptables-drop: " --log-level 4

# 保存
iptables-save > /etc/iptables/rules.v4
```

### firewalld（CentOS/RHEL）

```bash
# 查看默认 zone
firewall-cmd --get-default-zone

# 查看所有规则
firewall-cmd --list-all

# 添加服务
firewall-cmd --permanent --add-service=http
firewall-cmd --permanent --add-service=https

# 添加端口
firewall-cmd --permanent --add-port=8080/tcp

# 富规则（限制源IP）
firewall-cmd --permanent --add-rich-rule='
  rule family="ipv4"
  source address="10.0.0.0/8"
  port port="22" protocol="tcp"
  accept'

# 重新加载
firewall-cmd --reload
```

## 6. CDN 的工作原理？

**回答：**

```
CDN (Content Delivery Network) = 内容分发网络

原理：
  将内容缓存到全球各地的边缘节点
  用户访问时 DNS 解析到最近的节点
  → 减少延迟，加速访问

工作流程：
  1. 用户请求 www.example.com
  2. DNS 解析 → CDN 的智能 DNS
  3. 智能 DNS 返回离用户最近的边缘节点 IP
  4. 用户请求到边缘节点
  5. 边缘节点有缓存 → 直接返回（Cache Hit）
  6. 边缘节点无缓存 → 回源到源站获取（Cache Miss）

  用户 ──→ CDN 边缘节点(北京) ──→ 源站(深圳)
                ↑ 缓存命中直接返回
                
  用户(上海) ──→ CDN 边缘节点(上海)
  用户(北京) ──→ CDN 边缘节点(北京)
  用户(广州) ──→ CDN 边缘节点(广州)

CDN 的功能：
  - 静态资源加速（图片、CSS、JS、视频）
  - 动态加速（DSA, 优化回源路径）
  - DDoS 防护（分散流量）
  - WAF（Web 应用防火墙）
  - HTTPS 卸载（边缘节点终止 TLS）
  - 压缩（gzip/brotli）

缓存控制：
  CDN 依赖 HTTP 缓存头：
    Cache-Control: max-age=86400
    CDN 缓存 TTL 配置
    缓存刷新/预热 API

常用 CDN：
  Cloudflare, AWS CloudFront, 阿里云 CDN, 腾讯云 CDN, Fastly, Akamai
```

## 7. WAF（Web 应用防火墙）？

**回答：**

```
WAF = Web Application Firewall
工作在 HTTP/HTTPS 层（L7），防护 Web 应用攻击

防护内容：
  - SQL 注入
  - XSS（跨站脚本）
  - CSRF（跨站请求伪造）
  - 文件上传漏洞
  - 路径遍历
  - 命令注入
  - CC 攻击（HTTP Flood）

部署模式：
  1. 反向代理模式：WAF 在前，Web 服务器在后
     用户 → WAF → Web Server
  
  2. 旁路检测模式：流量镜像到 WAF 分析
     用户 → Web Server
              ↘ 镜像 → WAF（检测+告警）

  3. 云 WAF：作为 CDN 的一部分
     用户 → 云WAF(CDN) → 源站

常见 WAF：
  - 开源：ModSecurity (Nginx/Apache 模块)
  - 云上：AWS WAF, 阿里云 WAF, Cloudflare WAF
  - 商业：Imperva, F5

ModSecurity + Nginx 示例：
  # 加载 ModSecurity 模块
  load_module modules/ngx_http_modsecurity_module.so;
  
  server {
      modsecurity on;
      modsecurity_rules_file /etc/nginx/modsecurity/main.conf;
  }

  # OWASP CRS (Core Rule Set) 提供通用规则集
```

## 8. 零信任网络架构？

**回答：**

```
传统安全模型：
  城堡+护城河：内网可信，外网不可信
  → 一旦攻击者进入内网，可以横向移动

零信任（Zero Trust）：
  核心原则："永不信任，始终验证"
  → 不管在内网还是外网，每次访问都需要验证

零信任三大原则：
  1. 验证每个用户和设备
  2. 最小权限访问
  3. 假设被入侵，持续监控

关键技术：
  ├─ 身份认证
  │   ├─ 多因素认证 (MFA)
  │   ├─ SSO (SAML/OIDC)
  │   └─ 设备证书
  ├─ 微分段
  │   ├─ 网络策略 (K8s NetworkPolicy)
  │   ├─ 服务网格 (Istio mTLS)
  │   └─ 最小权限防火墙规则
  ├─ 持续验证
  │   ├─ 设备健康检查
  │   ├─ 行为分析
  │   └─ 风险评分
  └─ 加密一切
      ├─ mTLS (双向 TLS)
      ├─ 传输加密
      └─ 存储加密

DevOps 落地：
  - K8s: NetworkPolicy + Istio mTLS + OPA/Gatekeeper
  - 云: IAM 最小权限 + VPC 安全组 + PrivateLink
  - 接入: BeyondCorp 模型 (Google), Tailscale
  - CI/CD: OIDC 认证（不使用长期凭据）
```

## 9. 端口扫描和入侵检测？

**回答：**

### 端口扫描（安全审计视角）

```bash
# nmap 常用命令（仅用于授权测试）
nmap -sT 192.168.1.100            # TCP Connect 扫描
nmap -sS 192.168.1.100            # TCP SYN 扫描（半开扫描）
nmap -sU 192.168.1.100            # UDP 扫描
nmap -sV 192.168.1.100            # 服务版本探测
nmap -O 192.168.1.100             # 操作系统探测
nmap -p 1-65535 192.168.1.100     # 全端口扫描
nmap -sC -sV 192.168.1.100       # 默认脚本+版本探测

# 扫描结果状态
#   open:     端口开放
#   closed:   端口关闭（返回 RST）
#   filtered: 被防火墙过滤（无响应或ICMP不可达）
```

### 入侵检测系统（IDS/IPS）

```
IDS (Intrusion Detection System): 检测并告警
IPS (Intrusion Prevention System): 检测并阻断

类型：
  NIDS (网络型): 监控网络流量
    → Suricata, Snort, Zeek (Bro)
  
  HIDS (主机型): 监控主机行为
    → OSSEC, Wazuh, Falco

Suricata 示例：
  # /etc/suricata/suricata.yaml
  # 使用 ET Open 规则集
  
  # 规则示例：检测 SSH 暴力破解
  alert ssh any any -> $HOME_NET 22 (msg:"SSH brute force"; \
    flow:to_server; threshold: type both, track by_src, count 5, seconds 60; \
    sid:1000001; rev:1;)

Falco（K8s 运行时安全）：
  - 检测容器内异常行为
  - 规则示例：容器内执行 shell
  - 容器内写入 /etc/ 目录
  - 容器内安装软件包
```

### 安全审计工具

```bash
# lynis（系统安全审计）
lynis audit system

# CIS Benchmark 检查
# 使用 OpenSCAP 或 InSpec 对照 CIS 基准检查

# 检查开放端口
ss -tlnp                      # 确认只开了必要端口
lsof -i -P -n | grep LISTEN   # 查看监听进程
```

## 10. SSL/TLS 证书管理和安全？

**回答：**

### 证书生命周期

```
1. 生成密钥对（私钥+CSR）
2. 提交 CSR 给 CA
3. CA 验证身份后签发证书
4. 部署证书到服务器
5. 监控证书有效期
6. 到期前续期

Let's Encrypt 自动化流程：
  certbot certonly --webroot -w /var/www/html -d example.com
  # 自动续期 cron
  0 0 * * * certbot renew --quiet --post-hook "systemctl reload nginx"
```

### 证书安全最佳实践

```bash
# 1. 检查证书信息
openssl x509 -in cert.pem -text -noout

# 2. 检查证书过期时间
openssl x509 -in cert.pem -enddate -noout
echo | openssl s_client -connect example.com:443 2>/dev/null | openssl x509 -noout -dates

# 3. 检查证书链完整性
openssl verify -CAfile ca-bundle.crt cert.pem

# 4. 测试 SSL/TLS 配置
# 使用 testssl.sh
./testssl.sh https://example.com

# 5. Nginx SSL 推荐配置
ssl_protocols TLSv1.2 TLSv1.3;
ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;
ssl_prefer_server_ciphers on;
ssl_session_cache shared:SSL:10m;
ssl_session_timeout 10m;
ssl_stapling on;
ssl_stapling_verify on;
add_header Strict-Transport-Security "max-age=63072000; includeSubDomains" always;

# 6. 私钥安全
#   - 私钥权限 600，属主 root
#   - 不要把私钥提交到代码仓库
#   - 使用 Vault / AWS KMS 管理密钥
#   - 定期轮转证书和密钥
```

### 证书监控告警

```bash
# Prometheus blackbox_exporter 监控证书
# prometheus.yml
- job_name: 'ssl-check'
  metrics_path: /probe
  params:
    module: [http_2xx]
  static_configs:
    - targets:
      - https://example.com
  relabel_configs:
    - source_labels: [__address__]
      target_label: __param_target
    - target_label: __address__
      replacement: blackbox-exporter:9115

# 告警规则
# 证书 30 天内过期
- alert: SSLCertExpiringSoon
  expr: probe_ssl_earliest_cert_expiry - time() < 86400 * 30
  for: 1h
  labels:
    severity: warning
  annotations:
    summary: "SSL证书即将过期: {{ $labels.instance }}"
```

## 11. 常用安全加固清单？

**回答：**

```
系统层：
  □ 禁用 root SSH 登录
  □ 使用密钥认证，禁用密码登录
  □ 修改 SSH 默认端口
  □ 安装 fail2ban
  □ 定期更新系统补丁
  □ 最小化安装（不装不需要的包）
  □ 关闭不必要的服务
  □ 配置 NTP 时间同步
  □ 配置日志收集（rsyslog → 集中日志）

网络层：
  □ 配置防火墙（默认拒绝入站）
  □ 限制管理端口的源 IP
  □ 启用 SYN Cookie
  □ 禁止 IP 转发（非路由器）
  □ 启用反向路径过滤（rp_filter）
  □ 关闭 ICMP 重定向

应用层：
  □ HTTPS 强制（HSTS）
  □ 证书自动续期和监控
  □ 使用 WAF
  □ 配置 CORS
  □ 安全响应头（X-Frame-Options, CSP 等）
  □ 依赖漏洞扫描（Trivy, Snyk）

容器/K8s：
  □ 镜像漏洞扫描
  □ 非 root 运行容器
  □ 只读文件系统
  □ NetworkPolicy 网络隔离
  □ Pod Security Standards
  □ Secret 加密存储
  □ RBAC 最小权限
  □ 审计日志开启

持续安全：
  □ 定期渗透测试
  □ 定期安全审计（lynis）
  □ 安全事件响应预案
  □ 安全培训
```
