## 1. OSI 七层模型和 TCP/IP 四层模型分别是什么？各层的职责？

**回答：**

### OSI 七层 vs TCP/IP 四层

```
OSI 七层模型          TCP/IP 四层模型       典型协议/设备
┌──────────────┐    ┌──────────────┐
│   应用层      │    │              │    HTTP, HTTPS, DNS, FTP, SSH, SMTP
├──────────────┤    │   应用层      │    SNMP, Telnet, MQTT, gRPC
│   表示层      │    │              │    SSL/TLS, JPEG, ASCII, XML, JSON
├──────────────┤    │              │    RPC, NetBIOS, SOCKS
│   会话层      │    ├──────────────┤
├──────────────┤    │   传输层      │    TCP, UDP, SCTP, QUIC
│   传输层      │    ├──────────────┤    端口号 (0-65535)
├──────────────┤    │   网络层      │    IP, ICMP, ARP, OSPF, BGP
│   网络层      │    │  (网际层)     │    路由器
├──────────────┤    ├──────────────┤
│  数据链路层   │    │  网络接口层   │    Ethernet, Wi-Fi, PPP
├──────────────┤    │  (链路层)     │    交换机, 网桥
│   物理层      │    │              │    网线, 光纤, 集线器
└──────────────┘    └──────────────┘
```

### 各层核心职责

| 层 | 核心职责 | PDU（协议数据单元） | 关键概念 |
|----|---------|-------------------|---------|
| 应用层 | 为用户程序提供网络服务接口 | 消息(Message) | URL, 请求/响应 |
| 表示层 | 数据格式转换、加密、压缩 | — | 编码, 序列化 |
| 会话层 | 建立/管理/终止会话 | — | 同步点, 令牌管理 |
| 传输层 | 端到端可靠传输、流量/拥塞控制 | 段(Segment)/数据报(Datagram) | 端口号, 连接 |
| 网络层 | 寻址、路由选择、分片 | 包(Packet) | IP地址, 路由表 |
| 数据链路层 | 帧同步、差错检测、MAC寻址 | 帧(Frame) | MAC地址, VLAN |
| 物理层 | 比特流传输 | 比特(Bit) | 带宽, 编码方式 |

**面试要点：** 实际工作中更常用 TCP/IP 四层模型，OSI 七层主要用于教学和标准化参考。

## 2. 数据在各层的封装和解封装过程？

**回答：**

### 封装过程（发送方，自上而下）

```
应用层：      [HTTP 请求数据]
                 ↓ 加 TCP 头部
传输层：      [TCP头 | HTTP 请求数据]
                 ↓ 加 IP 头部
网络层：      [IP头 | TCP头 | HTTP 请求数据]
                 ↓ 加帧头和帧尾
数据链路层：  [帧头 | IP头 | TCP头 | HTTP 请求数据 | FCS]
                 ↓ 转为比特流
物理层：       01010110 11001010 ...
```

### 各层头部关键字段

| 层 | 头部关键字段 | 大小 |
|----|------------|------|
| TCP | 源端口, 目的端口, 序列号, 确认号, 标志位, 窗口大小 | 20-60 字节 |
| IP | 版本, 头部长度, TTL, 协议类型, 源IP, 目的IP | 20-60 字节 |
| Ethernet | 目的MAC, 源MAC, 类型/长度, FCS(帧校验) | 18 字节 |

### 解封装过程（接收方，自下而上）

```
物理层收到比特流 → 还原为帧
  → 数据链路层：校验 FCS，去掉帧头/帧尾，根据类型字段(0x0800=IP)上交
    → 网络层：检查目的 IP，去掉 IP 头，根据协议字段(6=TCP)上交
      → 传输层：根据目的端口(80)找到对应进程，去掉 TCP 头
        → 应用层：HTTP 服务器处理请求数据
```

## 3. 各层的典型设备和工作原理？

**回答：**

| 层 | 设备 | 工作原理 |
|----|------|---------|
| 物理层 | 集线器(Hub)、中继器(Repeater) | 只负责信号放大和转发，不理解数据内容，所有端口广播 |
| 数据链路层 | 交换机(Switch)、网桥(Bridge) | 根据 MAC 地址表转发帧，隔离冲突域 |
| 网络层 | 路由器(Router)、三层交换机 | 根据路由表转发 IP 包，隔离广播域 |
| 应用层 | 防火墙(L7)、负载均衡器、代理服务器 | 理解应用层协议，进行深度包检测/内容路由 |

### 交换机工作流程

```
1. 收到帧 → 学习源 MAC 地址和对应端口，记入 MAC 地址表
2. 查找目的 MAC 地址：
   - 找到 → 从对应端口转发（单播）
   - 未找到 → 除入端口外所有端口泛洪（Flooding）
   - 目的是广播 MAC（FF:FF:FF:FF:FF:FF）→ 泛洪

查看 MAC 地址表（Linux bridge）：
$ brctl showmacs br0
$ bridge fdb show
```

### 路由器工作流程

```
1. 收到 IP 包 → 检查目的 IP 地址
2. 查路由表：
   - 直连路由 → 直接转发
   - 静态/动态路由 → 找到下一跳，从对应接口转发
   - 默认路由 → 没有更精确匹配时使用
   - 无匹配 → 丢弃，发送 ICMP Destination Unreachable
3. TTL 减 1（为 0 则丢弃，发送 ICMP Time Exceeded）
4. 重新计算校验和，更新帧头（下一跳 MAC），发出

$ ip route show      # 查看路由表
$ route -n           # 传统方式
```

## 4. 什么是 MTU？什么是 MSS？它们的关系？

**回答：**

### 定义

| 概念 | 全称 | 说明 | 典型值 |
|------|------|------|--------|
| MTU | Maximum Transmission Unit | 数据链路层一帧能承载的最大数据量 | Ethernet: 1500 字节 |
| MSS | Maximum Segment Size | TCP 段能承载的最大数据量 | 1460 字节 |

### 关系

```
以太网帧：
┌────────┬────────┬─────────────────────────────┬─────┐
│  帧头   │  IP头  │          TCP段               │ FCS │
│ 14 字节 │ 20字节 │                              │ 4B  │
│         │        │ ┌───────┬────────────────┐  │     │
│         │        │ │TCP头  │    数据(MSS)    │  │     │
│         │        │ │20字节 │   ≤1460字节     │  │     │
│         │        │ └───────┴────────────────┘  │     │
└────────┴────────┴─────────────────────────────┴─────┘
          ←──────── MTU = 1500 ─────────────────→

MSS = MTU - IP头(20) - TCP头(20) = 1460 字节
```

### IP 分片

```bash
# 当包大于路径 MTU 时，会被分片
# 查看网卡 MTU
ip link show eth0 | grep mtu

# 修改 MTU
ip link set eth0 mtu 9000              # Jumbo Frame

# 探测路径 MTU（PMTUD）
ping -M do -s 1472 target              # 设置 DF 位，1472 + 28(IP+ICMP头) = 1500
# 如果返回 "Frag needed"，说明路径 MTU 小于 1500

# traceroute 探测 PMTU
tracepath target
```

**DevOps 场景：** VPN/隧道会增加开销（如 VXLAN 增加 50 字节），需要调低 MTU 避免分片。容器网络（Flannel/Calico）中 MTU 配置不当是常见的网络问题来源。

## 5. 什么是 ARP 协议？工作原理？

**回答：**

ARP（Address Resolution Protocol）将 IP 地址解析为 MAC 地址，工作在网络层和数据链路层之间。

### 工作流程

```
主机 A (192.168.1.10) 要发数据给 主机 B (192.168.1.20)

1. A 查本地 ARP 缓存 → 未找到 B 的 MAC
2. A 广播 ARP Request（目标 MAC = FF:FF:FF:FF:FF:FF）
   "谁是 192.168.1.20？请告诉 192.168.1.10"
3. 同一广播域内所有主机收到，只有 B 回复
4. B 单播 ARP Reply 给 A
   "192.168.1.20 的 MAC 是 AA:BB:CC:DD:EE:FF"
5. A 将映射存入 ARP 缓存（有 TTL，通常几分钟）
6. A 用 B 的 MAC 地址封装帧并发送
```

### 常用命令

```bash
# 查看 ARP 缓存
arp -n
ip neigh show

# 删除 ARP 条目
arp -d 192.168.1.20
ip neigh del 192.168.1.20 dev eth0

# 静态 ARP 绑定（防 ARP 欺骗）
arp -s 192.168.1.1 AA:BB:CC:DD:EE:FF
ip neigh add 192.168.1.1 lladdr AA:BB:CC:DD:EE:FF dev eth0 nud permanent

# 抓 ARP 包
tcpdump -i eth0 arp
```

### 跨网段的 ARP

```
主机 A (192.168.1.10) → 路由器 (192.168.1.1/192.168.2.1) → 主机 B (192.168.2.20)

A 发现 B 不在同一子网 → A 解析网关（路由器）的 MAC → 将帧发给路由器
→ 路由器收到后解封装，查路由表 → 在另一接口解析 B 的 MAC → 封装新帧发给 B

帧的 MAC 地址逐跳变化，但 IP 包的源/目的 IP 不变
```

## 6. 什么是 ICMP 协议？常见的 ICMP 消息类型？

**回答：**

ICMP（Internet Control Message Protocol）用于网络层的差错报告和网络诊断，封装在 IP 包中。

### 常见 ICMP 消息类型

| 类型 | 代码 | 名称 | 用途 |
|------|------|------|------|
| 0 | 0 | Echo Reply | ping 回复 |
| 3 | 0 | Destination Unreachable - Network | 网络不可达 |
| 3 | 1 | Destination Unreachable - Host | 主机不可达 |
| 3 | 3 | Destination Unreachable - Port | 端口不可达 |
| 3 | 4 | Fragmentation Needed | 需要分片但设了 DF |
| 3 | 13 | Administratively Prohibited | 被防火墙拒绝 |
| 5 | 0 | Redirect | 重定向（有更优路由） |
| 8 | 0 | Echo Request | ping 请求 |
| 11 | 0 | Time Exceeded | TTL 过期（traceroute 利用此原理） |

### 基于 ICMP 的工具

```bash
# ping：发送 Echo Request，接收 Echo Reply
ping -c 4 192.168.1.1
# 输出：64 bytes from 192.168.1.1: icmp_seq=1 ttl=64 time=0.5 ms

# traceroute：利用 TTL 递增 + ICMP Time Exceeded
# TTL=1 → 第一跳路由器返回 ICMP TTL 过期
# TTL=2 → 第二跳路由器返回 ICMP TTL 过期
# ...直到到达目的地
traceroute 8.8.8.8
```

### DevOps 注意事项

```bash
# 很多云环境和防火墙会禁用 ICMP
# ping 不通不代表服务不可用！

# 检查防火墙是否允许 ICMP
iptables -A INPUT -p icmp --icmp-type echo-request -j ACCEPT
iptables -A INPUT -p icmp --icmp-type echo-reply -j ACCEPT

# 建议：至少允许以下 ICMP 类型
# - Echo Request/Reply（ping）
# - Destination Unreachable（路径 MTU 发现需要）
# - Time Exceeded（traceroute 需要）
```

## 7. 什么是广播域和冲突域？

**回答：**

### 定义

| 概念 | 定义 | 范围 |
|------|------|------|
| 冲突域 | 同一时间只能有一个设备发送数据的区域 | 共享同一信道的所有设备 |
| 广播域 | 广播帧能到达的所有设备的范围 | 同一个 VLAN/子网 |

### 各设备对冲突域和广播域的影响

```
设备          分割冲突域？    分割广播域？
─────────────────────────────────────
集线器(Hub)       ✗              ✗
交换机(Switch)    ✓              ✗
路由器(Router)    ✓              ✓
VLAN              ✓              ✓
```

### 示例

```
Hub 连接 4 台主机：
  冲突域 = 1 个（全部共享）
  广播域 = 1 个

Switch 连接 4 台主机（同一 VLAN）：
  冲突域 = 4 个（每个端口是独立冲突域）
  广播域 = 1 个

Switch 连接 4 台主机（2 个 VLAN）：
  冲突域 = 4 个
  广播域 = 2 个

Router 连接 2 个子网：
  每个接口是独立的广播域
```

## 9. IP 包的 TTL 字段有什么作用？

**回答：**

TTL（Time To Live）限制 IP 包在网络中的生存时间/跳数，防止包在网络中无限循环。

### 工作机制

```
发送方设置 TTL（如 64）
  → 每经过一个路由器，TTL 减 1
    → TTL 变为 0 时，路由器丢弃该包
      → 发送 ICMP Time Exceeded 给源主机
```

### 各操作系统默认 TTL

| 操作系统 | 默认 TTL |
|---------|---------|
| Linux | 64 |
| Windows | 128 |
| macOS | 64 |
| Cisco IOS | 255 |

```bash
# 通过 TTL 判断对方操作系统（粗略）
$ ping -c 1 target
# TTL=64 → 大概率 Linux/macOS
# TTL=128 → 大概率 Windows
# TTL=56 → 可能 Linux，经过了 8 跳

# 设置出站 TTL
ping -t 10 target       # Windows
ping -m 10 target       # Linux（设置 TTL）
```

### traceroute 利用 TTL 的原理

```
发送 TTL=1 的包 → 第一跳路由器返回 ICMP Time Exceeded（得到第一跳 IP）
发送 TTL=2 的包 → 第二跳路由器返回 ICMP Time Exceeded
...
发送 TTL=N 的包 → 到达目的地，返回 ICMP Echo Reply / UDP Port Unreachable
```

## 10. TCP/IP 协议栈中，数据如何在主机间传输（完整流程）？

**回答：**

以主机 A（192.168.1.10）访问同网段主机 B（192.168.1.20）的 Web 服务（端口 80）为例：

### 发送方（主机 A）

```
1. 应用层
   浏览器构建 HTTP 请求: GET /index.html HTTP/1.1
   通过 Socket API 传给传输层

2. 传输层
   TCP 封装：
   ┌──────────────────────────────────────┐
   │ 源端口: 50000 │ 目的端口: 80        │
   │ 序列号: 1000   │ 确认号: ...         │
   │ 标志: PSH,ACK │ 窗口: 65535        │
   │ [HTTP请求数据]                       │
   └──────────────────────────────────────┘

3. 网络层
   IP 封装：
   ┌──────────────────────────────────────┐
   │ 版本: 4 │ TTL: 64 │ 协议: 6(TCP)    │
   │ 源IP: 192.168.1.10                   │
   │ 目的IP: 192.168.1.20                 │
   │ [TCP段]                              │
   └──────────────────────────────────────┘
   判断目的 IP 在同一子网 → 不需要经过网关

4. 数据链路层
   需要 B 的 MAC 地址 → 查 ARP 缓存或发 ARP 请求
   Ethernet 帧封装：
   ┌────────────────────────────────────────────┐
   │ 目的MAC: B的MAC │ 源MAC: A的MAC            │
   │ 类型: 0x0800 (IPv4)                         │
   │ [IP包]                                      │
   │ FCS(帧校验序列)                              │
   └────────────────────────────────────────────┘

5. 物理层
   将帧转换为电信号/光信号发送到网络介质上
```

### 接收方（主机 B）

```
物理层: 信号 → 比特流
  ↓
数据链路层: 校验 FCS → 检查目的 MAC = 自己 → 去帧头 → 上交网络层
  ↓
网络层: 检查目的 IP = 自己 → 去 IP 头 → 根据协议字段(6) 上交 TCP
  ↓
传输层: 根据目的端口(80) → 找到 Web 服务器进程 → 去 TCP 头 → 上交
  ↓
应用层: Web 服务器处理 HTTP 请求 → 生成响应
```

## 11. 什么是 Socket？它和 TCP/IP 的关系？

**回答：**

### 定义

Socket（套接字）是操作系统提供的网络编程抽象接口，位于应用层和传输层之间，是应用程序与 TCP/IP 协议栈交互的 API。

```
应用程序
   ↓ (Socket API: socket(), bind(), listen(), accept(), connect(), send(), recv())
Socket 抽象层
   ↓
传输层 (TCP/UDP)
   ↓
网络层 (IP)
   ↓
数据链路层/物理层
```

### Socket 五元组

一个 Socket 连接由以下五元组唯一标识：

```
(协议, 源IP, 源端口, 目的IP, 目的端口)
例如: (TCP, 192.168.1.10, 50000, 10.0.0.1, 80)
```

### TCP Socket 通信流程

```
服务器端                        客户端
socket()                       socket()
bind(port)
listen()
                               connect() ←→ [三次握手]
accept() → 新 socket
                  ←→ send()/recv() 数据交互 ←→
close()                        close()    [四次挥手]
```

### 常用查看命令

```bash
# 查看所有 Socket 连接
ss -tuanp

# 各状态连接数统计
ss -s

# 查看 ESTABLISHED 连接
ss -tnp state established

# 查看 TIME_WAIT 数量
ss -tan state time-wait | wc -l

# 查看某端口的连接
ss -tnp sport = :80
```

## 12. 什么是网络字节序？为什么需要字节序转换？

**回答：**

### 字节序

| 类型 | 英文 | 说明 | 示例（存储 0x12345678） |
|------|------|------|----------------------|
| 大端序 | Big-Endian | 高位字节在低地址 | 12 34 56 78 |
| 小端序 | Little-Endian | 低位字节在低地址 | 78 56 34 12 |

**网络字节序 = 大端序**（Big-Endian），这是 RFC 规定的标准。

```
为什么需要转换？
  x86/x64 CPU 使用小端序
  网络传输使用大端序
  → 发送前需要 主机字节序 → 网络字节序
  → 接收后需要 网络字节序 → 主机字节序
```

### 影响范围

```
需要转换的：IP 头部、TCP/UDP 头部中的多字节字段
  - IP 地址（4 字节）
  - 端口号（2 字节）
  - 序列号、确认号（4 字节）
  - 校验和等

不需要转换的：应用层数据（由应用协议自行定义）

抓包时注意：
  Wireshark 显示的已经是转换后的人类可读格式
  tcpdump -X 看到的原始数据是网络字节序
```

## 13. 常见的网络协议及其所在层次？（速查表）

### 应用层协议

| 协议 | 端口 | 传输层 | 用途 |
|------|------|--------|------|
| HTTP | 80 | TCP | 超文本传输 |
| HTTPS | 443 | TCP | 加密HTTP |
| SSH | 22 | TCP | 安全远程登录 |
| FTP | 20/21 | TCP | 文件传输 |
| SMTP | 25 | TCP | 邮件发送 |
| POP3 | 110 | TCP | 邮件接收 |
| IMAP | 143 | TCP | 邮件接收（高级） |
| DNS | 53 | TCP/UDP | 域名解析 |
| DHCP | 67/68 | UDP | 动态IP分配 |
| SNMP | 161/162 | UDP | 网络管理 |
| NTP | 123 | UDP | 时间同步 |
| TFTP | 69 | UDP | 简单文件传输 |
| MQTT | 1883/8883 | TCP | IoT消息 |
| gRPC | 自定义 | TCP(HTTP/2) | RPC框架 |

### 传输层协议

| 协议 | 特点 |
|------|------|
| TCP | 面向连接，可靠，字节流 |
| UDP | 无连接，不可靠，数据报 |
| SCTP | 多流，多宿，消息边界 |
| QUIC | 基于UDP，内置TLS，多路复用 |

### 网络层协议

| 协议 | 用途 |
|------|------|
| IPv4/IPv6 | 网络寻址与路由 |
| ICMP/ICMPv6 | 差错报告与诊断 |
| ARP | IP→MAC 地址解析 |
| RARP | MAC→IP（已弃用，被 DHCP 替代） |
| IGMP | 组播组管理 |
| OSPF | 链路状态路由协议 |
| BGP | 边界网关协议（互联网骨干路由） |
| IPSec | IP 层加密 |

### 数据链路层协议

| 协议 | 用途 |
|------|------|
| Ethernet (IEEE 802.3) | 有线局域网 |
| Wi-Fi (IEEE 802.11) | 无线局域网 |
| PPP | 点对点连接 |
| PPPoE | 以太网上的PPP（宽带拨号） |
| VXLAN | 虚拟可扩展局域网（覆盖网络） |
