## 1. 网络故障排查的整体思路？

**回答：**

### 自下而上排查法

```
从 OSI 模型底层开始逐层排查：

L1 物理层 → L2 数据链路层 → L3 网络层 → L4 传输层 → L7 应用层

Step 1: 物理层
  - 网线/光纤是否连接？
  - 网卡灯是否亮？
  - ip link show → 网卡状态 UP/DOWN
  - ethtool eth0 → Link detected: yes/no

Step 2: 数据链路层
  - ARP 是否正常？
  - ip neigh show → 查看 ARP 表
  - arping 192.168.1.1 → 测试二层连通性

Step 3: 网络层
  - IP 地址是否正确？ ip addr show
  - 路由是否正确？ ip route show
  - ping 网关 → 通则三层OK
  - ping 外网 IP (8.8.8.8) → 通则出口OK

Step 4: DNS
  - nslookup / dig 域名 → 能解析则 DNS OK
  - 不通改用 8.8.8.8 测试

Step 5: 传输层
  - telnet/nc 测试端口是否可达
  - ss -tlnp 查看服务是否监听
  - 防火墙规则是否放行？

Step 6: 应用层
  - curl -v 测试 HTTP 响应
  - 查看应用日志
  - 证书是否过期？
```

### 常用排查命令速查

```bash
ip link show            # 网卡状态
ip addr show            # IP 地址
ip route show           # 路由表
ip neigh show           # ARP 表
ping 8.8.8.8            # 三层连通性
traceroute 8.8.8.8      # 路径追踪
dig example.com         # DNS 解析
ss -tlnp                # 监听端口
curl -v https://url     # HTTP 测试
iptables -L -n -v       # 防火墙规则
```

## 2. ping、traceroute 和 mtr 的原理与使用？

**回答：**

### ping

```bash
# 原理：发送 ICMP Echo Request，等待 Echo Reply
# 测试：网络连通性、延迟、丢包

ping 8.8.8.8                    # 基本测试
ping -c 10 8.8.8.8              # 发 10 个包
ping -i 0.2 8.8.8.8             # 间隔 0.2 秒
ping -s 1472 -M do 8.8.8.8     # 测试 MTU（1472+28=1500）
ping -f 8.8.8.8                 # 洪泛 ping（需要 root）

# 输出分析
64 bytes from 8.8.8.8: icmp_seq=1 ttl=117 time=5.32 ms
#                                   ↑           ↑
#                                TTL剩余      往返时间
# 丢包率 > 1% = 网络质量问题
# TTL 值可以推断经过了多少跳
```

### traceroute

```bash
# 原理：逐步增加 TTL (1, 2, 3...)
# 每跳路由器 TTL 减 1，到 0 时返回 ICMP Time Exceeded
# → 从而发现每一跳的路由器 IP

traceroute 8.8.8.8              # UDP 探测（默认）
traceroute -I 8.8.8.8           # ICMP 探测
traceroute -T -p 443 8.8.8.8   # TCP 探测（穿透防火墙）

# 输出分析
 1  192.168.1.1 (192.168.1.1)  1.234 ms  0.987 ms  1.102 ms
 2  10.0.0.1 (10.0.0.1)  5.432 ms  5.654 ms  5.321 ms
 3  * * *                        ← 路由器不回复 ICMP（不一定是故障）
 4  8.8.8.8 (8.8.8.8)  10.123 ms  9.876 ms  10.234 ms

# 三个时间 = 三次探测的 RTT
# * = 超时未回复（可能被防火墙过滤）
```

### mtr（推荐）

```bash
# mtr = ping + traceroute 结合体
# 持续监控每一跳的延迟和丢包

mtr 8.8.8.8                    # 交互模式
mtr -r -c 100 8.8.8.8          # 报告模式（发100个包）
mtr -rw -c 100 8.8.8.8         # 宽格式报告（显示完整域名）

# 输出示例
HOST:                   Loss%   Snt   Last   Avg  Best  Wrst StDev
  1. gateway             0.0%   100    0.5   0.6   0.3   2.1   0.3
  2. isp-router          2.0%   100    5.2   5.8   4.1  15.3   2.1  ← 此处开始丢包
  3. backbone            2.0%   100   10.1  10.5   9.2  20.1   2.5
  4. 8.8.8.8             0.0%   100   12.3  12.8  11.1  18.4   1.6

分析要点：
  - 如果某一跳开始丢包，且后续跳也丢包 → 该跳是问题点
  - 如果某一跳丢包但后续正常 → 该路由器 ICMP 限速，不是故障
  - Loss% 关注终点，中间跳的丢包不一定是问题
```

## 3. tcpdump 抓包详解？

**回答：**

### 基本使用

```bash
# 监听指定网卡
tcpdump -i eth0

# 抓取特定主机
tcpdump host 192.168.1.100
tcpdump src 192.168.1.100             # 源 IP
tcpdump dst 192.168.1.100             # 目的 IP

# 抓取特定端口
tcpdump port 80
tcpdump portrange 8000-9000
tcpdump src port 443

# 抓取特定协议
tcpdump tcp
tcpdump udp
tcpdump icmp

# 组合条件
tcpdump -i eth0 'host 192.168.1.100 and port 80'
tcpdump -i eth0 'src 10.0.0.1 and (dst port 80 or dst port 443)'
tcpdump -i eth0 'tcp[tcpflags] & tcp-syn != 0'    # SYN 包
tcpdump -i eth0 'tcp[tcpflags] & (tcp-syn|tcp-ack) == tcp-syn'  # 只抓 SYN
```

### 输出与保存

```bash
# 详细输出
tcpdump -v    # 详细
tcpdump -vv   # 更详细
tcpdump -vvv  # 最详细

# 不解析域名和端口名（加速）
tcpdump -nn

# 显示 ASCII 内容
tcpdump -A

# 显示十六进制和 ASCII
tcpdump -XX

# 保存到文件（供 Wireshark 分析）
tcpdump -i eth0 -w capture.pcap
tcpdump -i eth0 -w capture.pcap -c 1000      # 抓 1000 个包
tcpdump -i eth0 -w capture.pcap -G 3600 -W 24  # 每小时一个文件，保留24个

# 读取文件
tcpdump -r capture.pcap
tcpdump -r capture.pcap 'port 80'
```

### 常用排查场景

```bash
# 排查 TCP 连接问题（观察三次握手）
tcpdump -i eth0 -nn 'tcp[tcpflags] & (tcp-syn|tcp-fin|tcp-rst) != 0' and host 10.0.0.1

# 排查 DNS 解析
tcpdump -i eth0 -nn port 53

# 抓 HTTP 请求（明文）
tcpdump -i eth0 -A -nn 'tcp port 80 and (((ip[2:2] - ((ip[0]&0xf)<<2)) - ((tcp[12]&0xf0)>>2)) != 0)'

# 排查 ICMP 不可达
tcpdump -i eth0 icmp

# 排查连接重置
tcpdump -i eth0 'tcp[tcpflags] & tcp-rst != 0'

# 限制包长度（只看包头）
tcpdump -i eth0 -s 96 -nn port 443
```

## 4. Wireshark 使用与过滤器？

**回答：**

### 抓包过滤器（Capture Filter，BPF 语法）

```
在抓包前设置，减少数据量：
  host 192.168.1.100
  port 80
  tcp and port 443
  not broadcast and not multicast
  src net 10.0.0.0/8
```

### 显示过滤器（Display Filter，Wireshark 语法）

```
在已抓取的包中过滤，语法不同于 BPF：

# IP 过滤
ip.addr == 192.168.1.100
ip.src == 10.0.0.1
ip.dst == 10.0.0.2
ip.addr == 10.0.0.0/8

# TCP 过滤
tcp.port == 80
tcp.dstport == 443
tcp.flags.syn == 1                 # SYN 包
tcp.flags.rst == 1                 # RST 包
tcp.analysis.retransmission        # TCP 重传
tcp.analysis.duplicate_ack         # 重复 ACK
tcp.analysis.zero_window           # 零窗口
tcp.analysis.window_full           # 窗口满

# HTTP 过滤
http.request.method == "GET"
http.response.code == 500
http.host contains "example.com"
http.request.uri contains "/api"

# DNS 过滤
dns.qry.name == "example.com"
dns.flags.rcode != 0               # DNS 错误响应

# TLS/SSL
tls.handshake.type == 1           # Client Hello
tls.handshake.extensions_server_name == "example.com"

# 组合过滤
ip.addr == 10.0.0.1 && tcp.port == 80
http || dns
!(arp || icmp)
```

### Wireshark 实用功能

```
1. Follow TCP Stream：右键包 → Follow → TCP Stream
   → 完整查看一个 TCP 连接的所有数据

2. Statistics → Conversations：查看所有会话统计

3. Statistics → IO Graphs：流量时间曲线

4. Analyze → Expert Information：自动标记异常
   → 重传、RST、零窗口等

5. tshark（命令行版 Wireshark）:
   tshark -i eth0 -f "port 80"
   tshark -r capture.pcap -Y "http.response.code == 500"
   tshark -r capture.pcap -q -z conv,tcp          # 会话统计
   tshark -r capture.pcap -q -z io,stat,1         # 每秒流量统计
```

## 5. ss 和 netstat 命令？

**回答：**

### ss（推荐，替代 netstat）

```bash
# 查看所有 TCP 连接
ss -t -a

# 查看监听端口
ss -tlnp
# -t TCP, -l 监听, -n 不解析, -p 显示进程

# 查看 UDP
ss -ulnp

# 查看所有连接（TCP + UDP + Unix Socket）
ss -a

# 按状态过滤
ss -t state established
ss -t state time-wait
ss -t state close-wait

# 按地址/端口过滤
ss -tn dst 10.0.0.1
ss -tn sport = :80
ss -tn dport = :443

# 输出示例
State    Recv-Q  Send-Q   Local Address:Port    Peer Address:Port   Process
ESTAB    0       0        192.168.1.10:22       10.0.0.1:54321      users:(("sshd",pid=1234))
LISTEN   0       128      0.0.0.0:80            0.0.0.0:*           users:(("nginx",pid=5678))

# Recv-Q / Send-Q 分析
#   LISTEN 状态: Recv-Q = 全连接队列当前数, Send-Q = 全连接队列最大值（backlog）
#   ESTABLISHED: Recv-Q = 应用未读数据, Send-Q = 已发送未确认数据
#   Recv-Q 持续增长 → 应用读取慢
#   Send-Q 持续增长 → 网络拥塞或对方接收慢

# 统计连接状态
ss -s
# 或
ss -tan | awk '{print $1}' | sort | uniq -c | sort -rn
```

### netstat（较老，部分系统仍用）

```bash
netstat -tlnp               # TCP 监听端口
netstat -anp                # 所有连接
netstat -s                  # 各协议统计
netstat -r                  # 路由表（等同 route -n）
```

## 8. 常见网络性能问题排查？

**回答：**

### 延迟高

```bash
# 1. 确定延迟在哪个环节
curl -w "DNS:%{time_namelookup} TCP:%{time_connect} TLS:%{time_appconnect} Total:%{time_total}\n" \
     -o /dev/null -s https://example.com

# 2. 确定哪一跳延迟高
mtr -rw -c 50 target_ip

# 3. 检查系统负载
top / vmstat 1 / iostat -x 1

# 4. 检查网卡
ethtool -S eth0 | grep -i error    # 网卡错误统计
ifconfig eth0 | grep -i error       # 收发错误

# 常见原因：
#   - 路由绕行（traceroute 确认）
#   - TCP 拥塞导致缓冲膨胀（bufferbloat）
#   - 应用层处理慢（不是网络问题）
```

### 丢包

```bash
# 1. ping 测丢包率
ping -c 100 -i 0.1 target_ip

# 2. 检查网卡丢包
ip -s link show eth0
# → RX dropped / TX dropped / errors / overruns

# 3. 检查内核网络统计
netstat -s | grep -i drop
netstat -s | grep -i error
cat /proc/net/snmp | grep -i udp    # UDP 丢包

# 4. 检查 iptables 丢弃
iptables -L -n -v | grep DROP

# 5. 检查连接跟踪表满
dmesg | grep "nf_conntrack: table full"
sysctl net.netfilter.nf_conntrack_count
sysctl net.netfilter.nf_conntrack_max

# 常见原因及解决：
#   - conntrack 表满 → 增大 nf_conntrack_max
#   - 网卡队列溢出 → 增大 Ring Buffer: ethtool -G eth0 rx 4096
#   - 内核收包能力不足 → 调整 net.core.netdev_max_backlog
#   - 应用不及时读取 → 增大 Socket 缓冲区
```

### 连接超时

```bash
# 1. 确认端口是否可达
nc -zv target_ip 443 -w 5

# 2. 确认是否被防火墙阻挡
# 在目标机器上抓包看是否收到 SYN
tcpdump -i eth0 -nn 'tcp[tcpflags] & tcp-syn != 0 and port 443'

# 3. 确认 SYN 队列/全连接队列是否溢出
ss -tlnp | grep :443    # 看 Send-Q（backlog 大小）
netstat -s | grep -i overflow
netstat -s | grep -i "SYN"

# 4. 确认 TIME_WAIT 过多
ss -tan state time-wait | wc -l
# 优化：
sysctl -w net.ipv4.tcp_tw_reuse=1
```

## 9. strace 和 ltrace 排查网络问题？

**回答：**

```bash
# strace：追踪系统调用（含网络相关的 socket, connect, read, write 等）

# 追踪进程的网络相关系统调用
strace -e network -p <pid>

# 追踪特定系统调用
strace -e trace=connect,sendto,recvfrom -p <pid>

# 追踪新进程启动时的网络行为
strace -f -e network curl https://example.com

# 常见网络相关系统调用：
#   socket()    → 创建套接字
#   bind()      → 绑定地址
#   listen()    → 开始监听
#   accept()    → 接受连接
#   connect()   → 发起连接（看这里的返回值和延迟）
#   send/recv   → 发送/接收数据
#   close()     → 关闭连接

# 排查示例：应用连接超时
strace -tt -e connect -p <pid>
# 输出：
# 14:30:01.123456 connect(5, {sa_family=AF_INET, sin_port=htons(3306), sin_addr=inet_addr("10.0.0.1")}, 16) = -1 ETIMEDOUT
#                                                                          ↑ 目标地址              ↑ 超时

# 排查 DNS 解析慢
strace -tt -e sendto,recvfrom,connect -fp <pid> 2>&1 | grep -A2 '53'
```

## 10. 网络监控方案和指标？

**回答：**

### 关键网络指标

```
带宽（Bandwidth）: 网络吞吐量 (Mbps/Gbps)
延迟（Latency）:   网络往返时间 RTT (ms)
丢包率（Packet Loss）: 丢失数据包百分比 (%)
抖动（Jitter）:    延迟的波动程度 (ms)

TCP 指标：
  - 重传率 (Retransmission Rate)
  - 连接建立时间
  - 活跃连接数
  - TIME_WAIT 连接数

HTTP 指标：
  - 请求延迟 (P50/P95/P99)
  - QPS (每秒请求数)
  - 错误率 (4xx/5xx)
  - 响应大小
```

### 监控方案

```
Prometheus + Grafana 方案：

  node_exporter:
    - node_network_receive_bytes_total     (网卡入流量)
    - node_network_transmit_bytes_total    (网卡出流量)
    - node_network_receive_drop_total      (接收丢包)
    - node_network_transmit_drop_total     (发送丢包)
    - node_network_receive_errs_total      (接收错误)

  blackbox_exporter（探活/拨测）:
    - probe_success              (探测是否成功)
    - probe_duration_seconds     (探测总耗时)
    - probe_dns_lookup_time_seconds  (DNS解析耗时)
    - probe_tls_version_info     (TLS版本)
    - probe_ssl_earliest_cert_expiry (证书过期时间)

  cadvisor / kube-state-metrics (K8s):
    container_network_receive_bytes_total
    container_network_transmit_bytes_total

其他工具：
  - Smokeping: 持续延迟和丢包监控
  - Zabbix: 传统网络设备监控 (SNMP)
  - ntopng: 实时流量分析
  - Grafana Loki: 网络日志收集分析
```

### eBPF 网络可观测性

```bash
# 现代网络排障利器
# eBPF 可以在内核中安全地运行自定义程序

# bcc 工具集
tcpconnect    # 追踪 TCP 连接（替代 strace）
tcplife       # TCP 连接生命周期
tcpretrans    # TCP 重传
tcpdrop       # 内核丢包原因

# 示例
/usr/share/bcc/tools/tcpretrans
# 输出每一次重传的源/目地址、端口、状态

# Cilium Hubble (K8s 网络可观测性)
hubble observe --type drop     # 查看网络丢弃事件
hubble observe --namespace default  # 按命名空间过滤
```
