# 事务与锁机制

---

## 1. 事务的隔离级别？

**回答：**

```
四种隔离级别 (从低到高):

  ┌────────────────────┬──────────┬──────────┬──────────┐
  │ 隔离级别            │ 脏读     │ 不可重复读│ 幻读     │
  ├────────────────────┼──────────┼──────────┼──────────┤
  │ READ UNCOMMITTED   │ ✅ 可能  │ ✅ 可能  │ ✅ 可能  │
  │ READ COMMITTED     │ ❌ 不会  │ ✅ 可能  │ ✅ 可能  │
  │ REPEATABLE READ    │ ❌ 不会  │ ❌ 不会  │ ✅ 可能  │
  │ SERIALIZABLE       │ ❌ 不会  │ ❌ 不会  │ ❌ 不会  │
  └────────────────────┴──────────┴──────────┴──────────┘

  MySQL 默认: REPEATABLE READ (RR)
  PostgreSQL 默认: READ COMMITTED (RC)
  Oracle 默认: READ COMMITTED (RC)

三种并发问题:
  脏读:       读到其他事务未提交的数据
  不可重复读: 同一事务内两次读同一行, 结果不同 (被其他事务修改)
  幻读:       同一事务内两次范围查询, 行数不同 (被其他事务插入/删除)

InnoDB 的 RR 如何解决幻读:
  快照读 (SELECT): MVCC 解决
  当前读 (SELECT FOR UPDATE): Next-Key Lock 解决
```

```sql
-- 查看/设置隔离级别
SELECT @@transaction_isolation;       -- MySQL 8.0
SELECT @@tx_isolation;                -- MySQL 5.7

SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
SET GLOBAL TRANSACTION ISOLATION LEVEL REPEATABLE READ;
```

## 2. MVCC (多版本并发控制)？

**回答：**

```
MVCC: 实现非锁定读, 提高并发性能

核心组件:
  1. 隐藏列:
     DB_TRX_ID:    最后修改的事务 ID
     DB_ROLL_PTR:  指向 undo log 的指针
     DB_ROW_ID:    隐含自增 ID (无主键时)

  2. Undo Log (版本链):
     每次修改都保存旧版本
     通过 ROLL_PTR 形成版本链

  3. Read View (读视图):
     定义事务可以看到哪些版本
     包含: 活跃事务 ID 列表, 最小活跃事务 ID, 下一个事务 ID

版本链示例:
  ┌─────────────────────────┐
  │ 当前数据 (TRX_ID = 300) │
  │ name = '王五'           │
  │       │ ROLL_PTR        │
  │       ↓                 │
  │ Undo (TRX_ID = 200)    │
  │ name = '李四'           │
  │       │ ROLL_PTR        │
  │       ↓                 │
  │ Undo (TRX_ID = 100)    │
  │ name = '张三'           │
  └─────────────────────────┘

Read View 可见性判断:
  1. TRX_ID < min_trx_id     → 可见 (事务已提交)
  2. TRX_ID >= max_trx_id    → 不可见 (事务还没开始)
  3. TRX_ID 在活跃列表中      → 不可见 (事务未提交)
  4. TRX_ID 不在活跃列表中    → 可见 (事务已提交)

RC vs RR 的 Read View 区别:
  RC:  每次 SELECT 都创建新 Read View
       → 能读到其他事务已提交的最新数据
  RR:  第一次 SELECT 创建 Read View, 之后复用
       → 整个事务内看到的快照一致
```

## 3. InnoDB 锁的类型？

**回答：**

```
按粒度分:
  ┌──────────────────┬──────────────────────────────────┐
  │ 锁类型            │ 说明                             │
  ├──────────────────┼──────────────────────────────────┤
  │ 全局锁            │ FTWRL, 整个库只读 (备份用)        │
  │ 表级锁            │ 表锁, MDL 锁, 意向锁             │
  │ 行级锁            │ Record Lock, Gap Lock, Next-Key  │
  └──────────────────┴──────────────────────────────────┘

按模式分:
  共享锁 (S Lock / 读锁):
    多个事务可以同时持有
    SELECT ... LOCK IN SHARE MODE  (5.7)
    SELECT ... FOR SHARE           (8.0)

  排他锁 (X Lock / 写锁):
    只有一个事务可以持有
    SELECT ... FOR UPDATE
    INSERT / UPDATE / DELETE 自动加排他锁

  兼容性:
    ┌────┬────┬────┐
    │    │ S  │ X  │
    ├────┼────┼────┤
    │ S  │ ✅ │ ❌ │
    │ X  │ ❌ │ ❌ │
    └────┴────┴────┘

意向锁 (Intention Lock):
  表级锁, InnoDB 自动加
  IS (意向共享锁): 事务想给某行加 S 锁
  IX (意向排他锁): 事务想给某行加 X 锁
  → 快速判断表中是否有行锁, 避免逐行检查
```

## 4. InnoDB 行锁详解？

**回答：**

```
三种行锁:

  1. Record Lock (记录锁):
     锁定索引上的单条记录
     WHERE id = 5 → 锁住 id=5 这条记录

  2. Gap Lock (间隙锁):
     锁定索引记录之间的间隙 (不包含记录本身)
     防止其他事务在间隙中插入, 解决幻读
     
     例: 索引有 3, 7, 10
     Gap Lock (3, 7): 阻止插入 4, 5, 6
     
     只在 RR 隔离级别下生效
     RC 下没有 Gap Lock

  3. Next-Key Lock (临键锁):
     = Record Lock + Gap Lock
     锁定记录本身 + 记录前面的间隙
     
     例: 索引有 3, 7, 10, 15
     Next-Key Lock for 10: (7, 10]
     
     InnoDB 默认使用 Next-Key Lock
     
     退化规则:
       等值查询 + 唯一索引 + 命中 → 退化为 Record Lock
       等值查询 + 未命中 → 退化为 Gap Lock
       范围查询 → 保持 Next-Key Lock
```

```sql
-- 行锁加锁示例 (索引值: 3, 7, 10, 15)

-- 1. 等值查询 + 主键 + 命中
SELECT * FROM t WHERE id = 7 FOR UPDATE;
-- 加锁: Record Lock on id=7 (退化)

-- 2. 等值查询 + 主键 + 未命中
SELECT * FROM t WHERE id = 8 FOR UPDATE;
-- 加锁: Gap Lock (7, 10) (退化)

-- 3. 范围查询
SELECT * FROM t WHERE id >= 7 AND id < 10 FOR UPDATE;
-- 加锁: Next-Key Lock (3, 7] + (7, 10)

-- 4. 非唯一索引等值查询
SELECT * FROM t WHERE age = 7 FOR UPDATE;
-- 加锁: Next-Key Lock (3, 7] + Gap Lock (7, 10)
```

## 5. 死锁及处理？

**回答：**

```
死锁: 两个或多个事务相互等待对方持有的锁

示例:
  事务 A:                    事务 B:
  BEGIN;                     BEGIN;
  UPDATE t SET ... WHERE id=1;  UPDATE t SET ... WHERE id=2;
  -- 持有 id=1 的锁            -- 持有 id=2 的锁
  UPDATE t SET ... WHERE id=2;  UPDATE t SET ... WHERE id=1;
  -- 等待 id=2 的锁 ← 阻塞     -- 等待 id=1 的锁 ← 阻塞
  -- 死锁!

InnoDB 死锁检测:
  wait-for graph (等待图) 检测
  发现死锁 → 回滚代价最小的事务
  innodb_deadlock_detect = ON (默认开启)

查看死锁信息:
  SHOW ENGINE INNODB STATUS\G
  → LATEST DETECTED DEADLOCK 段

避免死锁:
  1. 固定加锁顺序: 多表/多行操作按固定顺序
  2. 缩小事务: 减少锁持有时间
  3. 避免长事务: 尽快 COMMIT
  4. 合理索引: 减小锁范围 (避免锁升级为表锁)
  5. 降低隔离级别: RR → RC (减少 Gap Lock)
```

```sql
-- 死锁监控
-- 1. 查看锁等待
SELECT * FROM information_schema.INNODB_LOCK_WAITS;     -- 5.7
SELECT * FROM performance_schema.data_lock_waits;        -- 8.0

-- 2. 查看当前锁
SELECT * FROM information_schema.INNODB_LOCKS;           -- 5.7
SELECT * FROM performance_schema.data_locks;             -- 8.0

-- 3. 查看长事务
SELECT * FROM information_schema.INNODB_TRX
WHERE TIME_TO_SEC(TIMEDIFF(NOW(), trx_started)) > 60;

-- 4. 超时设置
SET innodb_lock_wait_timeout = 50;  -- 默认 50 秒
-- 等锁超时 → 回滚当前语句 (不是整个事务)
```

## 6. 乐观锁 vs 悲观锁？

**回答：**

```
悲观锁 (Pessimistic Locking):
  假设: 冲突经常发生, 先加锁再操作
  实现: 数据库行锁 (SELECT FOR UPDATE)
  场景: 写冲突高, 如库存扣减

  -- 悲观锁操作
  START TRANSACTION;
  SELECT stock FROM goods WHERE id = 1 FOR UPDATE;  -- 加排他锁
  -- 判断 stock > 0
  UPDATE goods SET stock = stock - 1 WHERE id = 1;
  COMMIT;

乐观锁 (Optimistic Locking):
  假设: 冲突很少发生, 更新时才检查
  实现: 版本号 / CAS
  场景: 读多写少

  -- 版本号方式
  SELECT stock, version FROM goods WHERE id = 1;
  -- 假设 stock=10, version=5
  UPDATE goods SET stock = stock - 1, version = version + 1
  WHERE id = 1 AND version = 5;  -- 版本号匹配才更新
  -- 影响行数 = 0 → 被其他事务修改, 需重试

  -- CAS 方式 (比较并交换)
  UPDATE goods SET stock = stock - 1
  WHERE id = 1 AND stock = 10;   -- 旧值匹配才更新

对比:
  ┌──────────┬────────────────┬────────────────┐
  │ 维度      │ 悲观锁          │ 乐观锁          │
  ├──────────┼────────────────┼────────────────┤
  │ 加锁时机  │ 操作前加锁      │ 提交时检查       │
  │ 性能      │ 锁等待, 并发低  │ 无锁, 并发高     │
  │ 冲突处理  │ 阻塞等待        │ 失败重试         │
  │ 适用场景  │ 写多/高冲突     │ 读多/低冲突      │
  │ 死锁      │ 可能            │ 不会             │
  │ ABA 问题  │ 无              │ 版本号可防, CAS 有│
  └──────────┴────────────────┴────────────────┘
```

## 7. 当前读 vs 快照读？

**回答：**

```
快照读 (Snapshot Read):
  普通 SELECT 语句
  读取 MVCC 版本链中的历史版本
  不加锁, 不阻塞
  
  SELECT * FROM users WHERE id = 1;
  → 读取 Read View 可见的版本

当前读 (Current Read):
  读取最新版本, 加锁
  
  触发当前读的语句:
    SELECT ... FOR UPDATE      — 加 X 锁
    SELECT ... FOR SHARE       — 加 S 锁
    INSERT                     — 加 X 锁 (唯一键检查)
    UPDATE                     — 加 X 锁
    DELETE                     — 加 X 锁

幻读场景:
  事务 A (RR):
    T1: SELECT * FROM t WHERE age = 25;     — 快照读, 2 行
    T2: (事务 B INSERT age=25 并 COMMIT)
    T3: SELECT * FROM t WHERE age = 25;     — 快照读, 仍 2 行 (MVCC)
    T4: SELECT * FROM t WHERE age = 25 FOR UPDATE;  — 当前读, 3 行!
    
    或者:
    T3: UPDATE t SET name = 'x' WHERE age = 25;  — 当前读, 影响 3 行
    T4: SELECT * FROM t WHERE age = 25;     — 快照读, 3 行! (幻读)

  → RR 下 MVCC 解决快照读幻读, 但快照读 + 当前读混用时可能幻读
  → 完全防止: SELECT FOR UPDATE (加 Next-Key Lock)
```

## 8. 分布式事务？

**回答：**

```
分布式事务: 跨多个数据库/服务的事务

方案:
  ┌──────────────┬─────────────────────────────────┐
  │ 方案          │ 说明                             │
  ├──────────────┼─────────────────────────────────┤
  │ 2PC (XA)     │ 两阶段提交, 强一致性, 性能差     │
  │ TCC          │ Try-Confirm-Cancel, 业务侵入大   │
  │ Saga         │ 长事务拆分, 补偿事务              │
  │ 本地消息表    │ 可靠消息, 最终一致性              │
  │ 消息事务      │ RocketMQ 事务消息                │
  └──────────────┴─────────────────────────────────┘

2PC (两阶段提交):
  阶段一 (Prepare): 协调者问所有参与者 "能提交吗?"
  阶段二 (Commit):  所有回答 Yes → 提交; 有 No → 回滚

  问题:
    同步阻塞: 等待所有参与者
    单点故障: 协调者挂了
    数据不一致: 部分收到 Commit 部分没收到

TCC (Try-Confirm-Cancel):
  Try:     预留资源 (冻结库存)
  Confirm: 确认 (扣减冻结库存)
  Cancel:  取消 (释放冻结库存)
  → 业务层面实现, 每个操作都要写三个接口

Saga:
  长事务拆分为多个本地事务
  每个本地事务都有对应的补偿事务
  T1 → T2 → T3 → 若 T3 失败 → C3 → C2 → C1
  → 最终一致性

最终一致性 (本地消息表):
  1. 业务操作 + 写消息到本地消息表 (同一事务)
  2. 定时任务扫描消息表 → 发消息
  3. 消费者处理 + 确认
  4. 重试直到成功
```

## 9. Online DDL？

**回答：**

```sql
-- DDL 操作可能锁表! (特别是大表)

-- MySQL 5.6+ Online DDL:
ALTER TABLE t ADD INDEX idx_name(name), ALGORITHM=INPLACE, LOCK=NONE;

-- ALGORITHM:
--   COPY:    拷贝到临时表, 锁表 (旧方式)
--   INPLACE: 原地修改, 不需要全表拷贝
--   INSTANT: 只改元数据, 瞬间完成 (MySQL 8.0)

-- LOCK:
--   DEFAULT: 允许最大并发
--   NONE:    不加锁 (DDL 期间可读写)
--   SHARED:  只读, 不可写
--   EXCLUSIVE: 完全锁表

-- 常见操作对照:
-- ┌──────────────────┬───────────┬──────────┐
-- │ 操作              │ Algorithm │ Lock     │
-- ├──────────────────┼───────────┼──────────┤
-- │ 添加索引          │ INPLACE   │ NONE     │
-- │ 删除索引          │ INPLACE   │ NONE     │
-- │ 添加列           │ INSTANT   │ NONE     │  (MySQL 8.0)
-- │ 删除列           │ INPLACE   │ NONE     │  (rebuild table)
-- │ 修改列类型        │ COPY      │ SHARED   │  (需锁表!)
-- │ 修改列名          │ INPLACE   │ NONE     │
-- └──────────────────┴───────────┴──────────┘

-- 大表 DDL 工具 (避免锁表):
-- pt-online-schema-change (Percona):
--   创建新表 → 触发器同步 → 切换表名
-- gh-ost (GitHub):
--   创建新表 → binlog 同步 → 切换表名 (无触发器)
--   推荐! 更安全

-- gh-ost 示例:
-- gh-ost \
--   --host=master \
--   --database=mydb \
--   --table=orders \
--   --alter="ADD INDEX idx_status(status)" \
--   --execute
```

## 10. 事务与锁面试速答？

**回答：**

```
Q: MySQL 默认隔离级别?
A: REPEATABLE READ (RR)

Q: RR 下如何解决幻读?
A: 快照读: MVCC (Read View)
   当前读: Next-Key Lock (Record + Gap Lock)

Q: MVCC 的实现原理?
A: 隐藏列 (TRX_ID + ROLL_PTR) + Undo Log 版本链
   + Read View 判断可见性

Q: RC 和 RR 的 MVCC 区别?
A: RC 每次 SELECT 创建新 Read View
   RR 只在第一次 SELECT 创建, 后续复用

Q: 行锁加在哪?
A: 加在索引上! 不走索引 → 退化为表锁!

Q: Gap Lock 的作用?
A: 锁住索引间隙, 防止其他事务插入, 解决幻读

Q: 如何避免死锁?
A: 固定加锁顺序, 缩小事务, 合理索引, 降隔离级别

Q: 乐观锁和悲观锁选哪个?
A: 写多冲突高 → 悲观锁 (FOR UPDATE)
   读多冲突低 → 乐观锁 (版本号)

Q: 什么是两阶段提交?
A: 写 redo log (prepare) → 写 binlog → redo log (commit)
   保证 redo log 和 binlog 一致, 崩溃恢复正确

Q: 大表加索引怎么不锁表?
A: 用 gh-ost 或 pt-online-schema-change
   Online DDL: ALGORITHM=INPLACE, LOCK=NONE
```
