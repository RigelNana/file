# 索引原理与优化

---

## 1. 索引的数据结构？

**回答：**

```
B+ Tree (InnoDB 默认索引结构):

  为什么选 B+ Tree?
  ┌──────────────┬──────────────────────────────────┐
  │ 数据结构      │ 问题                             │
  ├──────────────┼──────────────────────────────────┤
  │ 二叉查找树    │ 极端情况退化为链表 O(n)           │
  │ AVL/红黑树    │ 树高太高, 磁盘 IO 多              │
  │ Hash         │ 不支持范围查询, 不支持排序         │
  │ B Tree       │ 非叶子节点存数据, 每页放的 key 少  │
  │ B+ Tree ✓   │ 只有叶子存数据, 叶子链表范围查高效  │
  └──────────────┴──────────────────────────────────┘

B+ Tree 特点:
  1. 非叶子节点只存 key (不存数据)
     → 每个节点可以放更多 key → 树更矮 → IO 更少
  2. 叶子节点存所有数据, 且通过双向链表连接
     → 范围查询只需遍历链表, 无需回溯
  3. 所有查询都走到叶子 → 查询性能稳定 O(log n)

  树高 vs 数据量:
    每个页 16KB, 每个 key 约 8B + 指针 6B
    非叶子每页 ≈ 16KB / 14B ≈ 1170 个 key
    树高 3: 1170 × 1170 × 16 ≈ 2000 万行!
    → 2000 万行数据只需 3 次磁盘 IO

                  [10, 20, 30]             ← 非叶子 (只存 key)
                 /    |    |    \
          [1..9] [10..19] [20..29] [30..39]  ← 叶子 (存数据)
            ↔       ↔        ↔       ↔       ← 双向链表
```

## 2. 聚簇索引 vs 非聚簇索引？

**回答：**

```
聚簇索引 (Clustered Index):
  数据按主键顺序存储
  叶子节点 = 完整数据行
  一个表只有一个聚簇索引

  InnoDB 聚簇索引选择顺序:
    1. 有主键 → 主键作为聚簇索引
    2. 无主键 → 第一个非空唯一索引
    3. 都没有 → InnoDB 自动生成 6 字节 rowid

非聚簇索引 (Secondary Index / 二级索引):
  叶子节点存储: 索引列值 + 主键值
  需要回表: 先查二级索引 → 得到主键 → 再查聚簇索引

回表过程:
  SELECT * FROM users WHERE name = '张三';
  
  1. 查 name 索引 (B+ Tree)
     → 找到 name='张三' 的叶子节点 → 得到 id=1001
  2. 用 id=1001 查聚簇索引 (回表)
     → 得到完整行 (id, name, age, email, ...)
  3. 返回结果

  → 回表 = 额外一次 B+ Tree 查询 = 额外磁盘 IO!

覆盖索引 (Covering Index):
  SELECT id, name FROM users WHERE name = '张三';
  
  如果 name 索引包含了 id 和 name
  → 直接从索引返回, 不需要回表!
  → EXPLAIN 中 Extra 显示 "Using index"

  创建覆盖索引:
  CREATE INDEX idx_name_age ON users(name, age);
  SELECT name, age FROM users WHERE name = '张三';
  → Using index, 无需回表
```

## 3. 联合索引与最左前缀原则？

**回答：**

```
联合索引 (Composite Index):
  CREATE INDEX idx_a_b_c ON t(a, b, c);

  B+ Tree 排序规则: 先按 a 排序, a 相同按 b, b 相同按 c

最左前缀原则:
  ┌─────────────────────────────────┬──────────┬──────────────┐
  │ 查询条件                        │ 走索引?   │ 使用到的列    │
  ├─────────────────────────────────┼──────────┼──────────────┤
  │ WHERE a = 1                    │ ✅       │ a            │
  │ WHERE a = 1 AND b = 2         │ ✅       │ a, b         │
  │ WHERE a = 1 AND b = 2 AND c=3 │ ✅       │ a, b, c      │
  │ WHERE b = 2                    │ ❌       │ 无 (缺最左a) │
  │ WHERE c = 3                    │ ❌       │ 无           │
  │ WHERE b = 2 AND c = 3         │ ❌       │ 无           │
  │ WHERE a = 1 AND c = 3         │ ⚠️ 部分  │ 只用到 a     │
  │ WHERE a > 1 AND b = 2         │ ⚠️ 部分  │ a (范围后停) │
  │ WHERE a = 1 AND b > 2 AND c=3 │ ⚠️ 部分  │ a, b         │
  │ WHERE a = 1 ORDER BY b        │ ✅       │ a + 排序 b   │
  └─────────────────────────────────┴──────────┴──────────────┘

关键规则:
  1. 查询条件必须从最左列开始
  2. 遇到范围查询 (>, <, BETWEEN, LIKE 'xx%') 后面的列不走索引
  3. 等值查询不受顺序影响 (优化器会调整)
     WHERE b=2 AND a=1 → 优化器重排为 a=1 AND b=2 → 走索引

联合索引设计原则:
  1. 等值查询列放前面, 范围查询列放后面
  2. 区分度高的列放前面 (选择性 = 不同值数 / 总行数)
  3. 常用查询条件的列放前面
  4. 尽量利用覆盖索引减少回表
```

## 4. 索引失效场景详解？

**回答：**

```sql
-- 1. 对索引列使用函数
WHERE YEAR(create_time) = 2024         -- ❌ 失效
WHERE create_time >= '2024-01-01'
  AND create_time < '2025-01-01'       -- ✅ 范围查询

-- 2. 隐式类型转换
WHERE phone = 13812345678              -- ❌ phone 是 VARCHAR, 数字触发隐式转换
WHERE phone = '13812345678'            -- ✅ 类型匹配

-- 3. 隐式字符集转换
-- 两个表 JOIN 列字符集不同 → utf8 vs utf8mb4 → 转换 → 失效

-- 4. LIKE 左模糊
WHERE name LIKE '%test'                -- ❌ 全表扫描
WHERE name LIKE 'test%'                -- ✅ 走索引
WHERE name LIKE '%test%'               -- ❌ 全表扫描

-- 5. OR 条件
WHERE a = 1 OR b = 2                   -- ❌ 如果 a 有索引但 b 没有
-- 优化: 确保 OR 的所有列都有索引, 或改用 UNION
SELECT * FROM t WHERE a = 1
UNION ALL
SELECT * FROM t WHERE b = 2;

-- 6. NOT IN / NOT EXISTS / !=
WHERE status != 1                      -- ❌ 可能全表扫描
WHERE status IN (0, 2, 3)              -- ✅ 用 IN 代替

-- 7. 索引列参与运算
WHERE id + 1 = 10                      -- ❌
WHERE id = 9                           -- ✅

-- 8. 使用 IS NOT NULL
WHERE name IS NOT NULL                 -- ❌ 可能不走索引
-- 设计时: 给列设默认值, 避免 NULL

-- 9. 数据量太小或选择性太低
WHERE gender = '男'                     -- 50% 数据, 全表扫描更快
```

```
索引失效排查:
  EXPLAIN SELECT ... ;
  
  关注:
    type = ALL → 全表扫描 (索引失效)
    key = NULL → 没用到索引
    Extra: Using where → 存储引擎返回后 Server 层过滤
```

## 5. 如何创建高效的索引？

**回答：**

```
索引设计原则:

  1. 选择性高的列
     选择性 = COUNT(DISTINCT col) / COUNT(*)
     性别: 0.5 (差) → 不适合单独建索引
     手机号: ~1.0 (好) → 适合建索引

  2. 前缀索引 (节省空间)
     ALTER TABLE t ADD INDEX idx_email(email(6));
     → 只索引前 6 个字符
     查找最优前缀长度:
       SELECT COUNT(DISTINCT LEFT(email, n)) / COUNT(*) FROM t;
       → n 从小到大, 找到选择性接近完整列的值

  3. 覆盖索引减少回表
     -- 频繁查询: SELECT name, age FROM users WHERE name = ?
     CREATE INDEX idx_name_age ON users(name, age);
     → 不需要回表

  4. 索引下推 (ICP, Index Condition Pushdown)
     MySQL 5.6+ 自动优化
     WHERE name LIKE '张%' AND age = 25
     → 在索引层就过滤 age, 减少回表次数
     → EXPLAIN Extra: Using index condition

  5. 避免过多索引
     ✗ 每个列都建索引 → 插入/更新变慢
     ✓ 根据查询模式建联合索引
     ✓ 一个表索引不超过 5-6 个

  6. 不可见索引 (MySQL 8.0)
     ALTER TABLE t ALTER INDEX idx INVISIBLE;
     → 优化器不使用该索引, 但索引仍在维护
     → 测试删索引的影响, 安全后再真删
```

## 6. EXPLAIN 执行计划详解？

**回答：**

```sql
EXPLAIN SELECT * FROM orders WHERE user_id = 100 AND status = 1;
```

```
EXPLAIN 关键列:

  id: 查询序号 (子查询/UNION 有多个 id)

  select_type:
    SIMPLE:      简单查询
    PRIMARY:     最外层查询
    SUBQUERY:    子查询
    DERIVED:     派生表 (FROM 子查询)
    UNION:       UNION 第二个及后续查询

  type (访问类型, 从好到差):
    system:  表只有一行
    const:   主键/唯一索引等值查询
    eq_ref:  JOIN 时被驱动表用主键/唯一索引
    ref:     非唯一索引等值查询
    range:   索引范围扫描 (>, <, IN, BETWEEN)
    index:   扫描整个索引树 (覆盖索引)
    ALL:     全表扫描 ← 需要优化!

  possible_keys: 可能使用的索引
  key:           实际使用的索引 (NULL = 全表扫描)
  key_len:       使用索引的字节长度 (越短越好)
  ref:           索引查找的比较值
  rows:          预估扫描行数 (越少越好)
  filtered:      满足条件的行百分比

  Extra (重要):
    Using index:          覆盖索引, 无需回表 ✓
    Using where:          存储引擎返回后再过滤
    Using index condition: 索引下推 (ICP) ✓
    Using temporary:      使用临时表 ← 需优化
    Using filesort:       额外排序 ← 需优化
    Using join buffer:    JOIN 缓冲
```

```sql
-- 优化示例: Using filesort → 利用索引排序
-- 优化前:
EXPLAIN SELECT * FROM orders WHERE user_id = 100 ORDER BY create_time;
-- type=ref, Extra: Using filesort

-- 优化: 创建联合索引
CREATE INDEX idx_user_time ON orders(user_id, create_time);
-- type=ref, Extra: 无 filesort (索引本身有序)
```

## 7. 索引的代价与权衡？

**回答：**

```
索引的代价:

  1. 空间代价
     每个索引都是一棵 B+ Tree
     联合索引多列 → 索引更大
     → 关注 information_schema.TABLES.INDEX_LENGTH

  2. 写入代价
     INSERT: 数据 + 所有索引都要更新
     UPDATE: 修改索引列 → 索引需要维护
     DELETE: 索引标记删除 (不立即清理)
     → 索引越多, 写入越慢

  3. 维护代价
     页分裂: 随机插入 → B+ Tree 页分裂 → 性能下降
     碎片: 频繁删除 → 索引出现空洞
     重建: ALTER TABLE t ENGINE=InnoDB; 或 OPTIMIZE TABLE

权衡:
  ┌──────────────────┬──────────────────────────┐
  │ 适合建索引        │ 不适合建索引              │
  ├──────────────────┼──────────────────────────┤
  │ WHERE 条件列      │ 低选择性列 (性别/状态)    │
  │ JOIN 关联列       │ 频繁更新的列             │
  │ ORDER BY 列      │ 很少查询的列             │
  │ GROUP BY 列      │ 表数据量很小             │
  │ 覆盖索引需要的列  │ 已有太多索引的表          │
  └──────────────────┴──────────────────────────┘

查看索引使用情况:
  SELECT * FROM sys.schema_unused_indexes;      -- 未使用的索引
  SELECT * FROM sys.schema_redundant_indexes;    -- 冗余索引
```

## 8. 索引优化实战案例？

**回答：**

```sql
-- 案例 1: 分页优化
-- 问题: OFFSET 大时性能差
SELECT * FROM orders ORDER BY id LIMIT 1000000, 10;
-- 扫描 1000010 行, 丢弃前 100 万行!

-- 优化: 延迟关联 (Deferred Join)
SELECT o.* FROM orders o
INNER JOIN (
  SELECT id FROM orders ORDER BY id LIMIT 1000000, 10
) AS tmp ON o.id = tmp.id;
-- 子查询只扫描索引 (覆盖索引), 然后回表 10 行!

-- 或: 游标分页 (记住上次最大 id)
SELECT * FROM orders WHERE id > 1000000 ORDER BY id LIMIT 10;


-- 案例 2: 字符串模糊查询优化
-- 问题: LIKE '%keyword%' 全表扫描
SELECT * FROM articles WHERE content LIKE '%MySQL%';

-- 优化方案:
-- 1. 全文索引
ALTER TABLE articles ADD FULLTEXT INDEX ft_content(content);
SELECT * FROM articles WHERE MATCH(content) AGAINST('MySQL');

-- 2. ElasticSearch 全文搜索 (推荐大规模)


-- 案例 3: 排序优化
-- 问题: ORDER BY 导致 filesort
SELECT * FROM orders WHERE status = 1 ORDER BY create_time DESC;

-- 优化: 联合索引覆盖 WHERE + ORDER BY
CREATE INDEX idx_status_time ON orders(status, create_time);
-- 等值 status 后按 create_time 有序 → 无需排序
```

## 9. 自增主键 vs UUID？

**回答：**

```
自增主键 (AUTO_INCREMENT):
  ✓ 顺序插入, B+ Tree 追加写入, 无页分裂
  ✓ 占用空间小 (BIGINT 8 字节)
  ✓ 聚簇索引紧凑, 范围查询高效
  ✗ 分布式环境 ID 冲突
  ✗ 暴露数据量 (id=10000 → 有 1 万条数据)
  ✗ 数据迁移/合并时 ID 冲突

UUID:
  ✓ 全局唯一, 不依赖数据库
  ✓ 不暴露数据量
  ✗ 36 字节字符串, 占用空间大
  ✗ 随机插入, 导致页分裂, 性能差
  ✗ 不可排序

雪花算法 (Snowflake):
  ✓ 全局唯一 + 有序 (趋势递增)
  ✓ 64 位整数 (BIGINT)
  ✓ 包含时间信息, 可大致排序
  ✗ 依赖时钟 (时钟回拨问题)

  结构: 1位符号 + 41位时间戳 + 10位机器ID + 12位序列号
  → 每毫秒可生成 4096 个 ID

建议:
  单库:     BIGINT AUTO_INCREMENT
  分布式:   雪花算法 / 自研 ID 生成器
  业务需要: UUID 存为 BINARY(16) (比 VARCHAR(36) 好)
```

## 10. 索引面试高频问题？

**回答：**

```
Q: 为什么 InnoDB 用 B+ Tree 不用 B Tree?
A: B+ Tree 非叶子不存数据 → 每页放更多 key → 树更矮
   叶子链表 → 范围查询只需遍历链表
   所有查询到叶子 → 性能稳定

Q: 什么是回表? 怎么避免?
A: 二级索引查到主键 → 再查聚簇索引获取完整行
   避免: 使用覆盖索引 (查询列都在索引中)

Q: 联合索引 (a,b,c), WHERE a=1 AND c=3 用到几个列?
A: 只用到 a (中间缺 b, c 无法利用)

Q: 为什么推荐自增主键?
A: 顺序插入无页分裂, 占用空间小, 二级索引存储主键值也小

Q: 索引越多越好吗?
A: 不是! 索引加速读但拖慢写, 占用额外空间
   根据查询模式精准建索引, 一般不超过 5-6 个

Q: 什么是索引下推 (ICP)?
A: MySQL 5.6+, 在存储引擎层就用索引过滤
   减少回表次数, EXPLAIN Extra: Using index condition

Q: EXPLAIN type=ALL 怎么办?
A: 全表扫描! 检查:
   1. 是否缺索引 → 添加索引
   2. 索引是否失效 → 检查函数/类型转换
   3. 数据量太小优化器选择全表 → 正常

Q: 前缀索引的优缺点?
A: 优: 节省空间, 适合长字符串
   缺: 不能用于 ORDER BY, 不能做覆盖索引
   用 LEFT(col, n) 找最优长度
```
