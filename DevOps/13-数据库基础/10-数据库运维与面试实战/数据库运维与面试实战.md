## 2. 数据库容量规划？

**回答：**

```
容量评估维度:
  1. 数据量: 当前 + 预估增长
  2. QPS:    读 QPS + 写 QPS
  3. 延迟:   P50, P95, P99 要求
  4. 连接数: 并发连接数
  5. 可用性: SLA 要求 (99.9%? 99.99%?)

MySQL 容量基准:
  单表行数:   建议 < 2000 万 (超过考虑分表)
  单库大小:   建议 < 500GB
  QPS:        单实例 3000-8000 (取决于查询复杂度)
  连接数:     建议 < 500 (每个连接约 10MB 内存)

Redis 容量基准:
  单实例:     建议 < 30GB (大了 fork 慢)
  QPS:        单实例 10 万+
  Key 数量:   百万级可以, 千万级考虑 Cluster

评估公式:
  存储 = 单行大小 × 行数 × (1 + 索引膨胀系数)
  索引膨胀: 通常 1.5-3 倍数据大小
  
  例: 1KB/行 × 1000 万行 × 2.5 = 25GB
  加 binlog/备份: 预留 3 倍 = 75GB

扩容时机:
  CPU > 70% 持续              → 升配或读写分离
  磁盘 > 80%                  → 扩容或清理
  QPS 达到 70% 基准           → 加缓存或分库
  主从延迟持续 > 10s          → 检查从库或加从
  慢查询数持续增长             → 优化 SQL 或加索引
```

## 3. 数据库备份与灾备？

**回答：**

```
备份策略:
  ┌──────────┬──────────────────────────────────────┐
  │ 频率      │ 方式                                  │
  ├──────────┼──────────────────────────────────────┤
  │ 全量      │ 每天/每周 (xtrabackup / mysqldump)    │
  │ 增量      │ 每小时 (xtrabackup incremental)       │
  │ 实时      │ binlog 持续归档                       │
  │ 测试      │ 每月恢复演练                           │
  └──────────┴──────────────────────────────────────┘

RPO / RTO:
  RPO (Recovery Point Objective): 能接受丢多少数据
    RPO = 0: 不能丢数据 → 同步复制
    RPO <1h: 增量备份 + binlog

  RTO (Recovery Time Objective): 恢复要多快
    RTO <1h: 物理备份 (xtrabackup) + 热备从库
    RTO <5min: 主从自动切换

灾备方案:
  ┌──────────────┬──────┬──────┬─────────────────────┐
  │ 方案          │ RPO  │ RTO  │ 成本                 │
  ├──────────────┼──────┼──────┼─────────────────────┤
  │ 备份恢复      │ 小时 │ 小时 │ 低                   │
  │ 主从 + 手动切 │ 秒   │ 分钟 │ 中 (从库成本)        │
  │ 自动故障转移  │ 秒   │ 秒   │ 中 (Sentinel/MHA)   │
  │ 多 AZ 部署   │ 0    │ 秒   │ 高 (同步复制)        │
  │ 跨 Region    │ 秒   │ 分钟 │ 高 (异步复制)        │
  └──────────────┴──────┴──────┴─────────────────────┘

备份验证:
  定期恢复到测试环境
  校验: pt-table-checksum (主从数据一致性)
  自动化: 脚本恢复 + 数据校验 + 告警
```

## 4. 数据库读写分离架构设计？

**回答：**

```
完整读写分离架构:

  应用层
    │
  连接池 (HikariCP / SQLAlchemy)
    │
  代理层 (ProxySQL / RDS Proxy)
    ├── 写 → Master (主库)
    │         │
    │    Binlog 复制 (半同步)
    │         │
    └── 读 → Slave 1, Slave 2, Slave 3
            (轮询 / 权重 / 最少连接)

一致性保证:
  1. 强一致性场景 (支付/扣库存):
     → 读写都走 Master
     → SQL Hint: /* master */ SELECT ...
  
  2. 会话一致性 (写后读):
     → 写后 N 秒内读走 Master
     → ProxySQL: mysql_query_rules 控制
  
  3. 最终一致性 (列表/搜索):
     → 允许短暂延迟, 读走 Slave
     → 大多数场景可接受

监控:
  主从延迟: pt-heartbeat (精确到 ms)
  延迟 > 阈值 → 自动切读到 Master (降级)
  Slave 故障 → 自动摘除, 不路由读请求
```

## 5. 数据库安全？

**回答：**

```
安全层次:
  1. 网络层
     数据库不暴露公网
     VPC 内网 + Security Group (只开 3306)
     SSL/TLS 加密传输

  2. 认证层
     强密码策略
     禁用 root 远程登录
     最小权限原则
```

```sql
-- 用户权限管理
CREATE USER 'app'@'10.0.0.%' IDENTIFIED BY 'strong_password';

-- 最小权限: 只授予需要的表和操作
GRANT SELECT, INSERT, UPDATE ON mydb.users TO 'app'@'10.0.0.%';
GRANT SELECT ON mydb.orders TO 'app'@'10.0.0.%';

-- 只读用户
CREATE USER 'readonly'@'%' IDENTIFIED BY 'password';
GRANT SELECT ON mydb.* TO 'readonly'@'%';

-- 查看权限
SHOW GRANTS FOR 'app'@'10.0.0.%';
```

```
  3. 数据层
     敏感数据加密:
       传输中: SSL/TLS
       静态:   TDE (透明数据加密)
       字段级: 应用层加密 (AES)
     
     脱敏:
       姓名: 张** 
       手机: 138****5678
       身份证: 110***********1234

  4. 审计层
     开启审计日志 (谁在什么时候操作了什么)
     MySQL Enterprise Audit / Percona Audit Plugin
     定期审查权限和日志

  5. 防 SQL 注入
     应用层: 参数化查询 (Prepared Statement)
     WAF: 检测 SQL 注入特征
     测试: sqlmap 扫描
```

## 6. ElasticSearch 基础？

**回答：**

```
Elasticsearch: 分布式搜索和分析引擎

核心概念:
  ┌──────────────────┬──────────────────────────────┐
  │ 概念              │ 对应 RDBMS                    │
  ├──────────────────┼──────────────────────────────┤
  │ Index            │ Database                      │
  │ Type (7.x 移除)  │ Table                         │
  │ Document         │ Row                           │
  │ Field            │ Column                        │
  │ Mapping          │ Schema                        │
  │ Shard            │ 分区                           │
  │ Replica          │ 副本                           │
  └──────────────────┴──────────────────────────────┘

特点:
  倒排索引: 分词 → 文档列表 (全文搜索高效)
  分布式:   数据自动分片到多节点
  近实时:   写入后约 1 秒可搜索
  DSL 查询: JSON 格式查询语言

使用场景:
  全文搜索 (商品搜索, 内容搜索)
  日志分析 (ELK Stack: Elasticsearch + Logstash + Kibana)
  指标聚合 (时序数据分析)
  应用搜索 (App Search)

与数据库的关系:
  MySQL: 主数据库 (事务, CRUD)
  ES:    搜索引擎 (全文搜索, 复杂查询)
  同步:  Canal / Debezium 监听 binlog → 写入 ES
```

## 7. 数据库连接池？

**回答：**

```
连接池: 复用数据库连接, 避免频繁创建/销毁

工作原理:
  1. 启动时创建 N 个连接 (min idle)
  2. 应用获取连接 (从池中取)
  3. 使用完归还连接 (放回池中)
  4. 超过 max 时等待或拒绝

常用连接池:
  ┌──────────────────┬──────────────────────────────┐
  │ 连接池            │ 说明                          │
  ├──────────────────┼──────────────────────────────┤
  │ HikariCP         │ Java 最快, Spring Boot 默认   │
  │ Druid            │ 阿里巴巴, 监控强              │
  │ c3p0             │ 老牌, 不推荐新项目             │
  │ SQLAlchemy Pool  │ Python                       │
  │ pgbouncer        │ PostgreSQL 专用代理连接池      │
  │ ProxySQL         │ MySQL 代理连接池               │
  └──────────────────┴──────────────────────────────┘

关键参数:
  最小空闲连接: 保持的最小连接数
  最大连接数:   允许的最大连接数
  连接超时:     获取连接的最大等待时间
  空闲超时:     空闲连接的最大存活时间
  最大生命周期: 连接的最大存活时间 (防止使用过期连接)

连接数设置:
  经验公式: connections = (CPU 核数 × 2) + 有效磁盘数
  例: 8 核 + 1 SSD = 17 个连接 / 应用实例
  
  注意:
    应用实例数 × 最大连接数 < MySQL max_connections
    留余量给监控/管理连接
```

## 8. 数据库迁移与版本管理？

**回答：**

```
Schema 迁移工具:
  Flyway:   SQL 文件版本化迁移 (Java 生态)
  Liquibase: XML/YAML/SQL 迁移 (Java 生态)
  Alembic:  Python (SQLAlchemy 配套)
  Django:   内置 migrations
  prisma:   Node.js
  golang-migrate: Go

Flyway 示例:
  sql/
  ├── V1__create_users.sql
  ├── V2__add_email_column.sql
  └── V3__create_orders.sql

  -- V1__create_users.sql
  CREATE TABLE users (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
  );

  -- V2__add_email_column.sql
  ALTER TABLE users ADD COLUMN email VARCHAR(255);
  CREATE INDEX idx_email ON users(email);

  命名规则: V{版本号}__{描述}.sql

迁移最佳实践:
  1. 所有 Schema 变更通过迁移文件 (不手动改)
  2. 迁移文件提交到 Git (版本控制)
  3. CI/CD 自动执行迁移
  4. 向后兼容: 先加新列 → 应用代码更新 → 再删旧列
  5. 大表 DDL: 使用 gh-ost 或 pt-osc
  6. 测试环境先执行验证
  7. 每次迁移可回滚 (记录 DOWN 脚本)
```

## 9. 面试场景题：设计一个计数系统？

**回答：**

```
场景: 抖音点赞数/播放数 (高并发写, 读多)

方案:
  ┌─────────────────────────────────────────────┐
  │ 客户端                                      │
  │   ↓ 点赞请求                                │
  │ API 网关 (限流/去重)                         │
  │   ↓                                         │
  │ 服务层                                      │
  │   ├── 写: Redis INCR (实时计数)              │
  │   │     └── 异步 → Kafka → DB (持久化)       │
  │   └── 读: Redis GET (从缓存读)               │
  │                                              │
  │ 数据存储                                     │
  │   Redis: 实时计数 (INCR 原子操作, 10万+ QPS) │
  │   MySQL: 持久化 (定期同步)                    │
  └─────────────────────────────────────────────┘

设计要点:
  1. 写入: Redis INCR (原子, 高性能)
  2. 持久化: 异步批量写入 MySQL (不阻塞请求)
  3. 读取: 优先 Redis, 降级读 MySQL
  4. 去重: Redis Set 记录用户是否已点赞
  5. 精确 vs 近似: 播放数可以近似, 交易金额必须精确

代码流程:
  # 点赞
  if SISMEMBER(liked:{video_id}, user_id) == 0:
      SADD(liked:{video_id}, user_id)
      INCR(count:{video_id})
      send_to_kafka(video_id, user_id, "like")

  # 读计数
  count = GET(count:{video_id})
  if count is None:
      count = SELECT count FROM video_stats WHERE id = video_id
      SET(count:{video_id}, count, EX=3600)
    
  # Kafka 消费者 (异步持久化)
  batch_update: UPDATE video_stats SET count = count + N WHERE id = ?
```

## 10. 数据库面试综合速答？

**回答：**

```
Q: 项目中数据库怎么设计的?
A: 根据业务选型 (MySQL OLTP), utf8mb4 字符集
   核心表设计范式化, 热点查询反范式冗余
   自增 BIGINT 主键, 联合索引覆盖高频查询

Q: 数据库慢了怎么排查?
A: 1. 慢查询日志 + pt-query-digest 找到慢 SQL
   2. EXPLAIN 分析执行计划
   3. 优化索引/SQL (避免全表扫描)
   4. 架构优化: 缓存/读写分离/分库分表

Q: 秒杀场景数据库怎么设计?
A: Redis 预扣库存 (DECR) → 异步下单 → MySQL 持久化
   限流 + 队列削峰 + Redis 原子操作

Q: 如何保证缓存和 DB 一致性?
A: Cache Aside: 先更新 DB → 删缓存
   + 消息队列重试删除
   + 订阅 binlog 异步更新缓存 (Canal)

Q: MySQL 和 Redis 各自优势?
A: MySQL: 事务 + 持久化 + SQL 查询 + 关系
   Redis: 高性能 + 丰富数据结构 + 原子操作
   两者配合: MySQL 存数据, Redis 做缓存

Q: 分布式 ID 怎么生成?
A: 雪花算法 (时间+机器+序号, 有序唯一 BIGINT)
   号段模式 (DB 批量分配 ID 段)
   UUID (全局唯一但无序, 不推荐做主键)

Q: 千万级大表怎么优化?
A: 1. 索引优化 (覆盖索引, 联合索引)
   2. 分区表 (按时间/范围分区)
   3. 水平分表 (按 ID Hash)
   4. 归档历史数据
   5. 读写分离 + 缓存

Q: 数据库连接池怎么配?
A: 最大连接数: CPU核 × 2 + 磁盘数
   所有实例连接数之和 < max_connections
   设置 连接超时/空闲超时/最大生命周期
```
