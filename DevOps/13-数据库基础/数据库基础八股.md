## 一、关系型数据库基础

### 1. 关系型数据库和非关系型数据库的区别？

**答：**

| 特性 | 关系型 (SQL) | 非关系型 (NoSQL) |
|------|-------------|-----------------|
| 数据模型 | 表格、行列 | 文档、键值、列族、图 |
| Schema | 固定 Schema | 灵活/无 Schema |
| 事务 | 完整 ACID 支持 | 部分支持或最终一致 |
| 扩展方式 | 垂直扩展为主 | 水平扩展友好 |
| 查询语言 | SQL | 各有各的查询 API |
| 代表 | MySQL, PostgreSQL | MongoDB, Redis, Cassandra |

### 2. 什么是 ACID？

**答：**

| 特性 | 英文 | 含义 |
|------|------|------|
| **原子性** | Atomicity | 事务要么全部成功，要么全部回滚 |
| **一致性** | Consistency | 事务前后数据库处于一致状态 |
| **隔离性** | Isolation | 并发事务互不干扰 |
| **持久性** | Durability | 事务提交后数据持久保存 |

### 3. MySQL 的存储引擎有哪些？InnoDB 和 MyISAM 的区别？

**答：**

| 特性 | InnoDB | MyISAM |
|------|--------|--------|
| 事务 | ✅ 支持 | ❌ 不支持 |
| 行级锁 | ✅ 支持 | ❌ 仅表锁 |
| 外键 | ✅ 支持 | ❌ 不支持 |
| 崩溃恢复 | ✅ 支持 | ❌ 不支持 |
| 全文索引 | ✅ (5.6+) | ✅ 支持 |
| 存储方式 | 聚簇索引 | 非聚簇索引 |
| 适用场景 | OLTP（事务型） | OLAP（分析型/读多写少） |

**生产环境中，InnoDB 是 MySQL 默认且推荐的存储引擎。**

## 二、索引

### 4. 什么是索引？有哪些类型？

**答：**

索引是数据库用于加速查询的数据结构，类似书的目录。

| 索引类型 | 说明 |
|---------|------|
| **B+ Tree 索引** | 默认索引结构，范围查询高效 |
| **Hash 索引** | 等值查询 O(1)，不支持范围查询 |
| **全文索引** | 文本搜索 |
| **空间索引** | 地理数据 |

**按功能分：**
| 类型 | 说明 |
|------|------|
| 主键索引 | 唯一+非空，聚簇索引 |
| 唯一索引 | 列值唯一 |
| 普通索引 | 无约束 |
| 联合索引 | 多列组合索引 |
| 覆盖索引 | 查询所需列全在索引中 |

```sql
-- 创建索引
CREATE INDEX idx_name ON users(name);
CREATE UNIQUE INDEX idx_email ON users(email);
CREATE INDEX idx_name_age ON users(name, age);  -- 联合索引

-- 查看执行计划
EXPLAIN SELECT * FROM users WHERE name = 'test';
```

### 5. 联合索引的最左前缀原则？

**答：**

对于联合索引 `(a, b, c)`：

| 查询条件 | 是否使用索引 | 说明 |
|---------|------------|------|
| `WHERE a = 1` | ✅ | 最左列 |
| `WHERE a = 1 AND b = 2` | ✅ | 命中 (a, b) |
| `WHERE a = 1 AND b = 2 AND c = 3` | ✅ | 全命中 |
| `WHERE b = 2` | ❌ | 缺少最左列 a |
| `WHERE b = 2 AND c = 3` | ❌ | 缺少最左列 a |
| `WHERE a = 1 AND c = 3` | ⚠️ 部分 | 只用到 a |

### 6. 索引失效的常见场景？

**答：**

```sql
-- 1. 对索引列使用函数
WHERE YEAR(create_time) = 2024         -- ❌ 失效
WHERE create_time >= '2024-01-01'      -- ✅

-- 2. 隐式类型转换
WHERE phone = 13812345678              -- ❌ phone 是 varchar
WHERE phone = '13812345678'            -- ✅

-- 3. LIKE 左模糊
WHERE name LIKE '%test'                -- ❌
WHERE name LIKE 'test%'                -- ✅

-- 4. OR 条件（部分列无索引）
WHERE a = 1 OR b = 2                   -- ❌ 如果 b 无索引

-- 5. 不等于操作
WHERE status != 1                      -- ❌ 可能全表扫描

-- 6. IS NULL / IS NOT NULL
WHERE name IS NOT NULL                 -- ❌ 可能不走索引
```

## 三、SQL 优化

### 7. 慢查询排查和优化方法？

**答：**

```sql
-- 1. 开启慢查询日志
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1;    -- 超过 1 秒记录

-- 2. 使用 EXPLAIN 分析执行计划
EXPLAIN SELECT * FROM orders WHERE user_id = 100;
```

**EXPLAIN 关键字段：**

| 字段 | 含义 | 关注点 |
|------|------|-------|
| type | 连接类型 | system > const > eq_ref > ref > range > index > ALL |
| key | 使用的索引 | NULL 表示全表扫描 |
| rows | 扫描行数 | 越小越好 |
| Extra | 额外信息 | Using index(好), Using filesort(需优化) |

**常见优化手段：**
1. 添加合适的索引
2. 避免 `SELECT *`，只查需要的列
3. 大表分页优化：用游标代替 `OFFSET`
4. 避免子查询，改用 JOIN
5. 批量操作代替循环单条操作
6. 读写分离，主从架构

## 四、MySQL 主从复制与高可用

### 8. MySQL 主从复制的原理？

**答：**

```
  Master (主库)                         Slave (从库)
  ┌──────────────┐                    ┌──────────────┐
  │ 写入数据      │                    │ IO Thread    │
  │      ↓        │                    │    ↓         │
  │ Binary Log    │ ──── 网络 ────→    │ Relay Log    │
  │              │                     │    ↓         │
  └──────────────┘                    │ SQL Thread   │
                                      │    ↓         │
                                      │ 应用到数据    │
                                      └──────────────┘
```

**三个线程：**
1. **Master Binlog Dump Thread**：主库发送 binlog 给从库
2. **Slave IO Thread**：从库接收 binlog，写入 Relay Log
3. **Slave SQL Thread**：从库回放 Relay Log

**复制模式：**
| 模式 | 说明 | 特点 |
|------|------|------|
| 异步复制 | 默认模式，主库不等从库确认 | 性能好，可能丢数据 |
| 半同步复制 | 主库等至少一个从库确认 | 平衡性能与安全 |
| 组复制 (MGR) | 多主多从，Paxos 协议 | 强一致性 |

### 9. MySQL 高可用方案有哪些？

**答：**

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **主从复制 + MHA** | Master High Availability | 传统方案，自动故障转移 |
| **InnoDB Cluster (MGR)** | MySQL 官方方案，组复制 | MySQL 8.0+ |
| **ProxySQL / MySQL Router** | 数据库代理，读写分离 | 中间件路由 |
| **云托管 RDS** | AWS RDS Multi-AZ, 阿里云 RDS | 免运维 |

## 五、Redis

### 10. Redis 的数据类型和使用场景？

**答：**

| 数据类型 | 说明 | 使用场景 |
|---------|------|---------|
| **String** | 字符串/数字 | 缓存、计数器、分布式锁 |
| **Hash** | 哈希表 | 用户信息、对象存储 |
| **List** | 双向链表 | 消息队列、最新列表 |
| **Set** | 无序集合 | 去重、交集/并集运算 |
| **Sorted Set** | 有序集合（带分数） | 排行榜、延迟队列 |
| **Stream** | 消息流 (5.0+) | 消息队列、事件流 |

```bash
# String
SET key value EX 3600        # 设置并过期
INCR counter                 # 原子自增

# Hash
HSET user:1001 name "张三" age 25
HGETALL user:1001

# Sorted Set
ZADD leaderboard 100 "player1"
ZREVRANGE leaderboard 0 9    # 获取 Top 10
```

### 11. Redis 持久化方式？

**答：**

| 方式 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **RDB (快照)** | 定时将内存数据序列化到磁盘 | 恢复快，文件紧凑 | 可能丢失最后一次快照后的数据 |
| **AOF (追加日志)** | 记录每条写命令 | 数据安全性高 | 文件大，恢复慢 |
| **混合持久化** (4.0+) | RDB + AOF 结合 | 兼顾性能和安全 | 文件可读性差 |

```
# redis.conf
# RDB
save 900 1          # 900秒内至少1次修改则快照
save 300 10         # 300秒内至少10次修改
save 60 10000       # 60秒内至少10000次修改

# AOF
appendonly yes
appendfsync everysec    # 每秒同步（推荐）
```

### 12. Redis 缓存常见问题？

**答：**

| 问题 | 描述 | 解决方案 |
|------|------|---------|
| **缓存穿透** | 查询不存在的数据，绕过缓存直达数据库 | 布隆过滤器、缓存空值 |
| **缓存击穿** | 热点 key 过期，大量请求涌入数据库 | 互斥锁、热点 key 永不过期 |
| **缓存雪崩** | 大量 key 同时过期，数据库压力激增 | 过期时间加随机值、多级缓存 |

```
缓存穿透:  Client → Cache(miss) → DB(不存在) → 无结果
解决: Client → 布隆过滤器 → Cache → DB

缓存击穿:  热点key过期 → 大量并发 → DB 压力
解决: 加分布式锁，只允许一个请求回源

缓存雪崩:  大量key同时过期 → DB 雪崩
解决: 过期时间 = 基础时间 + random(0, 300)
```

## 六、MongoDB

### 13. MongoDB 的特点和适用场景？

**答：**

**特点：**
- 文档型数据库（JSON/BSON 格式）
- 灵活的 Schema（同一集合中文档结构可以不同）
- 水平扩展（分片 Sharding）
- 内置复制集高可用
- 丰富的查询和聚合功能

**适用场景：**
| 场景 | 原因 |
|------|------|
| 内容管理 | 灵活 Schema |
| 日志存储 | 写入性能好，支持 TTL |
| IoT 数据 | 高写入吞吐 |
| 实时分析 | 聚合管道 |
| 游戏后端 | 灵活数据结构 |

**不适用场景：**
- 强事务需求（如金融交易）→ MySQL/PostgreSQL
- 复杂多表关联 → 关系型数据库

### 14. MongoDB 副本集 (Replica Set) 的架构？

**答：**

```
  Primary (主节点)
  ├── 处理读写请求
  ├── oplog (操作日志)
  │
  ├──→ Secondary 1 (从节点)
  │    └── 同步 oplog，可处理读请求
  │
  ├──→ Secondary 2 (从节点)
  │    └── 同步 oplog，可处理读请求
  │
  └──→ Arbiter (仲裁节点，可选)
       └── 仅参与选举，不存储数据
```

- 最少 3 个节点（推荐奇数个）
- 主节点故障时自动选举新主节点（Raft 协议）
- 支持读写分离（readPreference 配置）

## 七、数据库运维

### 15. 数据库备份策略？

**答：**

| 策略 | 说明 | 工具 |
|------|------|------|
| **全量备份** | 完整数据备份 | mysqldump, xtrabackup, mongodump |
| **增量备份** | 只备份变化数据 | xtrabackup, binlog |
| **逻辑备份** | SQL 语句形式导出 | mysqldump (SQL文本) |
| **物理备份** | 直接复制数据文件 | xtrabackup (InnoDB物理文件) |

```bash
# MySQL 逻辑备份
mysqldump -u root -p --single-transaction --routines \
  --all-databases > full_backup.sql

# MySQL 使用 xtrabackup 物理备份
xtrabackup --backup --target-dir=/backup/full

# Redis 备份
redis-cli BGSAVE          # 手动触发 RDB
cp /var/lib/redis/dump.rdb /backup/

# MongoDB 备份
mongodump --uri="mongodb://localhost:27017" --out=/backup/
```

**备份策略建议：**
- 全量备份（每天/每周）+ 增量备份（每小时）
- 备份文件异地存储（不同机房/云区域）
- 定期做恢复演练验证备份有效性
- 自动化监控备份状态和告警

### 16. 如何做数据库的读写分离？

**答：**

```
          Write            Read
Client ──→ Proxy ──→ Master ──→ Slave 1
                              ↘ Slave 2
                              ↘ Slave 3
```

**常用方案：**
| 方案 | 工具 | 特点 |
|------|------|------|
| 代理层 | ProxySQL, MaxScale | 透明，应用无感知 |
| 应用层 | Spring + 多数据源 | 灵活，需要代码改造 |
| 云托管 | AWS RDS Read Replica, 阿里云 RDS | 免运维 |

**注意事项：**
- 主从延迟可能导致读到旧数据
- 关键业务读操作强制走主库
- 监控主从延迟指标 (`Seconds_Behind_Master`)
