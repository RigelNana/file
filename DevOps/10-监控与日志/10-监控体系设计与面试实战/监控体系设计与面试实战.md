---

## 1. 如何从零搭建监控体系？

**回答：**

```
监控体系搭建路径:

Phase 1: 基础监控 (Week 1-2)
  □ 部署 kube-prometheus-stack (Prometheus + Grafana + Alertmanager)
  □ node-exporter → 节点指标 (CPU, 内存, 磁盘)
  □ 基础告警规则 (实例宕机, 磁盘满, 内存高)
  □ 导入通用 Dashboard (Node Exporter Full)

Phase 2: 应用监控 (Week 3-4)
  □ 应用内置 /metrics 端点 (Prometheus client library)
  □ ServiceMonitor 自动发现
  □ RED 指标: Rate, Errors, Duration
  □ 业务指标: 订单数, 用户数, 转化率
  □ Blackbox Exporter → 外部探测 (HTTP, SSL 证书)

Phase 3: 日志系统 (Week 5-6)
  □ Loki + Promtail (或 ELK)
  □ 结构化日志标准
  □ 日志级别规范
  □ Grafana 关联 Metrics ↔ Logs

Phase 4: 链路追踪 (Week 7-8)
  □ OpenTelemetry SDK 接入
  □ Tempo / Jaeger 部署
  □ Exemplar 关联 (Metrics → Traces)
  □ 关键路径追踪

Phase 5: 完善告警 (Week 9-10)
  □ 告警分级 (P0-P3)
  □ 值班轮换 (PagerDuty/OpsGenie)
  □ Runbook 文档
  □ 告警通道 (Slack + PagerDuty + Email)
  □ 定期回顾和优化
```

指标监控:
  ┌──────────────┬────────────────┬──────────────────┬────────────────┐
  │ 维度          │ Prometheus     │ Datadog          │ CloudWatch     │
  ├──────────────┼────────────────┼──────────────────┼────────────────┤

  │ HA/扩展       │ Thanos/Mimir   │ 内置              │ 内置            │

  └──────────────┴────────────────┴──────────────────┴────────────────┘

日志系统:
  ┌──────────────┬────────────────┬──────────────────┬────────────────┐
  │ 维度          │ ELK Stack      │ Grafana Loki     │ Splunk         │
  ├──────────────┼────────────────┼──────────────────┼────────────────┤
  │ 索引方式      │ 全文索引        │ 标签索引          │ 全文索引        │
  │ 成本          │ 中             │ 低                │ 高             │
  │ 搜索速度      │ 快             │ 标签快/全文慢      │ 快             │
  │ 资源消耗      │ 高             │ 低                │ 高             │

推荐组合:
  开源栈: Prometheus + Loki + Tempo + Grafana (Grafana Stack)

  AWS:    CloudWatch + X-Ray (或混合 Prometheus)

## 3. 事故响应 (Incident Response) 流程？

**回答：**

```
事故响应流程:

1. 检测 (Detect)
   告警触发 → 值班人员收到通知
   自动分类: P0/P1/P2/P3

2. 响应 (Respond)
   确认告警 → 开始调查
   P0/P1: 5-15 分钟内响应
   P2:    30 分钟内响应
   创建事故频道 (Slack #incident-xxx)

3. 诊断 (Diagnose)
   查看 Dashboard → 定位异常指标
   查看日志 → 找到错误信息
   查看链路追踪 → 定位延迟瓶颈
   关联最近变更 → 是否有部署/配置变更

4. 缓解 (Mitigate)
   目标: 快速恢复, 不是根因分析
   常用操作:
     回滚部署 (kubectl rollout undo)
     扩容 (kubectl scale / HPA)
     流量切换 (DNS / LB)
     重启服务
     临时修复 (Feature Flag)

5. 恢复 (Resolve)
   确认服务恢复正常
   验证指标恢复基线
   关闭告警
   更新状态页

6. 复盘 (Post-mortem)
   事故时间线
   根因分析 (5 Whys)
   影响范围
   后续行动项 (Action Items)
   无指责文化 (Blameless)
```

## 4. SRE On-Call 最佳实践？

**回答：**

```
On-Call 体系:

值班轮换:
  工具: PagerDuty, OpsGenie, Grafana OnCall
  模式: 
    Primary + Secondary 双人值班
    每周轮换
    时区跨越: Follow-the-Sun 模式

升级策略 (Escalation):
  0 min:   通知 Primary On-Call (Push + Slack)
  5 min:   未确认 → 通知 Secondary
  15 min:  未响应 → 通知 Team Lead
  30 min:  P0 未解决 → 通知 Engineering Manager

On-Call 卫生:
  ✓ 每次值班前检查告警是否正常
  ✓ 确保 VPN/laptop 可用
  ✓ 交接班记录未解决问题
  ✓ 值班后统计: 告警数, 事故数, MTTA, MTTR
  ✓ 定期 Gameday 演练

告警统计:
  MTTA (Mean Time to Acknowledge): 平均确认时间
  MTTR (Mean Time to Resolve):     平均恢复时间
  
  目标:
    P0: MTTA < 5min, MTTR < 30min
    P1: MTTA < 15min, MTTR < 2h
    P2: MTTA < 1h, MTTR < 24h
```

## 5. 容量规划 (Capacity Planning)？

**回答：**

```
容量规划: 预测未来资源需求, 提前扩容

方法:
  1. 基于趋势预测
     predict_linear(node_filesystem_avail_bytes[30d], 7*86400)
     # 基于 30 天趋势, 预测 7 天后磁盘剩余空间
     
  2. 基于增长率
     当前 QPS × 预期增长率 → 所需资源
     100 QPS → 6 个月后 200 QPS → 需要翻倍资源

  3. 基于压力测试
     Locust/k6 压测 → 找到单实例瓶颈
     → 根据目标 QPS 计算所需实例数

关键容量指标:
  ┌──────────────┬──────────────────────────────────────┐
  │ 资源          │ 指标 / 告警                           │
  ├──────────────┼──────────────────────────────────────┤
  │ CPU          │ 使用率 > 70% 持续 → 考虑扩容          │
  │ 内存          │ 使用率 > 80% → 扩容                  │
  │ 磁盘          │ predict_linear 预测满                 │
  │ 网络带宽      │ 利用率 > 60% → 升级                  │
  │ 数据库连接    │ 连接数 > 80% max_connections          │
  │ K8s Pod      │ HPA 频繁到达 maxReplicas              │
  └──────────────┴──────────────────────────────────────┘

成本优化:
  使用率过低 → 缩容 / 降配
  使用 Spot Instance 非关键 workload
  请求/限制合理设置 (VPA 推荐)
  按需 vs 预留实例
```

## 6. APM (应用性能管理) 是什么？

**回答：**

```
APM = Application Performance Management

三大能力:
  1. 指标监控: RED (Rate, Errors, Duration)
  2. 分布式追踪: 请求链路和延迟分解
  3. 日志关联: 结合日志定位问题

APM 产品对比:
  ┌──────────────┬──────────────┬──────────────────────────┐
  │ 工具          │ 类型          │ 特点                     │
  ├──────────────┼──────────────┼──────────────────────────┤
  │ Datadog APM  │ SaaS 商业     │ 全功能, 自动注入          │
  │ New Relic    │ SaaS 商业     │ 全功能, 免费层            │
  │ Elastic APM  │ 开源+商业     │ 与 ELK 集成, Agent 侵入  │
  │ Jaeger       │ 开源          │ CNCF, 专注追踪           │
  │ Tempo        │ 开源          │ Grafana 生态, 低成本     │
  │ SkyWalking   │ 开源          │ Apache, Java 生态        │
  │ OpenTelemetry│ 开源标准      │ 统一 SDK, 多后端          │
  └──────────────┴──────────────┴──────────────────────────┘

OpenTelemetry 已成为事实标准:
  应用 → OTel SDK → OTel Collector → 多后端
                        ├→ Prometheus (Metrics)
                        ├→ Loki (Logs)
                        └→ Tempo/Jaeger (Traces)
```

## 7. 面试场景题：线上服务变慢怎么排查？

**回答：**

```
排查思路 (由外到内):

Step 1: 确认范围
  Q: 所有用户还是部分？所有接口还是特定？
  看: Grafana Dashboard → QPS, P99, Error Rate
  判断: 全局性 vs 局部性

Step 2: 看基础设施
  □ 节点 CPU/内存/磁盘/网络 → node-exporter Dashboard
  □ 是否有节点 NotReady
  □ Pod 是否有 OOM / CrashLooping
  □ 是否有 CPU Throttling

Step 3: 看应用层
  □ 应用 QPS 是否突增
  □ 错误率是否升高
  □ P99 延迟是否异常
  □ 最近是否有部署 → 回滚验证

Step 4: 看依赖
  □ 数据库慢查询 → MySQL Exporter
  □ Redis 命中率下降 → Redis Exporter
  □ 外部 API 延迟 → Blackbox Exporter
  □ MQ 积压 → Kafka/RabbitMQ Exporter

Step 5: 链路追踪
  □ Jaeger/Tempo → 找到慢请求
  □ Span 分析 → 哪个服务/哪步最慢
  □ 数据库调用数量 (N+1 问题)

Step 6: 日志
  □ Loki/ES → 搜索 ERROR/WARN 日志
  □ 关联 TraceID → 看完整请求日志
  □ 看是否有异常堆栈

Step 7: 缓解
  □ 扩容 → HPA / kubectl scale
  □ 回滚 → kubectl rollout undo
  □ 限流/降级 → 保护核心功能
```

## 8. 面试场景题：如何设计一个高可用监控系统？

**回答：**

```
高可用监控设计:

指标层 HA:
  Prometheus:
    方案1: 双副本 + Thanos Query (去重)
    方案2: remote_write → Mimir/VictoriaMetrics (多副本)
  
  Thanos 架构:
    2× Prometheus + Sidecar → S3 (持久存储)
    Thanos Query → 统一查询 + 去重
    Thanos Compact → 降采样 (长期存储)

告警层 HA:
  Alertmanager: 3 副本, Gossip 协议
  Prometheus 向所有 AM 发送告警
  AM 集群内部去重, 只发一次通知

日志层 HA:
  Loki: Simple Scalable 模式
    Write: 3 副本 (Distributor + Ingester)
    Read:  2 副本 (Querier)
    Backend: S3 (持久存储)
  
  ELK:
    ES: 3 Master + N Data + 副本分片
    Kafka 缓冲层

可视化层 HA:
  Grafana: 2+ 副本 + 外部数据库 (MySQL/PG)
  LB/Ingress 前面均衡

数据持久性:
  指标: S3 + Thanos (长期) + 本地 SSD (短期)
  日志: S3 (长期) + 本地 (短期)
  配置: Git (Dashboard JSON, Alert Rules)

故障隔离:
  监控系统和业务系统分离
  独立集群/独立账号
  监控系统自身也要监控 (Meta-monitoring)
```

## 9. 面试场景题：日志量太大怎么处理？

**回答：**

```
日志量控制策略:

1. 源头减量
   ✓ 合理日志级别 (生产 INFO+, 不要 DEBUG)
   ✓ 采样日志 (高频路径只记 1/100)
   ✓ 避免循环中打日志
   ✓ 避免记录大 payload (body, SQL 结果集)

2. 采集层过滤
   # Fluent Bit 丢弃不需要的日志
   [FILTER]
       Name   grep
       Match  *
       Exclude log healthcheck

   # Promtail 丢弃
   pipeline_stages:
     - drop:
         expression: '.*healthcheck.*'

3. 分级存储
   Hot (0-7天):   SSD, 全量
   Warm (7-30天): HDD, 只保留 ERROR/WARN
   Cold (30-90天): S3, 压缩归档
   Delete (>90天): 自动删除

4. 日志压缩
   Loki: 自动 gzip/snappy 压缩 (10-20x)
   ES: best_compression (zlib)
   归档: S3 Glacier (成本极低)

5. 索引优化 (ELK)
   不需要搜索的字段 → index: false
   不需要分词的字段 → keyword
   减少副本数 (warm 阶段 → 1 副本)
   ILM 自动管理生命周期

6. 架构优化
   Kafka 缓冲 → 削峰填谷
   多目的地: 重要日志 → ES (可搜索)
              全量日志 → S3 (归档备查)

日志量估算:
  1000 Pod × 100 行/秒 × 200 bytes/行
  = 20 MB/s = 1.7 TB/天
  
  压缩后: ~170 GB/天
  30 天保留: ~5 TB
```

## 10. 面试高频问答总结？

**回答：**

```
Q: Prometheus 为什么用 Pull 模式?
A: 自带健康检查, 便于调试 (curl /metrics), 集中控制

Q: Counter 和 Gauge 区别?
A: Counter 只增不减 (请求数), 需 rate()
   Gauge 可增可减 (CPU%), 直接用

Q: rate 和 irate 选哪个?
A: rate → 告警+趋势 (平滑)
   irate → 实时面板 (灵敏)

Q: Histogram 和 Summary 选哪个?
A: Histogram (95% 场景), 可聚合, 服务端计算

Q: 什么是高基数 (Cardinality)?
A: 标签唯一值太多 → 时间序列爆炸 → OOM
   避免 user_id, trace_id 作为 Metric Label

Q: Prometheus 存不下怎么办?
A: 远程存储: Thanos (S3), Mimir, VictoriaMetrics

Q: ELK 和 Loki 怎么选?
A: 需要强全文搜索 → ELK
   已有 Grafana + 预算有限 → Loki

Q: 告警太多怎么办?
A: 分组 (group_by), 抑制 (inhibit), 调阈值
   定期回顾清理, 只告可操作的

Q: 如何实现 Metrics → Logs → Traces 关联?
A: 统一标签体系 + Exemplar + Grafana 多数据源关联
   Metrics (traceID exemplar) → Traces
   Logs (derived field traceID) → Traces

Q: 监控系统自身挂了怎么办?
A: Meta-monitoring (监控的监控)
   独立的简单检查 (cron + curl + 判断)
   云厂商告警 (CloudWatch Alarm 监控 Prometheus)
   Watchdog 告警 (一直 firing 的告警, 不 firing = 有问题)

Q: SLO 99.9% 和 99.99% 差多少?
A: 99.9%:  每月 43.2 分钟宕机
   99.99%: 每月 4.3 分钟宕机
   差 10 倍, 成本指数增长

Q: On-Call 收到告警第一步做什么?
A: 1. 确认影响范围 (看 Dashboard)
   2. 判断是否需要升级
   3. 查看最近变更 (部署/配置)
   4. 快速缓解 (回滚/扩容)
   5. 再查根因
```
