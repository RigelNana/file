# 可观测性基础与方法论

---

## 1. 什么是可观测性 (Observability)？与监控有什么区别？

**回答：**

```
可观测性 (Observability):
  定义: 通过系统的外部输出来推断系统内部状态的能力
  来源: 控制理论 (Control Theory)
  核心: 系统能否回答「发生了什么？为什么？」

监控 (Monitoring):
  定义: 收集、处理、展示预定义的指标和日志
  核心: 系统是否在正常运行？

可观测性 vs 监控:
  ┌─────────────┬──────────────────────┬──────────────────────┐
  │   维度       │  监控                │  可观测性             │
  ├─────────────┼──────────────────────┼──────────────────────┤
  │  思路        │ 已知的未知 (Known    │ 未知的未知 (Unknown  │
  │             │ Unknowns)            │ Unknowns)            │
  │  方式        │ 预定义 Dashboard     │ 自由探索和查询        │
  │  目的        │ 检测已知故障         │ 调试未知问题          │
  │  数据        │ 固定指标             │ 高基数 (high cardinality) │
  │  演进        │ 监控是可观测性的子集  │ 包含监控 + 更多能力    │
  └─────────────┴──────────────────────┴──────────────────────┘

面试关键点:
  监控告诉你 "系统挂了"
  可观测性告诉你 "为什么挂了，哪里挂了"
```

## 2. 可观测性三大支柱是什么？

**回答：**

```
三大支柱 (Three Pillars):

1. Metrics（指标）
   特点: 可聚合的数值型时间序列数据
   示例: CPU 使用率、请求量、延迟 P99
   工具: Prometheus, Datadog, CloudWatch, InfluxDB
   优势: 存储成本低, 查询速度快, 适合告警
   局限: 缺乏上下文, 无法看到单个请求

2. Logs（日志）
   特点: 离散的、带时间戳的事件记录
   示例: "2024-01-15 10:00:05 ERROR UserService.login() user_id=123 timeout"
   工具: ELK Stack, Loki, Fluentd, Splunk
   优势: 包含丰富上下文, 适合调试
   局限: 数据量大, 存储成本高, 缺乏全局视角

3. Traces（链路追踪）
   特点: 请求在分布式系统中的完整调用路径
   示例: 用户请求 → API Gateway → UserService → DB → Cache
   工具: Jaeger, Zipkin, Tempo, AWS X-Ray, OpenTelemetry
   优势: 跨服务追踪, 定位瓶颈
   局限: 实施复杂, 采样率影响完整性

三者关系:
  Metrics → 发现问题 (CPU 异常升高)
  Logs    → 定位原因 (哪个错误导致 CPU 升高)
  Traces  → 追踪路径 (问题请求经过了哪些服务)
```

```
请求进入系统:
  ├── Metrics: 请求量 +1, 延迟记录 150ms
  ├── Logs:    "INFO POST /api/order user=123 status=500 latency=150ms"
  └── Traces:  
      TraceID: abc123
      ├── API Gateway  (20ms)
      ├── OrderService (50ms)
      ├── PaymentService (70ms) ← 超时
      └── InventoryService (10ms)
```

## 3. SLI、SLO、SLA 分别是什么？它们之间的关系？

**回答：**

```
SLI (Service Level Indicator) — 服务等级指标
  定义: 衡量服务质量的具体指标
  示例:
    可用性 SLI:  成功请求数 / 总请求数
    延迟 SLI:    延迟 < 200ms 的请求比例
    吞吐量 SLI:  每秒处理请求数
    错误率 SLI:  5xx 错误占比

SLO (Service Level Objective) — 服务等级目标
  定义: 基于 SLI 设定的目标值
  示例:
    可用性: 99.95% (每月停机 ≤ 21.9 分钟)
    延迟:   P99 < 200ms
    错误率: < 0.1%
  设定原则:
    不追求 100% (成本指数增长)
    基于用户体验设定
    留有 Error Budget (错误预算)

SLA (Service Level Agreement) — 服务等级协议
  定义: 与客户签订的合同, SLO + 违约惩罚
  示例: "可用性不低于 99.9%, 否则退还月费 10%"
  特点: 通常比 SLO 宽松 (Internal SLO < External SLA)
```

```
关系:
  SLI (实际测量) → SLO (内部目标) → SLA (客户协议)

  Example:
    SLI: 当前可用性 = 99.97%
    SLO: 内部目标   = 99.95% (达标 ✓)
    SLA: 客户协议   = 99.9%  (达标 ✓)
```

## 4. Error Budget（错误预算）是什么？如何使用？

**回答：**

```
Error Budget = 1 - SLO

SLO = 99.95%
Error Budget = 0.05% = 每月 21.9 分钟宕机

         ┌───────────────────────────────────────┐
月初      │██████████████████████████████████ 100% │ Error Budget
         └───────────────────────────────────────┘
         ┌───────────────────────────────────────┐
月中      │█████████████████████░░░░░░░░░░░░░ 65%  │ 用了一部分
         └───────────────────────────────────────┘
         ┌───────────────────────────────────────┐
月底      │████░░░░░░░░░░░░░░░░░░░░░░░░░░░░░ 12%  │ 快耗尽
         └───────────────────────────────────────┘

Error Budget 策略:
  预算充足:  可以发布新功能, 做实验
  预算紧张:  暂停新功能发布, 优先稳定性
  预算耗尽:  冻结变更, 全力修复

Error Budget 计算:
  30 天 × 24 小时 × 60 分钟 = 43200 分钟
  ┌─────────┬──────────────────────────────┐
  │  SLO    │  每月允许宕机时间             │
  ├─────────┼──────────────────────────────┤
  │  99%    │  7 小时 12 分钟               │
  │  99.9%  │  43.2 分钟                   │
  │  99.95% │  21.9 分钟                   │
  │  99.99% │  4.3 分钟                    │
  └─────────┴──────────────────────────────┘
```

## 5. 四个黄金指标 (Four Golden Signals) 是什么？

**回答：**

```
Google SRE 提出, 适用于所有面向用户的系统:

1. Latency（延迟）
   定义: 处理请求所需时间
   关注: 成功请求延迟 vs 失败请求延迟 (分开看)
   指标: P50, P90, P95, P99
   示例: P99 = 200ms (99% 的请求在 200ms 内完成)

2. Traffic（流量）
   定义: 系统承受的请求量/负载
   指标: QPS (每秒查询), RPS (每秒请求), 并发连接数
   示例: Web = HTTP QPS, 流媒体 = 并发会话数

3. Errors（错误率）
   定义: 请求失败的比率
   类型:
     显式错误: HTTP 5xx
     隐式错误: 返回 200 但内容错误
     策略错误: 延迟超过 SLO 视为错误
   指标: 5xx 占比, 错误 QPS

4. Saturation（饱和度）
   定义: 系统资源使用程度
   指标: CPU%, 内存%, 磁盘 I/O, 队列深度
   关键: 饱和度预警 (在耗尽前告警)
   建议: CPU > 80%, 内存 > 85% 告警
```

## 6. USE 方法和 RED 方法？

**回答：**

```
USE 方法 — 针对资源 (Resource-focused)
  提出: Brendan Gregg
  适用: CPU, 内存, 磁盘, 网络

  U: Utilization (利用率) — 资源忙碌的时间比例
  S: Saturation (饱和度)  — 排队等待的工作量
  E: Errors (错误)       — 错误事件数量

  示例:
  ┌──────────┬────────────────┬───────────────────┬────────────┐
  │ 资源      │ Utilization    │ Saturation        │ Errors     │
  ├──────────┼────────────────┼───────────────────┼────────────┤
  │ CPU      │ CPU% = 75%     │ Run Queue Length   │ ECC errors │
  │ 内存      │ Used% = 80%    │ Swap 使用量        │ OOM kills  │
  │ 磁盘      │ IO% = 60%      │ IO Queue Length   │ 坏块数      │
  │ 网络      │ 带宽利用率      │ 丢包率            │ CRC errors │
  └──────────┴────────────────┴───────────────────┴────────────┘


RED 方法 — 针对服务 (Service-focused)
  提出: Tom Wilkie (Grafana Labs)
  适用: 微服务, API, Web 服务

  R: Rate (速率)    — 每秒请求数
  E: Errors (错误率) — 每秒错误请求数
  D: Duration (时长) — 请求处理时间分布

  示例:
  ┌──────────────┬─────────────────────────────┐
  │ Rate         │ sum(rate(http_requests[5m])) │
  │ Errors       │ sum(rate(http_errors[5m]))   │
  │ Duration     │ histogram_quantile(0.99, ...) │
  └──────────────┴─────────────────────────────┘


四个黄金指标 vs USE vs RED:
  ┌─────────────┬──────────────┬──────────────┐
  │ 黄金指标     │  USE         │  RED         │
  ├─────────────┼──────────────┼──────────────┤
  │ 所有系统     │ 基础设施      │ 微服务       │
  │ Google SRE  │ Brendan Gregg │ Tom Wilkie  │
  │ 4 个指标     │ 3 个维度      │ 3 个维度     │
  └─────────────┴──────────────┴──────────────┘
```

## 7. 什么是高基数 (High Cardinality)？为什么重要？

**回答：**

```
基数 (Cardinality) = 某个标签/属性的唯一值数量

低基数:  status_code (200, 301, 404, 500) → 4 个值
中基数:  region (us-east-1, eu-west-1, ...) → ~20 个值
高基数:  user_id  → 百万级唯一值
超高基数: trace_id → 每个请求一个唯一值

高基数为什么重要:
  可观测性: 高基数 = 更细粒度 = 更好的调试能力
  成本:     高基数 = 更多时间序列 = 更多存储和内存

  Prometheus 痛点:
    每个唯一标签组合 = 1 条时间序列
    http_requests{method, status, endpoint, user_id}
    → 如果 user_id 有 100 万用户 → 100 万条时间序列 → OOM

解决方案:
  ┌─────────────────────────────┬─────────────────────────┐
  │ 方案                        │ 说明                     │
  ├─────────────────────────────┼─────────────────────────┤
  │ 避免高基数标签在 Metrics 中   │ user_id 放 Logs/Traces  │
  │ 使用 Exemplar               │ Metrics 关联 Trace      │
  │ 采用更适合的工具              │ Datadog 支持高基数       │
  │ 预聚合 (recording rules)    │ 减少查询时计算量          │
  └─────────────────────────────┴─────────────────────────┘
```

## 8. OpenTelemetry (OTel) 是什么？

**回答：**

```
OpenTelemetry:
  定义: CNCF 开源项目, 提供统一的可观测性数据收集标准
  目标: 一套 SDK, 收集 Metrics + Logs + Traces
  前身: OpenTracing + OpenCensus 合并

组件:
  ┌────────────────┬─────────────────────────────┐
  │ API            │ 定义接口和数据模型            │
  │ SDK            │ 实现 API (多语言)             │
  │ Collector      │ 接收、处理、导出数据           │
  │ 自动插桩        │ 自动注入追踪代码              │
  └────────────────┴─────────────────────────────┘

架构:
  App (SDK) → OTel Collector → Backend
                │
                ├── Prometheus (Metrics)
                ├── Jaeger (Traces)
                └── Loki (Logs)

OTel Collector 配置:
  receivers:    接收数据 (OTLP, Jaeger, Prometheus)
  processors:   处理数据 (批量, 采样, 过滤)
  exporters:    导出数据 (Prometheus, Jaeger, OTLP)
  pipelines:    连接 receivers → processors → exporters
```

```yaml
# otel-collector-config.yaml
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318
  prometheus:
    config:
      scrape_configs:
        - job_name: 'app'
          static_configs:
            - targets: ['app:8080']

processors:
  batch:
    timeout: 5s
    send_batch_size: 1000
  memory_limiter:
    check_interval: 1s
    limit_mib: 512

exporters:
  prometheus:
    endpoint: 0.0.0.0:8889
  jaeger:
    endpoint: jaeger:14250
    tls:
      insecure: true

service:
  pipelines:
    metrics:
      receivers: [otlp, prometheus]
      processors: [memory_limiter, batch]
      exporters: [prometheus]
    traces:
      receivers: [otlp]
      processors: [memory_limiter, batch]
      exporters: [jaeger]
```

## 9. 结构化日志 vs 非结构化日志？

**回答：**

```
非结构化日志 (传统):
  2024-01-15 10:00:05 ERROR [UserService] Login failed for user john from 10.0.0.1

结构化日志 (推荐):
  {
    "timestamp": "2024-01-15T10:00:05Z",
    "level": "ERROR",
    "service": "UserService",
    "message": "Login failed",
    "user": "john",
    "ip": "10.0.0.1",
    "trace_id": "abc123",
    "request_id": "req-456"
  }

对比:
  ┌──────────────┬────────────────────┬────────────────────┐
  │ 维度          │ 非结构化            │ 结构化              │
  ├──────────────┼────────────────────┼────────────────────┤
  │ 可读性        │ 人类友好            │ 机器友好            │
  │ 解析          │ 需要 Grok/正则      │ 原生 JSON 解析      │
  │ 搜索          │ 全文搜索            │ 字段精确查询         │
  │ 关联          │ 困难               │ trace_id 轻松关联    │
  │ 存储          │ 较小               │ 稍大 (字段名重复)    │
  │ 趋势          │ 逐渐被替代          │ 业界标准             │
  └──────────────┴────────────────────┴────────────────────┘

日志级别:
  TRACE → DEBUG → INFO → WARN → ERROR → FATAL
  
  生产环境建议: INFO 及以上
  调试时临时:   DEBUG
  避免: 过多 DEBUG 日志影响性能
```

## 10. 如何设计监控告警分级体系？

**回答：**

```
告警分级:
  ┌──────────┬──────────────────────────┬──────────────────────┐
  │ 级别      │ 定义                     │ 响应时间 / 通知方式    │
  ├──────────┼──────────────────────────┼──────────────────────┤
  │ P0/Fatal │ 服务完全不可用            │ 立即, 电话 + 短信      │
  │ P1/Critical│ 核心功能严重受影响       │ 15 分钟, PagerDuty    │
  │ P2/Warning│ 性能下降或次要功能异常    │ 30 分钟, Slack        │
  │ P3/Info  │ 需要关注但不紧急          │ 工作日处理, Email      │
  └──────────┴──────────────────────────┴──────────────────────┘

告警设计原则:
  1. 可操作性: 每条告警必须有明确的下一步操作
  2. 避免疲劳: 宁缺毋滥, 不重要的不告警
  3. 合适阈值: 基于基线 (baseline) 而非拍脑袋
  4. 分组收敛: 相同问题合并 (Alertmanager group_by)
  5. 逐步升级: Warning → Critical → 自动 Escalation
  6. 告警包含: What (什么问题) + Where (哪里) + Runbook (如何处理)

Runbook 模板:
  告警名:     HighCPUUsage
  描述:       节点 CPU 使用率 > 80% 持续 10 分钟
  影响:       可能导致请求延迟增加
  排查步骤:
    1. 登录节点, top 查看进程
    2. 检查是否有异常进程
    3. 检查最近是否有部署
  恢复操作:   kill 异常进程 / 扩容 / 回滚部署
  升级条件:   15 分钟未恢复, 升级到 P1
```
