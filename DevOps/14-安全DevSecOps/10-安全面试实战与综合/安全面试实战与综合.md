## 1. 安全事件应急响应流程？

**回答：**

```
应急响应六步法 (NIST SP 800-61):

  1. 准备 (Preparation)
     ├── 建立 IRT (Incident Response Team)
     ├── 制定应急预案 (Playbook)
     ├── 部署安全监控 (SIEM / Falco / WAF)
     ├── 定期演练 (红蓝对抗 / 桌面演练)
     └── 准备取证工具和隔离脚本

  2. 检测与分析 (Detection & Analysis)
     ├── 告警触发 (SIEM / CloudTrail / Falco)
     ├── 日志分析 (ELK / Splunk)
     ├── 确认是否误报
     ├── 评估影响范围 (受影响系统/数据/用户)
     └── 分级: P1 (数据泄露) / P2 (入侵) / P3 (异常行为)

  3. 遏制 (Containment)
     ├── 短期: 隔离受影响系统 (网络隔离/停止服务)
     │   kubectl cordon <node>
     │   kubectl delete pod <compromised-pod>
     │   iptables -A INPUT -s <attacker-ip> -j DROP
     ├── 长期: 加固配置, 修复漏洞
     └── 保留证据 (snapshot / 镜像快照)

  4. 消除 (Eradication)
     ├── 清除恶意程序 / 后门
     ├── 修复漏洞 (补丁 / 代码修复)
     ├── 重置被泄露的凭证 (密码/Token/Key 全部轮换)
     └── 容器场景: 销毁受影响容器, 重新构建镜像

  5. 恢复 (Recovery)
     ├── 从备份恢复数据 (验证完整性)
     ├── 逐步恢复服务 (灰度放量)
     ├── 加强监控 (增加告警规则)
     └── 验证修复有效性

  6. 复盘 (Lessons Learned)
     ├── 时间线梳理 (从发现到恢复)
     ├── 根因分析 (5 Why)
     ├── 改进措施 (技术 + 流程)
     ├── 更新应急预案
     └── 输出安全事件报告
```

**DevOps 视角的快速响应：**
```bash
# 自动化响应脚本示例

# 1. 隔离被入侵的 Pod
kubectl label pod <pod> quarantine=true
kubectl apply -f deny-all-network-policy.yaml

# 2. 保留证据快照
kubectl logs <pod> > /evidence/pod-logs-$(date +%s).log
kubectl get pod <pod> -o yaml > /evidence/pod-spec-$(date +%s).yaml

# 3. 封禁攻击者 IP (WAF / Security Group)
aws ec2 revoke-security-group-ingress \
  --group-id sg-xxx \
  --protocol tcp --port 443 --cidr <attacker-ip>/32

# 4. 快速回滚到安全版本
kubectl rollout undo deployment/<app>

# 5. 轮换被泄露的凭证
vault lease revoke -prefix secret/data/app/
vault write secret/data/app/db password=$(openssl rand -base64 32)
```

## 2. 红蓝对抗与渗透测试？

**回答：**

```
红蓝对抗:
  红队 (Red Team):  模拟攻击者, 尝试入侵
  蓝队 (Blue Team): 防守方, 检测和响应
  紫队 (Purple):     红蓝协作, 提升双方能力

渗透测试流程:
  1. 信息收集 (Recon)
     ├── 被动: DNS, WHOIS, 搜索引擎, GitHub
     └── 主动: Nmap 端口扫描, 服务识别
  
  2. 漏洞扫描
     ├── Nessus / OpenVAS (主机漏洞)
     ├── OWASP ZAP / Burp Suite (Web 漏洞)
     └── Trivy / Grype (容器/依赖漏洞)
  
  3. 漏洞利用
     ├── 验证漏洞可利用性 (PoC)
     └── 获取访问权限
  
  4. 后渗透
     ├── 权限提升
     ├── 横向移动
     └── 数据提取 (验证影响)
  
  5. 报告
     ├── 发现的漏洞清单
     ├── 风险等级 (CVSS 评分)
     ├── 修复建议
     └── 复测验证

DevOps 中的安全测试:
  ┌──────────────────┬─────────────────────────────┐
  │ 阶段              │ 安全活动                     │
  ├──────────────────┼─────────────────────────────┤
  │ 代码提交          │ git-secrets / gitleaks      │
  │ CI 构建           │ SAST (SonarQube / Semgrep)  │
  │ 依赖安装          │ SCA (Snyk / Trivy fs)       │
  │ 镜像构建          │ 镜像扫描 (Trivy image)      │
  │ 预发布            │ DAST (OWASP ZAP)            │
  │ 部署              │ K8s 准入策略 (OPA/Kyverno)  │
  │ 运行时            │ Falco 行为检测              │
  │ 定期              │ 渗透测试 (季度/年度)        │
  └──────────────────┴─────────────────────────────┘
```

## 3. 供应链安全？

**回答：**

```
软件供应链攻击:
  恶意依赖包 (npm / PyPI 投毒)
  CI/CD 流水线被入侵
  构建工具被篡改
  镜像被植入后门

防护措施:
  ┌──────────────────┬──────────────────────────────┐
  │ 环节              │ 措施                          │
  ├──────────────────┼──────────────────────────────┤
  │ 依赖管理          │ lock 文件 (package-lock.json) │
  │                   │ 依赖漏洞扫描 (Snyk/Trivy)    │
  │                   │ 私有 Registry 代理            │
  │                   │ 固定依赖版本 (不用 latest)    │
  ├──────────────────┼──────────────────────────────┤
  │ 构建安全          │ 可重复构建 (Reproducible)     │
  │                   │ 构建环境隔离 (沙箱)           │
  │                   │ SBOM 软件物料清单             │
  ├──────────────────┼──────────────────────────────┤
  │ 签名验证          │ 代码签名 (GPG / Sigstore)     │
  │                   │ 镜像签名 (Cosign)             │
  │                   │ Helm Chart 签名 (Provenance) │
  ├──────────────────┼──────────────────────────────┤
  │ CI/CD 安全        │ Runner 隔离 (不共享)          │
  │                   │ 最小权限 Token                │
  │                   │ 审计日志 (谁改了什么)         │
  └──────────────────┴──────────────────────────────┘

SBOM (Software Bill of Materials):
  软件物料清单 — 列出所有组件和依赖
  
  格式: SPDX / CycloneDX
  工具: syft (生成 SBOM), grype (基于 SBOM 扫描漏洞)

  # 生成 SBOM
  syft <image> -o spdx-json > sbom.json
  
  # 基于 SBOM 扫描漏洞
  grype sbom:sbom.json

SLSA (Supply-chain Levels for Software Artifacts):
  Level 0: 无保障
  Level 1: 构建过程有记录
  Level 2: 使用托管构建服务
  Level 3: 源码和构建平台经过审计
  Level 4: 双人审查 + 密封构建环境
```

## 4. 零信任架构？

**回答：**

```
零信任原则: "永不信任,始终验证" (Never Trust, Always Verify)

传统模型:
  网络边界 = 信任边界
  内网 = 可信, 外网 = 不可信
  ❌ 一旦攻破边界, 内部畅通无阻

零信任模型:
  没有隐含信任
  每次访问都要 认证 + 授权 + 加密
  最小权限 + 持续验证

零信任核心组件:
  ┌──────────────────┬──────────────────────────────┐
  │ 组件              │ 说明                          │
  ├──────────────────┼──────────────────────────────┤
  │ 身份认证          │ SSO + MFA + 设备信任          │
  │ 微分段            │ 网络细粒度隔离 (不是大平面)    │
  │ 最小权限          │ RBAC + ABAC + JIT 权限        │
  │ 加密通信          │ mTLS (双向 TLS)               │
  │ 持续验证          │ 每次请求都验证身份和权限       │
  │ 设备信任          │ 设备健康检查 (合规/补丁)      │
  │ 日志审计          │ 所有访问行为可追溯            │
  └──────────────────┴──────────────────────────────┘

K8s 中的零信任实践:
  Service Mesh (Istio):
    mTLS: 服务间强制双向 TLS
    AuthorizationPolicy: 细粒度访问控制
    RequestAuthentication: JWT 验证

  NetworkPolicy: 默认拒绝, 白名单放行
  RBAC: 最小权限 ServiceAccount
  OPA/Kyverno: 策略即代码
```

## 7. 运行时安全与 Falco？

**回答：**

```
Falco: 云原生运行时安全工具 (CNCF 项目)

原理:
  内核级: eBPF / 系统调用监控
  规则引擎: 检测异常行为
  告警: stdout / Slack / Webhook / Kafka

检测能力:
  容器中执行 Shell
  读取敏感文件 (/etc/shadow, /etc/passwd)
  向外发起异常连接
  权限提升行为
  容器逃逸尝试
  K8s API 异常调用
```

```yaml
# Falco 规则示例

# 检测容器中执行 Shell
- rule: Terminal shell in container
  desc: A shell was spawned in a container
  condition: >
    spawned_process and container and
    proc.name in (bash, sh, zsh, dash)
  output: >
    Shell spawned in container
    (user=%user.name container=%container.name
     image=%container.image.repository
     shell=%proc.name parent=%proc.pname
     cmdline=%proc.cmdline)
  priority: WARNING
  tags: [container, shell, mitre_execution]

# 检测读取敏感文件
- rule: Read sensitive file
  desc: Sensitive file was read in container
  condition: >
    open_read and container and
    fd.name in (/etc/shadow, /etc/passwd, /etc/sudoers)
  output: >
    Sensitive file read (file=%fd.name container=%container.name)
  priority: WARNING

# 检测容器网络异常
- rule: Unexpected outbound connection
  desc: Container making unexpected outbound conn
  condition: >
    outbound and container and
    not (fd.sport in (80, 443, 53))
  output: >
    Unexpected outbound (container=%container.name
     connection=%fd.name)
  priority: NOTICE
```

```bash
# Helm 部署 Falco
helm repo add falcosecurity https://falcosecurity.github.io/charts
helm install falco falcosecurity/falco \
  --set falcosidekick.enabled=true \
  --set falcosidekick.config.slack.webhookurl="https://hooks.slack.com/..."

# 查看 Falco 告警
kubectl logs -l app.kubernetes.io/name=falco -f
```

```yaml
# Kyverno: 禁止使用 latest 标签
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: disallow-latest-tag
spec:
  validationFailureAction: Enforce
  rules:
  - name: require-image-tag
    match:
      any:
      - resources:
          kinds:
          - Pod
    validate:
      message: "Image tag 'latest' is not allowed."
      pattern:
        spec:
          containers:
          - image: "!*:latest"

---
# Kyverno: 强制资源限制
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-resource-limits
spec:
  validationFailureAction: Enforce
  rules:
  - name: check-limits
    match:
      any:
      - resources:
          kinds:
          - Pod
    validate:
      message: "CPU and memory limits are required."
      pattern:
        spec:
          containers:
          - resources:
              limits:
                memory: "?*"
                cpu: "?*"
```

## 9. 安全左移实践总结？

**回答：**

```
安全左移 = 在开发早期就发现和解决安全问题

完整 DevSecOps Pipeline:

  代码提交 ──→ CI 构建 ──→ 测试 ──→ 部署 ──→ 运行
     │            │          │         │         │
  gitleaks    SonarQube   OWASP ZAP  准入策略   Falco
  Semgrep     Trivy fs    API 扫描   镜像签名   SIEM
  pre-commit  镜像扫描    渗透测试   合规检查   WAF
              SBOM 生成                         审计

不同阶段的安全投入收益:
  ┌──────────────┬───────┬───────────────────────┐
  │ 阶段          │ 成本   │ 说明                   │
  ├──────────────┼───────┼───────────────────────┤
  │ 需求/设计     │ 1x    │ 威胁建模, 安全需求      │
  │ 编码          │ 5x    │ 代码审查, SAST          │
  │ 测试          │ 15x   │ DAST, 渗透测试          │
  │ 生产          │ 60x   │ 事件响应, 补丁修复      │
  └──────────────┴───────┴───────────────────────┘
  越早发现, 修复成本越低!

安全文化建设:
  1. 开发者安全培训 (OWASP Top 10 培训)
  2. 安全 Champion 制度 (每个团队有安全代表)
  3. 安全编码规范 (团队文档化)
  4. 自动化门禁 (CI 中强制安全扫描)
  5. 安全度量 (漏洞发现到修复的 MTTR)

度量指标:
  漏洞发现率: 每千行代码漏洞数
  修复时长:   P1 漏洞平均修复时间
  扫描覆盖率: CI 管线安全扫描覆盖比例
  合规率:     通过安全检查的部署比例
  逃逸率:     生产环境发现的漏洞占比
```

## 10. 安全面试综合速答？

**回答：**

```
Q: DevSecOps 和传统安全有什么区别?
A: 传统安全是事后审查, DevSecOps 是安全左移,
   贯穿整个 SDLC, 全员负责, 自动化驱动

Q: SAST vs DAST vs SCA?
A: SAST 扫源代码 (早期/误报多)
   DAST 扫运行应用 (发现运行时漏洞)
   SCA 扫第三方依赖 (已知 CVE)
   三者互补, 不是替代

Q: 发现生产安全事件怎么处理?
A: 准备→检测→遏制→消除→恢复→复盘
   先隔离止血, 保留证据, 修复漏洞, 轮换凭证

Q: 容器安全怎么做?
A: 最小基础镜像 + 非 root 运行 + 镜像扫描
   只读文件系统 + 资源限制 + NetworkPolicy
   运行时 Falco 监控 + 准入策略 (Kyverno/OPA)

Q: K8s Secret 安全吗?
A: 默认 base64 编码, 不安全!
   方案: etcd 加密 (EncryptionConfig)
        + External Secrets + Vault 外部管理
        + Sealed Secrets (GitOps 加密)

Q: 零信任是什么?
A: 永不信任,始终验证. 每次访问都认证授权
   mTLS + 微分段 + 最小权限 + 持续验证
   K8s: Istio mTLS + NetworkPolicy + RBAC

Q: 供应链安全怎么保障?
```
