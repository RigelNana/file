## 1. 常见安全合规标准详解？

**回答：**

```
┌──────────────┬──────────┬───────────────────────────────────────────────┐
│ 标准          │ 领域      │ 核心要求                                      │
├──────────────┼──────────┼───────────────────────────────────────────────┤
│ SOC 2        │ 通用 SaaS│ 安全性/可用性/保密性/处理完整性/隐私 (5 信任原则)│
│ ISO 27001    │ 通用      │ 信息安全管理体系 (ISMS), 114 项控制            │
│ PCI DSS      │ 支付      │ 12 项要求 (防火墙/加密/访问控制/日志/测试等)    │
│ HIPAA        │ 医疗      │ PHI 保护 (隐私/安全/事务/唯一标识等)           │
│ GDPR         │ 隐私      │ 数据最小化/知情同意/被遗忘权/72h 通报          │
│ 等保 2.0     │ 国内      │ 5 级保护 (物理/网络/主机/应用/数据/管理)        │
│ FedRAMP      │ 美国政府  │ 云服务安全评估 (Low/Moderate/High)              │
│ NIST 800-53  │ 通用      │ 安全与隐私控制框架 (20 个控制族)               │
└──────────────┴──────────┴───────────────────────────────────────────────┘

SOC 2 五大信任原则:
  Security:           必选 (基础)
  Availability:       SLA 保证
  Confidentiality:    数据保密
  Processing Integrity: 处理准确
  Privacy:            个人信息保护

ISO 27001 核心流程:
  Plan → Do → Check → Act (PDCA 循环)
  定义范围 → 风险评估 → 控制措施 → 持续改进

等保 2.0 等级:
  一级: 自主保护     (用户自行保护)
  二级: 指导保护     (备案, 自查)
  三级: 监督保护     (年度测评, 多数互联网)
  四级: 强制保护     (半年测评, 金融/电信)
  五级: 专控保护     (绝密, 极少)
```

## 2. SOC 2 合规实践？

**回答：**

```
SOC 2 审计类型:
  Type I:   某一时间点, 控制设计是否合理
  Type II:  一段时间 (通常 6-12 月), 控制是否持续有效
  → 客户通常要求 Type II

DevOps 团队需要做什么:
  ┌──────────────────────┬──────────────────────────────────────┐
  │ 控制领域              │ 实践                                  │
  ├──────────────────────┼──────────────────────────────────────┤
  │ 变更管理              │ PR 审查 + CI/CD 自动化 + 审计日志      │
  │ 访问控制              │ RBAC + MFA + 最小权限 + 定期审查       │
  │ 系统监控              │ 集中日志 + 告警 + 事件响应             │
  │ 风险评估              │ 定期漏洞扫描 + 渗透测试               │
  │ 数据保护              │ 加密传输/存储 + 备份 + 数据分类       │
  │ 供应商管理            │ 第三方安全评估 + 合同约束              │
  │ 事件响应              │ 应急预案 + 演练 + 通报流程             │
  │ 物理安全              │ 云服务商提供 (AWS/Azure 已合规)       │
  └──────────────────────┴──────────────────────────────────────┘

证据收集自动化:
  Git 提交记录 → 变更管理证据
  CI/CD 日志 → 测试和部署证据
  CloudTrail → AWS API 调用审计
  IAM 策略导出 → 访问控制证据
  漏洞扫描报告 → 风险管理证据
  培训记录 → 安全意识证据
```

## 3. GDPR 合规要点？

**回答：**

```
GDPR 核心原则:
  1. 合法/公平/透明:  有合法依据处理数据
  2. 目的限制:        只为特定目的收集
  3. 数据最小化:      只收集必要数据
  4. 准确性:          保持数据准确
  5. 存储限制:        不再需要时删除
  6. 完整性和保密性:  适当安全措施
  7. 可问责性:        能证明自己合规

数据主体权利:
  知情权:     告知用户数据如何被使用
  访问权:     用户可查看自己的数据
  更正权:     用户可修改不准确数据
  被遗忘权:   用户可请求删除数据 (Right to Erasure)
  限制处理权: 限制数据处理方式
  数据可携权: 导出数据 (机器可读格式)
  反对权:     反对特定数据处理

技术实现:
  数据清单:  知道存了什么数据, 在哪里
  同意管理:  Cookie Banner / 隐私偏好中心
  数据加密:  传输 (TLS) + 存储 (AES-256)
  审计日志:  记录谁访问了什么数据
  数据删除:  自动化删除过期数据
  隐私设计:  Privacy by Design / by Default

违规处罚:
  一般违规: 最高 1000 万欧 或全球年营业额 2%
  严重违规: 最高 2000 万欧 或全球年营业额 4%
  72 小时内必须向监管机构报告数据泄露
```

## 4. 安全审计日志设计？

**回答：**

```
审计日志必须记录的信息:
  谁 (Who):   用户ID, 角色, IP 地址
  什么 (What): 操作类型, 操作对象, 操作参数
  何时 (When): 精确时间戳 (UTC)
  哪里 (Where): 来源系统, 服务名, 请求路径
  结果 (Result): 成功/失败, 变更前后值

审计日志格式 (JSON):
  {
    "timestamp": "2026-03-11T10:30:00Z",
    "event_type": "user.login",
    "actor": {
      "user_id": "u-12345",
      "role": "admin",
      "ip": "10.0.1.100",
      "user_agent": "Mozilla/5.0..."
    },
    "resource": {
      "type": "account",
      "id": "acc-67890"
    },
    "action": "login",
    "result": "success",
    "metadata": {
      "mfa_used": true,
      "session_id": "sess-abc"
    }
  }

关键审计事件:
  ┌────────────────────┬──────────────────────────────────┐
  │ 事件类型            │ 示例                              │
  ├────────────────────┼──────────────────────────────────┤
  │ 认证事件            │ 登录/登出/MFA/密码重置            │
  │ 授权事件            │ 权限变更/角色分配/越权尝试        │
  │ 数据访问            │ 查看敏感数据/导出数据/批量查询    │
  │ 数据修改            │ 创建/更新/删除关键记录            │
  │ 系统配置            │ 安全策略变更/防火墙规则修改       │
  │ 管理操作            │ 用户创建/停用/部署发布            │
  └────────────────────┴──────────────────────────────────┘
```

## 5. SIEM 安全信息与事件管理？

**回答：**

```
SIEM 架构:

  数据源
  ├── 操作系统日志 (syslog/Windows Event)
  ├── 应用日志 (access log/error log)
  ├── 网络设备 (防火墙/IDS/IPS)
  ├── 云平台 (CloudTrail/Azure Activity)
  ├── 安全设备 (WAF/EDR/DLP)
  └── 身份系统 (AD/LDAP/SSO)
      │
      ↓ 采集 (Agent/Syslog/API)
      │
  ┌──────────────────────────────────────┐
  │ SIEM 平台                            │
  │  日志归一化 → 关联分析 → 告警 → 响应  │
  │  (Normalization → Correlation → Alert)│
  └──────────────────────────────────────┘
      │
      ↓ 输出
      │
  ├── 实时告警 (PagerDuty/Slack/邮件)
  ├── 仪表盘 (安全态势感知)
  ├── 工单 (事件响应流程)
  └── 报告 (合规审计报告)

主流 SIEM 工具:
  ┌──────────────────┬──────────────────────────────────┐
  │ 工具              │ 特点                              │
  ├──────────────────┼──────────────────────────────────┤
  │ Splunk           │ 企业级, 功能强, 贵                 │
  │ ELK + Security   │ 开源, 灵活, Elastic SIEM          │
  │ AWS Security Hub │ AWS 原生, 集成 GuardDuty          │
  │ Azure Sentinel   │ Azure 原生, AI 驱动               │
  │ Google Chronicle │ Google 云, 大规模数据分析          │
  │ Wazuh            │ 开源, HIDS + SIEM                 │
  │ QRadar           │ IBM, 企业级                       │
  └──────────────────┴──────────────────────────────────┘

关联规则示例:
  同一 IP 5 分钟内 10 次登录失败 → 暴力破解告警
  管理员非工作时间登录 → 异常行为告警
  敏感数据大量导出 → 数据泄露预警
  新创建的 IAM 用户立即获得 Admin → 提权告警
```

## 6. Open Policy Agent (OPA) 策略即代码？

**回答：**

```
OPA: 通用策略引擎, 将安全/合规策略作为代码管理

核心概念:
  Policy: 策略 (用 Rego 语言编写)
  Data:   输入数据 (JSON)
  Query:  决策请求 → 允许/拒绝

使用场景:
  ┌────────────────────┬──────────────────────────────────┐
  │ 场景                │ 说明                              │
  ├────────────────────┼──────────────────────────────────┤
  │ K8s 准入控制        │ Gatekeeper (OPA + K8s Webhook)   │
  │ API 授权            │ 微服务间的访问控制                │
  │ Terraform 合规      │ IaC 部署前检查                   │
  │ CI/CD 策略门禁      │ 合规检查通过才能部署             │
  │ 数据过滤            │ 行级/列级数据访问控制             │
  └────────────────────┴──────────────────────────────────┘
```

```rego
# Rego 策略示例: K8s Pod 安全策略

package kubernetes.admission

# 拒绝使用 latest 标签
deny[msg] {
    input.request.kind.kind == "Pod"
    container := input.request.object.spec.containers[_]
    endswith(container.image, ":latest")
    msg := sprintf("容器 %v 不允许使用 latest 标签", [container.name])
}

# 拒绝特权容器
deny[msg] {
    input.request.kind.kind == "Pod"
    container := input.request.object.spec.containers[_]
    container.securityContext.privileged == true
    msg := sprintf("容器 %v 不允许特权模式", [container.name])
}

# 必须设置资源限制
deny[msg] {
    input.request.kind.kind == "Pod"
    container := input.request.object.spec.containers[_]
    not container.resources.limits
    msg := sprintf("容器 %v 必须设置资源限制", [container.name])
}
```

```
Conftest: OPA 的 CI/CD 集成工具

# 检查 Dockerfile 合规
conftest test Dockerfile --policy policy/

# 检查 Kubernetes YAML
conftest test deployment.yaml --policy policy/

# 检查 Terraform 计划
terraform plan -out=plan.bin
terraform show -json plan.bin > plan.json
conftest test plan.json --policy policy/
```

## 8. 供应链安全？

**回答：**

```
软件供应链攻击: 通过依赖/工具链/构建过程植入恶意代码

著名事件:
  SolarWinds (2020): 构建过程被植入后门
  Log4Shell (2021): 开源依赖 0day 漏洞
  Codecov (2021):   CI 脚本被篡改, 窃取环境变量
  XZ Utils (2024):  维护者社工攻击, 后门植入

SLSA 框架 (Supply-chain Levels for Software Artifacts):
  Level 0: 无保证
  Level 1: 构建过程有文档 (provenance)
  Level 2: 使用托管构建服务
  Level 3: 构建平台可审计, 不可篡改
  Level 4: 两人审查, 可复现构建

防护措施:
  ┌──────────────────────┬──────────────────────────────────┐
  │ 层面                  │ 实践                              │
  ├──────────────────────┼──────────────────────────────────┤
  │ 依赖管理              │ 锁定版本 + SCA 扫描 + 私有镜像    │
  │ 构建安全              │ SLSA + 构建隔离 + 签名            │
  │ 镜像安全              │ Cosign 签名 + Trivy 扫描          │
  │ 代码审查              │ PR 必须审查 + 提交签名 (GPG)      │
  │ CI/CD 安全            │ 最小权限 + Secret 管理 + 审计日志  │
  │ 运行时安全            │ SBOM + 运行时监控                 │
  └──────────────────────┴──────────────────────────────────┘

SBOM (Software Bill of Materials): 软件物料清单
  列出所有组件和依赖, 方便漏洞追踪
  格式: SPDX, CycloneDX
  工具: syft (生成), grype (扫描)
```

```bash
# Cosign: 容器镜像签名
cosign sign --key cosign.key myregistry/app:v1.0

# Cosign: 验证签名
cosign verify --key cosign.pub myregistry/app:v1.0

# syft: 生成 SBOM
syft packages myregistry/app:v1.0 -o cyclonedx-json > sbom.json

# grype: 基于 SBOM 扫描漏洞
grype sbom:sbom.json
```

## 9. 安全自动化与 DevSecOps Pipeline？

**回答：**

```
完整 DevSecOps Pipeline:

  开发阶段
  ├── IDE 插件 (安全提示)
  ├── Pre-commit Hook (git-secrets/gitleaks)
  └── 威胁建模 (设计阶段)
      │
  构建阶段
  ├── SAST (SonarQube/Semgrep/CodeQL)
  ├── SCA (Snyk/Dependabot/Trivy fs)
  ├── License 检查
  └── 单元测试 + 安全测试
      │
  测试阶段
  ├── DAST (OWASP ZAP)
  ├── IAST (运行时检测)
  ├── 容器镜像扫描 (Trivy image)
  └── IaC 安全检查 (tfsec/checkov)
      │
  发布阶段
  ├── 镜像签名 (Cosign)
  ├── SBOM 生成 (syft)
  ├── 合规门禁 (OPA/Conftest)
  └── 审批流程
      │
  运行阶段
  ├── K8s 准入控制 (Gatekeeper/Kyverno)
  ├── 运行时安全 (Falco)
  ├── 网络策略 (NetworkPolicy/Service Mesh)
  └── SIEM 监控 + 告警
```

```yaml
# GitHub Actions 完整 DevSecOps Pipeline
name: DevSecOps Pipeline
on: [push, pull_request]

jobs:
  security-scan:
    runs-on: ubuntu-latest
    steps:
      # 密钥泄漏扫描
      - name: Gitleaks
        uses: gitleaks/gitleaks-action@v2

      # SAST 代码扫描
      - name: SonarQube Scan
        uses: sonarsource/sonarqube-scan-action@v2
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}

      # SCA 依赖扫描
      - name: Snyk Dependencies
        uses: snyk/actions/node@master
        env:
          SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}

      # 容器镜像扫描
      - name: Trivy Image Scan
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: ${{ env.IMAGE }}
          exit-code: 1
          severity: HIGH,CRITICAL

      # IaC 安全检查
      - name: Checkov
        uses: bridgecrewio/checkov-action@master
        with:
          directory: terraform/

      # DAST 动态扫描 (staging 环境)
      - name: OWASP ZAP
        uses: zaproxy/action-baseline@v0.7.0
        with:
          target: ${{ env.STAGING_URL }}
```

## 10. 合规与审计面试速答？

**回答：**

```
Q: 公司做过哪些合规认证?
A: SOC 2 Type II + ISO 27001
   DevOps 团队负责: 变更管理(Git+CI/CD), 访问控制(RBAC+MFA),
   系统监控(ELK+告警), 漏洞管理(Trivy+Snyk)
   证据自动化收集: CloudTrail + CI/CD 日志 + IAM 导出

Q: 如何实现策略即代码?
A: OPA/Rego 编写策略 → Conftest CI 检查 →
   Gatekeeper K8s 准入控制 → Git 管理策略版本

Q: 审计日志怎么设计?
A: 5W: Who/What/When/Where/Result
   集中到 ELK/SIEM, 不可篡改 (S3 Object Lock)
   关键事件: 登录/权限变更/敏感数据访问/配置修改

Q: GDPR 对技术的影响?
A: 数据最小化 + 加密存储 + 用户删除接口 (被遗忘权)
   同意管理 + 数据可携带 (导出 API) + 72h 泄露通报

Q: 供应链安全怎么做?
A: 依赖锁定 + SCA 扫描 (Snyk) + 镜像签名 (Cosign)
   SBOM 生成 (syft) + SLSA 构建来源 + PR 审查

Q: 合规和效率冲突怎么办?
A: 自动化是关键
   策略即代码 (不手动审查) + CI 门禁 (不拖慢流程)
   合规检查融入 Pipeline, 开发无感知
   证据自动收集, 审计不打扰日常工作

Q: 容器运行时安全怎么做?
A: Falco 检测异常行为 (exec 进容器/读敏感文件)
   只读根文件系统 + 非 root + drop capabilities
   NetworkPolicy 限制网络访问
```
