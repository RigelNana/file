## 1. Docker 安全威胁模型？

**回答：**

```
容器安全攻击面:

  ┌─────────────────────────────────────────┐
  │          容器安全分层模型                  │
  ├─────────────────────────────────────────┤
  │ 5. 应用层: 代码漏洞、依赖漏洞             │
  │ 4. 容器运行时: 逃逸、权限提升             │
  │ 3. 镜像层: 恶意镜像、漏洞基础镜像          │
  │ 2. 编排层: K8s API暴露、RBAC不当          │
  │ 1. 主机层: 内核漏洞、Docker Daemon        │
  │ 0. 网络层: 容器间网络不隔离               │
  └─────────────────────────────────────────┘

容器 vs 虚拟机 安全对比:
  ┌──────────┬──────────────┬───────────────┐
  │ 特性      │ 容器          │ 虚拟机         │
  ├──────────┼──────────────┼───────────────┤
  │ 隔离级别  │ 进程级 (弱)    │ 硬件级 (强)    │
  │ 内核共享  │ 共享宿主内核   │ 独立内核        │
  │ 逃逸风险  │ 较高          │ 较低           │
  │ 攻击面    │ 共享内核 + ns  │ Hypervisor    │
  │ 启动速度  │ 秒级          │ 分钟级         │
  └──────────┴──────────────┴───────────────┘

关键风险:
  容器逃逸: 利用内核漏洞或配置不当从容器访问宿主机
  特权容器: --privileged 赋予容器几乎所有主机权限
  Docker Socket: 挂载 /var/run/docker.sock → 完全控制 Docker
  敏感信息泄露: 镜像中包含密钥/凭证
  供应链攻击: 使用了被篡改的基础镜像
```

## 2. Docker 镜像安全？

**回答：**

```dockerfile
# ==================== 安全 Dockerfile 模板 ====================

# 1. 使用最小基础镜像 (减少攻击面)
# ❌ FROM ubuntu:latest     (100+ MB, 大量包)
# ❌ FROM node:18            (900+ MB)
# ✅ FROM node:18-alpine     (小)
# ✅ FROM gcr.io/distroless/nodejs18  (最小, 无 shell)

FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
RUN npm run build

# 2. 多阶段构建 (不包含构建工具)
FROM gcr.io/distroless/nodejs18-debian12
COPY --from=builder /app/dist /app
COPY --from=builder /app/node_modules /app/node_modules
WORKDIR /app

# 3. 非 root 用户
USER nonroot:nonroot

# 4. 只暴露必要端口
EXPOSE 3000

ENTRYPOINT ["node", "server.js"]
```

```
镜像安全检查清单:
  ✅ 使用最小基础镜像 (Alpine / Distroless / Scratch)
  ✅ 固定镜像版本 (不用 :latest)
  ✅ 多阶段构建 (不包含编译器/调试工具)
  ✅ 非 root 用户运行
  ✅ 不在镜像中存储密钥 (用 Secret 挂载)
  ✅ .dockerignore 排除敏感文件
  ✅ 扫描镜像漏洞 (Trivy / Snyk)
  ✅ 镜像签名 (Cosign / Notary)
  ✅ 使用可信基础镜像 (官方或经过验证)
  ✅ 定期重建镜像 (获取安全补丁)
```

```bash
# Trivy 镜像漏洞扫描
trivy image --severity HIGH,CRITICAL myapp:latest

# 输出示例:
# myapp:latest (alpine 3.18)
# ┌──────────────┬────────────────┬──────────┬─────────────────┐
# │ Library      │ Vulnerability  │ Severity │ Fixed Version   │
# ├──────────────┼────────────────┼──────────┼─────────────────┤
# │ openssl      │ CVE-2023-xxxx  │ HIGH     │ 3.1.2           │
# └──────────────┴────────────────┴──────────┴─────────────────┘

# Cosign 镜像签名
cosign sign --key cosign.key myregistry.com/myapp:latest
cosign verify --key cosign.pub myregistry.com/myapp:latest
```

## 3. Docker 运行时安全？

**回答：**

```bash
# ==================== 安全运行容器 ====================

# ❌ 危险: 特权模式
docker run --privileged myapp

# ❌ 危险: 挂载 Docker Socket
docker run -v /var/run/docker.sock:/var/run/docker.sock myapp

# ❌ 危险: 挂载主机根目录
docker run -v /:/host myapp

# ✅ 安全: 最小权限运行
docker run \
  --user 1000:1000 \
  --read-only \
  --tmpfs /tmp:rw,noexec,nosuid \
  --cap-drop ALL \
  --cap-add NET_BIND_SERVICE \
  --security-opt no-new-privileges:true \
  --security-opt seccomp=default.json \
  --memory 512m \
  --cpus 1 \
  --pids-limit 100 \
  --network custom-net \
  myapp:1.0
```

```
运行时安全参数详解:
  ┌─────────────────────────────┬───────────────────────────┐
  │ 参数                         │ 作用                      │
  ├─────────────────────────────┼───────────────────────────┤
  │ --user 1000:1000            │ 非 root 运行               │
  │ --read-only                 │ 只读文件系统               │
  │ --cap-drop ALL              │ 移除所有 Linux Capabilities│
  │ --cap-add NET_BIND_SERVICE  │ 只添加需要的能力            │
  │ --no-new-privileges         │ 禁止提权                   │
  │ --security-opt seccomp=...  │ 系统调用白名单              │
  │ --memory 512m               │ 内存限制                   │
  │ --pids-limit 100            │ 进程数限制 (防 fork bomb)   │
  │ --network custom-net        │ 自定义网络隔离              │
  └─────────────────────────────┴───────────────────────────┘

Linux Capabilities (能力):
  传统: root 拥有所有权限
  Capabilities: 将 root 权限拆分为细粒度能力
  
  常见 Capabilities:
    NET_BIND_SERVICE: 绑定 < 1024 端口
    SYS_PTRACE:       调试进程
    SYS_ADMIN:        挂载/命名空间等 (危险, 相当于半个 root)
    NET_RAW:          原始套接字 (ping)
    
  最佳实践: drop ALL → 只 add 需要的
```

## 4. 容器逃逸防护？

**回答：**

```
容器逃逸: 从容器内部获取宿主机访问权限

常见逃逸途径:
  1. 内核漏洞
     容器共享宿主机内核
     CVE-2019-5736 (runc 漏洞): 覆盖 runc 二进制获取 root
     CVE-2022-0185 (内核漏洞): 利用文件系统上下文提权
  
  2. 配置不当
     --privileged 模式 (具有宿主机所有设备访问权)
     挂载 Docker Socket (/var/run/docker.sock)
     挂载宿主机敏感目录 (/proc, /sys, /)
     SYS_ADMIN capability
  
  3. Docker Socket 逃逸
     容器内有 docker.sock → 可以创建新的特权容器
     → 挂载宿主机文件系统 → 完全控制宿主机

防护措施:
  ┌──────────────────┬──────────────────────────────┐
  │ 层面              │ 措施                          │
  ├──────────────────┼──────────────────────────────┤
  │ 内核              │ 及时更新、启用 SELinux/AppArmor│
  │ 运行时            │ 禁止 --privileged, drop caps  │
  │ 文件系统          │ --read-only, 不挂载敏感路径    │
  │ Seccomp          │ 系统调用限制 (白名单)           │
  │ 用户命名空间      │ User Namespace Remapping      │
  │ 替代运行时        │ gVisor (沙箱) / Kata (微虚机)  │
  └──────────────────┴──────────────────────────────┘

沙箱容器运行时:
  gVisor (Google): 应用内核 → 拦截系统调用
    runsc 作为容器运行时, 不直接调用宿主机内核
    安全性高, 但有兼容性和性能损失

  Kata Containers: 轻量 VM + 容器接口
    每个 Pod 一个微虚拟机
    硬件级隔离, 兼容 OCI 标准
```

## 6. Docker Daemon 安全加固？

**回答：**

```json
// /etc/docker/daemon.json 安全配置
{
  "icc": false,
  "userns-remap": "default",
  "no-new-privileges": true,
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  },
  "storage-driver": "overlay2",
  "live-restore": true,
  "default-ulimits": {
    "nofile": { "Name": "nofile", "Hard": 65536, "Soft": 32768 },
    "nproc": { "Name": "nproc", "Hard": 4096, "Soft": 2048 }
  },
  "tls": true,
  "tlscacert": "/etc/docker/ca.pem",
  "tlscert": "/etc/docker/server-cert.pem",
  "tlskey": "/etc/docker/server-key.pem",
  "tlsverify": true
}
```

```
关键配置说明:
  icc: false             → 禁止容器间默认通信
  userns-remap: default  → 用户命名空间隔离 (容器root≠宿主root)
  no-new-privileges      → 禁止容器内提权
  tls + tlsverify        → Docker API 启用 TLS 认证
  live-restore           → Daemon 重启不影响运行中的容器

Docker Socket 保护:
  ❌ 不要暴露到网络: dockerd -H tcp://0.0.0.0:2375
  ✅ 只用 Unix Socket: /var/run/docker.sock
  ✅ 限制 Socket 访问权限: chmod 660
  ✅ 使用 Docker Context + SSH 远程管理

Docker Bench Security:
  # 自动检查 Docker 安全配置 (基于 CIS Benchmark)
  docker run --rm --net host --pid host \
    --userns host --cap-add audit_control \
    -v /var/lib:/var/lib:ro \
    -v /var/run/docker.sock:/var/run/docker.sock:ro \
    -v /etc:/etc:ro \
    docker/docker-bench-security
```

## 7. 容器网络安全？

**回答：**

```
Docker 网络模式:
  bridge (默认):  虚拟网桥, 容器间可通信
  host:           共享宿主网络 (不推荐, 无隔离)
  none:           无网络
  overlay:        跨主机网络 (Swarm / K8s)
  macvlan:        容器有独立 MAC 地址

网络隔离策略:
  1. 使用自定义网络 (不用默认 bridge)
     docker network create --driver bridge frontend
     docker network create --driver bridge backend
     → 不同网络的容器默认无法通信

  2. 禁止容器间通信 (ICC)
     daemon.json: "icc": false
     → 需要通信的用 --link 或自定义网络

  3. 限制出站流量
     iptables 规则限制容器访问外部网络

  4. 加密网络通信
     Overlay 网络: docker network create --opt encrypted
     服务间: mTLS (Istio / Linkerd)

容器防火墙:
  iptables / nftables: 主机级网络规则
  Calico:              K8s 网络策略
  Cilium:              eBPF 网络策略 (高性能)
```

## 8. 容器运行时安全监控？

**回答：**

```
运行时安全工具:
  ┌──────────────┬──────────────────────────────────┐
  │ 工具          │ 说明                              │
  ├──────────────┼──────────────────────────────────┤
  │ Falco        │ 容器运行时威胁检测 (CNCF)          │
  │ Sysdig       │ 系统调用监控 + 安全                │
  │ Aqua Security│ 全生命周期容器安全平台              │
  │ Twistlock    │ Prisma Cloud (Palo Alto)          │
  │ StackRox     │ Red Hat ACS (K8s 安全)            │
  └──────────────┴──────────────────────────────────┘

Falco (CNCF 运行时安全):
  工作原理: 内核模块/eBPF → 捕获系统调用 → 规则匹配 → 告警
  
  检测场景:
    容器内启动 shell
    读取敏感文件 (/etc/shadow, /etc/passwd)
    异常网络连接
    权限变更
    文件系统写入 (只读容器不应写)
```

```yaml
# Falco 规则示例
- rule: Terminal shell in container
  desc: 检测容器内启动交互式 shell
  condition: >
    spawned_process and container and
    proc.name in (bash, sh, zsh) and
    proc.tty != 0
  output: >
    Shell spawned in container
    (user=%user.name container=%container.name
     shell=%proc.name parent=%proc.pname)
  priority: WARNING

- rule: Read sensitive file in container
  desc: 容器内读取敏感文件
  condition: >
    open_read and container and
    fd.name in (/etc/shadow, /etc/passwd)
  output: >
    Sensitive file read in container
    (file=%fd.name container=%container.name)
  priority: CRITICAL
```

## 9. CI/CD 中的容器安全流水线？

**回答：**

```yaml
# 完整容器安全流水线
stages:
  - build
  - scan
  - sign
  - deploy

# 构建镜像
build:
  stage: build
  script:
    - docker build -t $IMAGE:$TAG .
    - docker push $IMAGE:$TAG

# 镜像漏洞扫描
vulnerability_scan:
  stage: scan
  script:
    - trivy image --exit-code 1 --severity CRITICAL $IMAGE:$TAG
  # CRITICAL 漏洞 → 构建失败

# Dockerfile 最佳实践检查
dockerfile_lint:
  stage: scan
  script:
    - hadolint Dockerfile
  # 检查: 是否用 root, 是否固定版本等

# 配置合规扫描
config_scan:
  stage: scan
  script:
    - conftest test --policy policy/ deployment.yaml
  # 检查 K8s 清单是否符合安全策略

# 镜像签名
sign:
  stage: sign
  script:
    - cosign sign --key env://COSIGN_KEY $IMAGE:$TAG
  needs: [vulnerability_scan, dockerfile_lint]

# 部署 (验证签名)
deploy:
  stage: deploy
  script:
    - cosign verify --key cosign.pub $IMAGE:$TAG
    - kubectl apply -f deployment.yaml
```

```
安全门禁 (Security Gates):
  ┌────────┬───────────────────┬───────────────┐
  │ 阶段    │ 检查项             │ 工具           │
  ├────────┼───────────────────┼───────────────┤
  │ 编码    │ 代码漏洞           │ SonarQube      │
  │ 构建    │ Dockerfile 规范    │ Hadolint       │
  │ 构建后  │ 镜像漏洞           │ Trivy          │
  │ 构建后  │ 镜像签名           │ Cosign         │
  │ 部署前  │ K8s 配置合规       │ OPA/Conftest   │
  │ 部署前  │ 签名验证           │ Cosign Verify  │
  │ 运行时  │ 异常行为           │ Falco          │
  └────────┴───────────────────┴───────────────┘
```

## 10. 容器安全面试速答？

**回答：**

```
Q: Docker 为什么不如 VM 安全?
A: 容器共享宿主机内核, 隔离靠 namespace/cgroup (进程级)
   VM 有独立内核 + Hypervisor (硬件级隔离)
   容器逃逸 = 获取宿主机 root, VM 逃逸难度更高

Q: 怎么加固容器安全?
A: 镜像: 最小基础镜像 + 多阶段构建 + 漏洞扫描
   运行: 非 root + read-only + drop ALL caps
   配置: 禁止 privileged + 不挂载 docker.sock
   网络: 自定义网络 + 网络策略
   监控: Falco 运行时检测

Q: 什么是容器逃逸?
A: 从容器突破隔离访问宿主机
   途径: 内核漏洞 / privileged / docker.sock / SYS_ADMIN
   防护: 更新内核 + seccomp + gVisor/Kata + 最小权限

Q: Distroless 镜像是什么?
A: Google 的最小镜像, 只包含应用运行时
   无 shell, 无包管理器, 无调试工具 → 攻击面最小
   缺点: 调试困难 (可用 debug 变体)

Q: 镜像签名有什么用?
A: 保证镜像完整性和来源可信
   构建时签名 (Cosign) → 部署时验证
   防止供应链攻击 (中间人篡改镜像)

Q: 什么是 Linux Capabilities?
A: 将 root 权限拆分为细粒度能力
   容器应 drop ALL → 只 add 需要的
   常用: NET_BIND_SERVICE, SYS_PTRACE
   危险: SYS_ADMIN (≈半个 root)
```
