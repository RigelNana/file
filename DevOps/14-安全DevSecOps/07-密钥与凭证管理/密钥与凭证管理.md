## 1. 为什么不能在代码中硬编码密钥？如何防止密钥泄漏？

**回答：**

```
密钥泄漏的危害:
  1. GitHub 公开仓库爬虫自动扫描 (AWS Key 平均 13 分钟被发现利用)
  2. 攻击者可直接操控云资源 (创建挖矿实例/删除数据)
  3. 数据库凭证泄漏 → 数据被拖库
  4. API Key 泄漏 → 服务被滥用 (产生巨额账单)

常见泄漏途径:
  硬编码:      代码中写死 password = "abc123"
  配置文件提交: .env / config.yaml 提交到 Git
  日志打印:     log.info("token: " + token)
  Docker 镜像:  ENV 或 COPY 把密钥打入镜像层
  CI/CD 输出:   构建日志中暴露 Secret

防泄漏工具:
  ┌──────────────────┬─────────────────────────────────────┐
  │ 工具              │ 说明                                │
  ├──────────────────┼─────────────────────────────────────┤
  │ gitleaks         │ 扫描 Git 历史中的密钥泄漏             │
  │ git-secrets      │ AWS 出品, 阻止提交含密钥的代码         │
  │ truffleHog       │ 深度扫描 Git 历史 (高熵字符串+正则)    │
  │ detect-secrets   │ Yelp 出品, 轻量级密钥检测             │
  │ GitHub Secret    │ GitHub 内置, 自动扫描推送中的密钥      │
  │  Scanning        │                                     │
  └──────────────────┴─────────────────────────────────────┘
```

```bash
# gitleaks: 扫描整个 Git 仓库历史
gitleaks detect --source . --report-format json --report-path report.json

# 在 pre-commit hook 中拦截
# .pre-commit-config.yaml
# repos:
#   - repo: https://github.com/gitleaks/gitleaks
#     rev: v8.18.0
#     hooks:
#       - id: gitleaks

# git-secrets: 安装并注册 AWS 模式
git secrets --install
git secrets --register-aws
git secrets --scan  # 扫描当前仓库

# truffleHog: 扫描高熵字符串
trufflehog git file://. --since-commit HEAD~50
```

```yaml
# CI/CD 集成 (GitHub Actions)
name: Secret Scan
on: [push, pull_request]
jobs:
  gitleaks:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: gitleaks/gitleaks-action@v2
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

## 2. HashiCorp Vault 的核心概念和使用？

**回答：**

```
Vault 架构:
  ┌──────────────────────────────────────────────┐
  │                  Vault Server                 │
  │                                               │
  │  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
  │  │ Secret   │  │  Auth    │  │  Audit   │   │
  │  │ Engines  │  │ Methods  │  │  Devices │   │
  │  │          │  │          │  │          │   │
  │  │ KV v2    │  │ Token    │  │ File     │   │
  │  │ AWS      │  │ AppRole  │  │ Syslog   │   │
  │  │ Database │  │ K8s      │  │ Socket   │   │
  │  │ PKI      │  │ LDAP     │  │          │   │
  │  │ Transit  │  │ OIDC     │  │          │   │
  │  └──────────┘  └──────────┘  └──────────┘   │
  │                                               │
  │  ┌──────────────────────────────────────┐     │
  │  │ Storage Backend (Consul / Raft / S3) │     │
  │  └──────────────────────────────────────┘     │
  └──────────────────────────────────────────────┘

核心概念:
  Secret Engine:  存储和生成密钥的后端 (KV, AWS, DB, PKI)
  Auth Method:    认证方式 (Token, AppRole, K8s ServiceAccount)
  Policy:         访问控制策略 (路径 + 能力)
  Lease:          密钥租约 (TTL, 自动过期/续租)
  Seal/Unseal:    密封/解封 (Shamir 密钥分割)
```

```bash
# 基本操作
# 启动 dev 模式 (仅测试)
vault server -dev

# KV Secret Engine (v2)
vault kv put secret/myapp/config \
  db_host=10.0.0.1 \
  db_user=app \
  db_password=s3cret

vault kv get secret/myapp/config
vault kv get -field=db_password secret/myapp/config

# 查看版本历史
vault kv metadata get secret/myapp/config

# Policy 定义
vault policy write app-policy - <<EOF
path "secret/data/myapp/*" {
  capabilities = ["read", "list"]
}
path "database/creds/myapp-role" {
  capabilities = ["read"]
}
EOF

# AppRole 认证 (应用程序使用)
vault auth enable approle
vault write auth/approle/role/myapp \
  token_policies="app-policy" \
  token_ttl=1h \
  token_max_ttl=4h

# 获取 Role ID 和 Secret ID
vault read auth/approle/role/myapp/role-id
vault write -f auth/approle/role/myapp/secret-id

# 登录获取 Token
vault write auth/approle/login \
  role_id="xxx" \
  secret_id="yyy"
```

## 3. Vault 动态凭证 (Dynamic Secrets) 是什么？

**回答：**

```
动态凭证: Vault 按需生成临时数据库凭证, 自动过期销毁

传统方式:
  应用 → 读取配置文件中的固定密码 → 连接数据库
  问题: 密码从不轮换, 多应用共用, 泄漏影响大

动态凭证:
  应用 → 向 Vault 请求 → Vault 在数据库创建临时用户 → 返回凭证
  优点: 每次不同, 自动过期, 最小权限, 可审计

工作流程:
  1. 管理员配置 Database Secret Engine
  2. 应用通过 API 请求凭证
  3. Vault 在 MySQL 中创建临时用户
  4. 返回 username/password (带 TTL)
  5. TTL 到期 → Vault 自动删除该用户
```

```bash
# 配置 MySQL Database Engine
vault secrets enable database

vault write database/config/mydb \
  plugin_name=mysql-database-plugin \
  connection_url="{{username}}:{{password}}@tcp(10.0.0.1:3306)/" \
  allowed_roles="myapp-role" \
  username="vault_admin" \
  password="admin_password"

# 创建角色 (定义临时用户的权限和 TTL)
vault write database/roles/myapp-role \
  db_name=mydb \
  creation_statements="CREATE USER '{{name}}'@'%' IDENTIFIED BY '{{password}}'; \
    GRANT SELECT, INSERT, UPDATE ON mydb.* TO '{{name}}'@'%';" \
  default_ttl="1h" \
  max_ttl="24h"

# 应用获取动态凭证
vault read database/creds/myapp-role
# 返回:
# username: v-approle-myapp-role-abc123
# password: A1a-随机生成的强密码
# lease_duration: 1h

# 续租 (如果需要延长)
vault lease renew database/creds/myapp-role/lease-id

# 撤销 (立即删除)
vault lease revoke database/creds/myapp-role/lease-id
```

```
支持的动态凭证:
  ┌──────────────────┬─────────────────────────────┐
  │ 类型              │ 说明                        │
  ├──────────────────┼─────────────────────────────┤
  │ Database         │ MySQL, PostgreSQL, MongoDB   │
  │ AWS              │ 临时 IAM 凭证 (STS)          │
  │ Azure            │ Service Principal            │
  │ GCP              │ Service Account Key          │
  │ Consul           │ ACL Token                    │
  │ RabbitMQ         │ 临时用户                      │
  │ PKI              │ 动态 TLS 证书                 │
  └──────────────────┴─────────────────────────────┘
```

## 4. Kubernetes 中如何集成 Vault？

**回答：**

```
三种集成方式:
  1. Vault Agent Sidecar Injector (注入 sidecar)
  2. Vault CSI Provider (CSI 驱动挂载)
  3. 应用直接调用 Vault API

方式一: Vault Agent Sidecar (最常用)

  工作流程:
    Pod 创建 → Mutating Webhook 注入 sidecar
    → Vault Agent 用 K8s ServiceAccount 认证
    → 获取 Secret 写入共享 Volume
    → 应用从文件读取
```

```yaml
# 使用 Vault Agent 注入 Secret
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  template:
    metadata:
      annotations:
        # Vault Agent Injector 注解
        vault.hashicorp.com/agent-inject: "true"
        vault.hashicorp.com/role: "myapp"
        vault.hashicorp.com/agent-inject-secret-db-creds: "database/creds/myapp-role"
        vault.hashicorp.com/agent-inject-template-db-creds: |
          {{- with secret "database/creds/myapp-role" -}}
          export DB_USER={{ .Data.username }}
          export DB_PASS={{ .Data.password }}
          {{- end }}
    spec:
      serviceAccountName: myapp
      containers:
        - name: myapp
          image: myapp:latest
          command: ["sh", "-c", "source /vault/secrets/db-creds && ./start.sh"]
```

```bash
# Vault 端配置 K8s Auth
vault auth enable kubernetes

vault write auth/kubernetes/config \
  kubernetes_host="https://kubernetes.default.svc:443"

vault write auth/kubernetes/role/myapp \
  bound_service_account_names=myapp \
  bound_service_account_namespaces=default \
  policies=app-policy \
  ttl=1h
```

```yaml
# 方式二: Vault CSI Provider
apiVersion: secrets-store.csi.x-k8s.io/v1
kind: SecretProviderClass
metadata:
  name: vault-db-creds
spec:
  provider: vault
  parameters:
    vaultAddress: "http://vault.vault:8200"
    roleName: "myapp"
    objects: |
      - objectName: "db-password"
        secretPath: "secret/data/myapp/config"
        secretKey: "db_password"
---
# Pod 挂载
spec:
  volumes:
    - name: secrets
      csi:
        driver: secrets-store.csi.k8s.io
        readOnly: true
        volumeAttributes:
          secretProviderClass: vault-db-creds
  containers:
    - volumeMounts:
        - name: secrets
          mountPath: "/mnt/secrets"
          readOnly: true
```

## 5. Sealed Secrets 与 SOPS 在 GitOps 中怎么用？

**回答：**

```
问题: GitOps 要求所有配置存 Git, 但 Secret 不能明文存

方案对比:
  ┌──────────────────┬──────────────────┬──────────────────┐
  │ 特性              │ Sealed Secrets   │ SOPS             │
  ├──────────────────┼──────────────────┼──────────────────┤
  │ 加密方式          │ 非对称加密        │ 对称/KMS 加密     │
  │ 密钥管理          │ K8s 集群内        │ AWS KMS/GCP/PGP  │
  │ 解密位置          │ 集群内 Controller │ CI/CD 或本地      │
  │ 支持平台          │ 仅 Kubernetes    │ 任何环境           │
  │ 与 ArgoCD         │ 原生支持          │ 需要插件          │
  │ 粒度              │ 整个 Secret      │ 字段级加密         │
  └──────────────────┴──────────────────┴──────────────────┘
```

```bash
# === Sealed Secrets ===
# 安装 Controller
helm install sealed-secrets sealed-secrets/sealed-secrets -n kube-system

# 加密 Secret
kubectl create secret generic db-creds \
  --from-literal=password=s3cret \
  --dry-run=client -o yaml | \
  kubeseal --format yaml > sealed-secret.yaml

# sealed-secret.yaml 可以安全提交到 Git
# Controller 在集群内解密成真正的 Secret

# === SOPS ===
# 用 AWS KMS 加密
sops --encrypt \
  --kms "arn:aws:kms:us-east-1:123:key/abc-def" \
  secrets.yaml > secrets.enc.yaml

# 加密后的文件 (只有 value 被加密, key 可读)
# db_password: ENC[AES256_GCM,data:abc123,iv:...,tag:...]
# api_key: ENC[AES256_GCM,data:xyz789,iv:...,tag:...]

# 解密
sops --decrypt secrets.enc.yaml

# 编辑 (自动解密 → 编辑 → 重新加密)
sops secrets.enc.yaml

# .sops.yaml 配置文件
# creation_rules:
#   - path_regex: .*\.enc\.yaml$
#     kms: "arn:aws:kms:us-east-1:123:key/abc-def"
```

```yaml
# SOPS + Kustomize + ArgoCD
# kustomization.yaml
generators:
  - ksops-generator.yaml

# ksops-generator.yaml
apiVersion: viaduct.ai/v1
kind: ksops
metadata:
  name: secret-generator
files:
  - secrets.enc.yaml
```

```python
# AWS Secrets Manager 使用
import boto3
import json

def get_secret(secret_name):
    client = boto3.client('secretsmanager', region_name='us-east-1')
    response = client.get_secret_value(SecretId=secret_name)

# 获取数据库凭证
creds = get_secret('prod/myapp/db')
db_host = creds['host']
db_password = creds['password']
```

## 7. 密钥轮换 (Key Rotation) 怎么做？

**回答：**

```
为什么要轮换:
  1. 降低凭证泄漏后的影响 (Window of Exposure)
  2. 合规要求 (PCI DSS 要求 90 天轮换)
  3. 最小化长期凭证风险

轮换策略:
  ┌──────────────────┬──────────┬──────────────────────────┐
  │ 凭证类型          │ 建议频率  │ 方法                      │
  ├──────────────────┼──────────┼──────────────────────────┤
  │ 数据库密码        │ 30-90 天 │ Vault 动态凭证 / SM 轮换   │
  │ API Key          │ 90 天    │ 双 Key 轮换               │
  │ TLS 证书         │ 90 天    │ cert-manager 自动续期      │
  │ SSH Key          │ 90 天    │ Vault SSH CA              │
  │ IAM Access Key   │ 90 天    │ 尽量用 IAM Role 代替       │
  │ 加密密钥 (KMS)   │ 365 天   │ KMS 自动轮换               │
  │ Service Account  │ 按需     │ Workload Identity          │
  └──────────────────┴──────────┴──────────────────────────┘

双 Key 无停机轮换:
  1. 生成新 Key (Key-B), 保留旧 Key (Key-A)
  2. 服务端同时接受 Key-A 和 Key-B
  3. 所有客户端切换到 Key-B
  4. 确认无 Key-A 流量后, 废弃 Key-A

  时间线:
  ──生成 Key-B──客户端切换──监控确认──废弃 Key-A──
```

```bash
# AWS IAM Access Key 轮换脚本
#!/bin/bash
USER="app-user"

# 1. 创建新 Key
NEW_KEY=$(aws iam create-access-key --user-name "$USER" --output json)
NEW_ACCESS_KEY=$(echo "$NEW_KEY" | jq -r '.AccessKey.AccessKeyId')
NEW_SECRET_KEY=$(echo "$NEW_KEY" | jq -r '.AccessKey.SecretAccessKey')

# 2. 更新到 Secrets Manager
aws secretsmanager update-secret \
  --secret-id "prod/$USER/access-key" \
  --secret-string "{\"access_key\":\"$NEW_ACCESS_KEY\",\"secret_key\":\"$NEW_SECRET_KEY\"}"

# 3. 等待应用切换 (获取新凭证)
sleep 300

# 4. 列出旧 Key 并禁用
OLD_KEYS=$(aws iam list-access-keys --user-name "$USER" \
  --query "AccessKeyMetadata[?AccessKeyId!=\`$NEW_ACCESS_KEY\`].AccessKeyId" \
  --output text)

for KEY in $OLD_KEYS; do
  aws iam update-access-key --user-name "$USER" \
    --access-key-id "$KEY" --status Inactive
done

# 5. 确认无影响后删除旧 Key
# aws iam delete-access-key --user-name "$USER" --access-key-id "$KEY"
```

## 8. Vault Transit Engine (加密即服务) 是什么？

**回答：**

```
Transit Engine: Vault 提供加密/解密 API, 应用不直接管理密钥

  传统方式:
    应用自行管理加密密钥 → 密钥存代码/配置 → 难以轮换

  Transit Engine:
    应用 → 发送明文到 Vault → Vault 加密返回密文
    应用 → 发送密文到 Vault → Vault 解密返回明文
    密钥永远不离开 Vault

使用场景:
  数据库字段加密 (信用卡号, 身份证)
  API 通信加密
  Token 签名
  数据脱敏
```

```bash
# 启用 Transit Engine
vault secrets enable transit

# 创建加密密钥
vault write -f transit/keys/myapp-key

# 加密
vault write transit/encrypt/myapp-key \
  plaintext=$(echo -n "4111-1111-1111-1111" | base64)
# 返回: ciphertext = vault:v1:xxx...

# 解密
vault write transit/decrypt/myapp-key \
  ciphertext="vault:v1:xxx..."
# 返回: plaintext = base64 编码的明文

# 密钥轮换 (旧数据仍可解密)
vault write -f transit/keys/myapp-key/rotate

# 重新加密旧数据 (用新版本密钥)
vault write transit/rewrap/myapp-key \
  ciphertext="vault:v1:xxx..."
# 返回: vault:v2:yyy... (用新密钥加密)
```

```python
# 应用集成示例
import hvac

client = hvac.Client(url='http://vault:8200', token='s.xxx')

# 加密敏感字段
def encrypt_field(plaintext):
    import base64
    encoded = base64.b64encode(plaintext.encode()).decode()
    result = client.secrets.transit.encrypt_data(
        name='myapp-key',
        plaintext=encoded
    )
    return result['data']['ciphertext']

# 解密
def decrypt_field(ciphertext):
    import base64
    result = client.secrets.transit.decrypt_data(
        name='myapp-key',
        ciphertext=ciphertext
    )
    return base64.b64decode(result['data']['plaintext']).decode()

# 存储到数据库的是密文
card_encrypted = encrypt_field("4111-1111-1111-1111")
# → "vault:v1:abc123..."

# 读取时解密
card_number = decrypt_field(card_encrypted)
# → "4111-1111-1111-1111"
```

## 9. 零信任架构 (Zero Trust) 与密钥管理？

**回答：**

```
零信任原则: "永不信任, 始终验证" (Never Trust, Always Verify)

  传统模型: 内网 = 安全, 外网 = 不安全 (城堡模型)
  零信任:   不区分内外网, 每次访问都需认证和授权

  核心原则:
    1. 验证每个请求 (身份 + 设备 + 上下文)
    2. 最小权限访问
    3. 假设已被入侵 (Assume Breach)

零信任在 DevOps 中的实践:
  ┌──────────────────┬──────────────────────────────────┐
  │ 层面              │ 实践                              │
  ├──────────────────┼──────────────────────────────────┤
  │ 身份认证          │ mTLS, OIDC, SPIFFE/SPIRE         │
  │ 网络              │ Service Mesh (Istio), 微分段      │
  │ 密钥              │ 动态凭证, 短 TTL, 自动轮换        │
  │ 访问控制          │ RBAC + ABAC, OPA Policy           │
  │ 设备              │ 设备证书, 健康检查                 │
  │ 数据              │ 传输加密 (mTLS) + 静态加密        │
  │ 监控              │ 持续验证, 异常检测                 │
  └──────────────────┴──────────────────────────────────┘

SPIFFE/SPIRE: 服务身份框架
  SPIFFE ID: spiffe://cluster.local/ns/prod/sa/myapp
  SPIRE: 颁发和管理 SVID (SPIFFE 可验证身份文档)
  用途: 服务间 mTLS 认证, 不依赖 Secret 传递
```

## 10. 密钥管理面试速答？

**回答：**

```
Q: 项目中密钥怎么管理的?
A: Vault 集中管理, K8s 通过 Agent Injector 注入
   CI/CD 用 GitHub Secrets / AWS Secrets Manager
   GitOps 场景用 Sealed Secrets 加密存 Git

Q: 为什么不用 K8s Secret?
A: K8s Secret 默认 base64 编码 (不是加密)
   需要开启 EncryptionConfiguration 加密 etcd
   缺少轮换/审计/动态凭证等高级功能

Q: Vault 高可用怎么部署?
A: Raft 存储后端 (Integrated Storage)
   3 或 5 节点集群, 自动选举 Leader
   Auto-Unseal (AWS KMS / Azure Key Vault)
   DR Replication 跨区域灾备

Q: 动态凭证 vs 静态凭证?
A: 动态: Vault 按需生成, 自动过期, 每请求唯一
   静态: 长期有效, 需手动轮换, 泄漏影响大
   推荐: 能用动态就用动态

Q: 密钥泄漏了怎么应急?
A: 1. 立即撤销/轮换泄漏的密钥
   2. 审查泄漏范围 (日志审计)
   3. 评估影响 (数据是否被访问)
   4. 修复泄漏源 (代码/配置)
   5. 加强防护 (pre-commit hook, 扫描)

Q: SOPS vs Sealed Secrets 怎么选?
A: Sealed Secrets: 纯 K8s, 简单, ArgoCD 原生支持
   SOPS: 多平台, 字段级加密, 更灵活
   推荐: K8s only → Sealed Secrets
         多环境 → SOPS + KMS
```
