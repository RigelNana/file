## 1. 常见网络攻击类型及防护？

**回答：**

```
│ 攻击类型          │ 原理                        │ 防护                         │
├──────────────────┼─────────────────────────────┼─────────────────────────────┤
│ DDoS             │ 海量请求耗尽资源             │ CDN/WAF/云防护/限流           │
│ SQL 注入          │ 恶意 SQL 拼接               │ 参数化查询/ORM/WAF           │
│ XSS              │ 注入恶意脚本                 │ 转义/CSP/HttpOnly            │
│ CSRF             │ 伪造用户请求                 │ CSRF Token/SameSite Cookie  │
│ SSRF             │ 服务端请求伪造               │ 白名单/网络隔离/禁内网访问    │

└──────────────────┴─────────────────────────────┴─────────────────────────────┘
```

## 2. SQL 注入攻击原理和防护？

**回答：**

```
原理: 用户输入被拼接到 SQL 语句中执行

攻击示例:
  输入: ' OR '1'='1' --
  拼接后: SELECT * FROM users WHERE name='' OR '1'='1' --' AND pass='xxx'
  效果: 绕过认证, 返回所有用户
```

```python
# ❌ 危险: 字符串拼接 (SQL 注入)
query = f"SELECT * FROM users WHERE name='{username}' AND pass='{password}'"
cursor.execute(query)

# ✅ 安全: 参数化查询 (Prepared Statement)
cursor.execute(
    "SELECT * FROM users WHERE name = %s AND pass = %s",
    (username, password)
)

# ✅ 安全: ORM (SQLAlchemy)
user = session.query(User).filter(
    User.name == username,
    User.password == hashed_password
).first()
```

```
防护层次:
  1. 代码层: 参数化查询 (MUST) + ORM
  2. 框架层: Django/Rails 等默认参数化
  3. WAF 层:  检测 SQL 注入特征模式
  4. DB 层:   最小权限 (应用账号不给 DROP/ALTER)
  5. 测试层: sqlmap 自动化扫描

SQL 注入类型:
  基于报错:   利用错误信息获取数据
  联合查询:   UNION SELECT 读取其他表
  盲注:       基于布尔/时间推断信息
  堆叠注入:   ; 分号执行多条语句
  带外注入:   DNS/HTTP 请求外带数据
```

## 3. XSS 跨站脚本攻击？

**回答：**

```
三种 XSS 类型:
  1. 反射型: 恶意脚本在 URL 中, 服务器原样返回
     ↳ http://site.com/search?q=<script>steal(cookie)</script>
  
  2. 存储型: 恶意脚本存入数据库, 每次访问页面都执行
     ↳ 评论/留言板中注入 <script>
  
  3. DOM 型: 前端 JS 直接操作 DOM 引发
     ↳ document.innerHTML = location.hash
```

```javascript
// ❌ 危险: 直接插入用户输入
element.innerHTML = userInput;
document.write(userInput);

// ✅ 安全: 使用 textContent (不解析 HTML)
element.textContent = userInput;

// ✅ 安全: DOMPurify 库清洗 HTML
import DOMPurify from 'dompurify';
element.innerHTML = DOMPurify.sanitize(userInput);

// ✅ 安全: React 默认转义 (除非用 dangerouslySetInnerHTML)
// Vue 默认转义 (除非用 v-html)
```

```
防护措施:
  1. 输出编码/转义:
     HTML: &lt; &gt; &amp; &quot;
     JS:   \x3C (不直接嵌入)
     URL:  encodeURIComponent()

  2. Content-Security-Policy (CSP):
     Content-Security-Policy: default-src 'self';
       script-src 'self' 'nonce-abc123';
       style-src 'self' 'unsafe-inline';
       img-src 'self' data: https:;

  3. Cookie 安全:
     Set-Cookie: session=xxx; HttpOnly; Secure; SameSite=Strict

  4. HTTP 头:
     X-Content-Type-Options: nosniff
     X-Frame-Options: DENY
     X-XSS-Protection: 0  (现代浏览器用 CSP 代替)
```

## 4. CSRF 跨站请求伪造？

**回答：**

```
攻击原理:
  1. 用户登录 bank.com (浏览器保存 Cookie)
  2. 用户访问恶意网站 evil.com
  3. evil.com 页面自动发送请求到 bank.com/transfer
  4. 浏览器自动带上 bank.com 的 Cookie
  5. 银行服务器认为是用户自己的操作

  evil.com 的恶意代码:
  <img src="https://bank.com/transfer?to=attacker&amount=10000">
  或
  <form action="https://bank.com/transfer" method="POST">
    <input type="hidden" name="to" value="attacker">
    <input type="hidden" name="amount" value="10000">
  </form>
  <script>document.forms[0].submit();</script>

防护:
  1. CSRF Token:
     服务端生成随机 Token → 嵌入表单隐藏字段
     提交时验证 Token → 攻击者无法获取正确 Token

  2. SameSite Cookie:
     Set-Cookie: session=xxx; SameSite=Strict  # 完全禁止跨站携带
     Set-Cookie: session=xxx; SameSite=Lax     # 允许 GET 导航携带
     # Chrome 默认 Lax

  3. Referer / Origin 校验:
     检查请求来源是否为合法域名

  4. 双重 Cookie 验证:
     Cookie 中设一个值, 请求头中也发一个, 服务端比对
```

## 5. DDoS 攻击防护？

**回答：**

```
DDoS 攻击层次:
  ┌──────────┬──────────────────┬───────────────────────────┐
  │ 层次      │ 攻击类型          │ 防护                      │
  ├──────────┼──────────────────┼───────────────────────────┤
  │ L3/L4    │ SYN Flood        │ SYN Cookie / 硬件防火墙    │
  │ (网络层) │ UDP Flood        │ 流量清洗 (Scrubbing)       │
  │          │ DNS Amplification│ 带宽过滤                   │
  ├──────────┼──────────────────┼───────────────────────────┤
  │ L7       │ HTTP Flood       │ WAF / 限流 / 验证码        │
  │ (应用层) │ Slowloris        │ 超时设置 / 连接数限制      │
  │          │ API 滥用          │ API Rate Limiting          │
  └──────────┴──────────────────┴───────────────────────────┘

防护架构:
  用户 → CDN (Cloudflare/AWS CloudFront)
       → WAF (规则过滤)
       → Load Balancer (连接限制)
       → 应用 (限流/验证)

  多层防御:
    1. 云厂商 DDoS 防护 (AWS Shield / Cloudflare)
    2. CDN 分散流量
    3. WAF 过滤恶意请求
    4. Nginx 限流 (limit_req / limit_conn)
    5. 应用层限流 (令牌桶/滑动窗口)
    6. 自动扩缩容 (吸收流量)
```

```nginx
# Nginx 限流配置
http {
    # 定义限流区域: 按 IP 限制, 每秒 10 个请求
    limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s;
    limit_conn_zone $binary_remote_addr zone=conn:10m;

    server {
        location /api/ {
            # 允许突发 20 个请求, 不延迟
            limit_req zone=api burst=20 nodelay;
            # 每 IP 最大 50 个并发连接
            limit_conn conn 50;
            
            proxy_pass http://backend;
        }
    }
}
```

## 6. SSL/TLS 原理与证书管理？

**回答：**

```
TLS 握手流程 (TLS 1.3 简化):
  Client                          Server
    │                               │
    ├── ClientHello ───────────────→│  (支持的密码套件, 随机数)
    │                               │
    │←── ServerHello ──────────────┤  (选择的密码套件, 随机数)
    │←── Certificate ──────────────┤  (服务器证书)
    │←── CertificateVerify ────────┤  (签名证明拥有私钥)
    │←── Finished ─────────────────┤
    │                               │
    ├── Finished ──────────────────→│
    │                               │
    ├═══ 加密通信 (对称加密) ════════┤  (AES-256-GCM)

  TLS 1.3 改进:
    握手 1-RTT (vs TLS 1.2 的 2-RTT)
    移除不安全算法 (RC4, SHA-1, RSA Key Exchange)
    0-RTT 恢复 (会话复用)

证书类型:
  DV (Domain Validation):    只验证域名所有权 (免费, Let's Encrypt)
  OV (Org Validation):       验证组织身份 (收费)
  EV (Extended Validation):  扩展验证 (绿色地址栏, 最贵)
  Wildcard:                  *.example.com (通配符)
  SAN:                       多域名 (Subject Alternative Name)
```

```bash
# Let's Encrypt 免费证书
certbot certonly --nginx -d example.com -d www.example.com

# 自动续期 (cron)
0 0 * * * certbot renew --quiet --post-hook "systemctl reload nginx"

# 查看证书信息
openssl x509 -in cert.pem -text -noout

# 检查远程证书
openssl s_client -connect example.com:443 -servername example.com

# 生成自签名证书 (仅测试)
openssl req -x509 -newkey rsa:4096 -keyout key.pem -out cert.pem \
  -sha256 -days 365 -nodes \
  -subj "/CN=localhost"
```


## 7. mTLS (双向 TLS) 是什么？

**回答：**

```
TLS vs mTLS:
  TLS:  客户端验证服务端证书 (单向)
  mTLS: 双方互相验证证书 (双向)

  普通 TLS:
    浏览器 ──验证──→ 服务器证书  ✅
    浏览器 ←─不验证─ 客户端       (任何客户端都能连)

  mTLS:
    客户端 ──验证──→ 服务器证书  ✅
    服务端 ──验证──→ 客户端证书  ✅  (只有持有有效证书的客户端)

使用场景:
  微服务间通信 (Service Mesh)
  零信任网络
  API 安全 (B2B 接口)
  IoT 设备认证

Service Mesh 中的 mTLS:
  ┌──────────────┐         ┌──────────────┐
  │   Service A  │  mTLS   │   Service B  │
  │ ┌──────────┐ │ ←─────→ │ ┌──────────┐ │
  │ │ App      │ │         │ │ App      │ │
  │ │          │ │         │ │          │ │
  │ │ Envoy    │ │         │ │ Envoy    │ │
  │ │ Sidecar  │ │         │ │ Sidecar  │ │
  │ └──────────┘ │         │ └──────────┘ │
  └──────────────┘         └──────────────┘
  
  Istio 自动:
    1. 颁发证书 (Citadel/istiod)
    2. 注入 Envoy sidecar
    3. 代理 mTLS 通信
    4. 自动轮换证书
```

```yaml
# Istio mTLS 策略
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: istio-system
spec:
  mtls:
    mode: STRICT  # 强制 mTLS (PERMISSIVE 允许明文)
```

## 8. WAF (Web Application Firewall) 是什么？

**回答：**

```
WAF: 应用层防火墙, 检测和拦截恶意 HTTP 请求

  传统防火墙: L3/L4 (IP, 端口)  → 不理解 HTTP 内容
  WAF:        L7 (HTTP)         → 分析请求内容

WAF 检测方式:
  1. 签名匹配:  已知攻击模式 (SQL 注入特征, XSS 特征)
  2. 异常检测:  偏离正常行为 (请求频率, 参数长度)
  3. 机器学习:  自动学习正常流量模式

常见 WAF 产品:
  ┌──────────────────┬──────────────────────────────┐
  │ 产品              │ 说明                          │
  ├──────────────────┼──────────────────────────────┤
  │ AWS WAF          │ 云原生 + 托管规则              │
  │ Cloudflare WAF   │ CDN 集成 + 免费版             │
  │ ModSecurity      │ 开源 + OWASP CRS 规则集       │
  │ Akamai WAF       │ 企业级                        │
  │ 阿里云 WAF       │ 国内常用                      │
  └──────────────────┴──────────────────────────────┘
```

```hcl
# AWS WAF 配置 (Terraform)
resource "aws_wafv2_web_acl" "main" {
  name  = "main-waf"
  scope = "REGIONAL"

  default_action {
    allow {}
  }

  # 限流规则: 5 分钟内超过 2000 次请求则封禁
  rule {
    name     = "rate-limit"
    priority = 1
    action {
      block {}
    }
    statement {
      rate_based_statement {
        limit              = 2000
        aggregate_key_type = "IP"
      }
    }
    visibility_config {
      sampled_requests_enabled   = true
      cloudwatch_metrics_enabled = true
      metric_name                = "RateLimit"
    }
  }

  # AWS 托管规则: SQL 注入防护
  rule {
    name     = "aws-sql-injection"
    priority = 2
    override_action {
      none {}
    }
    statement {
      managed_rule_group_statement {
        name        = "AWSManagedRulesSQLiRuleSet"
        vendor_name = "AWS"
      }
    }
    visibility_config {
      sampled_requests_enabled   = true
      cloudwatch_metrics_enabled = true
      metric_name                = "SQLInjection"
    }
  }

  visibility_config {
    sampled_requests_enabled   = true
    cloudwatch_metrics_enabled = true
    metric_name                = "MainWAF"
  }
}
```

## 9. SSRF (服务端请求伪造) 攻击？

**回答：**

```
SSRF 原理:
  攻击者控制服务端发起请求, 访问内网资源

  正常功能: 用户输入 URL → 服务器去获取内容
  攻击:     用户输入内网地址 → 服务器访问内网

  示例:
    # 正常请求
    GET /fetch?url=https://example.com/image.jpg

    # 攻击请求
    GET /fetch?url=http://169.254.169.254/latest/meta-data/  (AWS 元数据)
    GET /fetch?url=http://10.0.0.1:3306/                     (内网数据库)
    GET /fetch?url=file:///etc/passwd                         (本地文件)

  Capital One 2019 年数据泄露就是 SSRF → 获取 IAM 临时凭证

防护:
  1. 输入校验:
     URL 白名单 (只允许指定域名)
     禁止内网 IP (10.x, 172.16.x, 192.168.x, 169.254.x)
     禁止特殊协议 (file://, gopher://, dict://)
  
  2. 网络层:
     服务不应能访问 metadata endpoint
     使用 IMDSv2 (需要 Token, 防 SSRF)
     网络隔离 (处理用户 URL 的服务放 DMZ)
  
  3. AWS 特定:
     IMDSv2 强制 (HttpTokens: required)
     VPC Endpoint Policy 限制
```

```python
# ❌ 危险: 直接使用用户输入的 URL
import requests

def fetch_url(user_url):
    return requests.get(user_url).text  # SSRF!

# ✅ 安全: 校验 URL
import ipaddress
from urllib.parse import urlparse

ALLOWED_SCHEMES = {'http', 'https'}
BLOCKED_NETWORKS = [
    ipaddress.ip_network('10.0.0.0/8'),
    ipaddress.ip_network('172.16.0.0/12'),
    ipaddress.ip_network('192.168.0.0/16'),
    ipaddress.ip_network('169.254.0.0/16'),
    ipaddress.ip_network('127.0.0.0/8'),
]

def is_safe_url(url):
    parsed = urlparse(url)
    if parsed.scheme not in ALLOWED_SCHEMES:
        return False
    
    import socket
    try:
        ip = socket.gethostbyname(parsed.hostname)
        ip_addr = ipaddress.ip_address(ip)
        for network in BLOCKED_NETWORKS:
            if ip_addr in network:
                return False
    except socket.gaierror:
        return False
    
    return True

def fetch_url_safe(user_url):
    if not is_safe_url(user_url):
        raise ValueError("URL not allowed")
    return requests.get(user_url, timeout=5).text
```

## 10. 网络安全面试速答？

**回答：**

```
Q: HTTP vs HTTPS?
A: HTTPS = HTTP + TLS
   TLS 提供: 加密 + 身份验证 + 完整性
   默认端口: HTTP 80, HTTPS 443

Q: 对称加密 vs 非对称加密?
A: 对称: 同一密钥 (AES) — 快, 用于数据加密
   非对称: 公钥+私钥 (RSA/ECDSA) — 慢, 用于密钥交换
   TLS: 非对称协商 → 对称加密通信

Q: Cookie 安全属性?
A: HttpOnly: JS 不可读 (防 XSS 窃取)
   Secure:   仅 HTTPS 传输
   SameSite: 防 CSRF (Strict/Lax/None)
   Path:     限制 Cookie 路径
   Domain:   限制 Cookie 域

Q: CSP (Content Security Policy) 是什么?
A: HTTP 头, 控制页面可加载的资源来源
   防止 XSS: 限制 script 只从自身域加载
   示例: Content-Security-Policy: default-src 'self'

Q: CORS (跨域资源共享)?
A: 浏览器安全策略, 限制跨域 Ajax 请求
   服务端设置 Access-Control-Allow-Origin
   预检请求: OPTIONS (复杂请求先询问)
   不是安全机制, 是浏览器的限制

Q: VPN vs 零信任?
A: VPN: 连上就信任 (城堡模型, 内网=安全)
   零信任: 每次请求都验证 (不区分内外网)
   趋势: 零信任取代 VPN (BeyondCorp)

Q: DNS 安全?
A: DNS 劫持: 篡改解析结果 → DNSSEC
   DNS 泄漏: 暴露访问记录 → DoH/DoT
   DNS 放大: DDoS 利用 → 限制递归查询

Q: API 安全最佳实践?
A: 1. 认证: OAuth 2.0 / JWT
   2. 授权: RBAC + Scope
   3. 限流: Rate Limiting
   4. 输入校验: Schema 验证
   5. HTTPS + HSTS
   6. 日志审计
```
