## 1. count vs for_each 的区别？

**回答：**

```hcl
# count: 按数量创建
resource "aws_instance" "web" {
  count         = 3
  ami           = "ami-xxx"
  instance_type = "t3.micro"

  tags = {
    Name = "web-${count.index}"   # web-0, web-1, web-2
  }
}
# 引用: aws_instance.web[0], aws_instance.web[1]

# for_each: 按集合创建
resource "aws_instance" "web" {
  for_each      = toset(["app", "api", "worker"])
  ami           = "ami-xxx"
  instance_type = "t3.micro"

  tags = {
    Name = "web-${each.key}"      # web-app, web-api, web-worker
  }
}
# 引用: aws_instance.web["app"], aws_instance.web["api"]
```

```
对比:
  特性           count              for_each
  标识           索引 (0,1,2)       键名 (string)
  删除中间元素   后续索引偏移重建   只删除对应资源
  适用           同质资源           异构/命名资源
  数据源         number             set / map

关键区别 (面试重点):
  count = 3, 删除索引1:
    原: [0]=a, [1]=b, [2]=c
    删: [0]=a, [1]=c            ← b删了, c变成索引1, 被重建!

  for_each, 删除"b":
    原: ["a"]=a, ["b"]=b, ["c"]=c
    删: ["a"]=a, ["c"]=c        ← 只删b, c不受影响

结论: 生产环境优先用 for_each
```

## 2. for 表达式？

**回答：**

```hcl
# 列表 → 列表
locals {
  upper_names = [for name in var.names : upper(name)]
  # ["alice", "bob"] → ["ALICE", "BOB"]
}

# 列表 → 过滤列表
locals {
  large_instances = [for i in var.instances : i if i.size == "large"]
}

# 列表 → Map
locals {
  instance_map = { for i in var.instances : i.name => i.id }
  # [{name="web", id="i-1"}, ...] → {"web" = "i-1", ...}
}

# Map → Map (转换)
locals {
  upper_tags = { for k, v in var.tags : k => upper(v) }
  # {name = "web"} → {name = "WEB"}
}

# Map → 过滤 Map
locals {
  prod_tags = { for k, v in var.tags : k => v if k != "debug" }
}

# 分组 (groupby)
locals {
  by_az = { for s in var.subnets : s.az => s.id... }
  # 注意 ... 表示值为列表
  # {"us-east-1a" = ["subnet-1", "subnet-3"], "us-east-1b" = ["subnet-2"]}
}

# 嵌套 for + flatten
locals {
  all_ports = flatten([
    for sg_name, sg in var.security_groups : [
      for rule in sg.rules : {
        sg_name  = sg_name
        port     = rule.port
        protocol = rule.protocol
      }
    ]
  ])
}
```

## 3. dynamic Block？

**回答：**

```hcl
# 问题: 安全组规则数量不固定
# 不用 dynamic:
resource "aws_security_group" "web" {
  ingress {
    from_port = 80
    to_port   = 80
    protocol  = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  ingress {
    from_port = 443
    to_port   = 443
    protocol  = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  # ... 很多重复
}

# 用 dynamic:
variable "ingress_rules" {
  default = [
    { port = 80,  cidr = "0.0.0.0/0", description = "HTTP" },
    { port = 443, cidr = "0.0.0.0/0", description = "HTTPS" },
    { port = 22,  cidr = "10.0.0.0/8", description = "SSH" },
  ]
}

resource "aws_security_group" "web" {
  name   = "web-sg"
  vpc_id = var.vpc_id

  dynamic "ingress" {
    for_each = var.ingress_rules
    content {
      from_port   = ingress.value.port
      to_port     = ingress.value.port
      protocol    = "tcp"
      cidr_blocks = [ingress.value.cidr]
      description = ingress.value.description
    }
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}
```

```hcl
# iterator 自定义迭代器名
dynamic "ingress" {
  for_each = var.ingress_rules
  iterator = rule              # 使用 rule 代替 ingress

  content {
    from_port = rule.value.port
    to_port   = rule.value.port
    protocol  = "tcp"
  }
}

# 嵌套 dynamic
dynamic "setting" {
  for_each = var.settings
  content {
    name  = setting.value.name

    dynamic "value" {
      for_each = setting.value.values
      content {
        value = value.value
      }
    }
  }
}
```

## 4. 条件表达式与条件资源？

**回答：**

```hcl
# 三元运算符
locals {
  instance_type = var.environment == "production" ? "m5.xlarge" : "t3.micro"
}

# 条件创建资源
resource "aws_lb" "main" {
  count = var.create_lb ? 1 : 0
  name  = "${var.environment}-lb"
}

# 引用条件资源 (注意 [0])
output "lb_dns" {
  value = var.create_lb ? aws_lb.main[0].dns_name : null
}

# for_each 条件
resource "aws_cloudwatch_metric_alarm" "cpu" {
  for_each = var.enable_monitoring ? var.instances : {}
  # ...
}

# 条件模块
module "monitoring" {
  count  = var.enable_monitoring ? 1 : 0
  source = "./modules/monitoring"
}

# null 条件
resource "aws_instance" "web" {
  key_name = var.enable_ssh ? var.key_name : null
  # null = 不设置该属性
}
```

## 5. lifecycle 生命周期管理？

**回答：**

```hcl
resource "aws_instance" "web" {
  ami           = var.ami_id
  instance_type = var.instance_type

  lifecycle {
    # 先创建新的, 再销毁旧的 (零停机)
    create_before_destroy = true

    # 防止意外删除
    prevent_destroy = true

    # 忽略外部变更 (不回滚)
    ignore_changes = [
      tags,                    # 忽略 tag 变更
      ami,                     # 忽略 AMI 变更
      user_data,               # 忽略 user_data 变更
    ]

    # 忽略所有变更
    # ignore_changes = all

    # 触发替换 (Terraform 1.2+)
    replace_triggered_by = [
      aws_ami.custom.id,       # AMI 变更时替换实例
    ]
  }
}
```

```
各参数使用场景:
  create_before_destroy:
    RDS, ELB 等需要零停机的资源
    注意: 资源名称冲突时需要 name_prefix

  prevent_destroy:
    数据库、S3 存储桶等关键数据资源
    terraform destroy 时会报错 (需手动移除)

  ignore_changes:
    ASG 的 desired_capacity (外部自动扩缩)
    ECS 的 task_definition (CI/CD 单独更新)
    EC2 的 tags (其他工具管理标签)

  replace_triggered_by:
    AMI 更新时触发实例替换
    配置文件变更时触发服务重启
```

## 6. Provisioner 配置器？

**回答：**

```hcl
# remote-exec: 在远程机器执行命令
resource "aws_instance" "web" {
  ami           = var.ami_id
  instance_type = "t3.micro"
  key_name      = var.key_name

  provisioner "remote-exec" {
    inline = [
      "sudo yum install -y nginx",
      "sudo systemctl start nginx",
    ]

    connection {
      type        = "ssh"
      user        = "ec2-user"
      private_key = file("~/.ssh/id_rsa")
      host        = self.public_ip
    }
  }
}

# local-exec: 在本地执行命令
resource "aws_instance" "web" {
  # ...

  provisioner "local-exec" {
    command = "echo ${self.public_ip} >> hosts.txt"
  }

  # 销毁时执行
  provisioner "local-exec" {
    when    = destroy
    command = "echo 'Instance destroyed' >> cleanup.log"
  }
}

# file: 上传文件
resource "aws_instance" "web" {
  provisioner "file" {
    source      = "scripts/setup.sh"
    destination = "/tmp/setup.sh"

    connection {
      type        = "ssh"
      user        = "ec2-user"
      private_key = file("~/.ssh/id_rsa")
      host        = self.public_ip
    }
  }
}
```

```
⚠ HashiCorp 不推荐使用 Provisioner:
  替代方案:
    user_data     → EC2 启动脚本
    Packer        → 预制 AMI
    Ansible       → 配置管理
    cloud-init    → 初始化配置

  Provisioner 问题:
    ✗ 不在 plan 中体现
    ✗ 失败处理困难
    ✗ 增加执行时间
    ✗ 不可重复执行
```

## 7. terraform_data 与 null_resource？

**回答：**

```hcl
# null_resource (传统方式)
resource "null_resource" "deploy" {
  triggers = {
    version = var.app_version
  }

  provisioner "local-exec" {
    command = "deploy.sh ${var.app_version}"
  }
}

# terraform_data (Terraform 1.4+, 推荐)
resource "terraform_data" "deploy" {
  triggers_replace = [var.app_version]

  provisioner "local-exec" {
    command = "deploy.sh ${var.app_version}"
  }
}

# 区别:
# terraform_data: 内置, 无需 provider
# null_resource:  需要 hashicorp/null provider
```

```hcl
# 常见用途: 触发脚本
resource "terraform_data" "ansible" {
  triggers_replace = [
    aws_instance.web.id,
    var.ansible_playbook_hash,
  ]

  provisioner "local-exec" {
    command = "ansible-playbook -i '${aws_instance.web.public_ip},' playbook.yml"
  }

  depends_on = [aws_instance.web]
}

# 存储中间数据
resource "terraform_data" "computed" {
  input = aws_instance.web.private_ip
}
output "stored_ip" {
  value = terraform_data.computed.output
}
```

## 8. 模板文件 (templatefile)？

**回答：**

```hcl
# templates/user_data.sh.tpl
#!/bin/bash
echo "Hostname: ${hostname}"
echo "Environment: ${environment}"
echo "DB Host: ${db_host}"

%{ for port in ports ~}
iptables -A INPUT -p tcp --dport ${port} -j ACCEPT
%{ endfor ~}

%{ if enable_monitoring ~}
curl -sL https://monitoring-agent.example.com/install.sh | bash
%{ endif ~}

# main.tf
resource "aws_instance" "web" {
  user_data = templatefile("${path.module}/templates/user_data.sh.tpl", {
    hostname          = "web-server"
    environment       = var.environment
    db_host           = aws_db_instance.main.endpoint
    ports             = [80, 443, 8080]
    enable_monitoring = var.environment == "production"
  })
}
```

```hcl
# Nginx 配置模板
# templates/nginx.conf.tpl
server {
  listen ${port};
  server_name ${domain};

%{ for backend in backends ~}
  upstream ${backend.name} {
    server ${backend.host}:${backend.port};
  }
%{ endfor ~}

  location / {
    proxy_pass http://${backends[0].name};
  }
}

# 使用
locals {
  nginx_config = templatefile("${path.module}/templates/nginx.conf.tpl", {
    port     = 80
    domain   = "example.com"
    backends = [
      { name = "app", host = "10.0.1.10", port = 8080 },
      { name = "api", host = "10.0.1.20", port = 3000 },
    ]
  })
}
```

## 9. 特殊表达式与 Splat？

**回答：**

```hcl
# Splat 表达式 (*)
output "instance_ids" {
  value = aws_instance.web[*].id
  # 等价于: [for i in aws_instance.web : i.id]
}

# 仅适用于 count 资源, for_each 用 values()
output "instance_ids_for_each" {
  value = values(aws_instance.web)[*].id
}

# try() - 安全访问 (Terraform 1.0+)
locals {
  port = try(var.config.port, 8080)
  # config.port 存在则用, 否则默认 8080
}

# can() - 检查表达式是否合法
locals {
  is_valid_cidr = can(cidrhost(var.cidr, 0))
}

# coalesce() - 返回第一个非空值
locals {
  name = coalesce(var.custom_name, var.default_name, "unnamed")
}

# one() - 将 0-1 元素列表转为单值
data "aws_ami" "latest" {
  most_recent = true
  filter { ... }
}

# lookup() - Map 安全查找
locals {
  ami = lookup(var.ami_map, var.region, "ami-default")
}

# element() - 循环索引
resource "aws_instance" "web" {
  count             = 6
  availability_zone = element(var.azs, count.index)
  # 3 个 AZ 循环: az-a, az-b, az-c, az-a, az-b, az-c
}

# cidrsubnet() - 子网计算
locals {
  subnets = [for i in range(3) : cidrsubnet("10.0.0.0/16", 8, i)]
  # ["10.0.0.0/24", "10.0.1.0/24", "10.0.2.0/24"]
}
```

## 10. moved Block 代码重构？

**回答：**

```hcl
# 场景 1: 重命名资源
# 旧: resource "aws_instance" "web" { ... }
# 新: resource "aws_instance" "application" { ... }

moved {
  from = aws_instance.web
  to   = aws_instance.application
}

# 场景 2: 移入 Module
moved {
  from = aws_instance.web
  to   = module.compute.aws_instance.web
}

# 场景 3: count → for_each
# 旧: aws_instance.web[0], aws_instance.web[1]
# 新: aws_instance.web["app"], aws_instance.web["api"]
moved {
  from = aws_instance.web[0]
  to   = aws_instance.web["app"]
}
moved {
  from = aws_instance.web[1]
  to   = aws_instance.web["api"]
}

# 场景 4: 模块重命名
moved {
  from = module.old_vpc
  to   = module.networking
}
```

```
moved vs terraform state mv:
  moved block:
    ✓ 代码化, 可审查
    ✓ 团队协作安全
    ✓ plan 中可预览
    ✓ 适合 CI/CD

  terraform state mv:
    ✓ 临时操作
    ✗ 不可审查
    ✗ 团队需要协调

  推荐: 优先用 moved block
```
