# 制品管理与镜像仓库

---

## 1. 什么是制品 (Artifact)？类型有哪些？

**回答：**

```
制品 = CI/CD 流水线的构建产出物

类型                 示例                          存储位置
──────────          ──────────────────           ──────────
容器镜像             Docker Image                  Container Registry
包/库                npm, PyPI, Maven JAR          Package Registry
二进制文件           Go binary, .exe               通用制品仓库
Helm Chart          Chart tgz                     Chart Repository
OS 包               .deb, .rpm                    APT/YUM Repository
文档                 API docs, 静态站点             S3 / Pages
```

## 2. 容器镜像仓库 (Container Registry) 对比？

| **JFrog Artifactory** | 商业 | 全格式, 元数据, 高级安全 | 大企业 |

## 3. Harbor 企业级镜像仓库？

**回答：**

```
Harbor 核心功能:
  镜像管理          → 项目/仓库/Tag/Label 管理
  漏洞扫描          → Trivy 集成, 自动扫描
  镜像签名          → Cosign / Notary, 确保镜像可信
  RBAC             → 项目级别细粒度权限
  镜像复制          → 跨 Harbor 实例复制 (多数据中心)
  配额管理          → 项目存储配额
  垃圾回收          → 清理未使用的镜像层
  Webhook          → 推送/扫描事件通知
  Robot Account    → CI/CD 专用账户

部署:
  helm install harbor harbor/harbor \
    --set expose.type=ingress \
    --set expose.ingress.hosts.core=harbor.example.com \
    --set persistence.persistentVolumeClaim.registry.size=100Gi
```

```bash
# Harbor 使用
docker login harbor.example.com
docker tag myapp:v1 harbor.example.com/myproject/myapp:v1
docker push harbor.example.com/myproject/myapp:v1

# Robot Account (CI/CD 推荐)
# Harbor UI → 项目 → Robot Accounts → 创建
# 权限: push, pull (最小权限)
```

## 4. 镜像版本管理策略？

**回答：**

```
❌ 错误做法:
  image: myapp:latest          # 不可追溯, 不可复现
  image: myapp                 # 同上 (默认 latest)

✅ 正确做法:
  image: myapp:v1.2.3          # 语义版本
  image: myapp:abc1234         # Git commit SHA
  image: myapp:20240115-abc1234 # 日期 + SHA

推荐标记策略:
  1. Git SHA (短)              → 唯一标识构建
  2. 语义版本 (Semver)          → 发布版本
  3. 分支名                    → 开发/测试环境
  4. latest                   → 仅用于开发, 不用于生产
```

```yaml
# CI 中的标记策略
# GitHub Actions 示例
- name: Docker meta
  id: meta
  uses: docker/metadata-action@v5
  with:
    images: ghcr.io/${{ github.repository }}
    tags: |
      type=sha,prefix=                    # abc1234
      type=ref,event=branch               # main, develop
      type=semver,pattern={{version}}      # 1.2.3 (Tag 触发)
      type=semver,pattern={{major}}.{{minor}} # 1.2

- name: Build and push
  uses: docker/build-push-action@v5
  with:
    push: true
    tags: ${{ steps.meta.outputs.tags }}
```

## 5. 镜像安全扫描？

**回答：**

```
扫描工具:
  Trivy          → 开源, 速度快, CI 友好 (推荐)
  Grype          → Anchore 开源, 轻量
  Snyk Container → 商业, 修复建议
  Clair          → CoreOS/Quay, 老牌
  Harbor         → 内置 Trivy 扫描

扫描内容:
  OS 包漏洞       → CVE 数据库
  语言依赖漏洞     → npm, pip, maven 等
  配置错误        → Dockerfile 最佳实践
  密钥泄露        → 镜像中的密钥/密码
```

```yaml
# Trivy CI 集成 (GitHub Actions)
- name: Trivy Scan
  uses: aquasecurity/trivy-action@master
  with:
    image-ref: 'myapp:${{ github.sha }}'
    format: 'sarif'
    output: 'trivy-results.sarif'
    severity: 'CRITICAL,HIGH'
    exit-code: '1'                         # 发现漏洞则失败

- name: Upload Trivy results
  uses: github/codeql-action/upload-sarif@v3
  with:
    sarif_file: 'trivy-results.sarif'

# Trivy CLI
trivy image myapp:v1
trivy image --severity HIGH,CRITICAL --exit-code 1 myapp:v1
trivy fs .                                 # 扫描本地文件系统
trivy config .                             # 扫描 IaC 配置
```

## 6. 镜像签名与验证？

**回答：**

```
目的: 确保部署的镜像确实来自可信的 CI/CD 流水线

工具:
  Cosign (Sigstore)   → 推荐, CNCF 项目, 无密钥签名 (Keyless)
  Notary v2           → Docker 原生, OCI 规范

流程:
  CI 构建镜像 → 推送 → 签名 → Registry
  部署时 → 验证签名 → 通过则部署
```

```bash
# Cosign 签名
cosign sign --key cosign.key registry.example.com/myapp:v1

# Cosign 无密钥签名 (推荐, 基于 OIDC)
cosign sign registry.example.com/myapp:v1
# 自动通过 Fulcio CA 签名, 记录到 Rekor 透明日志

# 验证
cosign verify --key cosign.pub registry.example.com/myapp:v1
```

```yaml
# Kyverno 策略 — 强制镜像签名验证
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: verify-image-signature
spec:
  validationFailureAction: Enforce
  rules:
    - name: verify-cosign
      match:
        any:
          - resources:
              kinds: [Pod]
      verifyImages:
        - imageReferences:
            - "registry.example.com/*"
          attestors:
            - entries:
                - keys:
                    publicKeys: |
                      -----BEGIN PUBLIC KEY-----
                      ...
                      -----END PUBLIC KEY-----
```

## 7. 制品生命周期管理？

**回答：**

```
策略                   说明
──────────            ──────────────────────────
保留策略               保留最近 N 个版本或 N 天内的制品
Tag 不可变             推送后禁止同 Tag 覆盖 (Immutable Tags)
清理策略               定期删除未引用的镜像层 (Garbage Collection)
漏洞修复               定期重建 base image, 修复已知漏洞
升级基线               定期更新基础镜像到最新安全版本
```

```bash
# Harbor 清理策略 (项目级)
# Harbor UI → 项目 → Tag Retention
# 保留最近 10 个 Tag
# 保留最近 30 天的 Tag

# ECR Lifecycle Policy
aws ecr put-lifecycle-policy --repository-name myapp --lifecycle-policy-text '{
  "rules": [
    {
      "rulePriority": 1,
      "description": "Keep last 20 images",
      "selection": {
        "tagStatus": "tagged",
        "tagPrefixList": ["v"],
        "countType": "imageCountMoreThan",
        "countNumber": 20
      },
      "action": { "type": "expire" }
    },
    {
      "rulePriority": 2,
      "description": "Remove untagged after 7 days",
      "selection": {
        "tagStatus": "untagged",
        "countType": "sinceImagePushed",
        "countUnit": "days",
        "countNumber": 7
      },
      "action": { "type": "expire" }
    }
  ]
}'
```

## 8. 多架构镜像构建？

**回答：**

```bash
# Docker Buildx — 多架构构建
docker buildx create --name multiarch --use
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  -t registry.example.com/myapp:v1 \
  --push .
```

```yaml
# GitHub Actions 多架构构建
- name: Set up QEMU
  uses: docker/setup-qemu-action@v3

- name: Set up Docker Buildx
  uses: docker/setup-buildx-action@v3

- name: Build and push multi-arch
  uses: docker/build-push-action@v5
  with:
    platforms: linux/amd64,linux/arm64
    push: true
    tags: myapp:v1
    cache-from: type=gha
    cache-to: type=gha,mode=max
```

## 9. 通用制品仓库 (Nexus/Artifactory)？

**回答：**

```
Nexus Repository:
  开源版 (OSS) + 商业版 (Pro)
  支持格式: Docker, npm, PyPI, Maven, NuGet, Helm, APT, YUM, raw
  适用: 统一管理所有制品类型

JFrog Artifactory:
  商业为主 (开源版功能有限)
  支持 30+ 包格式
  高级: 元数据, Build Info, Xray 安全扫描
  适用: 大企业, 完整 DevOps 平台

对比:
  Nexus OSS (免费)    → 中小团队, 基本需求
  Nexus Pro (付费)    → 高可用, 复制, 高级安全
  Artifactory (付费)  → 企业级, 全功能
  Harbor (免费)       → 只做 Docker Registry
```

```yaml
# Maven + Nexus 集成
# settings.xml
<servers>
  <server>
    <id>nexus</id>
    <username>${env.NEXUS_USER}</username>
    <password>${env.NEXUS_PASS}</password>
  </server>
</servers>

# pom.xml
<distributionManagement>
  <repository>
    <id>nexus</id>
    <url>https://nexus.example.com/repository/maven-releases/</url>
  </repository>
</distributionManagement>

# CI 发布
mvn deploy -DskipTests
```

## 10. 镜像优化与最佳实践？

**回答：**

```dockerfile
# ===== 多阶段构建 (减小镜像) =====
# Stage 1: Build
FROM golang:1.22-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download                           # 利用缓存
COPY . .
RUN CGO_ENABLED=0 go build -o /app/server .

# Stage 2: Runtime
FROM alpine:3.19
RUN apk --no-cache add ca-certificates
COPY --from=builder /app/server /usr/local/bin/
USER 1000:1000                                # 非 root
EXPOSE 8080
ENTRYPOINT ["server"]
```

```
镜像优化清单:
  □ 使用多阶段构建
  □ 使用最小基础镜像 (alpine, distroless, scratch)
  □ 合并 RUN 指令 (减少层数)
  □ .dockerignore 排除不需要的文件
  □ 固定基础镜像版本 (不用 latest)
  □ COPY 之前先 COPY 依赖文件 (利用缓存)
  □ 非 root 用户运行
  □ 不在镜像中包含密钥/凭证

镜像大小对比:
  基础镜像        大小
  ubuntu:22.04    77 MB
  debian:12       139 MB
  alpine:3.19     7 MB
  distroless      ~2 MB (Go), ~50 MB (Python/Java)
  scratch         0 MB (仅静态二进制)
```
