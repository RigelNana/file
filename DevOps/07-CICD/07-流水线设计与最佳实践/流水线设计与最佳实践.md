# 流水线设计与最佳实践

---

## 1. 完整 CI/CD 流水线应包含哪些阶段？

**回答：**

```
┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐
│ Source │→│ Build  │→│ Test   │→│Scan &  │→│Package │→│Deploy  │
│        │ │        │ │        │ │Quality │ │& Push  │ │        │
└────────┘ └────────┘ └────────┘ └────────┘ └────────┘ └────────┘

详细:
1. Source (代码检出)
   - Git clone / checkout
   - Submodule 初始化
   - 分支/Tag/PR 触发

2. Build (构建)
   - 依赖安装 (npm ci, pip install, mvn dependency:resolve)
   - 编译 (javac, go build, tsc)
   - 静态资源构建 (webpack, vite)

3. Test (测试)
   - Lint / 格式检查
   - 单元测试 + 覆盖率
   - 集成测试

4. Scan & Quality (扫描与质检)
   - SAST 静态安全扫描
   - 依赖漏洞扫描
   - 代码质量门控 (SonarQube)

5. Package & Push (打包推送)
   - Docker build + push
   - Helm chart 打包
   - 制品上传

6. Deploy (部署)
   - Staging 自动部署 + 冒烟测试
   - Production 手动审批/自动部署
   - 上线验证 + 监控
```

## 2. 流水线性能优化？

**回答：**

```
策略                   方法                              效果
──────────            ──────────────────────           ──────
并行执行               parallel stages / matrix          2-5x 加速
缓存                  依赖缓存 (node_modules, .m2)       减少下载
增量构建               只构建变更模块                      Monorepo 适用
Docker 缓存            多阶段构建 + --cache-from           减少构建
制品传递               artifacts 在 stages 间传递          避免重复构建
快速失败               Lint 和单元测试最先执行              早发现问题
只触发相关流水线        文件变更过滤 (changes/paths)         避免无关构建
Runner/Agent 缓存      本地 Docker 层缓存                  减少拉取
```

```yaml
# GitHub Actions 优化示例
jobs:
  lint:                                 # 1. Lint 最先 (秒级)
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          cache: 'npm'                  # 缓存 node_modules
      - run: npm ci
      - run: npm run lint

  test:
    needs: lint                         # 2. Lint 通过后才测试
    runs-on: ubuntu-latest
    strategy:
      matrix:
        shard: [1, 2, 3, 4]            # 测试分片并行
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          cache: 'npm'
      - run: npm ci
      - run: npm test -- --shard=${{ matrix.shard }}/4

  build:
    needs: test
    if: github.ref == 'refs/heads/main'  # 只在 main 分支构建
    runs-on: ubuntu-latest
    steps:
      - uses: docker/build-push-action@v5
        with:
          cache-from: type=gha          # GitHub Actions 缓存
          cache-to: type=gha,mode=max
          push: true
          tags: myapp:${{ github.sha }}
```

## 3. Monorepo 流水线设计？

**回答：**

```
Monorepo 结构:
  repo/
  ├── services/
  │   ├── user-service/
  │   ├── order-service/
  │   └── payment-service/
  ├── libs/
  │   ├── common/
  │   └── auth/
  └── infrastructure/
      └── terraform/

挑战:
  - 不是每次提交都需要构建所有服务
  - 需要检测哪些服务/库发生了变更
  - 共享库变更可能影响多个服务
```

```yaml
# GitHub Actions — 路径过滤
name: User Service CI
on:
  push:
    paths:
      - 'services/user-service/**'
      - 'libs/common/**'              # 共享库变更也触发
      - 'libs/auth/**'

# GitLab CI — changes 规则
user-service-build:
  rules:
    - changes:
        - services/user-service/**
        - libs/common/**

# 高级: 动态决定构建哪些服务
detect-changes:
  script: |
    CHANGED=$(git diff --name-only HEAD~1)
    if echo "$CHANGED" | grep -q "services/user-service"; then
      echo "BUILD_USER=true" >> build.env
    fi
  artifacts:
    reports:
      dotenv: build.env
```

## 4. 多环境流水线设计？

**回答：**

```
环境流:

  Feature Branch → Dev → QA → Staging → Production
       │           │     │      │          │
       └── PR CI   └ Auto └ Auto └ Auto    └ Manual Approval

每个环境:
  独立 Namespace / 集群
  独立配置 (ConfigMap/Secret)
  独立数据库
  独立域名
```

```yaml
# GitLab CI 多环境
stages: [build, test, deploy-dev, deploy-staging, deploy-prod]

.deploy-template:
  image: bitnami/kubectl:latest
  script:
    - kubectl set image deployment/myapp myapp=$IMAGE -n $NAMESPACE
    - kubectl rollout status deployment/myapp -n $NAMESPACE

deploy-dev:
  extends: .deploy-template
  stage: deploy-dev
  variables:
    NAMESPACE: development
  environment:
    name: development
    url: https://dev.example.com
  rules:
    - if: '$CI_COMMIT_BRANCH == "develop"'

deploy-staging:
  extends: .deploy-template
  stage: deploy-staging
  variables:
    NAMESPACE: staging
  environment:
    name: staging
    url: https://staging.example.com
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'

deploy-prod:
  extends: .deploy-template
  stage: deploy-prod
  variables:
    NAMESPACE: production
  environment:
    name: production
    url: https://example.com
  when: manual
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'
```

## 5. 质量门控 (Quality Gate)？

**回答：**

```
质量门控 = 不满足标准则阻止部署

门控指标              阈值示例                    工具
──────────           ──────────────             ──────────
代码覆盖率            ≥ 80%                      JaCoCo, coverage.py, nyc
测试通过率            100%                       JUnit, pytest
代码质量              SonarQube Quality Gate      SonarQube
安全漏洞              0 Critical/High             Trivy, Snyk
依赖漏洞              0 Known Vulnerabilities      Dependabot, Snyk
代码规范              0 Lint Errors               ESLint, Flake8
重复代码              < 3%                        SonarQube
技术债务              < 30 min                    SonarQube
```

```yaml
# SonarQube 集成 (GitHub Actions)
- name: SonarQube Scan
  uses: SonarSource/sonarqube-scan-action@v2
  env:
    SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
    SONAR_HOST_URL: ${{ secrets.SONAR_HOST_URL }}

- name: SonarQube Quality Gate
  uses: SonarSource/sonarqube-quality-gate-action@v1
  timeout-minutes: 5
  env:
    SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
  # 不通过则 Job 失败 → 阻止合并/部署
```

```yaml
# 覆盖率门控
- name: Check Coverage
  run: |
    COV=$(pytest --cov=src --cov-report=term | grep TOTAL | awk '{print $NF}' | tr -d '%')
    if [ "$COV" -lt 80 ]; then
      echo "Coverage $COV% < 80% threshold"
      exit 1
    fi
```

## 6. 通知与告警？

**回答：**

```
CI/CD 通知场景:
  构建失败      → 立即通知开发者
  部署完成      → 通知团队
  部署失败      → 通知 + 自动回滚
  安全漏洞      → 通知安全团队
  审批请求      → 通知审批人

通知渠道:
  Slack / DingTalk / 飞书 / 企业微信
  Email
  PagerDuty (告警)
  GitHub / GitLab Comment (PR 评论)
```

```yaml
# GitHub Actions Slack 通知
- name: Notify Slack
  if: failure()
  uses: slackapi/slack-github-action@v1
  with:
    payload: |
      {
        "text": "❌ Build failed: ${{ github.repository }}",
        "blocks": [
          {
            "type": "section",
            "text": {
              "type": "mrkdwn",
              "text": "*Build Failed* on `${{ github.ref_name }}`\n<${{ github.server_url }}/${{ github.repository }}/actions/runs/${{ github.run_id }}|View Run>"
            }
          }
        ]
      }
  env:
    SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK }}
```

## 7. CI/CD 指标与改进？

**回答：**

```
DORA 指标 (DevOps Research and Assessment):

指标                   定义                         精英团队目标
────────────          ────────────────────         ─────────────
部署频率               多久部署一次到生产               按需 (每天多次)
变更前置时间            提交代码到部署到生产的时间         < 1 小时
变更失败率             部署导致的生产故障比例             < 5%
服务恢复时间            生产故障到恢复的时间              < 1 小时

如何度量:
  GitLab → Value Stream Analytics (内置)
  GitHub → DORA Metrics (GitHub Insights)
  自建 → 流水线数据 + 监控数据

其他 CI/CD 指标:
  构建时间              平均 Pipeline 执行时间
  构建成功率            Pipeline 成功/总次数
  测试覆盖率趋势         覆盖率随时间的变化
  排队时间              Job 等待 Runner 的时间
  制品大小              Docker 镜像大小趋势
```

## 8. 流水线模板化与标准化？

**回答：**

```
问题: 20个微服务, 每个 Jenkinsfile/gitlab-ci.yml 都差不多
解决: 模板化 → 统一标准, 减少重复

方式:
  Jenkins     → Shared Library
  GitLab CI   → include + extends + CI/CD Template 项目
  GitHub Actions → Reusable Workflows + Composite Actions
```

```yaml
# GitLab CI 模板项目: devops/ci-templates

# templates/python-service.yml
.python-ci:
  image: python:3.12
  cache:
    key: ${CI_COMMIT_REF_SLUG}
    paths: [.pip-cache/]
  variables:
    PIP_CACHE_DIR: "$CI_PROJECT_DIR/.pip-cache"

.python-build:
  extends: .python-ci
  stage: build
  script:
    - pip install -r requirements.txt
    - python -m build

.python-test:
  extends: .python-ci
  stage: test
  script:
    - pip install -r requirements.txt -r requirements-test.txt
    - pytest tests/ --junitxml=report.xml --cov=src
  artifacts:
    reports:
      junit: report.xml

# 各项目中使用:
include:
  - project: 'devops/ci-templates'
    ref: v1.0
    file: '/templates/python-service.yml'

build:
  extends: .python-build

test:
  extends: .python-test
```

## 9. Pipeline 调试与排错？

**回答：**

```
常见问题              排查方法
──────────           ──────────────────────────
Job 被卡住             检查 Runner 状态, 排队中?
构建失败               查看 Job 日志, 本地复现
权限不足               检查 Runner 权限, 凭证/Token 是否过期
缓存失效               清除缓存重试
Docker 构建失败          检查 Dockerfile, 网络连接
部署失败               kubectl describe, events 查看
超时                  增加 timeout, 检查网络
```

```bash
# 本地调试

# GitLab CI — 使用 gitlab-runner exec
gitlab-runner exec docker build

# GitHub Actions — 使用 act (本地运行 Actions)
act -j test                           # 运行特定 Job
act push                              # 模拟 push 事件
act --secret-file .secrets             # 提供 Secrets

# Jenkins — 使用 Replay 功能
# Jenkins UI → Build → Replay → 修改 Jenkinsfile → Run

# 通用调试技巧:
# 1. 添加 debug 输出
- run: |
    echo "PATH: $PATH"
    echo "PWD: $(pwd)"
    ls -la
    env | sort

# 2. SSH 进入 Runner (GitHub Actions debug)
- uses: mxschmitt/action-tmate@v3
  if: failure()

# 3. 查看 Pipeline 可视化 (DAG)
# GitLab: CI/CD → Pipelines → Pipeline Graph
# GitHub: Actions → Workflow Run → Graph
```

## 10. CI/CD 最佳实践清单？

**回答：**

```
===== 流水线设计 =====
□ Pipeline as Code (版本控制)
□ 快速反馈 (目标 < 10 分钟)
□ Lint/单元测试最先执行 (fail fast)
□ 并行执行无依赖的阶段
□ 缓存依赖 (npm, pip, maven)
□ 模板化标准流水线

===== 构建 =====
□ 可重复构建 (固定依赖版本)
□ Docker 多阶段构建 (减小镜像)
□ 镜像使用具体 Tag (不用 latest)
□ 构建编号/Git SHA 作为版本标识

===== 测试 =====
□ 测试金字塔 (单元 > 集成 > E2E)
□ 代码覆盖率门控 (≥ 80%)
□ 测试并行化
□ 测试报告上传 (JUnit XML)

===== 安全 =====
□ Secrets 使用 CI/CD 平台管理
□ 日志中不打印敏感信息
□ Runner/Agent 最小权限
□ SAST + 依赖扫描 + 镜像扫描
□ 分支保护 + PR 审查

===== 部署 =====
□ 零停机部署 (readinessProbe + PDB)
□ 回滚方案就绪
□ 生产部署需审批 (Continuous Delivery)
□ 上线后监控告警
□ 金丝雀/蓝绿发布 (高风险变更)

===== 运维 =====
□ DORA 指标追踪
□ 构建历史清理策略
□ Runner 定期维护
□ Pipeline 模板定期更新
```
