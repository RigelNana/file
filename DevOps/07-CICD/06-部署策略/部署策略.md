# 部署策略

---

## 1. 常见部署策略对比？

**回答：**

| 策略 | 说明 | 零停机 | 回滚速度 | 资源消耗 | 风险 |
|------|------|--------|----------|----------|------|
| **重建 (Recreate)** | 先停旧版再启新版 | ❌ | 快 | 低 | 有停机 |
| **滚动更新 (Rolling)** | 逐步替换实例 | ✅ | 慢 | 低 | 新旧共存 |
| **蓝绿 (Blue/Green)** | 两套环境切换流量 | ✅ | 秒级 | 2x | 资源翻倍 |
| **金丝雀 (Canary)** | 小比例流量给新版 | ✅ | 快 | 低~中 | 实现复杂 |
| **A/B 测试** | 按条件路由不同版本 | ✅ | 快 | 中 | 需路由能力 |
| **影子 (Shadow)** | 复制流量到新版但不返回 | ✅ | 快 | 2x | 数据副作用 |

```
风险从低到高:
  重建 > 滚动 > 蓝绿 > 金丝雀

复杂度从低到高:
  重建 < 滚动 < 蓝绿 < 金丝雀 < A/B
```

## 2. 滚动更新 (Rolling Update) 详解？

**回答：**

K8s Deployment 的默认策略。

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 10
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 25%            # 最多额外创建 25% Pod (3个)
      maxUnavailable: 25%       # 最多 25% Pod 不可用 (2个)
  template:
    spec:
      containers:
        - name: myapp
          image: myapp:v2
          readinessProbe:       # 必须配置! 控制流量切换
            httpGet:
              path: /healthz
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 5
      minReadySeconds: 30       # Pod Ready 后等待 30s 再继续
```

```
滚动过程 (10 replicas, maxSurge=3, maxUnavailable=2):

Step 1: 创建 3 个 v2 Pod
  v1: ██████████ (10)
  v2: ███░░░░░░░ (3 creating)
  可用: 10

Step 2: v2 Ready 后, 停止 2 个 v1
  v1: ████████░░ (8)
  v2: ███        (3)
  可用: 11

Step 3: 继续滚动...
  ...

Final: 所有切换到 v2
  v1: (0)
  v2: ██████████ (10)
```

### 回滚

```bash
kubectl rollout undo deployment/myapp                  # 回滚上一版本
kubectl rollout undo deployment/myapp --to-revision=3  # 回滚指定版本
kubectl rollout history deployment/myapp               # 查看历史
kubectl rollout status deployment/myapp                # 查看状态
kubectl rollout pause deployment/myapp                 # 暂停
kubectl rollout resume deployment/myapp                # 恢复
```

## 3. 蓝绿部署 (Blue/Green) 详解？

**回答：**

```
原理:
  维护两套完全相同的环境 (Blue + Green)
  Blue = 当前生产
  Green = 新版本
  验证 Green OK 后, 切换流量 (修改 Service selector / LB)
  出问题 → 立即切回 Blue (秒级回滚)

    ┌──────────┐
    │   LB     │
    │ / Service│
    └────┬─────┘
         │ selector: version=green  ←── 切换这里
    ┌────┴─────┐     ┌──────────┐
    │ Blue     │     │ Green    │
    │ v1 (3x)  │     │ v2 (3x)  │
    └──────────┘     └──────────┘
```

```yaml
# Blue Deployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-blue
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
      version: blue
  template:
    metadata:
      labels:
        app: myapp
        version: blue
    spec:
      containers:
        - name: myapp
          image: myapp:v1
---
# Green Deployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-green
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
      version: green
  template:
    metadata:
      labels:
        app: myapp
        version: green
    spec:
      containers:
        - name: myapp
          image: myapp:v2
---
# Service — 切换 selector
apiVersion: v1
kind: Service
metadata:
  name: myapp
spec:
  selector:
    app: myapp
    version: green    # blue → green 切换流量
  ports:
    - port: 80
      targetPort: 8080
```

```bash
# 切换流量
kubectl patch svc myapp -p '{"spec":{"selector":{"version":"green"}}}'

# 回滚
kubectl patch svc myapp -p '{"spec":{"selector":{"version":"blue"}}}'

# 验证OK后, 缩容旧版本
kubectl scale deployment myapp-blue --replicas=0
```

## 4. 金丝雀部署 (Canary) 详解？

**回答：**

先将小比例流量发送到新版本，逐步扩大。

### 方式一: K8s 原生 (副本数控制)

```yaml
# Stable: 9 副本
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-stable
spec:
  replicas: 9
  selector:
    matchLabels:
      app: myapp
      track: stable
  template:
    metadata:
      labels:
        app: myapp
        track: stable
    spec:
      containers:
        - name: myapp
          image: myapp:v1
---
# Canary: 1 副本 (10% 流量)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-canary
spec:
  replicas: 1
  selector:
    matchLabels:
      app: myapp
      track: canary
  template:
    metadata:
      labels:
        app: myapp
        track: canary
    spec:
      containers:
        - name: myapp
          image: myapp:v2
---
# Service 匹配 app: myapp (包含 stable + canary)
apiVersion: v1
kind: Service
metadata:
  name: myapp
spec:
  selector:
    app: myapp          # 匹配所有 track
  ports:
    - port: 80
```

### 方式二: Istio 流量管理

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: myapp
spec:
  hosts: [myapp]
  http:
    - route:
        - destination:
            host: myapp
            subset: stable
          weight: 90          # 90% 流量
        - destination:
            host: myapp
            subset: canary
          weight: 10          # 10% 流量
---
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: myapp
spec:
  host: myapp
  subsets:
    - name: stable
      labels:
        version: v1
    - name: canary
      labels:
        version: v2
```

### 方式三: Argo Rollouts

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: myapp
spec:
  replicas: 10
  strategy:
    canary:
      steps:
        - setWeight: 5
        - pause: { duration: 2m }
        - analysis:                    # 自动分析
            templates:
              - templateName: success-rate
            args:
              - name: service-name
                value: myapp
        - setWeight: 20
        - pause: { duration: 5m }
        - setWeight: 50
        - pause: { duration: 5m }
        - setWeight: 100

---
# AnalysisTemplate — 自动化金丝雀验证
apiVersion: argoproj.io/v1alpha1
kind: AnalysisTemplate
metadata:
  name: success-rate
spec:
  args:
    - name: service-name
  metrics:
    - name: success-rate
      interval: 1m
      successCondition: result[0] >= 0.95  # 成功率 >= 95%
      provider:
        prometheus:
          address: http://prometheus:9090
          query: |
            sum(rate(http_requests_total{service="{{args.service-name}}", code=~"2.."}[5m]))
            /
            sum(rate(http_requests_total{service="{{args.service-name}}"}[5m]))
```

## 5. 渐进式交付 (Progressive Delivery) 工具对比？

**回答：**

| 工具 | 类型 | 支持策略 | 分析能力 | 流量管理 |
|------|------|----------|----------|----------|
| **Argo Rollouts** | K8s CRD | Canary, Blue-Green | AnalysisTemplate | Istio, Nginx, ALB |
| **Flagger** | K8s Controller | Canary, A/B, Blue-Green | Prometheus, Datadog | Istio, Linkerd, Nginx, Contour |
| **Istio** | Service Mesh | 流量切分 | 手动 | 原生 VirtualService |
| **Spinnaker** | CD 平台 | 全部策略 | 自动化判断 | 多云 LB |

```
选型:
  已用 ArgoCD          → Argo Rollouts (无缝集成)
  已用 Istio/Linkerd   → Flagger (Mesh 原生)
  多云部署             → Spinnaker
  只需简单金丝雀        → K8s 原生副本数控制
```

## 6. Feature Flags (功能开关)？

**回答：**

```
Feature Flags = 代码中的开关, 运行时控制功能是否启用

场景:
  Trunk-Based Development → 功能未完成但代码已合并 → Flag 关闭
  金丝雀发布              → 只对部分用户开放新功能
  A/B 测试               → 不同用户看到不同功能
  快速回滚               → 不用重新部署, 关闭 Flag 即可
  灰度发布               → 按比例/条件逐步开放

工具:
  LaunchDarkly     → 商业, 功能丰富
  Unleash          → 开源, 自托管
  Flagsmith        → 开源, 多平台 SDK
  OpenFeature      → 标准规范, 统一 API

与部署策略结合:
  传统金丝雀: 基础设施层面 (流量切分)
  Feature Flag: 应用层面 (代码开关)
  最佳实践: 两者结合使用
```

## 7. 回滚策略？

**回答：**

```
场景                 回滚方式                          耗时
──────────          ────────────────────────         ──────
K8s Deployment       kubectl rollout undo              分钟级
蓝绿部署             切换 Service selector              秒级
金丝雀               停止 Rollout, 自动回滚              秒~分钟级
Helm                 helm rollback                     分钟级
ArgoCD               argocd app rollback / Git Revert   分钟级
Feature Flag         关闭 Flag                          秒级
数据库变更           不可自动回滚 (需手动)                分钟~小时
```

```bash
# K8s 回滚
kubectl rollout undo deployment/myapp
kubectl rollout history deployment/myapp

# Helm 回滚
helm rollback myapp 1
helm history myapp

# ArgoCD 回滚 (推荐: Git Revert 而非 ArgoCD rollback)
git revert HEAD
git push                    # ArgoCD 自动同步

# Argo Rollouts 回滚
kubectl argo rollouts abort myapp      # 中止当前发布
kubectl argo rollouts undo myapp       # 回滚

# 自动回滚 (HPA + Probe)
# readinessProbe 失败 → Pod 从 Service Endpoints 移除
# livenessProbe 失败 → 容器重启
# 滚动更新中 Probe 失败 → 更新暂停
```

## 8. 零停机部署的关键配置？

**回答：**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0        # 关键: 不允许不可用 → 零停机
  template:
    spec:
      terminationGracePeriodSeconds: 60
      containers:
        - name: myapp
          image: myapp:v2

          # 1. 就绪探针 — 确保新 Pod 就绪才接收流量
          readinessProbe:
            httpGet:
              path: /ready
              port: 8080
            initialDelaySeconds: 10
            periodSeconds: 5
            failureThreshold: 3

          # 2. 存活探针 — 检测容器健康
          livenessProbe:
            httpGet:
              path: /healthz
              port: 8080
            initialDelaySeconds: 15
            periodSeconds: 10

          # 3. 优雅终止
          lifecycle:
            preStop:
              exec:
                command: ["sh", "-c", "sleep 10"]
                # 等待 Service Endpoints 更新完成
                # 避免流量发到正在终止的 Pod
---
# 4. PDB — 保证最少可用数
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: myapp-pdb
spec:
  minAvailable: 2              # 至少 2 个可用
  selector:
    matchLabels:
      app: myapp
```

```
零停机检查清单:
  □ readinessProbe 配置正确
  □ maxUnavailable: 0
  □ preStop hook (sleep 5~15s)
  □ terminationGracePeriodSeconds 足够
  □ PDB 设置 minAvailable
  □ 应用处理 SIGTERM 优雅关闭连接
  □ 连接池/健康检查考虑
```

## 9. 数据库变更与部署配合？

**回答：**

```
挑战:
  数据库 Schema 变更不能简单回滚
  新旧代码版本可能同时运行 (滚动更新期间)
  大表 DDL 可能锁表

原则:
  1. 向后兼容 (Backward Compatible)
     新 Schema 兼容旧代码
     新代码兼容旧 Schema
  2. 拆分变更
     先修改 Schema → 部署新代码 → 清理旧 Schema
  3. 不可逆数据库变更单独管理

示例 — 重命名列:
  Step 1: 添加新列 + 双写 (新旧列都写)
  Step 2: 迁移旧数据到新列
  Step 3: 代码改为读新列
  Step 4: 停止写旧列
  Step 5: 删除旧列
```

### 数据库迁移工具

```
工具              语言/框架          特点
──────────       ──────────        ──────────
Flyway            Java              SQL 迁移, 版本化
Liquibase         Java              XML/YAML/SQL, 回滚支持
Alembic           Python/SQLAlchemy  自动生成迁移
Django Migrations Python/Django      内置
golang-migrate    Go                 多数据库支持
Knex              Node.js            JS 迁移

CI/CD 集成:
  Init Container 运行迁移 → 主容器启动应用
  或 CI Pipeline 中单独 Stage 运行迁移
```

## 10. 部署策略选型指南？

**回答：**

```
应用类型                     推荐策略
───────────────────         ────────────────────────
无状态 Web 应用              滚动更新 (默认) 或 金丝雀
微服务                      金丝雀 + Service Mesh
关键基础设施 (数据库等)       蓝绿 (快速回滚)
移动端 API                   金丝雀 (兼容性验证)
内部工具                     滚动更新 (简单足够)
面向消费者 (高流量)           金丝雀 + A/B + Feature Flags
批处理 / CronJob             重建 (Recreate)
```

```
决策树:

  能容忍停机?
  ├─ Yes → Recreate (最简单)
  └─ No
      ├─ 需要秒级回滚?
      │   ├─ Yes → Blue/Green
      │   └─ No
      │       ├─ 需要灰度验证?
      │       │   ├─ Yes → Canary
      │       │   │   ├─ 有 Service Mesh? → Istio/Linkerd 流量切分
      │       │   │   └─ 否 → Argo Rollouts
      │       │   └─ No → Rolling Update
      │       └─ 需要按用户特征分流?
      │           └─ Yes → A/B Testing + Feature Flags
      └─ 需要对比新旧版性能?
          └─ Shadow Traffic
```
