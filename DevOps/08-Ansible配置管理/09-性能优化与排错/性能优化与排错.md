# 性能优化与排错

---

## 1. Ansible 性能瓶颈在哪？

**回答：**

```
常见瓶颈:
  SSH 连接     → 每个任务建立新连接 (默认)
  Facts 收集   → 每台主机收集大量系统信息
  串行执行     → forks 默认只有 5
  模块传输     → 每个任务传输 Python 模块到远程
  DNS 解析     → 主机名解析慢
  大文件传输   → copy/synchronize 大文件

性能指标:
  ansible-playbook site.yml + profile_tasks 回调:
    显示每个任务耗时
    识别最慢的任务
```

## 2. SSH 连接优化？

**回答：**

```ini
# ansible.cfg
[ssh_connection]
# 开启 SSH 连接复用 (最重要的优化)
ssh_args = -o ControlMaster=auto -o ControlPersist=60s -o PreferDNS=no

# 启用流水线 (减少 SSH 操作次数)
pipelining = True
# 默认: 1.连接 → 2.传模块 → 3.执行 → 4.取结果
# 开启: 1.连接 → 2.传模块+执行+取结果 (通过 stdin)
# 要求: 目标主机 /etc/sudoers 中设置 Defaults !requiretty

# 使用 SSH 而非 Paramiko
transport = ssh
```

```
ControlMaster 原理:
  第一次连接: 建立 SSH 连接, 创建 socket 文件
  后续连接: 复用已有 socket, 无需重新握手
  ControlPersist=60s: 空闲 60 秒后关闭

pipelining 原理:
  关闭: SSH连接1(创建目录) → SSH连接2(传输模块) → SSH连接3(执行) → SSH连接4(清理)
  开启: SSH连接1(通过stdin传输模块并执行)
  效果: 减少 2-3 次 SSH 操作/任务, 大幅提升速度
```

## 3. Facts 收集优化？

**回答：**

```ini
# ansible.cfg
[defaults]
# 收集策略
gathering = smart        # smart/implicit/explicit
# smart:    只在缓存过期时才收集
# implicit: 每次都收集 (默认)
# explicit: 只有 gather_facts: true 时才收集

# Facts 缓存
fact_caching = jsonfile
fact_caching_connection = /tmp/ansible_facts_cache
fact_caching_timeout = 86400    # 24小时

# 其他缓存后端
# fact_caching = redis
# fact_caching_connection = localhost:6379:0
# fact_caching = memcached
# fact_caching_connection = ['localhost:11211']
```

```yaml
# Playbook 级别控制
- hosts: webservers
  gather_facts: false          # 不需要 Facts 时关闭

  tasks:
    - name: Only gather network facts
      setup:
        gather_subset:
          - network              # 只收集网络信息
          - "!hardware"          # 排除硬件信息
          - "!ohai"
          - "!facter"
      # gather_subset 选项:
      #   all, min, hardware, network, virtual, ohai, facter
      #   用 ! 排除

    - name: Get specific fact
      setup:
        filter: "ansible_default_ipv4"   # 只收集特定 Fact
```

## 4. 并行度优化 (forks)？

**回答：**

```ini
# ansible.cfg
[defaults]
forks = 50              # 并行连接数 (默认 5)

# 命令行覆盖
# ansible-playbook site.yml -f 50
```

```
forks 设置建议:
  控制节点资源    推荐 forks
  2 CPU / 4GB    10-20
  4 CPU / 8GB    30-50
  8 CPU / 16GB   50-100
  16 CPU / 32GB  100-200

注意事项:
  forks 过大 → 控制节点 CPU/内存/网络成为瓶颈
  forks > 目标主机数 → 无实际效果
  serial 限制 > forks → 实际并行 = forks

监控控制节点:
  top / htop 观察 CPU 和内存使用
  ansible-playbook + profile_tasks 观察任务耗时
```

## 5. 任务级别优化？

**回答：**

```yaml
# 1. 合并包安装 (一次 apt 安装多个包)
# 差:
- apt: name=nginx
- apt: name=python3
- apt: name=git

# 好:
- apt:
    name:
      - nginx
      - python3
      - git
    state: present

# 2. 使用 free 策略 (独立任务)
- hosts: webservers
  strategy: free          # 不等待慢主机

# 3. 异步任务
- name: Download large files in parallel
  get_url:
    url: "{{ item }}"
    dest: "/tmp/{{ item | basename }}"
  loop: "{{ download_urls }}"
  async: 300
  poll: 0
  register: downloads

- name: Wait for downloads
  async_status:
    jid: "{{ item.ansible_job_id }}"
  loop: "{{ downloads.results }}"
  register: results
  until: results.finished
  retries: 30
  delay: 10

# 4. copy vs synchronize
# copy = Python 实现, 逐文件, 较慢
# synchronize = rsync 封装, 增量同步, 快

# 大目录用 synchronize:
- synchronize:
    src: /opt/app/static/
    dest: /var/www/static/
    delete: yes
    rsync_opts:
      - "--exclude=*.log"

# 5. 减少不必要的 changed 检查
- command: get_status.sh
  changed_when: false      # 不触发 handler, 减少开销
  check_mode: false         # check 模式也执行

# 6. 条件跳过
- name: Skip if already deployed
  stat:
    path: /opt/app/releases/{{ version }}
  register: release_dir

- name: Deploy
  unarchive:
    src: "app-{{ version }}.tar.gz"
    dest: /opt/app/releases/
  when: not release_dir.stat.exists
```

## 6. Mitogen 加速？

**回答：**

```
Mitogen = Ansible 第三方加速插件
原理: 替换默认 SSH 连接, 使用持久化 Python 连接
效果: 1.25x - 7x 加速 (取决于场景)

安装:
  pip install mitogen

配置:
  # ansible.cfg
  [defaults]
  strategy_plugins = /path/to/mitogen/ansible_mitogen/plugins/strategy
  strategy = mitogen_linear

优化原理:
  1. 持久化连接: 不重复建立 SSH
  2. 进程复用: 远程 Python 进程不销毁
  3. 智能传输: 只传变化的代码
  4. 减少 fork: 更高效的进程管理

限制:
  不支持所有模块 (raw, script 等)
  不支持所有连接类型
  可能与某些插件冲突
  社区维护, 更新不及时

适用场景: 大量主机 + 大量任务
```

## 7. 调试方法？

**回答：**

```bash
# 详细输出 (verbosity)
ansible-playbook site.yml -v       # 显示结果
ansible-playbook site.yml -vv      # 显示输入参数
ansible-playbook site.yml -vvv     # 显示 SSH 连接信息
ansible-playbook site.yml -vvvv    # 显示连接插件详情

# Syntax Check
ansible-playbook site.yml --syntax-check

# Check Mode (Dry Run)
ansible-playbook site.yml --check --diff

# 逐步执行
ansible-playbook site.yml --step
# (N)o/(y)es/(c)ontinue

# 从指定任务开始
ansible-playbook site.yml --start-at-task "Deploy config"

# 列出任务/主机/标签
ansible-playbook site.yml --list-tasks
ansible-playbook site.yml --list-hosts
ansible-playbook site.yml --list-tags
```

```yaml
# debug 策略 — 交互式调试器
- hosts: webservers
  strategy: debug
  tasks:
    - name: Debug this
      command: /opt/fail.sh

# 进入调试器后:
#   p task        → 打印当前任务
#   p result      → 打印结果
#   p vars        → 打印变量
#   p host        → 打印主机
#   r             → 重执行当前任务
#   c             → 继续到下一个任务
#   q             → 退出

# debugger 关键字 (Ansible 2.5+)
- name: Suspect task
  command: /opt/check.sh
  debugger: on_failed     # always / never / on_failed / on_unreachable / on_skipped
```

## 9. 常见错误排查？

**回答：**

```
错误: "Unreachable" — 无法连接
排查:
  1. ssh user@host 手动测试
  2. 检查 ansible_host / ansible_port
  3. 检查 SSH 密钥 / 密码
  4. 检查防火墙规则
  5. ansible all -m ping -vvv 查看详情

错误: "Permission denied" — 权限不足
排查:
  1. 检查 ansible_user 权限
  2. 检查 become / become_user
  3. 检查 sudoers 配置
  4. 检查文件权限 (SSH key 600, .ssh 700)

错误: "Module not found" — 模块找不到
排查:
  1. ansible-doc <module_name> 检查是否存在
  2. 检查 Collection 是否安装
  3. 使用 FQCN: ansible.builtin.copy
  4. pip install 相关 Python 依赖

错误: "Variable undefined" — 变量未定义
排查:
  1. ansible-inventory --host <hostname> 检查变量
  2. 检查 group_vars / host_vars 文件名
  3. 检查 YAML 格式 (缩进)
  4. 使用 default() 过滤器

错误: "Jinja2 template error" — 模板错误
排查:
  1. 检查 {{ }} 配对
  2. 字符串比较加引号: when: var == "value"
  3. 布尔值判断: when: var | bool
  4. 嵌套变量: "{{ lookup('env', 'HOME') }}"

错误: "YAML syntax error" — YAML 格式
排查:
  1. yamllint 检查
  2. 检查缩进 (空格, 不能用 Tab)
  3. 特殊字符需引号: name: "Task: do stuff"
  4. 检查 : 后有空格
```

## 10. 性能优化检查清单？

**回答：**

```
配置优化 (ansible.cfg):
  □ forks = 50 (根据控制节点资源调整)
  □ pipelining = True
  □ ssh_args = -o ControlMaster=auto -o ControlPersist=60s
  □ gathering = smart
  □ fact_caching = jsonfile (或 redis)
  □ fact_caching_timeout = 86400
  □ host_key_checking = False (测试环境)
  □ stdout_callback = yaml
  □ callback_whitelist = timer, profile_tasks

Playbook 优化:
  □ gather_facts: false (不需要时关闭)
  □ setup filter: 只收集需要的 Facts
  □ 合并包安装 (一次 apt 多个包)
  □ 使用 synchronize 替代 copy (大文件)
  □ 使用 async/poll 并行长任务
  □ changed_when: false (只读任务)
  □ 条件跳过已完成的任务

架构优化:
  □ 使用 AWX/Tower 分布式执行
  □ 考虑 Pull 模式 (ansible-pull)
  □ 使用 Mitogen 加速
  □ 拆分大 Playbook (import_playbook)
  □ 合理设计 serial 批次

监控:
  □ profile_tasks 找出慢任务
  □ -vvv 分析 SSH 连接时间
  □ 控制节点 CPU/内存/网络监控
  □ 执行时间趋势分析

典型优化效果:
  pipelining           → 提升 20-40%
  ControlPersist       → 提升 10-20%
  fact_caching         → 提升 10-30%
  forks 5→50          → 提升 5-10x (多主机)
  gather_facts: false  → 节省 2-5s/主机
  Mitogen              → 提升 2-7x
```
