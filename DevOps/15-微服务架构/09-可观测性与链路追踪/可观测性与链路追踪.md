## 1. 微服务可观测性三大支柱？

**回答：**

```
可观测性 (Observability):
  从系统外部输出推断系统内部状态的能力

三大支柱:
  ┌──────────────────────────────────────────────────┐
  │                  可观测性                          │
  │                                                   │
  │  ┌──────────┐  ┌──────────┐  ┌──────────────┐    │
  │  │  Metrics  │  │  Logging │  │   Tracing    │    │
  │  │  指标     │  │  日志     │  │   链路追踪    │    │
  │  │          │  │          │  │              │    │
  │  │ 发生了    │  │ 为什么    │  │ 在哪里       │    │
  │  │ 什么?    │  │ 发生?    │  │ 发生?        │    │
  │  │          │  │          │  │              │    │
  │  │Prometheus│  │ ELK/Loki │  │Jaeger/Zipkin │    │
  │  │ Grafana  │  │          │  │OpenTelemetry │    │
  │  └──────────┘  └──────────┘  └──────────────┘    │
  └──────────────────────────────────────────────────┘

各支柱特点:
  ┌──────────────┬──────────────┬──────────────┬──────────────┐
  │              │ Metrics      │ Logging      │ Tracing      │
  ├──────────────┼──────────────┼──────────────┼──────────────┤
  │ 数据类型      │ 数值 (聚合)  │ 文本 (离散)  │ 树形调用链    │
  │ 存储成本      │ 低           │ 高           │ 中           │
  │ 查询方式      │ PromQL       │ 全文搜索     │ TraceID      │
  │ 时效性        │ 近实时       │ 近实时       │ 近实时       │
  │ 告警          │ 最适合       │ 可以         │ 不适合       │
  │ 排障深度      │ 发现问题     │ 定位原因     │ 定位链路     │
  │ 适用场景      │ 监控/告警    │ 错误诊断     │ 跨服务排障   │
  └──────────────┴──────────────┴──────────────┴──────────────┘

补充: 第四支柱 — Profiling (持续性能分析)
  Pyroscope / Parca: CPU/内存火焰图
  → 从代码级别分析性能瓶颈
```

## 2. 分布式链路追踪原理？

**回答：**

```
核心概念:
  Trace:   一次完整请求的调用链 (包含多个 Span)
  Span:    一个操作/调用 (有开始/结束时间)
  SpanContext: Span 的上下文 (TraceID, SpanID, 采样标记)
  
请求追踪过程:
  Client → API GW → 用户服务 → 订单服务 → 支付服务
  
  TraceID: abc-123 (全链路唯一)
  ┌────────────────────────────────────────────┐
  │ Span 1: API Gateway (0ms - 150ms)          │
  │  ├── Span 2: 用户服务 (5ms - 25ms)         │
  │  └── Span 3: 订单服务 (30ms - 140ms)       │
  │       └── Span 4: 支付服务 (35ms - 120ms)  │
  └────────────────────────────────────────────┘

上下文传播 (Context Propagation):
  HTTP Header 传递 TraceID + SpanID
  
  服务A → 服务B:
    Header:
      traceparent: 00-abc123-span001-01
      (版本-TraceID-ParentSpanID-采样标记)

  W3C Trace Context (标准):
    traceparent: 00-{trace-id}-{parent-span-id}-{trace-flags}
  
  B3 格式 (Zipkin):
    X-B3-TraceId: abc123
    X-B3-SpanId: span002
    X-B3-ParentSpanId: span001
    X-B3-Sampled: 1

采样策略:
  ┌──────────────┬──────────────────────────────┐
  │ 策略          │ 说明                          │
  ├──────────────┼──────────────────────────────┤
  │ 全量采样      │ 100% 采集, 数据量大            │
  │ 概率采样      │ 按比例采集 (如 1%)             │
  │ 速率采样      │ 每秒最多 N 条                  │
  │ 尾部采样      │ 先缓存, 出错/慢的请求才采集    │
  │ 自适应采样    │ 根据流量动态调整采样率          │
  └──────────────┴──────────────────────────────┘
  
  生产建议: 尾部采样 (Tail-based Sampling)
  → 正常请求低采样, 异常/慢请求 100% 采集
```

## 3. OpenTelemetry 标准？

**回答：**

```
OpenTelemetry (OTel):
  CNCF 可观测性统一标准
  合并了 OpenTracing + OpenCensus
  提供 Metrics + Logs + Traces 统一 SDK

架构:
  ┌──────────────────────────────────────────┐
  │              应用服务                      │
  │  ┌────────────────────────────────────┐  │
  │  │       OTel SDK (自动/手动埋点)      │  │
  │  │  Traces + Metrics + Logs           │  │
  │  └──────────────┬─────────────────────┘  │
  └─────────────────┼────────────────────────┘
                    │ OTLP 协议
                    ↓
  ┌──────────────────────────────────────────┐
  │         OTel Collector                    │
  │  Receivers → Processors → Exporters      │
  │                                           │
  │  接收数据     处理/过滤     导出到后端       │
  └──────────┬──────────┬──────────┬─────────┘
             ↓          ↓          ↓
         Jaeger    Prometheus   Loki
         (Traces)  (Metrics)   (Logs)

核心优势:
  厂商中立: 一套 SDK, 任意切换后端
  语言覆盖: Java/Go/Python/JS/Rust/C++ 等
  自动埋点: 自动检测常见框架 (HTTP/gRPC/DB)
  统一协议: OTLP (OpenTelemetry Protocol)
```

```yaml
# OTel Collector 配置
apiVersion: v1
kind: ConfigMap
metadata:
  name: otel-collector-config
data:
  config.yaml: |
    receivers:
      otlp:
        protocols:
          grpc:
            endpoint: 0.0.0.0:4317
          http:
            endpoint: 0.0.0.0:4318

    processors:
      batch:
        timeout: 5s
        send_batch_size: 1024
      memory_limiter:
        check_interval: 1s
        limit_mib: 512
      # 尾部采样
      tail_sampling:
        decision_wait: 10s
        policies:
          - name: error-policy
            type: status_code
            status_code: {status_codes: [ERROR]}
          - name: latency-policy
            type: latency
            latency: {threshold_ms: 1000}
          - name: probabilistic
            type: probabilistic
            probabilistic: {sampling_percentage: 5}

    exporters:
      otlp/jaeger:
        endpoint: jaeger-collector:4317
        tls:
          insecure: true
      prometheus:
        endpoint: 0.0.0.0:8889
      loki:
        endpoint: http://loki:3100/loki/api/v1/push

    service:
      pipelines:
        traces:
          receivers: [otlp]
          processors: [memory_limiter, batch]
          exporters: [otlp/jaeger]
        metrics:
          receivers: [otlp]
          processors: [memory_limiter, batch]
          exporters: [prometheus]
        logs:
          receivers: [otlp]
          processors: [memory_limiter, batch]
          exporters: [loki]
```

```
架构:
  ┌──────────┐     ┌──────────────┐     ┌──────────┐
  │ App+SDK  │────→│  Collector   │────→│  Storage  │

                                               │
                   ┌──────────────┐            │

                   │  (查询界面)   │
                   └──────────────┘

组件说明:
  ┌──────────────┬──────────────────────────────┐
  │ 组件          │ 说明                          │
  ├──────────────┼──────────────────────────────┤
  │ Client SDK   │ 应用内埋点 (OTel SDK)         │
  │ Agent        │ 本地代理 (已弃用, 直连)       │
  │ Collector    │ 接收/处理/存储 Span           │
  │ Storage      │ 后端存储 (ES/Cassandra/Kafka) │
```

## 5. 微服务日志聚合方案？

**回答：**

```
微服务日志挑战:
  服务分散 → 日志分散在多个容器/节点
  需要统一收集、聚合、搜索

主流方案对比:
  ┌──────────────┬──────────────────┬──────────────────┐
  │              │ ELK/EFK          │ Loki + Grafana   │
  ├──────────────┼──────────────────┼──────────────────┤
  │ 组成          │ ES + Logstash/   │ Loki + Promtail  │
  │              │ Fluentd + Kibana │ + Grafana         │
  │ 索引方式      │ 全文索引          │ 只索引标签        │
  │ 存储成本      │ 高 (全文索引)     │ 低 (压缩存储)    │
  │ 查询能力      │ 强 (全文搜索)     │ 中 (标签过滤)    │
  │ 资源占用      │ 高 (ES 吃内存)   │ 低               │
  │ 适用场景      │ 大规模/复杂查询   │ K8s/中小规模     │
  │ 学习成本      │ 高               │ 低 (类 PromQL)  │
  └──────────────┴──────────────────┴──────────────────┘

K8s 日志架构:
  ┌───────────────────────────────────────────┐
  │ Node                                       │
  │  Pod A → stdout/stderr → /var/log/pods/   │
  │  Pod B → stdout/stderr → /var/log/pods/   │
  │                                            │
  │  ┌──────────────┐                          │
  │  │ DaemonSet    │  采集日志                 │
  │  │ (Promtail/   │  ──→ Loki / ES          │
  │  │  Fluentd)    │                          │
  │  └──────────────┘                          │
  └───────────────────────────────────────────┘

日志规范:
  1. 结构化日志 (JSON 格式)
  2. 统一字段: timestamp, service, traceId, level, message
  3. 日志级别合理使用: ERROR/WARN/INFO/DEBUG
  4. 关联 TraceID → 日志与链路追踪关联
```

## 6. 微服务指标监控体系？

**回答：**

```
微服务关键指标 (RED + USE):

RED 方法 (面向服务):
  R (Rate):     请求速率 (QPS)
  E (Errors):   错误率
  D (Duration): 延迟分布 (P50/P95/P99)

USE 方法 (面向资源):
  U (Utilization): 资源使用率 (CPU/Memory)
  S (Saturation):  饱和度 (队列长度)
  E (Errors):      错误数

四大黄金指标 (Google SRE):
  ┌──────────────┬──────────────────────────────┐
  │ 指标          │ 说明                          │
  ├──────────────┼──────────────────────────────┤
  │ 延迟 Latency │ 请求处理时间 (P50/P95/P99)    │
  │ 流量 Traffic │ 请求速率 (req/s)              │
  │ 错误 Errors  │ 请求错误率 (5xx 占比)         │
  │ 饱和度 Satur │ 资源饱和程度 (CPU/内存/连接)  │
  └──────────────┴──────────────────────────────┘

监控层次:
  ┌─────────────────────────────────────┐
  │ 业务指标   订单量/转化率/收入         │
  ├─────────────────────────────────────┤
  │ 应用指标   QPS/延迟/错误率/JVM       │
  ├─────────────────────────────────────┤
  │ 中间件     DB 连接/Redis 命中率/MQ堆积│
  ├─────────────────────────────────────┤
  │ 基础设施   CPU/内存/磁盘/网络         │
  └─────────────────────────────────────┘
```

```yaml
# PrometheusRule 告警规则
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: microservice-alerts
spec:
  groups:
  - name: microservice.rules
    rules:
    - alert: HighErrorRate
      expr: |
        sum(rate(http_requests_total{status=~"5.."}[5m])) by (service)
        / sum(rate(http_requests_total[5m])) by (service) > 0.05
      for: 5m
      labels:
        severity: critical
      annotations:
        summary: "{{ $labels.service }} 错误率超过 5%"

    - alert: HighLatency
      expr: |
        histogram_quantile(0.99,
          sum(rate(http_request_duration_seconds_bucket[5m])) by (le, service)
        ) > 2
      for: 5m
      labels:
        severity: warning
      annotations:
        summary: "{{ $labels.service }} P99 延迟超过 2 秒"
```

## 7. SLI/SLO/SLA 与告警设计？

**回答：**

```
  ┌──────────────┬──────────────────────────────┐
  │ 概念          │ 说明                          │
  ├──────────────┼──────────────────────────────┤
  │ SLI          │ 服务级别指标 (Service Level   │
  │              │ Indicator) → 可量化的指标     │
  │              │ 如: 请求成功率, P99 延迟       │
  │              │                              │
  │ SLO          │ 服务级别目标 (Service Level   │
  │              │ Objective) → SLI 的目标值     │
  │              │ 如: 成功率 ≥ 99.9%            │
  │              │     P99 延迟 ≤ 500ms          │
  │              │                              │
  │ SLA          │ 服务级别协议 (Service Level   │
  │              │ Agreement) → 对外承诺 + 违约后果│
  │              │ 如: 可用性 99.95%, 否则赔偿   │
  └──────────────┴──────────────────────────────┘

SLO 示例:
  ┌──────────────┬──────────────┬──────────────┐
  │ 服务          │ SLI          │ SLO          │
  ├──────────────┼──────────────┼──────────────┤
  │ API 网关      │ 成功率        │ ≥ 99.95%     │
  │ 订单服务      │ P99 延迟      │ ≤ 500ms      │
  │ 支付服务      │ 成功率        │ ≥ 99.99%     │
  │ 搜索服务      │ P95 延迟      │ ≤ 200ms      │
  └──────────────┴──────────────┴──────────────┘

错误预算 (Error Budget):
  SLO = 99.9% → 每月允许 43 分钟不可用
  SLO = 99.95% → 每月允许 22 分钟不可用
  
  错误预算用完 → 冻结新功能, 专注稳定性

告警设计原则:
  1. 基于 SLO 告警 (非绝对阈值)
  2. 燃烧率告警 (Burn Rate)
     错误预算消耗速度 > 预期 → 告警
  3. 分级: Critical (立即处理) / Warning (关注)
  4. 可操作: 每条告警都有对应的 Runbook
  5. 避免告警疲劳: 宁缺毋滥
```

```yaml
# 基于 SLO 的燃烧率告警
# 错误预算消耗速度告警 (比绝对阈值更智能)
groups:
- name: slo-burn-rate
  rules:
  # 快速燃烧 (1h 窗口, 14.4x 燃烧率) → 紧急
  - alert: SLOBurnRateCritical
    expr: |
      (
        1 - sum(rate(http_requests_total{status!~"5.."}[1h])) by (service)
            / sum(rate(http_requests_total[1h])) by (service)
      ) > (14.4 * 0.001)
    for: 2m
    labels:
      severity: critical

  # 慢速燃烧 (6h 窗口, 6x 燃烧率) → 警告
  - alert: SLOBurnRateWarning
    expr: |
      (
        1 - sum(rate(http_requests_total{status!~"5.."}[6h])) by (service)
            / sum(rate(http_requests_total[6h])) by (service)
      ) > (6 * 0.001)
    for: 15m
    labels:
      severity: warning
```

## 8. 故障排查与根因分析？

**回答：**

```
微服务故障排查流程:
  1. 发现问题 → 告警/用户反馈
  2. 确定范围 → 影响了哪些服务
  3. 定位链路 → 哪个环节出了问题
  4. 根因分析 → 为什么出问题
  5. 修复恢复 → 解决并验证
  6. 复盘改进 → 避免再次发生

排查工具链:
  ┌──────────────────────────────────────────────────┐
  │ 步骤                  工具                        │
  ├──────────────────────────────────────────────────┤
  │ 1. 看告警/大盘       Grafana Dashboard           │
  │ 2. 看指标           Prometheus (QPS/延迟/错误率)  │
  │ 3. 看链路           Jaeger (哪个服务慢/报错)      │
  │ 4. 看日志           Loki/ELK (具体错误信息)       │
  │ 5. 看依赖           Kiali (服务拓扑/流量)         │
  │ 6. 看资源           K8s (Pod状态/资源使用)        │
  │ 7. 看代码           Profiling (CPU/内存火焰图)    │
  └──────────────────────────────────────────────────┘

常见故障模式:
  ┌──────────────────┬──────────────────────────────┐
  │ 故障              │ 排查思路                      │
  ├──────────────────┼──────────────────────────────┤
  │ 服务超时          │ 链路追踪 → 找到慢 Span        │
  │ 5xx 激增          │ 错误日志 → 下游依赖/DB/OOM    │
  │ 延迟突增          │ CPU/GC/连接池/下游慢           │
  │ 间歇性失败        │ 资源竞争/DNS 缓存/连接池耗尽   │
  │ 级联故障          │ 熔断缺失/无超时/资源未隔离     │
  │ 数据不一致        │ 异步消息丢失/重复/顺序问题     │
  └──────────────────┴──────────────────────────────┘
```

```bash
# K8s + Istio 故障排查命令

# 1. 查看 Pod 状态
kubectl get pods -n production --sort-by='.status.containerStatuses[0].restartCount'
kubectl top pods -n production --sort-by=memory

# 2. 查看异常事件
kubectl get events -n production --sort-by='.lastTimestamp' | tail -20

# 3. 查看服务日志
kubectl logs deploy/order-service -n production --tail=100 -f
```

## 10. 可观测性面试速答？

**回答：**

```
Q: 可观测性三大支柱?
A: Metrics (指标/告警), Logs (日志/诊断), Traces (链路/定位)
   Prometheus + Loki/ELK + Jaeger/Zipkin

Q: 分布式链路追踪原理?
A: TraceID 全链路传播 (HTTP Header)
   每个服务生成 Span (开始/结束/父子关系)
   Collector 收集 Span → 存储 → UI 展示调用链

Q: OpenTelemetry 是什么?
A: CNCF 可观测性统一标准
   合并 OpenTracing + OpenCensus
   一套 SDK 覆盖 Metrics/Logs/Traces
   厂商中立, 可切换任意后端

Q: 采样策略怎么选?
A: 尾部采样 (Tail Sampling) 最佳
   正常请求低比例, 异常/慢请求 100% 采集
   平衡数据量和排障需求

Q: ELK vs Loki?
A: ELK: 全文索引, 查询强大, 资源消耗大
   Loki: 只索引标签, 类 PromQL 查询, 轻量
   K8s + 中小规模用 Loki, 大规模复杂查询用 ELK

Q: 微服务监控哪些指标?
A: RED: Rate(QPS) + Errors(错误率) + Duration(延迟)
   四大黄金指标: 延迟/流量/错误/饱和度
   按层: 业务→应用→中间件→基础设施

Q: SLI/SLO/SLA 区别?
A: SLI: 可量化指标 (成功率/延迟)
   SLO: 内部目标 (成功率 ≥ 99.9%)
   SLA: 对外承诺 + 违约赔偿

Q: 怎么做告警?
A: 基于 SLO 燃烧率告警 (非绝对阈值)
   分级 Critical/Warning, 每条有 Runbook
   避免告警疲劳, 宁缺毋滥

Q: 微服务故障排查流程?
A: 告警 → Grafana 大盘 → Prometheus 指标
   → Jaeger 链路 → Loki 日志 → 定位根因
   关键: 三大支柱通过 TraceID 关联
```
