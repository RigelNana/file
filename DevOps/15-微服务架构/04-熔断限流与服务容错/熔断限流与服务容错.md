# 熔断限流与服务容错

---

## 1. 什么是熔断器模式？

**回答：**

```
熔断器 (Circuit Breaker): 防止级联故障的保护机制
类比: 家用电路的保险丝, 异常时自动断开

三种状态:
  ┌──────┐    失败率>阈值    ┌──────┐   超时后探测    ┌───────────┐
  │Closed │──────────────→ │ Open  │──────────────→│ Half-Open │
  │(正常)  │               │(断路)  │               │ (半开)     │
  └──────┘  ←──────────────└──────┘  ←─────────────└───────────┘
              探测成功恢复              探测失败回到 Open

工作流程:
  1. Closed (关闭): 正常请求通过
     ├── 统计失败率 (滑动窗口)
     └── 失败率 > 阈值 → 转为 Open

  2. Open (断开): 直接拒绝请求, 快速失败
     ├── 返回 fallback 响应
     └── 等待超时时间 → 转为 Half-Open

  3. Half-Open (半开): 放行少量探测请求
     ├── 探测成功 → 回到 Closed
     └── 探测失败 → 回到 Open

参数配置:
  ┌──────────────────┬───────────────────────────┐
  │ 参数              │ 说明                       │
  ├──────────────────┼───────────────────────────┤
  │ 失败率阈值        │ 触发熔断的失败率 (如 50%)   │
  │ 慢调用阈值        │ 响应时间超过 N ms 算慢调用  │
  │ 滑动窗口          │ 统计窗口大小 (如 10 次请求) │
  │ 最小请求数        │ 触发计算的最小请求量        │
  │ 等待时间          │ Open → Half-Open 的等待时间 │
  │ 半开放行数        │ Half-Open 放行的探测请求数  │
  └──────────────────┴───────────────────────────┘
```

## 2. 常见限流算法？

**回答：**

```
1. 固定窗口计数器 (Fixed Window)
   ┌────────┬────────┬────────┐
   │ 0-1秒   │ 1-2秒   │ 2-3秒  │
   │ 80/100  │ 90/100  │ 70/100 │
   └────────┴────────┴────────┘
   问题: 窗口边界可能出现 2× 流量突刺

2. 滑动窗口计数器 (Sliding Window)
   ┌─────────────────────────────┐
   │ [1.0, 1.1, 1.2 ... 2.0]    │ → 滑动统计
   └─────────────────────────────┘
   解决固定窗口边界问题, Redis 有序集合实现

3. 漏桶算法 (Leaky Bucket)
   请求 → [桶 (固定容量)] → 恒定速率流出
   特点: 输出平滑, 不允许突发
   问题: 突发流量全被排队, 延迟增大

4. 令牌桶算法 (Token Bucket) ★推荐
   [令牌生成器] → 恒定速率放入 → [令牌桶 (容量N)]
   请求到来 → 取令牌 → 有令牌则通过, 无则拒绝
   特点: 允许一定突发 (桶内累积令牌)
   
   实现: Guava RateLimiter, Redis + Lua

常用 Redis + Lua 令牌桶:
  local tokens = redis.call('get', KEYS[1])
  local last_time = redis.call('get', KEYS[2])
  local now = tonumber(ARGV[1])
  local rate = tonumber(ARGV[2])     -- 每秒生成速率
  local capacity = tonumber(ARGV[3]) -- 桶容量

  -- 计算新增令牌
  local elapsed = now - tonumber(last_time or now)
  local new_tokens = math.min(capacity, 
    tonumber(tokens or capacity) + elapsed * rate)

  if new_tokens >= 1 then
    redis.call('set', KEYS[1], new_tokens - 1)
    redis.call('set', KEYS[2], now)
    return 1  -- 允许
  else
    return 0  -- 拒绝
  end

限流算法对比:
  ┌──────────────┬─────────┬──────────┬─────────┐
  │ 算法          │ 突发流量 │ 平滑输出  │ 实现难度 │
  ├──────────────┼─────────┼──────────┼─────────┤
  │ 固定窗口      │ 有边界问题│ 否       │ 简单    │
  │ 滑动窗口      │ 较好     │ 否       │ 中等    │
  │ 漏桶          │ 不允许   │ 是       │ 中等    │
  │ 令牌桶        │ 允许     │ 否       │ 中等    │
  └──────────────┴─────────┴──────────┴─────────┘
```

## 3. 限流的实现层次？

**回答：**

```
限流层次 (由外到内):
  
  1. 网关层限流 (全局)
     ├── Nginx: limit_req_zone (漏桶)
     ├── Kong: rate-limiting 插件
     ├── APISIX: limit-req / limit-count
     └── AWS API Gateway: 请求配额

  2. 服务层限流 (单服务)
     ├── Sentinel: 流控规则 (QPS / 并发数)
     ├── Resilience4j: RateLimiter
     └── 自定义: Guava RateLimiter

  3. 分布式限流 (全局精确)
     ├── Redis + Lua (令牌桶 / 滑动窗口)
     └── Sentinel 集群模式

Nginx 限流配置:
  http {
    # 定义限流区域: 10MB 共享内存, 每 IP 10 请求/秒
    limit_req_zone $binary_remote_addr 
      zone=api_limit:10m rate=10r/s;
    
    server {
      location /api/ {
        # burst=20: 最多排队 20 个
        # nodelay: 不排队, 超出直接拒绝
        limit_req zone=api_limit burst=20 nodelay;
        limit_req_status 429;
      }
    }
  }

分布式限流 (Redis 滑动窗口):
  -- 滑动窗口限流 Lua 脚本
  local key = KEYS[1]
  local window = tonumber(ARGV[1])  -- 窗口大小 (秒)
  local limit = tonumber(ARGV[2])   -- 窗口内最大请求数
  local now = tonumber(ARGV[3])     -- 当前时间戳 (ms)

  -- 清除窗口外的记录
  redis.call('ZREMRANGEBYSCORE', key, 0, now - window * 1000)
  -- 统计窗口内请求数
  local count = redis.call('ZCARD', key)

  if count < limit then
    redis.call('ZADD', key, now, now .. math.random())
    redis.call('EXPIRE', key, window)
    return 1  -- 允许
  else
    return 0  -- 拒绝
  end
```

## 4. 服务降级策略？

**回答：**

```
降级: 系统压力大时, 牺牲非核心功能保核心功能

降级策略:
  ┌──────────────────┬─────────────────────────────┐
  │ 策略              │ 说明                         │
  ├──────────────────┼─────────────────────────────┤
  │ 返回默认值/缓存   │ 推荐服务降级返回热门列表      │
  │ 功能降级          │ 关闭评论/推荐/排行等非核心功能 │
  │ 读降级            │ 只读缓存, 不查数据库          │
  │ 写降级            │ 异步写入, 先返回成功          │
  │ 体验降级          │ 降低图片质量/关闭动画         │
  │ 限流降级          │ 超出容量的请求直接拒绝        │
  └──────────────────┴─────────────────────────────┘

实践场景:
  电商大促:
    ├── 关闭推荐算法 → 返回静态热门商品
    ├── 关闭评论功能 → 显示"暂不可用"
    ├── 库存查询降级 → 读缓存 (允许少量超卖)
    └── 物流查询降级 → 返回"查询中,稍后更新"

  降级开关:
    配置中心 (Nacos / Apollo) 动态切换
    不需要重启服务, 实时生效

Fallback 实现:
  // Resilience4j 降级示例
  @CircuitBreaker(name = "productService", 
                  fallbackMethod = "getProductFallback")
  public Product getProduct(Long id) {
      return productClient.getProduct(id);
  }

  // 降级方法: 返回缓存或默认值
  public Product getProductFallback(Long id, Exception e) {
      return cacheService.getCachedProduct(id)
          .orElse(Product.defaultProduct());
  }
```

## 5. 超时与重试策略？

**回答：**

```
超时设置原则:
  1. 每个调用都必须设超时 (绝不无限等待)
  2. 下游超时 < 上游超时 (避免超时传播)
  3. 总超时 = 上游超时 (链路总时间有限)

  Client → Gateway(5s) → ServiceA(3s) → ServiceB(1s)
          超时递减, 越下游越短

超时配置:
  连接超时 (Connect Timeout): 建立连接, 通常 1-3s
  读超时   (Read Timeout):    等待响应, 根据业务设定
  写超时   (Write Timeout):   发送请求, 通常 3-5s

重试策略:
  ┌──────────────────┬──────────────────────────────┐
  │ 策略              │ 说明                          │
  ├──────────────────┼──────────────────────────────┤
  │ 固定间隔          │ 每次间隔固定 (如 1s)           │
  │ 指数退避          │ 2^n × base (1s, 2s, 4s, 8s)  │
  │ 指数退避+抖动     │ 2^n + random (避免雷群效应) ★  │
  │ 最大重试次数      │ 通常 2-3 次                    │
  └──────────────────┴──────────────────────────────┘

重试注意事项:
  1. 只重试幂等操作 (GET/PUT/DELETE)
     ❌ POST 创建订单不应盲目重试
  2. 只重试可恢复的错误
     ✅ 503 Service Unavailable → 重试
     ✅ 网络超时 → 重试
     ❌ 400 Bad Request → 不重试
     ❌ 401 Unauthorized → 不重试
  3. 重试风暴: 全部客户端同时重试 → 雪崩
     解决: 指数退避 + 随机抖动 + 熔断器
  4. 重试预算: 全链路重试次数有限
     上游已重试 → 下游不再重试

Envoy 重试配置:
  retryPolicy:
    retryOn: 5xx,connect-failure,retriable-4xx
    numRetries: 3
    perTryTimeout: 2s
    retryBackOff:
      baseInterval: 500ms
      maxInterval: 10s
```

## 6. 隔离策略 (Bulkhead)？

**回答：**

```
舱壁模式 (Bulkhead): 资源隔离, 防止故障扩散
类比: 船的水密舱, 一个舱进水不影响其他舱

隔离维度:
  ┌──────────────┬──────────────────────────────┐
  │ 维度          │ 说明                          │
  ├──────────────┼──────────────────────────────┤
  │ 线程池隔离    │ 不同服务调用使用不同线程池      │
  │ 信号量隔离    │ 限制并发调用数 (轻量)          │
  │ 进程隔离      │ 不同功能部署为独立服务          │
  │ 容器隔离      │ K8s 资源限制 (CPU/Memory)     │
  │ 集群隔离      │ 不同业务使用不同集群            │
  └──────────────┴──────────────────────────────┘

线程池隔离:
  ┌─────────────────────────────┐
  │ 服务 A                      │
  │  ┌─────────┐ ┌─────────┐   │
  │  │线程池-订单│ │线程池-支付│  │
  │  │ 10个线程  │ │ 5个线程   │  │
  │  └────┬────┘ └────┬────┘   │
  │       ↓            ↓        │
  │   订单服务      支付服务     │
  └─────────────────────────────┘
  支付服务故障 → 只影响支付线程池
  订单服务不受影响!

信号量隔离 vs 线程池隔离:
  ┌──────────┬────────────┬────────────┐
  │          │ 信号量      │ 线程池      │
  ├──────────┼────────────┼────────────┤
  │ 资源消耗  │ 低          │ 高          │
  │ 超时控制  │ 不支持      │ 支持        │
  │ 异步      │ 不支持      │ 支持        │
  │ 适用      │ 高性能本地  │ 第三方调用  │
  └──────────┴────────────┴────────────┘

K8s 容器级隔离:
  resources:
    requests:
      cpu: "500m"
      memory: "256Mi"
    limits:
      cpu: "1000m"       # CPU 上限
      memory: "512Mi"    # OOM Kill 阈值
```

## 7. Sentinel 与 Resilience4j 对比？

**回答：**

```
主流服务容错框架:
  ┌──────────────┬──────────────┬──────────────┐
  │              │ Sentinel     │ Resilience4j │
  ├──────────────┼──────────────┼──────────────┤
  │ 来源          │ 阿里巴巴     │ 开源社区      │
  │ 语言          │ Java         │ Java         │
  │ 功能          │ 限流+熔断+降级│ 熔断+限流+重试│
  │ 限流算法      │ 滑动窗口+令牌桶│ 令牌桶       │
  │ 控制台        │ 有 (Dashboard)│ 无 (集成监控) │
  │ 规则持久化    │ 支持 (Nacos等)│ 代码/配置    │
  │ 动态规则      │ 实时修改      │ 需重启       │
  │ 集群限流      │ 支持          │ 不支持       │
  │ 适合场景      │ 国内/Spring Cloud Alibaba │ 轻量级/国际 │
  └──────────────┴──────────────┴──────────────┘

Sentinel 核心概念:
  资源:  要保护的代码/接口
  规则:  流控规则, 熔断规则, 热点规则
  效果:  直接拒绝 / 排队等待 / 慢启动

Resilience4j 核心模块:
  CircuitBreaker: 熔断器
  RateLimiter:    限流器
  Retry:          重试
  Bulkhead:       隔离
  TimeLimiter:    超时控制

选型建议:
  Spring Cloud Alibaba 生态 → Sentinel
  Spring Boot 轻量项目       → Resilience4j
  非 Java 项目              → Envoy/Istio (基础设施层)
```

## 8. 雪崩效应与防护？

**回答：**

```
雪崩效应 (Cascading Failure):
  一个服务故障 → 依赖方阻塞 → 连锁反应 → 整个系统崩溃

  User → A → B → C (故障)
              ↓
         B 线程池耗尽 (等待 C 响应)
              ↓
         A 线程池耗尽 (等待 B 响应)
              ↓
         全部服务不可用 → 雪崩!

防护体系 (多层防御):
  ┌─────────────────────────────────────────────┐
  │ 层次          措施            作用            │
  ├─────────────────────────────────────────────┤
  │ L1: 限流      令牌桶/滑动窗口  控制入口流量    │
  │ L2: 超时      合理超时设置     避免无限等待    │
  │ L3: 熔断      Circuit Breaker  快速失败       │
  │ L4: 隔离      线程池/信号量    故障不扩散      │
  │ L5: 降级      Fallback        保证核心可用    │
  │ L6: 重试      指数退避+抖动    恢复暂时故障    │
  └─────────────────────────────────────────────┘

最佳实践:
  1. 所有外部调用设超时 (连接 1s + 读 3s)
  2. 关键服务加熔断器 (失败率 50% 触发)
  3. 网关层全局限流 (保护总入口)
  4. 非核心功能提前准备降级开关
  5. 异步解耦 (消息队列削峰)
  6. 弹性扩容 (HPA 自动扩缩)
  7. 定期混沌工程演练 (Chaos Monkey)
```

## 9. 混沌工程？

**回答：**

```
混沌工程 (Chaos Engineering):
  主动注入故障, 验证系统弹性

原则:
  1. 建立正常行为假设
  2. 引入真实世界的故障
  3. 在生产环境运行实验
  4. 自动化持续运行
  5. 最小化影响范围

常见故障注入:
  ┌──────────────┬──────────────────────────────┐
  │ 类型          │ 工具/方法                     │
  ├──────────────┼──────────────────────────────┤
  │ Pod 重启/删除 │ Chaos Mesh, Litmus            │
  │ 网络延迟/丢包 │ tc netem, Chaos Mesh          │
  │ CPU/内存压力  │ stress-ng, Chaos Mesh         │
  │ 节点故障      │ kubelet stop, 关机            │
  │ AZ 故障       │ 关闭一个可用区的所有节点       │
  │ DNS 故障      │ CoreDNS 注入错误              │
  │ 磁盘故障      │ 填满磁盘, IO 延迟             │
  └──────────────┴──────────────────────────────┘

工具:
  Chaos Mesh:    K8s 原生, CNCF 项目
  Litmus:        K8s 混沌工程平台
  Chaos Monkey:  Netflix 开源
  Gremlin:       商业化混沌工程平台
  AWS FIS:       AWS 故障注入服务

Chaos Mesh 示例:
  apiVersion: chaos-mesh.org/v1alpha1

    action: pod-kill

      namespaces: [production]

      cron: "@every 1h"    # 每小时随机杀一个 Pod
```

## 10. 服务容错面试速答？

**回答：**

```
Q: 熔断器三种状态?
A: Closed(正常) → Open(断路,快速失败) → Half-Open(探测恢复)
   失败率超阈值触发, 超时后探测, 成功则恢复

Q: 令牌桶和漏桶的区别?
A: 漏桶: 恒定速率输出, 不允许突发
   令牌桶: 恒定速率生成令牌, 允许一定突发
   推荐令牌桶 (灵活, 允许短暂高峰)

Q: 限流在哪一层做?
A: 网关层: 全局限流 (Nginx/Kong)
   服务层: 单服务限流 (Sentinel)
   分布式: Redis + Lua (全局精确)

Q: 怎么防止服务雪崩?
A: 超时 + 熔断 + 限流 + 隔离 + 降级 + 异步
   多层防御, 快速失败, 故障隔离

Q: 重试要注意什么?
A: 只重试幂等操作和可恢复错误
   指数退避 + 随机抖动 (防重试风暴)
   设最大重试次数 (2-3次)

Q: 降级和熔断的区别?
A: 熔断: 自动触发, 下游故障时切断调用
   降级: 主动策略, 放弃非核心功能保核心

Q: 混沌工程是什么?
A: 主动注入故障验证系统弹性 (杀Pod/网络延迟)
   工具: Chaos Mesh / Litmus
   目的: 提前发现弱点, 而非等生产出事
```
