# 微服务基础与架构演进

---

## 1. 什么是微服务？

**回答：**

```
微服务: 将应用拆分为一组小型、独立部署的服务
每个服务:
  - 围绕特定业务能力构建
  - 独立开发、测试、部署
  - 拥有自己的数据存储
  - 通过 API 通信

核心特征:
  ┌──────────────────────────────────────────────┐
  │ 1. 服务组件化     - 以服务而非库的方式组件化   │
  │ 2. 围绕业务组织   - 按业务能力而非技术层划分   │
  │ 3. 产品化思维     - 你构建,你运维 (You build it│
  │                     you run it)               │
  │ 4. 去中心化治理   - 技术栈自由选择            │
  │ 5. 去中心化数据   - 每个服务有自己的数据库     │
  │ 6. 基础设施自动化 - CI/CD + 容器化            │
  │ 7. 容错设计       - 面向失败设计              │
  │ 8. 演进式设计     - 可替换,可重写              │
  └──────────────────────────────────────────────┘
```

## 2. 单体架构 vs 微服务架构？

**回答：**

```
单体:
  ┌────────────────────────────────┐
  │  用户模块 │ 订单模块 │ 支付模块  │ → 一个进程、一个部署包
  │  ─────────────────────────────│
  │           共享数据库           │
  └────────────────────────────────┘

微服务:
  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ 用户服务  │  │ 订单服务  │  │ 支付服务  │  → 独立进程
  └──────────┘  └──────────┘  └──────────┘
       ↕              ↕              ↕
   User DB       Order DB      Payment DB   → 独立数据库

详细对比:
  ┌──────────────┬────────────────┬────────────────────┐
  │ 维度          │ 单体            │ 微服务              │
  ├──────────────┼────────────────┼────────────────────┤
  │ 部署          │ 整体部署        │ 独立部署            │
  │ 技术栈        │ 统一            │ 每服务自由选择      │
  │ 扩展          │ 整体扩展        │ 按服务独立扩展      │
  │ 故障影响      │ 全局故障        │ 故障隔离            │
  │ 开发效率      │ 初期快,后期慢   │ 初期慢,后期快       │
  │ 代码复杂度    │ 高 (大泥球)     │ 低 (单个服务小)     │
  │ 运维复杂度    │ 低              │ 高 (需要平台支撑)   │
  │ 数据管理      │ 共享数据库      │ 每服务独立数据库    │
  │ 团队规模      │ 小团队适合      │ 大团队适合          │
  │ 测试          │ 简单 (端到端)   │ 复杂 (契约测试)     │
  │ 调试          │ 简单 (单进程)   │ 复杂 (分布式追踪)   │
  │ 事务          │ 本地事务        │ 分布式事务          │
  └──────────────┴────────────────┴────────────────────┘

什么时候不该用微服务:
  ❌ 团队 < 10 人
  ❌ 业务简单, 变化少
  ❌ 没有 DevOps 基础设施
  ❌ 没有容器/K8s 平台
  ❌ 创业初期 (先跑通再拆)
```

## 3. 架构演进路径？

**回答：**

```
典型演进路线:

  单体 → 模块化单体 → 微服务

  阶段 1: 单体 (Monolith)
    一个代码库, 一个部署包
    适合: 0-1 阶段, 快速验证

  阶段 2: 模块化单体 (Modular Monolith)
    代码按模块划分, 模块间接口清晰
    单一部署, 但内部解耦
    适合: 过渡期, 为拆分做准备

  阶段 3: 微服务 (Microservices)
    独立部署, 独立数据库
    需要完善的 DevOps 工具链
    适合: 业务复杂, 团队规模大

演进而非重写:
  ┌────────────────────────────────────────┐
  │ 绞杀者模式 (Strangler Fig Pattern)     │
  │                                        │
  │   1. 新功能用微服务写                   │
  │   2. 旧功能逐步迁移                     │
  │   3. 通过 API Gateway 路由              │
  │   4. 旧单体逐步缩小直到消失             │
  │                                        │
  │   Client                               │
  │     ↓                                   │
  │   API Gateway                          │
  │     ├── /new-feature → 新微服务         │
  │     ├── /migrated    → 已迁移微服务     │
  │     └── /legacy      → 旧单体          │
  └────────────────────────────────────────┘

反模式:
  ❌ 大爆炸式重写 (风险极高)
  ❌ 分布式单体 (拆了但强耦合)
  ❌ 微服务优先 (先单体,后拆分)
```

## 4. 微服务拆分原则？

**回答：**

```
拆分原则:
  ┌──────────────────┬──────────────────────────────┐
  │ 原则              │ 说明                          │
  ├──────────────────┼──────────────────────────────┤
  │ 单一职责          │ 一个服务只做一件事            │
  │ 高内聚低耦合      │ 服务内相关, 服务间独立        │
  │ 围绕业务能力      │ 按业务域而非技术层            │
  │ 数据隔离          │ 每个服务有自己的数据           │
  │ 自治性            │ 独立开发/测试/部署/扩展       │
  │ 可独立部署        │ 改一个不影响其他              │
  └──────────────────┴──────────────────────────────┘

拆分策略:
  1. 按业务功能: 用户服务、商品服务、订单服务、支付服务
  2. 按数据模型: 每个服务管理自己的数据实体
  3. 按变更频率: 频繁变化的功能独立
  4. 按团队: 一个小组负责一个服务 (康威定律)
  5. 按扩展需求: 高流量模块独立扩展

电商系统拆分示例:
  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ 用户服务  │  │ 商品服务  │  │ 搜索服务  │
  │ 注册/登录 │  │ 商品管理  │  │ ES 搜索   │
  │ 用户信息  │  │ 分类/品牌 │  │ 推荐      │
  └──────────┘  └──────────┘  └──────────┘
  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ 订单服务  │  │ 支付服务  │  │ 库存服务  │
  │ 下单/取消 │  │ 支付/退款 │  │ 库存管理  │
  │ 订单查询  │  │ 对账      │  │ 扣减/回滚 │
  └──────────┘  └──────────┘  └──────────┘
  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ 通知服务  │  │ 物流服务  │  │ 营销服务  │
  │ 短信/邮件 │  │ 发货/跟踪 │  │ 优惠券    │
  └──────────┘  └──────────┘  └──────────┘

拆分粒度判断:
  太粗: 服务像小单体, 变更频繁, 团队协作困难
  太细: 分布式开销大, 调用链过长, 运维复杂
  合适: 2-Pizza Team (6-8人) 能完全负责
```

## 5. 领域驱动设计 (DDD) 与微服务？

**回答：**

```
DDD 核心概念:
  ┌──────────────────┬──────────────────────────────┐
  │ 概念              │ 说明                          │
  ├──────────────────┼──────────────────────────────┤
  │ 领域 (Domain)     │ 业务问题空间                  │
  │ 子域 (Subdomain)  │ 核心域/支撑域/通用域          │
  │ 限界上下文 (BC)   │ 领域模型的边界, 对应一个微服务 │
  │ 聚合 (Aggregate)  │ 一组相关实体的一致性边界       │
  │ 聚合根             │ 聚合的入口实体                │
  │ 领域事件           │ 领域中发生的有意义的事件       │
  │ 上下文映射         │ BC 之间的关系                  │
  └──────────────────┴──────────────────────────────┘

限界上下文 = 微服务边界:
  电商领域:
    ┌────────────────┐  ┌────────────────┐
    │ 商品上下文      │  │ 订单上下文      │
    │ Product (聚合根)│  │ Order (聚合根)  │
    │ Category        │  │ OrderItem       │
    │ Brand           │  │ ShippingAddr    │
    └────────────────┘  └────────────────┘
    
    同一个 "商品" 在不同上下文含义不同:
      商品上下文: 名称, 描述, 分类, 品牌...
      订单上下文: 商品ID, 名称, 价格, 数量 (快照)
      库存上下文: 商品ID, 库存数, 锁定数

子域分类:
  核心域: 业务核心竞争力 (自研, 投入最多) 
    → 如电商的交易/推荐系统
  支撑域: 支撑核心域运作 (可外包)
    → 如物流跟踪, 内容管理
  通用域: 通用能力 (用现成方案)
    → 如认证, 邮件, 短信

DDD 指导拆分:
  1. 识别子域
  2. 划分限界上下文
  3. 每个 BC → 一个微服务
  4. BC 间通过 API/事件通信
```

## 6. 康威定律？

**回答：**

```
康威定律 (Conway's Law):
  "设计系统的组织, 其产出的设计等同于组织之间的沟通结构"

简单说:
  组织架构 → 决定 → 系统架构

案例:
  一个团队 → 单体应用
  三个团队 → 三个服务 (自然形成)
  前端+后端团队 → 前后端分离架构

正确应用:
  ✅ 逆康威定律 (Inverse Conway Maneuver):
     先设计目标架构 → 再调整团队组织

  微服务团队模型:
    ┌──────────────────────────────────┐
    │ 产品团队 A (用户域)              │
    │ 前端 + 后端 + 测试 + SRE        │
    │ 负责: 用户服务 (全生命周期)      │
    ├──────────────────────────────────┤
    │ 产品团队 B (交易域)              │
    │ 前端 + 后端 + 测试 + SRE        │
    │ 负责: 订单 + 支付服务            │
    ├──────────────────────────────────┤
    │ 平台团队                         │
    │ 负责: K8s / CI/CD / 监控 / 网关  │
    └──────────────────────────────────┘

  "Two Pizza Team": 一个团队 6-8 人,
  两个披萨能喂饱的团队规模 (Amazon 实践)
```

## 7. 微服务数据管理策略？

**回答：**

```
核心原则: Database per Service (每服务独立数据库)

  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ 用户服务  │  │ 订单服务  │  │ 商品服务  │
  └────┬─────┘  └────┬─────┘  └────┬─────┘
       │              │              │
   [MySQL]       [PostgreSQL]    [MongoDB]
   user_db        order_db       product_db

优点:
  ✅ 服务独立性 (技术栈自由)
  ✅ 独立扩展 (数据库独立扩容)
  ✅ 故障隔离 (一个库挂不影响其他)

挑战与解决:
  ┌──────────────────┬──────────────────────────────┐
  │ 挑战              │ 解决方案                      │
  ├──────────────────┼──────────────────────────────┤
  │ 跨服务查询        │ API 聚合 / BFF / CQRS        │
  │ 数据一致性        │ Saga / 事件驱动 / 最终一致    │
  │ 数据冗余          │ 允许适度冗余, 事件同步        │
  │ 报表/分析         │ 数据仓库 / CDC 同步           │
  │ 分布式事务        │ Saga / 本地消息表              │
  └──────────────────┴──────────────────────────────┘

数据同步模式:
  1. 事件驱动: 服务A 发事件 → 服务B 消费更新本地副本
  2. CDC (Change Data Capture): Debezium 监听 binlog
  3. API 调用: 需要时实时查询 (强一致但耦合)

反模式:
  ❌ 共享数据库: 多服务直接共享同一数据库
     → 强耦合, 无法独立演进
  ❌ 直接读其他服务的表:
     → 绕过 API, 破坏封装
```

## 8. 微服务与容器/K8s？

**回答：**

```
微服务 + K8s = 天然搭配

  微服务需要:            K8s 提供:
  ──────────            ──────────
  独立部署              Deployment / Pod
  服务发现              Service / DNS
  负载均衡              Service / Ingress
  弹性伸缩              HPA / VPA
  健康检查              Liveness / Readiness
  配置管理              ConfigMap / Secret
  滚动更新              RollingUpdate
  自我修复              ReplicaSet 自愈

典型部署架构:
  ┌──────────────────────────────────────┐
  │ Ingress Controller (Nginx / Traefik) │
  ├──────────────────────────────────────┤
  │ API Gateway (Kong / APISIX)          │
  ├──┬──────┬──────┬──────┬─────────────┤
  │  │User  │Order │Product│ Payment    │ ← Deployment
  │  │Svc   │Svc   │Svc    │ Svc       │
  │  │×3    │×3    │×2     │ ×2        │ ← 独立副本数
  ├──┴──────┴──────┴──────┴─────────────┤
  │ MySQL  │ Redis  │ Kafka │ MongoDB   │ ← StatefulSet
  └──────────────────────────────────────┘

一个微服务的 K8s 资源:
  Deployment    → Pod 管理和滚动更新
  Service       → 服务发现和负载均衡
  Ingress       → 外部访问路由
  HPA           → 自动水平扩展
  ConfigMap     → 非敏感配置
  Secret        → 敏感配置
  PDB           → Pod 中断预算
  ServiceMonitor→ Prometheus 监控
```

## 9. 12-Factor App？

**回答：**

```
12-Factor App: 构建 SaaS/微服务应用的 12 条最佳实践

  ┌────┬─────────────────┬──────────────────────────┐
  │ #  │ 因素             │ 说明                      │
  ├────┼─────────────────┼──────────────────────────┤
  │ 1  │ 代码基准         │ 一份代码, 多份部署         │
  │ 2  │ 依赖             │ 显式声明依赖               │
  │ 3  │ 配置             │ 配置存储在环境变量中       │
  │ 4  │ 后端服务         │ 把后端服务当资源           │
  │ 5  │ 构建/发布/运行   │ 严格分离三个阶段           │
  │ 6  │ 进程             │ 无状态进程                 │
  │ 7  │ 端口绑定         │ 通过端口绑定暴露服务       │
  │ 8  │ 并发             │ 通过进程模型扩展           │
  │ 9  │ 易处置性         │ 快速启动,优雅终止          │
  │ 10 │ 开发/生产等价    │ 保持环境尽可能相似         │
  │ 11 │ 日志             │ 把日志当事件流             │
  │ 12 │ 管理进程         │ 作为一次性进程运行管理任务  │
  └────┴─────────────────┴──────────────────────────┘

与微服务的映射:
  #3 配置 → K8s ConfigMap/Secret + Vault
  #6 无状态 → Pod 无状态 + 外部存储
  #8 并发 → K8s HPA 水平扩展
  #9 易处置 → Pod 快速启停 + Graceful Shutdown
  #11 日志 → stdout → Fluentd → ELK
```

## 10. 微服务基础面试速答？

**回答：**

```
Q: 什么是微服务?
A: 把应用拆成一组小型、独立部署的服务,
   每个服务围绕一个业务能力, 有自己的数据库,
   通过 API/消息通信

Q: 微服务的优缺点?
A: 优: 独立部署/扩展, 技术栈灵活, 故障隔离, 团队自治
   缺: 分布式复杂性, 数据一致性, 运维成本高, 调试困难

Q: 单体什么时候该拆?
A: 代码库太大难以维护, 部署频率受限,
   团队规模增大协作困难, 部分模块需要独立扩展
   前提: 有 DevOps 基础设施 (CI/CD + K8s + 监控)

Q: 微服务怎么拆分?
A: DDD 限界上下文指导拆分
   按业务能力 (非技术层) 划分
   每服务一数据库, Two Pizza Team 负责

Q: 什么是康威定律?
A: 系统架构反映组织沟通结构
   反向应用: 先设计架构再调整团队

Q: 12-Factor App 最重要的几条?
A: #3 配置外部化, #6 无状态进程,
   #9 快速启停, #11 日志到 stdout

Q: 微服务必须用 K8s 吗?
A: 不是必须, 但强烈推荐
   K8s 提供了微服务所需的几乎所有基础设施
   没有 K8s 则需要自建大量轮子
```
