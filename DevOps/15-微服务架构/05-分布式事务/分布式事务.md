# 分布式事务

---

## 1. 为什么微服务需要分布式事务？

**回答：**

```
单体架构:
  一个数据库, 本地事务 (ACID) 就够了
  BEGIN → UPDATE 库存 → UPDATE 余额 → COMMIT

微服务架构:
  每个服务独立数据库, 无法使用本地事务
  
  订单服务 (Order DB) → 库存服务 (Stock DB) → 支付服务 (Pay DB)
  
  问题: 订单创建了, 但库存扣减失败 → 数据不一致!

分布式事务本质:
  跨多个服务/数据库的操作, 要么全部成功, 要么全部回滚
  但分布式环境下, 网络不可靠 + 节点可能故障
  → 需要特殊的分布式事务方案
```

## 2. 2PC 两阶段提交？

**回答：**

```
2PC (Two-Phase Commit):

  阶段一: Prepare (准备)
    协调者 → 所有参与者: "你们能提交吗?"
    参与者: 执行事务但不提交, 记 Undo/Redo 日志
    参与者 → 协调者: "YES" 或 "NO"

  阶段二: Commit/Rollback
    如果所有 YES → 协调者: "提交!"  → 所有参与者 COMMIT
    如果有 NO    → 协调者: "回滚!"  → 所有参与者 ROLLBACK

时序图:
  协调者        参与者A       参与者B
    │─ Prepare ──→│             │
    │─ Prepare ──────────────→│
    │             │             │
    │←── YES ────│             │
    │←── YES ──────────────── │
    │             │             │
    │── Commit ──→│             │
    │── Commit ──────────────→│

问题:
  ┌──────────────────┬──────────────────────────────┐
  │ 问题              │ 说明                          │
  ├──────────────────┼──────────────────────────────┤
  │ 同步阻塞          │ 参与者在 Prepare 后锁资源等待  │
  │ 单点故障          │ 协调者挂了, 参与者不知道提交还是回滚│
  │ 数据不一致        │ Commit 阶段部分成功部分失败    │
  │ 性能差            │ 两轮网络通信 + 持锁等待        │
  └──────────────────┴──────────────────────────────┘

适用: 数据库层面 (XA 事务), 对性能要求不高的场景
```

## 3. 3PC 三阶段提交？

**回答：**

```
3PC: 在 2PC 基础上增加 CanCommit 阶段 + 超时机制

  阶段一: CanCommit (询问)
    协调者: "你们能执行这个事务吗?" (不执行, 只确认)
    参与者: 检查资源, 回复 YES/NO

  阶段二: PreCommit (预提交)
    协调者: "开始执行, 但先不提交"
    参与者: 执行事务, 记日志, 等待

  阶段三: DoCommit (提交)
    协调者: "提交!"
    参与者: 正式提交

改进:
  增加超时机制: 参与者超时未收到指令 → 默认提交
  → 解决了协调者单点故障时参与者永久阻塞问题
  
  但仍有数据不一致风险 (超时默认提交可能不正确)
  → 实际很少使用, TCC/Saga 更常见
```

## 4. TCC 模式？

**回答：**

```
TCC: Try - Confirm - Cancel

  Try:     预留资源 (冻结库存, 冻结余额)
  Confirm: 确认提交 (真正扣减)
  Cancel:  取消回滚 (解冻)

下单流程:
  ┌──────────┐     ┌──────────┐     ┌──────────┐
  │ 订单服务  │     │ 库存服务  │     │ 支付服务  │
  └────┬─────┘     └────┬─────┘     └────┬─────┘
       │                │                │
  Try: │  创建待确认订单  │  冻结10件库存   │  冻结100元
       │                │                │
  全部 Try 成功:
       │                │                │
  Confirm: 订单→已确认   │  库存-10        │  余额-100
       │                │                │
  任一 Try 失败:
       │                │                │
  Cancel: 取消订单       │  解冻库存       │  解冻余额

代码示例:
```

```python
# TCC 接口定义
class StockTccService:
    def try_deduct(self, order_id, sku_id, quantity):
        """Try: 冻结库存"""
        # UPDATE stock SET frozen = frozen + quantity
        # WHERE sku_id = ? AND available >= quantity
        pass

    def confirm_deduct(self, order_id, sku_id, quantity):
        """Confirm: 真正扣减"""
        # UPDATE stock SET frozen = frozen - quantity,
        #   total = total - quantity WHERE sku_id = ?
        pass

    def cancel_deduct(self, order_id, sku_id, quantity):
        """Cancel: 解冻"""
        # UPDATE stock SET frozen = frozen - quantity
        # WHERE sku_id = ?
        pass
```

```
TCC 注意事项:
  ┌────────────────┬──────────────────────────────┐
  │ 问题            │ 解决方案                      │
  ├────────────────┼──────────────────────────────┤
  │ 空回滚          │ Cancel 时检查 Try 是否执行过   │
  │ 幂等性          │ 用唯一事务 ID 防重复           │
  │ 悬挂            │ Try 晚于 Cancel 到达, 拒绝执行 │
  │ 业务侵入        │ 每个服务都要实现 3 个接口       │
  └────────────────┴──────────────────────────────┘

适用: 对一致性要求高, 资金类业务 (支付/转账)
框架: Seata TCC, ByteTCC, Hmily
```

## 5. Saga 模式？

**回答：**

```
Saga: 将长事务拆成一系列本地事务 + 补偿事务

  正向流程:
    T1 → T2 → T3 → T4  (每步是本地事务)
  
  补偿流程 (某步失败):
    T1 → T2 → T3(失败) → C2 → C1  (逆序补偿)

示例: 订单流程
  T1: 创建订单 (待支付)
  T2: 扣减库存
  T3: 扣减余额
  T4: 订单状态→已支付
  
  如果 T3 失败:
  C2: 恢复库存
  C1: 取消订单

两种实现:

  编排式 (Choreography):
    服务间通过事件驱动
    订单服务 → 发事件 → 库存服务 → 发事件 → 支付服务
    
    优点: 解耦, 简单
    缺点: 流程分散, 难追踪

  协调式 (Orchestration):
    中心协调器控制流程
    ┌────────────────────────┐
    │    Saga 协调器          │
    │  T1 → T2 → T3 → T4   │
    │  失败 → C3 → C2 → C1 │
    └────────────────────────┘
    
    优点: 流程清晰, 易监控
    缺点: 协调器是单点

Saga vs TCC:
  ┌──────────┬────────────────┬────────────────┐
  │          │ TCC            │ Saga           │
  ├──────────┼────────────────┼────────────────┤
  │ 隔离性    │ 强 (资源预留)   │ 弱 (中间状态可见)│
  │ 一致性    │ 强一致          │ 最终一致        │
  │ 业务侵入  │ 高 (3个接口)    │ 中 (补偿接口)   │
  │ 性能      │ 中              │ 高              │
  │ 适用      │ 资金/短事务     │ 长流程/跨服务   │
  └──────────┴────────────────┴────────────────┘

框架: Seata Saga, Temporal, Cadence, AWS Step Functions
```

## 6. 本地消息表 + 消息队列？

**回答：**

```
本地消息表: 最实用的分布式事务方案

原理:
  将 业务操作 和 消息写入 放在同一个本地事务中
  → 保证 "业务+消息" 的原子性
  → 消息异步发送, 消费者最终处理

流程:
  1. 订单服务:
     BEGIN
       INSERT INTO orders (...)          -- 业务操作
       INSERT INTO message_table (...)   -- 写消息到本地表
     COMMIT
  
  2. 定时任务/MQ:
     扫描 message_table → 发送到 MQ → 标记已发送
  
  3. 库存服务:
     消费 MQ 消息 → 扣减库存 → ACK

  4. 重试:
     失败的消息定时重发, 消费者保证幂等
```

```sql
-- 本地消息表
CREATE TABLE outbox_messages (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT,
    message_id  VARCHAR(64) UNIQUE NOT NULL,
    topic       VARCHAR(128) NOT NULL,
    payload     JSON NOT NULL,
    status      ENUM('PENDING', 'SENT', 'FAILED') DEFAULT 'PENDING',
    retry_count INT DEFAULT 0,
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_status_created (status, created_at)
);

-- 业务操作 + 消息写入 = 同一事务
START TRANSACTION;
  INSERT INTO orders (order_id, user_id, amount) VALUES (?, ?, ?);
  INSERT INTO outbox_messages (message_id, topic, payload)
    VALUES (UUID(), 'order.created', '{"orderId": "...", "items": [...]}');
COMMIT;
```

```
变体: Transactional Outbox + CDC

  Outbox 表 → CDC (Debezium) → Kafka → 消费者服务
  
  优点: 不需要定时轮询, 实时性更好
  Debezium 监听 binlog, 自动捕获 outbox 表变更

对比:
  ┌──────────────┬────────────────┬──────────────────┐
  │ 方案          │ 可靠性          │ 实时性            │
  ├──────────────┼────────────────┼──────────────────┤
  │ 定时轮询      │ 高              │ 秒级延迟          │
  │ CDC (Debezium)│ 高              │ 毫秒级延迟        │
  │ 事务消息 (RMQ)│ 高              │ 毫秒级延迟        │
  └──────────────┴────────────────┴──────────────────┘
```

## 7. Seata 分布式事务框架？

**回答：**

```
Seata: 阿里开源分布式事务框架

四种模式:
  AT:   自动补偿 (基于 SQL 解析, 无侵入)
  TCC:  手动编码 Try-Confirm-Cancel
  Saga: 长事务补偿
  XA:   标准 XA 两阶段提交

AT 模式 (最常用):
  基于 SQL 解析, 自动生成回滚 SQL
  
  1. 一阶段: 拦截 SQL, 保存前镜像 (Before Image)
     执行 SQL, 保存后镜像 (After Image)
     生成行锁, 提交本地事务
  
  2. 二阶段 Commit: 异步删除 Undo Log
  
  2. 二阶段 Rollback: 用 Before Image 还原数据

架构:
  TC (Transaction Coordinator): 事务协调器 (Seata Server)
  TM (Transaction Manager):     事务管理器 (发起者)
  RM (Resource Manager):         资源管理器 (参与者)

  TM 开启全局事务 → 获取 XID
  RM 注册分支事务 → 报告状态
  TM 决定全局提交/回滚
  TC 驱动所有 RM 提交/回滚
```

```yaml
# Seata 配置
seata:
  enabled: true
  tx-service-group: my_tx_group
  service:
    vgroup-mapping:
      my_tx_group: default
  registry:
    type: nacos
    nacos:
      server-addr: nacos:8848
```

```java
// AT 模式使用 (Java)
@GlobalTransactional(timeoutMills = 30000, name = "create-order")
public void createOrder(OrderDTO order) {
    // 1. 创建订单 (自动纳入全局事务)
    orderService.create(order);
    
    // 2. 扣减库存 (远程调用, 自动纳入)
    stockService.deduct(order.getSkuId(), order.getQuantity());
    
    // 3. 扣减余额 (远程调用, 自动纳入)
    accountService.debit(order.getUserId(), order.getAmount());
    
    // 任一步骤异常 → 自动全局回滚
}
```

## 8. 幂等性设计？

**回答：**

```
幂等性: 同一操作执行多次, 结果与执行一次相同

为什么重要:
  网络超时 → 重试 → 重复请求
  MQ 消费 → 至少一次投递 → 重复消息
  用户点击 → 重复提交

天然幂等:
  查询 (SELECT)       → 天然幂等
  删除 (DELETE by PK) → 天然幂等
  覆盖更新 (SET x=5)  → 天然幂等

非幂等:
  INSERT              → 重复插入
  UPDATE SET x=x+1    → 重复累加
  发送通知             → 重复发送

实现方案:
  ┌──────────────────┬──────────────────────────────┐
  │ 方案              │ 实现方式                      │
  ├──────────────────┼──────────────────────────────┤
  │ 唯一约束          │ 数据库 UNIQUE KEY             │
  │ Token 机制        │ 请求前获取 Token, 用后即删     │
  │ 去重表            │ 记录已处理的请求               │
  │ 乐观锁            │ WHERE version = ?             │
  │ 状态机             │ 只允许合法状态转换             │
  │ Redis SETNX       │ 分布式去重 (带过期时间)         │
  └──────────────────┴──────────────────────────────┘
```

```python
# Redis 幂等实现
import redis
import json

r = redis.Redis()

def process_with_idempotency(request_id, handler):
    """幂等处理包装"""
    key = f"idempotent:{request_id}"
    
    # 1. 检查是否已处理
    result = r.get(key)
    if result:
        return json.loads(result)  # 返回之前的结果
    
    # 2. 尝试获取处理权 (SETNX)
    if not r.set(key, "processing", nx=True, ex=3600):
        return {"status": "duplicate", "message": "请求正在处理中"}
    
    try:
        # 3. 执行业务逻辑
        result = handler()
        
        # 4. 保存结果
        r.set(key, json.dumps(result), ex=86400)
        return result
    except Exception as e:
        r.delete(key)  # 失败时删除, 允许重试
        raise
```

```sql
-- 去重表方案
CREATE TABLE idempotent_records (
    request_id VARCHAR(64) PRIMARY KEY,
    result     JSON,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_created (created_at)
);

-- 处理前检查, 利用 UNIQUE 约束防重
INSERT IGNORE INTO idempotent_records (request_id, result)
VALUES (?, ?);
-- affected_rows = 0 表示已处理过
```

## 9. 最终一致性方案选型？

**回答：**

```
分布式事务选型指南:

  强一致性场景 (资金/交易):
    推荐: TCC (Seata TCC / Hmily)
    原因: 资源预留, 隔离性强
    代价: 业务侵入大, 开发成本高

  一般业务 (订单/库存):
    推荐: Seata AT 模式
    原因: 无侵入, 自动补偿
    代价: 行锁竞争, 性能有影响

  长流程业务 (审批/物流):
    推荐: Saga (Temporal / Seata Saga)
    原因: 灵活补偿, 支持长事务
    代价: 中间状态可见

  异步解耦场景 (通知/同步):
    推荐: 本地消息表 + MQ
    原因: 简单可靠, 最终一致
    代价: 需要幂等消费

决策树:
  需要强一致性?
    ├── 是 → 金融/支付? → TCC
    │        一般业务?  → Seata AT
    │
    └── 否 (最终一致即可)
         ├── 长流程?    → Saga
         └── 短流程?    → 本地消息表 + MQ

实际经验:
  80% 的场景: 本地消息表 + MQ (够用且简单)
  15% 的场景: Seata AT (需要跨库一致性)
  5%  的场景: TCC (资金核心链路)

  不要过度使用分布式事务!
  能用 最终一致性 的场景, 就不要用 强一致性
```

## 10. 分布式事务面试速答？

**回答：**

```
Q: 分布式事务有哪些方案?
A: 2PC/3PC (理论), TCC (预留/确认/取消),
   Saga (补偿事务链), 本地消息表+MQ (最终一致),
   Seata AT (自动补偿)

Q: TCC 和 Saga 区别?
A: TCC: Try 预留资源 → 隔离性强, 业务侵入大
   Saga: 直接执行 → 失败则补偿, 中间状态可见
   TCC 适合资金短事务, Saga 适合长流程

Q: 实际项目中怎么用?
A: 大多数场景: 本地消息表 + MQ (简单可靠)
   需要跨库一致性: Seata AT (无侵入)
   资金核心: TCC 模式

Q: 如何保证消息不丢?
A: 生产端: 本地消息表 (同事务)
   MQ: 持久化 + 同步刷盘
   消费端: 手动 ACK + 幂等消费

Q: 幂等怎么做?
A: 数据库唯一索引 / Redis SETNX / 去重表
   核心: 用唯一 ID 标识每次操作

Q: CAP 和分布式事务什么关系?
A: CAP 说不能同时满足一致性和可用性
   TCC/2PC 选 CP (强一致, 性能低)
   Saga/消息 选 AP (最终一致, 可用性高)

Q: Seata AT 原理?
A: SQL 级别拦截, 自动生成 undo log
   一阶段: 执行 SQL + 保存前后镜像 + 本地提交
   二阶段: 成功删 log, 失败用前镜像回滚
```
