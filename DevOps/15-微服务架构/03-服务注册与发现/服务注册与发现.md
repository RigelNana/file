# 服务注册与发现

---

## 1. 为什么需要服务注册与发现？

**回答：**

```
传统方式: 硬编码 IP + 端口
  order-service → http://10.0.1.5:8080 (写死配置)
  ❌ 实例动态变化 (扩缩容/故障恢复) → 配置过期
  ❌ 多实例负载均衡 → 无法自动感知

微服务场景:
  服务实例数量多且动态变化
  容器/K8s 环境中 IP 随时变化
  需要自动感知服务上下线

服务注册:
  服务启动 → 将自身 (名称 + 地址 + 端口) 注册到注册中心
  服务停止 → 注销注册

服务发现:
  调用方 → 从注册中心查询目标服务的实例列表 → 选一个调用

  ┌────────────────────────────┐
  │      注册中心               │
  │  user-svc:                 │
  │    - 10.0.1.1:8080 ✅      │
  │    - 10.0.1.2:8080 ✅      │
  │    - 10.0.1.3:8080 ❌ (下线)│
  │  order-svc:                │
  │    - 10.0.2.1:8080 ✅      │
  └────────────────────────────┘
       ↑ 注册/心跳     ↓ 查询
   服务实例          调用方
```

## 2. 客户端发现 vs 服务端发现？

**回答：**

```
客户端发现 (Client-side Discovery):
  客户端直接从注册中心获取实例列表
  客户端自己做负载均衡

  调用方 ──查询──→ 注册中心
    │               │
    │ ←── 实例列表 ──┘
    │
    ├── 轮询/随机/权重 选择
    │
    └── 直连目标实例

  优点: 少一跳, 性能好
  缺点: 客户端逻辑重, 每种语言需要 SDK
  工具: Eureka + Ribbon, Nacos

服务端发现 (Server-side Discovery):
  通过负载均衡器/代理转发
  调用方只需要知道代理地址

  调用方 ──→ 负载均衡器/代理 ──→ 目标实例
                    │
                注册中心

  优点: 客户端简单, 语言无关
  缺点: 多一跳, 多一个组件
  工具: K8s Service, AWS ALB, Consul Connect

K8s 的服务发现:
  属于服务端发现模式
  Service → ClusterIP → kube-proxy 转发
  DNS: user-svc.namespace.svc.cluster.local
  调用方只需: http://user-svc:8080
```

## 3. 主流注册中心对比？

**回答：**

```
  ┌──────────────┬──────────┬──────────┬──────────┬──────────┐
  │ 维度          │ etcd     │ Consul   │ Nacos    │ ZooKeeper│
  ├──────────────┼──────────┼──────────┼──────────┼──────────┤
  │ 一致性协议    │ Raft     │ Raft     │ Raft     │ ZAB      │
  │              │          │          │ +Distro  │          │
  │ CAP          │ CP       │ CP       │ CP+AP可选│ CP       │
  │ 健康检查      │ TTL      │ 多种     │ 多种     │ 会话     │
  │ KV 存储      │ ✅       │ ✅       │ ✅       │ ✅       │
  │ 服务发现      │ 基础     │ 原生     │ 原生     │ 基础     │
  │ 配置中心      │ ✅       │ ✅       │ ✅       │ ✅       │
  │ 多数据中心    │ ❌       │ ✅       │ ✅       │ ❌       │
  │ 语言          │ Go       │ Go       │ Java     │ Java     │
  │ K8s集成       │ 内置     │ 好       │ 一般     │ 差       │
  │ 推荐场景      │ K8s/通用 │ 多语言   │ Java生态 │ 旧系统   │
  └──────────────┴──────────┴──────────┴──────────┴──────────┘

选型建议:
  K8s 环境                → K8s 内置 DNS + Service (够用)
  非 K8s + 多语言         → Consul (功能全面)
  Java / Spring Cloud     → Nacos (阿里生态)
  已有 K8s 需要更强功能    → Consul on K8s
  纯 KV 存储              → etcd
```

## 4. Consul 详解？

**回答：**

```
Consul: HashiCorp 开源的服务网格和配置中心

核心功能:
  1. 服务发现: 注册/发现/健康检查
  2. KV 存储: 动态配置
  3. Service Mesh: Consul Connect (mTLS)
  4. 多数据中心: 内置跨 DC 支持

架构:
  ┌──────────────────────────────────┐
  │ Consul Cluster                   │
  │  Server 1 (Leader) ←Raft→ Server 2│
  │        ↕                  ↕      │
  │  Server 3                       │
  └───────────┬──────────────────────┘
              │ Gossip (Serf)
  ┌───────────┴──────────────────────┐
  │  Agent (Client mode)             │
  │  每个节点/Pod 上的 Agent          │
  │  → 注册本地服务                   │
  │  → 转发查询到 Server              │
  │  → 执行本地健康检查               │
  └──────────────────────────────────┘

健康检查类型:
  HTTP:   定期 GET 请求, 检查返回码
  TCP:    TCP 连接检查
  Script: 执行脚本, 检查退出码
  TTL:    服务定期上报心跳
  gRPC:   gRPC 健康检查协议

服务注册示例:
  {
    "service": {
      "name": "user-service",
      "port": 8080,
      "tags": ["v1", "primary"],
      "check": {
        "http": "http://localhost:8080/health",
        "interval": "10s",
        "timeout": "3s"
      }
    }
  }

DNS 查询:
  dig @127.0.0.1 -p 8600 user-service.service.consul
  → 返回健康实例的 IP

HTTP API 查询:
  curl http://consul:8500/v1/health/service/user-service?passing=true
```

## 5. K8s 服务发现机制？

**回答：**

```
K8s 内置服务发现: Service + DNS

  ┌─────────────────────────────────────────┐
  │  CoreDNS                                │
  │  user-svc.default.svc.cluster.local     │
  │  → ClusterIP: 10.96.0.100              │
  └─────────────────────────────────────────┘
              ↓ DNS 解析
  ┌──────────────────────┐
  │ Service: user-svc    │
  │ ClusterIP: 10.96.0.100│
  │ → Endpoints:         │
  │   10.0.1.1:8080      │
  │   10.0.1.2:8080      │
  │   10.0.1.3:8080      │
  └──────────────────────┘
              ↓ kube-proxy (iptables/IPVS)
  ┌─────┐ ┌─────┐ ┌─────┐
  │Pod 1│ │Pod 2│ │Pod 3│
  └─────┘ └─────┘ └─────┘

DNS 命名规则:
  <svc>.<namespace>.svc.cluster.local
  同 namespace 内可以直接: http://user-svc:8080
  跨 namespace:           http://user-svc.other-ns:8080

Service 类型:
  ┌──────────────┬──────────────────────────────┐
  │ 类型          │ 说明                          │
  ├──────────────┼──────────────────────────────┤
  │ ClusterIP    │ 集群内部 VIP (默认)            │
  │ NodePort     │ 每个节点暴露端口               │
  │ LoadBalancer │ 云厂商 LB (外部访问)           │
  │ Headless     │ 无 VIP, 直接返回 Pod IP       │
  │ ExternalName │ CNAME 映射外部服务             │
  └──────────────┴──────────────────────────────┘

Headless Service:
  clusterIP: None
  DNS 直接返回所有 Pod IP (不经过 VIP)
  用于: StatefulSet / 客户端负载均衡 / gRPC

Readiness Probe 与服务发现:
  Pod 未 Ready → 从 Endpoints 移除 → 不接收流量
  Pod Ready    → 加入 Endpoints → 接收流量
  → 自动健康管理
```

## 6. 负载均衡策略？

**回答：**

```
负载均衡算法:
  ┌──────────────────┬──────────────────────────────┐
  │ 算法              │ 说明                          │
  ├──────────────────┼──────────────────────────────┤
  │ 轮询 (Round Robin)│ 依次分配请求                  │
  │ 加权轮询          │ 按权重分配 (高配机器多分)      │
  │ 随机              │ 随机选择                      │
  │ 最少连接          │ 选当前连接最少的实例           │
  │ 加权最少连接      │ 结合权重和连接数               │
  │ IP Hash           │ 相同 IP 固定到同一实例         │
  │ 一致性 Hash       │ 最小化实例变化的影响           │
  └──────────────────┴──────────────────────────────┘

不同层的负载均衡:
  L4 (传输层): TCP/UDP 转发
    K8s Service (iptables/IPVS)
    AWS NLB
    
  L7 (应用层): HTTP 路由
    Nginx / Envoy / Traefik
    K8s Ingress
    AWS ALB

gRPC 负载均衡:
  L4 不适用: HTTP/2 多路复用, 单连接多请求
  需要 L7 负载均衡:
    - Envoy / Linkerd (Service Mesh)
    - 客户端负载均衡 (Headless Service + gRPC客户端)
    - K8s: Headless Service + DNS 轮询
```

## 7. 服务优雅上下线？

**回答：**

```
优雅下线:
  1. Pod 收到 SIGTERM
  2. 从 Endpoints 移除 (不再接收新请求)
  3. 处理完正在进行的请求
  4. 关闭连接, 清理资源
  5. 超时强制 SIGKILL

K8s 下线流程:
  ┌─────────────────────────────────────────┐
  │ 1. kubectl delete pod / 滚动更新        │
  │    ↓                                    │
  │ 2. Pod 标记 Terminating                 │
  │    ↓                                    │
  │ 3. 并行:                                │
  │    a. 从 Service Endpoints 移除         │
  │    b. 执行 preStop Hook                 │
  │    c. 发 SIGTERM 给容器                 │
  │    ↓                                    │
  │ 4. 等待 terminationGracePeriodSeconds   │
  │    (默认 30s)                           │
  │    ↓                                    │
  │ 5. 仍未退出 → SIGKILL 强制杀死          │
  └─────────────────────────────────────────┘
```

```yaml
# 优雅下线配置
spec:
  terminationGracePeriodSeconds: 60  # 给足时间
  containers:
  - name: app
    lifecycle:
      preStop:
        exec:
          command: ["sh", "-c", "sleep 5"]
          # sleep 等待 Endpoints 更新传播
    # 应用需要:
    # 1. 捕获 SIGTERM 信号
    # 2. 停止接收新请求
    # 3. 处理完已有请求
    # 4. 关闭连接
    # 5. 退出
```

```
优雅上线:
  Readiness Probe 通过后才加入 Endpoints
  → 确保服务完全启动后才接收流量

  startupProbe:  → 启动检查 (慢启动应用)
  readinessProbe: → 持续就绪检查 (流量控制)
  livenessProbe:  → 存活检查 (自愈)
```

## 8. 服务发现与 DNS？

**回答：**

```
DNS 服务发现:
  最简单的服务发现方式
  服务名 → DNS 解析 → IP 地址

K8s CoreDNS:
  Pod 内置 DNS 配置, 自动解析 Service 名
  
  resolv.conf:
    nameserver 10.96.0.10
    search default.svc.cluster.local svc.cluster.local cluster.local

  查询顺序:
    user-svc
    → user-svc.default.svc.cluster.local  ✅
    → user-svc.svc.cluster.local
    → user-svc.cluster.local

DNS 缓存问题:
  DNS 有 TTL 缓存, 实例变化后可能延迟感知
  K8s CoreDNS TTL 默认 30s
  解决: 客户端 DNS 缓存短 + 重试 + 健康检查

外部服务接入:
  ExternalName Service:
    apiVersion: v1
    kind: Service
    metadata:
      name: external-db
    spec:
      type: ExternalName
      externalName: db.external.example.com
  
  → Pod 内访问 external-db 自动 CNAME 到外部
```

## 9. 服务发现的高可用？

**回答：**

```
注册中心挂了怎么办?

  本地缓存:
    客户端缓存实例列表
    注册中心不可用时使用缓存
    Nacos/Eureka 都支持

  多节点部署:
    注册中心本身集群化 (3/5 节点)
    Raft 协议保证一致性
    允许少数节点故障

  自我保护模式 (Eureka):
    大量实例同时心跳失败时
    → 可能是网络问题而非实例故障
    → 不立即注销, 等待恢复

K8s 的高可用:
  etcd 集群 (3/5 节点)
  CoreDNS 多副本
  kube-proxy 每节点
  → 即使部分组件故障, 服务发现仍可工作

最佳实践:
  ✅ 注册中心集群部署 (至少 3 节点)
  ✅ 客户端缓存实例列表
  ✅ 健康检查 + 自动注销
  ✅ 优雅上下线
  ✅ 监控注册中心状态
```

## 10. 服务注册与发现面试速答？

**回答：**

```
Q: 服务发现是什么?
A: 服务启动时注册到注册中心 (名称+地址)
   调用方从注册中心查询实例列表
   自动感知服务上下线

Q: K8s 怎么做服务发现?
A: Service + CoreDNS
   同 namespace: http://svc-name:port
   跨 namespace: http://svc-name.ns:port
   kube-proxy (iptables/IPVS) 负载均衡

Q: 注册中心怎么选?
A: K8s 环境: 内置 Service + DNS 够用
   跨平台/多语言: Consul
   Java Spring Cloud: Nacos
   已有 K8s 不够: Consul on K8s

Q: CP 还是 AP?
A: 注册中心一般选 AP (可用性优先)
   允许短暂的注册信息不一致
   Eureka/Nacos 默认 AP
   etcd/ZK/Consul 是 CP (强一致但可能短暂不可用)

Q: 注册中心挂了怎么办?
A: 客户端本地缓存实例列表
   注册中心集群化 (3/5节点)
   自我保护模式 (不误注销)

Q: gRPC 负载均衡怎么做?
A: L4 不行 (HTTP/2 多路复用)
   需要 L7: Envoy / Linkerd / 客户端负载均衡
   K8s: Headless Service + DNS
```
