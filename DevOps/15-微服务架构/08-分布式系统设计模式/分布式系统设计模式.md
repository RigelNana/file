# 分布式系统设计模式

---

## 1. CAP 定理详解？

**回答：**

```
CAP 定理 (Brewer 定理):
  分布式系统最多只能同时满足以下三项中的两项

  C (Consistency)        一致性: 所有节点在同一时刻看到相同数据
  A (Availability)       可用性: 每个请求都能收到非错误响应
  P (Partition Tolerance) 分区容错: 网络分区时系统仍能运行

  ┌────────────────────────────────────┐
  │           C (一致性)                │
  │          /  \                      │
  │        CP    CA                    │
  │       /        \                   │
  │      P ───AP─── A                  │
  │                                    │
  │  CA: 网络不分区才能同时满足 C+A     │
  │      → 实际分布式系统不存在         │
  │                                    │
  │  结论: 分布式系统只能在 CP 和 AP 选  │
  └────────────────────────────────────┘

CP vs AP:
  ┌────────────┬──────────────────┬──────────────────┐
  │            │ CP (牺牲可用性)   │ AP (牺牲一致性)   │
  ├────────────┼──────────────────┼──────────────────┤
  │ 网络分区时  │ 拒绝不一致的请求  │ 返回可能过时数据  │
  │ 适用场景    │ 金融交易/库存扣减 │ 社交/推荐/缓存    │
  │ 代表系统    │ ZooKeeper/etcd   │ Cassandra/Eureka │
  │            │ HBase/MongoDB    │ DynamoDB/CouchDB │
  │ 一致性模型  │ 强一致性          │ 最终一致性        │
  └────────────┴──────────────────┴──────────────────┘

微服务中的应用:
  注册中心: AP 优先 (Eureka/Nacos) → 允许短暂数据不一致
  配置中心: CP 优先 (etcd/ZooKeeper) → 配置必须一致
  订单/支付: CP → 数据正确性优先
  推荐/搜索: AP → 可用性优先
```

## 2. BASE 理论？

**回答：**

```
BASE: 对 CAP 中 AP 方案的补充

  BA (Basically Available)  基本可用
    → 出现故障时允许部分功能降级
    → 例: 响应时间从 50ms 增加到 2s, 但不宕机

  S  (Soft State)           软状态
    → 允许系统中的数据存在中间状态
    → 例: 订单 "处理中" 状态

  E  (Eventually Consistent) 最终一致性
    → 不要求实时一致, 但经过一段时间后达成一致
    → 例: 下单后库存异步扣减, 最终一致

ACID vs BASE:
  ┌──────────────┬──────────────────┬──────────────────┐
  │              │ ACID             │ BASE             │
  ├──────────────┼──────────────────┼──────────────────┤
  │ 一致性        │ 强一致性          │ 最终一致性        │
  │ 可用性        │ 可能不可用        │ 基本可用          │
  │ 适用场景      │ 单机数据库        │ 分布式系统        │
  │ 复杂度        │ 简单             │ 需要补偿机制      │
  │ 性能          │ 较低 (锁等待)    │ 较高             │
  │ 代表          │ MySQL 事务       │ 微服务 Saga       │
  └──────────────┴──────────────────┴──────────────────┘

最终一致性的实现方式:
  1. 消息队列异步同步 (最常用)
  2. 定时任务对账
  3. 事件驱动 + 重试
  4. 读时修复 (Read Repair)
```

## 3. 一致性模型与选择？

**回答：**

```
一致性级别 (由强到弱):
  ┌────────────────────────────────────────────┐
  │ 强一致性 (Strong Consistency)               │
  │   写入后立即可读最新值                       │
  │   代表: ZooKeeper, etcd                     │
  │                                             │
  │ 线性一致性 (Linearizability)                 │
  │   操作按全局时间顺序排列                     │
  │   → 最强的一致性保证                         │
  │                                             │
  │ 顺序一致性 (Sequential Consistency)          │
  │   所有进程看到相同的操作顺序                  │
  │                                             │
  │ 因果一致性 (Causal Consistency)              │
  │   有因果关系的操作按顺序, 无关操作可乱序      │
  │                                             │
  │ 最终一致性 (Eventual Consistency)             │
  │   无新写入后, 最终所有副本收敛到一致          │
  │   代表: DynamoDB, Cassandra                  │
  │                                             │
  │ 弱一致性 (Weak Consistency)                  │
  │   不保证后续读取能读到最新写入                │
  └────────────────────────────────────────────┘

微服务场景下的选择:
  ┌──────────────────┬──────────────────────────┐
  │ 业务场景          │ 一致性选择                │
  ├──────────────────┼──────────────────────────┤
  │ 账户余额/库存扣减 │ 强一致性 (2PC/TCC)       │
  │ 订单状态流转      │ 最终一致性 (Saga)         │
  │ 用户资料更新      │ 最终一致性 (消息队列)     │
  │ 搜索索引同步      │ 最终一致性 (CDC/Event)    │
  │ 分布式锁          │ 强一致性 (etcd/ZK)       │
  │ 配置下发          │ 强一致性 (etcd)          │
  │ 缓存同步          │ 最终一致性 (失效策略)     │
  └──────────────────┴──────────────────────────┘
```

## 4. Sidecar/Ambassador/Adapter 模式？

**回答：**

```
容器设计模式 (源自 K8s 多容器 Pod):

1. Sidecar 模式:
   主容器 + 辅助容器, 增强功能

   ┌──────────────────────────────┐
   │            Pod               │
   │  ┌──────────┐ ┌──────────┐  │
   │  │ 业务容器  │ │ Sidecar  │  │
   │  │ (App)    │ │ (辅助)   │  │
   │  └──────────┘ └──────────┘  │
   └──────────────────────────────┘

   用途:
     日志收集 (Fluentd/Filebeat Sidecar)
     代理 (Envoy/Linkerd Sidecar)
     同步 (Git-Sync Sidecar)
     监控 (Prometheus Exporter Sidecar)

2. Ambassador 模式:
   代理/中介容器, 简化外部服务访问

   ┌──────────────────────────────┐
   │            Pod               │
   │  ┌──────────┐ ┌──────────┐  │
   │  │ 业务容器  │→│Ambassador│→→→ 外部服务
   │  │          │ │ (代理)   │  │ (DB/Cache)
   │  └──────────┘ └──────────┘  │
   └──────────────────────────────┘

   用途:
     数据库代理 (连接池/读写分离)
     缓存代理 (Redis Proxy)
     API 代理 (处理认证/重试)

3. Adapter 模式:
   标准化/转换容器, 统一输出格式

   ┌──────────────────────────────┐
   │            Pod               │
   │  ┌──────────┐ ┌──────────┐  │
   │  │ 业务容器  │→│ Adapter  │→→→ 监控系统
   │  │ (自定义   │ │ (格式    │  │ (标准格式)
   │  │  格式)   │ │  转换)   │  │
   │  └──────────┘ └──────────┘  │
   └──────────────────────────────┘

   用途:
     日志格式标准化
     指标格式转换 (→ Prometheus 格式)
     协议转换
```

## 5. CQRS 与事件溯源？

**回答：**

```
CQRS (Command Query Responsibility Segregation):
  命令查询职责分离 → 读写使用不同模型

传统模式:
  ┌──────────────┐
  │   Service    │
  │  读 + 写     │ → 同一个模型/数据库
  │  同一模型     │
  └──────┬───────┘
         │
    ┌────┴────┐
    │   DB    │
    └─────────┘

CQRS 模式:
  ┌──────────────┐     ┌──────────────┐
  │ Command 端    │     │ Query 端      │
  │ (写模型)      │     │ (读模型)      │
  │ 复杂业务逻辑  │     │ 优化查询      │
  └──────┬───────┘     └──────┬───────┘
         │                     │
    ┌────┴────┐          ┌────┴────┐
    │ Write DB│───同步──→│ Read DB │
    │ (MySQL) │  (Event) │ (ES/Redis)│
    └─────────┘          └─────────┘

事件溯源 (Event Sourcing):
  不存储当前状态, 而是存储所有状态变更事件
  当前状态 = 重放所有事件

  传统: UPDATE account SET balance = 900 WHERE id = 1
  事件溯源:
    Event1: AccountCreated { id: 1, balance: 1000 }
    Event2: MoneyWithdrawn { id: 1, amount: 100 }
    → 当前余额 = 1000 - 100 = 900

  ┌─────────────────────────────────────┐
  │          Event Store                 │
  │  [Created] → [Deposited] → [Withdrawn] → [Deposited]│
  │    $0          +$1000       -$100         +$500      │
  │                                                      │
  │  当前余额 = 0 + 1000 - 100 + 500 = $1400            │
  └─────────────────────────────────────┘

CQRS + Event Sourcing 优势:
  完整审计日志 / 时间旅行 (回放到任意时刻)
  读写独立扩展 / 事件驱动天然解耦
  
代价:
  复杂度高 / 最终一致性 / 事件版本管理
```

## 6. 舱壁隔离与背压模式？

**回答：**

```
舱壁隔离 (Bulkhead Pattern):
  源自船舶的水密舱壁设计
  将系统资源隔离, 防止一个组件故障影响其他

  无隔离:
    所有请求共享线程池 → 一个慢服务耗尽线程 → 全部阻塞
    
  舱壁隔离:
    ┌─────────────┬─────────────┬─────────────┐
    │  用户服务     │  订单服务    │  支付服务     │
    │  线程池 50   │  线程池 30   │  线程池 20    │
    │  ═══════    │  ═══════    │  ═══════     │
    │  互相隔离     │  互相隔离    │  互相隔离     │
    └─────────────┴─────────────┴─────────────┘
    订单服务慢 → 只影响订单线程池 → 用户/支付不受影响

隔离维度:
  线程池隔离: 每个依赖分配独立线程池
  信号量隔离: 每个依赖限制并发数 (轻量)
  进程隔离:   独立进程/容器
  集群隔离:   不同服务部署在不同节点

背压模式 (Backpressure):
  下游处理不过来时, 向上游反馈压力

  无背压:
    生产者 100/s →→→→→→→→ 消费者 50/s → OOM / 崩溃

  有背压:
    生产者 100/s → [缓冲区满] → 通知生产者减速
                                → 或拒绝新请求
                                → 或丢弃低优先级

  背压实现方式:
    1. 消息队列: 消费者按能力拉取 (Pull 模式)
    2. 反应式流: Reactive Streams (request(n))
    3. 限流: 在入口处控制流量
    4. 动态扩容: 自动扩展消费者实例
```

## 7. 绞杀者模式（单体微服务化）？

**回答：**

```
绞杀者模式 (Strangler Fig Pattern):
  渐进式地将单体应用替换为微服务
  名称来源: 绞杀榕 → 逐渐包裹宿主树直到完全替代

  ┌──────────────────────────────────────────┐
  │ 阶段1: 单体 + 反向代理                    │
  │                                           │
  │  Client → [Proxy] → 单体应用              │
  │                                           │
  │ 阶段2: 逐步抽取微服务                      │
  │                                           │
  │  Client → [Proxy] ──→ 单体 (大部分功能)    │
  │                    └→ 用户微服务 (新)       │
  │                                           │
  │ 阶段3: 持续迁移                            │
  │                                           │
  │  Client → [Proxy] ──→ 单体 (少量功能)      │
  │                    ├→ 用户微服务            │
  │                    ├→ 订单微服务            │
  │                    └→ 商品微服务            │
  │                                           │
  │ 阶段4: 完全替代                            │
  │                                           │
  │  Client → [API GW] ──→ 用户微服务          │
  │                     ├→ 订单微服务           │
  │                     ├→ 商品微服务           │
  │                     └→ 支付微服务           │
  │                                           │
  │  单体下线 ✓                                │
  └──────────────────────────────────────────┘

实施原则:
  ┌──────────────────┬──────────────────────────┐
  │ 原则              │ 说明                      │
  ├──────────────────┼──────────────────────────┤
  │ 由外到内          │ 先抽取边缘/独立的功能     │
  │ 数据先分          │ 先拆分数据库, 再拆服务    │
  │ 增量替换          │ 小步快跑, 不要大爆炸      │
  │ 功能对等          │ 新服务功能必须覆盖旧逻辑  │
  │ 可回退            │ 代理层保留切回单体的能力  │
  │ 双写过渡          │ 迁移期间新旧系统双写      │
  └──────────────────┴──────────────────────────┘

优先抽取的服务:
  变更频繁的模块 → 独立发布
  性能瓶颈模块   → 独立扩展
  团队独立负责的 → 减少协调
  技术栈不同的   → 适合独立服务
```

## 8. 幂等性设计？

**回答：**

```
幂等性: 同一操作执行多次, 结果与执行一次相同

为什么微服务需要幂等:
  网络不可靠 → 自动重试 → 可能重复执行
  消息队列   → 至少一次投递 → 可能重复消费
  用户操作   → 重复提交 → 重复下单

天然幂等:   GET, DELETE (按 ID), PUT (全量更新)
非幂等操作: POST (创建), UPDATE x = x + 1 (增量)

幂等实现方案:
  ┌────────────────┬──────────────────────────────┐
  │ 方案            │ 原理                          │
  ├────────────────┼──────────────────────────────┤
  │ 唯一 ID        │ 每个请求携带唯一 ID, 重复忽略  │
  │ 数据库唯一约束  │ UNIQUE KEY 防重复插入          │
  │ 乐观锁         │ UPDATE ... WHERE version = N  │
  │ Token 机制     │ 先获取 Token, 用后失效         │
  │ 状态机         │ 状态只能单向流转, 重复无效      │
  │ Redis 去重     │ SETNX 判断是否已处理           │
  └────────────────┴──────────────────────────────┘
```

```python
# 方案1: Redis 去重 (最常用)
import redis
import hashlib

r = redis.Redis()

def process_order(request_id, order_data):
    # 用请求 ID 做幂等键
    key = f"idempotent:{request_id}"
    
    # SETNX: 不存在才设置, 返回是否成功
    if not r.set(key, "processing", nx=True, ex=3600):
        # 已存在, 说明重复请求
        return {"status": "duplicate", "message": "请求已处理"}
    
    try:
        result = create_order(order_data)
        r.set(key, "done", ex=86400)
        return result
    except Exception:
        r.delete(key)  # 失败时删除, 允许重试
        raise
```

```sql
-- 方案2: 数据库唯一约束
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    request_id VARCHAR(64) UNIQUE,  -- 幂等键
    user_id BIGINT,
    amount DECIMAL(10,2),
    created_at TIMESTAMP
);

-- 重复插入会报唯一约束冲突

VALUES ('req-abc-123', 1001, 99.00)
ON DUPLICATE KEY UPDATE id = id;  -- 冲突时不做操作
```

## 9. 分布式 ID 生成？

**回答：**

```
微服务场景: 每个服务独立数据库, 需要全局唯一 ID

方案对比:
  ┌──────────────┬──────────┬──────────┬────────────────┐
  │ 方案          │ 有序性    │ 性能     │ 说明            │
  ├──────────────┼──────────┼──────────┼────────────────┤
  │ UUID          │ 无序     │ 极高     │ 太长, 索引差    │
  │ 数据库自增     │ 有序     │ 低      │ 单点瓶颈        │
  │ 号段模式      │ 有序     │ 高      │ 批量获取, 减少DB │
  │ Redis INCR   │ 有序     │ 高      │ 依赖 Redis      │
  │ Snowflake    │ 趋势递增  │ 极高    │ 时钟回拨问题    │
  │ Leaf (美团)   │ 有序     │ 极高    │ 号段+Snowflake  │
  └──────────────┴──────────┴──────────┴────────────────┘

Snowflake 算法 (最常用):
  64 位 ID 结构:
  ┌───────┬───────────────────┬──────────┬──────────┐
  │ 1 bit │    41 bit         │  10 bit  │  12 bit  │
  │ 符号位│  时间戳 (毫秒)      │ 机器 ID  │  序列号  │
  └───────┴───────────────────┴──────────┴──────────┘
  
  时间戳: 支持约 69 年
  机器ID: 1024 台机器
  序列号: 每毫秒 4096 个 ID
  → 单机每秒可生成 400 万+ 个 ID

  优点: 趋势递增, 高性能, 不依赖外部
  缺点: 依赖时钟 (时钟回拨会重复)
```

```python
# Snowflake 简易实现
import time
import threading

class SnowflakeIDGenerator:
    def __init__(self, worker_id, datacenter_id):
        self.worker_id = worker_id & 0x1F       # 5 bit
        self.datacenter_id = datacenter_id & 0x1F # 5 bit
        self.sequence = 0
        self.last_timestamp = -1
        self.lock = threading.Lock()
        self.epoch = 1609459200000  # 2021-01-01

    def _current_millis(self):
        return int(time.time() * 1000)

    def next_id(self):
        with self.lock:
            timestamp = self._current_millis()
            if timestamp == self.last_timestamp:
                self.sequence = (self.sequence + 1) & 0xFFF
                if self.sequence == 0:
                    while timestamp <= self.last_timestamp:
                        timestamp = self._current_millis()
            else:
                self.sequence = 0
            self.last_timestamp = timestamp
            return (
                ((timestamp - self.epoch) << 22)
                | (self.datacenter_id << 17)
                | (self.worker_id << 12)
                | self.sequence
            )

# 使用
gen = SnowflakeIDGenerator(worker_id=1, datacenter_id=1)
order_id = gen.next_id()  # 如: 6842628722688000001
```

## 10. 分布式系统设计面试速答？

**回答：**

```
Q: CAP 定理是什么?
A: 分布式系统最多满足 C/A/P 中的两项
   网络分区必然存在, 所以只能选 CP 或 AP
   微服务通常选 AP + 最终一致性

Q: BASE 理论?
A: 基本可用 + 软状态 + 最终一致性
   是 CAP 中 AP 方案的落地指导
   通过牺牲强一致性换取高可用

Q: CQRS 是什么?
A: 读写分离的架构模式
   写端处理业务逻辑, 读端优化查询
   配合 Event Sourcing 实现事件驱动

Q: 事件溯源是什么?
A: 不存当前状态, 存所有变更事件
   当前状态 = 重放事件
   天然审计日志, 可时间旅行

Q: 舱壁隔离怎么做?
A: 线程池隔离: 每个依赖独立线程池
   信号量隔离: 限制并发数 (轻量)
   防止一个故障服务拖垮整个系统

Q: 绞杀者模式?
A: 渐进式单体→微服务的迁移策略
   代理层路由, 增量替换, 可回退
   先抽变更频繁/性能瓶颈的模块

Q: 怎么保证幂等性?
A: 唯一请求 ID + Redis SETNX 去重 (最常用)
   数据库唯一约束, 乐观锁, 状态机
   非幂等操作 (POST/增量更新) 必须做

Q: 分布式 ID 怎么生成?

   号段模式: 批量获取 ID 段, 减少 DB 压力
   生产推荐: 美团 Leaf (号段+Snowflake 双模式)

Q: 微服务数据一致性怎么保证?
A: 强一致: 2PC/TCC (核心交易)
   最终一致: Saga/消息队列 (大部分场景)
   定时对账: 兜底方案
```
