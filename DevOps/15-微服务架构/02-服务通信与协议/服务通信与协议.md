# 服务通信与协议

---

## 1. 微服务通信方式总览？

**回答：**

```
通信分类:

  同步通信 (请求-响应):
    客户端发请求, 等待响应
    ├── REST (HTTP/JSON)
    ├── gRPC (HTTP/2 + Protobuf)
    └── GraphQL

  异步通信 (事件驱动):
    客户端发消息, 不等待响应
    ├── 消息队列 (RabbitMQ / SQS)
    ├── 事件流 (Kafka)
    └── 发布订阅

  ┌──────────────────┬───────────────┬───────────────┐
  │ 维度              │ 同步           │ 异步           │
  ├──────────────────┼───────────────┼───────────────┤
  │ 耦合度            │ 高 (强依赖)    │ 低 (松耦合)    │
  │ 响应速度          │ 实时           │ 延迟           │
  │ 可用性            │ 依赖下游       │ 高 (消息持久)  │
  │ 复杂度            │ 低             │ 高             │
  │ 一致性            │ 强一致         │ 最终一致       │
  │ 适用场景          │ 查询/CRUD      │ 解耦/削峰/通知 │
  └──────────────────┴───────────────┴───────────────┘

通信模式:
  1. 请求-响应:    A ──请求──→ B ──响应──→ A
  2. 发布-订阅:    A ──事件──→ [Broker] ──→ B, C, D
  3. 点对点:       A ──消息──→ [Queue] ──→ B (一个消费者)
  4. 请求-异步回复: A ──请求──→ B, B ──回调──→ A
```

## 2. REST API 设计最佳实践？

**回答：**

```
RESTful 核心原则:
  资源导向: URL 表示资源, HTTP 方法表示操作
  无状态: 每个请求包含所有必要信息
  统一接口: 标准 HTTP 方法

HTTP 方法与 CRUD:
  ┌──────────┬──────────────────┬────────────────┐
  │ 方法      │ 操作              │ 示例            │
  ├──────────┼──────────────────┼────────────────┤
  │ GET      │ 查询              │ GET /users/123 │
  │ POST     │ 创建              │ POST /users    │
  │ PUT      │ 全量更新          │ PUT /users/123 │
  │ PATCH    │ 部分更新          │ PATCH /users/123│
  │ DELETE   │ 删除              │ DELETE /users/123│
  └──────────┴──────────────────┴────────────────┘

URL 设计:
  ✅ GET    /api/v1/users              # 获取用户列表
  ✅ GET    /api/v1/users/123          # 获取单个用户
  ✅ GET    /api/v1/users/123/orders   # 获取用户的订单
  ✅ POST   /api/v1/users              # 创建用户
  ✅ PATCH  /api/v1/users/123          # 更新用户
  
  ❌ GET    /api/getUsers              # 动词不用
  ❌ POST   /api/deleteUser            # 用 DELETE 方法
  ❌ GET    /api/user                  # 用复数

响应格式:
  {
    "code": 200,
    "data": { "id": 123, "name": "test" },
    "message": "success",
    "timestamp": "2026-03-11T10:00:00Z"
  }

状态码:
  200 OK          - 成功
  201 Created     - 创建成功
  204 No Content  - 删除成功
  400 Bad Request - 参数错误
  401 Unauthorized- 未认证
  403 Forbidden   - 无权限
  404 Not Found   - 资源不存在
  409 Conflict    - 冲突
  429 Too Many    - 限流
  500 Internal    - 服务器错误

版本管理:
  URL 路径: /api/v1/users (推荐, 直观)
  Header:   Accept: application/vnd.api.v1+json
  Query:    /api/users?version=1
```

## 3. gRPC 详解？

**回答：**

```
gRPC: Google 开源的高性能 RPC 框架

特点:
  HTTP/2 传输 (多路复用, 头部压缩)
  Protobuf 序列化 (二进制, 体积小)
  强类型契约 (.proto 文件定义接口)
  代码自动生成 (多语言支持)
  双向流式通信

REST vs gRPC:
  ┌──────────────┬────────────────┬─────────────────┐
  │ 维度          │ REST           │ gRPC             │
  ├──────────────┼────────────────┼─────────────────┤
  │ 协议          │ HTTP/1.1       │ HTTP/2           │
  │ 数据格式      │ JSON (文本)    │ Protobuf (二进制)│
  │ 性能          │ 较低           │ 高 (7-10x)       │
  │ 流式          │ 不支持         │ 支持 (4种模式)   │
  │ 代码生成      │ 手动/OpenAPI   │ 自动生成         │
  │ 浏览器        │ 原生支持       │ 需 grpc-web      │
  │ 可读性        │ 高 (JSON可读)  │ 低 (二进制)      │
  │ 调试          │ curl/Postman   │ grpcurl/BloomRPC │
  │ 适用          │ 外部 API       │ 内部服务通信     │
  └──────────────┴────────────────┴─────────────────┘

gRPC 四种通信模式:
  1. Unary:           Client ──→ Server (一对一)
  2. Server Streaming: Client ──→ Server ══→ (服务端流)
  3. Client Streaming: Client ══→ Server ──→ (客户端流)
  4. Bidirectional:   Client ══⇄══ Server  (双向流)
```

```protobuf
// user.proto - 接口定义
syntax = "proto3";
package user;

service UserService {
  // Unary
  rpc GetUser (GetUserRequest) returns (UserResponse);
  // Server streaming
  rpc ListUsers (ListUsersRequest) returns (stream UserResponse);
  // Client streaming
  rpc BatchCreateUsers (stream CreateUserRequest) returns (BatchResult);
  // Bidirectional streaming
  rpc Chat (stream ChatMessage) returns (stream ChatMessage);
}

message GetUserRequest {
  int64 id = 1;
}

message UserResponse {
  int64 id = 1;
  string name = 2;
  string email = 3;
  int64 created_at = 4;
}
```

```
使用场景:
  ✅ 微服务间内部通信 (高性能)
  ✅ 实时数据推送 (Server Streaming)
  ✅ 多语言项目 (自动代码生成)
  ✅ IoT 设备通信 (低带宽)
  
  ❌ 面向浏览器的 Web API (用 REST)
  ❌ 需要人可读的接口 (用 REST + JSON)
```

## 4. GraphQL 基础？

**回答：**

```
GraphQL: 客户端精准查询所需数据

REST 的问题:
  Over-fetching:  返回了不需要的字段
  Under-fetching: 需要多次请求凑齐数据

GraphQL 解决:
  一次请求, 精确获取所需字段

对比:
  REST:    GET /users/1          → 所有字段
           GET /users/1/orders   → 第二次请求
  
  GraphQL: query {
             user(id: 1) {
               name
               email
               orders {
                 id
                 total
               }
             }
           }
           → 一次请求, 精确字段

适用场景:
  ✅ BFF (Backend for Frontend)
  ✅ 移动端 (带宽敏感, 精简数据)
  ✅ 多客户端不同需求 (Web/App/小程序)
  ✅ 聚合多个微服务数据
  
  ❌ 简单 CRUD API
  ❌ 文件上传
  ❌ 实时性要求极高 (gRPC 更好)

GraphQL vs REST vs gRPC:
  REST:    外部公开 API, CRUD 简单
  gRPC:    内部微服务间高性能通信
  GraphQL: BFF 层, 聚合多源数据
```

## 5. 消息队列选型？

**回答：**

```
消息队列对比:
  ┌──────────────┬────────────┬────────────┬──────────┐
  │ 特性          │ Kafka      │ RabbitMQ   │ SQS/SNS  │
  ├──────────────┼────────────┼────────────┼──────────┤
  │ 模型          │ 日志流     │ 消息队列   │ 托管队列 │
  │ 吞吐量        │ 百万/秒    │ 万/秒      │ 按需     │
  │ 延迟          │ ms级       │ μs级       │ ms级     │
  │ 消息保留      │ 持久化     │ 消费即删   │ 14天     │
  │ 消费模型      │ Pull       │ Push/Pull  │ Pull     │
  │ 消息重放      │ 支持       │ 不支持     │ 不支持   │
  │ 有序性        │ 分区有序   │ 队列有序   │ FIFO可选 │
  │ 运维          │ 复杂       │ 中等       │ 免运维   │
  │ 适用          │ 大数据/日志│ 业务消息   │ AWS 云   │
  └──────────────┴────────────┴────────────┴──────────┘

选型建议:
  日志/事件流/大数据   → Kafka
  业务异步消息/可靠投递 → RabbitMQ
  AWS 环境/简单队列    → SQS
  发布订阅通知         → SNS + SQS
  云原生简单           → Redis Stream (轻量)

消息可靠性保证:
  生产端: 确认机制 (ack)
  Broker: 持久化存储
  消费端: 至少一次消费 + 幂等处理

  消息语义:
    At-most-once:  最多一次 (可能丢)
    At-least-once: 至少一次 (可能重复, 需幂等)
    Exactly-once:  精确一次 (Kafka 事务支持)
```

## 6. 事件驱动架构 (EDA)？

**回答：**

```
事件驱动架构: 服务通过发布/订阅事件通信

  ┌──────────┐   事件    ┌──────────┐   事件    ┌──────────┐
  │ 订单服务  │──创建──→ │ Event Bus│──通知──→ │ 库存服务  │
  │          │          │ (Kafka)  │          │ 通知服务  │
  └──────────┘          └──────────┘          │ 积分服务  │
                                              └──────────┘

  订单服务只管发 "订单已创建" 事件
  → 不需要知道谁消费
  → 新系统想订阅? 自己加消费者就行

事件类型:
  领域事件 (Domain Event):
    OrderCreated, PaymentCompleted, UserRegistered
    业务语义, 不可变, 包含发生时的快照

  集成事件 (Integration Event):
    跨服务边界的事件
    通过消息中间件传递

  命令 (Command):
    CreateOrder, ProcessPayment
    有明确目标, 期望被执行

事件 vs 命令:
  ┌──────────────┬──────────────┬──────────────┐
  │ 维度          │ 事件          │ 命令          │
  ├──────────────┼──────────────┼──────────────┤
  │ 语义          │ 已发生的事实  │ 要求执行      │
  │ 接收者        │ 0-N 个       │ 1 个          │
  │ 耦合度        │ 松耦合       │ 紧耦合        │
  │ 发送方        │ 不关心谁消费 │ 知道目标      │
  │ 命名          │ 过去式       │ 祈使句        │
  └──────────────┴──────────────┴──────────────┘

事件格式 (CloudEvents 标准):
  {
    "specversion": "1.0",
    "type": "com.example.order.created",
    "source": "/order-service",
    "id": "uuid-123",
    "time": "2026-03-11T10:00:00Z",
    "data": {
      "orderId": "ORD-001",
      "userId": "USR-123",
      "total": 99.99
    }
  }
```

## 7. 幂等性设计？

**回答：**

```
幂等性: 同一操作执行多次, 效果与执行一次相同

为什么需要:
  网络重试, 消息重复消费, 用户重复提交
  微服务间通信不可靠, 必须做幂等

天然幂等的操作:
  GET    /users/123          → 查询幂等
  PUT    /users/123 {全量}   → 覆盖幂等
  DELETE /users/123          → 删除幂等

非幂等需要处理:
  POST   /orders             → 可能重复创建
  PATCH  /account/balance +10 → 可能重复加钱

实现方案:
  ┌──────────────────┬──────────────────────────────┐
  │ 方案              │ 说明                          │
  ├──────────────────┼──────────────────────────────┤
  │ 唯一请求 ID       │ 客户端生成 requestId,         │
  │                   │ 服务端去重 (Redis/DB 记录)     │
  │ 数据库唯一约束    │ UNIQUE KEY 防止重复插入        │
  │ 乐观锁            │ UPDATE ... WHERE version = N  │
  │ Token 机制        │ 先获取 Token, 使用时校验       │
  │ 状态机            │ 订单状态只能单向流转           │
  └──────────────────┴──────────────────────────────┘

唯一请求 ID 示例:
  # 客户端
  POST /api/orders
  X-Request-Id: uuid-abc-123
  
  # 服务端
  1. 检查 Redis: GET idempotent:uuid-abc-123
  2. 如果存在 → 返回之前的结果 (幂等)
  3. 如果不存在:
     a. SET idempotent:uuid-abc-123 "processing" EX 3600
     b. 执行业务逻辑
     c. SET idempotent:uuid-abc-123 {result} EX 3600
     d. 返回结果

消息消费幂等:
  Kafka: 用 (topic + partition + offset) 去重
  通用: 消息有唯一 messageId, 消费前检查是否已处理
```

## 8. API 版本管理？

**回答：**

```
版本管理策略:
  ┌──────────────────┬──────────────────────────────┐
  │ 方式              │ 示例                          │
  ├──────────────────┼──────────────────────────────┤
  │ URL 路径 (推荐)   │ /api/v1/users                │
  │ Header           │ Accept: application/vnd.v2   │
  │ Query 参数        │ /api/users?version=2         │
  └──────────────────┴──────────────────────────────┘

向后兼容原则:
  ✅ 新增字段 (不影响老客户端)
  ✅ 新增可选参数
  ✅ 新增端点
  ❌ 删除字段 (破坏性变更)
  ❌ 修改字段类型
  ❌ 修改必填参数

API 生命周期:
  v1 (Current) → v2 (新版本)
    └── v1 标记 Deprecated
    └── 迁移期 (6个月)
    └── v1 下线 (通知客户端)

API 文档:
  OpenAPI/Swagger: REST API 标准文档
  Protobuf:        gRPC 接口定义即文档
  AsyncAPI:        异步 API/事件文档
```

## 9. 服务间数据传输安全？

**回答：**

```
传输安全:
  TLS:     HTTPS 加密传输
  mTLS:    双向 TLS (Service Mesh)
  JWT:     服务间身份验证
  API Key: 简单认证

mTLS 在微服务中:
  每个服务都有证书
  互相验证身份
  Istio 自动管理 mTLS

  服务 A ←──mTLS──→ 服务 B
  双方互相验证证书, 加密通信

认证传播:
  外部用户 → API Gateway (验证JWT)
    → 内部服务 (携带用户上下文)
    → 下一个服务 (透传用户信息)

  方案:
    JWT 透传: Gateway 验证后, 把 JWT 传给后端
    请求头传播: X-User-Id / X-Tenant-Id
    服务间 Token: 服务自身的 ServiceAccount Token
```

## 10. 服务通信面试速答？

**回答：**

```
Q: 微服务间怎么通信?
A: 同步: REST (简单CRUD) / gRPC (高性能内部)
   异步: 消息队列 (解耦/削峰/通知)
   选择取决于场景: 实时性 vs 解耦性

Q: REST vs gRPC 怎么选?
A: 外部 API → REST (通用, 浏览器友好)
   内部服务间 → gRPC (高性能, 强类型, 10x 快)
   BFF 聚合层 → GraphQL

Q: 消息队列怎么选?
A: 大数据/日志/事件流 → Kafka
   业务消息/可靠投递 → RabbitMQ
   AWS 简单 → SQS/SNS
   轻量级 → Redis Stream

Q: 如何保证消息不丢?
A: 生产端: 确认机制 (ack)
   Broker: 持久化 + 副本
   消费端: 手动 ack + 幂等处理

Q: 如何保证幂等?
A: 唯一请求 ID + Redis/DB 去重
   数据库唯一约束
   乐观锁版本号
   状态机约束

Q: 服务间安全怎么做?
A: mTLS 双向认证 (Istio 自动)
   JWT 透传用户上下文
   服务间最小权限
```
