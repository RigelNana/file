## 2. 灰度发布与金丝雀发布实践？

**回答：**

```
发布策略对比:
  ┌──────────────┬──────────────────────────────────┐
  │ 策略          │ 说明                              │
  ├──────────────┼──────────────────────────────────┤
  │ 滚动发布      │ 逐步替换旧实例, K8s 默认策略      │
  │ 蓝绿发布      │ 两套环境, 一键切换                │
  │ 金丝雀发布    │ 小比例流量验证新版本              │
  │ A/B 测试      │ 按用户特征分流, 对比效果          │
  │ 灰度发布      │ 按规则逐步放量 (金丝雀的泛化)    │
  └──────────────┴──────────────────────────────────┘

金丝雀发布流程:
  v1 (100%) → v1(95%) + v2(5%) → ... → v2 (100%)
  
  关键: 自动化指标验证
    部署 v2 → 切 5% 流量 → 监控指标 (错误率/延迟)
    → 指标正常 → 扩大流量 → 全量
    → 指标异常 → 自动回滚
```

```yaml
# Argo Rollouts: 自动化金丝雀发布
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: order-service
spec:
  replicas: 10
  strategy:
    canary:
      steps:
      - setWeight: 5
      - pause: {duration: 2m}
      - analysis:
          templates:
          - templateName: success-rate
      - setWeight: 20
      - pause: {duration: 5m}
      - analysis:
          templates:
          - templateName: success-rate
      - setWeight: 50
      - pause: {duration: 5m}
      - setWeight: 100
      canaryService: order-service-canary
      stableService: order-service-stable
      trafficRouting:
        istio:
          virtualServices:
          - name: order-service-vsvc
            routes:
            - primary
  template:
    metadata:
      labels:
        app: order-service
    spec:
      containers:
      - name: order-service
        image: order-service:v2
        ports:
        - containerPort: 8080

---
# 分析模板: 自动验证指标
apiVersion: argoproj.io/v1alpha1
kind: AnalysisTemplate
metadata:
  name: success-rate
spec:
  metrics:
  - name: success-rate
    interval: 60s
    count: 5
    successCondition: result[0] >= 0.99
    failureLimit: 2
    provider:
      prometheus:
        address: http://prometheus:9090
        query: |
          sum(rate(http_requests_total{status!~"5..",app="order-service",
            rollouts_pod_template_hash="{{args.canary-hash}}"}[2m]))
          / sum(rate(http_requests_total{app="order-service",
            rollouts_pod_template_hash="{{args.canary-hash}}"}[2m]))
```

## 3. 微服务容量规划与弹性伸缩？

**回答：**

```
容量规划:
  1. 基准测试: 单实例压测 → 确定单实例承载量
  2. 计算实例数: 预期 QPS / 单实例 QPS × 冗余系数
  3. 资源估算: 实例数 × 单实例资源 (CPU/Memory)
  4. 冗余设计: 至少 N+1, 核心服务 N+2

  例: 预期 10000 QPS, 单实例 500 QPS
      实例数 = 10000/500 × 1.5(冗余) = 30 实例
      资源 = 30 × (500m CPU + 512Mi Memory)

弹性伸缩层次:
  ┌──────────────┬──────────────────────────────┐
  │ 层次          │ 工具                          │
  ├──────────────┼──────────────────────────────┤
  │ Pod 水平伸缩  │ HPA (CPU/Memory/自定义指标)  │
  │ Pod 垂直伸缩  │ VPA (自动调整资源请求)        │
  │ 节点伸缩      │ Cluster Autoscaler           │
  │ 事件驱动伸缩  │ KEDA (消息队列/自定义事件)    │
  └──────────────┴──────────────────────────────┘
```

```yaml
# HPA: 基于自定义指标伸缩 (QPS)
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: order-service-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: order-service
  minReplicas: 3
  maxReplicas: 50
  metrics:
  - type: Pods
    pods:
      metric:
        name: http_requests_per_second
      target:
        type: AverageValue
        averageValue: "300"     # 每 Pod 300 QPS 时扩容
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70  # CPU 70% 时扩容
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 30
      policies:
      - type: Percent
        value: 50               # 每次最多扩 50%
        periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300  # 5分钟稳定期
      policies:
      - type: Percent
        value: 20               # 每次最多缩 20%
        periodSeconds: 120

---
# KEDA: 基于消息队列积压伸缩
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: order-consumer
spec:
  scaleTargetRef:
    name: order-consumer
  minReplicaCount: 1
  maxReplicaCount: 20
  triggers:
  - type: kafka
    metadata:
      bootstrapServers: kafka:9092
      consumerGroup: order-group
      topic: orders
      lagThreshold: "100"       # 积压 > 100 则扩容
```

## 4. 微服务配置中心？

**回答：**

```
为什么需要配置中心:
  微服务数量多 → 配置分散 → 修改需重新部署
  需要: 集中管理 / 动态更新 / 环境隔离 / 灰度配置

配置中心对比:
  ┌──────────────┬──────────────┬──────────────┬──────────────┐
  │              │ K8s ConfigMap│ Nacos        │ Apollo       │
  ├──────────────┼──────────────┼──────────────┼──────────────┤
  │ 动态更新      │ 需重启/Watch │ 实时推送      │ 实时推送      │
  │ 版本管理      │ Git          │ 内置         │ 内置          │
  │ 灰度发布      │ 不支持       │ 支持         │ 支持          │
  │ 权限管理      │ RBAC         │ 支持         │ 支持          │
  │ 多环境        │ Namespace    │ Namespace    │ Env + Cluster│
  │ 存储          │ etcd         │ MySQL        │ MySQL        │
  │ 适用场景      │ K8s 原生     │ Java/通用     │ Java 生态     │
  │ 复杂度        │ 简单         │ 中等         │ 中等          │
  └──────────────┴──────────────┴──────────────┴──────────────┘

配置管理最佳实践:
  1. 配置分层: 公共配置 → 应用配置 → 环境配置
  2. 敏感信息加密存储 (K8s Secret / Vault)
  3. 配置变更审计 (谁在什么时候改了什么)
  4. 配置灰度: 先灰度一批实例, 验证后全量
  5. 配置回滚: 保留历史版本, 可快速回退
```

```yaml

      new-checkout: false
      recommendation-v2: true
```

## 5. 微服务安全体系？

**回答：**

```
微服务安全层次:
  ┌──────────────────────────────────────────┐
  │ 层次1: 边界安全 (南北向)                   │
  │   WAF / DDoS 防护 / API 网关认证          │
  ├──────────────────────────────────────────┤
  │ 层次2: 服务间安全 (东西向)                  │
  │   mTLS / 服务身份认证 / 网络策略           │
  ├──────────────────────────────────────────┤
  │ 层次3: 应用安全                            │
  │   输入验证 / SQL注入防护 / XSS 防护        │
  ├──────────────────────────────────────────┤
  │ 层次4: 数据安全                            │
  │   数据加密 / 密钥管理 / 数据脱敏           │
  ├──────────────────────────────────────────┤
  │ 层次5: 运维安全                            │
  │   RBAC / 审计日志 / 合规检查              │
  └──────────────────────────────────────────┘

零信任网络 (Zero Trust):
  传统: 内网可信, 防火墙保护边界
  零信任: 不信任任何请求, 始终验证

  原则:
    永不信任, 始终验证 (Never Trust, Always Verify)
    最小权限原则 (Least Privilege)
    假设已被攻破 (Assume Breach)

  微服务落地:
    所有服务间通信 mTLS (Istio 自动)
    每个请求验证身份和权限
    网络策略限制 Pod 间通信
    密钥统一管理 (Vault)
```


## 6. 数据库拆分与数据一致性？

**回答：**

```
微服务数据库拆分原则:
  每个服务拥有独立数据库 (Database per Service)
  服务间只能通过 API 访问对方数据, 不能直连对方 DB

拆分方式:
  ┌──────────────────────────────────────────┐
  │ 阶段1: 共享数据库 (起步阶段)              │
  │   所有服务 → 同一个 DB (不同 Schema)      │
  │   简单但耦合                              │
  │                                           │
  │ 阶段2: 逻辑拆分                           │
  │   同一 DB, 每个服务用独立 Schema           │
  │   禁止跨 Schema 查询                      │
  │                                           │
  │ 阶段3: 物理拆分                           │
  │   每个服务独立 DB 实例                     │
  │   完全解耦                                │
  └──────────────────────────────────────────┘

跨服务数据查询方案:
  ┌──────────────────┬──────────────────────────┐
  │ 方案              │ 说明                      │
  ├──────────────────┼──────────────────────────┤
  │ API 组合          │ 调用多个服务 API 聚合数据 │
  │ CQRS 读模型       │ 独立读库聚合多源数据      │
  │ 事件驱动同步       │ 事件 → 本地冗余存储      │
  │ 数据湖/数仓       │ 全量同步到分析库          │
  └──────────────────┴──────────────────────────┘

数据一致性保障:
  1. 服务内: 本地事务 (ACID)
  2. 跨服务: Saga + 补偿 (最终一致)
  3. 定时对账: 兜底检查 + 修复
  4. CDC (Change Data Capture):
     监听 DB Binlog → 同步数据变更

CDC 架构:
  MySQL (Binlog) → Debezium → Kafka → 下游服务/ES/数仓
```

## 7. 微服务测试策略？

**回答：**

```
测试金字塔 (微服务版):
                  ┌──────┐
                  │ E2E  │      少量端到端测试
                  │ 测试  │      (关键业务路径)
                 ┌┴──────┴┐
                 │ 契约测试 │    服务间接口契约
                 │ (Pact) │    (防止接口变更破坏)
                ┌┴────────┴┐
                │ 集成测试   │  服务 + DB/MQ 集成
                │           │  (Testcontainers)
               ┌┴──────────┴┐
               │   单元测试    │  核心业务逻辑
               │   (大量)     │  (快速, Mock 依赖)
               └─────────────┘

测试类型:
  ┌──────────────┬──────────────────────────────┐
  │ 测试类型      │ 说明                          │
  ├──────────────┼──────────────────────────────┤
  │ 单元测试      │ 测试函数/方法, Mock 外部依赖  │
  │ 集成测试      │ 测试服务 + DB/MQ 的集成       │
  │ 契约测试      │ 验证服务间 API 契约不破坏      │
  │ 组件测试      │ 测试单个微服务完整功能         │
  │ E2E 测试      │ 端到端测试关键业务流程         │
  │ 混沌测试      │ 注入故障验证系统弹性           │
  │ 性能测试      │ 压测验证容量和延迟             │
  └──────────────┴──────────────────────────────┘

契约测试 (Consumer-Driven Contract):
  消费者定义期望 → Pact Broker → 提供者验证

  消费者 (前端/调用方):
    "我调用 GET /api/users/1 期望返回 {id, name, email}"
    → 生成 Pact 契约文件

  提供者 (后端/被调用方):
    CI 中验证: 我的 API 满足所有消费者的契约吗?
    → 不满足则构建失败

混沌工程 (Chaos Engineering):
  工具: Chaos Mesh (K8s) / LitmusChaos
  注入: Pod 杀死 / 网络延迟 / CPU 压力 / 磁盘故障
  目的: 验证熔断/重试/降级是否生效
```

## 8. 单体迁移微服务策略？

**回答：**

```
迁移策略选择:
  ┌──────────────────┬──────────────────────────────┐
  │ 策略              │ 说明                          │
  ├──────────────────┼──────────────────────────────┤
  │ 绞杀者模式        │ 渐进替换, 风险低, 推荐        │
  │ 大爆炸重写        │ 全部重写, 风险极高, 不推荐    │
  │ 分支抽象          │ 代码层引入抽象层, 逐步切换    │
  │ 并行运行          │ 新旧同时运行, 比对结果        │
  └──────────────────┴──────────────────────────────┘

绞杀者模式实施步骤:
  1. 识别边界: 按 DDD 划分限界上下文
  2. 加入代理层: Nginx/API GW 统一入口
  3. 选择首批服务: 边缘/独立/变更频繁的模块
  4. 数据拆分: 先拆数据库, 双写过渡
  5. 流量迁移: 代理层切流到新服务
  6. 验证回退: 持续监控, 可回退到单体
  7. 重复: 逐个模块迁移

数据迁移策略:
  ┌──────────────────────────────────────────┐
  │ 阶段1: 双写                              │
  │   新服务写 → 新旧两个 DB 同时写           │
  │   读还是从旧 DB 读                        │
  │                                           │
  │ 阶段2: 双读比对                           │
  │   同时读新旧 DB, 比对结果                  │
  │   修复数据不一致                           │
  │                                           │
  │ 阶段3: 切读                              │
  │   读切到新 DB, 旧 DB 降级为备份           │
  │                                           │
  │ 阶段4: 停写                              │
  │   停止旧 DB 写入, 完全切到新 DB           │
  └──────────────────────────────────────────┘

常见陷阱:
  过早拆分 → 业务还没理清就拆, 导致反复调整
  拆太细   → 管理复杂度爆炸 (Nano Service)
  共享数据库→ 看似拆了, 实际还是耦合
  忽略数据  → 只拆服务不拆数据, 后患无穷
```

## 9. 微服务常见问题与反模式？

**回答：**

```
常见反模式:
  ┌──────────────────┬──────────────────────────────┐
  │ 反模式            │ 说明                          │
  ├──────────────────┼──────────────────────────────┤
  │ 分布式单体        │ 拆了但强耦合, 必须一起部署     │
  │ 数据库共享        │ 多个服务共享 DB, 伪微服务      │
  │ 过度拆分          │ 拆太细, 运维复杂度爆炸         │
  │ 同步调用链过长    │ A→B→C→D→E, 延迟叠加, 脆弱    │
  │ 缺乏熔断/超时    │ 一个慢服务拖垮整个系统         │
  │ 硬编码服务地址    │ 没用服务发现, 改地址要重新部署  │
  │ 忽略数据一致性    │ 没有分布式事务方案, 数据混乱   │
  │ 测试不足          │ 缺少契约测试, 接口变更导致故障  │
  │ 日志/链路缺失    │ 出了问题无法排查               │
  │ 每个服务不同框架  │ 技术栈过于分散, 维护成本高     │
  └──────────────────┴──────────────────────────────┘

调用链过长的优化:
  A → B → C → D → E   (同步, 延迟叠加)
  
  优化方案:
    异步化:    A → [MQ] → B → [MQ] → C
    合并服务:  B + C 合并 (如果关系紧密)
    并行调用:  A → (B, C, D 并行) → 聚合
    缓存:      A → Cache → 减少下游调用

什么时候不适合微服务:
  ┌──────────────────────────────────────────┐
  │ 不适合微服务的情况:                        │
  │   · 团队 < 5 人 (运维开销 > 收益)         │
  │   · 业务简单, 无需独立扩展                 │
  │   · 项目初期, 业务边界不清晰              │
  │   · 无 K8s/容器化基础设施                 │
  │   · 对延迟极度敏感 (网络开销)             │
  │                                           │
  │ 适合微服务的情况:                          │
  │   · 大型团队 (多团队并行开发)              │
  │   · 业务复杂, 领域清晰                    │
  │   · 需要独立扩展特定服务                   │
  │   · 需要多技术栈                           │
  │   · 有成熟的容器化/DevOps 基础             │
  └──────────────────────────────────────────┘
```

## 10. 微服务生产实践面试速答？

**回答：**

```
Q: 微服务拆分原则?
A: 按 DDD 限界上下文拆分, 单一职责
   高内聚低耦合, 每个服务独立数据库
   团队对齐 (康威定律)

Q: 怎么做灰度发布?
A: Argo Rollouts + Istio 流量分割
   自动分析指标 (错误率/延迟)
   异常自动回滚, 正常逐步放量

Q: 怎么做弹性伸缩?
A: HPA: 基于 CPU/内存/自定义指标 (QPS)
   KEDA: 基于消息队列积压等事件驱动
   Cluster Autoscaler: 节点级伸缩

Q: 配置中心怎么选?
A: K8s 原生: ConfigMap + Secret
   需要动态推送: Nacos / Apollo
   敏感信息: Vault / Sealed Secret

Q: 微服务安全怎么做?
A: 边界: WAF + API 网关认证
   服务间: mTLS (Istio 自动) + NetworkPolicy
   数据: 加密 + Vault 密钥管理
   理念: 零信任, 永不信任始终验证

Q: 数据库怎么拆分?
A: 逻辑拆分 (独立 Schema) → 物理拆分 (独立实例)
   跨服务查询: API 组合 / CQRS 读模型 / CDC
   一致性: Saga 最终一致 + 定时对账

Q: 单体怎么迁移到微服务?
A: 绞杀者模式: 反向代理 + 逐步抽取
   先拆边缘/独立模块, 数据双写过渡
   小步快跑, 可回退

Q: 微服务有哪些反模式?
A: 分布式单体 (拆了但耦合)
   同步长调用链 (改用异步/并行)
   过度拆分 (Nano Service)
   缺乏熔断/超时/链路追踪

Q: 什么时候不该用微服务?
A: 团队少于 5 人, 业务简单
   项目初期边界不清, 缺乏 DevOps 基础
   单体能解决的问题不要用微服务
```
