# 微服务架构八股文

---

## 一、微服务基础

### 1. 什么是微服务？与单体架构的区别？

**答：**

微服务是一种将应用拆分为一组小型、独立部署的服务的架构风格，每个服务围绕特定业务能力构建。

| 特性 | 单体架构 | 微服务架构 |
|------|---------|-----------|
| 部署 | 整体部署 | 独立部署 |
| 技术栈 | 统一 | 每个服务自由选择 |
| 扩展 | 整体扩展 | 按服务独立扩展 |
| 故障影响 | 单点故障影响全局 | 故障隔离 |
| 开发效率 | 初期快，后期慢 | 初期慢，后期快 |
| 复杂度 | 代码复杂 | 运维复杂 |
| 数据管理 | 共享数据库 | 每服务独立数据库 |

```
单体:
┌────────────────────────────────┐
│  用户模块 │ 订单模块 │ 支付模块  │ → 一个进程、一个部署包
└────────────────────────────────┘

微服务:
┌──────────┐  ┌──────────┐  ┌──────────┐
│ 用户服务  │  │ 订单服务  │  │ 支付服务  │  → 独立进程、独立部署
└──────────┘  └──────────┘  └──────────┘
     ↕              ↕              ↕
   User DB       Order DB      Payment DB   → 独立数据库
```

### 2. 微服务拆分的原则？

**答：**

| 原则 | 说明 |
|------|------|
| **单一职责** | 每个服务只负责一个业务域 |
| **高内聚低耦合** | 服务内部高度相关，服务间松散依赖 |
| **领域驱动 (DDD)** | 按限界上下文 (Bounded Context) 拆分 |
| **团队组织** | 符合康威定律，一个团队一个服务 |
| **数据隔离** | 每个服务有自己的数据存储 |

**常见拆分策略：**
- 按业务功能拆分：用户服务、商品服务、订单服务
- 按数据模型拆分：每个服务管理自己的数据
- 按变更频率拆分：频繁变更的独立出去

## 二、服务通信

### 3. 微服务间的通信方式？

**答：**

| 方式 | 类型 | 协议 | 适用场景 |
|------|------|------|---------|
| **REST** | 同步 | HTTP/JSON | 简单 CRUD |
| **gRPC** | 同步 | HTTP/2 + Protobuf | 高性能、低延迟 |
| **消息队列** | 异步 | AMQP/Kafka | 解耦、削峰 |
| **事件驱动** | 异步 | Event Bus | 最终一致性 |

```
同步通信:
  服务A ──HTTP/gRPC──→ 服务B      (等待响应)

异步通信:
  服务A → [消息队列] → 服务B      (不等待, 解耦)
  
事件驱动:
  服务A → 发布事件 → [Event Bus] → 服务B (订阅消费)
                                  → 服务C (订阅消费)
```

### 4. gRPC 的优势和使用场景？

**答：**

| 特性 | REST (JSON) | gRPC (Protobuf) |
|------|------------|-----------------|
| 序列化 | JSON（文本） | Protobuf（二进制） |
| 性能 | 较低 | 高（体积小10倍，速度快7倍） |
| 协议 | HTTP/1.1 | HTTP/2（多路复用） |
| 流式 | 不支持 | 支持（双向流） |
| 代码生成 | 手动 | 自动生成客户端和服务端 |
| 浏览器支持 | 原生支持 | 需要 grpc-web |

**适用场景：**
- 微服务间内部通信（高性能）
- 实时数据流（双向流式）
- 多语言项目（代码自动生成）

## 三、服务治理

### 5. 服务注册与发现是什么？

**答：**

```
服务注册:
  服务启动 → 将自身地址注册到注册中心
  
服务发现:
  客户端 → 从注册中心获取目标服务地址 → 调用

┌────────────────────────────┐
│      注册中心               │
│  (Consul / Eureka / etcd)  │
│                            │
│  user-svc: [10.0.1.1:8080, │
│             10.0.1.2:8080]  │
│  order-svc: [10.0.2.1:8080] │
└────────────────────────────┘
       ↑ 注册          ↓ 发现
   服务实例          客户端/网关
```

**常见注册中心：**

| 工具 | 一致性协议 | 特点 |
|------|----------|------|
| etcd | Raft | K8s 内置，强一致 |
| Consul | Raft | 服务发现 + KV + 健康检查 |
| ZooKeeper | ZAB | Java 生态常用 |
| Nacos | Raft + Distro | 阿里开源，国内流行 |

### 6. 什么是熔断、限流、降级？

**答：**

```
                   请求
                    │
            ┌───────┴───────┐
            │    限流器      │  ← 超过阈值直接拒绝
            └───────┬───────┘
                    │
            ┌───────┴───────┐
            │    熔断器      │  ← 下游故障时断开电路
            │   ┌─────┐     │
            │   │Closed│→ Open → Half-Open │
            │   └─────┘     │
            └───────┬───────┘
                    │
            ┌───────┴───────┐
            │    降级策略     │  ← 返回兜底数据/缓存
            └───────────────┘
```

| 机制 | 说明 | 示例 |
|------|------|------|
| **熔断** | 下游异常时切断调用，快速失败 | 5秒内失败率>50%则熔断 |
| **限流** | 控制请求速率，保护系统 | 每秒最多100个请求 |
| **降级** | 非核心功能降级处理 | 推荐服务不可用时返回默认列表 |

**常用工具：** Sentinel、Hystrix（已停更）、Resilience4j

### 7. 分布式事务有哪些解决方案？

**答：**

| 方案 | 原理 | 一致性 | 性能 | 适用场景 |
|------|------|--------|------|---------|
| **2PC** | 两阶段提交 | 强一致 | 低 | 数据库层面 |
| **TCC** | Try-Confirm-Cancel | 强一致 | 中 | 资金交易 |
| **Saga** | 补偿事务链 | 最终一致 | 高 | 长事务、跨服务 |
| **本地消息表** | 本地事务 + 消息 | 最终一致 | 高 | 通用 |
| **事务消息** | RocketMQ 事务消息 | 最终一致 | 高 | 异步场景 |

```
Saga 模式 (编排式):
  步骤1 → 步骤2 → 步骤3
                      ↓ (失败)
  补偿1 ← 补偿2 ← 补偿3

例: 下单流程
  创建订单 → 扣减库存 → 扣减余额
                          ↓ (余额不足)
  取消订单 ← 恢复库存 ← (补偿)
```

## 五、Service Mesh

### 9. 什么是 Service Mesh？

**答：**

Service Mesh（服务网格）是处理微服务间通信的基础设施层，将网络功能从业务代码中解耦。

```
传统模式 (SDK侵入):
┌────────────────────┐
│   业务代码          │
│   + 服务发现 SDK    │
│   + 负载均衡 SDK    │
│   + 熔断器 SDK      │
│   + 链路追踪 SDK    │
└────────────────────┘

Service Mesh (Sidecar 代理):
┌──────────────────┐
│   业务代码        │ ← 只关注业务
│   (纯业务逻辑)    │
├──────────────────┤
│   Sidecar Proxy  │ ← 处理所有网络功能
│   (Envoy)        │
└──────────────────┘
```

**数据面 + 控制面：**
```
控制面 (Istio / Linkerd Control Plane)
  ├── 配置下发 (路由规则、策略)
  ├── 证书管理 (mTLS)
  └── 遥测收集

数据面 (Envoy Sidecar)
  ├── 服务发现
  ├── 负载均衡
  ├── 熔断限流
  ├── 重试超时
  ├── mTLS 加密
  └── 可观测性 (指标、日志、链路)
```

### 10. Istio 的核心功能？

**答：**

| 功能 | 说明 |
|------|------|
| **流量管理** | 路由、负载均衡、故障注入、金丝雀发布 |
| **安全** | mTLS 自动加密、RBAC、认证策略 |
| **可观测性** | 指标 (Prometheus)、链路追踪 (Jaeger)、日志 |
| **策略** | 限流、访问控制 |

```yaml
# Istio VirtualService 流量路由
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: reviews
spec:
  hosts:
  - reviews
  http:
  - match:
    - headers:
        end-user:
          exact: test-user
    route:
    - destination:
        host: reviews
        subset: v2         # 测试用户走 v2
  - route:
    - destination:
        host: reviews
        subset: v1         # 其他用户走 v1
      weight: 90
    - destination:
        host: reviews
        subset: v2
      weight: 10           # 10% 流量走 v2（金丝雀）

---
# DestinationRule 定义子集
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: reviews
spec:
  host: reviews
  subsets:
  - name: v1
    labels:
      version: v1
  - name: v2
    labels:
      version: v2
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100
    outlierDetection:       # 熔断
      consecutive5xxErrors: 5
      interval: 5s
      baseEjectionTime: 30s
```

## 六、分布式系统设计

### 11. CAP 定理是什么？

**答：**

在分布式系统中，最多只能同时满足三项中的两项：

| 特性 | 含义 |
|------|------|
| **C (Consistency)** | 一致性：所有节点数据相同 |
| **A (Availability)** | 可用性：每个请求都能得到响应 |
| **P (Partition Tolerance)** | 分区容错：网络分区时系统仍可运行 |

```
        C (一致性)
       / \
     CP    CA (实际不存在，因为网络总会出问题)
    /        \
  P ────AP──── A
```

| 选择 | 说明 | 代表 |
|------|------|------|
| CP | 牺牲可用性 | ZooKeeper, etcd, HBase |
| AP | 牺牲一致性（最终一致） | Cassandra, DynamoDB, Eureka |

**实际应用中通常选 AP + 最终一致性。**

### 12. 常见的分布式设计模式？

**答：**

| 模式 | 说明 | 场景 |
|------|------|------|
| **Sidecar** | 辅助容器/进程 | 日志收集、代理、监控 |
| **Ambassador** | 代理外部服务的调用 | 连接数据库、缓存的代理 |
| **CQRS** | 读写分离 | 读写负载不均 |
| **Event Sourcing** | 事件溯源 | 审计、回放 |
| **Saga** | 分布式事务补偿 | 跨服务事务 |
| **Bulkhead** | 舱壁隔离 | 资源隔离，防止级联故障 |
| **Circuit Breaker** | 熔断器 | 下游故障保护 |
| **Strangler Fig** | 绞杀者模式 | 单体渐进拆分为微服务 |

## 七、可观测性

### 13. 微服务的可观测性如何实现？

**答：**

**三大支柱：**

| 支柱 | 工具 | 用途 |
|------|------|------|
| **指标 (Metrics)** | Prometheus + Grafana | 系统/业务指标监控 |
| **日志 (Logging)** | ELK / Loki | 日志聚合分析 |
| **链路追踪 (Tracing)** | Jaeger / Zipkin / OpenTelemetry | 请求全链路追踪 |

```
分布式链路追踪:

  Client → API Gateway → 用户服务 → 订单服务 → 支付服务
  
  TraceID: abc123
  ├── SpanID: 1 (Gateway, 100ms)
  │   ├── SpanID: 2 (用户服务, 20ms)
  │   └── SpanID: 3 (订单服务, 60ms)
  │       └── SpanID: 4 (支付服务, 30ms)
```

```yaml
# OpenTelemetry Collector 配置
apiVersion: v1
kind: ConfigMap
metadata:
  name: otel-collector-config
data:
  config.yaml: |
    receivers:
      otlp:
        protocols:
          grpc:
            endpoint: 0.0.0.0:4317
    exporters:
      jaeger:
        endpoint: jaeger-collector:14250
      prometheus:
        endpoint: 0.0.0.0:8889
    service:
      pipelines:
        traces:
          receivers: [otlp]
          exporters: [jaeger]
        metrics:
          receivers: [otlp]
          exporters: [prometheus]
```

### 14. 如何实现微服务的灰度发布/金丝雀发布？

**答：**

```
金丝雀发布流程:

  1. 部署新版本（少量实例）
     v1 × 9  +  v2 × 1

  2. 切入少量流量（如5%~10%）
     流量 → [网关/Service Mesh] → 90% → v1
                                → 10% → v2

  3. 观察指标（错误率、延迟、业务指标）
     
  4. 逐步扩大流量
     → 30% → 50% → 100%

  5. 全量切换或回滚
```

**实现方式：**

| 方式 | 工具 | 特点 |
|------|------|------|
| K8s 原生 | 多 Deployment + Service | 简单，粒度粗 |
| Istio | VirtualService 流量权重 | 精细控制 |
| Argo Rollouts | Rollout CRD + 自动分析 | 自动化金丝雀 |
| Flagger | Istio/Linkerd + Prometheus | 自动渐进式 |
| Nginx Ingress | Canary 注解 | 简单实用 |

```yaml
# Argo Rollouts 金丝雀发布
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: my-app
spec:
  replicas: 10
  strategy:
    canary:
      steps:
      - setWeight: 10        # 10% 流量
      - pause: {duration: 5m}
      - setWeight: 30
      - pause: {duration: 5m}
      - setWeight: 60
      - pause: {duration: 5m}
      - setWeight: 100
      analysis:
        templates:
        - templateName: success-rate
        startingStep: 1
  template:
    spec:
      containers:
      - name: app
        image: my-app:v2
```
