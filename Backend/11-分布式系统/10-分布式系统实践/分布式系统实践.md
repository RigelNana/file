## 1. 幂等性设计？

**回答：**

```
  幂等 = 同一操作执行多次结果和执行一次相同

  需要幂等的场景：
  网络重试 → 请求重复发送
  MQ重复消费 → 消息投递至少一次(at-least-once)
  用户重复点击 → 前端防抖不够可靠

  实现方案：
  ┌──────────────────┬──────────────────────────┐
  │ 方案              │ 适用                     │
  ├──────────────────┼──────────────────────────┤
  │ 唯一索引          │ 插入操作                 │
  │ 去重表            │ 通用 用请求ID去重        │
  │ 状态机            │ 有状态流转的业务         │
  │ Token机制         │ 表单提交                 │
  │ 乐观锁(版本号)    │ 更新操作                 │
  └──────────────────┴──────────────────────────┘

  去重表方案（推荐通用方案）：
  CREATE TABLE idempotent_records (
      request_id VARCHAR(64) PRIMARY KEY,
      result     TEXT,
      created_at TIMESTAMP
  );

  func ProcessWithIdempotency(ctx context.Context, reqID string, fn func() (string, error)) (string, error) {
      // 1. 查询是否已处理
      var result string
      err := db.QueryRowContext(ctx,
          "SELECT result FROM idempotent_records WHERE request_id=?", reqID).Scan(&result)
      if err == nil {
          return result, nil // 已处理 直接返回
      }
      
      // 2. 执行业务逻辑
      result, err = fn()
      if err != nil {
          return "", err
      }
      
      // 3. 记录结果（唯一索引防并发）
      db.ExecContext(ctx,
          "INSERT IGNORE INTO idempotent_records (request_id, result) VALUES (?, ?)",
          reqID, result)
      return result, nil
  }

  Token机制：
  1. 客户端先获取Token → 服务端存Redis
  2. 提交时带Token → 服务端验证并删除
  3. 重复提交 → Token已删 → 拒绝
```

  func initTracer() *trace.TracerProvider {

      tp := trace.NewTracerProvider(
          trace.WithBatcher(exporter),
          trace.WithResource(resource.NewWithAttributes(
              semconv.SchemaURL,
              semconv.ServiceNameKey.String("order-service"),

      )

  func TracingMiddleware(next http.Handler) http.Handler {
      return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
          tracer := otel.Tracer("http")
          ctx, span := tracer.Start(r.Context(), r.URL.Path)
          defer span.End()
          next.ServeHTTP(w, r.WithContext(ctx))
      })
  }

  // gRPC传递trace context
  conn, _ := grpc.Dial(addr,
      grpc.WithUnaryInterceptor(
          otelgrpc.UnaryClientInterceptor()),

## 3. 分布式限流实践？

**回答：**

```
  单机限流 vs 分布式限流：

  单机：rate.NewLimiter（进程内 简单高效）
  分布式：Redis集中计数（跨实例统一限流）

  多维度限流：
  ┌──────────┬──────────────────────┐
  │ 维度      │ 示例                 │
  ├──────────┼──────────────────────┤
  │ 用户      │ 每用户100次/分       │
  │ IP        │ 每IP 1000次/分       │
  │ 接口      │ 每API 5000次/秒      │
  │ 全局      │ 总QPS不超过10万      │
  └──────────┴──────────────────────┘

  Go限流中间件：
  func RateLimitMiddleware(rdb *redis.Client) func(http.Handler) http.Handler {
      return func(next http.Handler) http.Handler {
          return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
              key := "rate:" + r.RemoteAddr
              count, _ := rdb.Incr(r.Context(), key).Result()
              if count == 1 {
                  rdb.Expire(r.Context(), key, time.Minute)
              }
              if count > 100 { // 每分钟100次
                  http.Error(w, "Too Many Requests", 429)
                  return
              }
              next.ServeHTTP(w, r)
          })
      }
  }

  限流响应设计：
  HTTP 429 Too Many Requests
  Header: Retry-After: 30
  Header: X-RateLimit-Limit: 100
  Header: X-RateLimit-Remaining: 0
  Body: {"error": "rate limit exceeded", "retry_after": 30}
```

## 4. 数据一致性实践？

**回答：**

```
  分布式系统数据一致性的工程实践

  1. 异步最终一致（最常用）
     业务操作 → 发消息 → 异步消费更新
     保证消息可靠（本地消息表/事务消息）

  2. 对账修复
     定期对比主从数据
     不一致 → 以主为准修复

  3. 版本号防并发覆盖
     UPDATE t SET data=?, version=version+1
     WHERE id=? AND version=?
     affected==0 → 冲突 重试

  4. 读写分离的一致性
  func GetOrder(ctx context.Context, id string) (*Order, error) {
      // 刚写入的数据走主库
      if isRecentWrite(ctx, id) {
          return masterDB.Get(ctx, id)
      }
      // 其他走从库
      return slaveDB.Get(ctx, id)
  }
  
  func isRecentWrite(ctx context.Context, id string) bool {
      // 写入后在Redis标记 3秒内走主库
      return rdb.Exists(ctx, "recent_write:"+id).Val() > 0
  }

  5. 最终一致性保障清单
     □ 消息可靠投递（本地消息表+重试）
     □ 消费幂等（去重表+唯一索引）
     □ 对账Job（T+1日对账）
     □ 告警监控（不一致告警）
     □ 人工修复工具（兜底）
```

## 5. 分布式Job调度？

**回答：**

```
  问题：多实例部署时 定时任务只能执行一次

  方案：
  ┌──────────────┬──────────────────────────────┐
  │ 方案          │ 特点                         │
  ├──────────────┼──────────────────────────────┤
  │ 分布式锁      │ 抢锁执行 简单               │
  │ 选主          │ Leader执行 etcd选举          │
  │ 调度框架      │ 专业调度 xxl-job/asynq       │
  └──────────────┴──────────────────────────────┘

  分布式锁方案（简单场景）：
  func RunCronJob(rdb *redis.Client) {
      ticker := time.NewTicker(time.Minute)
      for range ticker.C {
          ok, _ := rdb.SetNX(ctx, "cron:daily-report",
              "locked", 30*time.Minute).Result()
          if ok {
              doReport()
              rdb.Del(ctx, "cron:daily-report")
          }
      }
  }

  Go异步任务框架 - Asynq（推荐）：
  // 定义任务
  func NewEmailTask(to string) *asynq.Task {
      payload, _ := json.Marshal(EmailPayload{To: to})
      return asynq.NewTask("email:send", payload)
  }
  
  // 入队
  client := asynq.NewClient(asynq.RedisClientOpt{Addr: "redis:6379"})
  client.Enqueue(NewEmailTask("user@example.com"),
      asynq.MaxRetry(3),
      asynq.Timeout(5*time.Minute))
  
  // Worker处理
  mux := asynq.NewServeMux()
  mux.HandleFunc("email:send", handleEmailTask)
  srv := asynq.NewServer(asynq.RedisClientOpt{Addr: "redis:6379"},
      asynq.Config{Concurrency: 10})
  srv.Run(mux)

  Asynq特点：
  Redis作为Broker 轻量
  支持重试/定时/周期任务
  Go原生 接口简洁
```

## 7. 优雅上下线？

**回答：**

```
  优雅下线 = 不丢请求地停止服务

  流程：
  1. 收到停止信号（SIGTERM）
  2. 标记不健康（readiness probe失败）
  3. 停止接收新请求
  4. 等待在途请求完成（或超时）
  5. 关闭连接 释放资源
  6. 退出进程

Go优雅关闭：
  func main() {
      srv := &http.Server{Addr: ":8080", Handler: mux}
      
      go func() {
          if err := srv.ListenAndServe(); err != http.ErrServerClosed {
              log.Fatal(err)
          }
      }()
      
      // 等待停止信号
      quit := make(chan os.Signal, 1)
      signal.Notify(quit, syscall.SIGINT, syscall.SIGTERM)
      <-quit
      
      log.Info("shutting down...")
      
      // 给在途请求30秒完成
      ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
      defer cancel()
      
      if err := srv.Shutdown(ctx); err != nil {
          log.Error("forced shutdown", "err", err)
      }
      
      // 关闭其他资源
      db.Close()
      rdb.Close()
      log.Info("server exited")
  }

  K8s配置：
  spec:
    terminationGracePeriodSeconds: 60  # 给60秒关闭
    containers:
    - lifecycle:
        preStop:
          exec:
            command: ["sleep", "5"]  # 等Service摘除Endpoint

  优雅上线 = 服务准备好再接流量
  readinessProbe通过后才加入Service Endpoints
  预热缓存/连接池后再标记Ready
```

## 8. 灰度发布实践？

**回答：**

```
  灰度发布 = 逐步放量验证新版本

  发布策略：
  ┌──────────────┬──────────────────────────────┐
  │ 策略          │ 说明                         │
  ├──────────────┼──────────────────────────────┤
  │ 金丝雀        │ 小比例流量→观察→逐步放大     │
  │ 蓝绿部署      │ 两套环境 切换流量            │
  │ 滚动更新      │ K8s默认 逐个替换Pod          │
  │ A/B测试       │ 按用户特征分流               │
  └──────────────┴──────────────────────────────┘

  金丝雀发布流程：
  1% → 观察5min → 5% → 观察10min
  → 20% → 观察30min → 50% → 100%

  K8s Ingress金丝雀：
  # 金丝雀Ingress
  metadata:
    annotations:
      nginx.ingress.kubernetes.io/canary: "true"
      nginx.ingress.kubernetes.io/canary-weight: "10"  # 10%流量

  Argo Rollouts（推荐）：
  apiVersion: argoproj.io/v1alpha1
  kind: Rollout
  spec:
    strategy:
      canary:
        steps:
        - setWeight: 5
        - pause: {duration: 10m}
        - setWeight: 20
        - pause: {duration: 10m}
        - setWeight: 50
        - pause: {duration: 10m}

  发布检查清单：
  □ 监控大盘正常（错误率/延迟/QPS）
  □ 日志无异常ERROR
  □ 核心业务指标正常（下单/支付成功率）
  □ 自动回滚条件配置（错误率>5%自动回滚）
```

## 9. 故障排查方法论？

**回答：**

```
  线上故障排查五步法：
  1. 确认影响范围（哪些用户/功能受影响）
  2. 快速止血（回滚/切流/降级）
  3. 定位根因（日志/监控/链路追踪）
  4. 修复问题
  5. 复盘总结

  排查工具链：
  ┌──────────────┬──────────────────────────────┐
  │ 维度          │ 工具                         │
  ├──────────────┼──────────────────────────────┤
  │ 指标监控      │ Prometheus + Grafana         │
  │ 日志查询      │ ELK / Loki + Grafana         │
  │ 链路追踪      │ Jaeger / Tempo               │
  │ 容器状态      │ kubectl describe/logs/exec   │
  │ Go性能        │ pprof (CPU/Heap/Goroutine)   │
  │ 网络          │ tcpdump/wireshark            │
  └──────────────┴──────────────────────────────┘

  常见故障定位：
  OOM → kubectl describe pod → 查看OOMKilled
       → pprof heap → 找到内存泄漏点
  
  延迟飙升 → Grafana P99延迟图 → 定位时间点
           → Jaeger查慢Trace → 找到慢Span
           → 可能原因：GC/慢SQL/下游超时
  
  错误率升高 → 过滤ERROR日志 → 分析错误类型
             → 连接池满/超时/依赖故障

  故障等级：
  P0：核心业务不可用 全员响应 5min内响应
  P1：核心业务部分受损 15min内响应
  P2：非核心功能异常 1h内响应
  P3：体验问题 工单处理
```

## 10. 分布式系统实践面试速答？

**回答：**

```
Q: 幂等怎么实现？
A: 去重表(请求ID唯一索引)+状态机+Token
   核心：同一操作多次执行结果相同

Q: 分布式追踪怎么做？
A: OpenTelemetry统一标准 Jaeger存储
   trace_id串联Trace+Metrics+Logs

Q: 分布式限流怎么做？
A: 单机rate.NewLimiter 分布式Redis+Lua
   多维度：用户/IP/接口/全局

Q: 怎么保证数据一致性？
A: 最终一致：可靠消息+幂等消费+对账
   读写分离：近期写入走主库

Q: 定时任务多实例怎么办？
A: 分布式锁抢占/Leader选举
   推荐Asynq框架 Redis作Broker

Q: 优雅关闭怎么实现？
A: 信号监听→停止接收→等待在途→关闭资源
   Go用http.Server.Shutdown K8s配合preStop+terminationGrace

Q: 灰度发布怎么做？
A: 金丝雀逐步放量 1%→5%→20%→100%
   K8s Ingress canary或Argo Rollouts

Q: 线上故障怎么排查？
A: 确认影响→快速止血→定位根因→修复→复盘
   Prometheus看指标 Jaeger查链路 日志找错误
```
