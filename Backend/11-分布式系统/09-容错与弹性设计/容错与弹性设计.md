# 容错与弹性设计

---

## 1. 容错设计原则？

**回答：**

```
  Design for Failure = 默认一切都会失败

  核心原则：
  1. 冗余：任何单点都有备份
  2. 隔离：故障不扩散（舱壁模式）
  3. 快速失败：尽早发现 尽快失败
  4. 优雅降级：核心功能可用 非核心关闭
  5. 自我恢复：自动检测+自动修复

  弹性设计模式：
  ┌──────────────┬──────────────────────────────┐
  │ 模式          │ 作用                         │
  ├──────────────┼──────────────────────────────┤
  │ 超时          │ 避免无限等待                 │
  │ 重试          │ 暂时性故障恢复               │
  │ 熔断          │ 阻断故障传播                 │
  │ 隔离          │ 故障边界控制                 │
  │ 降级          │ 核心保可用 非核心关           │
  │ 限流          │ 过载保护                     │
  │ 回退          │ 备选方案                     │
  └──────────────┴──────────────────────────────┘

  组合使用：
  请求 → 限流 → 超时 → 重试 → 熔断 → 降级/回退
```

## 2. 超时设计？

**回答：**

```
  超时设计三原则：
  1. 所有外部调用必须有超时
  2. 上游超时 > 下游超时 × 重试次数
  3. 超时要可配置

  超时级联：
  ┌────────┐ 5s  ┌────────┐ 2s  ┌────────┐ 1s  ┌────┐
  │Gateway │────→│ServiceA│────→│ServiceB│────→│ DB │
  └────────┘     └────────┘     └────────┘     └────┘
  
  Go context 传递截止时间：
  func HandleRequest(w http.ResponseWriter, r *http.Request) {
      ctx, cancel := context.WithTimeout(r.Context(), 3*time.Second)
      defer cancel()
      
      result, err := callServiceA(ctx)
      if err != nil {
          if ctx.Err() == context.DeadlineExceeded {
              http.Error(w, "timeout", http.StatusGatewayTimeout)
              return
          }
      }
  }

  func callServiceA(ctx context.Context) (Result, error) {
      // ctx已带截止时间 自动传递给下游
      return grpcClient.DoSomething(ctx, req)
  }

  超时分类：
  ┌──────────────┬──────────────────────────────┐
  │ 类型          │ 说明                         │
  ├──────────────┼──────────────────────────────┤
  │ 连接超时      │ 建立连接（100-500ms）        │
  │ 读超时        │ 等待响应（1-5s）             │
  │ 写超时        │ 发送数据                     │
  │ 空闲超时      │ 连接空闲回收（60s）          │
  │ 请求总超时    │ 整个请求生命周期             │
  └──────────────┴──────────────────────────────┘
```

## 3. 舱壁隔离模式？

**回答：**

```
  舱壁(Bulkhead) = 将系统分成互不影响的隔离区
  
  源自船舶设计：一个舱进水不影响其他舱

  隔离维度：
  ┌───────────────┬──────────────────────────────┐
  │ 维度           │ 方法                         │
  ├───────────────┼──────────────────────────────┤
  │ 线程/协程隔离  │ 不同服务用独立的goroutine池  │
  │ 连接池隔离     │ 每个下游独立的连接池         │
  │ 进程隔离       │ 不同服务部署在不同容器       │
  │ 节点隔离       │ 关键服务独占节点             │
  └───────────────┴──────────────────────────────┘

  goroutine池隔离：
  type BulkHead struct {
      sem chan struct{}
  }
  
  func NewBulkHead(maxConcurrency int) *BulkHead {
      return &BulkHead{sem: make(chan struct{}, maxConcurrency)}
  }
  
  func (b *BulkHead) Execute(ctx context.Context, fn func() error) error {
      select {
      case b.sem <- struct{}{}:
          defer func() { <-b.sem }()
          return fn()
      case <-ctx.Done():
          return ctx.Err()
      default:
          return errors.New("bulkhead full, rejected")
      }
  }

  // 使用：每个下游服务独立的BulkHead
  var (
      orderBulkhead   = NewBulkHead(100)
      paymentBulkhead = NewBulkHead(50)
  )

  好处：
  Service B故障 → 只耗尽自己的并发槽
  不影响Service C的调用
```

## 4. 降级策略？

**回答：**

```
  降级 = 系统过载时主动关闭部分功能 保核心

  降级级别：
  ┌──────┬──────────────────────────────────────┐
  │ 级别  │ 策略                                 │
  ├──────┼──────────────────────────────────────┤
  │ P0    │ 核心链路不降级（登录/下单/支付）     │
  │ P1    │ 次核心（推荐/评论→返回空/默认）      │
  │ P2    │ 辅助功能（日志详情→简化）            │
  │ P3    │ 可关闭（营销弹窗/动效→直接关）       │
  └──────┴──────────────────────────────────────┘

  降级方式：
  1. 返回缓存数据（旧数据好过没数据）
  2. 返回默认值（兜底数据）
  3. 功能降级（全文搜索→标题搜索）
  4. 体验降级（实时排行→定时排行）
  5. 服务降级（停止非核心微服务）

  Go 降级实现：
  type Degrader struct {
      enabled atomic.Bool
  }
  
  func (d *Degrader) GetRecommendations(ctx context.Context, uid string) ([]Item, error) {
      if d.enabled.Load() {
          // 降级：返回热门商品兜底
          return getHotItems(), nil
      }
      // 正常：调用推荐服务
      return recommendService.GetPersonalized(ctx, uid)
  }

  触发降级的信号：
  CPU > 80% / 内存 > 85%
  下游响应时间 > 阈值
  错误率 > 阈值
  人工开关（运营后台一键降级）
```

## 5. 回退(Fallback)模式？

**回答：**

```
  Fallback = 主逻辑失败时的备选方案

  回退策略：
  ┌──────────────────┬──────────────────────────┐
  │ 策略              │ 场景                     │
  ├──────────────────┼──────────────────────────┤
  │ 缓存回退          │ 服务不可用→返回缓存      │
  │ 默认值回退        │ 配置获取失败→用默认值    │
  │ 备份服务回退      │ 主服务挂→调备用服务      │
  │ 静态数据回退      │ 推荐服务挂→返回热门列表  │
  └──────────────────┴──────────────────────────┘

Go 多级回退：
  func GetUserProfile(ctx context.Context, uid string) (*Profile, error) {
      // 1. 尝试主服务
      profile, err := userService.GetProfile(ctx, uid)
      if err == nil {
          cache.Set(profileKey(uid), profile, time.Hour)
          return profile, nil
      }
      log.Warn("primary service failed", "err", err)
      
      // 2. 回退到缓存
      if cached, ok := cache.Get(profileKey(uid)); ok {
          return cached, nil
      }
      
      // 3. 回退到默认值
      return &Profile{
          Name:   "用户" + uid,
          Avatar: defaultAvatar,
      }, nil
  }

  回退+熔断组合：
  熔断Open → 直接走Fallback
  不去调用已知故障的服务
  减少无意义的等待和超时
```

## 6. 故障检测与自愈？

**回答：**

```
  故障检测：
  ┌──────────────┬──────────────────────────────┐
  │ 方式          │ 说明                         │
  ├──────────────┼──────────────────────────────┤
  │ 心跳检测      │ 定时心跳 超时判定故障        │
  │ 健康检查      │ HTTP/TCP/gRPC Health         │
  │ Gossip探测    │ 多节点交叉探测 避免误判      │
  │ Phi检测       │ 根据历史心跳间隔概率判断     │
  └──────────────┴──────────────────────────────┘

  K8s 健康检查：
  livenessProbe → 检测进程是否存活 → 失败重启容器
  readinessProbe → 检测是否可服务 → 失败摘除流量
  startupProbe → 检测启动完成 → 慢启动保护

  spec:
    containers:
    - name: app
      livenessProbe:
        httpGet:
          path: /healthz
          port: 8080
        initialDelaySeconds: 5
        periodSeconds: 10
      readinessProbe:
        httpGet:
          path: /ready
          port: 8080
        periodSeconds: 5

  自愈机制：
  1. K8s自动重启（crashLoopBackoff）
  2. HPA自动扩容（CPU/内存/自定义指标）
  3. PDB（PodDisruptionBudget）保证最小可用数
  4. Node自动修复（Cluster Autoscaler替换故障节点）

  Go健康检查端点：
  http.HandleFunc("/healthz", func(w http.ResponseWriter, r *http.Request) {
      if err := db.Ping(); err != nil {
          w.WriteHeader(http.StatusServiceUnavailable)
          return
      }
      w.WriteHeader(http.StatusOK)
  })
```

## 7. 混沌工程？

**回答：**

```
  混沌工程 = 主动注入故障 验证系统弹性

  流程：
  1. 定义稳态假设（正常指标是什么样）
  2. 引入故障变量（杀进程/网络延迟/磁盘满）
  3. 观察系统行为
  4. 对比假设 发现弱点
  5. 修复并重新验证

  故障类型：
  ┌──────────────┬──────────────────────────────┐
  │ 类型          │ 工具                         │
  ├──────────────┼──────────────────────────────┤
  │ Pod故障       │ 杀Pod/容器                   │
  │ 网络故障      │ 延迟/丢包/断网/DNS故障       │
  │ 磁盘故障      │ 磁盘满/IO延迟               │
  │ CPU/内存压力  │ stress-ng                    │
  │ 依赖故障      │ 下游服务不可用               │
  └──────────────┴──────────────────────────────┘

  K8s混沌工程工具：
  Chaos Mesh（PingCAP开源 K8s原生）
  Litmus（CNCF项目）
  ChaosBlade（阿里开源）

  Chaos Mesh 示例（Pod杀死）：
  apiVersion: chaos-mesh.org/v1alpha1
  kind: PodChaos
  metadata:
    name: pod-kill-test
  spec:
    action: pod-kill
    mode: one
    selector:
      labelSelectors:
        app: my-service

  实践建议：
  先在测试环境验证
  从小范围开始（单Pod → 单节点 → 可用区）
  生产混沌必须有紧急停止开关
  建立GameDay机制 定期演练
```

## 8. 流量管理？

**回答：**

```
  流量管理 = 控制请求的路由和分配

  灰度发布/金丝雀：
  ┌──────────┐   90%   ┌──────────┐
  │          │────────→│  V1稳定  │
  │ Gateway  │         └──────────┘
  │          │   10%   ┌──────────┐
  │          │────────→│  V2新版  │
  └──────────┘         └──────────┘

  流量染色：
  请求打标签 → 路由到特定版本
  Header: X-Traffic-Tag: gray
  测试流量专走灰度链路

  AB测试：
  用户ID Hash → A组/B组 → 不同版本
  收集数据对比效果

  流量回放：
  录制线上流量 → 在测试环境回放
  验证新版本在真实流量下的表现

  Go 流量路由中间件：
  func TrafficRouter(next http.Handler) http.Handler {
      return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
          tag := r.Header.Get("X-Traffic-Tag")
          switch tag {
          case "gray":
              grayProxy.ServeHTTP(w, r)
          case "shadow":
              // 影子流量：同时转发 不影响正常响应
              go shadowProxy.ServeHTTP(io.Discard, r.Clone(r.Context()))
              next.ServeHTTP(w, r)
          default:
              next.ServeHTTP(w, r)
          }
      })
  }

  Feature Flag：
  func IsFeatureEnabled(uid string, feature string) bool {
      // 根据用户ID+特性名 判断是否开启
      // 可以按比例/白名单/条件控制
  }
```

## 9. 容量规划？

**回答：**

```
  容量规划 = 确保系统能承载预期流量

  步骤：
  1. 估算流量（QPS/TPS/带宽/存储）
  2. 压测获取单机能力
  3. 计算所需资源
  4. 预留Buffer（1.5-2倍）

  计算公式：
  实例数 = 峰值QPS / 单实例QPS × 安全系数
  eg: 10000QPS / 2000QPS × 1.5 = 7.5 → 8实例

  存储估算：
  日增数据 = 日活用户 × 人均操作 × 单条大小
  eg: 100万DAU × 10条 × 1KB = 10GB/天

  关键指标：
  ┌──────────┬──────────────────────────────────┐
  │ 指标      │ 计算                             │
  ├──────────┼──────────────────────────────────┤
  │ QPS       │ 日PV / 86400 × 峰值系数(3-5)    │
  │ 带宽      │ QPS × 响应大小                   │
  │ 连接数    │ 并发用户 × 连接保持比            │
  │ 存储      │ 日增量 × 保留天数 × 副本数       │
  └──────────┴──────────────────────────────────┘

  弹性伸缩：
  K8s HPA → CPU/内存/自定义指标自动扩缩
  apiVersion: autoscaling/v2
  kind: HorizontalPodAutoscaler
  spec:
    minReplicas: 3
    maxReplicas: 20
    metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70

  压测工具：
  wrk/vegeta/k6 → 得出单实例能力
  定期压测 保证容量评估准确
```

## 10. 容错与弹性面试速答？

**回答：**

```
Q: 容错设计核心原则？
A: Design for Failure 默认一切会失败
   冗余+隔离+快速失败+降级+自愈

Q: 超时设计要注意什么？
A: 所有外部调用必须超时
   上游>下游×重试次数 用context传递

Q: 舱壁模式是什么？
A: 隔离不同调用的资源池（goroutine/连接池）
   一个下游故障不拖垮其他调用

Q: 降级策略有哪些？
A: 返回缓存/默认值/功能简化/关闭非核心
   按P0-P3分级 P0不降级

Q: Fallback怎么设计？
A: 主服务→缓存→默认值 多级回退
   和熔断配合 Open时直接走Fallback

Q: K8s怎么做健康检查？
A: liveness(存活)→重启 readiness(就绪)→摘流量
   startup(启动)→慢启动保护

Q: 混沌工程怎么做？
A: 定义稳态→注入故障→观察→修复
   Chaos Mesh/Litmus 先测试环境后生产

Q: 容量怎么规划？
A: 实例数=峰值QPS/单机QPS×安全系数
   定期压测+HPA弹性伸缩
```
