# 分布式基础理论

---

## 1. CAP定理详解？

**回答：**

```
CAP 三要素：
  C（Consistency）：所有节点在同一时刻看到相同数据
  A（Availability）：每个请求都能收到非错误响应
  P（Partition tolerance）：网络分区时系统仍能运行

  网络分区不可避免 → P 必须保证
  → 实际选择是 CP 或 AP

  ┌──────────────┬──────────────────────────────┐
  │ 选择          │ 系统                         │
  ├──────────────┼──────────────────────────────┤
  │ CP            │ etcd, ZooKeeper, HBase       │
  │              │ 分区时拒绝服务 保证一致       │
  ├──────────────┼──────────────────────────────┤
  │ AP            │ Cassandra, DynamoDB, Eureka  │
  │              │ 分区时继续服务 允许不一致     │
  └──────────────┴──────────────────────────────┘

  注意：CAP 中的 C 是线性一致性（最强的一致性）
  实际系统通常在中间地带
  不是二选一的开关 而是一个连续光谱
```

## 2. BASE理论？

**回答：**

```
BASE = CAP 中 AP 的工程化实践

  BA（Basically Available）基本可用
    → 允许响应时间增加（正常50ms 出问题200ms）
    → 允许功能降级（搜索降级为热门推荐）

  S（Soft State）软状态
    → 数据允许存在中间状态
    → 不同副本之间存在同步延迟

  E（Eventually Consistent）最终一致性
    → 不要求实时一致
    → 在一定时间窗口后达成一致

一致性级别：
  ┌──────────────┬──────────────────────────────┐
  │ 级别          │ 说明                         │
  ├──────────────┼──────────────────────────────┤
  │ 强一致性      │ 写后立即可读到（线性一致性） │
  │ 顺序一致性    │ 操作顺序全局一致             │
  │ 因果一致性    │ 有因果关系的操作有序         │
  │ 最终一致性    │ 最终所有副本一致             │
  └──────────────┴──────────────────────────────┘

  大多数互联网系统选择最终一致性
  如：下单后余额不立即变 但几秒后一定变
```

## 3. 分布式系统八大谬误？

**回答：**

```
Peter Deutsch 提出的八大谬误：

  1. 网络是可靠的         → 实际：会丢包/断连
  2. 延迟为零             → 实际：跨机房几十ms
  3. 带宽是无限的         → 实际：有瓶颈
  4. 网络是安全的         → 实际：需加密认证
  5. 拓扑不会变化         → 实际：节点增减常见
  6. 只有一个管理员       → 实际：多团队多权限
  7. 传输成本为零         → 实际：序列化/网络有开销
  8. 网络是同质的         → 实际：不同协议/硬件

  工程启示：
  ┌──────────────┬──────────────────────────────┐
  │ 谬误          │ 设计对策                     │
  ├──────────────┼──────────────────────────────┤
  │ 网络不可靠    │ 超时+重试+幂等               │
  │ 延迟不为零    │ 异步+批量+缓存               │
  │ 带宽有限      │ 压缩+精简+分页               │
  │ 网络不安全    │ TLS+认证+最小权限            │
  │ 拓扑变化      │ 服务发现+健康检查            │
  └──────────────┴──────────────────────────────┘
```

## 4. 时钟与时序问题？

**回答：**

```
分布式系统中没有全局时钟

  物理时钟：NTP同步 有误差（几ms到几十ms）
  逻辑时钟：用来确定事件顺序

Lamport 逻辑时钟：
  每个节点维护一个计数器
  发送消息：counter++, 附带counter
  接收消息：counter = max(local, received) + 1
  → 可以判断因果关系

向量时钟（Vector Clock）：
  每个节点维护所有节点的计数器
  [A:2, B:3, C:1]
  
  比较规则：
  V1 < V2 当且仅当 V1所有分量≤V2 且至少一个<
  否则 V1 ∥ V2（并发 无法比较）

  用途：检测数据冲突
  Dynamo/Riak 使用向量时钟

Hybrid Logical Clock（HLC）：
  结合物理时钟和逻辑时钟
  TiDB/CockroachDB 使用
  
  物理部分提供近似真实时间
  逻辑部分保证因果序
```

## 5. 拜占庭将军问题？

**回答：**

```
  问题：部分节点可能发送错误/恶意信息
  如何在有叛徒的情况下达成共识？

  ┌─────────┐
  │ 将军A    │──正确消息──→ 将军C
  ├─────────┤
  │ 将军B    │──错误消息──→ 将军C（叛徒）
  ├─────────┤
  │ 将军C    │  收到矛盾信息 → 如何决策？
  └─────────┘

  BFT定理：N ≥ 3f+1（f个拜占庭故障需要至少3f+1个节点）

  实际应用：
  ┌──────────────┬──────────────────────────┐
  │ 场景          │ 是否需要BFT              │
  ├──────────────┼──────────────────────────┤
  │ 内部集群      │ 不需要（信任节点 用Raft） │
  │ 区块链        │ 需要（不信任节点）        │
  │ P2P网络       │ 需要                     │
  └──────────────┴──────────────────────────┘

CFT（Crash Fault Tolerance）：
  只处理节点宕机 不处理恶意
  Raft/Paxos 属于 CFT
  N ≥ 2f+1（f个宕机需2f+1个节点）
  大多数内部系统用 CFT 就够了
```

## 6. Quorum 机制？

**回答：**

```
  N = 副本总数
  W = 写成功需要的副本数
  R = 读需要的副本数

  保证强一致：W + R > N

  常见配置：
  ┌──────┬───┬───┬──────────────────┐
  │ N    │ W │ R │ 特点              │
  ├──────┼───┼───┼──────────────────┤
  │ 3    │ 2 │ 2 │ 读写都要多数确认  │
  │ 3    │ 3 │ 1 │ 写慢读快 强一致   │
  │ 3    │ 1 │ 3 │ 写快读慢 最终一致 │
  │ 3    │ 1 │ 1 │ 最快 但可能不一致 │
  └──────┴───┴───┴──────────────────┘

  例：3副本 W=2 R=2
  写入时2个副本确认 → 读取2个副本
  至少有1个副本是最新的 → 能读到最新

  Kafka ISR：
  min.insync.replicas = 2
  acks = all
  → 类似 W=2（ISR中所有副本确认）

  Cassandra 直接支持 Quorum 级别：
  CONSISTENCY QUORUM → 读/写都要多数
```

## 7. 数据分片策略？

**回答：**

```
  ┌──────────────┬──────────────────────────────┐
  │ 策略          │ 特点                         │
  ├──────────────┼──────────────────────────────┤
  │ Hash 分片     │ 均匀但扩容需迁移             │
  │ Range 分片    │ 范围查询友好 可能不均         │
  │ 一致性Hash    │ 扩容只迁少量 但可能不均       │
  │ 虚拟节点Hash  │ 一致性Hash+虚拟节点 均匀     │
  └──────────────┴──────────────────────────────┘

一致性Hash：
  ┌─────── 节点A ──────┐
  │                     │
  节点D         Key X → 节点B ← 顺时针找
  │                     │
  └─────── 节点C ──────┘
  
  加入新节点E（在A和B之间）：
  只有A~E之间的Key需要迁移到E
  其他Key不受影响

虚拟节点：
  每个物理节点映射 100~200 个虚拟节点
  分布更均匀
  
  Node1 → vn1_1, vn1_2, ..., vn1_150
  Node2 → vn2_1, vn2_2, ..., vn2_150

Redis Cluster：
  16384 个 slot（Hash 槽）
  CRC16(key) % 16384 → slot
  slot 分配到不同节点
  扩容迁移 slot 单位
```

## 8. 分布式共识？

**回答：**

```
共识 = 多个节点就某个值达成一致

  Paxos：
  理论完备但难以理解和实现
  角色：Proposer/Acceptor/Learner
  两阶段：Prepare(承诺) → Accept(接受)

  Raft（推荐理解）：
  三个角色：Leader/Follower/Candidate
  
  选举过程：
  1. Follower 超时 → 变成 Candidate
  2. Candidate 发 RequestVote 给所有节点
  3. 获得多数票 → 成为 Leader
  4. Leader 发心跳 维持领导权

  日志复制：
  1. Client → Leader（写请求）
  2. Leader 写本地日志
  3. Leader 复制日志给 Follower（AppendEntries）
  4. 多数确认 → Leader 提交
  5. Leader 通知 Follower 提交

  ┌────────┐  写请求   ┌────────┐
  │ Client │──────────→│ Leader │
  └────────┘           └───┬────┘
                      复制日志
                    ┌────┴────┐
                ┌───▼──┐  ┌──▼───┐
                │ F1   │  │ F2   │
                └──────┘  └──────┘
                多数确认 → 提交

  etcd 使用 Raft 的工业级实现
```

## 9. FLP 不可能定理？

**回答：**

```
FLP 定理（Fischer-Lynch-Paterson, 1985）：
  在异步系统中 即使只有一个节点可能崩溃
  也不存在能保证终止的共识算法

  异步 = 消息延迟无上界
  
  意味着：
  确定性算法 + 异步网络 + 1个故障
  → 无法同时保证 Safety + Liveness

  工程上如何绕过：
  1. 超时机制（引入部分同步假设）
     Raft 用超时触发选举
  2. 随机化算法
     随机选举等待时间避免冲突
  3. 故障检测器
     最终检测到故障节点

  所以实际的共识算法都引入了超时：
  Raft：选举超时 150-300ms 随机
  Paxos：Proposer 超时后重试
  
  FLP 是理论下限
  实际系统通过工程手段工作良好
```

## 10. 分布式基础面试速答？

**回答：**

```
Q: CAP定理？
A: 一致性/可用性/分区容忍三选二
   P必选 实际选CP或AP

Q: BASE理论？
A: 基本可用+软状态+最终一致
   AP方案的工程化实践

Q: 强一致和最终一致区别？
A: 强一致=写后立即读到最新
   最终一致=一定时间后所有副本一致

Q: Raft选举过程？
A: Follower超时→Candidate→发RequestVote
   获多数票→成为Leader→心跳维持

Q: 一致性Hash解决什么？
A: 扩缩容时只迁移少量数据
   虚拟节点解决数据倾斜

Q: Quorum机制？
A: W+R>N保证强一致
   3副本常用W=2 R=2

Q: 拜占庭故障？
A: 节点可能发恶意消息
   内部系统不需要BFT 用Raft(CFT)即可

Q: FLP不可能定理？
A: 异步系统中共识不能保证终止
   实际用超时和随机化绕过
```
