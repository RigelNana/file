# 服务治理

## 1. 服务治理概述？

**回答：**

```
  服务治理 = 管理微服务间的通信/可靠性/可观测性

  核心能力：
  ┌─────────────────┬──────────────────────────────┐
  │ 能力             │ 解决什么问题                 │
  ├─────────────────┼──────────────────────────────┤
  │ 服务注册与发现   │ 服务之间如何找到对方         │
  │ 负载均衡         │ 流量如何分配                 │
  │ 熔断降级         │ 故障如何隔离                 │
  │ 限流             │ 过载如何保护                 │
  │ 超时重试         │ 调用失败如何处理             │
  │ 链路追踪         │ 调用链如何观测               │
  │ 配置管理         │ 配置如何动态下发             │
  └─────────────────┴──────────────────────────────┘

  治理方式演进：
  SDK侵入式（Go框架如go-micro/Kratos）
  → Sidecar代理（Service Mesh Istio/Linkerd）
  → eBPF内核态（Cilium 未来方向）
```

## 2. 服务注册与发现？

**回答：**

```
  服务启动时注册自己 调用方查询可用实例

  ┌─────────┐ Register ┌──────────────┐
  │ Service │────────→ │ Registry     │
  │    A    │          │ (etcd/Consul)│
  └─────────┘          └──────┬───────┘
                              │ Watch/Query
  ┌─────────┐ Discover ┌─────▼───────┐
  │ Service │←─────────│   Client    │
  │    B    │          │  (服务B)    │
  └─────────┘          └─────────────┘

  注册中心选型：
  ┌──────────┬─────────┬──────────┬────────────┐
  │ 系统      │ 一致性  │ 健康检查 │ 语言       │
  ├──────────┼─────────┼──────────┼────────────┤
  │ etcd      │ CP(Raft)│ Lease TTL│ Go         │
  │ Consul    │ CP(Raft)│ 主动探测 │ Go         │
  │ Nacos     │ AP/CP   │ 心跳     │ Java       │
  │ ZooKeeper │ CP(ZAB) │ 会话     │ Java       │
  │ Eureka    │ AP      │ 心跳     │ Java       │
  └──────────┴─────────┴──────────┴────────────┘
  Go生态推荐：etcd

  K8s服务发现：
  Service + DNS → pod-name.svc.cluster.local
  Endpoints自动更新 无需额外注册中心

Go + etcd 注册：
  func Register(client *clientv3.Client, svcName, addr string) {
      lease, _ := client.Grant(context.TODO(), 10)
      client.Put(context.TODO(),
          "/services/"+svcName+"/"+addr, addr,
          clientv3.WithLease(lease.ID))
      // 续约
      ch, _ := client.KeepAlive(context.TODO(), lease.ID)
      go func() { for range ch {} }()
  }
```

## 3. 负载均衡？

**回答：**

```
  客户端负载均衡（gRPC/go-micro）vs 服务端负载均衡（Nginx/LB）

  常用算法：
  ┌──────────────────┬──────────────────────────┐
  │ 算法              │ 特点                     │
  ├──────────────────┼──────────────────────────┤
  │ 轮询(RR)          │ 简单均匀                 │
  │ 加权轮询(WRR)     │ 按权重分配               │
  │ 随机              │ 简单 大数定律趋均匀       │
  │ 最少连接(LC)      │ 选当前连接最少的节点     │
  │ 一致性Hash        │ 同Key固定到同节点        │
  │ P2C(两次随机)     │ gRPC默认 性能好          │
  └──────────────────┴──────────────────────────┘

  P2C（Power of Two Choices）：
  随机选两个节点 取负载低的那个
  比全局最优差一点 但无需全局信息

  gRPC负载均衡：
  // 客户端侧
  conn, _ := grpc.Dial(
      "etcd:///my-service",
      grpc.WithDefaultServiceConfig(
          `{"loadBalancingPolicy":"round_robin"}`),
  )

  健康检查 + 负载均衡：
  探活失败的节点自动摘除
  恢复后重新加入负载均衡池
```

## 4. 熔断器模式？

**回答：**

```
  熔断器 = 防止故障级联扩散

  三态转换：
  ┌────────┐ 连续失败    ┌──────┐
  │ Closed │────────────→│ Open │ 快速失败
  └───┬────┘ 达到阈值    └──┬───┘
      │                    │ 超时
      │                    ▼
      │               ┌──────────┐
      │               │Half-Open │ 试探请求
      │               └────┬─────┘
      │ 试探成功            │ 试探失败
      └────────────────────┘

Go 实现（gobreaker）：
  cb := gobreaker.NewCircuitBreaker(gobreaker.Settings{
      Name:        "my-service",
      MaxRequests: 3,                          // Half-Open允许3个试探
      Interval:    10 * time.Second,           // Closed状态计数周期
      Timeout:     30 * time.Second,           // Open→Half-Open超时
      ReadyToTrip: func(counts gobreaker.Counts) bool {
          return counts.ConsecutiveFailures > 5 // 连续5次失败开启
      },
  })

  result, err := cb.Execute(func() (interface{}, error) {
      return callRemoteService()
  })

  熔断指标：
  失败率（如50%以上）
  连续失败次数（如5次）
  慢调用比率（如30%超过2s）
  滑动窗口统计（最近10s/100次）

  和限流的区别：
  限流 = 保护自己不被打垮
  熔断 = 保护自己不被下游拖垮
```

## 5. 限流算法？

**回答：**

```
  四种主流限流算法：

  1. 固定窗口
     时间窗口内计数 超过阈值拒绝
     问题：窗口边界突发

  2. 滑动窗口
     细分多个小窗口 滑动统计
     解决边界突发

  3. 漏桶
     请求以固定速率处理（FIFO队列）
     流量整形 但突发能力差

  4. 令牌桶（推荐）
     以固定速率生成令牌 请求消耗令牌
     允许突发（桶中有存量令牌）

  ┌──────────┬──────────┬──────────┬──────────┐
  │ 算法      │ 突发流量 │ 平滑度   │ 实现     │
  ├──────────┼──────────┼──────────┼──────────┤
  │ 固定窗口  │ 允许     │ 差       │ 简单     │
  │ 滑动窗口  │ 较好     │ 较好     │ 中等     │
  │ 漏桶      │ 不允许   │ 好       │ 简单     │
  │ 令牌桶    │ 允许     │ 好       │ 中等     │
  └──────────┴──────────┴──────────┴──────────┘

Go 令牌桶：
  limiter := rate.NewLimiter(rate.Limit(100), 200)
  // 每秒100个令牌 桶容量200

  if !limiter.Allow() {
      return errors.New("rate limited")
  }

分布式限流（Redis + Lua）：
  -- 滑动窗口限流
  local key = KEYS[1]
  local limit = tonumber(ARGV[1])
  local window = tonumber(ARGV[2])
  local now = tonumber(ARGV[3])
  redis.call('ZREMRANGEBYSCORE', key, 0, now - window)
  local count = redis.call('ZCARD', key)
  if count < limit then
      redis.call('ZADD', key, now, now .. math.random())
      redis.call('EXPIRE', key, window / 1000)
      return 1
  end
  return 0
```

## 6. 超时与重试？

**回答：**

```
  超时原则：
  上游超时 > 下游超时 × 重试次数
  eg: Gateway 3s > Service 1s × 2次重试

  超时设置：
  ┌──────────┬──────────┬──────────────────┐
  │ 组件      │ 超时类型 │ 建议值           │
  ├──────────┼──────────┼──────────────────┤
  │ HTTP     │ 连接      │ 100-500ms       │
  │ HTTP     │ 读写      │ 1-5s            │
  │ gRPC     │ 截止时间  │ 1-3s            │
  │ DB       │ 查询      │ 1-3s            │
  │ Redis    │ 读写      │ 100-500ms       │
  └──────────┴──────────┴──────────────────┘

重试策略：
  立即重试 → 固定间隔 → 指数退避 → 指数退避+抖动

  指数退避 + 抖动（推荐）：
  func RetryWithBackoff(fn func() error, maxRetries int) error {
      for i := 0; i < maxRetries; i++ {
          err := fn()
          if err == nil {
              return nil
          }
          if !isRetryable(err) {
              return err // 不可重试错误
          }
          base := time.Duration(1<<uint(i)) * 100 * time.Millisecond
          jitter := time.Duration(rand.Int63n(int64(base)))
          time.Sleep(base + jitter)
      }
      return errors.New("max retries exceeded")
  }

  重试注意：
  只重试幂等操作（GET/PUT 不重试POST）
  只重试可恢复错误（5xx/超时 不重试4xx）
  限制重试次数（通常2-3次）
  避免重试风暴（加抖动 控制总重试比例）
```

## 7. 链路追踪？

**回答：**

```
  链路追踪 = 跟踪请求在微服务间的完整调用路径

  核心概念：
  Trace：完整调用链（一个请求的全部路径）
  Span：一次调用（有parent形成树形结构）
  Context：trace_id + span_id 跨服务传递

  ┌─────┐    ┌─────┐    ┌─────┐
  │ A   │───→│ B   │───→│ C   │
  │ T1  │    │ T1  │    │ T1  │ 同一trace_id
  │ S1  │    │ S2  │    │ S3  │ 不同span_id
  └─────┘    └─────┘    └─────┘

  技术选型：
  ┌──────────────┬──────────────────────────────┐
  │ 系统          │ 特点                         │
  ├──────────────┼──────────────────────────────┤
  │ Jaeger        │ Uber开源 Go实现 推荐         │
  │ Zipkin        │ Twitter开源 Java             │
  │ SkyWalking    │ Apache 无侵入字节码增强      │
  │ Tempo         │ Grafana推 对接Grafana        │
  └──────────────┴──────────────────────────────┘

  OpenTelemetry（OTEL）= 统一标准
  Traces + Metrics + Logs 三合一

Go接入OTEL：
  tp := trace.NewTracerProvider(
      trace.WithBatcher(exporter),
      trace.WithResource(resource.NewWithAttributes(
          semconv.ServiceName("my-service"),
      )),
  )
  otel.SetTracerProvider(tp)

  tracer := otel.Tracer("my-service")
  ctx, span := tracer.Start(ctx, "handleRequest")
  defer span.End()
  span.SetAttributes(attribute.String("user.id", userID))

  采样策略：
  全量采样（测试/小流量）
  概率采样（1%/10%）
  尾采样（出错/慢请求才采集）
```

## 8. Service Mesh？

**回答：**

```
  Service Mesh = 基础设施层处理服务治理

  核心思想：治理能力从业务代码下沉到Sidecar代理

  ┌─────────────────────────────┐
  │ Service A                   │
  │ ┌──────┐    ┌─────────────┐│
  │ │ App  │───→│ Sidecar     ││ (Envoy)
  │ └──────┘    │ 限流/熔断/  ││
  │             │ 重试/追踪   ││
  │             └──────┬──────┘│
  └────────────────────┼───────┘
                       │ mTLS
  ┌────────────────────┼───────┐
  │ Service B          │       │
  │ ┌─────────────┐  ┌▼─────┐ │
  │ │ Sidecar     │──│ App  │ │
  │ └─────────────┘  └──────┘ │
  └─────────────────────────────┘

  Istio = 最主流的Service Mesh

  数据平面：Envoy Sidecar（每个Pod一个）
  控制平面：Istiod（配置下发/证书管理/服务发现）

  Istio流量治理：
  VirtualService → 路由规则
  DestinationRule → 负载均衡/熔断
  Gateway → 入口流量管理

  优势：
  业务代码零侵入
  统一的治理策略
  灰度/金丝雀发布内置支持

  劣势：
  增加延迟（两次Sidecar代理）
  资源开销（每个Pod一个Envoy）
  运维复杂度高
```

## 9. 配置管理？

**回答：**

```
  配置管理 = 配置集中管理 + 动态更新

  方案对比：
  ┌──────────────┬────────────┬──────────────────┐
  │ 方案          │ 特点       │ 适用             │
  ├──────────────┼────────────┼──────────────────┤
  │ 文件/环境变量 │ 简单       │ 小项目           │
  │ K8s ConfigMap │ 原生       │ K8s环境          │
  │ etcd          │ 强一致     │ Go微服务         │
  │ Nacos         │ 功能丰富   │ Java生态         │
  │ Apollo        │ 灰度/审计  │ 大型系统         │
  │ Consul KV     │ 服务发现+配置│ HashiCorp生态  │
  └──────────────┴────────────┴──────────────────┘

  etcd配置管理（Go推荐）：
  func WatchConfig(client *clientv3.Client, key string) {
      // 初始加载
      resp, _ := client.Get(context.TODO(), key)
      applyConfig(resp.Kvs[0].Value)
      
      // 监听变更
      ch := client.Watch(context.TODO(), key)
      for wresp := range ch {
          for _, ev := range wresp.Events {
              applyConfig(ev.Kv.Value)
              log.Info("config updated", "key", key)
          }
      }
  }

  配置管理最佳实践：
  1. 分环境：dev/staging/prod 隔离
  2. 敏感信息加密：密码/密钥不明文存储
  3. 版本管理：配置可回滚
  4. 灰度发布：先推少量实例验证
  5. 变更审计：谁在何时改了什么
```

## 10. 服务治理面试速答？

**回答：**

```
Q: 服务注册发现怎么做？
A: etcd/Consul存储实例信息 Lease续约
   K8s用Service+DNS内置发现

Q: 负载均衡有哪些算法？
A: 轮询/加权轮询/最少连接/一致性Hash/P2C
   gRPC默认P2C 随机选两个取负载低的

Q: 熔断器三个状态？
A: Closed(正常)→Open(快速失败)→Half-Open(试探)
   连续失败超阈值触发 gobreaker库

Q: 限流怎么做？
A: 单机令牌桶(rate.NewLimiter)
   分布式Redis+Lua滑动窗口

Q: 重试策略要注意什么？
A: 只重试幂等操作+可恢复错误
   指数退避+抖动 防重试风暴

Q: 链路追踪原理？
A: Trace→Span树 trace_id跨服务传递
   OpenTelemetry统一标准 Jaeger存储查询

Q: Service Mesh是什么？
A: Sidecar代理处理治理（Envoy）
   Istio控制平面下发规则 业务零侵入

Q: 配置管理怎么做？
A: etcd/Nacos集中存储 Watch监听变更
   分环境隔离 敏感信息加密 支持回滚
```
