# 一致性协议

---

## 1. Paxos算法？

**回答：**

```
Paxos 角色：
  Proposer：提出提案
  Acceptor：投票接受
  Learner：学习结果

Basic Paxos 两阶段：
  Phase 1 - Prepare：
    Proposer 选编号 n → 发 Prepare(n) 给所有 Acceptor
    Acceptor：如果 n > 已承诺的编号
      → 承诺不再接受 < n 的提案
      → 返回已接受的最大编号提案（如有）

  Phase 2 - Accept：
    Proposer 收到多数 Promise
    → 选择已接受提案中编号最大的值（或自己的值）
    → 发 Accept(n, value) 给所有 Acceptor
    Acceptor：如果 n ≥ 已承诺编号 → 接受

  ┌──────────┐  Prepare(n)   ┌──────────┐
  │ Proposer │──────────────→│ Acceptor │
  └──────────┘               └──────────┘
       │          Promise(n)       │
       │←──────────────────────────│
       │                           │
       │    Accept(n, val)         │
       │──────────────────────────→│
       │                           │
       │    Accepted(n, val)       │
       │←──────────────────────────│

Multi-Paxos：
  选一个 Leader 跳过 Phase 1
  后续只需 Phase 2 → 性能大幅提升
  实际系统都用 Multi-Paxos 或 Raft
```

## 2. Raft算法详解？

**回答：**

```
Raft 三个子问题：
  1. Leader 选举
  2. 日志复制
  3. 安全性

状态转换：
  ┌─────────┐  超时  ┌───────────┐  获多数票  ┌────────┐
  │ Follower│──────→│ Candidate │──────────→│ Leader │
  └─────────┘       └───────────┘           └────────┘
       ↑                  │ 发现更高Term         │
       │                  ↓                     │
       └──────────────────┘←────────────────────┘
                    发现更高Term Leader

Leader 选举：
  每个节点随机选举超时（150-300ms）
  超时的 Follower → Candidate
  Term+1 → 给自己投票 → RequestVote
  多数票 → 成为 Leader → 发心跳

日志复制：
  type LogEntry struct {
      Term    int
      Index   int
      Command interface{}
  }

  Leader → AppendEntries → Followers
  多数确认 → commitIndex 前进
  已提交的日志不可更改

安全性保证：
  选举限制：候选人的日志必须至少和多数一样新
  → 保证新 Leader 一定包含所有已提交日志
  Leader 不删日志只追加
```

## 3. ZAB协议（ZooKeeper）？

**回答：**

```
ZAB = ZooKeeper Atomic Broadcast

  和 Raft 类似 都是 Leader-based

  两种模式：
  1. 恢复模式（选Leader+同步数据）
  2. 广播模式（Leader处理写请求）

  写流程：
  Client → Leader → Proposal → Followers
                              多数ACK → Commit

  与 Raft 的区别：
  ┌──────────────┬──────────────┬──────────────┐
  │ 对比          │ Raft          │ ZAB          │
  ├──────────────┼──────────────┼──────────────┤
  │ 日志          │ 连续索引      │ zxid(epoch+txn)│
  │ 选举          │ Term+日志最新 │ epoch+zxid最大│
  │ 提交          │ commitIndex  │ 两阶段Proposal│
  │ 恢复          │ 日志回滚对齐  │ 专门恢复模式  │
  └──────────────┴──────────────┴──────────────┘

ZooKeeper 特点：
  强一致性读（Read from Leader / Sync）
  顺序一致性（同一客户端的操作有序）
  Watch 机制（数据变更通知）
  
  用途：
  配置管理、分布式锁、选举、服务发现
```

## 4. Gossip协议？

**回答：**

```
Gossip = 最终一致的去中心化协议
  类似流言传播

  过程：
  每个节点定期随机选择几个节点
  交换自己知道的信息
  指数级传播 → 最终所有节点都知道

  ┌───┐  gossip  ┌───┐
  │ A │─────────→│ B │
  └───┘          └───┘
    │               │
    │ gossip        │ gossip
    ↓               ↓
  ┌───┐          ┌───┐
  │ C │          │ D │
  └───┘          └───┘
  
  N个节点 → O(logN) 轮后全部知道

  三种模式：
  ┌──────────────┬──────────────────────────┐
  │ 模式          │ 说明                     │
  ├──────────────┼──────────────────────────┤
  │ Push          │ 推送自己知道的新信息     │
  │ Pull          │ 主动拉取对方的信息       │
  │ Push-Pull     │ 双向交换（最快收敛）     │
  └──────────────┴──────────────────────────┘

应用：
  Redis Cluster：节点间状态同步
  Cassandra：故障检测+数据修复
  Consul：成员管理（Serf库）

优点：去中心化 容错高 扩展性好
缺点：最终一致（有延迟）消息冗余
```

## 5. etcd 与 Raft 实践？

**回答：**

```
etcd = 基于 Raft 的分布式KV存储

  特点：
  强一致性（线性一致性读写）
  Watch 实时监听
  租约（Lease）自动过期
  多版本（MVCC）

  架构：
  ┌────────────────────────────────────┐
  │ etcd 节点                          │
  │ ┌──────────┐ ┌──────────────────┐ │
  │ │ gRPC API │ │ Raft Module      │ │
  │ └──────────┘ │ Leader Election  │ │
  │              │ Log Replication  │ │
  │ ┌──────────┐ └──────────────────┘ │
  │ │ BoltDB   │ ← 存储引擎(B+树)     │
  │ └──────────┘                      │
  └────────────────────────────────────┘

Go 使用 etcd：
  cli, _ := clientv3.New(clientv3.Config{
      Endpoints: []string{"localhost:2379"},
      DialTimeout: 5 * time.Second,
  })
  
  // Put
  cli.Put(ctx, "key", "value")
  
  // Get
  resp, _ := cli.Get(ctx, "key")
  
  // Watch
  wch := cli.Watch(ctx, "prefix/", clientv3.WithPrefix())
  for wresp := range wch {
      for _, ev := range wresp.Events {
          fmt.Printf("%s %s\n", ev.Type, ev.Kv.Key)
      }
  }
  
  // 分布式锁
  session, _ := concurrency.NewSession(cli)
  mutex := concurrency.NewMutex(session, "/lock/")
  mutex.Lock(ctx)
  defer mutex.Unlock(ctx)
```

## 6. 线性一致性？

**回答：**

```
线性一致性 = 最强的一致性模型
  效果等同于单台机器

  要求：
  1. 写操作在调用和返回之间的某个时刻生效
  2. 后续读一定能看到已完成的写
  3. 所有操作存在一个全局合法顺序

  Timeline：
  Client1: |--W(x=1)--|
  Client2:     |--R(x)→1--|     ✓ 合法（W已完成）
  Client3:       |--R(x)→0--|  ✗ 非法（W已完成但读到旧值）

实现方式：
  单 Leader 读写（Raft Leader 处理所有读写）
  但 Leader 读也可能不一致（刚选出的Leader）

  Raft 线性一致性读：
  方案1：ReadIndex
    Leader 确认自己仍是 Leader（发心跳）
    等待 commitIndex ≥ readIndex
    然后读取本地状态机
  
  方案2：Lease Read
    Leader 维护一个租约期
    租约内不会选出新 Leader
    直接读本地（更快 但依赖时钟）

etcd 默认 serializable 读（可能读到旧数据）
  加 clientv3.WithSerializable() 读从库
  不加 → 线性一致性读（走Leader）
```

## 7. 多副本一致性方案对比？

**回答：**

```
  ┌──────────────┬────────┬────────┬────────┬──────────┐
  │ 方案          │ 一致性 │ 性能   │ 可用性 │ 复杂度   │
  ├──────────────┼────────┼────────┼────────┼──────────┤
  │ 单主复制      │ 强     │ 写受限 │ 主切换 │ 低       │
  │ 多主复制      │ 最终   │ 写高   │ 高     │ 高(冲突) │
  │ 无主复制      │ Quorum │ 灵活   │ 高     │ 中       │
  │ Raft/Paxos   │ 强     │ 中     │ 多数OK │ 高       │
  │ Gossip       │ 最终   │ 高     │ 很高   │ 低       │
  └──────────────┴────────┴────────┴────────┴──────────┘

选择建议：
  配置/元数据 → Raft（etcd）
  用户数据读多写少 → 单主+读从库
  用户数据写多 → 分片+单主
  跨地域 → 多主/无主（处理冲突）
  节点状态同步 → Gossip

冲突解决：
  Last Write Wins（最后写入胜）
  → 简单但可能丢数据
  
  CRDTs（无冲突复制数据类型）
  → 自动合并 不丢数据
  → 计数器/集合/Map 等
  
  应用层解决
  → 检测冲突 交给用户决定
```

## 8. 两阶段提交（2PC）？

**回答：**

```
  2PC = 分布式事务的经典协议

  角色：Coordinator（协调者）+ Participants（参与者）

  Phase 1 - Prepare（投票）：
    Coordinator → 所有 Participant：准备好了吗？
    Participant：执行事务 但不提交 → 回复 Yes/No

  Phase 2 - Commit/Abort：
    如果所有 Yes → Coordinator 发 Commit
    如果有 No   → Coordinator 发 Abort

  ┌──────────────┐   Prepare   ┌──────────────┐
  │ Coordinator  │────────────→│ Participant  │
  └──────────────┘             └──────────────┘
         │          Yes/No            │
         │←───────────────────────────│
         │                            │
         │      Commit/Abort          │
         │───────────────────────────→│

问题：
  1. 阻塞：Participant 在 Prepare 后必须等 Commit/Abort
  2. 单点：Coordinator 挂了全部阻塞
  3. 数据不一致：Phase 2 部分节点收到 部分没收到

3PC 改进：
  增加 PreCommit 阶段 + 超时自动提交
  减少阻塞但更复杂

实际选择：
  能用 Saga 就不用 2PC
  Saga 更灵活 → 最终一致性
```

## 9. 一致性协议在工程中的选择？

**回答：**

```
  ┌──────────────────┬──────────────────────────┐
  │ 场景              │ 推荐协议                 │
  ├──────────────────┼──────────────────────────┤
  │ 配置管理/服务发现 │ Raft（etcd/Consul）      │
  │ 分布式锁/选主    │ Raft（etcd）             │
  │ 数据库复制       │ 单主Raft/Multi-Paxos     │
  │ 集群成员管理     │ Gossip（Serf/memberlist）│
  │ 跨地域同步       │ 多主复制+冲突解决        │
  │ 区块链           │ BFT（PBFT/Tendermint）   │
  └──────────────────┴──────────────────────────┘

  Go 中常用组件：
  etcd/raft：工业级 Raft 库
  hashicorp/raft：HashiCorp 的 Raft 实现
  hashicorp/memberlist：Gossip 库
  hashicorp/serf：基于 memberlist 的集群管理

使用 hashicorp/raft 构建一致性存储：
  // 实现 raft.FSM 接口
  type fsm struct {
      mu    sync.RWMutex
      store map[string]string
  }
  
  func (f *fsm) Apply(log *raft.Log) interface{} {
      var cmd Command
      json.Unmarshal(log.Data, &cmd)
      f.mu.Lock()
      f.store[cmd.Key] = cmd.Value
      f.mu.Unlock()
      return nil
  }

  // 写入通过 Leader
  raftNode.Apply(data, 10*time.Second)
```

## 10. 一致性协议面试速答？

**回答：**

```
Q: Paxos和Raft区别？
A: Paxos理论完备但难理解实现
   Raft分解为选举+日志复制 更工程化

Q: Raft怎么保证安全性？
A: 候选人日志必须至少和多数一样新
   已提交日志不会被覆盖

Q: Gossip协议特点？
A: 去中心化 最终一致
   O(logN)轮收敛 适合状态同步

Q: 2PC最大问题？
A: 阻塞和单点故障
   Coordinator挂了全部Participant卡住

Q: 线性一致性读怎么实现？
A: ReadIndex(确认Leader+等commitIndex)
   Lease Read(租约内直接读 更快)

Q: etcd为什么用Raft？
A: 强一致性需求(配置/锁/选主)
   Raft易理解 工业实现成熟

Q: 多主复制冲突怎么解决？
A: LWW(最后写入胜) 简单但可能丢数据
   CRDT自动合并 或应用层解决

Q: 什么时候用Gossip什么时候用Raft？
A: 状态同步/成员管理→Gossip(弱一致)
   配置/锁/选主→Raft(强一致)
```
