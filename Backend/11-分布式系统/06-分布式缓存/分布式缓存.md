# 分布式缓存

---

## 1. 分布式缓存架构？

**回答：**

```
  多级缓存架构：
  ┌────────┐
  │ Client │ → 客户端缓存（浏览器/App）
  └───┬────┘
      ↓
  ┌────────┐
  │  CDN   │ → 静态资源缓存
  └───┬────┘
      ↓
  ┌────────┐
  │ Nginx  │ → 接入层缓存（Lua+共享内存）
  └───┬────┘
      ↓
  ┌────────┐
  │ 本地    │ → 进程内缓存（BigCache/Ristretto）
  └───┬────┘
      ↓
  ┌────────┐
  │ Redis  │ → 分布式缓存
  └───┬────┘
      ↓
  ┌────────┐
  │  DB    │ → 持久存储
  └────────┘

缓存模式：
  Cache-Aside（旁路缓存）：应用管理读写，最常用
  Read-Through：缓存层自动加载
  Write-Through：同步写入缓存+DB
  Write-Behind：异步批量写DB
  Refresh-Ahead：主动刷新即将过期的缓存

Cache-Aside 读写：
  读：cache.Get → miss → db.Query → cache.Set
  写：db.Update → cache.Delete（先更新DB再删缓存）
```

## 2. Redis Cluster原理？

**回答：**

```
  Redis Cluster = 去中心化 + Hash Slot分片

  Hash Slot：
  16384个槽位
  CRC16(key) % 16384 → 确定槽位
  每个节点负责一部分槽位

  ┌──────────┐ ┌──────────┐ ┌──────────┐
  │ Master A │ │ Master B │ │ Master C │
  │ 0-5460   │ │ 5461-10922│ │10923-16383│
  │ Slave A' │ │ Slave B' │ │ Slave C' │
  └──────────┘ └──────────┘ └──────────┘

  Gossip协议：节点间通信 交换状态
  故障检测：多数节点认为某节点不可达 → PFAIL→FAIL
  自动故障转移：Slave提升为Master

  MOVED重定向：
  客户端访问错误节点 → 返回 MOVED slot ip:port
  客户端缓存slot→node映射 减少重定向

  扩缩容：
  添加节点 → 迁移部分slot到新节点
  redis-cli --cluster reshard

  限制：
  不支持跨slot事务（MULTI）
  多key操作必须在同一slot（用HashTag {tag}）
```

## 3. 缓存穿透/击穿/雪崩？

**回答：**

```
  ┌──────────┬────────────────────┬─────────────────┐
  │ 问题      │ 原因                │ 解决方案         │
  ├──────────┼────────────────────┼─────────────────┤
  │ 穿透      │ 查不存在的数据      │ 布隆过滤器/空值  │
  │ 击穿      │ 热点key过期         │ 互斥锁/永不过期  │
  │ 雪崩      │ 大量key同时过期     │ 随机过期/多级    │
  └──────────┴────────────────────┴─────────────────┘

穿透 - 布隆过滤器：
  func Get(key string) (string, error) {
      if !bloomFilter.Test(key) {
          return "", nil // 肯定不存在
      }
      val, err := cache.Get(key)
      if err == redis.Nil {
          val, err = db.Get(key)
          if val == "" {
              cache.Set(key, "", 60*time.Second) // 缓存空值
          }
      }
      return val, err
  }

击穿 - singleflight：
  var g singleflight.Group
  
  func GetHotData(key string) (interface{}, error) {
      val, err := cache.Get(key)
      if err == redis.Nil {
          v, err, _ := g.Do(key, func() (interface{}, error) {
              data, err := db.Get(key)
              cache.Set(key, data, time.Hour)
              return data, err
          })
          return v, err
      }
      return val, err
  }

雪崩 - 随机过期：
  ttl := baseTTL + rand.Intn(300) // 基础TTL + 随机0-300秒
```

## 4. 热Key与大Key？

**回答：**

```
热Key = 单个key被大量访问
  eg: 热门商品详情 热搜话题

  解决方案：
  1. 本地缓存：bigcache/ristretto 挡住大部分请求
  2. Key分片：key_1 key_2 ... key_N 分散读压力
  3. 读写分离：Redis从节点分担读
  4. 多级缓存：客户端→进程→Redis→DB

  热Key分片：
  func GetShardedKey(key string) string {
      n := rand.Intn(10) // 10个分片
      return fmt.Sprintf("%s_%d", key, n)
  }
  // 写入时写所有分片 读取时随机读一个

大Key = 单个key的value很大
  eg: 100万元素的Set  5MB的String

  危害：
  阻塞其他请求（Redis单线程）
  网络带宽占满
  主从同步延迟

  解决方案：
  1. 拆分：大Hash→按前缀拆成多个Hash
  2. 压缩：JSON→Protobuf/gzip
  3. 异步删除：UNLINK代替DEL
  4. 渐进式扫描：HSCAN/SSCAN 不用HGETALL

  发现大Key：
  redis-cli --bigkeys
  redis-cli --memkeys
  MEMORY USAGE key
```

## 5. 缓存一致性？

**回答：**

```
  经典问题：数据库更新后缓存如何同步？

  方案对比：
  ┌─────────────────────┬───────┬──────────────┐
  │ 方案                 │ 一致性│ 说明         │
  ├─────────────────────┼───────┼──────────────┤
  │ 先更新DB再删缓存     │ 较好  │ 推荐         │
  │ 先删缓存再更新DB     │ 差    │ 并发问题     │
  │ 先更新DB再更新缓存   │ 差    │ 并发覆盖     │
  │ 延迟双删             │ 较好  │ 删→更新→延迟删│
  │ 订阅Binlog           │ 好    │ Canal异步     │
  └─────────────────────┴───────┴──────────────┘

  先更新DB再删缓存（Cache-Aside）：
  func Update(ctx context.Context, id int, data Data) error {
      // 1. 更新数据库
      if err := db.Update(ctx, id, data); err != nil {
          return err
      }
      // 2. 删除缓存
      return cache.Del(ctx, cacheKey(id))
  }

  问题：删缓存失败怎么办？
  → 重试（消息队列异步重试）
  → 订阅Binlog（Canal监听→删缓存）

  延迟双删：
  func UpdateWithDoubleDelete(ctx context.Context, id int, data Data) error {
      cache.Del(ctx, cacheKey(id))        // 1. 先删缓存
      db.Update(ctx, id, data)            // 2. 更新DB
      go func() {
          time.Sleep(500 * time.Millisecond) // 3. 延迟
          cache.Del(ctx, cacheKey(id))      // 4. 再删缓存
      }()
      return nil
  }

  Canal方案：
  MySQL Binlog → Canal → MQ → 消费者删缓存
  最终一致 对业务无侵入
```

## 6. 进程内缓存？

**回答：**

```
  Go常用本地缓存库：
  ┌────────────┬──────────┬──────────┬─────────────┐
  │ 库          │ 淘汰策略 │ 并发安全 │ 特点         │
  ├────────────┼──────────┼──────────┼─────────────┤
  │ sync.Map   │ 无       │ 是       │ 标准库       │
  │ bigcache   │ FIFO     │ 是       │ 零GC 高性能  │
  │ ristretto  │ LFU(TinyLFU) │ 是  │ 高命中率     │
  │ freecache  │ LRU近似  │ 是       │ 零GC         │
  └────────────┴──────────┴──────────┴─────────────┘

bigcache示例：
  cache, _ := bigcache.New(bigcache.DefaultConfig(10 * time.Minute))
  cache.Set("key", []byte("value"))
  data, _ := cache.Get("key")

ristretto示例（推荐）：
  cache, _ := ristretto.NewCache(&ristretto.Config{
      NumCounters: 1e7,     // 跟踪频率的计数器
      MaxCost:     1 << 30, // 最大缓存大小
      BufferItems: 64,
  })
  cache.Set("key", "value", 1) // cost=1
  cache.Wait()                  // 等待写入完成
  val, found := cache.Get("key")

本地缓存要点：
  大小限制：防止OOM
  过期机制：避免数据陈旧
  一致性：集群多节点缓存不一致
  → 短过期时间 + 变更广播（Redis Pub/Sub）
```

## 7. 缓存预热与降级？

**回答：**

```
缓存预热 = 系统启动时提前加载缓存

  场景：
  服务重启后大量请求直接打DB → 预热避免冷启动

  方案：
  1. 启动时加载：查询热点数据 Set到缓存
  2. 定时预热：定时Job刷新即将过期的热点
  3. 流量预热：灰度释放流量 逐步加热缓存

  func WarmupCache(ctx context.Context) {
      // 加载TOP1000热门商品
      items, _ := db.GetTopItems(ctx, 1000)
      for _, item := range items {
          data, _ := json.Marshal(item)
          cache.Set(ctx, itemKey(item.ID), data, time.Hour)
      }
  }

缓存降级 = 缓存不可用时的兜底策略

  降级层次：
  1. Redis故障 → 读本地缓存
  2. 本地也没有 → 返回默认值/静态数据
  3. 写操作 → 降级为异步写

  func GetWithFallback(ctx context.Context, key string) (Data, error) {
      // 1. 尝试Redis
      val, err := redis.Get(ctx, key)
      if err == nil {
          return val, nil
      }
      // 2. 降级到本地缓存
      if v, ok := localCache.Get(key); ok {
          return v, nil
      }
      // 3. 降级到默认值
      return defaultData, nil
  }
```

## 8. 一致性Hash详解？

**回答：**

```
  一致性Hash = 解决节点增减时大量缓存失效

  原理：
  节点和Key都映射到0~2^32的环上
  Key顺时针找最近的节点

  ┌───────────────────────┐
  │        0              │
  │    N1 ●              │
  │       ↑ K1           │
  │                      │
  │ N3 ●         ● N2    │
  │       K3  K2↗        │
  │                      │
  └───────────────────────┘

  增加节点只影响相邻区间 大部分Key不变

  虚拟节点：
  解决数据倾斜 每个物理节点映射多个虚拟节点
  通常 100-200 个虚拟节点

Go实现：
  type ConsistentHash struct {
      ring     map[uint32]string // hash → node
      sorted   []uint32         // 排序的hash值
      replicas int              // 虚拟节点数
  }
  
  func (h *ConsistentHash) Add(node string) {
      for i := 0; i < h.replicas; i++ {
          hash := crc32.ChecksumIEEE(
              []byte(fmt.Sprintf("%s-%d", node, i)))
          h.ring[hash] = node
          h.sorted = append(h.sorted, hash)
      }
      sort.Slice(h.sorted, func(i, j int) bool {
          return h.sorted[i] < h.sorted[j]
      })
  }
  
  func (h *ConsistentHash) Get(key string) string {
      hash := crc32.ChecksumIEEE([]byte(key))
      idx := sort.Search(len(h.sorted), func(i int) bool {
          return h.sorted[i] >= hash
      })
      if idx >= len(h.sorted) {
          idx = 0
      }
      return h.ring[h.sorted[idx]]
  }
```

## 9. Redis高可用方案？

**回答：**

```
  三种方案对比：
  ┌──────────────┬──────────┬──────────┬──────────┐
  │ 方案          │ 数据分片 │ 自动故障转移│ 复杂度 │
  ├──────────────┼──────────┼──────────┼──────────┤
  │ 主从复制      │ 否       │ 否         │ 低     │
  │ Sentinel哨兵 │ 否       │ 是         │ 中     │
  │ Cluster集群  │ 是       │ 是         │ 高     │
  └──────────────┴──────────┴──────────┴──────────┘

  Sentinel：
  ┌─────────┐ ┌─────────┐ ┌─────────┐
  │Sentinel1│ │Sentinel2│ │Sentinel3│  >=3个奇数
  └────┬────┘ └────┬────┘ └────┬────┘
       │           │           │
  ┌────▼──────────▼──────────▼────┐
  │          Master                │
  │     ┌──────┐  ┌──────┐       │
  │     │Slave1│  │Slave2│       │
  │     └──────┘  └──────┘       │
  └────────────────────────────────┘

  Master故障 → Sentinel选举 → Slave提升
  客户端通过Sentinel获取Master地址

  Cluster（推荐大规模）：
  3主3从起步
  数据自动分片到16384个slot
  节点间Gossip通信
  故障自动切换

  选型：
  数据量小+单机能抗 → 主从+Sentinel
  数据量大/高并发 → Cluster
```

## 10. 分布式缓存面试速答？

**回答：**

```
Q: 缓存穿透怎么解决？
A: 布隆过滤器+缓存空值
   布隆过滤器预判key是否存在

Q: 缓存击穿怎么解决？
A: singleflight/互斥锁 合并并发请求
   或热点key永不过期 异步刷新

Q: 缓存雪崩怎么解决？
A: 随机过期时间+多级缓存+熔断降级
   避免大量key同时过期

Q: 如何保证缓存一致性？
A: 先更新DB再删缓存（Cache-Aside）
   失败重试或订阅Binlog异步删

Q: 热Key怎么处理？
A: 本地缓存挡流量+Key分片分散读
   ristretto/bigcache做进程内缓存

Q: 大Key怎么处理？
A: 拆分为多个小Key+UNLINK异步删除
   redis-cli --bigkeys扫描发现

Q: Redis Cluster怎么分片？
A: 16384个Hash Slot
   CRC16(key)%16384定位slot→节点

Q: 一致性Hash原理？
A: 节点和Key映射到环上 顺时针查找
   虚拟节点解决数据倾斜
```
