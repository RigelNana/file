# 分布式存储

---

## 1. 分布式存储概述？

**回答：**

```
  分布式存储 = 数据分散在多台机器 对外提供统一服务

  分类：
  ┌──────────────────┬──────────────────────────────┐
  │ 类型              │ 代表系统                     │
  ├──────────────────┼──────────────────────────────┤
  │ 分布式文件系统    │ HDFS、Ceph、MinIO            │
  │ 分布式KV存储      │ etcd、TiKV、Consul           │
  │ 分布式数据库      │ TiDB、CockroachDB、Spanner   │
  │ 分布式对象存储    │ S3、MinIO、Ceph RGW          │
  │ 分布式消息存储    │ Kafka、Pulsar                │
  └──────────────────┴──────────────────────────────┘

  核心挑战：
  数据分片：如何划分数据到不同节点
  数据复制：如何保证多副本一致
  故障处理：节点挂了如何恢复
  扩缩容：如何在线增减节点
```

## 2. 数据复制策略？

**回答：**

```
  ┌───────────────┬──────────┬──────────────────────┐
  │ 策略           │ 一致性   │ 说明                 │
  ├───────────────┼──────────┼──────────────────────┤
  │ 同步复制       │ 强       │ 写入所有副本才返回   │
  │ 异步复制       │ 弱       │ 写入主节点就返回     │
  │ 半同步复制     │ 较强     │ 写入多数节点返回     │
  └───────────────┴──────────┴──────────────────────┘

  同步复制：
  Client → Master → Slave1 + Slave2 → 全部ACK → 返回
  优点：强一致  缺点：延迟高 可用性受限

  异步复制：
  Client → Master → 返回  ← Master异步→Slave
  优点：延迟低  缺点：主挂了会丢数据

  半同步（Raft/Kafka ISR）：
  Client → Master → 多数节点ACK → 返回
  W + R > N → 保证能读到最新
  Kafka: min.insync.replicas=2, acks=all

  复制拓扑：
  1. 单主复制：Master写 Slave读（MySQL主从）
  2. 多主复制：多Master可写（多数据中心）
  3. 无主复制：任意节点可写（Cassandra/DynamoDB）
```

## 3. 数据分片策略？

**回答：**

```
  目标：均匀分布 + 扩缩容影响小

  1. Hash分片
     Hash(key) % N → 节点编号
     问题：扩容时几乎所有key重新映射
  
  2. 一致性Hash
     环形空间 顺时针查找最近节点
     虚拟节点解决倾斜（100~200个）
     扩容只影响相邻节点
  
  3. Range分片
     按范围划分 eg: [A-M]→节点1 [N-Z]→节点2
     适合范围查询 但可能热点
  
  4. Hash Slot（Redis Cluster方式）
     固定16384个slot
     CRC16(key) % 16384 → slot → 节点
     扩容时迁移部分slot

  对比：
  ┌──────────────┬──────────┬──────────┬──────────┐
  │ 方式          │ 均匀性   │ 扩缩容   │ 范围查询 │
  ├──────────────┼──────────┼──────────┼──────────┤
  │ Hash取模      │ 好       │ 差       │ 不支持   │
  │ 一致性Hash    │ 好(虚拟) │ 好       │ 不支持   │
  │ Range         │ 可能倾斜 │ 好       │ 支持     │
  │ Hash Slot     │ 好       │ 好       │ 不支持   │
  └──────────────┴──────────┴──────────┴──────────┘

  热点分片处理：
  记录读写计数 → 发现热分片 → 自动拆分
  TiDB Region：默认96MB 自动分裂
```

## 4. HDFS架构？

**回答：**

```
  HDFS = Hadoop Distributed File System

  ┌─────────────┐
  │  NameNode   │ 主节点（元数据）
  │  文件→Block  │ 文件名/目录/Block位置
  │  映射关系    │
  └──────┬──────┘
         │
  ┌──────┼──────┬──────────────┐
  ┌──────▼──┐ ┌─▼────────┐ ┌──▼───────┐
  │DataNode1│ │DataNode2 │ │DataNode3 │
  │Block A  │ │Block A   │ │Block B   │
  │Block B  │ │Block C   │ │Block C   │
  └─────────┘ └──────────┘ └──────────┘

  核心特点：
  Block大小：默认128MB（大文件友好）
  复制因子：默认3副本
  一次写入多次读取（不支持修改）
  NameNode单点 → Standby NameNode
  
  写入流程：
  1. Client请求NameNode → 分配Block+DataNode列表
  2. Client → DataNode1 → DataNode2 → DataNode3（Pipeline）
  3. 全部ACK → 写入成功

  适用场景：大文件批处理（MapReduce/Spark）
  不适用：小文件/随机读写/低延迟访问
```

## 5. 对象存储？

**回答：**

```
  对象存储 = 海量非结构化数据存储

  核心概念：
  Bucket（桶）→ Object（对象）→ Data + Metadata

  S3 API（事实标准）：
  PUT    /bucket/key   → 上传对象
  GET    /bucket/key   → 下载对象
  DELETE /bucket/key   → 删除对象
  LIST   /bucket       → 列出对象

  MinIO = 开源S3兼容对象存储
  特点：Go编写 轻量 兼容S3 API

Go SDK：
  client, _ := minio.New("localhost:9000", &minio.Options{
      Creds:  credentials.NewStaticV4("access", "secret", ""),
      Secure: false,
  })
  
  // 上传
  client.PutObject(ctx, "bucket", "key", reader, size,
      minio.PutObjectOptions{ContentType: "image/png"})
  
  // 下载
  obj, _ := client.GetObject(ctx, "bucket", "key", minio.GetObjectOptions{})
  
  // 预签名URL（有效期7天）
  url, _ := client.PresignedGetObject(ctx, "bucket", "key",
      7*24*time.Hour, nil)

  选型：
  云上 → S3/OSS/COS（推荐）
  私有化 → MinIO（S3兼容 运维简单）
  大规模 → Ceph RGW
```

## 6. 纠删码与多副本？

**回答：**

```
  多副本：数据存N份（通常3份）
  纠删码：数据拆分+编码 用更少空间容错

  多副本（Replication）：
  原始数据D → D D D（3副本）
  存储开销：3x
  容错：可丢2份 最多N-1份
  恢复：直接复制

  纠删码（Erasure Coding）：
  原始数据拆为k块 编码为m块校验
  总共k+m块 任意丢m块都能恢复
  
  例：RS(4,2) → 4数据块+2校验块
  存储开销：1.5x（比3副本省一半）
  容错：可丢任意2块

  对比：
  ┌──────────┬──────────┬──────────┬──────────┐
  │ 方案      │ 存储开销 │ 恢复速度 │ 适用     │
  ├──────────┼──────────┼──────────┼──────────┤
  │ 3副本     │ 300%     │ 快       │ 热数据   │
  │ RS(4,2)  │ 150%     │ 较慢     │ 温/冷数据│
  │ RS(10,4) │ 140%     │ 慢       │ 冷数据   │
  └──────────┴──────────┴──────────┴──────────┘

  实际应用：
  HDFS: 3副本（默认）+ EC策略（冷数据）
  Ceph: 副本池 + EC池
  MinIO: 内置EC
```

## 7. 分布式数据库？

**回答：**

```
  分布式数据库 = 数据分散+SQL引擎+分布式事务

  NewSQL代表：
  ┌──────────────┬──────────────────────────────┐
  │ 系统          │ 特点                         │
  ├──────────────┼──────────────────────────────┤
  │ TiDB          │ MySQL兼容 Raft+MVCC          │
  │ CockroachDB   │ PG兼容 Raft+分布式事务       │
  │ Spanner       │ Google TrueTime 全球一致     │
  │ OceanBase     │ 蚂蚁金融级 Paxos             │
  └──────────────┴──────────────────────────────┘

  TiDB架构：
  ┌────────┐ ┌────────┐ ┌────────┐
  │ TiDB   │ │ TiDB   │ │ TiDB   │ SQL层（无状态）
  └───┬────┘ └───┬────┘ └───┬────┘
      │          │          │
  ┌───▼──────────▼──────────▼───┐
  │           PD (调度)          │ 元数据+调度
  └───┬──────────┬──────────┬───┘
  ┌───▼────┐ ┌───▼────┐ ┌───▼────┐
  │ TiKV   │ │ TiKV   │ │ TiKV   │ 存储层
  │ Region │ │ Region │ │ Region │ Raft组
  └────────┘ └────────┘ └────────┘

  Region = TiKV的数据分片单元
  每个Region对应一个Raft组（3副本）
  自动分裂/合并
  PD负责调度Region到不同TiKV节点
```

## 8. 分布式存储一致性？

**回答：**

```
  一致性模型层次：

  强一致性（Linearizability）：
  读到最新写入 如单机
  代价高 需要共识协议
  代表：etcd  ZooKeeper  Spanner

  顺序一致性（Sequential）：
  所有操作按某全局顺序执行
  不保证实时性

  因果一致性（Causal）：
  有因果关系的操作保序
  并发操作不保序

  最终一致性（Eventual）：
  停止写入后 最终所有副本一致
  代表：Cassandra  DynamoDB  S3

  读一致性保证：
  ┌────────────────────┬──────────────────────┐
  │ 保证               │ 实现                 │
  ├────────────────────┼──────────────────────┤
  │ Read-after-Write   │ 读主/版本号检查      │
  │ Monotonic Reads    │ 固定版本/Session粘滞 │
  │ Bounded Staleness  │ 最大延迟限制         │
  └────────────────────┴──────────────────────┘

  工程选择：
  配置/元数据 → 强一致（etcd/ZooKeeper）
  业务数据 → 最终一致+对账
  用户可见数据 → Read-after-Write
```

## 9. 分布式存储选型？

**回答：**

```
  按场景选型：

  键值存储：
  配置/服务发现 → etcd（Raft强一致 <5GB）
  缓存 → Redis Cluster
  大规模KV → TiKV / Cassandra

  关系数据：
  MySQL兼容 → TiDB
  PG兼容 → CockroachDB
  海量分析 → ClickHouse

  文件/对象：
  云上 → S3/OSS
  私有化 → MinIO（轻量）/ Ceph（重量级）
  大数据 → HDFS

  消息/日志：
  高吞吐 → Kafka
  低延迟 → Pulsar
  有序日志 → Kafka（Partition有序）

  选型原则：
  1. 优先用云服务 减少运维
  2. 数据量决定架构（<100GB单机够用）
  3. 读写比决定方案（读多→缓存 写多→分片）
  4. 一致性要求决定复制策略
  5. 团队能力决定复杂度上限
```

## 10. 分布式存储面试速答？

**回答：**

```
Q: 数据复制有哪些策略？
A: 同步(强一致高延迟) 异步(低延迟可能丢)
   半同步(多数ACK 如Raft W+R>N)

Q: 数据分片怎么做？
A: Hash取模/一致性Hash/Range/Hash Slot
   推荐一致性Hash或Hash Slot

Q: HDFS架构？
A: NameNode管元数据 DataNode存Block
   默认128MB Block 3副本 Pipeline写入

Q: 纠删码和多副本区别？
A: 多副本=3份存3x空间 EC=编码存1.5x
   热数据用副本 冷数据用EC省空间

Q: TiDB架构？
A: TiDB(无状态SQL) + PD(调度) + TiKV(存储)
   Region为分片单元 每个是Raft组

Q: 对象存储核心API？
A: PUT/GET/DELETE/LIST
   S3是事实标准 MinIO开源兼容

Q: 怎么选分布式存储？
A: 按场景：KV→etcd/Redis 关系→TiDB
   文件→MinIO/S3 消息→Kafka

Q: 一致性模型有哪些？
A: 强一致(Linearizable) 顺序 因果
   最终一致 按业务需求选择
```
