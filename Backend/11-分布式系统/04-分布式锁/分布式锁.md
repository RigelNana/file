# 分布式锁

---

## 1. 分布式锁的需求与特性？

**回答：**

```
  单机锁（sync.Mutex）在分布式环境下无效
  多个进程/机器需要互斥访问共享资源

  特性要求：
  ┌──────────────┬──────────────────────────────┐
  │ 特性          │ 说明                         │
  ├──────────────┼──────────────────────────────┤
  │ 互斥          │ 同一时刻只有一个持有者       │
  │ 可重入        │ 同一持有者可重复加锁         │
  │ 超时释放      │ 持有者宕机后自动释放         │
  │ 高可用        │ 锁服务本身不单点             │
  │ 高性能        │ 加锁解锁操作快               │
  └──────────────┴──────────────────────────────┘

  实现方式：
  ┌──────────────┬────────┬────────┬──────────┐
  │ 方式          │ 性能   │ 可靠性 │ 复杂度   │
  ├──────────────┼────────┼────────┼──────────┤
  │ Redis         │ 高     │ 中     │ 低       │
  │ etcd          │ 中     │ 高     │ 中       │
  │ ZooKeeper     │ 中     │ 高     │ 中       │
  │ MySQL         │ 低     │ 中     │ 低       │
  └──────────────┴────────┴────────┴──────────┘
```

## 2. Redis分布式锁？

**回答：**

```
基本实现：SET key val NX EX

  加锁：
  SET lock:order:123 <random_value> NX EX 30
  NX：不存在才设置（互斥）
  EX：30秒过期（防死锁）
  random_value：随机值（防误释放）

  释放锁（Lua原子操作）：
  if redis.call("GET", KEYS[1]) == ARGV[1] then
      return redis.call("DEL", KEYS[1])
  else
      return 0
  end

Go 实现：
  func (l *RedisLock) Lock(ctx context.Context) error {
      l.value = uuid.New().String()
      ok, err := l.rdb.SetNX(ctx, l.key, l.value, l.ttl).Result()
      if err != nil {
          return err
      }
      if !ok {
          return ErrLockFailed
      }
      return nil
  }
  
  func (l *RedisLock) Unlock(ctx context.Context) error {
      script := redis.NewScript(`
          if redis.call("GET", KEYS[1]) == ARGV[1] then
              return redis.call("DEL", KEYS[1])
          end
          return 0
      `)
      return script.Run(ctx, l.rdb, []string{l.key}, l.value).Err()
  }

为什么要随机值？
  A 加锁 → A 超时释放 → B 加锁 → A 释放（误删B的锁）
  用随机值：A 只能释放自己的锁
```

## 3. Redis锁续期？

**回答：**

```
  问题：业务执行时间超过锁过期时间
  → 锁自动释放 其他人可以加锁 → 并发问题

  看门狗（Watchdog）机制：
  加锁成功后启动后台goroutine
  定期续期（每过期时间的1/3续一次）
  业务完成后停止续期

Go 实现：
  func (l *RedisLock) Lock(ctx context.Context) error {
      l.value = uuid.New().String()
      ok, _ := l.rdb.SetNX(ctx, l.key, l.value, l.ttl).Result()
      if !ok {
          return ErrLockFailed
      }
      
      // 启动看门狗
      l.stopCh = make(chan struct{})
      go l.watchdog()
      return nil
  }
  
  func (l *RedisLock) watchdog() {
      ticker := time.NewTicker(l.ttl / 3)
      defer ticker.Stop()
      for {
          select {
          case <-ticker.C:
              // 续期：只续期自己的锁
              script := `
                  if redis.call("GET", KEYS[1]) == ARGV[1] then
                      return redis.call("PEXPIRE", KEYS[1], ARGV[2])
                  end
                  return 0`
              redis.NewScript(script).Run(context.Background(),
                  l.rdb, []string{l.key}, l.value, l.ttl.Milliseconds())
          case <-l.stopCh:
              return
          }
      }
  }
  
  func (l *RedisLock) Unlock(ctx context.Context) error {
      close(l.stopCh) // 停止看门狗
      // Lua 释放锁...
  }
```

## 4. Redlock算法？

**回答：**

```
  问题：单 Redis 主从切换时锁可能丢失
  A在Master加锁 → Master宕机 → Slave升主(没有锁) → B也加锁

  Redlock（Redis官方方案）：
  使用 N 个独立Redis实例（推荐5个）

  加锁过程：
  1. 记录起始时间 T1
  2. 依次向 N 个实例加锁（相同key+value+TTL）
  3. 加锁成功的实例数 ≥ N/2+1（多数）
  4. 加锁耗时 T2-T1 < TTL
  5. 锁的有效时间 = TTL - (T2-T1)
  → 同时满足3,4 → 加锁成功

  ┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐
  │R1 │ │R2 │ │R3 │ │R4 │ │R5 │
  │ ✓ │ │ ✓ │ │ ✗ │ │ ✓ │ │ ✗ │
  └───┘ └───┘ └───┘ └───┘ └───┘
  3/5 成功 ≥ 3 → 加锁成功

  释放锁：向所有实例发送释放

争议（Martin Kleppmann的质疑）：
  依赖系统时钟 → 时钟跳变可能导致问题
  GC暂停可能导致锁过期但客户端不知道
  
  如果要求绝对正确 → 用 etcd/ZooKeeper
  Redis锁适合大多数场景（性能优先）
```

## 5. etcd分布式锁？

**回答：**

```
  基于 Raft 强一致性 → 不存在主从丢锁问题

  原理：
  创建带Lease的Key → 谁先创建谁持有锁
  Lease 自动续期 → 客户端宕机自动释放
  Watch 机制 → 等待锁释放

Go 实现（官方 concurrency 包）：
  import "go.etcd.io/etcd/client/v3/concurrency"
  
  cli, _ := clientv3.New(clientv3.Config{
      Endpoints: []string{"localhost:2379"},
  })
  
  // 创建 session（自动续期Lease）
  session, _ := concurrency.NewSession(cli, 
      concurrency.WithTTL(10))
  defer session.Close()
  
  // 创建锁
  mutex := concurrency.NewMutex(session, "/lock/order/123")
  
  // 加锁（阻塞直到获取）
  if err := mutex.Lock(ctx); err != nil {
      return err
  }
  defer mutex.Unlock(ctx)
  
  // 业务逻辑
  doBusinessLogic()

  TryLock（非阻塞）：
  if err := mutex.TryLock(ctx); err != nil {
      // 未获取到锁
      return err
  }

etcd锁 vs Redis锁：
  etcd：强一致 不丢锁 性能稍低
  Redis：性能高 有极小概率丢锁
  
  金融级 → etcd
  一般业务 → Redis
```

## 6. 可重入锁？

**回答：**

```
  可重入 = 同一个持有者可以多次加锁
  内部用计数器 加锁+1 解锁-1 到0才释放

  Redis 可重入锁（Lua）：
  -- 加锁
  local key = KEYS[1]
  local owner = ARGV[1]
  local ttl = ARGV[2]
  
  if redis.call("EXISTS", key) == 0 then
      redis.call("HSET", key, owner, 1)
      redis.call("PEXPIRE", key, ttl)
      return 1
  end
  if redis.call("HEXISTS", key, owner) == 1 then
      redis.call("HINCRBY", key, owner, 1)
      redis.call("PEXPIRE", key, ttl)
      return 1
  end
  return 0
  
  -- 解锁
  if redis.call("HEXISTS", key, owner) == 0 then
      return 0
  end
  local count = redis.call("HINCRBY", key, owner, -1)
  if count <= 0 then
      redis.call("DEL", key)
  end
  return 1

使用 Hash 存储：
  Key: lock:resource
  Field: owner_id
  Value: 重入次数

注意：
  实际开发中很少需要可重入
  如果调用链路需要同一个锁
  → 将锁传递下去比可重入更清晰
```

## 7. 读写锁？

**回答：**

```
  读锁共享 写锁互斥
  读多写少场景性能更好

  Redis 实现思路：
  读锁：INCR readers_count
  写锁：SET write_lock + 检查 readers_count == 0

  etcd 实现（官方支持）：
  import "go.etcd.io/etcd/client/v3/concurrency"
  
  // 读写锁
  rwMutex := concurrency.NewRWMutex(session, "/rwlock/config")
  
  // 读锁（多个可同时持有）
  rwMutex.RLock(ctx)
  config := readConfig()
  rwMutex.RUnlock(ctx)
  
  // 写锁（独占）
  rwMutex.Lock(ctx)
  writeConfig(newConfig)
  rwMutex.Unlock(ctx)

应用场景：
  配置更新（写少读多）
  缓存刷新（写时独占 读时共享）
  库存查询 vs 扣减
```

## 8. 分布式锁的常见陷阱？

**回答：**

```
  ┌──────────────┬──────────────────────────────┐
  │ 陷阱          │ 解决                         │
  ├──────────────┼──────────────────────────────┤
  │ 死锁          │ 必须设过期时间               │
  │ 误释放        │ 随机值+Lua原子释放           │
  │ 锁过期        │ 看门狗续期                   │
  │ 主从丢锁      │ Redlock/etcd                 │
  │ 锁粒度太大    │ 细粒度Key（lock:order:123）  │
  │ 锁等待方式    │ 重试间隔+超时 不要spin       │
  └──────────────┴──────────────────────────────┘

  锁等待最佳实践：
  func AcquireLock(ctx context.Context, key string) error {
      for i := 0; i < maxRetries; i++ {
          if tryLock(ctx, key) {
              return nil
          }
          // 指数退避 + 随机抖动
          backoff := time.Duration(1<<i) * 100 * time.Millisecond
          jitter := time.Duration(rand.Intn(50)) * time.Millisecond
          
          select {
          case <-ctx.Done():
              return ctx.Err()
          case <-time.After(backoff + jitter):
          }
      }
      return ErrLockTimeout
  }

  Fencing Token（防脑裂）：
  加锁时获取一个递增Token
  操作资源时携带Token
  资源端检查Token是否最新
  → 过期锁的操作会被拒绝
```

## 9. MySQL分布式锁？

**回答：**

```
  方案1：唯一索引
  CREATE TABLE distributed_lock (
      lock_key VARCHAR(64) PRIMARY KEY,
      owner    VARCHAR(64),
      expire_at TIMESTAMP
  );
  
  加锁：INSERT（成功=获取锁）
  解锁：DELETE WHERE lock_key=? AND owner=?
  过期清理：定时删除 expire_at < NOW()

  方案2：FOR UPDATE
  SELECT * FROM lock_table 
  WHERE lock_key = 'order:123' FOR UPDATE;
  -- 事务结束自动释放

  方案3：GET_LOCK()
  SELECT GET_LOCK('lock_name', 10);  -- 10秒超时
  -- 业务操作
  SELECT RELEASE_LOCK('lock_name');

缺点：
  性能差（数据库IO）
  可用性依赖数据库
  行锁可能升级为表锁

适用：
  已有MySQL 不想引入Redis/etcd
  对性能要求不高
  小规模系统
```

## 10. 分布式锁面试速答？

**回答：**

```
Q: Redis分布式锁怎么实现？
A: SET key val NX EX + 随机值
   释放用Lua保证原子性

Q: 为什么要随机值？
A: 防止误释放别人的锁
   释放时检查值是否是自己的

Q: 锁过期了业务没完成怎么办？
A: 看门狗机制 后台goroutine定期续期
   每过期时间1/3续一次

Q: Redis主从切换锁丢失？
A: Redlock(5个独立Redis多数加锁)
   或用etcd(Raft强一致)

Q: etcd锁原理？
A: Lease+Watch 基于Raft强一致
   Lease自动续期 宕机自动释放

Q: Redis锁和etcd锁怎么选？
A: 一般业务→Redis(性能高)
   金融级→etcd(强一致不丢锁)

Q: 分布式锁的陷阱？
A: 死锁(设TTL) 误释放(随机值)
   锁过期(看门狗) 粒度太大(细化key)

Q: Fencing Token是什么？
A: 加锁时获取递增Token
   资源端校验Token防止过期锁操作
```
