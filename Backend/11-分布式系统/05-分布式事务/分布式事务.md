# 分布式事务

---

## 1. 分布式事务的挑战？

**回答：**

```
单机事务：ACID由数据库保证
分布式事务：跨多个服务/数据库 无法靠单DB保证

  场景：下单 = 创建订单 + 扣库存 + 扣余额
  三个操作在三个服务/数据库 → 要么全成功要么全失败

  方案概览：
  ┌──────────────┬────────┬────────┬──────────┐
  │ 方案          │ 一致性 │ 性能   │ 侵入性   │
  ├──────────────┼────────┼────────┼──────────┤
  │ 2PC           │ 强     │ 差     │ 低       │
  │ TCC           │ 强     │ 中     │ 高       │
  │ Saga          │ 最终   │ 好     │ 中       │
  │ 本地消息表    │ 最终   │ 好     │ 中       │
  │ 事务消息      │ 最终   │ 好     │ 低       │
  └──────────────┴────────┴────────┴──────────┘

  原则：能不用分布式事务就不用
  优先最终一致性方案
```

## 2. TCC模式？

**回答：**

```
TCC = Try - Confirm - Cancel

  Try：预留资源（冻结库存/冻结余额）
  Confirm：确认提交（扣减冻结的资源）
  Cancel：取消回滚（释放冻结的资源）

  流程：
  ┌──────────────┐
  │ TCC协调者     │
  │ 1. Try All   │→ 订单Try + 库存Try + 余额Try
  │    全部OK?    │
  │ 2a. Confirm  │→ 订单Confirm + 库存Confirm + 余额Confirm
  │ 2b. Cancel   │→ 订单Cancel + 库存Cancel + 余额Cancel
  └──────────────┘

Go 接口设计：
  type TCCService interface {
      Try(ctx context.Context, req Request) error
      Confirm(ctx context.Context, req Request) error
      Cancel(ctx context.Context, req Request) error
  }

库存TCC示例：
  Try：UPDATE stock SET frozen=frozen+1 WHERE available>=1
  Confirm：UPDATE stock SET frozen=frozen-1, sold=sold+1
  Cancel：UPDATE stock SET frozen=frozen-1, available=available+1

TCC问题：
  空回滚：Try没执行 直接Cancel → Cancel要判断
  悬挂：Cancel先到 Try后到 → Try要检查已Cancel
  幂等：Confirm/Cancel可能重试 → 必须幂等

  → 用事务状态表记录每步状态
```

## 3. Saga模式？

**回答：**

```
  Saga = 一系列本地事务 + 补偿操作

  正向流程：T1 → T2 → T3
  补偿流程：T3失败 → C2 → C1

  编舞 Choreography：
  ┌─────┐ Event  ┌─────┐ Event  ┌─────┐
  │ T1  │───────→│ T2  │───────→│ T3  │
  └─────┘        └─────┘        └─────┘
  各服务自行监听事件 无中心

  编排 Orchestration：
  ┌───────────────────────┐
  │ Saga Orchestrator     │
  │  Step1: CreateOrder   │──→ OK
  │  Step2: DeductStock   │──→ OK
  │  Step3: Debit         │──→ FAIL
  │  Compensate:          │
  │    RestoreStock       │
  │    CancelOrder        │
  └───────────────────────┘

Go 编排器：
  type SagaStep struct {
      Name       string
      Action     func(ctx context.Context) error
      Compensate func(ctx context.Context) error
  }
  
  func RunSaga(ctx context.Context, steps []SagaStep) error {
      executed := make([]SagaStep, 0, len(steps))
      for _, step := range steps {
          if err := step.Action(ctx); err != nil {
              // 逆序补偿已执行步骤
              for i := len(executed) - 1; i >= 0; i-- {
                  if cErr := executed[i].Compensate(ctx); cErr != nil {
                      log.Error("compensate failed", "step", executed[i].Name)
                  }
              }
              return err
          }
          executed = append(executed, step)
      }
      return nil
  }

Saga要点：
  每步必须幂等
  补偿操作必须可靠（重试保证）
  需要对账机制兜底
```

## 4. 本地消息表？

**回答：**

```
  核心思想：利用本地事务保证业务操作和消息写入的原子性

  ┌─────────────────────────────┐
  │ 同一个数据库事务              │
  │  INSERT INTO orders (...)   │
  │  INSERT INTO messages (...) │ 消息表
  └─────────────────────────────┘
      ↓ 定时扫描
  ┌──────────┐    ┌──────────┐    ┌──────────┐
  │ Message  │───→│   MQ     │───→│ Consumer │
  │ Scanner  │    │          │    │ 扣库存   │
  └──────────┘    └──────────┘    └──────────┘

消息表：
  CREATE TABLE outbox_messages (
      id         BIGINT PRIMARY KEY,
      event_type VARCHAR(64),
      payload    TEXT,
      status     TINYINT DEFAULT 0,  -- 0待发送 1已发送
      retry_count INT DEFAULT 0,
      created_at TIMESTAMP,
      updated_at TIMESTAMP
  );

Go 实现：
  func CreateOrder(ctx context.Context, tx *sql.Tx, order Order) error {
      // 1. 业务操作
      tx.ExecContext(ctx, "INSERT INTO orders ...", ...)
      
      // 2. 写消息表（同一事务）
```

## 5. 事务消息？

**回答：**

```
  RocketMQ 原生支持事务消息

  流程：
  1. Producer 发送 Half Message（对Consumer不可见）
  2. Broker 返回 OK
  3. Producer 执行本地事务
  4. 本地事务成功 → Commit（Consumer可见）
     本地事务失败 → Rollback
  5. Broker 定时回查未决事务

  ┌──────────┐  Half Msg   ┌────────┐
  │ Producer │────────────→│ Broker │
  └─────┬────┘             └────┬───┘
        │                       │
   执行本地事务              暂存消息
        │                       │
   Commit/Rollback             │
        │───────────────────→  │
                               │
                          消息可见/删除
                               │
                          ┌────▼───┐
                          │Consumer│
                          └────────┘

  回查机制：
  Broker 发现 Half Message 超时未 Commit
  → 回调 Producer 检查本地事务状态
  → 根据结果 Commit 或 Rollback

  优势：
  不需要消息表
  不需要定时扫描
  MQ 内置支持 可靠性高
```

## 6. Seata分布式事务框架？

**回答：**

```
  Seata = 阿里开源分布式事务框架

  支持模式：
  ┌──────────┬──────────────────────────────┐
  │ 模式      │ 说明                         │
  ├──────────┼──────────────────────────────┤
  │ AT        │ 自动补偿 无侵入（推荐）      │
  │ TCC       │ 手动编写Try/Confirm/Cancel   │
  │ Saga      │ 长事务 编排/编舞             │
  │ XA        │ 基于数据库XA协议             │
  └──────────┴──────────────────────────────┘

  AT模式原理：
  1. 解析SQL → 记录前镜像(Before Image)
  2. 执行SQL
  3. 记录后镜像(After Image)
  4. 生成undo_log
  5. 提交本地事务 + undo_log
  
  回滚时：根据undo_log做反向SQL
  
  优势：对业务无侵入 只加注解
  劣势：有全局锁 性能开销

  Go 生态：
  Seata 主要面向 Java（注解驱动）
  Go 推荐用 DTM（分布式事务管理器）
  
  DTM 支持：Saga/TCC/XA/二阶段消息
  接入简单 通过HTTP/gRPC调用
```

## 7. 最终一致性保障？

**回答：**

```
最终一致性四件套：

  1. 可靠投递
     本地消息表 / 事务消息
     确保消息一定发出

  2. 幂等消费
     消费者处理同一消息多次结果相同
     方法：唯一索引/去重表/状态机

  3. 重试机制
     消费失败重试（指数退避）
     配置最大重试次数
     超过次数进死信队列

  4. 对账补偿
     定期对比各系统数据
     发现不一致 → 修复

  ┌──────────┐    ┌──────┐    ┌──────────┐
  │ Producer │───→│  MQ  │───→│ Consumer │
  │ 可靠投递  │    │ 重试  │    │ 幂等消费  │
  └──────────┘    └──────┘    └──────────┘
                                    │
                              ┌─────▼─────┐
                              │ 对账补偿   │ 定时任务
                              └───────────┘

Go 幂等消费：
  func HandleMessage(ctx context.Context, msg Message) error {
      // 利用唯一索引保证幂等
      _, err := db.ExecContext(ctx,
          "INSERT IGNORE INTO processed_msgs (msg_id) VALUES (?)",
          msg.ID)
      if err != nil {
          return err // DB错误
      }
      // affected==0 说明已处理过
      // 执行业务逻辑
      return processOrder(ctx, msg)
  }
```

## 8. 分布式事务选型？

**回答：**

```
  选型决策树：
  
  需要强一致？
  ├── 是 → 性能要求高？
  │       ├── 是 → TCC
  │       └── 否 → 2PC/XA
  └── 否 → 最终一致
          ├── 有MQ → 事务消息 / 本地消息表
          └── 跨多服务 → Saga

  实际经验：
  ┌──────────────┬──────────────────────────────┐
  │ 场景          │ 推荐方案                     │
  ├──────────────┼──────────────────────────────┤
  │ 订单+支付     │ Saga + 对账                  │
  │ 订单+库存     │ 本地消息表 / 事务消息        │
  │ 转账          │ TCC / Saga                   │
  │ 跨库写入      │ Saga / XA(性能不敏感)        │
  │ 异步通知      │ 本地消息表                   │
  └──────────────┴──────────────────────────────┘

  90% 的场景：Saga + 幂等 + 对账
  不要追求强一致 除非业务强制要求
```

## 9. 对账与补偿设计？

**回答：**

```
  对账 = 分布式事务的最后防线

  设计思路：
  1. 每个服务记录操作流水
  2. 对账Job定期比对各服务流水
  3. 发现不一致 → 触发补偿

  对账表设计：
  CREATE TABLE biz_journal (
      id          BIGINT PRIMARY KEY,
      biz_type    VARCHAR(32),
      biz_id      VARCHAR(64),    -- 业务ID（订单号）
      action      VARCHAR(32),    -- 操作（create/pay/ship）
      status      VARCHAR(16),    -- 状态
      amount      BIGINT,
      created_at  TIMESTAMP
  );

Go 对账任务：
  func Reconcile(orderDB, stockDB *sql.DB) {
      // 查询昨天的订单
      orders := queryOrders(orderDB, yesterday)
      // 查询库存扣减记录
      deductions := queryDeductions(stockDB, yesterday)
      
      // 比对
      for _, order := range orders {
          if !deductions.Contains(order.ID) {
              // 订单存在但库存未扣减 → 补偿
              log.Warn("inconsistency found", "orderID", order.ID)
              compensate(order)
          }
      }
  }

  对账频率建议：
  T+1 日对账（覆盖绝大多数场景）
  实时对账（资金相关 10分钟级）

  补偿方式：
  自动补偿：规则明确 自动修复
  人工介入：规则复杂 告警+工单
```

## 10. 分布式事务面试速答？

**回答：**

```
Q: 分布式事务有哪些方案？
A: 2PC(强一致) TCC(强一致) Saga(最终一致)
   本地消息表/事务消息(最终一致)

Q: TCC三个阶段？
A: Try=预留资源 Confirm=确认提交
   Cancel=取消回滚 注意空回滚和悬挂

Q: Saga和TCC区别？
A: TCC有Try预留 可回滚 强一致
   Saga无预留 靠补偿 最终一致

Q: 本地消息表原理？
A: 业务操作和消息写入同一个事务
   定时扫描发送到MQ→消费者处理

Q: 最终一致性怎么保证？
A: 可靠投递+幂等消费+重试+对账
   四件套缺一不可

Q: 幂等怎么实现？
A: 唯一索引/去重表/状态机
   同一操作多次执行结果相同

Q: 怎么选方案？
A: 90%场景Saga+幂等+对账
   不要追求强一致 除非业务强制

Q: 对账怎么做？
A: 各服务记录流水 定期比对
   不一致→自动补偿/人工介入
```
