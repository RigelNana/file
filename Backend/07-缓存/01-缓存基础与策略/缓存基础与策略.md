# 缓存基础与策略

---

## 1. 缓存的核心价值和适用场景？

**回答：**

```
缓存价值：
  ┌──────────────┬──────────────┬──────────────┐
  │ 存储介质      │ 读取速度     │ 典型代表      │
  ├──────────────┼──────────────┼──────────────┤
  │ CPU 寄存器   │ < 1ns       │ -             │
  │ L1 Cache     │ ~1ns        │ -             │
  │ L2 Cache     │ ~3ns        │ -             │
  │ L3 Cache     │ ~10ns       │ -             │
  │ 内存(DRAM)   │ ~100ns      │ Redis         │
  │ SSD          │ ~100μs      │ RocksDB       │
  │ HDD          │ ~10ms       │ MySQL         │
  │ 网络请求      │ ~1-1000ms   │ API           │
  └──────────────┴──────────────┴──────────────┘
  
  内存比磁盘快 10 万倍

适用场景：
  ✅ 热点数据（频繁读取）
  ✅ 读多写少
  ✅ 数据有一定容忍度（短暂不一致可接受）
  ✅ 计算结果缓存（复杂查询/聚合）
  
  ❌ 不适合：
  强一致性要求 / 写多读少 / 数据量极大
```

## 2. 常见缓存策略（读写模式）？

**回答：**

```
Cache-Aside（旁路缓存，最常用）：
  读：缓存命中→返回
      缓存未命中→查DB→写缓存→返回
  写：更新DB→删除缓存

  ┌──────┐  1.查缓存   ┌──────┐
  │ 应用  │───────────→│ 缓存 │
  │      │←───────────│      │
  │      │  2.未命中    │      │
  │      │─────┐      └──────┘
  │      │     ↓
  │      │  ┌──────┐
  │      │←─│ 数据库│
  │      │  └──────┘
  │      │  3.写缓存
  │      │───────────→ 缓存
  └──────┘

Read-Through：
  应用只和缓存交互
  缓存未命中→缓存自己去查DB并填充
  适合：缓存即服务（如 Guava Cache）

Write-Through：
  写入时同步写缓存和DB
  保证一致性但写入慢

Write-Behind（Write-Back）：
  写入只写缓存→异步批量写DB
  写入快但可能丢数据
  适合：写密集场景（如日志）
```

## 3. 多级缓存架构？

**回答：**

```
  ┌─────────────────────────────────────┐
  │              客户端                  │
  ├─────────────────────────────────────┤
  │ L1: 浏览器缓存 / CDN 缓存           │
  │     HTTP Cache-Control/ETag         │
  ├─────────────────────────────────────┤
  │ L2: 应用本地缓存                    │
  │     进程内缓存（Go sync.Map/bigcache）│
  │     优：无网络延迟                   │
  │     缺：不同实例不一致/GC 压力       │
  ├─────────────────────────────────────┤
  │ L3: 分布式缓存（Redis）              │
  │     优：集中管理/一致性好            │
  │     缺：网络延迟（~1ms）             │
  ├─────────────────────────────────────┤
  │ L4: 数据库                          │
  │     MySQL / PostgreSQL              │
  └─────────────────────────────────────┘

实际应用：
  热点key →本地缓存（ms级，防Redis压力）
  普通缓存 →Redis（高性能共享缓存）
  持久数据 →数据库
```

```go
// Go 本地缓存示例 (sync.Map)
var localCache sync.Map

func GetUser(id int64) (*User, error) {
    // L1: 本地缓存
    if v, ok := localCache.Load(id); ok {
        return v.(*User), nil
    }
    // L2: Redis
    user, err := getFromRedis(id)
    if err == nil {
        localCache.Store(id, user)
        return user, nil
    }
    // L3: DB
    user, err = getFromDB(id)
    if err == nil {
        setToRedis(id, user, 5*time.Minute)
        localCache.Store(id, user)
    }
    return user, err
}
```

## 4. 缓存预热与更新？

**回答：**

```
缓存预热：
  系统启动时提前加载热点数据到缓存
  避免冷启动时大量请求打到 DB

  方式：
  1. 启动脚本：从 DB 批量加载到 Redis
  2. 定时任务：每天凌晨预热次日热点
  3. 手动预热：运营活动前提前加载

缓存更新策略：
  1. 定时过期（TTL）：
     最简单，设置合理的过期时间
     适合实时性要求不高的数据

  2. 主动更新：
     数据变更时删除/更新缓存
     实时性好但实现复杂

  3. 异步更新：
     MQ消费/订阅 binlog 触发更新
     最终一致性

  4. 逻辑过期：
     缓存不设 TTL
     value 中存逻辑过期时间
     过期后仍返回旧值+异步更新
     适合热点key防击穿

TTL 设置建议：
  用户信息：5-30 分钟
  商品详情：1-5 分钟
  配置信息：5-60 分钟
  排行榜：1-5 分钟
  加随机偏移防止雪崩
```

## 5. 缓存的序列化方案？

**回答：**

```
  ┌──────────────┬──────┬──────┬──────────┐
  │ 方案          │ 速度 │ 空间 │ 跨语言    │
  ├──────────────┼──────┼──────┼──────────┤
  │ JSON         │ 中   │ 大   │ ✅       │
  │ Protobuf     │ 快   │ 小   │ ✅       │
  │ MessagePack  │ 快   │ 较小 │ ✅       │
  │ Gob (Go)     │ 快   │ 中   │ ❌ Go专用 │
  └──────────────┴──────┴──────┴──────────┘

推荐：
  简单场景 → JSON（可读性/调试）
  高性能 → Protobuf / MessagePack
```

```go
// Go Redis 序列化示例
import "encoding/json"

func SetCache(key string, value interface{}, ttl time.Duration) error {
    data, err := json.Marshal(value)
    if err != nil {
        return err
    }
    return rdb.Set(ctx, key, data, ttl).Err()
}

func GetCache(key string, dest interface{}) error {
    data, err := rdb.Get(ctx, key).Bytes()
    if err != nil {
        return err
    }
    return json.Unmarshal(data, dest)
}
```

## 6. 缓存 Key 设计规范？

**回答：**

```
Key 命名规范：
  格式：业务:模块:标识
  示例：
  user:info:12345        用户信息
  product:detail:678     商品详情
  order:list:user:123    用户订单列表
  rank:daily:20240115    每日排行榜

  规范：
  1. 使用 : 分隔层级（Redis 可视化工具友好）
  2. 不要太长（影响性能），不超过 128 字节
  3. 不要太短（可读性差）
  4. 不包含特殊字符
  5. 有意义的前缀（便于批量管理）

Key 管理：
  避免使用 KEYS * （阻塞）→ 用 SCAN 迭代
  设置合理 TTL（防止内存泄漏）
  监控 key 数量和内存使用

Value 大小控制：
  String < 10KB
  集合元素 < 5000
  大 Value → 拆分/压缩
```

## 7. 缓存命中率监控？

**回答：**

```
命中率 = 命中次数 / (命中 + 未命中)
  目标：> 95%

Redis 内置统计：
  INFO stats
  → keyspace_hits: 1000000    # 命中
  → keyspace_misses: 50000    # 未命中
  → 命中率 = 1000000 / 1050000 ≈ 95.2%

应用层监控：
```

```go
// 业务层缓存命中率统计
type CacheStats struct {
    hits   atomic.Int64
    misses atomic.Int64
}

func (s *CacheStats) RecordHit()  { s.hits.Add(1) }
func (s *CacheStats) RecordMiss() { s.misses.Add(1) }

func (s *CacheStats) HitRate() float64 {
    h, m := s.hits.Load(), s.misses.Load()
    total := h + m
    if total == 0 {
        return 0
    }
    return float64(h) / float64(total) * 100
}
```

```
命中率低排查：
  1. 过期时间太短 → 适当延长
  2. 缓存未覆盖热点查询 → 补充缓存
  3. 缓存穿透 → 布隆过滤器/缓存空值
  4. 内存不足淘汰 → 扩容
```

## 8. HTTP 缓存机制？

**回答：**

```
强缓存（不发请求）：
  Cache-Control: max-age=3600    # 缓存 1 小时
  Cache-Control: no-cache        # 每次要验证
  Cache-Control: no-store        # 完全不缓存
  Expires: Thu, 01 Jan 2025 ...  # HTTP/1.0 过期时间

协商缓存（发请求验证）：
  Last-Modified / If-Modified-Since
  → 基于修改时间（秒级精度）
  
  ETag / If-None-Match
  → 基于内容哈希（精确）
  → 304 Not Modified

  流程：
  浏览器 → 发请求（带 If-None-Match: "abc"）
  服务器 → ETag 没变 → 304（不传body）
  服务器 → ETag 变了 → 200 + 新内容

CDN 缓存：
  静态资源（JS/CSS/图片）→ CDN 节点缓存
  Cache-Control: public, max-age=31536000
  文件名带 hash → 内容变更则 URL 变更
```

## 9. 本地缓存（Go 进程内缓存）？

**回答：**

```
Go 常用本地缓存库：
  ┌──────────────┬──────────────────────────┐
  │ 库            │ 特点                      │
  ├──────────────┼──────────────────────────┤
  │ sync.Map     │ 标准库/读多写少场景        │
  │ bigcache     │ 零GC/大量数据/字节存储     │
  │ freecache    │ 零GC/并发安全/内存限制     │
  │ ristretto    │ 高性能/LFU/TinyLFU        │
  │ groupcache   │ 分布式场景/防击穿          │
  └──────────────┴──────────────────────────┘

注意事项：
  1. 内存限制（设置最大大小）
  2. 多实例不一致（各进程缓存不同）
  3. GC 压力（大量指针对象→用 bigcache）
  4. 过期清理
```

```go
// ristretto 示例
import "github.com/dgraph-io/ristretto"

cache, _ := ristretto.NewCache(&ristretto.Config{
    NumCounters: 1e7,     // 用于估算频率
    MaxCost:     1 << 30, // 最大内存 1GB
    BufferItems: 64,      // 每个 Get buffer
})

// 设置
cache.Set("key", value, cost)
cache.Wait() // 等待写入完成

// 获取
value, found := cache.Get("key")
```

## 10. 缓存基础面试速答？

**回答：**

```
Q: 为什么用缓存？
A: 内存比磁盘快10万倍
   降低DB压力/提升响应速度

Q: Cache-Aside 模式怎么用？
A: 读：先缓存后DB（miss则写缓存）
   写：先更新DB再删缓存

Q: 多级缓存的层次？
A: L1本地缓存(进程内)
   L2分布式缓存(Redis)
   L3数据库

Q: TTL 怎么设置？
A: 根据业务容忍度和更新频率
   加随机偏移防雪崩

Q: Key 命名规范？
A: 业务:模块:标识
   如 user:info:12345

Q: 缓存命中率目标？
A: > 95%
   低于90%需要排查优化

Q: 缓存序列化用什么？
A: 简单用 JSON
   高性能用 Protobuf

Q: Go 本地缓存推荐哪个？
A: ristretto（高性能/TinyLFU）
   bigcache（零GC/大数据量）
```
