# 缓存一致性

---

## 1. 缓存与数据库一致性问题本质？

**回答：**

```
核心矛盾：
  缓存和数据库是两个独立系统
  无法用单机事务保证两者一致
  → 只能做到最终一致性

不一致场景：
  1. 并发读写   → 旧值覆盖新值
  2. 操作中断   → 一个成功一个失败
  3. 主从延迟   → 读到从库旧值

一致性级别：
  强一致：读写串行化（性能差，不现实）
  最终一致：短暂不一致，最终收敛（常用）
  弱一致：不保证一致（适合非关键数据）
```

---

## 2. 经典方案：Cache-Aside 模式？

**回答：**

```
读流程：
  1. 读缓存 → 命中 → 返回
  2. 读缓存 → 未命中 → 读 DB → 写缓存 → 返回

写流程：
  1. 更新 DB
  2. 删除缓存

  ┌──────┐   ①读Cache    ┌──────┐
  │      │ ──────────→   │      │
  │      │   ②Cache Miss │      │
  │ App  │ ──────────→   │Cache │
  │      │   ⑤写Cache    │      │
  │      │ ──────────→   │      │
  └──┬───┘               └──────┘
     │ ③读DB  ④更新DB
  ┌──┴───┐
  │  DB  │
  └──────┘

为什么删缓存而不是更新缓存？
  → 避免并发写导致脏数据
  → 写DB1(v1) → 写DB2(v2) → 更新缓存2(v2) → 更新缓存1(v1)
  → 最终缓存=v1 但 DB=v2 → 不一致！
  
  删缓存 → 下次读自动重建 → 更安全
```

## 3. 先更新 DB 还是先删缓存？

**回答：**

```
方案1：先删缓存 → 后更新 DB（不推荐）
  ┌────────┐    ┌────────┐
  │Thread A│    │Thread B│
  │删缓存  │    │        │
  │        │    │读缓存miss│
  │        │    │读DB旧值 │
  │更新DB  │    │写缓存旧值│ ← 脏数据！
  └────────┘    └────────┘
  结果：缓存=旧值，DB=新值 → 不一致

方案2：先更新 DB → 后删缓存（推荐）
  ┌────────┐    ┌────────┐
  │Thread A│    │Thread B│
  │更新DB  │    │        │
  │        │    │读缓存命中│ ← 短暂读到旧值
  │删缓存  │    │        │
  └────────┘    └────────┘
  结果：删缓存后读到新值，短暂不一致可接受

  极端场景（概率极低）：
  Thread A: 读DB旧值 → Thread B: 更新DB+删缓存
  → Thread A: 写缓存旧值
  但要求 DB 写比读慢 → 概率极低
```

## 4. 延迟双删方案？

**回答：**

```
流程：
  1. 删除缓存
  2. 更新数据库
  3. 延迟 N ms
  4. 再次删除缓存

  ┌──────────────────────────────────────────┐
  │ 删缓存 → 更新DB → sleep(500ms) → 删缓存 │
  └──────────────────────────────────────────┘

作用：
  第二次删除 → 清理并发读写导致的脏缓存

延迟时间 N 怎么设？
  N > 一次读请求耗时（读DB + 写缓存）
  通常 200ms ~ 1s
```

```go
// 延迟双删实现
func UpdateUser(ctx context.Context, user *User) error {
    key := fmt.Sprintf("user:%d", user.ID)
    
    // 1. 删缓存
    rdb.Del(ctx, key)
    
    // 2. 更新 DB
    if err := db.UpdateUser(user); err != nil {
        return err
    }
    
    // 3. 延迟双删（异步）
    go func() {
        time.Sleep(500 * time.Millisecond)
        rdb.Del(context.Background(), key)
    }()
    
    return nil
}
```

**问题：**
- 延迟时间难以精确设定
- 删缓存失败怎么办？

## 5. 删缓存失败怎么办？重试方案？

**回答：**

```
方案1：消息队列重试（推荐）
  ┌─────┐  更新DB  ┌────┐
  │ App │─────────→│ DB │
  │     │          └────┘
  │     │  发消息   ┌────┐
  │     │─────────→│ MQ │
  └─────┘          └──┬─┘
                      │ 消费消息
                 ┌────┴────┐
                 │删缓存    │
                 │失败→重试 │
                 └─────────┘

方案2：订阅 binlog（推荐）
  ┌────┐  写入    ┌────────┐
  │ DB │────────→│ binlog │
  └────┘         └────┬───┘
                      │ 订阅
                 ┌────┴────┐
                 │Canal/   │
                 │Debezium │
                 └────┬────┘
                      │ 解析变更
                 ┌────┴────┐
                 │删除缓存  │
                 └─────────┘

  优势：
  与业务代码解耦
  可靠性高（binlog 可靠）
  自动捕获所有变更
```

```go
// 消息队列重试方案
func UpdateWithRetry(ctx context.Context, user *User) error {
    // 1. 更新 DB
    if err := db.UpdateUser(user); err != nil {
        return err
    }
    
    // 2. 发送删缓存消息到 MQ
    msg := DeleteCacheMsg{
        Key:       fmt.Sprintf("user:%d", user.ID),
        RetryCount: 0,
    }
    return mq.Publish("cache-delete", msg)
}

// 消费者
func HandleCacheDelete(msg DeleteCacheMsg) error {
    err := rdb.Del(ctx, msg.Key).Err()
    if err != nil && msg.RetryCount < 3 {
        msg.RetryCount++
        // 延迟重试
        return mq.PublishDelay("cache-delete", msg, time.Second)
    }
    return err
}
```

## 6. 基于 Binlog 的一致性方案？

**回答：**

```
架构：
  ┌─────┐    ┌─────┐    ┌───────┐    ┌──────┐
  │ App │───→│MySQL│───→│Canal  │───→│消费者│───→删缓存
  └─────┘    └─────┘    └───────┘    └──────┘
             binlog      解析        处理

Canal 工作原理：
  伪装成 MySQL 从库
  接收 MySQL 推送的 binlog
  解析成结构化数据
  发送到 MQ 或直接消费

处理流程：
  1. 应用只写 DB（不管缓存）
  2. Canal 监听 binlog 变更
  3. 解析出变更的表和行
  4. 构造 key → 删除/更新缓存
  5. 消费失败 → MQ 重试

优势：
  业务无侵入
  所有变更都能捕获（包括直接修改 DB）
  可靠性高

注意：
  Canal 本身高可用
  消费延迟（通常 ms 级）
  需要处理幂等性
```

## 7. 读写分离场景的一致性？

**回答：**

```
问题：
  主库写入 → 删缓存 → 从库还没同步
  → 读请求 → 缓存 miss → 读从库旧值 → 写缓存旧值
  
  ┌────────┐  写   ┌──────┐  复制   ┌──────┐
  │  App   │──────→│Master│────────→│Slave │
  │        │       └──────┘  延迟   └──┬───┘
  │        │  删缓存  ↓               │
  │        │────→ Cache               │
  │        │  读缓存miss → 读Slave旧值 │
  └────────┘                          │

解决方案：

1. 强制读主库（简单有效）：
   写入后短时间内路由到主库读
   set flag，TTL = 主从延迟时间

2. 延迟双删：
   删缓存 → 等主从同步完 → 再删一次
   延迟时间 > 主从复制延迟

3. 缓存加标记：
   写入时设标记 key（短 TTL）
   读时检查标记 → 有则读主库
```

```go
// 强制读主库方案
func UpdateAndInvalidate(ctx context.Context, user *User) error {
    if err := masterDB.UpdateUser(user); err != nil {
        return err
    }
    
    key := fmt.Sprintf("user:%d", user.ID)
    rdb.Del(ctx, key)
    
    // 标记短时间内必须读主库
    flagKey := fmt.Sprintf("read_master:%d", user.ID)
    rdb.Set(ctx, flagKey, "1", 3*time.Second)
    return nil
}

func GetUser(ctx context.Context, id int64) (*User, error) {
    key := fmt.Sprintf("user:%d", id)
    val, err := rdb.Get(ctx, key).Result()
    if err == nil {
        return unmarshalUser(val)
    }
    
    // 检查是否需要读主库
    flagKey := fmt.Sprintf("read_master:%d", id)
    if rdb.Exists(ctx, flagKey).Val() > 0 {
        return masterDB.GetUser(id)
    }
    return slaveDB.GetUser(id)
}
```

## 8. 强一致性方案？

**回答：**

```
方案1：串行化（不推荐，性能差）
  读写都加分布式锁
  → 同一数据同一时刻只有一个操作
  → 性能极差

方案2：分布式事务
  用 2PC/TCC 保证 DB 和缓存一致
  → 复杂度太高

方案3：Write Behind + 读屏障
  所有写通过缓存 → 异步 flush 到 DB
  读屏障确保最新数据

实际选择：
  绝大多数场景：最终一致性足够
  短时间不一致对业务影响很小

  真的需要强一致 → 不用缓存
  用 DB + 本地缓存 + 极短 TTL
```

## 9. 一致性方案选型？

**回答：**

```
按场景选择：

  普通业务（容忍秒级不一致）：
  → Cache-Aside + 先更新DB后删缓存
  → 加合理 TTL 兜底

  高并发写（如秒杀库存）：
  → 延迟双删 + MQ 重试

  强一致需求（如金融余额）：
  → 不用缓存 / 读写都走 DB
  → 或 Redis + Lua 原子操作

  数据变更频繁（多来源修改）：
  → Binlog 订阅（Canal/Debezium）
  → 与业务解耦

  ┌────────────────────────────────────┐
  │ 最终一致性黄金组合：               │
  │ Cache-Aside                       │
  │ + 先更新DB后删缓存                │
  │ + MQ 异步补偿                     │
  │ + TTL 兜底                        │
  │ + Binlog 订阅（可选）             │
  └────────────────────────────────────┘
```

## 10. 缓存一致性面试速答？

**回答：**

```
Q: 为什么删缓存而不是更新缓存？
A: 并发更新可能导致旧值覆盖新值
   删除后读时重建更安全

Q: 先更新DB还是先删缓存？
A: 推荐先更新DB后删缓存
   先删缓存会导致并发读写脏数据

Q: 延迟双删是什么？
A: 删缓存→更新DB→延迟N ms→再删缓存
   清理并发读产生的脏缓存

Q: 删缓存失败怎么办？
A: MQ 重试 或 订阅 binlog 异步删除

Q: 读写分离如何保证一致？
A: 写入后设标记强制读主库
   或延迟双删等主从同步

Q: 能做到强一致吗？
A: 很难。缓存和DB是两个系统
   通常做最终一致 + TTL 兜底

Q: Channel/Canal 方案的优势？
A: 业务无侵入，捕获所有变更
   可靠性高，解耦彻底

Q: 最佳实践是什么？
A: Cache-Aside + 先更新DB后删缓存
   + MQ补偿 + TTL兜底
```
