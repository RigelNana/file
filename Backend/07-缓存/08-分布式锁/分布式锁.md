# 分布式锁

---

## 1. 为什么需要分布式锁？

**回答：**

```
单机锁 vs 分布式锁：
  单机：sync.Mutex / channel → 只在一个进程内有效
  分布式：多个服务实例 → 需要跨进程的锁

场景：
  1. 防止库存超卖（多实例并发扣减）
  2. 防止重复操作（定时任务只在一个节点执行）
  3. 分布式幂等（同一请求只处理一次）

  ┌─────────┐  ┌─────────┐  ┌─────────┐
  │Instance1│  │Instance2│  │Instance3│
  └────┬────┘  └────┬────┘  └────┬────┘
       │            │            │
       └────────────┼────────────┘
                    │
              ┌─────┴─────┐
              │分布式锁服务 │
              │Redis/ZK/  │
              │etcd       │
              └───────────┘

实现方式：
  Redis（SETNX）→ 最常用，性能高
  ZooKeeper    → 可靠性高
  etcd         → 强一致性
  MySQL        → 简单但性能差
```

## 2. Redis 分布式锁基本实现？

**回答：**

```
加锁：SET key value NX EX seconds
  NX → 不存在才设置（互斥）
  EX → 设置过期时间（防死锁）
  value → 唯一标识（UUID，防误删）

解锁：先判断 value → 再 DEL（Lua 保证原子）

  加锁：
  SET lock:order:1001 uuid-xxx NX EX 30

  解锁（Lua 脚本）：
  if redis.call('GET', KEYS[1]) == ARGV[1] then
      return redis.call('DEL', KEYS[1])
  end
  return 0
```

```go
// Go 实现基本分布式锁
type RedisLock struct {
    rdb    *redis.Client
    key    string
    value  string
    ttl    time.Duration
}

func NewRedisLock(rdb *redis.Client, key string, ttl time.Duration) *RedisLock {
    return &RedisLock{
        rdb:   rdb,
        key:   key,
        value: uuid.New().String(),
        ttl:   ttl,
    }
}

func (l *RedisLock) Lock(ctx context.Context) (bool, error) {
    return l.rdb.SetNX(ctx, l.key, l.value, l.ttl).Result()
}

var unlockScript = redis.NewScript(`
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('DEL', KEYS[1])
end
return 0
`)

func (l *RedisLock) Unlock(ctx context.Context) error {
    _, err := unlockScript.Run(ctx, l.rdb, []string{l.key}, l.value).Result()
    return err
}
```

## 3. 过期时间难以设定？ 看门狗续期？

**回答：**

```
问题：
  TTL 设太短 → 任务没完成锁已过期 → 其他人拿到锁
  TTL 设太长 → 持锁进程崩溃 → 等待时间过长

看门狗（Watchdog）方案：
  加锁后启动后台 goroutine
  定期续期（每 TTL/3 续一次）
  任务完成 → 停止续期 + 释放锁
  进程崩溃 → goroutine 也死了 → 锁自动过期

  ┌──────────────────────────┐
  │ Lock(TTL=30s)            │
  │  → 启动 watchdog         │
  │    → 每 10s 续期到 30s   │
  │                          │
  │ 业务逻辑执行...           │
  │                          │
  │ Unlock()                 │
  │  → 停止 watchdog         │
  │  → 删除锁                │
  └──────────────────────────┘
```

```go
// 看门狗续期
func (l *RedisLock) LockWithWatchdog(ctx context.Context) (bool, error) {
    ok, err := l.rdb.SetNX(ctx, l.key, l.value, l.ttl).Result()
    if err != nil || !ok {
        return false, err
    }
    
    // 启动看门狗
    l.stopCh = make(chan struct{})
    go l.watchdog()
    return true, nil
}

func (l *RedisLock) watchdog() {
    ticker := time.NewTicker(l.ttl / 3)
    defer ticker.Stop()
    
    for {
        select {
        case <-l.stopCh:
            return
        case <-ticker.C:
            // 续期（只续自己的锁）
            l.rdb.Eval(context.Background(), `
                if redis.call('GET', KEYS[1]) == ARGV[1] then
                    return redis.call('PEXPIRE', KEYS[1], ARGV[2])
                end
                return 0
            `, []string{l.key}, l.value, l.ttl.Milliseconds())
        }
    }
}

func (l *RedisLock) Unlock(ctx context.Context) error {
    close(l.stopCh) // 停止看门狗
    return l.unlock(ctx)
}
```

## 4. Redlock 算法？

**回答：**

```
Redlock：Redis 作者提出的分布式锁算法
  解决单 Redis 实例故障导致锁失效问题

算法流程（N 个独立 Redis 实例，通常 5 个）：
  1. 获取当前时间 T1
  2. 依次向 N 个实例发送加锁请求
     SET key value NX PX ttl
  3. 计算耗时 = 当前时间 - T1
  4. 如果多数实例(N/2+1)加锁成功
     且耗时 < TTL
     → 加锁成功
  5. 否则 → 向所有实例发送解锁

  ┌───────┐ ┌───────┐ ┌───────┐ ┌───────┐ ┌───────┐
  │Redis 1│ │Redis 2│ │Redis 3│ │Redis 4│ │Redis 5│
  │  ✅   │ │  ✅   │ │  ✅   │ │  ❌   │ │  ❌   │
  └───────┘ └───────┘ └───────┘ └───────┘ └───────┘
  3/5 成功 > 2(N/2) → 加锁成功

争议：
  Martin Kleppmann 质疑：
  - GC 暂停/时钟跳变可能导致不安全
  - 理论上无法保证强互斥
  
  实际使用：
  大多数场景 单节点 Redis + 看门狗 够用
  强一致需求 → 用 etcd / ZooKeeper
```

## 5. 可重入锁？

**回答：**

```
可重入：同一线程/goroutine 可以多次加锁
  ┌────────────────────────┐
  │ func A() {             │
  │   lock.Lock()          │ → 加锁成功 count=1
  │   B()                  │
  │   lock.Unlock()        │ → count=0 → 真正释放
  │ }                      │
  │ func B() {             │
  │   lock.Lock()          │ → 同一持有者 count=2
  │   ...                  │
  │   lock.Unlock()        │ → count=1
  │ }                      │
  └────────────────────────┘

Redis 实现：用 Hash 存锁信息
  HSET lock:xxx holder goroutine-id
  HSET lock:xxx count  1
```

```lua
-- 加锁 Lua
local key = KEYS[1]
local holder = ARGV[1]
local ttl = ARGV[2]

if redis.call('EXISTS', key) == 0 then
    redis.call('HSET', key, 'holder', holder)
    redis.call('HSET', key, 'count', 1)
    redis.call('PEXPIRE', key, ttl)
    return 1
end

if redis.call('HGET', key, 'holder') == holder then
    redis.call('HINCRBY', key, 'count', 1)
    redis.call('PEXPIRE', key, ttl)
    return 1
end

return 0  -- 别人持有

-- 解锁 Lua
local key = KEYS[1]
local holder = ARGV[1]

if redis.call('HGET', key, 'holder') ~= holder then
    return -1  -- 不是持有者
end

local count = redis.call('HINCRBY', key, 'count', -1)
if count <= 0 then
    redis.call('DEL', key)
    return 1  -- 完全释放
end
return 0  -- 还有重入
```

## 6. etcd 分布式锁？

**回答：**

```
etcd 实现分布式锁的优势：
  强一致性（Raft 协议）
  Lease 租约机制（自动过期）
  Watch 机制（阻塞等待锁释放）
  Revision 全局有序（公平锁）

实现原理：
  1. 创建 Lease（TTL）
  2. 用 Lease 创建 key（带前缀）
  3. 检查自己的 Revision 是否最小
     最小 → 拿到锁
     不是 → Watch 前一个 key，等待删除
  4. 释放：删除 key / 撤销 Lease

  优势 vs Redis：
  - Raft 保证强一致（不会脑裂丢锁）
  - Watch 代替轮询（效率高）
  - 自带公平排队
  
  劣势：
  - 性能不如 Redis
  - 部署复杂度高
```

```go
// Go etcd 分布式锁（官方 concurrency 包）
import clientv3 "go.etcd.io/etcd/client/v3"
import "go.etcd.io/etcd/client/v3/concurrency"

cli, _ := clientv3.New(clientv3.Config{
    Endpoints: []string{"localhost:2379"},
})

session, _ := concurrency.NewSession(cli, concurrency.WithTTL(10))
defer session.Close()

mutex := concurrency.NewMutex(session, "/lock/order/1001")

// 加锁（阻塞直到获取）
if err := mutex.Lock(ctx); err != nil {
    log.Fatal(err)
}

// 业务逻辑...

// 解锁
if err := mutex.Unlock(ctx); err != nil {
    log.Fatal(err)
}
```

## 7. ZooKeeper 分布式锁？

**回答：**

```
ZooKeeper 实现原理：
  利用临时顺序节点 + Watch 机制

流程：
  1. 在锁节点下创建临时顺序节点
     /lock/order → /lock/order/node_0001
  
  2. 获取所有子节点，判断自己是否最小
     最小 → 获得锁
     不是 → Watch 前一个节点

  3. 前一个节点删除 → 收到通知 → 再次判断

  4. 释放锁 → 删除节点（或会话断开自动删除）

  /lock/order/
    ├── node_0001  ← 持有锁
    ├── node_0002  ← Watch 0001
    └── node_0003  ← Watch 0002

  优势：
  - 强一致性（ZAB 协议）
  - 临时节点自动清理（进程崩溃）
  - 顺序节点实现公平锁
  - 避免惊群效应（只 Watch 前一个）

  劣势：
  - 性能最差
  - 部署运维复杂
```

## 8. Redis vs etcd vs ZooKeeper 分布式锁？

**回答：**

```
  ┌──────────┬──────────┬──────────┬──────────┐
  │ 对比      │ Redis    │ etcd     │ZooKeeper │
  ├──────────┼──────────┼──────────┼──────────┤
  │ 一致性   │ 弱一致   │ 强一致   │ 强一致   │
  │ 性能     │ 最高     │ 高       │ 中等     │
  │ 可靠性   │ 中等     │ 高       │ 高       │
  │ 自动续期 │ 需实现   │ Lease    │ 会话     │
  │ 公平锁   │ 需实现   │ Revision │ 顺序节点 │
  │ 部署     │ 简单     │ 中等     │ 复杂     │
  │ AP/CP    │ AP       │ CP       │ CP       │
  │ 适用场景 │ 高性能   │ K8s生态  │ 大数据   │
  └──────────┴──────────┴──────────┴──────────┘

选择建议：
  性能优先 + 容忍极端情况 → Redis
  安全性优先 + K8s 体系   → etcd
  安全性优先 + Java 体系  → ZooKeeper
  简单业务 → Redis 单节点就够
```

## 9. 分布式锁最佳实践？

**回答：**

```
1. 锁粒度尽量细：
   ❌ lock("order")
   ✅ lock("order:1001")

2. 超时时间合理：
   评估业务最大执行时间
   加上看门狗自动续期

3. 加锁失败处理：
   重试 + 退避（指数退避）
   设置最大重试次数
   超过 → 返回错误 / 降级

4. 防止误删：
   value 用 UUID
   Lua 脚本原子判断

5. 保证幂等性：
   拿到锁后仍需做幂等检查
   锁只是减少并发，不能替代幂等

6. 异常处理：
   defer unlock
   panic recovery 中也要 unlock
```

```go
// 完整的分布式锁使用模板
func ProcessOrder(ctx context.Context, orderID string) error {
    lock := NewRedisLock(rdb, "lock:order:"+orderID, 30*time.Second)
    
    // 加锁（带重试）
    var locked bool
    for i := 0; i < 3; i++ {
        var err error
        locked, err = lock.LockWithWatchdog(ctx)
        if err != nil {
            return fmt.Errorf("lock error: %w", err)
        }
        if locked {
            break
        }
        time.Sleep(time.Duration(100*(i+1)) * time.Millisecond)
    }
    if !locked {
        return ErrLockFailed
    }
    defer lock.Unlock(ctx)
    
    // 幂等检查
    if isProcessed(orderID) {
        return nil
    }
    
    // 业务逻辑
    return doProcess(ctx, orderID)
}
```

## 10. 分布式锁面试速答？

**回答：**

```
Q: Redis 分布式锁怎么实现？
A: SET key uuid NX EX ttl 加锁
   Lua 脚本判断 uuid 后 DEL 解锁

Q: 为什么 value 要用 UUID？
A: 防止误删别人的锁
   只有持锁者才能释放

Q: 锁过期但任务未完成怎么办？
A: 看门狗机制，后台 goroutine
   定期续期（TTL/3 间隔）

Q: Redlock 是什么？
A: 在 N 个独立 Redis 上加锁
   多数成功(N/2+1) → 加锁成功

Q: Redis 锁和 etcd 锁怎么选？
A: 性能优先 → Redis
   强一致优先 → etcd

Q: 可重入锁怎么实现？
A: 用 Hash 存 holder + count
   同一持有者 count++
   解锁 count-- 到 0 才删除

Q: 分布式锁的注意事项？
A: 锁粒度要细 + 看门狗续期
   + UUID防误删 + 幂等检查

Q: 公平锁怎么实现？
A: etcd: Revision 排序
   ZK: 顺序节点 + Watch 前一个
```
