# 缓存八股文

## 一、缓存基础

### 1. 为什么要用缓存？

**答：** 降低数据库压力、提升系统响应速度。内存读取速度比磁盘快几个数量级。缓存常用于热点数据（频繁读取、不经常变化），典型场景：用户信息、商品详情、配置数据、排行榜、会话信息。缓存策略：Cache-Aside（旁路缓存）、Read-Through、Write-Through、Write-Behind。

### 2. Redis 为什么快？

**答：** 纯内存操作（ns级）、单线程避免锁竞争和上下文切换、IO多路复用（epoll）处理大量连接、高效的数据结构（SDS/ziplist/skiplist/intset）、6.0+引入多线程只用于网络IO（命令执行仍单线程）。

## 二、Redis 数据结构

### 3. Redis 的数据类型和底层实现？

**答：** String（SDS，支持二进制安全/预分配/惰性释放）、Hash（ziplist→hashtable，阈值128个字段或64字节）、List（quicklist=ziplist+双向链表）、Set（intset→hashtable）、ZSet（ziplist→skiplist+hashtable）、Stream（基数树，5.0+消息流）。还有 Bitmap、HyperLogLog、GEO 等。

### 4. 跳表为什么用在 ZSet？

**答：** 跳表实现简单（相比平衡树）、范围查询高效（相比哈希）、支持 O(logN) 的增删查。Redis ZSet 同时用跳表（范围查询）和字典（O(1)单元素查找）。跳表最多32层，概率1/4晋升。

## 三、持久化

### 5. RDB 和 AOF 的区别？

**答：** RDB 是某时刻的全量快照（fork子进程+COW），文件紧凑恢复快但可能丢几分钟数据。AOF 记录每条写命令（appendonly），appendfsync可选 always/everysec/no，支持重写压缩。4.0+混合持久化：AOF重写时前部分RDB+后部分AOF，兼顾恢复速度和安全性。

## 四、缓存问题

### 6. 缓存穿透、击穿、雪崩？

**答：** 穿透：查询不存在的数据（缓存和DB都无），恶意攻击场景多。防御：布隆过滤器、缓存空值。击穿：热点key过期瞬间大量请求打到DB。防御：互斥锁、热点不过期、逻辑过期。雪崩：大量key同时过期或Redis宕机。防御：过期时间加随机值、集群部署、限流降级、多级缓存。

### 7. 缓存与数据库一致性？

**答：** Cache-Aside 模式：读→先缓存后DB；写→先更新DB再删缓存。先删缓存再更新DB有并发问题（其他线程读到旧值写入缓存）。延迟双删可以缓解。最终一致性方案：Canal订阅binlog→更新缓存、消息队列异步删除缓存。强一致性需要分布式锁。

## 五、集群与分布式锁

### 8. Redis 集群模式？

**答：** 主从复制（手动故障切换）→ Sentinel（自动故障检测和切换，3+哨兵节点）→ Redis Cluster（16384个slot分片，去中心化Gossip协议，官方推荐）。Cluster支持自动分片和再平衡，节点故障时从库自动提升。限制：不支持跨slot事务、多key操作需要用hash tag。

### 9. Redis 分布式锁？

**答：** SETNX+过期时间（SET key value NX EX 30，原子操作）。释放锁需要判断value是自己的（Lua脚本原子操作）。问题：过期时间难估计→看门狗续期（Redisson）。集群下用RedLock（多数节点加锁），但有争议（Martin Kleppmann vs Antirez 之争）。

## 六、高级特性

### 10. Redis 内存淘汰策略？

**答：** 8种策略：noeviction（默认，满了报错）、allkeys-lru（LRU淘汰）、allkeys-lfu（LFU，4.0+）、allkeys-random、volatile-lru（有过期的key中LRU）、volatile-lfu、volatile-random、volatile-ttl。推荐 allkeys-lfu（最近最不常用，4.0+）或 allkeys-lru。

### 11. Redis 的过期删除策略？

**答：** 惰性删除：访问key时检查是否过期→删除，节省CPU但可能内存浪费。定期删除：每100ms随机检查过期key删除一部分（限制时间25ms），平衡CPU和内存。两者结合使用。

### 12. Pipeline 和 Lua 脚本？

**答：** Pipeline：批量发送命令减少网络往返（RTT），非原子。Lua脚本：Redis内置Lua解释器，EVAL执行脚本，在server端原子执行多条命令。分布式锁释放、限流计数等场景常用Lua保证原子性。

### 13. Redis 事务？

**答：** MULTI/EXEC 包裹命令队列，顺序执行不被中断（单线程），但不支持回滚（命令错误会跳过继续执行）。WATCH实现乐观锁（CAS），被修改则事务失败。实际用Lua脚本更多（真正原子+可编程逻辑）。

### 14. 大key和热key问题？

**答：** 大key：String>10KB或集合元素>5000。危害：阻塞/内存不均/网络拥塞。解决：拆分（Hash分段）、压缩、异步删除（UNLINK）。热key：某个key被高频访问。危害：单节点过载。解决：本地缓存（多级缓存）、key复制到多节点（加后缀分散读取）。
