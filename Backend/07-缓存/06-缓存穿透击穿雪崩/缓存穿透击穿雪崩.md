# 缓存穿透击穿雪崩

---

## 1. 缓存穿透是什么？怎么解决？

**回答：**

```
缓存穿透：查询不存在的数据
  缓存没有 → 数据库没有 → 每次都打到数据库

  Client → Cache(MISS) → DB(不存在) → 返回空
                                ↑ 每次都到这里

攻击场景：
  恶意请求 id=-1 或随机不存在的 id
  大量请求绕过缓存直达数据库 → DB 崩溃

解决方案：

1. 缓存空值：
   查 DB 不存在 → 缓存空值 + 短过期时间
   SET user:-1 "" EX 60
   问题：大量不同的无效 key → 内存浪费

2. 布隆过滤器（推荐）：
   ┌──────┐    ┌──────────┐    ┌──────┐    ┌────┐
   │Client│───→│BloomFilter│───→│Cache │───→│ DB │
   └──────┘    └──────────┘    └──────┘    └────┘
               不存在则直接返回

   预先将所有合法 ID 加入布隆过滤器
   请求先过布隆过滤器
   → 一定不存在 → 直接拒绝
   → 可能存在 → 继续查缓存/DB

   误判率可控（0.1%）
   不支持删除（可用 Cuckoo Filter）

3. 参数校验：
   前端/网关层校验 id 格式、范围
   拦截明显非法请求
```

```go
// 布隆过滤器 + 缓存空值
func GetUser(ctx context.Context, id int64) (*User, error) {
    // 1. 布隆过滤器判断
    if !bloomFilter.Test([]byte(fmt.Sprint(id))) {
        return nil, ErrNotFound
    }
    
    // 2. 查缓存
    key := fmt.Sprintf("user:%d", id)
    val, err := rdb.Get(ctx, key).Result()
    if err == nil {
        if val == "" { // 空值缓存
            return nil, ErrNotFound
        }
        return unmarshalUser(val)
    }
    
    // 3. 查数据库
    user, err := db.GetUser(id)
    if err != nil {
        // 缓存空值，短过期
        rdb.Set(ctx, key, "", time.Minute)
        return nil, ErrNotFound
    }
    
    // 4. 写入缓存
    data, _ := json.Marshal(user)
    rdb.Set(ctx, key, data, time.Hour)
    return user, nil
}
```

## 2. 缓存击穿是什么？怎么解决？

**回答：**

```
缓存击穿：热点 key 过期瞬间，大量请求打到 DB

  时间线：
  ──────────────────────────────────────→
  [  热点 key 存在  ] 过期 [ 大量请求涌入 DB ]
                      ↓
                   DB 压力暴增

解决方案：

1. 互斥锁（推荐）
   key 过期后，只有一个请求去查 DB 并重建缓存
   其他请求等待或返回旧值

   ┌──────────────────────────────┐
   │ 请求来了 → 缓存未命中        │
   │ → 尝试加锁                   │
   │   → 成功：查 DB → 写缓存     │
   │   → 失败：等待/重试           │
   └──────────────────────────────┘

2. 逻辑过期
   缓存不设 TTL，在 value 中存逻辑过期时间
   发现过期 → 异步更新 → 返回旧值

3. 永不过期
   热点数据不设过期时间
   通过后台任务定时更新

4. 二级缓存
   L1(本地) + L2(Redis)
   Redis 过期了 L1 还有
```

```go
// 互斥锁方案
func GetHotData(ctx context.Context, key string) (string, error) {
    // 1. 查缓存
    val, err := rdb.Get(ctx, key).Result()
    if err == nil {
        return val, nil
    }
    
    // 2. 缓存未命中 → 加锁
    lockKey := "lock:" + key
    locked, _ := rdb.SetNX(ctx, lockKey, "1", 10*time.Second).Result()
    
    if locked {
        defer rdb.Del(ctx, lockKey)
        // 3. 双重检查
        val, err = rdb.Get(ctx, key).Result()
        if err == nil {
            return val, nil
        }
        // 4. 查 DB 并重建缓存
        val, err = queryDB(key)
        if err != nil {
            return "", err
        }
        rdb.Set(ctx, key, val, randomExpire(time.Hour))
        return val, nil
    }
    
    // 5. 没拿到锁 → 短暂等待后重试
    time.Sleep(50 * time.Millisecond)
    return GetHotData(ctx, key)
}
```

## 3. 缓存雪崩是什么？怎么解决？

**回答：**

```
缓存雪崩：大量 key 同时过期 或 Redis 宕机
  → 所有请求涌入数据库 → DB 崩溃

场景1：大量 key 同时过期
  批量导入缓存设了相同 TTL
  → 某一刻全部过期

场景2：Redis 实例宕机
  → 所有请求失去缓存

解决方案：

1. 随机过期时间：
   避免同时过期
   TTL = base + random(0, range)
   如：1h + random(0, 10min)

2. Redis 高可用：
   Sentinel / Cluster 避免单点故障

3. 限流降级：
   DB 前加限流（令牌桶/漏桶）
   超过阈值 → 返回降级响应

4. 多级缓存：
   L1 本地缓存 + L2 Redis
   Redis 挂了 L1 还能挡一层

5. 缓存预热：
   服务启动时加载热点数据
   过期时间错开

6. 后台刷新：
   key 快过期时后台异步续期
```

## 5. 布隆过滤器原理与实现？

**回答：**

```
布隆过滤器原理：
  位数组 + 多个哈希函数

  添加元素：
    key → hash1(key)=3, hash2(key)=7, hash3(key)=11
    BitArray[3]=1, BitArray[7]=1, BitArray[11]=1

  查询元素：
    key → hash1=3, hash2=7, hash3=11
    全部为1 → 可能存在（有误判）
    任一为0 → 一定不存在

  特性：
    空间效率极高（1亿数据 ≈ 120MB）
    不支持删除
    有误判率（可控，如 0.1%）
    没有漏判

  参数选择（0.1% 误判率）：
    k ≈ 10 个哈希函数
    m ≈ 14.4 * n（n 是元素数量）
```

```go
// 使用 Redis Bitmap 实现简易布隆过滤器
type BloomFilter struct {
    rdb    *redis.Client
    key    string
    hashes int
    size   uint
}

func (bf *BloomFilter) Add(ctx context.Context, item string) {
    for i := 0; i < bf.hashes; i++ {
        offset := bf.hash(item, i)
        bf.rdb.SetBit(ctx, bf.key, int64(offset), 1)
    }
}

func (bf *BloomFilter) Exists(ctx context.Context, item string) bool {
    for i := 0; i < bf.hashes; i++ {
        offset := bf.hash(item, i)
        bit, _ := bf.rdb.GetBit(ctx, bf.key, int64(offset)).Result()
        if bit == 0 {
            return false  // 一定不存在
        }
    }
    return true  // 可能存在
}

// Redis 4.0+ 模块：RedisBloom
// BF.ADD key item
// BF.EXISTS key item
// BF.MADD key item1 item2
```

## 6. 热点 Key 问题？

**回答：**

```
热点 Key：某些 key 访问量极大
  → 单个 Redis 节点压力过大

发现热点 key：
  redis-cli --hotkeys（Redis 4.0+）
  MONITOR 命令（线上慎用）
  客户端统计
  proxy 层统计

解决方案：

1. 本地缓存：
   热点数据放进程内缓存
   L1(本地) → L2(Redis) → DB
   减少对 Redis 的请求

2. key 分散：
   将一个 key 拆成多个
   key_1, key_2, ..., key_N
   读时随机选一个
   写时全部更新

3. 读写分离：
   从库分担读压力

4. 专用实例：
   热点 key 独立 Redis 实例
   与普通数据隔离
```

```go
// 本地缓存 + Redis 多级缓存
type MultiLevelCache struct {
    local *ristretto.Cache  // L1
    redis *redis.Client     // L2
}

func (c *MultiLevelCache) Get(ctx context.Context, key string) (string, error) {
    // L1 本地缓存
    if val, ok := c.local.Get(key); ok {
        return val.(string), nil
    }
    // L2 Redis
    val, err := c.redis.Get(ctx, key).Result()
    if err == nil {
        c.local.SetWithTTL(key, val, 1, 10*time.Second)
        return val, nil
    }
    return "", err
}
```

## 7. 大 Key 问题？

**回答：**

```
大 Key 判断标准：
  String > 10KB
  Hash/List/Set/ZSet 元素 > 5000

大 Key 危害：
  内存不均衡（Cluster 数据倾斜）
  网络带宽占用
  阻塞主线程（DEL 大 key）
  AOF/RDB 影响

发现大 key：
  redis-cli --bigkeys
  MEMORY USAGE key
  RDB 分析工具（redis-rdb-tools）

解决方案：

1. 拆分：
   Hash 5w 个 field → 拆成 10 个 Hash，每个 5000
   key → key:0, key:1, ..., key:9
   field 取模决定在哪个子 key

2. 压缩：
   String 类型用 gzip/snappy 压缩
   JSON 换 Protobuf/MessagePack

3. 安全删除：
   不要直接 DEL → 阻塞
   Redis 4.0+ UNLINK（异步删除）
   老版本 HSCAN + HDEL 分批删除

4. 合理设计：
   避免把大对象整体存 Redis
   只缓存必要字段
```

## 8. 缓存预热与缓存降级？

**回答：**

```
缓存预热（Warm Up）：
  服务启动时预先加载热点数据到缓存
  避免冷启动大量请求打到 DB

预热策略：
  1. 启动时预热：从 DB 加载 TOP N 热点数据
  2. 定时预热：定时任务刷新即将过期的缓存
  3. 监听 binlog：数据变更主动更新缓存

缓存降级（Fallback）：
  缓存不可用时的兜底策略

降级层级：
  ┌────────────────────────────────┐
  │ 正常：Cache → DB               │
  │ 降级1：返回本地缓存（可能过期） │
  │ 降级2：返回默认值              │
  │ 降级3：返回错误提示            │
  └────────────────────────────────┘
```

```go
// 缓存降级示例
func GetProduct(ctx context.Context, id int64) (*Product, error) {
    key := fmt.Sprintf("product:%d", id)
    
    // 1. 尝试 Redis
    val, err := rdb.Get(ctx, key).Result()
    if err == nil {
        return unmarshalProduct(val)
    }
    
    // 2. Redis 异常 → 本地缓存
    if localVal, ok := localCache.Get(key); ok {
        return localVal.(*Product), nil
    }
    
    // 3. 查数据库（限流保护）
    if !rateLimiter.Allow() {
        // 4. 降级 → 返回默认值
        return defaultProduct, nil
    }
    
    product, err := db.GetProduct(id)
    if err != nil {
        return defaultProduct, nil
    }
    
    // 回写缓存
    data, _ := json.Marshal(product)
    rdb.Set(ctx, key, data, time.Hour)
    localCache.Set(key, product, 10*time.Second)
    return product, nil
}
```

## 9. 缓存更新策略？

**回答：**

```
缓存更新策略对比：

1. Cache-Aside（旁路缓存）— 最常用
   读：先读缓存，miss 则读 DB 回填缓存
   写：更新 DB → 删除缓存
   → 后续读自动重建

2. Read/Write Through
   应用只与缓存交互
   缓存负责同步读写 DB
   → 对应用透明

3. Write Behind（写回）
   写只写缓存 → 异步批量写 DB
   性能高但可能丢数据

Cache-Aside 的坑：
  先删缓存再更新 DB → 并发问题
  ┌────────┐  ┌────────┐
  │Thread A│  │Thread B│
  │删缓存  │  │        │
  │        │  │读缓存miss│
  │        │  │读DB旧值 │
  │更新DB  │  │写缓存旧值│← 脏数据！
  └────────┘  └────────┘

  推荐：先更新 DB → 再删缓存
  用延迟双删进一步保证：
  1. 删缓存
  2. 更新 DB
  3. 延迟 N ms 再删缓存
```

## 10. 缓存穿透击穿雪崩面试速答？

**回答：**

```
Q: 缓存穿透是什么？
A: 查询不存在的数据，缓存和DB都没有
   大量请求打到DB

Q: 怎么防缓存穿透？
A: 布隆过滤器（推荐）+ 缓存空值
   + 参数校验

Q: 缓存击穿是什么？
A: 热点key过期瞬间，大量请求
   涌入DB

Q: 怎么防缓存击穿？
A: 互斥锁（setnx）只让一个请求查DB
   或设置逻辑过期/永不过期

Q: 缓存雪崩是什么？
A: 大量key同时过期或Redis宕机
   DB压力暴增崩溃

Q: 怎么防缓存雪崩？
A: 过期时间加随机值 + Redis高可用
   + 限流降级 + 多级缓存

Q: 布隆过滤器的缺点？
A: 有误判率（可控）
   不支持删除（可用Cuckoo Filter）

Q: 大key怎么删？
A: Redis 4.0+ 用 UNLINK（异步删除）
   老版本 SCAN + 分批删除
```
