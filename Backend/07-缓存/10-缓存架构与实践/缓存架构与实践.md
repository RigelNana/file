# 缓存架构与实践

---

## 1. 多级缓存架构设计？

**回答：**

```
多级缓存架构：
  ┌────────┐    ┌──────────┐    ┌──────────┐    ┌────┐
  │ Client │───→│ L1 本地  │───→│ L2 Redis │───→│ DB │
  │        │    │ 进程内   │    │ 分布式   │    │    │
  └────────┘    └──────────┘    └──────────┘    └────┘
                bigcache         Redis Cluster    MySQL
                ristretto
                freecache

各层特点：
  ┌──────┬──────────┬────────────┬──────────┐
  │ 层级 │ 速度      │ 容量        │ 一致性   │
  ├──────┼──────────┼────────────┼──────────┤
  │ L1   │ ns 级    │ 几百MB      │ 多实例不同│
  │ L2   │ ms 级    │ 几十GB      │ 全局一致  │
  │ DB   │ ms~s 级  │ 无限       │ 权威数据  │
  └──────┴──────────┴────────────┴──────────┘

L1 本地缓存选型：
  bigcache   → 零 GC，适合大量 key
  ristretto  → 高命中率（TinyLFU）
  freecache  → 零 GC，支持过期

L1 适合：
  热点数据 / 变化不频繁 / 容忍短暂不一致
  配置信息 / 字典数据 / 热门商品
```

```go
// 多级缓存实现
type MultiLevelCache struct {
    l1    *ristretto.Cache
    l2    *redis.Client
    l1TTL time.Duration
    l2TTL time.Duration
}

func (c *MultiLevelCache) Get(ctx context.Context, key string) ([]byte, error) {
    // L1
    if val, ok := c.l1.Get(key); ok {
        return val.([]byte), nil
    }
    
    // L2
    val, err := c.l2.Get(ctx, key).Bytes()
    if err == nil {
        c.l1.SetWithTTL(key, val, int64(len(val)), c.l1TTL)
        return val, nil
    }
    
    return nil, ErrCacheMiss
}

func (c *MultiLevelCache) Set(ctx context.Context, key string, val []byte) error {
    c.l1.SetWithTTL(key, val, int64(len(val)), c.l1TTL)
    return c.l2.Set(ctx, key, val, c.l2TTL).Err()
}

func (c *MultiLevelCache) Delete(ctx context.Context, key string) error {
    c.l1.Del(key)
    return c.l2.Del(ctx, key).Err()
}
```

## 2. L1 本地缓存一致性？

**回答：**

```
问题：多个实例的 L1 缓存不一致

  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │Instance 1│  │Instance 2│  │Instance 3│
  │L1: v2    │  │L1: v1    │  │L1: v1    │
  └──────────┘  └──────────┘  └──────────┘
  只有 Instance 1 更新了

解决方案：

1. Redis Pub/Sub 广播通知：
   更新时 PUBLISH 通知所有实例清除 L1
   ┌──────┐ PUBLISH  ┌──────┐ 广播  ┌──────┐
   │Inst 1│─────────→│Redis │──────→│Inst 2│
   │更新L1│          │Pub/Sub│──────→│清L1  │
   └──────┘          └──────┘      └──────┘

2. MQ 广播（更可靠）：
   发送广播消息通知所有实例

3. 短 TTL：
   L1 TTL 设 5~30s
   容忍短暂不一致

4. 版本号机制：
   L2 存版本号，L1 定期比对
```

```go
// Redis Pub/Sub 广播清除 L1
func (c *MultiLevelCache) InvalidateL1(ctx context.Context, key string) {
    // 广播通知所有实例
    c.l2.Publish(ctx, "cache:invalidate", key)
}

// 启动订阅
func (c *MultiLevelCache) StartSubscriber(ctx context.Context) {
    sub := c.l2.Subscribe(ctx, "cache:invalidate")
    ch := sub.Channel()
    
    go func() {
        for msg := range ch {
            c.l1.Del(msg.Payload)
        }
    }()
}
```

## 3. 缓存在秒杀场景的应用？

**回答：**

```
秒杀架构：
  ┌──────┐    ┌──────┐    ┌──────┐    ┌────┐
  │前端  │───→│网关  │───→│Redis │───→│ DB │
  │倒计时│    │限流  │    │扣库存│    │落库│
  └──────┘    └──────┘    └──────┘    └────┘

设计要点：

1. 预热库存到 Redis：
   SET seckill:item:1001 100  # 100 件

2. Lua 原子扣减：
   local stock = redis.call('GET', KEYS[1])
   if tonumber(stock) > 0 then
       redis.call('DECR', KEYS[1])
       return 1
   end
   return 0

3. 异步落库：
   Redis 扣减成功 → 发 MQ 消息
   消费者异步写入 DB 订单

4. 防超卖：
   Lua 脚本保证判断+扣减原子
   分布式锁不适合（太慢）

5. 流量控制：
   网关令牌桶限流
   前端随机延迟 + 验证码
   队列削峰
```

## 4. 缓存在排行榜的应用？

**回答：**

```
Sorted Set 实现排行榜：

  ZADD rank:game score member
  ZREVRANGE rank:game 0 9        → TOP 10
  ZREVRANK rank:game "player1"   → 排名（0-based）
  ZSCORE rank:game "player1"     → 分数
  ZINCRBY rank:game 10 "player1" → 加分
```

```go
// 排行榜服务
type RankService struct {
    rdb *redis.Client
    key string
}

// 更新分数
func (s *RankService) UpdateScore(ctx context.Context, 
    userID string, score float64) error {
    return s.rdb.ZAdd(ctx, s.key, redis.Z{
        Score:  score,
        Member: userID,
    }).Err()
}

// 获取 TOP N
func (s *RankService) TopN(ctx context.Context, n int64) ([]redis.Z, error) {
    return s.rdb.ZRevRangeWithScores(ctx, s.key, 0, n-1).Result()
}

// 获取用户排名（从 0 开始）
func (s *RankService) GetRank(ctx context.Context, userID string) (int64, error) {
    rank, err := s.rdb.ZRevRank(ctx, s.key, userID).Result()
    if err != nil {
        return -1, err
    }
    return rank + 1, nil // 转为 1-based
}

// 获取我的排名和前后 N 名
func (s *RankService) GetAroundMe(ctx context.Context, 
    userID string, n int64) ([]redis.Z, error) {
    rank, err := s.rdb.ZRevRank(ctx, s.key, userID).Result()
    if err != nil {
        return nil, err
    }
    start := rank - n
    if start < 0 {
        start = 0
    }
    return s.rdb.ZRevRangeWithScores(ctx, s.key, start, rank+n).Result()
}
```

## 5. Session 与 Token 缓存？

**回答：**

```
Session 存储方案：

方案1：Redis 存 Session
  登录 → 生成 sessionID → 存 Redis
  SET session:{id} {user_data} EX 1800

  请求带 Cookie/Header → 从 Redis 获取 Session
  → 好处：服务无状态，支持水平扩展

方案2：JWT + Redis 黑名单
  JWT 自包含用户信息
  Redis 只存黑名单（注销的 token）
  SET blacklist:{token_hash} 1 EX {remain_ttl}

  验证时检查黑名单
```

```go
// Session 管理
type SessionStore struct {
    rdb *redis.Client
    ttl time.Duration
}

func (s *SessionStore) Create(ctx context.Context, 
    userID string, data map[string]interface{}) (string, error) {
    
    sessionID := generateSessionID()
    key := "session:" + sessionID
    
    jsonData, _ := json.Marshal(data)
    if err := s.rdb.Set(ctx, key, jsonData, s.ttl).Err(); err != nil {
        return "", err
    }
    return sessionID, nil
}

func (s *SessionStore) Get(ctx context.Context, 
    sessionID string) (map[string]interface{}, error) {
    
    key := "session:" + sessionID
    val, err := s.rdb.Get(ctx, key).Bytes()
    if err != nil {
        return nil, ErrSessionNotFound
    }
    
    // 续期
    s.rdb.Expire(ctx, key, s.ttl)
    
    var data map[string]interface{}
    json.Unmarshal(val, &data)
    return data, nil
}

func (s *SessionStore) Destroy(ctx context.Context, sessionID string) error {
    return s.rdb.Del(ctx, "session:"+sessionID).Err()
}
```

## 6. 计数器与限流？

**回答：**

```
1. 简单计数器：
   INCR page:views:{page_id}
   INCRBY user:points:{uid} 10

2. 固定窗口限流：
   key = rate:{api}:{minute}
   INCR key → 如果 == 1 则 EXPIRE 60
   如果 > limit → 拒绝

3. 滑动窗口限流（ZSet）：
   ZADD key now now
   ZREMRANGEBYSCORE key 0 (now-window)
   ZCARD key → 当前窗口请求数

4. 令牌桶（Lua 脚本）：
   定时添加令牌 + 请求消耗令牌
```

```go
// 滑动窗口限流
func (rl *RateLimiter) Allow(ctx context.Context, 
    key string, limit int64, window time.Duration) bool {
    
    now := time.Now().UnixMilli()
    windowStart := now - window.Milliseconds()
    
    pipe := rl.rdb.Pipeline()
    pipe.ZRemRangeByScore(ctx, key, "0", fmt.Sprint(windowStart))
    pipe.ZAdd(ctx, key, redis.Z{Score: float64(now), Member: now})
    pipe.ZCard(ctx, key)
    pipe.Expire(ctx, key, window)
    
    cmds, err := pipe.Exec(ctx)
    if err != nil {
        return true // 降级放行
    }
    
    count := cmds[2].(*redis.IntCmd).Val()
    return count <= limit
}
```

## 7. 分布式 ID 生成？

**回答：**

```
Redis INCR 生成全局唯一 ID：
  INCR global:id:{biz_type}
  
  优势：简单、有序、高性能
  劣势：依赖 Redis 可用性

优化：批量获取
  每次取 1000 个号段
  本地消耗完再取下一批
  → 减少 Redis 调用
```

```go
// 号段模式分布式ID
type IDGenerator struct {
    rdb     *redis.Client
    key     string
    step    int64
    current int64
    max     int64
    mu      sync.Mutex
}

func (g *IDGenerator) NextID(ctx context.Context) (int64, error) {
    g.mu.Lock()
    defer g.mu.Unlock()
    
    if g.current >= g.max {
        // 获取新号段
        newMax, err := g.rdb.IncrBy(ctx, g.key, g.step).Result()
        if err != nil {
            return 0, err
        }
        g.current = newMax - g.step
        g.max = newMax
    }
    
    g.current++
    return g.current, nil
}

// Snowflake 更常用：
// 时间戳(41) + 机器ID(10) + 序列号(12)
// 不依赖外部存储
```

## 8. 延迟队列实现？

**回答：**

```
ZSet 实现延迟队列：
  score = 执行时间戳
  member = 任务内容

  添加任务：
  ZADD delay_queue {execute_time} {task_data}

  消费任务：
  ZRANGEBYSCORE delay_queue 0 {now} LIMIT 0 1
  → 取出到期任务 → ZREM 删除
```

```go
// 延迟队列
type DelayQueue struct {
    rdb *redis.Client
    key string
}

func (q *DelayQueue) Add(ctx context.Context, 
    taskID string, data interface{}, delay time.Duration) error {
    executeAt := time.Now().Add(delay).Unix()
    taskData, _ := json.Marshal(data)
    
    return q.rdb.ZAdd(ctx, q.key, redis.Z{
        Score:  float64(executeAt),
        Member: fmt.Sprintf("%s:%s", taskID, taskData),
    }).Err()
}

func (q *DelayQueue) Consume(ctx context.Context, 
    handler func(string) error) {
    
    for {
        now := float64(time.Now().Unix())
        // 获取到期任务
        results, err := q.rdb.ZRangeByScore(ctx, q.key, &redis.ZRangeBy{
            Min:   "0",
            Max:   fmt.Sprintf("%f", now),
            Count: 1,
        }).Result()
        
        if err != nil || len(results) == 0 {
            time.Sleep(100 * time.Millisecond)
            continue
        }
        
        task := results[0]
        // 原子取出（防止重复消费）
        removed, _ := q.rdb.ZRem(ctx, q.key, task).Result()
        if removed > 0 {
            handler(task)
        }
    }
}
```

## 9. Redis 缓存设计规范？

**回答：**

```
1. Key 命名规范：
   {业务}:{对象}:{id}
   user:profile:1001
   order:detail:2001
   统一前缀便于管理和查找

2. Value 规范：
   避免 > 10KB → 拆分 / 压缩
   选合适的数据结构
   Hash 代替多个 String key

3. TTL 规范：
   必须设置过期时间
   加随机值避免同时过期
   TTL = base + random(0, range)

4. 读写规范：
   避免 KEYS * → 用 SCAN
   避免大量 DEL → 用 UNLINK
   批量操作用 Pipeline

5. 容量规划：
   预估 key 数量和大小
   maxmemory = 物理内存 × 70%
   预留 COW + 碎片开销

6. 一致性规范：
   默认 Cache-Aside 模式
   先更新 DB 后删缓存
   MQ 补偿 + TTL 兜底

7. 监控规范：
   内存 / QPS / 命中率 / 延迟
   告警阈值设定
   慢查询定期检查
```

## 10. 缓存架构面试速答？

**回答：**

```
Q: 多级缓存怎么设计？
A: L1 本地缓存(bigcache/ristretto)
   L2 Redis + L3 数据库

Q: 多实例 L1 缓存怎么保一致？
A: Redis Pub/Sub 广播清除通知
   或设短 TTL 容忍短暂不一致

Q: 秒杀怎么用缓存？
A: 库存预热到Redis → Lua原子扣减
   → MQ异步落库

Q: 排行榜怎么实现？
A: ZSet：ZADD 更新分数
   ZREVRANGE 获取 TOP N

Q: 限流怎么用 Redis 实现？
A: 固定窗口：INCR+EXPIRE
   滑动窗口：ZSet+时间戳
   令牌桶：Lua 脚本

Q: 延迟队列怎么实现？
A: ZSet: score=执行时间戳
   消费者轮询 ZRANGEBYSCORE

Q: Key 命名有什么规范？
A: 业务:对象:id 格式
   如 user:profile:1001

Q: 缓存容量怎么规划？
A: maxmemory = 物理内存 × 70%
   预留fork+碎片开销
```
