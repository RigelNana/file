# 代码整洁原则

---

## 1. Clean Code的核心理念？

**回答：**

```
  Bob大叔《Clean Code》核心：
  代码被读的次数远多于写 → 可读性第一

  整洁代码的特征：
  ┌──────────────────────┬──────────────────────────────┐
  │ 特征                  │ 说明                         │
  ├──────────────────────┼──────────────────────────────┤
  │ 有意义的命名          │ 名字表达意图                 │
  │ 小函数                │ 每个函数只做一件事           │
  │ 最少注释              │ 代码自解释                   │
  │ 统一格式              │ 团队一致的代码风格           │
  │ 没有重复              │ DRY原则                      │
  │ 最小惊讶              │ 行为符合预期                 │
  └──────────────────────┴──────────────────────────────┘

  判断标准：
  WTF/min → 代码审查时的困惑次数越少越好

  实践原则：
  1. 童子军规则：离开时比来时更干净
  2. 破窗理论：不容忍第一个烂代码
  3. KISS：Keep It Simple, Stupid
  4. YAGNI：You Ain't Gonna Need It
     不写用不到的代码
```

## 2. 可读性为什么是第一优先级？

**回答：**

```
  代码生命周期：
  写1次 → 读N次（Code Review/调试/修改/交接）

  可读性直接影响：
  1. 维护成本
     80%的时间花在维护而非开发
     可读性差 → 修改一个Bug引入三个
  
  2. 协作效率
     新人上手速度
     Code Review效率
  
  3. 正确性
     不理解的代码无法确认正确性
     越难读越容易改出Bug

  可读性优先于：
  ✗ 性能：先写清楚 再优化瓶颈
  ✗ 简洁：过度简洁反而难读
  ✗ 聪明：聪明的代码是维护噩梦

  Go示例：
  // 差：聪明但难读
  if v := m[k]; v != nil && v.f() && !v.d {
      return v.p(x, y, z)
  }
  
  // 好：清晰易读
  user, exists := userCache[userID]
  if !exists {
      return nil, ErrUserNotFound
  }
  if !user.IsActive() {
      return nil, ErrUserInactive
  }
  return user.GetProfile()
```

## 3. DRY原则与合理复用？

**回答：**

```
  DRY = Don't Repeat Yourself
  每一份知识在系统中只有唯一、明确的表示

  DRY不是消灭所有重复代码：
  ┌──────────────────────┬──────────────────────┐
  │ 场景                  │ 处理方式             │
  ├──────────────────────┼──────────────────────┤
  │ 真正的重复（同一逻辑）│ 提取函数/方法        │
  │ 偶然的重复（目前一样）│ 保持独立 各自演化    │
  │ 跨层重复（DTO转换）   │ 可以接受的重复       │
  └──────────────────────┴──────────────────────┘

  三次法则（Rule of Three）：
  第1次：写
  第2次：忍
  第3次：重构

  过度DRY的坏处：
  // 强行复用 → 需要大量参数/条件分支
  func ProcessEntity(entityType string, id int, 
      withCache bool, validate bool, ...) {
      if entityType == "user" { ... }
      if entityType == "order" { ... }
      // 越来越复杂
  }

  // 分开更清晰
  func ProcessUser(id int) { ... }
  func ProcessOrder(id int) { ... }

  判断是否需要DRY：
  问：如果这个逻辑变了 两处都要改吗？
  是 → 提取复用
  否 → 各自独立
```

## 4. KISS与YAGNI？

**回答：**

```
  KISS = Keep It Simple, Stupid
  用最简单的方案解决问题

  YAGNI = You Ain't Gonna Need It
  不写现在用不到的功能

  KISS违反示例：
  // 过度设计：只需要一个简单的配置读取
  type ConfigLoader interface { ... }
  type ConfigLoaderFactory struct { ... }
  type ConfigStrategy interface { ... }
  type YAMLConfigStrategy struct { ... }
  type JSONConfigStrategy struct { ... }
  type ConfigWatcher struct { ... }
  
  // KISS：当前只需要读YAML
  func LoadConfig(path string) (*Config, error) {
      data, err := os.ReadFile(path)
      if err != nil { return nil, err }
      var cfg Config
      return &cfg, yaml.Unmarshal(data, &cfg)
  }

  YAGNI违反示例：
  // "将来可能需要支持多种数据库"
  type DatabaseAdapter interface { ... }
  type MySQLAdapter struct { ... }
  type PostgresAdapter struct { ... }  // 从来没用过
  type MongoAdapter struct { ... }     // 从来没用过
  
  // 当前只用MySQL → 直接用MySQL
  // 等真需要时再抽象

  核心：
  简单 > 复杂
  当下 > 将来
  必要 > 可能
```

## 5. 最小惊讶原则？

**回答：**

```
  Principle of Least Astonishment
  代码的行为应该符合读者的预期

  违反示例：
  // GetUser 居然会修改数据？
  func GetUser(id int) (*User, error) {
      user, err := db.Find(id)
      user.LastAccessTime = time.Now()
      db.Save(user)  // 副作用！Get不应该修改
      return user, err
  }
  
  // Delete 返回了被删除的对象？
  func DeleteOrder(id int) *Order { ... }
  
  // IsValid 结果还纠正了数据？
  func (u *User) IsValid() bool {
      if u.Name == "" {
          u.Name = "Unknown"  // 偷偷修改！
      }
      return true
  }

  正确做法：
  func GetUser(id int) (*User, error)    // 只读
  func UpdateLastAccess(id int) error    // 明确修改
  func DeleteOrder(id int) error          // 只删除
  func (u *User) IsValid() bool           // 只验证

  关键规则：
  1. 命名暗示行为（Get=读 Set=写 Is=判断）
  2. 函数无隐藏副作用
  3. 参数顺序符合习惯（src, dst）
  4. 返回值类型符合预期
  5. 错误处理方式一致
```

## 6. 单一职责怎么落地？

**回答：**

```
  SRP = 一个模块只有一个变更的理由

  判断是否违反SRP：
  "这个包/结构体修改的原因有几个？"
  > 1个 → 需要拆分

  Go包级别SRP：
  // 差：user包做了太多事
  package user
  - CreateUser()
  - ValidatePassword()
  - SendWelcomeEmail()    // 邮件不应在user包
  - GenerateReport()       // 报表不应在user包
  
  // 好：各司其职
  package user      → CRUD
  package auth      → 认证/密码
  package email     → 邮件发送
  package report    → 报表生成

  Go结构体级别SRP：
  // 差：Service太大
  type OrderService struct { ... }
  func (s *OrderService) CreateOrder() { ... }
  func (s *OrderService) SendNotification() { ... }
  func (s *OrderService) GenerateInvoice() { ... }
  func (s *OrderService) CalculateShipping() { ... }
  
  // 好：拆分
  type OrderService struct { ... }       // 订单核心
  type NotificationService struct { ... } // 通知
  type InvoiceService struct { ... }      // 发票
  type ShippingService struct { ... }     // 物流

  衡量标准：
  能用一句话描述这个包/结构体做什么
  如果用了"和"/"并且" → 需要拆
```

## 7. 关注点分离？

**回答：**

```
  Separation of Concerns
  不同关注点放在不同层/模块

  后端分层示例：
  ┌──────────────────────────┐
  │   Handler/Controller     │ → HTTP/请求解析/响应
  ├──────────────────────────┤
  │   Service/UseCase        │ → 业务逻辑
  ├──────────────────────────┤
  │   Repository/Store       │ → 数据访问
  ├──────────────────────────┤
  │   Model/Entity           │ → 数据结构
  └──────────────────────────┘

  每一层只关心自己的事：
  // Handler → 只管HTTP 不管业务逻辑
  func (h *Handler) CreateOrder(w http.ResponseWriter, r *http.Request) {
      var req CreateOrderRequest
      if err := json.NewDecoder(r.Body).Decode(&req); err != nil {
          http.Error(w, "bad request", 400)
          return
      }
      
      order, err := h.orderService.Create(r.Context(), req)
      if err != nil {
          handleError(w, err)
          return
      }
      
      json.NewEncoder(w).Encode(order)
  }
  
  // Service → 只管业务 不管HTTP
  func (s *OrderService) Create(ctx context.Context, req CreateOrderRequest) (*Order, error) {
      // 业务校验
      // 业务逻辑
      // 调用Repository
  }
  
  // Repository → 只管数据 不管业务
  func (r *OrderRepo) Save(ctx context.Context, order *Order) error {
      return r.db.WithContext(ctx).Create(order).Error
  }

  好处：
  改HTTP框架 → 只改Handler层
  改数据库 → 只改Repository层
  业务逻辑独立可测试
```

## 8. 代码格式与一致性？

**回答：**

```
  格式规则：
  1. 团队统一 > 个人偏好
  2. 用工具自动化 不靠自觉

  Go格式化工具链：
  gofmt        → 官方格式化（必须）
  goimports    → 自动管理import
  golangci-lint → 集成40+检查器

  格式要点：
  垂直格式：
  - 相关代码靠近（相关函数放在一起）
  - 调用者在被调用者上方
  - 变量声明靠近使用处
  
  水平格式：
  - 行宽不超过120字符
  - 适当的空格和对齐
  
  分组：
  import (
      // 标准库
      "context"
      "fmt"
      
      // 第三方
      "github.com/gin-gonic/gin"
      
      // 本项目
      "myapp/internal/service"
  )

  .golangci.yml 示例：
  linters:
    enable:
      - gofmt
      - goimports
      - govet
      - errcheck
      - staticcheck
      - gosimple
      - unused
      - misspell
      - gocyclo
  
  linters-settings:
    gocyclo:
      min-complexity: 15

  CI强制：
  golangci-lint run ./...
  不通过不允许合并
```

## 9. 技术债与代码腐化？

**回答：**

```
  代码腐化的征兆：
  ┌──────────────────────┬──────────────────────────────┐
  │ 症状                  │ 表现                         │
  ├──────────────────────┼──────────────────────────────┤
  │ 僵化(Rigidity)        │ 改一处牵动多处               │
  │ 脆弱(Fragility)       │ 修一个Bug引入新Bug           │
  │ 不可移植(Immobility)  │ 无法复用 只能复制            │
  │ 粘滞(Viscosity)       │ 做正确的事比做错的事更难     │
  │ 不必要的复杂           │ 过度设计                     │
  │ 不必要的重复           │ 到处Copy-Paste              │
  └──────────────────────┴──────────────────────────────┘

  技术债管理：
  1. 识别
     Code Review发现
     静态分析工具检测
     开发者主动标记TODO
  
  2. 记录
     Issue/Backlog跟踪
     标注优先级和影响范围
  
  3. 偿还
     每个Sprint分配20%时间
     大重构做技术方案评审
     渐进式重构（不要大爆炸）
  
  4. 预防
     Code Review把关
     CI自动检查
     编码规范文档
     新人培训

  TODO规范：
  // TODO(zhangsan): 优化N+1查询 #issue-123
  // FIXME: 并发竞态 需要加锁
  // HACK: 临时方案 下版本重构
```

**回答：**

```

   好命名+小函数+无重复

   偶然重复可以保留 三次法则

Q: KISS和YAGNI区别？
A: KISS强调方案简单
   YAGNI强调不做用不到的功能

Q: 什么是最小惊讶原则？
A: 代码行为符合命名暗示
   Get只读 没有副作用

Q: 单一职责怎么判断？

   如果描述用了"和" 就需要拆

Q: 技术债怎么管理？
A: 识别→记录→偿还→预防
   每Sprint分配20%时间偿还

Q: 代码腐化的信号？
A: 改一处牵动多处(僵化)
   修Bug引入新Bug(脆弱)

Q: Go怎么保证代码规范？

   CI强制通过 Code Review把关
```
