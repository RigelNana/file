# 命名与注释规范

---

## 1. 命名的基本原则？

**回答：**

```
  核心：名副其实 看到名字就知道含义

  ┌──────────────┬───────────────┬───────────────┐
  │ 规则          │ 差             │ 好             │
  ├──────────────┼───────────────┼───────────────┤
  │ 表达意图      │ d             │ elapsedDays   │
  │ 避免误导      │ accountList   │ accounts      │
  │ 有意义区分    │ data/info     │ rawInput/parsed│
  │ 可搜索        │ 7             │ MaxRetryCount │
  │ 不编码类型    │ nameString    │ name          │
  │ 统一概念      │ get/fetch混用 │ 统一用Get      │
  └──────────────┴───────────────┴───────────────┘

  命名长度 ∝ 作用域大小：
  for i := 0; i < 10; i++     // 局部短命名OK
  var userRepository *UserRepo  // 包级别长命名

  布尔命名用肯定形式：
  isActive / hasPermission / canDelete
  不用 isNotActive / noPermission
```

## 2. Go语言命名规范？

**回答：**

```
  Go特有规范：
  1. 大小写控制可见性
     大写开头 = 导出（Public）
     小写开头 = 未导出（Private）
  
  2. 包名
     小写单词 不用下划线/驼峰
     短且有意义
     ✗ string_utils → ✓ stringutil
  
  3. 缩写全大写
     HTTP / URL / ID / API / JSON
     不是 Http / Url / Id
  
  4. 接口命名
     单方法接口：方法名 + er
     Reader / Writer / Closer / Stringer
  
  5. 错误变量
     var ErrNotFound = errors.New("not found")
     var ErrTimeout  = errors.New("timeout")
  
  6. Getter不用Get前缀
     user.Name()   ✓      // 不是 user.GetName()
     user.SetName() ✓     // Setter保留Set
  
  7. 构造函数
     NewServer()          // 返回 *Server
     NewServerWithConfig() // 带选项

  Go惯用命名：
  ctx   → context.Context
  err   → error
  mu    → sync.Mutex
  wg    → sync.WaitGroup
  ok    → 布尔判断
  n     → 数量/长度
  i,j,k → 循环变量
```

## 3. 变量命名策略？

**回答：**

```
  按类型命名：
  布尔：is/has/can/should + 形容词/过去分词
    isActive, hasPermission, canRetry, shouldNotify

  集合：复数名词
    users, orderItems, validEmails

  Map：key + To + value
    userIDToName, emailToUser

  Channel：描述内容
    jobs := make(chan Job)
    results := make(chan Result)
    done := make(chan struct{})

  函数/方法：动词开头
    CreateUser, ValidateInput, ParseConfig
    handleRequest, buildQuery, formatOutput

  避免上下文重复：
  // 差：包名已经说了user
  package user
  type UserService struct{}
  func (s *UserService) GetUser() {}
  
  // 好
  package user
  type Service struct{}
  func (s *Service) Get() {}
  // 调用：user.NewService().Get()

  常量命名：
  const (
      MaxRetryCount    = 3
      DefaultTimeout   = 30 * time.Second
      StatusActive     = "active"
  )
```

## 4. 好注释与坏注释？

**回答：**

```
  好注释（应该写的）：
  1. 法律信息
     // Copyright 2024 Company. All rights reserved.
  
  2. 解释"为什么"（而非"是什么"）
     // 使用指数退避避免压垮下游服务
     delay := baseDelay * time.Duration(math.Pow(2, float64(attempt)))
  
  3. 警示
     // WARNING: 此函数非并发安全，调用方需持有锁
  
  4. TODO
     // TODO(zhangsan): 优化为批量查询 #issue-456
  
  5. 公共API文档（godoc）
     // Server 是HTTP服务器的核心结构体。
     // 它管理路由注册和请求处理。
     type Server struct { ... }

  坏注释（不应该写的）：
  // 差：重复代码内容
  // 将i加1
  i++
  
  // 差：注释掉的代码（用Git管理）
  // oldFunction()
  // anotherOldFunction()
  
  // 差：日志式注释
  // 2024-01-01 zhangsan 添加了验证逻辑
  
  // 差：括号闭合注释
  } // end if
  } // end for
  
  // 差：多余的注释
  // GetUser 获取用户
  func GetUser() {}

  原则：如果能通过改名让代码自解释
  就不需要注释
```

## 5. godoc注释规范？

**回答：**

```
  Go文档注释规范：
  1. 导出符号都要文档注释
  2. 以被注释的名字开头
  3. 完整句子 句号结尾

  包注释：
  // Package user provides user management functionality
  // including CRUD operations, authentication, and profile management.
  package user

  类型注释：
  // Order represents a customer order in the system.
  // It tracks the order lifecycle from creation to fulfillment.
  type Order struct {
      ID        string
      UserID    string
      Status    OrderStatus
      CreatedAt time.Time
  }

  函数注释：
  // CreateOrder creates a new order for the given user.
  // It validates the order items, calculates pricing,
  // and persists the order to the database.
  //
  // It returns ErrInvalidItem if any order item is invalid.
  // It returns ErrInsufficientStock if stock is not available.
  func CreateOrder(ctx context.Context, req CreateOrderRequest) (*Order, error) {

  示例测试（出现在文档中）：
  func ExampleCreateOrder() {
      order, err := CreateOrder(ctx, req)
      if err != nil {
          log.Fatal(err)
      }
      fmt.Println(order.ID)
      // Output: order-123
  }

  查看文档：
  go doc ./...
  godoc -http=:6060
```

## 6. 魔法数字与常量提取？

**回答：**

```
  魔法数字 = 代码中裸露的字面量
  不知道含义 无法搜索 改一处漏一处

  // 差：魔法数字
  if retries > 3 { return err }
  if status == 1 { ... }
  time.Sleep(5 * time.Second)
  buf := make([]byte, 4096)
  
  // 好：命名常量
  const MaxRetries = 3
  if retries > MaxRetries { return err }
  
  const StatusActive = 1
  if status == StatusActive { ... }
  
  const RetryDelay = 5 * time.Second
  time.Sleep(RetryDelay)
  
  const DefaultBufSize = 4096
  buf := make([]byte, DefaultBufSize)

  枚举常量用iota：
  type OrderStatus int
  const (
      OrderPending   OrderStatus = iota // 0
      OrderConfirmed                     // 1
      OrderShipped                       // 2
      OrderDelivered                     // 3
      OrderCancelled                     // 4
  )
  
  func (s OrderStatus) String() string {
      names := [...]string{
          "pending", "confirmed", "shipped",
          "delivered", "cancelled",
      }
      if int(s) >= len(names) { return "unknown" }
      return names[s]
  }

  例外：0、1、""等显而易见的值不需要常量
  for i := 0; i < len(items); i++  // 0是显而易见的
```

## 7. 命名中的一致性？

**回答：**

```
  同一概念用同一个词：
  ┌──────────────────────┬──────────────────────┐
  │ 混乱                  │ 一致                 │
  ├──────────────────────┼──────────────────────┤
  │ Get/Fetch/Retrieve   │ 统一用Get             │
  │ Create/Add/Insert    │ 统一用Create          │
  │ Delete/Remove/Destroy│ 统一用Delete          │
  │ Update/Modify/Change │ 统一用Update          │
  │ List/Find/Query      │ 统一用List            │
  └──────────────────────┴──────────────────────┘

  CRUD命名一致：
  type UserService interface {
      CreateUser(ctx context.Context, req *CreateUserReq) (*User, error)
      GetUser(ctx context.Context, id string) (*User, error)
      UpdateUser(ctx context.Context, id string, req *UpdateUserReq) error
      DeleteUser(ctx context.Context, id string) error
      ListUsers(ctx context.Context, filter *UserFilter) ([]*User, error)
  }
  
  type OrderService interface {
      CreateOrder(...)  // 同样的命名模式
      GetOrder(...)
      UpdateOrder(...)
      DeleteOrder(...)
      ListOrders(...)
  }

  分层命名一致：
  Handler层: HandleCreateUser / HandleGetUser
  Service层: CreateUser / GetUser
  Repo层:    InsertUser / FindUser / ... 
             (Repo可以用数据库术语)

  项目内形成命名词典：
  统一发送通知 → Notify
  统一验证 → Validate
  统一转换 → Convert
```

## 8. 领域术语与上下文？

**回答：**

```
  使用业务领域术语命名：
  // 差：技术术语
  type DataProcessor struct{}
  func (p *DataProcessor) ProcessData()
  
  // 好：领域术语 
  type PaymentGateway struct{}
  func (g *PaymentGateway) ChargeCard()

  领域词汇表（Ubiquitous Language）：
  团队统一的业务术语 代码和文档一致
  
  电商领域：
  Order（订单）  SKU（库存单位）
  Cart（购物车）  Checkout（结账）
  Refund（退款）  Fulfillment（履约）

  避免过度缩写：
  // 差
  func CalcOrdTotAmt(ord *Ord) float64
  
  // 好
  func CalculateOrderTotal(order *Order) float64

  上下文决定命名长度：
  // 包内使用 短名称
  package order
  type Service struct{}     // order.Service 已经够清晰
  
  // 跨包使用 需要更多上下文
  orderSvc := order.NewService()
  paymentGW := payment.NewGateway()

  不要在命名中加类型后缀：
  userMap → users
  nameString → name
  itemList → items
```

## 9. 注释驱动重构？

**回答：**

```
  当你想写注释时 先想能否重构让代码自解释

  重构前（需要注释）：
  // 检查用户是否有权限访问订单
  if user.Role == "admin" || 
     (user.Role == "user" && order.UserID == user.ID) ||
     (user.Role == "support" && order.Status == "disputed") {
      // 处理订单
  }

  重构后（代码自解释）：
  if user.CanAccessOrder(order) {
      // 处理订单
  }
  
  func (u *User) CanAccessOrder(order *Order) bool {
      switch u.Role {
      case "admin":
          return true
      case "user":
          return order.UserID == u.ID
      case "support":
          return order.Status == "disputed"
      default:
          return false
      }
  }

  常见重构手法：
  ┌──────────────────────┬──────────────────────────┐
  │ 注释描述              │ 重构方式                 │
  ├──────────────────────┼──────────────────────────┤
  │ 解释复杂条件          │ 提取为HasXxx/IsXxx方法   │
  │ 解释代码块做什么      │ 提取为独立函数           │
  │ 解释变量含义          │ 改用更好的变量名         │
  │ 解释参数含义          │ 使用Options结构体        │
  └──────────────────────┴──────────────────────────┘

  如果重构后仍需注释 → 说明"为什么"
  // 使用事务是因为订单和库存需要原子更新
  // 而库存服务不支持saga模式
```

## 10. 命名与注释面试速答？

**回答：**

```
Q: 好命名的标准？
A: 名副其实 看名字知道含义
   长度与作用域成正比

Q: Go命名特点？
A: 大小写控制可见性
   缩写全大写(HTTP/ID)
   接口加er(Reader)
   Getter不加Get

Q: 变量怎么命名？
A: 布尔is/has/can 集合用复数
   函数动词开头 避免上下文重复

Q: 什么时候写注释？
A: 解释"为什么"不是"是什么"
   公共API文档/警示/TODO

Q: 什么是坏注释？
A: 重复代码/注释掉的代码
   日志式注释/括号闭合注释

Q: 魔法数字怎么处理？
A: 提取为命名常量
   枚举用iota

Q: 怎么保持命名一致？
A: 同概念同一词(Get/Create/Delete)
   项目形成命名词典

Q: 注释太多说明什么？
A: 代码可读性不够
   先尝试重构让代码自解释
```
