# 重构技巧

---

## 1. 什么是重构？

**回答：**

```
  重构 = 在不改变外部行为的前提下
        改善代码内部结构

  关键前提：
  1. 有测试保护（重构前先补测试）
  2. 小步前进（每次改一点 运行测试）
  3. 不同时改功能和结构

  重构 vs 重写：
  ┌──────────────┬───────────────┬───────────────┐
  │ 维度          │ 重构           │ 重写           │
  ├──────────────┼───────────────┼───────────────┤
  │ 风险          │ 低（渐进式）   │ 高（一次性）   │
  │ 时间          │ 持续进行       │ 集中较长       │
  │ 测试          │ 现有测试可用   │ 需要全部重测   │
  │ 适用          │ 结构尚可改善   │ 完全不可维护   │
  └──────────────┴───────────────┴───────────────┘

  什么时候重构：
  1. 添加功能前 → 让代码更容易扩展
  2. 修Bug时 → 让代码更清晰
  3. Code Review时 → 发现问题及时改
  4. 三次法则 → 第三次重复时
  
  什么时候不重构：
  ✗ 即将上线赶工期
  ✗ 代码完全不理解（先读懂再说）
  ✗ 没有测试保护
```

## 2. 提取函数/方法？

**回答：**

```
  最常用的重构手法
  将代码块提取为独立函数

  重构前：
  func CreateOrder(ctx context.Context, req *CreateOrderReq) (*Order, error) {
      // 验证输入 — 20行
      if req.UserID == "" { return nil, ErrInvalid }
      if len(req.Items) == 0 { return nil, ErrEmptyCart }
      for _, item := range req.Items {
          if item.Quantity <= 0 { return nil, ErrInvalidQuantity }
          product, err := productRepo.Get(ctx, item.ProductID)
          if err != nil { return nil, err }
          if product.Stock < item.Quantity { return nil, ErrInsufficientStock }
      }
      
      // 计算价格 — 15行
      var total float64
      for _, item := range req.Items {
          // 复杂的价格计算逻辑...
      }
      
      // 创建订单 — 10行
      order := &Order{...}
      if err := orderRepo.Save(ctx, order); err != nil {
          return nil, err
      }
      
      // 发送通知 — 5行
      // ...
      
      return order, nil
  }

  重构后：
  func CreateOrder(ctx context.Context, req *CreateOrderReq) (*Order, error) {
      if err := validateOrderRequest(ctx, req); err != nil {
          return nil, err
      }
      
      total, err := calculateTotal(req.Items)
      if err != nil {
          return nil, err
      }
      
      order, err := saveOrder(ctx, req, total)
      if err != nil {
          return nil, err
      }
      
      notifyOrderCreated(ctx, order) // 异步 不影响主流程
      return order, nil
  }

  提取时机：看到注释分隔代码块 → 提取
```

## 3. 消除重复代码？

**回答：**

```
  三种常见重复模式及处理：

  1. 相同代码块 → 提取函数
  // 重构前：多处重复的错误响应
  func HandlerA(w http.ResponseWriter, r *http.Request) {
      // ...
      w.Header().Set("Content-Type", "application/json")
      w.WriteHeader(400)
      json.NewEncoder(w).Encode(map[string]string{"error": msg})
  }
  
  // 重构后
  func writeError(w http.ResponseWriter, status int, msg string) {
      w.Header().Set("Content-Type", "application/json")
      w.WriteHeader(status)
      json.NewEncoder(w).Encode(map[string]string{"error": msg})
  }

  2. 相似代码（结构相同 细节不同）→ 参数化/模板方法
  // 重构前：CRUD逻辑重复
  func CreateUser() { validate(); save(); notify(); }
  func CreateOrder() { validate(); save(); notify(); }
  
  // 重构后：泛型Repository（Go 1.18+）
  type Repository[T any] struct { db *gorm.DB }
  func (r *Repository[T]) Create(ctx context.Context, entity *T) error {
      return r.db.WithContext(ctx).Create(entity).Error
  }

  3. 跨模块重复 → 提取公共包
  // internal/response/ 统一响应
  func JSON(w http.ResponseWriter, status int, data interface{}) {
      w.Header().Set("Content-Type", "application/json")
      w.WriteHeader(status)
      json.NewEncoder(w).Encode(data)
  }

  注意：只消除真正的重复（同一知识）
  偶然重复不要强行合并
```

## 4. 简化条件表达式？

**回答：**

```
  1. 卫语句替代嵌套if
  // 差：深层嵌套
  func Process(req *Request) error {
      if req != nil {
          if req.IsValid() {
              if req.HasPermission() {
                  // 真正的逻辑
                  return execute(req)
              } else {
                  return ErrNoPermission
              }
          } else {
              return ErrInvalid
          }
      }
      return ErrNilRequest
  }
  
  // 好：卫语句 提前返回
  func Process(req *Request) error {
      if req == nil {
          return ErrNilRequest
      }
      if !req.IsValid() {
          return ErrInvalid
      }
      if !req.HasPermission() {
          return ErrNoPermission
      }
      return execute(req)
  }

  2. 提取条件为方法
  // 差
  if user.Age >= 18 && user.Status == "active" && !user.IsBanned {
  
  // 好
  if user.CanPlaceOrder() {
  
  func (u *User) CanPlaceOrder() bool {
      return u.Age >= 18 && u.Status == "active" && !u.IsBanned
  }

  3. 用Map替代switch（当分支纯数据时）
  // 差：长switch
  func GetStatusText(code int) string {
      switch code {
      case 1: return "pending"
      case 2: return "active"
      // ...20个case
      }
  }
  
  // 好：Map查表
  var statusTexts = map[int]string{
      1: "pending", 2: "active", // ...
  }
  func GetStatusText(code int) string { return statusTexts[code] }
```

## 5. 引入参数对象？

**回答：**

```
  参数列表太长 → 封装为结构体

  重构前：
  func SearchOrders(
      userID string,
      status string,
      startDate time.Time,
      endDate time.Time,
      minAmount float64,
      maxAmount float64,
      page int,
      pageSize int,
      sortBy string,
      sortOrder string,
  ) ([]*Order, int64, error)

  重构后：
  type OrderFilter struct {
      UserID    string
      Status    string
      StartDate time.Time
      EndDate   time.Time
      MinAmount float64
      MaxAmount float64
  }
  
  type Pagination struct {
      Page     int
      PageSize int
      SortBy   string
      Order    string  // asc/desc
  }
  
  func (p *Pagination) Offset() int {
      return (p.Page - 1) * p.PageSize
  }
  
  type PageResult[T any] struct {
      Items []*T
      Total int64
      Page  int
  }
  
  func SearchOrders(ctx context.Context, filter *OrderFilter, 
      page *Pagination) (*PageResult[Order], error)

  好处：
  1. 参数有意义的分组
  2. 参数对象可复用（Pagination各处通用）
  3. 可以给参数对象加方法
  4. 新增字段不改函数签名
```

## 6. 用策略替换条件？

**回答：**

```
  当switch/if-else处理不同策略时 → 用接口+策略模式

  重构前：
  func CalculateShipping(method string, weight float64) float64 {
      switch method {
      case "standard":
          if weight <= 1 { return 10 }
          return 10 + (weight-1)*5
      case "express":
          if weight <= 1 { return 20 }
          return 20 + (weight-1)*8
      case "overnight":
          return 50 + weight*10
      default:
          return 0
      }
  }

  重构后：
  type ShippingCalculator interface {
      Calculate(weight float64) float64
  }
  
  type StandardShipping struct{}
  func (s StandardShipping) Calculate(weight float64) float64 {
      if weight <= 1 { return 10 }
      return 10 + (weight-1)*5
  }
  
  type ExpressShipping struct{}
  func (e ExpressShipping) Calculate(weight float64) float64 {
      if weight <= 1 { return 20 }
      return 20 + (weight-1)*8
  }

  策略注册表：
  var shippingStrategies = map[string]ShippingCalculator{
      "standard":  StandardShipping{},
      "express":   ExpressShipping{},
      "overnight": OvernightShipping{},
  }
  
  func CalculateShipping(method string, weight float64) float64 {
      calc, ok := shippingStrategies[method]
      if !ok { return 0 }
      return calc.Calculate(weight)
  }

  适用场景：
  分支逻辑复杂 且会继续增长
  简单的2-3个分支不需要过度设计
```

## 7. 拆分大型结构体/包？

**回答：**

```
  识别大型结构体：
  方法超过15个 → 需要拆分
  字段超过10个 → 考虑分组

  拆分方法１ — 按职责拆分：
  // 差：God Object
  type OrderService struct {
      db     *gorm.DB
      cache  *redis.Client
      mq     *kafka.Producer
      email  *EmailClient
      sms    *SMSClient
  }
  func (s *OrderService) Create()      // 订单
  func (s *OrderService) SendEmail()   // 邮件
  func (s *OrderService) SendSMS()     // 短信
  func (s *OrderService) SyncCache()   // 缓存
  func (s *OrderService) PublishEvent()// 事件
  
  // 好：按职责拆分
  type OrderService struct { ... }     // 核心订单逻辑
  type NotificationService struct { ... } // 通知
  type OrderCacheService struct { ... }   // 缓存
  type EventPublisher struct { ... }      // 事件发布

  拆分包：
  // 差：大杂烩包
  package utils
  ├── string_utils.go
  ├── time_utils.go
  ├── http_utils.go
  └── crypto_utils.go
  
  // 好：按功能拆分
  package stringutil  → 字符串工具
  package timeutil    → 时间工具
  package httputil    → HTTP工具
  package cryptoutil  → 加密工具

  Go包拆分原则：
  一个包一个职责
  避免循环依赖
  包名即用途
```

## 8. 渐进式重构策略？

**回答：**

```
  大规模重构 = 多次小步重构

  策略：Strangler Fig Pattern（绞杀者模式）
  ┌──────────────────────────────────┐
  │ 新代码逐步替换旧代码              │
  │ 旧代码和新代码共存                │
  │ 流量逐步迁移到新代码              │
  │ 旧代码最终完全废弃                │
  └──────────────────────────────────┘

  步骤：
  1. 写测试保护现有行为
  2. 新建新的实现（不动旧代码）
  3. 用接口/适配器切换新旧实现
  4. 灰度验证新实现
  5. 完全切换后删除旧代码

  示例：重构数据访问层
  // 阶段1：抽象接口
  type UserStore interface {
      FindByID(ctx context.Context, id string) (*User, error)
  }
  
  // 阶段2：旧实现和新实现并存
  type OldUserStore struct { ... }  // 旧实现
  type NewUserStore struct { ... }  // 新实现
  
  // 阶段3：通过Feature Flag切换
  func NewUserStoreSelector(old, new UserStore) UserStore {
      return &userStoreSelector{old: old, new: new}
  }
  func (s *userStoreSelector) FindByID(ctx context.Context, id string) (*User, error) {
      if featureFlag.IsEnabled("new_user_store") {
          return s.new.FindByID(ctx, id)
      }
      return s.old.FindByID(ctx, id)
  }
  
  // 阶段4：验证通过后删除旧实现

  每一步都可以发布 可以回滚
```

## 9. 重构的安全网？

**回答：**

```
  重构前必须有安全网 = 测试

  安全网层次：
  1. 单元测试 → 函数级验证
  2. 集成测试 → 模块间验证
  3. E2E测试  → 用户场景验证

  没有测试怎么办？
  → 先补测试 再重构
  
  补测试策略（遗留代码）：
  1. 特征测试（Characterization Test）
     不管对不对 先记录当前行为
  func TestLegacyProcess(t *testing.T) {
      result := LegacyProcess(input)
      // 不确定对不对 先记录当前结果
      assert.Equal(t, currentOutput, result)
  }
  
  2. 金丝雀测试
     新旧实现并行运行 对比结果
  func TestCompare(t *testing.T) {
      oldResult := oldImpl(input)
      newResult := newImpl(input)
      assert.Equal(t, oldResult, newResult)
  }

  重构检查清单：
  ✓ 所有测试通过
  ✓ 每次改动小且可理解
  ✓ 频繁提交（每个小步骤一次）
  ✓ 不同时改结构和功能
  ✓ Code Review确认
  ✓ 灰度发布验证
```

## 10. 重构面试速答？

**回答：**

```
Q: 什么是重构？
A: 不改变外部行为 改善内部结构
   前提是有测试保护

Q: 什么时候重构？
A: 加功能前/修Bug时/Code Review时
   三次法则：第三次重复时

Q: 最常用的重构手法？
A: 提取函数/方法
   看到注释分隔代码块 → 提取

Q: 怎么消除重复？
A: 相同代码→提取函数
   相似代码→参数化/泛型
   只消除真正重复 不合并偶然重复

Q: 条件表达式怎么简化？
A: 卫语句替代嵌套
   提取条件为方法
   Map替代纯数据switch

Q: 大类怎么拆？
A: 按职责拆分 一个类一个职责
   God Object → 多个专职Service

Q: 大规模重构怎么做？
A: 绞杀者模式 渐进式替换
   接口隔离+Feature Flag+灰度

Q: 重构的安全网？
A: 测试(单元+集成+E2E)
   没测试先补特征测试
   小步前进 频繁提交
```
