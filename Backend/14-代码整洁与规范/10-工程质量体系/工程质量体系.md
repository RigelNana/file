# 工程质量体系

---

## 1. 工程质量全景？

**回答：**

```
  质量保障体系：
  ┌──────────────────────────────────────┐
  │           编码阶段                    │
  │  IDE检查 + pre-commit + 格式化       │
  ├──────────────────────────────────────┤
  │           提交阶段                    │
  │  Code Review + CI自动检查            │
  ├──────────────────────────────────────┤
  │           测试阶段                    │
  │  单元测试 + 集成测试 + E2E测试       │
  ├──────────────────────────────────────┤
  │           发布阶段                    │
  │  灰度发布 + 监控告警 + 回滚机制      │
  ├──────────────────────────────────────┤
  │           线上阶段                    │
  │  SLO监控 + 故障复盘 + 持续改进       │
  └──────────────────────────────────────┘

  质量不是测试出来的 是设计和构建出来的

  质量度量指标：
  ┌──────────────────────┬──────────────────┐
  │ 指标                  │ 目标             │
  ├──────────────────────┼──────────────────┤
  │ 测试覆盖率            │ 核心>80%         │
  │ 圈复杂度              │ <15              │
  │ 代码重复率            │ <5%              │
  │ PR合并时间            │ <1个工作日       │
  │ Bug逃逸率             │ <5%              │
  │ 线上故障频率          │ 月均<1次         │
  └──────────────────────┴──────────────────┘
```

## 2. 测试金字塔？

**回答：**

```
  测试金字塔：
          /\
         /  \         E2E测试 10%
        /    \        验证用户场景
       /──────\
      /        \      集成测试 20%
     /          \     验证模块协作
    /────────────\
   /              \   单元测试 70%
  /________________\  验证函数逻辑

  各层对比：
  ┌──────────────┬──────────┬──────────┬──────────┐
  │ 维度          │ 单元测试  │ 集成测试  │ E2E测试  │
  ├──────────────┼──────────┼──────────┼──────────┤
  │ 速度          │ 毫秒     │ 秒       │ 分钟     │
  │ 稳定性        │ 高       │ 中       │ 低       │
  │ 维护成本      │ 低       │ 中       │ 高       │
  │ 反馈速度      │ 快       │ 中       │ 慢       │
  │ 覆盖范围      │ 细粒度   │ 模块间   │ 全流程   │
  └──────────────┴──────────┴──────────┴──────────┘

  Go测试策略：
  单元测试：go test ./internal/...
  集成测试：testcontainers + -tags=integration
  E2E测试：httptest.Server + 完整请求流程

  反模式 — 冰淇淋锥（倒金字塔）：
  大量E2E → 少量单元测试
  结果：慢、不稳定、维护成本高
```

## 3. 持续集成最佳实践？

**回答：**

```
  CI原则：
  1. 每次提交都触发CI
  2. 构建应该快（<10分钟）
  3. 失败立即修复（不过夜）
  4. 主分支永远是绿色的

  CI Pipeline设计：
  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐
  │ Format  │→│  Lint   │→│  Test   │→│  Build  │
  └─────────┘  └─────────┘  └─────────┘  └─────────┘
      gofmt     golangci    go test      go build
                  -lint      -race

  CI加速策略：
  1. 缓存Go模块
     uses: actions/cache@v4
     with:
       path: ~/go/pkg/mod
       key: go-mod-${{ hashFiles('go.sum') }}
  
  2. 并行运行
     lint和test并行
     不同Go版本矩阵并行
  
  3. 增量测试
     只测试变更的包
     go test $(go list ./... | grep changed)
  
  4. 分层测试
     PR阶段：单元测试（快）
     合并后：集成测试（慢）
     定时：E2E测试
```

## 4. 代码质量度量？

**回答：**

```
  度量工具：
  ┌──────────────────────┬──────────────────────┐
  │ 指标                  │ 工具                 │
  ├──────────────────────┼──────────────────────┤
  │ 覆盖率                │ go test -cover       │
  │ 圈复杂度              │ gocyclo              │
  │ 代码重复              │ dupl / jscpd         │
  │ 依赖分析              │ godepgraph           │
  │ 综合评分              │ SonarQube            │
  │ 安全漏洞              │ gosec / govulncheck  │
  └──────────────────────┴──────────────────────┘

  SonarQube集成：
  # sonar-project.properties
  sonar.projectKey=myapp
  sonar.sources=.
  sonar.exclusions=**/*_test.go,**/vendor/**
  sonar.go.coverage.reportPaths=coverage.out
  sonar.go.tests.reportPaths=test-report.json

  关键度量看板：
  1. 代码健康度
     覆盖率趋势（不能持续下降）
     圈复杂度分布
     技术债务评分
  
  2. 交付效率
     PR合并时间
     CI构建时间
     部署频率
  
  3. 质量结果
     线上Bug数
     Bug逃逸率
     MTTR（平均修复时间）

  度量不是目的 改进才是：
  覆盖率从60%→80%有价值
  从95%→100%可能浪费时间
```

## 5. 故障复盘与改进？

**回答：**

```
  故障复盘（Postmortem）流程：

  1. 时间线
     何时发现 → 何时响应 → 何时恢复
  
  2. 影响范围
     影响了多少用户/多少请求
     持续了多长时间
  
  3. 根因分析（5 Whys）
     为什么服务挂了？→ OOM
     为什么OOM？→ 内存泄漏
     为什么泄漏？→ goroutine没释放
     为什么没释放？→ 缺少context取消
     为什么没context？→ 没有相关规范
  
  4. 改进措施
     短期：修复Bug + 回归测试
     中期：添加goroutine数监控告警
     长期：制定context使用规范 + 培训

  复盘模板：
  ## 故障标题
  ## 影响：P0/P1/P2 影响X用户 持续Y分钟
  ## 时间线
  ## 根因分析
  ## 修复措施
  ## 改进项（附负责人和截止日期）

  复盘原则：
  ✓ Blameless（不追责 追因）
  ✓ 改进项必须落实跟踪
  ✓ 分享给全团队学习
  ✗ 不是走形式
```

## 6. 技术债看板管理？

**回答：**

```
  技术债分类：
  ┌──────────────┬──────────────┬──────────────────┐
  │ 类别          │ 优先级        │ 示例             │
  ├──────────────┼──────────────┼──────────────────┤
  │ 架构债        │ 高           │ 单体拆微服务     │
  │ 代码债        │ 中           │ 重复代码/God类   │
  │ 测试债        │ 中           │ 缺少单元测试     │
  │ 文档债        │ 低           │ API文档过时       │
  │ 依赖债        │ 高           │ 依赖有CVE漏洞    │
  │ 基础设施债    │ 中           │ CI/CD不完善      │
  └──────────────┴──────────────┴──────────────────┘

  管理方法：
  1. 识别
     每个Sprint Review时收集
     Code Review时标记
     工具自动检测
  
  2. 评估
     影响范围（几个模块受影响）
     修复成本（人天）
     不修复的风险
  
  3. 排期
     每个Sprint分配20%时间
     高优先级单独排Sprint
     与产品需求平衡
  
  4. 跟踪
     Issue/Jira看板
     定期Review进度
     度量技术债趋势

  技术债不会自己消失
  越早还 利息越少
```

## 7. 编码规范落地？

**回答：**

```
  规范落地四步法：

  1. 制定规范
     基于官方(Effective Go) + 团队特有
     不要太多 聚焦高频问题
     Example > 规则描述
  
  2. 工具强制
     能用工具检查的不靠人
     golangci-lint配置规则
     CI门禁不通过不合并
  
  3. Code Review补充
     工具检查不了的靠Review
     设计合理性/业务正确性
     统一Review标准
  
  4. 培训传播
     新人培训包含编码规范
     定期分享好/坏代码案例
     结对编程传递经验

  规范文档维护：
  放在代码仓库中（和代码一起Review）
  定期更新（每季度Review一次）
  接受团队讨论和修改

  避免的反模式：
  ✗ 规范写了没人看
  ✗ 规范和工具不一致
  ✗ 只管新代码不管旧代码
  ✗ 规范太多记不住

  好的规范 = 少而精 + 工具强制 + 持续迭代
```

## 8. 可观测性与质量？

**回答：**

```
  可观测性三支柱保障质量：

  1. Metrics（指标）
     RED指标：Rate/Error/Duration
     用于监控和告警
     Prometheus + Grafana
  
  2. Logging（日志）
     结构化日志（JSON）
     TraceID贯穿
     错误现场记录
  
  3. Tracing（链路追踪）
     请求全链路可见
     性能瓶颈定位
     OpenTelemetry + Jaeger

  质量相关的关键指标：
  ┌──────────────────────┬──────────────────────┐
  │ 指标                  │ 阈值                 │
  ├──────────────────────┼──────────────────────┤
  │ 错误率                │ <0.1%                │
  │ P99延迟               │ <500ms               │
  │ 可用性                │ >99.9%               │
  │ panic次数             │ 0                    │
  │ goroutine泄漏         │ 趋势不增长           │
  │ 内存使用              │ 无持续增长           │
  └──────────────────────┴──────────────────────┘

  SLO（服务级别目标）：
  可用性SLO: 99.95%（月停机<22分钟）
  延迟SLO: P99 < 200ms
  错误SLO: 错误率 < 0.1%

  Error Budget（错误预算）：
  本月允许的不可用时间 = 30天 × 0.05% ≈ 22分钟
  预算用完 → 停止新功能 专注稳定性
```

## 9. 团队协作规范？

**回答：**

```
  Git工作流：
  ┌──────────────────────────────────────┐
  │ main ──────────────────────────────  │
  │    \               /    \       /    │
  │     develop ──────/──────\─────/     │
  │       \    /                         │
  │       feature                        │
  └──────────────────────────────────────┘

  分支命名：
  feature/add-user-auth
  fix/order-timeout-issue
  refactor/auth-module
  hotfix/critical-bug

  Commit规范：
  <type>(<scope>): <subject>
  
  type: feat/fix/docs/refactor/test/chore
  scope: user/order/auth（模块）
  subject: 简洁描述

  PR流程：
  1. 从develop创建feature分支
  2. 开发 + 本地测试
  3. push → 创建PR
  4. CI自动检查
  5. Code Review（至少1人）
  6. 合并到develop
  7. 定期从develop合并到main发布

  沟通规范：
  技术方案 → 文档评审后再开发
  接口变更 → 通知所有消费者
  依赖升级 → CHANGELOG说明
  故障 → 群通报 + 复盘
```

## 10. 工程质量面试速答？

**回答：**

```
Q: 质量保障体系包含什么？
A: 编码(lint)+提交(Review+CI)
   测试(单元+集成)+发布(灰度)+监控

Q: 测试金字塔？
A: 单元70%+集成20%+E2E 10%
   反模式：冰淇淋锥(E2E太多)

Q: CI怎么做？
A: 每次提交触发 <10分钟完成
   lint+test+build 失败不过夜

Q: 怎么度量代码质量？
A: 覆盖率+圈复杂度+重复率
   Bug逃逸率+MTTR

Q: 故障复盘怎么做？
A: 时间线+影响+5Whys根因
   Blameless 改进项必须落实

Q: 技术债怎么管理？
A: 识别→评估→排期→跟踪
   每Sprint 20%时间偿还

Q: 编码规范怎么落地？
A: 工具强制(golangci-lint+CI门禁)
   Review补充 新人培训

Q: SLO和Error Budget？
A: SLO=可用性/延迟/错误率目标
   Error Budget用完→停新功能保稳定
```
