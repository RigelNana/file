# 单元测试

---

## 1. 单元测试基础？

**回答：**

```
  单元测试 = 验证最小可测试单元的正确性

  特征（FIRST）：
  ┌──────────────┬──────────────────────────────┐
  │ 原则          │ 说明                         │
  ├──────────────┼──────────────────────────────┤
  │ Fast          │ 毫秒级完成                   │
  │ Independent   │ 测试间不依赖                 │
  │ Repeatable    │ 任何环境结果一致             │
  │ Self-Validating │ 自动判断通过/失败          │
  │ Timely        │ 及时编写 最好TDD             │
  └──────────────┴──────────────────────────────┘

  Go测试基本结构：
  // user_test.go
  func TestCreateUser(t *testing.T) {
      // Arrange（准备）
      svc := NewUserService(mockRepo)
      req := &CreateUserReq{Name: "test", Email: "t@t.com"}
      
      // Act（执行）
      user, err := svc.Create(context.Background(), req)
      
      // Assert（断言）
      if err != nil {
          t.Fatalf("expected no error, got %v", err)
      }
      if user.Name != "test" {
          t.Errorf("expected name 'test', got %q", user.Name)
      }
  }

  运行测试：
  go test ./...                  # 所有测试
  go test -v ./pkg/user/...     # 指定包 详细输出
  go test -run TestCreateUser   # 指定测试
  go test -count=1              # 禁用缓存
  go test -race                 # 竞态检测
  go test -cover                # 覆盖率
```

## 2. 表驱动测试（Table-Driven Tests）？

**回答：**

```
  Go最推荐的测试模式 减少重复

  func TestCalculateDiscount(t *testing.T) {
      tests := []struct {
          name     string
          amount   float64
          vipLevel int
          want     float64
      }{
          {"普通用户无折扣", 100, 0, 100},
          {"VIP1 九五折", 100, 1, 95},
          {"VIP2 九折", 100, 2, 90},
          {"VIP3 八五折", 100, 3, 85},
          {"零金额", 0, 3, 0},
          {"大额订单", 10000, 2, 9000},
      }
      
      for _, tt := range tests {
          t.Run(tt.name, func(t *testing.T) {
              got := CalculateDiscount(tt.amount, tt.vipLevel)
              if got != tt.want {
                  t.Errorf("CalculateDiscount(%v, %v) = %v, want %v",
                      tt.amount, tt.vipLevel, got, tt.want)
              }
          })
      }
  }

  优势：
  1. 用例一目了然
  2. 加新用例只加一行
  3. 每个用例有名字 失败定位清晰
  4. t.Run生成子测试 可单独运行

  边界值测试用例：
  空值/nil/零值
  最大值/最小值/边界-1/边界+1
  特殊字符/超长字符串
  并发场景
```

## 3. Mock与依赖注入？

**回答：**

```
  核心：通过接口注入依赖 测试时替换为Mock

  1. 定义接口
  type UserRepository interface {
      FindByID(ctx context.Context, id string) (*User, error)
      Save(ctx context.Context, user *User) error
  }

  2. Service依赖接口
  type UserService struct {
      repo UserRepository
  }
  
  func NewUserService(repo UserRepository) *UserService {
      return &UserService{repo: repo}
  }

  3. 手写Mock
  type MockUserRepo struct {
      FindByIDFunc func(ctx context.Context, id string) (*User, error)
      SaveFunc     func(ctx context.Context, user *User) error
  }
  
  func (m *MockUserRepo) FindByID(ctx context.Context, id string) (*User, error) {
      return m.FindByIDFunc(ctx, id)
  }
  
  func (m *MockUserRepo) Save(ctx context.Context, user *User) error {
      return m.SaveFunc(ctx, user)
  }

  4. 测试中使用
  func TestGetUser(t *testing.T) {
      mock := &MockUserRepo{
          FindByIDFunc: func(ctx context.Context, id string) (*User, error) {
              if id == "123" {
                  return &User{ID: "123", Name: "test"}, nil
              }
              return nil, ErrNotFound
          },
      }
      
      svc := NewUserService(mock)
      user, err := svc.GetUser(context.Background(), "123")
      if err != nil {
          t.Fatal(err)
      }
      if user.Name != "test" {
          t.Errorf("got %q, want 'test'", user.Name)
      }
  }

  mockgen自动生成：
  //go:generate mockgen -source=repository.go -destination=mock_repository.go
```

## 4. 测试覆盖率？

**回答：**

```
  覆盖率类型：
  行覆盖率：代码行执行百分比
  分支覆盖率：if/switch分支覆盖
  函数覆盖率：函数被调用百分比

  Go覆盖率工具：
  # 查看覆盖率
  go test -cover ./...
  
  # 生成覆盖率报告
  go test -coverprofile=coverage.out ./...
  go tool cover -html=coverage.out -o coverage.html
  
  # 按函数查看
  go tool cover -func=coverage.out

  覆盖率目标（建议 非强制）：
  ┌──────────────────┬──────────────┐
  │ 模块              │ 目标         │
  ├──────────────────┼──────────────┤
  │ 核心业务逻辑      │ >80%         │
  │ 工具/算法库       │ >90%         │
  │ 数据访问层        │ >60%（集成测试）│
  │ Handler层         │ >60%         │
  │ 总体              │ >60%         │
  └──────────────────┴──────────────┘

  覆盖率的陷阱：
  1. 高覆盖率 ≠ 高质量
     100%覆盖率但没有断言 → 无意义
  
  2. 不要为了覆盖率写无意义测试
     // 差：只测了能运行 没测正确性
     func TestGetUser(t *testing.T) {
         GetUser("123")  // 没有断言
     }
  
  3. 关注有意义的测试 不追求100%
     简单的getter/setter不需要测
     集中测试复杂逻辑和边界条件
```

## 5. 测试替身（Test Doubles）？

**回答：**

```
  五种测试替身：
  ┌──────────────┬──────────────────────────────┐
  │ 类型          │ 说明                         │
  ├──────────────┼──────────────────────────────┤
  │ Dummy         │ 占位 不使用                  │
  │ Stub          │ 返回固定值                   │
  │ Spy           │ 记录调用 可验证              │
  │ Mock          │ 预设期望 自动验证            │
  │ Fake          │ 简化实现（内存DB等）         │
  └──────────────┴──────────────────────────────┘

  Stub示例：
  type StubUserRepo struct{}
  func (s *StubUserRepo) FindByID(ctx context.Context, id string) (*User, error) {
      return &User{ID: id, Name: "stub"}, nil  // 固定返回
  }

  Spy示例：
  type SpyEmailSender struct {
      SentEmails []string
  }
  func (s *SpyEmailSender) Send(to, body string) error {
      s.SentEmails = append(s.SentEmails, to)
      return nil
  }
  // 测试中验证
  if len(spy.SentEmails) != 1 {
      t.Error("expected 1 email sent")
  }

  Fake示例（内存Repository）：
  type FakeUserRepo struct {
      users map[string]*User
  }
  func (f *FakeUserRepo) FindByID(ctx context.Context, id string) (*User, error) {
      u, ok := f.users[id]
      if !ok { return nil, ErrNotFound }
      return u, nil
  }
  func (f *FakeUserRepo) Save(ctx context.Context, user *User) error {
      f.users[user.ID] = user
      return nil
  }

  选择依据：
  简单场景 → Stub
  需要验证调用 → Spy/Mock
  需要状态 → Fake
```

## 6. 集成测试与testcontainers？

**回答：**

```
  集成测试 = 验证多个模块协作
  通常测试：Service + 真实DB / 真实Redis

  testcontainers-go（推荐）：
  import "github.com/testcontainers/testcontainers-go"

  func TestUserRepo_Integration(t *testing.T) {
      if testing.Short() {
          t.Skip("skipping integration test")
      }
      
      // 启动PostgreSQL容器
      ctx := context.Background()
      pg, err := testcontainers.GenericContainer(ctx, 
          testcontainers.GenericContainerRequest{
              ContainerRequest: testcontainers.ContainerRequest{
                  Image:        "postgres:15",
                  ExposedPorts: []string{"5432/tcp"},
                  Env: map[string]string{
                      "POSTGRES_PASSWORD": "test",
                      "POSTGRES_DB":       "testdb",
                  },
                  WaitingFor: wait.ForListeningPort("5432/tcp"),
              },
              Started: true,
          })
      defer pg.Terminate(ctx)
      
      // 获取连接信息
      host, _ := pg.Host(ctx)
      port, _ := pg.MappedPort(ctx, "5432")
      dsn := fmt.Sprintf("postgres://postgres:test@%s:%s/testdb", host, port.Port())
      
      // 执行测试
      repo := NewUserRepo(dsn)
      user := &User{Name: "test"}
      err = repo.Save(ctx, user)
      assert.NoError(t, err)
      
      found, err := repo.FindByID(ctx, user.ID)
      assert.NoError(t, err)
      assert.Equal(t, "test", found.Name)
  }

  运行：
  go test ./...               # 跳过集成测试
  go test -short=false ./...  # 包含集成测试
```

## 7. Benchmark测试？

**回答：**

```
  Go内置性能基准测试：

  func BenchmarkMarshalJSON(b *testing.B) {
      user := &User{ID: "123", Name: "test", Email: "t@t.com"}
      
      b.ResetTimer()
      for i := 0; i < b.N; i++ {
          json.Marshal(user)
      }
  }
  
  // 对比不同实现
  func BenchmarkMarshalJSON_StdLib(b *testing.B) {
      for i := 0; i < b.N; i++ {
          json.Marshal(data)
      }
  }
  
  func BenchmarkMarshalJSON_Sonic(b *testing.B) {
      for i := 0; i < b.N; i++ {
          sonic.Marshal(data)
      }
  }

  运行：
  go test -bench=. -benchmem ./...
  
  输出：
  BenchmarkMarshalJSON_StdLib-8   500000   3200 ns/op   1024 B/op   8 allocs/op
  BenchmarkMarshalJSON_Sonic-8   2000000    800 ns/op    512 B/op   2 allocs/op

  关注指标：
  ns/op     → 每次操作耗时
  B/op      → 每次操作内存分配
  allocs/op → 每次操作分配次数

  避免编译器优化干扰：
  var result []byte
  func BenchmarkMarshal(b *testing.B) {
      var r []byte
      for i := 0; i < b.N; i++ {
          r, _ = json.Marshal(data)
      }
      result = r  // 防止编译器消除
  }
```

## 8. 模糊测试（Fuzz Testing）？

**回答：**

```
  Go 1.18+ 内置模糊测试
  自动生成随机输入发现意外行为

  func FuzzParseJSON(f *testing.F) {
      // 种子语料
      f.Add([]byte(`{"name":"test"}`))
      f.Add([]byte(`{}`))
      f.Add([]byte(`null`))
      
      f.Fuzz(func(t *testing.T, data []byte) {
          var result map[string]interface{}
          err := json.Unmarshal(data, &result)
          if err != nil {
              return  // 解析失败是预期的
          }
          
          // 验证：能解析的一定能序列化回去
          encoded, err := json.Marshal(result)
          if err != nil {
              t.Fatalf("marshal after unmarshal failed: %v", err)
          }
          
          // 验证：再次解析结果一致
          var result2 map[string]interface{}
          if err := json.Unmarshal(encoded, &result2); err != nil {
              t.Fatalf("second unmarshal failed: %v", err)
          }
      })
  }

  运行：
  go test -fuzz=FuzzParseJSON -fuzztime=30s
  
  发现crash时自动保存到 testdata/fuzz/ 
  后续 go test 会自动回归

  适合模糊测试的场景：
  解析器（JSON/XML/Protocol）
  编解码（Base64/加密）
  验证函数（邮箱/URL格式）
  任何处理外部输入的函数
```

## 9. 测试最佳实践？

**回答：**

```
  1. 测试命名清晰
  // 好：描述输入和预期结果
  func TestCreateUser_EmptyName_ReturnsError(t *testing.T)
  func TestCreateUser_DuplicateEmail_ReturnsConflict(t *testing.T)
  func TestCalculateTotal_EmptyCart_ReturnsZero(t *testing.T)

  2. 一个测试验证一个行为
  // 差：一个测试验证太多
  func TestUser(t *testing.T) {
      // 测试创建、查询、更新、删除...太多了
  }

  3. 测试独立不共享状态
  // 用t.Cleanup清理
  func TestWithDB(t *testing.T) {
      db := setupTestDB(t)
      t.Cleanup(func() { db.Close() })
  }

  4. TestMain统一Setup/Teardown
  func TestMain(m *testing.M) {
      setup()
      code := m.Run()
      teardown()
      os.Exit(code)
  }

  5. 使用testify简化断言（可选）
  import "github.com/stretchr/testify/assert"
  assert.Equal(t, expected, actual)
  assert.NoError(t, err)
  assert.Contains(t, str, "substring")

  6. 测试辅助函数
  func newTestUser(t *testing.T) *User {
      t.Helper()  // 报错时显示调用者行号
      return &User{
          ID:   uuid.New().String(),
          Name: "test-" + t.Name(),
      }
  }
```

## 10. 单元测试面试速答？

**回答：**

```
Q: 单元测试FIRST原则？
A: Fast快速 Independent独立 Repeatable可重复
   Self-Validating自验证 Timely及时

Q: Go推荐什么测试模式？
A: 表驱动测试(Table-Driven Tests)
   减少重复 用例清晰 t.Run子测试

Q: 怎么Mock依赖？
A: 依赖接口注入 测试时替换Mock
   手写Mock或mockgen生成

Q: 覆盖率多少合适？
A: 核心业务>80% 总体>60%
   不追求100% 关注有意义的测试

Q: 五种测试替身？
A: Dummy(占位) Stub(固定值) Spy(记录调用)
   Mock(预设期望) Fake(简化实现)

Q: 怎么做集成测试？
A: testcontainers启动真实DB容器
   -short标志区分单元/集成测试

Q: Benchmark怎么用？
A: func BenchmarkXxx(b *testing.B)
   关注ns/op B/op allocs/op

Q: 模糊测试是什么？
A: Go 1.18+内置 自动随机输入
   发现解析器/编解码等的边界Bug
```
