# 错误处理规范

---

## 1. Go错误处理哲学？

**回答：**

```
  Go错误处理核心：errors are values（错误是值）

  vs 异常机制：
  ┌──────────────┬──────────────────┬──────────────────┐
  │ 维度          │ Go (error)       │ Java (Exception) │
  ├──────────────┼──────────────────┼──────────────────┤
  │ 机制          │ 返回值           │ 抛出/捕获        │
  │ 流程控制      │ 显式if检查       │ try/catch        │
  │ 性能          │ 无额外开销       │ 栈展开开销       │
  │ 可见性        │ 签名中可见       │ 可能被忽略       │
  │ 强制处理      │ 可忽略(但不应该) │ Checked强制      │
  └──────────────┴──────────────────┴──────────────────┘

  Go的选择理由：
  1. 显式优于隐式 → 错误处理在代码中可见
  2. 简单 → 不需要try/catch/finally
  3. 函数签名即契约 → 看签名就知道会不会出错

  基本模式：
  result, err := DoSomething()
  if err != nil {
      return fmt.Errorf("do something: %w", err)
  }
  // 使用result

  不要忽略错误：
  // 差
  result, _ := DoSomething()
  json.Unmarshal(data, &v)  // 忽略了错误
  
  // 如果确实不需要 显式说明
  _ = conn.Close() // 连接关闭错误不影响业务
```

## 2. 错误包装与链？

**回答：**

```
  Go 1.13+ 用 %w 包装错误：
  
  func GetUserOrders(ctx context.Context, userID string) ([]*Order, error) {
      user, err := userRepo.FindByID(ctx, userID)
      if err != nil {
          return nil, fmt.Errorf("find user %s: %w", userID, err)
      }
      
      orders, err := orderRepo.ListByUser(ctx, userID)
      if err != nil {
          return nil, fmt.Errorf("list orders for user %s: %w", userID, err)
      }
      
      return orders, nil
  }
  
  // 错误链：
  // list orders for user 123: query database: connection refused

  %w vs %v：
  %w → 保留原始错误 可以用Is/As解包
  %v → 仅字符串拼接 丢失原始错误类型

  errors.Is 和 errors.As：
  // 判断错误链中是否包含特定错误
  if errors.Is(err, sql.ErrNoRows) {
      return nil, ErrNotFound
  }
  
  // 提取错误链中的特定类型
  var pgErr *pgconn.PgError
  if errors.As(err, &pgErr) {
      if pgErr.Code == "23505" {
          return ErrDuplicate
      }
  }

  包装层次不要太深：
  每层加一句上下文 不要重复信息
  "handler: service: repo: query: connection refused"
  → 每层只加自己的上下文
```

## 3. 自定义错误类型？

**回答：**

```
  三种自定义错误方式：

  1. 哨兵错误（Sentinel Error）
  var (
      ErrNotFound     = errors.New("not found")
      ErrUnauthorized = errors.New("unauthorized")
      ErrConflict     = errors.New("conflict")
  )
  
  // 使用
  if errors.Is(err, ErrNotFound) { ... }

  2. 自定义错误类型（携带额外信息）
  type AppError struct {
      Code    int    // 业务错误码
      Message string // 用户可见消息
      Err     error  // 原始错误
  }
  
  func (e *AppError) Error() string {
      return fmt.Sprintf("[%d] %s: %v", e.Code, e.Message, e.Err)
  }
  
  func (e *AppError) Unwrap() error { return e.Err }
  
  // 使用
  var appErr *AppError
  if errors.As(err, &appErr) {
      writeJSON(w, appErr.Code, appErr.Message)
  }

  3. 错误行为接口
  type TemporaryError interface {
      Temporary() bool
  }
  
  type NetworkError struct {
      Err error
  }
  func (e *NetworkError) Temporary() bool { return true }
  
  // 根据行为决定是否重试
  var tempErr TemporaryError
  if errors.As(err, &tempErr) && tempErr.Temporary() {
      // 重试
  }

  选择依据：
  简单错误 → 哨兵错误
  需要额外信息 → 自定义类型
  需要多态行为 → 错误接口
```

## 4. panic和recover？

**回答：**

```
  panic使用场景（极少）：
  程序不可恢复的错误 继续运行没意义
  
  适合panic：
  1. 程序初始化失败（无法启动）
  2. 不可能到达的代码路径
  3. 明确的编程错误（nil解引用等）

  func MustParseURL(raw string) *url.URL {
      u, err := url.Parse(raw)
      if err != nil {
          panic(fmt.Sprintf("invalid URL %q: %v", raw, err))
      }
      return u
  }
  // Must前缀表示会panic

  不适合panic：
  ✗ 业务逻辑错误
  ✗ 用户输入验证
  ✗ 外部服务调用失败
  → 这些都用error返回

  recover使用场景：
  1. HTTP中间件防止单个请求crash整个服务
  func RecoveryMiddleware(next http.Handler) http.Handler {
      return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
          defer func() {
              if rec := recover(); rec != nil {
                  log.Printf("panic: %v\n%s", rec, debug.Stack())
                  http.Error(w, "Internal Server Error", 500)
              }
          }()
          next.ServeHTTP(w, r)
      })
  }

  2. goroutine保护
  go func() {
      defer func() {
          if r := recover(); r != nil {
              log.Printf("goroutine panic: %v", r)
          }
      }()
      doWork()
  }()

  原则：正常业务永远用error 不用panic
```

## 5. 错误处理在分层架构中？

**回答：**

```
  每层的错误处理职责：
  ┌──────────────────────────────────┐
  │ Handler层                        │
  │ 将错误转为HTTP响应               │
  │ 记录日志（日志只记一次）         │
  ├──────────────────────────────────┤
  │ Service层                        │
  │ 业务错误判断和包装               │
  │ 跨服务错误转换                   │
  ├──────────────────────────────────┤
  │ Repository层                     │
  │ 数据层错误转换为业务错误         │
  │ sql.ErrNoRows → ErrNotFound      │
  └──────────────────────────────────┘

  Repository层：
  func (r *UserRepo) FindByID(ctx context.Context, id string) (*User, error) {
      var user User
      err := r.db.WithContext(ctx).First(&user, "id = ?", id).Error
      if errors.Is(err, gorm.ErrRecordNotFound) {
          return nil, ErrNotFound  // 转换为业务错误
      }
      if err != nil {
          return nil, fmt.Errorf("find user %s: %w", id, err)
      }
      return &user, nil
  }

  Service层：
  func (s *OrderService) Create(ctx context.Context, req CreateReq) error {
      user, err := s.userRepo.FindByID(ctx, req.UserID)
      if errors.Is(err, ErrNotFound) {
          return fmt.Errorf("user not found: %w", ErrInvalidOrder)
      }
      // ...
  }

  Handler层：
  func (h *Handler) CreateOrder(w http.ResponseWriter, r *http.Request) {
      err := h.orderSvc.Create(r.Context(), req)
      if err != nil {
          handleError(w, r, err)  // 统一错误处理
          return
      }
  }
  
  func handleError(w http.ResponseWriter, r *http.Request, err error) {
      var appErr *AppError
      if errors.As(err, &appErr) {
          writeJSON(w, appErr.Code, appErr.Message)
          return
      }
      log.Printf("unhandled error: %v", err)
      writeJSON(w, 500, "internal error")
  }
```

## 6. 错误日志最佳实践？

**回答：**

```
  核心原则：错误只记录一次

  // 差：每层都记录 日志重复
  // Repo层
  func (r *Repo) Find(id string) error {
      err := db.Query(...)
      log.Printf("query error: %v", err)  // 记录1
      return err
  }
  // Service层
  func (s *Svc) Get(id string) error {
      err := s.repo.Find(id)
      log.Printf("find error: %v", err)   // 记录2 重复
      return err
  }
  // Handler层
  func (h *H) Handle(w http.ResponseWriter, r *http.Request) {
      err := h.svc.Get(id)
      log.Printf("handle error: %v", err) // 记录3 重复
  }

  // 好：底层只包装上传 顶层统一记录
  // Repo层
  func (r *Repo) Find(id string) error {
      return fmt.Errorf("query user %s: %w", id, err)
  }
  // Service层
  func (s *Svc) Get(id string) error {
      return fmt.Errorf("get user: %w", err)
  }
  // Handler层（唯一记录日志的地方）
  func (h *H) Handle(w http.ResponseWriter, r *http.Request) {
      if err != nil {
          log.Printf("error: %v", err)  // 只记一次
          // "error: get user: query user 123: connection refused"
      }
  }

  错误日志要包含：
  ✓ 错误链完整信息
  ✓ 请求上下文（traceID/userID）
  ✓ 关键参数值
  ✗ 不要记录密码/Token
```

## 7. 超时与取消处理？

**回答：**

```
  Go用context传递超时和取消信号：

  设置超时：
  func (h *Handler) GetUser(w http.ResponseWriter, r *http.Request) {
      ctx, cancel := context.WithTimeout(r.Context(), 5*time.Second)
      defer cancel()
      
      user, err := h.svc.GetUser(ctx, userID)
      if errors.Is(err, context.DeadlineExceeded) {
          http.Error(w, "request timeout", 504)
          return
      }
  }

  传递context：
  func (s *Service) GetUser(ctx context.Context, id string) (*User, error) {
      // context往下传 每层都检查
      user, err := s.repo.FindByID(ctx, id)
      if err != nil {
          return nil, err
      }
      
      // 长操作前检查是否已取消
      select {
      case <-ctx.Done():
          return nil, ctx.Err()
      default:
      }
      
      profile, err := s.profileSvc.Get(ctx, id)
      // ...
  }

  超时错误处理策略：
  ┌──────────────────────┬──────────────────────┐
  │ 场景                  │ 处理                 │
  ├──────────────────────┼──────────────────────┤
  │ DeadlineExceeded     │ 上游超时 返回504     │
  │ Canceled             │ 客户端取消 不记告警   │
  │ 下游超时             │ 重试/降级/熔断       │
  └──────────────────────┴──────────────────────┘

  注意：
  context取消后不要继续写数据库
  用独立context做清理操作
  cleanupCtx := context.WithoutCancel(ctx) // Go 1.21+
```

## 8. 错误码设计？

**回答：**

```
  API错误响应设计：
  {
      "code": 10001,
      "message": "用户不存在",
      "detail": "user_id: 12345"
  }

  错误码规范：
  ┌──────────────┬──────────────────────────────┐
  │ 范围          │ 说明                         │
  ├──────────────┼──────────────────────────────┤
  │ 10000-19999  │ 用户模块                     │
  │ 20000-29999  │ 订单模块                     │
  │ 30000-39999  │ 支付模块                     │
  │ 40000-49999  │ 商品模块                     │
  │ 90000-99999  │ 系统通用                     │
  └──────────────┴──────────────────────────────┘

  Go错误码实现：
  type ErrCode struct {
      Code       int    `json:"code"`
      HTTPStatus int    `json:"-"`
      Message    string `json:"message"`
  }
  
  func (e *ErrCode) Error() string { return e.Message }
  
  var (
      ErrUserNotFound   = &ErrCode{10001, 404, "用户不存在"}
      ErrUserDisabled   = &ErrCode{10002, 403, "用户已禁用"}
      ErrOrderNotFound  = &ErrCode{20001, 404, "订单不存在"}
      ErrOrderCancelled = &ErrCode{20002, 400, "订单已取消"}
      ErrInternal       = &ErrCode{90001, 500, "服务器内部错误"}
  )
  
  // 统一响应
  func handleError(w http.ResponseWriter, err error) {
      var ec *ErrCode
      if errors.As(err, &ec) {
          writeJSON(w, ec.HTTPStatus, map[string]interface{}{
              "code":    ec.Code,
              "message": ec.Message,
          })
          return
      }
      // 未知错误不暴露细节
      writeJSON(w, 500, map[string]interface{}{
          "code":    90001,
          "message": "服务器内部错误",
      })
  }

  原则：
  对外 → 固定错误码+友好消息
  对内 → 完整错误链+堆栈+参数
```

## 9. 重试与容错？

**回答：**

```
  可重试的错误：
  网络超时 / 连接拒绝 / 503 / 429
  不可重试：400/401/403/404/业务逻辑错误

  指数退避+抖动：
  func RetryWithBackoff(ctx context.Context, maxRetries int, 
      fn func() error) error {
      
      var lastErr error
      for i := 0; i < maxRetries; i++ {
          lastErr = fn()
          if lastErr == nil {
              return nil
          }
          
          // 不可重试的错误直接返回
          if !IsRetryable(lastErr) {
              return lastErr
          }
          
          // 指数退避 + 随机抖动
          backoff := time.Duration(1<<uint(i)) * 100 * time.Millisecond
          jitter := time.Duration(rand.Int63n(int64(backoff / 2)))
          delay := backoff + jitter
          
          select {
          case <-ctx.Done():
              return ctx.Err()
          case <-time.After(delay):
          }
      }
      return fmt.Errorf("max retries exceeded: %w", lastErr)
  }
  
  func IsRetryable(err error) bool {
      if errors.Is(err, context.DeadlineExceeded) { return true }
      if errors.Is(err, context.Canceled) { return false }
      var netErr net.Error
      if errors.As(err, &netErr) { return netErr.Timeout() }
      return false
  }

  重试必须配合幂等：
  非幂等操作不能盲目重试
  用唯一请求ID保证幂等性
```

## 10. 错误处理面试速答？

**回答：**

```
Q: Go错误处理哲学？
A: errors are values 显式检查
   不用异常 函数签名即契约

Q: %w和%v区别？
A: %w保留原始错误 可Is/As解包
   %v仅字符串拼接 丢失类型信息

Q: 什么时候用panic？
A: 初始化失败/不可恢复的编程错误
   业务逻辑永远用error

Q: 错误日志怎么记？
A: 底层只包装 顶层统一记录
   错误只记一次 不重复

Q: 错误码怎么设计？
A: 模块分段(10000用户/20000订单)
   对外友好消息 对内完整错误链

Q: context超时怎么处理？
A: DeadlineExceeded→504
   Canceled→客户端取消 不告警

Q: 哪些错误可以重试？
A: 网络超时/连接拒绝/503/429
   400/401/404不可重试

Q: 分层架构错误怎么处理？
A: Repo层转换为业务错误
   Service层判断和包装
   Handler层转HTTP响应+记日志
```
