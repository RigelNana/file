# 代码审查

---

## 1. Code Review的目的？

**回答：**

```
  Code Review的价值：
  ┌──────────────────────┬──────────────────────────────┐
  │ 目的                  │ 说明                         │
  ├──────────────────────┼──────────────────────────────┤
  │ 发现Bug               │ 早期发现 修复成本最低        │
  │ 知识共享              │ 团队成员了解彼此代码         │
  │ 提升代码质量          │ 保持一致的代码标准           │
  │ 安全审查              │ 发现安全漏洞                 │
  │ 设计讨论              │ 讨论架构和设计决策           │
  │ 指导新人              │ 传授经验和最佳实践           │
  └──────────────────────┴──────────────────────────────┘

  Code Review不是：
  ✗ 找茬/指责
  ✗ 代码格式纠正（交给工具）
  ✗ 证明自己更厉害
  ✗ 阻塞开发进度的借口

  统计数据：
  Code Review能发现60-90%的缺陷
  修复成本：开发阶段 vs 生产环境 = 1:100
```

## 2. 审查什么内容？

**回答：**

```
  审查检查清单：

  1. 正确性
     业务逻辑是否正确
     边界条件处理
     并发安全
     错误处理完整性

  2. 设计
     职责划分是否合理
     抽象层次是否一致
     是否过度设计

  3. 可读性
     命名是否清晰
     函数是否太长
     注释是否必要且正确

  4. 性能
     N+1查询
     不必要的内存分配
     大循环中的重复操作
     缺少超时设置

  5. 安全性
     SQL注入
     XSS/CSRF
     越权访问
     敏感信息泄露
     输入校验

  6. 测试
     关键逻辑是否有测试
     测试用例是否覆盖边界
     测试是否可靠（不flaky）

  7. 一致性
     是否符合项目规范
     是否和现有代码风格一致

  不要审查的（交给工具）：
  代码格式 → gofmt
  静态检查 → golangci-lint
  依赖漏洞 → govulncheck
```

## 3. 怎么写好Review意见？

**回答：**

```
  原则：对事不对人 给建议不给命令

  差的评论：
  ✗ "这代码写得太烂了"
  ✗ "你不知道要用context吗？"
  ✗ "这个不对"（没说哪里不对/为什么）
  
  好的评论：
  ✓ "建议将这个函数拆分为两个 分别处理验证和持久化
     这样更容易测试和维护"
  
  ✓ "Nit: 这里的变量名改为 userOrders 更清晰"
  
  ✓ "这里没有检查 err 如果数据库连接失败
     会导致nil pointer dereference"

  评论分级：
  ┌──────────────┬──────────────────────────────┐
  │ 前缀          │ 含义                         │
  ├──────────────┼──────────────────────────────┤
  │ Blocker      │ 必须修改才能合并             │
  │ Suggestion   │ 建议修改 不强制              │
  │ Nit          │ 小问题 可忽略                │
  │ Question     │ 不理解 请解释                │
  │ Praise       │ 写得好 值得表扬              │
  └──────────────┴──────────────────────────────┘

  给出建议时附代码示例：
  "建议改为：
  ```go
  if err := validate(req); err != nil {
      return fmt.Errorf("validate: %w", err)
  }
  ```
  这样错误链更清晰"
```

## 4. PR的规范？

**回答：**

```
  PR大小：
  理想：200-400行改动
  最大：不超过800行
  大变更拆为多个PR

  PR标题：
  feat: 添加用户注册功能
  fix: 修复订单超时未取消的问题
  refactor: 重构用户认证模块
  
  遵循Conventional Commits：
  <type>(<scope>): <description>

  PR描述模板：
  ## What
  简要描述改了什么

  ## Why
  为什么需要这个改动（关联Issue）

  ## How
  实现方案简述

  ## Testing
  如何测试/测试了什么

  ## Checklist
  - [ ] 单元测试通过
  - [ ] 代码符合规范
  - [ ] 需要的文档已更新
  - [ ] 无安全问题

  PR注意事项：
  1. 自己先Review一遍再提交
  2. CI通过后再请人Review
  3. 及时响应Review意见
  4. 不要在PR中偷偷加无关改动
```

## 5. Review的效率？

**回答：**

```
  时间管理：
  每天固定时段做Review（如上午10点）
  单次Review不超过60分钟（注意力下降）
  一次Review不超过400行

  提升效率的方法：
  1. 自动化检查
     CI自动跑lint/test
     只Review机器不能检查的

  2. 优先级
     先看核心逻辑 再看辅助代码
     安全问题 > 正确性 > 设计 > 风格

  3. 两遍Review
     第一遍：整体设计和架构
     第二遍：细节和边界

  4. 预Review
     提交者先自己Review
     附上设计说明减少Reviewer理解时间

  Review响应时间：
  ┌──────────────┬──────────────────┐
  │ 目标          │ 时间             │
  ├──────────────┼──────────────────┤
  │ 首次响应      │ <4小时           │
  │ 完成Review    │ <1个工作日       │
  │ 紧急修复      │ <1小时           │
  └──────────────┴──────────────────┘

  避免来回：
  一次把所有问题提完
  不要挤牙膏式Review
```

## 6. 作为被Review者？

**回答：**

```
  心态：
  Review的是代码 不是人
  每条意见都是学习机会
  不要defensive（防御性回应）

  提交前自检：
  1. git diff 自己看一遍
  2. 删除调试代码和临时注释
  3. 确保CI通过
  4. PR描述清楚写好

  收到意见后：
  ┌──────────────────────┬──────────────────────┐
  │ 情况                  │ 处理                 │
  ├──────────────────────┼──────────────────────┤
  │ 同意                  │ 修改 + 回复"已修改"  │
  │ 不同意                │ 解释理由 讨论         │
  │ 不理解                │ 请对方解释           │
  │ Nit/小问题            │ 可创建后续Issue       │
  └──────────────────────┴──────────────────────┘

  不好的回应：
  ✗ "这样也能工作啊"
  ✗ "之前就是这么写的"
  ✗ 不回复直接merge
  
  好的回应：
  ✓ "好建议 已修改"
  ✓ "这里我考虑的是... 你觉得呢？"
  ✓ "认同 我创建了follow-up issue #123"
```

## 7. Go代码常见Review问题？

**回答：**

```
  高频问题：

  1. 忽略error
  // ✗
  json.Unmarshal(data, &v)
  // ✓
  if err := json.Unmarshal(data, &v); err != nil { ... }

  2. goroutine泄漏
  // ✗ channel没消费者 goroutine永远阻塞
  go func() { ch <- result }()
  // ✓ 带超时或context
  
  3. 缺少context传递
  // ✗
  func GetUser(id string) (*User, error)
  // ✓
  func GetUser(ctx context.Context, id string) (*User, error)

  4. 竞态条件
  // ✗ 并发写map
  go func() { m[key] = value }()
  // ✓ sync.Map 或 加锁

  5. defer在循环中
  // ✗ 文件句柄在循环结束后才关闭
  for _, f := range files {
      fp, _ := os.Open(f)
      defer fp.Close()  // 不会立即关闭！
  }
  // ✓ 提取为函数或手动关闭

  6. 切片append陷阱
  // ✗ 可能影响原切片
  newSlice := append(oldSlice, elem)
  // ✓ 明确copy
  newSlice := make([]int, len(oldSlice))
  copy(newSlice, oldSlice)
  newSlice = append(newSlice, elem)

  7. 硬编码配置
  // ✗
  timeout := 30 * time.Second
  // ✓
  timeout := cfg.RequestTimeout
```

## 8. 自动化Review工具？

**回答：**

```
  Go工具链：
  ┌──────────────────────┬──────────────────────┐
  │ 工具                  │ 检查内容             │
  ├──────────────────────┼──────────────────────┤
  │ gofmt/goimports      │ 格式化               │
  │ go vet               │ 常见错误             │
  │ staticcheck          │ 高级静态分析         │
  │ errcheck             │ 未检查的error        │
  │ gosec                │ 安全漏洞             │
  │ gocyclo              │ 圈复杂度             │
  │ ineffassign          │ 无效赋值             │
  │ golangci-lint        │ 以上全部集成         │
  └──────────────────────┴──────────────────────┘

  CI集成（GitHub Actions）：
  name: Code Review
  on: [pull_request]
  jobs:
    lint:
      runs-on: ubuntu-latest
      steps:
      - uses: actions/checkout@v4
      - uses: golangci/golangci-lint-action@v4
        with:
          version: latest
    
    test:
      runs-on: ubuntu-latest
      steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
        with:
          go-version: '1.22'
      - run: go test -race -cover ./...
    
    security:
      runs-on: ubuntu-latest
      steps:
      - run: |
          go install golang.org/x/vuln/cmd/govulncheck@latest
          govulncheck ./...

  PR门禁：
  以上全部通过才允许合并
  至少1人Approve
```

## 9. Review文化建设？

**回答：**

```
  建设健康Review文化：

  1. 制度保障
     所有代码必须Review（包括高级工程师）
     至少1人Approve才能合并
     定义SLA（4小时内首次响应）

  2. 文档规范
     编写Review指南
     常见问题检查清单
     好Review/坏Review示例

  3. 培训
     Reviewer培训
     新人如何提交好的PR
     定期分享Review中发现的好/坏Pattern

  4. 度量
     Review响应时间
     Review轮次（越少越好）
     Review发现的Bug比例

  5. 正向激励
     表扬好的Review意见
     表扬优质PR
     不惩罚被Review出问题的人

  团队约定示例：
  ┌──────────────────────────────────────┐
  │ 1. 对事不对人                         │
  │ 2. 所有意见都值得讨论                 │
  │ 3. 使用前缀分级(Blocker/Suggestion)   │
  │ 4. 承认不同方案的合理性               │
  │ 5. PR小而频繁                         │
  │ 6. 及时响应 不阻塞他人                │
  └──────────────────────────────────────┘
```

   分级(Blocker/Suggestion/Nit)

   提交者先自Review

Q: 收到Review意见怎么处理？
A: 同意就改 不同意就讨论
   不defensive 每条意见是学习机会

   竞态条件/defer在循环中

Q: 需要什么自动化工具？

   gosec+govulncheck CI门禁
