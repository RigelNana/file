# 代码整洁与规范八股文

## 一、代码整洁基础

### 1. 什么是Clean Code？为什么重要？

**答：** Clean Code是可读、可理解、可修改的代码。代码被读的次数远多于写的次数（10:1），整洁的代码降低维护成本、减少Bug、提高协作效率。核心特征：有意义的命名、小而专注的函数、最少的注释（代码自解释）、统一的格式、没有重复。Bob大叔总结：整洁的代码读起来像散文。

### 2. 好的命名有哪些原则？

**答：** ① 名副其实：变量名表达意图（`elapsedTimeInDays`而非`d`）；② 避免误导：不用`accountList`表示非List类型；③ 有意义的区分：避免`a1/a2`或`data/info`这类无意义差异；④ 可搜索：常量用命名替代魔法数字；⑤ 类名用名词（`User`、`OrderService`），方法名用动词（`GetUser`、`CreateOrder`）；⑥ 一致性：同一概念用同一个词（`Get/Fetch/Retrieve`选一个统一）。

### 3. 函数设计有什么原则？

**答：** ① 小：函数应该尽可能短，20行以内最佳；② 只做一件事（SRP）；③ 参数尽量少（0-2个，多了用结构体）；④ 无副作用：不隐式修改外部状态；⑤ 命令与查询分离（CQS）：修改状态的不要返回值，返回值的不要修改状态；⑥ DRY：不重复自己，提取公共逻辑；⑦ 错误处理不要用返回码，Go用`error`、其他语言用异常。

### 4. Go语言错误处理最佳实践？

**答：** ① 错误是值，显式检查不忽略：`if err != nil`；② 用`fmt.Errorf("context: %w", err)`包装错误，保留调用链；③ 用`errors.Is()`和`errors.As()`判断错误类型；④ 定义哨兵错误`var ErrNotFound = errors.New("not found")`；⑤ 不用panic处理业务错误，panic仅用于不可恢复的程序错误；⑥ 在最外层（main/handler）统一处理错误，中间层向上传播。

### 5. 注释的正确用法？

**答：** 好注释：法律信息、解释意图（为什么而非是什么）、TODO标记、公共API文档。坏注释：重复代码内容、注释掉的代码、日志式注释、位置标记。原则：注释是代码表达力不足的补偿，优先通过好的命名和结构让代码自解释。Go语言用`godoc`风格注释导出符号。

### 6. 什么是SOLID原则？

**答：**

| 原则 | 全称 | 说明 |
|------|------|------|
| S | 单一职责 | 一个类/模块只有一个变更的理由 |
| O | 开闭原则 | 对扩展开放，对修改关闭 |
| L | 里氏替换 | 子类可替换父类使用 |
| I | 接口隔离 | 接口小而专，不强迫实现不需要的方法 |
| D | 依赖倒置 | 依赖抽象而非具体实现 |

Go语言通过接口（隐式实现）天然支持I和D原则，组合优于继承体现了L原则。

### 7. 单元测试的原则？

**答：** ① FIRST原则：Fast快速、Independent独立、Repeatable可重复、Self-Validating自验证、Timely及时；② 一个测试一个断言（概念）；③ AAA模式：Arrange准备 → Act执行 → Assert断言；④ 测试覆盖率是参考不是目标，关注关键路径；⑤ Go用Table-Driven Tests（表驱动测试）减少重复；⑥ Mock外部依赖（数据库/API），用接口注入。

### 8. 重构是什么？什么时候重构？

**答：** 重构是在不改变外部行为的前提下改善代码内部结构。时机：① 三次法则（Rule of Three）——第三次重复时重构；② 添加新功能前先重构让代码更容易扩展；③ 修Bug时重构使问题不再发生；④ Code Review时发现的问题。前提：必须有测试保护。常见手法：提取函数、内联变量、移动方法、引入参数对象、以策略替换条件。

### 9. 代码审查（Code Review）关注什么？

**答：** ① 功能正确性：逻辑是否正确、边界条件；② 设计合理性：职责划分、抽象层次；③ 可读性：命名、注释、代码结构；④ 安全性：输入校验、SQL注入、越权；⑤ 性能：N+1查询、无效循环、内存泄漏；⑥ 测试：是否覆盖核心逻辑；⑦ 一致性：是否符合项目规范。原则：对事不对人，给出建议而非命令，及时Review不阻塞。

### 10. 代码质量度量指标？

**答：** ① 圈复杂度（Cyclomatic Complexity）：分支越多越复杂，建议<10；② 代码覆盖率：核心模块>80%；③ 重复率：重复代码比例<5%；④ 认知复杂度：代码理解难度；⑤ 依赖耦合度：包间依赖关系；⑥ 代码行数：函数<50行，文件<500行。Go工具：`gocyclo`检测圈复杂度，`golangci-lint`集成多种检查，`go test -cover`覆盖率。

## 二、工程规范

### 11. Go项目代码规范？

**答：** ① 使用`gofmt/goimports`自动格式化；② 遵循Effective Go和Go Code Review Comments；③ 变量命名：局部短命名、导出长命名、缩写全大写（`HTTP`不是`Http`）；④ 包名小写单词、不用下划线；⑤ 接口命名加`er`后缀（`Reader`、`Writer`）；⑥ 错误变量`Err`开头（`ErrNotFound`）。使用`golangci-lint`集成检查，CI强制通过。

### 12. 代码坏味道有哪些？

**答：** ① 过长函数/方法；② 过大的类/包；③ 过长参数列表；④ 重复代码；⑤ 发散式修改（一个类因多种原因修改）；⑥ 霰弹式修改（一个改动影响多个类）；⑦ 特性依恋（方法大量使用其他类的数据）；⑧ 数据泥团（多个数据总是一起出现）；⑨ 基本类型偏执（不封装领域概念）；⑩ 过度耦合（包间循环依赖）。

### 13. 什么是技术债？如何管理？

**答：** 技术债是为了短期目标而做的不理想技术决策，类似金融债务会产生"利息"（维护成本增加）。分类：有意债务（明知不理想但赶工期）和无意债务（经验不足导致）。管理：① 记录技术债（TODO/Issue）；② 定期偿还（每个Sprint分配20%时间）；③ 新代码不引入新债务；④ 重大债务做技术方案评审。

### 14. 代码覆盖率与测试策略？

**答：** 测试金字塔：单元测试（70%）> 集成测试（20%）> E2E测试（10%）。单元测试快速反馈，集成测试验证模块协作，E2E测试验证用户流程。Go测试策略：`go test`单元测试、`testcontainers`集成测试、表驱动减少重复、`-race`检测竞态、Benchmark测性能。覆盖率目标：核心业务>80%，工具类>90%，总体>60%。不追求100%，关注有意义的测试。
