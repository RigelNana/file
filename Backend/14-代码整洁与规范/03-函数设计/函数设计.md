## 1. 函数应该多短？

**回答：**

```
  原则：一个函数只做一件事 做完做好

  建议长度：
  ┌──────────────┬──────────────────┐
  │ 建议          │ 说明             │
  ├──────────────┼──────────────────┤
  │ 理想          │ 5-15行           │
  │ 可接受        │ 15-30行          │
  │ 需要关注      │ 30-50行          │
  │ 必须重构      │ >50行            │
  └──────────────┴──────────────────┘

  判断标准不是行数 而是抽象层次：
  函数内的每条语句都在同一个抽象层次

  // 差：混合了不同抽象层次
  func ProcessOrder(order *Order) error {
      // 验证（高层）
      if order.UserID == "" { return ErrInvalid }
      // SQL操作（低层）
      _, err := db.Exec("INSERT INTO orders ...")
      // 发送HTTP（低层）
      resp, err := http.Post("https://notify.svc/send", ...)
      // 日志（低层）
      log.Printf("order %s created", order.ID)
  }
  
  // 好：统一抽象层次
  func ProcessOrder(order *Order) error {
      if err := validateOrder(order); err != nil {
          return err
      }
      if err := saveOrder(order); err != nil {
          return err
      }
      if err := notifyUser(order); err != nil {
          log.Printf("notify failed: %v", err)
      }
      return nil
  }
```

## 2. 函数参数设计？

**回答：**

```
  参数数量指南：
  0个（最佳） > 1个 > 2个 > 3个（上限）
  超过3个 → 封装为结构体

  // 差：参数过多
  func CreateUser(name, email, phone string, age int, 
      role string, departmentID int, isActive bool) error

  // 好：参数对象
  type CreateUserRequest struct {
      Name         string
      Email        string
      Phone        string
      Age          int
      Role         string
      DepartmentID int
  }
  func CreateUser(req *CreateUserRequest) error

  Options模式（Go惯用）：
  type ServerOption func(*Server)
  
  func WithPort(port int) ServerOption {
      return func(s *Server) { s.port = port }
  }
  
  func WithTimeout(d time.Duration) ServerOption {
      return func(s *Server) { s.timeout = d }
  }
  
  func NewServer(addr string, opts ...ServerOption) *Server {
      s := &Server{addr: addr, port: 8080, timeout: 30*time.Second}
      for _, opt := range opts {
          opt(s)
      }
      return s
  }
  
  // 使用
  srv := NewServer("0.0.0.0",
      WithPort(9090),
      WithTimeout(60*time.Second),
  )

  参数顺序惯例：
  context.Context永远第一个
  输出参数放后面
  可选参数用Options模式
```

## 3. 函数返回值设计？

**回答：**

```
  Go多返回值模式：
  1. (result, error) — 最常见
  func GetUser(id string) (*User, error)
  
  2. (result, bool) — 查找/类型断言
  func (m *Map) Get(key string) (value string, ok bool)
  
  3. (result) — 绝不失败
  func NewDefaultConfig() *Config
  
  4. error — 只关心是否成功
  func DeleteUser(id string) error

  返回值原则：
  1. 不返回nil切片 返回空切片
  func ListUsers() []*User {
      users := make([]*User, 0)  // 不是 var users []*User
      // ...
      return users  // JSON序列化为[] 而非null
  }
  
  2. 错误要有上下文
  func GetUser(id string) (*User, error) {
      user, err := repo.Find(id)
      if err != nil {
          return nil, fmt.Errorf("get user %s: %w", id, err)
      }
      return user, nil
  }
  
  3. 避免返回太多值
  // 差
  func Parse() (name, email, phone string, age int, err error)
  // 好
  func Parse() (*UserInfo, error)

  命名返回值 — 仅用于文档说明：
  func Divide(a, b float64) (result float64, err error)
  // 不要用naked return 容易出错
```

## 4. 命令与查询分离（CQS）？

**回答：**

```
  CQS = Command-Query Separation

  查询（Query）：返回结果 不修改状态
  命令（Command）：修改状态 不返回结果

  // 差：混合了命令和查询
  func (s *Stack) Pop() interface{} {
      // 既修改了栈 又返回了值
  }
  
  // 差：查询有副作用
  func GetUserCount() int {
      count := db.Count()
      cache.Set("user_count", count)  // 副作用！
      return count
  }

  Go实践中的CQS：
  // 查询 - 只读
  func (s *OrderService) GetOrder(id string) (*Order, error)
  func (s *OrderService) ListOrders(filter Filter) ([]*Order, error)
  func (s *OrderService) CountOrders() (int64, error)
  
  // 命令 - 只写
  func (s *OrderService) CreateOrder(req CreateReq) error
  func (s *OrderService) UpdateStatus(id string, status Status) error
  func (s *OrderService) DeleteOrder(id string) error

  CQRS（架构级CQS）：
  ┌──────────────┐   ┌──────────────┐
  │ Write Model  │   │ Read Model   │
  │ 写模型(复杂)  │   │ 读模型(简单)  │
  │ 规范化存储    │   │ 冗余/物化视图 │
  └──────┬───────┘   └──────┬───────┘
         │     同步/事件      │
         └────────────────────┘

  例外：Pop/Dequeue等操作天然需要命令查询合一
  但要在文档中明确说明
```

## 5. 纯函数与副作用？

**回答：**

```
  纯函数 = 相同输入总是相同输出 + 无副作用

  副作用包括：
  - 修改全局/外部状态
  - IO操作（文件/数据库/网络）
  - 修改传入的参数
  - 随机数/时间

  // 纯函数（可预测 易测试）
  func CalculateTotal(items []Item) float64 {
      total := 0.0
      for _, item := range items {
          total += item.Price * float64(item.Quantity)
      }
      return total
  }
  
  // 非纯函数（依赖外部 有副作用）
  func CalculateTotal(items []Item) float64 {
      total := 0.0
      for _, item := range items {
          total += item.Price * float64(item.Quantity)
      }
      db.SaveTotal(total)  // 副作用
      return total
  }

  实践策略 — 函数式核心 命令式外壳：
  // 核心逻辑 = 纯函数（易测试）
  func CalculateDiscount(order Order, rules []Rule) float64 {
      // 纯计算 无IO
  }
  
  // 外壳 = 编排IO（薄层）
  func ProcessOrder(ctx context.Context, orderID string) error {
      order, _ := repo.GetOrder(ctx, orderID)      // IO
      rules, _ := repo.GetRules(ctx)                // IO
      discount := CalculateDiscount(order, rules)    // 纯计算
      order.Discount = discount
      return repo.SaveOrder(ctx, order)              // IO
  }
```

## 6. 函数的错误处理？

**回答：**

```
  Go错误处理最佳实践：

  1. 不忽略错误
  // 差
  result, _ := DoSomething()
  // 好
  result, err := DoSomething()
  if err != nil { return fmt.Errorf("do something: %w", err) }

  2. 错误包装提供上下文
  func GetUserOrders(userID string) ([]*Order, error) {
      user, err := userRepo.Get(userID)
      if err != nil {
          return nil, fmt.Errorf("get user %s: %w", userID, err)
      }
      orders, err := orderRepo.ListByUser(userID)
      if err != nil {
          return nil, fmt.Errorf("list orders for user %s: %w", userID, err)
      }
      return orders, nil
  }
  // 错误链：list orders for user 123: query: connection refused

  3. Happy Path左对齐
  // 好：正常流程在左 错误提前返回
  func ProcessRequest(r *Request) error {
      if err := validate(r); err != nil {
          return err
      }
      if err := authorize(r); err != nil {
          return err
      }
      if err := execute(r); err != nil {
          return err
      }
      return nil
  }

  4. defer处理资源清理
  func ReadFile(path string) ([]byte, error) {
      f, err := os.Open(path)
      if err != nil {
          return nil, err
      }
      defer f.Close()
      return io.ReadAll(f)
  }
```

## 7. 函数组合与管道？

**回答：**

```
  函数组合 = 小函数组合成大功能

  中间件模式（Go最常用的函数组合）：
  type Middleware func(http.Handler) http.Handler
  
  func Logging(next http.Handler) http.Handler {
      return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
          start := time.Now()
          next.ServeHTTP(w, r)
          log.Printf("%s %s %v", r.Method, r.URL.Path, time.Since(start))
      })
  }
  
  func Auth(next http.Handler) http.Handler {
      return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
          if !isAuthenticated(r) {
              http.Error(w, "unauthorized", 401)
              return
          }
          next.ServeHTTP(w, r)
      })
  }
  
  // 组合
  handler := Logging(Auth(RateLimit(myHandler)))

  管道模式：
  type Transform func(string) string
  
  func Pipeline(input string, transforms ...Transform) string {
      result := input
      for _, t := range transforms {
          result = t(result)
      }
      return result
  }
  
  output := Pipeline(rawInput,
      strings.TrimSpace,
      strings.ToLower,
      removeSpecialChars,
      truncateToMaxLength,
  )

  优势：
  每个函数独立可测试
  组合灵活 可插拔
  符合单一职责
```

## 8. 递归vs迭代？

**回答：**

```
  选择依据：
  ┌──────────────┬───────────────┬───────────────┐
  │ 维度          │ 递归           │ 迭代           │
  ├──────────────┼───────────────┼───────────────┤
  │ 可读性        │ 树/图遍历更清晰│ 简单循环更清晰 │
  │ 性能          │ 有栈开销       │ 通常更快       │
  │ 栈溢出风险    │ 深度大时有     │ 无             │
  │ 适用场景      │ 分治/回溯/树   │ 线性遍历       │
  └──────────────┴───────────────┴───────────────┘

  Go没有尾递归优化 → 深度递归用迭代

  // 递归（适合树遍历）
  func WalkTree(node *Node, visit func(*Node)) {
      if node == nil { return }
      visit(node)
      WalkTree(node.Left, visit)
      WalkTree(node.Right, visit)
  }
  
  // 深度大时改迭代
  func WalkTreeIter(root *Node, visit func(*Node)) {
      stack := []*Node{root}
      for len(stack) > 0 {
          node := stack[len(stack)-1]
          stack = stack[:len(stack)-1]
          if node == nil { continue }
          visit(node)
          stack = append(stack, node.Right, node.Left)
      }
  }

  实际建议：
  树/图遍历 → 递归更清晰（深度可控）
  列表处理 → 迭代
  深度不确定 → 迭代或加深度限制
```

## 9. 函数文档与契约？

**回答：**

```
  函数契约 = 前置条件 + 后置条件 + 不变式

  Go通过文档注释表达契约：
  // Transfer transfers amount from src to dst account.
  //
  // Preconditions:
  //   - amount must be positive
  //   - src and dst must be different accounts
  //   - src must have sufficient balance
  //
  // It returns ErrInsufficientFunds if src balance < amount.
  // It returns ErrSameAccount if src == dst.
  // The operation is atomic: either both accounts update or neither does.
  func Transfer(ctx context.Context, src, dst string, amount int64) error

  防御式编程 vs 契约式编程：
  // 防御式：函数内部检查（面向外部输入）
  func CreateUser(req *CreateUserReq) error {
      if req == nil { return ErrNilRequest }
      if req.Email == "" { return ErrInvalidEmail }
      // ...
  }
  
  // 契约式：调用者保证（面向内部调用）
  // 内部函数可以假定参数已验证
  func saveUser(user *User) error {
      // user不会为nil 上层已验证
  }

  实践原则：
  系统边界（Handler）→ 防御式 严格验证
  内部函数 → 契约式 信任调用者
  导出函数 → 文档说明前置条件
```

## 10. 函数设计面试速答？

**回答：**

```
Q: 函数多长合适？
A: 理想5-15行 同一抽象层次
   只做一件事

Q: 函数参数多少合适？
A: 0-2个最佳 超过3个用结构体
   Go用Options模式处理可选参数

Q: 什么是CQS？
A: 命令查询分离
   查询不修改状态 命令不返回结果

Q: 纯函数有什么好处？
A: 可预测+易测试+无副作用
   核心逻辑用纯函数 IO放外层

Q: Go错误处理原则？
A: 不忽略错误+包装上下文
   Happy Path左对齐+defer清理

Q: Go的Options模式？
A: func WithPort(p int) Option
   NewServer(addr, WithPort(9090))
   解决参数过多的问题

Q: 什么时候用递归？
A: 树/图遍历 分治算法
   Go无尾递归优化 深度大用迭代

Q: 函数组合怎么实现？
A: 中间件模式 func(Handler) Handler
   管道模式 按顺序执行Transform
```
