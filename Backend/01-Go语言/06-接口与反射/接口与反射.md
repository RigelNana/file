# 接口与反射

---

## 1. interface 底层实现？iface 和 eface 的区别？

**回答：**

```
Go 接口的两种底层结构:

eface (empty interface): interface{}
  ┌─────────────┐
  │ _type *_type │ → 类型信息 (大小, 对齐, hash, 方法等)
  ├─────────────┤
  │ data  unsafe│ → 指向实际数据的指针
  │     .Pointer│
  └─────────────┘

iface (non-empty interface): 有方法的接口
  ┌─────────────┐
  │ tab  *itab  │ → 接口类型 + 具体类型 + 方法表
  ├─────────────┤
  │ data  unsafe│ → 指向实际数据的指针
  │     .Pointer│
  └─────────────┘

itab 结构:
  ┌──────────────┐
  │ inter *type  │ → 接口类型信息
  │ _type *type  │ → 具体类型信息
  │ hash  uint32 │ → 类型 hash (加速断言)
  │ fun   [1]ptr │ → 方法表 (虚函数表)
  └──────────────┘

比较:
  ┌──────────┬───────────────┬───────────────┐
  │          │ eface         │ iface         │
  ├──────────┼───────────────┼───────────────┤
  │ 场景      │ interface{}   │ 有方法的接口   │
  │ 头部      │ _type         │ itab          │
  │ 方法表    │ 无            │ 有 (fun[])    │
  │ 大小      │ 16B           │ 16B           │
  │ 断言开销  │ 比较 _type    │ 比较 itab     │
  └──────────┴───────────────┴───────────────┘
```

## 2. 类型断言和类型开关？

**回答：**

```go
// 类型断言 (Type Assertion)
var i interface{} = "hello"

// 单值断言: 失败 panic
s := i.(string)

// 双值断言: 安全
s, ok := i.(string)
if !ok {
    // 断言失败
}

// 类型开关 (Type Switch)
func describe(i interface{}) string {
    switch v := i.(type) {
    case int:
        return fmt.Sprintf("int: %d", v)
    case string:
        return fmt.Sprintf("string: %s", v)
    case []byte:
        return fmt.Sprintf("bytes: len=%d", len(v))
    case nil:
        return "nil"
    default:
        return fmt.Sprintf("unknown: %T", v)
    }
}
```

```
底层实现:
  类型断言 → 比较 itab/type 指针
    iface 断言: 查找 itab 缓存表 (hash 表)
    eface 断言: 直接比较 _type 指针

  itab 缓存:
    全局 hash 表, key = (接口类型, 具体类型)
    第一次断言计算并缓存, 后续 O(1)

性能:
  类型断言 ≈ 比较指针 (纳秒级, 有缓存)
  比 reflect 快得多
```

## 3. nil interface 陷阱？

**回答：**

```go
// 经典陷阱: interface 为 nil 的条件

type MyError struct{ msg string }
func (e *MyError) Error() string { return e.msg }

func getError() error {
    var p *MyError = nil
    return p // 返回 non-nil interface!
}

func main() {
    err := getError()
    fmt.Println(err == nil) // false!
    // err 的 iface: { tab: *itab(error, *MyError), data: nil }
    // tab 不为 nil → interface 不为 nil
}

// 正确写法:
func getError() error {
    var p *MyError = nil
    if p == nil {
        return nil // 返回真正的 nil interface
    }
    return p
}
```

```
interface == nil 的条件:
  iface: tab == nil && data == nil
  eface: _type == nil && data == nil

  ┌──────────────┬───────┬───────┬────────────┐
  │ 情况          │ type  │ data  │ == nil ?    │
  ├──────────────┼───────┼───────┼────────────┤
  │ var e error   │ nil   │ nil   │ true       │
  │ e = (*T)(nil) │ *T    │ nil   │ false!     │
  │ e = &T{}      │ *T    │ 0xABC │ false      │
  └──────────────┴───────┴───────┴────────────┘

  要判断 data 是否 nil:
    reflect.ValueOf(err).IsNil()
```

## 4. 接口的隐式实现和鸭子类型？

**回答：**

```go
// Go 接口: 隐式实现 (无需声明 implements)
type Writer interface {
    Write(p []byte) (n int, err error)
}

// 只要有 Write 方法就实现了 Writer
type MyWriter struct{}
func (w MyWriter) Write(p []byte) (int, error) {
    return len(p), nil
}

// 编译期检查 (常用技巧)
var _ Writer = (*MyWriter)(nil) // 编译期确认实现
var _ Writer = MyWriter{}       // 值类型也实现

// 鸭子类型 vs 接口
// "如果它走路像鸭子, 叫声像鸭子, 那它就是鸭子"
// Go: 结构化类型 (structural typing)
// Python/JS: 鸭子类型 (运行时检查)
// Java/C#: 名义类型 (显式声明)

// 接口设计原则:
// 1. 小接口原则 (1-3 个方法)
type Reader interface { Read(p []byte) (n int, err error) }
type Writer interface { Write(p []byte) (n int, err error) }
type ReadWriter interface { Reader; Writer } // 组合

// 2. 在消费端定义接口, 不在实现端
// 好: 哪里用就在哪里定义
// 差: 在 package 里定义大而全的接口

// 3. 接受接口, 返回结构体
func NewService(repo Repository) *Service { ... }
```

## 5. reflect 包的核心 API？

**回答：**

```go
import "reflect"

// 两大核心类型: Type 和 Value
t := reflect.TypeOf(x)   // 获取类型信息 → reflect.Type
v := reflect.ValueOf(x)  // 获取值信息   → reflect.Value

// Type 常用操作
t.Kind()       // 底层类型: struct, ptr, slice...
t.Name()       // 类型名: "User"
t.NumField()   // 字段数量 (struct)
t.Field(i)     // 第 i 个字段信息
t.NumMethod()  // 方法数量
t.Method(i)    // 第 i 个方法
t.Elem()       // 元素类型 (ptr/slice/map/chan)
t.Implements() // 是否实现接口

// Value 常用操作
v.Kind()       // 底层类型
v.Type()       // → reflect.Type
v.Interface()  // → interface{}
v.Int()        // → int64
v.String()     // → string
v.Field(i)     // 第 i 个字段值
v.Elem()       // 解引用指针
v.Set(newVal)  // 设置值 (必须可设置)
v.CanSet()     // 是否可设置
v.Call(args)   // 调用函数

// 反射三定律:
// 1. interface{} → reflect.Value (ValueOf)
// 2. reflect.Value → interface{} (Interface())
// 3. 要修改反射对象, 值必须可设置 (CanSet)

// 修改值必须传指针:
x := 42
v := reflect.ValueOf(&x).Elem() // 必须 &x 然后 Elem()
v.SetInt(100)
fmt.Println(x) // 100
```

## 6. Struct Tag 和反射读取？

**回答：**

```go
type User struct {
    Name  string `json:"name" db:"user_name" validate:"required"`
    Age   int    `json:"age,omitempty" db:"age" validate:"min=0,max=150"`
    Email string `json:"email" db:"email" validate:"email"`
}

// 读取 tag:
t := reflect.TypeOf(User{})
for i := 0; i < t.NumField(); i++ {
    field := t.Field(i)
    fmt.Println(field.Name)
    fmt.Println(field.Tag.Get("json"))     // "name"
    fmt.Println(field.Tag.Get("db"))       // "user_name"
    fmt.Println(field.Tag.Get("validate")) // "required"
}

// Tag 格式: `key:"value" key2:"value2"`
// key:value 之间用空格分隔
// value 中逗号分隔选项

// 常见 Tag:
// json:     encoding/json
// xml:      encoding/xml
// db/gorm:  ORM 映射
// yaml:     YAML 解析
// validate: 验证规则
// form:     表单绑定

// Tag 在反射中的使用:
// json.Marshal/Unmarshal 通过反射读取 json tag
// 决定字段的 JSON key, 是否 omitempty 等
// 这也是为什么 JSON 操作需要反射, 性能较低

// 性能优化方案:
// 1. 缓存反射结果 (encoding/json 已做)
// 2. 用 code generation: easyjson, jsoniter
// 3. 用 unsafe: jsoniter 在某些场景使用
```

## 7. 反射的性能开销和替代方案？

**回答：**

```
反射性能对比 (大致):
  ┌──────────────────┬──────────────┐
  │ 操作              │ 相对耗时      │
  ├──────────────────┼──────────────┤
  │ 直接调用          │ 1x (基准)    │
  │ 接口方法调用      │ ~1.5x        │
  │ reflect.Call      │ ~100x        │
  │ reflect.ValueOf   │ ~50x         │
  │ json.Marshal      │ ~10-30x      │
  └──────────────────┴──────────────┘

原因:
  - 大量 allocation (interface{} 频繁装箱)
  - 运行时类型检查
  - 无法内联

替代方案:
  ┌──────────────────┬──────────────────────────────┐
  │ 方案              │ 说明                          │
  ├──────────────────┼──────────────────────────────┤
  │ 代码生成          │ go generate + 模板生成代码    │
  │                  │ 如 easyjson, protobuf        │
  │ 接口 + 类型断言   │ 比反射快, 编译时可检查        │
  │ go:linkname      │ 链接到未导出函数 (慎用)       │
  │ unsafe.Pointer   │ 跳过类型系统 (极慎用)         │
  │ 泛型 (Go 1.18+)  │ 编译时特化, 无运行时开销      │
  └──────────────────┴──────────────────────────────┘
```

```go
// 代码生成示例: 手动序列化
//go:generate easyjson -all user.go

// 泛型替代反射 (Go 1.18+)
func Map[T any, R any](s []T, f func(T) R) []R {
    result := make([]R, len(s))
    for i, v := range s {
        result[i] = f(v)
    }
    return result
}
// 编译时特化, 零反射开销
```

## 8. 接口组合和嵌入？

**回答：**

```go
// 小接口原则: Go 标准库典型设计
type Reader interface {
    Read(p []byte) (n int, err error)
}

type Writer interface {
    Write(p []byte) (n int, err error)
}

type Closer interface {
    Close() error
}

// 组合接口
type ReadWriter interface {
    Reader
    Writer
}

type ReadWriteCloser interface {
    Reader
    Writer
    Closer
}

// 标准库典型接口大小:
// io.Reader      → 1 method
// io.Writer      → 1 method
// io.Closer      → 1 method
// fmt.Stringer   → 1 method
// error          → 1 method  (最常用)
// sort.Interface → 3 methods
// http.Handler   → 1 method
// context.Context → 4 methods (偏大, 但核心接口)

// 接口嵌入 struct:
type Cacher struct {
    io.Reader       // 嵌入接口
    io.Writer
    buf []byte
}

// c.Read() → 调用嵌入的 Reader.Read()
// c.Write() → 调用嵌入的 Writer.Write()

// 常见模式: 接口守卫
type responseWriter struct {
    http.ResponseWriter // 嵌入接口
    status int
    written bool
}

func (rw *responseWriter) WriteHeader(code int) {
    rw.status = code
    rw.written = true
    rw.ResponseWriter.WriteHeader(code)
}
```

## 9. 方法集规则与接口满足？

**回答：**

```
方法集规则:
  ┌──────────────┬──────────────────────────┐
  │ 类型          │ 方法集                    │
  ├──────────────┼──────────────────────────┤
  │ T (值类型)    │ T 的值接收者方法          │
  │ *T (指针类型) │ T 和 *T 的所有方法        │
  └──────────────┴──────────────────────────┘

  → *T 方法集 ⊇ T 方法集
```

```go
type Animal interface {
    Speak() string
    Move()
}

type Dog struct{ Name string }

func (d Dog) Speak() string { return "Woof" }  // 值接收者
func (d *Dog) Move()        { /* ... */ }       // 指针接收者

var a Animal

a = &Dog{"Rex"}  // OK: *Dog 有 Speak + Move
a = Dog{"Rex"}   // 编译错误! Dog 只有 Speak, 缺 Move

// 原因:
// Dog 值可能是不可寻址的 (如函数返回值)
// 不能对不可寻址的值取指针调用指针方法
// 但 &Dog 总能调用值方法 (编译器自动解引用)

// 实际影响:
// 如果接口方法任一是指针接收者
//   → 只能用指针类型实现接口
// 如果接口方法全是值接收者
//   → 值类型和指针类型都能实现接口

// 经验法则:
// 需要修改接收者 → 指针接收者
// 结构体大 → 指针接收者 (避免拷贝)
// 一致性: 一个类型尽量统一用一种接收者
```

## 10. 接口与反射面试速答？

**回答：**

```
Q: interface 底层实现?
A: 两种: eface(空接口,_type+data) 和 iface(有方法,itab+data)
   itab 含方法表, 全局缓存加速断言

Q: nil interface 陷阱?
A: 给 interface 赋 (*T)(nil), type 非 nil → interface 非 nil
   正确: 先判 nil 再返回 nil (不要返回 typed nil)

Q: 类型断言底层?
A: 比较 itab 指针, 有全局 hash 缓存
   首次计算后缓存, O(1) 查找

Q: 方法集规则?
A: T 方法集=值接收者方法, *T 方法集=所有方法
   指针接收者方法只能通过指针类型满足接口

Q: reflect 性能?
A: 比直接调用慢 ~100x
   替代: 代码生成/泛型/接口断言

Q: struct tag 怎么读?
A: reflect.TypeOf(s).Field(i).Tag.Get("json")
   json/db/validate 等库通过反射读取 tag

Q: Go 接口 vs Java 接口?
A: Go 隐式实现(鸭子类型), Java 显式 implements
   Go 小接口组合, Java 继承层次
   Go 运行时多态, Java 编译时+运行时
```
