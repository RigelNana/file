## 1. pprof 性能分析工具？

**回答：**

```go
// pprof: Go 内置的性能分析工具

// 方式1: HTTP 服务内嵌 pprof
import _ "net/http/pprof" // 匿名导入注册路由
go func() {
    http.ListenAndServe("localhost:6060", nil)
}()
// 访问: http://localhost:6060/debug/pprof/

// 方式2: 代码中手动收集
import "runtime/pprof"

f, _ := os.Create("cpu.prof")
pprof.StartCPUProfile(f)
defer pprof.StopCPUProfile()

// Profile 类型:
// /debug/pprof/profile    → CPU Profile
// /debug/pprof/heap       → 内存分配
// /debug/pprof/goroutine  → Goroutine 栈
// /debug/pprof/block      → 阻塞分析
// /debug/pprof/mutex      → 锁竞争
// /debug/pprof/trace      → 执行追踪
```

```bash
# 采集和分析
# CPU 分析 (30秒)
go tool pprof http://localhost:6060/debug/pprof/profile?seconds=30

# 内存分析
go tool pprof http://localhost:6060/debug/pprof/heap

# 交互模式常用命令:
#   top          → 最耗资源的函数
#   list funcName → 源码级分析
#   web          → 生成火焰图 (需要 graphviz)
#   svg          → SVG 图

# 基准测试 + pprof
go test -bench=. -cpuprofile=cpu.prof -memprofile=mem.prof
go tool pprof cpu.prof

# 火焰图 (推荐 go tool pprof -http=:8080)
go tool pprof -http=:8080 cpu.prof
# 浏览器打开火焰图面
```

```
Profile 含义:
  ┌──────────────┬──────────────────────────────┐
  │ Profile      │ 排查场景                      │
  ├──────────────┼──────────────────────────────┤
  │ CPU          │ CPU 使用率高, 找热点函数       │
  │ Heap (alloc) │ 内存大, 找分配多的地方         │
  │ Heap (inuse) │ 内存泄漏, 找驻留内存           │
  │ Goroutine    │ Goroutine 泄漏, 数量暴增      │
  │ Block        │ Channel/Lock 阻塞             │
  │ Mutex        │ 锁竞争严重                     │
  │ Trace        │ 调度延迟, GC 暂停              │
  └──────────────┴──────────────────────────────┘
```

## 2. 内存逃逸分析？

**回答：**

```
逃逸分析 (Escape Analysis):
  编译器决定变量分配在栈还是堆

  栈分配: 快, 函数返回自动释放, 无 GC 压力
  堆分配: 慢, 需要 GC 回收, 有 GC 压力

常见逃逸场景:
  ┌──────────────────────────┬──────────────────────┐
  │ 场景                      │ 原因                  │
  ├──────────────────────────┼──────────────────────┤
  │ 返回指针                  │ 变量生命周期超出函数   │
  │ 接口参数 (interface{})    │ 编译时不知道具体类型   │
  │ 闭包引用外部变量           │ 变量被闭包捕获        │
  │ slice/map 太大            │ 超过栈大小限制        │
  │ 发送到 channel            │ 变量生命周期不确定    │
  │ fmt.Println(x)           │ 参数是 interface{}    │
  └──────────────────────────┴──────────────────────┘
```

```go
// 查看逃逸分析结果:
// go build -gcflags="-m" main.go
// go build -gcflags="-m -m" main.go  (更详细)

// 逃逸示例:
func newUser() *User {
    u := User{Name: "Alice"} // u 逃逸到堆
    return &u                // 返回了指针
}

// 不逃逸:
func process() {
    u := User{Name: "Alice"} // u 在栈上
    fmt.Println(u.Name)      // 但如果传给 interface{} 参数...
    // 实际上 u.Name 会逃逸!
}

// 减少逃逸的技巧:
// 1. 传值而非传指针 (小结构体)
// 2. 避免不必要的 interface{} 参数
// 3. 预分配 slice/map
// 4. 用 sync.Pool 复用对象
// 5. 使用 strings.Builder 而非 fmt.Sprintf
```

## 3. 常见性能优化手段？

**回答：**

```
性能优化层次:
  ┌──────────────────────────────────────┐
  │ 1. 算法/数据结构优化 (效果最大)        │
  ├──────────────────────────────────────┤
  │ 2. 减少内存分配 (减轻 GC 压力)        │
  ├──────────────────────────────────────┤
  │ 3. 并发优化 (充分利用 CPU)            │
  ├──────────────────────────────────────┤
  │ 4. IO 优化 (减少系统调用)             │
  ├──────────────────────────────────────┤
  │ 5. 编译器/汇编优化 (极少需要)          │
  └──────────────────────────────────────┘
```

```go
// 1. 字符串拼接: strings.Builder (非 + 号)
var b strings.Builder
b.Grow(1024) // 预分配
for i := 0; i < 1000; i++ {
    b.WriteString("x")
}
s := b.String()

// 2. sync.Pool 复用对象
var bufPool = sync.Pool{
    New: func() interface{} {
        return new(bytes.Buffer)
    },
}
func process() {
    buf := bufPool.Get().(*bytes.Buffer)
    defer func() {
        buf.Reset()
        bufPool.Put(buf)
    }()
    // 使用 buf...
}

// 3. slice 预分配
users := make([]User, 0, len(ids)) // 预知容量
for _, id := range ids {
    users = append(users, getUser(id))
}

// 4. 避免 map 频繁扩容
m := make(map[string]int, 1000) // 预估大小

// 5. 用 strconv 替代 fmt
s := strconv.Itoa(42)  // 快
s := fmt.Sprintf("%d", 42) // 慢 (用反射)

// 6. 减少锁粒度
type ShardedMap struct {
    shards [256]struct {
        mu sync.RWMutex
        m  map[string]interface{}
    }
}
// 按 key hash 分片, 减少锁竞争

// 7. 批量操作
// 差: 逐条插入 DB
// 好: 批量 INSERT, 减少网络往返
```

## 4. Goroutine 泄漏检测与排查？

**回答：**

```go
// Goroutine 泄漏: goroutine 数量持续增长不释放

// 常见泄漏场景:
// 1. Channel 未关闭, goroutine 永远阻塞在读
func leak1() {
    ch := make(chan int)
    go func() { ch <- 1 }() // 无人读, 永远阻塞
    // goroutine 泄漏!
}

// 2. 忘记取消 Context
func leak2() {
    ctx, cancel := context.WithCancel(context.Background())
    // 忘了 defer cancel() → goroutine 泄漏
    go func() {
        <-ctx.Done() // 永远等不到
    }()
}

// 3. HTTP Response Body 未关闭
func leak3() {
    resp, _ := http.Get("https://example.com")
    // 忘了 resp.Body.Close() → 连接泄漏
}

// 检测方法:
// 1. runtime.NumGoroutine()
func monitorGoroutines() {
    for range time.Tick(10 * time.Second) {
        log.Printf("goroutines: %d", runtime.NumGoroutine())
    }
}

// 2. pprof goroutine
// go tool pprof http://localhost:6060/debug/pprof/goroutine

// 3. goleak (uber 的测试工具)
import "go.uber.org/goleak"

func TestMain(m *testing.M) {
    goleak.VerifyTestMain(m) // 测试结束检查 goroutine 泄漏
}

// 防泄漏最佳实践:
// - 每个 goroutine 都要有退出机制 (context/done channel)
// - defer cancel() 是标配
// - defer resp.Body.Close() 是标配
// - 用 errgroup / WaitGroup 管理 goroutine 生命周期
```

## 5. 竞态检测与并发调试？

**回答：**

```bash
# Go Race Detector: 运行时检测数据竞态
go test -race ./...
go run -race main.go
go build -race -o myapp && ./myapp

# 原理: 编译时插桩, 运行时检测并发读写冲突
# 性能: 慢 2-10x, 内存 5-10x → 测试环境使用
```

```go
// 竞态示例
var counter int
func increment() { counter++ } // 非原子操作!

// 修复方案:
// 1. sync.Mutex
var mu sync.Mutex
func increment() {
    mu.Lock()
    counter++
    mu.Unlock()
}

// 2. atomic
var counter int64
func increment() {
    atomic.AddInt64(&counter, 1)
}

// 3. Channel
func increment(ch chan<- int) {
    ch <- 1 // 通过 channel 通信
}

// 并发调试工具:
// -race            → 竞态检测
// GODEBUG=schedtrace=1000 → 调度器追踪
// runtime.Stack()  → 当前 goroutine 栈
// pprof goroutine  → 所有 goroutine 栈
// trace 工具       → 可视化调度

// runtime/trace: 执行追踪
import "runtime/trace"
f, _ := os.Create("trace.out")
trace.Start(f)
defer trace.Stop()
// go tool trace trace.out → 浏览器可视化
```

## 6. GC 调优与 GOGC？

**回答：**

```
Go GC 调优参数:

  GOGC (默认 100):
    控制 GC 触发频率
    GOGC=100 → 堆增长 100% 时触发 GC
    GOGC=50  → 堆增长 50% 时触发 (更频繁, 内存低)
    GOGC=200 → 堆增长 200% 时触发 (更少, CPU低)
    GOGC=off → 关闭 GC (特殊场景)

  GOMEMLIMIT (Go 1.19+):
    设置内存软上限
    GOMEMLIMIT=1GiB → 堆接近 1GiB 时积极 GC
    → 替代 GOGC 的推荐方案

  ┌─────────────────────────────────────────────┐
  │ 场景              │ 建议                      │
  ├─────────────────────────────────────────────┤
  │ 低延迟服务        │ GOGC=50 (更频繁但STW更短)  │
  │ 高吞吐批处理      │ GOGC=200 或 off           │
  │ 内存受限 (容器)   │ GOMEMLIMIT=容器内存×70%    │
  │ 默认              │ GOGC=100 + GOMEMLIMIT     │
  └─────────────────────────────────────────────┘
```

```go
// 运行时调整
import "runtime/debug"
debug.SetGCPercent(50)           // 等同 GOGC=50
debug.SetMemoryLimit(1 << 30)    // 1GiB 限制

// 监控 GC
import "runtime"
var stats runtime.MemStats
runtime.ReadMemStats(&stats)
fmt.Printf("HeapAlloc: %d MB\n", stats.HeapAlloc/1024/1024)
fmt.Printf("NumGC: %d\n", stats.NumGC)
fmt.Printf("PauseTotalNs: %d ms\n", stats.PauseTotalNs/1e6)

// GC 可视化:
// GODEBUG=gctrace=1 → 打印每次 GC 信息
// go tool trace → 可视化 GC 暂停时间
```

## 7. 编译优化与构建技巧？

**回答：**

```bash
# 编译优化标志
go build -ldflags="-s -w" -o app  # 去除符号表和调试信息 (减小体积)
# -s: 去符号表
# -w: 去 DWARF 调试信息
# 体积可减少 ~30%

# 注入版本信息
go build -ldflags="-X main.version=v1.2.3 -X main.buildTime=$(date -u +%Y%m%d)" -o app

# 交叉编译
GOOS=linux GOARCH=amd64 go build -o app-linux
GOOS=darwin GOARCH=arm64 go build -o app-mac
GOOS=windows GOARCH=amd64 go build -o app.exe

# CGO 控制
CGO_ENABLED=0 go build -o app  # 纯 Go, 静态链接

# 进一步压缩: UPX
upx --best app  # 可再压缩 50-70%
```

```go
// 编译器指令 (Compiler Directives)

//go:inline       // 强制内联
//go:noinline     // 禁止内联
//go:nosplit      // 禁止栈分裂
//go:noescape     // 标记不逃逸

//go:build linux  // 条件编译 (Go 1.17+)

// 查看编译器优化:
// go build -gcflags="-m"       → 逃逸分析
```

## 8. 常用调试工具 dlv？

**回答：**

```bash
# Delve (dlv): Go 专用调试器

# 安装
go install github.com/go-delve/delve/cmd/dlv@latest

# 调试模式
dlv debug main.go              # 调试源码
dlv test ./pkg/user            # 调试测试
dlv attach <pid>               # 附加到运行进程
dlv exec ./myapp               # 调试编译后的程序
dlv core ./myapp core.dump     # 分析 core dump

# 常用命令:
# break (b)    main.go:42       → 设置断点
# continue (c)                  → 继续执行
# next (n)                      → 单步 (不进入)
# step (s)                      → 单步 (进入函数)
# stepout                       → 跳出当前函数
# print (p)    variableName     → 打印变量
# locals                        → 所有局部变量
# args                          → 函数参数
# goroutines                    → 所有 goroutine
# goroutine <id>                → 切换 goroutine
# stack (bt)                    → 调用栈
# condition (cond) <bp> <expr>  → 条件断点

# 远程调试 (容器/K8s)
dlv debug --headless --listen=:2345 --api-version=2
# VS Code / GoLand 连接 :2345 远程调试
```

## 10. 性能优化与调试面试速答？

**回答：**

```
Q: Go 性能分析用什么工具?
A: pprof (CPU/内存/goroutine/阻塞/锁)
   trace (调度/GC 可视化)
   -race (竞态检测)

Q: 逃逸分析是什么?
A: 编译器决定变量在栈还是堆分配
   返回指针/接口参数/闭包捕获 → 逃逸到堆
   go build -gcflags="-m" 查看

Q: 怎么减少 GC 压力?
A: 减少堆分配: 预分配/sync.Pool/避免逃逸
   调 GOGC/GOMEMLIMIT 控制 GC 频率
   用 strconv 代替 fmt 减少反射

Q: Goroutine 泄漏怎么排查?
A: pprof goroutine 看数量和栈
   goleak 在测试中检测
   防泄漏: context+cancel, close channel

Q: GOGC 和 GOMEMLIMIT?
A: GOGC=100: 堆增长 100% 触发 GC (默认)
   GOMEMLIMIT: 内存软上限, Go 1.19+
   容器中推荐 GOMEMLIMIT=容器内存×70%

Q: 字符串拼接怎么优化?
A: strings.Builder + Grow 预分配
   比 + 号快 100 倍以上 (避免中间分配)

Q: 怎么做竞态检测?
A: go test -race / go run -race
   编译时插桩, 运行时检测并发冲突
   测试/CI 环境必须开启

Q: dlv 调试器怎么用?
A: dlv debug/attach/test
   break/continue/next/step/print
   支持远程调试 (容器/K8s)
```
