## 1. 性能优化方法论？

**回答：**

```
  性能优化三步：量化 → 定位 → 优化

  Step 1：量化指标
  ┌──────────┬──────────────────────────┐
  │ 指标      │ 说明                     │
  ├──────────┼──────────────────────────┤
  │ QPS/TPS  │ 吞吐量                   │
  │ RT       │ 响应时间(P50/P95/P99)    │
  │ 并发数   │ 同时处理的请求数         │
  │ 错误率   │ 失败请求占比             │
  └──────────┴──────────────────────────┘
  
  QPS = 并发数 / 平均RT

  Step 2：定位瓶颈
  CPU密集 → pprof/top
  IO密集 → iostat/netstat
  内存 → 内存泄漏/GC
  锁竞争 → mutex profile

  Step 3：优化
  先优化瓶颈最大的点
  每次只改一个变量
  对比优化前后数据
```

## 2. Go 性能分析工具？

**回答：**

```
pprof 是 Go 核心性能分析工具：

  import _ "net/http/pprof"
  
  go func() {
      http.ListenAndServe(":6060", nil)
  }()

  常用命令：
  go tool pprof http://localhost:6060/debug/pprof/profile?seconds=30
  go tool pprof http://localhost:6060/debug/pprof/heap
  go tool pprof http://localhost:6060/debug/pprof/goroutine

  pprof 交互命令：
  top 10        → 最耗资源的10个函数
  list funcName → 查看函数逐行耗时
  web           → 生成调用图

  Benchmark 测试：
  func BenchmarkXxx(b *testing.B) {
      for i := 0; i < b.N; i++ {
          // 要测试的函数
      }
  }
  
  go test -bench=. -benchmem -cpuprofile=cpu.prof

  Trace 工具：
  go tool trace trace.out
  → 可视化 goroutine 调度、GC、阻塞
```

## 3. 数据库性能优化？

**回答：**

```
  ┌────────────────┬───────────────────────────┐
  │ 方向            │ 措施                      │
  ├────────────────┼───────────────────────────┤
  │ 索引优化        │ 合理索引 覆盖索引 避免回表│
  │ SQL优化         │ 避免SELECT * 减少子查询   │
  │ 连接池          │ 复用连接 减少建连开销     │
  │ 读写分离        │ 写主读从 分担压力         │
  │ 分库分表        │ 水平拆分 降低单表数据量   │
  │ 缓存            │ 热数据放Redis             │
  └────────────────┴───────────────────────────┘

  索引优化要点：
  最左前缀原则（联合索引）
  避免索引失效：函数/类型转换/LIKE '%xx'
  EXPLAIN 分析执行计划

  慢查询排查：
  1. 开启慢查询日志
  2. EXPLAIN 看执行计划
  3. type: ALL(全表扫描) → 需要加索引
  4. rows: 扫描行数过多 → 优化索引

  批量操作优化：
  // 差：循环单条插入
  for _, u := range users {
      db.Exec("INSERT ...", u.Name)
  }
  
  // 好：批量插入
  db.Exec("INSERT INTO users(name) VALUES (?),(?),(?)", 
      names[0], names[1], names[2])
```

## 4. 缓存性能优化？

**回答：**

```
  三大缓存问题：
  ┌──────────┬──────────────────┬──────────────────┐
  │ 问题      │ 原因             │ 解决             │
  ├──────────┼──────────────────┼──────────────────┤
  │ 缓存穿透  │ 查询不存在的数据 │ 布隆过滤器/空值  │
  │ 缓存击穿  │ 热点Key过期瞬间  │ singleflight/锁 │
  │ 缓存雪崩  │ 大量Key同时过期  │ 随机过期时间     │
  └──────────┴──────────────────┴──────────────────┘

Go singleflight 防击穿：
  import "golang.org/x/sync/singleflight"
  
  var g singleflight.Group
  
  func GetUser(id string) (*User, error) {
      v, err, _ := g.Do("user:"+id, func() (interface{}, error) {
          // 只有一个请求会真正查DB
          return queryDB(id)
      })
      if err != nil {
          return nil, err
      }
      return v.(*User), nil
  }

  热Key处理：
  1. 本地缓存（不走Redis网络）
  2. Key打散（user:123 → user:123:{0-9}）
  3. 读写分离（Redis读从库）
```

## 5. 网络性能优化？

**回答：**

```
  ┌────────────────┬──────────────────────────┐
  │ 优化方向        │ 方法                     │
  ├────────────────┼──────────────────────────┤
  │ 减少请求数      │ 合并API/批量接口         │
  │ 减少传输量      │ 压缩/精简字段            │
  │ 减少连接开销    │ 长连接/连接池            │
  │ 减少延迟        │ CDN/就近部署             │
  │ 协议优化        │ HTTP/2/gRPC/protobuf     │
  └────────────────┴──────────────────────────┘

  HTTP/2 优势：
  多路复用（一个连接并行多个请求）
  Header 压缩（HPACK）
  服务端推送

  gRPC 性能：
  基于 HTTP/2
  Protobuf 序列化（比 JSON 快5-10倍）
  流式传输

  JSON vs Protobuf 对比：
  JSON:     {"user_id":123,"name":"test"}  → ~40 bytes
  Protobuf: 二进制编码                      → ~10 bytes
  
  序列化速度：Protobuf 比 JSON 快5倍+
  反序列化：同样快几倍

  连接复用：
  HTTP Keep-Alive（默认开启）
  gRPC 长连接 + 连接池
  数据库连接池
```

## 6. Go 并发性能优化？

**回答：**

```
  Goroutine 优化：
  1. 控制 goroutine 数量
     → Worker Pool 模式
     → 无限创建 goroutine 会 OOM
  
  2. 减少锁竞争
     → sync.RWMutex（读多写少）
     → sync.Map（读多写少的map）
     → atomic 代替 Mutex（简单计数）
     → 分段锁（按key分桶）

  3. 减少内存分配
     → sync.Pool 复用对象
     → 预分配 slice（make([]T, 0, n)）
     → strings.Builder 代替 string 拼接

sync.Pool 示例：
  var bufPool = sync.Pool{
      New: func() interface{} {
          return new(bytes.Buffer)
      },
  }
  
  func process() {
      buf := bufPool.Get().(*bytes.Buffer)
      defer func() {
          buf.Reset()
          bufPool.Put(buf)
      }()
      // 使用 buf
  }

  4. 减少 GC 压力
     → 减少堆分配（逃逸分析）
     → 用值类型代替指针（小对象）
     → GOGC 调参（默认100）
```

## 7. 零拷贝与IO优化？

**回答：**

```
传统IO（4次拷贝）：
  磁盘 → 内核缓冲区 → 用户buffer → Socket缓冲区 → 网卡
  + 4次上下文切换

sendfile 零拷贝（2次拷贝）：
  磁盘 → 内核缓冲区 → 网卡（DMA直接传输）
  + 2次上下文切换

  应用场景：
  Nginx 发送静态文件
  Kafka 消费消息
  → sendfile 系统调用

IO 多路复用：
  epoll（Linux）：O(1) 事件通知
  Go net 包底层已用 epoll
  → 一个 goroutine 一个连接也不怕（netpoller）

Go IO 优化：
  bufio.Reader/Writer → 减少系统调用次数
  io.Copy → 内部可能用 sendfile
  
  // 大文件传输
  func serveFile(w http.ResponseWriter, path string) {
      f, _ := os.Open(path)
      defer f.Close()
      io.Copy(w, f) // Go 底层优化
  }
```

## 8. 序列化性能？

**回答：**

```
  ┌──────────────┬────────┬────────┬──────────┐
  │ 格式          │ 速度   │ 体积   │ 可读性   │
  ├──────────────┼────────┼────────┼──────────┤
  │ JSON          │ 慢     │ 大     │ ✓        │
  │ Protobuf      │ 快     │ 小     │ ✗        │
  │ MessagePack   │ 较快   │ 较小   │ ✗        │
  │ FlatBuffers   │ 最快   │ 小     │ ✗        │
  └──────────────┴────────┴────────┴──────────┘

  JSON 优化：
  encoding/json → 反射慢
  推荐 sonic (字节跳动)/json-iterator → 快2-5倍
  
  // sonic 示例
  import "github.com/bytedance/sonic"
  data, _ := sonic.Marshal(obj)
  sonic.Unmarshal(data, &obj)

  Protobuf 推荐场景：
  微服务内部 gRPC 通信
  消息队列消息格式
  大数据量传输

  减少序列化的方式：
  结果缓存（序列化后的字符串直接缓存）
  避免重复序列化
  选择合适的格式
```

  func BenchmarkSort(b *testing.B) {
      data := generateData(1000)
      b.ResetTimer() // 重置计时器
      for i := 0; i < b.N; i++ {
          d := make([]int, len(data))
          copy(d, data)
          sort.Ints(d)

  // 运行并对比
  go test -bench=. -benchmem -count=5 | tee old.txt

  预热后再测（JIT/缓存预热）

## 10. 高性能面试速答？

**回答：**

```
Q: 性能优化先做什么？
A: 先量化指标 再pprof定位瓶颈
   每次只改一个变量 对比数据

Q: Go怎么做性能分析？
A: pprof分析CPU/内存/goroutine
   Benchmark做微基准测试 benchstat对比

Q: DB性能优化关键？
A: 索引优化(最左前缀/覆盖索引)
   避免全表扫描 批量操作 连接池

Q: 缓存三大问题？
A: 穿透(查不存在)→布隆过滤器
   击穿(热Key过期)→singleflight
   雪崩(批量过期)→随机过期时间

Q: gRPC为什么快？
A: HTTP/2多路复用+Protobuf序列化
   比JSON快5-10倍 体积小3-5倍

Q: Go并发性能优化？
A: Worker Pool控制goroutine数
   sync.Pool复用对象 减少GC

Q: 零拷贝原理？
A: sendfile跳过用户态
   磁盘→内核→网卡 减少拷贝次数

Q: JSON序列化太慢怎么办？
A: 用sonic/json-iterator替换标准库
   内部通信用Protobuf
```
