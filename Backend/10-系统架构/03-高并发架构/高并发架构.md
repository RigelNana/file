# 高并发架构

---

## 1. 高并发的核心思路？

**回答：**

```
高并发 = 分流 + 缓冲 + 异步

  ┌──────────┬───────────────────────────────┐
  │ 策略      │ 实现方式                      │
  ├──────────┼───────────────────────────────┤
  │ 横向扩展  │ 加机器、负载均衡              │
  │ 缓存      │ 多级缓存减少DB访问            │
  │ 异步      │ 消息队列削峰填谷              │
  │ 分库分表  │ 数据层水平拆分                │
  │ 读写分离  │ 写主库读从库                  │
  │ CDN       │ 静态资源就近访问              │
  │ 连接池    │ 复用连接减少开销              │
  └──────────┴───────────────────────────────┘

  请求链路优化：
  Client → CDN → LB → Gateway → Cache → Service → DB
                 ↑                  ↑              ↑
               分流              挡住90%       分库分表
```

## 2. 水平扩展？

**回答：**

```
垂直扩展（Scale Up）：加CPU/内存 → 有天花板
水平扩展（Scale Out）：加机器 → 理论无上限

  水平扩展前提：
  1. 服务无状态
  2. 数据可分片
  3. 有负载均衡

  服务层水平扩展：
  ┌────────┐
  │  LB    │
  └──┬─────┘
  ┌──┼──┬──┬──┐
  │S1│S2│S3│S4│  加机器即可
  └──┴──┴──┴──┘

  数据层水平扩展：
  ┌──────────────┐
  │ Sharding Key │
  └──────┬───────┘
    hash(key) % N
  ┌──┬──┬──┬──┐
  │D1│D2│D3│D4│  分库分表
  └──┴──┴──┴──┘

Go 服务扩展容易（编译成二进制直接部署）：
  K8s HPA：基于CPU/QPS自动扩缩
  apiVersion: autoscaling/v2
  kind: HorizontalPodAutoscaler
  spec:
    minReplicas: 3
    maxReplicas: 20
    metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          averageUtilization: 70
```

## 3. 多级缓存架构？

**回答：**

```
  Client Cache → CDN → Nginx(Lua) → Local Cache → Redis → DB

  ┌──────────────┬──────────┬────────────┬──────────┐
  │ 层级          │ 容量     │ 延迟       │ 命中率   │
  ├──────────────┼──────────┼────────────┼──────────┤
  │ 客户端缓存    │ 小       │ 0ms        │ 30%      │
  │ CDN           │ 大       │ 5ms        │ 50%      │
  │ 本地缓存      │ 小(GB级) │ <1ms       │ 80%      │
  │ 分布式缓存    │ 大(TB级) │ 1-5ms      │ 95%      │
  │ 数据库        │ 很大     │ 10-100ms   │ 100%     │
  └──────────────┴──────────┴────────────┴──────────┘

Go 本地缓存方案：
  sync.Map       → 简单场景
  bigcache       → 大量KV（无GC压力）
  ristretto      → 带准入策略的LFU
  freecache      → 零GC

缓存更新策略：
  Cache Aside：先更新DB 再删缓存
  Read Through：缓存层自动加载
  Write Behind：异步批量写DB
  删除优于更新（避免并发不一致）
```

## 4. 消息队列削峰？

**回答：**

```
场景：秒杀 瞬间QPS可能是平时100倍

  不用MQ：
  100万请求 ──→ DB 扛不住 → 崩溃

  用MQ削峰：
  100万请求 ──→ MQ(缓冲) ──→ Consumer恒速消费 ──→ DB
                 ↑                     ↑
               承接洪峰          按DB承受力消费

Go 消费者限速消费：
  func consume(ch <-chan Message, db *sql.DB) {
      // 控制消费速率
      limiter := rate.NewLimiter(rate.Limit(1000), 100)
      
      for msg := range ch {
          limiter.Wait(context.Background())
          processOrder(db, msg)
      }
  }

MQ 削峰适用场景：
  秒杀/抢购
  批量导入
  大促活动
  突发流量

不适用：
  需要同步结果的场景
  延迟敏感的核心链路
```

## 5. 读写分离？

**回答：**

```
  写 → 主库(Master)
  读 → 从库(Slave)

  ┌────────┐
  │ Client │
  └──┬─────┘
  ┌──┴──────────┐
  │ Proxy       │  ShardingSphere / ProxySQL
  ├─────────────┤
  │ 写 → Master │
  │ 读 → Slave  │
  └──┬──────┬───┘
  ┌──┴──┐ ┌─┴────┐
  │Master│ │Slave │ × N
  └──────┘ └──────┘
     │  binlog复制  ↑
     └──────────────┘

问题：主从延迟
  解决：
  1. 关键读走主库（强一致场景）
  2. 写后短暂从主库读（flag标记）
  3. 半同步复制减少延迟

Go 实现读写分离：
  type DBCluster struct {
      master *sql.DB
      slaves []*sql.DB
  }
  
  func (c *DBCluster) Master() *sql.DB {
      return c.master
  }
  
  func (c *DBCluster) Slave() *sql.DB {
      idx := rand.Intn(len(c.slaves))
      return c.slaves[idx]
  }
  
  // 写操作
  cluster.Master().Exec(...)
  // 读操作
  cluster.Slave().Query(...)
```

## 6. 分库分表？

**回答：**

```
何时分库分表：
  单表 > 2000万行 或 > 10GB
  单库连接数不够

  ┌──────────┬──────────────────────────┐
  │ 方式      │ 说明                     │
  ├──────────┼──────────────────────────┤
  │ 垂直分库  │ 按业务拆（用户/订单/商品）│
  │ 水平分库  │ 同一业务数据分到多个库   │
  │ 垂直分表  │ 按字段拆（主表+扩展表）  │
  │ 水平分表  │ 同一表数据分到多个表     │
  └──────────┴──────────────────────────┘

分片策略：
  Hash取模：hash(userID) % 16
    → 均匀分布 但扩容要迁移数据
  Range范围：userID 1~100万库1 100万~200万库2
    → 扩容简单 但可能不均匀
  一致性Hash：虚拟节点环
    → 扩容只迁移少量数据

分库分表后的问题：
  ┌──────────────┬──────────────────────┐
  │ 问题          │ 解决                 │
  ├──────────────┼──────────────────────┤
  │ 跨库 Join     │ 冗余字段/应用层Join  │
  │ 分布式事务    │ Saga/TCC             │
  │ 全局排序分页  │ 各分片取数 合并排序  │
  │ 全局唯一ID    │ Snowflake/UUID       │
  │ 扩容迁移      │ 双写迁移/一致性Hash  │
  └──────────────┴──────────────────────┘
```

## 7. 连接池设计？

**回答：**

```
为什么用连接池：
  建立TCP连接（三次握手）+ 认证 = 昂贵
  复用连接 = 省去建立/销毁开销

  ┌─────────────────────────┐
  │ Connection Pool          │
  │ ┌───┐ ┌───┐ ┌───┐      │
  │ │C1 │ │C2 │ │C3 │ idle │
  │ └───┘ └───┘ └───┘      │
  │ ┌───┐ ┌───┐            │
  │ │C4 │ │C5 │ in-use     │
  │ └───┘ └───┘            │
  └─────────────────────────┘

Go database/sql 内置连接池：
  db.SetMaxOpenConns(100)     // 最大连接
  db.SetMaxIdleConns(20)      // 空闲连接
  db.SetConnMaxLifetime(time.Hour) // 连接存活时间
  db.SetConnMaxIdleTime(10 * time.Minute) // 空闲超时

参数调优：
  MaxOpen = 业务并发数 × 1.2
  MaxIdle = MaxOpen × 0.2
  注意：MaxOpen 不能超过 DB 的 max_connections / 实例数

Redis 连接池（go-redis）：
  rdb := redis.NewClient(&redis.Options{
      PoolSize:     100,
      MinIdleConns: 10,
      PoolTimeout:  3 * time.Second,
  })

HTTP Client 连接池：
  transport := &http.Transport{
      MaxIdleConns:        100,
      MaxIdleConnsPerHost: 10,
      IdleConnTimeout:     90 * time.Second,
  }
```

## 8. 负载均衡算法？

**回答：**

```
  ┌────────────────┬─────────────────────────────┐
  │ 算法            │ 特点                        │
  ├────────────────┼─────────────────────────────┤
  │ 轮询            │ 简单均匀                    │
  │ 加权轮询        │ 按机器性能分配权重          │
  │ 随机            │ 简单，大量请求趋于均匀      │
  │ 加权随机        │ 按权重随机                  │
  │ 最少连接        │ 优先分给连接最少的          │
  │ 一致性Hash      │ 相同Key到相同节点           │
  │ IP Hash         │ 相同IP到相同节点（会话保持） │
  └────────────────┴─────────────────────────────┘

四层 vs 七层负载均衡：
  L4（传输层）：LVS/F5，按IP+端口转发，性能高
  L7（应用层）：Nginx/HAProxy，按URL/Header路由，灵活

Go 实现简单加权轮询：
  type Backend struct {
      URL    string
      Weight int
  }
  
  type WeightedRR struct {
      backends       []Backend
      currentWeights []int
  }
  
  func (w *WeightedRR) Next() string {
      total := 0
      maxIdx := 0
      for i := range w.backends {
          w.currentWeights[i] += w.backends[i].Weight
          total += w.backends[i].Weight
          if w.currentWeights[i] > w.currentWeights[maxIdx] {
              maxIdx = i
          }
      }
      w.currentWeights[maxIdx] -= total
      return w.backends[maxIdx].URL
  }
```

## 9. 流量治理？

**回答：**

```
流量治理 = 限流 + 熔断 + 降级 + 隔离

  ┌──────────────────────────────────────────┐
  │                 流量入口                  │
  │  ┌────────┐ ┌────────┐ ┌────────────┐   │
  │  │ 限流   │ │ 熔断   │ │ 降级/隔离  │   │
  │  └────────┘ └────────┘ └────────────┘   │
  │  超出阈值   下游故障    故障时兜底       │
  │  → 拒绝     → 停止调用  → 返回默认值    │
  └──────────────────────────────────────────┘

流量隔离：
  线程池隔离（Java Hystrix）
  信号量隔离
  泳道隔离（按用户等级划分独立链路）

  VIP用户 → 独立服务实例 → 独立DB
  普通用户 → 共享服务实例 → 共享DB

流量染色：
  在请求Header加标记
  全链路透传（TraceID + 环境标记）
  用途：灰度路由、压测流量识别

  X-Traffic-Tag: canary
  X-Traffic-Tag: shadow（压测影子流量）

全链路压测：
  影子库/影子表存压测数据
  标记压测流量 不影响真实数据
  验证系统真实承载能力
```

## 10. 高并发面试速答？

**回答：**

```
Q: 高并发核心思路？
A: 分流(LB)+缓冲(MQ)+缓存+异步
   水平扩展 无状态服务

Q: 多级缓存顺序？
A: 客户端→CDN→Nginx→本地缓存→Redis→DB
   越靠前命中率越好

Q: 读写分离的问题？
A: 主从延迟 导致读到旧数据
   关键读走主库 写后标记读主

Q: 分库分表怎么分？
A: 垂直分库按业务 水平分表按分片键
   Hash取模或Range 优先选Hash

Q: 连接池参数怎么配？
A: MaxOpen=业务并发×1.2
   MaxIdle=MaxOpen×20%
   不超过DB max_connections/实例数

Q: L4和L7负载均衡区别？
A: L4按IP+端口转发 性能高(LVS)
   L7按URL/Header路由 更灵活(Nginx)

Q: MQ削峰原理？
A: 突发流量写入MQ缓冲
   消费者按DB承受力恒速消费

Q: 全链路压测怎么做？
A: 流量染色+影子库/表
   标记压测数据不影响线上
```
