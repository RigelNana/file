# 架构设计基础与原则

---

## 1. 架构设计的核心目标？

**回答：**

```
架构设计 = 在约束条件下做权衡取舍

四个核心目标：
  ┌────────────────┬──────────────────────────┐
  │ 目标            │ 说明                     │
  ├────────────────┼──────────────────────────┤
  │ 可用性          │ 系统持续可用             │
  │ 可扩展性        │ 应对增长                 │
  │ 可维护性        │ 容易修改和理解           │
  │ 性能            │ 快速响应                 │
  └────────────────┴──────────────────────────┘

约束条件：
  业务约束（需求、法规）
  技术约束（团队能力、技术债）
  资源约束（时间、预算、人力）

架构 ≠ 追求完美
架构 = 在约束下做最优权衡
```

## 2. 分层架构？

**回答：**

```
经典三层/四层架构：

  ┌─────────────────────────┐
  │ Presentation Layer      │  表现层（API/UI）
  ├─────────────────────────┤
  │ Business Logic Layer    │  业务逻辑层
  ├─────────────────────────┤
  │ Data Access Layer       │  数据访问层
  ├─────────────────────────┤
  │ Database / External     │  基础设施
  └─────────────────────────┘

  原则：
  - 上层依赖下层，不跨层调用
  - 每层只做自己的事
  - 层间通过接口通信

Go 项目中的分层：
  handler/     → 表现层（HTTP/gRPC处理）
  service/     → 业务逻辑层
  repository/  → 数据访问层
  model/       → 数据模型
  pkg/         → 公共工具

  handler → service → repository → DB
  每层通过接口依赖
```

## 3. 整洁架构（Clean Architecture）？

**回答：**

```
Robert Martin 提出：依赖方向由外向内

  ┌──────────────────────────────────┐
  │ Frameworks & Drivers             │ 外层
  │  ┌──────────────────────────┐    │
  │  │ Interface Adapters       │    │
  │  │  ┌──────────────────┐    │    │
  │  │  │ Use Cases        │    │    │
  │  │  │  ┌──────────┐    │    │    │
  │  │  │  │ Entities │    │    │    │ 内层
  │  │  │  └──────────┘    │    │    │
  │  │  └──────────────────┘    │    │
  │  └──────────────────────────┘    │
  └──────────────────────────────────┘

  内层不知道外层的存在
  外层依赖内层的接口

  Entities：领域对象（最核心）
  Use Cases：业务用例
  Interface Adapters：转换（Controller/Presenter/Gateway）
  Frameworks：框架/驱动（DB/Web框架/UI）

  核心价值：
  - 业务逻辑不依赖框架
  - 可独立测试
  - 可替换外层实现
```

## 4. 六边形架构（端口适配器）？

**回答：**

```
业务核心在中间，通过端口与外部交互

  ┌────────┐    Port    ┌──────────┐    Port    ┌────────┐
  │ HTTP   │───────────→│          │←───────────│ MySQL  │
  │ Client │   Adapter  │ Business │  Adapter   │ DB     │
  └────────┘            │   Core   │            └────────┘
  ┌────────┐            │          │            ┌────────┐
  │ gRPC   │───────────→│          │←───────────│ Redis  │
  │ Client │   Adapter  │          │  Adapter   │ Cache  │
  └────────┘            └──────────┘            └────────┘
                          左侧：驱动端口（入）
                          右侧：被驱动端口（出）

Go 实现：
  Port = interface
  Adapter = interface 的具体实现
  
  业务核心只定义接口
  适配器实现接口
  通过 DI 注入
```

## 5. 可用性怎么衡量？

**回答：**

```
可用性 = MTBF / (MTBF + MTTR)
  MTBF：平均无故障时间
  MTTR：平均修复时间

  ┌────────┬──────────────────┬──────────┐
  │ 等级   │ 可用性           │ 年停机   │
  ├────────┼──────────────────┼──────────┤
  │ 2个9   │ 99%              │ 3.65天   │
  │ 3个9   │ 99.9%            │ 8.76小时 │
  │ 4个9   │ 99.99%           │ 52分钟   │
  │ 5个9   │ 99.999%          │ 5.26分钟 │
  └────────┴──────────────────┴──────────┘

  一般业务：3个9
  核心交易：4个9
  基础设施：5个9

SLI / SLO / SLA：
  SLI（指标）：请求成功率、P99延迟
  SLO（目标）：成功率 > 99.9%
  SLA（协议）：不达标的赔偿
```

## 6. 架构设计原则？

**回答：**

```
  ┌──────────────┬──────────────────────────────┐
  │ 原则          │ 说明                         │
  ├──────────────┼──────────────────────────────┤
  │ 简单原则      │ KISS，能简单就不复杂         │
  │ 合适原则      │ 合适的才是最好的             │
  │ 演进原则      │ 渐进式演进，不大重构         │
  │ 单一职责      │ 每个组件只做一件事           │
  │ 关注点分离    │ 不同关注点分开处理           │
  │ DRY           │ 不重复业务逻辑               │
  │ 面向失败      │ 设计时就考虑会出什么问题     │
  │ 无状态        │ 服务无状态才能水平扩展       │
  │ 幂等设计      │ 重复操作结果相同             │
  └──────────────┴──────────────────────────────┘

最重要的三个：
  1. 简单（刚开始别过度设计）
  2. 合适（匹配团队和业务阶段）
  3. 演进（允许架构随业务生长）
```

## 7. 面向失败设计？

**回答：**

```
Design for Failure：假设一切都会出错

  ┌──────────┬──────────────────────────────┐
  │ 失败类型  │ 应对策略                     │
  ├──────────┼──────────────────────────────┤
  │ 网络故障  │ 重试+超时+熔断               │
  │ 服务宕机  │ 冗余+自动切换+健康检查       │
  │ 磁盘故障  │ RAID+多副本+备份             │
  │ 机房故障  │ 异地多活+跨机房复制          │
  │ 人为错误  │ 灰度发布+回滚+变更审批       │
  │ 流量洪峰  │ 限流+降级+弹性伸缩          │
  └──────────┴──────────────────────────────┘

核心思想：
  1. 不信任任何外部调用（设超时）
  2. 不信任单点（做冗余）
  3. 不信任网络（做重试和幂等）
  4. 预留降级方案
```

## 8. 无状态设计？

**回答：**

```
无状态服务：请求之间不共享状态

  有状态：
    Server A (session=abc)
    → 用户只能去 Server A
    → Session 丢失用户登出

  无状态：
    Server A / B / C 都可处理
    → 状态存在外部（Redis/JWT）
    → 水平扩展自由

  ┌──────────┐     ┌──────────┐
  │ Client   │────→│LB 随机分配│
  └──────────┘     └───┬──────┘
                    ┌──┴──┬──────┐
                  ┌─┴─┐ ┌┴──┐ ┌─┴─┐
                  │S1 │ │S2 │ │S3 │  无状态服务
                  └───┘ └───┘ └───┘
                       ↓
                  ┌──────────┐
                  │  Redis   │  状态外部存储
                  └──────────┘

Go 实践：
  不用全局变量存状态
  Session 存 Redis
  用 JWT 代替 Session
  文件上传用 OSS
```

## 9. 容量评估方法？

**回答：**

```
Step 1：业务量估算
  日活用户(DAU) = 100万
  人均请求 = 30次/天
  日总请求 = 3000万

Step 2：QPS 计算
  平均 QPS = 3000万 / 86400 ≈ 350
  峰值系数 = 3~5倍
  峰值 QPS = 350 × 5 = 1750

Step 3：单机能力（压测）
  单机 QPS = 500（压测结果）

Step 4：所需机器
  机器数 = 1750 / 500 = 3.5
  冗余系数 1.5 → 6 台

Step 5：其他资源
  带宽 = QPS × 单请求大小
  存储 = 日新增数据 × 保留天数
  连接数 = 并发数 × 连接复用率

注意：
  峰值才是关键，不是平均
  预留 50% 以上余量
  最好做压力测试验证
```
