# 高可用架构

---

## 1. 高可用核心策略？

**回答：**

```
高可用 = 消除单点 + 快速恢复

  ┌──────────────┬─────────────────────────────┐
  │ 策略          │ 实现方式                    │
  ├──────────────┼─────────────────────────────┤
  │ 冗余          │ 多实例/多副本/多机房        │
  │ 自动切换      │ 健康检查+故障摘除+主备切换  │
  │ 熔断降级      │ 故障隔离+降级兜底           │
  │ 限流保护      │ 防止过载击垮系统            │
  │ 异步解耦      │ 消息队列缓冲写入            │
  │ 数据备份      │ 定期备份+跨机房/跨地域复制  │
  └──────────────┴─────────────────────────────┘

  关键指标：
  MTBF（平均无故障时间）→ 越长越好
  MTTR（平均修复时间）→ 越短越好
  可用性 = MTBF / (MTBF + MTTR)
```

## 2. 冗余设计？

**回答：**

```
服务冗余：
  ┌──────┐
  │  LB  │
  └──┬───┘
  ┌──┴──┬──────┬──────┐
  │ S1  │ S2   │ S3   │  ≥ 2 个实例
  └─────┴──────┴──────┘
  任何一个挂掉 LB 自动摘除

数据冗余：
  MySQL 主从：1主2从 + 半同步复制
  Redis Cluster：每个分片一主一从
  Kafka：三副本 ISR ≥ 2

基础设施冗余：
  网络：双链路、双交换机
  电源：双路供电 + UPS
  机房：异地多活

冗余公式：
  单机可用性 = 99%
  双机可用性 = 1 - (1-0.99)² = 99.99%
  三机可用性 = 1 - (1-0.99)³ = 99.9999%
```

## 3. 故障检测与自动切换？

**回答：**

```
健康检查方式：
  ┌──────────┬──────────────────────────┐
  │ 方式      │ 说明                     │
  ├──────────┼──────────────────────────┤
  │ TCP 探测  │ 端口是否可连             │
  │ HTTP 探测 │ /health 返回 200         │
  │ 心跳检测  │ 主动上报心跳             │
  │ 业务探测  │ 执行真实业务（读写一行）  │
  └──────────┴──────────────────────────┘

Go 健康检查端点：
  func healthHandler(w http.ResponseWriter, r *http.Request) {
      // 检查依赖
      if err := db.Ping(); err != nil {
          w.WriteHeader(http.StatusServiceUnavailable)
          json.NewEncoder(w).Encode(map[string]string{
              "status": "unhealthy",
              "error":  err.Error(),
          })
          return
      }
      w.WriteHeader(http.StatusOK)
      json.NewEncoder(w).Encode(map[string]string{"status": "ok"})
  }

自动切换：
  K8s：Liveness Probe 失败 → 重启容器
  Nginx：upstream 失败 → 摘除节点
  MySQL：MHA/Orchestrator 主从切换
  Redis：Sentinel 哨兵自动选主
```

## 4. 熔断与降级？

**回答：**

```
熔断器三种状态：

  Closed ──失败率>阈值──→ Open
    ↑                       │
    │                   超时后
    │                       ↓
    └──── 探测成功 ←── Half-Open

  Closed：正常通过请求
  Open：直接拒绝，不调下游
  Half-Open：放少量请求探测

降级策略：
  ┌──────────────┬──────────────────────────┐
  │ 级别          │ 做法                     │
  ├──────────────┼──────────────────────────┤
  │ P0 核心功能   │ 保障不降级               │
  │ P1 重要功能   │ 简化逻辑                 │
  │ P2 一般功能   │ 返回缓存/默认值          │
  │ P3 次要功能   │ 直接关闭                 │
  └──────────────┴──────────────────────────┘

  推荐信息流：完整 → 去掉推荐 → 返回热门缓存
  搜索：精确搜索 → 模糊搜索 → 返回热门榜单
  下单：正常 → 排队 → 限购
```

## 5. 限流算法？

**回答：**

```
  ┌────────────────┬───────────────────────────────┐
  │ 算法            │ 特点                          │
  ├────────────────┼───────────────────────────────┤
  │ 固定窗口        │ 简单，有临界突刺              │
  │ 滑动窗口        │ 精确，内存开销大              │
  │ 漏桶            │ 恒定速率，不抗突发            │
  │ 令牌桶          │ 允许突发，最常用              │
  └────────────────┴───────────────────────────────┘

令牌桶原理：
  定速 r 往桶放令牌
  桶容量 b（允许突发）
  每个请求取一个令牌
  没令牌 → 拒绝

Go 标准库限流：
  import "golang.org/x/time/rate"

  limiter := rate.NewLimiter(rate.Limit(100), 200)
  // 每秒100个令牌 桶容量200

  if !limiter.Allow() {
      // 限流 返回 429
      http.Error(w, "Too Many Requests", 429)
      return
  }

分布式限流：
  Redis + Lua 脚本实现
  按用户ID/IP/接口做Key
```

## 6. 异地多活？

**回答：**

```
部署模式对比：
  ┌──────────┬──────────────────────┬────────┐
  │ 模式      │ 说明                 │ 成本   │
  ├──────────┼──────────────────────┼────────┤
  │ 冷备      │ 备机房不提供服务     │ 低     │
  │ 热备      │ 备机房只读           │ 中     │
  │ 双活      │ 两机房都读写         │ 高     │
  │ 多活      │ 多机房都读写         │ 很高   │
  └──────────┴──────────────────────┴────────┘

异地多活挑战：
  数据一致性（延迟导致的）
  流量路由（用户分配到哪个机房）
  数据冲突（同一数据多处修改）

常见方案：
  1. 按用户分片（UserID % N 路由到不同机房）
     → 同一用户请求总在同一机房，无冲突
  2. 只读多活（写统一到主机房）
     → 简单但写入有跨机房延迟
  3. 单元化（Set化）
     → 用户+其关联数据封闭在一个单元
     → 蚂蚁金服 LDC 方案

数据同步：
  MySQL：Canal/DTS 双向复制
  Redis：CRDT 或 双向同步中间件
  关键：解决数据冲突（时间戳/版本号）
```

## 7. 超时设计？

**回答：**

```
超时是高可用的基础防线

  Client ──2s──→ Gateway ──1s──→ ServiceA ──500ms──→ DB
                                          ──300ms──→ Redis

  原则：
  1. 每个外部调用都设超时
  2. 上游超时 > 下游超时之和
  3. 超时后必须处理（重试/降级）

Go 超时设置方式：
  // HTTP Client
  client := &http.Client{Timeout: 3 * time.Second}

  // Context 超时
  ctx, cancel := context.WithTimeout(ctx, 2*time.Second)
  defer cancel()

  // DB 连接
  db.SetConnMaxLifetime(5 * time.Minute)

超时 + 重试策略：
  第1次：立即
  第2次：100ms
  第3次：200ms
  最多3次 总耗时 < 上游超时

  重试必须幂等！
  非幂等操作不能盲目重试
```

## 8. 灰度发布？

**回答：**

```
灰度 = 逐步放量验证

  ┌──────────────┬──────────────────────────┐
  │ 策略          │ 说明                     │
  ├──────────────┼──────────────────────────┤
  │ 金丝雀发布    │ 1%→5%→20%→100%          │
  │ 蓝绿部署      │ 新旧两套 切流量          │
  │ A/B 测试      │ 按特征分流 对比效果      │
  │ Feature Flag  │ 代码中开关 动态控制      │
  └──────────────┴──────────────────────────┘

金丝雀发布流程：
  V1 (100%)  → V1 (99%) + V2 (1%)
             → V1 (80%) + V2 (20%)
             → V1 (0%)  + V2 (100%)
  每步都监控指标 异常立刻回滚

K8s 灰度（Nginx Ingress 注解）：
  annotations:
    nginx.ingress.kubernetes.io/canary: "true"
    nginx.ingress.kubernetes.io/canary-weight: "10"
  # 10% 流量到新版本

回滚标准：
  错误率上升 > 0.1%
  P99 延迟增加 > 20%
  业务指标下降
```

## 9. 混沌工程？

**回答：**

```
Chaos Engineering：主动注入故障验证系统韧性

  Netflix 提出 → Chaos Monkey

  原则：
  1. 建立稳态假设（正常指标是什么）
  2. 引入真实世界事件（断网/杀进程/磁盘满）
  3. 在生产环境实验（非生产验证不够）
  4. 最小化爆炸半径

  常见实验：
  ┌──────────────┬──────────────────────────┐
  │ 故障类型      │ 注入方式                 │
  ├──────────────┼──────────────────────────┤
  │ 进程崩溃      │ 随机 kill 容器           │
  │ 网络延迟      │ tc netem delay 100ms     │
  │ 网络丢包      │ tc netem loss 10%        │
  │ CPU 满载      │ stress-ng --cpu 4        │
  │ 磁盘满        │ 写满磁盘                 │
  │ 依赖不可用    │ iptables 断开下游网络    │
  └──────────────┴──────────────────────────┘

  工具：
  Chaos Mesh（K8s 原生）
  Litmus Chaos
  Gremlin

  实施步骤：
  1. 先在测试环境验证
  2. 生产环境小范围
  3. 建立完善监控告警
  4. 有预案可快速停止
```

## 10. 高可用面试速答？

**回答：**

```
Q: 高可用核心思路？
A: 冗余消除单点 + 快速故障发现 + 自动恢复
   MTTR越短可用性越高

Q: 4个9怎么做到？
A: 多副本+自动切换+熔断降级
   年停机≤52分钟

Q: 熔断和降级区别？

Q: 限流用什么算法？
A: 令牌桶（允许突发）最常用
   分布式用Redis+Lua

Q: 异地多活最大挑战？
A: 数据一致性 跨机房延迟导致
   常用方案：按用户分片路由

Q: 超时设计原则？
A: 每个外部调用必须设超时
   上游超时>下游超时之和

Q: 灰度发布怎么做？
A: 金丝雀 1%→5%→20%→100%
   监控指标异常立刻回滚

Q: 混沌工程是什么？
A: 主动注入故障验证系统韧性
   在可控范围内模拟真实故障
```
