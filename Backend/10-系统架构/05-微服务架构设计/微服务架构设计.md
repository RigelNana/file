# 微服务架构设计

---

## 1. 微服务架构核心概念？

**回答：**

```
微服务 = 按业务域拆分的独立可部署服务

  单体 vs 微服务：
  ┌──────────┬────────────────┬────────────────┐
  │ 维度      │ 单体            │ 微服务          │
  ├──────────┼────────────────┼────────────────┤
  │ 部署      │ 一个包          │ 独立部署        │
  │ 开发      │ 一个仓库        │ 独立仓库        │
  │ 扩展      │ 整体扩展        │ 按服务扩展      │
  │ 技术栈    │ 统一            │ 可以不同        │
  │ 团队      │ 一个大团队      │ 小团队各负其责  │
  │ 故障      │ 一挂全挂        │ 故障隔离        │
  │ 复杂度    │ 代码复杂        │ 运维复杂        │
  └──────────┴────────────────┴────────────────┘

适用场景：
  团队 > 10人
  业务复杂度高
  需要独立扩展/部署
  有 DevOps 基础设施

不适用：
  初创团队（先单体跑通）
  简单业务
  没有运维能力
```

## 2. 服务拆分原则？

**回答：**

```
拆分依据：
  ┌──────────────┬──────────────────────────────┐
  │ 方法          │ 说明                         │
  ├──────────────┼──────────────────────────────┤
  │ 按业务域      │ 用户/订单/支付/库存          │
  │ 按变更频率    │ 频繁变化的独立出来            │
  │ 按团队边界    │ 康威定律 → 组织=架构          │
  │ 按扩展需求    │ 高并发模块独立扩展            │
  └──────────────┴──────────────────────────────┘

DDD 限界上下文指导拆分：
  ┌──────────┐ ┌──────────┐ ┌──────────┐
  │ 用户上下文│ │ 订单上下文│ │ 支付上下文│
  │ User     │ │ Order    │ │ Payment  │
  │ Address  │ │ OrderItem│ │ Transaction│
  └──────────┘ └──────────┘ └──────────┘
  各上下文独立建模 通过API/事件通信

拆分反模式：
  ✗ 拆得太细（纳米服务）→ 运维成本爆炸
  ✗ 共享数据库 → 服务间强耦合
  ✗ 循环依赖 → A调B B调A
  ✗ 没有自动化 → 手工运维跟不上

经验法则：
  一个服务 2-3 人维护
  一个服务一个数据库
  服务间只通过API通信
```

## 3. 服务间通信？

**回答：**

```
  ┌──────────┬──────────────┬──────────────┐
  │ 方式      │ 特点          │ 场景          │
  ├──────────┼──────────────┼──────────────┤
  │ HTTP/REST│ 简单通用      │ 对外API       │
  │ gRPC     │ 高性能 强类型 │ 内部通信      │
  │ 消息队列  │ 异步解耦      │ 事件驱动      │
  └──────────┴──────────────┴──────────────┘

同步 vs 异步：
  同步（RPC/HTTP）：
    + 简单 实时响应
    - 强耦合 级联故障
  
  异步（MQ/Event）：
    + 解耦 削峰
    - 复杂 最终一致性

gRPC 定义服务：
  syntax = "proto3";
  
  service UserService {
      rpc GetUser(GetUserRequest) returns (User);
      rpc ListUsers(ListUsersRequest) returns (stream User);
  }

通信模式选择：
  查询类 → 同步 gRPC/HTTP
  命令类（不需要结果）→ 异步 MQ
  事件通知 → 异步 Event
  数据同步 → CDC/Event Sourcing
```

## 4. 服务注册与发现？

**回答：**

```
  服务注册中心：
  ┌────────┐  注册   ┌──────────┐  发现   ┌────────┐
  │ServiceA│───────→│ Registry │←───────│ServiceB│
  └────────┘        └──────────┘        └────────┘
                    etcd/Consul/Nacos

  客户端发现 vs 服务端发现：
  客户端发现：
    Client → Registry 获取地址列表 → 自己负载均衡
    例：gRPC + etcd（Go常用）
  
  服务端发现：
    Client → LB/DNS → Service
    例：K8s Service（DNS解析）

K8s 中的服务发现：
  通过 Service + DNS 自动实现
  svc-name.namespace.svc.cluster.local
  不需要额外注册中心

Go + etcd 服务注册：
  // 注册：写key+租约
  lease, _ := client.Grant(ctx, 10)
  client.Put(ctx, "/services/user/node1", addr,
      clientv3.WithLease(lease.ID))
  client.KeepAlive(ctx, lease.ID)
  
  // 发现：Watch前缀
  resp, _ := client.Get(ctx, "/services/user/",
      clientv3.WithPrefix())
  wch := client.Watch(ctx, "/services/user/",
      clientv3.WithPrefix())
```

## 5. API 网关？

**回答：**

```
  ┌──────────┐              ┌─────────┐
  │ Client   │──→ API GW ──→│Services │
  └──────────┘    │         └─────────┘
                  │ 功能：
                  ├─ 路由转发
                  ├─ 认证鉴权
                  ├─ 限流熔断
                  ├─ 协议转换
                  ├─ 请求聚合
                  └─ 日志监控

  常用网关：
  ┌──────────────┬──────────────────────────┐
  │ 网关          │ 特点                     │
  ├──────────────┼──────────────────────────┤
  │ Kong          │ OpenResty 插件丰富       │
  │ APISIX        │ 国产 高性能              │
  │ Envoy         │ Service Mesh 数据面      │
  │ Traefik       │ Cloud Native 自动发现    │
  │ 自研          │ Go 实现 完全可控         │
  └──────────────┴──────────────────────────┘

BFF 模式（Backend For Frontend）：
  ┌──────┐   ┌──────────┐   ┌──────────┐
  │ Web  │──→│ Web BFF  │──→│          │
  └──────┘   └──────────┘   │ Services │
  ┌──────┐   ┌──────────┐   │          │
  │Mobile│──→│Mobile BFF│──→│          │
  └──────┘   └──────────┘   └──────────┘
  
  每种前端一个BFF
  各自聚合/裁剪所需数据
```

## 6. 分布式事务？

**回答：**

```
  微服务下一个操作跨多个服务/数据库

  ┌──────────────┬──────────────────────────┐
  │ 方案          │ 特点                     │
  ├──────────────┼──────────────────────────┤
  │ 2PC          │ 强一致 性能差 阻塞       │
  │ TCC          │ 强一致 侵入大            │
  │ Saga         │ 最终一致 补偿机制        │
  │ 本地消息表    │ 最终一致 简单可靠        │
  │ 事务消息      │ 最终一致 RocketMQ原生    │
  └──────────────┴──────────────────────────┘

  Saga 编排(Orchestration)：
  ┌──────────────┐
  │ Orchestrator │
  │  ┌───────┐   │
  │  │Step 1 │──→ 创建订单
  │  │Step 2 │──→ 扣库存
  │  │Step 3 │──→ 扣款
  │  └───────┘   │
  │  失败时      │
  │  补偿Step3   │
  │  补偿Step2   │
  │  补偿Step1   │
  └──────────────┘

最佳实践：
  能不用分布式事务就不用
  优先考虑最终一致性
  Saga 是微服务中最常用方案
  每步操作必须幂等
  补偿操作必须可靠
```

## 7. 服务治理？

**回答：**

```
  ┌──────────────┬──────────────────────────┐
  │ 治理能力      │ 实现方式                 │
  ├──────────────┼──────────────────────────┤
  │ 超时控制      │ Context + Deadline       │
  │ 重试策略      │ 指数退避 + 抖动          │
  │ 熔断器        │ gobreaker / 自研         │
  │ 限流          │ 令牌桶 / Redis + Lua     │
  │ 降级          │ 返回缓存/默认值          │
  │ 负载均衡      │ 加权轮询 / P2C           │
  │ 链路追踪      │ OpenTelemetry            │
  │ 日志聚合      │ ELK / Loki              │
  └──────────────┴──────────────────────────┘

Service Mesh：
  将治理能力下沉到基础设施层

  ┌──────────────────────────────────────┐
  │ Pod                                  │
  │ ┌──────────┐  ┌──────────────────┐  │
  │ │ App      │←→│ Sidecar (Envoy)  │  │
  │ └──────────┘  └──────────────────┘  │
  └──────────────────────────────────────┘
  
  Istio 架构：
  控制面（istiod）：配置下发、证书管理
  数据面（Envoy）：流量代理、观测

  优势：对应用透明 不侵入代码
  劣势：延迟增加 运维复杂
```

## 8. 微服务数据管理？

**回答：**

```
原则：Database per Service
  每个服务拥有自己的数据库
  服务间不共享数据库

  ┌──────────┐    ┌──────────┐    ┌──────────┐
  │ 用户服务  │    │ 订单服务  │    │ 支付服务  │
  └────┬─────┘    └────┬─────┘    └────┬─────┘
  ┌────┴─────┐    ┌────┴─────┐    ┌────┴─────┐
  │ User DB  │    │ Order DB │    │ Pay DB   │
  └──────────┘    └──────────┘    └──────────┘

跨服务数据查询：
  1. API 组合查询（BFF聚合）
  2. CQRS（维护读视图）
  3. 数据冗余（关键字段冗余到本地）

数据一致性方案：
  事件驱动 + 最终一致
  订单创建 → 发Event → 库存服务消费 → 扣库存

  ┌────────┐  Event  ┌────────┐  Event  ┌────────┐
  │ Order  │────────→│ MQ     │────────→│ Stock  │
  │ Service│        │        │        │ Service│
  └────────┘        └────────┘        └────────┘

  对账补偿：
  定期对比各服务数据
  发现不一致 → 触发修复
```

## 9. 微服务测试策略？

**回答：**

```
  测试金字塔：
       ┌───────┐
       │ E2E   │  少  端到端测试
      ┌┴───────┴┐
      │ 集成测试 │  中
     ┌┴─────────┴┐
     │ 契约测试   │  服务间接口
    ┌┴───────────┴┐
    │  单元测试    │  多  覆盖核心逻辑
    └─────────────┘

契约测试（Pact）：
  消费方定义期望 → 生成契约
  提供方验证契约 → 确保兼容
  接口改动自动发现

Go 集成测试（testcontainers）：
  func TestUserRepo(t *testing.T) {
      ctx := context.Background()
      // 启动真实 MySQL 容器
      container, _ := testcontainers.GenericContainer(ctx,
          testcontainers.GenericContainerRequest{
              ContainerRequest: testcontainers.ContainerRequest{
                  Image: "mysql:8.0",
                  ExposedPorts: []string{"3306/tcp"},
                  Env: map[string]string{
                      "MYSQL_ROOT_PASSWORD": "test",
                  },
              },
              Started: true,
          })
      defer container.Terminate(ctx)
      // 获取端口 连接测试
  }

微服务测试要点：
  单元测试 Mock 外部依赖
  集成测试用 testcontainers 启动真实依赖
  契约测试保证服务间兼容
  E2E 测试覆盖核心流程（少量）
```

## 10. 微服务面试速答？

**回答：**

```
Q: 微服务拆分依据？
A: 按业务域(DDD限界上下文)
   一个服务2-3人维护 独立数据库

Q: 内部通信用什么？
A: gRPC(同步 高性能 强类型)
   MQ(异步事件驱动 解耦)

Q: 服务发现怎么做？
A: K8s中用Service+DNS
   非K8s用etcd/Consul+客户端发现

Q: 微服务分布式事务？
A: 优先避免 → 最终一致Saga
   每步幂等 有补偿 有对账

Q: Service Mesh用来做什么？
A: 治理能力下沉到Sidecar
   应用无感知的超时/重试/熔断/观测

Q: 每服务一个数据库的好处？
A: 服务独立 互不影响
   可以选最适合的数据库

Q: API网关的作用？
A: 统一入口：认证/限流/路由/监控
   BFF模式为不同端裁剪API

Q: 微服务测试策略？
A: 单元测试为主 契约测试保兼容
   集成测试用容器 E2E少量核心
```
