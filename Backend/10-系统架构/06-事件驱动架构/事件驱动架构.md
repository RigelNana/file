# 事件驱动架构

---

## 1. 事件驱动架构核心概念？

**回答：**

```
EDA（Event-Driven Architecture）：
  系统组件通过事件进行异步通信

  ┌──────────┐  Event   ┌──────────┐  Event   ┌──────────┐
  │ Producer │────────→│ Broker   │────────→│ Consumer │
  │ 事件生产  │         │ 事件中介  │         │ 事件消费  │
  └──────────┘         └──────────┘         └──────────┘

  事件 vs 命令：
  ┌──────────┬──────────────────┬──────────────────┐
  │ 对比      │ 事件(Event)      │ 命令(Command)    │
  ├──────────┼──────────────────┼──────────────────┤
  │ 方向      │ 广播通知          │ 定向请求         │
  │ 语义      │ 已发生的事实      │ 期望执行的动作   │
  │ 耦合      │ 生产者不关心消费者│ 调用方依赖被调方 │
  │ 命名      │ 过去式 OrderCreated│ 祈使句 CreateOrder│
  └──────────┴──────────────────┴──────────────────┘

EDA 三种模式：
  1. 事件通知(Notification)：只通知，消费者自行查询
  2. 事件溯源(Event Sourcing)：事件即数据源
  3. 事件携带状态(Event-Carried State Transfer)：事件带完整数据
```

## 2. 事件溯源（Event Sourcing）？

**回答：**

```
传统方式：只保存最终状态
Event Sourcing：保存所有事件，状态由事件推算

  ┌──────────────────────────────────────┐
  │ Event Store                          │
  │ Event 1: AccountCreated(balance=0)   │
  │ Event 2: MoneyDeposited(amount=100)  │
  │ Event 3: MoneyWithdrawn(amount=30)   │
  │ Event 4: MoneyDeposited(amount=50)   │
  │                                      │
  │ 当前状态 = replay 所有事件            │
  │ balance = 0 + 100 - 30 + 50 = 120    │
  └──────────────────────────────────────┘

Go 实现：
  type Event struct {
      Type      string
      Data      interface{}
      Timestamp time.Time
      Version   int
  }
  
  type Account struct {
      ID      string
      Balance int
      Version int
      events  []Event
  }
  
  func (a *Account) Apply(e Event) {
      switch e.Type {
      case "MoneyDeposited":
          a.Balance += e.Data.(int)
      case "MoneyWithdrawn":
          a.Balance -= e.Data.(int)
      }
      a.Version = e.Version
  }

优势：
  完整审计日志
  可回溯到任意时刻的状态
  天然适合CQRS

劣势：
  事件数量多 需要快照优化
  事件schema演进复杂
  查询不方便 需要读视图
```

## 3. CQRS 模式？

**回答：**

```
CQRS = Command Query Responsibility Segregation
读写职责分离

  传统：同一模型处理读和写
  CQRS：读模型和写模型分离

  ┌────────────┐                        ┌────────────┐
  │ Command    │                        │ Query      │
  │ (写操作)    │                        │ (读操作)    │
  └──────┬─────┘                        └──────┬─────┘
         │                                     │
  ┌──────▼─────┐    Event/Sync    ┌────────────▼─┐
  │ Write DB   │─────────────────→│  Read DB     │
  │ (规范化)    │                  │  (反规范化)   │
  └────────────┘                  └──────────────┘

  Write Model：规范化 事务保证
  Read Model：反规范化 查询优化

同步方式：
  1. 同步双写（强一致 性能差）
  2. 事件驱动（最终一致 性能好）
  3. CDC（Change Data Capture）
     → Debezium 监听 binlog → 更新读库

Go 实现：
  // 写命令
  type CreateOrderCommand struct {
      UserID string
      Items  []OrderItem
  }
  
  // 读查询
  type OrderQuery struct {
      UserID string
      Status string
  }
  
  // 写处理器操作写库
  func (h *CommandHandler) CreateOrder(cmd CreateOrderCommand) error {
      order := NewOrder(cmd)
      if err := h.writeRepo.Save(order); err != nil {
          return err
      }
      h.eventBus.Publish(OrderCreatedEvent{...})
      return nil
  }
  
  // 读处理器查询读库
  func (h *QueryHandler) GetOrders(q OrderQuery) ([]OrderView, error) {
      return h.readRepo.FindByUser(q.UserID, q.Status)
  }
```

## 4. 事件设计原则？

**回答：**

```
事件命名：
  领域 + 实体 + 过去式动词
  order.created / payment.completed / user.registered

事件结构：
  type DomainEvent struct {
      ID        string    `json:"id"`         // 事件唯一ID
      Type      string    `json:"type"`       // 事件类型
      Source    string    `json:"source"`     // 来源服务
      Time      time.Time `json:"time"`       // 发生时间
      Data      any       `json:"data"`       // 事件数据
      Version   string    `json:"version"`    // Schema版本
  }

设计原则：
  ┌──────────────┬──────────────────────────────┐
  │ 原则          │ 说明                         │
  ├──────────────┼──────────────────────────────┤
  │ 不可变        │ 事件一旦发生不能修改         │
  │ 自描述        │ 包含足够理解事件的信息       │
  │ 有版本        │ schema演进需要版本控制       │
  │ 幂等消费      │ 消费者处理同一事件多次结果一样│
  │ 有序性        │ 同一聚合根的事件有序         │
  └──────────────┴──────────────────────────────┘

事件粒度：
  太粗：DataChanged → 消费者不知道改了什么
  太细：UserEmailCharacterAdded → 没意义
  合适：UserEmailUpdated → 清晰有意义
```

## 5. Outbox Pattern？

**回答：**

```
  问题：业务操作和发事件要原子性

  // 错误方式：不原子
  db.Save(order)     // 1. 保存成功
  mq.Publish(event)  // 2. 发送可能失败 → 数据不一致

  Outbox Pattern：
  ┌─────────────────────────────────┐
  │ 同一个数据库事务                 │
  │  1. INSERT INTO orders (...)    │
  │  2. INSERT INTO outbox (...)    │ 发件箱表
  └─────────────────────────────────┘
  
  独立进程/CDC 扫描 outbox 表 → 发送到 MQ

Go 实现：
  func CreateOrder(ctx context.Context, tx *sql.Tx, order Order) error {
      // 1. 保存订单
      _, err := tx.ExecContext(ctx,
          "INSERT INTO orders (id, user_id, total) VALUES (?, ?, ?)",
          order.ID, order.UserID, order.Total)
      if err != nil {
          return err
      }
      
      // 2. 同事务写outbox
      eventData, _ := json.Marshal(OrderCreatedEvent{
          OrderID: order.ID,
          UserID:  order.UserID,
      })
      _, err = tx.ExecContext(ctx,
          "INSERT INTO outbox (id, event_type, payload, status) VALUES (?, ?, ?, ?)",
          uuid.New(), "order.created", eventData, "pending")
      return err
  }

  Outbox 扫描器 / CDC(Debezium) 负责可靠投递
```

## 6. 事件流处理？

**回答：**

```
  批处理 vs 流处理：
  ┌──────────┬──────────────────────────┐
  │ 批处理    │ 定时处理一批数据（分钟/小时）│
  │ 流处理    │ 实时处理每条数据（毫秒级）  │
  └──────────┴──────────────────────────┘

  流处理架构：
  ┌──────────┐   ┌──────┐   ┌──────────┐   ┌──────┐
  │ 数据源    │──→│Kafka │──→│ 流处理    │──→│ 输出  │
  │ App/DB   │   │      │   │ Flink    │   │ DB   │
  └──────────┘   └──────┘   └──────────┘   └──────┘

  流处理常见操作：
  过滤(Filter)：只保留满足条件的事件
  映射(Map)：转换事件格式
  聚合(Aggregate)：窗口聚合（1分钟内的总额）
  Join：关联多个事件流

Go 简单流处理：
  // Pipeline 模式实现流处理
  func FilterEvents(in <-chan Event, pred func(Event) bool) <-chan Event {
      out := make(chan Event)
      go func() {
          defer close(out)
          for e := range in {
              if pred(e) {
                  out <- e
              }
          }
      }()
      return out
  }

  流处理工具：
  Apache Flink（最强大）
  Kafka Streams（轻量）
  Go：自研 Pipeline / Benthos
```

## 7. 事件总线设计？

**回答：**

```
  进程内事件总线：
  type EventBus struct {
      mu       sync.RWMutex
      handlers map[string][]EventHandler
  }
  
  type EventHandler func(ctx context.Context, event Event) error
  
  func (bus *EventBus) Subscribe(eventType string, handler EventHandler) {
      bus.mu.Lock()
      defer bus.mu.Unlock()
      bus.handlers[eventType] = append(bus.handlers[eventType], handler)
  }
  
  func (bus *EventBus) Publish(ctx context.Context, event Event) error {
      bus.mu.RLock()
      handlers := bus.handlers[event.Type]
      bus.mu.RUnlock()
      
      for _, h := range handlers {
          if err := h(ctx, event); err != nil {
              return err // 或记录错误继续
          }
      }
      return nil
  }

分布式事件总线：
  底层用 Kafka/RabbitMQ/NATS

  ┌──────────────────────────┐
  │ EventBus Interface       │
  │  Publish(event)          │
  │  Subscribe(type, handler)│
  └──────────────────────────┘
      │                │
  ┌───┴────┐     ┌─────┴────┐
  │ Kafka  │     │ Memory   │
  │ Impl   │     │ Impl     │
  └────────┘     └──────────┘
  
  生产用Kafka 测试用Memory
  通过接口切换
```

## 8. Saga 编排与编舞？

**回答：**

```
  Choreography（编舞）：无中心 各服务监听事件
  ┌─────┐ Event  ┌─────┐ Event  ┌─────┐
  │Order│───────→│Stock│───────→│Pay  │
  │     │←───────│     │←───────│     │
  └─────┘ 补偿   └─────┘ 补偿   └─────┘
  
  优：简单 去中心化
  劣：流程不直观 难调试

  Orchestration（编排）：有中心协调者
  ┌────────────────────────────┐
  │ Saga Orchestrator          │
  │  Step1: CreateOrder → OK   │
  │  Step2: DeductStock → OK   │
  │  Step3: DeductBalance → Fail│
  │  Compensate:               │
  │    RestoreStock             │
  │    CancelOrder              │
  └────────────────────────────┘
  
  优：流程清晰 易调试
  劣：协调者是单点

Go Saga 编排器：
  type SagaStep struct {
      Name       string
      Action     func(ctx context.Context) error
      Compensate func(ctx context.Context) error
  }
  
  func ExecuteSaga(ctx context.Context, steps []SagaStep) error {
      var completed []SagaStep
      for _, step := range steps {
          if err := step.Action(ctx); err != nil {
              // 逆序补偿
              for i := len(completed) - 1; i >= 0; i-- {
                  completed[i].Compensate(ctx)
              }
              return fmt.Errorf("step %s failed: %w", step.Name, err)
          }
          completed = append(completed, step)
      }
      return nil
  }
```

## 9. EDA 常见问题与解决方案？

**回答：**

```
  ┌────────────────┬──────────────────────────┐
  │ 问题            │ 解决方案                 │
  ├────────────────┼──────────────────────────┤
  │ 事件丢失        │ 持久化+ACK+重试          │
  │ 重复消费        │ 幂等性设计               │
  │ 事件顺序        │ 分区键保证局部有序       │
  │ 事件schema演进  │ 版本号+向后兼容          │
  │ 事件风暴        │ 限流+背压               │
  │ 调试困难        │ 分布式追踪+事件日志      │
  └────────────────┴──────────────────────────┘

幂等性实现：
  // 去重表
  func HandleEvent(ctx context.Context, event Event) error {
      // 检查是否已处理
      processed, _ := repo.IsProcessed(ctx, event.ID)
      if processed {
          return nil // 已处理 跳过
      }
      
      // 处理事件
      if err := processEvent(ctx, event); err != nil {
          return err
      }
      
      // 标记已处理
      return repo.MarkProcessed(ctx, event.ID)
  }

事件溯源 Schema 演进：
  v1: { "amount": 100 }
  v2: { "amount": 100, "currency": "CNY" }
  
  向上兼容：新增字段给默认值
  不兼容变更：新事件类型
```

## 10. 事件驱动面试速答？

**回答：**

```
Q: 事件和命令区别？
A: 事件=已发生的事实(过去式)
   命令=期望执行的动作(祈使句)

Q: Event Sourcing是什么？
A: 不存状态 只存事件
   当前状态由事件回放得出

Q: CQRS核心思想？
A: 读写模型分离
   写库规范化 读库反规范化

Q: Outbox Pattern解决什么？
A: 业务操作和发事件的原子性
   同一事务写业务表+outbox表

Q: Saga编排和编舞区别？
A: 编舞=各服务自行监听事件 无中心
   编排=中心协调者统一调度 流程清晰

Q: 事件重复消费怎么办？
A: 幂等性设计 通过事件ID去重
   处理前检查 处理后标记

Q: EDA的优缺点？
A: 优：解耦+异步+可扩展
   缺：最终一致+调试难+事件管理复杂

Q: 流处理和批处理区别？
A: 批处理=定时处理一批(分钟/小时级)
   流处理=实时逐条处理(毫秒级)
```
