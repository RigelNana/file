## 1. 系统设计面试方法论？

**回答：**

```
四步法（推荐 45分钟分配）：

  Step 1：需求分析（5分钟）
  ┌──────────────────────────────────┐
  │ 功能需求：核心功能是什么？       │
  │ 非功能需求：QPS/延迟/可用性？    │
  │ 约束条件：数据量/用户规模？      │
  │ 不做什么：明确边界               │
  └──────────────────────────────────┘

  Step 2：高层设计（10分钟）
  画出核心组件和交互
  Client → LB → Service → Cache → DB

  Step 3：详细设计（20分钟）
  深入核心组件
  数据模型/API设计/算法选择

  Step 4：扩展讨论（10分钟）
  扩展性/容错/监控

面试技巧：
  - 主动沟通 不闷头设计
  - 先问清需求再设计
  - 承认 trade-off 说出理由
  - 从简单方案出发 逐步优化
```

## 2. 短链接系统设计？

**回答：**

```
需求分析：
  功能：长URL→短URL、短URL→重定向
  QPS：写100/s 读10000/s
  短链长度：7位

核心设计：
  ┌────────┐    ┌──────┐    ┌───────┐    ┌──────┐
  │ Client │───→│ LB   │───→│Service│───→│Cache │
  └────────┘    └──────┘    └───────┘    └──┬───┘
                                            │miss
                                         ┌──▼───┐
                                         │ DB   │
                                         └──────┘

  短链生成方案：
  方案1：自增ID + Base62编码
    ID=1234567 → Base62 → "w7E"
    优：不冲突 有序
    缺：可预测
  
  方案2：Hash取前N位
    MD5(longURL)[:7]
    优：简单
    缺：可能冲突 需要检测

  推荐：Snowflake生成ID → Base62编码

数据模型：
  CREATE TABLE url_mapping (
      id         BIGINT PRIMARY KEY,
      short_code VARCHAR(10) UNIQUE,
      long_url   TEXT NOT NULL,
      created_at TIMESTAMP
  );

读优化：
  Redis 缓存（90%+ 命中）
  301 永久重定向（浏览器缓存）
  或 302 临时重定向（可统计点击）

写优化：
  Snowflake 生成ID（无DB竞争）
  分库分表（按 short_code Hash）
```

## 3. 秒杀系统设计？

**回答：**

```
核心挑战：
  瞬时高并发 库存不超卖

架构设计：
  ┌──────┐   ┌─────┐   ┌──────┐   ┌───────┐   ┌────┐
  │ CDN  │──→│ LB  │──→│限流层│──→│业务层 │──→│ MQ │
  └──────┘   └─────┘   └──────┘   └───────┘   └──┬─┘
  静态化      Nginx     令牌桶     库存预检      │
                                                ┌─▼──┐
                                                │DB  │
                                                └────┘

  分层拦截：
  1. 前端：按钮禁用+答题验证（挡掉部分流量）
  2. CDN/Nginx：静态资源分发
  3. 接入层：限流（令牌桶 每秒放N个）
  4. 服务层：Redis预减库存
  5. 队列层：MQ异步下单
  6. 数据层：DB最终扣减

Redis 预减库存（Lua 原子操作）：
  -- KEYS[1] = stock:item_id
  local stock = tonumber(redis.call('GET', KEYS[1]))
  if stock <= 0 then
      return -1  -- 售罄
  end
  redis.call('DECR', KEYS[1])
  return stock - 1

关键设计：
  库存预加载到 Redis
  Lua 脚本原子扣减
  成功后发 MQ 异步创建订单
  订单超时未支付 → 回滚库存
```

  │ 业务系统  │──→│ Push MQ  │──→│ Push Service │

  连接管理：
  type ConnManager struct {
      mu    sync.RWMutex

  // 用户连接到哪台Gateway → Redis记录
  // userID → gatewayID

  2. 通过内部 RPC 通知该 Gateway
  3. Gateway 通过长连接推送给用户
  4. 用户离线 → 存离线消息表 → 上线拉取

## 5. 分布式ID生成系统？

**回答：**

```
需求：全局唯一、有序、高性能

方案对比：
  ┌──────────────┬──────────┬────────┬──────────┐
  │ 方案          │ 有序     │ 性能   │ 可用性   │
  ├──────────────┼──────────┼────────┼──────────┤
  │ UUID          │ ✗        │ 高     │ 高       │
  │ DB自增        │ ✓        │ 低     │ 低       │
  │ Redis INCR   │ ✓        │ 中     │ 中       │
  │ Snowflake    │ ✓(趋势)  │ 高     │ 高       │
  │ Leaf(美团)   │ ✓        │ 高     │ 高       │
  └──────────────┴──────────┴────────┴──────────┘

  Snowflake 结构（64位）：
  +--+-------------------+----------+------------+
  |0 | 41位时间戳         | 10位机器 | 12位序列号  |
  +--+-------------------+----------+------------+
  
  支持：4096个/ms/机器 → 单机 400万/s

Go 实现核心：
  type Snowflake struct {
      mu        sync.Mutex
      timestamp int64
      workerID  int64
      sequence  int64
  }
  
  func (s *Snowflake) Generate() int64 {
      s.mu.Lock()
      defer s.mu.Unlock()
      
      now := time.Now().UnixMilli() - epoch
      if now == s.timestamp {
          s.sequence = (s.sequence + 1) & 0xFFF
          if s.sequence == 0 {
              for now <= s.timestamp {
                  now = time.Now().UnixMilli() - epoch
              }
          }
      } else {
          s.sequence = 0
      }
      s.timestamp = now
      return (now << 22) | (s.workerID << 12) | s.sequence
  }

时钟回拨问题：
  拒绝生成 等待时钟追上
  或预留3位给时钟回拨次数
```

## 6. 评论系统设计？

**回答：**

```
需求：帖子评论 + 楼中楼回复 + 排序

数据模型：
  -- 评论表
  CREATE TABLE comments (
      id         BIGINT PRIMARY KEY,
      content_id BIGINT NOT NULL,   -- 帖子ID
      parent_id  BIGINT DEFAULT 0,  -- 父评论ID（0=一级评论）
      root_id    BIGINT DEFAULT 0,  -- 根评论ID
      user_id    BIGINT NOT NULL,
      content    TEXT NOT NULL,
      like_count INT DEFAULT 0,
      created_at TIMESTAMP
  );
  
  INDEX: (content_id, root_id, created_at)

楼中楼实现：
  root_id 指向一级评论
  parent_id 指向直接回复的评论
  查询一级评论 → 展开时查 root_id 子评论

分页方案：
  时间排序：WHERE created_at < cursor ORDER BY created_at DESC
  热度排序：WHERE content_id = ? ORDER BY like_count DESC, id DESC

性能优化：
  评论计数缓存（Redis INCR）
  热帖评论缓存（前N条）
  点赞用 Redis Set（防重复）

写入优化：
  评论写 MQ 异步落库
  计数异步更新（Redis → 定期同步DB）
```

## 7. 限流系统设计？

**回答：**

```
需求：分布式限流 多维度 可配置

多维度限流：
  ┌──────────┬──────────────────────────┐
  │ 维度      │ 示例                     │
  ├──────────┼──────────────────────────┤
  │ 全局      │ 整个API 10000/s          │
  │ 用户      │ 每用户 100/min           │
  │ IP        │ 每IP 50/min             │
  │ 接口      │ /api/sms 10/min/user    │
  └──────────┴──────────────────────────┘

Redis + Lua 滑动窗口限流：
  -- KEYS[1] = rate_limit:{user}:{api}
  -- ARGV[1] = window (秒)
  -- ARGV[2] = limit
  -- ARGV[3] = now (毫秒)
  local key = KEYS[1]
  local window = tonumber(ARGV[1]) * 1000
  local limit = tonumber(ARGV[2])
  local now = tonumber(ARGV[3])
  
  -- 移除窗口外的数据
  redis.call('ZREMRANGEBYSCORE', key, 0, now - window)
  local count = redis.call('ZCARD', key)
  
  if count < limit then
      redis.call('ZADD', key, now, now .. math.random())
      redis.call('EXPIRE', key, ARGV[1])
      return 1  -- 允许
  end
  return 0  -- 拒绝

Go 限流中间件：
  func RateLimitMiddleware(rdb *redis.Client) gin.HandlerFunc {
      return func(c *gin.Context) {
          userID := c.GetString("userID")
          key := fmt.Sprintf("rate:%s:%s", userID, c.FullPath())
          allowed, _ := checkRateLimit(rdb, key, 60, 100)
          if !allowed {
              c.JSON(429, gin.H{"error": "too many requests"})
              c.Abort()
              return
          }
          c.Next()
      }
  }
```

## 8. Feed 流系统设计？

**回答：**

```
需求：关注的人发动态 看自己的 Timeline

两种模式：
  推模式(Push/Fan-out on write)：
    发动态时推到所有粉丝的收件箱
    ✓ 读取快
    ✗ 写放大（大V百万粉丝）

  拉模式(Pull/Fan-out on read)：
    读取时去关注人的发件箱拉取
    ✓ 写入快
    ✗ 读放大（关注1000人要查1000次）

  推拉结合：
  ┌──────────────────────────────────┐
  │ 普通用户（粉丝<1000）→ 推模式   │
  │ 大V（粉丝>10000）   → 拉模式   │
  │ 活跃用户            → 推        │
  │ 不活跃用户          → 不推      │
  └──────────────────────────────────┘

数据结构：
  发件箱：user:{uid}:outbox → Sorted Set (score=timestamp)
  收件箱：user:{uid}:inbox  → Sorted Set (score=timestamp)

推送时：
  func PublishFeed(userID string, feed Feed) {
      // 写入发件箱
      rdb.ZAdd(ctx, "outbox:"+userID, redis.Z{
          Score: float64(feed.Timestamp),
          Member: feed.ID,
      })
      // 获取粉丝列表
      followers := getFollowers(userID)
      // 推到粉丝收件箱（大V跳过）
      for _, fid := range followers {
          rdb.ZAdd(ctx, "inbox:"+fid, redis.Z{...})
      }
  }

读取时：
  Timeline = 收件箱内容 + 拉取关注大V的发件箱
  合并排序 → 返回
```

## 9. 配置中心设计？

**回答：**

```
需求：集中管理配置 实时推送变更

架构：
  ┌──────────┐    ┌──────────────┐    ┌──────────┐
  │ Admin UI │───→│Config Center │←───│ Client   │
  └──────────┘    │  (Server)    │    │ SDK      │
                  └──────┬───────┘    └──────────┘
                         │
                  ┌──────▼───────┐
                  │  存储         │
                  │ etcd/MySQL   │
                  └──────────────┘

  推送方式：
  1. 长轮询（Apollo 方案）
     Client 30s 长连接等待
     有变更立即返回 无变更超时后重连
  
  2. Watch（etcd 方案）
     etcd Watch key prefix
     有变更通过 gRPC stream 推送

  3. WebSocket
     实时双向通信

Go Client SDK：
  type ConfigClient struct {
      cache   sync.Map
      watchers map[string][]func(string, string)
  }
  
  func (c *ConfigClient) Get(key string) string {
      if v, ok := c.cache.Load(key); ok {
          return v.(string)
      }
      // 从 Server 拉取
      val := c.fetchFromServer(key)
      c.cache.Store(key, val)
      return val
  }
  
  func (c *ConfigClient) Watch(key string, cb func(old, new string)) {
      c.watchers[key] = append(c.watchers[key], cb)
  }

灰度发布：
  配置也需要灰度
  先推给1%实例 → 观察 → 全量
```

## 10. 系统设计面试速答？

**回答：**

```
Q: 系统设计面试怎么答？
A: 需求分析→高层设计→详细设计→扩展
   主动沟通 说清trade-off

Q: 短链系统核心设计？
A: Snowflake生成ID+Base62编码
   Redis缓存热点 分库分表

Q: 秒杀怎么防超卖？
A: Redis Lua原子预减库存
   分层拦截 MQ异步下单

Q: 分布式ID怎么生成？
A: Snowflake(41位时间+10位机器+12位序列)
   单机400万/s 注意时钟回拨

Q: Feed流怎么设计？
A: 推拉结合 普通用户推 大V拉
   收件箱(Redis SortedSet)

Q: 评论楼中楼怎么做？
A: root_id指一级评论 parent_id指回复
   查一级评论+展开子评论

Q: 限流怎么做？
A: 单机令牌桶(rate.Limiter)
   分布式Redis+Lua滑动窗口

Q: 配置中心推送方式？
A: 长轮询(Apollo) / Watch(etcd)
   客户端本地缓存+回调通知
```
