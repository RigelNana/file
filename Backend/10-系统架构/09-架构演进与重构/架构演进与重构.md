# 架构演进与重构

---

## 1. 架构演进路径？

**回答：**

```
典型演进路径：
  单体 → 垂直拆分 → SOA → 微服务 → 服务网格

  ┌────────────┐
  │ 单体应用    │  初创期 快速迭代
  └──────┬─────┘
         ↓
  ┌────────────┐
  │ 垂直拆分    │  按业务拆成几个独立应用
  └──────┬─────┘
         ↓
  ┌────────────┐
  │ SOA        │  服务化 ESB中间件
  └──────┬─────┘
         ↓
  ┌────────────┐
  │ 微服务      │  细粒度 独立部署
  └──────┬─────┘
         ↓
  ┌────────────┐
  │ 服务网格    │  治理下沉到基础设施
  └────────────┘

  关键原则：
  1. 业务驱动演进（不是技术驱动）
  2. 渐进式迁移（不大爆炸重写）
  3. 演进伴随基础设施升级
  4. 每次演进解决当前最痛的问题
```

## 2. Strangler Fig 模式？

**回答：**

```
  绞杀者模式：逐步替换旧系统

  ┌───────────────────────────────────────┐
  │ API Gateway / 反向代理                │
  │                                       │
  │ /api/users  → 新用户服务（已迁移）    │
  │ /api/orders → 新订单服务（已迁移）    │
  │ /api/pay    → 旧单体（待迁移）        │
  │ /api/stock  → 旧单体（待迁移）        │
  └───────────────────────────────────────┘

  步骤：
  1. 在旧系统前加代理层
  2. 新功能在新服务开发
  3. 逐个模块从旧系统迁出
  4. 最终旧系统完全退役

  Nginx 路由切换：
  upstream legacy { server old-monolith:8080; }
  upstream new_user { server user-service:8080; }
  
  location /api/users {
      proxy_pass http://new_user;  # 已迁移
  }
  location / {
      proxy_pass http://legacy;     # 未迁移
  }

  优势：
  - 风险可控（逐步切换）
  - 可随时回滚
  - 新旧并存 不影响业务
```

## 3. 技术债管理？

**回答：**

```
技术债 = 为了短期速度牺牲长期质量

  类型：
  ┌──────────────┬──────────────────────────┐
  │ 类型          │ 例子                     │
  ├──────────────┼──────────────────────────┤
  │ 有意识的      │ "先硬编码 以后改"       │
  │ 无意识的      │ 不知道有更好的设计       │
  │ 审慎的        │ 明知有债 评估后接受      │
  │ 鲁莽的        │ 不管了先跑起来           │
  └──────────────┴──────────────────────────┘

  技术债信号：
  - 改一个功能要动很多文件
  - 新人上手很困难
  - Bug修复引入新Bug
  - 部署频率越来越低
  - 团队抱怨代码质量

管理策略：
  1. 量化：建立技术债看板
  2. 优先级：影响开发效率的优先
  3. 预算：每个Sprint分配20%时间还债
  4. 渐进：Boy Scout Rule（每次让代码好一点）
  5. 防止新增：Code Review + CI检查

不要试图一次性还清！
  → 持续小步改进比大重构安全
```

## 4. 重构技巧？

**回答：**

```
Martin Fowler 重构原则：
  小步重构 每步可验证
  有测试保护的重构

  常用重构手法：
  ┌──────────────────┬──────────────────────┐
  │ 手法              │ 说明                 │
  ├──────────────────┼──────────────────────┤
  │ 提取函数          │ 长函数拆成小函数     │
  │ 提取接口          │ 抽取公共行为         │
  │ 移动方法          │ 方法放到合适的类     │
  │ 用多态替换条件    │ if/switch → 策略模式 │
  │ 引入参数对象      │ 多个参数 → 结构体   │
  │ 封装集合          │ 不暴露内部集合       │
  └──────────────────┴──────────────────────┘

Go 重构示例：
  // Before：巨大函数
  func ProcessOrder(order *Order) error {
      // 验证...50行
      // 计算价格...30行
      // 扣库存...20行
      // 发通知...20行
  }
  
  // After：职责分离
  func ProcessOrder(order *Order) error {
      if err := validateOrder(order); err != nil {
          return err
      }
      total := calculateTotal(order)
      if err := deductStock(order); err != nil {
          return err
      }
      return notifyUser(order, total)
  }

安全重构三步：
  1. 加测试（保护现有行为）
  2. 重构（小步改进）
  3. 验证（测试通过）
```

## 5. 单体拆分微服务？

**回答：**

```
  拆分步骤：
  1. 识别边界（DDD限界上下文）
  2. 解耦数据（每服务独立数据库）
  3. 抽取服务（从边缘模块开始）
  4. 切换流量（Strangler Fig）
  5. 废弃旧代码

  先拆什么？
  ┌──────────────┬──────────────────────────┐
  │ 优先拆        │ 原因                     │
  ├──────────────┼──────────────────────────┤
  │ 独立业务模块  │ 耦合少 风险低            │
  │ 高频变更模块  │ 部署解耦收益大           │
  │ 性能瓶颈模块  │ 独立扩展                 │
  └──────────────┴──────────────────────────┘

  数据拆分策略：
  Phase 1：代码分离 共享数据库（过渡）
  ┌──────┐ ┌──────┐
  │Svc A │ │Svc B │ 各自独立服务
  └──┬───┘ └──┬───┘
     └────┬────┘
     ┌────┴────┐
     │共享 DB  │     还在用同一个库
     └─────────┘

  Phase 2：数据分离 各自独立库
  ┌──────┐ ┌──────┐
  │Svc A │ │Svc B │
  └──┬───┘ └──┬───┘
  ┌──┴───┐ ┌──┴───┐
  │DB A  │ │DB B  │  独立数据库
  └──────┘ └──────┘

  数据同步：事件驱动 / CDC
```

## 6. 数据迁移策略？

**回答：**

```
双写迁移（最安全）：
  
  Phase 1：双写
  写入 → 旧库 + 新库（同时写）
  读取 → 旧库（主）
  
  Phase 2：对比验证
  定期对比新旧库数据一致性
  
  Phase 3：切读
  读取 → 新库（主）
  写入 → 新旧都写
  
  Phase 4：切完
  停止写旧库
  旧库下线

Go 双写示例：
  type DualWriteRepo struct {
      oldRepo OrderRepository
      newRepo OrderRepository
  }
  
  func (r *DualWriteRepo) Save(ctx context.Context, order *Order) error {
      // 先写旧库（主）
      if err := r.oldRepo.Save(ctx, order); err != nil {
          return err
      }
      // 异步写新库（允许失败 后续对账补偿）
      go func() {
          if err := r.newRepo.Save(context.Background(), order); err != nil {
              log.Error("dual write to new repo failed", "err", err)
          }
      }()
      return nil
  }

数据校验：
  定时任务对比两库数据
  发现不一致 → 以旧库为准修复
  连续多天一致 → 可以切换
```

## 7. 灰度迁移策略？

**回答：**

```
按用户灰度迁移：
  
  Phase 1：1% 内部用户 → 新服务
  Phase 2：5% 用户 → 新服务
  Phase 3：20% 用户 → 新服务
  Phase 4：100% 用户 → 新服务

流量分配方式：
  1. 按 UserID Hash 分流
  2. 按地域分流
  3. 按客户端版本分流
  4. Feature Flag 控制

Go Feature Flag 控制：
  type FeatureFlag struct {
      mu    sync.RWMutex
      flags map[string]int // feature → percentage
  }
  
  func (f *FeatureFlag) IsEnabled(feature, userID string) bool {
      f.mu.RLock()
      pct := f.flags[feature]
      f.mu.RUnlock()
      
      // 根据 userID 的 hash 决定
      h := fnv.New32a()
      h.Write([]byte(userID))
      return int(h.Sum32()%100) < pct
  }
  
  // 使用
  if ff.IsEnabled("new-order-service", userID) {
      return newOrderService.CreateOrder(ctx, req)
  }
  return legacyService.CreateOrder(ctx, req)

监控指标对比：
  新旧服务的 RT/错误率/业务指标
  异常立刻回滚
```

## 8. 架构决策记录（ADR）？

**回答：**

```
ADR = Architecture Decision Record
  记录架构决策的原因和上下文

格式：
  # ADR-001: 选择 PostgreSQL 作为主数据库
  
  ## 状态
  已采纳 | 2024-01-15
  
  ## 上下文
  需要选择主数据库。业务需要 JSON 查询、
  全文搜索、事务支持。团队有 MySQL 经验。
  
  ## 决定
  选择 PostgreSQL。
  
  ## 理由
  - 原生 JSON 支持
  - 丰富的索引类型（GiST, GIN）
  - 强一致性保证
  - 社区活跃 生态好
  
  ## 备选方案
  - MySQL：JSON 支持不如 PG
  - MongoDB：缺乏事务（旧版本）
  
  ## 后果
  - 团队需要学习 PostgreSQL
  - 需要搭建 PG 运维能力

存放位置：
  project/
  └── docs/
      └── adr/
          ├── 001-选择postgresql.md
          ├── 002-采用微服务架构.md
          └── README.md

好处：
  新人理解为什么这么设计
  避免反复讨论同一问题
  记录决策上下文 日后评估
```

## 9. 架构评审？

**回答：**

```
  架构评审检查清单：
  ┌──────────────┬──────────────────────────────┐
  │ 维度          │ 检查点                       │
  ├──────────────┼──────────────────────────────┤
  │ 功能性        │ 是否满足业务需求？           │
  │ 可用性        │ 单点？容灾？降级方案？       │
  │ 性能          │ 容量评估？瓶颈在哪？         │
  │ 可扩展性      │ 能水平扩展吗？               │
  │ 安全性        │ 认证？授权？数据加密？       │
  │ 可维护性      │ 分层清晰？职责明确？         │
  │ 可测试性      │ 如何测试？Mock方案？         │
  │ 成本          │ 机器/带宽/存储成本？         │
  │ 合规          │ 数据合规？隐私保护？         │
  └──────────────┴──────────────────────────────┘

技术方案模板：
  1. 背景与目标
  2. 需求分析
  3. 方案对比（≥2个方案）
  4. 详细设计（架构图/时序图/数据模型）
  5. 容量评估
  6. 风险评估与应对
  7. 里程碑排期
  8. 监控与告警

评审常见问题：
  Q: 你的方案和 XXX 比有什么优势？
  Q: 最大的风险是什么？怎么应对？
  Q: 这个设计能扛多大流量？
  Q: 数据一致性怎么保证？
  Q: 上线计划和回滚方案？
```

**回答：**

```
Q: 架构演进的关键原则？

Q: Strangler Fig模式？

   新旧并存 可随时回滚

Q: 技术债怎么管？
A: 量化→优先级→每Sprint留20%时间
   小步持续改进 不试图一次清完

Q: 单体拆微服务先拆什么？
A: 独立业务/高频变更/性能瓶颈优先
   从边缘开始 先代码分离后数据分离

Q: 数据迁移怎么做？

   以旧库为准 对账补偿

Q: 灰度迁移策略？

   监控对比新旧指标 异常回滚

Q: ADR是什么？
A: 架构决策记录
   记录决策的上下文/理由/备选方案

Q: 架构评审看什么？
```
