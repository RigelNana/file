# 设计模式八股文

## 一、设计原则

### 1. SOLID 原则分别是什么？

**答：** S-单一职责（一个类只做一件事）；O-开闭原则（对扩展开放，对修改关闭）；L-里氏替换（子类可替换父类）；I-接口隔离（客户端不依赖不需要的接口）；D-依赖倒置（依赖抽象不依赖具体）。

### 2. 组合优于继承怎么理解？

**答：** 继承是"is-a"关系，耦合度高，父类变化影响所有子类；组合是"has-a"关系，通过持有对象引用获得功能，更灵活。Go 语言没有继承，用组合（嵌入结构体）+ 接口实现多态，避免了菱形继承等问题。

## 二、创建型模式

### 3. 常用创建型模式有哪些？

**答：** 单例模式（全局唯一实例，Go 用 sync.Once）；工厂方法（定义创建接口，子类决定实例化哪个类）；抽象工厂（创建相关对象族）；建造者模式（分步构建复杂对象）；原型模式（克隆现有对象）。

### 4. Go 中如何实现单例？

**答：** 使用 `sync.Once` 保证线程安全的懒加载：`var once sync.Once; once.Do(func() { instance = &Singleton{} })`。不推荐 init() 饿汉式（无法延迟初始化）和双重检查锁（Go 有更优雅的 sync.Once）。

## 三、结构型模式

### 5. 常用结构型模式？

**答：** 适配器模式（转换接口兼容）；装饰器模式（动态添加功能，Go 用函数包装如 middleware）；代理模式（控制访问，如 RPC 代理、缓存代理）；外观模式（提供统一简单接口）；组合模式（树形结构）；桥接模式（抽象与实现分离）。

### 6. 装饰器和代理的区别？

**答：** 装饰器重在增强功能（增加新行为），可以多层嵌套；代理重在控制访问（权限校验、远程调用、缓存等），客户端感知不到代理的存在。Go 中 HTTP middleware 就是装饰器模式的典型应用。

## 四、行为型模式

### 7. 常用行为型模式？

**答：** 策略模式（封装算法族，运行时替换）；观察者模式（事件通知，一对多）；模板方法（固定骨架，子类实现具体步骤）；责任链模式（多个处理器依次处理）；状态模式（对象行为随状态变化）；命令模式（请求封装为对象）。

### 8. 策略模式和工厂模式区别？

**答：** 工厂模式关注对象创建（创建什么），策略模式关注行为选择（怎么做）。工厂返回对象，策略替换算法。常配合使用：工厂创建策略对象。

## 五、并发与微服务模式

### 9. Go 中常用并发模式？

**答：** Pipeline（管道链式处理）；Fan-Out/Fan-In（扇出并行、扇入聚合）；Worker Pool（固定 goroutine 池处理任务）；Context 取消传播；errgroup 并发错误收集；Pub/Sub 发布订阅。

### 10. 微服务中常用设计模式？

**答：** 服务发现（注册中心）；熔断器（Hystrix/gobreaker，防止级联故障）；限流器（令牌桶、滑动窗口）；重试+退避（指数退避+抖动）；Sidecar（如 Envoy 代理）；Saga（分布式事务编排）；CQRS（读写分离）；事件溯源（Event Sourcing）。

## 六、实践

### 11. 怎么避免过度设计？

**答：** KISS 原则（保持简单）；YAGNI（不需要就不做）；先写最简单的实现，重构时再引入模式；代码坏味道出现时再用模式解决。设计模式是工具不是目的。

### 12. Go 中最常用的设计模式？

**答：** 函数选项模式（Functional Options，构造复杂对象）；接口+组合（替代继承）；中间件模式（HTTP/gRPC 拦截链）；sync.Once 单例；策略模式（接口多态）；观察者/事件驱动。

### 13. 接口隔离在 Go 中怎么体现？

**答：** Go 推崇小接口：`io.Reader`（1个方法）、`io.Writer`（1个方法），按需组合成 `io.ReadWriter`。接受接口返回结构体，让调用方决定需要什么能力。

### 14. 如何判断该用哪种模式？

**答：** 根据问题场景：创建对象复杂→建造者/工厂；接口不兼容→适配器；动态增加功能→装饰器；多种算法可切换→策略；一对多通知→观察者；控制访问→代理。先识别问题再选模式，不要为用模式而用模式。
