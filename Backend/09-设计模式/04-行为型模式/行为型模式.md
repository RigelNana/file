# 行为型模式

---

## 1. 行为型模式有哪些？

**回答：**

```
行为型模式：定义对象之间的职责分配和通信

  ┌──────────┬────────────────────────────────┐
  │ 模式      │ 目的                           │
  ├──────────┼────────────────────────────────┤
  │ 策略      │ 封装算法族，运行时替换         │
  │ 观察者    │ 一对多事件通知                 │
  │ 模板方法  │ 固定骨架，子类实现步骤         │
  │ 责任链    │ 多个处理器依次处理             │
  │ 状态      │ 对象行为随状态变化             │
  │ 命令      │ 请求封装为对象                 │
  │ 迭代器    │ 统一遍历集合                   │
  │ 中介者    │ 减少多对多通信                 │
  │ 备忘录    │ 保存和恢复对象状态             │
  │ 访问者    │ 不修改类增加操作               │
  └──────────┴────────────────────────────────┘
```

## 2. 策略模式？

**回答：**

```
定义一系列算法，封装每一个并使它们可互换

  ┌──────────┐     ┌──────────┐
  │ Context  │────→│ Strategy │ (接口)
  └──────────┘     └────┬─────┘
                    ┌───┴───┐
                 ┌──┴──┐ ┌──┴──┐
                 │ 策A │ │ 策B │
                 └─────┘ └─────┘

场景：支付方式、排序算法、压缩算法、限流策略
```

```go
// 策略接口
type RateLimiter interface {
    Allow(key string) bool
}

// 策略1：固定窗口
type FixedWindowLimiter struct {
    limit    int
    window   time.Duration
    counters sync.Map
}

func (l *FixedWindowLimiter) Allow(key string) bool {
    now := time.Now().Unix() / int64(l.window.Seconds())
    windowKey := fmt.Sprintf("%s:%d", key, now)
    
    val, _ := l.counters.LoadOrStore(windowKey, new(int64))
    count := atomic.AddInt64(val.(*int64), 1)
    return count <= int64(l.limit)
}

// 策略2：令牌桶
type TokenBucketLimiter struct {
    rate     float64
    capacity int
    buckets  sync.Map
}

func (l *TokenBucketLimiter) Allow(key string) bool {
    // 令牌桶实现
    return true
}

// 使用策略
type APIServer struct {
    limiter RateLimiter  // 注入不同策略
}

func (s *APIServer) HandleRequest(key string) error {
    if !s.limiter.Allow(key) {
        return errors.New("rate limited")
    }
    return nil
}

// 运行时切换
server := &APIServer{limiter: &FixedWindowLimiter{limit: 100, window: time.Second}}
// 需要换策略：
server.limiter = &TokenBucketLimiter{rate: 10, capacity: 100}
```

## 3. 观察者模式？

**回答：**

```
一对多的依赖关系
当一个对象状态改变时自动通知所有依赖者

  ┌──────────┐     ┌──────────┐
  │ Subject  │────→│ Observer │ × N
  │ (发布者) │     │ (订阅者) │
  └──────────┘     └──────────┘

场景：事件系统、配置变更通知、UI 更新
```

```go
// 事件总线
type EventBus struct {
    mu       sync.RWMutex
    handlers map[string][]EventHandler
}

type Event struct {
    Type string
    Data interface{}
}

type EventHandler func(event Event)

func NewEventBus() *EventBus {
    return &EventBus{handlers: make(map[string][]EventHandler)}
}

func (b *EventBus) Subscribe(eventType string, handler EventHandler) {
    b.mu.Lock()
    defer b.mu.Unlock()
    b.handlers[eventType] = append(b.handlers[eventType], handler)
}

func (b *EventBus) Publish(event Event) {
    b.mu.RLock()
    handlers := make([]EventHandler, len(b.handlers[event.Type]))
    copy(handlers, b.handlers[event.Type])
    b.mu.RUnlock()
    
    for _, handler := range handlers {
        handler(event)
    }
}

// 使用
bus := NewEventBus()

// 订阅
bus.Subscribe("user.created", func(e Event) {
    user := e.Data.(*User)
    fmt.Printf("Send welcome email to %s\n", user.Email)
})

bus.Subscribe("user.created", func(e Event) {
    user := e.Data.(*User)
    fmt.Printf("Init wallet for %s\n", user.ID)
})

// 发布
bus.Publish(Event{Type: "user.created", Data: newUser})
```

```go
// Go channel 实现观察者
type Watcher struct {
    updates chan Config
}

type ConfigCenter struct {
    mu       sync.RWMutex
    config   Config
    watchers []*Watcher
}

func (c *ConfigCenter) Watch() *Watcher {
    w := &Watcher{updates: make(chan Config, 1)}
    c.mu.Lock()
    c.watchers = append(c.watchers, w)
    c.mu.Unlock()
    return w
}

func (c *ConfigCenter) Update(cfg Config) {
    c.mu.Lock()
    c.config = cfg
    watchers := make([]*Watcher, len(c.watchers))
    copy(watchers, c.watchers)
    c.mu.Unlock()
    
    for _, w := range watchers {
        select {
        case w.updates <- cfg:
        default: // 非阻塞
        }
    }
}
```

## 4. 责任链模式？

**回答：**

```
将请求沿处理器链传递，每个处理器决定是否处理或传递

  Request → [Handler A] → [Handler B] → [Handler C] → Response

场景：
  HTTP Middleware 链
  日志级别过滤
  审批流程
  输入验证
```

```go
// 通用责任链
type Handler interface {
    Handle(ctx context.Context, req *Request) (*Response, error)
    SetNext(handler Handler)
}

type BaseHandler struct {
    next Handler
}

func (h *BaseHandler) SetNext(handler Handler) {
    h.next = handler
}

func (h *BaseHandler) HandleNext(ctx context.Context, req *Request) (*Response, error) {
    if h.next != nil {
        return h.next.Handle(ctx, req)
    }
    return nil, errors.New("no handler")
}

// 认证处理器
type AuthHandler struct {
    BaseHandler
}

func (h *AuthHandler) Handle(ctx context.Context, req *Request) (*Response, error) {
    if req.Token == "" {
        return &Response{Code: 401, Msg: "unauthorized"}, nil
    }
    return h.HandleNext(ctx, req)
}

// 限流处理器
type RateLimitHandler struct {
    BaseHandler
    limiter RateLimiter
}

func (h *RateLimitHandler) Handle(ctx context.Context, req *Request) (*Response, error) {
    if !h.limiter.Allow(req.ClientIP) {
        return &Response{Code: 429, Msg: "too many requests"}, nil
    }
    return h.HandleNext(ctx, req)
}

// 业务处理器
type BusinessHandler struct {
    BaseHandler
}

func (h *BusinessHandler) Handle(ctx context.Context, req *Request) (*Response, error) {
    return &Response{Code: 200, Msg: "ok"}, nil
}

// 构建链
auth := &AuthHandler{}
rateLimit := &RateLimitHandler{limiter: limiter}
business := &BusinessHandler{}

auth.SetNext(rateLimit)
rateLimit.SetNext(business)

// 调用
resp, err := auth.Handle(ctx, req)
```

## 5. 模板方法模式？

**回答：**

```
定义算法骨架，将某些步骤延迟到子类实现

  Go 没有抽象方法
  用接口 + 组合实现
```

```go
// 定义步骤接口
type DataExporter interface {
    FetchData(ctx context.Context) ([]map[string]interface{}, error)
    Transform(data []map[string]interface{}) ([]byte, error)
    Write(data []byte) error
}

// 模板：固定流程
func Export(ctx context.Context, exporter DataExporter) error {
    // Step 1: 获取数据
    data, err := exporter.FetchData(ctx)
    if err != nil {
        return fmt.Errorf("fetch: %w", err)
    }
    
    // Step 2: 转换
    result, err := exporter.Transform(data)
    if err != nil {
        return fmt.Errorf("transform: %w", err)
    }
    
    // Step 3: 写出
    if err := exporter.Write(result); err != nil {
        return fmt.Errorf("write: %w", err)
    }
    
    return nil
}

// 具体实现：CSV导出
type CSVExporter struct {
    db       *sql.DB
    filePath string
}

func (e *CSVExporter) FetchData(ctx context.Context) ([]map[string]interface{}, error) {

    return nil, nil
}

func (e *CSVExporter) Transform(data []map[string]interface{}) ([]byte, error) {
    // 转为 CSV 格式
    return nil, nil
}

func (e *CSVExporter) Write(data []byte) error {
    return os.WriteFile(e.filePath, data, 0644)
}

// 具体实现：JSON导出到 S3
type S3JsonExporter struct {

}
// ... 实现接口方法
```

## 6. 状态模式？

**回答：**

```
对象行为随内部状态变化而变化

  场景：订单状态机、工作流、TCP连接状态

  ┌──────┐ 支付  ┌──────┐ 发货  ┌──────┐ 签收  ┌──────┐
  │待支付│──────→│已支付│──────→│已发货│──────→│已完成│
  └──┬───┘       └──┬───┘       └──┬───┘       └──────┘
     │ 取消          │ 退款         │ 退货
  ┌──┴───┐       ┌──┴───┐       ┌──┴───┐
  │已取消│       │已退款│       │退货中│
  └──────┘       └──────┘       └──────┘
```

```go
// 状态接口
type OrderState interface {
    Pay(order *Order) error
    Ship(order *Order) error
    Cancel(order *Order) error
    Name() string
}

type Order struct {
    ID    string
    State OrderState
}

func (o *Order) SetState(state OrderState) {
    log.Printf("Order %s: %s -> %s", o.ID, o.State.Name(), state.Name())
    o.State = state
}

// 具体状态：待支付
type PendingState struct{}
func (s *PendingState) Name() string { return "pending" }
func (s *PendingState) Pay(order *Order) error {
    // 执行支付逻辑
    order.SetState(&PaidState{})
    return nil
}
func (s *PendingState) Ship(order *Order) error {
    return errors.New("cannot ship unpaid order")
}
func (s *PendingState) Cancel(order *Order) error {
    order.SetState(&CancelledState{})
    return nil
}

// 具体状态：已支付
type PaidState struct{}
func (s *PaidState) Name() string { return "paid" }
func (s *PaidState) Pay(order *Order) error {
    return errors.New("already paid")
}
func (s *PaidState) Ship(order *Order) error {
    order.SetState(&ShippedState{})
    return nil
}
func (s *PaidState) Cancel(order *Order) error {
    // 退款逻辑
    order.SetState(&RefundedState{})
    return nil
}

// 使用
order := &Order{ID: "001", State: &PendingState{}}
order.State.Pay(order)   // pending → paid
order.State.Ship(order)  // paid → shipped
```

## 7. 命令模式？

**回答：**

```
将请求封装为对象，支持撤销/重做/队列化

场景：任务队列、Undo/Redo、宏命令
```

```go
type Command interface {
    Execute() error
    Undo() error
}

// 具体命令
type CreateFileCommand struct {
    path    string
    content []byte
}

func (c *CreateFileCommand) Execute() error {
    return os.WriteFile(c.path, c.content, 0644)
}

func (c *CreateFileCommand) Undo() error {
    return os.Remove(c.path)
}

// 命令管理器（支持撤销）
type CommandManager struct {
    history []Command
    pos     int
}

func (m *CommandManager) Execute(cmd Command) error {
    if err := cmd.Execute(); err != nil {
        return err
    }
    m.history = append(m.history[:m.pos], cmd)
    m.pos++
    return nil
}

func (m *CommandManager) Undo() error {
    if m.pos <= 0 {
        return errors.New("nothing to undo")
    }
    m.pos--
    return m.history[m.pos].Undo()
}

func (m *CommandManager) Redo() error {
    if m.pos >= len(m.history) {
        return errors.New("nothing to redo")
    }
    err := m.history[m.pos].Execute()
    m.pos++
    return err
}

// 宏命令：批量执行
type MacroCommand struct {
    commands []Command
}

func (c *MacroCommand) Execute() error {
    for _, cmd := range c.commands {
        if err := cmd.Execute(); err != nil {
            return err
        }
    }
    return nil
}

func (c *MacroCommand) Undo() error {
    for i := len(c.commands) - 1; i >= 0; i-- {
        if err := c.commands[i].Undo(); err != nil {
            return err
        }
    }
    return nil
}
```

## 8. 迭代器模式？

**回答：**

```
提供统一的方式遍历集合

  Go 中通常用 channel 或回调实现
  Go 1.23+ 支持 range over func
```

```go
// Channel 迭代器
type TreeNode struct {
    Value int
    Left  *TreeNode
    Right *TreeNode
}

// 中序遍历迭代器
func (t *TreeNode) InOrder() <-chan int {
    ch := make(chan int)
    go func() {
        defer close(ch)
        t.inOrder(ch)
    }()
    return ch
}

func (t *TreeNode) inOrder(ch chan<- int) {
    if t == nil {
        return
    }
    t.Left.inOrder(ch)
    ch <- t.Value
    t.Right.inOrder(ch)
}

// 使用
for val := range root.InOrder() {
    fmt.Println(val)
}

// 回调迭代器
type Collection struct {
    items []interface{}
}

func (c *Collection) ForEach(fn func(index int, item interface{}) bool) {
    for i, item := range c.items {
        if !fn(i, item) {
            break
        }
    }
}
```

## 9. 中介者模式？

**回答：**

```
用中介对象封装多对多交互

  不用中介：           用中介：
  A ←→ B              A → M ← B
  A ←→ C              C → M ← D
  B ←→ C
  B ←→ D              M = Mediator
  C ←→ D              (集中管理通信)

场景：聊天室、交易所撮合、事件调度
```

```go
// 中介者：聊天室
type ChatRoom struct {
    mu      sync.RWMutex
    members map[string]*Member
}

type Member struct {
    Name string
    room *ChatRoom
    msgs chan string
}

func NewChatRoom() *ChatRoom {
    return &ChatRoom{members: make(map[string]*Member)}
}

func (r *ChatRoom) Join(name string) *Member {
    m := &Member{Name: name, room: r, msgs: make(chan string, 100)}
    r.mu.Lock()
    r.members[name] = m
    r.mu.Unlock()
    return m
}

func (r *ChatRoom) Broadcast(from string, msg string) {
    r.mu.RLock()
    defer r.mu.RUnlock()
    
    for name, member := range r.members {
        if name != from {
            select {
            case member.msgs <- fmt.Sprintf("[%s]: %s", from, msg):
            default:
            }
        }
    }
}

func (m *Member) Send(msg string) {
    m.room.Broadcast(m.Name, msg)
}
```

## 10. 行为型模式面试速答？

**回答：**

```
Q: 策略和工厂区别？
A: 工厂关注创建什么
   策略关注怎么做（算法替换）

Q: 观察者模式 Go 怎么实现？
A: EventBus(map+回调)
   或 channel 通知

Q: 责任链的典型例子？
A: HTTP Middleware 链
   每层处理或传递

Q: 模板方法 Go 怎么实现？
A: 定义接口+独立函数作骨架
   不同实现完成具体步骤

Q: 状态模式和策略模式区别？
A: 状态模式自动转换状态
   策略模式外部指定策略

Q: 命令模式有什么好处？
A: 请求封装为对象
   支持撤销/重做/队列化

Q: 中介者模式解决什么？
A: 减少多对多耦合
   集中管理对象通信

Q: Go 中最常用的行为型模式？
A: 策略(接口多态)
   观察者(event/channel)
   责任链(middleware)
```
