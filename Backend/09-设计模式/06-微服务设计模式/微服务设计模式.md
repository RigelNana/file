## 2. API Gateway 模式？

**回答：**

```
统一入口，聚合路由/认证/限流

  ┌────────────┐
  │   Client   │
  └──────┬─────┘
         │
  ┌──────┴──────┐
  │ API Gateway │ ← 认证 / 限流 / 路由
  └──┬───┬───┬──┘
     │   │   │
  ┌──┴┐ ┌┴──┐┌┴──┐
  │SvcA│ │SvcB││SvcC│
  └───┘ └───┘└───┘

职责：
  路由分发
  认证授权
  限流熔断
  协议转换（HTTP ↔ gRPC）
  请求聚合
  日志监控

常见实现：
  Kong / APISIX / Envoy / Traefik
  自研（Go + net/http/httputil.ReverseProxy）
```

```go
// 简单 API Gateway 骨架
type Gateway struct {
    routes map[string]*httputil.ReverseProxy
}

func (g *Gateway) ServeHTTP(w http.ResponseWriter, r *http.Request) {
    // 1. 认证
    if err := authenticate(r); err != nil {
        http.Error(w, "unauthorized", 401)
        return
    }
    
    // 2. 限流
    if !rateLimiter.Allow(r.RemoteAddr) {
        http.Error(w, "too many requests", 429)
        return
    }
    
    // 3. 路由
    service := extractService(r.URL.Path)
    proxy, ok := g.routes[service]
    if !ok {
        http.Error(w, "not found", 404)
        return
    }
    
    proxy.ServeHTTP(w, r)
}
```

## 3. BFF（Backend For Frontend）模式？

**回答：**

```
为不同前端定制的后端

  ┌───────┐  ┌───────┐  ┌───────┐
  │ Web   │  │Mobile │  │ Mini  │
  └───┬───┘  └───┬───┘  └───┬───┘
      │          │          │
  ┌───┴───┐  ┌───┴───┐  ┌───┴───┐
  │Web BFF│  │ App   │  │ Mini  │
  │       │  │ BFF   │  │ BFF   │
  └───┬───┘  └───┬───┘  └───┴───┘
      │          │          │
  ┌───┴──────────┴──────────┴───┐
  │      微服务集群              │
  └─────────────────────────────┘

优势：
  不同端有不同的数据需求
  Web 需要完整数据，App 需要精简
  BFF 做数据裁剪和聚合
  避免通用 API 的过度获取

注意：
  BFF 不应包含业务逻辑
  只做数据编排和适配
```

## 4. 熔断器模式？

**回答：**

```
防止级联故障，快速失败

状态机：
  ┌──────┐  失败率>阈值  ┌──────┐
  │Closed│──────────────→│ Open │
  │(正常)│               │(熔断)│
  └──┬───┘               └──┬───┘
     ↑                      │ 超时后
     │   ┌─────────────┐    │
     │   │ Half-Open   │←───┘
     │   │(半开/试探)   │
     │   └──┬──────────┘
     │      │ 成功→关闭
     └──────┘ 失败→打开
```

```go
import "github.com/sony/gobreaker"

// 配置熔断器
cb := gobreaker.NewCircuitBreaker(gobreaker.Settings{
    Name:        "external-api",
    MaxRequests: 3,                    // Half-Open 状态允许的请求数
    Interval:    10 * time.Second,     // Closed 状态统计窗口
    Timeout:     30 * time.Second,     // Open → Half-Open 超时
    ReadyToTrip: func(counts gobreaker.Counts) bool {
        failureRatio := float64(counts.TotalFailures) / float64(counts.Requests)
        return counts.Requests >= 10 && failureRatio >= 0.5
    },
    OnStateChange: func(name string, from, to gobreaker.State) {
        log.Printf("breaker %s: %s → %s", name, from, to)
    },
})

// 使用
result, err := cb.Execute(func() (interface{}, error) {
    return callExternalAPI(ctx)
})
if err != nil {
    if errors.Is(err, gobreaker.ErrOpenState) {
        // 熔断中，走降级逻辑
        return fallback()
    }
    return nil, err
}
```

## 5. 重试与退避模式？

**回答：**

```
失败时自动重试，间隔递增

退避策略：
  固定间隔：1s, 1s, 1s
  指数退避：1s, 2s, 4s, 8s
  指数退避+抖动：1.2s, 2.5s, 3.8s（推荐）

  ⚠️ 无抖动 → 所有客户端同时重试 → 惊群效应
```

```go
type RetryConfig struct {
    MaxRetries  int
    BaseDelay   time.Duration
    MaxDelay    time.Duration
    Retryable   func(error) bool
}

func Retry(ctx context.Context, cfg RetryConfig, fn func() error) error {
    var lastErr error
    for i := 0; i <= cfg.MaxRetries; i++ {
        lastErr = fn()
        if lastErr == nil {
            return nil
        }
        
        // 检查是否可重试
        if cfg.Retryable != nil && !cfg.Retryable(lastErr) {
            return lastErr
        }
        
        if i == cfg.MaxRetries {
            break
        }
        
        // 指数退避 + 抖动
        delay := cfg.BaseDelay * time.Duration(1<<uint(i))
        if delay > cfg.MaxDelay {
            delay = cfg.MaxDelay
        }
        jitter := time.Duration(rand.Int63n(int64(delay) / 2))
        delay = delay/2 + jitter
        
        select {
        case <-time.After(delay):
        case <-ctx.Done():
            return ctx.Err()
        }
    }
    return fmt.Errorf("max retries exceeded: %w", lastErr)
}

// 使用
err := Retry(ctx, RetryConfig{
    MaxRetries: 3,
    BaseDelay:  100 * time.Millisecond,
    MaxDelay:   5 * time.Second,
    Retryable: func(err error) bool {
        // 只重试临时错误
        var netErr net.Error
        return errors.As(err, &netErr) && netErr.Timeout()
    },
}, func() error {
    return callService(ctx)
})
```

## 6. Sidecar 模式？

**回答：**

```
在主应用旁部署辅助容器

  ┌─────────────────────────┐ Pod
  │ ┌──────────┐ ┌────────┐ │
  │ │ App      │ │Sidecar │ │
  │ │ 容器     │ │ 容器   │ │
  │ │          │←→│        │ │
  │ └──────────┘ └────────┘ │
  └─────────────────────────┘

Sidecar 职责：
  日志收集（Fluentd/Filebeat）
  服务网格代理（Envoy/Istio）
  配置加载
  证书管理
  监控采集

优势：
  关注点分离
  不修改应用代码
  语言无关
  独立升级
```

## 7. 服务发现模式？

**回答：**

```
客户端发现 vs 服务端发现

客户端发现：
  ┌──────────┐  查询  ┌──────────┐
  │ Client   │──────→│ Registry │
  └────┬─────┘       └──────────┘
       │ 直连               ↑ 注册
  ┌────┴─────┐       ┌──────┴───┐
  │ Service  │       │ Service  │
  └──────────┘       └──────────┘

服务端发现：
  ┌──────────┐       ┌───────────┐  转发  ┌─────────┐
  │ Client   │──────→│Load Balancer│──────→│ Service │
  └──────────┘       └───────────┘       └─────────┘

注册中心选型：
  ┌──────────┬──────┬──────┬───────┐
  │          │ etcd │Consul│ Nacos │
  ├──────────┼──────┼──────┼───────┤
  │ 一致性   │ CP   │ CP   │ AP/CP │
  │ 健康检查 │ TTL  │ 多种 │ 多种  │
  │ 语言     │ Go   │ Go   │ Java  │
  └──────────┴──────┴──────┴───────┘
```

## 8. 限流降级模式？

**回答：**

```go
// 令牌桶限流
import "golang.org/x/time/rate"

type RateLimitMiddleware struct {
    limiters sync.Map
    rate     rate.Limit
    burst    int
}

func (m *RateLimitMiddleware) getLimiter(key string) *rate.Limiter {
    if l, ok := m.limiters.Load(key); ok {
        return l.(*rate.Limiter)
    }
    l := rate.NewLimiter(m.rate, m.burst)
    m.limiters.Store(key, l)
    return l
}

func (m *RateLimitMiddleware) Handle(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        limiter := m.getLimiter(r.RemoteAddr)
        if !limiter.Allow() {
            http.Error(w, "rate limited", http.StatusTooManyRequests)
            return
        }
        next.ServeHTTP(w, r)
    })
}

// 降级策略
type DegradeService struct {
    primary   Service
    fallback  Service
    breaker   *gobreaker.CircuitBreaker
}

func (s *DegradeService) Query(ctx context.Context, id string) (*Result, error) {
    result, err := s.breaker.Execute(func() (interface{}, error) {
        return s.primary.Query(ctx, id)
    })
    
    if err != nil {
        // 降级到备用服务/缓存/默认值
        log.Printf("primary failed, using fallback: %v", err)
        return s.fallback.Query(ctx, id)
    }
    
    return result.(*Result), nil
}
```

## 9. CQRS 与 Event Sourcing？

**回答：**

```
CQRS：命令查询职责分离

  ┌──────┐  Write  ┌───────────┐  同步  ┌───────────┐
  │Client│────────→│Write Model│──────→│Read Model │
  │      │←────────│  (MySQL)  │       │ (ES/Redis)│
  │      │  Read   └───────────┘       └───────────┘
  └──────┘                     ↑ Event  ↑
                               └────────┘

Event Sourcing：用事件记录状态变化

  传统：只存当前状态
    users 表: {name: "Bob", balance: 100}

  事件溯源：存所有事件
    Event 1: UserCreated {name: "Bob"}
    Event 2: BalanceAdded {amount: 200}
    Event 3: BalanceDeducted {amount: 100}
    → 重放事件得到当前状态
```

```go
// 简化的 CQRS
type CommandHandler struct {
    writeDB  *sql.DB
    eventBus EventBus
}

func (h *CommandHandler) CreateOrder(ctx context.Context, cmd CreateOrderCmd) error {
    order := &Order{ID: uuid.New(), Items: cmd.Items, Total: cmd.Total}
    
    // 写入写库
    if err := h.writeDB.Save(ctx, order); err != nil {
        return err
    }
    
    // 发布事件 → 更新读模型
    return h.eventBus.Publish(Event{
        Type: "order.created",
        Data: order,
    })
}

type QueryHandler struct {
    readDB *elasticsearch.Client  // 读优化的存储
}

func (h *QueryHandler) SearchOrders(ctx context.Context, query string) ([]Order, error) {
    // 从读库查询
    return h.readDB.Search(ctx, query)
}
```

## 10. 微服务设计模式面试速答？

**回答：**

```
Q: API Gateway 有什么用？
A: 统一入口 路由/认证/限流/监控
   Kong/APISIX/Envoy

Q: BFF 模式是什么？
A: 为不同前端定制后端
   数据裁剪和聚合 不含业务逻辑

Q: 熔断器的三个状态？
A: Closed(正常) → Open(熔断)
   → Half-Open(试探) → Closed

Q: 重试为什么要加抖动？
A: 指数退避避免同时重试
   +随机抖动防惊群效应

Q: Sidecar 模式的好处？
A: 关注点分离 语言无关
   不改应用代码 独立升级

Q: 服务发现两种方式？
A: 客户端发现(直连)
   服务端发现(LB转发)

Q: CQRS 是什么？
A: 读写分离 写模型优化写
   读模型优化查 事件同步

Q: Event Sourcing？
A: 存事件而非当前状态
   重放事件得到状态
   可审计 可回溯
```
