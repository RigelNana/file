# 反模式与代码坏味道

---

## 1. 什么是反模式？

**回答：**

```
反模式：看似合理但实际有害的设计方案

  ┌──────────────────┬──────────────────────────┐
  │ 反模式            │ 问题                     │
  ├──────────────────┼──────────────────────────┤
  │ God Object       │ 一个类承担所有职责       │
  │ Spaghetti Code   │ 代码缠绕无结构           │
  │ Golden Hammer    │ 到处用同一个工具/模式     │
  │ Lava Flow        │ 无人敢删的废弃代码       │
  │ Copy-Paste       │ 复制粘贴代替复用         │
  │ Premature Opt    │ 过早优化                 │
  │ Over-Engineering │ 过度设计                 │
  │ Magic Numbers    │ 硬编码的数字/字符串       │
  └──────────────────┴──────────────────────────┘
```


## 3. 常见代码坏味道？

**回答：**

```
Martin Fowler 《重构》中的坏味道：

1. 过长函数：一个函数 > 50 行
2. 过长参数列表：> 3 个参数
3. 重复代码：相同逻辑出现多处
4. 发散式变化：一个类因多种原因修改
5. 霰弹式修改：一个变化需要改多个类
6. 特性依恋：函数大量使用其他类的数据
7. 数据泥团：总是一起出现的数据
8. 基本类型偏执：用基本类型代替小对象
9. 过多的注释：注释掩盖了糟糕的代码
10. 死代码：永远不会执行的代码
```

```go
// 坏味道1：过长参数列表
func CreateUser(name, email, phone, address, city, state, zip string, age int) error { }
// 修复：使用结构体
type CreateUserReq struct {
    Name    string
    Email   string
    Phone   string
    Address Address
    Age     int
}
func CreateUser(req CreateUserReq) error { }

// 坏味道2：基本类型偏执
func Transfer(from string, to string, amount float64, currency string) {}
// 修复：值对象
type AccountID string
type Money struct {
    Amount   decimal.Decimal
    Currency string
}
func Transfer(from, to AccountID, money Money) error { return nil }

// 坏味道3：数据泥团
func SendNotification(email, phone, pushToken string) {}
func UpdateContact(email, phone, pushToken string) {}
// 修复：提取结构体
type ContactInfo struct {
    Email     string
    Phone     string
    PushToken string
}
```

## 4. Go 中常见反模式？

**回答：**

```go
// 1. init() 滥用
func init() {
    db = connectDB()        // 副作用
    cache = connectRedis()  // 难以测试
    loadConfig()            // 隐式依赖
}
// 修复：显式初始化，构造函数注入

// 2. 忽略错误
result, _ := doSomething()  // ❌
// 修复：处理每个错误
result, err := doSomething()
if err != nil {
    return fmt.Errorf("doSomething: %w", err)
}

// 3. 裸 goroutine
go func() {
    // 没有 recover，panic 会崩溃整个进程
    process()
}()
// 修复：安全的 goroutine
go func() {
    defer func() {
        if r := recover(); r != nil {
            log.Printf("panic recovered: %v", r)
        }
    }()
    process()
}()

// 4. 过度使用 interface{}
func Process(data interface{}) interface{} { }
// 修复：使用泛型或具体类型
func Process[T any](data T) T { }

// 5. 锁粒度过大
type Service struct {
    mu    sync.Mutex
    users map[string]*User
    cache map[string][]byte
}
func (s *Service) GetUser(id string) *User {
    s.mu.Lock()         // 锁住了所有操作 ❌
    defer s.mu.Unlock()
    return s.users[id]
}
// 修复：细粒度锁或 sync.Map
```

## 5. 过度设计的表现？

**回答：**

```
过度设计的信号：

  1. 用了设计模式但只有一种实现
     → 策略模式只有一个策略
  
  2. 建了抽象层但只有一个具体类
     → 接口只有一个实现且以后也不会有
  
  3. 预测未来需求
     → "以后可能会用到"（YAGNI 违反）
  
  4. 深层嵌套的继承/组合
     → 3 层以上的装饰器
  
  5. 过度 DRY
     → 为消除 2 行重复建了 5 个文件
  
  6. 过度的工厂/构建器
     → 简单对象用了 Builder + Factory + Pool

判断标准：
  当前是否有 2 个以上的实现？
  这个抽象是否简化了代码？
  删除它会不会更简单？
  
  "能用简单 if-else 解决的别用设计模式"
```

## 6. 分布式系统中的反模式？

**回答：**

```
1. 分布式单体：
   拆了微服务但紧耦合
   改一个服务要改一堆
   部署要全部一起发布

2. 共享数据库：
   多个服务共用一个数据库
   schema 变更互相影响
   → 每个服务独立数据库

3. 同步链式调用：
   A → B → C → D → E
   链条越长延迟越高、可靠性越低
   → 异步事件驱动

4. 过细的微服务：
   "Nano Service"
   一个函数拆一个服务
   → 运维成本 > 收益

5. 缺少超时：
   调用下游没有超时
   一个服务阻塞导致级联阻塞
   → 所有外部调用设超时

6. 重试风暴：
   全部客户端同时重试
   → 指数退避 + 抖动 + 熔断
```

## 7. 错误处理反模式？

**回答：**

```go
// 1. 吞掉错误
func process() {
    result, err := fetch()
    _ = err // ❌ 错误被吞掉
    use(result)
}

// 2. 错误信息不够
if err != nil {
    return errors.New("failed") // ❌ 没有上下文
}
// 修复
if err != nil {
    return fmt.Errorf("fetch user %s: %w", userID, err)
}

// 3. 过度包装
if err != nil {
    return fmt.Errorf("layer1: %w",
        fmt.Errorf("layer2: %w",
            fmt.Errorf("layer3: %w", err))) // ❌ 嵌套过深
}

// 4. panic 当错误用
func GetUser(id string) *User {
    user, err := db.Find(id)
    if err != nil {
        panic(err) // ❌ 不应该 panic
    }
    return user
}

// 5. 错误类型混乱
var (
    ErrNotFound     = errors.New("not found")
    ErrNotExist     = errors.New("not exist")    // ❌ 语义重复
    ErrNotFoundUser = errors.New("user not found") // ❌ 和 ErrNotFound 重复
)
// 修复：统一错误定义
var ErrNotFound = errors.New("not found")
// 使用时带上下文：fmt.Errorf("user %s: %w", id, ErrNotFound)
```

## 8. 并发编程反模式？

**回答：**

```go
// 1. goroutine 泄露
func leaky() {
    ch := make(chan int)
    go func() {
        val := <-ch // 永远阻塞，goroutine 泄露 ❌
        fmt.Println(val)
    }()
    // ch 从未写入，函数返回了
}

// 2. 在锁内做 IO
func (s *Service) GetData(id string) ([]byte, error) {
    s.mu.Lock()
    defer s.mu.Unlock()
    return http.Get("http://api/" + id) // ❌ 锁内做网络请求
}

// 3. 数据竞争
var counter int
for i := 0; i < 100; i++ {
    go func() {
        counter++ // ❌ 多个 goroutine 写
    }()
}
// 修复：atomic.AddInt64 或 sync.Mutex

// 4. channel 误用
ch := make(chan int, 1000000) // ❌ 巨大缓冲区掩盖问题
// 修复：合理大小，配合 context 控制

// 5. 忘记 WaitGroup
for _, item := range items {
    go process(item)
}
// 函数返回了但 goroutine 还在跑 ❌
// 修复：wg.Add → go func{ defer wg.Done() } → wg.Wait
```

## 9. 性能相关反模式？

**回答：**

```go
// 1. 不必要的内存分配
func concat(strs []string) string {
    result := ""
    for _, s := range strs {
        result += s // ❌ 每次创建新字符串
    }
    return result
}
// 修复：strings.Builder
func concat(strs []string) string {
    var b strings.Builder
    for _, s := range strs {
        b.WriteString(s)
    }
    return b.String()
}

    orders, _ := db.Query("SELECT * FROM orders WHERE user_id = ?", user.ID) // ❌

// 4. 不用 sync.Pool 复用对象
```

## 10. 反模式面试速答？

**回答：**

```
Q: God Object 是什么？
A: 一个类承担所有职责
   应拆分为多个单一职责的类

Q: Go 中最常见的反模式？
A: 忽略错误、裸goroutine
   init()滥用、锁粒度过大

Q: 过度设计怎么判断？
A: 只有一种实现却建抽象
   预测未来需求(YAGNI)
   删掉会不会更简单

Q: 分布式单体是什么？
A: 拆了微服务但紧耦合
   改一个要改一堆 一起部署

Q: N+1 查询问题？
A: 循环中逐条查关联数据
   批量查询IN或JOIN解决

Q: goroutine 泄露怎么发现？
A: pprof查goroutine数量
   确保channel有写入或ctx取消

Q: 代码坏味道有哪些？
A: 过长函数/参数列表
   重复代码 死代码
   基本类型偏执

Q: 怎么避免反模式？
A: Code Review + SOLID原则
   KISS/YAGNI + 适度重构
```
