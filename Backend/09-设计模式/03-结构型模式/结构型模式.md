# 结构型模式

---

## 1. 结构型模式有哪些？

**回答：**

```
结构型模式：组合类和对象以获得更大的结构

  ┌──────────┬────────────────────────────────┐
  │ 模式      │ 目的                           │
  ├──────────┼────────────────────────────────┤
  │ 适配器    │ 转换接口使不兼容的类协作       │
  │ 装饰器    │ 动态添加功能                   │
  │ 代理      │ 控制对象访问                   │
  │ 外观      │ 提供统一简单接口               │
  │ 组合      │ 树形结构表示部分-整体           │
  │ 桥接      │ 分离抽象与实现                 │
  │ 享元      │ 共享细粒度对象减少内存开销     │
  └──────────┴────────────────────────────────┘
```

## 2. 适配器模式？

**回答：**

```
将一个接口转换为客户端期望的另一个接口

  ┌──────────┐     ┌──────────┐     ┌──────────┐
  │ Client   │────→│ Adapter  │────→│ Adaptee  │
  │          │     │ (转换层) │     │ (被适配) │
  └──────────┘     └──────────┘     └──────────┘

场景：
  - 集成第三方库（接口不匹配）
  - 旧系统改造
  - 统一多个不同接口
```

```go
// 目标接口
type MessageSender interface {
    Send(ctx context.Context, to string, content string) error
}

// 被适配者1：某短信 SDK
type TencentSMS struct{}
func (s *TencentSMS) SendSMS(phoneNumber, templateID string, params []string) error {
    fmt.Printf("Tencent SMS to %s\n", phoneNumber)
    return nil
}

// 被适配者2：某邮件 SDK  
type SendGridEmail struct{}
func (e *SendGridEmail) SendMail(from, to, subject, body string) error {
    fmt.Printf("SendGrid email to %s\n", to)
    return nil
}

// 适配器1：短信
type SMSAdapter struct {
    client     *TencentSMS
    templateID string
}

func (a *SMSAdapter) Send(ctx context.Context, to string, content string) error {
    return a.client.SendSMS(to, a.templateID, []string{content})
}

// 适配器2：邮件
type EmailAdapter struct {
    client *SendGridEmail
    from   string
}

func (a *EmailAdapter) Send(ctx context.Context, to string, content string) error {
    return a.client.SendMail(a.from, to, "Notification", content)
}

// 统一使用
func Notify(sender MessageSender, to, content string) error {
    return sender.Send(context.Background(), to, content)
}
```

## 3. 装饰器模式？

**回答：**

```
动态地给对象添加额外的功能

  不改变原有代码，层层包装
  
  ┌────────────────────────────────┐
  │ Logging                       │
  │  ┌──────────────────────────┐ │
  │  │ Metrics                  │ │
  │  │  ┌────────────────────┐  │ │
  │  │  │ Caching            │  │ │
  │  │  │  ┌──────────────┐  │  │ │
  │  │  │  │ Core Service  │  │  │ │
  │  │  │  └──────────────┘  │  │ │
  │  │  └────────────────────┘  │ │
  │  └──────────────────────────┘ │
  └────────────────────────────────┘

Go 惯用方式：HTTP Middleware
```

```go
// 通用装饰器
type Service interface {
    Do(ctx context.Context, req Request) (Response, error)
}

type CoreService struct{}
func (s *CoreService) Do(ctx context.Context, req Request) (Response, error) {
    return Response{Data: "result"}, nil
}

// 日志装饰器
type LoggingDecorator struct {
    next   Service
    logger *log.Logger
}

func (d *LoggingDecorator) Do(ctx context.Context, req Request) (Response, error) {
    d.logger.Printf("request: %+v", req)
    resp, err := d.next.Do(ctx, req)
    d.logger.Printf("response: %+v, error: %v", resp, err)
    return resp, err
}

// 指标装饰器
type MetricsDecorator struct {
    next    Service
    counter int64
}

func (d *MetricsDecorator) Do(ctx context.Context, req Request) (Response, error) {
    start := time.Now()
    resp, err := d.next.Do(ctx, req)
    atomic.AddInt64(&d.counter, 1)
    log.Printf("duration: %v", time.Since(start))
    return resp, err
}

// 组合使用
var svc Service = &CoreService{}
svc = &LoggingDecorator{next: svc, logger: log.Default()}
svc = &MetricsDecorator{next: svc}
resp, err := svc.Do(ctx, req) // Metrics → Logging → Core
```

```go
// HTTP Middleware（装饰器的经典应用）
type Middleware func(http.Handler) http.Handler

func LoggingMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        start := time.Now()
        next.ServeHTTP(w, r)
        log.Printf("%s %s %v", r.Method, r.URL.Path, time.Since(start))
    })
}

func AuthMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        token := r.Header.Get("Authorization")
        if token == "" {
            http.Error(w, "unauthorized", 401)
            return
        }
        next.ServeHTTP(w, r)
    })
}

// 链式包装
handler := AuthMiddleware(LoggingMiddleware(myHandler))
```

## 4. 代理模式？

**回答：**

```
为对象提供代理以控制访问

  ┌──────────┐     ┌──────────┐     ┌──────────┐
  │ Client   │────→│  Proxy   │────→│  Real    │
  │          │     │          │     │  Subject │
  └──────────┘     └──────────┘     └──────────┘

代理类型：
  远程代理：RPC/gRPC 客户端
  缓存代理：查缓存再查原始
  保护代理：权限校验
  延迟代理：延迟初始化
```

```go
// 缓存代理
type UserRepository interface {
    GetByID(ctx context.Context, id string) (*User, error)
}

type MySQLUserRepo struct{ db *sql.DB }
func (r *MySQLUserRepo) GetByID(ctx context.Context, id string) (*User, error) {
    // 查数据库
    return &User{ID: id}, nil
}

// 缓存代理
type CachedUserRepo struct {
    repo  UserRepository
    cache *redis.Client
    ttl   time.Duration
}

func (r *CachedUserRepo) GetByID(ctx context.Context, id string) (*User, error) {
    // 1. 查缓存
    key := "user:" + id
    data, err := r.cache.Get(ctx, key).Bytes()
    if err == nil {
        var user User
        json.Unmarshal(data, &user)
        return &user, nil
    }
    
    // 2. 缓存未命中，查 DB
    user, err := r.repo.GetByID(ctx, id)
    if err != nil {
        return nil, err
    }
    
    // 3. 写缓存
    data, _ = json.Marshal(user)
    r.cache.Set(ctx, key, data, r.ttl)
    
    return user, nil
}

// 使用方无感知
var repo UserRepository = &CachedUserRepo{
    repo:  &MySQLUserRepo{db: db},
    cache: redisClient,
    ttl:   5 * time.Minute,
}
```

## 5. 外观模式？

**回答：**

```
为复杂子系统提供统一的简单接口

  ┌──────────┐
  │  Client  │
  └────┬─────┘
       │
  ┌────┴─────┐  Facade
  │ 统一接口 │
  └┬───┬───┬─┘
   │   │   │
  ┌┴┐ ┌┴┐ ┌┴┐  子系统
  │A│ │B│ │C│
  └─┘ └─┘ └─┘
```

```go
// 子系统
type UserService struct{}
func (s *UserService) CreateUser(ctx context.Context, name, email string) (string, error) {
    return "user-123", nil
}

type WalletService struct{}
func (s *WalletService) CreateWallet(ctx context.Context, userID string) error {
    return nil
}

type NotifyService struct{}
func (s *NotifyService) SendWelcome(ctx context.Context, userID, email string) error {
    return nil
}

// 外观：注册流程
type RegistrationFacade struct {
    userSvc   *UserService
    walletSvc *WalletService
    notifySvc *NotifyService
}

func (f *RegistrationFacade) Register(ctx context.Context, name, email string) error {
    // 统一编排复杂流程
    userID, err := f.userSvc.CreateUser(ctx, name, email)
    if err != nil {
        return fmt.Errorf("create user: %w", err)
    }
    
    if err := f.walletSvc.CreateWallet(ctx, userID); err != nil {
        return fmt.Errorf("create wallet: %w", err)
    }
    
    if err := f.notifySvc.SendWelcome(ctx, userID, email); err != nil {
        log.Printf("send welcome failed: %v", err) // 非关键步骤不阻断
    }
    
    return nil
}

// 调用方只需：
facade.Register(ctx, "Bob", "bob@example.com")
```

## 6. 组合模式？

**回答：**

```
树形结构表示"部分-整体"关系

  场景：文件系统、菜单、组织架构

  ┌────────┐
  │ Company│
  ├────┬───┘
  │    ├── Department A
  │    │   ├── Team 1
  │    │   └── Team 2
  │    └── Department B
  │        └── Team 3
```

```go
type Component interface {
    Name() string
    Size() int64
}

// 叶子：文件
type File struct {
    name string
    size int64
}

func (f *File) Name() string { return f.name }
func (f *File) Size() int64  { return f.size }

// 组合：目录
type Directory struct {
    name     string
    children []Component
}

func (d *Directory) Name() string { return d.name }
func (d *Directory) Size() int64 {
    var total int64
    for _, child := range d.children {
        total += child.Size()
    }
    return total
}

func (d *Directory) Add(c Component) {
    d.children = append(d.children, c)
}

// 使用
root := &Directory{name: "root"}
src := &Directory{name: "src"}
src.Add(&File{name: "main.go", size: 1024})
src.Add(&File{name: "util.go", size: 512})
root.Add(src)
root.Add(&File{name: "README.md", size: 256})

fmt.Println(root.Size()) // 1792
```

## 7. 桥接模式？

**回答：**

```
将抽象与实现分离，使两者独立变化

  ┌───────────┐       ┌───────────┐
  │ Abstraction│──────→│Implementor│
  └─────┬─────┘       └─────┬─────┘
     ┌──┴──┐            ┌───┴───┐
  ┌──┴──┐┌─┴─┐     ┌───┴──┐┌───┴──┐
  │具体A ││具B│     │实现X ││实现Y │
  └─────┘└───┘     └──────┘└──────┘

  抽象维度 × 实现维度 → 桥接连接

场景：通知(邮件/短信) × 紧急程度(普通/紧急)
```

```go
// 实现维度：发送方式
type Sender interface {
    Send(ctx context.Context, to, content string) error
}

type EmailSender struct{}
func (s *EmailSender) Send(ctx context.Context, to, content string) error {
    fmt.Printf("Email to %s: %s\n", to, content)
    return nil
}

type SMSSender struct{}
func (s *SMSSender) Send(ctx context.Context, to, content string) error {
    fmt.Printf("SMS to %s: %s\n", to, content)
    return nil
}

// 抽象维度：通知级别
type Notification struct {
    sender Sender  // 桥接
}

type NormalNotification struct {
    Notification
}

func (n *NormalNotification) Notify(ctx context.Context, to, msg string) error {
    return n.sender.Send(ctx, to, "[Normal] "+msg)
}

type UrgentNotification struct {
    Notification
}

func (n *UrgentNotification) Notify(ctx context.Context, to, msg string) error {
    // 紧急通知可能还有额外逻辑（重复发送等）
    return n.sender.Send(ctx, to, "[URGENT] "+msg)
}

// 自由组合
normalEmail := &NormalNotification{Notification{sender: &EmailSender{}}}
urgentSMS := &UrgentNotification{Notification{sender: &SMSSender{}}}
```

## 8. 享元模式？

**回答：**

```
共享细粒度对象以减少内存开销

  将对象状态分为：
  - 内部状态（共享，不变的）
  - 外部状态（不共享，变化的）

场景：大量相似对象（棋子、粒子、字符渲染）
```

```go
// 享元：共享的棋子类型
type ChessPiece struct {
    Name  string // 内部状态（共享）
    Color string
    Icon  string
}

// 享元工厂
type ChessFactory struct {
    pieces map[string]*ChessPiece
}

func NewChessFactory() *ChessFactory {
    f := &ChessFactory{pieces: make(map[string]*ChessPiece)}
    f.pieces["black_king"] = &ChessPiece{Name: "King", Color: "Black", Icon: "♚"}
    f.pieces["white_king"] = &ChessPiece{Name: "King", Color: "White", Icon: "♔"}
    // ... 其他棋子
    return f
}

func (f *ChessFactory) GetPiece(key string) *ChessPiece {
    return f.pieces[key]
}

// 使用：外部状态（位置）不共享
type ChessPosition struct {
    Piece *ChessPiece // 共享的享元
    X, Y  int         // 外部状态
}

factory := NewChessFactory()
board := []ChessPosition{
    {Piece: factory.GetPiece("black_king"), X: 4, Y: 7},
    {Piece: factory.GetPiece("white_king"), X: 4, Y: 0},
}
// 相同类型的棋子共享同一个 ChessPiece 对象
```

## 9. 装饰器、代理、适配器对比？

**回答：**

```
  ┌──────────┬────────────┬────────────┬────────────┐
  │          │ 适配器     │ 装饰器     │ 代理       │
  ├──────────┼────────────┼────────────┼────────────┤
  │ 目的     │ 接口转换   │ 功能增强   │ 控制访问   │
  │ 接口     │ 不同→相同  │ 相同       │ 相同       │
  │ 嵌套     │ 通常单层   │ 可多层     │ 通常单层   │
  │ 被包装者 │ 已有的     │ 通用的     │ 特定的     │
  │ 关注点   │ 兼容性     │ 扩展性     │ 安全/性能  │
  ├──────────┼────────────┼────────────┼────────────┤
  │ Go例子   │ 第三方SDK  │ middleware │ 缓存代理   │
  │          │ 包装       │ 链         │ RPC stub   │
  └──────────┴────────────┴────────────┴────────────┘
```

## 10. 结构型模式面试速答？

**回答：**

```
Q: 适配器模式什么时候用？
A: 接口不兼容时做转换
   集成第三方SDK最常见

Q: 装饰器在 Go 中怎么实现？
A: HTTP Middleware 就是装饰器
   层层包装 Handler

Q: 代理和装饰器区别？
A: 代理控制访问(缓存/权限)
   装饰器增强功能(日志/监控)

Q: 外观模式解决什么？
A: 简化复杂子系统的调用
   提供统一入口

Q: 组合模式适用什么场景？
A: 树形结构(文件系统/菜单)
   部分和整体一致对待

Q: 桥接模式的好处？
A: 抽象和实现独立变化
   避免类爆炸(M×N → M+N)

Q: 享元模式的核心？
A: 共享不变的内部状态
   外部状态由客户端传入
   减少内存开销
```
