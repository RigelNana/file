# 数据库八股文

## 一、MySQL 基础与架构

### 1. MySQL 整体架构？

**答：** MySQL 采用分层架构：连接层（连接管理、认证授权）→ 服务层（SQL 解析、优化器、查询缓存）→ 存储引擎层（InnoDB/MyISAM，可插拔）→ 文件系统层。一条 SQL 的执行流程：客户端连接 → 查询缓存（8.0已移除）→ 解析器语法分析 → 优化器生成执行计划 → 执行器调用存储引擎接口 → 返回结果。

### 2. InnoDB 与 MyISAM 区别？

**答：** InnoDB 支持事务（ACID）、行级锁、外键、MVCC、崩溃恢复（redo log），聚簇索引组织数据；MyISAM 不支持事务，仅表级锁，适合读多写少场景，支持全文索引（InnoDB 5.6+也支持）。生产环境几乎都用 InnoDB。

## 二、索引

### 3. B+ 树索引原理？

**答：** InnoDB 使用 B+ 树作为索引结构：非叶子节点只存 key 用于导航，叶子节点存完整数据（聚簇）或主键值（二级索引），叶子节点间用双向链表连接便于范围查询。相比 B 树：更矮（同样数据扇出更大，IO 更少）、叶子链表支持范围扫描、查询稳定（都要到叶子节点）。

### 4. 聚簇索引与非聚簇索引？

**答：** 聚簇索引（主键索引）：叶子节点存完整行数据，一张表只有一个，数据按主键顺序物理存储。非聚簇索引（二级索引）：叶子节点存主键值，查询需要回表（先查二级索引得到主键，再查聚簇索引获取数据）。覆盖索引：查询字段都在索引中，无需回表。

### 5. 索引失效场景？

**答：** 联合索引不满足最左前缀、对索引列做函数/计算、隐式类型转换（如 varchar 与 int 比较）、LIKE 以 `%` 开头、OR 条件中有非索引列、NOT IN/NOT EXISTS（部分场景）、数据量小时优化器选择全表扫描。用 EXPLAIN 分析执行计划确认是否走索引。

## 三、事务

### 6. 事务 ACID 特性？

**答：** A（原子性）：事务要么全部成功要么全部回滚，通过 undo log 实现。C（一致性）：事务前后数据满足约束，是目的。I（隔离性）：并发事务互不干扰，通过 MVCC + 锁实现。D（持久性）：提交后数据不丢失，通过 redo log + WAL 实现。

### 7. 隔离级别？

**答：** 读未提交（脏读）→ 读已提交（RC，不可重复读）→ 可重复读（RR，InnoDB 默认，MVCC 快照读 + Next-Key Lock 防幻读）→ 串行化（最高隔离，性能最差）。InnoDB 的 RR 级别通过 MVCC（快照读）和 Next-Key Lock（当前读）基本解决了幻读问题。

## 四、锁

### 8. InnoDB 的锁类型？

**答：** 行锁：Record Lock（记录锁）、Gap Lock（间隙锁，锁定间隙防插入）、Next-Key Lock（记录锁+间隙锁，左开右闭）。表锁：意向锁（IS/IX，行锁前自动加，用于快速判断表中是否有行锁）、MDL 锁（元数据锁，DDL/DML 互斥）、Auto-Inc 锁（自增主键）。

### 9. MVCC 原理？

**答：** 多版本并发控制：每行数据有隐藏列 trx_id（最近修改事务ID）和 roll_pointer（指向 undo log 版本链）。事务开始时生成 ReadView（包含活跃事务ID列表、最小/最大事务ID），读取时沿版本链找到对自己可见的版本。RC 每次 SELECT 生成新 ReadView；RR 只在第一次 SELECT 生成，后续复用。

## 五、日志与复制

### 10. redo log、undo log、binlog 的区别？

**答：** redo log（InnoDB 引擎层）：物理日志，记录数据页修改，保证持久性，循环写入固定大小文件，用于崩溃恢复。undo log（InnoDB 引擎层）：逻辑日志，记录修改前数据，用于事务回滚和 MVCC。binlog（Server 层）：逻辑日志（SQL 语句或行变更），用于主从复制和数据恢复，追加写入。

### 11. 主从复制原理？

**答：** 主库写 binlog → 从库 IO 线程拉取 binlog 写入 relay log → 从库 SQL 线程重放 relay log。复制方式：异步复制（默认，可能丢数据）、半同步复制（至少一个从库确认收到 relay log）、组复制（Paxos 协议，强一致）。延迟原因：网络延迟、从库单线程重放（5.6+支持多线程）。

## 六、SQL 优化

### 12. 慢查询优化思路？

**答：** 开启慢查询日志定位 → EXPLAIN 分析执行计划（关注 type、key、rows、Extra）→ 优化索引（添加/调整索引）→ 优化 SQL（避免 SELECT *、减少子查询改 JOIN）→ 分页优化（延迟关联/游标分页）→ 表结构优化（合适的数据类型、垂直/水平拆分）。

### 13. 分库分表？

**答：** 垂直分库：按业务拆分不同数据库。垂直分表：将大字段拆到扩展表。水平分库分表：按分片键（如用户ID hash）拆分到多个库/表。中间件：ShardingSphere、Vitess。问题：跨分片 JOIN、分布式事务、全局 ID（雪花算法）、数据迁移、扩容。

### 14. MySQL 和 Redis 数据一致性？

**答：** 常用方案：Cache-Aside（先更新 DB 再删缓存）+ 延迟双删 + 消息队列保证最终一致性。不推荐：先更新缓存再更新 DB（并发问题）。读取：先查缓存，未命中查 DB 并写缓存。删除缓存而非更新缓存（避免并发写覆盖）。强一致性需求用分布式锁或 Binlog 订阅（Canal）同步。
