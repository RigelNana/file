# 索引原理与优化

---

## 1. B+ 树为什么适合做数据库索引？

**回答：**

```
B+ 树 vs 其他数据结构：

  ┌──────────────┬──────────────────────────────┐
  │ 数据结构      │ 不适合的原因                  │
  ├──────────────┼──────────────────────────────┤
  │ 哈希表        │ 不支持范围查询和排序           │
  │ 二叉搜索树    │ 退化为链表 O(n)               │
  │ AVL/红黑树    │ 树高大，IO 次数多             │
  │ B 树         │ 非叶子也存数据，扇出小         │
  │ 跳表          │ 随机指针，不利于磁盘 IO        │
  └──────────────┴──────────────────────────────┘

B+ 树优势：
  ┌──────────── B+ 树 ────────────┐
  │      [30 | 60]                │ ← 非叶子：只存 key
  │     /    |     \              │    扇出大(~1000)
  │   [10|20] [40|50] [70|80]    │    树高 3-4 层可存
  │    ↓↓      ↓↓      ↓↓       │    千万级数据
  │  [数据]  [数据]   [数据]      │ ← 叶子：存完整数据
  │    ← → ← → ← →              │    双向链表连接
  └──────────────────────────────┘    支持范围查询

关键特点：
  1. 矮胖树：3-4层可存几千万数据（IO 3-4次）
  2. 非叶子只存 key：单个节点可存更多 key
  3. 叶子链表：WHERE id > 100 范围扫描高效
  4. 查询稳定：所有查询都要到叶子 O(logN)
```

## 2. 聚簇索引和二级索引的区别？

**回答：**

```
  聚簇索引（主键索引）：
  ┌──────────────────────────────┐
  │      [主键30 | 主键60]        │
  │     /          \             │
  │   [10|20]    [40|50]         │
  │     ↓               ↓       │
  │  ┌────────┐  ┌────────┐     │
  │  │id=10   │  │id=40   │     │
  │  │name=张三│  │name=王五│     │
  │  │age=25  │  │age=30  │     │
  │  │完整行.. │  │完整行.. │     │
  │  └────────┘  └────────┘     │
  └──────────────────────────────┘
  叶子节点存完整行数据

  二级索引（name 索引）：
  ┌──────────────────────────────┐
  │      [李四 | 王五]            │
  │     /        \               │
  │  ┌──────┐ ┌──────┐          │
  │  │张三→10│ │王五→40│          │
  │  │李四→20│ │赵六→50│          │
  │  └──────┘ └──────┘          │
  └──────────────────────────────┘
  叶子节点存 (索引值, 主键ID)

  回表查询：
  SELECT * FROM users WHERE name = '张三'
  1. 查二级索引(name) → 得到主键 id=10
  2. 查聚簇索引(id=10) → 得到完整行  ← 回表

  覆盖索引（避免回表）：
  SELECT id, name FROM users WHERE name = '张三'
  索引 (name) 中已有 id(主键) → 直接返回，无需回表
  EXPLAIN 中 Extra: Using index
```

## 3. 联合索引与最左前缀原则？

**回答：**

```
联合索引 (a, b, c)：
  B+ 树按 a 排序 → a 相同按 b 排序 → b 相同按 c 排序

  ┌──────────────┬──────┬──────┬──────┐
  │ 查询条件      │ a    │ b    │ c    │
  ├──────────────┼──────┼──────┼──────┤
  │ a=1          │ ✅   │      │      │
  │ a=1 AND b=2  │ ✅   │ ✅   │      │
  │ a=1 AND b=2  │ ✅   │ ✅   │ ✅   │
  │ AND c=3      │      │      │      │
  │ b=2          │ ❌   │      │      │
  │ b=2 AND c=3  │ ❌   │      │      │
  │ a=1 AND c=3  │ ✅   │      │ ❌   │
  │ a>1 AND b=2  │ ✅   │ ❌   │      │
  └──────────────┴──────┴──────┴──────┘

  最左前缀原则：
  - 查询从联合索引最左列开始连续匹配
  - 遇到范围查询(> < BETWEEN LIKE)后面的列索引失效
  - 等值条件顺序无关（优化器会调整）

  联合索引设计原则：
  1. 高频查询条件放最左
  2. 区分度高的列放前面
  3. 尽量覆盖查询字段（覆盖索引）
```

## 4. 索引失效的场景有哪些？

**回答：**

```
索引失效场景：

  1. 违反最左前缀：
     INDEX(a,b) → WHERE b=1 ❌

  2. 对索引列使用函数/计算：
     WHERE YEAR(create_time) = 2024 ❌
     → WHERE create_time >= '2024-01-01'
       AND create_time < '2025-01-01' ✅

  3. 隐式类型转换：
     phone VARCHAR(20)
     WHERE phone = 13800138000 ❌ (字符串和数字比较)
     WHERE phone = '13800138000' ✅

  4. LIKE 左模糊：
     WHERE name LIKE '%张' ❌
     WHERE name LIKE '张%' ✅

  5. OR 条件中有非索引列：
     WHERE a=1 OR b=2 (b无索引) ❌
     → 两列都有索引可能用 index_merge

  6. NOT IN / NOT EXISTS：
     部分情况优化器放弃索引

  7. 优化器判断全表更快：
     小表或数据量少时

  8. 字符集不匹配：
     JOIN 时两表字符集不同 → 隐式转换
```

## 5. 如何用 EXPLAIN 分析执行计划？

**回答：**

```
EXPLAIN SELECT * FROM users WHERE name = '张三';

关键字段解读：
  ┌──────────────┬──────────────────────────────┐
  │ 字段          │ 说明                          │
  ├──────────────┼──────────────────────────────┤
  │ id           │ 查询序号（相同=同层，大的先执行）│
  │ select_type  │ SIMPLE/PRIMARY/SUBQUERY/UNION │
  │ table        │ 访问的表                       │
  │ type ★       │ 访问类型（性能排序见下）        │
  │ possible_keys│ 可能使用的索引                 │
  │ key ★        │ 实际使用的索引                 │
  │ key_len      │ 索引使用的字节数               │
  │ ref          │ 索引关联的列                   │
  │ rows ★       │ 预估扫描行数                   │
  │ filtered     │ 过滤百分比                     │
  │ Extra ★      │ 额外信息                       │
  └──────────────┴──────────────────────────────┘

type 性能排序（好→差）：
  system > const > eq_ref > ref > range
  > index > ALL
  
  const: 主键/唯一索引等值
  eq_ref: JOIN 时主键/唯一索引
  ref: 非唯一索引等值
  range: 索引范围查询
  index: 全索引扫描
  ALL: 全表扫描 ← 需要优化

Extra 重要值：
  Using index → 覆盖索引（好）
  Using where → Server 层过滤
  Using temporary → 用了临时表（需优化）
  Using filesort → 额外排序（需优化）
  Using index condition → 索引下推 ICP
```

## 6. 什么是索引下推（ICP）？

**回答：**

```
Index Condition Pushdown (5.6+)：
  将部分 WHERE 条件下推到存储引擎层
  在索引层就过滤，减少回表次数

示例：INDEX(name, age)
  SELECT * FROM users WHERE name LIKE '张%' AND age > 20

  无 ICP：
  ┌──────────┐   回表100行   ┌──────────┐
  │ 二级索引  │──────────────→│ 聚簇索引 │
  │ name LIKE│  逐行检查age   │ 读取完整行│
  │ 匹配100行│               │ 过滤age>20│
  └──────────┘               └──────────┘
  回表 100 次

  有 ICP：
  ┌──────────┐   回表20行    ┌──────────┐
  │ 二级索引  │─────────────→│ 聚簇索引 │
  │ name LIKE│               │ 读取完整行│
  │ 且 age>20│               │          │
  │ 匹配20行 │               │          │
  └──────────┘               └──────────┘
  索引层就过滤 age，只回表 20 次

  EXPLAIN Extra 中显示：Using index condition
```

## 7. 前缀索引和全文索引？

**回答：**

```
前缀索引：
  对字符串列的前 N 个字符建索引
  CREATE INDEX idx_email ON users(email(6));

  优点：减少索引空间
  缺点：不能用于 ORDER BY 和覆盖索引

  选择前缀长度：
  -- 计算区分度
  SELECT 
    COUNT(DISTINCT LEFT(email, 5)) / COUNT(*) AS sel5,
    COUNT(DISTINCT LEFT(email, 6)) / COUNT(*) AS sel6,
    COUNT(DISTINCT LEFT(email, 7)) / COUNT(*) AS sel7
  FROM users;
  -- 选择区分度 > 90% 的最短长度

全文索引（FULLTEXT）：
  适用于大文本搜索
  
  ALTER TABLE articles ADD FULLTEXT(content);
  
  -- 自然语言模式
  SELECT * FROM articles 
  WHERE MATCH(content) AGAINST('MySQL 索引');
  
  -- 布尔模式
  SELECT * FROM articles 
  WHERE MATCH(content) AGAINST(
    '+MySQL -Oracle' IN BOOLEAN MODE);

  生产环境：大规模文本搜索推荐 Elasticsearch
```

## 8. 索引设计原则？

**回答：**

```
  1. 适合建索引：
     • WHERE 条件列
     • JOIN 关联列
     • ORDER BY / GROUP BY 列
     • 高选择性列（区分度高）

  2. 不适合建索引：
     • 数据量小的表
     • 频繁更新的列（维护索引开销大）
     • 区分度低的列（如性别）
     • WHERE 中不使用的列

  3. 索引设计技巧：
     • 覆盖索引减少回表
     • 联合索引代替多个单列索引
     • 短索引（前缀索引）节省空间
     • 避免冗余索引
       (a,b) 已包含 (a)，不需要单独建 (a)

  4. 主键设计：
     • 自增 BIGINT（插入有序，页分裂少）
     • 避免 UUID 做主键（随机写入，页分裂多）
     • 业务无关（不用身份证号等）

  5. 索引数量控制：
     • 单表索引不宜过多（5-6个以内）
     • 索引占空间，影响写入性能
```

## 9. 自适应哈希索引和 Change Buffer？

**回答：**

```
自适应哈希索引（AHI）：
  InnoDB 自动为热点页建立哈希索引
  将 B+ 树查找 O(logN) → 哈希查找 O(1)
  
  条件：某个索引页被等值查询频繁访问
  自动管理：无需手动干预
  参数：innodb_adaptive_hash_index = ON（默认）
  
  注意：高并发下可能有 AHI 锁竞争
  监控：SHOW ENGINE INNODB STATUS → AHI 部分

Change Buffer（写缓冲）：
  非唯一二级索引的 DML 操作缓存

  ┌─────────┐     ┌──────────────┐
  │ INSERT   │────→│ Change Buffer│
  │ UPDATE   │     │ (缓存变更)    │
  │ DELETE   │     └──────┬───────┘
  └─────────┘            │
                   后台 merge 或
                   读到该页时 merge
                         │
                    ┌────↓────┐
                    │ 二级索引 │
                    │ 数据页   │
                    └─────────┘

  作用：减少随机 IO（二级索引是随机写入）
  限制：唯一索引不能用（需要检查唯一性）
  参数：innodb_change_buffer_max_size（默认25%）
```

## 10. 索引优化面试速答？

**回答：**

```
Q: 为什么用 B+ 树不用 B 树？
A: B+ 非叶子只存 key → 扇出更大 → 更矮
   叶子链表 → 范围查询高效

Q: 什么是回表？
A: 二级索引查到主键后
   再查聚簇索引获取完整行

Q: 怎么避免回表？
A: 覆盖索引
   查询字段都在索引中

Q: 最左前缀原则？
A: 联合索引(a,b,c)
   查询必须从最左列开始连续匹配
   范围查询后的列索引失效

Q: 为什么自增主键好？
A: 有序插入，减少页分裂
   紧凑存储，索引更小

Q: EXPLAIN 最重要的字段？
A: type（ALL要优化）
   key（实际用的索引）
   rows（扫描行数越少越好）
   Extra（Using index好/filesort坏）

Q: 索引失效常见原因？
A: 函数计算/隐式转换
   左模糊 LIKE '%x'
   违反最左前缀

Q: 索引建多了有什么问题？
A: 占用磁盘空间
   INSERT/UPDATE/DELETE 变慢（维护索引）
```
