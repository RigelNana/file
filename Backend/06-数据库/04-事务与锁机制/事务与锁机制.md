## 1. 事务的 ACID 特性如何实现？

**回答：**

```
  ┌──────────┬──────────────┬──────────────────┐
  │ 特性      │ 实现机制      │ 说明              │
  ├──────────┼──────────────┼──────────────────┤
  │ A 原子性  │ undo log     │ 回滚时恢复旧数据  │
  │ C 一致性  │ 约束+应用逻辑│ 其他三者的目的     │
  │ I 隔离性  │ MVCC + 锁    │ 并发事务互不干扰  │
  │ D 持久性  │ redo log     │ WAL 先写日志再刷盘│
  └──────────┴──────────────┴──────────────────┘

redo log 保证持久性：
  1. 修改数据前先写 redo log (WAL)
  2. redo log 顺序写入（快）
  3. 事务提交 → redo log 刷盘
  4. 崩溃恢复 → 重放 redo log

undo log 保证原子性：
  1. 修改数据前记录旧值到 undo log
  2. 事务回滚 → 用 undo log 恢复
  3. 同时用于 MVCC（版本链）

两阶段提交（binlog + redo log 一致性）：
  1. InnoDB: 写 redo log（prepare 状态）
  2. Server: 写 binlog
  3. InnoDB: redo log 改为 commit 状态
  → 保证主从数据一致
```

## 2. InnoDB 有哪些锁？

**回答：**

```
锁分类：

  按粒度：
  ┌────────────────────────────────────┐
  │ 表级锁                             │
  │  意向锁 IS/IX（自动加）             │
  │  MDL 锁（元数据锁）                │
  │  AUTO-INC 锁                      │
  ├────────────────────────────────────┤
  │ 行级锁                             │
  │  Record Lock（记录锁）             │
  │  Gap Lock（间隙锁）                │
  │  Next-Key Lock（临键锁）           │
  │  Insert Intention Lock（插入意向锁）│
  └────────────────────────────────────┘

  按模式：
  共享锁 S（读锁）：SELECT ... LOCK IN SHARE MODE
  排他锁 X（写锁）：SELECT ... FOR UPDATE
                     INSERT / UPDATE / DELETE

  兼容性：
  ┌──────┬──────┬──────┐
  │      │ S    │ X    │
  ├──────┼──────┼──────┤
  │ S    │ ✅   │ ❌   │
  │ X    │ ❌   │ ❌   │
  └──────┴──────┴──────┘

意向锁作用：
  事务加行锁前先加表级意向锁(IS/IX)
  其他事务加表锁时无需逐行检查是否有行锁
  只需检查意向锁即可快速判断
```

## 3. Record Lock、Gap Lock、Next-Key Lock？

**回答：**

```
假设索引值：10, 20, 30

Record Lock（记录锁）：
  锁住单条索引记录
  SELECT * FROM t WHERE id = 20 FOR UPDATE;
  → 锁住 id=20 这一行

Gap Lock（间隙锁）：
  锁住索引间的间隙（左开右开）
  防止其他事务在间隙中插入
  
  间隙：(-∞,10) (10,20) (20,30) (30,+∞)
  
  SELECT * FROM t WHERE id = 15 FOR UPDATE;
  → id=15 不存在 → 锁住间隙(10,20)
  → 其他事务无法 INSERT id=11~19

Next-Key Lock（临键锁）：
  Record Lock + Gap Lock（左开右闭）
  InnoDB 默认锁类型（RR 级别）
  
  锁范围：(-∞,10] (10,20] (20,30] (30,+∞)
  
  SELECT * FROM t WHERE id = 20 FOR UPDATE;
  → Next-Key: (10,20]  
  → 等值命中唯一索引时退化为 Record Lock

Insert Intention Lock（插入意向锁）：
  特殊的间隙锁
  不同事务插入同一间隙的不同位置不互斥
  → 提高并发插入性能
```

## 4. 死锁的原因和解决？

**回答：**

```
死锁条件（同时满足四个）：
  1. 互斥：资源独占
  2. 持有等待：持有锁的同时等待其他锁
  3. 不可剥夺：锁不能被强制释放
  4. 循环等待：A等B，B等A

MySQL 死锁示例：
  事务A:                    事务B:
  UPDATE t SET ..           UPDATE t SET ..
  WHERE id = 1; (锁id=1)   WHERE id = 2; (锁id=2)
  
  UPDATE t SET ..           UPDATE t SET ..
  WHERE id = 2; (等id=2)   WHERE id = 1; (等id=1)
  → 死锁！

InnoDB 处理机制：
  1. 等待超时：innodb_lock_wait_timeout（默认50s）
  2. 死锁检测：innodb_deadlock_detect = ON（默认）
     → 检测到死锁后回滚代价较小的事务

避免死锁：
  1. 按固定顺序访问资源（如按 id 从小到大）
  2. 减小事务粒度（事务尽量短）
  3. 一次性获取所有需要的锁
  4. 使用较低隔离级别（RC 无 Gap Lock）

排查死锁：
  SHOW ENGINE INNODB STATUS;  → LATEST DETECTED DEADLOCK
  日志：innodb_print_all_deadlocks = ON
```

## 5. 乐观锁与悲观锁？

**回答：**

```
悲观锁：
  假设冲突一定发生 → 先加锁再操作
  
  -- 行锁（FOR UPDATE）
  BEGIN;
  SELECT * FROM inventory WHERE id = 1 FOR UPDATE;
  -- 处理业务逻辑
  UPDATE inventory SET stock = stock - 1 WHERE id = 1;
  COMMIT;
  
  优点：数据安全
  缺点：阻塞等待，并发低

乐观锁：
  假设冲突很少发生 → 更新时检查版本

  方式一：版本号
  -- 读取
  SELECT id, stock, version FROM inventory WHERE id = 1;
  -- version = 5
  
  -- 更新（CAS）
  UPDATE inventory 
  SET stock = stock - 1, version = version + 1
  WHERE id = 1 AND version = 5;
  -- 影响行数=0 → 重试

  方式二：时间戳
  UPDATE inventory 
  SET stock = stock - 1, update_time = NOW()
  WHERE id = 1 AND update_time = '2024-01-01 12:00:00';

对比：
  ┌──────────┬──────────┬──────────┐
  │          │ 悲观锁    │ 乐观锁   │
  ├──────────┼──────────┼──────────┤
  │ 冲突场景  │ 高冲突    │ 低冲突   │
  │ 性能      │ 低（阻塞）│ 高       │
  │ 实现      │ DB行锁    │ 版本号   │
  │ 重试      │ 不需要    │ 需要重试 │
  └──────────┴──────────┴──────────┘
```

## 6. 当前读与快照读？

**回答：**

```
快照读（Consistent Read）：
  普通 SELECT → 读取 MVCC 快照版本
  不加锁 → 不阻塞
  
  SELECT * FROM users WHERE id = 1;
  → 读取事务启动时的快照版本

当前读（Locking Read）：
  读取最新数据 + 加锁
  
  SELECT ... LOCK IN SHARE MODE;  → S 锁
  SELECT ... FOR UPDATE;          → X 锁
  INSERT / UPDATE / DELETE;       → X 锁
  → 读取最新已提交的数据

RR 级别下的区别：
  事务A:                    事务B:
  BEGIN;                    
  SELECT * FROM t;          
  → 快照版本(无新数据)       INSERT INTO t VALUES(5);
                            COMMIT;
  SELECT * FROM t;          
  → 仍是旧快照(无新数据) ← 快照读：可重复读

  SELECT * FROM t 
  FOR UPDATE;
  → 看到新数据(id=5) ← 当前读：读最新

  幻读：当前读能看到其他事务插入的数据
  InnoDB 通过 Next-Key Lock 防止当前读的幻读
```

## 7. 意向锁和 MDL 锁？

**回答：**

```
意向锁（Intention Lock）：
  表级锁，行锁的"预告"
  
  意向共享锁 IS：事务准备加行级 S 锁
  意向排他锁 IX：事务准备加行级 X 锁

  作用：
  事务加表锁时无需逐行检查
  只查意向锁就知道表中是否有行锁

  兼容矩阵：
  ┌──────┬──────┬──────┬──────┬──────┐
  │      │ IS   │ IX   │ S    │ X    │
  ├──────┼──────┼──────┼──────┼──────┤
  │ IS   │ ✅   │ ✅   │ ✅   │ ❌   │
  │ IX   │ ✅   │ ✅   │ ❌   │ ❌   │
  │ S    │ ✅   │ ❌   │ ✅   │ ❌   │
  │ X    │ ❌   │ ❌   │ ❌   │ ❌   │
  └──────┴──────┴──────┴──────┴──────┘
  意向锁之间互相兼容

MDL 锁（Metadata Lock）：
  5.5 引入，自动加
  DML (SELECT/INSERT...) → MDL 读锁
  DDL (ALTER TABLE...)   → MDL 写锁
  
  读-读兼容 / 读-写互斥 / 写-写互斥
  
  危险场景：
  长事务持有 MDL 读锁 → ALTER TABLE 等 MDL 写锁
  → 后续所有 SELECT 也被阻塞（排队等 MDL 写锁）
  
  解决：
  - 避免长事务
  - DDL 前检查 SHOW PROCESSLIST
  - 使用 pt-online-schema-change 或 gh-ost
```

## 8. 行锁加锁规则（RR 级别）？

**回答：**

```
InnoDB RR 级别加锁规则（重要）：

规则1：加锁的基本单位是 Next-Key Lock（左开右闭）
规则2：查找过程中访问到的对象才会加锁
规则3：唯一索引等值查询，命中记录 → 退化为 Record Lock
规则4：等值查询，向右遍历到不满足条件时，
       Next-Key Lock 退化为 Gap Lock
规则5：范围查询也会访问到不满足条件的第一个值

示例（索引值：5, 10, 15, 20）：

  等值查询（唯一索引命中）：
  WHERE id = 10
  → Record Lock: id=10（规则3退化）

  等值查询（唯一索引未命中）：
  WHERE id = 12
  → Gap Lock: (10, 15)（规则4退化）

  等值查询（非唯一索引命中）：
  WHERE name = 'B' (name 索引值: A,B,C)
  → Next-Key: (A,B] + Gap: (B,C)

  范围查询：
  WHERE id >= 10 AND id < 15
  → Record Lock: id=10
  → Next-Key: (10,15]
  → 实际锁范围：[10,15]

特别注意：
  RC 级别无 Gap Lock → 可能幻读但并发更好
```

## 9. 如何减少锁冲突？

**回答：**

```
减少锁冲突的方法：

  1. 缩短事务：
     事务中不要有网络请求/耗时计算
     只在必要的代码段使用事务

  2. 降低隔离级别：
     RC 级别无 Gap Lock（大多数互联网场景够用）
     减少锁范围

  3. 合理使用索引：
     WHERE 条件走索引 → 行锁
     不走索引 → 锁全表（因为要全表扫描）

  4. 按顺序访问：
     多行操作按主键顺序 → 避免死锁

  5. 批量操作拆分：
     大事务拆为小批次（LIMIT）

  6. 乐观锁替代：
     低冲突场景用版本号代替 FOR UPDATE
```

```go
// Go 事务最佳实践
func Transfer(db *sql.DB, from, to int64, amount float64) error {
    tx, err := db.Begin()
    if err != nil {
        return err
    }
    defer tx.Rollback()

    // 按 ID 顺序加锁（避免死锁）
    first, second := from, to
    if from > to {
        first, second = to, from
    }

    // 锁定两行
    var balance1, balance2 float64
    tx.QueryRow("SELECT balance FROM accounts WHERE id = ? FOR UPDATE", first).Scan(&balance1)
    tx.QueryRow("SELECT balance FROM accounts WHERE id = ? FOR UPDATE", second).Scan(&balance2)

    // 业务逻辑
    tx.Exec("UPDATE accounts SET balance = balance - ? WHERE id = ?", amount, from)
    tx.Exec("UPDATE accounts SET balance = balance + ? WHERE id = ?", amount, to)

    return tx.Commit()
}
```

## 10. 事务与锁面试速答？

**回答：**

```
Q: ACID 分别靠什么实现？
A: A→undo log / C→约束+逻辑
   I→MVCC+锁 / D→redo log(WAL)

Q: InnoDB 默认行锁是什么？
A: Next-Key Lock（记录锁+间隙锁）
   RR 级别下防止幻读

Q: 什么时候行锁退化为表锁？
A: WHERE 条件不走索引
   InnoDB 对所有行加锁 ≈ 表锁

Q: 快照读和当前读区别？
A: 快照读：普通 SELECT，读 MVCC 版本，不加锁
   当前读：FOR UPDATE / DML，读最新+加锁

Q: 死锁怎么排查？
A: SHOW ENGINE INNODB STATUS
   看 LATEST DETECTED DEADLOCK

Q: 悲观锁和乐观锁场景？
A: 高冲突→悲观锁(FOR UPDATE)
   低冲突→乐观锁(版本号CAS)

Q: 两阶段提交是什么？
A: 写redo log(prepare)→写binlog
   →redo log(commit)
   保证两份日志一致性

Q: 如何避免死锁？
A: 按固定顺序访问资源
   事务尽量短/降隔离级别
```
