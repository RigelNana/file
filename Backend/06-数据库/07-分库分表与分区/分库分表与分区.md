# 分库分表与分区

---

## 1. 为什么要分库分表？

**回答：**

```
单库瓶颈：
  数据量过大：单表超过千万级 → 查询变慢
  并发过高：单库连接数有限 → 支撑不了
  磁盘空间不足：单机存不下

  ┌──────────────────────────────────────┐
  │ 阈值参考（经验值）：                  │
  │ 单表行数 > 500万~2000万 → 考虑分表   │
  │ 单库大小 > 几百GB → 考虑分库          │
  │ 单库 QPS > 几千 → 考虑分库            │
  └──────────────────────────────────────┘

分库分表类型：
  ┌──────────┬──────────────────────────┐
  │ 类型      │ 说明                      │
  ├──────────┼──────────────────────────┤
  │ 垂直分库  │ 按业务拆（用户库/订单库）  │
  │ 垂直分表  │ 宽表拆分（主表+扩展表）   │
  │ 水平分库  │ 同结构数据分到多个库      │
  │ 水平分表  │ 同结构数据分到多个表      │
  └──────────┴──────────────────────────┘

优先级：
  先优化 SQL/索引 → 读写分离 → 缓存
  → 垂直拆分 → 水平拆分（最后手段）
```

## 2. 水平分表的分片策略？

**回答：**

```
分片键选择：
  高频查询字段（如 user_id）
  数据分布均匀
  避免跨分片查询

常用分片策略：

  1. 哈希取模：
     shard = hash(user_id) % N
     优点：分布均匀
     缺点：扩容需要数据迁移

  2. 范围分片：
     0-1000万 → 表1
     1000万-2000万 → 表2
     优点：扩容简单（新建表即可）
     缺点：热点问题（新数据集中在最后一个表）

  3. 一致性哈希：
     虚拟节点 + 哈希环
     优点：扩容只迁移部分数据
     缺点：实现复杂

  4. 日期分片（适合日志/订单）：
     按月/年建表
     orders_202401, orders_202402

推荐：
  用户相关 → user_id 哈希取模
  时间序列 → 日期范围分片
  订单 → user_id 哈希（查询都带 user_id）
```

## 3. 分库分表带来哪些问题？

**回答：**

```
核心问题：

  1. 跨分片 JOIN：
     用户表和订单表在不同库
     → 不能直接 JOIN
     解决：应用层关联 / 冗余字段 / 宽表

  2. 跨分片排序/分页：
     SELECT * FROM orders ORDER BY create_time LIMIT 10
     → 每个分片取前 10 条，合并后再排序
     深分页性能更差

  3. 分布式事务：
     跨库操作如何保证一致性
     → XA 事务 / TCC / Saga / 最终一致性

  4. 全局唯一 ID：
     每个库自增主键会重复
     → 雪花算法 / UUID / 号段模式

  5. 数据迁移和扩容：
     分片数变化 → 数据重新分配
     → 双写迁移 / 停机迁移

  6. 运维复杂度：
     监控、备份、DDL 变更成本倍增

  7. 非分片键查询：
     按 order_id 查询但分片键是 user_id
     → 需要额外索引表或广播查询
```

## 4. 全局唯一 ID 方案？

**回答：**

```
  ┌──────────────┬──────┬──────┬──────────┐
  │ 方案          │ 有序 │ 性能 │ 依赖      │
  ├──────────────┼──────┼──────┼──────────┤
  │ UUID         │ ❌   │ 高   │ 无        │
  │ 数据库自增    │ ✅   │ 中   │ 数据库    │
  │ 号段模式      │ ✅   │ 高   │ 数据库    │
  │ 雪花算法 ★   │ ✅   │ 高   │ 时钟      │
  │ Redis INCR   │ ✅   │ 高   │ Redis     │
  └──────────────┴──────┴──────┴──────────┘

雪花算法（Snowflake）：
  64 位 = 1(符号) + 41(时间戳) + 10(机器ID) + 12(序列号)
  
  0 | 00000000 00000000 00000000 00000000 00000000 0
    | ←────────── 41位时间戳(ms) ──────────────→|
    | ←─ 10位机器ID ─→| ←─ 12位序列号 ─→|

  特点：趋势递增，本地生成无依赖
  问题：时钟回拨 → 等待或报错
```

```go
// 简化版雪花算法
type Snowflake struct {
    mu        sync.Mutex
    epoch     int64 // 起始时间戳
    workerID  int64
    sequence  int64
    lastStamp int64
}

func (s *Snowflake) NextID() int64 {
    s.mu.Lock()
    defer s.mu.Unlock()

    now := time.Now().UnixMilli()
    if now == s.lastStamp {
        s.sequence = (s.sequence + 1) & 0xFFF // 4096
        if s.sequence == 0 {
            for now <= s.lastStamp { // 等待下一毫秒
                now = time.Now().UnixMilli()
            }
        }
    } else {
        s.sequence = 0
    }
    s.lastStamp = now

    return ((now - s.epoch) << 22) |
        (s.workerID << 12) | s.sequence
}
```

## 5. 分库分表中间件？

**回答：**

```
常用中间件：

  ShardingSphere（Apache）：
  ┌──────────────────────────────────┐
  │ ShardingSphere-JDBC              │
  │ Java 库，嵌入应用中              │
  │ 透明分片/读写分离/分布式事务     │
  ├──────────────────────────────────┤
  │ ShardingSphere-Proxy             │
  │ 独立代理服务                     │
  │ 支持任何语言（MySQL 协议兼容）    │
  └──────────────────────────────────┘

  Vitess（Go，YouTube 开源）：
  - 支持 MySQL 分片和管理
  - Kubernetes 原生部署
  - 适合大规模场景

  TiDB（PingCAP）：
  - 分布式 NewSQL 数据库
  - 兼容 MySQL 协议
  - 自动分片/自动扩容
  - 不用改代码（推荐）

  选择建议：
  Java 项目 → ShardingSphere-JDBC
  多语言 → ShardingSphere-Proxy / Vitess
  全新项目 → 考虑 TiDB（免分库分表）
```

## 6. MySQL 分区表？

**回答：**

```
分区 vs 分表：
  分区：MySQL 原生支持，一张表多个物理文件
  分表：应用层拆分，多张独立表

分区类型：
  RANGE：按范围（常用）
  LIST：按值列表
  HASH：哈希取模
  KEY：MySQL 内部哈希

  -- RANGE 分区（按日期）
  CREATE TABLE orders (
      id BIGINT,
      user_id BIGINT,
      create_time DATETIME,
      amount DECIMAL(10,2)
  ) PARTITION BY RANGE (YEAR(create_time)) (
      PARTITION p2022 VALUES LESS THAN (2023),
      PARTITION p2023 VALUES LESS THAN (2024),
      PARTITION p2024 VALUES LESS THAN (2025),
      PARTITION pmax VALUES LESS THAN MAXVALUE
  );

  -- 快速删除历史数据
  ALTER TABLE orders DROP PARTITION p2022;

优点：
  分区裁剪（只查相关分区）
  历史数据管理方便（DROP PARTITION）
  对应用透明

缺点：
  分区键必须在主键/唯一索引中
  跨分区查询性能差
  分区数有限（8192）

适用场景：日志/时序数据/按时间清理
```

## 7. 数据迁移与扩容方案？

**回答：**

```
停机迁移（简单但有停机）：
  1. 停止写入
  2. 全量数据迁移到新分片
  3. 切换路由配置
  4. 恢复服务

双写迁移（推荐，无停机）：
  ┌─────────────────────────────────┐
  │ 阶段1：双写                      │
  │ 新旧库同时写入 + 旧库读取        │
  │                                  │
  │ 阶段2：全量迁移                  │
  │ 后台将旧库数据迁移到新库          │
  │ 对比校验数据一致性               │
  │                                  │
  │ 阶段3：切读                      │
  │ 读流量切到新库                   │
  │                                  │
  │ 阶段4：停写旧库                  │
  │ 停止旧库写入                     │
  │ 观察一段时间后下线旧库            │
  └─────────────────────────────────┘

binlog 迁移：
  用 Canal 监听 binlog → 写入新分片
  全量导出 + 增量 binlog 同步
  切换后停止 Canal

扩容策略：
  翻倍扩容法：4 → 8 → 16
  数据只需一半迁移
  hash(key) % 4 的路由可以通过位运算升级
```

## 8. 读写分离方案？

**回答：**

```
架构：
  ┌──────────┐  写请求   ┌──────────┐
  │ 应用/中间件│────────→│ 主库      │
  │          │          │ (Master)  │
  │          │  读请求   ├──────────┤
  │          │────────→│ 从库1     │
  │          │────────→│ 从库2     │
  └──────────┘          └──────────┘

实现方式：

  1. 应用层代码（手动）：
     写操作 → 主库 DataSource
     读操作 → 从库 DataSource

  2. 中间件代理：
     ProxySQL / MySQL Router
     应用不感知主从

  3. ORM 层（GORM）：
```

```go
// GORM 读写分离
db, _ := gorm.Open(mysql.Open(masterDSN))

db.Use(dbresolver.Register(
    dbresolver.Config{
        Sources:  []gorm.Dialector{
            mysql.Open(masterDSN),
        },
        Replicas: []gorm.Dialector{
            mysql.Open(slave1DSN),
            mysql.Open(slave2DSN),
        },
        Policy: dbresolver.RandomPolicy{},
    },
))

// 自动路由：写→主库，读→从库
db.Create(&user)          // → 主库
db.Find(&users)           // → 从库
db.Clauses(
    dbresolver.Write,     // 强制主库读
).Find(&users)
```

```
注意事项：
  主从延迟 → 写后立即读走主库
  关键业务读主库
  监控从库延迟
```

## 9. NewSQL 数据库了解吗？

**回答：**

```
NewSQL：兼顾关系型 + 分布式

  ┌──────────────┬──────────────────────────┐
  │ 数据库        │ 特点                      │
  ├──────────────┼──────────────────────────┤
  │ TiDB         │ MySQL 兼容/自动分片       │
  │              │ Raft 一致性/在线扩缩容    │
  ├──────────────┼──────────────────────────┤
  │ CockroachDB  │ PostgreSQL 兼容           │
  │              │ 全球分布/强一致            │
  ├──────────────┼──────────────────────────┤
  │ OceanBase    │ 蚂蚁自研/MySQL 兼容       │
  │              │ 混合负载(HTAP)            │
  ├──────────────┼──────────────────────────┤
  │ PolarDB     │ 阿里云                    │
  │              │ 计算存储分离              │
  └──────────────┴──────────────────────────┘

TiDB 架构：
  TiDB Server → SQL 解析/优化（无状态可扩展）
  TiKV        → 分布式 KV 存储（Raft 多副本）
  PD          → 调度和元数据管理
  TiFlash     → 列式存储引擎（OLAP加速）

与分库分表对比：
  分库分表：改造成本大/运维复杂/自动化程度低
  NewSQL：开箱即用/自动分片/但学习和运维有门槛
```

## 10. 分库分表面试速答？

**回答：**

```
Q: 什么时候分库分表？
A: 单表超千万行性能下降时
   先优化SQL/索引/缓存/读写分离
   最后才分库分表

Q: 常用分片策略？
A: 哈希取模（均匀）
   范围分片（扩容简单但有热点）

Q: 分片键怎么选？
A: 高频查询条件/数据均匀分布
   如用户相关用 user_id

Q: 分库分表最大问题？
A: 跨分片 JOIN / 分布式事务
   全局 ID / 非分片键查询

Q: 全局 ID 方案？
A: 雪花算法（趋势递增/本地生成）

Q: 数据迁移怎么做？
A: 双写迁移（无停机）
   全量导出+增量 binlog 同步

Q: 不想分库分表有什么方案？
A: TiDB 等 NewSQL 数据库
   兼容 MySQL，自动分片

Q: 读写分离注意什么？
A: 主从延迟→写后读走主库
   关键业务读主库
```
