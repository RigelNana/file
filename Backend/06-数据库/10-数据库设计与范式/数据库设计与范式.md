# 数据库设计与范式

---

## 1. 数据库三大范式？

**回答：**

```
第一范式（1NF）- 原子性：
  每个字段不可再分
  
  ❌ 地址: "北京市海淀区xxx路"
  ✅ 省: "北京" 市: "北京" 区: "海淀" 街道: "xxx路"
  （是否拆分取决于业务需求）

第二范式（2NF）- 消除部分依赖：
  在 1NF 基础上，非主键列完全依赖于主键
  
  ❌ 订单项表(订单ID, 商品ID, 商品名, 数量)
     商品名只依赖商品ID（部分依赖）
  ✅ 拆分：
     订单项(订单ID, 商品ID, 数量)
     商品(商品ID, 商品名)

第三范式（3NF）- 消除传递依赖：
  在 2NF 基础上，非主键不依赖其他非主键
  
  ❌ 员工(员工ID, 部门ID, 部门名)
     部门名依赖部门ID（传递依赖）
  ✅ 拆分：
     员工(员工ID, 部门ID)
     部门(部门ID, 部门名)

实际开发：
  适度反范式化
  为了查询性能可以冗余字段
  减少 JOIN 提升查询速度
```

## 2. 反范式设计何时使用？

**回答：**

```
反范式：适当冗余，减少 JOIN

场景1 - 冗余字段：
  订单表冗余商品名称和价格
  → 避免查询时 JOIN 商品表
  → 保留下单时的快照（价格可能变）

场景2 - 计数冗余：
  用户表增加 post_count 字段
  → 避免每次 COUNT(*)
  → 发帖时 +1，删帖时 -1

场景3 - 宽表：
  数据仓库中的事实表
  包含所有维度信息（预 JOIN）
  → 分析查询无需 JOIN

反范式原则：
  ┌──────────────┬──────────┬──────────┐
  │              │ 范式化    │ 反范式化  │
  ├──────────────┼──────────┼──────────┤
  │ 数据冗余     │ 少        │ 多       │
  │ 写操作       │ 简单      │ 复杂     │
  │ 读操作       │ 需 JOIN   │ 直接查   │
  │ 一致性维护   │ 容易      │ 需要维护 │
  │ 适用场景     │ 写多      │ 读多     │
  └──────────────┴──────────┴──────────┘

  互联网项目通常读多写少 → 适度反范式
```

## 3. 表设计最佳实践？

**回答：**

```
基本规范：
  1. 表名/字段名用小写+下划线
  2. 每表必有主键（自增 BIGINT）
  3. 每表加 create_time / update_time
  4. 字段 NOT NULL + DEFAULT（避免 NULL）
  5. 注释每个表和重要字段

数据类型选择：
  主键 → BIGINT UNSIGNED AUTO_INCREMENT
  金额 → DECIMAL(10,2)
  状态 → TINYINT + 注释
  时间 → DATETIME
  短文本 → VARCHAR(N)
  JSON → JSON 类型
  布尔 → TINYINT(1)

避免 NULL 的原因：
  NULL 参与计算结果为 NULL
  NULL 影响索引效率
  COUNT(字段) 忽略 NULL
  IS NULL / IS NOT NULL 索引使用受限

示例：
  CREATE TABLE users (
      id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
      username VARCHAR(50) NOT NULL COMMENT '用户名',
      email VARCHAR(100) NOT NULL DEFAULT '' COMMENT '邮箱',
      status TINYINT NOT NULL DEFAULT 1 COMMENT '1正常 0禁用',
      create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
      update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
          ON UPDATE CURRENT_TIMESTAMP,
      INDEX idx_username (username),
      INDEX idx_create_time (create_time)
  ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户表';
```

## 4. 如何设计订单系统的表？

**回答：**

```
核心表设计：

  users（用户表）:
  id, username, phone, email, status, create_time

  products（商品表）:
  id, name, description, price, stock, status

  orders（订单表）:
  id, order_no, user_id, total_amount,
  status(待支付/已支付/已发货/已完成/已取消),
  payment_time, create_time

  order_items（订单项表）:
  id, order_id, product_id,
  product_name, product_price,  ← 冗余快照
  quantity, subtotal

  payments（支付记录表）:
  id, order_id, payment_no, amount,
  payment_method, status, create_time

设计要点：
  1. 订单号(order_no)用雪花算法/业务规则
     不用自增 ID 暴露给外部
  2. 订单项冗余商品名和价格（快照）
  3. 状态字段用 TINYINT + 状态机管理
  4. 金额用 DECIMAL（不用 FLOAT）
  5. 软删除（is_deleted）而非物理删除
  6. 分库分表键选 user_id
```

## 5. 如何设计权限系统的表？

**回答：**

```
RBAC（基于角色的访问控制）：

  users（用户表）:
  id, username, password_hash, status

  roles（角色表）:
  id, name, description

  permissions（权限表）:
  id, name, resource, action
  -- resource: '/api/users', action: 'read'

  user_roles（用户-角色关联）:
  user_id, role_id
  PRIMARY KEY (user_id, role_id)

  role_permissions（角色-权限关联）:
  role_id, permission_id
  PRIMARY KEY (role_id, permission_id)

  ER 关系：
  用户 ←多对多→ 角色 ←多对多→ 权限

  权限查询：
  SELECT DISTINCT p.* FROM permissions p
  JOIN role_permissions rp ON p.id = rp.permission_id
  JOIN user_roles ur ON rp.role_id = ur.role_id
  WHERE ur.user_id = ?;

扩展设计：
  部门表 → 部门角色
  数据权限 → 行级别控制
  菜单权限 → 前端路由控制
  操作日志 → 审计追踪
```

## 6. 软删除与数据归档？

**回答：**

```
软删除：
  不物理删除数据，用标记标识已删除

  方案1 - 删除标记：
  ALTER TABLE users ADD is_deleted TINYINT DEFAULT 0;
  -- 删除
  UPDATE users SET is_deleted = 1 WHERE id = ?;
  -- 查询
  SELECT * FROM users WHERE is_deleted = 0;
  
  问题：唯一索引冲突
  username 唯一，删除后无法创建同名用户
  解决：唯一索引加 is_deleted
  UNIQUE KEY (username, is_deleted)
  或用 deleted_at（NULL 表示未删除）

  方案2 - 删除时间：
  deleted_at DATETIME DEFAULT NULL
  -- 删除
  UPDATE users SET deleted_at = NOW() WHERE id = ?;
  -- 查询
  SELECT * FROM users WHERE deleted_at IS NULL;
  
  GORM 默认使用此方案

数据归档：
  历史数据迁移到归档表/冷存储
  
  1. 定时任务迁移
     INSERT INTO orders_archive SELECT * FROM orders
     WHERE create_time < DATE_SUB(NOW(), INTERVAL 1 YEAR);
     DELETE FROM orders WHERE create_time < ...;

  2. 分区表方案
     按月分区 → DROP 过期分区

  3. 冷热分离
     热数据：MySQL（最近3个月）
     冷数据：ClickHouse/S3（历史）
```

## 7. 多租户数据库设计？

**回答：**

```
方案对比：

  ┌──────────────┬──────────┬──────┬──────┐
  │ 方案          │ 隔离级别 │ 成本 │ 运维 │
  ├──────────────┼──────────┼──────┼──────┤
  │ 独立数据库    │ 最高     │ 高   │ 复杂 │
  │ 共享库独立表  │ 中       │ 中   │ 中   │
  │ 共享表(tenant)│ 低       │ 低   │ 简单 │
  └──────────────┴──────────┴──────┴──────┘

共享表方案（最常用）：
  每个表加 tenant_id 字段
  
  CREATE TABLE orders (
      id BIGINT PRIMARY KEY,
      tenant_id BIGINT NOT NULL,  ← 租户ID
      user_id BIGINT NOT NULL,
      amount DECIMAL(10,2),
      INDEX idx_tenant (tenant_id)
  );
  
  -- 所有查询都带 tenant_id
  SELECT * FROM orders 
  WHERE tenant_id = ? AND user_id = ?;
```

```go
// Go 中间件自动注入 tenant_id
func TenantMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        tenantID := r.Header.Get("X-Tenant-ID")
        ctx := context.WithValue(r.Context(), "tenant_id", tenantID)
        next.ServeHTTP(w, r.WithContext(ctx))
    })
}

// GORM Scope 自动过滤
func TenantScope(tenantID int64) func(db *gorm.DB) *gorm.DB {
    return func(db *gorm.DB) *gorm.DB {
        return db.Where("tenant_id = ?", tenantID)
    }
}
// 使用
db.Scopes(TenantScope(tenantID)).Find(&orders)
```

## 8. 树形结构存储方案？

**回答：**

```
场景：部门树/菜单树/评论回复

方案1 - 邻接表（最常用）：
  id | parent_id | name
  1  | 0         | 总公司
  2  | 1         | 技术部
  3  | 1         | 市场部
  4  | 2         | 后端组

  优点：结构简单，增删改容易
  缺点：查询子树需要递归

  MySQL 8.0 递归查询：
  WITH RECURSIVE tree AS (
      SELECT * FROM departments WHERE id = 1
      UNION ALL
      SELECT d.* FROM departments d
      JOIN tree t ON d.parent_id = t.id
  )
  SELECT * FROM tree;

方案2 - 路径枚举（path）：
  id | path       | name
  1  | /1/        | 总公司
  2  | /1/2/      | 技术部
  4  | /1/2/4/    | 后端组
  
  查子树：WHERE path LIKE '/1/2/%'
  优点：查询子树简单高效
  缺点：路径字段长度有限，移动节点需更新子树

方案3 - 闭包表（Closure Table）：
  ancestor | descendant | depth
  1        | 1          | 0
  1        | 2          | 1
  1        | 4          | 2
  2        | 2          | 0
  2        | 4          | 1

  查子树：WHERE ancestor = 1
  优点：查询灵活，支持任意深度
  缺点：额外空间，维护复杂
```

## 9. 数据库设计评审要点？

**回答：**

```
评审清单：

  命名规范：
  □ 表名/字段名小写+下划线
  □ 表名用复数或业务前缀
  □ 关联表用 A_B 命名

  主键设计：
  □ 自增 BIGINT UNSIGNED
  □ 不使用业务字段做主键

  字段设计：
  □ NOT NULL + DEFAULT
  □ 数据类型合理（DECIMAL/DATETIME）
  □ 避免 TEXT/BLOB（必要时拆分）
  □ 枚举用 TINYINT（不用 ENUM）

  索引设计：
  □ 常用查询条件有索引
  □ 联合索引顺序合理
  □ 不超过 5-6 个索引
  □ 无冗余索引

  扩展性：
  □ 预留扩展字段或 JSON 列
  □ 考虑未来数据量增长
  □ 合理的分库分表策略

  安全性：
  □ 敏感字段加密（手机号/身份证）
  □ 密码 bcrypt 存储
  □ SQL 注入防护

  其他：
  □ 外键在应用层维护（不用 DB 外键）
  □ 注释完整
  □ create_time / update_time
  □ 软删除方案
```

## 10. 数据库设计面试速答？

**回答：**

```
Q: 三大范式是什么？
A: 1NF 字段不可再分
   2NF 消除部分依赖
   3NF 消除传递依赖

Q: 实际开发遵循三范式吗？
A: 适度反范式
   为查询性能冗余字段（空间换时间）

Q: 为什么不建议用 NULL？
A: NULL 计算结果为 NULL
   影响索引/COUNT 忽略 NULL

Q: 外键用不用？
A: 生产不用数据库外键
   应用层维护关联关系
   外键影响性能和扩展

Q: 主键用 UUID 好不好？
A: 不好，随机写入导致页分裂
   用自增 BIGINT（有序插入效率高）

Q: 树形结构怎么存？
A: 邻接表最常用(parent_id)
   MySQL 8.0 用递归 CTE 查询子树

Q: 多租户怎么设计？
A: 共享表加 tenant_id 字段
   所有查询自动过滤 tenant_id

Q: 软删除怎么做？
A: deleted_at 字段
   NULL 表示未删除
   唯一索引需要特殊处理
```
