# 主从复制与高可用

---

## 1. MySQL 主从复制架构？

**回答：**

```
基本架构：
  ┌──────────┐     binlog     ┌──────────┐
  │  Master  │───────────────→│  Slave   │
  │  主库    │   IO Thread    │  从库    │
  │          │                │          │
  │ binlog   │                │ relay log│
  │ dump线程 │                │ SQL线程  │
  └──────────┘                └──────────┘

复制线程（3个）：
  主库: Binlog Dump Thread
    → 读取 binlog 发送给从库

  从库: IO Thread
    → 接收 binlog 写入 relay log

  从库: SQL Thread
    → 读取 relay log 执行 SQL

配置步骤：
  主库:
  [mysqld]
  server-id = 1
  log-bin = mysql-bin
  binlog_format = ROW
  
  CREATE USER 'repl'@'%' IDENTIFIED BY 'password';
  GRANT REPLICATION SLAVE ON *.* TO 'repl'@'%';

  从库:
  [mysqld]
  server-id = 2
  relay-log = relay-bin
  read_only = 1
  
  CHANGE MASTER TO
    MASTER_HOST='master_ip',
    MASTER_USER='repl',
    MASTER_PASSWORD='password',
    MASTER_AUTO_POSITION=1;  -- GTID模式
  START SLAVE;
```

## 2. 异步、半同步、组复制的区别？

**回答：**

```
  ┌──────────────┬────────────┬──────────────┐
  │ 模式          │ 数据安全   │ 性能          │
  ├──────────────┼────────────┼──────────────┤
  │ 异步复制      │ 可能丢数据 │ 最高          │
  │ 半同步复制    │ 基本不丢   │ 较高          │
  │ 组复制(MGR)  │ 不丢       │ 中等          │
  └──────────────┴────────────┴──────────────┘

异步复制（默认）：
  主库提交 → 立即返回客户端
  不等从库确认
  主库崩溃 → binlog 未传输部分丢失

半同步复制：
  主库提交 → 等至少一个从库确认接收 relay log
  → 返回客户端
  
  INSTALL PLUGIN rpl_semi_sync_master ...;
  SET GLOBAL rpl_semi_sync_master_enabled = 1;
  SET GLOBAL rpl_semi_sync_master_timeout = 1000; -- ms

  增强半同步（5.7 AFTER_SYNC）：
  redo log → binlog → 等从库ACK → 引擎提交
  → 主库崩溃不会出现"幻影读"

组复制（MGR, Group Replication）：
  基于 Paxos 协议的多主/单主模式
  自动成员管理
  冲突检测和处理
  强一致性
```

## 3. 高可用架构方案？

**回答：**

```
  ┌──────────────┬──────────────────────────┐
  │ 方案          │ 说明                      │
  ├──────────────┼──────────────────────────┤
  │ 主从+手动切换│ 最简单，但需要人工介入    │
  │ MHA          │ 自动故障检测+切换          │
  │ MySQL InnoDB │ 官方组复制方案             │
  │  Cluster     │ MGR + MySQL Router        │
  │ Orchestrator │ 拓扑管理+自动故障转移     │
  │ PXC(Percona) │ 多主同步复制              │
  │ 云数据库RDS  │ 托管服务，自动高可用       │
  └──────────────┴──────────────────────────┘

MHA（Master High Availability）：
  ┌─────────┐
  │ MHA     │ 监控主库
  │ Manager │ 检测故障
  └────┬────┘ 选举新主
       │      切换 VIP
  ┌────┴────┐  ┌────────┐  ┌────────┐
  │ Master  │  │ Slave1 │  │ Slave2 │
  │ (VIP)   │  │        │  │        │
  └─────────┘  └────────┘  └────────┘

  故障切换流程：
  1. MHA 检测主库不可用
  2. 选择数据最新的从库
  3. 补齐差异的 relay log
  4. 提升为新主库
  5. 其他从库指向新主
  6. VIP 漂移到新主
  → 切换时间约 10-30 秒
```

## 4. GTID 复制？

**回答：**

```
GTID（Global Transaction Identifier）：
  全局唯一事务标识符
  格式：server_uuid:transaction_id
  例如：3E11FA47-71CA-11E1-9E33-C80AA9429562:1-100

优势：
  1. 自动定位复制位点（无需手动指定 binlog 文件和位置）
  2. 故障切换简单（从库自动找到正确位置）
  3. 一致性保证（同一事务全局唯一）

配置：
  [mysqld]
  gtid_mode = ON
  enforce_gtid_consistency = ON
  
  -- 从库
  CHANGE MASTER TO
    MASTER_HOST='master_ip',
    MASTER_AUTO_POSITION = 1;  -- 自动定位

  -- 查看 GTID
  SHOW MASTER STATUS;  -- Executed_Gtid_Set
  SHOW SLAVE STATUS;   -- Retrieved_Gtid_Set

GTID 限制：
  不能在事务中创建临时表
  不能使用 CREATE TABLE ... SELECT
  需要 ROW 格式 binlog
```

## 5. 故障切换（Failover）流程？

**回答：**

```
手动切换（计划内维护）：
  1. 主库设只读：SET GLOBAL read_only = ON;
  2. 等从库追上主库（Seconds_Behind_Master = 0）
  3. 从库停止复制：STOP SLAVE;
  4. 从库取消只读：SET GLOBAL read_only = OFF;
  5. 应用修改配置指向新主库
  6. 其他从库 CHANGE MASTER 到新主

自动切换（MHA 流程）：
  ┌─────────────────────────────────┐
  │ 1. 检测主库不可用              │
  │    ping 失败 + SSH 失败         │
  │                                │
  │ 2. 选举新主库                  │
  │    数据最新 + 配置的优先级      │
  │                                │
  │ 3. 数据补齐                    │
  │    从其他从库获取差异 relay log  │
  │    应用到新主库                 │
  │                                │
  │ 4. 切换                        │
  │    新主库取消只读               │
  │    其他从库指向新主             │
  │    VIP 漂移                    │
  │                                │
  │ 5. 通知应用                    │
  │    DNS 更新 / VIP 漂移         │
  └─────────────────────────────────┘

数据一致性保证：
  半同步复制 → 主从数据差异最小
  binlog 补齐 → 尽量不丢事务
  人工确认 → 关键业务
```

## 6. 多主架构（Multi-Master）？

**回答：**

```
双主架构（Active-Standby）：
  ┌──────────┐  相互复制  ┌──────────┐
  │ Master A │←─────────→│ Master B │
  │ (写入)   │            │ (备用)   │
  └──────────┘            └──────────┘
  
  只有一个接受写入
  另一个热备（宕机后接管）
  VIP/Keepalived 实现自动切换

  防止自增冲突：
  Master A: auto_increment_offset=1, increment=2 → 1,3,5...
  Master B: auto_increment_offset=2, increment=2 → 2,4,6...

MySQL InnoDB Cluster（真正多主）：
  ┌──────────────────────────────────┐
  │        MySQL Router              │
  │  (读写分离 + 故障感知路由)        │
  ├──────┬──────────┬───────────────┤
  │ Node1│ Node2    │ Node3         │
  │ (RW) │ (RO)     │ (RO)          │
  │      │          │               │
  │  ←── Group Replication ──→      │
  │      (Paxos 协议)               │
  └──────┴──────────┴───────────────┘
  
  单主模式（默认）：一个 RW + 多个 RO
  多主模式：所有节点都可写（冲突检测）
```

## 7. ProxySQL 与 MySQL Router？

**回答：**

```
ProxySQL：
  高性能 MySQL 代理
  
  功能：
  - 读写分离（自动路由）
  - 查询缓存
  - 查询重写和过滤
  - 连接池和多路复用
  - 后端健康检查
  - 查询限流
  
  优势：灵活/功能丰富/社区活跃

MySQL Router（官方）：
  InnoDB Cluster 配套的路由组件
  
  功能：
  - 自动感知集群拓扑变化
  - 读写分离
  - 故障自动切换
  
  配置简单，与 InnoDB Cluster 深度集成

选择：
  InnoDB Cluster → MySQL Router
  传统主从 → ProxySQL
  需要高级功能 → ProxySQL
```

## 8. 如何保证主从数据一致性？

**回答：**

```
数据不一致的原因：
  1. 异步复制丢 binlog
  2. 从库直接写入
  3. binlog 格式为 STATEMENT 有不确定性
  4. 主从版本/配置不同

保证一致性措施：

  1. 半同步或组复制
  2. 从库设 read_only（super_read_only 更安全）
  3. binlog 使用 ROW 格式
  4. 主从版本和配置保持一致

数据校验工具：
  pt-table-checksum（一致性检查）：
  pt-table-checksum \
    --host=master \
    --user=root --password=xxx \
    --databases=mydb
  → 在主库计算校验和，从库对比

  pt-table-sync（数据修复）：
  pt-table-sync \
    --execute \
    --sync-to-master \
    slave_host
  → 修复从库与主库的差异数据

定期巡检：
  每天/每周运行 pt-table-checksum
  发现差异及时 pt-table-sync 修复
  记录不一致次数，分析原因
```

## 9. 跨机房/多地多活？

**回答：**

```
跨机房方案：

  同城双机房：
  ┌────────────┐   专线   ┌────────────┐
  │ 机房A (主)  │←───────→│ 机房B (从)  │
  │ 写入 + 读  │  延迟<1ms│ 只读        │
  └────────────┘         └────────────┘
  同城延迟低，半同步可行

  两地三中心：
  ┌─── 同城 A ───┐     ┌── 异地 ──┐
  │ 主中心(写)    │     │ 灾备中心  │
  │ 同城灾备(读)  │────→│ 异步复制  │
  └──────────────┘     └──────────┘
  
  异地多活（最复杂）：
  用户按地域路由到最近机房
  每个机房有完整的读写能力
  数据通过 DTS 双向同步
  → 需要解决数据冲突（最后写入wins/业务定制）

关键挑战：
  数据冲突解决
  跨地域延迟（几十ms~几百ms）
  故障时的流量切换
  数据一致性保证
```

## 10. 主从复制与高可用面试速答？

**回答：**

```
Q: 主从复制有几个线程？
A: 3个：主库 BinlogDump
   从库 IO Thread + SQL Thread

Q: 异步复制有什么风险？
A: 主库崩溃时 binlog 未传输
   → 数据丢失

Q: 半同步复制原理？
A: 主库等至少一个从库确认
   收到 relay log 后才提交

Q: GTID 有什么好处？
A: 全局唯一事务标识
   故障切换自动定位/无需手动指定位点

Q: 高可用推荐方案？
A: MHA（自动切换 10-30s）
   InnoDB Cluster（官方组复制）

Q: 主从数据不一致怎么办？
A: pt-table-checksum 检查
   pt-table-sync 修复

Q: 双主架构自增ID怎么处理？
A: offset 和 increment 错开
   A产生奇数 B产生偶数

Q: 多地多活最难的是什么？
A: 数据冲突和一致性
   跨地域延迟影响用户体验
```
