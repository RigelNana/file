# 日志与备份恢复

---

## 1. MySQL 有哪些日志？

**回答：**

```
MySQL 日志体系：

  ┌──────────────┬──────────┬──────────────────────┐
  │ 日志          │ 层级     │ 作用                  │
  ├──────────────┼──────────┼──────────────────────┤
  │ redo log     │ InnoDB   │ 崩溃恢复（持久性）    │
  │ undo log     │ InnoDB   │ 事务回滚 + MVCC      │
  │ binlog       │ Server   │ 主从复制 + 数据恢复   │
  │ relay log    │ 从库     │ 中继日志，重放 binlog  │
  │ slow query   │ Server   │ 慢查询记录            │
  │ error log    │ Server   │ 错误和启动信息        │
  │ general log  │ Server   │ 所有 SQL 记录（调试用）│
  └──────────────┴──────────┴──────────────────────┘

三大核心日志关系：
  ┌─────────┐
  │ binlog  │ ← Server 层，所有引擎都有
  │ 逻辑日志 │    主从复制/数据恢复
  └────┬────┘
       │ 两阶段提交
  ┌────┴────┐  ┌─────────┐
  │ redo log│  │ undo log│
  │ 物理日志 │  │ 逻辑日志 │
  │ 崩溃恢复 │  │ 回滚+MVCC│
  └─────────┘  └─────────┘
   └── InnoDB 引擎层 ──┘
```

## 2. redo log 详解？

**回答：**

```
redo log 作用：
  WAL（Write-Ahead Logging）
  修改数据前先写日志 → 崩溃后可恢复

  为什么不直接写数据文件？
  数据页是随机 IO（慢）
  redo log 是顺序 IO（快）
  → 先写 redo log 提交，后台异步刷数据页

redo log 结构：
  固定大小文件组，循环写入
  
  ┌──────────────────────────────┐
  │  ib_logfile0   ib_logfile1   │
  │  ┌─────────┐  ┌─────────┐  │
  │  │ ████░░░░│  │░░░░░░░░ │  │
  │  └─────────┘  └─────────┘  │
  │       ↑write_pos    ↑checkpoint │
  └──────────────────────────────┘
  
  write_pos: 当前写入位置（前进）
  checkpoint: 已刷盘的位置（前进）
  write_pos 追上 checkpoint → 需要等待刷脏页

刷盘策略（innodb_flush_log_at_trx_commit）：
  0: 每秒刷一次（可能丢 1 秒数据）
  1: 每次提交都刷盘（最安全，默认）
  2: 每次提交写 OS 缓存，每秒刷盘

  推荐：核心业务用 1，性能要求高可用 2
```

## 3. binlog 详解？

**回答：**

```
binlog 作用：
  1. 主从复制
  2. 数据恢复（mysqlbinlog 工具）

binlog 格式：
  ┌──────────────┬──────────────────────────┐
  │ 格式          │ 说明                      │
  ├──────────────┼──────────────────────────┤
  │ STATEMENT    │ 记录 SQL 语句             │
  │              │ 优：日志量小               │
  │              │ 缺：NOW()等不确定函数不安全│
  ├──────────────┼──────────────────────────┤
  │ ROW          │ 记录行变更（推荐）         │
  │              │ 优：确定性，不会不一致     │
  │              │ 缺：批量操作日志量大       │
  ├──────────────┼──────────────────────────┤
  │ MIXED        │ 混合模式                  │
  │              │ 普通用 STATEMENT          │
  │              │ 不安全时自动切 ROW         │
  └──────────────┴──────────────────────────┘

  推荐：ROW 格式（数据安全第一）

binlog 刷盘（sync_binlog）：
  0: OS 自行决定刷盘（最快，不安全）
  1: 每次提交都刷盘（最安全，推荐）
  N: 每 N 次提交刷一次

双 1 配置（最安全）：
  innodb_flush_log_at_trx_commit = 1
  sync_binlog = 1
```

## 4. 两阶段提交为什么必要？

**回答：**

```
问题：binlog 和 redo log 是两个独立的日志
  如果写入不一致 → 主从数据不一致

  场景1：先写 redo log 再写 binlog
  redo log 写完，binlog 未写 → 崩溃
  主库用 redo log 恢复（有数据）
  从库没有 binlog（没数据）→ 不一致

  场景2：先写 binlog 再写 redo log
  binlog 写完，redo log 未写 → 崩溃
  主库 redo log 无记录（没数据）
  从库重放 binlog（有数据）→ 不一致

两阶段提交：
  ┌─────────────────────────────────┐
  │ 1. redo log: PREPARE            │
  │    → 写入 redo log，标记 prepare │
  │                                  │
  │ 2. 写 binlog                     │
  │    → 写入 binlog 并刷盘          │
  │                                  │
  │ 3. redo log: COMMIT              │
  │    → redo log 标记 commit        │
  └─────────────────────────────────┘

  崩溃恢复判断：
  redo log=prepare + binlog完整 → 提交
  redo log=prepare + binlog不完整 → 回滚
  → 保证两份日志数据一致
```

## 5. 数据备份策略？

**回答：**

```
备份分类：
  物理备份：复制数据文件（xtrabackup）
  逻辑备份：导出 SQL（mysqldump）
  
  全量备份：备份所有数据
  增量备份：只备份上次备份后的变化

常用工具：

  mysqldump（逻辑备份）：
  # 全库备份
  mysqldump -u root -p --all-databases \
    --single-transaction > full.sql
  
  # 单库备份
  mysqldump -u root -p dbname \
    --single-transaction > db.sql

  # 恢复
  mysql -u root -p dbname < db.sql

  xtrabackup（物理备份，推荐）：
  # 全量备份
  xtrabackup --backup --target-dir=/backup/full

  # 增量备份
  xtrabackup --backup --target-dir=/backup/inc1 \
    --incremental-basedir=/backup/full

  # 恢复
  xtrabackup --prepare --target-dir=/backup/full
  xtrabackup --copy-back --target-dir=/backup/full

备份策略（生产环境）：
  每周一次全量 + 每天增量
  binlog 实时备份到远程
  定期验证备份可恢复
  RPO 目标决定备份频率
```

## 6. binlog 数据恢复？

**回答：**

```
使用 binlog 恢复（基于时间点恢复）：

  场景：误操作 DROP TABLE，需要恢复

  步骤：
  1. 全量备份恢复
     mysql -u root -p dbname < full_backup.sql

  2. 找到误操作时间点
     mysqlbinlog --start-datetime="2024-01-01 00:00:00" \
       --stop-datetime="2024-01-15 14:30:00" \
       mysql-bin.000001 | head

  3. 重放 binlog 到误操作前
     mysqlbinlog --start-datetime="2024-01-10 00:00:00" \
       --stop-datetime="2024-01-15 14:29:00" \
       mysql-bin.000001 mysql-bin.000002 \
       | mysql -u root -p dbname

  也可以用 position 定位：
  mysqlbinlog --start-position=1234 \
    --stop-position=5678 \
    mysql-bin.000001 | mysql -u root -p

  闪回工具（逆向binlog）：
  - MyFlash
  - binlog2sql
  将 DELETE 反转为 INSERT
  将 UPDATE 反转为逆向 UPDATE
```

## 7. 主从复制的日志流转？

**回答：**

```
主从复制架构：

  ┌────────────┐         ┌────────────┐
  │  主库 Master │         │  从库 Slave │
  │            │         │            │
  │  写入数据   │         │            │
  │    ↓       │         │            │
  │  binlog    │────────→│  relay log │
  │  (二进制)   │ IO Thread│  (中继日志) │
  │            │         │    ↓       │
  │            │         │ SQL Thread │
  │            │         │  重放 SQL   │
  └────────────┘         └────────────┘

详细流程：
  1. 主库执行写操作 → 写入 binlog
  2. 从库 IO 线程连接主库
  3. 主库 Binlog Dump 线程发送 binlog 事件
  4. 从库 IO 线程写入 relay log
  5. 从库 SQL 线程读取 relay log 并执行
  6. 从库数据与主库保持（最终）一致

复制模式：
  异步复制：默认，主库不等从库确认
  半同步：至少一个从库确认收到 relay log
  组复制：Paxos 协议，强一致（MGR）

GTID 复制（推荐）：
  全局事务 ID，简化故障切换
  自动定位复制位点
  SET @@GLOBAL.gtid_mode = ON;
```

## 8. 主从延迟原因与解决？

**回答：**

```
延迟原因：
  1. 从库单线程重放（5.6之前）
  2. 主库大事务（批量更新几百万行）
  3. 从库机器性能差
  4. 网络延迟
  5. 从库上有大查询阻塞 SQL 线程

解决方案：
  1. 多线程复制（5.7+）：
     slave_parallel_workers = 8
     slave_parallel_type = LOGICAL_CLOCK
     → 同一组提交的事务并行重放

  2. 读写分离中间件：
     读从库时感知延迟
     延迟超阈值 → 读主库

  3. 避免大事务：
     拆分为小批次

  4. 从库硬件与主库一致

  5. 使用半同步复制：
     确保从库收到日志（但不保证重放完成）

监控延迟：
  SHOW SLAVE STATUS;
  → Seconds_Behind_Master 字段

  -- 更精确的方式
  SELECT * FROM performance_schema.replication_applier_status;
```

## 9. 生产环境日志配置建议？

**回答：**

```
核心配置（my.cnf）：

  # redo log
  innodb_flush_log_at_trx_commit = 1   # 每次提交刷盘
  innodb_log_file_size = 1G            # 日志文件大小
  innodb_log_files_in_group = 2        # 日志文件数量

  # binlog
  sync_binlog = 1                      # 每次提交刷盘
  binlog_format = ROW                  # ROW 格式
  expire_logs_days = 7                 # binlog 保留天数
  max_binlog_size = 1G                 # 单个文件大小

  # 慢查询
  slow_query_log = ON
  long_query_time = 1                  # 阈值 1 秒
  log_queries_not_using_indexes = ON   # 未用索引的查询

  # 错误日志
  log_error = /var/log/mysql/error.log

日志管理：
  定期清理 binlog：PURGE BINARY LOGS BEFORE '2024-01-01';
  定期轮转慢查询日志
  监控磁盘空间（日志可能占大量空间）

  安全建议：
  日志文件权限限制
  备份日志到远程存储
  不在 general log 中记录敏感 SQL
```

## 10. 日志与备份面试速答？

**回答：**

```
Q: redo log 和 binlog 的区别？
A: redo log: InnoDB 物理日志/循环写/崩溃恢复
   binlog: Server 逻辑日志/追加写/复制+恢复

Q: 为什么需要两阶段提交？
A: 保证 redo log 和 binlog 数据一致
   prepare → binlog → commit

Q: binlog 用什么格式？
A: ROW 格式，确定性最强
   STATEMENT 有不确定函数风险

Q: innodb_flush_log_at_trx_commit 设多少？
A: 1=每次提交刷盘（最安全）
   2=写OS缓存每秒刷（折中）

Q: 主从复制流程？
A: 主写 binlog → 从 IO 线程拉取
   → 写 relay log → SQL 线程重放

Q: 数据误删怎么恢复？
A: 全量备份恢复 + binlog 重放到误操作前
   闪回工具逆向 binlog

Q: 备份推荐什么工具？
A: xtrabackup（物理备份，支持增量）
   mysqldump（逻辑备份，小库）

Q: 主从延迟怎么解决？
A: 多线程复制/避免大事务
   读写分离感知延迟
```
