## 1. PostgreSQL 与 MySQL 的区别？

**回答：**

```
  ┌──────────────┬──────────────┬──────────────┐
  │ 对比维度      │ PostgreSQL   │ MySQL        │
  ├──────────────┼──────────────┼──────────────┤
  │ 定位          │ 功能丰富     │ 简单易用     │
  │ MVCC 实现    │ 多版本存储   │ undo log     │
  │ 索引类型      │ B-tree/GIN/  │ B+tree/Hash  │
  │              │ GiST/BRIN等  │              │
  │ JSON 支持    │ JSONB(强)    │ JSON(较弱)   │
  │ 扩展性        │ 插件丰富     │ 存储引擎     │
  │ 全文搜索      │ 内置 tsvector│ FULLTEXT     │
  │ 地理信息      │ PostGIS(强)  │ 基本支持     │
  │ 窗口函数      │ 早期支持     │ 8.0 才支持   │
  │ 物化视图      │ ✅           │ ❌           │
  │ 社区          │ 学术/严谨    │ 互联网/活跃  │
  │ 生态          │ 较小         │ 最大         │
  └──────────────┴──────────────┴──────────────┘

PostgreSQL 独有特性：
  CTE 递归查询（WITH RECURSIVE）
  数组/Range/自定义类型
  表继承
  规则系统
  丰富的索引（部分索引/表达式索引）

选择建议：
  互联网 OLTP → MySQL（生态/运维成熟）
  复杂查询/GIS/JSON → PostgreSQL
  数据分析/OLAP → PostgreSQL
```

## 2. Redis 基础与数据结构？

**回答：**

```
Redis：内存键值数据库
  单线程命令处理（6.0+ IO 多线程）
  持久化（RDB/AOF）
  丰富数据结构

数据结构：
  ┌──────────┬──────────────┬──────────────┐
  │ 类型      │ 底层实现      │ 常用场景      │
  ├──────────┼──────────────┼──────────────┤
  │ String   │ SDS          │ 缓存/计数器   │
  │ Hash     │ ziplist/dict │ 对象存储      │
  │ List     │ quicklist    │ 消息队列/时间线│
  │ Set      │ intset/dict  │ 标签/交集     │
  │ ZSet     │ ziplist/     │ 排行榜/延迟队列│
  │          │ skiplist+dict│              │
  │ Stream   │ radix tree   │ 消息流(5.0+) │
  │ Bitmap   │ String       │ 签到/布隆过滤 │
  │ HyperLog │ 概率算法     │ UV 统计      │
  │ GEO      │ ZSet         │ 附近的人      │
  └──────────┴──────────────┴──────────────┘

  -- 常用命令
  SET key value EX 60       -- 设置带过期
  GET key
  HSET user:1 name "张三"   -- Hash
  ZADD rank 100 "user1"     -- 有序集合
  LPUSH queue "msg1"        -- 列表
```

## 3. MongoDB 基础？

**回答：**

```
MongoDB：文档型 NoSQL 数据库
  BSON（Binary JSON）存储
  灵活 Schema（无需预定义表结构）
  支持丰富查询和聚合

核心概念映射：
  ┌──────────┬──────────┬──────────┐
  │ RDBMS    │ MongoDB  │ 说明     │
  ├──────────┼──────────┼──────────┤
  │ Database │ Database │ 数据库   │
  │ Table    │Collection│ 集合/表  │
  │ Row      │ Document │ 文档/行  │
  │ Column   │ Field    │ 字段/列  │
  │ JOIN     │ 嵌套/引用│ 关联     │
  │ PRIMARY  │ _id      │ 主键     │
  └──────────┴──────────┴──────────┘

适用场景：
  日志存储
  CMS 内容管理（灵活 Schema）
  IoT 设备数据
  游戏用户数据
  产品目录（属性不固定）

不适用场景：
  强事务需求（4.0 开始支持多文档事务）
  复杂 JOIN 查询
  严格数据一致性
```

## 4. Elasticsearch 基础？

**回答：**

```
Elasticsearch：分布式搜索和分析引擎
  基于 Lucene
  倒排索引
  RESTful API
  近实时搜索（NRT）

核心概念：
  Index（索引）→ 类似数据库
  Type（类型）→ 7.x 废弃
  Document（文档）→ 类似行/JSON
  Field（字段）→ 类似列
  Shard（分片）→ 水平拆分
  Replica（副本）→ 高可用

倒排索引原理：
  文档1: "Redis is fast"
  文档2: "MySQL is reliable"
  文档3: "Redis is reliable and fast"

  倒排索引：
  ┌──────────┬──────────────┐
  │ 词项      │ 文档列表      │
  ├──────────┼──────────────┤
  │ redis    │ [1, 3]       │
  │ mysql    │ [2]          │
  │ fast     │ [1, 3]       │
  │ reliable │ [2, 3]       │
  └──────────┴──────────────┘
  搜索 "redis fast" → 文档1和3

使用场景：
  全文搜索（电商/内容搜索）
  日志分析（ELK 栈）
  业务数据分析（聚合统计）
```

## 5. SQL vs NoSQL 如何选型？

**回答：**

```
  ┌─────────────┬──────────────┬──────────────┐
  │ 维度         │ SQL          │ NoSQL        │
  ├─────────────┼──────────────┼──────────────┤
  │ 数据模型     │ 关系型/表    │ 文档/KV/图   │
  │ Schema      │ 固定         │ 灵活/动态    │
  │ 事务         │ ACID         │ BASE(最终一致)│
  │ 扩展方式     │ 垂直+分库分表│ 水平扩展     │
  │ 查询         │ SQL(复杂JOIN)│ 简单查询     │
  │ 一致性       │ 强一致       │ 最终一致     │
  │ 适合         │ 结构化数据   │ 半/非结构化  │
  └─────────────┴──────────────┴──────────────┘

选型指南：
  核心业务（订单/支付）→ MySQL/PostgreSQL
  缓存/会话/排行 → Redis
  日志/非结构化数据 → MongoDB
  全文搜索 → Elasticsearch
  时序数据 → InfluxDB/TimescaleDB
  图关系 → Neo4j
  大数据分析 → ClickHouse/Hive

实际项目中通常多种混用：
  MySQL 存核心数据
  Redis 做缓存
  ES 做搜索
  MongoDB 存日志/配置
```

## 6. Redis 持久化方案？

**回答：**

```
RDB（快照）：
  某个时间点的数据快照
  
  触发方式：
  - save 900 1     # 900秒内1次修改
  - save 300 10    # 300秒内10次修改
  - BGSAVE（手动）→ fork 子进程
  
  优点：文件紧凑/恢复快
  缺点：可能丢失最后几分钟数据

AOF（追加日志）：
  记录每个写命令
  
  刷盘策略（appendfsync）：
  - always: 每条命令都刷（最安全，最慢）
  - everysec: 每秒刷一次（推荐，最多丢1秒）
  - no: 由 OS 决定

  AOF 重写：
  后台重写压缩 AOF 文件
  多条命令合并为最终状态

混合持久化（4.0+,推荐）：
  AOF 重写时：前半部分 RDB + 后半部分 AOF
  兼顾恢复速度和数据安全

  aof-use-rdb-preamble yes
```

## 7. Redis 集群方案？

**回答：**

```
  ┌──────────────┬──────────────────────────┐
  │ 方案          │ 说明                      │
  ├──────────────┼──────────────────────────┤
  │ 主从复制      │ 读写分离，手动故障切换    │
  │ Redis Sentinel│ 自动故障检测和切换        │
  │ Redis Cluster │ 分片+高可用（官方推荐）   │
  └──────────────┴──────────────────────────┘

Redis Cluster：
  16384 个哈希槽（slot）
  key → CRC16(key) % 16384 → 对应节点
  
  ┌─────────┐  ┌─────────┐  ┌─────────┐
  │ Node A  │  │ Node B  │  │ Node C  │
  │ 0-5460  │  │5461-10922│ │10923-16383│
  │ Slave A'│  │ Slave B'│  │ Slave C'│
  └─────────┘  └─────────┘  └─────────┘

  特点：
  - 去中心化（Gossip 协议通信）
  - 自动分片和再平衡
  - 部分节点故障可继续服务
  - 不支持跨节点事务
  - 不支持多 key 操作（除非同一 slot）
```

## 8. 缓存与数据库一致性？

**回答：**

```
Cache-Aside 模式（最常用）：

  读：缓存命中→返回，未命中→查DB→写缓存→返回
  写：更新 DB → 删除缓存

  为什么删缓存而不是更新缓存？
  并发写时：
  A 更新DB=10 → B 更新DB=20
  B 更新缓存=20 → A 更新缓存=10
  → DB=20 缓存=10 不一致

  为什么先更新 DB 再删缓存？
  先删缓存再更新 DB 的问题：
  A 删缓存 → B 读缓存未命中 → B 读DB旧值写缓存
  → A 更新DB → 缓存=旧值 DB=新值 不一致

  延迟双删：
  删缓存 → 更新 DB → sleep(短暂) → 再删缓存
  解决并发不一致

  最终一致性（推荐）：
  更新 DB → 发消息(MQ) → 消费者删缓存
  或：Canal 订阅 binlog → 更新/删除缓存
```

## 9. 时序数据库与列式数据库？

**回答：**

```
时序数据库：
  专为时间序列数据优化
  物联网/监控/金融

  ┌──────────────┬──────────────────────────┐
  │ 数据库        │ 特点                      │
  ├──────────────┼──────────────────────────┤
  │ InfluxDB     │ Go 编写，最流行            │
  │ TimescaleDB  │ PostgreSQL 扩展           │
  │ Prometheus   │ 监控专用（拉模式）         │
  │ TDengine     │ 国产，超高性能             │
  └──────────────┴──────────────────────────┘

  特点：高写入吞吐/时间范围查询/自动过期/压缩

列式数据库（OLAP）：
  按列存储，适合分析型查询

  ┌──────────────┬──────────────────────────┐
  │ 数据库        │ 特点                      │
  ├──────────────┼──────────────────────────┤
  │ ClickHouse   │ 极快分析查询              │
  │ Apache Doris │ MPP 分析引擎              │
  │ StarRocks    │ Doris 分支，更强           │
  │ BigQuery     │ Google 云数据仓库          │
  └──────────────┴──────────────────────────┘

  行存 vs 列存：
  行存（MySQL）：SELECT * 快，适合 OLTP
  列存（ClickHouse）：SELECT SUM(amount) 快，适合 OLAP
  列存优势：同类型数据压缩率高/向量化计算
```

## 10. PostgreSQL与NoSQL面试速答？

**回答：**

```
Q: PostgreSQL 和 MySQL 选哪个？
A: 互联网 OLTP → MySQL（生态成熟）
   复杂查询/GIS/JSONB → PostgreSQL

Q: Redis 用什么持久化？
A: 混合持久化（推荐）
   RDB快照 + AOF日志

Q: Redis Cluster 怎么分片？
A: 16384 个哈希槽
   CRC16(key) % 16384

Q: 缓存和 DB 一致性怎么做？
A: Cache-Aside：读查缓存/写更新DB+删缓存
   最终一致：Canal 订阅 binlog 同步

Q: MongoDB 适合什么场景？
A: 灵活 Schema/日志/CMS
   不适合强事务和复杂 JOIN

Q: ES 倒排索引原理？
A: 词项→文档列表
   搜索时查词项对应的文档

Q: 什么时候用 ClickHouse？
A: OLAP 分析：聚合/统计/报表
   列式存储压缩率高查询快

Q: 项目中怎么混用多种数据库？
A: MySQL 核心数据 + Redis 缓存
   ES 搜索 + MongoDB 日志
```
