# 延迟消息与死信队列

---

## 1. 延迟消息的应用场景？

**回答：**

```
延迟消息：消息发出后不立即消费，延迟一段时间后投递

场景：
  1. 订单超时关闭：下单 30 分钟未支付 → 关闭
  2. 定时通知：会议开始前 15 分钟提醒
  3. 延迟重试：消费失败 → 延迟 N 秒后重试
  4. 延时生效：优惠券 24 小时后生效
  5. 数据清理：临时文件 1 小时后删除

  ┌──────┐  发送时指定延迟   ┌──────┐  延迟到期   ┌──────┐
  │ App  │──────────────────→│ MQ   │────────────→│消费者│
  └──────┘   delay=30min     └──────┘             └──────┘
                              等待30min后投递
```

---

## 2. RabbitMQ 延迟消息实现？

**回答：**

```
方案1：TTL + 死信队列

  ┌──────────────┐  过期  ┌──────┐  路由  ┌──────────┐
  │ Delay Queue  │──────→│ DLX  │──────→│ Real     │
  │ (无消费者)   │       │      │       │ Queue    │
  │ TTL=30min    │       │      │       │ (有消费者)│
  └──────────────┘       └──────┘       └──────────┘

  队列级 TTL：所有消息统一延迟时间
  消息级 TTL：每条消息不同延迟时间

  问题：消息级 TTL 有序列头阻塞
  队列头 TTL=60s, 第二条 TTL=10s
  → 第二条也要等 60s 才检查

方案2：延迟消息插件（推荐）
  rabbitmq-delayed-message-exchange
  
  声明 Exchange 类型为 x-delayed-message
  消息设 x-delay header

  优势：
  - 不同延迟时间互不影响
  - 使用简单
  劣势：
  - 需要安装插件
  - 性能不如原生

方案3：多级延迟队列
  不同延迟时间 → 不同 TTL 的队列
  delay_5s_queue (TTL=5s)
  delay_30s_queue (TTL=30s)
  delay_5m_queue (TTL=5min)
  delay_30m_queue (TTL=30min)
```

## 3. Kafka 延迟消息方案？

**回答：**

```
Kafka 不原生支持延迟消息

方案1：外部定时服务
  发消息到 DB/Redis → 定时扫描 → 到期发 Kafka

  ┌──────┐   ┌──────────┐   ┌──────┐
  │ App  │──→│ 定时服务 │──→│Kafka │
  └──────┘   │ Redis/DB │   └──────┘
             └──────────┘

方案2：Redis ZSet
  ZADD delay_queue {execute_time} {msg}
  定时扫描 ZRANGEBYSCORE → 发到 Kafka

方案3：时间轮（Timing Wheel）
  内部多级 Topic：
  delay_1s / delay_5s / delay_1m / delay_30m
  定时服务消费延迟 Topic → 到期投递到目标 Topic

方案4：Kafka 内部 Topic 转发
  delay Topic → 消费者检查时间戳
  → 未到期 → 重新发到 delay Topic
  → 到期 → 发到目标 Topic
  简单但有空转消费
```

```go
// Redis ZSet 延迟队列 + Kafka
func AddDelayMessage(ctx context.Context, msg *Message, delay time.Duration) error {
    executeAt := time.Now().Add(delay).Unix()
    data, _ := json.Marshal(msg)
    return rdb.ZAdd(ctx, "delay:kafka", redis.Z{
        Score:  float64(executeAt),
        Member: string(data),
    }).Err()
}

// 定时扫描并发送到 Kafka
func ScanAndPublish(ctx context.Context) {
    for {
        now := float64(time.Now().Unix())
        msgs, _ := rdb.ZRangeByScore(ctx, "delay:kafka", &redis.ZRangeBy{
            Min: "0", Max: fmt.Sprintf("%f", now), Count: 100,
        }).Result()
        
        for _, raw := range msgs {
            removed, _ := rdb.ZRem(ctx, "delay:kafka", raw).Result()
            if removed > 0 {
                var msg Message
                json.Unmarshal([]byte(raw), &msg)
                kafkaProducer.Send(msg.Topic, msg.Key, msg.Value)
            }
        }
        time.Sleep(100 * time.Millisecond)
    }
}
```

## 4. RocketMQ 延迟消息？

**回答：**

```
RocketMQ 原生支持延迟消息

延迟级别（固定级别）：
  Level 1: 1s
  Level 2: 5s
  Level 3: 10s
  Level 4: 30s
  Level 5: 1min
  Level 6: 2min
  ...
  Level 18: 2h

RocketMQ 5.0+ 支持任意延迟时间

原理：
  1. 消息发到 SCHEDULE_TOPIC（内部Topic）
  2. 按延迟级别放入对应队列
  3. 定时服务扫描到期消息
  4. 转投到目标 Topic

优势：
  原生支持，无需额外组件
  可靠性与普通消息一致
```

## 5. 死信队列（DLQ）概念？

**回答：**

```
死信队列（Dead Letter Queue）：
  存放无法正常消费的消息

消息成为死信的条件：
  1. 消费者拒绝（Reject/Nack）且不重入队列
  2. 消息 TTL 过期
  3. 队列溢出（超过 max-length）
  4. 消费失败超过最大重试次数

  ┌──────────┐  消费失败   ┌──────────┐
  │ Normal   │ ──────────→│  DLQ     │
  │ Queue    │  超过重试   │          │
  └──────────┘  次数       └──────────┘
                            ↓
                    人工处理 / 告警
                    排查问题后重新投递

死信队列作用：
  1. 隔离异常消息，不阻塞正常消费
  2. 保留失败消息供排查
  3. 支持问题修复后重新投递
  4. 作为告警数据源
```

## 6. 死信队列实践？

**回答：**

```
Kafka 死信队列实现：

  Kafka 不原生支持 DLQ
  需要在消费者代码中实现

  消费流程：
  1. 消费消息
  2. 处理失败 → 重试 N 次
  3. 仍然失败 → 发到 DLQ Topic
  4. 提交 offset（不阻塞后续消费）
  5. DLQ 消费者单独处理/告警
```

```go
// Kafka 死信队列实现
const maxRetries = 3

func consumeWithDLQ(consumer *kafka.Consumer, producer *kafka.Producer) {
    dlqTopic := "orders.DLQ"
    
    for {
        msg, err := consumer.ReadMessage(time.Second)
        if err != nil {
            continue
        }
        
        // 获取重试次数
        retryCount := getRetryCount(msg.Headers)
        
        err = processMessage(msg)
        if err != nil {
            if retryCount < maxRetries {
                // 重试：发回原 Topic（带延迟更好）
                setRetryCount(msg.Headers, retryCount+1)
                producer.Produce(msg, nil)
            } else {
                // 超过重试次数 → 发到 DLQ
                dlqMsg := &kafka.Message{
                    TopicPartition: kafka.TopicPartition{
                        Topic: &dlqTopic,
                    },
                    Key:     msg.Key,
                    Value:   msg.Value,
                    Headers: msg.Headers,
                }
                producer.Produce(dlqMsg, nil)
                log.Error("msg sent to DLQ", "key", string(msg.Key))
            }
        }
        
        // 无论成功失败都提交 offset
        consumer.CommitMessage(msg)
    }
}
```

## 7. 重试策略设计？

**回答：**

```
重试策略：

1. 固定间隔：
   每次重试间隔相同
   retry_interval = 5s

2. 指数退避（推荐）：
   间隔指数增长
   1s → 2s → 4s → 8s → 16s
   加随机 jitter 避免惊群

3. 多级延迟重试：
   第1次重试：5s 后
   第2次重试：30s 后
   第3次重试：5min 后
   第4次重试：30min 后
   超过 → DLQ

  ┌──────────────────────────────┐
  │ 消费失败                     │
  │  ↓                          │
  │ 重试1(5s) → 重试2(30s)      │
  │  ↓                          │
  │ 重试3(5min) → 重试4(30min)  │
  │  ↓                          │
  │ 死信队列 → 人工处理         │
  └──────────────────────────────┘
```

```go
// 指数退避重试
func retryWithBackoff(maxRetries int, fn func() error) error {
    for i := 0; i < maxRetries; i++ {
        err := fn()
        if err == nil {
            return nil
        }
        
        if i < maxRetries-1 {
            backoff := time.Duration(1<<uint(i)) * time.Second
            jitter := time.Duration(rand.Intn(1000)) * time.Millisecond
            time.Sleep(backoff + jitter)
        }
    }
    return fmt.Errorf("max retries exceeded")
}
```

## 8. 定时任务 vs 延迟消息？

**回答：**

```
  ┌──────────┬──────────────┬──────────────┐
  │ 对比      │ 定时扫描     │ 延迟消息     │
  ├──────────┼──────────────┼──────────────┤
  │ 精度     │ 秒~分钟级    │ 秒级         │
  │ 实时性   │ 取决于扫描间隔│ 较高         │
  │ DB 压力  │ 频繁扫描大表 │ 无 DB 查询   │
  │ 可靠性   │ 高（DB 持久化）│ 取决于 MQ   │
  │ 复杂度   │ 简单         │ 中等         │
  │ 数据量   │ 大表性能差   │ MQ 天然分布式│
  │ 适用     │ 少量定时任务 │ 大量延迟任务 │
  └──────────┴──────────────┴──────────────┘

选择建议：
  任务量小 + 精度要求不高 → 定时扫描
  大量延迟任务 + 精度高 → 延迟消息
  订单关闭这种场景 → 延迟消息更优
```

## 9. 时间轮算法？

**回答：**

```
时间轮（Timing Wheel）：
  高效的定时任务调度算法
  Kafka/Netty/Go 标准库都在用

单层时间轮：
  ┌───┬───┬───┬───┬───┬───┬───┬───┐
  │ 0 │ 1 │ 2 │ 3 │ 4 │ 5 │ 6 │ 7 │
  └───┴───┴───┴─↑─┴───┴───┴───┴───┘
                │ 指针
                当前 tick

  槽位数 = 8，tick 间隔 = 1s
  指针每秒前进一格
  任务放入对应槽位链表
  指针到达 → 执行该槽位的所有任务

多层时间轮（Hierarchical）：
  秒轮：8 个槽 (0-7s)
  分轮：6 个槽 (0-5min)
  时轮：24 个槽 (0-23h)
  
  延迟 65s 的任务：
  先放入分轮第1槽
  → 降级到秒轮第5槽
  → 到期执行

  复杂度：
  添加任务：O(1)
  取消任务：O(1)
  执行检查：O(1)

应用：
  Kafka 内部延迟操作用时间轮
  Netty HashedWheelTimer
```

## 10. 延迟消息与死信面试速答？

**回答：**

```
Q: 延迟消息有哪些方案？
A: RabbitMQ: TTL+DLX / 延迟插件
   RocketMQ: 原生支持
   Kafka: Redis ZSet / 定时服务

Q: RabbitMQ 延迟队列有什么坑？
A: 消息级TTL有队列头阻塞
   第一条60s未到期,后面都等

Q: 死信队列是什么？
A: 存放消费失败/过期/被拒绝的消息
   隔离异常不阻塞正常消费

Q: Kafka 怎么实现死信队列？
A: 消费者代码实现
   重试N次失败→发到DLQ Topic

Q: 重试策略推荐什么？
A: 指数退避+随机jitter
   或多级延迟(5s/30s/5min/30min)

Q: 时间轮算法优势？
A: 添加/取消任务O(1)
   Kafka/Netty都在用

Q: 定时扫描和延迟消息怎么选？
A: 少量任务→定时扫描(简单)
   大量高精度→延迟消息

Q: 订单30分钟超时关闭怎么实现？
A: 推荐延迟消息
   下单时发delay=30min的消息
   到期消费者检查并关闭未支付订单
```
