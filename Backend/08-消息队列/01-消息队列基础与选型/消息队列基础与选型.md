# 消息队列基础与选型

---

## 1. 消息队列的核心价值？

**回答：**

```
三大核心价值：

1. 解耦：
   ┌──────┐  直接调用  ┌──────┐
   │订单  │───────────→│库存  │
   │服务  │───────────→│积分  │ 强耦合，每加一个下游要改代码
   │      │───────────→│通知  │
   └──────┘            └──────┘

   ┌──────┐  发消息    ┌──────┐    ┌──────┐
   │订单  │──────────→│  MQ  │───→│库存  │
   │服务  │           │      │───→│积分  │ 松耦合
   └──────┘           │      │───→│通知  │
                      └──────┘    └──────┘

2. 异步：
   同步：下单 → 扣库存 → 加积分 → 发短信 → 返回（500ms）
   异步：下单 → 扣库存 → 返回（100ms）
         └→ MQ → 加积分/发短信（异步）

3. 削峰：
   秒杀：瞬间 10w 请求 → MQ 缓冲 → 消费者按 1000/s 处理
   保护下游系统不被流量打垮
```

```
  ├──────────┼──────────┼──────────┼──────────┼──────────┤
  │ 语言      │ Scala/   │ Erlang   │ Java     │ Java     │
  │          │ Java     │          │          │          │
  │ 吞吐量   │ 百万/s   │ 万级/s   │ 十万/s   │ 百万/s   │
  │ 延迟     │ ms 级    │ μs 级    │ ms 级    │ ms 级    │
  │ 可靠性   │ 高       │ 高       │ 高       │ 高       │
  │ 功能     │ 简单     │ 丰富     │ 丰富     │ 丰富     │
  │ 事务消息 │ 有       │ ❌       │ ✅       │ ✅       │
  │ 延迟消息 │ ❌       │ 插件     │ ✅       │ ✅       │
  │ 顺序消息 │ 分区有序 │ 队列有序 │ 队列有序 │ 分区有序 │
  │ 社区     │ 活跃     │ 活跃     │ 中等     │ 成长中   │

选型建议：
  日志采集/大数据流 → Kafka
  复杂路由/低延迟/中小规模 → RabbitMQ
  电商/金融（事务/延迟消息）→ RocketMQ
  云原生/多租户 → Pulsar
```

## 3. 消息模型：点对点 vs 发布订阅？

**回答：**

```
1. 点对点（Queue 模式）：
   一条消息只被一个消费者消费

   Producer → Queue → Consumer1（消费）
                  → Consumer2（不会收到同一条）

   RabbitMQ 默认模式

2. 发布订阅（Pub/Sub 模式）：
   一条消息被多个订阅者消费

   Producer → Topic → Subscriber1（消费）
                  → Subscriber2（也消费）
                  → Subscriber3（也消费）

   Kafka Consumer Group 实现两种模式：
   同一组 → 点对点（组内竞争消费）
   不同组 → 发布订阅（各组都收到）

   ┌──────────────────────────┐
   │ Topic: orders            │
   │  Partition 0  Partition 1│
   └────┬──────────┬──────────┘
        │          │
   ┌────┴────┐ ┌───┴─────┐
   │Group A  │ │Group B  │
   │C1    C2 │ │C3    C4 │ 每个Group独立消费
   └─────────┘ └─────────┘
```

## 4. 消息队列的通信协议？

**回答：**

```
1. AMQP（RabbitMQ）：
   高级消息队列协议
   定义了交换机、队列、绑定等语义
   功能丰富，互操作性好

2. Kafka 自定义协议：
   基于 TCP 的二进制协议
   高性能，批量传输优化
   社区生态丰富

3. MQTT：
   物联网协议，轻量级
   QoS 0/1/2 三级服务质量
   适合嵌入式设备

4. STOMP：
   简单文本协议，类似 HTTP
   RabbitMQ 支持

5. gRPC/HTTP：
   Pulsar 支持 HTTP 接入
   方便不同语言接入
```

## 5. 引入消息队列的副作用？

**回答：**

```
1. 系统可用性降低：
   MQ 挂了  → 整个链路受影响
   → 需要 MQ 集群高可用

2. 系统复杂度增加：
   需处理：消息丢失/重复/顺序/积压
   需保证：幂等性/最终一致性
   需运维：MQ 集群监控告警

3. 一致性问题：
   A 事务成功 → 发消息 → B 消费失败
   → 需要补偿机制/事务消息

4. 排查难度增加：
   异步链路 → 问题排查跨系统
   → 需要分布式链路追踪

何时不该用 MQ：
  ❌ 调用链路简单直接
  ❌ 对实时性要求极高
  ❌ 数据量小且同步足够
  ✅ 多下游 / 异步 / 削峰 / 事件驱动
```

## 6. Push vs Pull 消费模式？

**回答：**

```
Push 模式（RabbitMQ 默认）：
  Broker 主动推送消息给消费者
  优势：实时性高
  劣势：可能压垮消费者

Pull 模式（Kafka）：
  消费者主动拉取消息
  优势：消费者按自己的速度消费
  劣势：轮询有延迟（long polling 优化）

  ┌──────┐   Push    ┌──────┐
  │Broker│──────────→│Client│  RabbitMQ
  └──────┘           └──────┘

  ┌──────┐   Pull    ┌──────┐
  │Broker│←──────────│Client│  Kafka
  └──────┘           └──────┘

Kafka Long Polling：
  fetch.min.bytes = 1
  fetch.max.wait.ms = 500
  → 没有数据时等待最多 500ms
  → 兼顾实时性和效率
```

## 7. 消息的生命周期？

**回答：**

```
Kafka 消息生命周期：

  Producer                Broker                Consumer
  ┌──────┐            ┌──────────┐            ┌──────┐
  │序列化│            │写入分区  │            │拉取  │
  │分区  │───发送────→│持久化    │←──拉取────│反序列│
  │压缩  │            │副本同步  │            │处理  │
  │重试  │            │保留策略  │            │提交  │
  └──────┘            └──────────┘            └──────┘

消息保留策略（Broker 端）：
  log.retention.hours = 168     # 保留 7 天
  log.retention.bytes = -1      # 不限大小
  log.segment.bytes = 1073741824 # 1GB 一个段文件
  log.cleanup.policy = delete   # 删除/compact

消息过了保留期 → 自动删除
  不影响未消费的消息（只要在保留期内消费即可）
```

## 8. 消息序列化？

**回答：**

```
序列化格式选择：

  ┌──────────┬────────┬────────┬──────────┐
  │ 格式      │ 体积   │ 速度   │ Schema   │
  ├──────────┼────────┼────────┼──────────┤
  │ JSON     │ 大     │ 慢     │ 无       │
  │ Protobuf │ 小     │ 快     │ .proto   │
  │ Avro     │ 小     │ 快     │ Schema   │
  │ MsgPack  │ 中     │ 快     │ 无       │
  └──────────┴────────┴────────┴──────────┘

推荐：
  内部服务 → Protobuf（最紧凑、最快）
  需要 Schema Evolution → Avro + Schema Registry
  调试方便 → JSON（开发环境）

Kafka + Schema Registry：
  Schema 集中管理
  Producer/Consumer 自动校验
  支持 Schema 演进（forward/backward 兼容）
```

## 9. 消息队列监控指标？

**回答：**

```
Kafka 关键指标：

1. 消费者 Lag：
   = 最新 Offset - 消费者 Offset
   lag 持续增大 → 消费能力不足
   核心告警指标

2. 吞吐量：
   Messages/sec（生产和消费）
   Bytes/sec（网络流量）

3. 延迟：
   生产延迟（发送到 ACK）
   端到端延迟（生产到消费）

4. 分区指标：
   ISR 数量（ISR 收缩 → 副本落后）
   Under-replicated partitions

5. Broker 指标：
   CPU / 内存 / 磁盘 IO / 网络
   请求队列长度
   GC 暂停时间

监控工具：
  Kafka Manager / CMAK
  Burrow（消费 Lag 监控）
  Prometheus + kafka_exporter + Grafana
```

## 10. 消息队列基础面试速答？

**回答：**

```
Q: 为什么用消息队列？
A: 解耦、异步、削峰
   降低系统间依赖

Q: Kafka 和 RabbitMQ 怎么选？
A: 大数据/高吞吐 → Kafka
   复杂路由/低延迟 → RabbitMQ

Q: Push 和 Pull 区别？
A: Push(RabbitMQ):实时但可能压垮消费者
   Pull(Kafka):按自己速度消费

Q: 消息模型有哪些？
A: 点对点(Queue) + 发布订阅(Pub/Sub)
   Kafka Consumer Group 支持两种

Q: 引入 MQ 有什么问题？
A: 可用性降低、复杂度增加
   一致性和排查难度增加

Q: 消息序列化推荐什么？
A: 内部服务用 Protobuf
   需要 Schema 演进用 Avro

Q: 最重要的监控指标？
A: 消费者 Lag（积压量）
   Lag 持续增大需告警

Q: 消息保留多久？
A: Kafka 默认 7 天
   可按时间或大小配置
```
