# 消息顺序与幂等

---

## 1. 为什么消息顺序很重要？

**回答：**

```
场景示例：
  订单状态变更：创建 → 支付 → 发货
  如果乱序：发货 → 创建 → 支付 → 状态错误

  数据库 binlog 同步：
  INSERT → UPDATE → DELETE
  乱序 → 数据不一致

  用户操作序列：
  注册 → 实名认证 → 开户
  乱序 → 业务逻辑错误

不需要全局有序的场景（大多数）：
  不同订单之间不需要有序
  只需要同一订单的消息有序
  → 局部有序（分区/队列级别）
```

---

## 2. Kafka 怎么保证顺序？

**回答：**

```
Kafka 保证：单分区内消息有序

方案：同一业务 key → 同一分区
  Producer 设置 key = orderId
  hash(orderId) % partitions → 固定分区
  
  ┌──────────────────────────────────┐
  │ Topic: orders (3 partitions)     │
  │ P0: [order-1001 的所有消息]      │
  │ P1: [order-1002 的所有消息]      │
  │ P2: [order-1003 的所有消息]      │
  └──────────────────────────────────┘
  
  同一分区 → 一个消费者消费 → 有序

注意事项：
  1. 消费者单线程消费该分区
     多线程可能乱序

  2. 重试不能打乱顺序
     max.in.flight.requests.per.connection = 1
     或开启幂等 (enable.idempotence=true, ≤5)

  3. 分区数变更 → key 路由改变
     hash(key) % 新分区数 ≠ 旧结果
     → 在线扩分区需谨慎
```

## 3. RabbitMQ 怎么保证顺序？

**回答：**

```
RabbitMQ 保证：单队列单消费者有序

方案1：一个队列一个消费者
  简单但吞吐低

方案2：按 key 路由到不同队列
  同一个 key → 同一个队列 → 有序
  
  N 个队列，每个队列一个消费者
  hash(key) % N → 选择队列

  Exchange(topic) → Queue_0 → Consumer_0
                 → Queue_1 → Consumer_1
                 → Queue_2 → Consumer_2

方案3：消费者内排序
  收到消息后放内存队列
  按序号排序后处理
  复杂度高，超时处理困难
```

## 4. 消费端保证顺序的方案？

**回答：**

```
问题：即使分区内有序，多线程消费也可能乱序

方案1：单线程消费（简单）
  每个分区一个 goroutine
  串行处理 → 天然有序
  缺点：吞吐受限

方案2：内存队列分发（推荐）
  单线程拉取 → 按 key hash 到 N 个内存队列
  每个内存队列单 goroutine 消费

  ┌──────────┐  按key分发  ┌──────────┐
  │Partition │────────────→│Queue 0   │→ goroutine 0
  │Consumer  │            │Queue 1   │→ goroutine 1
  │(单线程)  │            │Queue 2   │→ goroutine 2
  └──────────┘            └──────────┘

  同一 key 在同一 Queue → 有序
  不同 key 并行 → 吞吐高
```

```go
// 内存队列分发保证顺序
type OrderedConsumer struct {
    queues   []chan *Message
    numSlots int
}

func NewOrderedConsumer(slots int) *OrderedConsumer {
    oc := &OrderedConsumer{
        queues:   make([]chan *Message, slots),
        numSlots: slots,
    }
    for i := 0; i < slots; i++ {
        oc.queues[i] = make(chan *Message, 1000)
        go oc.worker(oc.queues[i])
    }
    return oc
}

func (oc *OrderedConsumer) Dispatch(msg *Message) {
    // 同一 key 固定到同一 slot
    slot := hash(msg.Key) % oc.numSlots
    oc.queues[slot] <- msg
}

func (oc *OrderedConsumer) worker(ch chan *Message) {
    for msg := range ch {
        processMessage(msg) // 串行处理，保证顺序
    }
}
```

## 5. 顺序消息的性能权衡？

**回答：**

```
全局有序 vs 局部有序：

  全局有序：1 个分区 → 吞吐极低
  局部有序：按 key 分区 → 推荐

  ┌──────────┬──────────┬──────────┐
  │ 方案      │ 有序范围 │ 吞吐量   │
  ├──────────┼──────────┼──────────┤
  │ 单分区   │ 全局有序 │ 极低     │
  │ 按key分区│ key有序  │ 高       │
  │ 无序     │ 无保证   │ 最高     │
  └──────────┴──────────┴──────────┘

大多数场景只需要局部有序：
  同一用户的操作有序
  同一订单的状态变更有序
  不同用户/订单之间不需要有序

如果确实需要全局有序：
  单分区 + 单消费者
  或在消费端用序号排序重组
```

## 6. 幂等性的核心原理？

**回答：**

```
幂等性：f(x) = f(f(x))
  同一操作执行多次，结果与执行一次相同

为什么需要幂等？
  MQ 保证 At Least Once → 必然有重复
  网络重试 / Rebalance / Broker 重发

天然幂等的操作：
  SELECT → 查询天然幂等
  SET a = 5 → 赋值天然幂等
  DELETE WHERE id = 1 → 删除天然幂等

非幂等的操作：
  INSERT → 可能重复插入
  UPDATE set a = a + 1 → 多次累加
  扣款 balance -= 100 → 多次扣

需要做幂等处理的操作：
  发送通知（短信/邮件/推送）
  创建订单
  扣减库存
  转账扣款
```

## 7. 幂等方案详解？

**回答：**

```
方案1：唯一消息 ID + 去重
  每条消息携带全局唯一 ID
  消费前检查 → 已存在则跳过

  实现：
  a) 数据库唯一索引
  b) Redis SETNX（性能更好）

方案2：数据库唯一约束
  CREATE UNIQUE INDEX uk_order_no ON orders(order_no);
  INSERT → 重复则报错或 IGNORE

方案3：乐观锁（版本号）
  UPDATE orders SET status='paid', version=6
  WHERE id=1001 AND version=5;
  → 影响 0 行 = 已处理

方案4：Token 机制
  1. 请求前获取 token（Redis 生成）
  2. 请求时带 token
  3. 服务端验证并删除 token
  4. 重复请求 → token 已删 → 拒绝

方案5：状态机
  订单状态：待支付 → 已支付 → 已发货
  只有当前状态 = 待支付 时才能改为已支付
  重复消息 → 状态不匹配 → 跳过
```

```go
// 状态机幂等
func PayOrder(ctx context.Context, orderID string) error {
    result := db.Exec(
        `UPDATE orders SET status = 'paid', paid_at = NOW()
         WHERE id = ? AND status = 'pending'`, orderID)
    
    if result.RowsAffected == 0 {
        // 已支付或不存在，幂等跳过
        return nil
    }
    // 后续逻辑...
    return nil
}
```

## 8. 消息去重的存储选择？

**回答：**

```
去重存储对比：

  ┌──────────┬──────────┬──────────┬──────────┐
  │ 方案      │ 性能     │ 可靠性   │ 容量     │
  ├──────────┼──────────┼──────────┼──────────┤
  │ Redis    │ 最高     │ 中等     │ 受内存限 │
  │ MySQL    │ 中等     │ 高       │ 大       │
  │ 内存Map  │ 最高     │ 低       │ 受内存限 │
  │ 布隆过滤 │ 高       │ 中等     │ 极省     │
  └──────────┴──────────┴──────────┴──────────┘

推荐：
  短期去重（24h内）→ Redis SETNX + TTL
  永久去重 → MySQL 唯一索引
  海量去重 → 布隆过滤器 + Redis/MySQL 兜底

Redis 去重 TTL 设多久？
  > 消息可能重复的最大时间窗口
  通常 24h 足够
  结合 Kafka retention 时间考虑
```

## 9. 生产者幂等？

**回答：**

```
Kafka Producer 幂等：
  enable.idempotence = true
  
  原理：
  每个 Producer 分配 PID（Producer ID）
  每个 <PID, Partition> 维护 Sequence Number
  Broker 检查 seq：
    seq == expected → 接受
    seq < expected → 重复，丢弃
    seq > expected → 乱序，报错

  保证范围：
  ✅ 单会话 + 单分区内不重复
  ❌ Producer 重启（PID 改变）
  ❌ 跨分区

  需要跨分区 Exactly Once → 事务 Producer
  transactional.id = "my-tx-id"
  → Producer 重启也能延续事务状态

业务层面的生产幂等：
  发消息前检查是否已发过
  本地消息表记录发送状态
  重启后继续发送未完成的消息
```

## 10. 顺序与幂等面试速答？

**回答：**

```
Q: Kafka 怎么保证消息顺序？
A: 同一 key → 同一分区
   分区内有序 + 单消费者

Q: 全局有序怎么做？
A: 单分区（吞吐极低）
   大多数场景只需局部有序

Q: 多线程消费怎么保持顺序？
A: 按 key hash 到内存队列
   每个队列单 goroutine 串行处理

Q: 幂等性是什么？
A: 同一操作多次执行
   结果与一次执行相同

Q: 消费端幂等怎么做？
A: 唯一ID+Redis去重
   数据库唯一约束/乐观锁/状态机

Q: Kafka Producer 幂等原理？
A: PID + Sequence Number
   Broker 检查序号重复则丢弃

Q: 去重用 Redis 还是 MySQL？
A: 短期去重用 Redis(SETNX+TTL)
   永久去重用 MySQL 唯一索引

Q: 顺序和性能怎么权衡？
A: 按 key 分区(局部有序)性能好
   全局有序性能极差,通常不需要
```
