# 消息队列八股文

## 一、消息队列基础

### 1. 为什么要用消息队列？

**答：** 三大核心作用：**解耦**（上下游系统通过 MQ 通信，互不依赖）；**异步**（非核心流程异步处理，降低响应时间，如下单后异步发短信/加积分）；**削峰**（高并发写入 MQ，消费者按能力消费，保护下游系统）。引入 MQ 的代价是系统复杂度增加、需考虑消息丢失/重复/顺序性等问题。

### 2. 消息队列怎么选型？

**答：** **Kafka**：高吞吐（百万级 TPS）、分区有序、持久化到磁盘，适合日志采集、大数据流式处理和事件驱动。**RabbitMQ**：延迟低（μs 级）、功能全（交换机/路由/死信/延迟队列）、Erlang 开发运维复杂，适合对可靠性和灵活路由要求高的业务。**RocketMQ**：阿里开源、支持事务消息/延迟消息/顺序消息，适合电商交易链路。**Pulsar**：计算存储分离，支持多租户，适合云原生场景。

### 3. Kafka 核心概念？

**答：** **Broker**（服务节点）、**Topic**（逻辑分类）、**Partition**（分区，有序，分布在不同 Broker）、**Offset**（消费位移）、**Producer**（生产者）、**Consumer Group**（消费者组，组内每个分区只被一个消费者消费）、**Replica**（副本，Leader 读写、Follower 同步）、**ISR**（In-Sync Replicas，保持同步的副本集合）。

### 4. 消息怎么保证不丢失？

**答：** **生产端**：Kafka 设 `acks=all` + `retries` 重试，确保所有 ISR 副本确认。**Broker 端**：`min.insync.replicas>=2`，至少两个副本同步写入；副本机制保证单节点故障不丢数据。**消费端**：关闭自动提交 offset，处理完成后手动提交；幂等消费防止重试导致问题。

### 5. 消息怎么保证顺序？

**答：** Kafka 只保证**单分区有序**。需要全局有序 → 只用 1 个分区（牺牲吞吐）。通常按业务 key（如订单 ID）路由到同一分区，保证同一业务线的消息有序。消费端单线程消费或按 key 分发到有序队列，避免多线程乱序。

### 6. 消息重复消费怎么解决（幂等性）？

**答：** MQ 本身至少一次语义（at-least-once），重复消费不可避免。解决靠消费端幂等：① 唯一 ID + 去重表（数据库唯一索引或 Redis SETNX）；② 数据库乐观锁（版本号/条件更新）；③ 业务本身幂等（如 `SET balance = 100` 而非 `balance += 10`）。

### 7. 消息积压怎么处理？

**答：** **紧急处理**：扩容消费者实例（Kafka 增加消费者 ≤ 分区数）；临时创建更多分区分散负载；跳过非关键消息。**根因排查**：消费者处理慢（优化逻辑/减少 IO）、消费者异常挂了、生产端突增（限流）。**预防**：监控消费 lag、消费者和分区数匹配、异步处理耗时操作。

### 8. 延迟消息怎么实现？

**答：** **RabbitMQ**：TTL + 死信队列（消息过期转入 DLX）或 rabbitmq-delayed-message-exchange 插件。**RocketMQ**：原生支持延迟级别（1s/5s/10s/30s/1m...）。**Kafka**：不原生支持，常见方案：时间轮 + 内部 topic 转发、外部定时服务扫描、或用 Redis ZSet 做延迟调度。

### 9. 事务消息？

**答：** 解决本地事务和发消息的一致性。**RocketMQ 事务消息**：① 发送 Half 消息（对消费者不可见）；② 执行本地事务；③ 根据事务结果 Commit/Rollback；④ Broker 定期回查未确认消息。**Kafka 事务**：Transactional Producer 支持跨分区原子写入（Exactly-Once 语义），配合 `isolation.level=read_committed`。**本地消息表**：事务中写业务数据 + 消息表，定时任务扫描发送。

### 10. Kafka 为什么吞吐量高？

**答：** ① **顺序写磁盘**（append-only log，顺序 IO 接近内存速度）；② **Page Cache**（利用 OS 页缓存，减少磁盘 IO）；③ **零拷贝**（sendfile 系统调用，数据直接从磁盘到网卡）；④ **批量处理**（Producer 批量发送、Broker 批量持久化、Consumer 批量拉取）；⑤ **分区并行**（多 Partition 并行读写）；⑥ **压缩**（snappy/lz4/zstd 减少网络传输）。

## 二、高级特性

### 11. Kafka 消费者 Rebalance？

**答：** 消费者组成员变化（加入/离开/崩溃）时触发 Rebalance，重新分配分区。**问题**：Rebalance 期间消费暂停（STW），大量分区时耗时长。**优化**：增量 Rebalance（Kafka 2.4+ Cooperative Rebalancing）只迁移需要变化的分区；合理设 `session.timeout.ms` 和 `max.poll.interval.ms` 避免误判；静态成员（`group.instance.id`）减少重启触发 Rebalance。

### 12. RabbitMQ 交换机类型？

**答：** **Direct**（精确匹配 routing key）、**Fanout**（广播到所有绑定队列）、**Topic**（通配符匹配，`*` 匹配一个单词，`#` 匹配零或多个单词）、**Headers**（按消息头匹配，少用）。消息流向：Producer → Exchange → Binding(routing key) → Queue → Consumer。

### 13. 如何保证消息队列的高可用？

**答：** **Kafka**：多 Broker 集群 + 分区多副本（Leader-Follower），Leader 挂了从 ISR 选新 Leader；`min.insync.replicas` 保证写入可靠性。**RabbitMQ**：镜像队列（Mirrored Queue，3.8+ 用 Quorum Queue 替代），多节点冗余。集群部署 + 监控 + 自动故障转移。

### 14. 消息队列在微服务中的应用场景？

**答：** **异步通信**（订单服务 → MQ → 库存/积分/通知）；**事件驱动**（领域事件发布订阅，服务间松耦合）；**CQRS**（命令写入 → 事件 → 读模型更新）；**Saga 事务**（通过事件编排分布式事务）；**日志采集**（应用日志 → Kafka → ELK）；**CDC**（数据库变更 → Debezium → Kafka → 下游同步）。
