# 消息积压与消费延迟

---

## 1. 消息积压的原因？

**回答：**

```
消息积压 = 生产速度 > 消费速度

常见原因：

1. 消费者处理慢：
   - 业务逻辑复杂/耗时
   - 外部依赖慢（DB/API/网络）
   - 锁竞争/死锁

2. 消费者故障：
   - 消费者进程崩溃
   - OOM / GC 频繁
   - 处理异常导致反复重试

3. 流量突增：
   - 促销/秒杀流量暴增
   - 上游批量数据导入
   - 重放历史消息

4. 消费能力不足：
   - 消费者实例太少
   - 分区数少，消费者多了也没用
   - 单线程消费，未充分并行

5. 配置问题：
   - max.poll.records 太大
   - max.poll.interval.ms 太小 → 频繁 Rebalance
```

---

## 2. 消息积压紧急处理方案？

**回答：**

```
紧急处理步骤：

1. 快速定位原因：
   查看消费者日志 → 是否有异常
   查看消费 Lag → kafka-consumer-groups --describe
   查看消费者状态 → 是否在运行

2. 增加消费能力：
   a) 增加消费者实例（≤ 分区数）
   b) 分区不够 → 临时扩分区
   c) 优化消费逻辑（去掉不必要的 IO）

3. 转储方案（严重积压）：
   ┌──────┐    ┌──────┐    ┌──────────┐
   │原Topic│──→│转储  │──→│新Topic   │
   │积压   │   │消费者│   │(更多分区)│
   └──────┘    └──────┘    └──────────┘
                            ↑ N个消费者
   
   快速消费者读取原 Topic → 转发到临时 Topic
   临时 Topic 有更多分区 → 更多消费者并行

4. 降级跳过：
   非关键消息 → 跳过或记录日志后续补偿
   只处理关键消息

5. 限流生产端：
   暂时降低生产速度
   给消费者追赶时间
```

## 3. 消费延迟优化？

**回答：**

```
消费延迟 = 消息产生到消费完成的时间

优化方向：

1. 并行消费：
   增加分区数 + 消费者实例
   单消费者内多线程处理
   批量处理减少 IO

2. 减少处理耗时：
   异步化（非核心逻辑异步）
   批量写 DB（攒一批再写）
   本地缓存减少网络调用

3. 优化拉取配置：
   fetch.min.bytes = 1          # 有数据就返回
   fetch.max.wait.ms = 100      # 最多等 100ms
   max.poll.records = 500       # 单次拉取量

4. 减少 Rebalance：
   session.timeout.ms = 45000
   heartbeat.interval.ms = 3000
   max.poll.interval.ms = 600000
   group.instance.id = "static-1"

5. 消费者端优化：
   连接池（DB/Redis/HTTP）
   预取/缓存
   减少序列化/反序列化开销
```

```go
// 批量消费优化
func batchConsume(consumer *kafka.Consumer) {
    batch := make([]*kafka.Message, 0, 200)
    ticker := time.NewTicker(100 * time.Millisecond)
    
    for {
        select {
        case <-ticker.C:
            if len(batch) > 0 {
                processBatch(batch)
                commitBatch(consumer, batch)
                batch = batch[:0]
            }
        default:
            msg, err := consumer.ReadMessage(10 * time.Millisecond)
            if err != nil {
                continue
            }
            batch = append(batch, msg)
            if len(batch) >= 200 {
                processBatch(batch)
                commitBatch(consumer, batch)
                batch = batch[:0]
            }
        }
    }
}
```

## 4. Kafka 分区扩容？

**回答：**

```
增加分区以提升消费并行度：

  kafka-topics.sh --alter \
    --topic orders \
    --partitions 12 \   # 从 6 增加到 12
    --bootstrap-server broker:9092

注意事项：
  1. 分区只能增加不能减少
  
  2. 按 key 路由的消息会重新分配
     hash(key) % 6 ≠ hash(key) % 12
     → 短暂的顺序性破坏
  
  3. 扩容后增加消费者实例
     消费者数 ≤ 分区数
  
  4. 分区太多的副作用：
     Leader 选举更慢
     打开文件数增加
     端到端延迟增加
     建议单集群 < 10000 分区

扩容时机：
  消费 Lag 持续增大
  单分区写入 QPS > 阈值
  消费者实例已 = 分区数但仍积压
```

## 5. 消费者扩容策略？

**回答：**

```
Kafka 消费者扩容：
  组内消费者数 ≤ 分区数
  超过分区数的消费者 → 空闲

  分区数 = 6
  消费者 = 3 → 每人 2 分区
  消费者 = 6 → 每人 1 分区
  消费者 = 8 → 2 人空闲 ❌

扩容步骤：
  1. 启动新消费者实例（同 group.id）
  2. 触发 Rebalance → 自动分配
  3. 确认分配均匀

弹性伸缩：
  基于 Lag 自动扩缩容
  Lag > 阈值 → 扩容
  Lag < 阈值 → 缩容

K8s HPA + 自定义指标：
  监控消费 Lag → Prometheus
  HPA 根据 Lag 调整 Pod 数
```

## 6. 消息积压监控与告警？

**回答：**

```
监控指标：

1. Consumer Lag：
   最核心的积压指标
   Lag = Latest Offset - Committed Offset
   
2. 消费速率：
   records-consumed-per-second
   异常下降 → 积压预警

3. 端到端延迟：
   消息时间戳 → 消费时间
   P99 延迟 > 阈值 → 告警

4. 消费者状态：
   alive / rebalancing / dead
   消费者数量变化

告警规则示例：
  Lag > 10000 持续 5min → Warning
  Lag > 100000 持续 5min → Critical
  消费速率 = 0 持续 1min → Critical
  端到端延迟 P99 > 10s → Warning

工具：
  kafka_exporter → Prometheus → Grafana
  Burrow（专门的 Lag 监控）
  告警 → AlertManager → 钉钉/Slack
```

## 7. 批量处理优化？

**回答：**

```
批量处理减少 IO 次数：

  逐条处理（慢）：
  for msg in messages:
      db.insert(msg)         # N 次 DB 操作
      redis.set(msg.key)     # N 次 Redis 操作

  批量处理（快）：
  batch = collect(messages, 200)
  db.batchInsert(batch)      # 1 次 DB 操作
  redis.pipeline(batch)      # 1 次 Redis 操作
```

```go
// 批量写 DB
func batchInsertOrders(orders []*Order) error {
    if len(orders) == 0 {
        return nil
    }
    
    query := "INSERT INTO orders (id, user_id, amount, status) VALUES "
    values := make([]interface{}, 0, len(orders)*4)
    placeholders := make([]string, 0, len(orders))
    
    for _, o := range orders {
        placeholders = append(placeholders, "(?, ?, ?, ?)")
        values = append(values, o.ID, o.UserID, o.Amount, o.Status)
    }
    
    query += strings.Join(placeholders, ",")
    query += " ON DUPLICATE KEY UPDATE status=VALUES(status)"
    
    _, err := db.Exec(query, values...)
    return err
}
```

## 8. 背压（Backpressure）机制？

**回答：**

```
背压：下游处理不过来时反压上游减速

Kafka Pull 模式天然支持背压：
  消费者按自己的速度拉取
  拉取慢 → 数据在 Broker 堆积
  不会压垮消费者

RabbitMQ 的背压：
  Prefetch Count 控制
  channel.Qos(prefetchCount=10)
  → 消费者最多 10 条未 ACK
  → 超过不再推送

应用层背压：
  1. 有界队列
     内存队列满 → 阻塞/丢弃
  
  2. 限流
     消费者内部令牌桶
     控制处理速率

  3. 动态调节
     根据下游负载动态调节消费速度
     下游 DB 负载高 → 减慢消费

  ┌──────┐   背压    ┌──────┐   背压    ┌────┐
  │Source│ ←────── │  MQ  │ ←────── │ DB │
  │减速  │          │缓冲  │          │慢  │
  └──────┘          └──────┘          └────┘
```

## 9. 消费者健康检查？

**回答：**

```
消费者可能假死：
  进程还在但不消费了
  GC 频繁 / 死锁 / 线程卡住

健康检查方案：

1. Kafka 内置心跳：
   heartbeat.interval.ms = 3000
   session.timeout.ms = 45000
   → 心跳超时 → 踢出组 → Rebalance

2. max.poll.interval.ms：
   两次 poll 的最大间隔
   超过 → 认为处理卡住 → Rebalance
   设太小 → 正常处理也被踢

3. 应用层健康检查：
   定期上报消费进度
   定期检查最后消费时间
   异常 → 重启/告警

4. K8s 探针：
   Liveness Probe：进程是否存活
   Readiness Probe：是否准备好消费
   自定义端点返回消费状态
```

## 10. 消息积压面试速答？

**回答：**

```
Q: 消息积压怎么处理？
A: 紧急：扩消费者/扩分区/跳过非关键
   根因：优化消费逻辑/修复故障

Q: 消费者实例数和分区数关系？
A: Kafka 消费者数 ≤ 分区数
   超出的消费者会空闲

Q: 怎么提升消费速度？
A: 增加并行度(分区+消费者)
   批量处理/异步化/减少IO

Q: Lag 持续增大怎么办？
A: 先扩容消费者
   不够就扩分区+消费者
   严重积压转储到临时Topic

Q: 什么是背压？
A: 下游处理不过来反压上游减速
   Kafka Pull 天然背压

Q: 消费者假死怎么办？
A: heartbeat/session.timeout 检测
   max.poll.interval.ms 防卡住
   K8s 探针+应用层监控

Q: 批量消费怎么做？
A: 攒一批消息(如200条)
   批量写DB/Pipeline写Redis

Q: 分区扩容有什么注意事项？
A: 只能增加不能减少
   按key路由会被打乱
   分区不要太多(<10000)
```
