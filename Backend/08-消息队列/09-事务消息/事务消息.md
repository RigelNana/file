# 事务消息

---

## 1. 为什么需要事务消息？

**回答：**

```
问题：本地事务和发消息的一致性

  场景：下单扣减库存 + 发消息通知下游

  方案1：先发消息再执行事务
    发消息成功 → 事务失败 → 下游收到了但上游没成功

  方案2：先执行事务再发消息
    事务成功 → 发消息失败 → 上游成功但下游不知道

  ┌──────────────────────────────────────┐
  │ 无论先后，两步操作不是原子的         │
  │ → 可能只成功一个                     │
  │ → 数据不一致                         │
  └──────────────────────────────────────┘

解决方案：
  1. 事务消息（RocketMQ / Kafka）
  2. 本地消息表（Outbox Pattern）
  3. 事件溯源（Event Sourcing）
```

## 2. RocketMQ 事务消息原理？

**回答：**

```
RocketMQ 事务消息流程：

  ┌──────────┐         ┌──────────┐
  │ Producer │         │  Broker  │
  └────┬─────┘         └────┬─────┘
       │ ① 发送 Half 消息   │
       │────────────────────→│
       │ ② 返回发送结果      │（消费者不可见）
       │←────────────────────│
       │                     │
       │ ③ 执行本地事务      │
       │ ┌────────────────┐  │
       │ │ BEGIN           │  │
       │ │ UPDATE orders..│  │
       │ │ COMMIT          │  │
       │ └────────────────┘  │
       │                     │
       │ ④ 根据事务结果     │
       │   Commit/Rollback  │
       │────────────────────→│
       │                     │ ⑤ Commit: 消息可见
       │                     │    Rollback: 丢弃
       │                     │
       │ ⑥ 回查（未收到④时）│
       │←────────────────────│
       │ 检查本地事务状态    │
       │────────────────────→│
       └─────────────────────┘

Half 消息：
  存在 Broker 但对消费者不可见
  等待 Commit 后才投递

回查机制：
  Broker 定期检查未确认的 Half 消息
  回调 Producer 查询本地事务状态
  → COMMIT / ROLLBACK / UNKNOWN
```

## 3. Kafka 事务消息？

**回答：**

```
Kafka Exactly-Once Semantics：
  支持跨分区原子写入
  典型场景：consume-transform-produce

配置：
  # Producer
  transactional.id = "my-tx-producer"
  enable.idempotence = true
  
  # Consumer
  isolation.level = read_committed

流程：
  1. producer.initTransactions()
  2. producer.beginTransaction()
  3. producer.send(record1)  # 发到 Partition A
  4. producer.send(record2)  # 发到 Partition B
  5. producer.sendOffsetsToTransaction(offsets, group)
  6. producer.commitTransaction()
     或 producer.abortTransaction()

  所有操作原子：全部成功或全部失败

  Consumer(read_committed)：
  只能读到已 commit 的消息
  未 commit 的消息被过滤

注意：
  Kafka 事务 ≠ 数据库事务
  不能把数据库操作包在 Kafka 事务里
  只保证 Kafka 内部的原子性
```

```go
// Go Kafka 事务消息
producer, _ := kafka.NewProducer(&kafka.ConfigMap{
    "bootstrap.servers": "broker:9092",
    "transactional.id":  "my-tx-producer",
})

producer.InitTransactions(nil)

// 开始事务
producer.BeginTransaction()

// 发送多条消息（原子）
producer.Produce(&kafka.Message{
    TopicPartition: kafka.TopicPartition{Topic: &topic1},
    Value:          []byte("msg1"),
}, nil)
producer.Produce(&kafka.Message{
    TopicPartition: kafka.TopicPartition{Topic: &topic2},
    Value:          []byte("msg2"),
}, nil)

// 提交事务
err := producer.CommitTransaction(nil)
if err != nil {
    producer.AbortTransaction(nil)
}
```

## 4. 本地消息表（Outbox Pattern）？

**回答：**

```
本地消息表：
  业务操作 + 消息记录在同一个数据库事务中

  ┌─────────────────────────────────────┐
  │ BEGIN                               │
  │   INSERT INTO orders (...) VALUES   │
  │   INSERT INTO outbox (topic, body)  │
  │ COMMIT                              │
  └─────────────────────────────────────┘
  
  定时任务/CDC 扫描 outbox → 发到 MQ

  ┌──────┐  事务  ┌──────────┐  扫描  ┌────────┐
  │业务DB│ ←────→│ outbox表 │──────→│ MQ     │
  └──────┘       └──────────┘       └────────┘

outbox 表设计：
  id          BIGINT PRIMARY KEY
  topic       VARCHAR(128)
  msg_key     VARCHAR(256)
  msg_body    TEXT
  status      TINYINT (0:待发送 1:已发送 2:失败)
  retry_count INT
  created_at  TIMESTAMP
  sent_at     TIMESTAMP
```

```go
// 本地消息表实现
func CreateOrder(ctx context.Context, order *Order) error {
    tx, _ := db.BeginTx(ctx, nil)
    defer tx.Rollback()
    
    // 1. 创建订单
    _, err := tx.Exec("INSERT INTO orders (id, user_id, amount) VALUES (?, ?, ?)",
        order.ID, order.UserID, order.Amount)
    if err != nil {
        return err
    }
    
    // 2. 写消息表（同一事务）
    msgBody, _ := json.Marshal(order)
    _, err = tx.Exec(
        "INSERT INTO outbox (topic, msg_key, msg_body) VALUES (?, ?, ?)",
        "order-created", order.ID, msgBody)
    if err != nil {
        return err
    }
    
    return tx.Commit()
}

// 定时扫描发送
func ScanOutbox(ctx context.Context) {
    rows, _ := db.Query(
        "SELECT id, topic, msg_key, msg_body FROM outbox WHERE status = 0 LIMIT 100")
    
    for rows.Next() {
        var msg OutboxMsg
        rows.Scan(&msg.ID, &msg.Topic, &msg.Key, &msg.Body)
        
        err := kafkaProducer.Send(msg.Topic, msg.Key, msg.Body)
        if err == nil {
            db.Exec("UPDATE outbox SET status = 1, sent_at = NOW() WHERE id = ?", msg.ID)
        } else {
            db.Exec("UPDATE outbox SET retry_count = retry_count + 1 WHERE id = ?", msg.ID)
        }
    }
}
```

## 5. Outbox + CDC 方案？

**回答：**

```
用 CDC 替代定时扫描：

  ┌──────┐  事务  ┌──────────┐  binlog  ┌────────┐  投递  ┌────┐
  │业务DB│ ←────→│ outbox表 │────────→│Debezium│──────→│ MQ │
  └──────┘       └──────────┘         └────────┘       └────┘

  Debezium 监听 binlog
  outbox 表有新记录 → 自动捕获
  发送到 Kafka Topic

优势：
  - 实时性高（ms 级延迟）
  - 不需要定时扫描（无 DB 轮询压力）
  - 可靠性高（binlog 保证）

Debezium Outbox Event Router：
  专门为 Outbox 模式设计的组件
  自动解析 outbox 表 → 路由到目标 Topic
  自动删除已处理记录

  Debezium 配置：
  - transforms=outbox
  - transforms.outbox.type=
      io.debezium.transforms.outbox.EventRouter
  - table.include.list=public.outbox
```

## 6. Saga 模式与事务消息？

**回答：**

```
Saga：通过事件编排的分布式事务

两种实现方式：

1. 编排式（Choreography）：
   每个服务监听事件 → 执行 → 发布新事件
   
   订单服务 →(订单创建)→ 库存服务
   库存服务 →(库存扣减)→ 支付服务
   支付服务 →(支付成功)→ 通知服务
   
   失败 → 发补偿事件 → 各服务回滚

2. 编排式（Orchestration）：
   中央协调器控制整个流程
   
   ┌──────────┐
   │Saga 协调器│
   │          │→ 订单服务.创建
   │          │→ 库存服务.扣减
   │          │→ 支付服务.扣款
   │          │← 失败 → 补偿
   └──────────┘

与事务消息的关系：
  Saga 中的每一步通过消息通信
  事务消息保证每一步的可靠性
  整个 Saga 通过补偿实现最终一致
```

## 7. 事务消息方案对比？

**回答：**

```
  ┌──────────┬────────────┬────────────┬────────────┐
  │ 方案      │RocketMQ    │ Kafka      │ 本地消息表 │
  │          │ 事务消息   │ 事务       │ + CDC     │
  ├──────────┼────────────┼────────────┼────────────┤
  │ 一致性   │ 最终一致   │ 精确一次   │ 最终一致   │
  │ 范围     │ 单消息+DB  │ Kafka内部  │ DB+MQ     │
  │ 回查     │ ✅         │ ❌         │ ❌         │
  │ DB参与   │ 事务       │ ❌         │ 事务      │
  │ 延迟     │ 低         │ 低         │ 秒级      │
  │ 复杂度   │ 中等       │ 低         │ 中等      │
  │ 侵入性   │ 低         │ 低         │ 需建表    │
  │ 适用     │ 电商交易   │流处理      │ 通用      │
  └──────────┴────────────┴────────────┴────────────┘

选择：
  用 RocketMQ → RocketMQ 事务消息
  用 Kafka 且只需 Kafka 内部原子 → Kafka 事务
  通用方案 → 本地消息表 + CDC
```

## 8. 最终一致性保障？

**回答：**

```
分布式系统中的最终一致性：
  放弃即时一致，保证最终收敛

保障机制：

1. 事务消息/本地消息表：
   保证消息和业务操作一致

2. 消费端幂等：
   重复消费结果相同

3. 重试机制：
   失败自动重试 + 指数退避

4. 补偿机制：
   定时对账发现不一致 → 补偿

5. 人工干预：
   死信队列 + 告警通知 → 人工处理

  ┌──────────────────────────────────┐
  │ 最终一致性 = 可靠发送            │
  │            + 可靠消费(幂等)      │
  │            + 重试补偿            │
  │            + 对账兜底            │
  └──────────────────────────────────┘

CAP 权衡：
  分布式系统无法同时满足 C+A+P
  消息系统选择 AP + 最终一致
  通过补偿达到最终 C
```

## 9. 事务消息实战案例？

**回答：**

```
场景：电商下单

  用户下单 → 扣库存 → 创建订单 → 发通知

  本地消息表方案：

  ┌──────────────────────────────────┐
  │ 订单服务                         │
  │ BEGIN                            │
  │   INSERT order (创建订单)        │
  │   INSERT outbox (库存扣减消息)   │
  │ COMMIT                          │
  └──────────────────────────────────┘
        ↓ CDC/定时扫描
  ┌──────────────────────────────────┐
  │ Kafka: inventory-deduct         │
  └──────────────────────────────────┘
        ↓
  ┌──────────────────────────────────┐
  │ 库存服务（消费者）               │
  │ 幂等扣减库存                     │
  │ 扣减成功 → 发 notification 消息  │
  │ 扣减失败 → 发 order-cancel 消息  │
  └──────────────────────────────────┘
        ↓
  ┌──────────────────────────────────┐
  │ 订单服务（消费 cancel 消息）     │
  │ 取消订单 → 退款（补偿）          │
  └──────────────────────────────────┘

关键点：
  每一步都幂等
  每一步失败都有补偿
  定时对账发现遗漏
```

## 10. 事务消息面试速答？

**回答：**

```
Q: 为什么需要事务消息？
A: 保证本地事务和发消息一致
   两步操作不是原子的

Q: RocketMQ 事务消息原理？
A: Half消息(不可见) → 执行事务
   → Commit/Rollback
   未确认 → Broker 回查

Q: Kafka 事务和 RocketMQ 事务区别？
A: Kafka:只保证Kafka内部原子性
   RocketMQ:保证本地事务+消息一致

Q: 本地消息表是什么？
A: 业务+消息记录在同一DB事务
   定时扫描/CDC发到MQ

Q: Outbox + CDC 优势？
A: 实时(ms级) + 无DB轮询压力
   + 可靠性高(binlog保证)

Q: Saga 模式是什么？
A: 通过事件编排的分布式事务
   每步操作有对应的补偿操作

Q: 怎么保证最终一致？
A: 可靠发送 + 幂等消费
   + 重试补偿 + 对账兜底

Q: 事务消息方案怎么选？
A: RocketMQ → 用其事务消息
   通用 → 本地消息表+CDC
   Kafka内部 → Kafka事务
```
