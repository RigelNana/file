# 消息可靠性保障

---

## 1. 消息丢失的三个环节？

**回答：**

```
消息从生产到消费经过三个环节，每个都可能丢失：

  ┌──────────┐    ┌──────────┐    ┌──────────┐
  │ Producer │───→│  Broker  │───→│ Consumer │
  │          │ ①  │          │ ②  │          │ ③
  └──────────┘    └──────────┘    └──────────┘

① 生产端丢失：
   网络故障、发送失败未重试
   → acks=all + 重试

② Broker 端丢失：
   节点宕机、磁盘故障
   → 多副本 + min.insync.replicas

③ 消费端丢失：
   自动提交 offset 后处理失败
   → 手动提交 offset，处理成功再提交
```

## 2. Kafka 端到端不丢消息方案？

**回答：**

```
生产端配置：
  acks = all                    # 所有 ISR 确认
  retries = Integer.MAX_VALUE   # 无限重试
  enable.idempotence = true     # 幂等，防重复
  max.in.flight.requests.per.connection = 5

Broker 端配置：
  replication.factor = 3        # 3 副本
  min.insync.replicas = 2       # 至少 2 个同步
  unclean.leader.election.enable = false  # 禁止非 ISR 选举

消费端：
  enable.auto.commit = false    # 关闭自动提交
  处理成功 → 手动 commitSync()
  处理失败 → 不提交，重试或发死信队列

  ┌──────────────────────────────────────┐
  │ 可靠性三板斧：                       │
  │ 1. 生产: acks=all + 幂等 + 重试     │
  │ 2. Broker: 3副本 + min.isr=2        │
  │ 3. 消费: 手动提交 + 幂等处理        │
  └──────────────────────────────────────┘
```

## 3. RabbitMQ 不丢消息方案？

**回答：**

```
生产端：
  Publisher Confirm 模式
  channel.ConfirmSelect()
  → 等待 Broker ACK
  → NACK 或超时 → 重发

  备选：事务模式（性能差，不推荐）
  channel.TxSelect()
  channel.TxCommit()

Broker 端：
  Exchange: durable = true
  Queue: durable = true
  Message: deliveryMode = 2（持久化）

  Quorum Queue（3.8+）：
  Raft 多数确认写入
  比镜像队列更可靠

消费端：
  autoAck = false
  处理完成 → BasicAck
  处理失败 → BasicNack(requeue=true) 重新入队
  多次失败 → BasicNack(requeue=false) → 死信队列

  ┌──────────────────────────────────────┐
  │ RabbitMQ 可靠性三板斧：              │
  │ 1. 生产: Publisher Confirm          │
  │ 2. Broker: 持久化 + Quorum Queue   │
  │ 3. 消费: 手动 ACK + 重试 + 死信     │
  └──────────────────────────────────────┘
```

## 4. 消息投递语义？

**回答：**

```
三种投递语义：

1. At Most Once（最多一次）：
   消息可能丢，但不重复
   → 先提交 offset，再处理
   处理失败 → 消息丢了

2. At Least Once（至少一次）— 最常用：
   消息不丢，但可能重复
   → 先处理，再提交 offset
   提交 offset 失败 → 重复消费

3. Exactly Once（恰好一次）：
   消息不丢不重复
   → 生产端幂等 + 消费端幂等
   → Kafka 事务（跨分区原子写入）

实际选择：
  大多数场景用 At Least Once + 消费端幂等
  
  ┌──────────────────────────────────┐
  │ At Least Once + 幂等 ≈ Exactly  │
  │ 业界通用方案                     │
  └──────────────────────────────────┘
```

## 5. 消费端幂等方案？

**回答：**

```
为什么需要幂等？
  重试/Rebalance/网络闪断 → 重复消费
  相同消息处理多次 → 结果应该一样

方案1：唯一 ID + 去重表
  每条消息带唯一 messageId
  消费前查去重表 → 已存在则跳过
  
  INSERT INTO msg_consumed(msg_id, ...) 
    VALUES(?, ...)
  -- 唯一索引 msg_id → 重复插入失败

方案2：Redis SETNX 去重
  SET msg:{id} 1 NX EX 86400
  返回 true → 首次消费
  返回 false → 重复消费

方案3：数据库乐观锁
  UPDATE account SET balance = balance + 100,
    version = version + 1
  WHERE id = 1001 AND version = 5
  影响行数 = 0 → 已处理

方案4：业务本身幂等
  SET balance = 100（而非 balance += 100）
  UPSERT / ON DUPLICATE KEY UPDATE
```

```go
// Redis 去重幂等消费
func ConsumeIdempotent(ctx context.Context, msg *Message) error {
    dedupeKey := fmt.Sprintf("msg:consumed:%s", msg.ID)
    
    // 去重检查
    ok, _ := rdb.SetNX(ctx, dedupeKey, "1", 24*time.Hour).Result()
    if !ok {
        return nil // 已消费，跳过
    }
    
    // 处理业务
    if err := processMessage(msg); err != nil {
        rdb.Del(ctx, dedupeKey) // 处理失败，删除标记
        return err
    }
    return nil
}
```

## 6. 本地消息表？

**回答：**

```
本地消息表：保证本地事务和发消息的一致性

  ┌─────────────────────────────────┐
  │ 同一个数据库事务内：             │
  │  1. 执行业务操作                 │
  │  2. 写入本地消息表               │
  │  → 事务commit                   │
  └─────────────────────────────────┘
        ↓
  ┌─────────────────────────────────┐
  │ 定时任务/CDC：                   │
  │  3. 扫描未发送的消息             │
  │  4. 发送到 MQ                   │
  │  5. 标记为已发送                 │
  └─────────────────────────────────┘

消息表结构：
  CREATE TABLE outbox (
      id         BIGINT PRIMARY KEY,
      topic      VARCHAR(128),
      msg_key    VARCHAR(256),
      msg_body   TEXT,
      status     TINYINT DEFAULT 0,  -- 0:待发送 1:已发送
      retry_count INT DEFAULT 0,
      created_at  TIMESTAMP,
      updated_at  TIMESTAMP
  );

优势：
  事务一致性强
  实现简单

劣势：
  侵入业务表（耦合）
  定时轮询有延迟
  → 可用 CDC (Debezium) 替代轮询
```

## 7. 消息回溯消费？

**回答：**

```
消息回溯：重新消费历史消息

Kafka 支持：
  消息在保留期内都可回溯
  通过 seek(offset) 指定位置消费
  
  kafka-consumer-groups.sh \
    --reset-offsets \
    --to-earliest \        # 从头消费
    --to-datetime 2024-01-01T00:00:00.000 \
    --topic orders \
    --group my-group \
    --execute

场景：
  1. Bug 修复后重新处理
  2. 新增消费者从头消费
  3. 数据修复

RabbitMQ 不支持：
  消息消费后就删除了
  → 需要额外方案（如日志归档）

注意：
  回溯消费必须保证幂等
  大量数据回溯可能影响集群性能
  可创建新 Consumer Group 回溯
```

## 8. 消息补偿机制？

**回答：**

```
补偿场景：
  消息丢失或处理失败需要恢复

1. 定时扫描补偿：
   定时检查业务状态
   发现未完成 → 重新发消息或直接处理
   
   如：订单 30 分钟未支付 → 扫描关闭

2. 对账补偿：
   每日对比上下游数据
   发现不一致 → 补发消息或修复数据

3. 手动补偿：
   死信队列中的消息
   人工审核后重新投递

4. 反查补偿：
   消费者处理前查源系统确认
   如：收到扣库存消息 → 查订单服务确认

  ┌──────────────────────────────┐
  │ 补偿 = 兜底机制              │
  │ 消息系统做到 At Least Once   │
  │ + 幂等 + 补偿 ≈ 最终一致    │
  └──────────────────────────────┘
```

## 9. 端到端可靠性方案设计？

**回答：**

```
完整可靠性方案：

  ┌──────────┐                     ┌──────────┐
  │上游服务  │                     │下游服务  │
  │          │                     │          │
  │ 1.业务DB │                     │ 6.幂等   │
  │ 2.消息表 │                     │   消费   │
  │   事务   │                     │ 7.手动   │
  └────┬─────┘                     │   ACK    │
       │                           └────┬─────┘
  ┌────┴─────┐                          │
  │ 3.定时   │     ┌──────────┐         │
  │   扫描   │────→│ MQ       │────────→│
  │ 4.发送MQ │     │ 持久化   │         │
  │ 5.标记   │     │ 多副本   │         │
  └──────────┘     └──────────┘         │
                        │               │
                   ┌────┴─────┐    ┌────┴─────┐
                   │ 死信队列 │    │ 对账补偿 │
                   └──────────┘    └──────────┘

关键组件：
  上游：本地消息表 + 定时扫描
  MQ：持久化 + 多副本 + 确认机制
  下游：幂等消费 + 手动 ACK
  兜底：死信队列 + 定时对账
```

## 10. 消息可靠性面试速答？

**回答：**

```
Q: 消息可能在哪丢失？
A: 生产端(网络) + Broker(宕机)
   + 消费端(提交后处理失败)

Q: Kafka 怎么不丢消息？
A: acks=all + min.isr=2 + 手动提交
   + 生产端重试 + 消费端幂等

Q: RabbitMQ 怎么不丢消息？
A: Publisher Confirm + 持久化
   + Quorum Queue + 手动ACK

Q: 三种投递语义？
A: At Most Once/At Least Once/
   Exactly Once
   常用 At Least Once + 幂等

Q: 幂等怎么做？
A: 唯一ID+去重表 / Redis SETNX
   / 乐观锁 / 业务幂等

Q: 本地消息表是什么？
A: 业务和消息在同一事务中写入
   定时扫描发送到 MQ

Q: 消息处理失败怎么办？
A: 重试N次 → 死信队列
   定时对账补偿兜底

Q: 消息回溯怎么做？
A: Kafka 支持 seek 到指定 offset
   RabbitMQ 不支持(消费即删除)
```
