## 2. Topic 设计规范？

**回答：**

```
命名规范：
  {环境}.{业务域}.{事件类型}.{版本}
  
  prod.order.created.v1
  prod.payment.completed.v1
  staging.user.registered.v1

分区数确定：
  目标吞吐 / 单分区吞吐 = 分区数
  
  例：目标 10万 msg/s，单分区 1万 msg/s
  → 10个分区（预留50%余量 → 15个）
  
  注意：分区只能增不能减
  建议先少后多，按需扩展

副本因子：
  生产环境至少 3 副本
  min.insync.replicas = 副本数 - 1

消息大小：
  默认 max.message.bytes = 1MB
  大消息 → 压缩 / 拆分 / 外部存储引用

Key 设计：
  保证同一业务实体到同一分区
  order_id → 订单相关消息有序
  user_id → 用户相关消息有序
  避免热点 Key（大客户单独处理）
```

## 3. 消息 Schema 管理？

**回答：**

```
Schema 演进问题：
  消息格式变更 → 消费者能否兼容？

序列化方案：

  ┌──────────┬───────┬────────┬─────────┐
  │ 方案      │ 可读  │ Schema │ 性能    │
  ├──────────┼───────┼────────┼─────────┤
  │ JSON     │ ✅    │ 弱     │ 一般    │
  │ Protobuf │ ❌    │ 强     │ 高      │
  │ Avro     │ ❌    │ 强     │ 高      │
  │ Thrift   │ ❌    │ 强     │ 高      │
  └──────────┴───────┴────────┴─────────┘

Schema Registry：
  集中管理 Schema 版本
  Producer 注册 → Consumer 获取
  支持兼容性检查

  兼容性策略：
  - BACKWARD：新Schema能读旧数据（可加可选字段）
  - FORWARD：旧Schema能读新数据（可减可选字段）
  - FULL：双向兼容
  - NONE：不检查

最佳实践：
  - 使用 Protobuf / Avro
  - 部署 Schema Registry
  - CI 中检查兼容性
  - 字段只加不删（BACKWARD 兼容）
  - 消息包含版本号
```

## 4. 消息监控体系？

**回答：**

```
核心监控指标：

  ┌─────────────────┬───────────────────────┐
  │ 类别            │ 监控项                │
  ├─────────────────┼───────────────────────┤
  │ Broker          │ 存活状态              │
  │                 │ ISR 缩减次数          │
  │                 │ Under-replicated分区  │
  │                 │ 磁盘使用率            │
  │                 │ 网络流量 in/out       │
  ├─────────────────┼───────────────────────┤
  │ Producer        │ 发送速率 records/s    │
  │                 │ 发送延迟 P99          │
  │                 │ 发送失败/重试次数     │
  │                 │ batch 大小            │
  ├─────────────────┼───────────────────────┤
  │ Consumer        │ 消费速率 records/s    │
  │                 │ Consumer Lag          │
  │                 │ Rebalance 次数        │
  │                 │ 处理延迟             │
  ├─────────────────┼───────────────────────┤
  │ Topic           │ 消息量/字节数         │
  │                 │ 分区 Leader 分布      │
  │                 │ 消息大小分布          │
  └─────────────────┴───────────────────────┘

监控工具栈：
  Kafka Exporter → Prometheus → Grafana
  Burrow（LinkedIn 开源，专注 Lag 监控）
  Kafka Manager / AKHQ / Kafka UI

告警规则：
  Consumer Lag > 10000 → Warning
  Consumer Lag > 100000 → Critical
  Under-replicated > 0 → Critical
  ISR Shrink > 0 → Warning
  Broker 离线 → Critical
  磁盘 > 80% → Warning
```

```go
// Go 获取 Consumer Lag
func GetConsumerLag(adminClient *kafka.AdminClient, group, topic string) {
    // 获取 committed offsets
    committed, _ := adminClient.ListConsumerGroupOffsets(
        context.Background(),
        []kafka.ConsumerGroupTopicPartitions{{Group: group}},
    )
    
    // 获取 topic 最新 offsets
    meta, _ := adminClient.GetMetadata(&topic, false, 5000)
    
    for _, partition := range meta.Topics[topic].Partitions {
        low, high, _ := consumer.QueryWatermarkOffsets(topic, partition.ID, 5000)
        
        var committedOffset int64
        // 从 committed 中查找对应分区
        lag := high - committedOffset
        
        log.Printf("Topic=%s Partition=%d Lag=%d (Low=%d High=%d)",
            topic, partition.ID, lag, low, high)
    }
}
```

## 5. 容量规划与扩缩容？

**回答：**

```
容量规划：

  维度1：存储
    日消息量 × 消息大小 × 副本数 × 保留天数
    = 1亿 × 1KB × 3 × 7 = 2.1TB
    预留 30% Buffer → 2.73TB

  维度2：网络
    峰值 QPS × 消息大小 × (1 + 副本数-1)
    = 10万 × 1KB × 3 = 300MB/s
    10Gbps 网卡利用率约 25%

  维度3：分区
    目标吞吐 / 单分区吞吐
    注意 Broker 文件句柄数限制

扩容操作：

  增加 Broker：
  1. 部署新 Broker 节点
  2. 使用 kafka-reassign-partitions 重新分配
  3. 验证副本复制完成
  4. 淘汰旧节点（可选）

  增加分区：
  kafka-topics.sh --alter --topic test --partitions 12
  ⚠️ 分区增加后 Key 路由会变化
  
  缩容（谨慎）：
  1. 分区不可减少
  2. 移除 Broker 需先迁移分区
  3. 迁移期间监控复制进度和 Lag

热点分区处理：
  某个分区数据量远大于其他
  → Key 修改（加随机后缀）
  → 单独 Topic 隔离
```

## 6. 多集群与跨机房部署？

**回答：**

```
多集群方案：

  ┌─────────┐  MirrorMaker  ┌─────────┐
  │ 集群 A  │──────────────→│ 集群 B  │
  │ (主)    │  ← 异步复制   │ (灾备)  │
  └─────────┘               └─────────┘

  MirrorMaker 2 (MM2)：
  基于 Kafka Connect
  Topic 名自动加前缀（source.topic1）
  offset 自动映射
  支持 ACL 同步

跨机房架构：
  
  方案1：异步复制（常用）
    一地写入 → MM2复制到另一地
    RPO > 0（可能丢消息）
    延迟 = 网络延迟

  方案2：同城双活
    两个集群各写各读
    业务路由到其中一个
    另一个热备

  方案3：Stretch Cluster
    同一集群跨机房部署
    ISR 包含两个机房的副本
    min.insync.replicas=2
    强一致但延迟高

选择：
  同城容灾 → Stretch Cluster
  异地容灾 → MM2 异步复制
  全球部署 → 多集群 + 区域路由
```

## 7. 消息迁移与升级？

**回答：**

```
Kafka 版本升级：
  支持滚动升级（Rolling Upgrade）

  步骤：
  1. 备份配置
  2. 更新 inter.broker.protocol.version 为旧版本
  3. 逐个 Broker 停机 → 替换二进制 → 启动
  4. 验证集群稳定
  5. 升级 inter.broker.protocol.version 到新版本
  6. 升级 log.message.format.version

  注意：
  先升 Broker，再升客户端
  客户端向后兼容老版 Broker

MQ 迁移（如 RabbitMQ → Kafka）：

  ┌──────────┐   ┌──────────┐   ┌──────────┐
  │ Producer │──→│ RabbitMQ │──→│ Consumer │
  └──────────┘   └──────────┘   └──────────┘
       ↓ 双写
  ┌──────────┐   ┌──────────┐   ┌──────────┐
  │ Producer │──→│  Kafka   │──→│ Consumer │
  └──────────┘   └──────────┘   └──────────┘

  阶段：
  1. 双写期：Producer 同时写两个 MQ
  2. 验证期：对比两边消费结果一致
  3. 切换期：停旧 MQ 消费，只用 Kafka
  4. 清理期：移除旧 MQ 代码和配置
```

## 8. 常见故障排查？

**回答：**

```
故障1：消息消费延迟
  排查：
  - kafka-consumer-groups.sh --describe --group xxx
  - 看 Lag 和 消费速率
  处理：
  - 增加消费者 / 分区
  - 优化消费逻辑
  - 检查是否频繁 Rebalance

故障2：Broker OOM
  排查：
  - jstack / jmap 分析
  - 检查 page cache 使用
  处理：
  - 调整 JVM 堆大小（6-8GB）
  - 减少 Topic / 分区数
  - 启用压缩减少内存占用

故障3：消息丢失
  排查：
  - 检查 acks 配置是否 = all
  - 检查 min.insync.replicas
  - 检查 Consumer 是否自动提交
  - 检查 unclean.leader.election
  处理：
  - acks=all + min.isr=2 + 手动提交

故障4：Rebalance 频繁
  排查：
  - 检查 heartbeat/session timeout
  - 消费者处理时间 > max.poll.interval
  处理：
  - 增大 max.poll.interval.ms
  - 减少 max.poll.records
  - 使用 Static Membership

故障5：磁盘满
  排查：
  - du -sh /data/kafka-logs/*
  处理：
  - 调整 log.retention.hours
  - 清理过期 Topic
  - 扩容磁盘
```

## 9. MQ 与微服务最佳实践？

**回答：**

```
微服务中的 MQ 使用场景：

  1. 异步解耦：订单→库存→支付
  2. 事件驱动：用户注册→发邮件→加积分
  3. 数据同步：主服务→数据仓库
  4. CQRS：写模型→事件→读模型

封装建议：

  ┌───────────────────────────────┐
  │ Application Layer            │
  │   EventPublisher             │
  │   EventSubscriber            │
  ├───────────────────────────────┤
  │ Infrastructure Layer         │
  │   KafkaPublisher             │
  │   KafkaSubscriber            │
  │   RabbitMQPublisher          │
  │   RabbitMQSubscriber         │
  └───────────────────────────────┘

  业务层面向接口编程
  基础设施层实现具体 MQ
  便于 MQ 切换
```

```go
// 统一消息接口
type EventPublisher interface {
    Publish(ctx context.Context, topic string, event Event) error
}

type EventSubscriber interface {
    Subscribe(ctx context.Context, topic string, handler EventHandler) error
}

type Event struct {
    ID        string            `json:"id"`
    Type      string            `json:"type"`
    Source    string            `json:"source"`
    Time      time.Time         `json:"time"`
    Data      json.RawMessage   `json:"data"`
    Metadata  map[string]string `json:"metadata"`
}

type EventHandler func(ctx context.Context, event Event) error

// Kafka 实现
type KafkaPublisher struct {
    producer *kafka.Producer
}

func (p *KafkaPublisher) Publish(ctx context.Context, topic string, event Event) error {
    data, _ := json.Marshal(event)
    
    deliveryChan := make(chan kafka.Event)
    err := p.producer.Produce(&kafka.Message{
        TopicPartition: kafka.TopicPartition{Topic: &topic},
        Key:            []byte(event.ID),
        Value:          data,
        Headers: []kafka.Header{
            {Key: "event-type", Value: []byte(event.Type)},
            {Key: "trace-id", Value: []byte(ctx.Value("traceID").(string))},
        },
    }, deliveryChan)
    if err != nil {
        return err
    }
    
    e := <-deliveryChan
    m := e.(*kafka.Message)
    return m.TopicPartition.Error
}
```

## 10. 消息队列生产实践面试速答？

**回答：**

```
Q: Kafka 生产环境关键配置？
A: acks=all + min.isr=2 + 手动提交
   + 幂等Producer + 3副本

Q: 如何确定分区数？
A: 目标吞吐/单分区吞吐
   预留余量 只增不减

Q: Schema 怎么管理？
A: Protobuf/Avro + Schema Registry
   BACKWARD 兼容 字段只加不删

Q: Consumer Lag 怎么监控？
A: Kafka Exporter + Prometheus
   + Burrow + Grafana 仪表盘
   Lag > 10000 告警

Q: 容量怎么规划？
A: 消息量×大小×副本×保留天数
   预留30%Buffer

Q: 跨机房怎么部署？
A: MM2 异步复制（常用）

Q: MQ 迁移怎么做？
A: 双写→验证→切换→清理
   先升 Broker 再升客户端

Q: 微服务中 MQ 怎么用？
A: 面向接口封装 EventPublisher
   业务层不感知具体 MQ 实现
   场景：解耦/事件驱动/CQRS
```
