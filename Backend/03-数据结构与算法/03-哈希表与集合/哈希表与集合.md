# 哈希表与集合

---

## 1. 哈希表的底层实现原理是什么？

**回答：**

哈希表通过**哈希函数**将 key 映射到数组下标，实现 O(1) 平均查找。

```
┌─────────────────────────────────────┐
│  key → hash(key) → index → bucket  │
│                                     │
│  "alice" → 0x7A3F → 3 → bucket[3]  │
│  "bob"   → 0x1B2C → 7 → bucket[7]  │
└─────────────────────────────────────┘

桶数组：
[0] → nil
[1] → nil
[2] → nil
[3] → ("alice", 100)
[4] → nil
[5] → nil
[6] → nil
[7] → ("bob", 200)
```

| 操作 | 平均 | 最坏 |
|------|------|------|
| 查找 | O(1) | O(n) |
| 插入 | O(1) | O(n) |
| 删除 | O(1) | O(n) |

最坏情况是所有 key 哈希到同一个桶（退化为链表）。好的哈希函数 + 合适的负载因子可保证平均 O(1)。

## 2. 哈希冲突的解决方案有哪些？

**回答：**

**1. 链地址法（Separate Chaining）：**

```
bucket[3] → [alice:100] → [charlie:300] → nil
bucket[7] → [bob:200] → nil

优点：实现简单，负载因子可 > 1
缺点：链表长时退化 O(n)
优化：链表长度超阈值转红黑树（Java HashMap 阈值=8）
```

**2. 开放寻址法（Open Addressing）：**

```
线性探测：h(k), h(k)+1, h(k)+2, ...
二次探测：h(k), h(k)+1², h(k)+2², ...
双重哈希：h(k), h(k)+h2(k), h(k)+2*h2(k), ...

优点：缓存友好（数据都在数组中）
缺点：负载因子必须 < 1，删除需要标记（墓碑）
```

**各语言 Map 的实现：**

| 语言 | 实现方式 | 特点 |
|------|---------|------|
| Go | 链地址法 + 溢出桶 | 每桶 8 个 KV，渐进式扩容 |
| Java | 链地址法 + 红黑树 | 链表长度 >8 转红黑树 |
| Python | 开放寻址（二次探测）| 负载因子 2/3 触发扩容 |
| C++ | 链地址法 | unordered_map |

## 3. Go map 的底层实现是怎样的？

**回答：**

```
hmap 结构：
┌──────────────┐
│ count         │  元素个数
│ B            │  桶数 = 2^B
│ buckets      │──→ [bucket0][bucket1]...[bucket 2^B-1]
│ oldbuckets   │──→ 扩容时旧桶（渐进式迁移）
│ nevacuate    │  迁移进度
└──────────────┘

每个 bucket（bmap）：
┌──────────────────────────────┐
│ tophash [8]uint8             │  高8位哈希，快速比较
│ keys    [8]keyType           │  8个key
│ values  [8]valueType         │  8个value
│ overflow *bmap               │  溢出桶指针
└──────────────────────────────┘
```

**关键特性：**

- **负载因子 > 6.5** 时翻倍扩容
- **溢出桶过多** 时等量扩容（整理碎片）
- **渐进式迁移**：每次访问时迁移 1-2 个桶
- **并发不安全**：读写并发会 fatal（检测到 hashWriting 标志）
- **遍历随机**：故意随机起始桶+桶内偏移

## 4. 什么是布隆过滤器（Bloom Filter）？

**回答：**

布隆过滤器是**概率型数据结构**，用于判断元素**可能存在**或**一定不存在**。

```
插入 "hello"：
k 个哈希函数分别计算位置，置位为 1

  h1("hello")=2  h2("hello")=5  h3("hello")=9
       ↓              ↓              ↓
  [0][0][1][0][0][1][0][0][0][1][0][0]

查询 "world"：
  h1("world")=1  h2("world")=5  h3("world")=7
       ↓              ↓              ↓
  [0][0][1][0][0][1][0][0][0][1][0][0]
   位1=0 → 一定不存在！

查询 "test"：
  所有位都是1 → 可能存在（可能误判）
```

| 特性 | 说明 |
|------|------|
| 假阳性 | 可能判断存在实际不存在 |
| 假阴性 | 不会发生（判断不存在则一定不存在）|
| 空间 | 远小于存原始数据 |
| 删除 | 标准布隆过滤器不支持，计数布隆过滤器可以 |

**应用场景：**
- Redis 缓存穿透防护
- 爬虫 URL 去重
- 垃圾邮件过滤
- 数据库查询优化（HBase、LevelDB）

## 5. HashMap 和 TreeMap / HashSet 和 TreeSet 的区别？

**回答：**

| 特性 | HashMap/HashSet | TreeMap/TreeSet |
|------|----------------|-----------------|
| 底层 | 哈希表 | 红黑树 |
| 查找 | O(1) 平均 | O(log n) |
| 有序 | 无序 | 键有序（自然序或 Comparator）|
| null key | 允许1个 | 不允许 |
| 遍历 | 无序 | 升序 |

**Go 语言对应：**

```go
// Go 只有 map（哈希表），没有 TreeMap
m := map[string]int{"b": 2, "a": 1, "c": 3}

// 需要有序遍历时手动排序 key
keys := make([]string, 0, len(m))
for k := range m {
    keys = append(keys, k)
}
sort.Strings(keys)
for _, k := range keys {
    fmt.Println(k, m[k])
}
```

**Python 对应：**

```python
# dict 从 3.7+ 保持插入顺序（不是排序序）
d = {"b": 2, "a": 1, "c": 3}

# 排序遍历
for k in sorted(d):
    print(k, d[k])

# set：无序不重复集合
s = {1, 2, 3}

# sortedcontainers 库提供 SortedDict, SortedSet
```

## 6. 如何设计一个好的哈希函数？

**回答：**

**好的哈希函数特征：**
- 确定性：同输入同输出
- 均匀分布：减少冲突
- 高效计算：O(1) 时间
- 雪崩效应：输入微小变化 → 输出巨大变化

**常见哈希函数：**

```
字符串哈希（多项式滚动哈希）：
hash = 0
for each char c in s:
    hash = hash * 31 + c    // 31 是常用质数

整数哈希（位混合）：
// Java HashMap 扰动函数
h = key.hashCode()
h = h ^ (h >>> 16)  // 高16位异或低16位

// Go map 哈希
基于 AES 指令（amd64）或 memhash
```

**为什么哈希表大小用 2 的幂？**

```
// 取模优化：n % 2^k = n & (2^k - 1)
index = hash & (len(buckets) - 1)  // 位运算比取模快

// 缺点：低位相同的哈希值会冲突
// 解决：扰动函数（如 Java 的 h ^ h>>>16）
```

## 7. Go sync.Map 的实现原理和适用场景？

**回答：**

```
sync.Map 结构：
┌────────────────────────────────┐
│ read  (atomic.Value)           │ → readOnly{m: map, amended: bool}
│ dirty (map + mutex)            │ → map[key]*entry
│ misses int                     │ → read 未命中计数
└────────────────────────────────┘

读操作：
1. 先读 read map（无锁，atomic）
2. 命中 → 直接返回
3. 未命中 → 加锁读 dirty

写操作：
1. read 中存在 → CAS 更新（无锁）
2. read 中不存在 → 加锁写 dirty

提升（promotion）：
misses >= len(dirty) 时，dirty 提升为 read
```

**适用场景：**

| 场景 | sync.Map | map + RWMutex |
|------|----------|--------------|
| 读多写少 | ✅ 优秀 | 一般 |
| 读写均衡 | 一般 | ✅ 更好 |
| key 稳定 | ✅ 优秀 | 一般 |
| key 频繁新增 | 较差 | ✅ 更好 |

## 8. 一致性哈希的原理和应用？

**回答：**

```
一致性哈希环：
        0
       / \
  node1   node2
     |     |
     ↓     ↓
  ┌──────────────┐
  │  0° ——→ 90°  │  node1 负责
  │ 90° ——→ 180° │  node2 负责
  │180° ——→ 270° │  node3 负责
  │270° ——→ 360° │  node1 负责
  └──────────────┘

增加节点 node4 在 135°：
只需迁移 90°~135° 的数据（从 node2 到 node4）
```

**虚拟节点解决数据倾斜：**

```go
type ConsistentHash struct {
    ring     map[uint32]string  // 哈希值 → 节点名
    sorted   []uint32           // 排序的哈希值
    replicas int                // 每个节点的虚拟节点数
}

func (c *ConsistentHash) Add(node string) {
    for i := 0; i < c.replicas; i++ {
        hash := hashFunc(fmt.Sprintf("%s#%d", node, i))
        c.ring[hash] = node
        c.sorted = append(c.sorted, hash)
    }
    sort.Slice(c.sorted, func(i, j int) bool {
        return c.sorted[i] < c.sorted[j]
    })
}

func (c *ConsistentHash) Get(key string) string {
    hash := hashFunc(key)
    idx := sort.Search(len(c.sorted), func(i int) bool {
        return c.sorted[i] >= hash
    })
    if idx >= len(c.sorted) {
        idx = 0  // 环绕
    }
    return c.ring[c.sorted[idx]]
}
```

**应用：** Redis Cluster 哈希槽（16384个）、Nginx 一致性哈希负载均衡、分布式缓存。

## 9. 如何实现 O(1) 的 LFU 缓存？

**回答：**

LFU（Least Frequently Used）：淘汰使用频率最低的数据。

```
结构：两个哈希表 + 频率链表
┌──────────┐
│ keyMap    │  key → node
├──────────┤
│ freqMap   │  freq → 双向链表
│  freq=1  │──→ [C] ↔ [D]    ← minFreq 指向这里
│  freq=2  │──→ [B]
│  freq=3  │──→ [A]
└──────────┘
```

```go
type LFUCache struct {
    capacity int
    minFreq  int
    keyMap   map[int]*Node          // key → node
    freqMap  map[int]*DoublyList    // freq → list
}

type Node struct {
    key, val, freq int
    prev, next     *Node
}

func (l *LFUCache) Get(key int) int {
    node, ok := l.keyMap[key]
    if !ok { return -1 }
    l.increaseFreq(node) // 频率+1，移到新频率链表
    return node.val
}

func (l *LFUCache) Put(key, value int) {
    if node, ok := l.keyMap[key]; ok {
        node.val = value
        l.increaseFreq(node)
        return
    }
    if len(l.keyMap) >= l.capacity {
        // 删除 minFreq 链表的尾部节点
        list := l.freqMap[l.minFreq]
        deleted := list.RemoveTail()
        delete(l.keyMap, deleted.key)
    }
    node := &Node{key: key, val: value, freq: 1}
    l.keyMap[key] = node
    l.addToFreqList(1, node)
    l.minFreq = 1
}
```

**关键点：** `minFreq` 变量跟踪当前最小频率，保证 O(1) 找到淘汰目标。

## 10. 哈希表与集合面试速答？

**回答：**

```
Q: 哈希表的平均和最坏时间复杂度？
A: 平均 O(1)，最坏 O(n)（所有 key 冲突）

Q: Go map 是并发安全的吗？
A: 不是，并发读写会 fatal error，需要 sync.Map 或加锁

Q: Python dict 从哪个版本保持插入顺序？
A: CPython 3.6 实现，3.7 正式保证

Q: 负载因子是什么？
A: 元素数/桶数，衡量哈希表满的程度。Go map 阈值 6.5，Java HashMap 阈值 0.75

Q: 为什么 Java HashMap 链表长度 > 8 转红黑树？
A: 泊松分布下链表长度 > 8 概率极低（亿分之六），转红黑树保证最坏 O(log n)

Q: 布隆过滤器能删除元素吗？
A: 标准的不能，计数布隆过滤器（CBF）可以

Q: set 和 map 的关系？
A: set 本质是只有 key 没有 value 的 map

Q: 哈希表扩容为什么选 2 倍？
A: 2 的幂方便位运算取模；也有用质数大小的（如 C++ 某些实现）

Q: Go map 遍历为什么是随机顺序？
A: 故意的，runtime 随机化起始位置，防止依赖遍历顺序

Q: 一致性哈希和普通哈希的区别？
A: 一致性哈希增删节点只影响相邻数据，普通哈希会导致大量数据重新分布
```
