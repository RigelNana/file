# 虚拟化与容器原理

---

## 1. 虚拟化技术分类？

**回答：**

```
虚拟化层次：
  ┌──────────────┬──────────────────────────────┐
  │ 类型          │ 说明                          │
  ├──────────────┼──────────────────────────────┤
  │ 硬件虚拟化    │ 整个硬件虚拟（VMware/KVM/Xen）│
  │ 操作系统虚拟化│ 共享内核（容器/Docker/LXC）   │
  │ 语言级虚拟化  │ JVM / Python VM              │
  └──────────────┴──────────────────────────────┘

  虚拟机 vs 容器：
  ┌──────────────┬──────────────┬──────────────┐
  │ 对比          │ 虚拟机        │ 容器          │
  ├──────────────┼──────────────┼──────────────┤
  │ 隔离级别      │ 硬件级        │ 进程级        │
  │ 启动速度      │ 分钟级        │ 秒级          │
  │ 资源开销      │ GB 级         │ MB 级         │
  │ 性能损耗      │ 5-20%        │ <2%           │
  │ 安全隔离      │ 强            │ 较弱          │
  │ 内核          │ 独立内核      │ 共享宿主内核  │
  │ 典型代表      │ VMware/KVM   │ Docker/LXC   │
  └──────────────┴──────────────┴──────────────┘
```

## 2. Linux Namespace？

**回答：**

```
Namespace：内核级资源隔离机制
  让每个容器看到独立的系统视图

  ┌──────────────┬──────────────────────────────┐
  │ Namespace     │ 隔离内容                      │
  ├──────────────┼──────────────────────────────┤
  │ PID          │ 进程 ID（容器内 PID 1）        │
  │ NET          │ 网络栈（独立 IP/端口/路由）    │
  │ MNT          │ 挂载点（独立文件系统视图）     │
  │ UTS          │ 主机名和域名                   │
  │ IPC          │ 进程间通信（信号量/消息队列）  │
  │ USER         │ 用户和组 ID                    │
  │ CGROUP       │ cgroup 根目录视图              │
  │ TIME         │ 系统时间（Linux 5.6+）         │
  └──────────────┴──────────────────────────────┘

  创建方式：
  - clone(CLONE_NEWPID | CLONE_NEWNET | ...)
  - unshare(CLONE_NEWNS)
  - setns(fd, nstype)

  容器本质：
  一组 Namespace + Cgroup + rootfs
  → 看起来像独立系统，实际共享宿主内核
```

## 3. Linux Cgroup？

**回答：**

```
Cgroup (Control Group)：
  限制、记录、隔离进程组使用的资源

  ┌──────────────┬──────────────────────────────┐
  │ 控制器        │ 限制内容                      │
  ├──────────────┼──────────────────────────────┤
  │ cpu          │ CPU 时间分配                   │
  │ cpuset       │ 绑定 CPU 核心                  │
  │ memory       │ 内存使用上限                   │
  │ blkio        │ 块设备 IO 速率                 │
  │ net_cls      │ 网络流量分类                   │
  │ pids         │ 进程数量限制                   │
  │ devices      │ 设备访问控制                   │
  └──────────────┴──────────────────────────────┘

  # 限制 CPU 使用 50%
  echo 50000 > /sys/fs/cgroup/cpu/mygroup/cpu.cfs_quota_us
  echo 100000 > /sys/fs/cgroup/cpu/mygroup/cpu.cfs_period_us

  # 限制内存 256MB
  echo 268435456 > /sys/fs/cgroup/memory/mygroup/memory.limit_in_bytes

  # 将进程加入 cgroup
  echo $PID > /sys/fs/cgroup/cpu/mygroup/cgroup.procs

Cgroup v1 vs v2：
  v1：每个控制器独立层级，管理复杂
  v2：统一层级，更简洁（推荐）
```

## 4. 容器网络原理？

**回答：**

```
Docker 网络模式：

  Bridge 模式（默认）：
  ┌────────────────────────────────────────┐
  │ 宿主机                                  │
  │  eth0 (物理网卡)                        │
  │    ↕ iptables NAT                       │
  │  docker0 (网桥 172.17.0.1)              │
  │    ├── veth1 ←→ 容器A eth0 (172.17.0.2)│
  │    └── veth2 ←→ 容器B eth0 (172.17.0.3)│
  └────────────────────────────────────────┘

  veth pair：虚拟网线，一端在容器，一端连网桥
  容器间通信：通过 docker0 网桥二层转发
  容器访外网：SNAT/MASQUERADE
  外部访容器：DNAT 端口映射

其他网络模式：
  Host：共享宿主网络栈（性能最好，无隔离）
  None：无网络
  Container：共享另一容器的网络栈
  Overlay：跨主机网络（VXLAN 隧道）
```

## 5. 容器存储原理？

**回答：**

```
联合文件系统 (UnionFS)：
  多个只读层 + 一个可写层 叠加

  ┌──────────────────────┐
  │ 可写层 (容器层)       │ ← 容器运行时修改
  ├──────────────────────┤
  │ 镜像层3 (应用代码)    │ ← 只读
  ├──────────────────────┤
  │ 镜像层2 (依赖安装)    │ ← 只读
  ├──────────────────────┤
  │ 镜像层1 (基础镜像)    │ ← 只读
  └──────────────────────┘

  写时复制 (CoW)：
  修改只读层文件 → 复制到可写层 → 修改副本

  存储驱动：
  overlay2（主流）：性能好，层数多
  devicemapper：块设备级别
  btrfs/zfs：高级文件系统特性

数据持久化：
  Volume：Docker 管理的持久卷
    docker run -v mydata:/app/data
  Bind Mount：绑定宿主目录
    docker run -v /host/path:/container/path
  tmpfs：内存临时存储
```

## 6. KVM 虚拟化原理？

**回答：**

```
KVM (Kernel-based Virtual Machine)：
  Linux 内核模块，利用硬件虚拟化扩展

  ┌────────────────────────────────────────┐
  │ 用户空间                                │
  │  ┌──────────────────────────────────┐  │
  │  │ QEMU（设备模拟）                   │  │
  │  │  VM1: [vCPU] [虚拟设备]            │  │
  │  │  VM2: [vCPU] [虚拟设备]            │  │
  │  └──────────────────────────────────┘  │
  ├────────────────────────────────────────┤
  │ 内核空间                                │
  │  KVM 模块 (/dev/kvm)                   │
  │  → 管理 vCPU、内存映射                  │
  ├────────────────────────────────────────┤
  │ 硬件                                    │
  │  VT-x/AMD-V (CPU 虚拟化)               │
  │  EPT/NPT (内存虚拟化)                  │
  │  VT-d (IO 虚拟化)                      │
  └────────────────────────────────────────┘

  CPU 虚拟化：
  Guest 运行在非 Root 模式
  特权指令 → VM Exit → KVM 处理 → VM Entry

  内存虚拟化：
  GVA → GPA → HPA (两级页表)
  EPT (Extended Page Table) 硬件加速
```

## 7. 容器安全？

**回答：**

```
容器安全风险：
  共享内核 → 内核漏洞可能逃逸
  默认 root → 权限过大
  镜像漏洞 → 基础镜像含已知 CVE

安全加固措施：
  ┌──────────────┬──────────────────────────────┐
  │ 措施          │ 说明                          │
  ├──────────────┼──────────────────────────────┤
  │ 非 root 运行 │ USER 指令指定普通用户          │
  │ 只读 rootfs  │ --read-only                   │
  │ 能力限制     │ --cap-drop ALL --cap-add NET   │
  │ Seccomp      │ 限制可用系统调用               │
  │ AppArmor     │ 强制访问控制                   │
  │ SELinux      │ 安全增强标签                   │
  │ 镜像扫描     │ Trivy/Clair 扫描 CVE          │
  │ 网络策略     │ NetworkPolicy 限制通信         │
  └──────────────┴──────────────────────────────┘

  Linux Capabilities：
  传统：root 有所有权限，普通用户无
  Capabilities：细粒度权限分割
    CAP_NET_BIND_SERVICE → 绑定 <1024 端口
    CAP_SYS_ADMIN → 系统管理（危险）
    CAP_NET_RAW → 原始套接字

  容器逃逸防护：
  不挂载 Docker Socket
  禁用 privileged 模式
  使用 gVisor/Kata 增强隔离
```

## 8. 容器运行时？

**回答：**

```
容器运行时层次：
  ┌──────────────────────────────────┐
  │ Docker / Podman / nerdctl        │ ← 用户工具
  ├──────────────────────────────────┤
  │ containerd / CRI-O              │ ← 高级运行时
  ├──────────────────────────────────┤
  │ runc / crun / gVisor / Kata     │ ← 低级运行时(OCI)
  ├──────────────────────────────────┤
  │ Linux Kernel                     │
  │ (Namespace + Cgroup + Seccomp)  │
  └──────────────────────────────────┘

  runc：
  OCI 标准参考实现
  Go 编写，调用 Linux 系统调用创建容器

  gVisor：
  Google 的沙箱运行时
  用户态内核 → 拦截系统调用 → 更强隔离

  Kata Containers：
  轻量级 VM + 容器接口
  每个容器运行在独立微型 VM 中

  containerd vs CRI-O：
  containerd：Docker 拆分出的运行时，K8s默认
  CRI-O：Red Hat 贡献，专为 K8s 设计
```

## 9. 进程沙箱与安全隔离？

**回答：**

```
沙箱技术对比：
  ┌──────────────┬──────────────────────────────┐
  │ 技术          │ 隔离方式                      │
  ├──────────────┼──────────────────────────────┤
  │ chroot       │ 仅限制根目录视图              │
  │ Namespace    │ 各种资源隔离                  │
  │ Seccomp      │ 限制可用系统调用              │
  │ AppArmor     │ 基于路径的强制访问控制        │
  │ SELinux      │ 基于标签的强制访问控制        │
  │ gVisor       │ 用户态内核                    │
  │ Firecracker  │ 微虚拟机（AWS Lambda 用）     │
  └──────────────┴──────────────────────────────┘

Seccomp (Secure Computing)：
  过滤进程可调用的系统调用
  Docker 默认 seccomp profile 禁用 ~44 个调用
  如：reboot, mount, kexec_load

  模式：
  SECCOMP_MODE_STRICT：只允许 read/write/exit/sigreturn
  SECCOMP_MODE_FILTER：BPF 程序过滤（Docker 用这个）

chroot vs pivot_root：
  chroot：改变根目录，但可逃逸
  pivot_root：更安全，交换根文件系统
  容器用 pivot_root
```

## 10. 虚拟化与容器面试速答？

**回答：**

```
Q: 容器和虚拟机区别？
A: 容器共享宿主内核，进程级隔离，秒级启动
   虚拟机独立内核，硬件级隔离，分钟级启动

Q: Docker 用了哪些 Linux 技术？
A: Namespace（隔离）+ Cgroup（限制）
   + UnionFS（镜像分层）+ chroot/pivot_root

Q: Namespace 有哪些？
A: PID/NET/MNT/UTS/IPC/USER/CGROUP/TIME
   分别隔离进程ID/网络/挂载/主机名/IPC/用户/cgroup/时间

Q: Cgroup 能限制什么？
A: CPU 时间/内存上限/磁盘 IO/进程数/网络带宽

Q: Docker 网络怎么实现的？
A: veth pair + 网桥 + iptables NAT
   容器间通过 docker0 网桥互联

Q: 联合文件系统是什么？
A: 多个只读镜像层 + 一个可写容器层叠加
   写时复制（CoW）修改只读文件

Q: 容器安全怎么做？
A: 非root运行/限制Capabilities/Seccomp过滤
   镜像扫描/只读rootfs/网络策略

Q: 容器逃逸怎么防？
A: 禁用 privileged/不挂载 docker.sock
   用 gVisor/Kata 增强隔离
```
