# 锁与同步

---

## 1. 锁的分类与对比？

**回答：**

```
  ┌──────────────┬──────────────────────────────────┐
  │ 锁类型        │ 特点                              │
  ├──────────────┼──────────────────────────────────┤
  │ 互斥锁 Mutex │ 同时只一个线程持有，阻塞等待       │
  │ 自旋锁        │ 忙等待不让出 CPU，临界区极短       │
  │ 读写锁        │ 读共享写独占，适合读多写少         │
  │ 条件变量      │ 等待特定条件满足再继续             │
  │ 信号量        │ 计数器，控制并发访问数             │
  │ 原子操作      │ CPU 指令级别，无锁，最轻量         │
  └──────────────┴──────────────────────────────────┘

互斥锁 vs 自旋锁：
  互斥锁：竞争失败 → 线程休眠 → 上下文切换开销
  自旋锁：竞争失败 → 忙等 → 浪费 CPU 但无切换开销
  → 临界区极短用自旋锁，长则用互斥锁

读写锁细分：
  读优先：读锁优先获取，写锁可能饥饿
  写优先：写锁优先，读锁等待
  公平模式：先来先得

Go 中的锁：
  sync.Mutex：互斥锁
  sync.RWMutex：读写锁
  sync/atomic：原子操作
  channel：CSP 模型通信
```

## 2. 死锁的条件与解决？

**回答：**

```
四个必要条件（缺一不可）：
  1. 互斥：资源同一时刻只能被一个进程使用
  2. 占有等待：持有资源的同时等待其他资源
  3. 不可抢占：已获取的资源不能被强制释放
  4. 循环等待：进程间形成环形等待链

  线程A: 持有锁1 → 等待锁2
  线程B: 持有锁2 → 等待锁1
  → 死锁！

预防死锁（打破条件）：
  ┌──────────────┬──────────────────────────────┐
  │ 策略          │ 方法                          │
  ├──────────────┼──────────────────────────────┤
  │ 破坏互斥      │ 使用不需要锁的算法（如CAS）   │
  │ 破坏占有等待  │ 一次性申请所有资源             │
  │ 破坏不可抢占  │ 获取失败时释放已有资源         │
  │ 破坏循环等待  │ 按固定顺序加锁                │
  └──────────────┴──────────────────────────────┘

检测死锁：
  资源分配图检测环
  go: runtime 检测 goroutine 死锁
  工具: pprof goroutine 分析

实际最佳实践：
  按固定顺序加锁
  设置锁超时
  使用 tryLock
  减小锁粒度
```

## 3. CAS 与无锁编程？

**回答：**

```
CAS (Compare And Swap)：
  原子操作：比较并交换
  CAS(addr, old, new) → 如果 *addr == old，则 *addr = new

  CPU 指令级支持（x86 CMPXCHG）

ABA 问题：
  线程1: 读到 A
  线程2: A → B → A
  线程1: CAS 成功（以为没变过）
  解决：加版本号 (A1 → B2 → A3)

自旋 CAS 伪代码：
  for {
      old = load(addr)
      new = compute(old)
      if CAS(addr, old, new) {
          break  // 成功
      }
      // 失败，重试
  }
```

```go
// Go 中的原子操作
import "sync/atomic"

var counter int64

// 原子加
atomic.AddInt64(&counter, 1)

// CAS
for {
    old := atomic.LoadInt64(&counter)
    if atomic.CompareAndSwapInt64(&counter, old, old+1) {
        break
    }
}

// Go 1.19+ atomic.Int64
var cnt atomic.Int64
cnt.Add(1)
cnt.Load()
```

## 4. 内存屏障与可见性？

**回答：**

```
CPU 缓存一致性问题：
  多核 CPU 各有 L1/L2 缓存
  一个核修改变量，另一个核可能看不到最新值

内存屏障 (Memory Barrier)：
  强制 CPU 按指定顺序执行内存操作
  保证指令不被重排序

  LoadBarrier：  之前的读操作必须完成
  StoreBarrier：之前的写操作必须完成
  FullBarrier：  读写都必须完成

Go 的内存模型 (happens-before)：
  channel 发送 happens-before 接收
  sync.Mutex Unlock happens-before 下次 Lock
  sync.Once Do 的执行 happens-before 任何 Do 返回
  atomic 操作提供顺序保证

  → 正确使用同步原语就不需关心内存屏障细节
```

## 5. 条件变量与生产者消费者？

**回答：**

```
条件变量：线程等待特定条件，条件满足时被唤醒

  Wait()：释放锁 + 睡眠 + 被唤醒后重新获取锁
  Signal()：唤醒一个等待线程
  Broadcast()：唤醒所有等待线程

  必须配合互斥锁使用
  Wait 前必须持有锁
  唤醒后要重新检查条件（while 而非 if）
```

```go
// Go: sync.Cond 生产者消费者
type Queue struct {
    items []int
    mu    sync.Mutex
    cond  *sync.Cond
    max   int
}

func NewQueue(max int) *Queue {
    q := &Queue{max: max}
    q.cond = sync.NewCond(&q.mu)
    return q
}

func (q *Queue) Put(item int) {
    q.mu.Lock()
    defer q.mu.Unlock()
    for len(q.items) >= q.max { // while 循环检查
        q.cond.Wait() // 队列满，等待
    }
    q.items = append(q.items, item)
    q.cond.Signal() // 通知消费者
}

func (q *Queue) Get() int {
    q.mu.Lock()
    defer q.mu.Unlock()
    for len(q.items) == 0 { // while 循环检查
        q.cond.Wait() // 队列空，等待
    }
    item := q.items[0]
    q.items = q.items[1:]
    q.cond.Signal() // 通知生产者
    return item
}

// Go 更地道的方式：用 channel
ch := make(chan int, 100) // 带缓冲 channel
ch <- item                // 生产（满时阻塞）
item := <-ch              // 消费（空时阻塞）
```

## 6. 信号量？

**回答：**

```
信号量 (Semaphore)：计数器，控制并发访问数

  P 操作 (Wait/Down)：计数器 -1，为 0 时阻塞
  V 操作 (Signal/Up)：计数器 +1，唤醒等待线程

  二元信号量 = 互斥锁（初始值 1）
  计数信号量：限制并发数（如连接池）

经典问题：
  生产者-消费者：两个信号量（empty + full）+ 互斥锁
  读者-写者：读计数 + 互斥锁 + 写信号量
  哲学家就餐：防止死锁（限制同时就餐人数/按序拿筷）
```

```go
// Go 实现信号量（channel 模拟）
type Semaphore struct {
    ch chan struct{}
}

func NewSemaphore(n int) *Semaphore {
    return &Semaphore{ch: make(chan struct{}, n)}
}

func (s *Semaphore) Acquire() {
    s.ch <- struct{}{} // P 操作
}

func (s *Semaphore) Release() {
    <-s.ch // V 操作
}

// 或使用 golang.org/x/sync/semaphore
```

## 7. 锁粒度优化？

**回答：**

```
锁粒度：锁保护的数据范围大小

  粗粒度锁：一把大锁保护所有数据 → 简单但并发低
  细粒度锁：多把锁保护不同数据 → 并发高但复杂

优化策略：
  ┌──────────────────┬──────────────────────────────┐
  │ 策略              │ 说明                          │
  ├──────────────────┼──────────────────────────────┤
  │ 减小锁范围        │ 只锁必要的代码段              │
  │ 锁分离            │ 读写锁分离（sync.RWMutex）    │
  │ 锁分段            │ 如 ConcurrentHashMap 分段锁   │
  │ 无锁              │ CAS 原子操作                  │
  │ 消除锁            │ 线程本地存储 / 不可变数据      │
  │ 乐观锁            │ 版本号/CAS，冲突时重试         │
  └──────────────────┴──────────────────────────────┘

Go sync.Map 的实现思路：
  read map (原子读，无锁) + dirty map (加锁写)
  读多写少场景高效
  → 适合于 key 稳定、读远多于写的场景
```

## 8. 用户态锁 vs 内核态锁？

**回答：**

```
  ┌──────────────┬──────────────────┬──────────────────┐
  │              │ 用户态锁          │ 内核态锁          │
  ├──────────────┼──────────────────┼──────────────────┤
  │ 切换开销      │ 无需进内核        │ 需要系统调用      │
  │ 性能          │ 快               │ 慢               │
  │ 适用          │ 无竞争/轻竞争     │ 有竞争需阻塞      │
  │ 代表          │ 自旋锁/CAS       │ futex/mutex      │
  └──────────────┴──────────────────┴──────────────────┘

Futex (Fast Userspace Mutex)：
  Linux 混合锁实现
  无竞争：纯用户态 CAS（快）
  有竞争：陷入内核态等待（阻塞）
  → 兼顾性能和功能

Go sync.Mutex 实现：
  正常模式：CAS 尝试 + 自旋 + 信号量阻塞
  饥饿模式：等待超过 1ms → 直接交给等待者
  → 防止某个 goroutine 长时间拿不到锁
```

## 9. 分布式锁？

**回答：**

```
分布式锁：跨进程/跨机器的互斥访问

  ┌──────────────┬──────────────────────────────┐
  │ 方案          │ 特点                          │
  ├──────────────┼──────────────────────────────┤
  │ Redis        │ SETNX + 过期时间，简单高效     │
  │ etcd         │ 租约 + Watch，强一致            │
  │ ZooKeeper    │ 临时顺序节点，可靠              │
  │ 数据库        │ 行锁/唯一约束，性能较低         │
  └──────────────┴──────────────────────────────┘

Redis 分布式锁：
  加锁：SET key uuid NX PX 30000
  解锁：Lua 脚本（check uuid + DEL 原子操作）
  续期：Watchdog 机制（Redisson）

  问题：主从切换可能丢锁
  RedLock：多节点(N/2+1)成功才算获锁

etcd 分布式锁：
  创建租约 → 绑定 key → 竞争（唯一前缀最小 revision 获锁）
  自动续租 → 持有锁期间续约
  释放锁 → 删除 key / 租约过期
```

## 10. 锁与同步面试速答？

**回答：**

```
Q: 互斥锁和自旋锁区别？
A: 互斥锁：竞争失败休眠，有上下文切换
   自旋锁：忙等待，适合临界区极短的场景

Q: 死锁四个条件？
A: 互斥/占有等待/不可抢占/循环等待
   最常用预防：按固定顺序加锁

Q: CAS 是什么？ABA 问题？
A: Compare And Swap 原子操作
   ABA: 值被改回原值，CAS 误以为没变
   解决：加版本号

Q: Go sync.Mutex 原理？
A: 正常模式：CAS+自旋+信号量
   饥饿模式：等待>1ms 直接交给等待者

Q: 读写锁适用场景？
A: 读多写少（如配置缓存）
   读并发，写独占

Q: channel vs mutex？
A: channel: 通信而非共享，CSP 模型，Go 推荐
   mutex: 保护共享数据，传统方式
   "Don't communicate by sharing; share by communicating"

Q: 分布式锁怎么选？
A: Redis SETNX: 简单高效，AP
   etcd: 强一致，适合关键业务
   ZooKeeper: 可靠但笨重

Q: 什么是 Futex？
A: 快速用户态互斥锁
   无竞争在用户态 CAS，有竞争才进内核
   Go/C 的 Mutex 底层实现
```
