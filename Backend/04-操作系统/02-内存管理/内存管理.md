# 内存管理

---

## 1. 虚拟内存原理？

**回答：**

```
虚拟内存：每个进程拥有独立的虚拟地址空间
  通过页表映射到物理内存

  虚拟地址 → MMU(页表) → 物理地址

  ┌──────────────────────────────────────────┐
  │ 进程A 虚拟空间          物理内存          │
  │ ┌────────────┐    ┌──────────────────┐   │
  │ │ 0x0000     │    │ 物理页帧 0       │   │
  │ │ 代码段      │───→│ 进程A 代码       │   │
  │ │ 数据段      │    │ 进程B 数据       │   │
  │ │ 堆         │───→│ 进程A 堆         │   │
  │ │ ...        │    │ ...              │   │
  │ │ 栈         │───→│ 进程A 栈         │   │
  │ │ 0xFFFF     │    │ 内核空间         │   │
  │ └────────────┘    └──────────────────┘   │
  └──────────────────────────────────────────┘

作用：
  1. 进程隔离：每个进程独立地址空间
  2. 内存扩展：虚拟空间可大于物理内存
  3. 内存保护：页表权限位控制读写执行
  4. 共享内存：不同虚拟页映射同一物理页
```

## 2. 分页与分段？

**回答：**

```
  ┌──────────────┬──────────────────┬──────────────────┐
  │              │ 分页              │ 分段              │
  ├──────────────┼──────────────────┼──────────────────┤
  │ 大小          │ 固定（4KB）       │ 可变             │
  │ 划分依据      │ 物理划分          │ 逻辑划分         │
  │ 内碎片        │ 有（页内浪费）    │ 无               │
  │ 外碎片        │ 无               │ 有               │
  │ 地址          │ 页号+页内偏移     │ 段号+段内偏移     │
  │ 共享/保护     │ 页粒度            │ 更自然（按逻辑段）│
  └──────────────┴──────────────────┴──────────────────┘

现代系统：段页式（x86 分段 + 分页结合）
Linux：主要使用分页，段机制弱化

多级页表：
  解决页表过大问题
  32位系统：二级页表（页目录+页表）
  64位系统：四级页表（PGD→PUD→PMD→PTE）
  
TLB (Translation Lookaside Buffer)：
  页表的高速缓存，命中率 >99%
  上下文切换需刷新 TLB → 性能损失
```

## 3. 页面置换算法？

**回答：**

```
当物理内存不足时，选择哪个页面换出到磁盘

  ┌──────────────┬──────────────────────────────┐
  │ 算法          │ 原理                          │
  ├──────────────┼──────────────────────────────┤
  │ OPT（最优）   │ 淘汰最长时间不再使用的，理论最优│
  │ FIFO         │ 淘汰最早进入的，可能 Belady 异常│
  │ LRU          │ 淘汰最近最少使用的，效果好      │
  │ Clock        │ 近似 LRU，引用位+循环扫描      │
  │ LFU          │ 淘汰使用频率最低的             │
  └──────────────┴──────────────────────────────┘

Linux 实际方案：
  两个 LRU 链表：Active List + Inactive List
  新页先入 Inactive
  再次访问 → 晋升到 Active
  Active 满 → 降级到 Inactive
  淘汰 Inactive 尾部的页

Belady 异常：
  FIFO 分配更多页帧反而缺页更多
  LRU 不会出现此问题（栈算法）
```

## 4. 进程内存布局？

**回答：**

```
Linux 进程虚拟地址空间布局（低→高）：
  ┌────────────────────────┐ 0xFFFFFFFF (高地址)
  │   内核空间 (1GB)        │ 所有进程共享
  ├────────────────────────┤ 0xC0000000
  │   栈 (Stack)           │ ↓ 向低地址增长
  │   ...                  │
  │   内存映射段 (mmap)     │ 动态库/文件映射
  │   ...                  │
  │   堆 (Heap)            │ ↑ 向高地址增长
  ├────────────────────────┤
  │   BSS 段               │ 未初始化全局变量
  │   数据段 (Data)         │ 已初始化全局变量
  │   代码段 (Text)         │ 可执行指令（只读）
  └────────────────────────┘ 0x00000000 (低地址)

各段说明：
  代码段：程序指令，只读共享
  数据段：全局/静态变量（已初始化）
  BSS：  全局/静态变量（未初始化，零填充）
  堆：   malloc/new 动态分配，向上增长
  mmap： 动态库加载/文件映射/匿名映射
  栈：   函数调用（局部变量/返回地址），向下增长

栈 vs 堆：
  栈：自动管理，速度快，空间小（默认 8MB）
  堆：手动管理，速度慢，空间大，可能碎片
```

## 5. 内存分配策略？malloc 原理？

**回答：**

```
malloc 底层实现（glibc ptmalloc2）：
  小内存（< 128KB）：brk() 扩展堆
  大内存（≥ 128KB）：mmap() 分配独立区域

内存分配器对比：
  ┌──────────────┬──────────────────────────────┐
  │ 分配器        │ 特点                          │
  ├──────────────┼──────────────────────────────┤
  │ ptmalloc2    │ glibc 默认，多线程用 Arena    │
  │ tcmalloc     │ Google，线程本地缓存，高并发   │
  │ jemalloc     │ FreeBSD/Firefox，低碎片       │
  │ Go runtime   │ mcache/mcentral/mheap 三级   │
  └──────────────┴──────────────────────────────┘

内存碎片：
  内碎片：分配的内存大于实际需要
  外碎片：空闲内存不连续，无法分配大块
  解决：伙伴系统（Buddy）/ Slab 分配器

Linux 内核内存管理：
  伙伴系统：管理物理页帧，按 2^n 页分配
  Slab：管理内核对象（task_struct 等），减少碎片
```

## 6. 内存映射 mmap？

**回答：**

```
mmap：将文件/设备映射到进程虚拟地址空间

  进程 → 虚拟地址读写 → 自动同步到文件
  无需 read()/write()，减少拷贝

使用场景：
  文件映射：大文件随机读写
  匿名映射：分配大块内存（malloc 底层）
  共享映射：进程间共享内存
  私有映射：写时复制（动态库加载）

  ┌──────────────┬──────────────┬──────────────┐
  │              │ 共享映射      │ 私有映射      │
  ├──────────────┼──────────────┼──────────────┤
  │ 文件映射      │ 进程间共享    │ COW，独立修改 │
  │              │ 修改写回文件  │ 不写回文件    │
  │ 匿名映射      │ 父子进程共享  │ malloc 大内存 │
  └──────────────┴──────────────┴──────────────┘

mmap vs read/write：
  mmap：少一次拷贝（无需内核→用户缓冲区）
  适合：大文件/随机访问/进程共享
  不适合：小文件顺序读取（page fault 开销）
```

## 7. OOM Killer 机制？

**回答：**

```
OOM (Out of Memory)：
  系统物理内存+Swap 耗尽时触发
  内核 OOM Killer 选择进程杀掉释放内存

选择策略（oom_score）：
  分数越高越先被杀
  因素：进程内存使用量/运行时间/优先级

  oom_score_adj：手动调整（-1000 ~ 1000）
    -1000：永不被 OOM Kill
    1000：优先被 Kill

  # 查看进程 OOM 分数
  cat /proc/<pid>/oom_score
  
  # 保护关键进程
  echo -1000 > /proc/<pid>/oom_score_adj

避免 OOM：
  1. Cgroups 限制容器内存（K8s resources.limits）
  2. 应用层内存池/限制
  3. 监控内存使用告警
  4. 合理设置 overcommit 策略
     /proc/sys/vm/overcommit_memory
     0：启发式（默认），1：总是允许，2：不允许过量

容器 OOM：
  K8s Pod 被 OOM Kill → 状态 OOMKilled
  检查：kubectl describe pod → 看 Last State
```

## 8. Swap 与内存回收？

**回答：**

```
Swap：将不活跃的内存页写到磁盘，释放物理内存

回收流程：
  内存不足 → kswapd 后台回收 → 直接回收 → OOM Kill

可回收的页：
  文件页：有磁盘文件对应（page cache），直接丢弃/写回
  匿名页：无文件对应（堆/栈），写入 Swap

  ┌──────────────┬──────────────┬──────────────┐
  │              │ 文件页        │ 匿名页        │
  ├──────────────┼──────────────┼──────────────┤
  │ 来源          │ 文件读写缓存  │ 堆/栈/mmap   │
  │ 回收方式      │ 丢弃/写回磁盘│ 写入 Swap     │
  │ 代价          │ 低            │ 高（IO 慢）   │
  └──────────────┴──────────────┴──────────────┘

swappiness 参数（0-200）：
  0：尽量不使用 Swap（优先回收文件页）
  100：平等对待文件页和匿名页
  生产建议：10-30（对延迟敏感设为 0-10）

容器场景：
  K8s 要求关闭 Swap（swapoff -a）
  原因：Swap 导致延迟不可预测，影响调度
```

## 9. 大页内存（Huge Pages）？

**回答：**

```
普通页：4KB → 管理大内存需大量页表项
大页：2MB / 1GB → 减少页表项数量

优点：
  减少 TLB Miss（一个 TLB 条目覆盖更多内存）
  减少页表层级/大小
  减少缺页中断次数

使用场景：
  数据库（MySQL/PostgreSQL）
  大内存应用（Redis/Java 大堆）
  虚拟化（KVM 虚拟机）

Linux 大页类型：
  HugePages（静态）：启动时预留，不可 Swap
  Transparent Huge Pages (THP)：自动合并为大页
  
  # 配置 HugePages
  echo 1024 > /proc/sys/vm/nr_hugepages  # 预留 1024 个 2MB 大页

注意：
  THP 可能引起延迟抖动（碎片整理）
  Redis/数据库通常建议关闭 THP
  echo never > /sys/kernel/mm/transparent_hugepage/enabled
```

## 10. 内存管理面试速答？

**回答：**

```
Q: 虚拟内存是什么？
A: 每个进程独立虚拟地址空间
   通过页表映射到物理内存
   实现进程隔离/内存扩展/保护

Q: 分页和分段区别？
A: 分页：固定大小(4KB)，物理划分，有内碎片
   分段：可变大小，按逻辑划分，有外碎片
   Linux 主要用分页

Q: 页面置换算法？
A: LRU 最常考（淘汰最近最少使用）
   Linux 用改进方案：Active/Inactive 双链表

Q: malloc 底层原理？
A: 小内存 <128KB → brk() 扩展堆
   大内存 ≥128KB → mmap() 独立映射
   分配器管理空闲链表/内存池

Q: 栈和堆区别？
A: 栈：自动管理，快，空间小，向下增长
   堆：手动管理，慢，空间大，向上增长

Q: 什么是 OOM？怎么解决？
A: 内存耗尽，内核杀高分进程
   预防：Cgroups 限制/监控/合理 overcommit
   容器：设 resources.limits.memory

Q: Swap 对性能影响？
A: 匿名页换到磁盘，延迟增大
   生产服务：swappiness 设低或关闭
   K8s 要求关闭 Swap

Q: mmap 和 read 区别？
A: mmap 文件映射到虚拟地址，减少一次拷贝
   适合大文件随机读写/进程间共享
```
