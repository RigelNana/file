# CPU与系统调用

---

## 1. 用户态与内核态？

**回答：**

```
CPU 特权级别（x86 Ring）：
  Ring 0：内核态（最高权限）→ 操作硬件/内存管理
  Ring 3：用户态（受限权限）→ 应用程序运行

  用户态不能：
    直接操作 IO 设备
    访问内核内存
    执行特权指令（cli/sti/hlt）

切换场景：
  ┌──────────────────┬──────────────────────────────┐
  │ 触发方式          │ 说明                          │
  ├──────────────────┼──────────────────────────────┤
  │ 系统调用          │ 应用主动请求内核服务           │
  │ 异常              │ 缺页/除零/非法指令             │
  │ 硬件中断          │ 定时器/网卡/磁盘中断           │
  └──────────────────┴──────────────────────────────┘

切换开销：
  保存/恢复用户态寄存器
  切换栈（用户栈→内核栈）
  TLB 刷新（可能）
  约 1-10 微秒

减少切换的方法：
  批量系统调用（io_uring）
  mmap 替代 read/write
  用户态网络栈（DPDK）
```

## 2. 系统调用过程？

**回答：**

```
系统调用流程（以 read() 为例）：

  用户空间               内核空间
  ┌──────────┐           ┌──────────┐
  │ read()   │           │ sys_read()│
  │   ↓      │           │          │
  │ glibc    │           │          │
  │ wrapper  │──syscall──→│ 中断处理 │
  │          │ (int 0x80 │ 查系统调用│
  │          │  /syscall)│ 表       │
  │          │           │   ↓      │
  │ 返回值   │←──返回────│ 执行并返回│
  └──────────┘           └──────────┘

详细步骤：
  1. libc 封装函数设置系统调用号到 eax
  2. 参数放入寄存器（edi/esi/edx...）
  3. 执行 syscall 指令（x86_64）
  4. CPU 切换到内核态
  5. 根据调用号查 sys_call_table
  6. 执行内核函数
  7. 返回用户态

常见系统调用：
  进程：fork/exec/exit/wait/kill
  文件：open/read/write/close/stat
  内存：mmap/munmap/brk
  网络：socket/bind/listen/accept/connect
  同步：futex
```

## 3. 中断机制？

**回答：**

```
中断分类：
  ┌──────────────┬──────────────────────────────┐
  │ 类型          │ 说明                          │
  ├──────────────┼──────────────────────────────┤
  │ 硬件中断      │ 外设触发（网卡/磁盘/定时器）  │
  │ 软中断        │ 内核触发（系统调用/softirq）  │
  │ 异常          │ CPU 执行异常（缺页/除零）     │
  └──────────────┴──────────────────────────────┘

中断处理流程：
  1. 设备发出中断信号
  2. CPU 完成当前指令
  3. 保存现场（寄存器/PC/标志位）
  4. 根据中断向量号查中断描述符表
  5. 跳转到中断处理程序执行
  6. 恢复现场，返回被中断的程序

Linux 中断上下半部：
  上半部（硬中断）：快速处理，不可睡眠
    → 读取数据/清中断标志/唤醒下半部
  下半部（软中断/tasklet/workqueue）：延迟处理
    → 复杂耗时的操作

网卡中断示例：
  网卡收到数据包 → 硬中断 → 将数据放入 Ring Buffer
  → 触发 softirq/NAPI → 协议栈处理（下半部）
```

## 4. CPU 缓存与伪共享？

**回答：**

```
CPU 缓存层次：
  CPU Core → L1 Cache (32KB, ~1ns)
           → L2 Cache (256KB, ~3ns)
           → L3 Cache (共享, 数MB, ~10ns)
           → 内存 (~100ns)
           → 磁盘 (~ms)

缓存行 (Cache Line)：
  缓存的最小单位，通常 64 字节
  读取一个变量会加载整条缓存行

伪共享 (False Sharing)：
  不同核上的线程修改同一缓存行中的不同变量
  导致缓存行在核间反复失效 → 性能下降

  ┌────────────────────────────────────────┐
  │ Cache Line (64 bytes)                  │
  │ [变量A (核1修改) | 变量B (核2修改)]     │
  │  → 核1改A，核2缓存行失效               │
  │  → 核2改B，核1缓存行失效               │
  │  → 反复失效，性能骤降                  │
  └────────────────────────────────────────┘

解决：缓存行填充（Padding）
```

```go
// 伪共享问题示例
type BadStruct struct {
    a int64 // 核1频繁修改
    b int64 // 核2频繁修改 → 同一缓存行，伪共享！
}

// 解决：缓存行填充
type GoodStruct struct {
    a int64
    _ [56]byte // 填充到 64 字节
    b int64    // 独立缓存行
}
```

## 5. CPU 亲和性与 NUMA？

**回答：**

```
CPU 亲和性 (Affinity)：
  将进程/线程绑定到特定 CPU 核心
  减少 CPU 迁移和缓存失效

  # 将进程绑定到 CPU 0,1
  taskset -c 0,1 ./myapp

  # Go 中设置
  runtime.LockOSThread()  // 绑定 goroutine 到 OS 线程

NUMA (Non-Uniform Memory Access)：
  多路 CPU 架构，每个 CPU 有本地内存
  访问本地内存快，访问远程内存慢

  ┌────────────┐        ┌────────────┐
  │  CPU 0     │        │  CPU 1     │
  │  L1/L2/L3  │        │  L1/L2/L3  │
  │     ↓      │        │     ↓      │
  │  本地内存   │←互联→│  本地内存   │
  │  (快)      │ (慢)  │  (快)      │
  └────────────┘        └────────────┘

NUMA 优化：
  进程绑定 NUMA 节点
  内存分配优先本地节点
  Redis/数据库需要 NUMA 感知

  # 在 NUMA Node 0 上运行
  numactl --cpunodebind=0 --membind=0 ./redis-server
```

## 6. 进程优先级与 nice 值？

**回答：**

```
Linux 优先级体系：
  ┌──────────────┬──────────────────────────────┐
  │ 类型          │ 范围                          │
  ├──────────────┼──────────────────────────────┤
  │ 实时优先级    │ 0-99（数字越大优先级越高）     │
  │ 普通优先级    │ 100-139（nice -20 ~ 19）      │
  └──────────────┴──────────────────────────────┘

  nice 值：-20（最高优先级）~ 19（最低优先级）
  默认 nice = 0

  # 查看进程优先级
  ps -l
  top  # PR 列

  # 以高优先级运行
  nice -n -10 ./myapp

  # 修改运行中进程优先级
  renice -5 -p <pid>

调度策略：
  SCHED_NORMAL：普通进程（CFS 调度）
  SCHED_FIFO：实时先进先出
  SCHED_RR：实时轮转
  SCHED_BATCH：批处理任务
  SCHED_IDLE：空闲时才运行
```

## 7. 系统性能指标？

**回答：**

```
关键性能指标：
  ┌──────────────┬──────────────────────────────┐
  │ 指标          │ 说明                          │
  ├──────────────┼──────────────────────────────┤
  │ CPU 使用率    │ user/system/iowait/idle       │
  │ 负载 (Load)  │ 运行+等待进程数/核数           │
  │ 内存使用      │ free/used/buffer/cache        │
  │ IO 等待      │ iowait（CPU 等 IO 完成比例）   │
  │ 上下文切换    │ cs/s（vmstat）                │
  │ 网络吞吐      │ 带宽利用/包速率               │
  └──────────────┴──────────────────────────────┘

Load Average 解读（如 4 核 CPU）：
  < 4.0：正常
  = 4.0：刚好满载
  > 4.0：过载（进程在排队）
  查看：uptime / cat /proc/loadavg

  # CPU 高 → top 看哪个进程
  # Load 高但 CPU 低 → 可能 IO 等待（iowait）
  # 内存高 → free -h, 注意 buffer/cache 可回收
```

## 8. Linux 性能分析工具？

**回答：**

```
  ┌──────────────┬──────────────────────────────┐
  │ 工具          │ 用途                          │
  ├──────────────┼──────────────────────────────┤
  │ top/htop     │ 进程 CPU/内存实时概览          │
  │ vmstat       │ 系统总体（CPU/内存/IO/上下文） │
  │ iostat       │ 磁盘 IO 统计                  │
  │ sar          │ 历史性能数据                   │
  │ pidstat      │ 进程级性能统计                 │
  │ free         │ 内存使用                      │
  │ netstat/ss   │ 网络连接                      │
  │ strace       │ 系统调用追踪                   │
  │ ltrace       │ 库调用追踪                    │
  │ perf         │ CPU 性能分析（火焰图）         │
  │ ftrace       │ 内核函数追踪                   │
  │ bpftrace/eBPF│ 动态追踪框架                  │
  │ pprof        │ Go 程序性能分析                │
  │ dstat        │ vmstat+iostat+netstat 综合     │
  └──────────────┴──────────────────────────────┘

故障排查步骤：
  1. top：看 CPU/内存哪个高
  2. vmstat 1：看 CPU/IO/上下文切换
  3. iostat -x 1：磁盘 IO（%util/await）
  4. pidstat -u 1：哪个进程占 CPU
  5. strace -p <pid>：卡在哪个系统调用
  6. perf top / perf record：CPU 热点函数
```

## 9. eBPF 技术？

**回答：**

```
eBPF (Extended Berkeley Packet Filter)：
  Linux 内核可编程框架
  在内核中安全运行用户定义的程序

  ┌────────────────────────────────────────┐
  │ 用户空间                                │
  │  ┌──────────┐                          │
  │  │ BPF 程序  │ → 编译为 BPF 字节码      │
  │  └──────┬───┘                          │
  │         │ bpf() 系统调用               │
  ├─────────┼──────────────────────────────┤
  │ 内核空间 │                              │
  │         ↓                              │
  │  ┌──────────┐                          │
  │  │ Verifier │ → 安全检查                │
  │  └──────┬───┘                          │
  │         │ JIT 编译                     │
  │  ┌──────┴───┐                          │
  │  │ 内核Hook  │ → 网络/系统调用/追踪等    │
  │  └──────────┘                          │
  └────────────────────────────────────────┘

应用场景：
  网络：高性能包过滤/负载均衡（Cilium）
  可观测性：低开销追踪（bpftrace/Pixie）
  安全：运行时安全监控（Falco）
  性能：CPU/内存/IO 分析

优势：
  无需改内核/无需重启
  JIT 编译接近原生性能
  安全（Verifier 检查，不会崩内核）
```

## 10. CPU与系统调用面试速答？

**回答：**

```
Q: 用户态和内核态区别？
A: 用户态权限受限，不能操作硬件
   内核态拥有全部权限
   通过系统调用/中断/异常切换

Q: 系统调用开销？
A: 保存寄存器 + 切换栈 + 查表 + 执行 + 恢复
   约 1-10 微秒
   减少方法：批量(io_uring)/mmap/用户态协议栈

Q: 什么是伪共享？
A: 不同核修改同一缓存行中不同变量
   缓存行反复失效导致性能差
   解决：缓存行填充(padding)

Q: NUMA 是什么？
A: 多 CPU 架构，本地内存快远程内存慢
   优化：绑核+本地内存分配

Q: 中断上下半部？
A: 上半部：快速处理，不可睡眠（读数据/清标志）
   下半部：延迟处理耗时操作（softirq/tasklet）

Q: CPU Load 和 CPU 使用率区别？
A: Load：就绪+运行+不可中断进程数
   使用率：CPU 忙碌时间占比
   Load 高 CPU 低 → IO 等待

Q: eBPF 是什么？
A: 内核可编程框架，安全运行自定义代码
   应用：Cilium 网络/追踪/安全监控

Q: perf 怎么用？
A: perf top：实时看 CPU 热点函数
   perf record/report：记录并分析
   生成火焰图：perf script → FlameGraph
```
