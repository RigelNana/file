# 进程间通信

---

## 1. IPC 方式汇总？

**回答：**

```
Linux IPC 机制：
  ┌──────────────┬──────────────┬──────────────────────────┐
  │ IPC 方式      │ 数据流向      │ 特点                      │
  ├──────────────┼──────────────┼──────────────────────────┤
  │ 管道 Pipe     │ 单向          │ 父子进程，内核缓冲区 64KB  │
  │ 命名管道 FIFO │ 单向          │ 无亲缘关系进程             │
  │ 消息队列      │ 双向          │ 带类型的消息，内核持久化    │
  │ 共享内存      │ 双向          │ 最快，需要同步机制         │
  │ 信号量        │ 同步控制      │ 计数器，进程同步           │
  │ 信号 Signal   │ 异步通知      │ 事件通知（SIGKILL等）      │
  │ Socket        │ 双向          │ 跨主机/本机，最通用        │
  │ Unix Socket   │ 双向          │ 本机进程，比 TCP 快        │
  └──────────────┴──────────────┴──────────────────────────┘

选型建议：
  同机高性能 → 共享内存 + 信号量
  简单父子通信 → 管道
  跨机器通信 → Socket
  微服务间 → Unix Socket / gRPC
```

## 2. 管道原理？

**回答：**

```
管道（Pipe）：
  内核维护的环形缓冲区（默认 64KB）
  半双工（单向数据流）

  进程A (写端) ──→ [内核缓冲区] ──→ 进程B (读端)
                    ↑ 64KB 环形

  特性：
  - 读空管道阻塞
  - 写满管道阻塞
  - 写端关闭，读端读到 EOF
  - 读端关闭，写端收到 SIGPIPE

  Shell 中的管道：
  cat file | grep "error" | wc -l
  → 三个进程通过两个管道连接
```

```go
// Go 中使用管道
func main() {
    r, w, _ := os.Pipe()

    // 子进程写入
    go func() {
        w.Write([]byte("hello from child"))
        w.Close()
    }()

    // 父进程读取
    buf := make([]byte, 128)
    n, _ := r.Read(buf)
    fmt.Println(string(buf[:n]))
}
```

```
命名管道 FIFO：
  mkfifo /tmp/myfifo
  文件系统中有名字，无亲缘关系进程可用
  仍然是单向的
```

## 3. 共享内存？

**回答：**

```
共享内存原理：
  两个进程的页表映射到同一物理内存区域

  进程A 虚拟地址 ──→ ┌──────────┐ ←── 进程B 虚拟地址
                      │ 物理内存  │
                      │ (共享区)  │
                      └──────────┘

  IPC 中最快的方式：
  - 不需要内核介入数据拷贝
  - 直接读写内存
  - 需要额外同步机制（信号量/互斥锁）

POSIX 共享内存 API：
  shm_open()   → 创建/打开共享内存对象
  ftruncate()  → 设置大小
  mmap()       → 映射到进程地址空间
  munmap()     → 取消映射
  shm_unlink() → 删除共享内存对象

使用场景：
  高频数据交换（如行情数据推送）
  共享大数据集（如缓存）
  数据库共享缓冲区（PostgreSQL）
```

## 4. 消息队列（内核）？

**回答：**

```
System V 消息队列：
  内核维护的消息链表
  每条消息有类型标签 → 可选择性接收

  进程A ──msgsnc()──→ [内核消息队列] ──msgrcv()──→ 进程B
                       ↑ 带类型的消息

  API：
  msgget()  → 创建/获取消息队列
  msgsnd()  → 发送消息
  msgrcv()  → 接收消息（可按类型过滤）
  msgctl()  → 控制/删除

POSIX 消息队列：
  mq_open()
  mq_send()
  mq_receive()
  优势：支持优先级、异步通知

消息队列 vs 管道：
  ┌──────────────┬──────────────┬──────────────┐
  │ 对比          │ 管道          │ 消息队列      │
  ├──────────────┼──────────────┼──────────────┤
  │ 数据格式      │ 字节流        │ 结构化消息    │
  │ 方向          │ 单向          │ 双向          │
  │ 选择性读取    │ 不支持        │ 按类型过滤    │
  │ 生命周期      │ 进程退出消失  │ 内核持久      │
  └──────────────┴──────────────┴──────────────┘
```

## 5. 信号机制？

**回答：**

```
信号：异步通知机制
  内核/进程向目标进程发送软中断

  常见信号：
  ┌──────────────┬────────┬──────────────────────┐
  │ 信号          │ 编号    │ 默认行为              │
  ├──────────────┼────────┼──────────────────────┤
  │ SIGINT       │ 2      │ 终止（Ctrl+C）        │
  │ SIGKILL      │ 9      │ 强制终止（不可捕获）   │
  │ SIGSEGV      │ 11     │ 段错误 → core dump     │
  │ SIGTERM      │ 15     │ 优雅终止（可捕获）     │
  │ SIGSTOP      │ 19     │ 暂停（不可捕获）       │
  │ SIGCHLD      │ 17     │ 子进程状态变化         │
  │ SIGUSR1/2    │ 10/12  │ 用户自定义             │
  │ SIGPIPE      │ 13     │ 写入无读者管道         │
  └──────────────┴────────┴──────────────────────┘

  信号处理方式：
  1. 默认行为（终止/忽略/暂停/core dump）
  2. 捕获处理（注册 handler）
  3. 忽略（SIG_IGN）
  注意：SIGKILL 和 SIGSTOP 不可捕获/忽略
```

```go
// Go 中信号处理
func main() {
    sigs := make(chan os.Signal, 1)
    signal.Notify(sigs, syscall.SIGINT, syscall.SIGTERM)

    go func() {
        sig := <-sigs
        fmt.Println("收到信号:", sig)
        // 优雅退出：关闭连接、保存数据...
        os.Exit(0)
    }()

    // 业务逻辑...
    select {}
}
```

## 6. Unix Domain Socket？

**回答：**

```
Unix Domain Socket（UDS）：
  本机进程间通信，走内核但不经过网络协议栈

  TCP Socket vs Unix Socket：
  ┌──────────────┬──────────────┬──────────────┐
  │ 对比          │ TCP Socket   │ Unix Socket  │
  ├──────────────┼──────────────┼──────────────┤
  │ 地址          │ IP:Port      │ 文件路径      │
  │ 性能          │ 较慢         │ 快 2-3 倍     │
  │ 跨主机        │ 支持         │ 不支持        │
  │ 传输类型      │ 流/数据报    │ 流/数据报     │
  │ 特殊能力      │ 无           │ 可传递 FD     │
  └──────────────┴──────────────┴──────────────┘

  UDS 快的原因：
  - 不需要 TCP 握手/挥手
  - 不经过网络协议栈（无校验和/序列号）
  - 内核直接拷贝数据

  使用场景：
  Docker CLI → dockerd（/var/run/docker.sock）
  MySQL 本地连接
  Nginx → PHP-FPM
  Redis 本地通信
```

```go
// Go Unix Socket 服务端
func main() {
    os.Remove("/tmp/app.sock")
    ln, _ := net.Listen("unix", "/tmp/app.sock")
    defer ln.Close()

    for {
        conn, _ := ln.Accept()
        go handleConn(conn)
    }
}
```

## 7. 文件描述符传递？

**回答：**

```
Unix Socket 可以在进程间传递文件描述符：
  进程A 打开文件 → 通过 UDS 传 FD → 进程B 直接使用该 FD

  ┌──────────┐    sendmsg()     ┌──────────┐
  │ 进程A     │───SCM_RIGHTS──→│ 进程B     │
  │ fd=5     │                 │ fd=3     │
  │ 已打开文件 │                 │ 得到同一  │
  │          │                 │ 打开文件  │
  └──────────┘                 └──────────┘

  不是传数字，是传内核中的 file 结构引用
  进程B 得到新的 fd 号，但指向同一资源

应用场景：
  Nginx 平滑升级：
    旧 master → 新 master 传递 listen fd
    不中断已有连接

  特权端口代理：
    root 进程打开 80 端口 → 传 fd 给普通进程

  多进程服务器：
    父进程 accept → 传 conn fd 给工作进程

实现要点：
  sendmsg() + cmsg + SCM_RIGHTS
  接收方 recvmsg() 获取 fd
```

## 8. mmap 用于 IPC？

**回答：**

```
mmap 共享映射用于 IPC：

  方式一：文件映射共享（MAP_SHARED）
    两个进程 mmap 同一文件
    修改直接反映到文件和另一个进程

    进程A: addr = mmap(file, MAP_SHARED)
    进程B: addr = mmap(file, MAP_SHARED)
    → 修改通过页缓存共享

  方式二：匿名映射共享（fork 场景）
    addr = mmap(NULL, size, PROT_READ|PROT_WRITE,
                MAP_SHARED|MAP_ANONYMOUS, -1, 0)
    fork() 后父子进程共享该区域

mmap IPC vs 共享内存：
  ┌──────────────┬──────────────┬──────────────┐
  │ 对比          │ mmap         │ shmget       │
  ├──────────────┼──────────────┼──────────────┤
  │ 接口          │ POSIX        │ System V     │
  │ 文件关联      │ 可有可无      │ 无           │
  │ 持久化        │ 有文件即持久  │ 内核持久     │
  │ 灵活性        │ 高           │ 较低         │
  └──────────────┴──────────────┴──────────────┘

实际使用：
  数据库引擎（SQLite/LMDB 用 mmap 共享数据）
  高性能日志（多进程写入 mmap 文件）
```

## 9. Go 中的 IPC 实践？

**回答：**

```go
// 1. Channel（goroutine 间，非进程间）
ch := make(chan int, 10)
go func() { ch <- 42 }()
val := <-ch

// 2. 进程间 - 使用 net/rpc
// 服务端
type Arith struct{}
func (a *Arith) Multiply(args *Args, reply *int) error {
    *reply = args.A * args.B
    return nil
}
rpc.Register(new(Arith))
ln, _ := net.Listen("tcp", ":1234")
rpc.Accept(ln)

// 客户端
client, _ := rpc.Dial("tcp", "localhost:1234")
var reply int
client.Call("Arith.Multiply", &Args{3, 4}, &reply)

// 3. 进程间 - 使用 gRPC（推荐）
// 定义 proto → 生成代码 → 实现接口

// 4. 进程间 - 使用 Unix Socket
// net.Listen("unix", "/tmp/app.sock")

// 5. 进程间 - 使用共享内存（syscall.Mmap）
data, _ := syscall.Mmap(
    fd, 0, size,
    syscall.PROT_READ|syscall.PROT_WRITE,
    syscall.MAP_SHARED,
)
```

```
Go 进程间通信选型：
  同进程 goroutine → Channel
  本机进程间高性能 → Unix Socket
  本机进程间简单 → net/rpc
  跨机器微服务 → gRPC
  极致性能 → 共享内存 + sync
```

## 10. 进程间通信面试速答？

**回答：**

```
Q: 列举 IPC 方式？
A: 管道、命名管道、消息队列、共享内存
   信号量、信号、Socket、Unix Socket

Q: 最快的 IPC？
A: 共享内存，不需要内核拷贝数据
   但需要同步机制（信号量/互斥锁）

Q: 管道特点？
A: 半双工、内核缓冲区 64KB
   读空阻塞、写满阻塞
   写端关闭读端 EOF

Q: Unix Socket vs TCP？
A: UDS 不经过网络协议栈，快 2-3 倍
   用文件路径寻址，只能本机

Q: 共享内存怎么同步？
A: 信号量（semaphore）
   互斥锁（放共享内存中，PTHREAD_PROCESS_SHARED）
   文件锁（flock）

Q: 信号和信号量区别？
A: 信号：异步通知机制（SIGKILL/SIGTERM等）
   信号量：同步互斥计数器（PV操作）

Q: Go 进程间通信推荐？
A: goroutine 间用 Channel
   本机进程用 Unix Socket
   跨机器用 gRPC

Q: Docker 怎么和 daemon 通信？
A: Unix Domain Socket
   /var/run/docker.sock
```
