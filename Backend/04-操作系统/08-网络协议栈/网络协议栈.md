# 网络协议栈

---

## 1. Linux 网络协议栈架构？

**回答：**

```
Linux 网络数据包收发路径：

  发送路径（自顶向下）：
  ┌──────────────────────┐
  │ 应用层（send/write） │
  ├──────────────────────┤
  │ Socket 层            │ → 发送缓冲区
  ├──────────────────────┤
  │ 传输层（TCP/UDP）    │ → 分段/校验和
  ├──────────────────────┤
  │ 网络层（IP）         │ → 路由查找/TTL
  ├──────────────────────┤
  │ 链路层（Ethernet）   │ → ARP/MAC
  ├──────────────────────┤
  │ 网卡驱动             │ → DMA 发送
  ├──────────────────────┤
  │ 物理网卡（NIC）      │
  └──────────────────────┘

  接收路径（自底向上）：
  网卡 → 硬中断 → DMA 到 Ring Buffer
  → 软中断(NAPI轮询) → 协议栈逐层解析
  → Socket 接收缓冲区 → 唤醒进程
```

## 2. 网卡收包流程？

**回答：**

```
详细收包过程：

  1. 网卡收到数据帧
  2. DMA 将数据写入 Ring Buffer（sk_buff）
  3. 触发硬中断
  4. 硬中断处理：
     - 关闭网卡中断（避免频繁中断）
     - 触发 NAPI 软中断
  5. 软中断/NAPI 轮询：
     - 从 Ring Buffer 批量取包
     - 交给网络协议栈处理
  6. IP 层：校验和/路由/分片重组
  7. TCP 层：序列号/确认/拥塞控制
  8. 数据放入 Socket 接收缓冲区
  9. 唤醒阻塞的读进程

  Ring Buffer 满了怎么办？
  → 丢包！ifconfig 看 RX dropped
  → 调大：ethtool -G eth0 rx 4096

NAPI 机制：
  中断 + 轮询 混合模式
  低负载：中断驱动（低延迟）
  高负载：切换轮询（高吞吐）
  → 避免中断风暴
```

## 3. sk_buff 结构？

**回答：**

```
sk_buff（Socket Buffer）：
  Linux 网络子系统的核心数据结构
  表示一个网络数据包

  ┌─────────────────────────────┐
  │ sk_buff 结构                 │
  ├─────────────────────────────┤
  │ head      → 缓冲区起始       │
  │ data      → 当前协议头起始   │
  │ tail      → 数据末尾         │
  │ end       → 缓冲区末尾       │
  │ mac_header → MAC 头          │
  │ network_header → IP 头       │
  │ transport_header → TCP 头    │
  │ len       → 数据长度         │
  │ protocol  → 协议类型         │
  │ dev       → 网络设备         │
  └─────────────────────────────┘

  协议栈处理过程中：
  接收方向：data 指针上移（剥离各层头部）
  发送方向：data 指针下移（添加各层头部）

  零拷贝相关：
  skb_clone()：共享数据区，独立 sk_buff 头
  skb 支持分散聚集 IO（scatter-gather）
```

## 4. TCP 内核参数调优？

**回答：**

```
关键 TCP 内核参数：

连接建立：
  net.ipv4.tcp_max_syn_backlog = 65535
    → SYN 队列大小（防 SYN Flood）
  net.core.somaxconn = 65535
    → Accept 队列大小
  net.ipv4.tcp_syncookies = 1
    → 开启 SYN Cookie 防攻击

连接复用：
  net.ipv4.tcp_tw_reuse = 1
    → TIME_WAIT 连接复用
  net.ipv4.tcp_fin_timeout = 30
    → FIN_WAIT_2 超时时间

缓冲区：
  net.core.rmem_max = 16777216
  net.core.wmem_max = 16777216
    → 全局收发缓冲区最大值
  net.ipv4.tcp_rmem = "4096 87380 16777216"
  net.ipv4.tcp_wmem = "4096 65536 16777216"
    → TCP 收发缓冲区（最小/默认/最大）

Keep-Alive：
  net.ipv4.tcp_keepalive_time = 600
    → 空闲多久发探测包
  net.ipv4.tcp_keepalive_intvl = 15
    → 探测间隔
  net.ipv4.tcp_keepalive_probes = 5
    → 探测次数

文件描述符：
  fs.file-max = 1000000
    → 系统级最大文件描述符
  ulimit -n 1000000
    → 进程级最大文件描述符
```

## 5. Netfilter 与 iptables？

**回答：**

```
Netfilter：Linux 内核网络包过滤框架
iptables：用户空间配置工具

  数据包经过 Netfilter 的 5 个钩子点：

  入站：  PREROUTING → INPUT → 本机进程
  转发：  PREROUTING → FORWARD → POSTROUTING
  出站：  本机进程 → OUTPUT → POSTROUTING

  ┌──────────┐     ┌──────────┐     ┌──────────┐
  │PREROUTING│────→│ FORWARD  │────→│POSTROUTING│
  └────┬─────┘     └──────────┘     └────┬─────┘
       │ 目标是本机                       │
       ↓                                 ↑
  ┌──────────┐                      ┌──────────┐
  │  INPUT   │──→ 本机进程 ──→      │  OUTPUT  │
  └──────────┘                      └──────────┘

四个表（按优先级）：
  raw → mangle → nat → filter

  filter：包过滤（默认）→ ACCEPT/DROP/REJECT
  nat：网络地址转换 → SNAT/DNAT/MASQUERADE
  mangle：包修改 → TTL/TOS/MARK
  raw：连接跟踪控制

Docker/K8s 大量使用 iptables/nftables 规则
实现容器端口映射和 Service 负载均衡
```

## 6. 网络虚拟化设备？

**回答：**

```
Linux 网络虚拟设备：

  veth pair（虚拟以太网线对）：
    成对出现，一端进一端出
    容器网络核心 → 连接容器和网桥
    ip link add veth0 type veth peer name veth1

  Bridge（网桥）：
    虚拟交换机，二层转发
    docker0 就是一个网桥
    brctl addbr br0

  tun/tap：
    tun：三层虚拟设备（IP 包）→ VPN
    tap：二层虚拟设备（以太网帧）→ 虚拟机

  macvlan：
    在物理网卡上创建多个虚拟 MAC 地址
    容器可直接获得与宿主同网段 IP

  VXLAN：
    在三层网络上构建二层覆盖网络
    跨主机容器通信（K8s Flannel/Calico）
    封装开销 50 字节

  ipvlan：
    类似 macvlan，但共享 MAC 地址
    用 IP 区分，适合 MAC 数量受限场景
```

## 7. 高性能网络技术？

**回答：**

```
DPDK（Data Plane Development Kit）：
  用户态网络库，绕过内核协议栈
  性能：10x-100x → 百万级包处理/秒

  ┌────────────────┐     ┌────────────────┐
  │ 传统路径        │     │ DPDK 路径       │
  │ 网卡            │     │ 网卡            │
  │  ↓ 中断         │     │  ↓ 轮询(PMD)   │
  │ 内核协议栈      │     │ 用户态驱动      │
  │  ↓              │     │  ↓              │
  │ Socket         │     │ 应用直接处理    │
  └────────────────┘     └────────────────┘

  核心技术：
  UIO/VFIO：用户态驱动框架
  Hugepage：大页内存减少 TLB miss
  PMD：轮询模式驱动（无中断）
  Ring Buffer：无锁环形队列

XDP (eXpress Data Path)：
  在网卡驱动层用 eBPF 处理包
  比 iptables 快 10 倍
  动作：XDP_PASS / XDP_DROP / XDP_TX / XDP_REDIRECT

io_uring：
  异步 IO 框架（Linux 5.1+）
  共享内存环形缓冲（SQ/CQ）
  减少系统调用开销
```

## 8. 网络 IO 优化？

**回答：**

```
TCP 优化策略：

  1. Nagle 算法 vs TCP_NODELAY
     Nagle：小包合并发送（减少小包）
     TCP_NODELAY：禁用 Nagle（降低延迟）
     低延迟场景（游戏/交易）要开 NODELAY

  2. TCP_CORK
     积攒数据直到取消 CORK 或超时
     适合 sendfile() 等大块传输

  3. SO_REUSEPORT
     多个进程/线程监听同一端口
     内核级负载均衡到不同 Socket
     Nginx 1.9.1+ 支持

  4. 零拷贝技术
     sendfile()：文件内容直接发送到 Socket
     splice()：管道零拷贝
     MSG_ZEROCOPY：发送时零拷贝（4.14+）

  5. Scatter-Gather IO
     writev()：多缓冲区合并一次写
     避免数据复制到连续缓冲区

  6. 连接池
     复用 TCP 连接，避免频繁三次握手
     HTTP Keep-Alive / gRPC 连接复用

  7. 拥塞控制算法
     BBR：基于带宽和 RTT 估计（推荐）
     sysctl net.ipv4.tcp_congestion_control=bbr
```

## 9. 常见网络问题排查？

**回答：**

```
网络故障排查流程：

  1. 连通性检查
     ping：ICMP 连通
     telnet/nc：端口连通
     traceroute：路径追踪

  2. DNS 问题
     nslookup / dig：DNS 解析
     /etc/resolv.conf：DNS 配置
     systemd-resolve --status

  3. 网络连接状态
     ss -tlnp：监听端口
     ss -s：连接统计
     ss state time-wait | wc -l

  4. 抓包分析
     tcpdump -i eth0 port 80 -w cap.pcap
     Wireshark 分析 pcap 文件

  5. 网络指标
     sar -n DEV 1：网络吞吐
     sar -n TCP 1：TCP 统计
     ethtool -S eth0：网卡统计

常见问题及排查：
  ┌──────────────┬──────────────────────────────┐
  │ 问题          │ 排查方法                      │
  ├──────────────┼──────────────────────────────┤
  │ 连接超时      │ 查防火墙/路由/对端状态        │
  │ TIME_WAIT 多  │ ss 统计 + 调 tw_reuse        │
  │ 丢包          │ ifconfig/ethtool -S 看计数器 │
  │ 延迟高        │ ping/traceroute/mtr          │
  │ 带宽瓶颈      │ iperf3 测带宽                │
  │ RST 包多      │ tcpdump 抓包分析              │
  └──────────────┴──────────────────────────────┘
```

## 10. 网络协议栈面试速答？

**回答：**

```
Q: 数据包从网卡到应用的路径？
A: 网卡DMA→Ring Buffer→硬中断→NAPI软中断
   →IP层→TCP层→Socket缓冲区→唤醒进程

Q: NAPI 是什么？
A: 中断+轮询混合机制
   低负载中断驱动，高负载轮询
   避免中断风暴

Q: TCP 调优关键参数？
A: somaxconn/tcp_max_syn_backlog（队列）
   tcp_tw_reuse（TIME_WAIT复用）
   tcp_rmem/tcp_wmem（缓冲区）

Q: iptables 五链？
A: PREROUTING → INPUT（入站）
   PREROUTING → FORWARD → POSTROUTING（转发）
   OUTPUT → POSTROUTING（出站）

Q: DPDK 为什么快？
A: 绕过内核协议栈，用户态轮询驱动
   大页内存/无锁队列
   性能提升 10-100 倍

Q: 怎么排查网络丢包？
A: ifconfig（RX dropped）
   ethtool -S（网卡级统计）
   netstat -s（协议级统计）
   tcpdump 抓包分析

Q: 什么是 XDP？
A: 在网卡驱动层用 eBPF 处理包
   比 iptables 快 10 倍
   可做 DDoS 防护/负载均衡
```
