# 日志体系

---

## 1. 日志设计原则？

**回答：**

```
  日志的目的：
  1. 问题排查（线上Bug、性能问题）
  2. 审计追踪（谁在什么时候做了什么）
  3. 业务分析（用户行为、业务指标）
  4. 安全监控（异常访问、攻击检测）

  设计原则：
  ┌──────────────────────┬──────────────────────────┐
  │ 原则                  │ 说明                     │
  ├──────────────────────┼──────────────────────────┤
  │ 结构化                │ JSON格式 可机器解析       │
  │ 可追踪                │ TraceID贯穿全链路         │
  │ 分级                  │ Debug/Info/Warn/Error     │
  │ 高性能                │ 异步写/零分配              │
  │ 可收集                │ 统一输出到日志系统         │
  └──────────────────────┴──────────────────────────┘

  不要在日志中包含：
  ✗ 密码/Token/密钥
  ✗ 信用卡号/身份证号（脱敏后可以）
  ✗ 完整请求体（大数据量）
  ✗ 高频循环体内逐条日志

  日志 ≠ print调试工具
  日志是生产环境的眼睛
```

## 2. Go日志库选型？

**回答：**

```
  主流日志库对比：
  ┌──────────────┬──────────┬──────────┬──────────┐
  │ 库            │ 性能     │ 结构化    │ 特点     │
  ├──────────────┼──────────┼──────────┼──────────┤
  │ log/slog     │ 中       │ ✓        │ 标准库1.21│
  │ zap          │ 最高     │ ✓        │ Uber出品  │
  │ zerolog      │ 极高     │ ✓        │ 零分配    │
  │ logrus       │ 中       │ ✓        │ 早期流行  │
  └──────────────┴──────────┴──────────┴──────────┘

  推荐优先级：
  Go 1.21+ 项目 → slog（标准库 无额外依赖）
  高性能场景 → zap / zerolog
  
  zap示例：
```

```go
import "go.uber.org/zap"

func NewLogger(env string) (*zap.Logger, error) {
    if env == "production" {
        cfg := zap.NewProductionConfig()
        cfg.OutputPaths = []string{"stdout"}
        cfg.ErrorOutputPaths = []string{"stderr"}
        return cfg.Build()
    }
    return zap.NewDevelopment()
}

// 使用SugaredLogger（方便但稍慢）
logger, _ := NewLogger("production")
sugar := logger.Sugar()
sugar.Infow("user created",
    "user_id", 12345,
    "email", "user@example.com",
)

// 使用Logger（高性能 强类型）
logger.Info("user created",
    zap.Int64("user_id", 12345),
    zap.String("email", "user@example.com"),
)

// slog示例（Go 1.21+）
slog.Info("user created",
    slog.Int64("user_id", 12345),
    slog.String("email", "user@example.com"),
)
```

## 3. 结构化日志怎么设计？

**回答：**

```
  结构化日志字段规范：
  
  {
    "timestamp": "2024-01-15T10:30:00.123Z",
    "level": "INFO",
    "msg": "order created",
    "trace_id": "abc-123-def",
    "span_id": "span-456",
    "caller": "service/order.go:42",
    "service": "order-svc",
    "env": "production",
    
    // 业务字段
    "user_id": 12345,
    "order_id": "ORD-20240115-001",
    "amount": 99.99,
    
    // 错误信息（Error级别）
    "error": "connection refused",
    "stack": "..."
  }

  字段分类：
  ┌──────────────┬──────────────────────────┐
  │ 类别          │ 字段                     │
  ├──────────────┼──────────────────────────┤
  │ 上下文        │ timestamp, level, caller │
  │ 追踪          │ trace_id, span_id       │
  │ 服务          │ service, env, host      │
  │ 业务          │ user_id, order_id 等    │
  │ 错误          │ error, stack            │
  └──────────────┴──────────────────────────┘

  Go实现 — 通过context传递日志字段：
```

```go
// middleware/logger.go
func LoggerMiddleware(logger *zap.Logger) func(http.Handler) http.Handler {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            traceID := r.Header.Get("X-Trace-ID")
            if traceID == "" {
                traceID = uuid.New().String()
            }
            // 注入到context
            reqLogger := logger.With(
                zap.String("trace_id", traceID),
                zap.String("method", r.Method),
                zap.String("path", r.URL.Path),
            )
            ctx := context.WithValue(r.Context(), loggerKey, reqLogger)
            
            start := time.Now()
            ww := middleware.NewWrapResponseWriter(w, r.ProtoMajor)
            next.ServeHTTP(ww, r.WithContext(ctx))
            
            reqLogger.Info("request completed",
                zap.Int("status", ww.Status()),
                zap.Duration("duration", time.Since(start)),
            )
        })
    }
}

// 从context取Logger
func LoggerFromCtx(ctx context.Context) *zap.Logger {
    if l, ok := ctx.Value(loggerKey).(*zap.Logger); ok {
        return l
    }
    return zap.L() // 全局兜底
}
```

## 4. 日志分级策略？

**回答：**

```
  日志级别与使用场景：
  ┌──────────┬──────────────────────────────────┐
  │ 级别      │ 使用场景                          │
  ├──────────┼──────────────────────────────────┤
  │ DEBUG    │ 开发调试信息 生产不开启            │
  │ INFO     │ 正常业务流程 关键操作记录          │
  │ WARN     │ 有潜在问题但不影响运行            │
  │ ERROR    │ 出错了 需要关注和处理             │
  │ FATAL    │ 致命错误 程序无法继续（慎用）      │
  └──────────┴──────────────────────────────────┘

  各级别示例：

  DEBUG：
  数据库查询SQL、请求参数详情
  → 仅开发/排查时开启

  INFO：
  用户注册成功、订单创建、服务启动
  → 记录关键业务节点

  WARN：
  重试成功、降级触发、配置缺少（用默认值）
  → 不影响当次请求 但需关注

  ERROR：
  数据库连接失败、外部API调用失败、数据不一致
  → 需要排查和修复

  生产环境建议：
  默认级别：INFO
  排查时临时调为DEBUG（支持动态调级）
  ERROR必须配告警

  动态调级实现：
```

```go
import "go.uber.org/zap"

var atomicLevel = zap.NewAtomicLevelAt(zap.InfoLevel)

func init() {
    cfg := zap.NewProductionConfig()
    cfg.Level = atomicLevel
    logger, _ := cfg.Build()
    zap.ReplaceGlobals(logger)
}

// HTTP接口动态调级
// PUT /admin/log-level {"level": "debug"}
func SetLogLevel(w http.ResponseWriter, r *http.Request) {
    var req struct{ Level string }
    json.NewDecoder(r.Body).Decode(&req)
    var lv zapcore.Level
    if err := lv.UnmarshalText([]byte(req.Level)); err != nil {
        http.Error(w, "invalid level", 400)
        return
    }
    atomicLevel.SetLevel(lv)
    w.Write([]byte("ok"))
}
```

## 5. 日志收集架构？

**回答：**

```
  典型日志架构：

  ┌─────────┐  ┌─────────┐  ┌─────────┐
  │ App-1   │  │ App-2   │  │ App-3   │
  │ stdout  │  │ stdout  │  │ stdout  │
  └────┬────┘  └────┬────┘  └────┬────┘
       │            │            │
       └────────────┼────────────┘
                    │
           ┌────────▼────────┐
           │   日志采集器      │
           │ Filebeat/Fluent  │
           └────────┬────────┘
                    │
           ┌────────▼────────┐
           │   消息队列(Kafka)│  缓冲削峰
           └────────┬────────┘
                    │
           ┌────────▼────────┐
           │  日志存储/索引    │
           │ Elasticsearch   │
           │ 或 Loki         │
           └────────┬────────┘
                    │
           ┌────────▼────────┐
           │   可视化查询      │
           │ Kibana / Grafana│
           └─────────────────┘

  两种主流方案对比：
  ┌──────────────┬────────────┬────────────┐
  │ 维度          │ ELK        │ Loki       │
  ├──────────────┼────────────┼────────────┤
  │ 存储          │ 全文索引    │ 只索引标签 │
  │ 资源消耗      │ 高         │ 低         │
  │ 查询能力      │ 强（全文）  │ 标签+grep  │
  │ 运维复杂度    │ 高         │ 低         │
  │ 适合          │ 大规模/全文 │ K8s/轻量  │
  └──────────────┴────────────┴────────────┘

  K8s日志方案：
  应用输出到stdout → K8s收集容器日志
  → DaemonSet部署Fluent Bit/Filebeat
  → 发送到Loki/ES

  日志要输出到stdout/stderr
  不要写到容器内文件（容器销毁就丢失）
```

## 6. 请求链路追踪？

**回答：**

```
  链路追踪作用：

  用户请求 → API Gateway → 服务A → 服务B → DB
                                  → 服务C → Redis

  TraceID贯穿整个调用链
  每个服务片段是一个Span

  OpenTelemetry实现：
```

```go
import (
    "go.opentelemetry.io/otel"
    "go.opentelemetry.io/otel/exporters/otlp/otlptrace"
    sdktrace "go.opentelemetry.io/otel/sdk/trace"
)

// 初始化Tracer
func initTracer() (*sdktrace.TracerProvider, error) {
    exporter, err := otlptrace.New(context.Background(),
        otlptrace.NewGRPCClient(
            otlptrace.WithEndpoint("jaeger:4317"),
            otlptrace.WithInsecure(),
        ),
    )
    if err != nil {
        return nil, err
    }
    
    tp := sdktrace.NewTracerProvider(
        sdktrace.WithBatcher(exporter),
        sdktrace.WithResource(resource.NewWithAttributes(
            semconv.SchemaURL,
            semconv.ServiceNameKey.String("order-service"),
        )),
    )
    otel.SetTracerProvider(tp)
    return tp, nil
}

// 在Service中使用
func (s *OrderService) CreateOrder(ctx context.Context, req CreateOrderReq) error {
    ctx, span := otel.Tracer("order").Start(ctx, "CreateOrder")
    defer span.End()
    
    span.SetAttributes(attribute.Int64("user_id", req.UserID))
    
    // 调用其他服务（context自动传播TraceID）
    user, err := s.userClient.GetUser(ctx, req.UserID)
    if err != nil {
        span.RecordError(err)
        return err
    }
    // ...
}
```

```
  日志与追踪关联：
  在日志中打印TraceID
  在Grafana中从日志跳转到Jaeger追踪

  追踪 → 链路耗时、调用关系
  日志 → 具体错误信息
  指标 → 宏观趋势
  三者关联形成完整可观测性
```

## 7. 日志告警怎么做？

**回答：**

```
  告警分级：
  ┌──────────┬──────────────┬──────────────────┐
  │ 级别      │ 响应时间     │ 通知方式          │
  ├──────────┼──────────────┼──────────────────┤
  │ P0 致命   │ 5分钟内      │ 电话 + 短信      │
  │ P1 严重   │ 30分钟内     │ 即时消息 + 短信  │
  │ P2 一般   │ 工作时间内   │ 即时消息         │
  │ P3 轻微   │ 下个Sprint  │ 邮件/Issue       │
  └──────────┴──────────────┴──────────────────┘

  基于日志的告警规则：
  1. ERROR日志频率突增
     5分钟内ERROR > 100条 → P1告警
  
  2. 关键字匹配
     出现"panic"/"OOM"/"deadlock" → P0告警
  
  3. 特定错误模式
     "connection refused"连续出现 → 依赖服务挂了
  
  4. 业务异常
     "payment failed"频率突增 → 支付通道异常

  告警原则：
  ✓ 告警要可操作（收到告警知道该做什么）
  ✓ 告警要有Runbook链接
  ✓ 避免告警疲劳（太多就被忽视）
  ✗ 不要WARN级别告警
  ✗ 不要可以自愈的告警

  Prometheus + AlertManager：
  基于指标告警（更推荐 比日志告警更可靠）
  错误率 > 1% 持续5分钟 → 告警
  P99延迟 > 2s 持续5分钟 → 告警
```

## 8. 审计日志？

**回答：**

```
  审计日志 vs 应用日志：
  ┌──────────────┬──────────────┬──────────────────┐
  │ 维度          │ 应用日志      │ 审计日志          │
  ├──────────────┼──────────────┼──────────────────┤
  │ 目的          │ 问题排查      │ 合规审计          │
  │ 内容          │ 技术细节      │ 谁做了什么        │
  │ 保留期        │ 7-30天       │ 1-7年            │
  │ 完整性        │ 可丢失       │ 不可丢失不可篡改  │
  │ 存储          │ ES/Loki      │ 独立存储          │
  └──────────────┴──────────────┴──────────────────┘

  审计日志结构：
```

```go
type AuditLog struct {
    ID        string    `json:"id"`
    Timestamp time.Time `json:"timestamp"`
    ActorID   string    `json:"actor_id"`    // 谁
    ActorType string    `json:"actor_type"`  // user/admin/system
    Action    string    `json:"action"`      // create/update/delete
    Resource  string    `json:"resource"`    // order/user
    ResourceID string   `json:"resource_id"` // 操作了什么
    Changes   string    `json:"changes"`     // 变更内容(JSON diff)
    IP        string    `json:"ip"`
    UserAgent string    `json:"user_agent"`
    Result    string    `json:"result"`      // success/failure
}

// 中间件自动记录
func AuditMiddleware(auditSvc AuditService) func(http.Handler) http.Handler {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            if r.Method == http.MethodGet {
                next.ServeHTTP(w, r)
                return // 读操作不记审计
            }
            ww := middleware.NewWrapResponseWriter(w, r.ProtoMajor)
            next.ServeHTTP(ww, r)
            
            result := "success"
            if ww.Status() >= 400 {
                result = "failure"
            }
            auditSvc.Log(r.Context(), AuditLog{
                ActorID:  getUserID(r.Context()),
                Action:   methodToAction(r.Method),
                Resource: extractResource(r.URL.Path),
                IP:       r.RemoteAddr,
                Result:   result,
            })
        })
    }
}
```

## 9. 日志最佳实践清单？

**回答：**

```
  日志DO：
  ✓ 使用结构化日志（JSON）
  ✓ 每条日志包含TraceID
  ✓ 记录关键业务操作
  ✓ 错误日志包含上下文（参数/状态）
  ✓ 输出到stdout（K8s标准）
  ✓ 支持动态调级
  ✓ 错误日志配告警
  ✓ 定义日志字段规范

  日志DON'T：
  ✗ fmt.Println调试日志留在代码里
  ✗ 日志中打印密码/Token
  ✗ 循环体内逐条记录
  ✗ 重复记录（底层记了顶层又记）
  ✗ 只记录"error occurred"没有上下文
  ✗ 用字符串拼接（性能差）
  ✗ 所有日志都用Error级别

  日志格式示例：
  
  好：
  logger.Error("create order failed",
      zap.Int64("user_id", userID),
      zap.String("product_id", productID),
      zap.Error(err),
  )
  
  坏：
  log.Printf("error: %v", err)
  log.Println("创建订单失败")

  性能注意：
  zap比logrus快10倍+
  用Logger而非SugaredLogger（避免interface{}分配）
  高频路径用采样日志（每秒只记录N条）
```

## 10. 日志体系面试速答？

**回答：**

```
Q: 为什么要用结构化日志？
A: JSON格式可机器解析+搜索+统计
   字段标准化 方便日志系统收集

Q: Go日志库推荐？
A: Go 1.21+ → slog 高性能→zap/zerolog
   logrus已不推荐（性能差）

Q: TraceID怎么传递？
A: 中间件生成注入context
   下游服务通过Header传递
   每条日志带trace_id字段

Q: 日志分几个级别？
A: Debug/Info/Warn/Error
   生产默认Info 支持动态调为Debug

Q: 日志收集方案？
A: K8s：stdout → Fluent Bit → Loki/ES → Grafana
   轻量用Loki 全文搜索用ELK

Q: 日志和链路追踪什么关系？
A: 日志记录细节 追踪记录链路
   通过TraceID关联 互相跳转

Q: 审计日志和应用日志区别？
A: 审计：谁做了什么 合规用 不可丢失
   应用：技术排查用 可丢失

Q: 日志性能怎么优化？
A: zap/zerolog零分配 异步写
   高频路径采样 不在循环中打日志
```
