# 性能优化方法论

---

## 1. 性能优化原则？

**回答：**

```
  核心原则：
  ┌──────────────────────────────────────┐
  │ 1. 先度量后优化（不要猜）            │
  │ 2. 优化瓶颈（帕累托定律 20/80）     │
  │ 3. 有基准测试对比（优化前后）        │
  │ 4. 不要过早优化                      │
  │ 5. 优化有代价（复杂度/可读性）       │
  └──────────────────────────────────────┘

  性能优化流程：
  发现问题 → 度量定位 → 分析根因
     → 制定方案 → 实施优化 → 验证效果

  性能指标：
  ┌──────────────┬──────────────────────────┐
  │ 指标          │ 说明                     │
  ├──────────────┼──────────────────────────┤
  │ 吞吐量        │ QPS每秒请求数            │
  │ 延迟          │ P50/P95/P99             │
  │ 错误率        │ 错误请求占比             │
  │ 资源使用      │ CPU/内存/连接数          │
  │ 饱和度        │ 队列/线程池使用率         │
  └──────────────┴──────────────────────────┘

  为什么用P99而不是平均值？
  平均值掩盖长尾问题
  P99 = 99%的请求在这个时间内完成
  10000个请求 P99=200ms → 100个请求>200ms
```

## 3. 数据库性能优化？

**回答：**

```
  数据库优化层次：

  1. SQL优化
     ┌──────────────────────────────────────┐
     │ EXPLAIN ANALYZE查看执行计划          │
     │ 避免SELECT *                         │
     │ 避免N+1查询（用JOIN或批量查询）      │
     │ 分页用游标而非OFFSET                 │
     │ 避免在WHERE中对字段做函数运算         │
     └──────────────────────────────────────┘

  2. 索引优化
     根据查询条件建索引
     组合索引注意最左前缀
     覆盖索引避免回表
     定期检查慢查询日志

  3. 连接池配置
```

```go
db.SetMaxOpenConns(50)       // 最大连接数
db.SetMaxIdleConns(25)       // 空闲连接数
db.SetConnMaxLifetime(30 * time.Minute)  // 连接最大生命周期
db.SetConnMaxIdleTime(5 * time.Minute)   // 空闲连接超时
```

```
  N+1查询问题：
  // 反模式：先查订单列表 再逐个查用户
  orders := db.Query("SELECT * FROM orders LIMIT 100")
  for _, o := range orders {
      user := db.Query("SELECT * FROM users WHERE id = ?", o.UserID)
      // 101次查询！
  }
  
  // 正确：JOIN或IN查询
  SELECT o.*, u.name FROM orders o
  JOIN users u ON o.user_id = u.id LIMIT 100
  // 1次查询

  游标分页（大数据量）：
  // 反模式
  SELECT * FROM orders ORDER BY id OFFSET 10000 LIMIT 20
  // OFFSET越大越慢
  
  // 正确：游标分页
  SELECT * FROM orders WHERE id > ? ORDER BY id LIMIT 20
```

## 4. 缓存优化？

**回答：**

```
  缓存策略：

  1. Cache-Aside（旁路缓存 最常用）
     读：先查缓存→命中返回→未命中查DB→写缓存
     写：先更新DB→删除缓存

  2. 缓存层级
     ┌──────────────────────────────────────┐
     │ 进程内缓存（sync.Map/bigcache）     │
     │ 延迟：纳秒级 容量：有限              │
     ├──────────────────────────────────────┤
     │ 分布式缓存（Redis）                  │
     │ 延迟：毫秒级 容量：大                │
     ├──────────────────────────────────────┤
     │ 数据库                               │
     │ 延迟：毫秒-秒级                      │
     └──────────────────────────────────────┘

  Go实现示例：
```

```go
func (s *UserService) GetUser(ctx context.Context, id int64) (*User, error) {
    key := fmt.Sprintf("user:%d", id)
    
    // 1. 查Redis缓存
    cached, err := s.rdb.Get(ctx, key).Result()
    if err == nil {
        var user User
        json.Unmarshal([]byte(cached), &user)
        return &user, nil
    }
    
    // 2. 缓存未命中 查DB
    user, err := s.repo.GetByID(ctx, id)
    if err != nil {
        return nil, err
    }
    
    // 3. 写入缓存（带过期时间）
    data, _ := json.Marshal(user)
    s.rdb.Set(ctx, key, data, 30*time.Minute)
    
    return user, nil
}
```

```
  缓存问题与解决：
  ┌──────────────┬──────────────────────────┐
  │ 问题          │ 解决                     │
  ├──────────────┼──────────────────────────┤
  │ 缓存穿透      │ 布隆过滤器/空值缓存      │
  │ 缓存击穿      │ singleflight/互斥锁      │
  │ 缓存雪崩      │ 过期时间加随机值          │
  │ 一致性        │ 延迟双删/消息队列         │
  └──────────────┴──────────────────────────┘
```

## 5. 并发优化？

**回答：**

```
  Go并发优化手段：

  1. sync.Pool复用对象
```

```go
var bufferPool = sync.Pool{
    New: func() interface{} {
        return new(bytes.Buffer)
    },
}

func processRequest() {
    buf := bufferPool.Get().(*bytes.Buffer)
    defer func() {
        buf.Reset()
        bufferPool.Put(buf)
    }()
    // 使用buf...
}
```

```
  2. singleflight合并请求
```

```go
var group singleflight.Group

func GetUser(ctx context.Context, id int64) (*User, error) {
    key := fmt.Sprintf("user:%d", id)
    v, err, _ := group.Do(key, func() (interface{}, error) {
        // 并发请求同一个用户 只查一次DB
        return repo.GetByID(ctx, id)
    })
    if err != nil {
        return nil, err
    }
    return v.(*User), nil
}
```

```
  3. 并行化独立任务
```

```go
func GetOrderDetail(ctx context.Context, orderID int64) (*OrderDetail, error) {
    var (
        order    *Order
        user     *User
        products []*Product
    )
    
    g, ctx := errgroup.WithContext(ctx)
    
    g.Go(func() error {
        var err error
        order, err = orderRepo.GetByID(ctx, orderID)
        return err
    })
    g.Go(func() error {
        var err error
        user, err = userClient.GetUser(ctx, order.UserID)
        return err
    })
    g.Go(func() error {
        var err error
        products, err = productClient.BatchGet(ctx, order.ProductIDs)
        return err
    })
    
    if err := g.Wait(); err != nil {
        return nil, err
    }
    return &OrderDetail{Order: order, User: user, Products: products}, nil
}
```

```
  4. 限制并发度
```

```go
// 信号量限制并发
sem := make(chan struct{}, 10) // 最多10个并发
for _, item := range items {
    sem <- struct{}{}
    go func(it Item) {
        defer func() { <-sem }()
        process(it)
    }(item)
}
```

## 7. 内存优化？

**回答：**

```
  Go内存优化手段：

  1. 减少内存分配
```

```go
// 反模式：字符串拼接大量分配
var s string
for i := 0; i < 1000; i++ {
    s += fmt.Sprintf("item-%d,", i)
}

// 正确：strings.Builder
var b strings.Builder
for i := 0; i < 1000; i++ {
    fmt.Fprintf(&b, "item-%d,", i)
}
s := b.String()
```

```
  2. 预分配切片
```

```go
// 反模式：不断append扩容
var result []int
for i := 0; i < 10000; i++ {
    result = append(result, i)
}

// 正确：预分配
result := make([]int, 0, 10000)
for i := 0; i < 10000; i++ {
    result = append(result, i)
}
```

```
  3. 逃逸分析
  go build -gcflags="-m" ./...
  
  逃逸到堆 = 需要GC回收 = 更慢
  尽量让变量留在栈上：
  - 不返回局部变量的指针（如果不必要）
  - 小对象用值传递
  - 接口类型会导致逃逸

  4. sync.Pool复用
  适合高频创建销毁的临时对象
  如：bytes.Buffer / 临时结构体

  5. 内存泄漏排查
  pprof heap → 看哪些对象占内存最多
  goroutine泄漏 → 本质也是内存泄漏

  常见泄漏原因：
  ✗ goroutine阻塞未退出
  ✗ time.After在循环中使用
  ✗ 全局Map不断增长没有清理
  ✗ 子字符串引用大字符串
```

## 8. 性能测试方法？

**回答：**

```
  压测工具：
  ┌──────────────┬──────────────────────────┐
  │ 工具          │ 特点                     │
  ├──────────────┼──────────────────────────┤
  │ wrk          │ 轻量HTTP压测             │
  │ k6           │ 脚本化 支持复杂场景       │
  │ vegeta       │ Go编写 恒定速率          │
  │ pprof        │ 代码级别Benchmark        │
  └──────────────┴──────────────────────────┘

  k6压测示例：
  import http from 'k6/http';
  import { check } from 'k6';
  
  export const options = {
    stages: [
      { duration: '1m', target: 100 },   // 1分钟升到100并发
      { duration: '3m', target: 100 },   // 维持3分钟
      { duration: '1m', target: 0 },     // 1分钟降到0
    ],
    thresholds: {
      http_req_duration: ['p(99)<200'],  // P99<200ms
      http_req_failed: ['rate<0.01'],    // 错误率<1%
    },
  };
  
  export default function() {
    const res = http.get('http://localhost:8080/api/v1/users/1');
    check(res, { 'status 200': (r) => r.status === 200 });
  }

  压测原则：
  1. 生产级别环境压测（不在本地）
  2. 逐步加压（不要一开始就满载）
  3. 关注P99而非平均值
  4. 压测时监控资源（CPU/内存/DB连接数）
  5. 找到系统瓶颈后再优化
```

## 9. 性能优化实战清单？

**回答：**

```
  按收益排序（高→低）：

  1. 数据库优化（通常是最大瓶颈）
     □ 加索引
     □ 消除N+1查询
     □ 慢查询分析
     □ 连接池调优
  
  2. 缓存
     □ 热数据放Redis
     □ 本地缓存（进程内）
     □ singleflight合并请求
  
  3. 并发优化
     □ 并行化独立任务（errgroup）
     □ 限制并发度
     □ 异步处理非关键路径
  
  4. 网络优化
     □ HTTP连接复用
     □ gRPC替代REST（内部调用）
     □ 批量操作替代逐条
  
  5. 代码优化
     □ sync.Pool复用对象
     □ 预分配切片
     □ strings.Builder替代拼接
  
  6. 架构优化
     □ 异步化（消息队列）
     □ 读写分离
     □ 分库分表
     □ CDN加速静态资源

  优化时要问自己：
  这个优化值得吗？（收益/成本）
  有Benchmark证明吗？
  增加了多少复杂度？
```

## 10. 性能优化面试速答？

**回答：**

```
Q: 性能优化原则？
A: 先度量后优化 优化瓶颈
   有Benchmark对比 不过早优化

Q: Go用什么工具做性能分析？
A: pprof(CPU/heap/goroutine)
   火焰图可视化 Benchmark基准测试

Q: 数据库怎么优化？
A: 索引+消除N+1+连接池调优
   EXPLAIN分析执行计划 游标分页

Q: 缓存策略？
A: Cache-Aside最常用
   穿透→布隆 击穿→singleflight 雪崩→随机过期

Q: Go并发怎么优化？
A: sync.Pool复用+singleflight合并
   errgroup并行+信号量限并发

Q: 内存怎么优化？
A: 预分配切片+strings.Builder
   逃逸分析+sync.Pool+pprof heap

Q: 压测怎么做？
A: k6/wrk压测 关注P99/错误率
   逐步加压 压测时监控资源

Q: 最常见的性能瓶颈？
A: 数据库(索引/N+1/慢查询)最常见
   其次是缓存缺失和网络调用
```
