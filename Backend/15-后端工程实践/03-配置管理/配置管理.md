# 配置管理

---

## 1. 配置管理原则？

**回答：**

```
  12-Factor App 配置原则：
  配置与代码严格分离
  同一份代码 不同配置 = 不同环境

  配置分层加载（优先级递增）：
  ┌────────────────────────────────────┐
  │ 4. 命令行参数     --port=8080     │ ← 最高优先
  ├────────────────────────────────────┤
  │ 3. 环境变量       APP_PORT=8080   │
  ├────────────────────────────────────┤
  │ 2. 配置文件       config.yaml     │
  ├────────────────────────────────────┤
  │ 1. 代码默认值     port: 8080      │ ← 最低优先
  └────────────────────────────────────┘

  配置分类：
  ┌──────────────┬──────────────┬──────────────────┐
  │ 类别          │ 示例          │ 管理方式          │
  ├──────────────┼──────────────┼──────────────────┤
  │ 基础配置      │ 端口/超时     │ 配置文件           │
  │ 环境配置      │ DB地址       │ 环境变量           │
  │ 敏感配置      │ 密码/密钥     │ Vault/K8s Secret  │
  │ 特性开关      │ Feature Flag │ 远程配置中心       │
  │ 动态配置      │ 限流阈值     │ 配置中心(实时推送)  │
  └──────────────┴──────────────┴──────────────────┘
```

## 2. Go配置管理实践？

**回答：**

```
  Viper使用示例：
```

```go
// internal/config/config.go
type Config struct {
    Server   ServerConfig   `mapstructure:"server"`
    Database DatabaseConfig `mapstructure:"database"`
    Redis    RedisConfig    `mapstructure:"redis"`
    Log      LogConfig      `mapstructure:"log"`
}

type ServerConfig struct {
    Port         int           `mapstructure:"port"`
    ReadTimeout  time.Duration `mapstructure:"read_timeout"`
    WriteTimeout time.Duration `mapstructure:"write_timeout"`
}

type DatabaseConfig struct {
    Host     string `mapstructure:"host"`
    Port     int    `mapstructure:"port"`
    User     string `mapstructure:"user"`
    Password string `mapstructure:"password"` // 从环境变量取
    DBName   string `mapstructure:"dbname"`
    MaxConns int    `mapstructure:"max_conns"`
}

func Load() (*Config, error) {
    v := viper.New()
    
    // 1. 设置默认值
    v.SetDefault("server.port", 8080)
    v.SetDefault("server.read_timeout", "10s")
    v.SetDefault("database.max_conns", 20)
    
    // 2. 读取配置文件
    v.SetConfigName("config")
    v.SetConfigType("yaml")
    v.AddConfigPath("./configs")
    v.AddConfigPath(".")
    if err := v.ReadInConfig(); err != nil {
        return nil, fmt.Errorf("read config: %w", err)
    }
    
    // 3. 环境变量覆盖
    v.SetEnvPrefix("APP")
    v.SetEnvKeyReplacer(strings.NewReplacer(".", "_"))
    v.AutomaticEnv()
    // APP_DATABASE_PASSWORD=xxx 自动覆盖 database.password
    
    // 4. 解析到结构体
    var cfg Config
    if err := v.Unmarshal(&cfg); err != nil {
        return nil, fmt.Errorf("unmarshal config: %w", err)
    }
    return &cfg, nil
}
```

```
  配置文件示例（configs/config.yaml）：

  server:
    port: 8080
    read_timeout: 10s
    write_timeout: 10s
  
  database:
    host: localhost
    port: 5432
    user: myapp
    # password通过环境变量 APP_DATABASE_PASSWORD 注入
    dbname: myapp_db
    max_conns: 20
  
  redis:
    addr: localhost:6379
    db: 0
  
  log:
    level: info
    format: json
```

## 3. 敏感配置怎么管理？

**回答：**

```
  原则：敏感信息绝不入代码仓库

  方案对比：
  ┌──────────────┬──────────┬──────────┬──────────┐
  │ 方案          │ 安全性   │ 复杂度    │ 适用     │
  ├──────────────┼──────────┼──────────┼──────────┤
  │ 环境变量      │ 中       │ 低       │ 小项目   │
  │ K8s Secret   │ 中       │ 中       │ K8s环境  │
  │ Vault        │ 高       │ 高       │ 大规模   │
  │ 云KMS        │ 高       │ 中       │ 云原生   │
  └──────────────┴──────────┴──────────┴──────────┘

  1. 环境变量（最简单）：
  export APP_DATABASE_PASSWORD=secret123
  docker run -e APP_DATABASE_PASSWORD=secret123

  2. K8s Secret + ExternalSecrets：
  apiVersion: external-secrets.io/v1beta1
  kind: ExternalSecret
  spec:
    secretStoreRef:
      name: vault-backend
    data:
      - secretKey: db-password
        remoteRef:
          key: myapp/database
          property: password

  3. HashiCorp Vault：
  vault kv put secret/myapp/db password=xxx
  应用启动时通过API获取

  防护措施：
  ✓ .gitignore排除配置文件
  ✓ pre-commit检查是否有密钥泄露（truffleHog/gitleaks）
  ✓ 定期轮换密钥
  ✓ 日志中不打印敏感配置
```

## 4. 多环境配置管理？

**回答：**

```
  环境分类：
  ┌──────────────┬──────────────────────────┐
  │ 环境          │ 特点                     │
  ├──────────────┼──────────────────────────┤
  │ local        │ 本地开发 连本地DB         │
  │ dev          │ 开发环境 共享数据          │
  │ staging      │ 预发布 配置同生产         │
  │ production   │ 生产 最严格              │
  └──────────────┴──────────────────────────┘

  管理方式：
  configs/
  ├── config.yaml          # 通用默认值
  ├── config.local.yaml    # 本地覆盖
  ├── config.dev.yaml      # 开发环境覆盖
  ├── config.staging.yaml  # 预发覆盖
  └── config.prod.yaml     # 生产覆盖

  加载逻辑：
```

```go
func Load(env string) (*Config, error) {
    v := viper.New()
    
    // 加载通用配置
    v.SetConfigName("config")
    v.AddConfigPath("./configs")
    v.ReadInConfig()
    
    // 加载环境特定配置（覆盖）
    v.SetConfigName("config." + env)
    v.MergeInConfig() // 合并 不是替换
    
    // 环境变量最终覆盖
    v.AutomaticEnv()
    
    var cfg Config
    v.Unmarshal(&cfg)
    return &cfg, nil
}

// 使用
// APP_ENV=staging go run ./cmd/server/
```

```
  K8s多环境管理：
  用Kustomize overlay：
  k8s/
  ├── base/
  │   ├── deployment.yaml
  │   └── kustomization.yaml
  ├── overlays/
  │   ├── dev/
  │   │   └── kustomization.yaml
  │   ├── staging/
  │   │   └── kustomization.yaml
  │   └── prod/
  │       └── kustomization.yaml

  原则：
  各环境差异只在配置 代码完全相同
  staging配置尽量和prod一致（只改地址）
```

## 5. 特性开关（Feature Flag）？

**回答：**

```
  特性开关用途：
  1. 渐进发布（灰度放量）
  2. 快速回滚（不需要部署 关闭开关即可）
  3. A/B测试
  4. 开发中隐藏未完成功能

  实现方式：
```

```go
// 简单实现 — 配置驱动
type FeatureFlags struct {
    NewPaymentFlow  bool `mapstructure:"new_payment_flow"`
    AsyncNotify     bool `mapstructure:"async_notify"`
    DarkMode        bool `mapstructure:"dark_mode"`
}

func (s *OrderService) CreateOrder(ctx context.Context, req CreateOrderReq) error {
    if s.flags.NewPaymentFlow {
        return s.newPaymentFlow(ctx, req)
    }
    return s.legacyPaymentFlow(ctx, req)
}

// 进阶实现 — 按用户/百分比
type Flag struct {
    Name       string
    Enabled    bool
    Percentage int      // 0-100 放量比例
    Whitelist  []string // 白名单用户ID
}

func (f *Flag) IsEnabled(userID string) bool {
    if !f.Enabled {
        return false
    }
    // 白名单优先
    for _, id := range f.Whitelist {
        if id == userID {
            return true
        }
    }
    // 按百分比
    hash := fnv.New32()
    hash.Write([]byte(userID + f.Name))
    return int(hash.Sum32()%100) < f.Percentage
}
```

```
  专业Feature Flag平台：
  开源：Unleash / Flagsmith
  商业：LaunchDarkly / Split
  
  特性开关生命周期：
  开发完成 → 开关控制灰度 → 全量放开 → 删除开关
  
  注意：
  开关不要累积太多
  全量后及时清理（否则成技术债）
  关键开关变更要记审计日志
```

## 6. 配置热更新？

**回答：**

```
  热更新场景：
  不重启服务 动态修改配置
  如：限流阈值、日志级别、特性开关

  Viper文件监听：
```

```go
func WatchConfig(v *viper.Viper, cfg *Config, mu *sync.RWMutex) {
    v.WatchConfig()
    v.OnConfigChange(func(e fsnotify.Event) {
        log.Info("config changed", "file", e.Name)
        mu.Lock()
        defer mu.Unlock()
        if err := v.Unmarshal(cfg); err != nil {
            log.Error("reload config failed", "error", err)
        }
    })
}

// 使用时加读锁
func (s *Service) GetRateLimit() int {
    s.cfgMu.RLock()
    defer s.cfgMu.RUnlock()
    return s.cfg.RateLimit
}
```

```
  配置中心方案：
  ┌──────────────┬──────────────────────────┐
  │ 方案          │ 特点                     │
  ├──────────────┼──────────────────────────┤
  │ etcd         │ K8s底层 强一致性          │
  │ Consul       │ 服务发现+配置 一体化      │
  │ Nacos        │ 阿里 配置+服务发现        │
  │ Apollo       │ 携程 专注配置管理          │
  │ K8s ConfigMap│ Watch API 简单场景        │
  └──────────────┴──────────────────────────┘

  注意事项：
  1. 热更新的配置必须线程安全（RWMutex/atomic）
  2. 更新失败要回滚 不能用半成品配置
  3. 配置变更要有审计日志
  4. 有些配置不适合热更新（DB连接池大小）
```

## 7. 配置验证？

**回答：**

```
  启动时验证配置 及早失败：
```

```go
func (c *Config) Validate() error {
    if c.Server.Port <= 0 || c.Server.Port > 65535 {
        return fmt.Errorf("invalid server port: %d", c.Server.Port)
    }
    if c.Database.Host == "" {
        return errors.New("database host is required")
    }
    if c.Database.MaxConns <= 0 {
        return errors.New("database max_conns must be positive")
    }
    if c.Server.ReadTimeout <= 0 {
        return errors.New("server read_timeout must be positive")
    }
    return nil
}

// main.go
func main() {
    cfg, err := config.Load()
    if err != nil {
        log.Fatal("load config", "error", err)
    }
    if err := cfg.Validate(); err != nil {
        log.Fatal("invalid config", "error", err)
    }
    // ...
}
```

```
  使用validator库（推荐）：
```

```go
import "github.com/go-playground/validator/v10"

type Config struct {
    Server struct {
        Port         int           `mapstructure:"port" validate:"required,min=1,max=65535"`
        ReadTimeout  time.Duration `mapstructure:"read_timeout" validate:"required,min=1s"`
    } `mapstructure:"server"`
    Database struct {
        Host     string `mapstructure:"host" validate:"required"`
        Port     int    `mapstructure:"port" validate:"required,min=1,max=65535"`
        User     string `mapstructure:"user" validate:"required"`
        Password string `mapstructure:"password" validate:"required"`
        DBName   string `mapstructure:"dbname" validate:"required"`
        MaxConns int    `mapstructure:"max_conns" validate:"required,min=1,max=1000"`
    } `mapstructure:"database"`
}

func (c *Config) Validate() error {
    validate := validator.New()
    return validate.Struct(c)
}
```

```
  验证原则：
  必填检查（required）
  范围检查（端口/连接数/超时）
  格式检查（URL/邮箱）
  依赖检查（启用Redis缓存时Redis地址必填）
  启动时验证失败 = 立即退出（fail fast）
```

## 8. 配置即代码？

**回答：**

```
  Configuration as Code：
  配置文件纳入版本控制
  变更通过PR审核
  自动化部署应用配置

  K8s ConfigMap管理：
  # k8s/configmap.yaml
  apiVersion: v1
  kind: ConfigMap
  metadata:
    name: myapp-config
  data:
    config.yaml: |
      server:
        port: 8080
      log:
        level: info

  Kustomize配置差异化：
  # k8s/overlays/prod/kustomization.yaml
  resources:
    - ../../base
  patches:
    - patch: |-
        - op: replace
          path: /data/config.yaml
          value: |
            server:
              port: 8080
            log:
              level: warn

  GitOps流程：
  修改配置文件 → 提交PR → Code Review → 合并
  → ArgoCD检测变更 → 自动同步到K8s

  好处：
  ✓ 配置变更有审计（Git历史）
  ✓ 可以回滚（git revert）
  ✓ 多环境差异清晰
  ✓ 自动化部署 减少人工操作失误
```

## 9. 配置管理反模式？

**回答：**

```
  反模式清单：

  1. 硬编码
     ✗ db, _ := sql.Open("postgres", "host=prod...")
     ✓ db, _ := sql.Open("postgres", cfg.Database.DSN())

  2. 配置散落各处
     ✗ 有的在代码里 有的在环境变量 有的在文件
     ✓ 统一入口Load() 分层加载

  3. 敏感信息入仓库
     ✗ password: "real-password" 提交到Git
     ✓ 环境变量/Vault 配置文件只放占位符

  4. 配置无默认值
     ✗ 忘设某个配置 启动时空指针panic
     ✓ 所有配置有合理默认值 + 启动校验

  5. 环境间配置差异大
     ✗ dev和prod代码结构不同需要不同配置Key
     ✓ 相同Key 不同值 通过overlay覆盖

  6. 全局变量传配置
     ✗ var GlobalConfig Config 各处直接访问
     ✓ 依赖注入 构造函数传入需要的配置

  7. 配置缺少文档
     ✗ 新人不知道有哪些配置 每个配置什么含义
     ✓ config.example.yaml + 注释说明

  最佳实践总结：
  统一入口 + 分层加载 + 启动校验
  敏感数据外置 + 环境差异最小化
  配置即代码 + Git管理 + 自动部署
```

## 10. 配置管理面试速答？

**回答：**

```
Q: 配置管理的核心原则？
A: 配置与代码分离 分层加载
   默认值→文件→环境变量→命令行

Q: Go用什么配置库？
A: Viper 支持多格式/多来源/环境变量/热更新
   结构体映射用mapstructure tag

Q: 敏感配置怎么管？
A: 绝不入Git 用环境变量/K8s Secret/Vault
   pre-commit检查泄露(gitleaks)

Q: 多环境怎么管理？
A: 基础配置+环境override
   K8s用Kustomize overlay

Q: Feature Flag什么场景用？
A: 灰度发布/快速回滚/A/B测试
   全量后及时清理开关

Q: 配置热更新注意什么？
A: 线程安全(RWMutex) 更新失败不能用半成品
   有些配置不适合热更(连接池)

Q: 配置验证什么时候做？
A: 启动时立即验证 fail fast
   必填+范围+格式+依赖检查

Q: 什么是配置即代码？
A: 配置纳入Git版本控制 PR审核
   GitOps自动同步 支持回滚
```
