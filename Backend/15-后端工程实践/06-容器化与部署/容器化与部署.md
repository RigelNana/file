## 1. Docker核心概念？

**回答：**

```
  Docker三要素：

  ┌──────────────────────────────────────┐
  │ 镜像（Image）                        │
  │ 只读模板 包含运行环境+代码           │
  │ 分层存储 共享基础层                   │
  ├──────────────────────────────────────┤
  │ 容器（Container）                    │
  │ 镜像的运行实例                       │
  │ 可写层 + 网络 + 存储                  │
  ├──────────────────────────────────────┤
  │ 仓库（Registry）                     │
  │ 镜像的存储分发中心                   │
  │ Docker Hub / ECR / Harbor            │
  └──────────────────────────────────────┘

  容器 vs 虚拟机：
  ┌──────────────┬──────────────┬──────────────┐
  │ 维度          │ 容器          │ 虚拟机       │
  ├──────────────┼──────────────┼──────────────┤
  │ 隔离级别      │ 进程级       │ 系统级        │
  │ 启动速度      │ 秒级         │ 分钟级        │
  │ 资源占用      │ MB级         │ GB级          │
  │ 镜像大小      │ MB级         │ GB级          │
  │ 性能          │ 接近原生     │ 有损耗        │
  └──────────────┴──────────────┴──────────────┘

  容器底层技术：
  Namespace：进程/网络/文件系统隔离
  Cgroups：CPU/内存资源限制
  UnionFS：分层文件系统（overlay2）
```

## 2. Dockerfile最佳实践？

**回答：**

```
  Go项目Dockerfile（生产级）：
```

```dockerfile
# syntax=docker/dockerfile:1

# 构建阶段
FROM golang:1.22-alpine AS builder
RUN apk add --no-cache ca-certificates tzdata
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux GOARCH=amd64 \
    go build -ldflags="-s -w -X main.version=$(git describe --tags)" \
    -o /server ./cmd/server/

# 运行阶段
FROM gcr.io/distroless/static-debian12
COPY --from=builder /usr/share/zoneinfo /usr/share/zoneinfo
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/
COPY --from=builder /server /server
COPY --from=builder /app/configs /configs
ENV TZ=Asia/Shanghai
EXPOSE 8080
USER nonroot:nonroot
ENTRYPOINT ["/server"]
```

```
  .dockerignore：
  .git
  .github
  .vscode
  *.md
  Makefile
  docker-compose*.yml
  vendor/

  最佳实践清单：
  ┌──────────────────────────────────────┐
  │ ✓ 多阶段构建（减少镜像体积）        │
  │ ✓ 先COPY go.mod再COPY .（利用缓存）│
  │ ✓ 最小基础镜像（distroless/scratch）│
  │ ✓ 非root用户运行                     │
  │ ✓ CGO_ENABLED=0（纯Go静态链接）     │
  │ ✓ .dockerignore排除无关文件          │
  │ ✓ 固定基础镜像版本（不用latest）     │
  │ ✓ 健康检查 HEALTHCHECK              │
  └──────────────────────────────────────┘
```

## 3. K8s部署配置？

**回答：**

```
  完整的K8s部署清单：
```

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
  labels:
    app: myapp
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1
      maxSurge: 1
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
        - name: myapp
          image: registry.example.com/myapp:v1.2.0
          ports:
            - containerPort: 8080
          env:
            - name: APP_ENV
              value: production
            - name: DATABASE_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: db-secret
                  key: password
          resources:
            requests:
              cpu: 100m
              memory: 128Mi
            limits:
              cpu: 500m
              memory: 512Mi
          livenessProbe:
            httpGet:
              path: /healthz
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 10
          readinessProbe:
            httpGet:
              path: /readyz
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 5
          lifecycle:
            preStop:
              exec:
                command: ["sh", "-c", "sleep 5"]
      terminationGracePeriodSeconds: 30
---
apiVersion: v1
kind: Service
metadata:
  name: myapp
spec:
  selector:
    app: myapp
  ports:
    - port: 80
      targetPort: 8080
  type: ClusterIP
```

```
  关键配置说明：
  resources：资源限制防止单Pod吃满节点
  livenessProbe：存活检查 失败重启Pod
  readinessProbe：就绪检查 失败摘除流量
  preStop+sleep：优雅关闭 等LB摘除后再停
  terminationGracePeriodSeconds：关闭等待时间
```

## 4. 健康检查怎么实现？

**回答：**

```
  三种健康检查：
  ┌──────────────┬──────────────────────────┐
  │ 类型          │ 用途                     │
  ├──────────────┼──────────────────────────┤
  │ liveness     │ 是否存活（失败则重启）    │
  │ readiness    │ 是否就绪（失败则摘流量）  │
  │ startup      │ 是否启动完成（慢启动）    │
  └──────────────┴──────────────────────────┘

  Go实现：
```

```go
func main() {
    // ... 初始化
    
    // 健康检查路由
    r.Get("/healthz", func(w http.ResponseWriter, r *http.Request) {
        w.WriteHeader(http.StatusOK)
        w.Write([]byte("ok"))
    })
    
    r.Get("/readyz", func(w http.ResponseWriter, r *http.Request) {
        // 检查关键依赖
        if err := db.PingContext(r.Context()); err != nil {
            http.Error(w, "db not ready", http.StatusServiceUnavailable)
            return
        }
        if err := rdb.Ping(r.Context()).Err(); err != nil {
            http.Error(w, "redis not ready", http.StatusServiceUnavailable)
            return
        }
        w.WriteHeader(http.StatusOK)
        w.Write([]byte("ready"))
    })
}
```

```
  设计原则：
  /healthz（存活）：只检查进程是否正常（简单快速）
  /readyz （就绪）：检查依赖是否可用（DB/Redis/外部服务）
  
  配置建议：
  liveness：
    initialDelaySeconds: 10（等应用启动）
    periodSeconds: 10
    failureThreshold: 3（连续3次失败才重启）
  
  readiness：
    periodSeconds: 5（更频繁检查）
    failureThreshold: 1（1次失败就摘流量）

  常见问题：
  ✗ liveness检查DB → DB挂了所有Pod重启（雪崩）
  ✓ liveness只检查进程 readiness检查依赖
```

## 5. 优雅关闭怎么做？

**回答：**

```
  优雅关闭流程（K8s）：

  1. K8s发送SIGTERM
  2. preStop hook执行（sleep 5s）
     这期间K8s从Service Endpoints移除Pod
  3. 应用收到信号 停止接受新请求
  4. 等待进行中的请求完成（有超时）
  5. 关闭资源（DB连接/消息队列等）
  6. 进程退出

  Go实现：
```

```go
func main() {
    srv := &http.Server{
        Addr:    ":8080",
        Handler: router,
    }

    // 启动服务
    go func() {
        if err := srv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
            log.Fatal("listen error", "error", err)
        }
    }()

    // 等待终止信号
    quit := make(chan os.Signal, 1)
    signal.Notify(quit, syscall.SIGINT, syscall.SIGTERM)
    <-quit
    log.Info("shutting down...")

    // 给进行中的请求最多30秒完成
    ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
    defer cancel()

    // 停止接受新请求 等待已有请求完成
    if err := srv.Shutdown(ctx); err != nil {
        log.Error("forced shutdown", "error", err)
    }

    // 关闭其他资源
    db.Close()
    rdb.Close()
    log.Info("server exited")
}
```

```
  K8s preStop配置：
  lifecycle:
    preStop:
      exec:
        command: ["sh", "-c", "sleep 5"]
  terminationGracePeriodSeconds: 35
  # 5(preStop) + 30(Shutdown超时) = 35

  为什么需要preStop sleep？
  K8s Endpoints更新和SIGTERM几乎同时
  kube-proxy/ingress可能还没更新路由
  sleep几秒让流量先切走
```

## 6. 资源管理与HPA？

**回答：**

```
  资源配置（requests/limits）：
  resources:
    requests:          # 调度保证（Node必须有这么多空闲）
      cpu: 100m        # 0.1核
      memory: 128Mi    # 128MB
    limits:            # 上限（超过被throttle/OOMKill）
      cpu: 500m        # 0.5核
      memory: 512Mi

  配置建议：
  ┌──────────────────────────────────────┐
  │ requests = 日常用量                   │
  │ limits = 峰值用量的1.5-2倍           │
  │ 不设limits可能影响其他Pod            │
  │ limits过小会频繁OOMKill              │
  └──────────────────────────────────────┘

  HPA（水平自动扩缩容）：
  apiVersion: autoscaling/v2
  kind: HorizontalPodAutoscaler
  metadata:
    name: myapp
  spec:
    scaleTargetRef:
      apiVersion: apps/v1
      kind: Deployment
      name: myapp
    minReplicas: 2
    maxReplicas: 10
    metrics:
      - type: Resource
        resource:
          name: cpu
          target:
            type: Utilization
            averageUtilization: 70
      - type: Resource
        resource:
          name: memory
          target:
            type: Utilization
            averageUtilization: 80

  HPA工作原理：
  CPU使用率 > 70% → 增加Pod
  CPU使用率 < 70% → 减少Pod（有冷却期）
  
  注意：
  必须设置requests HPA才生效
  冷却期默认5分钟（防止频繁扩缩）
  需要Metrics Server
```

## 7. 镜像安全？

**回答：**

```
  镜像安全扫描：
  ┌──────────────┬──────────────────────────┐
  │ 工具          │ 特点                     │
  ├──────────────┼──────────────────────────┤
  │ Trivy        │ 开源 全面 CI友好          │
  │ Snyk         │ 商业 集成好              │
  │ Cosign       │ 镜像签名验证              │
  └──────────────┴──────────────────────────┘

  CI中扫描：
  - name: Trivy scan
    uses: aquasecurity/trivy-action@master
    with:
      image-ref: myapp:${{ github.sha }}
      severity: CRITICAL,HIGH
      exit-code: 1  # 有高危漏洞CI失败

  安全最佳实践：
  1. 最小基础镜像（distroless没Shell无法被利用）
  2. 非root运行（USER nonroot）
  3. 固定版本标签（不用latest）
  4. 定期更新基础镜像
  5. 镜像签名（Cosign）
  6. 私有仓库（Harbor/ECR）

  镜像签名流程：
  构建 → 扫描 → 签名 → 推送
  部署时验证签名 → 只允许签名过的镜像运行

  K8s准入控制（只允许可信镜像）：
  OPA Gatekeeper / Kyverno
  规则：只允许来自内部Registry的镜像
```

## 8. Docker Compose开发环境？

**回答：**

```
  完整开发环境配置：
```

```yaml
# docker-compose.yml
services:
  app:
    build:
      context: .
      dockerfile: Dockerfile
      target: builder  # 用构建阶段做开发
    volumes:
      - .:/app        # 代码热重载
    ports:
      - "8080:8080"
    environment:
      - APP_ENV=local
      - DATABASE_URL=postgres://dev:dev@postgres:5432/myapp?sslmode=disable
      - REDIS_ADDR=redis:6379
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_started
    command: go run ./cmd/server/  # 开发模式

  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: myapp
      POSTGRES_USER: dev
      POSTGRES_PASSWORD: dev
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U dev"]
      interval: 5s
      timeout: 5s
      retries: 5

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"

  # 可选：消息队列
  kafka:
    image: bitnami/kafka:3.7
    ports:
      - "9092:9092"
    environment:
      - KAFKA_CFG_NODE_ID=0
      - KAFKA_CFG_PROCESS_ROLES=controller,broker
      - KAFKA_CFG_LISTENERS=PLAINTEXT://:9092,CONTROLLER://:9093
      - KAFKA_CFG_CONTROLLER_QUORUM_VOTERS=0@kafka:9093
      - KAFKA_CFG_CONTROLLER_LISTENER_NAMES=CONTROLLER

volumes:
  pgdata:
```

## 10. 容器化部署面试速答？

**回答：**

```
Q: Docker镜像怎么优化？
A: 多阶段构建+distroless基础镜像
   禁CGO+去调试符号 镜像<10MB

Q: K8s Deployment关键配置？
A: resources/livenessProbe/readinessProbe
   preStop hook+优雅关闭

Q: liveness和readiness区别？
A: liveness失败重启Pod（只检查进程）
   readiness失败摘流量（检查依赖）

Q: 优雅关闭怎么实现？
A: SIGTERM→preStop sleep→停止接新请求
   →等待进行中请求→关闭资源→退出

Q: HPA怎么配置？
A: 基于CPU/内存利用率自动扩缩
   必须设requests minReplicas≥2

Q: 镜像安全怎么保证？
A: Trivy扫描+Cosign签名+非root
   distroless+固定版本+私有仓库

Q: 部署策略怎么选？
A: 滚动(默认)蓝绿(快速回滚)金丝雀(大流量)
   Argo Rollouts支持自动分析回滚

Q: K8s中数据库密码怎么管？
A: K8s Secret + ExternalSecrets
   从Vault同步 不硬编码在YAML中
```
