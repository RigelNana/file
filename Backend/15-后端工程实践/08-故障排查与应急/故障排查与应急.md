# 故障排查与应急

---

## 1. 故障排查方法论？

**回答：**

```
  排查流程：
  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ 1.发现   │→│ 2.止血   │→│ 3.定位   │
  │ 监控/告警 │  │ 回滚/降级│  │ 日志/追踪│
  └──────────┘  └──────────┘  └──────────┘
       ↓
  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ 4.修复   │→│ 5.验证   │→│ 6.复盘   │
  │ 根因修复  │  │ 功能验证 │  │ 改进项   │
  └──────────┘  └──────────┘  └──────────┘

  先止血 后排查！
  不要在线上问题发生时去分析根因

  止血手段（按优先级）：
  1. 回滚（最快最可靠）
  2. Feature Flag关闭
  3. 降级（关闭非关键功能）
  4. 限流（保护系统不崩溃）
  5. 热修复（最后手段）

  定位方法：
  ┌──────────────────────────────────────┐
  │ 看监控 → 什么时候开始异常？           │
  │ 看日志 → 有什么错误信息？             │
  │ 看变更 → 最近部署/配置改了什么？      │
  │ 看依赖 → 上下游服务是否正常？         │
  │ 看资源 → CPU/内存/磁盘/连接数         │
  └──────────────────────────────────────┘
```

## 2. 常见故障类型与处理？

**回答：**

```
  故障分类速查表：
  ┌──────────────┬──────────────────┬──────────────┐
  │ 故障类型      │ 典型表现          │ 快速处理     │
  ├──────────────┼──────────────────┼──────────────┤
  │ OOM          │ Pod被Kill/重启   │ 增大limits   │
  │ CPU打满      │ 响应变慢          │ 扩容/限流    │
  │ 连接池耗尽   │ 超时错误          │ 调大连接池   │
  │ goroutine泄漏│ 内存持续增长     │ 重启+修代码  │
  │ 死锁         │ 请求卡住          │ 重启+修代码  │
  │ DB慢查询     │ 接口超时          │ kill慢SQL    │
  │ 依赖服务挂   │ 特定接口报错      │ 降级/熔断    │
  │ 磁盘满       │ 日志写入失败      │ 清理日志     │
  │ 证书过期     │ TLS握手失败       │ 更新证书     │
  │ DNS故障      │ 无法解析域名      │ 检查DNS配置  │
  └──────────────┴──────────────────┴──────────────┘

  OOM排查：
  1. kubectl describe pod → 看Reason: OOMKilled
  2. pprof heap → 看内存分配热点
  3. 常见原因：
     - 大查询一次加载太多数据
     - goroutine泄漏
     - 全局缓存无限增长
     - 大文件一次性读入内存

  连接池耗尽排查：
  1. 监控活跃连接数
  2. 查看是否有慢查询占用连接
  3. 检查连接池配置是否合理
  4. 检查是否有连接泄漏（忘记Close）
```

## 4. goroutine泄漏排查？

**回答：**

```
  表现：
  goroutine数量持续增长 内存持续上升
  pprof goroutine → 大量相同的goroutine阻塞在同一行

  常见原因与修复：
```

```go
// 1. channel没有消费者
// 反模式
func process() {
    ch := make(chan int)
    go func() {
        ch <- 1 // 永远阻塞 没人读
    }()
    // 忘了从ch读取
}

// 2. HTTP响应体没有Close
// 反模式
resp, _ := http.Get(url)
// 忘了 defer resp.Body.Close()
// 连接无法复用 goroutine泄漏

// 正确
resp, err := http.Get(url)
if err != nil {
    return err
}
defer resp.Body.Close()
io.Copy(io.Discard, resp.Body) // 读完body才能复用连接

// 3. 没有用context控制超时
// 反模式
go func() {
    result := callExternalService() // 可能永远不返回
    ch <- result
}()

// 正确
go func() {
    ctx, cancel := context.WithTimeout(ctx, 5*time.Second)
    defer cancel()
    result, err := callExternalServiceWithCtx(ctx)
    if err != nil {
        return
    }
    ch <- result
}()

// 4. time.After在循环中
// 反模式（每次循环创建一个timer goroutine）
for {
    select {
    case msg := <-ch:
        process(msg)
    case <-time.After(5 * time.Second):
        // time.After在上一个未触发时不会被GC
    }
}
// 正确
timer := time.NewTimer(5 * time.Second)
defer timer.Stop()
for {
    select {
    case msg := <-ch:
        if !timer.Stop() {
            <-timer.C
        }
        timer.Reset(5 * time.Second)
        process(msg)
    case <-timer.C:
        // timeout
    }
}
```

```
  排查步骤：
  1. pprof goroutine找到泄漏的goroutine
  2. 看它阻塞在哪一行代码
  3. 分析为什么没有退出
  4. 添加context超时/done channel
```

## 5. 应急预案怎么做？

**回答：**

```
  应急响应流程：
  
  1. 发现告警（5分钟内响应）
     On-Call工程师确认告警
     判断影响范围和严重程度
  
  2. 通报拉群
     P0/P1 → 立即拉应急群
     通知相关人员（技术负责人/产品）
     持续更新进展
  
  3. 止血操作
     优先恢复服务 而不是分析根因
     回滚/降级/限流/重启
  
  4. 根因定位
     服务恢复后开始分析
     日志+监控+链路追踪
  
  5. 彻底修复
     短期：临时修复+测试+部署
     长期：根本性解决
  
  6. 故障复盘
     48小时内完成复盘
     Blameless + 改进项跟踪

  Runbook（运维手册）：
  ┌──────────────────────────────────────┐
  │ ## Redis连接超时                      │
  │ ### 现象                              │
  │ 接口超时 日志有redis timeout           │
  │ ### 排查步骤                          │
  │ 1. 检查Redis是否存活                  │
  │ 2. 检查连接池使用率                   │
  │ 3. 检查慢命令（SLOWLOG）             │
  │ ### 处理方案                          │
  │ 1. 连接池满 → 调大PoolSize            │
  │ 2. 慢命令 → 优化命令/加缓存           │
  │ 3. Redis本身问题 → 联系DBA            │
  └──────────────────────────────────────┘
```

## 6. 降级与熔断怎么做？

**回答：**

```
  降级策略：
  ┌──────────────┬──────────────────────────┐
  │ 降级类型      │ 示例                     │
  ├──────────────┼──────────────────────────┤
  │ 服务降级      │ 推荐服务挂→返回热门列表   │
  │ 功能降级      │ 关闭评论/搜索建议        │
  │ 数据降级      │ 缓存挂→读旧数据/默认值   │
  │ 体验降级      │ 关闭动画/降低图片质量     │
  └──────────────┴──────────────────────────┘

  熔断器（Circuit Breaker）：
  ┌──────┐  错误率>阈值  ┌──────┐  超时后探测  ┌──────────┐
  │ 关闭  │────────────→│ 打开  │──────────→│ 半开     │
  │(正常) │←────────────│(拒绝) │←──────────│(部分请求)│
  └──────┘  错误率下降   └──────┘  探测失败   └──────────┘

  Go实现（gobreaker）：
```

```go
import "github.com/sony/gobreaker"

cb := gobreaker.NewCircuitBreaker(gobreaker.Settings{
    Name:        "external-api",
    MaxRequests: 3,              // 半开时允许3个请求
    Interval:    10 * time.Second, // 统计窗口
    Timeout:     30 * time.Second, // 打开后30秒进入半开
    ReadyToTrip: func(counts gobreaker.Counts) bool {
        failureRatio := float64(counts.TotalFailures) / float64(counts.Requests)
        return counts.Requests >= 10 && failureRatio >= 0.5
        // 10个请求中50%失败 → 打开
    },
    OnStateChange: func(name string, from, to gobreaker.State) {
        log.Info("circuit breaker state changed",
            "name", name, "from", from, "to", to)
    },
})

result, err := cb.Execute(func() (interface{}, error) {
    return callExternalAPI(ctx)
})
if err != nil {
    // 熔断打开时 err = gobreaker.ErrOpenState
    return fallbackResult() // 返回降级结果
}
```

## 7. 限流怎么实现？

**回答：**

```
  限流算法对比：
  ┌──────────────┬──────────────┬──────────────────┐
  │ 算法          │ 特点          │ 适用场景          │
  ├──────────────┼──────────────┼──────────────────┤
  │ 固定窗口      │ 简单          │ 粗粒度限流        │
  │ 滑动窗口      │ 更平滑        │ 精确限流          │
  │ 令牌桶        │ 允许突发      │ API限流           │
  │ 漏桶          │ 匀速输出      │ 流量整形          │
  └──────────────┴──────────────┴──────────────────┘

  Go实现（rate包）：
```

```go
import "golang.org/x/time/rate"

// 服务级限流
limiter := rate.NewLimiter(rate.Limit(1000), 100)
// 每秒1000个请求 burst=100

func RateLimitMiddleware(limiter *rate.Limiter) func(http.Handler) http.Handler {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            if !limiter.Allow() {
                http.Error(w, "Too Many Requests", http.StatusTooManyRequests)
                return
            }
            next.ServeHTTP(w, r)
        })
    }
}

// 用户级限流
type UserRateLimiter struct {
    mu       sync.Mutex
    limiters map[string]*rate.Limiter
}

func (l *UserRateLimiter) GetLimiter(userID string) *rate.Limiter {
    l.mu.Lock()
    defer l.mu.Unlock()
    lim, exists := l.limiters[userID]
    if !exists {
        lim = rate.NewLimiter(10, 20) // 每用户10 QPS
        l.limiters[userID] = lim
    }
    return lim
}
```

```
  分布式限流（Redis）：
  使用Redis + Lua脚本实现令牌桶
  保证多实例共享限流计数

  限流返回：
  HTTP 429 Too Many Requests
  Header: Retry-After: 5
  Header: X-RateLimit-Remaining: 0
```

## 8. 故障复盘怎么写？

**回答：**

```
  复盘模板：

  # [P0] 订单服务不可用 2024-01-15

  ## 影响
  - 等级：P0
  - 持续时间：14:30 - 15:15（45分钟）
  - 影响范围：所有用户无法下单
  - 损失评估：约XXXX笔订单受影响

  ## 时间线
  - 14:30 监控告警：订单服务错误率>50%
  - 14:32 On-Call确认问题
  - 14:35 拉应急群 通知相关人员
  - 14:40 尝试回滚到上一版本
  - 14:45 回滚完成 错误率下降到5%
  - 14:50 排查：发现数据库连接池耗尽
  - 15:00 根因：新版本有N+1查询
  - 15:15 确认服务完全恢复

  ## 根因分析（5 Whys）
  1. 为什么服务不可用？→ 数据库连接池耗尽
  2. 为什么连接池耗尽？→ 查询数量暴增
  3. 为什么查询暴增？→ 新版本有N+1查询
  4. 为什么没发现N+1？→ Code Review没注意
  5. 为什么Review没发现？→ 缺少数据库查询Review清单

  ## 改进项
  | 改进项 | 负责人 | 截止日期 | 状态 |
  |--------|--------|----------|------|
  | 添加SQL查询数量监控 | 张三 | 1/22 | 待完成 |
  | Code Review增加DB检查项 | 李四 | 1/20 | 完成 |
  | 增加连接池使用率告警 | 王五 | 1/25 | 待完成 |

  复盘原则：
  ✓ Blameless（追因不追人）
  ✓ 改进项必须有负责人和截止日期
  ✓ 48小时内完成初步复盘
  ✓ 分享给全团队
```

## 9. 混沌工程？

**回答：**

```
  混沌工程理念：
  主动注入故障 验证系统韧性
  在生产中发现问题 而不是等用户发现

  故障注入类型：
  ┌──────────────┬──────────────────────────┐
  │ 类型          │ 模拟场景                  │
  ├──────────────┼──────────────────────────┤
  │ Pod Kill     │ 服务实例崩溃              │
  │ 网络延迟      │ 网络抖动                  │
  │ 网络丢包      │ 网络质量差                │
  │ CPU压力       │ 资源竞争                  │
  │ 内存压力      │ 内存不足                  │
  │ 磁盘IO延迟   │ 磁盘性能下降              │
  │ DNS故障       │ DNS解析失败              │
  └──────────────┴──────────────────────────┘

  工具选择：
  K8s环境：Chaos Mesh / Litmus
  AWS：AWS FIS
  通用：Gremlin（商业）

  Chaos Mesh示例：
  apiVersion: chaos-mesh.org/v1alpha1
  kind: PodChaos
  metadata:
    name: kill-order-service
  spec:
    action: pod-kill
    mode: one
    selector:
      labelSelectors:
        app: order-service
    scheduler:
      cron: '@every 1h'  # 每小时随机杀一个Pod

  实施步骤：
  1. 从非生产环境开始
  2. 定义稳态假设（预期行为）
  3. 注入故障
  4. 观察系统行为
  5. 验证或推翻假设
  6. 修复发现的问题

  问自己：
  杀掉一个Pod 服务还正常吗？
  数据库延迟增加 会级联故障吗？
  缓存全部失效 系统能否存活？
```

## 10. 故障排查面试速答？

**回答：**

```
Q: 线上故障排查流程？
A: 先止血(回滚/降级/限流)再定位
   看监控→看日志→看变更→看依赖→看资源

Q: OOM怎么排查？
A: pprof heap看内存分配热点
   常见原因：大查询/goroutine泄漏/全局缓存

Q: goroutine泄漏怎么查？
A: pprof goroutine看阻塞位置
   原因：channel无消费/缺context取消/Body未Close

Q: 什么是熔断器？
A: 关闭→打开→半开 三态
   错误率达阈值自动断开 防级联故障

Q: 限流算法？
A: 令牌桶(允许突发)漏桶(匀速)
   Go用rate包 分布式用Redis

Q: 降级策略？
A: 服务降级(返回兜底)功能降级(关非核心)
   数据降级(缓存旧数据/默认值)

Q: 故障复盘怎么做？
A: 时间线+影响+5Whys+改进项
   Blameless 改进必须有负责人和deadline

Q: 什么是混沌工程？
A: 主动注入故障验证系统韧性
   K8s用Chaos Mesh 从非生产开始
```
