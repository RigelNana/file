# 数据库迁移与版本

---

## 1. 为什么需要数据库迁移？

**回答：**

```
  问题场景：
  手动改数据库 → 忘了改哪些
  多人开发 → 不知道谁改了什么
  多环境 → dev/staging/prod结构不一致
  回滚 → 不知道怎么回到之前的状态

  数据库迁移解决的问题：
  ┌──────────────────────────────────────┐
  │ 1. Schema变更版本化（可追溯）        │
  │ 2. 自动化执行（不手动SQL）           │
  │ 3. 可回滚（每个变更有up/down）       │
  │ 4. 多环境一致（同一套迁移文件）      │
  │ 5. CI/CD集成（自动迁移）             │
  └──────────────────────────────────────┘

  核心概念：
  每次Schema变更 = 一个迁移文件
  迁移文件有序号/时间戳 按顺序执行
  数据库记录已执行的版本号
  新部署时自动执行未执行的迁移
```

## 2. Go迁移工具选型？

**回答：**

```
  主流工具对比：
  ┌──────────────┬──────────────┬──────────────────┐
  │ 工具          │ 特点          │ 适用场景          │
  ├──────────────┼──────────────┼──────────────────┤
  │ golang-migrate│ 纯SQL迁移    │ Raw SQL偏好      │
  │ goose        │ SQL+Go混合    │ 需要复杂迁移逻辑 │
  │ Atlas        │ 声明式+版本化 │ 现代化方案        │
  │ GORM Migrate │ ORM集成      │ GORM项目         │
  └──────────────┴──────────────┴──────────────────┘

  golang-migrate使用：

  # 安装
  go install -tags 'postgres' github.com/golang-migrate/migrate/v4/cmd/migrate@latest

  # 创建迁移
  migrate create -ext sql -dir migrations -seq create_users

  # 生成文件：
  # migrations/000001_create_users.up.sql
  # migrations/000001_create_users.down.sql

  # 执行迁移
  migrate -source file://migrations -database "postgres://..." up

  # 回滚一步
  migrate -source file://migrations -database "postgres://..." down 1

  goose使用：
  goose -dir migrations postgres "postgres://..." up
  goose -dir migrations postgres "postgres://..." down
  goose create add_email_column sql
```

## 3. 迁移文件怎么写？

**回答：**

```
  迁移文件示例：

  -- 000001_create_users.up.sql
  CREATE TABLE users (
      id         BIGSERIAL PRIMARY KEY,
      username   VARCHAR(50)  NOT NULL UNIQUE,
      email      VARCHAR(255) NOT NULL UNIQUE,
      password   VARCHAR(255) NOT NULL,
      created_at TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
      updated_at TIMESTAMPTZ  NOT NULL DEFAULT NOW()
  );
  CREATE INDEX idx_users_email ON users(email);

  -- 000001_create_users.down.sql
  DROP TABLE IF EXISTS users;

  -- 000002_add_avatar_to_users.up.sql
  ALTER TABLE users ADD COLUMN avatar_url VARCHAR(500);

  -- 000002_add_avatar_to_users.down.sql
  ALTER TABLE users DROP COLUMN avatar_url;

  命名规范：
  {序号}_{描述}.up.sql
  {序号}_{描述}.down.sql
  
  描述用动词+名词：
  create_users
  add_email_to_orders
  drop_legacy_table
  create_index_on_orders_status

  编写原则：
  ┌──────────────────────────────────────┐
  │ 1. up和down必须成对 且互为逆操作    │
  │ 2. 每个迁移只做一件事                │
  │ 3. 迁移一旦执行 永远不要修改          │
  │ 4. 先测试再上线（staging先跑一遍）   │
  │ 5. 大表操作评估影响（锁/时间）       │
  └──────────────────────────────────────┘
```

## 4. 大表变更怎么做？

**回答：**

```
  大表变更的风险：
  ALTER TABLE大表 → 可能锁表几分钟到几小时
  影响线上业务（无法读写）

  安全变更策略：

  1. 加列（PostgreSQL）
     ALTER TABLE ADD COLUMN — 不锁表（无默认值时）
     但 ADD COLUMN ... DEFAULT xxx — PG 11+不锁表

  2. 删列（三步法）
     Step 1: 代码停止读写该列（部署）
     Step 2: ALTER TABLE DROP COLUMN（迁移）
     Step 3: 清理相关代码
  
  3. 改列类型
     不要直接ALTER TABLE ALTER COLUMN TYPE
     ┌─────────────────────────────────────┐
     │ Step 1: 加新列                       │
     │ Step 2: 双写（新旧列同时写）         │
     │ Step 3: 数据回填（老数据写到新列）   │
     │ Step 4: 代码切换到读新列             │
     │ Step 5: 删旧列                       │
     └─────────────────────────────────────┘

  4. 加索引
     CREATE INDEX CONCURRENTLY idx_xxx ON table(col);
     -- CONCURRENTLY不锁表 但更慢

  在线DDL工具：
  MySQL → gh-ost / pt-online-schema-change
  原理：创建影子表 + 触发器同步 + 最终切换

  大表变更检查清单：
  □ 表有多大（行数/大小）
  □ 操作会不会锁表
  □ 预计执行时间
  □ 回滚方案是什么
  □ 在staging测过了吗
```

## 5. 迁移与CI/CD集成？

**回答：**

```
  自动化迁移流程：

  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ PR合并   │→│ 执行迁移  │→│ 部署代码  │→│ 健康检查 │
  └──────────┘  └──────────┘  └──────────┘  └──────────┘
                   ↓失败           不执行
                 中止部署

  CI中执行迁移：
```

```go
// cmd/migrate/main.go
func main() {
    cfg, _ := config.Load()
    
    m, err := migrate.New(
        "file://migrations",
        cfg.Database.DSN(),
    )
    if err != nil {
        log.Fatal(err)
    }
    
    if err := m.Up(); err != nil && err != migrate.ErrNoChange {
        log.Fatal("migration failed:", err)
    }
    log.Println("migration completed")
}
```

```
  Makefile集成：
  migrate-up:
      migrate -source file://migrations \
              -database "$(DATABASE_URL)" up

  migrate-down:
      migrate -source file://migrations \
              -database "$(DATABASE_URL)" down 1

  migrate-create:
      migrate create -ext sql -dir migrations -seq $(name)

  K8s中执行迁移：
  用Init Container或Job在Pod启动前执行
  
  apiVersion: batch/v1
  kind: Job
  metadata:
    name: db-migration
  spec:
    template:
      spec:
        containers:
          - name: migrate
            image: myapp:latest
            command: ["./migrate", "up"]
        restartPolicy: Never

  原则：
  迁移和代码同一个PR
  先迁移 后部署代码
  迁移失败 整个部署中止
```

## 6. 数据迁移（Data Migration）？

**回答：**

```
  Schema迁移 vs 数据迁移：
  Schema：表结构变更（DDL）
  Data：数据变换/回填/清洗（DML）

  数据迁移场景：
  1. 新列需要根据旧数据回填
  2. 数据格式变更（JSON→结构化列）
  3. 数据拆分（一张表拆多张）
  4. 数据清洗（修正脏数据）

  数据迁移注意事项：
  ┌──────────────────────────────────────┐
  │ 1. 分批执行 不要一个大事务           │
  │ 2. 可中断可恢复（记录进度）          │
  │ 3. 有进度日志                         │
  │ 4. 先在staging验证                   │
  │ 5. 执行前备份                         │
  └──────────────────────────────────────┘

  分批回填示例：
```

```go
func BackfillUserFullName(db *sql.DB) error {
    batchSize := 1000
    lastID := int64(0)
    
    for {
        rows, err := db.Query(`
            SELECT id, first_name, last_name FROM users
            WHERE id > $1 AND full_name IS NULL
            ORDER BY id LIMIT $2
        `, lastID, batchSize)
        if err != nil {
            return err
        }
        
        count := 0
        for rows.Next() {
            var id int64
            var first, last string
            rows.Scan(&id, &first, &last)
            
            _, err := db.Exec(`
                UPDATE users SET full_name = $1 WHERE id = $2
            `, first+" "+last, id)
            if err != nil {
                log.Printf("failed to update user %d: %v", id, err)
                continue // 容错继续
            }
            lastID = id
            count++
        }
        rows.Close()
        
        log.Printf("backfilled %d users, last_id=%d", count, lastID)
        if count < batchSize {
            break // 没有更多数据
        }
    }
    return nil
}
```

## 7. 迁移版本冲突怎么处理？

**回答：**

```
  冲突场景：
  两个开发者同时创建了 000005 号迁移
  合并时产生版本冲突

  解决方案：

  1. 用时间戳代替序号（推荐）
     20240115103000_add_column.up.sql
     20240116090000_add_index.up.sql
     时间戳几乎不会冲突

  2. 合并时重命名
     发现冲突 → 调整序号 → 重新测试

  3. 团队流程预防
     迁移文件通过PR管理
     Review时检查序号是否冲突
     约定：先合并先得

  Dirty数据库修复：
  迁移执行到一半失败 → 数据库标记为dirty
  
  # 查看状态
  migrate -source file://migrations \
          -database "..." version
  
  # 强制修复（确认当前版本正确后）
  migrate -source file://migrations \
          -database "..." force 5

  预防措施：
  1. 迁移文件中每条SQL一个事务
  2. 复杂迁移分多个小迁移
  3. staging先测完整流程
```

## 8. 数据库版本回滚？

**回答：**

```
  回滚策略：

  1. 向后兼容迁移（推荐）
     新代码兼容旧Schema
     旧代码兼容新Schema
     → 出问题只需回滚代码 不需回滚Schema

  2. Down迁移回滚
     执行down.sql恢复旧结构
     适用于简单变更（加列/删列）
     复杂变更可能无法完美回滚

  向后兼容示例：
  需要重命名列 email → contact_email

  Step 1（迁移1）：
  ALTER TABLE users ADD COLUMN contact_email VARCHAR(255);
  UPDATE users SET contact_email = email;

  Step 2（代码部署）：
  代码同时读写email和contact_email（双写）

  Step 3（确认无问题后 迁移2）：
  ALTER TABLE users DROP COLUMN email;

  → 任何一步出问题 只回滚当步即可

  回滚检查清单：
  ┌──────────────────────────────────────┐
  │ □ down.sql在staging测试通过          │
  │ □ 评估数据损失风险                    │
  │ □ 回滚后应用代码是否兼容              │
  │ □ 是否需要先回滚代码再回滚Schema      │
  │ □ 有备份可恢复（最坏情况）            │
  └──────────────────────────────────────┘

  黄金法则：
  向前修复优先于回滚
  Schema变更尽量做到向后兼容
  迁移和代码解耦 可以独立回滚
```

## 9. 数据库分支管理？

**回答：**

```
  分支数据库问题：
  feature分支创建了迁移文件
  另一个feature也创建了迁移
  合并到main时可能冲突

  解决方案：

  1. 短生命周期分支
     分支生命周期短 冲突概率低
     频繁rebase main 保持同步

  2. 每个开发者独立数据库
     docker-compose快速创建本地DB
     互不干扰

  3. 迁移审核
     PR中重点Review迁移文件
     合并前确认序号和依赖

  docker-compose开发数据库：
  services:
    db:
      image: postgres:16-alpine
      volumes:
        - pgdata:/var/lib/postgresql/data
      environment:
        POSTGRES_DB: myapp_dev
        POSTGRES_USER: dev
        POSTGRES_PASSWORD: dev
      ports:
        - "5432:5432"

  测试数据管理：
  种子数据（Seed）：
  每个环境需要的基础数据
  不放在迁移中 单独管理

  fixtures：
  测试用数据
  每次测试前重建
```

```go
// testutil/database.go
func SetupTestDB(t *testing.T) *sql.DB {
    t.Helper()
    db, _ := sql.Open("postgres", testDSN)
    
    // 应用所有迁移
    m, _ := migrate.New("file://../../migrations", testDSN)
    m.Up()
    
    // 清理数据（每次测试干净）
    t.Cleanup(func() {
        db.Exec("TRUNCATE users, orders CASCADE")
        db.Close()
    })
    return db
}
```

## 10. 数据库迁移面试速答？

**回答：**

```
Q: 为什么需要数据库迁移工具？
A: Schema变更版本化+自动执行+可回滚
   多环境一致+CI/CD集成

Q: Go用什么迁移工具？
A: golang-migrate(纯SQL) goose(SQL+Go)
   Atlas(声明式) 按项目需求选

Q: 大表怎么改？
A: 加列不锁表 删列三步法 改类型加新列双写
   加索引CONCURRENTLY MySQL用gh-ost

Q: 迁移怎么和CI/CD集成？
A: 先迁移后部署 迁移失败中止部署
   K8s用Job/InitContainer

Q: 版本冲突怎么解？
A: 用时间戳命名避免冲突
   短分支+频繁rebase+PR审核

Q: 怎么安全回滚Schema？
A: 优先向后兼容（不需要回滚Schema）
   简单变更用down.sql 复杂变更向前修复

Q: 数据迁移注意什么？
A: 分批执行 可中断可恢复
   有进度日志 先备份再执行

Q: Schema变更黄金法则？
A: 迟量向后兼容 迁移和代码解耦
   可独立回滚 向前修复优于回滚
```
