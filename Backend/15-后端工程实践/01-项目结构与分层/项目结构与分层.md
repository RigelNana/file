# 项目结构与分层

---

## 1. Go项目标准目录结构？

**回答：**

```
  推荐目录结构（golang-standards/project-layout）：

  myapp/
  ├── cmd/                    # 主入口
  │   ├── server/             # HTTP/gRPC服务
  │   │   └── main.go
  │   └── worker/             # 异步Worker
  │       └── main.go
  ├── internal/               # 私有代码（不可外部导入）
  │   ├── handler/            # HTTP/gRPC处理层
  │   ├── service/            # 业务逻辑层
  │   ├── repository/         # 数据访问层
  │   ├── model/              # 数据模型
  │   └── middleware/         # 中间件
  ├── pkg/                    # 可复用公共库
  │   ├── logger/
  │   └── httpclient/
  ├── api/                    # 接口定义
  │   ├── proto/              # Protobuf
  │   └── openapi/            # OpenAPI/Swagger
  ├── configs/                # 配置文件
  ├── deployments/            # 部署清单
  │   ├── docker/
  │   └── k8s/
  ├── migrations/             # 数据库迁移
  ├── scripts/                # 脚本
  ├── docs/                   # 文档
  ├── Makefile
  ├── Dockerfile
  ├── go.mod
  └── go.sum

  关键原则：
  1. cmd/ 只放main入口 不放业务逻辑
  2. internal/ 保证封装 外部包无法import
  3. pkg/ 放通用工具 可以被其他项目导入
  4. 按领域分包 不按技术分包

  反模式：
  ✗ 所有代码放根目录
  ✗ internal里按controller/service/dao分
     而不按业务领域分
  ✗ 循环依赖（handler导入service反过来也导入）
```

## 2. 三层架构怎么分？

**回答：**

```
  经典三层架构：

  ┌──────────────────────┐
  │     Handler层         │ 请求解析、参数校验、响应封装
  │  (Transport/API)      │ 不含业务逻辑
  ├──────────────────────┤
  │     Service层         │ 核心业务逻辑
  │  (Business/UseCase)   │ 编排Repository调用
  ├──────────────────────┤
  │     Repository层      │ 数据库/缓存/外部API
  │  (Data Access)        │ 封装存储细节
  └──────────────────────┘

  依赖方向：Handler → Service → Repository
  接口定义在消费者侧（Go惯例）

  代码示例：
```

```go
// internal/repository/user.go
type UserRepository interface {
    GetByID(ctx context.Context, id int64) (*model.User, error)
    Create(ctx context.Context, user *model.User) error
}

type userRepo struct {
    db *sql.DB
}

func (r *userRepo) GetByID(ctx context.Context, id int64) (*model.User, error) {
    var u model.User
    err := r.db.QueryRowContext(ctx,
        "SELECT id, name, email FROM users WHERE id = $1", id,
    ).Scan(&u.ID, &u.Name, &u.Email)
    return &u, err
}

// internal/service/user.go
type UserService struct {
    repo   repository.UserRepository
    cache  cache.Cache
}

func NewUserService(repo repository.UserRepository, cache cache.Cache) *UserService {
    return &UserService{repo: repo, cache: cache}
}

func (s *UserService) GetUser(ctx context.Context, id int64) (*model.User, error) {
    // 先查缓存
    if user, err := s.cache.GetUser(ctx, id); err == nil {
        return user, nil
    }
    // 查数据库
    user, err := s.repo.GetByID(ctx, id)
    if err != nil {
        return nil, fmt.Errorf("get user %d: %w", id, err)
    }
    // 写缓存（异步）
    go s.cache.SetUser(context.Background(), user)
    return user, nil
}

// internal/handler/user.go
type UserHandler struct {
    svc *service.UserService
}

func (h *UserHandler) GetUser(w http.ResponseWriter, r *http.Request) {
    id, err := strconv.ParseInt(chi.URLParam(r, "id"), 10, 64)
    if err != nil {
        respondError(w, http.StatusBadRequest, "invalid id")
        return
    }
    user, err := h.svc.GetUser(r.Context(), id)
    if err != nil {
        respondError(w, http.StatusInternalServerError, "internal error")
        return
    }
    respondJSON(w, http.StatusOK, user)
}
```

```
  层间职责边界：
  ┌──────────────┬────────────────────────────┐
  │ 层            │ 不应该做的                  │
  ├──────────────┼────────────────────────────┤
  │ Handler      │ 不直接操作DB               │
  │ Service      │ 不知道HTTP状态码            │
  │ Repository   │ 不包含业务判断逻辑          │
  └──────────────┴────────────────────────────┘
```

## 3. DDD分层怎么落地？

**回答：**

```
  DDD四层架构：

  ┌──────────────────────┐
  │    Interface层        │ HTTP/gRPC入口、DTO转换
  ├──────────────────────┤
  │   Application层       │ 用例编排、事务管理
  ├──────────────────────┤
  │     Domain层          │ 领域模型、领域服务、值对象
  │                       │ 不依赖任何外部（纯逻辑）
  ├──────────────────────┤
  │  Infrastructure层     │ 数据库、消息队列、外部API
  └──────────────────────┘

  依赖方向：Interface → Application → Domain ← Infrastructure
  核心：Domain层不依赖Infrastructure
  Infrastructure实现Domain定义的接口（依赖反转）

  目录结构：
  internal/
  ├── domain/
  │   └── order/
  │       ├── entity.go       # Order实体
  │       ├── valueobject.go  # Money、Address值对象
  │       ├── repository.go   # 接口定义
  │       └── service.go      # 领域服务
  ├── application/
  │   └── order/
  │       ├── command.go      # CreateOrderCmd
  │       ├── query.go        # GetOrderQuery
  │       └── service.go      # 用例编排
  ├── infrastructure/
  │   └── persistence/
  │       └── order_repo.go   # 接口实现
  └── interfaces/
      └── http/
          └── order_handler.go

  何时用DDD：
  ✓ 业务复杂、领域概念多
  ✓ 需要长期演进的核心系统
  ✗ CRUD为主的简单服务
  ✗ 追求快速交付的原型
```

## 4. 依赖注入怎么做？

**回答：**

```
  三种方式：
  ┌──────────────┬──────────┬────────────┬──────────┐
  │ 方式          │ 适用     │ 优点        │ 缺点     │
  ├──────────────┼──────────┼────────────┼──────────┤
  │ 手动注入      │ 小项目   │ 简单直观    │ 样板多   │
  │ Wire         │ 中大项目  │ 编译时检查  │ 学习成本 │
  │ fx           │ 大项目   │ 运行时灵活  │ 调试难   │
  └──────────────┴──────────┴────────────┴──────────┘

  1. 手动注入（推荐入门）：
```

```go
func main() {
    // 基础设施
    db := database.NewDB(cfg.DB)
    rdb := redis.NewClient(cfg.Redis)
    
    // Repository
    userRepo := repository.NewUserRepo(db)
    orderRepo := repository.NewOrderRepo(db)
    
    // Service
    userSvc := service.NewUserService(userRepo, rdb)
    orderSvc := service.NewOrderService(orderRepo, userSvc)
    
    // Handler
    userHandler := handler.NewUserHandler(userSvc)
    orderHandler := handler.NewOrderHandler(orderSvc)
    
    // Router
    r := chi.NewRouter()
    r.Get("/users/{id}", userHandler.GetUser)
    r.Post("/orders", orderHandler.CreateOrder)
    
    http.ListenAndServe(":8080", r)
}
```

```
  2. Google Wire（编译时DI）：
```

```go
// wire.go
//go:build wireinject

func InitializeApp(cfg *config.Config) (*App, error) {
    wire.Build(
        database.NewDB,
        redis.NewClient,
        repository.NewUserRepo,
        service.NewUserService,
        handler.NewUserHandler,
        NewApp,
    )
    return nil, nil
}
// 运行 wire ./cmd/server/ 生成 wire_gen.go
```

```
  依赖注入核心原则：
  1. 依赖接口 不依赖实现
  2. 在最外层（main）组装依赖
  3. 不在内部创建依赖 而是从外部传入
  4. 方便单元测试时替换Mock
```

## 5. 单体vs微服务怎么选？

**回答：**

```
  对比：
  ┌──────────────┬──────────────┬──────────────────┐
  │ 维度          │ 单体          │ 微服务            │
  ├──────────────┼──────────────┼──────────────────┤
  │ 开发速度      │ 初期更快      │ 初期慢           │
  │ 部署          │ 整体部署      │ 独立部署          │
  │ 团队          │ 小团队适合    │ 多团队适合        │
  │ 一致性        │ 强一致(事务)  │ 最终一致          │
  │ 运维复杂度    │ 低           │ 高(网络/监控/调试)│
  │ 技术栈        │ 统一         │ 多语言可选        │
  │ 扩展          │ 整体扩展     │ 独立扩展          │
  └──────────────┴──────────────┴──────────────────┘

  选择建议：
  团队<10人 → 单体优先
  业务初期不确定 → 单体 先跑通再拆
  已有明确领域边界 → 微服务
  流量各模块差异大 → 微服务（独立扩缩）

  演进路径：
  单体 → 模块化单体 → 微服务

  模块化单体（推荐过渡方案）：
  1. internal/ 按领域分模块
  2. 模块间通过接口通信 不直接访问另一模块数据库
  3. 需要时沿模块边界拆服务

  拆分信号：
  ✓ 某模块需要独立扩展
  ✓ 不同模块发布周期差异大
  ✓ 团队已经按模块划分
  ✗ 不要为了微服务而微服务
```

## 6. 模块间通信怎么设计？

**回答：**

```
  通信方式选择：
  ┌──────────────┬──────────────┬──────────────────┐
  │ 方式          │ 适用场景      │ 特点             │
  ├──────────────┼──────────────┼──────────────────┤
  │ HTTP/REST    │ 对外API      │ 通用/易调试       │
  │ gRPC         │ 内部服务间    │ 高性能/强类型     │
  │ 消息队列      │ 异步解耦     │ 削峰/最终一致     │
  │ 事件驱动      │ 领域事件     │ 松耦合/可追溯     │
  └──────────────┴──────────────┴──────────────────┘

  单体内模块通信：
```

```go
// 模块间通过接口通信 不直接导入另一模块
// internal/order/service.go
type OrderService struct {
    userQuerier UserQuerier  // 由user模块实现
}

type UserQuerier interface {
    GetUser(ctx context.Context, id int64) (*User, error)
}

// 在main中组装
userSvc := user.NewService(userRepo)
orderSvc := order.NewService(orderRepo, userSvc) // userSvc实现UserQuerier
```

```
  微服务间通信：
  同步调用（gRPC）：
  实时查询/需要立即返回结果
  加超时+重试+熔断
  
  异步消息（Kafka/RabbitMQ）：
  不需要立即返回
  允许最终一致
  削峰填谷

  通信原则：
  1. 尽量减少服务间调用
  2. 同步调用必须有超时
  3. 重要操作要幂等
  4. 可以用事件代替查询（本地缓存事件数据）
```

## 7. 错误码与响应规范？

**回答：**

```
  统一响应格式：
```

```go
// pkg/response/response.go
type Response struct {
    Code    int         `json:"code"`    // 业务错误码
    Message string      `json:"message"` // 错误信息
    Data    interface{} `json:"data"`    // 响应数据
}

// 成功：{"code": 0, "message": "success", "data": {...}}
// 失败：{"code": 10001, "message": "用户不存在", "data": null}

func Success(w http.ResponseWriter, data interface{}) {
    respondJSON(w, http.StatusOK, Response{
        Code:    0,
        Message: "success",
        Data:    data,
    })
}

func Error(w http.ResponseWriter, httpCode int, bizCode int, msg string) {
    respondJSON(w, httpCode, Response{
        Code:    bizCode,
        Message: msg,
    })
}
```

```
  错误码设计：
  ┌──────────────┬──────────────────────────┐
  │ 范围          │ 含义                     │
  ├──────────────┼──────────────────────────┤
  │ 0            │ 成功                     │
  │ 10000-19999  │ 用户模块错误              │
  │ 20000-29999  │ 订单模块错误              │
  │ 30000-39999  │ 支付模块错误              │
  │ 40000-49999  │ 通用参数错误              │
  │ 50000-59999  │ 系统内部错误              │
  └──────────────┴──────────────────────────┘

  HTTP状态码映射：
  200 → 成功
  400 → 参数错误
  401 → 未认证
  403 → 无权限
  404 → 资源不存在
  429 → 限流
  500 → 服务内部错误

  错误信息原则：
  给用户看的：友好可理解
  给开发看的：日志中记录详细错误栈
  永远不要把堆栈信息返回给前端
```

## 8. 中间件设计？

**回答：**

```
  中间件洋葱模型：

  请求 → [Recovery → Logging → Auth → RateLimit → Handler]
                                                     ↓
  响应 ← [Recovery ← Logging ← Auth ← RateLimit ← Handler]

  Go中间件模式：
```

```go
type Middleware func(http.Handler) http.Handler

// Recovery中间件
func Recovery(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        defer func() {
            if err := recover(); err != nil {
                log.Error("panic recovered", "error", err,
                    "stack", string(debug.Stack()))
                http.Error(w, "Internal Server Error", 500)
            }
        }()
        next.ServeHTTP(w, r)
    })
}

// RequestID中间件
func RequestID(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        id := r.Header.Get("X-Request-ID")
        if id == "" {
            id = uuid.New().String()
        }
        ctx := context.WithValue(r.Context(), requestIDKey, id)
        w.Header().Set("X-Request-ID", id)
        next.ServeHTTP(w, r.WithContext(ctx))
    })
}

// 中间件注册（chi示例）
r := chi.NewRouter()
r.Use(Recovery)
r.Use(RequestID)
r.Use(Logger)
r.Use(Auth)
```

```
  常用中间件清单：
  ┌──────────────┬──────────────────────────┐
  │ 中间件        │ 职责                     │
  ├──────────────┼──────────────────────────┤
  │ Recovery     │ panic恢复 防止进程崩溃    │
  │ RequestID    │ 请求追踪标识              │
  │ Logger       │ 请求日志（方法/路径/耗时）│
  │ Auth         │ 认证（JWT/Session）       │
  │ CORS         │ 跨域配置                  │
  │ RateLimit    │ 限流                      │
  │ Timeout      │ 请求超时控制              │
  │ Compress     │ 响应压缩                  │
  └──────────────┴──────────────────────────┘

  注册顺序很重要：
  Recovery要在最外层（兜底所有panic）
  Logger在Auth外面（未认证的也要记日志）
  Auth在业务Handler前面
```

## 9. main函数怎么组织？

**回答：**

```
  推荐结构 — 优雅启动与关闭：
```

```go
func main() {
    // 1. 加载配置
    cfg, err := config.Load()
    if err != nil {
        log.Fatal("load config", "error", err)
    }

    // 2. 初始化基础设施
    logger := logger.New(cfg.Log)
    db, err := database.New(cfg.DB)
    if err != nil {
        log.Fatal("init db", "error", err)
    }
    defer db.Close()

    rdb := redis.NewClient(&redis.Options{Addr: cfg.Redis.Addr})
    defer rdb.Close()

    // 3. 依赖注入
    userRepo := repository.NewUserRepo(db)
    userSvc := service.NewUserService(userRepo, rdb)
    userHandler := handler.NewUserHandler(userSvc)

    // 4. 路由注册
    r := chi.NewRouter()
    r.Use(middleware.Recovery, middleware.RequestID, middleware.Logger)
    r.Route("/api/v1", func(r chi.Router) {
        r.Get("/users/{id}", userHandler.GetUser)
    })

    // 5. 启动HTTP服务
    srv := &http.Server{
        Addr:         cfg.Server.Addr,
        Handler:      r,
        ReadTimeout:  10 * time.Second,
        WriteTimeout: 10 * time.Second,
    }

    go func() {
        logger.Info("server started", "addr", cfg.Server.Addr)
        if err := srv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
            log.Fatal("listen", "error", err)
        }
    }()

    // 6. 优雅关闭
    quit := make(chan os.Signal, 1)
    signal.Notify(quit, syscall.SIGINT, syscall.SIGTERM)
    <-quit

    logger.Info("shutting down...")
    ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
    defer cancel()
    if err := srv.Shutdown(ctx); err != nil {
        log.Fatal("shutdown", "error", err)
    }
    logger.Info("server exited")
}
```

```
  要点：
  1. 配置加载在最前面
  2. 所有资源都有defer Close
  3. 服务在goroutine中启动
  4. 主goroutine监听信号优雅退出
  5. Shutdown给足超时（处理完进行中请求）

  K8s场景注意：
  增加健康检查端点 /healthz /readyz
  preStop hook等待几秒再shutdown
  让负载均衡器先摘除流量
```

## 10. 项目结构面试速答？

**回答：**

```
Q: Go项目标准目录？
A: cmd(入口)+internal(私有)+pkg(公共)
   configs+api+deployments+migrations

Q: 三层架构怎么分？
A: Handler(请求响应)+Service(业务逻辑)
   +Repository(数据访问) 依赖向下

Q: DDD什么时候用？
A: 业务复杂且需要长期演进时
   Interface→App→Domain←Infrastructure

Q: 依赖注入怎么做？
A: 小项目手动注入 中大项目用Wire
   依赖接口不依赖实现 在main组装

Q: 单体还是微服务？
A: 小团队/<10人用单体
   先模块化单体 需要时按边界拆

Q: 模块间怎么通信？
A: 单体：接口隔离 不直接导入
   微服务：gRPC同步 + MQ异步

Q: 中间件执行顺序？
A: Recovery(最外)→Logger→Auth→业务
   洋葱模型 请求和响应对称执行

Q: main函数怎么组织？
A: 配置→基础设施→DI→路由→启动→优雅关闭
   signal.Notify+Shutdown
```
