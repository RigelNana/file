# 安全架构实践

---

## 1. 安全架构设计原则？

**回答：**

```
  核心原则：
  ┌──────────────────────┬──────────────────────────────┐
  │ 原则                  │ 说明                         │
  ├──────────────────────┼──────────────────────────────┤
  │ 纵深防御              │ 多层安全 不依赖单一措施       │
  │ 最小权限              │ 只给必要权限                  │
  │ 安全默认              │ 默认配置是安全的              │
  │ 零信任                │ 永不信任 始终验证             │
  │ 分离职责              │ 关键操作需多人                │
  │ 安全左移              │ 开发阶段就考虑安全            │
  │ 失败安全              │ 出错时倾向拒绝而非放行        │
  └──────────────────────┴──────────────────────────────┘

  纵深防御示例：
  用户请求 → CDN/WAF → 网关鉴权 → 限流
           → 服务间mTLS → 业务鉴权 → 数据权限
           → 加密存储 → 审计日志

  安全默认示例：
  Cookie默认HttpOnly+Secure+SameSite
  CORS默认不开
  接口默认需要认证（白名单放行少数公开接口）
  密码策略默认强制复杂度
```

## 2. 威胁建模？

**回答：**

```
  STRIDE模型：
  ┌──────────────┬──────────────┬──────────────────┐
  │ 威胁          │ 全称          │ 对应安全属性     │
  ├──────────────┼──────────────┼──────────────────┤
  │ S-Spoofing   │ 假冒          │ 认证             │
  │ T-Tampering  │ 篡改          │ 完整性           │
  │ R-Repudiation│ 抵赖          │ 不可否认         │
  │ I-Info Disc. │ 信息泄露      │ 保密性           │
  │ D-DoS        │ 拒绝服务      │ 可用性           │
  │ E-Elevation  │ 权限提升      │ 授权             │
  └──────────────┴──────────────┴──────────────────┘

  威胁建模流程：
  1. 画数据流图（DFD）
     标识信任边界/数据流/存储
  
  2. 识别威胁（STRIDE）
     每个数据流/存储逐一分析
  
  3. 评估风险（DREAD评分）
     Damage/Reproducibility/Exploitability
     /Affected Users/Discoverability
  
  4. 设计缓解措施
     每个威胁对应一个或多个防御

  示例 - 用户登录接口：
  S: 暴力破解/凭证填充 → 限流+账号锁定+MFA
  T: 中间人篡改 → HTTPS
  R: 否认登录行为 → 审计日志
  I: 密码泄露 → bcrypt+TLS
  D: 频繁请求 → 限流+验证码
  E: 越权 → RBAC+最小权限
```

## 3. 零信任架构？

**回答：**

```
  零信任 = Never Trust, Always Verify

  传统模型 vs 零信任：
  传统：内网可信 → VPN进入后自由访问
  零信任：不区分内外网 → 每次请求都验证

  零信任核心组件：
  ┌──────────────────────────────────────┐
  │         策略决策点（PDP）              │
  │   身份验证 + 设备信任 + 风险评估      │
  └──────────┬─────────────┬─────────────┘
             │             │
  ┌──────────▼──────┐ ┌───▼──────────────┐
  │ 策略执行点（PEP）│ │  持续监控/分析     │
  │ 网关/Sidecar    │ │  行为/异常检测     │
  └─────────────────┘ └──────────────────┘

  实施要点：
  1. 身份为中心
     每个请求携带身份凭证
     服务间也要mTLS/JWT
  
  2. 设备信任
     设备健康状态检查
     MDM/证书/合规检查
  
  3. 微分段
     网络细粒度隔离
     K8s NetworkPolicy
  
  4. 持续验证
     不是一次认证永久信任
     定期重新验证+风险评估
  
  5. 最小权限
     按需授权 细粒度控制

  Service Mesh实现零信任：
  Istio mTLS → 服务间加密认证
  AuthorizationPolicy → 细粒度访问控制
  PeerAuthentication → 强制mTLS
```

## 4. WAF与API防护？

**回答：**

```
  WAF = Web Application Firewall

  部署位置：
  客户端 → CDN → WAF → 负载均衡 → 应用

  WAF防护能力：
  ┌──────────────┬──────────────────────────────┐
  │ 防护类型      │ 说明                         │
  ├──────────────┼──────────────────────────────┤
  │ SQL注入       │ 检测SQL关键词/特殊字符       │
  │ XSS           │ 检测脚本标签                 │
  │ 文件包含      │ 路径穿越检测                 │
  │ CC攻击        │ 频率限制                     │
  │ Bot检测       │ 区分人机                     │
  │ 自定义规则    │ 业务特定防护                 │
  └──────────────┴──────────────────────────────┘

  WAF选型：
  云WAF: Cloudflare/阿里云WAF/AWS WAF
  开源: ModSecurity/Coraza(Go)
  K8s: Ingress Nginx + ModSecurity

  API网关安全：
  统一认证 → JWT验证/OAuth2
  限流 → 多维度(IP/用户/API)
  请求校验 → JSON Schema验证
  日志审计 → 所有API调用记录

  Bot防护：
  1. 速率限制 → 超频触发
  2. 验证码 → 人机验证
  3. 指纹识别 → 设备/浏览器指纹
  4. 行为分析 → 异常行为检测
```

## 5. 安全测试？

**回答：**

```
  安全测试类型：
  ┌──────────────┬──────────────────────────────┐
  │ 类型          │ 说明                         │
  ├──────────────┼──────────────────────────────┤
  │ SAST          │ 静态分析 扫描源码            │
  │ DAST          │ 动态测试 运行时扫描          │
  │ SCA           │ 依赖成分分析 检查CVE         │
  │ 渗透测试      │ 人工模拟攻击                 │
  │ 模糊测试      │ 随机输入发现崩溃             │
  └──────────────┴──────────────────────────────┘

  Go安全工具链：
  # SAST - 静态安全分析
  gosec ./...
  
  # 漏洞检查
  govulncheck ./...
  
  # 依赖审计
  go list -m -json all | nancy sleuth
  
  # 模糊测试（Go内置）
  func FuzzParseInput(f *testing.F) {
      f.Add("normal input")
      f.Fuzz(func(t *testing.T, input string) {
          result, err := ParseInput(input)
          if err != nil { return }
          // 验证结果不会panic
          _ = result
      })
  }

  CI/CD集成：
  PR → SAST(gosec) → SCA(govulncheck)
     → 构建 → DAST(OWASP ZAP)
     → 镜像扫描(Trivy) → 部署

  镜像扫描：
  trivy image myapp:latest
  trivy fs --security-checks vuln,config .
```

## 6. K8s安全加固？

**回答：**

```
  K8s安全层面：
  1. Pod安全
  apiVersion: v1
  kind: Pod
  spec:
    securityContext:
      runAsNonRoot: true
      runAsUser: 1000
      fsGroup: 2000
    containers:
    - name: app
      securityContext:
        allowPrivilegeEscalation: false
        readOnlyRootFilesystem: true
        capabilities:
          drop: ["ALL"]
      resources:
        limits:
          cpu: 500m
          memory: 256Mi

  2. 网络策略
  apiVersion: networking.k8s.io/v1
  kind: NetworkPolicy
  metadata:
    name: deny-all
  spec:
    podSelector: {}
    policyTypes: ["Ingress", "Egress"]
    # 默认拒绝所有 再逐条放行

  3. RBAC最小权限
  不使用cluster-admin
  ServiceAccount按需创建
  Role限制到namespace

  4. Secret管理
  etcd加密
  External Secrets Operator
  不用环境变量传密钥（ps可看到）

  5. 镜像安全
  只拉取可信Registry
  镜像签名(Cosign)
  准入控制(Kyverno/OPA)

  6. 审计
  启用API Server审计日志
  监控异常API调用
```

## 7. 安全事件响应？

**回答：**

```
  事件响应流程（NIST框架）：
  1. 准备 → 2. 检测 → 3. 遏制 → 4. 根除 → 5. 恢复 → 6. 复盘

  各阶段要点：
  ┌──────────────┬──────────────────────────────┐
  │ 阶段          │ 关键动作                     │
  ├──────────────┼──────────────────────────────┤
  │ 准备          │ 预案/工具/团队/演练         │
  │ 检测          │ 监控告警/异常发现           │
  │ 遏制          │ 隔离受影响系统/阻断攻击     │
  │ 根除          │ 修复漏洞/清除恶意代码       │
  │ 恢复          │ 恢复服务/验证安全           │
  │ 复盘          │ RCA/改进措施/更新预案       │
  └──────────────┴──────────────────────────────┘

  常见安全事件处理：
  1. 数据泄露
     遏制：revoke所有API Key/Token
     根除：修复泄露点
     恢复：重置受影响用户密码
     通知：72小时内通知监管（GDPR）
  
  2. 服务被入侵
     遏制：隔离服务器/切断网络
     根除：从干净镜像重建
     恢复：检查数据完整性
  
  3. DDoS
     遏制：启用CDN/Anti-DDoS
     根除：分析流量特征/封禁源
     恢复：确认服务正常

  事件严重等级：
  P0：生产数据泄露/服务被接管
  P1：关键漏洞被利用中
  P2：发现高危漏洞未被利用
  P3：低风险安全问题
```

## 8. OAuth2/OIDC安全风险？

**回答：**

```
  常见攻击：
  1. 授权码截获
     防御：PKCE（code_verifier + code_challenge）
  
  2. 重定向URI篡改
     防御：严格匹配redirect_uri 不支持通配符
  
  3. CSRF攻击
     防御：state参数（随机值绑定用户会话）
  
  4. Token泄露
     前端localStorage不安全
     防御：BFF模式 Token存后端
  
  5. 令牌劫持
     防御：Token绑定（DPoP）
     证明Token持有者身份

  安全检查清单：
  ✓ 使用授权码模式+PKCE
  ✓ 验证state参数
  ✓ 严格redirect_uri白名单
  ✓ Access Token短有效期（15min）
  ✓ Refresh Token单次使用+轮换
  ✓ Scope最小化
  ✓ Token不在URL中传递
  ✓ ID Token验证iss/aud/exp/nonce

  Go安全验证ID Token：
  func VerifyIDToken(tokenString string, config OIDCConfig) (*Claims, error) {
      token, err := jwt.Parse(tokenString, func(t *jwt.Token) (interface{}, error) {
          // 验证签名算法
          if t.Method.Alg() != "RS256" {
              return nil, fmt.Errorf("unexpected algorithm: %s", t.Method.Alg())
          }
          return getPublicKey(config.JWKSURI, t.Header["kid"].(string))
      })
      
      claims := token.Claims.(jwt.MapClaims)
      // 验证issuer
      if claims["iss"] != config.Issuer { return nil, fmt.Errorf("invalid issuer") }
      // 验证audience
      if claims["aud"] != config.ClientID { return nil, fmt.Errorf("invalid audience") }
      
      return parseClaims(claims), nil
  }
```

## 9. 供应链安全？

**回答：**

```
  供应链攻击 = 通过依赖/工具链注入恶意代码

  攻击面：
  ┌──────────────┬──────────────────────────────┐
  │ 攻击面        │ 示例                         │
  ├──────────────┼──────────────────────────────┤
  │ 依赖包        │ NPM恶意包/包名typosquatting │
  │ 构建工具      │ SolarWinds事件              │
  │ 基础镜像      │ 投毒的Docker镜像            │
  │ CI/CD         │ Pipeline注入                │
  │ 开源代码      │ 恶意PR/维护者账号被盗       │
  └──────────────┴──────────────────────────────┘

  防护措施：
  1. 依赖管理
     go.sum校验完整性
     govulncheck定期扫描
     固定依赖版本（不用latest）
  
  2. SBOM（软件物料清单）
     syft myapp:latest -o spdx-json
     记录所有依赖及版本
  
  3. 镜像安全
     基础镜像用distroless/alpine
     Trivy扫描
     镜像签名(Cosign)
  
  4. 构建安全
     可重复构建
     构建环境隔离
     SLSA框架合规

  Go依赖安全：
  # 检查已知漏洞
  govulncheck ./...
  
  # 验证模块校验
  GONOSUMCHECK=off go build
  
  # 检查依赖许可证
  go-licenses check ./...

  Dockerfile安全：
  FROM golang:1.22 AS builder
  RUN CGO_ENABLED=0 go build -o /app
  
  FROM gcr.io/distroless/static-debian12
  COPY --from=builder /app /app
  USER nonroot:nonroot
  ENTRYPOINT ["/app"]
```
