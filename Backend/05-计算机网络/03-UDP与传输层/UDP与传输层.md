# UDP与传输层

---

## 1. UDP 协议特点？

**回答：**

```
UDP（User Datagram Protocol）：

  ┌──────────────┬──────────────────────────────┐
  │ 特性          │ 说明                          │
  ├──────────────┼──────────────────────────────┤
  │ 无连接        │ 不需要握手，直接发送            │
  │ 不可靠        │ 无确认、无重传、无排序          │
  │ 面向数据报    │ 保留消息边界（不粘包）          │
  │ 头部小        │ 仅 8 字节                      │
  │ 无拥塞控制    │ 发送速率恒定                   │
  │ 支持广播/组播 │ 一对多通信                     │
  └──────────────┴──────────────────────────────┘

UDP 头部格式（8 字节）：
  ┌──────────┬──────────┐
  │ 源端口    │ 目标端口  │  4 字节
  ├──────────┼──────────┤
  │ 长度      │ 校验和    │  4 字节
  └──────────┴──────────┘

使用场景：
  DNS 查询（53端口）
  DHCP（67/68端口）
  实时音视频（RTP）
  在线游戏
  直播推流
  IoT 传感器数据
```

## 2. TCP vs UDP 完整对比？

**回答：**

```
  ┌──────────────┬──────────────┬──────────────┐
  │ 维度          │ TCP          │ UDP          │
  ├──────────────┼──────────────┼──────────────┤
  │ 连接          │ 三次握手建连  │ 无连接       │
  │ 可靠性        │ 保证送达和顺序│ 不保证       │
  │ 传输方式      │ 字节流        │ 数据报       │
  │ 头部大小      │ 20-60 字节   │ 8 字节       │
  │ 拥塞控制      │ 有            │ 无           │
  │ 流量控制      │ 有（滑动窗口）│ 无           │
  │ 消息边界      │ 无（需自处理）│ 有           │
  │ 广播/组播     │ 不支持        │ 支持         │
  │ 速度          │ 较慢          │ 快           │
  │ 适用          │ 可靠传输      │ 实时/效率    │
  └──────────────┴──────────────┴──────────────┘

选型指南：
  需要可靠 → TCP（HTTP/数据库/文件传输）
  需要速度 → UDP（视频流/游戏/DNS）
  两者兼顾 → QUIC（基于 UDP 实现可靠）
```

## 3. 基于 UDP 的可靠传输？

**回答：**

```
为什么要基于 UDP 做可靠传输？
  TCP 的问题：
  - 队头阻塞（一个包丢失阻塞后续所有包）
  - 握手开销（1-2 RTT）
  - 内核协议栈不可定制

在应用层实现可靠性：
  ┌──────────────────────────────┐
  │ 应用层（可靠性/拥塞控制）     │
  ├──────────────────────────────┤
  │ UDP（传输）                   │
  └──────────────────────────────┘

典型实现：
  QUIC（Google/HTTP3）
    多路复用无队头阻塞
    0-RTT 建连
    连接迁移（换 IP 不断连）

  KCP：
    快速重传（不等超时）
    非退让流控（牺牲带宽换低延迟）
    适合游戏/实时通信

  WebRTC（数据通道）：
    P2P 音视频通信
    DTLS + SRTP

需要实现的组件：
  序列号 + 确认机制
  重传策略
  排序与去重
  拥塞控制
  连接管理
```

## 4. QUIC 协议？

**回答：**

```
QUIC (Quick UDP Internet Connections)：
  Google 设计，HTTP/3 的传输协议

  ┌──────────────┬──────────────────────────────┐
  │ 特性          │ 说明                          │
  ├──────────────┼──────────────────────────────┤
  │ 基于 UDP      │ 用户态实现，易部署升级         │
  │ 0-RTT        │ 首次 1-RTT，恢复连接 0-RTT    │
  │ 多路复用      │ 流独立，无队头阻塞             │
  │ 连接迁移      │ 用 Connection ID 标识         │
  │ 内建 TLS 1.3  │ 加密与握手合并                 │
  │ 改进拥塞控制  │ 默认 Cubic，可用 BBR          │
  └──────────────┴──────────────────────────────┘

QUIC vs TCP+TLS：
  TCP+TLS:  TCP 握手(1RTT) + TLS 握手(1-2RTT) = 2-3 RTT
  QUIC:     合并握手 = 1 RTT（恢复 0 RTT）

  TCP 多路复用问题：
    一条 TCP 连接中一个包丢 → 所有流阻塞
  QUIC 多路复用：
    每条流独立，一条流丢包不影响其他流

连接迁移：
  TCP 用四元组标识连接（IP+端口变了就断）
  QUIC 用 Connection ID（WiFi 切 4G 不断连）
```

## 5. 端口与多路复用？

**回答：**

```
端口号：16 位，范围 0-65535

  ┌──────────────┬──────────────────────────────┐
  │ 范围          │ 说明                          │
  ├──────────────┼──────────────────────────────┤
  │ 0-1023       │ 知名端口（需 root）            │
  │ 1024-49151   │ 注册端口                      │
  │ 49152-65535  │ 临时端口（客户端自动分配）     │
  └──────────────┴──────────────────────────────┘

常见端口：
  22(SSH) 53(DNS) 80(HTTP) 443(HTTPS)
  3306(MySQL) 5432(PostgreSQL) 6379(Redis)
  8080(HTTP备选) 9090(Prometheus)

传输层多路复用/解复用：
  多路复用：多个应用数据通过一个网络连接发送
  解复用：根据端口号将数据分发给正确的应用

  Socket 五元组唯一标识一个连接：
  {协议, 源IP, 源端口, 目标IP, 目标端口}

  服务端同一端口可有多个连接：
  因为每个连接的源 IP+源端口不同
  accept() 返回新的 Socket（新五元组）
```

## 6. 广播与组播？

**回答：**

```
通信模型：
  单播（Unicast）：一对一（TCP/大部分UDP）
  广播（Broadcast）：一对全部（仅 UDP，限局域网）
  组播（Multicast）：一对多（UDP，可跨网络）
  任播（Anycast）：一对最近一个（DNS/CDN）

广播：
  目标地址 255.255.255.255（有限广播）
  或子网广播地址（如 192.168.1.255）
  仅限本网段，路由器不转发

组播：
  组播地址范围：224.0.0.0 ~ 239.255.255.255
  主机加入组播组接收数据
  IGMP 协议管理组成员

  应用：
  视频直播/IPTV
  集群心跳
  服务发现

  ┌──────────┐
  │ 发送方    │
  │ 发一次    │
  └────┬─────┘
       │ 组播
  ┌────┼────┬─────┐
  │    │    │     │
  ↓    ↓    ↓     ↓
  A    B    C     D  （组成员）
```

## 7. NAT 穿透？

**回答：**

```
NAT 类型（由宽松到严格）：
  完全锥型：任何外部主机都能通过映射端口访问
  限制锥型：只有内部先向其发过包的外部 IP 可访问
  端口限制锥：外部 IP+端口 都要匹配
  对称型：每个目标产生不同映射（最难穿透）

NAT 穿透方法：
  ┌──────────────┬──────────────────────────────┐
  │ 方法          │ 说明                          │
  ├──────────────┼──────────────────────────────┤
  │ UPnP         │ 自动端口映射（需路由器支持）   │
  │ STUN         │ 发现自己的公网 IP:端口         │
  │ TURN         │ 中继转发（保底方案）           │
  │ ICE          │ 综合框架（STUN+TURN）         │
  │ 打洞 (Punch) │ 双方同时向对方发包             │
  └──────────────┴──────────────────────────────┘

UDP 打洞流程：
  1. A 和 B 都连接中心服务器 S
  2. S 记录 A 和 B 的公网 IP:端口
  3. S 通知双方对方的公网地址
  4. A 和 B 同时向对方发 UDP 包
  5. NAT 设备看到出站包后允许入站

对称型 NAT 穿透难：
  每个目标映射不同端口 → 无法预测
  只能用 TURN 中继
```

## 8. 传输层安全？

**回答：**

```
传输层安全协议：

  TLS（Transport Layer Security）：
    在传输层和应用层之间
    TCP + TLS → HTTPS/gRPCS
    提供：加密 + 身份验证 + 完整性

  DTLS（Datagram TLS）：
    UDP 版本的 TLS
    用于 WebRTC / VPN(OpenVPN)

  TLS 1.3 vs TLS 1.2：
    ┌──────────────┬──────────────┬──────────────┐
    │ 特性          │ TLS 1.2      │ TLS 1.3      │
    ├──────────────┼──────────────┼──────────────┤
    │ 握手 RTT      │ 2 RTT       │ 1 RTT        │
    │ 恢复连接      │ 1 RTT       │ 0 RTT        │
    │ 密钥交换      │ RSA/ECDHE   │ 仅 ECDHE     │
    │ 对称加密      │ AES-CBC等   │ AEAD 专属    │
    │ 前向保密      │ 可选         │ 强制          │
    └──────────────┴──────────────┴──────────────┘

  前向保密 (PFS)：
    使用临时密钥（ECDHE）
    私钥泄露不影响历史通信
    TLS 1.3 强制要求
```

## 9. 传输层在 Go 中的使用？

**回答：**

```go
// UDP 服务端
func udpServer() {
    addr, _ := net.ResolveUDPAddr("udp", ":8080")
    conn, _ := net.ListenUDP("udp", addr)
    defer conn.Close()

    buf := make([]byte, 1024)
    for {
        n, remoteAddr, _ := conn.ReadFromUDP(buf)
        fmt.Printf("from %s: %s\n", remoteAddr, buf[:n])
        conn.WriteToUDP([]byte("reply"), remoteAddr)
    }
}

// UDP 客户端
func udpClient() {
    conn, _ := net.Dial("udp", "localhost:8080")
    defer conn.Close()
    conn.Write([]byte("hello"))
    buf := make([]byte, 1024)
    n, _ := conn.Read(buf)
    fmt.Println(string(buf[:n]))
}
```

// TLS TCP 服务端
func tlsServer() {
    cert, _ := tls.LoadX509KeyPair("cert.pem", "key.pem")

## 10. UDP与传输层面试速答？

**回答：**

```
Q: TCP 和 UDP 区别？
A: TCP 面向连接可靠字节流，头部20字节
   UDP 无连接不可靠数据报，头部8字节

Q: UDP 适用场景？
A: 实时音视频、游戏、DNS、直播
   允许丢包但要求低延迟的场景

Q: UDP 怎么实现可靠传输？
A: 应用层加序列号+ACK+重传+排序
   典型实现：QUIC/KCP

Q: QUIC 的优势？
A: 0-RTT建连、流独立无队头阻塞
   连接迁移（WiFi切4G不断）
   HTTP/3 使用 QUIC

Q: 为什么 DNS 用 UDP？
A: DNS 查询小（<512字节），一次往返
   UDP 无握手开销更快
   超过 512 字节或区域传输时用 TCP

Q: 一个端口能建多少连接？
A: 理论 2^48（源IP×源端口组合）
   实际受文件描述符和内存限制
   C10K/C1M 问题

Q: NAT 穿透怎么做？
A: STUN 发现公网地址 + UDP 打洞
   对称型 NAT 用 TURN 中继
   ICE = STUN + TURN 综合方案
```
