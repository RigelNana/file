# 计算机网络八股文

## 一、网络分层模型

### 1. OSI 七层和 TCP/IP 四层？
**答：**
OSI：应用层→表示层→会话层→传输层→网络层→数据链路层→物理层。实际使用 TCP/IP 四层：应用层（HTTP/DNS/gRPC）、传输层（TCP/UDP）、网络层（IP/ICMP）、网络接口层（以太网/WiFi）。每层只与相邻层交互，上层使用下层服务。

### 2. 各层常见协议？
**答：**
应用层：HTTP/HTTPS/DNS/FTP/SMTP/gRPC/WebSocket。传输层：TCP/UDP/QUIC。网络层：IP/ICMP/ARP/OSPF/BGP。链路层：以太网/WiFi/PPP。ARP 严格说在网络层和链路层之间。

## 二、TCP 协议

### 3. TCP 三次握手？
**答：**
客户端发 SYN(seq=x) → 服务端回 SYN+ACK(seq=y,ack=x+1) → 客户端发 ACK(ack=y+1)。三次而非两次是为了防止已失效的连接请求突然到达服务端导致资源浪费；也确保双方都确认了收发能力。

### 4. TCP 四次挥手？
**答：**
主动方 FIN → 被动方 ACK（半关闭）→ 被动方 FIN → 主动方 ACK → 主动方进入 TIME_WAIT（等 2MSL=60s）。四次是因为 TCP 全双工，每个方向需要独立关闭。TIME_WAIT 确保最后一个 ACK 到达以及旧连接包过期。

### 5. TCP 可靠传输机制？
**答：**
序列号+确认号保证有序；超时重传+快速重传（3个重复ACK）保证不丢；滑动窗口实现流量控制；校验和检查数据完整性。拥塞控制（慢启动→拥塞避免→快恢复）防止网络过载。

### 6. TCP 拥塞控制？
**答：**
慢启动：cwnd 从 1 指数增长到 ssthresh。拥塞避免：cwnd 线性增长。快重传：3 个重复 ACK 立即重传。快恢复：ssthresh = cwnd/2，cwnd = ssthresh+3 继续传输。BBR 算法：基于带宽和 RTT 估计，不依赖丢包，更适合现代网络。

### 7. TCP 粘包拆包？
**答：**
TCP 是字节流协议无消息边界。解决：固定长度；特殊分隔符（\r\n）；长度字段前缀（最常用，如 4 字节表示消息长度）；更高层协议（HTTP Content-Length/Transfer-Encoding: chunked）。

## 三、UDP 与传输层

### 8. TCP vs UDP？
**答：**

| 对比 | TCP | UDP |
|------|-----|-----|
| 连接 | 面向连接 | 无连接 |
| 可靠性 | 可靠（重传/排序） | 不可靠 |
| 速度 | 较慢（握手开销） | 快 |
| 传输方式 | 字节流 | 数据报 |
| 头部开销 | 20 字节 | 8 字节 |

UDP 适用：实时音视频、DNS、游戏、直播。可在应用层实现可靠性（如 QUIC/KCP）。

## 四、HTTP 协议

### 9. HTTP 常见状态码？
**答：**
200 OK、201 Created、204 No Content、301 永久重定向、302 临时重定向、304 未修改（缓存）、400 请求错误、401 未认证、403 禁止、404 未找到、405 方法不允许、429 频率限制、500 服务器错误、502 网关错误、503 服务不可用、504 网关超时。

### 10. HTTP/1.1 vs HTTP/2 vs HTTP/3？
**答：**
HTTP/1.1：Keep-Alive 长连接，管线化（但有队头阻塞）。HTTP/2：二进制帧、多路复用（一个连接多个流）、头部压缩（HPACK）、服务器推送，解决了 HTTP 层队头阻塞但仍有 TCP 层队头阻塞。HTTP/3：基于 QUIC（UDP），解决 TCP 队头阻塞，0-RTT 握手，连接迁移。

### 11. GET vs POST？
**答：**
GET 用于获取资源，参数在 URL 中，幂等可缓存，有长度限制；POST 用于提交数据，参数在 Body 中，非幂等不缓存。本质都是 HTTP 请求，技术上 GET 也可以带 Body（但不推荐）。语义上 GET 读取 POST 写入。

## 五、HTTPS 与 TLS

### 12. HTTPS 握手过程？
**答：**
TLS 1.2：Client Hello(支持加密套件/随机数) → Server Hello(选定套件/随机数/证书) → 客户端验证证书→ 生成预主密钥用服务端公钥加密发送 → 双方算出会话密钥 → Change Cipher Spec → 加密通信。TLS 1.3 简化为 1-RTT（支持 0-RTT）。

### 13. 对称加密 vs 非对称加密？
**答：**
对称：相同密钥加解密（AES/ChaCha20），速度快用于数据传输。非对称：公钥加密私钥解密（RSA/ECDHE），速度慢用于密钥交换和签名。HTTPS 用非对称加密交换对称密钥，然后用对称密钥加密数据。

## 六、DNS

### 14. DNS 解析过程？
**答：**
浏览器缓存 → 系统缓存 → hosts 文件 → 本地 DNS 服务器 → 根域名服务器(.） → 顶级域名服务器(.com) → 权威域名服务器(example.com) → 返回 IP。递归查询（客户端到本地 DNS）+ 迭代查询（DNS 服务器间）。
