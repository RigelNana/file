# 接口幂等性

---

## 1. 什么是幂等性？

**回答：**

```
  幂等 = 同一操作执行一次和执行多次效果一致

  HTTP方法幂等性：
  ┌──────────┬──────────┬──────────────────────────┐
  │ 方法      │ 幂等     │ 说明                     │
  ├──────────┼──────────┼──────────────────────────┤
  │ GET       │ ✅       │ 查询 不修改数据           │
  │ PUT       │ ✅       │ 全量替换 结果一致         │
  │ DELETE    │ ✅       │ 删除 多次删同一个结果一致 │
  │ HEAD      │ ✅       │ 获取元数据                │
  │ OPTIONS   │ ✅       │ 查询支持方法              │
  │ POST      │ ❌       │ 创建 多次会创建多条       │
  │ PATCH     │ ❌       │ 取决于实现                │
  └──────────┴──────────┴──────────────────────────┘

  为什么需要幂等？
  网络超时 → 客户端重试 → 可能重复执行
  消息重复消费
  前端按钮连续点击

  场景示例：
  支付100元 → 网络超时 → 重试
  没有幂等 → 扣了200元（灾难！）
  有幂等 → 只扣100元（正确）
```

## 2. 幂等性实现方案？

**回答：**

```
  方案对比：
  ┌──────────────────┬──────────────────┬──────────────────┐
  │ 方案              │ 适用场景          │ 复杂度           │
  ├──────────────────┼──────────────────┼──────────────────┤
  │ 唯一请求ID        │ 通用              │ 中               │
  │ 数据库唯一约束    │ 创建操作          │ 低               │
  │ 状态机            │ 有状态流转的操作  │ 中               │
  │ Token机制         │ 表单提交          │ 中               │
  │ 乐观锁            │ 更新操作          │ 低               │
  │ Redis SETNX       │ 通用 高性能       │ 低               │
  └──────────────────┴──────────────────┴──────────────────┘

  方案1: 唯一请求ID（Idempotency-Key）
  客户端生成唯一ID → 服务端去重

  方案2: 数据库唯一约束
  CREATE UNIQUE INDEX ON orders(order_no)
  重复插入报错 → 返回已有结果

  方案3: 状态机
  订单: 待支付→已支付→已发货→已完成
  已支付状态不能再次支付

  方案4: Token机制
  先请求Token → 提交时带Token → 使用后作废

  方案5: 乐观锁
  UPDATE WHERE version = ?
  版本不匹配 → 更新失败 → 重新读取
```

## 3. Idempotency-Key实现？

**回答：**

```
  流程：
  1. 客户端生成UUID作为Idempotency-Key
  2. 请求Header: Idempotency-Key: <uuid>
  3. 服务端检查Key是否已处理过
  4. 已处理 → 返回缓存的响应
  5. 未处理 → 执行业务 → 存储响应

  Go实现：
  func IdempotencyMiddleware(store IdempotencyStore) func(http.Handler) http.Handler {
      return func(next http.Handler) http.Handler {
          return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
              // 只对写操作做幂等
              if r.Method == "GET" || r.Method == "HEAD" {
                  next.ServeHTTP(w, r)
                  return
              }
              
              key := r.Header.Get("Idempotency-Key")
              if key == "" {
                  next.ServeHTTP(w, r)
                  return
              }
              
              // 检查是否已处理
              cached, err := store.Get(r.Context(), key)
              if err == nil {
                  // 返回缓存响应
                  w.WriteHeader(cached.StatusCode)
                  w.Write(cached.Body)
                  return
              }
              
              // 防止并发重复执行
              if !store.Lock(r.Context(), key, 30*time.Second) {
                  http.Error(w, "request in progress", 409)
                  return
              }
              
              // 捕获响应
              rec := &responseRecorder{ResponseWriter: w}
              next.ServeHTTP(rec, r)
              
              // 存储响应（TTL 24小时）
              store.Set(r.Context(), key, &CachedResponse{
                  StatusCode: rec.statusCode,
                  Body:       rec.body.Bytes(),
              }, 24*time.Hour)
          })
      }
  }

  Redis存储：
  Key: idempotency:<key>
  Value: JSON(statusCode + body)
  TTL: 24h
```

## 4. 数据库级幂等？

**回答：**

```
  唯一约束（最简单有效）：
  CREATE TABLE orders (
      id BIGINT PRIMARY KEY,
      order_no VARCHAR(64) UNIQUE,  -- 业务唯一号
      user_id BIGINT NOT NULL,
      amount DECIMAL(10,2)
  );
  
  // Go - INSERT忽略重复
  func CreateOrder(ctx context.Context, db *sql.DB, order *Order) error {
      _, err := db.ExecContext(ctx,
          `INSERT INTO orders (order_no, user_id, amount)
           VALUES (?, ?, ?)
           ON DUPLICATE KEY UPDATE id=id`, // MySQL
          order.OrderNo, order.UserID, order.Amount)
      return err
  }

  乐观锁（更新操作）：
  func UpdateBalance(ctx context.Context, db *sql.DB, userID int64, amount float64) error {
      for i := 0; i < 3; i++ {
          var balance float64
          var version int
          db.QueryRowContext(ctx,
              "SELECT balance, version FROM accounts WHERE user_id = ?",
              userID).Scan(&balance, &version)
          
          result, err := db.ExecContext(ctx,
              `UPDATE accounts SET balance = ?, version = version + 1
               WHERE user_id = ? AND version = ?`,
              balance+amount, userID, version)
          
          rows, _ := result.RowsAffected()
          if rows > 0 { return nil }
          // 版本冲突 重试
      }
      return fmt.Errorf("optimistic lock failed")
  }

  INSERT IF NOT EXISTS（PostgreSQL）：
  INSERT INTO orders (order_no, user_id, amount)
  VALUES ($1, $2, $3)
  ON CONFLICT (order_no) DO NOTHING
  RETURNING id;
```

## 5. Redis幂等去重？

**回答：**

```
  Redis SETNX（最快方案）：
  func CheckIdempotent(ctx context.Context, rdb *redis.Client, key string, ttl time.Duration) (bool, error) {
      // SETNX: 不存在才设置 → 返回true
      ok, err := rdb.SetNX(ctx, "idem:"+key, "1", ttl).Result()
      return ok, err
  }
  
  func ProcessPayment(ctx context.Context, req *PayRequest) (*PayResponse, error) {
      // 1. 幂等检查
      isNew, err := CheckIdempotent(ctx, rdb, req.IdempotencyKey, 24*time.Hour)
      if err != nil {
          return nil, err
      }
      if !isNew {
          // 已处理过 返回缓存结果
          return getCachedResult(ctx, req.IdempotencyKey)
      }
      
      // 2. 执行业务
      result, err := doPayment(ctx, req)
      if err != nil {
          // 失败需要删除幂等Key（允许重试）
          rdb.Del(ctx, "idem:"+req.IdempotencyKey)
          return nil, err
      }
      
      // 3. 缓存结果
      cacheResult(ctx, req.IdempotencyKey, result)
      return result, nil
  }

  注意事项：
  1. 失败要删除Key（允许重试）
  2. 成功后缓存结果（重复请求返回同样结果）
  3. TTL不能太短（客户端可能很久才重试）
  4. Key = 业务唯一标识（订单号/请求ID）
  
  并发安全：
  SETNX是原子操作
  多个请求同时到达 → 只有一个SETNX成功
```

## 6. 状态机幂等？

**回答：**

```
  状态流转（订单为例）：
  待支付 → 已支付 → 已发货 → 已完成
     ↓        ↓
    已取消   已退款

  状态机保证幂等：
  当前状态不允许的操作直接拒绝

  func PayOrder(ctx context.Context, db *sql.DB, orderID int64) error {
      tx, _ := db.BeginTx(ctx, nil)
      defer tx.Rollback()
      
      var status string
      tx.QueryRowContext(ctx,
          "SELECT status FROM orders WHERE id = ? FOR UPDATE",
          orderID).Scan(&status)
      
      // 状态检查
      switch status {
      case "paid":
          // 已支付 幂等返回成功
          return nil
      case "pending":
          // 待支付 可以继续
      default:
          return fmt.Errorf("order status %s cannot be paid", status)
      }
      
      // 执行支付
      _, err := tx.ExecContext(ctx,
          "UPDATE orders SET status = 'paid', paid_at = NOW() WHERE id = ?",
          orderID)
      if err != nil { return err }
      
      return tx.Commit()
  }

  状态机幂等的特点：
  1. 已处理的操作 → 直接返回成功
  2. 不允许的操作 → 拒绝
  3. 配合SELECT FOR UPDATE防并发
  4. 适合有明确状态流转的业务
```

## 7. Token机制幂等？

**回答：**

```
  流程：
  1. 客户端请求Token → GET /api/token
  2. 服务端生成Token → 存Redis → 返回Token
  3. 客户端提交表单 → 带上Token
  4. 服务端验证Token → 删除Token → 执行业务
  5. 重复提交 → Token已不存在 → 拒绝

  Go实现：
  // 获取Token
  func GetToken(w http.ResponseWriter, r *http.Request) {
      token := uuid.New().String()
      rdb.Set(r.Context(), "form_token:"+token, "1", 10*time.Minute)
      json.NewEncoder(w).Encode(map[string]string{"token": token})
  }
  
  // 验证并消费Token
  func SubmitForm(w http.ResponseWriter, r *http.Request) {
      token := r.Header.Get("X-Form-Token")
      if token == "" {
          http.Error(w, "missing token", 400)
          return
      }
      
      // 原子操作：获取并删除
      result, err := rdb.GetDel(r.Context(), "form_token:"+token).Result()
      if err != nil || result == "" {
          http.Error(w, "invalid or expired token", 409)
          return
      }
      
      // Token有效且已消费 执行业务
      processForm(r)
  }

  适用场景：
  表单提交（防重复提交）
  前端按钮防连击的后端保障
  
  注意：Token有过期时间
  过期后需要重新获取
```

## 8. 消息队列幂等消费？

**回答：**

```
  消息可能重复消费：
  消费者处理成功 → ACK失败 → 消息重新投递

  幂等消费方案：
  1. 去重表
  func ConsumeMessage(ctx context.Context, msg *Message) error {
      // 检查是否已消费
      _, err := db.ExecContext(ctx,
          `INSERT INTO consumed_messages (msg_id, created_at)
           VALUES (?, NOW())`,
          msg.ID)
      if err != nil {
          // 唯一键冲突 → 已消费 跳过
          if isDuplicateError(err) {
              log.Printf("message %s already consumed, skip", msg.ID)
              return nil
          }
          return err
      }
      
      // 执行业务
      return processMessage(ctx, msg)
  }

  2. 业务状态判断
  func HandleOrderPaid(ctx context.Context, msg *OrderPaidMsg) error {
      order, _ := db.GetOrder(ctx, msg.OrderID)
      
      // 已处理过 跳过
      if order.Status == "paid" {
          return nil
      }
      
      // 处理业务
      return processOrderPaid(ctx, order)
  }

  3. Redis去重（高性能）
  func IsProcessed(ctx context.Context, msgID string) bool {
      ok, _ := rdb.SetNX(ctx, "msg:"+msgID, "1", 7*24*time.Hour).Result()
      return !ok // true=已处理
  }

  最佳实践：
  消息ID全局唯一
  去重窗口足够长
  业务操作本身幂等
```

## 9. 分布式场景幂等？

**回答：**

```
  分布式幂等难点：
  1. 多实例并发 → 同一请求打到不同节点
  2. 数据库和缓存一致性
  3. 跨服务调用幂等

  解决方案：
  1. Redis分布式锁 + 幂等Key
  func ProcessWithIdempotent(ctx context.Context, key string, fn func() error) error {
      // 分布式锁防并发
      lock, err := redislock.Obtain(ctx, rdb, "lock:"+key, 30*time.Second, nil)
      if err != nil {
          return fmt.Errorf("another request in progress")
      }
      defer lock.Release(ctx)
      
      // 检查是否已处理
      exists, _ := rdb.Exists(ctx, "done:"+key).Result()
      if exists > 0 {
          return nil // 已处理
      }
      
      // 执行业务
      if err := fn(); err != nil {
          return err
      }
      
      // 标记完成
      rdb.Set(ctx, "done:"+key, "1", 24*time.Hour)
      return nil
  }

  2. 跨服务幂等
  服务A调用服务B → 传递Idempotency-Key
  服务B也做幂等检查
  整个调用链都是幂等的

  3. Saga事务幂等
  每一步都要幂等
  补偿操作也要幂等
  eg: 退款操作 多次调用只退一次
```

## 10. 幂等性面试速答？

**回答：**

```
Q: 什么是幂等性？
A: 同一操作执行多次结果一致
   网络重试/消息重复不会导致重复处理

Q: 哪些HTTP方法天然幂等？
A: GET/PUT/DELETE/HEAD/OPTIONS
   POST不幂等 需要额外设计

Q: 最简单的幂等方案？
A: Redis SETNX + 唯一请求ID
   原子操作 高性能

Q: 数据库怎么保证幂等？
A: 唯一约束（UNIQUE KEY）
   ON DUPLICATE KEY UPDATE / ON CONFLICT DO NOTHING

Q: 支付怎么做幂等？
A: 订单号唯一约束+状态机
   同一订单号只能支付一次

Q: 消息消费怎么幂等？
A: 消息ID去重表/Redis去重
   业务状态判断也是幂等手段

Q: Token机制怎么用？
A: 先获取Token→提交时带Token→用后作废
   适合表单防重复提交

Q: 分布式场景怎么做幂等？
A: Redis分布式锁+幂等Key
   跨服务传递Idempotency-Key
```
