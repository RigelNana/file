## 一、回滚基础

### 1. 什么是回滚？有哪些回滚类型？

**答：**

```
回滚 = 将系统恢复到已知正常状态

回滚类型：
┌─────────────────┬──────────────┬──────────┬────────────┐
│ 类型             │ 回滚对象      │ 回滚速度  │ 风险       │
├─────────────────┼──────────────┼──────────┼────────────┤
│ 代码回滚         │ 应用版本      │ 分钟级   │ 低         │
│ 配置回滚         │ 配置变更      │ 秒-分钟  │ 低         │
│ 数据库回滚       │ Schema/数据   │ 小时级   │ 高         │
│ 基础设施回滚     │ IaC 变更      │ 分钟级   │ 中         │
│ Feature Flag    │ 功能开关      │ 秒级     │ 极低       │
└─────────────────┴──────────────┴──────────┴────────────┘

回滚决策优先级：
  1. Feature Flag 关闭（最快最安全）
  2. 配置回滚（如果是配置变更）
  3. 代码回滚（重新部署旧版本）
  4. 数据库回滚（最后手段）
```

### 2. 回滚的触发条件如何定义？

**答：**

```
自动回滚触发条件：

1. 健康检查失败
   - Pod 持续 CrashLoopBackOff
   - Readiness Probe 连续失败

2. SLO 违反
   - 错误率超过错误预算燃烧率
   - 延迟 P99 超过阈值 > 2分钟

3. 金丝雀分析失败
   - Flagger/Argo Rollouts 自动判定

手动回滚触发：

1. 业务指标异常
   - 转化率下降
   - 用户投诉增加

2. 安全事件
   - 发现安全漏洞
   - 数据泄露风险

回滚决策矩阵：
  影响范围  │ 影响 < 1%  │ 影响 1-10% │ 影响 > 10%
  ─────────┼───────────┼───────────┼───────────
  已有缓解  │ 观察       │ 准备回滚    │ 立即回滚
  无法缓解  │ 准备回滚    │ 立即回滚    │ 立即回滚
```

### 3. K8s 中的回滚机制？

**答：**

```bash
# 查看部署历史
kubectl rollout history deployment/myapp

# 回滚到上一个版本
kubectl rollout undo deployment/myapp

# 回滚到指定版本
kubectl rollout undo deployment/myapp --to-revision=3

# 查看回滚状态
kubectl rollout status deployment/myapp

# 暂停/恢复滚动更新
kubectl rollout pause deployment/myapp
kubectl rollout resume deployment/myapp
```

```yaml
# Deployment 保留历史版本数
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  revisionHistoryLimit: 10  # 保留 10 个历史版本
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0     # 回滚时不允许不可用
```

## 二、回滚实践

### 4. Argo Rollouts 的自动回滚如何实现？

**答：**

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: payment-api
spec:
  strategy:
    canary:
      steps:
        - setWeight: 5
        - pause: { duration: 5m }
        - analysis:
            templates:
              - templateName: success-rate
            args:
              - name: service-name
                value: payment-api
        - setWeight: 20
        - pause: { duration: 5m }
        - setWeight: 50
        - pause: { duration: 10m }
        - setWeight: 100
      
      # 自动回滚配置
      abortScaleDownDelaySeconds: 30
      
      # 分析失败时的行为
      analysis:
        successfulRunHistoryLimit: 3
        unsuccessfulRunHistoryLimit: 3

---
apiVersion: argoproj.io/v1alpha1
kind: AnalysisTemplate
metadata:
  name: success-rate
spec:
  args:
    - name: service-name
  metrics:
    - name: success-rate
      interval: 1m
      count: 5
      failureLimit: 2
      provider:
        prometheus:
          address: http://prometheus:9090
          query: |
            sum(rate(http_requests_total{
              service="{{args.service-name}}",
              code!~"5.."
            }[2m])) /
            sum(rate(http_requests_total{
              service="{{args.service-name}}"
            }[2m]))
      successCondition: result[0] > 0.99
```

### 5. 数据库变更如何安全回滚？

**答：**

```
数据库回滚的核心原则：
  所有 Schema 变更必须向前兼容
  每个变更都有对应的回滚脚本

不可回滚的操作（必须避免）：
  ✗ 直接删除列/表
  ✗ 重命名列（旧代码找不到）
  ✗ 缩小字段长度
  ✗ 修改字段类型（不兼容）

安全的变更模式：

  添加列：
    UP:   ALTER TABLE ADD COLUMN new_col
    DOWN: ALTER TABLE DROP COLUMN new_col
    
  删除列（分步）：
    Step 1: 代码停止使用该列（发布）
    Step 2: 确认无查询引用
    Step 3: 添加列注释 "deprecated"
    Step 4: 一个迭代后删除
```

```python
# Alembic 迁移示例（双向）
def upgrade():
    op.add_column('users',
        sa.Column('contact_email', sa.String(255))
    )
    # 数据迁移
    op.execute(
        "UPDATE users SET contact_email = email"
    )

def downgrade():
    op.drop_column('users', 'contact_email')
```

## 三、面试

### 6. 面试题：如何设计一个完善的回滚体系？

**答：**

```
回滚体系四层设计：

第一层：预防
  - 变更前必须有回滚方案
  - 变更审批包含"回滚步骤"字段
  - 每个变更都做回滚演练

第二层：检测
  - 自动化金丝雀分析
  - SLO 违反告警
  - 业务指标异常检测

第三层：执行
  - 一键回滚脚本
  - 自动回滚（Argo/Flagger）
  - Feature Flag 快速关闭

第四层：验证
  - 回滚后自动执行 Smoke Test
  - 确认指标恢复正常
  - 通知相关人员

回滚 SLA：
  ┌──────────┬─────────────┐
  │ 回滚类型  │ 目标时间     │
  ├──────────┼─────────────┤
  │ Flag     │ < 1 分钟     │
  │ 配置     │ < 5 分钟     │
  │ 代码     │ < 15 分钟    │
  │ 数据库   │ < 1 小时     │
  └──────────┴─────────────┘
```

### 7. 面试题：回滚本身失败了怎么办？

**答：**

```
回滚失败的应急方案：

原因分析：
  1. 旧版本镜像不存在（清理了）
  2. 数据库不兼容（新数据旧代码读不了）
  3. 依赖服务已变更
  4. 配置中心已变更

预防措施：
  1. 镜像保留策略：至少保留 30 天
  2. 数据兼容：永远向后兼容 N-1
  3. 定期回滚演练
  4. 配置版本化管理

应急处理：
  回滚失败
    ├── 尝试修复前滚 (hotfix)
    ├── 手动部署指定版本
    ├── 数据库手动修复
    └── 最终手段：从备份恢复
    
  关键：
    不要在回滚失败后反复尝试
    → 启动事件响应流程
    → IC 介入决策
```
