# 变更冻结与风险窗口八股文

## 一、变更冻结

### 1. 什么是变更冻结（Change Freeze）？

**答：**

```
变更冻结 = 在特定时间段禁止或限制生产环境变更

目的：
  在高风险时段减少人为故障

典型冻结场景：
┌─────────────────┬────────────────────────┐
│ 场景             │ 冻结时间                │
├─────────────────┼────────────────────────┤
│ 节假日（双11等）  │ 活动前3天 ~ 活动后1天   │
│ 季度财报         │ 财报发布前1周           │
│ 重大活动         │ 活动前1天 ~ 活动后      │
│ 周末             │ 周五下午 ~ 周一上午     │
│ 夜间             │ 22:00 ~ 08:00          │
└─────────────────┴────────────────────────┘

冻结级别：
  Level 0: 完全冻结 — 禁止所有变更
  Level 1: 紧急冻结 — 仅允许 P0 修复
  Level 2: 部分冻结 — 禁止高风险变更
  Level 3: 观察期   — 允许变更但需额外审批
```

### 2. 变更冻结的流程如何设计？

**答：**

```
变更冻结生命周期：

  ┌────────────┐
  │ 1. 冻结通知 │ → 提前 N 天通知所有团队
  │   (T-7天)   │   → 完成在途变更或延期
  └─────┬──────┘
        ▼
  ┌────────────┐
  │ 2. 预冻结   │ → 只允许低风险变更
  │   (T-3天)   │   → 停止新的大型变更
  └─────┬──────┘
        ▼
  ┌────────────┐
  │ 3. 完全冻结 │ → 禁止所有非紧急变更
  │   (T-0)     │   → CI/CD Pipeline 自动阻断
  └─────┬──────┘
        ▼
  ┌────────────┐
  │ 4. 解冻评估 │ → 确认冻结期间无异常
  │   (T+1天)   │   → 逐步恢复变更
  └─────┬──────┘
        ▼
  ┌────────────┐
  │ 5. 完全解冻 │ → 恢复正常变更流程
  │   (T+2天)   │   → 积压变更有序发布
  └────────────┘

紧急变更例外流程：
  冻结期间 P0 故障
    → IC 发起紧急变更申请
    → VP 级别审批
    → 最小化变更范围
    → 额外 Reviewer 陪同
    → 立即验证 + 持续监控
```

### 3. 如何技术实现变更冻结？

**答：**

```yaml
# GitLab CI 变更冻结检查
deploy:
  stage: deploy
  rules:
    - if: '$CHANGE_FREEZE == "true"'
      when: never
    - if: '$CI_COMMIT_BRANCH == "main"'
      when: on_success
  script:
    - ./deploy.sh
```

```python
# 变更冻结中间件示例
import datetime

FREEZE_WINDOWS = [
    {
        "name": "双11冻结",
        "start": "2024-11-08T00:00:00",
        "end": "2024-11-13T00:00:00",
        "level": 0,           # 完全冻结
        "exceptions": ["P0"]  # 仅 P0 例外
    },
    {
        "name": "周末冻结",
        "weekday_start": 4,   # 周五
        "hour_start": 17,
        "weekday_end": 0,     # 周一
        "hour_end": 9,
        "level": 2,           # 部分冻结
    }
]

def is_change_allowed(change_risk_level):
    now = datetime.datetime.now()
    for window in FREEZE_WINDOWS:
        if in_freeze_window(now, window):
            if window["level"] == 0:
                return change_risk_level == "P0"
            elif window["level"] == 2:
                return change_risk_level in ("P0", "low")
    return True
```

## 二、风险窗口

### 4. 什么是变更风险窗口？如何定义？

**答：**

```
变更风险窗口 = 限定安全的变更时间段

设计原则：
  ▸ 在有充足人员监控时变更
  ▸ 避开业务高峰期
  ▸ 预留足够回滚时间
  ▸ 考虑全球时区

典型风险窗口设置：

  高风险变更：
    周二/周三 10:00-14:00
    （避开周一积压、周五影响周末）

  中风险变更：
    周一-周四 09:00-16:00

  低风险变更：
    工作日 09:00-18:00

  禁止变更：
    周五 17:00 - 周一 09:00
    节假日前后

时间线示例：
  ┌───┬───┬───┬───┬───┬───┬───┐
  │Mon│Tue│Wed│Thu│Fri│Sat│Sun│
  ├───┼───┼───┼───┼───┼───┼───┤
  │ M │ H │ H │ M │ L │ ✗ │ ✗ │
  └───┴───┴───┴───┴───┴───┴───┘
  H=高风险可  M=中风险可  L=低风险可  ✗=冻结
```

### 5. 如何计算变更风险分数？

**答：**

```
变更风险评分模型：

风险维度        权重    评分(1-5)
──────────────────────────────
变更范围         25%    影响多少服务
变更频率         15%    该服务多久没变更
回滚难度         20%    是否可快速回滚
测试覆盖         15%    自动化测试覆盖率
变更人员经验     10%    操作人对系统的熟悉度
历史故障率       15%    该服务变更引发故障的历史

风险分 = Σ (权重 × 评分)

风险等级：
  1.0 - 2.0  →  低风险（自动审批）
  2.0 - 3.0  →  中风险（团队 Lead 审批）
  3.0 - 4.0  →  高风险（SRE + 管理层审批）
  4.0 - 5.0  →  极高风险（VP 审批 + 陪同变更）
```

## 三、面试

### 6. 面试题：如何平衡变更速度和安全性？

**答：**

```
核心矛盾：
  冻结太严格 → 变更积压 → 解冻后大量变更 → 更危险
  冻结太宽松 → 频繁故障

平衡策略：

1. 渐进式信任
   - 自动化测试覆盖率高的服务 → 宽松策略
   - 新服务/低测试覆盖 → 严格策略

2. 小批量频繁发布
   - 每次变更范围小 → 风险小 → 不需要严格冻结
   - DORA 研究证明：高频发布 = 低故障率

3. 自动化保障
   - 自动回滚替代人工守护
   - 金丝雀分析替代冻结
   - Feature Flag 替代延迟发布

4. 差异化策略
   核心支付系统 → 严格冻结
   内部工具     → 宽松策略
   无状态服务   → 自动金丝雀即可

最终目标：
  不是"何时允许变更"
  而是"让每次变更都安全"
```

### 7. 面试题：冻结期间积压的变更怎么处理？

**答：**

```
冻结后变更积压处理：

错误做法：
  ✗ 解冻后所有变更同时发布
  ✗ 合并多个变更一起发
  ✗ 不测试直接发

正确做法：

1. 解冻缓冲期（1-2天）
   - 按优先级排序积压变更
   - 每天限定发布数量

2. 逐步发布
   Day 1: 只发最紧急的 2-3 个变更
   Day 2: 发中优先级变更
   Day 3: 恢复正常节奏

3. 每个变更独立发布
   - 不合并不相关变更
   - 每个变更间隔观察
   - 保持可回滚性

4. 额外监控
   - 解冻后一周内加强监控
   - 降低告警阈值
```
