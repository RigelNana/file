# 变更管理原则与流程八股文

---

## 一、变更管理基础

### 1. 为什么变更是事故的主要来源？

**答：**

```
Google SRE 统计：
  约 70% 的生产事故与变更相关

变更风险来源：
  ┌──────────────────────────────────┐
  │ 代码变更: 新 Bug、性能退化       │
  │ 配置变更: 错误配置、不兼容       │
  │ 基础设施: 网络/存储变更          │
  │ 依赖升级: 第三方库不兼容         │
  │ Schema 变更: 数据库迁移出错      │
  └──────────────────────────────────┘

变更管理的目标：
  不是阻止变更 → 而是让变更更安全
  "Move fast with guardrails"
```

### 2. 变更管理的核心原则是什么？

**答：**

| 原则 | 说明 | 实践 |
|------|------|------|
| **渐进式** | 小步快跑，逐步放量 | 金丝雀 → 灰度 → 全量 |
| **可观测** | 每次变更都能被监控到 | 部署后观察仪表盘 |
| **可回滚** | 任何变更都能快速回退 | 一键回滚机制 |
| **自动化** | 减少人为操作错误 | CI/CD Pipeline |
| **审批** | 高风险变更需要审核 | 变更评审流程 |

### 3. 变更分类和审批流程？

**答：**

```
变更分类：

Standard Change（标准变更）:
  ├── 预先审批的低风险变更
  ├── 例：更新应用配置、扩缩容
  ├── 审批：无需每次审批
  └── 流程：CI/CD 自动执行

Normal Change（普通变更）:
  ├── 需要评审的常规变更
  ├── 例：新功能发布、依赖升级
  ├── 审批：Peer Review + TL 审批
  └── 流程：PR → Review → 合并 → 自动发布

Emergency Change（紧急变更）:
  ├── 事件处理中的紧急修复
  ├── 例：Hotfix、紧急回滚
  ├── 审批：事后补审
  └── 流程：快速执行 → 事后 Review

审批矩阵：
  ┌────────────┬──────────┬──────────┬──────────┐
  │ 变更类型   │ Code     │ Config   │ Infra    │
  │            │ Review   │ Review   │ Review   │
  ├────────────┼──────────┼──────────┼──────────┤
  │ Standard   │ 自动     │ 自动     │ SRE      │
  │ Normal     │ Peer+TL  │ Peer     │ SRE+TL   │
  │ Emergency  │ 事后     │ 事后     │ SRE 当场 │
  └────────────┴──────────┴──────────┴──────────┘
```

## 二、变更流程设计

### 4. 完整的变更管理流程是什么？

**答：**

```
变更生命周期：

提交 → 评审 → 测试 → 发布 → 验证 → 关闭

详细流程：
┌─ 提交阶段 ──────────────────────────┐
│ 1. 开发提交 PR/MR                   │
│ 2. CI 自动化测试                     │
│ 3. 代码 Review                       │
└───────────┬──────────────────────────┘
            ▼
┌─ 评审阶段 ──────────────────────────┐
│ 4. 变更影响评估                      │
│ 5. 回滚方案确认                      │
│ 6. 发布窗口确认                      │
└───────────┬──────────────────────────┘
            ▼
┌─ 发布阶段 ──────────────────────────┐
│ 7. Pipeline 自动部署                 │
│ 8. 金丝雀/灰度发布                   │
│ 9. 自动化健康检查                    │
└───────────┬──────────────────────────┘
            ▼
┌─ 验证阶段 ──────────────────────────┐
│ 10. 监控仪表盘确认                   │
│ 11. SLO 指标检查                     │
│ 12. 全量发布或回滚                   │
└───────────┬──────────────────────────┘
            ▼
┌─ 关闭阶段 ──────────────────────────┐
│ 13. 标记变更完成                     │
│ 14. 如有问题，触发 Postmortem       │
└──────────────────────────────────────┘
```

### 5. 变更冻结窗口如何设计？

**答：**

```
变更冻结（Change Freeze）类型：

1. 周期性冻结
   ├── 每周五 16:00 后不发布（避免周末事故）
   ├── 节假日前 3 天冻结
   └── 月末/季末财务关账期间冻结

2. 事件驱动冻结
   ├── P0/P1 事件期间冻结所有变更
   └── 错误预算耗尽后冻结（SLO策略）

3. 业务驱动冻结
   ├── 大促前 1-3 天全面冻结
   ├── 重要活动期间冻结
   └── 审计期间冻结

例外流程：
  冻结期间的紧急变更需要：
  ├── SRE Manager 审批
  ├── 变更影响评估
  ├── 回滚方案确认
  └── 加强监控
```

## 三、面试高频题

### 6. 面试题：如何减少变更导致的事故？

**答：**

```
五层防护模型：

Layer 1: 预防
  ├── Code Review（多人 Review）
  ├── 自动化测试（单元+集成+E2E）
  ├── 变更影响分析
  └── 预发布环境验证

Layer 2: 检测
  ├── 金丝雀发布自动分析
  ├── 部署后 SLO 自动检查
  ├── 异常变更自动告警
  └── 发布与指标的关联展示

Layer 3: 限制
  ├── 渐进式发布（小批量放量）
  ├── 特性标志控制
  ├── 流量百分比控制
  └── 变更冻结窗口

Layer 4: 恢复
  ├── 一键回滚
  ├── 自动回滚（SLO 违规时）
  ├── 特性标志即时关闭
  └── 回滚时间 < 5 分钟

Layer 5: 学习
  ├── 变更相关事故 Postmortem
  ├── 改进 CI/CD 流程
  └── 分享 Lessons Learned
```

### 7. 面试题：如何平衡发布速度和稳定性？

**答：**

| 做法 | 提升速度 | 保障稳定 |
|------|----------|----------|
| CI/CD 自动化 | ✅ 减少手动步骤 | ✅ 减少人为错误 |
| 特性标志 | ✅ 代码随时发布 | ✅ 功能逐步开放 |
| 金丝雀发布 | ✅ 快速推进 | ✅ 小范围验证 |
| 自动回滚 | ✅ 减少恐惧 | ✅ 快速恢复 |
| 微服务 | ✅ 独立发布 | ⚠️ 增加复杂度 |

```
核心理念：
  "快不是不安全的理由，
   安全不是慢的借口。"

  正确的做法是让安全变得自动化，
  让快速发布成为默认操作。
```
