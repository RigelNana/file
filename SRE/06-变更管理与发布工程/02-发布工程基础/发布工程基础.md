## 一、发布工程概念

### 1. 什么是发布工程（Release Engineering）？

**答：**

```
发布工程定义：
  将代码变更安全、可靠地交付到生产环境的工程实践。

Google 对发布工程的定义：
  "Release engineering is a specific job function 
   at Google that builds and delivers software."

发布工程的四大原则（Google SRE）：
  ┌──────────────────────────────────────┐
  │ 1. 自助服务 (Self-Service)           │
  │    开发者能独立发布                   │
  │ 2. 高速度 (High Velocity)            │
  │    频繁小批量发布                     │
  │ 3. 封闭构建 (Hermetic Builds)        │
  │    构建结果确定性、可复现              │
  │ 4. 策略执行 (Enforcement of Policy)  │
  │    自动化的安全和合规检查              │
  └──────────────────────────────────────┘
```

### 2. 发布频率和批量大小的关系？

**答：**

```
DORA 指标的启示：

高绩效团队:
  发布频率: 每天多次
  变更批量: 小（单个 PR）
  故障率: 低
  MTTR: < 1 小时

低绩效团队:
  发布频率: 每月/每季度
  变更批量: 大（数十个变更）
  故障率: 高
  MTTR: > 1 周

大批量变更的问题：
  ├── 风险高（变量太多难以排查）
  ├── 回滚困难（哪个变更导致的？）
  ├── 反馈慢（问题隔很久才发现）
  └── 合并冲突多（长时间分支）

小批量发布的优势：
  ├── 问题容易定位（变化少好排查）
  ├── 回滚简单（只回退一个变更）
  ├── 反馈快（快速验证效果）
  └── 风险低（影响范围可控）
```

### 3. 发布的主要模型有哪些？

**答：**

| 模型 | 说明 | 适用场景 |
|------|------|----------|
| **持续部署** | 代码合并自动发布 | 微服务、SaaS |
| **发布列车** | 定时发布窗口 | 中大型团队 |
| **手动发布** | 人工触发 | 合规要求严格 |
| **GitOps** | Git 变更驱动部署 | K8s 环境 |

```
持续部署流水线：
  commit → build → test → deploy staging
  → integration test → canary → production

发布列车模型：
  每周二 10:00 AM 发布
  窗口前 24 小时 Code Freeze
  所有已合并的变更一起发布
  
  优点：有计划、可预测
  缺点：等待时间、批量可能偏大

GitOps 模型：
  代码仓库变更 → ArgoCD / Flux 检测
  → 自动同步到 K8s 集群
  
  优点：Git 即真理、审计友好
  缺点：学习曲线、需要成熟的 Git 流程
```

## 二、构建与制品管理

### 4. 封闭构建（Hermetic Build）的重要性？

**答：**

```
封闭构建 = 结果确定性

非封闭构建的问题：
  "在我机器上能跑啊"
  ├── 依赖版本不一致
  ├── 环境变量不同
  ├── 系统库版本不同
  └── 缓存状态不同

封闭构建的要求：
  ┌──────────────────────────────────────┐
  │ 1. 所有依赖版本锁定（go.sum/lock）  │
  │ 2. 构建环境标准化（容器化构建）      │
  │ 3. 不依赖外部状态                    │
  │ 4. 同一 commit 永远构建出同一制品    │
  └──────────────────────────────────────┘

容器化构建示例（多阶段 Dockerfile）:
  FROM golang:1.22 AS builder    # 固定版本
  COPY go.mod go.sum ./         # 锁定依赖
  RUN go mod download
  COPY . .
  RUN CGO_ENABLED=0 go build -o /app

  FROM gcr.io/distroless/static  # 最小基础镜像
  COPY --from=builder /app /app
  ENTRYPOINT ["/app"]
```

### 5. 制品管理的最佳实践？

**答：**

```
制品生命周期：

代码 → 构建 → 制品 → 测试 → 签名 → 发布

制品管理原则：
  1. 不可变（Immutable）
     同一个 Tag 永远对应同一个镜像
     禁止覆盖已有 Tag

  2. 版本化
     使用语义化版本: v1.2.3
     或使用 Git SHA: abc123
     绝不使用 latest（不确定性）

  3. 签名验证
     构建时签名，部署时验证
     防止供应链攻击

  4. 扫描
     每个制品经过安全扫描
     已知漏洞的镜像不允许部署

制品仓库选型：
  ├── Docker Registry (Harbor)
  ├── AWS ECR / GCR / ACR
  ├── Artifactory (通用)
  └── GHCR (GitHub)
```

## 三、面试高频题

### 6. 面试题：如何设计一个 CI/CD 流水线？

**答：**

```yaml
# GitHub Actions CI/CD 流水线
name: CI/CD Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Unit Tests
        run: go test ./...
      - name: Lint
        run: golangci-lint run
      - name: Security Scan
        run: gosec ./...

  build:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - name: Build Image
        run: docker build -t app:${{ github.sha }} .
      - name: Push to Registry
        run: docker push registry/app:${{ github.sha }}

  deploy-canary:
    needs: build
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - name: Deploy Canary (5%)
        run: kubectl set image deploy/app app=registry/app:${{ github.sha }}
      - name: Wait & Verify
        run: ./scripts/canary-check.sh --duration=10m

  deploy-production:
    needs: deploy-canary
    runs-on: ubuntu-latest
    steps:
      - name: Full Rollout
        run: kubectl rollout status deploy/app
```

### 7. 面试题：发布失败如何处理？

**答：**

```
发布失败处理决策树：

发布异常检测
  │
  ├── 金丝雀阶段发现 → 自动回滚金丝雀
  │   └── 影响范围: <5% 用户, 恢复: 秒级
  │
  ├── 灰度阶段发现 → 暂停灰度 + 评估
  │   ├── 严重 → 回滚
  │   └── 轻微 → 修复后继续
  │
  └── 全量后发现 → 全量回滚
      └── 回滚操作:
          kubectl rollout undo deploy/app
          预期恢复: < 5 分钟

回滚后流程：
  1. 确认恢复
  2. 分析失败原因
  3. 修复 + 加测试
  4. 重新走发布流程
```
