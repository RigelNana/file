## 一、金丝雀分析

### 1. 金丝雀分析（Canary Analysis）的核心是什么？

**答：**

```
金丝雀分析 = 用统计方法对比新旧版本指标

核心流程：
  部署金丝雀 → 收集指标 → 统计对比 → 判定结果

分析维度：
  ├── 成功率：新版本 ≥ 旧版本
  ├── 延迟分布：P50/P99 不退化
  ├── 资源效率：CPU/内存不异常增长
  ├── 业务指标：转化率/点击率不下降
  └── 错误模式：无新类型错误

判定结果：
  ┌──────────────────────────────────┐
  │ Pass:      所有指标通过 → 继续   │
  │ Marginal:  部分指标边界 → 人工判│
  │ Fail:      关键指标失败 → 回滚   │
  └──────────────────────────────────┘
```

### 2. Kayenta 自动金丝雀分析如何工作？

**答：**

```
Kayenta (Netflix/Google 开源):

架构：
  金丝雀 Pod ──→ Prometheus ──→ Kayenta ──→ 评分
  基线 Pod   ──→ Prometheus ──↗

分析过程：
  1. 定义 Canary Config（指标+阈值）
  2. 并行收集金丝雀和基线的指标
  3. 对每个指标做 Mann-Whitney U 检验
  4. 综合评分（0-100）
  5. 评分 > 阈值 → Pass

评分示例：
  指标            基线    金丝雀   结论
  error_rate     0.1%    0.12%   Pass (差异不显著)
  p99_latency    120ms   180ms   Fail (差异显著)
  cpu_usage      45%     42%     Pass
  gc_pause       5ms     5ms     Pass
  
  综合评分: 75/100
  阈值: 80
  结果: Marginal → 人工 Review
```

### 3. Flagger 如何实现自动化金丝雀？

**答：**

```yaml
# Flagger Canary 资源
apiVersion: flagger.app/v1beta1
kind: Canary
metadata:
  name: payment-api
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: payment-api
  
  # Istio 流量管理
  service:
    port: 8080
    targetPort: 8080
    
  analysis:
    # 分析间隔
    interval: 1m
    # 分析迭代次数
    iterations: 10
    # 失败阈值
    threshold: 2
    # 每次递增权重
    stepWeight: 10
    # 最大权重
    maxWeight: 50
    
    metrics:
      - name: request-success-rate
        thresholdRange:
          min: 99
        interval: 1m
      - name: request-duration
        thresholdRange:
          max: 200
        interval: 1m
    
    webhooks:
      - name: load-test
        url: http://flagger-loadtester/
        metadata:
          cmd: "hey -z 1m -q 10 http://payment-api-canary:8080/"
```

```
Flagger 自动化流程：

1. 检测到新镜像 → 创建金丝雀 Deployment
2. 路由 0% 流量到金丝雀
3. 每分钟递增 10% 流量
4. 每步检查 success-rate > 99% 和 duration < 200ms
5. 连续 2 次失败 → 自动回滚
6. 10 次迭代全部通过 → 切换到新版本
7. 清理金丝雀 Pod
```

## 二、高级金丝雀技术

### 4. 基于 SLO 的金丝雀分析如何实现？

**答：**

```
基于 SLO 的金丝雀判定：

传统方式：对比新旧版本指标
SLO 方式：检查金丝雀是否满足 SLO 目标

SLO 金丝雀规则：
  IF 金丝雀的 error_budget_burn_rate > 1.0:
    FAIL → 立即回滚
  IF 金丝雀的 error_budget_burn_rate > 0.5:
    WARN → 暂停放量，等待更多数据
  ELSE:
    PASS → 继续放量
```

```promql
# 金丝雀 SLO 燃烧率
(
  sum(rate(http_requests_total{
    deployment="payment-api-canary",
    code=~"5.."
  }[5m]))
  /
  sum(rate(http_requests_total{
    deployment="payment-api-canary"
  }[5m]))
) / 0.001  # SLO 目标 99.9%
```

### 5. 金丝雀发布中如何处理有状态服务？

**答：**

```
有状态服务金丝雀的挑战：
  ├── 数据格式变更可能不兼容
  ├── 新旧版本可能读写同一存储
  └── 回滚可能导致数据问题

解决策略：

1. 数据兼容性
   新版本必须能读旧数据
   旧版本必须能读新版本写的数据
   → 双向兼容

2. Feature Flag 控制写入
   新写入格式用 Feature Flag 控制
   金丝雀只读新格式，不写
   全量发布后才开启新格式写入

3. Shadow Traffic
   金丝雀接收影子流量（只读不写）
   对比金丝雀处理结果和线上结果
   验证逻辑正确性
```

## 三、面试高频题

### 6. 面试题：金丝雀发布的流量管理如何实现？

**答：**

```
流量分流技术：

方式一：K8s Service 权重（Istio）
  VirtualService:
    route:
      - destination: v1 (weight: 95)
      - destination: v2 (weight: 5)
  精度：1% 级别

方式二：Nginx Ingress 权重
  canary-weight: "5"
  精度：1% 级别

方式三：应用层分流
  请求头/Cookie 标记
  API Gateway 路由
  精度：用户/请求级别

方式四：DNS 权重
  权重 DNS 记录
  精度：较粗（不推荐）

推荐组合：
  Service Mesh (Istio) + Argo Rollouts
  → 最精细的流量控制 + 自动化分析
```

### 7. 面试题：金丝雀的样本量不够怎么办？

**答：**

```
低流量服务的金丝雀策略：

挑战：
  如果 QPS 很低（<10），
  金丝雀 5% 流量 = 0.5 QPS
  → 很难收集足够统计样本

解决方案：

1. 延长观察时间
   正常: 10 分钟/步
   低流量: 1-4 小时/步

2. 提高金丝雀比例
   正常: 5% 起步
   低流量: 20-50% 起步

3. 合成流量辅助
   发送额外测试流量到金丝雀
   验证功能正确性

4. 预发布环境验证
   在 Staging 环境做充分测试
   金丝雀主要验证生产环境差异

5. Feature Flag 替代
   用 Feature Flag 按用户控制
   更精细的控制粒度
```
