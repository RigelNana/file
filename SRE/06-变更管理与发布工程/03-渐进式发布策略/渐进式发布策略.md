# 渐进式发布策略八股文

---

## 一、渐进式发布概述

### 1. 什么是渐进式发布？有哪些策略？

**答：**

| 策略 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **金丝雀发布** | 小比例流量验证 | 风险最低、灵活 | 实现复杂 |
| **蓝绿部署** | 两套环境切换 | 回滚秒级 | 资源 2x |
| **滚动更新** | 逐个替换 Pod | K8s 原生 | 回滚较慢 |
| **A/B 测试** | 按用户分组 | 数据驱动 | 需要分流设施 |
| **灰度发布** | 按地域/比例放量 | 控制精细 | 需要灰度平台 |

```
渐进式发布全景：

安全性从高到低：
  金丝雀(1%) → 灰度(10%) → 灰度(50%) → 全量(100%)
  
  每一步都需要：
  ├── 自动健康检查
  ├── SLO 指标对比（新版 vs 旧版）
  ├── 异常自动回滚
  └── 观察窗口（至少 10 分钟）
```

### 2. 金丝雀发布的详细流程？

**答：**

```
金丝雀发布阶段：

Phase 1: 部署金丝雀（1-5% 流量）
  ├── 部署一个新版本 Pod
  ├── 路由少量流量到新 Pod
  ├── 观察 10-30 分钟
  └── 对比指标：新版 vs 旧版
       ├── 错误率
       ├── P99 延迟
       ├── 资源使用
       └── 业务指标（转化率等）

Phase 2: 扩大范围（10-25%）
  ├── 通过金丝雀检查
  ├── 增加新版本 Pod 数量
  ├── 路由更多流量
  └── 继续观察 10-30 分钟

Phase 3: 半量发布（50%）
  ├── 进一步验证
  ├── 关注长尾问题
  └── 确认无回归

Phase 4: 全量发布（100%）
  ├── 完成所有 Pod 替换
  ├── 移除旧版本
  └── 保持旧版本镜像可回滚

自动化判断标准：
  通过条件：
    error_rate(new) ≤ error_rate(old) × 1.1
    p99_latency(new) ≤ p99_latency(old) × 1.2
    
  回滚条件：
    error_rate(new) > error_rate(old) × 2
    p99_latency(new) > p99_latency(old) × 3
```

### 3. 如何在 K8s 中实现金丝雀发布？

**答：**

```yaml
# 方式一：Argo Rollouts
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: payment-api
spec:
  replicas: 10
  strategy:
    canary:
      steps:
        - setWeight: 5          # 5% 流量
        - pause: { duration: 10m }
        - setWeight: 20         # 20% 流量
        - pause: { duration: 10m }
        - setWeight: 50         # 50% 流量
        - pause: { duration: 15m }
        - setWeight: 100        # 全量
      canaryMetadata:
        labels:
          role: canary
      analysis:
        templates:
          - templateName: success-rate
        startingStep: 1
        args:
          - name: service-name
            value: payment-api

---
# 金丝雀分析模板
apiVersion: argoproj.io/v1alpha1
kind: AnalysisTemplate
metadata:
  name: success-rate
spec:
  metrics:
    - name: success-rate
      interval: 1m
      successCondition: result[0] >= 0.99
      provider:
        prometheus:
          address: http://prometheus:9090
          query: |
            sum(rate(http_requests_total{
              service="{{args.service-name}}",
              code!~"5.."
            }[5m]))
            /
            sum(rate(http_requests_total{
              service="{{args.service-name}}"
            }[5m]))
```

## 二、灰度发布设计

### 4. 灰度发布有哪些维度？

**答：**

```
灰度维度：

1. 流量比例
   5% → 10% → 25% → 50% → 100%

2. 用户维度
   ├── 内部员工先体验
   ├── 白名单用户
   ├── 新用户优先
   └── 按用户 ID 哈希

3. 地域维度
   ├── 先发布到流量最小的地域
   ├── 单城市/单可用区
   └── 逐步扩展

4. 设备/平台维度
   ├── 先 Android 后 iOS
   ├── 先 Web 后 App
   └── 按设备类型

组合策略示例：
  Phase 1: 内部员工全量（1 天）
  Phase 2: 北京 5% 用户（1 天）
  Phase 3: 北京 100% 用户（2 天）
  Phase 4: 全国 10% 用户（2 天）
  Phase 5: 全国 100%
```

### 5. 灰度发布中如何做指标对比？

**答：**

```
A/B 比较方法：

1. 基线对比组
   Control Group: 旧版本（基线）
   Treatment Group: 新版本（金丝雀）

2. 关键指标对比
   ┌──────────┬──────────┬──────────┬──────────┐
   │ 指标     │ 旧版本   │ 新版本   │ 判断     │
   ├──────────┼──────────┼──────────┼──────────┤
   │ 错误率   │ 0.05%    │ 0.04%    │ ✅ 更好  │
   │ P99 延迟 │ 120ms    │ 180ms    │ ⚠️ 退化  │
   │ CPU 使用 │ 45%      │ 40%      │ ✅ 更好  │
   │ 内存使用 │ 1.2GB    │ 1.8GB    │ ⚠️ 退化  │
   └──────────┴──────────┴──────────┴──────────┘

3. 统计显著性
   样本量足够大时（通常需要 1000+ 请求），
   使用统计方法判断差异是否显著

4. 自动化分析工具
   ├── Argo Rollouts Analysis
   ├── Kayenta（Netflix/Google）
   ├── Flagger（Weaveworks）
   └── 自定义 Prometheus 查询
```

## 三、面试高频题

### 6. 面试题：金丝雀发布发现问题后如何处理？

**答：**

```
金丝雀失败处理流程：

1. 自动检测到问题
   金丝雀分析器发现 error_rate 升高

2. 自动暂停发布
   停止进一步放量

3. 判断严重程度
   ├── 严重（错误率 >2x）→ 自动回滚
   └── 轻微（延迟小幅升高）→ 人工判断

4. 回滚操作
   kubectl argo rollouts abort payment-api
   → 流量 100% 回到旧版本
   → 金丝雀 Pod 被清理

5. 事后分析
   ├── 排查根因
   ├── 修复问题
   ├── 添加回归测试
   └── 重新走发布流程
```

### 7. 面试题：金丝雀发布和蓝绿部署如何选择？

**答：**

| 维度 | 金丝雀 | 蓝绿 |
|------|--------|------|
| **资源** | 新增少量 Pod | 完整两套环境 |
| **影响** | 小比例用户验证 | 瞬间全部切换 |
| **回滚速度** | 快（减少流量即可） | 极快（切换路由） |
| **验证深度** | 真实流量逐步验证 | 切换前需充分测试 |
| **适用** | 需要真实流量验证 | 有状态/数据库迁移 |
| **复杂度** | 高（需要流量分流） | 中（双环境管理） |

```
选择建议：
  ├── 无状态微服务 → 金丝雀
  ├── 有状态/DB 迁移 → 蓝绿 + 金丝雀组合
  ├── 资源受限 → 金丝雀（资源消耗少）
  └── 合规要求 → 蓝绿（环境可审计）
```
