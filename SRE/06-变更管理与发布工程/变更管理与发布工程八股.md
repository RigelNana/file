# 变更管理与发布工程八股文

---

## 一、变更管理

### 1. 为什么变更管理对 SRE 如此重要？

**答：** Google 的统计数据显示，**约 70% 的生产事故是由变更引起的**。变更管理是减少故障最有效的手段之一。

变更类型：
- **代码变更**：新功能发布、Bug 修复
- **配置变更**：参数调整、Feature Flag 切换
- **基础设施变更**：扩缩容、网络变更、数据库迁移
- **依赖变更**：第三方服务升级、库版本更新

### 2. 渐进式发布策略有哪些？

**答：**

| 策略 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **滚动更新** | 分批替换旧实例 | 简单、零停机 | 回滚慢、新旧版本共存 |
| **蓝绿部署** | 两套环境切换流量 | 回滚快（秒级） | 需要双倍资源 |
| **金丝雀发布** | 先给小比例流量测试 | 风险最小、可控 | 流程复杂、需完善监控 |
| **A/B 测试** | 按用户分组对比版本 | 可量化效果差异 | 需要实验平台支持 |
| **影子流量** | 复制真实流量到新版本 | 零用户影响 | 需处理副作用（写操作） |

### 3. 金丝雀发布的流程和关键指标是什么？

**答：**

```
1% 流量 → 监控 SLI → 正常 → 5% → 监控 → 正常 → 25% → ... → 100%
             ↓异常
         自动回滚
```

**金丝雀发布关键指标：**
- **错误率**：新版本的错误率是否高于旧版本
- **延迟**：P50/P99 延迟是否恶化
- **资源消耗**：CPU/内存是否异常升高
- **业务指标**：转化率、订单量等是否下降

### 4. 什么是 Feature Flag？在 SRE 中的作用？

**答：** Feature Flag（特性标志/功能开关）将**代码部署和功能发布解耦**。

SRE 视角的价值：
- **快速回滚**：关闭 Flag 即可禁用新功能，无需重新部署
- **渐进式发布**：逐步开放给部分用户
- **紧急降级**：在高负载时关闭非核心功能
- **A/B 实验**：支持按用户分组实验
