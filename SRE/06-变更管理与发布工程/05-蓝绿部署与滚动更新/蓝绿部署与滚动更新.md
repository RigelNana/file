# 蓝绿部署与滚动更新八股文

---

## 一、蓝绿部署

### 1. 蓝绿部署的原理和流程是什么？

**答：**

```
蓝绿部署 = 维护两套完全相同的环境

流程：
  ┌──────────────────────────────────┐
  │  阶段1: 蓝环境（当前生产版本 v1）  │
  │  用户 ──→ LB ──→ 蓝 (v1)        │
  │                   绿 (空闲)       │
  ├──────────────────────────────────┤
  │  阶段2: 部署到绿环境             │
  │  用户 ──→ LB ──→ 蓝 (v1)        │
  │                   绿 (v2 部署中)  │
  ├──────────────────────────────────┤
  │  阶段3: 验证绿环境               │
  │  QA ──→ 绿 (v2) ✓               │
  ├──────────────────────────────────┤
  │  阶段4: 切换流量                 │
  │  用户 ──→ LB ──→ 绿 (v2)        │
  │                   蓝 (v1 待机)    │
  ├──────────────────────────────────┤
  │  阶段5: 观察，确认正常后释放蓝    │
  └──────────────────────────────────┘

回滚：只需切换 LB 指向蓝环境
速度：秒级回滚（流量切换）
```

### 2. 蓝绿部署的优缺点是什么？

**答：**

| 方面 | 优点 | 缺点 |
|------|------|------|
| 回滚速度 | 秒级切换 | — |
| 资源成本 | — | 需要双倍资源 |
| 测试 | 可在绿环境充分测试 | — |
| 复杂度 | 实现简单 | — |
| 数据库 | — | Schema 变更需兼容 |
| 切换风险 | — | 一次切换全部流量 |
| 验证 | 生产环境真实验证 | — |

### 3. K8s 中如何实现蓝绿部署？

**答：**

```yaml
# 蓝环境 Deployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-blue
  labels:
    app: myapp
    version: blue
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
      version: blue
  template:
    metadata:
      labels:
        app: myapp
        version: blue
    spec:
      containers:
        - name: myapp
          image: myapp:v1.0
---
# 绿环境 Deployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-green
  labels:
    app: myapp
    version: green
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
      version: green
  template:
    metadata:
      labels:
        app: myapp
        version: green
    spec:
      containers:
        - name: myapp
          image: myapp:v2.0
---
# Service 切换
apiVersion: v1
kind: Service
metadata:
  name: myapp
spec:
  selector:
    app: myapp
    version: green  # 切换这里：blue ↔ green
  ports:
    - port: 80
      targetPort: 8080
```

```bash
# 切换命令
kubectl patch service myapp \
  -p '{"spec":{"selector":{"version":"green"}}}'

# 回滚命令
kubectl patch service myapp \
  -p '{"spec":{"selector":{"version":"blue"}}}'
```

## 二、滚动更新

### 4. K8s 滚动更新的机制和参数？

**答：**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 10
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 25%        # 最多多创建 25% 的 Pod
      maxUnavailable: 25%   # 最多 25% 不可用
  template:
    spec:
      containers:
        - name: myapp
          image: myapp:v2.0
          readinessProbe:
            httpGet:
              path: /health
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 5
          livenessProbe:
            httpGet:
              path: /health
              port: 8080
            initialDelaySeconds: 15
            periodSeconds: 10
```

```
滚动更新过程（10 个 Pod，maxSurge=2, maxUnavailable=2）:

阶段    旧v1    新v2    可用Pod
初始     10      0       10
步骤1    8       2       8+2=10  (终止2旧, 创建2新)
步骤2    6       4       6+4=10
步骤3    4       6       4+6=10
步骤4    2       8       2+8=10
步骤5    0       10      10
```

### 5. 滚动更新有哪些注意事项？

**答：**

```
关键配置要点：

1. Readiness Probe 必须正确
   - 只有 Ready 的 Pod 才接收流量
   - Probe 应验证真正的服务就绪状态
   - 初始延迟要给足启动时间

2. minReadySeconds
   - Pod Ready 后再等待 N 秒才认为可用
   - 避免 Pod 刚启动就承接大量流量
   spec:
     minReadySeconds: 30

3. 优雅终止
   - terminationGracePeriodSeconds 足够大
   - 收到 SIGTERM 先停止接收新请求
   - 处理完已有请求再退出
   - preStop hook 等待连接排干

4. PodDisruptionBudget
   apiVersion: policy/v1
   kind: PodDisruptionBudget
   spec:
     minAvailable: 80%
     selector:
       matchLabels:
         app: myapp
```

## 三、策略对比与面试

### 6. 蓝绿 vs 金丝雀 vs 滚动更新如何选择？

**答：**

| 维度 | 蓝绿部署 | 金丝雀发布 | 滚动更新 |
|------|---------|-----------|---------|
| 回滚速度 | 秒级 | 分钟级 | 分钟级 |
| 资源开销 | 双倍 | 少量额外 | 无额外 |
| 风险范围 | 全量切换 | 渐进放量 | 渐进替换 |
| 验证能力 | 上线前验证 | 线上流量验证 | 基本健康检查 |
| 实现复杂度 | 低 | 高 | 极低（K8s原生） |
| 数据库变更 | 需兼容 | 需兼容 | 需兼容 |
| 适用场景 | 关键服务、快速切换 | 高流量、需要精细验证 | 通用、低风险变更 |

```
选择决策树：

变更风险高？
  ├── 是 → 流量大（>1000 QPS）？
  │     ├── 是 → 金丝雀发布
  │     └── 否 → 蓝绿部署
  └── 否 → 滚动更新
```

### 7. 面试题：数据库 Schema 变更时如何安全发布？

**答：**

```
数据库兼容性发布（Expand-Contract 模式）：

阶段1: Expand（扩展）
  - 添加新列，NOT NULL 带默认值
  - 旧版本代码不受影响
  
阶段2: Migrate（迁移）
  - 新代码同时写新旧列
  - 后台任务迁移历史数据
  
阶段3: Contract（收缩）
  - 确认所有数据已迁移
  - 新代码只读写新列
  
阶段4: Cleanup（清理）
  - 删除旧列（可选，观察一段时间）

示例（重命名列 email → contact_email）：
  v1: 读写 email
  v2: 写 email + contact_email, 读 contact_email
  v3: 只写 contact_email, 读 contact_email
  v4: 删除 email 列

原则：
  - 永远向后兼容
  - 一次变更只做一步
  - 每步都可以安全回滚
```
