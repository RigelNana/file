## 一、发布工程体系

### 1. 发布工程的成熟度模型是什么？

**答：**

```
发布工程成熟度五级模型：

Level 1: 手动（Ad-hoc）
  - 手动 SSH 部署
  - 无版本管理
  - 靠人记住步骤

Level 2: 脚本化
  - 部署脚本（Shell/Ansible）
  - 基本的版本记录
  - Runbook 文档

Level 3: 自动化
  - CI/CD Pipeline
  - 自动测试 + 自动部署
  - 一键回滚

Level 4: 可度量
  - DORA 指标追踪
  - 变更失败率 < 15%
  - 部署频率：每天多次

Level 5: 自优化
  - 自动金丝雀分析
  - SLO 驱动的发布决策
  - 混沌工程验证发布
  - 持续优化 Pipeline

大多数团队在 Level 2-3
目标：至少达到 Level 4
```

### 2. 发布工程的核心原则有哪些？

**答：**

```
八大核心原则：

1. 小批量频繁发布
   ✗ 月度大版本
   ✓ 每天多次小变更

2. 部署 ≠ 发布
   部署 = 代码到服务器
   发布 = 功能对用户可见
   → Feature Flag 分离两者

3. 所有变更可追溯
   → Git 历史 + 审计日志
   → 谁在什么时间改了什么

4. 所有发布可回滚
   → 每个变更都有回滚方案
   → 回滚比修复优先

5. 环境一致性
   → Dev = Staging = Prod
   → 容器 + IaC 保证一致

6. 不可变制品
   → 构建一次，部署多次
   → 不在部署后修改制品

7. 自动化优先
   → 人工步骤 = 风险
   → 能自动化的必须自动化

8. 安全左移
   → 安全检查在 Pipeline 中
   → 不是发布后才补
```

## 二、实战经验

### 4. 如何处理紧急发布（Hotfix）？

**答：**

```
紧急发布流程：

触发条件：
  - P0/P1 生产事故
  - 安全漏洞修复
  - 数据丢失风险

紧急发布 vs 正常发布：
┌──────────┬────────────────┬─────────────┐
│ 环节      │ 正常发布        │ 紧急发布     │
├──────────┼────────────────┼─────────────┤
│ Code Rev  │ 2人 Review     │ 1人 Review  │
│ 测试      │ 完整测试套件    │ 关键路径测试 │
│ 审批      │ 标准审批       │ IC 直接审批  │
│ 环境      │ Dev→Stg→Prod  │ Stg→Prod    │
│ 部署方式  │ 金丝雀         │ 快速全量     │
│ 监控      │ 标准           │ 增强监控     │
└──────────┴────────────────┴─────────────┘

紧急发布约束：
  1. 修改范围最小化
  2. 发布后必须补齐完整测试
  3. 24小时内补 Postmortem
  4. 紧急发布比例 < 5%（月度）
```

### 5. 如何度量和改进发布效率？

**答：**

```
关键度量指标：

发布速度：
  - 部署频率：每天/每周部署次数
  - Lead Time：代码提交到上线的时间
  - Pipeline 执行时间

发布质量：
  - 变更失败率 = 导致故障的发布 / 总发布数
  - 回滚率 = 回滚次数 / 总发布数
  - 发布后事件数

发布效率：
  - Pipeline 成功率 = 成功 / 总执行数
  - Flaky Test 率
  - 等待审批时间

改进方向：
  指标差           │ 改进措施
  ─────────────────┼──────────────────
  Lead Time 长     │ 并行化 Pipeline
  失败率高         │ 加强测试和金丝雀
  回滚率高         │ 改进预发布验证
  Pipeline 慢      │ 构建缓存 + 并行
  审批时间长       │ 自动化审批规则
```

## 三、面试

### 6. 面试题：如何从零搭建一套发布体系？

**答：**

```
分阶段搭建发布体系：

Phase 1（第1-2周）: 基本自动化
  ✓ Dockerfile + 镜像构建
  ✓ 基本 CI Pipeline（构建+测试）
  ✓ K8s Deployment + 滚动更新
  ✓ 基本 Readiness/Liveness Probe

Phase 2（第3-4周）: 安全和质量
  ✓ 代码扫描（SAST/SCA）
  ✓ 镜像扫描
  ✓ 集成测试
  ✓ Staging 环境

Phase 3（第5-8周）: 高级发布
  ✓ GitOps（ArgoCD）
  ✓ 金丝雀发布（Argo Rollouts）
  ✓ Feature Flag 平台
  ✓ 变更管理流程

Phase 4（持续）: 优化
  ✓ 自动金丝雀分析
  ✓ SLO 驱动发布决策
  ✓ DORA 指标追踪
  ✓ 混沌工程验证
```

### 7. 面试题：多团队共享微服务时如何协调发布？

**答：**

```
多团队发布协调策略：

1. 服务契约
   - 所有 API 版本化（v1/v2）
   - 契约测试（Pact）
   - Breaking Change 需提前通知

2. 独立部署
   - 每个服务独立 Pipeline
   - 团队可以自主发布
   - 不需要协调发布时间

3. 兼容性保证
   - 新版本兼容旧 API 至少 2 个版本
   - 消费者驱动的契约测试
   - API 废弃需提前 N 个迭代通知

4. 共享基础设施
   - 共享 CI/CD 模板
   - 共享 Helm Chart 基础模板
   - 统一的发布标准和流程

5. 沟通机制
   - 发布日历（Google Calendar）
   - Slack 发布通知频道
   - 变更影响分析自动化

原则：
  最大化团队自主性
  最小化跨团队协调
  用技术手段替代人工沟通
```
