# SLO 设定方法论八股文

---

## 一、SLO 基础概念

### 1. 什么是 SLO（Service Level Objective）？

**答：** SLO 是基于 SLI 设定的**目标值**或**目标范围**，定义了服务可靠性的预期水平。

```
SLO = SLI ≥ 目标值

示例：
SLO(可用性) = 可用性 SLI ≥ 99.9%（30天窗口）
SLO(延迟)   = 延迟 SLI   ≥ 95% 请求 < 200ms（30天窗口）
```

**SLO 三要素**：

| 要素 | 描述 | 示例 |
|------|------|------|
| **SLI** | 要度量什么 | HTTP 请求成功率 |
| **目标值** | 要达到多少 | 99.9% |
| **时间窗口** | 在什么时间范围内 | 30 天滚动窗口 |

### 2. SLO 和 SLA 有什么区别？

**答：**

| 对比项 | SLO | SLA |
|--------|-----|-----|
| **性质** | 内部目标 | 商业合同 |
| **后果** | 触发内部策略 | 触发赔偿/违约 |
| **严格度** | 更严格 | 更宽松 |
| **可见性** | 工程团队 | 客户可见 |
| **制定者** | SRE + 开发 | 产品 + 法务 |

```
典型关系：
SLO > SLA（SLO 比 SLA 更严格）

SLA = 99.9%   ← 对客户承诺
SLO = 99.95%  ← 内部目标（留缓冲）
```

## 二、SLO 设定流程

### 3. 如何从零开始设定 SLO？

**答：**

**六步法**：

```
第1步：识别关键用户旅程
          │
          ▼
第2步：为每个旅程选择 SLI
          │
          ▼
第3步：收集历史数据（2-4周）
          │
          ▼
第4步：基于历史确定初始 SLO
          │
          ▼
第5步：试运行和调整（4-6周）
          │
          ▼
第6步：正式发布并建立告警
```

### 4. 初始 SLO 目标值应该怎么定？

**答：**

**方法一：基于历史数据**

```
收集过去 30 天数据：
  可用性 SLI 实际表现 = 99.97%

初始 SLO 设定：
  不要设为 99.97%（历史最好表现）
  建议设为 99.9%（低于实际，留出空间）

原则：SLO = 历史表现 - 合理缓冲
```

**方法二：基于业务需求**

| 业务优先级 | SLO 范围 | 适用场景 |
|------------|----------|----------|
| 核心变现 | 99.95%~99.99% | 支付、订单 |
| 核心体验 | 99.9%~99.95% | 搜索、推荐 |
| 重要功能 | 99.5%~99.9% | 个人设置、历史记录 |
| 辅助功能 | 99%~99.5% | 日志查询、数据分析 |

### 5. 什么是 SLO 时间窗口？滚动窗口和日历窗口有什么区别？

**答：**

| 窗口类型 | 定义 | 优点 | 缺点 |
|----------|------|------|------|
| **滚动窗口** | 过去 N 天 | 平滑、无边界效应 | 不与业务周期对齐 |
| **日历窗口** | 本月/本季度 | 与业务报告对齐 | 月初"满血复活"效应 |

```
滚动窗口（30天）：
... ─── Day1 ─── Day2 ─── ... ─── Day30 ───→
     └───────── 窗口 ──────────┘
         每天向右滑动一天

日历窗口（月）：
|─── 1月 ───|─── 2月 ───|─── 3月 ───|
  月末预算紧    月初重置     月末紧
```

**推荐**：滚动窗口用于工程决策，日历窗口用于业务报告。

## 三、SLO 设定的常见陷阱

### 6. 设定 SLO 有哪些常见错误？

**答：**

| 错误 | 描述 | 正确做法 |
|------|------|----------|
| **SLO 过高** | 设为 99.999% 导致无法创新 | 从实际需求出发 |
| **SLO 过低** | 设为 99% 导致用户不满 | 参考竞争对手水平 |
| **指标太多** | 每个服务 20 个 SLO | 精选 2-5 个关键 SLI |
| **指标不反映用户** | 用 CPU 利用率做 SLI | 选择用户感知指标 |
| **固定不变** | 设定后从不调整 | 定期回顾和调整 |
| **没有 Action** | 有 SLO 但违约无后果 | 建立错误预算策略 |

### 7. 为什么不应该追求 100% SLO？

**答：**

```
追求 100% 的问题：

1. 成本爆炸
   99.9%  → 1名 SRE 就能维护
   99.99% → 需要3-5名 SRE + 冗余架构
   100%   → 理论上需要无限投入

2. 创新停滞
   100% uptime = 0% 变更空间
   任何变更都可能导致不可用 → 不敢变更

3. 用户感知上限
   用户网络：99.5%
   + 你的服务：100%
   = 用户实际体验：≤ 99.5%
   你的 100% 对用户没有额外价值
```

## 四、多层 SLO 设计

### 8. 什么是多层 SLO？如何设计？

**答：** 对同一个 SLI 设定不同严格度的 SLO。

```
示例：API 延迟 SLO

SLO 层级 1（宽松）：90% 请求 < 100ms
SLO 层级 2（标准）：99% 请求 < 500ms
SLO 层级 3（严格）：99.9% 请求 < 2000ms

含义：
- 大部分请求应该很快（< 100ms）
- 几乎所有请求不应太慢（< 500ms）
- 极端情况也不能太离谱（< 2s）
```

### 9. 如何为微服务架构设计端到端 SLO？

**答：**

```
用户请求路径：
用户 → API GW → 服务A → 服务B → DB
                          ↓
                        服务C

每个环节的可用性 SLO：
  API GW: 99.99%
  服务A:  99.95%
  服务B:  99.95%
  服务C:  99.9%
  DB:     99.99%

端到端可用性 ≈ 0.9999 × 0.9995 × 0.9995 × 0.999 × 0.9999
             ≈ 99.78%
```

**关键点**：
- 链路越长，端到端可靠性越低
- 需要在关键路径上提高单点 SLO
- 或者引入冗余和降级策略

## 五、SLO 的回顾与调整

### 10. SLO 应该多久回顾一次？如何调整？

**答：**

| 回顾频率 | 内容 | 参与人 |
|----------|------|--------|
| **每周** | 检查 SLO 达标状态 | SRE |
| **每月** | 分析趋势、讨论调整 | SRE + Dev |
| **每季度** | 全面评审、确认下季度 SLO | SRE + Dev + PM |

**调整信号**：

```
需要调高 SLO：
  - 连续 3 个月轻松达标（预算剩余 > 80%）
  - 用户反馈质量期望更高
  - 竞争对手提供了更好的服务

需要调低 SLO：
  - 连续违约且改进代价过大
  - 业务确认当前水平可接受
  - 架构限制短期无法突破
```

### 11. 面试题：设计一个电商平台的 SLO 体系

**答：**

| 服务 | SLI | SLO（30天滚动） | 优先级 |
|------|-----|-----------------|--------|
| 商品列表 | 可用性 | ≥ 99.9% | P1 |
| 商品列表 | 延迟 P99 | < 500ms | P1 |
| 订单创建 | 可用性 | ≥ 99.95% | P0 |
| 订单创建 | 延迟 P99 | < 1000ms | P0 |
| 支付 | 可用性 | ≥ 99.99% | P0 |
| 支付 | 正确性 | ≥ 99.999% | P0 |
| 搜索 | 可用性 | ≥ 99.9% | P1 |
| 搜索 | 延迟 P90 | < 200ms | P1 |
| 用户中心 | 可用性 | ≥ 99.5% | P2 |

**设计思路**：
1. 支付和订单最关键——影响营收
2. 搜索和商品列表次之——影响转化
3. 用户中心可以适当放松——影响有限
