## 一、多窗口告警深入

### 1. 多窗口多燃烧率告警的完整参数是如何推导出来的？

**答：** 基于 Google SRE Workbook 的推导逻辑：

**核心约束**：
- 30 天错误预算只允许消耗 100%
- 希望在预算消耗 **2%** 时就通知（Page 级别）
- 希望在预算消耗 **5%** 时用工单跟踪
- 希望在预算消耗 **10%** 时必须关注

**推导过程**：
```
对于 Page 级别：消耗 2% 预算时告警

方案1：1小时窗口
  1小时 = 1/720 月（30天 = 720小时）
  1小时内消耗 2% 预算
  燃烧率 = 2% / (1/720) = 14.4x

方案2：6小时窗口
  6小时 = 6/720 月
  6小时内消耗 5% 预算
  燃烧率 = 5% / (6/720) = 6x
```

**完整参数表**：

| 告警级别 | 长窗口 | 短窗口 | 燃烧率 | 消耗预算% | 耗尽天数 |
|----------|--------|--------|--------|-----------|----------|
| Page | 1h | 5m | 14.4x | 2% | 2.08天 |
| Page | 6h | 30m | 6x | 5% | 5天 |
| Ticket | 1d | 2h | 3x | 10% | 10天 |
| Ticket | 3d | 6h | 1x | 10% | 30天 |

### 2. 短窗口大小是如何确定的？

**答：** 短窗口一般是长窗口的 **1/12 到 1/6**。

```
长窗口 → 短窗口（经验值）：
  1h  → 5m    (1/12)
  6h  → 30m   (1/12)
  1d  → 2h    (1/12)
  3d  → 6h    (1/12)
```

**短窗口的作用**：
- 太短（如 1 分钟）→ 个别请求失败就触发 → 噪音大
- 太长（如 = 长窗口）→ 失去了"当前仍在发生"的验证意义
- 1/12 是经验平衡点

### 3. 多窗口告警的误报率和漏报率如何？

**答：**

```
对比不同告警策略的效果：

策略              误报率    漏报率    检测时间
──────           ──────   ──────   ────────
纯阈值告警        高        低       快
单窗口燃烧率      中        中       中
多窗口燃烧率      低        低       中
```

**实际数据参考（基于 Google 经验）**：
- 多窗口告警误报率降低 **90%+** 相比静态阈值
- 检测时间在 **2-30 分钟**，取决于严重程度
- 重置时间在 **5 分钟-6 小时**

## 二、实际部署配置

### 4. 完整的 Prometheus + Alertmanager 多窗口告警配置是什么？

**答：**

```yaml
# prometheus-rules.yaml
groups:
  - name: slo-recording-rules
    interval: 30s
    rules:
      # 各窗口错误率
      - record: http_errors:burnrate5m
        expr: |
          sum(rate(http_requests_total{code=~"5.."}[5m]))
          / sum(rate(http_requests_total[5m]))
      - record: http_errors:burnrate30m
        expr: |
          sum(rate(http_requests_total{code=~"5.."}[30m]))
          / sum(rate(http_requests_total[30m]))
      - record: http_errors:burnrate1h
        expr: |
          sum(rate(http_requests_total{code=~"5.."}[1h]))
          / sum(rate(http_requests_total[1h]))
      - record: http_errors:burnrate6h
        expr: |
          sum(rate(http_requests_total{code=~"5.."}[6h]))
          / sum(rate(http_requests_total[6h]))
      - record: http_errors:burnrate1d
        expr: |
          sum(rate(http_requests_total{code=~"5.."}[1d]))
          / sum(rate(http_requests_total[1d]))
      - record: http_errors:burnrate3d
        expr: |
          sum(rate(http_requests_total{code=~"5.."}[3d]))
          / sum(rate(http_requests_total[3d]))

  - name: slo-alerts
    rules:
      # P0：快速燃烧（14.4x，1h/5m）
      - alert: SLO_ErrorBudget_FastBurn
        expr: |
          http_errors:burnrate1h > (14.4 * 0.001)
          and
          http_errors:burnrate5m > (14.4 * 0.001)
        for: 2m
        labels:
          severity: critical
          slo: availability
        annotations:
          summary: "SLO 快速消耗：14.4x 燃烧率"
          description: "当前 1h 错误率={{ $value }}，预计 2 天内耗尽预算"

      # P1：中速燃烧（6x，6h/30m）
      - alert: SLO_ErrorBudget_MediumBurn
        expr: |
          http_errors:burnrate6h > (6 * 0.001)
          and
          http_errors:burnrate30m > (6 * 0.001)
        for: 5m
        labels:
          severity: warning
          slo: availability

      # P2：慢速燃烧（3x，1d/2h）
      - alert: SLO_ErrorBudget_SlowBurn
        expr: |
          http_errors:burnrate1d > (3 * 0.001)
          and
          http_errors:burnrate2h > (3 * 0.001)
        for: 10m
        labels:
          severity: ticket

      # P3：缓慢泄漏（1x，3d/6h）
      - alert: SLO_ErrorBudget_SlowLeak
        expr: |
          http_errors:burnrate3d > (1 * 0.001)
          and
          http_errors:burnrate6h > (1 * 0.001)
        for: 30m
        labels:
          severity: info
```

### 5. Alertmanager 路由如何配合多级告警？

**答：**

```yaml
# alertmanager.yaml
route:
  group_by: ['slo', 'service']
  group_wait: 30s
  group_interval: 5m
  repeat_interval: 4h
  receiver: 'default'
  routes:
    - match:
        severity: critical
      receiver: 'pagerduty-critical'
      repeat_interval: 15m
    - match:
        severity: warning
      receiver: 'slack-oncall'
      repeat_interval: 1h
    - match:
        severity: ticket
      receiver: 'jira-ticket'
      repeat_interval: 24h

receivers:
  - name: 'pagerduty-critical'
    pagerduty_configs:
      - service_key: '<key>'
  - name: 'slack-oncall'
    slack_configs:
      - channel: '#sre-alerts'
  - name: 'jira-ticket'
    webhook_configs:
      - url: 'http://jira-webhook/create'
```

## 三、告警调优与验证

### 6. 如何验证多窗口告警是否正确？

**答：**

**验证方法**：

| 方法 | 描述 |
|------|------|
| **历史回放** | 用历史数据回放，检查告警是否在预期时间触发 |
| **注入测试** | 人工注入错误，验证告警触发和恢复 |
| **Amixr 模型** | 数学模拟不同错误模式下的告警行为 |

```
验证场景清单：
1. 瞬间错误尖峰（1分钟 100% 错误后恢复）
   → 期望：不触发 Page
   
2. 完全宕机（持续 5 分钟 100% 错误）
   → 期望：2 分钟内触发 Critical

3. 缓慢泄漏（持续 0.2% 错误率，刚好 2x 燃烧）
   → 期望：24 小时窗口触发 Ticket

4. 已恢复的事故（10 分钟 50% 错误后恢复）
   → 期望：短窗口恢复后告警消除
```

### 7. 告警抑制和去重如何配置？

**答：**

```yaml
# 同一个 SLO 的不同级别告警：高级别抑制低级别
inhibit_rules:
  - source_match:
      severity: 'critical'
    target_match:
      severity: 'warning'
    equal: ['slo', 'service']
  
  - source_match:
      severity: 'warning'
    target_match:
      severity: 'ticket'
    equal: ['slo', 'service']
```

## 四、面试高频题

### 8. 面试题：解释为什么多窗口比单窗口好

**答：**

```
单窗口（只看 1 小时）：
  场景：00:00 发生 5 分钟故障，00:05 恢复
  00:10 → 1h 内有高错误，触发告警 → 但已恢复！ → 误报
  01:05 → 过了整整 1 小时后告警才消除 → 重置太慢

多窗口（1小时 + 5分钟）：
  00:10 → 1h 窗口异常，但 5m 窗口已正常
  → AND 条件不满足 → 不告警 ✓
  → 几分钟内就确认恢复 → 快速重置 ✓
```

### 9. 面试题：如何处理多窗口告警告不出来的情况？

**答：** 多窗口告警可能在边界情况下遗漏，需要补充机制：

1. **错误预算消耗报告**：即使不触发告警，每日报告预算状态
2. **预算阈值告警**：当月度预算消耗 > 50% 时，无论燃烧率多少都告警
3. **合成监控兜底**：外部拨测作为最后防线

```promql
# 兜底告警：总预算消耗超过阈值
- alert: SLO_ErrorBudget_HighConsumption
  expr: |
    1 - (
      (1 - http_errors:ratio_30d)
      / (1 - 0.999)
    ) > 0.8
  labels:
    severity: warning
  annotations:
    summary: "30 天错误预算已消耗 80%"
```
