# SLI 定义与选择八股文

---

## 一、SLI 基础概念

### 1. 什么是 SLI（Service Level Indicator）？

**答：** SLI（服务等级指标）是对服务行为某一属性的**定量度量**，用来衡量用户体验的好坏。

**核心特征**：
- 是一个**比率**（0% ~ 100%），表示"好事件 / 总事件"
- 直接反映**用户感知**
- 是 SLO 和 SLA 的基础

```
SLI = 好事件数 / 有效事件总数 × 100%

示例：
SLI(可用性) = 成功请求数 / 总请求数 × 100%
SLI(延迟)   = P99 < 300ms 的请求数 / 总请求数 × 100%
```

### 2. SLI 与一般监控指标有什么区别？

**答：**

| 对比项 | 一般指标 | SLI |
|--------|----------|-----|
| **视角** | 系统内部指标 | 用户感知指标 |
| **形式** | 任意（计数、均值等） | 比率（0%~100%） |
| **示例** | CPU 利用率 80% | 请求成功率 99.95% |
| **目的** | 诊断和排查 | 衡量服务质量 |
| **数量** | 可能很多 | 精心选择少数几个 |

```
           ┌─ SLI ──────────────────────┐
           │  用户视角、比率形式、少量精选  │
           └──────────┬─────────────────┘
                      │
    ┌─────────────────┼─────────────────┐
    │                 │                 │
  CPU利用率      请求成功率          QPS
  (内部指标)      (SLI ✓)          (内部指标)
```

### 3. 为什么 SLI 要用比率而非绝对值？

**答：**

1. **可比性**：不同服务的 QPS 差异巨大，但成功率可以直接比较
2. **直觉性**：99.9% 比"每天 86 秒不可用"更容易理解
3. **可组合**：多个 SLI 可以用相同方式聚合
4. **与 SLO 对齐**：SLO 天然是一个阈值（如 ≥ 99.9%）

## 二、SLI 的分类与选择

### 4. 常见的 SLI 类型有哪些？

**答：**

| SLI 类型 | 定义 | 适用场景 |
|----------|------|----------|
| **可用性** | 成功请求 / 总请求 | 所有在线服务 |
| **延迟** | 快速请求 / 总请求 | 用户交互型服务 |
| **吞吐量** | 成功处理量 / 请求量 | 数据管道、批处理 |
| **正确性** | 正确响应 / 总响应 | 数据服务、计算服务 |
| **新鲜度** | 数据新鲜的查询 / 总查询 | 缓存、数据同步 |
| **覆盖率** | 已处理量 / 应处理量 | 批处理、ETL |
| **耐久性** | 成功读取 / 总写入 | 存储服务 |

### 5. 如何为不同类型的服务选择 SLI？

**答：**

**按服务类型选择**：

```
┌─────────────────┬────────────────────────────┐
│ 服务类型         │ 推荐 SLI                    │
├─────────────────┼────────────────────────────┤
│ API 服务         │ 可用性 + 延迟               │
│ Web 前端         │ 延迟 + 可用性               │
│ 数据管道         │ 新鲜度 + 正确性 + 覆盖率     │
│ 存储系统         │ 耐久性 + 可用性 + 延迟       │
│ 消息队列         │ 可用性 + 延迟               │
│ 流处理           │ 吞吐量 + 新鲜度             │
│ 批处理           │ 覆盖率 + 正确性             │
│ CDN              │ 延迟 + 可用性               │
└─────────────────┴────────────────────────────┘
```

### 6. 选择 SLI 有哪些原则？

**答：**

| 原则 | 描述 |
|------|------|
| **用户视角** | 反映用户实际感知，而非内部指标 |
| **可度量** | 能够准确、可靠地采集数据 |
| **少而精** | 每个服务 2-5 个 SLI 足够 |
| **可操作** | 能指导工程决策 |
| **有区分度** | 能区分"好"和"坏"的状态 |

## 三、SLI 的度量方法

### 7. SLI 有哪几种度量方式？各自优缺点是什么？

**答：**

| 度量方式 | 数据来源 | 优点 | 缺点 |
|----------|----------|------|------|
| **服务端日志** | 应用日志、访问日志 | 详细、包含所有请求 | 不反映网络延迟 |
| **服务端指标** | Prometheus 等 | 实时、精确 | 不包含客户端问题 |
| **负载均衡器** | LB 日志/指标 | 接近用户视角 | 不包含 LB 之前的问题 |
| **客户端埋点** | 前端/SDK 上报 | 最接近真实用户体验 | 数据可能不完整 |
| **合成监控** | 拨测探针 | 稳定基线、主动发现 | 不代表真实用户 |

**推荐组合**：
```
最佳实践：多层度量叠加

客户端 SLI ← 最接近用户感知（选做）
     │
负载均衡 SLI ← 推荐作为主要 SLI
     │
服务端 SLI ← 辅助排查
     │
合成监控 ← 基准线
```

### 8. 如何处理 SLI 中的"有效事件"和"好事件"？

**答：**

**关键定义**：
- **有效事件**：纳入 SLI 计算的事件（排除噪音）
- **好事件**：符合质量标准的事件

**示例**：

```
HTTP API 可用性 SLI：

有效事件 = 总请求 - 排除项
排除项：
  - 健康检查请求（/health, /ping）
  - 内部测试流量
  - 客户端明显错误（4xx 中 400、404 等）

好事件 = 有效事件中的非 5xx 响应
  - 200, 201, 204 → 好事件 ✓
  - 301, 302     → 好事件 ✓
  - 429          → 看情况（限流是否预期）
  - 500, 502     → 坏事件 ✗
```

### 9. 延迟 SLI 应该用平均值还是百分位数？

**答：** 应该用**百分位数**，不要用平均值。

| 指标 | 含义 | 问题 |
|------|------|------|
| **平均延迟** | 所有请求的平均响应时间 | 掩盖长尾延迟 |
| **P50** | 50% 请求在此时间内完成 | 典型用户体验 |
| **P90** | 90% 请求在此时间内完成 | 大部分用户体验 |
| **P99** | 99% 请求在此时间内完成 | 长尾用户体验 |

**举例**：
```
场景：100 个请求
- 99 个请求耗时 10ms
- 1 个请求耗时 10000ms

平均延迟 = (99×10 + 10000) / 100 = 109.9ms  ← 看起来还行
P99     = 10000ms                              ← 暴露了长尾问题
```

## 四、SLI 的实现

### 10. 如何用 Prometheus 实现可用性 SLI？

**答：**

```promql
# HTTP 可用性 SLI
# 好事件：非 5xx 的请求
# 有效事件：总请求（排除健康检查）

# 30 天窗口的可用性 SLI
sum(rate(http_requests_total{
  status!~"5..",
  path!="/health"
}[30d]))
/
sum(rate(http_requests_total{
  path!="/health"
}[30d]))
```

```promql
# gRPC 可用性 SLI
sum(rate(grpc_server_handled_total{
  grpc_code!="Internal",
  grpc_code!="Unavailable"
}[30d]))
/
sum(rate(grpc_server_handled_total[30d]))
```

### 11. 如何用 Prometheus 实现延迟 SLI？

**答：**

```promql
# 延迟 SLI：P99 < 300ms 的请求占比
# 使用 histogram bucket

sum(rate(http_request_duration_seconds_bucket{
  le="0.3"
}[30d]))
/
sum(rate(http_request_duration_seconds_count[30d]))
```

## 五、面试高频题

### 12. 面试题：为你的服务设计 SLI

**答：** 以电商订单服务为例：

| SLI | 定义 | 好事件 |
|-----|------|--------|
| 可用性 | 订单操作成功率 | 非 5xx 响应的请求 |
| 延迟 | 快速订单响应率 | 响应时间 < 500ms 的请求 |
| 正确性 | 订单金额准确率 | 金额计算正确的订单 |

**选择理由**：
1. **可用性**：下单失败直接影响营收
2. **延迟**：支付页面响应慢影响转化率
3. **正确性**：金额错误带来资金风险

### 13. 面试题：SLI 与业务指标的关系是什么？

**答：**

```
业务指标         技术 SLI          关系
─────────       ──────────       ──────
用户满意度  ←── 延迟 SLI         延迟越低，满意度越高
转化率      ←── 可用性 SLI       可用性越高，转化越好
数据准确性  ←── 正确性 SLI       正确率影响数据信任
SEO 排名    ←── 延迟 SLI         Google 将速度纳入排名
```

**关键点**：好的 SLI 应该与业务指标有**因果关系**，而不仅仅是相关性。
