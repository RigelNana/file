# 错误预算策略八股文

---

## 一、错误预算核心概念

### 1. 什么是错误预算（Error Budget）？

**答：** 错误预算是 SLO 允许的**不可靠性额度**，量化了服务在一定时间窗口内可以容忍的失败量。

```
错误预算 = 1 - SLO 目标

示例：SLO = 99.9%
错误预算 = 1 - 0.999 = 0.1%

30天窗口换算：
总分钟 = 30 × 24 × 60 = 43200 分钟
允许不可用 = 43200 × 0.001 = 43.2 分钟
```

**各可靠性等级的月度错误预算**：

| SLO | 错误预算 | 30天允许宕机 |
|-----|----------|-------------|
| 99% | 1% | 7.2 小时 |
| 99.5% | 0.5% | 3.6 小时 |
| 99.9% | 0.1% | 43.2 分钟 |
| 99.95% | 0.05% | 21.6 分钟 |
| 99.99% | 0.01% | 4.32 分钟 |

### 2. 错误预算的核心价值是什么？

**答：**

```
错误预算解决的核心矛盾：

开发团队：想要快速发布新功能  ← 可靠性风险
SRE 团队：想要保持系统稳定   ← 创新阻力

         错误预算（仲裁者）
              │
    ┌─────────┼─────────┐
    │                   │
预算充足：             预算耗尽：
  可以冒险               必须稳定
  快速发布               停止高风险变更
  尝试新技术             投入可靠性改进
```

## 二、错误预算策略（Error Budget Policy）

### 3. 什么是错误预算策略？

**答：** 错误预算策略是一份**提前约定的文档**，明确当错误预算消耗到不同程度时，各方应该采取什么行动。

**策略模板**：

| 预算剩余 | 状态 | 开发团队行动 | SRE 行动 |
|----------|------|-------------|----------|
| > 50% | 🟢 正常 | 正常发布、正常迭代 | 常规监控 |
| 25%-50% | 🟡 注意 | 加强变更审查 | 增加监控粒度 |
| 5%-25% | 🟠 警告 | 暂停非关键发布 | 主导可靠性改进 |
| 0%-5% | 🔴 危险 | 仅允许紧急修复 | 全力恢复可靠性 |
| ≤ 0% | ⛔ 耗尽 | 冻结所有变更 | 启动紧急可靠性项目 |

### 4. 错误预算耗尽后应该怎么做？

**答：**

```
错误预算耗尽（≤ 0%）
         │
         ▼
┌──────────────────────┐
│ 冻结所有非安全性变更    │
│ 开发团队投入可靠性修复  │
│ SRE 主导改进计划       │
└──────────┬───────────┘
           │
           ▼
排查根因并制定 Action Items：
  - 基础设施问题 → 架构改进
  - 代码缺陷 → Bug 修复
  - 容量不足 → 扩容
  - 运维缺陷 → 流程改进
           │
           ▼
改进后恢复正常发布节奏
```

### 5. 错误预算策略需要哪些人签字确认？

**答：**

| 角色 | 责任 | 签字理由 |
|------|------|----------|
| **SRE 负责人** | 定义策略和阈值 | 确保策略合理可执行 |
| **开发负责人** | 遵守发布冻结 | 明确自己的责任 |
| **产品负责人** | 理解功能交付影响 | 接受可靠性优先 |
| **VP/总监** | 提供管理支撑 | 确保策略能被执行 |

## 三、错误预算的计算与消耗

### 6. 如何计算错误预算消耗？

**答：**

**基于事件的计算**：
```
# 请求级别
错误预算（请求数）= 总请求 × (1 - SLO)
已消耗预算 = 失败请求数
剩余预算 = 错误预算 - 已消耗预算
消耗百分比 = 已消耗预算 / 错误预算 × 100%

示例：
SLO = 99.9%, 30天总请求 = 1,000,000
错误预算 = 1,000,000 × 0.001 = 1,000 次失败
已失败 = 300 次
消耗百分比 = 300/1000 = 30%
```

**基于时间的计算**：
```
# 时间级别
错误预算（分钟）= 窗口总分钟 × (1 - SLO)
已消耗预算 = 不可用分钟数
剩余预算 = 错误预算 - 已消耗

示例：
SLO = 99.9%, 30天窗口
错误预算 = 43200 × 0.001 = 43.2 分钟
本月已宕机 = 15 分钟
剩余 = 43.2 - 15 = 28.2 分钟（65.3%）
```

### 7. 错误预算消耗速率（Burn Rate）是什么？

**答：**

```
Burn Rate = 实际错误率 / 允许错误率

SLO = 99.9%（允许错误率 0.1%）

场景1：当前错误率 = 0.1%
  Burn Rate = 0.1% / 0.1% = 1x → 正常消耗

场景2：当前错误率 = 1%
  Burn Rate = 1% / 0.1% = 10x → 10倍速消耗！

场景3：当前错误率 = 0.01%
  Burn Rate = 0.01% / 0.1% = 0.1x → 消耗很慢
```

**Burn Rate 与耗尽时间**：

| Burn Rate | 30天预算耗尽时间 |
|-----------|-----------------|
| 1x | 30 天（正常） |
| 2x | 15 天 |
| 10x | 3 天 |
| 14.4x | ~2 天 |
| 36x | 20 小时 |
| 720x | 1 小时 |
| 1440x | 30 分钟 |

## 四、错误预算的分配与归因

### 8. 当多个团队共享一个服务时，错误预算如何分配？

**答：**

**方案一：按责任归因**

```
服务A 月度错误预算 = 43.2 分钟

本月事故：
  事故1（Dev 代码 Bug）：15 分钟 → Dev 消耗
  事故2（SRE 配置错误）：5 分钟  → SRE 消耗
  事故3（第三方宕机）：  10 分钟  → 排除或共担

Dev 消耗：15/43.2 = 34.7%
SRE 消耗：5/43.2 = 11.6%
```

**方案二：按变更类型归因**

| 消耗来源 | 归责方 |
|----------|--------|
| 代码变更导致 | 开发团队 |
| 配置变更导致 | SRE 团队 |
| 基础设施故障 | 平台团队 |
| 第三方故障 | 不归因或平摊 |

### 9. 面试题：如果产品经理坚持要发布功能但错误预算已经耗尽，你怎么处理？

**答：**

**按策略执行**：

1. **出示错误预算策略文档**——这是提前签字确认的
2. **展示数据**——图表展示预算消耗趋势和原因
3. **提出替代方案**：
   - 修复可靠性问题后再发布（推迟 1-2 周）
   - 小范围灰度（降低风险）
   - 配合特性标志（可快速回滚）
4. **升级决策**——如果僵持不下，升级到双方共同的上级

**关键原则**：
- 错误预算策略是"法律"，不是"建议"
- 但也不是死板执行，需要判断风险大小
- 如果是零风险变更（如改文案），可以豁免

### 10. Prometheus 中如何实现错误预算监控？

**答：**

```promql
# 错误预算剩余百分比

# 步骤1：计算 SLI
- record: sli:http_availability:ratio_30d
  expr: |
    sum(rate(http_requests_total{status!~"5.."}[30d]))
    /
    sum(rate(http_requests_total[30d]))

# 步骤2：计算错误预算消耗
- record: error_budget:consumption:ratio_30d
  expr: |
    1 - (
      (1 - sli:http_availability:ratio_30d)
      /
      (1 - 0.999)
    )

# 步骤3：告警
- alert: ErrorBudgetNearlyExhausted
  expr: error_budget:consumption:ratio_30d > 0.9
  for: 5m
  labels:
    severity: critical
  annotations:
    summary: "错误预算已消耗超过 90%"
```
