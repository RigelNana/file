# SLO 与用户旅程八股文

---

## 一、用户旅程与 CUJ

### 1. 什么是 CUJ（Critical User Journey）？

**答：** CUJ 是用户在使用产品时完成核心目标的**端到端路径**。SLO 应该以 CUJ 为基础来定义。

**电商平台 CUJ 示例**：

```
CUJ 1：搜索并浏览商品
  用户 → 搜索框 → 搜索结果 → 商品详情页

CUJ 2：加入购物车并下单
  用户 → 加入购物车 → 购物车页 → 确认订单 → 支付

CUJ 3：查看订单状态
  用户 → 我的订单 → 订单详情 → 物流追踪
```

### 2. 为什么 SLO 要以用户旅程为基础？

**答：**

```
传统 SLO（按服务定义）：
  订单服务 SLO = 99.9%     ✓ 达标
  支付服务 SLO = 99.95%    ✓ 达标
  库存服务 SLO = 99.9%     ✓ 达标

但用户体验：
  "下单"旅程 = 订单 × 支付 × 库存
  = 0.999 × 0.9995 × 0.999
  = 99.75%  ← 明显低于任何单个服务！

用户关心的是旅程，不是服务！
```

| 对比 | 按服务定义 SLO | 按用户旅程定义 SLO |
|------|---------------|-------------------|
| 视角 | 工程内部 | 用户感知 |
| 问题 | 各服务达标但体验差 | 直接反映用户体验 |
| 优先级 | 不知道哪个服务更重要 | CUJ 优先级决定投资 |

### 3. 如何识别关键用户旅程？

**答：**

**识别方法**：

| 维度 | 问题 | 示例 |
|------|------|------|
| **营收影响** | 失败了会损失多少钱？ | 支付流程 > 推荐 |
| **用户频率** | 多少用户多频繁使用？ | 搜索 > 管理后台 |
| **替代方案** | 失败后用户有替代吗？ | 核心功能无替代 > 锦上添花 |
| **用户期望** | 用户对速度/可靠的期望？ | 支付 > 浏览 |

```
CUJ 优先级矩阵：

          高收入影响
              │
   P0 支付     │  P1 搜索
   ──────────┼──────────
   P2 设置     │  P3 推荐
              │
          低用户频率 ──→ 高用户频率
```

## 二、基于 CUJ 定义 SLO

### 4. 如何将 CUJ 转化为 SLO？

**答：**

**步骤**：

```
Step 1：定义 CUJ
  "用户搜索商品并查看详情"

Step 2：识别涉及的服务
  前端 → API Gateway → 搜索服务 → 商品服务 → 缓存

Step 3：定义端到端 SLI
  可用性 SLI = 搜索请求返回有效结果的比例
  延迟 SLI = 从输入搜索到结果展示的时间

Step 4：设定 SLO
  可用性 ≥ 99.9%（30天滚动)
  延迟 P95 < 500ms

Step 5：定义度量点
  在 API Gateway 层度量（接近用户端）
```

### 5. 端到端 SLI 应该在哪里度量？

**答：**

```
用户旅程度量点选择：

用户浏览器
     │ ← 最接近用户，但数据可能不完整
     ▼
   CDN
     │
     ▼
负载均衡器 ← ✅ 推荐主要度量点
     │
     ▼
API Gateway ← ✅ 推荐辅助度量点
     │
     ▼
  微服务层 ← 排查使用
     │
     ▼
  数据库层 ← 排查使用
```

**推荐**：
- **主要 SLI**：在负载均衡器或 API Gateway 处度量
- **辅助 SLI**：在客户端/前端度量（如果可行）
- **排查用**：各微服务内部指标

## 三、多步骤旅程的 SLO

### 6. 多步骤用户旅程的 SLO 如何设计？

**答：**

以"下单"旅程为例：

```
下单旅程包含 4 步：
1. 选择商品加入购物车
2. 确认收货地址
3. 选择支付方式
4. 完成支付

方案一：端到端 SLO
  SLI = 从"加入购物车"到"支付成功"的完成率
  SLO ≥ 99.5%
  优点：直接反映用户体验
  缺点：任何一步出问题都会影响

方案二：分步 SLO + 端到端 SLO
  Step1 SLO ≥ 99.9%
  Step2 SLO ≥ 99.9%
  Step3 SLO ≥ 99.95%
  Step4 SLO ≥ 99.99%
  端到端 SLO ≥ 99.5%
  优点：可以精确定位瓶颈
  缺点：维护成本高
```

### 7. 如何处理跨渠道的用户旅程？

**答：**

```
同一旅程，不同渠道：

           Web        Mobile App     API
           │              │            │
           ▼              ▼            ▼
      ┌────────────────────────────────┐
      │        搜索服务               │
      └────────────────────────────────┘

SLO 策略：
  选项A：统一 SLO（所有渠道共用）
    搜索可用性 ≥ 99.9%
    
  选项B：分渠道 SLO
    Web 搜索 ≥ 99.9%
    App 搜索 ≥ 99.95%（比 Web 更严格）
    API 搜索 ≥ 99.99%（面向企业客户）
```

## 四、用户旅程 SLO 的实践

### 8. 如何构建 CUJ 到 SLO 的映射文档？

**答：**

```markdown
## CUJ-SLO 映射表

### CUJ-001：商品搜索与浏览
- **描述**：用户在搜索框输入关键词并浏览搜索结果
- **优先级**：P0
- **涉及服务**：API Gateway, 搜索服务, 商品服务, CDN
- **SLI**：
  - 可用性：返回有效搜索结果的请求比例
  - 延迟：搜索请求 P95 延迟
- **SLO**：
  - 可用性 ≥ 99.9%（30天滚动）
  - P95 延迟 < 300ms
- **度量点**：API Gateway 入口
- **Owner**：搜索团队 + SRE
```

### 9. CUJ SLO 和服务级 SLO 如何共存？

**答：**

```
两层 SLO 体系：

┌─ CUJ SLO（对外、业务视角）────────────┐
│                                       │
│  CUJ: 搜索商品        SLO ≥ 99.9%   │
│    ├── 搜索服务 SLO    ≥ 99.95%      │
│    ├── 商品服务 SLO    ≥ 99.95%      │
│    └── CDN SLO         ≥ 99.99%      │
│                                       │
│  CUJ: 完成下单        SLO ≥ 99.5%   │
│    ├── 订单服务 SLO    ≥ 99.9%       │
│    ├── 支付服务 SLO    ≥ 99.95%      │
│    └── 库存服务 SLO    ≥ 99.9%       │
│                                       │
└── 服务级 SLO（内部、技术视角）─────────┘

CUJ SLO 驱动业务决策
服务 SLO 驱动工程排查
```

## 五、面试高频题

### 10. 面试题：如何识别一个新产品的 CUJ？

**答：**

```
三步走：

Step 1：利益相关者访谈
  - 产品经理：最核心的用户流程是什么？
  - 销售：客户最关心什么？
  - 客服：用户投诉最多的是什么？

Step 2：数据分析
  - 用户行为分析：最高频的操作路径
  - 漏斗分析：转化率最低的步骤
  - 投诉分析：用户报告最多的问题

Step 3：优先级排序
  - 按营收影响 × 用户频率排序
  - 选择 Top 3-5 个作为初始 CUJ
  - 定期回顾和补充
```

### 11. 面试题：CUJ SLO 不达标时如何排查？

**答：**

```
CUJ "完成下单" SLO 不达标

Step 1：确认哪个维度不达标
  可用性 ✗（98.5%，目标 99.5%）
  延迟 ✓（P95 = 400ms，目标 500ms）

Step 2：拆解到服务级别
  订单服务  99.9% ✓
  支付服务  99.0% ✗ ← 问题在这里
  库存服务  99.9% ✓

Step 3：深入支付服务
  按接口拆分：
    /pay/create    99.5% ✓
    /pay/callback  97.0% ✗ ← 第三方回调问题

Step 4：定位根因
  第三方支付渠道B的回调成功率下降
  → 联系第三方 或 增加重试 + 降级到渠道A
```
