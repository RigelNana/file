## 一、SLO 驱动的工程决策

### 1. SLO 如何驱动工程优先级排序？

**答：**

```
传统方式：
  PM 说"这个功能很重要" → 优先开发
  运维说"系统不稳定" → 不知道排多高优先级

SLO 驱动：
  SLO 达标 + 预算充足 → 优先开发新功能
  SLO 达标 + 预算紧张 → 新功能 + 可靠性并行
  SLO 不达标          → 优先可靠性改进
```

**决策矩阵**：

| 错误预算状态 | 功能开发 | 可靠性工作 | 技术债务 |
|-------------|----------|-----------|----------|
| 充足（>50%） | ✅ 正常推进 | 常规维护 | 可选 |
| 中等（25-50%） | ⚠️ 谨慎推进 | 增加投入 | 优先高风险的 |
| 紧张（5-25%） | ❌ 仅低风险 | 主要精力 | 暂缓 |
| 耗尽（≤5%） | ❌ 冻结 | 全力投入 | 冻结 |

### 2. 如何用 SLO 数据说服产品团队投入可靠性工作？

### 3. SLO 如何指导架构决策？

**答：**

| 场景 | SLO 数据 | 决策 |
|------|----------|------|
| 单点故障 | 数据库宕机消耗 80% 预算 | 投资读写分离/多活 |

| 扩展性 | 高峰期 SLI 下降 | 自动扩缩容 |

## 二、SLO 驱动的发布决策

### 4. 如何基于 SLO 决定是否发布？

**答：**

```
发布前检查清单：

□ 错误预算剩余 > 25%？
  ├── 是 → 可以发布
  └── 否 → 评估发布风险
           │
           ├── 低风险（配置变更/文案修改）→ 允许发布
           ├── 中风险（小功能）→ 使用特性标志 + 金丝雀
           └── 高风险（大功能/架构变更）→ 推迟到预算恢复
```

### 5. 发布后如何用 SLO 评估发布质量？

**答：**

```
发布后 SLO 观察窗口：

T+0    发布开始（金丝雀 1%）
T+10m  检查 SLI：成功率、延迟、错误率
T+30m  检查 SLO 是否仍然达标
T+1h   检查燃烧率是否正常（≤ 1x）
T+24h  确认 SLO 稳定

回滚条件（任一满足即回滚）：
  - SLI 下降 > 0.1%
  - 燃烧率 > 3x
  - 错误预算消耗速度明显加快
```

## 三、SLO 驱动的运维决策

### 6. SLO 如何指导容量规划？

**答：**

```
SLO 驱动的容量规划流程：

1. 分析历史 SLI 与负载的关系
   负载增长 10% → SLI 从 99.95% → 99.92%

2. 预测业务增长
   下季度预计流量增长 30%

3. 计算 SLI 影响
   负载 +30% → 预计 SLI ≈ 99.86%
   当前 SLO = 99.9%
   → 将不达标！

4. 制定扩容计划
   扩容 50%（含缓冲）→ 预计 SLI ≈ 99.94%
   → 满足 SLO 要求
```

### 7. SLO 如何驱动 On-Call 改进？

**答：**

| SLO 数据信号 | On-Call 改进措施 |
|-------------|-----------------|
| 频繁 Page 告警但 SLO 达标 | 告警阈值太敏感，调高阈值 |
| SLO 不达标但无 Page 告警 | 告警覆盖不足，增加告警规则 |
| 凌晨频繁告警 | 分析是否可以延迟到工作时间处理 |
| 同类告警反复出现 | 自动化修复或根因修复 |

## 四、SLO 驱动的组织决策

### 8. SLO 如何影响团队资源分配？

**答：**

```
季度资源分配模型：

团队总人力：10 人

SLO 状态评估：
  服务A：SLO 达标，预算充足  → 无需额外投入
  服务B：SLO 勉强达标        → 需要 1 人做可靠性改进
  服务C：SLO 不达标          → 需要 2 人集中修复

分配：
  功能开发：6 人
  服务B 可靠性：1 人
  服务C 可靠性：2 人
  平台建设：1 人

如果没有 SLO 数据 → 管理层无法量化分配
有了 SLO 数据 → 分配有据可依
```

### 9. SLO 如何影响 SRE 是否接管服务？

**答：**

**接管评估标准**：

| 条件 | 要求 | 不满足的后果 |
|------|------|-------------|
| 有明确的 SLI/SLO | 必须 | 无法度量服务质量 |
| SLO 连续达标 3 个月 | 推荐 | 说明服务不够成熟 |
| 有完善的 Runbook | 必须 | On-Call 无法高效响应 |
| Toil < 50% | 必须 | 否则 SRE 变成纯运维 |
| 开发团队配合 | 必须 | 否则问题无法闭环 |

## 五、面试高频题

### 10. 面试题：SLO 达标但用户投诉增多，你怎么分析？

**答：**

```
可能原因排查：

1. SLI 选择不当
   → SLI 没有覆盖用户关心的维度
   示例：只监控成功率，没监控响应时间

2. SLO 目标太低
   → SLO = 99%，但用户期望 99.9%
   → 需要提高 SLO 目标

3. 用户旅程覆盖不全
   → 只监控了 API，没监控前端加载
   → 需要增加端到端 SLI

4. 分布不均
   → 整体 SLO 达标，但某地区/某时段不达标
   → 需要按维度拆分 SLI

5. 阈值定义问题
   → 延迟 SLI 定义为 P99<2s
   → 但用户觉得 >500ms 就慢
   → 需要调整延迟阈值
```

### 11. 面试题：如何平衡功能迭代速度和 SLO 达标？

**答：**

**核心方法：错误预算就是天然的平衡器。**

```
预算充足（>50%）：
  → 加速迭代，可以承担更多风险
  → 尝试更频繁的发布
  → 做一些创新性的变更

预算紧张（<25%）：
  → 减慢迭代速度
  → 每次发布必须有回滚方案
  → 优先修复可靠性问题

长期平衡：
  → 投资自动化测试和 CI/CD → 每次发布风险更低
  → 投资可观测性 → 问题检测更快
  → 投资渐进式发布 → 爆炸半径更小
  → 最终：更快的迭代 + 更高的可靠性
```
