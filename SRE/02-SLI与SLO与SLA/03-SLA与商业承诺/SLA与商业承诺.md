# SLA 与商业承诺八股文

---

## 一、SLA 基础概念

### 1. 什么是 SLA（Service Level Agreement）？

**答：** SLA 是服务提供方与客户之间的**正式商业协议**，包含可靠性承诺和违约后果。

**SLA 组成要素**：

| 要素 | 描述 | 示例 |
|------|------|------|
| **服务描述** | 承诺覆盖的服务范围 | 计算实例、对象存储 |
| **可用性目标** | 承诺的服务可靠性 | 月度可用性 ≥ 99.95% |
| **度量方式** | 如何计算可用性 | 不可用分钟数 / 总分钟数 |
| **排除条件** | 哪些情况不算违约 | 计划维护、不可抗力 |
| **赔偿方案** | 违约后客户获得什么 | 服务信用额度 |
| **申诉流程** | 客户如何主张赔偿 | 30 天内提交工单 |

### 2. SLA、SLO、SLI 三者的关系是什么？

**答：**

```
SLI  →  SLO  →  SLA
度量     目标     合同

SLI = 可用性实际为 99.97%
SLO = 内部目标 ≥ 99.95%    ← 工程团队看这个
SLA = 客户承诺 ≥ 99.9%     ← 客户看这个

关键：SLO > SLA（留缓冲）
```

**层次关系**：

| 层次 | 严格度 | 可见性 | 后果 |
|------|--------|--------|------|
| SLI | - | 工程内部 | 数据采集 |
| SLO | 最严格 | 工程 + 产品 | 触发错误预算策略 |
| SLA | 最宽松 | 客户外部 | 触发商业赔偿 |

### 3. 为什么 SLO 要比 SLA 更严格？

**答：**

```
场景：
  SLA = 99.9%（月度可用性）
  允许宕机 = 43.2 分钟/月

如果 SLO = SLA = 99.9%：
  当 SLO 告警触发时，你已经在违约边缘
  没有修复时间！

如果 SLO = 99.95%（比 SLA 严 0.05%）：
  SLO 告警触发 → 还有一半错误预算
  有时间修复，避免违反 SLA
```

## 二、SLA 设计与谈判

### 4. SLA 可用性有哪些常见等级和对应的停机时间？

**答：**

| 可用性 | 年宕机 | 月宕机 | 日宕机 | 俗称 |
|--------|--------|--------|--------|------|
| 99% | 3.65 天 | 7.31 小时 | 14.4 分钟 | 两个9 |
| 99.9% | 8.77 小时 | 43.2 分钟 | 1.44 分钟 | 三个9 |
| 99.95% | 4.38 小时 | 21.6 分钟 | 43.2 秒 | 三个半9 |
| 99.99% | 52.6 分钟 | 4.32 分钟 | 8.6 秒 | 四个9 |
| 99.999% | 5.26 分钟 | 25.9 秒 | 0.86 秒 | 五个9 |

### 5. 主流云厂商的 SLA 是怎样的？

**答：**

| 云服务 | 产品 | SLA | 赔偿 |
|--------|------|-----|------|
| AWS EC2 | 计算实例 | 99.99% | 10%-100% 信用额 |
| AWS S3 | 对象存储 | 99.9% | 10%-25% 信用额 |
| GCP Compute | 计算实例 | 99.99% | 10%-50% 信用额 |
| Azure VM | 虚拟机 | 99.99% | 10%-100% 信用额 |
| 阿里云 ECS | 计算实例 | 99.975% | 10%-30% 信用额 |

### 6. SLA 中常见的排除条件有哪些？

**答：**

| 排除项 | 描述 |
|--------|------|
| **计划维护** | 提前通知的维护窗口不算宕机 |
| **不可抗力** | 自然灾害、战争等 |
| **客户操作** | 客户自身造成的服务中断 |
| **超出用量限制** | 超过服务配额导致的问题 |
| **Beta/Preview** | 预览版服务不受 SLA 保护 |
| **第三方原因** | 因第三方供应商导致的中断 |

## 三、SLA 赔偿机制

### 7. SLA 赔偿通常是什么形式？

**答：** 最常见的是**服务信用额度（Service Credits）**，而非现金退款。

**典型赔偿阶梯**：

```
月度可用性          服务信用额度
────────           ──────────
99.0% - 99.9%     10%
95.0% - 99.0%     25%
< 95.0%           100%

示例：
月费 $10,000，可用性实际为 98%
赔偿 = $10,000 × 25% = $2,500 信用额度
（下月账单抵扣，不是现金退款）
```

### 8. 为什么 SLA 赔偿通常远低于客户的实际损失？

**答：**

```
示例：
电商平台每小时 GMV = 100 万元
使用的云服务月费 = 10 万元
云服务宕机 2 小时 → 客户损失 200 万元
SLA 赔偿 = 10 万 × 25% = 2.5 万元

客户实际损失：200 万
SLA 赔偿：    2.5 万  （1.25%!）
```

**所以**：SLA 赔偿不是保险，客户需要自己做：
- 多云灾备
- 降级兜底
- 业务连续性计划

## 四、SLA 管理实践

### 9. SRE 如何确保不违反 SLA？

**答：**

**多层防护**：

```
第1层：SLO 告警（比 SLA 更严格）
  SLO = 99.95%，SLA = 99.9%
  SLO 违约时还有 SLA 缓冲
         │
         ▼
第2层：错误预算告警
  预算消耗 > 50% → 减少高风险变更
  预算消耗 > 75% → 暂停非关键发布
  预算消耗 > 90% → 全力修复
         │
         ▼
第3层：SLA 预警
  接近 SLA 阈值 → 升级管理层
  → 启动紧急响应
```

### 10. 面试题：如何为你的 SaaS 产品设计 SLA？

**答：**

**设计步骤**：

1. **确定内部 SLO**：先有可靠的 SLO 数据
2. **确定 SLA 目标**：SLA = SLO - 缓冲（通常低 0.05% 以上）
3. **定义度量方式**：明确如何计算可用性
4. **设计赔偿阶梯**：合理但不过度承诺
5. **列出排除条件**：保护合理的运维需求
6. **法务审核**：确保法律条款无风险

**SLA 文档模板（关键部分）**：

```markdown
## 服务等级协议

### 1. 可用性承诺
本服务月度可用性不低于 99.9%。

### 2. 可用性计算
可用性 = (总分钟数 - 不可用分钟数) / 总分钟数 × 100%

### 3. 排除事项
以下情况不计入不可用时间：
- 已提前 48 小时通知的计划维护
- 客户自身操作导致的服务中断
- 不可抗力事件

### 4. 服务信用
| 月度可用性 | 信用额度 |
|-----------|---------|
| 99.0%-99.9% | 月费的 10% |
| 95.0%-99.0% | 月费的 25% |
| < 95.0% | 月费的 50% |

### 5. 申诉流程
客户须在事件发生后 30 个自然日内提交申诉。
```
