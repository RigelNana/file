# 可观测性平台建设八股文

---

## 一、平台整体架构

### 1. 可观测性平台的整体架构是什么？

**答：**

```
┌──────────────────────────────────────────────┐
│                 展示层                        │
│  Grafana（统一可视化） + PagerDuty（告警通知）  │
├──────────────────────────────────────────────┤
│                 查询层                        │
│  PromQL + LogQL + TraceQL                    │
├──────────────────────────────────────────────┤
│                 存储层                        │
│  Prometheus/Mimir  Loki    Tempo             │
│  (Metrics)        (Logs)  (Traces)           │
├──────────────────────────────────────────────┤
│                 处理层                        │
│  OTel Collector (采样/过滤/路由/富化)          │
├──────────────────────────────────────────────┤
│                 采集层                        │
│  OTel SDK + Promtail + node_exporter         │
├──────────────────────────────────────────────┤
│                 应用层                        │
│  微服务 + 中间件 + 基础设施                    │
└──────────────────────────────────────────────┘
```

### 2. 常见的可观测性技术栈组合有哪些？

**答：**

| 方案 | Metrics | Logs | Traces | 可视化 | 适合 |
|------|---------|------|--------|--------|------|
| **PLG 栈** | Prometheus | Loki | Tempo | Grafana | 中小规模、低成本 |
| **ELK + Prometheus** | Prometheus | ELK | Jaeger | Grafana+Kibana | 中大规模 |
| **Datadog** | Datadog | Datadog | Datadog | Datadog | 全托管、预算充足 |
| **自建 + OTel** | Mimir | Loki | Tempo | Grafana | 大规模、自主可控 |

### 3. 如何选择自建 vs 商业方案？

**答：**

| 维度 | 自建 | 商业（Datadog/New Relic） |
|------|------|--------------------------|
| 成本 | 低（基础设施+人力） | 高（按量付费） |
| 运维 | 需要专人运维 | 零运维 |
| 定制 | 完全可控 | 受限 |
| 上手 | 慢（需要学习） | 快（开箱即用） |
| 规模 | 需要自行扩展 | 自动扩展 |

## 二、平台能力建设

### 4. 可观测性平台需要具备哪些核心能力？

**答：**

| 能力 | 描述 | 实现 |
|------|------|------|
| **统一接入** | 标准化的数据接入规范 | OTel SDK + Collector |
| **数据关联** | Metrics-Traces-Logs 互联 | TraceID 关联 |
| **SLO 管理** | SLO 定义、监控、报告 | Sloth/Pyrra |
| **智能告警** | 燃烧率告警 + 异常检测 | Alertmanager |
| **快速排查** | 从告警到根因的快速链路 | Grafana 下钻 |
| **自助接入** | 开发团队自助配置 | 平台化工具 |

### 5. 数据关联具体怎么实现？

**答：**

```
三大信号关联矩阵：

Metrics ←→ Traces
  方式：Exemplar（指标样本关联 Trace ID）
  场景：延迟指标异常 → 点击查看对应 Trace

Traces ←→ Logs
  方式：Trace ID 嵌入日志
  场景：某个 Span 出错 → 查看对应日志

Metrics ←→ Logs  
  方式：通过相同的时间范围和标签关联
  场景：错误率上升 → 查看同时段错误日志

Grafana 配置：
  Prometheus DS → Exemplar → Tempo DS
  Tempo DS → Trace to Logs → Loki DS
  Loki DS → Derived Fields → Tempo DS
```

## 三、平台运营

### 6. 可观测性平台的运营指标有哪些？

**答：**

| 指标 | 描述 | 目标 |
|------|------|------|
| **接入覆盖率** | 接入平台的服务比例 | > 95% |
| **数据延迟** | 从生成到可查询的延迟 | < 30s |
| **查询延迟** | Dashboard 加载时间 | < 5s |
| **存储成本** | 每 GB 存储成本 | 持续优化 |
| **可用性** | 平台自身可用性 | > 99.9% |

### 7. 如何控制可观测性平台的成本？

**答：**

```
成本构成：
  存储（最大头）> 计算 > 网络

控制策略：
┌─────────────────────────────────────────┐
│ 策略              │ 效果              │
├─────────────────────────────────────────┤
│ 指标降采样         │ 远期数据 5m 精度   │
│ 日志采样           │ 正常日志 1% 采样   │
│ Trace 尾部采样     │ 只保留错误/慢请求  │
│ 数据保留策略       │ 7天/30天/90天分层  │
│ 压缩              │ Loki 压缩 10:1    │
│ 标签管控           │ 避免高基数         │
│ 冷热分层           │ SSD → HDD → S3   │
└─────────────────────────────────────────┘
```

## 四、成熟度评估

### 8. 如何评估组织的可观测性成熟度？

**答：**

| 等级 | 描述 | 能力 |
|------|------|------|
| **L1 基础** | 有基本监控 | 基础设施指标 + 简单告警 |
| **L2 标准** | 三大支柱齐全 | Metrics + Logs + Traces |
| **L3 关联** | 数据互联互通 | 三大支柱通过 TraceID 关联 |
| **L4 SLO 驱动** | SLO 为核心 | SLO 仪表盘 + 燃烧率告警 + 错误预算 |
| **L5 智能** | AI 辅助 | 异常检测 + 自动关联 + 根因推荐 |

### 9. 面试题：如何从零建设可观测性平台？

**答：**

```
Phase 1（1-2月）：基础建设
  ├── 部署 Prometheus + Grafana
  ├── 节点监控（node_exporter）
  ├── 应用 RED 指标
  └── 基础告警

Phase 2（3-4月）：日志体系
  ├── 部署 Loki + Promtail
  ├── 结构化日志规范
  ├── 日志搜索和分析
  └── 关键服务日志告警

Phase 3（5-6月）：链路追踪
  ├── 部署 Tempo + OTel Collector
  ├── 核心服务接入 OTel SDK
  ├── 配置 Metrics-Traces-Logs 关联
  └── 采样策略

Phase 4（7-9月）：SLO 体系
  ├── 定义核心服务 SLI/SLO
  ├── 多窗口燃烧率告警
  ├── SLO 仪表盘
  └── 错误预算策略

Phase 5（10-12月）：平台化
  ├── 自助接入工具
  ├── 成本优化
  ├── 平台自身可靠性
  └── 推广和培训
```

### 10. 面试题：可观测性平台自身如何保证可靠性？

**答：**

```
"谁来监控监控系统？"

策略：
1. 独立的健康检查
   - 外部拨测（如 UptimeRobot）监控 Grafana/Prometheus
   - 与主系统隔离的独立告警通道

2. 高可用部署
   - Prometheus HA（双写）
   - Alertmanager HA（集群）
   - Grafana 多副本 + LB

3. 降级方案
   - 主 Prometheus 宕 → 备 Prometheus 接管
   - Grafana 宕 → PromQL 直接查询
   - 所有都宕 → SSH 到服务器看本地日志

4. 容量预留
   - 监控系统的资源不与业务共享
   - 预留 20% 容量缓冲
```
