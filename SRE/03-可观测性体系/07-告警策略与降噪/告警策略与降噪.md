# 告警策略与降噪八股文

---

## 一、告警设计原则

### 1. 好的告警应该满足什么条件？

**答：**

| 原则 | 描述 | 反模式 |
|------|------|--------|
| **可操作** | 收到后知道该做什么 | "CPU 80%" → 然后呢？ |
| **紧急** | 需要人工立即处理 | 非紧急的用工单 |
| **真实** | 对应真实问题（低误报） | 80% 告警都是误报 |
| **关联 SLO** | 影响用户体验的才 Page | 不影响用户的不告警 |

**Google SRE 的告警判断标准**：
```
每条 Page 告警都应能回答：
1. 数据表明这是一个真实问题吗？     → 减少误报
2. 需要人工立即介入吗？             → 区分紧急性
3. 我能采取什么行动？               → 确保可操作
4. 是否影响用户？                   → 关联业务影响
```

### 2. 告警分级体系如何设计？

**答：**

| 级别 | 含义 | 通知方式 | 响应时间 |
|------|------|----------|----------|
| **P0/Critical** | 严重影响核心业务 | 电话 + 短信 | 5 分钟内 |
| **P1/High** | 影响部分用户 | Slack/企微 + 短信 | 15 分钟内 |
| **P2/Medium** | 有风险但暂未影响 | 工单系统 | 工作时间内 |
| **P3/Low** | 信息类/优化建议 | 邮件/日报 | 下次迭代 |

## 二、告警降噪

### 3. 告警疲劳是什么？有哪些表现？

**答：**

```
告警疲劳循环：

告警太多 → 工程师开始忽略告警
    │           │
    ▼           ▼
误报太多     静默所有告警
    │           │
    ▼           ▼
真实告警被忽略 → 故障响应延迟 → 更大的事故
```

**告警疲劳指标**：
- 每周 On-Call Page 数 > 2 次/天（太多）
- 告警 ACK 平均时长持续增长
- 超过 30% 的告警被直接静默
- 团队成员抱怨 On-Call 体验差

### 4. 告警降噪有哪些策略？

**答：**

| 策略 | 描述 | 效果 |
|------|------|------|
| **聚合（Grouping）** | 相关告警合并为一条 | 减少通知数量 |
| **抑制（Inhibition）** | 高级别抑制低级别 | 避免级联通知 |
| **静默（Silence）** | 维护期间暂时屏蔽 | 消除已知噪音 |
| **延迟（for 子句）** | 持续 N 分钟才告警 | 过滤瞬间尖峰 |
| **阈值调优** | 基于历史数据优化阈值 | 减少误报 |
| **SLO 告警** | 基于燃烧率而非静态阈值 | 根本性降噪 |

### 5. 如何系统性地优化告警？

**答：**

```
告警优化流程：

Step 1：收集数据（2-4周）
  记录每条告警的：
  - 是否真实问题？
  - 是否需要立即处理？
  - 采取了什么操作？
  - 花了多长时间？

Step 2：分类分析
  有效告警（真实+需要立即处理）：保留
  可延迟告警（真实+不紧急）：降为工单
  误报告警（不代表真实问题）：调阈值或删除
  重复告警（同一问题反复告）：合并或根因修复

Step 3：实施优化
  目标：每周 On-Call Page 不超过 2 次
  每条告警都有对应的 Runbook
```

## 三、告警路由与通知

### 6. 告警升级（Escalation）机制如何设计？

**答：**

```
升级路径：

T+0m    告警触发 → 通知一线 On-Call
T+5m    未 ACK → 通知二线 On-Call
T+15m   未解决 → 通知 SRE Manager
T+30m   未解决 → 通知 VP（如果是 P0）
T+60m   未解决 → 启动全面战争室（War Room）

PagerDuty 升级策略配置：
  Level 1: Primary On-Call     (立即)
  Level 2: Secondary On-Call   (5分钟后)
  Level 3: SRE Manager         (15分钟后)
  Level 4: VP of Engineering   (30分钟后)
```

### 7. 如何设计多渠道告警通知？

**答：**

| 级别 | 渠道 | 原因 |
|------|------|------|
| P0 | PagerDuty 电话 + Slack + 短信 | 必须确保收到 |
| P1 | Slack + 短信 | 紧急但不需要电话 |
| P2 | Slack/企微 + 工单 | 工作时间处理 |
| P3 | 邮件日报 | 仅供参考 |

## 四、告警运营

### 8. On-Call 告警的关键运营指标有哪些？

**答：**

| 指标 | 定义 | 目标 |
|------|------|------|
| **Pages/周** | 每周 Page 告警数 | ≤ 2 次/天 |
| **MTTA** | 平均确认时间 | < 5 分钟 |
| **MTTR** | 平均恢复时间 | < 30 分钟 |
| **误报率** | 误报 / 总告警 | < 5% |
| **无操作率** | 不需要操作的告警 / 总告警 | < 10% |
| **告警覆盖率** | 有告警覆盖的故障 / 总故障 | > 95% |

### 9. 面试题：你如何评估一个告警体系的质量？

**答：**

```
评估维度：

1. 覆盖率（是否能发现问题）
   - 过去 3 个月的故障中有多少是告警发现的？
   - 有多少是用户报告的？

2. 精确率（是否有太多噪音）
   - 误报率是多少？
   - 不需要操作的告警占比？

3. 及时性（是否足够快）
   - 平均检测时间（TTD）
   - 平均确认时间（TTA）

4. 可操作性（收到后能否行动）
   - 每条告警是否有 Runbook？
   - 工程师是否知道下一步做什么？

5. 工程师体验
   - On-Call 满意度调查
   - 每周 Page 数量趋势
```

### 10. 面试题：告警风暴如何处理？

**答：**

```
告警风暴特征：短时间内大量告警同时触发

应急处理：
  1. 不要逐条处理 → 先看全局
  2. 确认是否有全局性故障（如机房级别）
  3. 找到根源告警（可能只有一个）
  4. 批量静默非根源告警

预防措施：
  - 告警聚合（group_by service）
  - 告警抑制（infra 故障抑制应用告警）
  - 告警速率限制（同类告警最多 N 条/分钟）
  - 告警依赖建模（先告"因"再告"果"）
```
