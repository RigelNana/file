# 可观测性三大支柱八股文

---

## 一、可观测性基础概念

### 1. 什么是可观测性（Observability）？

**答：** 可观测性是指通过系统的**外部输出**（指标、日志、链路追踪），推断系统**内部状态**的能力。

```
可观测性 ≠ 监控

监控（Monitoring）：    已知的已知 → 对预定义指标设定告警
可观测性（Observability）：未知的未知 → 能回答任意问题
```

| 对比 | 监控 | 可观测性 |
|------|------|----------|
| 核心问题 | "系统正常吗？" | "系统为什么不正常？" |
| 方式 | 预定义告警 | 灵活查询和探索 |
| 能力 | 检测已知问题 | 分析未知问题 |

### 2. 可观测性三大支柱是什么？

**答：**

| 支柱 | 英文 | 核心内容 | 典型工具 |
|------|------|----------|----------|
| **指标（Metrics）** | Metrics | 聚合的数值时序数据 | Prometheus, Datadog |
| **日志（Logs）** | Logs | 离散的事件记录 | ELK, Loki, Splunk |
| **链路追踪（Traces）** | Traces | 请求的端到端路径 | Jaeger, Zipkin, Tempo |

```
三大支柱的关系：

Metrics（全局概览）
  "错误率上升了"
       │
       ▼ 定位哪些请求出错
Traces（请求路径）
  "这个请求在支付服务超时了"
       │
       ▼ 查看详细上下文
Logs（详细上下文）
  "支付服务: 连接数据库超时 timeout=30s"
```

### 3. 三大支柱如何协同排查问题？

**答：**

```
排查流程：

1. Metrics 发现问题
   "支付服务 P99 延迟从 200ms 升到 5s"
          │
          ▼
2. Traces 定位瓶颈
   找到慢请求的 Trace
   "数据库查询耗时 4.8s"
          │
          ▼
3. Logs 确认根因
   "SLOW QUERY: SELECT * FROM orders WHERE ... (4.8s)"
   "缺少索引：full table scan on orders"
          │
          ▼
4. 修复
   "ALTER TABLE orders ADD INDEX idx_user_id(user_id)"
```

## 二、指标（Metrics）

### 4. Metrics 有哪些类型？

**答：**

| 类型 | 英文 | 描述 | 示例 |
|------|------|------|------|
| **计数器** | Counter | 只增不减的累积值 | 总请求数、总错误数 |
| **仪表盘** | Gauge | 可增可减的瞬时值 | 当前连接数、内存使用量 |
| **直方图** | Histogram | 值的分布 | 请求延迟分布 |
| **摘要** | Summary | 预计算百分位 | P50/P99 延迟 |

```
Counter: 0 → 1 → 2 → 5 → 10 → ...（只增不减）
Gauge:   50 → 72 → 63 → 88 → 45（上下波动）
Histogram: 将值分桶计数
  [0-10ms]=500, [10-50ms]=300, [50-100ms]=100, [>100ms]=10
```

### 5. 四大黄金信号（Golden Signals）是什么？

**答：**

| 信号 | 英文 | 含义 | SRE 关注点 |
|------|------|------|-----------|
| **延迟** | Latency | 处理时间 | 区分成功和失败请求的延迟 |
| **流量** | Traffic | 请求量 | QPS、并发连接数 |
| **错误率** | Errors | 失败比例 | 5xx 比例、业务错误率 |
| **饱和度** | Saturation | 资源使用 | CPU、内存、队列深度 |

## 三、日志（Logs）

### 6. 日志有哪些级别和最佳实践？

**答：**

| 级别 | 用途 | 生产环境 |
|------|------|----------|
| **TRACE** | 极详细的调试信息 | 通常关闭 |
| **DEBUG** | 调试信息 | 通常关闭 |
| **INFO** | 正常操作记录 | 开启 |
| **WARN** | 潜在问题警告 | 开启 |
| **ERROR** | 错误事件 | 开启 |
| **FATAL** | 致命错误 | 开启 |

**结构化日志最佳实践**：

```json
{
  "timestamp": "2024-01-15T10:30:00Z",
  "level": "ERROR",
  "service": "payment-service",
  "trace_id": "abc123def456",
  "span_id": "span789",
  "message": "支付回调处理失败",
  "error": "connection timeout",
  "order_id": "ORD-2024-001",
  "duration_ms": 30500,
  "retry_count": 3
}
```

### 7. 日志和 Metrics 的边界在哪里？

**答：**

| 场景 | 用 Metrics | 用 Logs |
|------|-----------|---------|
| 错误率趋势 | ✅ | |
| 单个错误详情 | | ✅ |
| 资源利用率 | ✅ | |
| 请求参数和上下文 | | ✅ |
| 告警判断 | ✅ | |
| 审计追踪 | | ✅ |

## 四、链路追踪（Traces）

### 8. 分布式追踪的核心概念有哪些？

**答：**

```
一个 Trace（完整请求路径）：

Trace ID: abc-123
├── Span A: API Gateway (0-100ms)
│   ├── Span B: 订单服务 (10-80ms)
│   │   ├── Span C: 数据库查询 (20-50ms)
│   │   └── Span D: 缓存查询 (15-18ms)
│   └── Span E: 支付服务 (30-95ms)
│       └── Span F: 第三方支付 (35-90ms)
```

| 概念 | 描述 |
|------|------|
| **Trace** | 一个完整请求的端到端路径 |
| **Span** | Trace 中的一个操作单元 |
| **Trace ID** | 全局唯一的追踪标识 |
| **Span ID** | Span 的唯一标识 |
| **Parent Span** | 父 Span，表示调用关系 |
| **Baggage** | 跨服务传递的上下文数据 |

### 9. 采样策略有哪些？

**答：**

| 策略 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **头部采样** | 在请求入口决定是否采样 | 简单、完整链路 | 可能错过异常请求 |
| **尾部采样** | 请求完成后决定是否保留 | 保留所有异常 | 实现复杂、需缓存 |
| **速率采样** | 固定比例采样（如 1%） | 简单易配 | 低流量时样本不足 |
| **自适应采样** | 根据流量动态调整比例 | 平衡成本和覆盖 | 配置复杂 |

## 五、面试高频题

### 10. 面试题：三大支柱分别适合解答什么问题？

**答：**

```
Metrics 回答：
  "系统现在正常吗？"
  "过去一周的趋势如何？"
  "哪个服务的错误率最高？"

Logs 回答：
  "这个错误的具体原因是什么？"
  "用户 X 的操作记录是什么？"
  "这个异常发生在什么上下文中？"

Traces 回答：
  "这个请求经过了哪些服务？"
  "延迟主要消耗在哪里？"
  "服务 A 和服务 B 的调用关系是什么？"
```

### 11. 面试题：如何评估一个系统的可观测性成熟度？

**答：**

| 成熟度 | 描述 | 能力 |
|--------|------|------|
| **L0** | 无可观测性 | 出问题靠 SSH 进服务器看 |
| **L1** | 基础监控 | 有 Metrics 和基本告警 |
| **L2** | 结构化日志 | Metrics + 结构化 Logs + 搜索 |
| **L3** | 完整三支柱 | Metrics + Logs + Traces 关联 |
| **L4** | 智能可观测 | 异常检测 + 自动关联 + AIOps |
