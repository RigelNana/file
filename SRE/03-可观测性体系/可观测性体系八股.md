# 可观测性体系八股文

---

## 一、可观测性基础

### 1. 什么是可观测性？与传统监控有什么区别？

**答：** 可观测性（Observability）是指从系统的**外部输出**推断系统**内部状态**的能力。

| 维度 | 传统监控 | 可观测性 |
|------|----------|----------|
| 关注点 | 已知的故障模式 | 未知的未知（Unknown Unknowns） |
| 方式 | 预设仪表盘和告警 | 自由探索和查询 |
| 数据维度 | 低维度聚合数据 | 高基数、高维度原始数据 |
| 问题发现 | 告警通知"什么坏了" | 可以回答"为什么坏了" |
| 典型做法 | 固定 Dashboard | 即席查询（Ad-hoc Query） |

### 2. 可观测性的三大支柱是什么？

**答：**

| 支柱 | 英文 | 说明 | 典型工具 |
|------|------|------|----------|
| **指标** | Metrics | 可聚合的数值型时序数据 | Prometheus, Datadog |
| **日志** | Logs | 离散的事件记录 | ELK, Loki, Splunk |
| **追踪** | Traces | 请求在分布式系统中的完整路径 | Jaeger, Zipkin, Tempo |

**三者的关系：**
- Metrics 告诉你**"出了问题"**（如错误率升高）
- Logs 告诉你**"问题的细节"**（如具体的错误信息）
- Traces 告诉你**"问题在哪里"**（如哪个微服务、哪个调用链）

### 3. Google 的四大黄金信号是什么？RED 方法和 USE 方法呢？

**答：**

**四大黄金信号（Four Golden Signals）：**
1. **延迟（Latency）**：区分成功/失败请求的延迟
2. **流量（Traffic）**：系统承载的请求量
3. **错误率（Errors）**：失败请求的比例
4. **饱和度（Saturation）**：资源使用率

**RED 方法**（面向请求的微服务）：
- **R**ate：请求速率
- **E**rrors：错误率
- **D**uration：请求耗时

**USE 方法**（面向资源的基础设施）：
- **U**tilization：资源利用率
- **S**aturation：饱和度（排队等待）
- **E**rrors：错误计数

## 二、Prometheus 监控

### 4. Prometheus 的架构是怎样的？

**答：**

```
┌──────────────────────────────────────────────────────┐
│                   Prometheus Server                    │
│  ┌─────────────┐  ┌──────────┐  ┌─────────────────┐ │
│  │  Retrieval   │  │   TSDB   │  │  HTTP Server    │ │
│  │  (Pull拉取)  │  │ (存储引擎)│  │  (查询接口)     │ │
│  └──────┬──────┘  └──────────┘  └────────┬────────┘ │
└─────────┼────────────────────────────────┼──────────┘
          │                                │
    ┌─────▼─────┐                    ┌─────▼─────┐
    │  Targets   │                    │  Grafana   │
    │ (采集目标)  │                    │  Alertmgr  │
    └───────────┘                    └───────────┘
```

核心组件：
- **Prometheus Server**：拉取指标、存储、查询
- **Exporters**：暴露指标的代理（node_exporter, mysqld_exporter 等）
- **Pushgateway**：短生命周期 Job 推送指标
- **Alertmanager**：告警路由、分组、抑制、静默
- **TSDB**：高效的时序数据库存储引擎

### 5. Prometheus 的指标类型有哪些？

**答：**

| 类型 | 英文 | 特点 | 使用场景 |
|------|------|------|----------|
| **Counter** | 计数器 | 只增不减，重启归零 | 请求总数、错误总数 |
| **Gauge** | 仪表盘 | 可增可减 | 温度、内存使用量、连接数 |
| **Histogram** | 直方图 | 分桶统计分布 | 请求延迟分布、响应大小 |
| **Summary** | 摘要 | 客户端计算分位数 | 精确分位数（P50/P99） |

## 三、告警策略

### 6. 如何设计好的告警策略？

**答：** 好的告警应遵循以下原则：

1. **基于症状而非原因告警**：
   - ✅ "HTTP 错误率超过 1%"
   - ❌ "CPU 使用率超过 80%"

2. **告警必须可行动**：
   - 收到告警后必须有明确的行动步骤
   - 如果不需要立即行动，就不应该是紧急告警

3. **分级告警**：
   - **Page（紧急）**：需要立即人工响应，如影响用户的故障
   - **Ticket（工单）**：需要在工作时间处理，如磁盘快满
   - **Log（记录）**：仅记录不通知，如非关键告警信息

4. **避免告警疲劳**：
   - 控制告警数量（每个 On-Call 轮次不超过 2 个 Page）
   - 消除重复告警和无价值告警
   - 定期审查告警质量
