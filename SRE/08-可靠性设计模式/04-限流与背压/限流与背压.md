## 一、限流基础

### 1. 什么是限流（Rate Limiting）？为什么需要？

**答：**

```
限流定义：
  对系统的请求速率进行控制，
  防止过多请求压垮服务。

为什么需要限流：
  ┌────────────────────────────────────────┐
  │ 1. 保护后端服务不被流量洪峰击垮        │
  │ 2. 防止单个客户端占用过多资源          │
  │ 3. 防御 DDoS 和爬虫                   │
  │ 4. 保证公平性（多租户场景）            │
  │ 5. 控制 API 调用成本                   │
  └────────────────────────────────────────┘
```

### 2. 常见的限流算法有哪些？

**答：**

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **固定窗口** | 每个时间窗口计数 | 简单 | 窗口边界突发 |
| **滑动窗口** | 滑动时间窗口计数 | 平滑 | 实现稍复杂 |
| **漏桶** | 固定速率处理请求 | 平滑输出 | 不能处理突发 |
| **令牌桶** | 按速率放令牌，有桶容量 | 允许突发 | 稍复杂 |

```
令牌桶算法图示：

  令牌产生速率 r（如 100/s）
       │
       ▼
  ┌──────────┐    桶容量 b（如 200）
  │ ○○○○○○○○ │
  │ ○○○○○○○○ │
  └────┬─────┘
       │ 每个请求消耗一个令牌
       ▼
    [请求通过]
    
  桶满时：新令牌丢弃（限制长期速率）
  桶空时：请求被拒绝或排队
  桶内有令牌：允许突发流量（最多 b 个）
```

```go
// Go 标准库令牌桶限流
package main

import (
    "golang.org/x/time/rate"
    "net/http"
)

func rateLimitMiddleware(limiter *rate.Limiter) func(http.Handler) http.Handler {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            if !limiter.Allow() {
                http.Error(w, "Too Many Requests", http.StatusTooManyRequests)
                return
            }
            next.ServeHTTP(w, r)
        })
    }
}

func main() {
    // 每秒 100 个请求，允许突发 200
    limiter := rate.NewLimiter(100, 200)
    
    mux := http.NewServeMux()
    mux.HandleFunc("/api", handler)
    
    http.ListenAndServe(":8080", rateLimitMiddleware(limiter)(mux))
}
```

### 3. 分布式限流如何实现？

**答：**

```
单机限流 vs 分布式限流：

单机限流：
  每个实例独立计数
  问题：10 个实例各限 100 QPS = 总共 1000 QPS

分布式限流：
  集中式计数（Redis）
  优势：全局精确控制
```

```python
# Redis 滑动窗口限流（Python 示例）
import redis
import time

r = redis.Redis()

def is_rate_limited(key: str, limit: int, window: int) -> bool:
    """
    滑动窗口限流
    key: 限流标识（如 user:123）
    limit: 窗口内允许的请求数
    window: 窗口大小（秒）
    """
    now = time.time()
    pipe = r.pipeline()
    
    # 移除窗口外的记录
    pipe.zremrangebyscore(key, 0, now - window)
    # 添加当前请求
    pipe.zadd(key, {str(now): now})
    # 统计窗口内请求数
    pipe.zcard(key)
    # 设置过期时间
    pipe.expire(key, window)
    
    results = pipe.execute()
    count = results[2]
    
    return count > limit
```

## 二、背压机制

### 4. 什么是背压（Backpressure）？

**答：**

```
背压定义：
  当下游处理速度跟不上上游发送速度时，
  通过信号机制让上游减速或停止发送。

类比水管：
  水龙头(上游) ──→ 水管(通道) ──→ 水杯(下游)
  
  没有背压：水杯溢出（系统崩溃）
  有背压：  水杯快满时，关小水龙头

背压 vs 限流的区别：
┌──────────────┬────────────────────┐
│   限流        │   背压              │
├──────────────┼────────────────────┤
│ 在入口控制    │ 在系统内部传递       │
│ 丢弃多余请求  │ 减慢上游速度         │
│ 保护服务端    │ 端到端流量控制       │
│ 粗粒度控制    │ 细粒度自适应         │
└──────────────┴────────────────────┘
```

### 5. 背压的实现方式有哪些？

**答：**

```
方式一：有界队列
  Producer ──→ [Queue (max=1000)] ──→ Consumer
  队列满时：拒绝/阻塞 Producer

方式二：响应式流（Reactive Streams）
  Subscriber 告诉 Publisher 自己能处理多少
  request(100) → 发送 100 条 → request(50) → ...

方式三：TCP 流量控制
  接收方通过滑动窗口通告可用缓冲
  窗口为 0 → 发送方暂停

方式四：消息队列背压
  ┌────────┐    ┌──────┐    ┌────────┐
  │Producer│ →  │Kafka │ →  │Consumer│
  └────────┘    └──────┘    └────────┘
  
  Consumer Lag 增大 → 告警
  Consumer 处理不过来 → 
    1. 增加 Consumer 实例
    2. 或让 Producer 降低发送速率

方式五：HTTP 429 响应
  服务端返回 429 Too Many Requests
  客户端收到后主动降低请求频率
  配合 Retry-After Header
```

### 6. 面试题：限流被触发后应该怎么处理？

**答：**

```
客户端策略：
  1. 指数退避重试
     wait = min(base * 2^attempt, max_wait)
     加随机 Jitter 防止惊群

  2. 尊重 Retry-After
     resp.header("Retry-After") → 等待指定时间

  3. 自适应降级
     检测到 429 → 降低本地发送速率
     持续一段时间 → 恢复

服务端策略：
  1. 优先级限流
     VIP 客户 > 普通用户 > 爬虫
     
  2. 优雅降级
     超出限制时返回缓存数据（而非直接拒绝）
     
  3. 公平排队
     请求入队而非直接拒绝
     但需设置队列超时
```

## 三、面试高频题

### 7. 面试题：如何设计一个多级限流系统？

**答：**

```
多级限流架构：

  CDN/WAF 层 ─→ 网关层 ─→ 服务层 ─→ 依赖层

第一层：CDN/WAF (IP 级限流)
  ├── 单 IP 1000 QPS 上限
  ├── 防 DDoS / CC 攻击
  └── 地理区域限制

第二层：API Gateway (API + 用户级限流)
  ├── 每用户 100 QPS
  ├── 每 API 独立限额
  └── 全局总量限制

第三层：Service (自我保护)
  ├── 自适应限流（根据 CPU/队列深度）
  ├── 断路器保护依赖
  └── 降级策略

第四层：依赖保护 (连接池/并发控制)
  ├── DB 连接池上限
  ├── Redis 连接数限制
  └── 外部 API 调用限流
```

### 8. 面试题：自适应限流是什么？

**答：**

```
自适应限流：
  不使用固定阈值，而是根据系统实时状态
  动态调整限流阈值。

核心思想：
  系统健康 → 放宽限制
  系统压力大 → 收紧限制

判断指标：
  ├── CPU 使用率
  ├── 请求队列长度
  ├── P99 延迟
  └── 错误率

示例逻辑：
  if cpu > 80% || p99 > 2x_baseline:
      limit = limit * 0.8  # 减少 20% 通过量
  elif cpu < 50% && p99 < baseline:
      limit = limit * 1.1  # 增加 10% 通过量

优势：
  ✓ 不需要提前知道精确容量
  ✓ 适应不同负载模式
  ✓ 硬件变化后自动适配
```
