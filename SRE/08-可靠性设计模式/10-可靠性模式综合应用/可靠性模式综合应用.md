## 一、模式组合

### 1. 可靠性设计模式如何组合使用？

**答：**

```
单一模式不足以保证可靠性，需要多层防护：

                    ┌──────────────┐
                    │   限流       │ ← 入口防护
                    └──────┬───────┘
                           ▼
                    ┌──────────────┐
                    │   超时       │ ← 快速失败
                    └──────┬───────┘
                           ▼
                    ┌──────────────┐
                    │   重试+退避  │ ← 容忍瞬时故障
                    └──────┬───────┘
                           ▼
                    ┌──────────────┐
                    │   熔断器     │ ← 防止雪崩
                    └──────┬───────┘
                           ▼
                    ┌──────────────┐
                    │   降级       │ ← 保证核心功能
                    └──────┬───────┘
                           ▼
                    ┌──────────────┐
                    │   幂等       │ ← 保证数据正确
                    └──────────────┘

调用链中的组合示例：
  Client
    → 限流检查（令牌桶）
    → 设置 Deadline（3s）
    → 调用服务
      → 超时检查（1s）
      → 如果失败 → 重试（指数退避，最多 2 次）
      → 如果持续失败 → 熔断器打开
      → 熔断时 → 降级兜底
    → 幂等检查（去重）
```

### 2. 典型微服务调用的可靠性设计全景？

**答：**

```yaml
# 微服务 A 调用服务 B 的完整可靠性配置
service_b:
  # 超时
  timeout:
    connect: 1s
    read: 2s
    
  # 重试
  retry:
    max_attempts: 3
    backoff:
      initial: 100ms
      max: 2s
      jitter: true
    retryable_codes: [502, 503, 504]
    retry_budget: 0.1  # 10%
    
  # 熔断
  circuit_breaker:
    failure_threshold: 50%
    window: 10s
    half_open_requests: 5
    recovery_time: 30s
    
  # 限流
  rate_limit:
    qps: 1000
    burst: 200
    
  # 降级
  fallback:
    type: "cached_response"
    cache_ttl: 60s
    
  # 负载均衡
  load_balancer:
    algorithm: "round_robin"
    health_check:
      interval: 5s
      unhealthy_threshold: 3
```

### 3. Service Mesh 如何统一实现可靠性模式？

**答：**

```
Service Mesh（Istio/Linkerd）优势：
  将可靠性逻辑从应用代码移到基础设施层

  传统方式：                    Service Mesh 方式：
  ┌──────────┐                 ┌──────────┐
  │ App Code │                 │ App Code │ (纯业务)
  │ + 重试   │                 └────┬─────┘
  │ + 熔断   │                      │
  │ + 超时   │                 ┌────┴─────┐
  │ + 限流   │                 │ Sidecar  │ (所有可靠性)
  └──────────┘                 │ (Envoy)  │
                               └──────────┘
```

```yaml
# Istio VirtualService + DestinationRule
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: service-b
spec:
  hosts:
    - service-b
  http:
    - route:
        - destination:
            host: service-b
      timeout: 2s
      retries:
        attempts: 3
        perTryTimeout: 1s
        retryOn: "5xx,reset,connect-failure"
---
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: service-b
spec:
  host: service-b
  trafficPolicy:
    connectionPool:
      http:
        h2UpgradePolicy: DEFAULT
        maxRequestsPerConnection: 100
    outlierDetection:
      consecutiveErrors: 5
      interval: 10s
      baseEjectionTime: 30s
      maxEjectionPercent: 50
```

## 二、架构级可靠性

### 4. 如何设计一个高可用的电商系统？

**答：**

```
高可用电商架构：

用户 → CDN → WAF → LB → API Gateway → 微服务 → DB

各层可靠性设计：

CDN 层：
  ├── 静态资源缓存
  ├── DDoS 防护
  └── 多节点冗余

网关层：
  ├── 限流（用户级/API级）
  ├── 认证鉴权
  ├── 灰度路由
  └── 降级开关

服务层：
  ├── 无状态设计（水平扩展）
  ├── 超时 + 重试 + 熔断
  ├── 功能降级
  └── 多可用区部署

数据层：
  ├── 主从复制 + 自动切换
  ├── 读写分离
  ├── 分库分表
  └── 多活/灾备

缓存层：
  ├── 多级缓存（本地 + Redis）
  ├── 缓存雪崩防护
  ├── 热key打散
  └── 缓存穿透防护

可用性目标：
  整体 SLO: 99.99%（每年 < 52 分钟停机）
```

### 5. 如何计算多组件系统的可用性？

**答：**

```
串联系统（所有组件必须可用）：
  A (99.9%) → B (99.9%) → C (99.9%)
  总可用性 = 0.999 × 0.999 × 0.999 = 99.7%

并联系统（任一组件可用即可）：
  A (99.9%) ─┐
              ├─→ 总可用性 = 1 - (0.001)² = 99.9999%
  B (99.9%) ─┘

实际系统（混合）：
  ┌─ Web(99.9%) ─┐         ┌─ DB主(99.9%) ─┐
  │               │         │                │
  └─ Web(99.9%) ─┘ → API → └─ DB从(99.9%) ─┘
  = 99.9999%        99.9%   = 99.9999%
  
  总可用性 = 0.999999 × 0.999 × 0.999999 ≈ 99.9%

关键结论：
  系统可用性 ≤ 最弱环节的可用性
  要提高整体，先加固最弱的那个环节
```

## 三、面试高频题

### 6. 面试题：设计一个可靠的支付系统

**答：**

```
支付系统可靠性要点：

1. 幂等性
   每笔支付有唯一流水号
   重复请求返回相同结果

2. 一致性
   强一致：扣款操作（CP）
   最终一致：通知/对账（AP）

3. 高可用
   支付核心链路独立部署
   多可用区/多机房
   核心依赖有 Fallback

4. 容错
   超时 + 重试 + 熔断
   支付网关故障自动切换
   限流防止突发流量

5. 对账
   T+1 与银行对账
   差异自动告警
   自动补单/退款

6. 监控
   实时支付成功率
   支付延迟 P99
   各渠道可用率
```

### 7. 面试题：可靠性模式在微服务治理中的优先级？

**答：**

```
实施优先级（从必须到加分）：

P0 必须有：
  ├── 超时（无超时 = 定时炸弹）
  ├── 重试（基本的容错能力）
  └── 健康检查（故障检测基础）

P1 强烈建议：
  ├── 熔断（防止级联故障）
  ├── 限流（入口保护）
  └── 幂等（数据安全）

P2 成熟团队：
  ├── 降级（精细化容错）
  ├── Deadline 传播
  └── 自适应限流

P3 追求卓越：
  ├── Service Mesh 统一管理
  ├── 混沌工程验证
  └── 全链路压测
```

### 8. 面试题：如何向非技术人员解释可靠性设计？

**答：**

```
类比解释：

限流 = 餐厅限流
  "一次只接待 100 位客人，其他排队"

超时 = 等餐时间
  "点餐后超过 30 分钟没上菜，可以催或退餐"

重试 = 再试一次
  "打电话没人接，过一会儿再打"

熔断 = 保险丝
  "电路过载时保险丝断开，保护整个房子"

降级 = 备用方案
  "暴雨航班取消，改乘高铁"

幂等 = 防重复
  "电梯按了一次和按了十次效果一样"

冗余 = 备胎
  "汽车有备胎，爆胎了还能换上继续跑"
```
