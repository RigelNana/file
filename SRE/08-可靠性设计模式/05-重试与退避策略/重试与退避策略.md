## 一、重试基础

### 1. 为什么需要重试？什么时候应该重试？

**答：**

```
重试的价值：
  分布式系统中，暂时性故障是常态。
  适当重试可以掩盖这些瞬时错误，提升用户体验。

应该重试的场景：
  ✓ 网络超时（暂时性）
  ✓ 503 Service Unavailable
  ✓ 429 Too Many Requests（配合退避）
  ✓ 连接被重置
  ✓ DNS 解析失败

不应该重试的场景：
  ✗ 400 Bad Request（参数错误，重试也无用）
  ✗ 401/403（认证/授权失败）
  ✗ 404 Not Found
  ✗ 非幂等操作且无法确认结果
  ✗ 下游已明确过载
```

### 2. 重试策略有哪些？

**答：**

| 策略 | 公式 | 特点 |
|------|------|------|
| **立即重试** | wait = 0 | 最快但可能加剧问题 |
| **固定间隔** | wait = const | 简单但不够灵活 |
| **指数退避** | wait = base × 2^n | 逐渐降低重试频率 |
| **指数退避+Jitter** | wait = random(0, base × 2^n) | 避免惊群效应 |
| **装饰性退避** | wait = base × 2^n + random(0, base) | 折中方案 |

```
指数退避 + Jitter 图示：

重试次数:    1      2       3        4
退避上限:    2s     4s      8s       16s
Jitter后:    ·      ·       ·        ·
          [0-2s] [0-4s] [0-8s]  [0-16s]

为什么需要 Jitter：
  无 Jitter：1000 个客户端同时在 T+2s 重试
              → 又一波重试风暴
  有 Jitter：1000 个客户端在 [0, 2s] 间随机重试
              → 负载平滑分散
```

### 3. 如何实现指数退避？

**答：**

```go
// Go 指数退避实现
func retryWithBackoff(ctx context.Context, maxRetries int, fn func() error) error {
    baseDelay := 100 * time.Millisecond
    maxDelay := 30 * time.Second
    
    for attempt := 0; attempt <= maxRetries; attempt++ {
        err := fn()
        if err == nil {
            return nil
        }
        
        if attempt == maxRetries {
            return fmt.Errorf("max retries exceeded: %w", err)
        }
        
        // 指数退避 + Full Jitter
        delay := baseDelay * time.Duration(1<<uint(attempt))
        if delay > maxDelay {
            delay = maxDelay
        }
        jitter := time.Duration(rand.Int63n(int64(delay)))
        
        select {
        case <-ctx.Done():
            return ctx.Err()
        case <-time.After(jitter):
        }
    }
    return nil
}
```

```python
# Python 指数退避
import random
import time

def retry_with_backoff(fn, max_retries=3, base_delay=0.1, max_delay=30):
    for attempt in range(max_retries + 1):
        try:
            return fn()
        except RetryableError as e:
            if attempt == max_retries:
                raise
            delay = min(base_delay * (2 ** attempt), max_delay)
            jitter = random.uniform(0, delay)
            time.sleep(jitter)
```

## 二、重试风险

### 4. 重试风暴（Retry Storm）是什么？如何避免？

**答：**

```
重试风暴场景：

正常时：
  Client ─(100 QPS)─→ Server ✓

Server 故障时（无重试控制）：
  Client ─(100 QPS)──→ Server ✗
  Client ─(重试 100)──→ Server ✗ ← 200 QPS
  Client ─(再重试 100)→ Server ✗ ← 300 QPS
  
  多层级联时更严重：
  A → B → C
  C 故障：
  B 重试 3 次 → C 收到 3x 流量
  A 重试 3 次 → C 收到 9x 流量！

避免策略：
  ┌──────────────────────────────────────────┐
  │ 1. 限制重试次数（最多 3 次）              │
  │ 2. 指数退避 + Jitter                     │
  │ 3. 重试预算（如 10% 请求允许重试）        │
  │ 4. 熔断器（故障多时直接不调用）           │
  │ 5. 只在最外层重试（避免级联放大）         │
  │ 6. 截止时间（Deadline）传播               │
  └──────────────────────────────────────────┘
```

### 5. 什么是重试预算（Retry Budget）？

**答：**

```
重试预算：限制一段时间内重试请求占总请求的比例。

例：重试预算 = 10%
  正常请求 1000 QPS → 最多允许 100 QPS 重试
  超出预算 → 放弃重试，直接返回错误

实现方式：
  retry_budget = 0.1  # 10%
  
  recent_requests = count_requests(last_1min)
  recent_retries = count_retries(last_1min)
  
  if recent_retries / recent_requests > retry_budget:
      # 超出预算，不重试
      return error
  else:
      # 允许重试
      retry()

优势：
  ├── 自适应：流量大时绝对重试量也大，但比例受控
  ├── 防止重试风暴：有全局上限
  └── Google SRE 推荐的做法
```

### 6. 面试题：幂等性和重试的关系？

**答：**

```
幂等性：同一操作执行多次，结果和执行一次相同

重试前必须考虑幂等性：
  GET  /user/123        → 天然幂等 ✓
  PUT  /user/123        → 天然幂等 ✓
  DELETE /user/123      → 天然幂等 ✓
  POST /orders          → 非幂等 ✗ 可能创建重复订单

非幂等操作如何安全重试：
  方案一：幂等键（Idempotency Key）
    Client 生成唯一 ID，Server 去重
    POST /orders
    Idempotency-Key: uuid-xxx-yyy
    
  方案二：先检查再操作
    if not exists(order_id):
        create_order()
    
  方案三：乐观锁
    UPDATE ... WHERE version = expected_version
```

## 三、面试高频题

### 7. 面试题：如何设计一个健壮的 HTTP 客户端重试策略？

**答：**

```
完整重试策略设计：

1. 可重试判断
   ├── 5xx → 重试
   ├── 429 → 退避后重试
   ├── 超时 → 重试
   ├── 连接错误 → 重试
   └── 4xx → 不重试

2. 退避策略
   ├── 指数退避 + Full Jitter
   ├── base=100ms, max=30s
   └── 最多 3 次重试

3. 安全保障
   ├── 总超时（Deadline）限制
   ├── 重试预算 10%
   ├── 配合熔断器
   └── 只对幂等操作重试

4. 可观测性
   ├── 记录重试次数指标
   ├── 记录重试成功/失败
   └── 告警：重试率异常升高

配置示例：
  retry:
    max_attempts: 3
    backoff:
      initial: 100ms
      max: 30s
      multiplier: 2
      jitter: true
    retryable_codes: [502, 503, 504, 429]
    retry_budget: 0.1
    deadline: 60s
```

### 8. 面试题：Deadline 传播是什么？为什么重要？

**答：**

```
Deadline 传播：
  请求的截止时间在整个调用链中传递。

A ──→ B ──→ C ──→ D
Deadline = T+5s

A 调用 B 时：remaining = 5s
B 调用 C 时：remaining = 4s（已用 1s）
C 调用 D 时：remaining = 2s（已用 3s）

如果 remaining ≤ 0：直接返回超时，不再调用

没有 Deadline 传播的问题：
  A 设了 5s 超时
  B 设了 10s 超时
  C 设了 10s 超时
  
  A 已经超时返回了
  但 B 和 C 还在处理 → 浪费资源

实现方式：
  gRPC：原生支持 Deadline 传播
  HTTP：通过 Header 传递
    X-Request-Deadline: 2024-01-15T14:00:05Z
```
