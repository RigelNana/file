# 冗余与高可用架构八股文

---

## 一、高可用基础

### 1. 什么是高可用？如何衡量？

**答：**

```
高可用 (High Availability) = 系统持续正常运行的能力

可用性计算：
  可用性 = MTTF / (MTTF + MTTR)
  MTTF = 平均无故障时间
  MTTR = 平均修复时间

可用性等级：
┌─────────┬───────────────┬──────────────────┐
│ 等级     │ 年度停机时间   │ 典型场景          │
├─────────┼───────────────┼──────────────────┤
│ 99%     │ 3.65 天       │ 内部系统          │
│ 99.9%   │ 8.76 小时     │ 普通业务系统       │
│ 99.95%  │ 4.38 小时     │ 电商/社交          │
│ 99.99%  │ 52.6 分钟     │ 支付/核心系统      │
│ 99.999% │ 5.26 分钟     │ 电信/金融核心      │
└─────────┴───────────────┴──────────────────┘

提升可用性的两条路：
  1. 增加 MTTF: 减少故障发生（冗余/容错）
  2. 减少 MTTR: 快速恢复（自动化/预案）
```

### 2. 冗余设计的模式有哪些？

**答：**

```
冗余模式：

1. 主动-被动 (Active-Passive)
   ┌──────┐    ┌──────┐
   │Active│    │Standby│
   │ (RW) │    │ (只读) │
   └──┬───┘    └──┬───┘
      │   同步/异步  │
      └─────────────┘
   
   特点：被动节点不处理请求
   切换时间：秒~分钟级
   适用：数据库、有状态服务

2. 主动-主动 (Active-Active)
   ┌──────┐    ┌──────┐
   │Active│    │Active│
   │ (RW) │    │ (RW) │
   └──┬───┘    └──┬───┘
      │   双向同步  │
      └─────────────┘
   
   特点：所有节点都处理请求
   切换时间：无需切换
   挑战：数据冲突解决
   适用：无状态服务、CDN

3. N+1 冗余
   N 个节点处理全部流量
   1 个备用节点作为余量
   任一节点故障 → 备用接管

4. N+N (双倍冗余)
   任何时刻可以损失 50% 节点
   用于极高可用要求
```

### 3. 多层高可用架构如何设计？

**答：**

```
全链路高可用：

┌─────────────────────────────────────┐
│ DNS 层: GSLB 多活                    │
│  → 多区域 DNS 解析                   │
│  → 健康检查自动切换                  │
├─────────────────────────────────────┤
│ 接入层: LB 集群                      │
│  → 多台 LB 互为备份                  │
│  → ECMP / VIP 漂移                   │
├─────────────────────────────────────┤
│ 应用层: 多副本 + 无状态               │
│  → K8s Deployment replicas >= 3      │
│  → 跨 AZ 反亲和调度                  │
├─────────────────────────────────────┤
│ 缓存层: Redis Cluster/Sentinel       │
│  → 主从 + 自动故障切换               │
│  → 多副本分布在不同 AZ              │
├─────────────────────────────────────┤
│ 数据层: 数据库主从 + 读写分离         │
│  → 同步/半同步复制                   │
│  → 自动故障切换                      │
├─────────────────────────────────────┤
│ 存储层: 分布式存储                    │
│  → 多副本                            │
│  → 跨 AZ/Region 复制                │
└─────────────────────────────────────┘

关键原则：
  每一层都有冗余
  单点故障 = 可用性短板
```

## 二、实践

### 4. K8s 中如何实现高可用部署？

**答：**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: payment-api
spec:
  replicas: 3  # 至少 3 副本
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1
  template:
    spec:
      # 跨 AZ 反亲和
      affinity:
        podAntiAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
            - weight: 100
              podAffinityTerm:
                labelSelector:
                  matchLabels:
                    app: payment-api
                topologyKey: topology.kubernetes.io/zone
      
      # 拓扑分散约束
      topologySpreadConstraints:
        - maxSkew: 1
          topologyKey: topology.kubernetes.io/zone
          whenUnsatisfiable: DoNotSchedule
          labelSelector:
            matchLabels:
              app: payment-api
      
      containers:
        - name: payment-api
          resources:
            requests:
              cpu: "500m"
              memory: "512Mi"
            limits:
              memory: "1Gi"
---
# PDB 保护
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: payment-api-pdb
spec:
  minAvailable: 2
  selector:
    matchLabels:
      app: payment-api
```

### 5. 面试题：如何计算组合系统的可用性？

**答：**

```
可用性计算公式：

串联系统（A → B → C 都要正常）：
  可用性 = A × B × C
  例：99.9% × 99.9% × 99.9% = 99.7%
  → 链路越长可用性越低

并联系统（A 或 B 有一个正常即可）：
  可用性 = 1 - (1-A) × (1-B)
  例：1 - (1-99%) × (1-99%) = 99.99%
  → 冗余提升可用性

实际计算：
  Web(3副本) → API(3副本) → DB(主从)
  
  Web: 1-(1-0.99)^3 = 99.9999%
  API: 1-(1-0.99)^3 = 99.9999%
  DB:  1-(1-0.99)^2 = 99.99%
  
  端到端 = 99.9999% × 99.9999% × 99.99%
         ≈ 99.99%

关键结论：
  系统可用性取决于最弱环节
  串联环节越多越不可靠
  用冗余提升单组件可用性
```

## 三、面试

### 6. 面试题：设计一个 99.99% 可用的支付系统？

**答：**

```
99.99% = 年停机 < 52 分钟

关键设计：

1. 多 AZ 部署
   至少 3 个 AZ，任一 AZ 故障不影响

2. 无状态应用层
   水平扩展 + 健康检查 + 自动重建

3. 数据库高可用
   MySQL Group Replication 或 PG Patroni
   自动故障切换 < 30秒

4. 缓存层
   Redis Sentinel/Cluster
   缓存失效有降级方案

5. 消息队列
   Kafka 3+ Broker，ISR ≥ 2
   消息持久化 + 消费幂等

6. 降级策略
   非核心依赖可降级
   核心链路保持最简

7. 监控和快速恢复
   MTTD < 1 分钟（自动告警）
   MTTR < 5 分钟（自动回滚/切换）
```

### 7. 面试题：单机房 vs 多机房 vs 异地多活？

**答：**

```
架构演进：

单机房:
  所有服务在一个数据中心
  可用性: ~99.9%
  风险: 机房级故障 = 全部不可用

同城双活:
  两个机房处理流量，数据同步复制
  可用性: ~99.99%
  延迟: 亚毫秒
  适用: 大多数互联网公司

异地多活:
  多个地域独立运行
  可用性: ~99.999%
  延迟: 几十毫秒
  挑战: 数据一致性
  适用: 超大规模/监管要求

异地多活关键技术：
  数据分片: 按地域划分用户数据
  流量路由: 用户就近接入
  数据同步: 异步复制 + 冲突解决
  切流方案: 单元故障时快速切换
```
