# 优雅降级八股文

---

## 一、降级基础

### 1. 什么是优雅降级（Graceful Degradation）？

**答：**

```
优雅降级定义：
  当系统部分功能不可用时，
  主动关闭非核心功能，
  保证核心功能正常运行。

类比：
  飞机一个引擎故障 → 另一个引擎继续飞 → 安全降落
  不是直接坠毁

降级 vs 熔断：
  ┌────────────────┬────────────────────┐
  │ 熔断            │ 降级                │
  ├────────────────┼────────────────────┤
  │ 自动触发        │ 主动/自动均可       │
  │ 停止调用下游    │ 返回替代结果        │
  │ 保护下游        │ 保证用户体验        │
  │ 全有或全无      │ 渐进式              │
  └────────────────┴────────────────────┘
```

### 2. 降级策略有哪些类型？

**答：**

| 类型 | 做法 | 示例 |
|------|------|------|
| **功能降级** | 关闭非核心功能 | 关闭推荐、评论 |
| **数据降级** | 返回缓存/默认数据 | 返回昨天的排行榜 |
| **体验降级** | 降低体验但保持功能 | 关闭动画、降低图片质量 |
| **流量降级** | 限制部分用户访问 | 排队机制 |
| **写入降级** | 暂停非关键写入 | 暂停日志上报 |

```
电商网站降级示例：

Level 0（正常）:
  全部功能可用

Level 1（轻微降级）:
  ├── 关闭个性化推荐 → 展示热门商品
  ├── 关闭实时评论   → 展示缓存评论
  └── 关闭实时库存   → 最终一致性

Level 2（中度降级）:
  ├── 搜索降级为分类浏览
  ├── 商品详情用缓存版本
  └── 关闭优惠计算 → 原价展示

Level 3（重度降级）:
  ├── 只保留：浏览商品 + 加购 + 支付
  ├── 关闭所有非交易功能
  └── 静态化核心页面

Level 4（极端降级）:
  ├── 启用排队机制
  ├── 静态化首页
  └── 只接受已在购物车的订单
```

### 3. 降级开关如何设计？

**答：**

```yaml
# 降级开关配置示例
degradation:
  # 推荐服务降级
  recommendation:
    enabled: true
    trigger:
      type: "manual"  # manual | auto | slo-based
    fallback: "hot_products"  # 热门商品列表
    
  # 搜索服务降级
  search:
    enabled: true
    trigger:
      type: "auto"
      conditions:
        error_rate: "> 5%"
        latency_p99: "> 2s"
    fallback: "category_browse"  # 降级为分类浏览
    
  # 评论服务降级
  review:
    enabled: true
    trigger:
      type: "slo-based"
      slo_budget_remaining: "< 20%"
    fallback: "cached_reviews"  # 缓存评论
```

```go
// Go 降级开关实现
type DegradeSwitch struct {
    mu       sync.RWMutex
    switches map[string]bool
}

func (d *DegradeSwitch) IsEnabled(feature string) bool {
    d.mu.RLock()
    defer d.mu.RUnlock()
    return d.switches[feature]
}

func GetRecommendations(userID string) []Product {
    if degradeSwitch.IsEnabled("recommendation") {
        // 降级：返回热门商品缓存
        return getHotProductsFromCache()
    }
    // 正常：调用推荐服务
    return recommendationService.GetForUser(userID)
}
```

## 二、降级实践

### 4. 自动降级如何实现？

**答：**

```
自动降级触发链：

监控指标 → 判断条件 → 触发降级 → 执行回退 → 通知

具体实现：

1. 基于错误率自动降级
   if error_rate > 5% for 2min:
     activate_degrade("search")
   if error_rate < 1% for 5min:
     deactivate_degrade("search")

2. 基于 SLO 自动降级
   if slo_budget_remaining < 20%:
     activate_degrade("non_essential_features")
   
3. 基于延迟自动降级
   if p99_latency > 3x_baseline for 3min:
     activate_degrade("heavy_features")

注意事项：
  ┌────────────────────────────────────────┐
  │ 1. 降级后不要立即恢复（防止抖动）       │
  │ 2. 恢复条件应比触发条件更严格           │
  │ 3. 降级操作本身不能失败                 │
  │ 4. 降级时必须有告警通知                 │
  │ 5. 定期演练确保降级逻辑正确             │
  └────────────────────────────────────────┘
```

### 5. 面试题：如何对降级策略做测试？

**答：**

```
降级测试方法：

1. 单元测试
   - 模拟依赖服务不可用
   - 验证降级逻辑被正确触发
   - 验证返回的降级数据正确

2. 集成测试
   - 在测试环境注入故障
   - 验证降级开关工作正常
   - 验证降级恢复流程

3. 混沌测试
   - 在预发环境关闭依赖服务
   - 观察系统是否正确降级
   - 测量降级对用户体验的影响

4. 大促前演练
   - 手动触发各级降级开关
   - 验证降级后核心功能正常
   - 验证降级恢复流程
```

## 三、面试高频题

### 6. 面试题：降级和兜底有什么区别？

**答：**

```
降级（Degradation）：
  主动放弃部分功能以保全整体
  例：关闭推荐模块

兜底（Fallback）：
  某功能失败时提供替代结果
  例：推荐服务超时 → 返回缓存的热门商品

关系：
  降级策略中通常包含兜底方案
  
  if 推荐服务.call() 失败:
      return 兜底数据        ← 这是兜底
  
  if 系统压力大:
      关闭推荐服务          ← 这是降级
      所有请求直接走兜底
```

### 7. 面试题：大促场景如何做降级预案？

**答：**

```
大促降级预案方法论：

1. 功能分级
   P0: 浏览、加购、下单、支付（不能降级）
   P1: 搜索、库存查询（可降级为缓存）
   P2: 推荐、评论、积分（可关闭）
   P3: 客服、帮助中心（可延迟）

2. 预案表
   ┌─────────────┬───────┬────────────┐
   │ 触发条件     │ 操作  │ 预期效果    │
   ├─────────────┼───────┼────────────┤
   │ CPU>80%     │ 关P2  │ 降低30%负载 │
   │ DB延迟>1s   │ 缓存化│ 减少DB查询  │
   │ 错误率>5%   │ 关P2P3│ 降低50%负载 │
   │ 支付异常    │ 排队  │ 削峰填谷    │
   └─────────────┴───────┴────────────┘

3. 演练验证
   大促前 1 周逐一验证每个降级开关
```
