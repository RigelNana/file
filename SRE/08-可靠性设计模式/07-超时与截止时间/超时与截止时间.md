## 一、超时基础

### 1. 为什么分布式系统中超时设置至关重要？

**答：**

```
没有超时的后果：
  Client ──请求──→ Server（卡住）
  Client 一直等待...
  线程/连接被占用...
  更多请求堆积...
  连接池耗尽...
  系统崩溃 💥

超时是分布式系统的"安全阀"：
  ┌────────────────────────────────────────┐
  │ 1. 防止请求无限等待                     │
  │ 2. 释放资源（连接/线程/内存）           │
  │ 3. 使故障可检测（超时即失败信号）        │
  │ 4. 为重试提供基础                       │
  │ 5. 控制用户等待时间                     │
  └────────────────────────────────────────┘
```

### 2. 超时有哪些类型？

**答：**

| 类型 | 说明 | 默认值建议 |
|------|------|-----------|
| **连接超时** | TCP 建立连接的超时 | 1-3s |
| **读超时** | 等待服务端响应的超时 | 根据 SLO |
| **写超时** | 发送请求数据的超时 | 5-10s |
| **空闲超时** | 连接空闲多久被关闭 | 60-300s |
| **总超时** | 单次请求从开始到结束 | 含重试的总时间 |

```
超时层次：

Client
  ├── 连接超时 (1s)      ← TCP 握手
  ├── TLS 超时 (2s)      ← TLS 协商
  ├── 请求超时 (5s)      ← 等待响应
  └── 总超时 (10s)       ← 包含所有重试

调用链超时：
  A ───(5s)──→ B ───(3s)──→ C ───(1s)──→ D
  
  总超时应该 > ∑ 各层超时
  但每层超时应递减（越深越短）
```

### 3. 如何设置合理的超时时间？

**答：**

```
超时设置方法：

方法一：基于 SLO
  SLO: P99 延迟 < 200ms
  超时设置: 200ms × 3 = 600ms（留 3 倍余量）

方法二：基于历史数据
  查看 P99.9 延迟
  超时 = P99.9 × 1.5（覆盖绝大多数正常请求）

方法三：基于用户体验
  用户最多等待 3 秒
  减去前端渲染 0.5s
  后端总超时 ≤ 2.5s

常见服务超时参考值：
  ┌───────────────────────────────────┐
  │ 内部 API 调用:    200ms - 1s     │
  │ 数据库查询:       500ms - 2s     │
  │ Redis 查询:       50ms - 200ms   │
  │ 外部 API:         2s - 10s       │
  │ 文件上传:         30s - 120s     │
  │ 批处理任务:       按需设定        │
  └───────────────────────────────────┘

超时设置反模式：
  ✗ 不设超时（最危险）
  ✗ 超时设得太长（30s）→ 资源占用
  ✗ 所有服务用同一个超时值
  ✗ 下游超时 > 上游超时
```

## 二、Deadline 传播

### 4. 什么是 Deadline 传播？如何实现？

**答：**

```
Deadline = 请求的绝对截止时间

传播机制：
  Client (Deadline: T+5s)
    → A (remaining: 5s)
      → B (remaining: 4s, 已用 1s)
        → C (remaining: 2s, 已用 3s)

每一层都检查：
  if time.Now() > deadline:
      return DeadlineExceeded  // 不再往下调用

好处：
  1. 避免"无用功"（上游已超时，下游还在处理）
  2. 全链路超时一致性
  3. 资源及时释放
```

```go
// Go + gRPC Deadline 传播（原生支持）
func (s *Server) GetOrder(ctx context.Context, req *pb.OrderRequest) (*pb.Order, error) {
    // ctx 自动携带 Deadline
    
    // 检查是否已超时
    if deadline, ok := ctx.Deadline(); ok {
        if time.Now().After(deadline) {
            return nil, status.Error(codes.DeadlineExceeded, "deadline exceeded")
        }
    }
    
    // 调用下游时传递 ctx（Deadline 自动传播）
    payment, err := paymentClient.GetPayment(ctx, &pb.PaymentRequest{
        OrderId: req.Id,
    })
    if err != nil {
        return nil, err
    }
    
    return &pb.Order{Payment: payment}, nil
}
```

```python
# HTTP Deadline 传播（通过 Header）
import time
import requests

def call_with_deadline(url, deadline):
    remaining = deadline - time.time()
    if remaining <= 0:
        raise TimeoutError("deadline already exceeded")
    
    headers = {
        "X-Request-Deadline": str(deadline),
    }
    return requests.get(url, timeout=remaining, headers=headers)
```

### 5. 超时与重试如何配合？

**答：**

```
正确的超时 + 重试配置：

单次请求超时: 1s
最大重试次数: 3
退避: 100ms, 200ms, 400ms
总 Deadline: 3s

时间线：
  T+0.0s: 第 1 次请求（超时 1s）
  T+1.0s: 超时失败
  T+1.1s: 等待 100ms 后第 2 次请求
  T+2.1s: 超时失败
  T+2.3s: 等待 200ms 后第 3 次请求
  T+3.0s: 总 Deadline 到达 → 放弃

关键原则：
  ┌─────────────────────────────────────────┐
  │ 总 Deadline > 单次超时 × 重试次数 + 退避 │
  │ 但不能太长，要考虑用户耐心               │
  │                                          │
  │ 每次重试时重算剩余时间：                  │
  │ remaining = deadline - now               │
  │ if remaining < 单次超时:                 │
  │     单次超时 = remaining                 │
  └─────────────────────────────────────────┘
```

## 三、面试高频题

### 6. 面试题：连接超时和读超时应该怎么设？

**答：**

```
连接超时（Connect Timeout）：
  "TCP 握手要多久"
  通常 1-3 秒足够
  如果超过 → 网络问题或服务端 Down

读超时（Read Timeout）：
  "连上了之后，等响应要多久"
  取决于接口复杂度
  
  快接口（缓存查询）: 200ms
  中接口（DB 操作）: 1-2s
  慢接口（聚合计算）: 5-10s

常见错误：
  ✗ 连接超时 = 读超时 = 30s
    原因：如果服务端 Down，每个请求要等 30s
    
  ✓ 连接超时 = 1s, 读超时按接口设置
    原因：快速发现连接问题，分别控制
```

### 7. 面试题：如何处理超时引起的数据不一致？

**答：**

```
场景：
  Client → 转账服务 (超时) → 返回错误
  但实际转账已成功（服务端处理完了，但响应丢了）

解决方案：

1. 幂等设计
   每个请求携带唯一 ID
   服务端去重：同一 ID 只处理一次

2. 最终一致性
   异步对账系统定期检查
   发现不一致自动修复

3. 两阶段确认
   Client → 预扣款（锁定） → 确认/取消
   超时时：锁定自动释放

4. 查询确认
   超时后先查询结果
   if 已成功 → 不重试
   if 未找到 → 重试
```
