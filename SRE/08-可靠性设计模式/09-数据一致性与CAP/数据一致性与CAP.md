# 数据一致性与 CAP 八股文

---

## 一、CAP 定理

### 1. 什么是 CAP 定理？

**答：**

```
CAP 定理（Brewer 定理）：
  分布式系统最多只能同时满足以下三项中的两项：

  C - Consistency（一致性）
      所有节点看到的数据相同
      
  A - Availability（可用性）
      每个请求都能收到响应（非错误）
      
  P - Partition Tolerance（分区容忍）
      网络分区时系统继续运行

为什么只能选两个：
  网络分区（P）在分布式系统中不可避免
  → 实际上是在 C 和 A 之间选择

  ┌─────────────────────────────────────┐
  │ CP 系统：分区时保证一致性，牺牲可用性│
  │ 例：ZooKeeper, etcd, HBase         │
  │                                     │
  │ AP 系统：分区时保证可用性，牺牲一致性│
  │ 例：Cassandra, DynamoDB, Redis     │
  └─────────────────────────────────────┘
```

### 2. CAP 在实际系统设计中怎么应用？

**答：**

| 场景 | 选择 | 原因 | 代表系统 |
|------|------|------|----------|
| 金融转账 | CP | 数据正确性优先 | ZooKeeper |
| 电商商品浏览 | AP | 可用性优先 | Cassandra |
| 配置中心 | CP | 配置一致性关键 | etcd |
| 社交动态 | AP | 短暂不一致可接受 | DynamoDB |
| 分布式锁 | CP | 锁必须一致 | Redis (Redlock) |

```
实际不是非黑即白：

PACELC 扩展：
  如果有 Partition (P)：选 A 还是 C？
  Else (E)：选 Latency (L) 还是 Consistency (C)？

  PA/EL：高可用 + 低延迟（最终一致）
  PC/EC：强一致（高延迟）
  PA/EC：分区时可用，正常时一致
```

## 二、一致性模型

### 3. 常见的一致性模型有哪些？

**答：**

```
一致性强度从强到弱：

强一致性 (Strong Consistency)
│  所有节点在任何时刻看到的数据完全一致
│  实现：两阶段提交、Paxos/Raft
│  代价：高延迟
│
线性一致性 (Linearizability)
│  操作按全局时间排序，像单机一样
│  实现：Raft (etcd)
│
顺序一致性 (Sequential Consistency)
│  所有操作按某个全局顺序执行
│  每个客户端的操作顺序被保持
│
因果一致性 (Causal Consistency)
│  有因果关系的操作顺序被保持
│  无因果关系的操作可能乱序
│
最终一致性 (Eventual Consistency)
│  如果没有新的写入，最终所有节点一致
│  实现：DNS、CDN 缓存
│  延迟最低
▼
```

### 4. 最终一致性在 SRE 实践中如何处理？

**答：**

```
最终一致性的挑战：
  写入节点 A → 读取节点 B → 可能读到旧数据

SRE 需要关注的问题：

1. 复制延迟监控
   promql: replication_lag_seconds
   告警: lag > 5s → 调查原因

2. 一致性 SLI
   "99% 的读操作在写入后 1 秒内读到新值"

3. 读己之写（Read Your Writes）
   用户写入后立即读取自己的数据
   解决：写入后从主节点读取

4. 冲突解决
   多主复制下同时写入可能冲突
   策略：
   ├── Last-Write-Wins (LWW)
   ├── 向量时钟
   └── CRDT（无冲突复制数据类型）
```

## 三、一致性实践

### 5. SRE 如何在一致性和可用性之间做权衡？

**答：**

```
按业务场景选择：

强一致场景：
  ├── 账户余额     → 不能多扣/多加
  ├── 分布式锁     → 不能多人同时持锁
  ├── 配置变更     → 所有节点必须一致
  └── 库存扣减     → 不能超卖

最终一致场景：
  ├── 用户头像更新  → 几秒延迟可接受
  ├── 商品评论     → 不影响交易
  ├── 浏览记录     → 最终一致即可
  └── 通知消息     → 延迟可接受

混合策略：
  ┌──────────────────────────────────────┐
  │ 核心交易链路：强一致（CP）            │
  │ 非核心查询：  最终一致（AP）          │
  │ 同一系统中不同路径用不同一致性级别    │
  └──────────────────────────────────────┘
```

### 6. 分布式事务的常见方案？

**答：**

| 方案 | 一致性 | 性能 | 复杂度 | 适用场景 |
|------|--------|------|--------|----------|
| **2PC** | 强一致 | 低 | 中 | 数据库事务 |
| **TCC** | 强一致 | 中 | 高 | 金融转账 |
| **Saga** | 最终一致 | 高 | 中 | 长事务 |
| **本地消息表** | 最终一致 | 高 | 低 | 异步一致 |
| **事务消息** | 最终一致 | 高 | 中 | RocketMQ |

```
Saga 模式示例（电商下单）：

正向操作：
  创建订单 → 扣库存 → 扣款 → 发货

补偿操作（任何一步失败时反向执行）：
  取消发货 ← 退款 ← 回滚库存 ← 取消订单

优势：
  ├── 不需要长时间锁定资源
  ├── 每步是独立的本地事务
  └── 适合微服务架构

挑战：
  ├── 需要设计补偿逻辑
  ├── 中间状态对外可见
  └── 补偿操作本身也要幂等
```

## 四、面试高频题

### 7. 面试题：如何检测分布式系统中的数据不一致？

**答：**

```
检测方法：

1. 主动对账
   ├── 定时比对各节点/系统的数据
   ├── 发现不一致自动修复或告警
   └── 例：支付对账（每天与银行对账）

2. 一致性检查工具
   ├── pt-table-checksum（MySQL 主从一致性）
   ├── 自研 Checksum 比对工具
   └── 读修复（Read Repair）

3. 监控指标
   ├── 主从复制延迟
   ├── 数据版本差异
   └── 冲突计数

4. 混沌测试
   ├── 注入网络分区
   ├── 验证分区恢复后数据一致
   └── 减为单副本再恢复
```

### 8. 面试题：CAP 中的 P（分区容忍）能不能放弃？

**答：**

```
理论上不能：
  网络分区在分布式系统中不可避免。
  即使在同一数据中心，也可能出现：
  ├── 交换机故障
  ├── 网线松了
  ├── 防火墙配置错误
  └── 内核 bug 导致网络异常

如果放弃 P（单机系统）：
  可以同时有 C 和 A
  但失去了分布式系统的优势（容灾/扩展）

实际做法：
  接受 P 存在 → 在 C 和 A 之间权衡
  
  更精确的理解：
  "大多数时候网络正常时，C 和 A 都可以满足。
   分区发生时（极少），必须做出选择。"
```
