# 故障域与爆炸半径八股文

---

## 一、概念基础

### 1. 什么是故障域（Failure Domain）？

**答：**

```
故障域 = 一个故障能影响的最大范围

故障域层次（从小到大）：
┌──────────────────────────────────┐
│ Level 1: 进程/容器                │
│  一个 Pod 崩溃                    │
├──────────────────────────────────┤
│ Level 2: 节点/主机                │
│  一台服务器宕机                   │
├──────────────────────────────────┤
│ Level 3: 机架                    │
│  交换机/电源故障                  │
├──────────────────────────────────┤
│ Level 4: 可用区 (AZ)             │
│  数据中心部分设施故障              │
├──────────────────────────────────┤
│ Level 5: 区域 (Region)           │
│  整个数据中心/城市级故障           │
├──────────────────────────────────┤
│ Level 6: 云服务商                │
│  全球性服务故障                   │
└──────────────────────────────────┘

设计原则：
  关键服务跨越多个故障域
  任一故障域失效不影响整体可用性
```

### 2. 什么是爆炸半径（Blast Radius）？

**答：**

```
爆炸半径 = 一次变更或故障能影响的最大范围

控制爆炸半径的策略：

1. 分区（Cell/Shard Architecture）
   用户 → Cell 1 (10%)
   用户 → Cell 2 (10%)
   ...
   用户 → Cell 10 (10%)
   
   一个 Cell 故障 → 最多影响 10%

2. 灰度发布
   新版本先给 1% → 5% → 20% → 100%
   及时发现问题时只影响小比例

3. Feature Flag
   新功能先对内部用户开放
   逐步扩大范围

4. 多区域部署
   Region A + Region B + Region C
   单 Region 故障 → 流量切到其他

爆炸半径量化：
  最差情况影响比例 < 100%
  目标：任何单一故障影响 < 10% 用户
```

### 3. Cell 架构如何限制爆炸半径？

**答：**

```
Cell 架构（单元化架构）：

传统架构：
  所有用户 → 同一套服务
  故障 = 100% 影响

Cell 架构：
  ┌────────┐  ┌────────┐  ┌────────┐
  │ Cell 1 │  │ Cell 2 │  │ Cell 3 │
  │ 用户A-H│  │ 用户I-P│  │ 用户Q-Z│
  │ ┌App─┐ │  │ ┌App─┐ │  │ ┌App─┐ │
  │ │Cache│ │  │ │Cache│ │  │ │Cache│ │
  │ │ DB  │ │  │ │ DB  │ │  │ │ DB  │ │
  │ └────┘ │  │ └────┘ │  │ └────┘ │
  └────────┘  └────────┘  └────────┘
  
  Cell 2 故障 → 只影响用户 I-P（33%）

Cell 路由：
  路由层根据用户 ID 分配到对应 Cell
  每个 Cell 是独立的小型完整系统
  Cell 间不共享状态

适用场景：
  社交网络（按用户分片）
  SaaS 平台（按租户分片）
  支付系统（按商户分片）
```

## 二、实践

### 4. 在 K8s 中如何利用故障域？

**答：**

```yaml
# 跨 AZ 部署 — 利用拓扑域

# 方式1: Pod 反亲和
affinity:
  podAntiAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchLabels:
            app: critical-service
        topologyKey: topology.kubernetes.io/zone

# 方式2: 拓扑分散约束（推荐）
topologySpreadConstraints:
  - maxSkew: 1
    topologyKey: topology.kubernetes.io/zone
    whenUnsatisfiable: DoNotSchedule
    labelSelector:
      matchLabels:
        app: critical-service
  - maxSkew: 1
    topologyKey: kubernetes.io/hostname
    whenUnsatisfiable: ScheduleAnyway
    labelSelector:
      matchLabels:
        app: critical-service
```

```
K8s 故障域映射：
  Node → topology.kubernetes.io/hostname
  Rack → topology.kubernetes.io/rack (自定义)
  AZ   → topology.kubernetes.io/zone
  Region → topology.kubernetes.io/region

部署策略：
  3 副本 → 分布在 3 个 AZ
  6 副本 → 每个 AZ 2 个，不同 Node
```

### 5. 如何评估和减小爆炸半径？

**答：**

```
爆炸半径评估清单：

对于每个变更/故障场景，评估：
  □ 影响多少用户？（百分比）
  □ 影响多少服务？（依赖链）
  □ 影响多长时间？（恢复时间）
  □ 是否有降级方案？

减小爆炸半径的方法：
┌────────────────────┬──────────────────┐
│ 方法                │ 效果              │
├────────────────────┼──────────────────┤
│ 多 AZ 部署          │ AZ 故障不影响     │
│ Cell 架构           │ 限制影响百分比    │
│ 金丝雀发布          │ 变更只影响小比例  │
│ Feature Flag       │ 功能级别控制范围  │
│ 限流/熔断          │ 防止级联扩大      │
│ 服务隔离           │ 故障不跨服务传播  │
│ 多集群             │ 集群故障不影响    │
│ 多云               │ 云商故障不影响    │
└────────────────────┴──────────────────┘
```

## 三、面试

### 6. 面试题：如何设计一个最大爆炸半径 < 5% 的系统？

**答：**

```
目标：任何单一故障影响 < 5% 用户

方案：20+ Cell 架构

1. 用户分片
   按 user_id % 20 分配到 20 个 Cell
   每个 Cell 承载 5% 用户

2. 每个 Cell 独立
   独立的应用实例
   独立的数据库
   独立的缓存
   不共享有状态组件

3. 全局层
   DNS/LB 全局（但有冗余）
   路由层决定 Cell 分配
   认证服务可以共享（但有降级）

4. 共享组件高可用
   路由层: 多区域 + 本地缓存
   认证: 多副本 + Token 缓存
   配置: 本地缓存 + 定期同步

5. 验证
   混沌工程随机关闭单个 Cell
   确认影响限制在 5% 以内
```

### 7. 面试题：故障域隔离和服务隔离的关系？

**答：**

```
两个维度的隔离：

故障域隔离（物理维度）：
  跨 AZ、跨 Region 部署
  确保物理故障不影响全局

服务隔离（逻辑维度）：
  服务间边界清晰
  A 服务故障不拖垮 B 服务

两者结合：
  ┌──── AZ-1 ────┐  ┌──── AZ-2 ────┐
  │ ServiceA (1)  │  │ ServiceA (2)  │
  │ ServiceB (1)  │  │ ServiceB (2)  │
  │ ┌─熔断──┐    │  │ ┌─熔断──┐    │
  │ │A 不影响 B│  │  │ │A 不影响 B│  │
  │ └────────┘    │  │ └────────┘    │
  └───────────────┘  └───────────────┘
  
  AZ-1 故障 → AZ-2 接管（故障域隔离）
  ServiceA 故障 → ServiceB 不受影响（服务隔离）

完整的隔离策略：
  物理: 多 AZ + 多 Region
  网络: NetworkPolicy + ServiceMesh
  逻辑: 熔断 + 限流 + 降级
  数据: 分库分表 + 独立存储
```
