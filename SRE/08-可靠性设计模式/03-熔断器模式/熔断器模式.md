## 一、熔断器原理

### 1. 什么是熔断器模式？

**答：**

```
熔断器 (Circuit Breaker) = 防止级联故障的保护机制

类比：家用电路保险丝
  电流过大 → 保险丝熔断 → 保护电路

软件中：
  下游故障 → 熔断器打开 → 快速失败 → 保护上游

三种状态：
  ┌───────────┐      连续失败达阈值     ┌───────────┐
  │   关闭     │ ──────────────────→   │   打开     │
  │  (正常)    │                        │  (快速失败) │
  │           │ ←──────────────────    │           │
  └───────────┘     探测成功恢复        └─────┬─────┘
        ↑                                     │
        │          冷却时间到                   │
        │           ┌───────────┐              │
        └───────────│   半开     │←─────────────┘
           探测成功  │ (少量请求) │
                    └───────────┘
                      探测失败 → 回到打开状态

参数：
  failure_threshold: 触发打开的失败次数（如 5 次）
  reset_timeout: 打开后多久进入半开（如 30 秒）
  success_threshold: 半开状态恢复需要的成功次数
```

### 2. 熔断器的实现原理？

**答：**

```go
// 简化的熔断器实现
type CircuitBreaker struct {
    state        State  // Closed/Open/HalfOpen
    failures     int
    successes    int
    threshold    int
    resetTimeout time.Duration
    lastFailTime time.Time
    mu           sync.Mutex
}

func (cb *CircuitBreaker) Execute(fn func() error) error {
    cb.mu.Lock()
    
    switch cb.state {
    case Open:
        if time.Since(cb.lastFailTime) > cb.resetTimeout {
            cb.state = HalfOpen
            cb.mu.Unlock()
            return cb.tryRequest(fn)
        }
        cb.mu.Unlock()
        return ErrCircuitOpen // 快速失败
        
    case HalfOpen:
        cb.mu.Unlock()
        return cb.tryRequest(fn)
        
    case Closed:
        cb.mu.Unlock()
        return cb.tryRequest(fn)
    }
    
    return nil
}

func (cb *CircuitBreaker) tryRequest(fn func() error) error {
    err := fn()
    cb.mu.Lock()
    defer cb.mu.Unlock()
    
    if err != nil {
        cb.failures++
        cb.lastFailTime = time.Now()
        if cb.failures >= cb.threshold {
            cb.state = Open
        }
        return err
    }
    
    cb.successes++
    if cb.state == HalfOpen && cb.successes >= 3 {
        cb.state = Closed
        cb.failures = 0
    }
    return nil
}
```

### 3. 熔断策略有哪些维度？

**答：**

```
熔断触发策略：

1. 失败次数
   连续 N 次失败 → 触发
   简单直接但对突发敏感

2. 失败率
   滑动窗口内失败率 > X%
   更稳定，推荐使用
   例: 10秒内失败率 > 50%

3. 慢调用率
   响应时间 > 阈值的请求占比 > X%
   例: P99 > 2s 的请求 > 60%

4. 异常类型
   只对特定异常熔断
   例: 5xx 触发，4xx 不触发

熔断恢复策略：
  线性恢复: 半开状态放固定少量请求
  渐进恢复: 逐步增加放行比例 1%→5%→20%→100%
  探针恢复: 定期发探测请求验证

推荐：
  触发 = 失败率 + 慢调用率
  恢复 = 渐进式放量
```

## 二、实践

### 4. Istio 中如何配置熔断？

**答：**

```yaml
# Istio DestinationRule 配置熔断
apiVersion: networking.istio.io/v1alpha3
kind: DestinationRule
metadata:
  name: payment-api
spec:
  host: payment-api
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100       # 最大连接数
      http:
        h2UpgradePolicy: DEFAULT
        http1MaxPendingRequests: 100  # 最大等待请求
        http2MaxRequests: 1000        # 最大并发请求
        maxRequestsPerConnection: 10  # 每连接最大请求
        maxRetries: 3                 # 最大重试
    
    outlierDetection:
      consecutive5xxErrors: 5     # 连续 5 个 5xx
      interval: 10s               # 检测间隔
      baseEjectionTime: 30s       # 驱逐基础时间
      maxEjectionPercent: 50      # 最多驱逐 50% 实例
```

```
Istio 熔断工作原理：

1. 连接池限制
   超过 maxConnections → 排队
   超过 maxPendingRequests → 拒绝（503）

2. 异常点驱逐
   连续 5xx > 5 次 → 该实例被驱逐 30s
   驱逐期间不接收流量
   最多驱逐 50%（保证最低服务能力）
```

### 5. 应用层熔断器（如 resilience4j/gobreaker）？

**答：**

```go
// Go gobreaker 示例
import "github.com/sony/gobreaker"

cb := gobreaker.NewCircuitBreaker(gobreaker.Settings{
    Name:        "payment-api",
    MaxRequests: 3,              // 半开最大请求
    Interval:    10 * time.Second, // 统计窗口
    Timeout:     30 * time.Second, // 打开后超时
    ReadyToTrip: func(counts gobreaker.Counts) bool {
        failureRatio := float64(counts.TotalFailures) /
            float64(counts.Requests)
        return counts.Requests >= 10 && failureRatio >= 0.5
    },
    OnStateChange: func(name string, from, to gobreaker.State) {
        log.Printf("CircuitBreaker %s: %s -> %s", name, from, to)
        // 发送指标/告警
    },
})

// 使用
result, err := cb.Execute(func() (interface{}, error) {
    return callPaymentAPI()
})
```

## 三、面试

### 6. 面试题：熔断和降级的区别？

**答：**

```
熔断 vs 降级：

熔断 (Circuit Breaker):
  → 自动行为
  → 检测到下游故障 → 自动停止请求
  → 保护调用方不被拖垮
  → "不再调你了"

降级 (Degradation):
  → 主动策略
  → 服务不可用时返回备选结果
  → 保证用户有基本体验
  → "给你一个替代方案"

关系：
  熔断触发 → 执行降级逻辑
  
  正常: A → B (完整功能)
  熔断: A → 本地缓存/默认值 (降级功能)

示例：
  商品推荐服务熔断后：
  → 不再调推荐服务（熔断）
  → 返回热门商品列表（降级）
```

### 7. 面试题：熔断器参数如何调优？

**答：**

```
熔断器调优策略：

过于敏感（误触发）：
  症状: 偶尔几个超时就触发熔断
  调整: 
    → 增大失败阈值
    → 增大统计窗口
    → 排除慢请求只看 5xx

过于迟钝（不触发）：
  症状: 下游已经崩了还在发请求
  调整:
    → 减小失败阈值
    → 减小统计窗口
    → 加入慢调用检测

恢复太慢：
  症状: 下游已恢复但流量一直很少
  调整:
    → 减小 reset timeout
    → 使用渐进式恢复

推荐初始参数：
  统计窗口: 10-30 秒
  失败率阈值: 50%
  最小请求数: 10（避免少量请求误触发）
  打开超时: 30-60 秒
  半开请求数: 3-5
  
  然后根据实际监控数据调整
```
