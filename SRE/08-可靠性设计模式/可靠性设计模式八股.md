# 可靠性设计模式八股文

---

## 一、高可用架构

### 1. 什么是高可用（High Availability）？如何计算可用性？

**答：**

$$\text{可用性} = \frac{MTTF}{MTTF + MTTR} = \frac{\text{正常运行时间}}{\text{总时间}}$$

| 术语 | 全称 | 含义 |
|------|------|------|
| **MTTF** | Mean Time To Failure | 平均无故障时间 |
| **MTTR** | Mean Time To Recovery | 平均恢复时间 |
| **MTBF** | Mean Time Between Failures | 平均故障间隔 = MTTF + MTTR |

**提高可用性的两个方向：**
- 增大 MTTF → 减少故障发生（冗余、测试、变更管理）
- 减小 MTTR → 加快故障恢复（自动化、Runbook、回滚）

### 2. 什么是故障域和爆炸半径？

**答：**

- **故障域（Failure Domain）**：共享同一故障源的资源集合
  - 示例：同一个机架、同一个可用区、同一个区域
- **爆炸半径（Blast Radius）**：一次故障能影响的最大范围

**缩小爆炸半径的策略：**
1. 多可用区/多区域部署
2. 微服务隔离（故障不传播）
3. 分片（Sharding）：一个分片故障不影响其他
4. 细胞架构（Cell-based Architecture）：独立的部署单元
5. 渐进式发布：限制变更的影响范围

## 二、弹性设计模式

### 3. 熔断器模式（Circuit Breaker）是什么？

**答：** 熔断器防止一个服务的失败级联传播到整个系统。

```
    关闭（正常）──── 失败达到阈值 ────→ 打开（熔断）
        ↑                                    │
        │                               超时后尝试
        │                                    │
        └──── 探测请求成功 ←──── 半开（探测）
```

三种状态：
- **关闭（Closed）**：正常转发请求
- **打开（Open）**：直接返回错误/降级响应，不转发
- **半开（Half-Open）**：允许少量探测请求，成功则关闭

### 4. 限流策略有哪些？

**答：**

| 算法 | 原理 | 特点 |
|------|------|------|
| **固定窗口** | 固定时间窗口内计数 | 简单，但有边界突发问题 |
| **滑动窗口** | 滑动时间窗口计数 | 更精确，计算量略大 |
| **令牌桶** | 以固定速率生成令牌 | 允许突发（桶容量） |
| **漏桶** | 以固定速率处理请求 | 严格匀速，平滑流量 |

### 5. 重试策略的最佳实践是什么？

**答：**

| 策略 | 说明 |
|------|------|
| **指数退避** | 重试间隔指数增长：1s → 2s → 4s → 8s |
| **抖动（Jitter）** | 在退避基础上加随机偏移，避免重试风暴 |
| **最大重试次数** | 限制重试次数，防止无限重试 |
| **幂等检查** | 确保重试的请求是幂等的 |
| **重试预算** | 全局限制重试比例（如不超过总请求的 10%） |
| **区分错误类型** | 只对可重试错误重试（5xx），不重试客户端错误（4xx） |

### 6. 什么是优雅降级？

**答：** 在系统过载或部分故障时，**主动降低非核心功能质量**以保障核心功能可用。

**降级策略示例：**
- 电商大促：关闭推荐系统、使用缓存数据、简化页面
- 搜索引擎：返回缓存的旧结果而非实时计算
- 社交平台：暂停非核心功能（如用户在线状态）
- API 服务：返回部分数据而非完全失败
