## 一、RCA 方法论

### 1. 什么是 RCA？在 SRE 中为什么重要？

**答：**

```
RCA (Root Cause Analysis) = 根因分析

目的：
  不是找"谁的错"
  而是找"系统哪里有缺陷"

RCA 在 SRE 中的定位：
  事件响应 → 恢复服务 → RCA → 改进系统

RCA 的层次：
  ┌─────────────────────────────────────┐
  │ 表面原因: 部署了有 Bug 的代码        │
  ├─────────────────────────────────────┤
  │ 中间原因: 测试没覆盖该场景          │
  ├─────────────────────────────────────┤
  │ 根本原因: 缺少集成测试流程          │
  ├─────────────────────────────────────┤
  │ 系统原因: 团队没有测试文化和标准     │
  └─────────────────────────────────────┘

RCA 的目标：
  找到根本原因和系统原因
  改进流程和机制
  防止同类问题再次发生
```

### 2. 常用的 RCA 方法有哪些？

**答：**

```
RCA 方法工具箱：

┌──────────────┬──────────────────────┬─────────────────┐
│ 方法          │ 适用场景              │ 特点             │
├──────────────┼──────────────────────┼─────────────────┤
│ 5 Why        │ 单一明确的故障        │ 简单直接         │
│ 鱼骨图        │ 多因素复杂故障        │ 系统化分析       │
│ 故障树分析    │ 安全关键系统          │ 自顶向下逻辑      │
│ 时间线分析    │ 事件序列复杂          │ 还原现场         │
│ 变更分析      │ 变更引发的故障        │ 对比前后差异      │
│ 人因分析      │ 人为操作引发          │ 分析认知因素      │
└──────────────┴──────────────────────┴─────────────────┘

选择建议：
  简单故障 → 5 Why 足够
  复杂故障 → 时间线 + 鱼骨图
  生产事故 → 5 Why + 时间线 + Action Items
```

### 3. RCA 的完整流程？

**答：**

```
RCA 六步流程：

Step 1: 定义问题
  什么出了问题？
  影响范围多大？
  持续了多长时间？

Step 2: 收集数据
  ├── 时间线（什么时间发生了什么）
  ├── 监控数据（指标/日志/追踪）
  ├── 变更记录
  ├── 操作记录
  └── 通信记录（聊天/电话）

Step 3: 重建时间线
  09:00 部署 v2.3
  09:15 异常指标出现
  09:20 告警触发
  09:25 IC 确认事件
  09:35 回滚 v2.3
  09:40 服务恢复

Step 4: 分析原因
  使用 5 Why / 鱼骨图
  区分触发因素和根本原因

Step 5: 制定 Action Items
  每个 Item 必须：
  ├── 可执行（动作明确）
  ├── 有 Owner（责任人）
  ├── 有截止日期
  └── 可验证（如何确认完成）

Step 6: 跟踪关闭
  定期 Review Action Items 进度
  确认改进措施有效
```

## 二、RCA 实践

### 4. 如何写一份好的 RCA 报告？

**答：**

```
RCA 报告模板：

# 事件标题: [P0] 支付服务不可用 45 分钟

## 摘要
2024-01-15 09:15-10:00，支付服务因数据库连接池
耗尽导致全部请求失败，影响约 50 万笔交易。

## 影响
- 持续时间: 45 分钟
- 影响用户: 约 10 万
- 影响交易: 约 50 万笔
- 财务影响: 预估 ¥200 万
- SLO: Error Budget 消耗 30%

## 时间线
(详细的时间线)

## 根因分析
触发因素: 新版本引入了未关闭的 DB 连接
根本原因: 
  1. Code Review 未发现连接泄漏
  2. 缺少连接池使用的集成测试
  3. 连接池监控告警缺失

## Action Items
| # | Action | Owner | 截止日 | 状态 |
|---|--------|-------|-------|------|
| 1 | 添加连接池监控告警 | SRE-张三 | 1/22 | P0 |
| 2 | 添加连接泄漏单测 | Dev-李四 | 1/29 | P1 |
| 3 | Code Review Checklist 加连接管理 | TL-王五 | 1/29 | P2 |

## 经验教训
- 什么做得好
- 什么可以改进
- 哪里有运气成分
```

### 5. Action Item 如何确保落地？

**答：**

```
Action Item 管理：

SMART 原则：
  S - Specific: "添加 DB 连接池告警" 而不是 "改进监控"
  M - Measurable: 可以验证完成
  A - Assignable: 明确 Owner
  R - Relevant: 和根因直接相关
  T - Time-bound: 有截止日期

优先级分类：
  P0: 防止再次发生（必须在 1 周内完成）
  P1: 改进检测能力（2 周内）
  P2: 长期改进（1 个月内）

跟踪机制：
  1. 每周 Postmortem Review 会议
     → 检查所有 Open 的 Action Items
  
  2. 自动化提醒
     → Jira/GitHub Issue 到期提醒
     → Slack Bot 自动跟踪
  
  3. 月度统计
     → Action Item 完成率
     → 超期 Item 数量
     → 同类事件是否再次发生

完成标准：
  ✗ "已部署代码" 不算完成
  ✓ "已部署 + 告警已验证触发 + 文档已更新"
```

## 三、面试

### 6. 面试题：RCA 中常见的陷阱？

**答：**

```
RCA 五大陷阱：

1. 归咎个人
   ✗ "张三点错了按钮"
   ✓ "系统缺少操作确认机制"
   → 人会犯错，系统应容错

2. 停在表面原因
   ✗ "缓存过期导致"
   ✓ 继续问 Why: 为什么缓存策略不合理？
     为什么没有防击穿？

3. 过度追求单一根因
   ✗ 非要找到"那个"根因
   ✓ 多因素共同导致，分别改进
   → 瑞士奶酪模型：多层防御漏洞重合

4. Action Item 空洞
   ✗ "加强监控" "注意代码质量"
   ✓ "为 payment-api 添加连接池 >80% 的告警"

5. 只修不防
   ✗ 只修复了这次的 Bug
   ✓ 建立机制防止同类 Bug
   → "举一反三"
```

### 7. 面试题：如何在无责文化中开展 RCA？

**答：**

```
无责 RCA 实践要点：

1. 语言规范
   ✗ "谁犯了错误"
   ✓ "系统哪里可以改进"
   ✗ "如果当时没有..."
   ✓ "在那个时间点，基于可用信息..."

2. 会议规则
   → 聚焦系统改进，不讨论个人责任
   → 鼓励所有人分享当时的决策依据
   → 假设每个人都是基于善意和当时的信息做决定

3. 管理层承诺
   → 不因诚实分享而受惩罚
   → 不在绩效评估中引用事故
   → 表扬主动报告问题的行为

4. 持续建设
   → 定期分享 RCA 学习
   → 新人培训包含 RCA 案例
   → 建立组织级别的事故知识库
```
