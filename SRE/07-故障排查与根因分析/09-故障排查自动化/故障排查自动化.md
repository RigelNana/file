## 一、自动化排查

### 1. 故障排查自动化的层次模型？

**答：**

```
排查自动化四层模型：

Level 0: 纯手动
  人工登录 → 手动检查 → 凭经验判断
  问题：慢、易遗漏、依赖个人经验

Level 1: 诊断脚本
  预置排查脚本 → 一键收集信息
  例：health-check.sh 检查所有组件状态
  
Level 2: 智能诊断
  自动关联指标/日志/追踪
  告警触发 → 自动收集上下文
  例：告警附带最近变更、相关指标图

Level 3: 自动修复
  检测到问题 → 自动执行修复
  例：Pod OOM → 自动重启 + 扩容
  
Level 4: 预测性
  预测即将发生的问题
  在故障发生前自动处理
```

### 2. 如何构建自动化诊断脚本？

**答：**

```bash
#!/bin/bash
# SRE 快速诊断脚本

SERVICE=$1
NAMESPACE=${2:-production}

echo "=== 服务诊断: $SERVICE ==="
echo "时间: $(date)"
echo ""

# 1. Pod 状态
echo "--- Pod 状态 ---"
kubectl -n $NAMESPACE get pods -l app=$SERVICE \
  -o wide --show-labels

# 2. 最近事件
echo "--- 最近事件 ---"
kubectl -n $NAMESPACE get events \
  --field-selector involvedObject.name=$SERVICE \
  --sort-by='.lastTimestamp' | tail -20

# 3. 资源使用
echo "--- 资源使用 ---"
kubectl -n $NAMESPACE top pods -l app=$SERVICE

# 4. 最近日志错误
echo "--- 最近错误日志 ---"
kubectl -n $NAMESPACE logs -l app=$SERVICE \
  --tail=50 --since=5m | grep -i "error\|fatal\|panic"

# 5. 端点健康
echo "--- Endpoints ---"
kubectl -n $NAMESPACE get endpoints $SERVICE

# 6. 最近部署
echo "--- 部署历史 ---"
kubectl -n $NAMESPACE rollout history deployment/$SERVICE

echo ""
echo "=== 诊断完成 ==="
```

### 3. 告警自动上下文收集如何实现？

**答：**

```yaml
# Alertmanager webhook → 自动诊断服务
route:
  receiver: auto-diagnosis
  
receivers:
  - name: auto-diagnosis
    webhook_configs:
      - url: 'http://diagnosis-service:8080/alert'
        send_resolved: true
```

```python
# 自动诊断服务
from flask import Flask, request
import subprocess

app = Flask(__name__)

@app.route('/alert', methods=['POST'])
def handle_alert(self):
    alerts = request.json.get('alerts', [])
    
    for alert in alerts:
        service = alert['labels'].get('service')
        alert_name = alert['labels'].get('alertname')
        
        # 收集诊断信息
        context = collect_diagnosis(service)
        
        # 发送到 Slack（包含上下文）
        send_to_slack(
            channel='#incidents',
            title=f'🚨 {alert_name} - {service}',
            fields=[
                ('最近变更', context['recent_deploys']),
                ('Pod 状态', context['pod_status']),
                ('错误日志', context['error_logs']),
                ('相关指标', context['grafana_url']),
            ]
        )

def collect_diagnosis(service):
    return {
        'recent_deploys': get_recent_deploys(service),
        'pod_status': get_pod_status(service),
        'error_logs': get_recent_errors(service),
        'grafana_url': build_grafana_url(service),
    }
```

## 二、自动修复

### 4. Auto-Remediation 自动修复如何设计？

**答：**

```
自动修复设计原则：

1. 只自动化已知问题的已知修复
   ✓ OOM → 重启 Pod
   ✓ 证书即将过期 → 自动续期
   ✗ 未知错误 → 需要人工判断

2. 修复操作必须安全且幂等
   ✓ 重启 Pod（K8s 自动处理）
   ✓ 清理磁盘临时文件
   ✗ 删除数据库记录

3. 有次数限制和冷却期
   同一修复操作 1 小时内最多 3 次
   超过限制 → 升级到人工

常见自动修复场景：
┌───────────────────┬─────────────────────┐
│ 检测条件           │ 自动修复动作         │
├───────────────────┼─────────────────────┤
│ Pod OOM Killed    │ 重启 + 增大 limits  │
│ 磁盘 >90%         │ 清理日志/临时文件    │
│ 证书 <30天过期     │ 自动续期            │
│ 连接数 >80%       │ 重启连接池          │
│ 节点 NotReady     │ Cordon + 通知       │
│ 5xx >5%           │ 回滚到上个版本       │
└───────────────────┴─────────────────────┘
```

### 5. K8s 自带的自修复机制有哪些？

**答：**

```
K8s 内置自修复：

1. Pod 重启策略
   restartPolicy: Always
   → 容器崩溃后自动重启
   → 退避时间指数增长

2. Liveness Probe
   → 探测失败 → 重启容器
   → 处理死锁/卡死

3. ReplicaSet 控制器
   → 维持期望副本数
   → Pod 被删除/节点故障 → 自动重建

4. Node 自愈
   → 节点 NotReady → Pod 被驱逐
   → 在其他节点重建

5. HPA 自动扩缩
   → 负载增加 → 自动扩容
   → 负载降低 → 自动缩容

6. PDB 保护
   → Disruption Budget
   → 确保最少可用 Pod 数

局限性：
  - 只能处理 Pod/节点级故障
  - 应用层逻辑故障无法处理
  - 需要配合自定义 Operator
```

## 三、面试

### 6. 面试题：如何设计一个 Runbook 自动化系统？

**答：**

```
Runbook 自动化系统设计：

架构：
  告警 → 匹配 Runbook → 执行步骤 → 验证 → 通知

核心组件：
  1. Runbook 引擎
     - YAML 定义排查和修复步骤
     - 条件分支
     - 步骤间传递数据

  2. 执行器
     - K8s API 操作
     - SSH 远程命令
     - 数据库查询
     - HTTP API 调用

  3. 安全控制
     - 操作白名单
     - 权限最小化
     - 操作审计

示例 Runbook YAML：
  name: high-error-rate
  trigger:
    alert: ErrorRateHigh
  steps:
    - name: check-recent-deploy
      action: k8s.rollout_history
      params:
        deployment: "{{ .service }}"
    - name: check-if-new-deploy
      condition: "steps.check-recent-deploy.age < 30m"
      action: k8s.rollout_undo
      params:
        deployment: "{{ .service }}"
    - name: verify
      action: wait_for
      params:
        condition: "error_rate < 0.01"
        timeout: 5m
```

### 7. 面试题：自动修复会有什么风险？

**答：**

```
自动修复的风险与防护：

风险：
  1. 误判导致错误修复
     → 正常的批处理被当做异常重启
  
  2. 修复循环
     → 自动重启 → 问题重现 → 又重启
     → 消耗资源却无效
  
  3. 掩盖根因
     → 自动修复后不追查原因
     → 定时炸弹
  
  4. 级联影响
     → 自动修复一个服务影响另一个

防护措施：
  1. 冷却期：同一修复 1h 内最多 3 次
  2. 升级机制：超限后必须人工介入
  3. 审计日志：所有自动操作有记录
  4. 模拟模式：先 dry-run 再实际执行
  5. 影响范围限制：单次最多影响 N% 实例
  6. 强制 RCA：自动修复触发后仍需 Postmortem

平衡：
  高频低风险操作 → 自动修复
  低频高风险操作 → 人工确认
```
