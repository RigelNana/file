# 系统化故障排查方法八股文

---

## 一、故障排查方法论

### 1. SRE 排查故障的系统化方法是什么？

**答：**

```
Google SRE 系统化排查框架：

1. 定义问题
   什么出了问题？
   ├── 症状是什么？（错误率升高/延迟增大）
   ├── 影响范围？（哪些用户/服务受影响）
   └── 什么时候开始的？

2. 分诊（Triage）
   先止血还是先查因？
   ├── 影响严重 → 先恢复（重启/回滚/扩容）
   └── 影响可控 → 边排查边缓解

3. 检查（Examine）
   ├── 监控面板：近期指标变化
   ├── 日志：错误日志、访问日志
   ├── 变更记录：最近有什么变更？
   └── 关联事件：其他服务有异常吗？

4. 诊断（Diagnose）
   ├── 提出假设
   ├── 验证或否定假设
   └── 二分法缩小范围

5. 测试/修复（Test/Treat）
   ├── 实施修复
   ├── 验证修复有效
   └── 确认无副作用

6. 治愈（Cure）
   ├── 根治根因
   ├── 添加监控/告警
   └── 更新 Runbook
```

### 2. 排查故障时如何快速定位变更因素？

**答：**

```
"最近改了什么" 检查清单：

代码变更：
  - git log --oneline --since="2h ago"
  - 最近部署的版本是什么？
  - ArgoCD/CI 部署历史

配置变更：
  - 配置中心变更记录
  - Feature Flag 变更
  - K8s ConfigMap/Secret 变更

基础设施变更：
  - Terraform apply 记录
  - K8s 节点变更（扩缩容）
  - 网络策略变更

外部因素：
  - 云服务商状态页
  - 依赖的第三方服务
  - 流量模式变化（突增？爬虫？）
  - DNS/证书过期

统一变更时间线：
  09:00  部署 payment-api v2.3    ← 可疑
  09:15  修改 Redis 配置
  09:20  错误率开始上升           ← 故障开始
  09:25  收到告警
  
  → 最高优先级排查 09:00 的部署
```

### 3. 二分法排查思路如何应用？

**答：**

```
二分排查 = 系统性缩小问题范围

示例：API 返回 500 错误

第一刀：前端还是后端？
  → curl 直接调 API → 后端返回 500
  → 后端问题

第二刀：应用层还是基础设施层？
  → 应用日志有异常堆栈 → 应用层

第三刀：哪个服务？
  → 分布式追踪显示 order-service 报错
  → order-service

第四刀：哪个接口？
  → /api/v1/orders/create 报错
  → 订单创建接口

第五刀：代码还是数据？
  → 日志: "column 'discount_type' not found"
  → 数据库 Schema 问题

根因定位：
  昨天有人加了新字段但没有部署迁移脚本

关键原则：
  每一步都排除 50% 的可能性
  做实验验证假设，不要靠猜
```

## 二、排查工具与实践

### 4. SRE 排查问题的第一步应该看什么？

**答：**

```
"黄金信号" 快速排查法：

第一步：看四个黄金信号
  1. 流量（Traffic）：QPS 是否正常？
  2. 错误率（Errors）：5xx 比例？
  3. 延迟（Latency）：P50/P99 变化？
  4. 饱和度（Saturation）：CPU/内存/磁盘？

第二步：看 RED 指标（每个服务）
  Rate（请求率）→ 流量分布
  Errors（错误率）→ 哪个服务出错
  Duration（延迟）→ 哪里慢了

第三步：看 USE 指标（每个资源）
  Utilization（利用率）→ 资源够不够
  Saturation（饱和度）→ 队列是否堆积
  Errors（错误）→ 硬件/内核错误
```

```promql
# 快速排查 PromQL 示例

# 1. 各服务错误率排行
topk(5,
  sum by(service)(rate(http_requests_total{code=~"5.."}[5m]))
  /
  sum by(service)(rate(http_requests_total[5m]))
)

# 2. 延迟突增检测
histogram_quantile(0.99,
  rate(http_request_duration_seconds_bucket[5m])
) > 1.0

# 3. CPU 饱和度
1 - avg by(instance)(rate(node_cpu_seconds_total{mode="idle"}[5m]))
```

### 5. 如何利用分布式追踪定位故障？

**答：**

```
分布式追踪排查流程：

1. 找到异常 Trace
   → 从错误日志中提取 Trace ID
   → 或在 Jaeger/Tempo 中按条件搜索

2. 分析 Trace 瀑布图
   Client ─[200ms]──────────────────────┐
     API Gateway ─[5ms]─┐               │
       Auth Service ─[3ms]               │
     API Gateway ─[190ms]────────┐       │
       Order Service ─[180ms]────┐       │
         DB Query ─[170ms] ← 慢！│       │
                                 │       │
   → DB 查询占 170ms/200ms = 85%
   → 数据库是瓶颈

3. 检查 Span 标签
   span.attributes:
     db.statement: "SELECT * FROM orders WHERE..."
     db.type: "postgresql"
     error: true
     error.message: "timeout"
   
   → 定位到具体 SQL 语句

4. 采取行动
   → 检查该 SQL 的执行计划
   → 发现缺少索引
   → 添加索引解决
```

## 三、面试

### 6. 面试题：线上服务延迟突增，你的排查思路？

**答：**

```
延迟突增排查清单：

第一步（1分钟）：确认范围
  □ 全部接口还是特定接口？
  □ 所有用户还是部分用户？
  □ 所有 Pod 还是部分 Pod？

第二步（2分钟）：排查变更
  □ 最近有部署吗？
  □ 配置变更？
  □ 依赖服务有变更吗？

第三步（5分钟）：资源检查
  □ CPU/内存是否打满
  □ GC 是否频繁
  □ 线程池/连接池是否满
  □ 磁盘 IO 是否异常

第四步（5分钟）：下游依赖
  □ 数据库响应时间
  □ Redis 响应时间
  □ 依赖的微服务响应时间
  □ 网络延迟

第五步：深入分析
  □ 慢查询日志
  □ 火焰图/Profiling
  □ 分布式追踪
  □ 热点 Key / 锁争用

常见根因 TOP 5：
  1. 数据库慢查询（缺索引/锁等待）
  2. GC 暂停（内存泄漏/堆不够）
  3. 连接池耗尽（下游超时堆积）
  4. 外部依赖变慢（级联延迟）
  5. 流量突增（突发访问/攻击）
```

### 7. 面试题：如何区分是自身问题还是依赖方问题？

**答：**

```
自身 vs 依赖方问题的判断：

方法1：检查自身资源
  CPU/MEM/IO 正常？
  ├── 不正常 → 自身问题
  └── 正常 → 可能是依赖方

方法2：检查出站请求
  对外请求延迟增大？
  ├── 是 → 依赖方问题嫌疑大
  └── 否 → 自身逻辑问题

方法3：分布式追踪
  耗时主要在哪个 Span？
  ├── 自身处理 Span → 自身问题
  └── 外部调用 Span → 依赖方问题

方法4：隔离实验
  切换到 Mock/降级：
  ├── 恢复正常 → 确认是依赖方
  └── 仍然异常 → 自身问题

关键原则：
  先自查再查他人
  用数据说话，不用主观判断
  和依赖方同步排查，不互相甩锅
```
