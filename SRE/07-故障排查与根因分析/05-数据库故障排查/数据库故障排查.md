## 一、数据库排查基础

### 1. 数据库故障的常见分类？

**答：**

```
数据库故障分类：

┌──────────────┬────────────────────┬──────────────────┐
│ 类型          │ 表现                │ 影响              │
├──────────────┼────────────────────┼──────────────────┤
│ 连接耗尽      │ 无法建立新连接      │ 服务不可用        │
│ 慢查询        │ 响应延迟大          │ 性能下降          │
│ 锁等待/死锁   │ 查询挂起           │ 服务阻塞          │
│ 复制延迟      │ 从库数据不一致      │ 读到旧数据        │
│ 磁盘空间满    │ 写入失败           │ 数据丢失风险      │
│ 主从切换      │ 短暂不可用          │ 连接中断          │
│ 数据损坏      │ 查询报错           │ 数据完整性问题    │
└──────────────┴────────────────────┴──────────────────┘
```

### 2. MySQL 慢查询如何排查？

**答：**

```sql
-- 1. 查看当前执行中的查询
SHOW PROCESSLIST;
-- 或详细版
SELECT id, user, host, db, command, time, state, info
FROM information_schema.processlist
WHERE command != 'Sleep'
ORDER BY time DESC;

-- 2. 查看慢查询日志
-- 确认慢查询日志已开启
SHOW VARIABLES LIKE 'slow_query%';
SHOW VARIABLES LIKE 'long_query_time';

-- 3. 分析执行计划
EXPLAIN ANALYZE SELECT * FROM orders 
WHERE user_id = 12345 AND status = 'pending';

-- 关键看：
-- type: ALL(全表扫描) → 需要索引
-- rows: 扫描行数太多 → 需要优化
-- Extra: Using filesort / Using temporary → 性能问题

-- 4. 查看索引使用情况
SELECT * FROM sys.schema_unused_indexes;
SELECT * FROM sys.schema_redundant_indexes;

-- 5. 常见慢查询原因
-- a. 缺少索引 → 全表扫描
-- b. 索引失效 → 隐式类型转换/函数操作
-- c. 大结果集 → 缺少 LIMIT
-- d. JOIN 过多 → 笛卡尔积
-- e. 锁等待   → 事务冲突
```

### 3. 数据库连接耗尽如何排查？

**答：**

```sql
-- MySQL
SHOW STATUS LIKE 'Threads_connected';
SHOW STATUS LIKE 'Max_used_connections';
SHOW VARIABLES LIKE 'max_connections';

-- 按来源分析连接
SELECT user, host, db, command, count(*) as cnt
FROM information_schema.processlist
GROUP BY user, host, db, command
ORDER BY cnt DESC;

-- PostgreSQL
SELECT count(*) FROM pg_stat_activity;
SELECT max_conn, used, max_conn - used AS available
FROM (SELECT count(*) AS used FROM pg_stat_activity) t,
     (SELECT setting::int AS max_conn 
      FROM pg_settings WHERE name='max_connections') s;

-- 按应用分析
SELECT client_addr, datname, state, count(*)
FROM pg_stat_activity
GROUP BY client_addr, datname, state
ORDER BY count DESC;
```

```
连接耗尽排查步骤：

1. 确认连接数现状
   当前连接数 / 最大连接数

2. 分析连接来源
   哪个应用/IP 连接最多？
   是否有异常连接源？

3. 分析连接状态
   Sleep/Idle 连接过多？
   → 应用连接池配置不合理
   → 连接未正确归还

4. 紧急处理
   → 杀掉空闲连接
   → 临时增大 max_connections
   → 但这只是缓解，需查根因

5. 根因修复
   → 应用端设置合理的连接池大小
   → 设置连接最大空闲时间
   → 设置连接最大存活时间
```

## 二、锁与复制

### 4. 数据库锁问题如何排查？

**答：**

```sql
-- MySQL 锁排查

-- 查看当前锁等待
SELECT * FROM information_schema.innodb_lock_waits;

-- 详细锁信息
SELECT 
  r.trx_id AS waiting_trx_id,
  r.trx_mysql_thread_id AS waiting_thread,
  r.trx_query AS waiting_query,
  b.trx_id AS blocking_trx_id,
  b.trx_mysql_thread_id AS blocking_thread,
  b.trx_query AS blocking_query
FROM information_schema.innodb_lock_waits w
JOIN information_schema.innodb_trx b 
  ON b.trx_id = w.blocking_trx_id
JOIN information_schema.innodb_trx r 
  ON r.trx_id = w.requesting_trx_id;

-- 查看死锁日志
SHOW ENGINE INNODB STATUS\G
-- 找 LATEST DETECTED DEADLOCK 部分

-- PostgreSQL 锁排查
SELECT 
  blocked.pid AS blocked_pid,
  blocked.query AS blocked_query,
  blocking.pid AS blocking_pid,
  blocking.query AS blocking_query
FROM pg_catalog.pg_locks bl
JOIN pg_stat_activity blocked ON bl.pid = blocked.pid
JOIN pg_catalog.pg_locks kl ON bl.locktype = kl.locktype
  AND bl.relation = kl.relation AND bl.pid != kl.pid
JOIN pg_stat_activity blocking ON kl.pid = blocking.pid
WHERE NOT bl.granted;
```

```
死锁预防策略：
  1. 所有事务按相同顺序访问表/行
  2. 事务尽量简短
  3. 使用 SELECT ... FOR UPDATE NOWAIT
  4. 设置合理的锁超时时间
  5. 避免大事务
```

### 5. 主从复制延迟如何排查？

**答：**

```sql
-- MySQL 复制延迟
SHOW SLAVE STATUS\G
-- Seconds_Behind_Master: 延迟秒数
-- Relay_Log_Space: 未回放的日志量

-- PostgreSQL 复制延迟
SELECT client_addr, state, 
  pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn) AS lag_bytes,
  now() - pg_last_xact_replay_timestamp() AS lag_time
FROM pg_stat_replication;
```

```
复制延迟排查：

1. 确认延迟程度
   → 秒级：通常可接受
   → 分钟级：需要排查
   → 小时级：紧急处理

2. 常见原因
   ├── 大事务：单个事务修改百万行
   │   → 拆分为小批量
   ├── 从库硬件弱：CPU/IO 跟不上
   │   → 升级从库或减少从库查询
   ├── 从库查询负载高：分析查询占用资源
   │   → 分散读流量
   ├── 网络延迟：主从跨机房
   │   → 检查网络/优化 binlog 传输
   └── 长事务阻塞回放
       → 设置 slave_parallel_workers

3. 监控
   # Prometheus 指标
   mysql_slave_status_seconds_behind_master
   # 告警阈值
   > 30s: Warning
   > 300s: Critical
```

## 三、面试

### 6. 面试题：数据库 CPU 100% 排查？

**答：**

```
数据库 CPU 100% 排查：

1. 找到高消耗的查询
   -- MySQL
   SELECT * FROM sys.statements_with_runtimes_in_95th_percentile;
   
   -- PostgreSQL
   SELECT query, calls, total_time/calls AS avg_time,
          rows, shared_blks_hit, shared_blks_read
   FROM pg_stat_statements
   ORDER BY total_time DESC
   LIMIT 10;

2. 分析查询
   → EXPLAIN 查看执行计划
   → 是否全表扫描？
   → 是否隐式类型转换？

3. 常见 CPU 高的原因
   a. 全表扫描: 缺少索引或索引失效
   b. 排序: 大结果集 ORDER BY
   c. 聚合: 无索引的 COUNT/SUM
   d. 锁争用: CPU 空转等待锁
   e. 连接风暴: 大量短连接

4. 快速缓解
   → KILL 掉最耗 CPU 的查询
   → 应用层限流
   → 临时指向从库

5. 根治
   → 优化 SQL + 添加索引
   → 分库分表（数据量大）
   → 读写分离
```

### 7. 面试题：如何安全处理数据库磁盘空间满？

**答：**

```
磁盘空间满紧急处理：

紧急恢复（5分钟内）：
  1. 清理 binlog（如果安全）
     PURGE BINARY LOGS BEFORE DATE(NOW() - INTERVAL 3 DAY);
  2. 删除临时文件
     ls -la /tmp/mysql* 
  3. 清理慢查询日志
     > /var/log/mysql/slow.log
  4. 压缩归档旧的归档日志

根本解决：
  1. 扩容磁盘
  2. 历史数据归档/清理策略
  3. binlog 过期时间设置
     expire_logs_days = 7
  4. 分区表管理
  5. 监控磁盘使用率告警
     > 70% Warning
     > 85% Critical

预防措施：
  □ 磁盘使用率监控 + 告警
  □ 定期数据归档（月度/季度）
  □ 大表分区管理
  □ binlog/wal 清理策略
  □ 预留至少 20% 可用空间
```
