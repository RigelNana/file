## 一、5Why 分析法

### 1. 什么是 5Why 分析法？如何使用？

**答：**

```
5Why = 连续追问"为什么"直到找到根本原因

核心思想：
  每个"为什么"深入一层
  通常 5 次左右到达根因
  不一定刚好 5 次，可能 3-7 次

示例：支付服务 P0 故障

Why 1: 为什么支付服务不可用？
  → 数据库连接池耗尽

Why 2: 为什么连接池耗尽？
  → 有大量连接未被释放

Why 3: 为什么连接未被释放？
  → 新版本代码在错误路径没有关闭连接

Why 4: 为什么 Code Review 没发现？
  → 团队没有连接管理的检查清单

Why 5: 为什么没有自动化检测？
  → 缺少静态分析规则检测资源泄漏

Action Items:
  1. 修复代码（P0）
  2. 添加连接泄漏的静态分析规则（P1）
  3. Code Review Checklist 增加资源管理项（P2）
```

### 2. 5Why 的注意事项和技巧？

**答：**

```
5Why 常见错误：

✗ 停得太早：
  Why 1: 为什么宕机？→ 部署了有 Bug 的版本
  "修紧 Bug 就好了"
  → 没有深入到流程和系统层面

✗ 归咎个人：
  Why: 为什么部署出错？→ "Because 张三 didn't test"
  → 应该问：为什么系统允许未测试的代码部署？

✗ 分支太多迷失方向：
  一次 5Why 聚焦一条主链路
  多个触发因素分别做 5Why

✓ 正确技巧：

1. 用事实不用主观判断
   ✗ "因为代码写得不好"
   ✓ "因为 defer close 放在了 err check 之后"

2. 区分触发原因和系统原因
   触发原因: 这次具体是什么引起的
   系统原因: 为什么防御机制没生效

3. 每层都要有证据
   不是凭空猜测
   每个"因为"都要有日志/数据支撑

4. 多条根因链路
   一个事故可能有多个根因
   主链路 + 辅助因素
```

## 二、鱼骨图分析

### 3. 鱼骨图（石川图）如何应用于 SRE？

**答：**

```
鱼骨图 = 因果分析图

结构：
                             ┌─ 问题 ─┐
                             │        │
  ┌──────────┐  ┌──────────┐│        │┌──────────┐  ┌──────────┐
  │ 人员      │  │ 流程      ││        ││ 技术      │  │ 环境      │
  │ ├─原因1  │  │ ├─原因3  ││ 支付   ││ ├─原因5  │  │ ├─原因7  │
  │ └─原因2  │  │ └─原因4  ││ 服务   ││ └─原因6  │  │ └─原因8  │
  └──────────┘  └──────────┘│ P0故障 │└──────────┘  └──────────┘
                             │        │
  ┌──────────┐  ┌──────────┐│        │
  │ 工具      │  │ 监控      ││        │
  │ ├─原因9  │  │ ├─原因11 ││        │
  │ └─原因10 │  │ └─原因12 ││        │
  └──────────┘  └──────────┘└────────┘

SRE 常用六大分类（6M）：
  Man（人员）: 操作失误、经验不足、培训缺失
  Method（流程）: 流程缺失、审批不严、文档过期
  Machine（技术）: Bug、架构缺陷、容量不足
  Material（数据）: 数据异常、配置错误
  Measurement（监控）: 告警缺失、指标不全
  Milieu（环境）: 网络异常、硬件故障、第三方故障
```

### 4. 鱼骨图实战案例？

**答：**

```
案例：电商大促期间订单服务崩溃

问题：订单服务在双11高峰期持续30分钟不可用

鱼骨分析：

人员:
  ├── 容量评估把经验预估翻倍不够
  └── On-Call 人员对该服务不熟悉

流程:
  ├── 大促前没有做压力测试
  ├── 变更冻结期有紧急变更未充分测试
  └── 扩容审批流程太长

技术:
  ├── 数据库连接池配置太小
  ├── 缓存击穿保护缺失
  └── 限流阈值设置不合理

监控:
  ├── 连接池利用率没有告警
  ├── 自动扩缩策略反应太慢
  └── 缓存命中率没有监控

环境:
  ├── CDN 回源比例异常高
  └── 云服务商限流

根因总结（按权重）：
  1. 缓存击穿 + 连接池小 = 直接原因
  2. 缺少大促压测 = 流程原因
  3. 监控不全 = 检测能力不足
```

## 三、综合与面试

### 5. 5Why 和鱼骨图如何组合使用？

**答：**

```
组合使用策略：

1. 先用鱼骨图发散
   → 列出所有可能的原因
   → 按类别整理
   → 识别最可能的原因

2. 再用 5Why 收敛
   → 对鱼骨图中最关键的原因
   → 做 5Why 深入分析
   → 找到根本原因

鱼骨图: 广度优先（不遗漏）
5Why:   深度优先（找根因）

实际操作：
  Step 1: 10分钟鱼骨图头脑风暴
  Step 2: 投票选出 Top 3 原因
  Step 3: 每个原因做 3-5 次 Why
  Step 4: 汇总 Action Items
```

### 6. 面试题：请用 5Why 分析一个你经历过的故障？

**答：**

```
STAR 格式回答框架：

Situation:
  "我们的搜索服务在某天下午出现 50% 请求超时"

Task:
  "作为 On-Call 工程师，我负责恢复服务和后续 RCA"

Action（5Why）:
  Why 1: 为什么搜索超时？
    → Elasticsearch 集群响应慢

  Why 2: 为什么 ES 响应慢？
    → JVM Old GC 频繁（每分钟 Full GC）

  Why 3: 为什么 GC 频繁？
    → 索引大小增长了 3 倍，堆内存不够

  Why 4: 为什么索引增长了 3 倍？
    → 上周上线了新字段但没评估存储影响

  Why 5: 为什么没评估存储影响？
    → Schema 变更流程没有容量评估步骤

Result:
  "我们添加了 Schema 变更必须包含容量评估的流程
   ES 堆内存扩容到 32G
   添加了 JVM 内存使用率 > 80% 的告警
   此后再未发生类似问题"
```

### 7. 面试题：故障树分析（FTA）和 5Why 的区别？

**答：**

```
故障树分析 vs 5Why：

故障树分析(FTA):
  ├── 自顶向下，从顶事件出发
  ├── 用逻辑门（AND/OR）连接
  ├── 可以量化概率
  ├── 适合安全关键系统
  └── 预防性分析（事前）

5Why:
  ├── 线性追问
  ├── 简单直觉
  ├── 快速定性
  ├── 适合日常事故
  └── 事后分析（事后）

SRE 日常实践：
  事后复盘: 5Why + 鱼骨图（最常用）
  系统设计: 故障模式分析（FMEA）
  安全领域: 故障树分析（FTA）

推荐组合：
  事故复盘 = 时间线 + 5Why + Action Items
  → 简单高效，适合工程文化
```
