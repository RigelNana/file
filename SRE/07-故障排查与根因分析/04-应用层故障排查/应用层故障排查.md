## 一、应用层排查方法

### 1. 应用层故障的常见类型和排查入口？

**答：**

```
应用层故障分类：

┌──────────────┬────────────────────┬──────────────────┐
│ 故障类型      │ 表现                │ 排查入口          │
├──────────────┼────────────────────┼──────────────────┤
│ 进程崩溃      │ Pod CrashLoop      │ 日志/Core dump   │
│ 内存泄漏      │ OOM Kill           │ 内存 Profiling   │
│ CPU 热点      │ 延迟高/CPU 满       │ 火焰图           │
│ 死锁          │ 请求挂起不返回      │ 线程/协程 dump   │
│ 连接泄漏      │ 连接数持续增长      │ 连接池监控       │
│ 逻辑错误      │ 返回错误结果        │ 日志 + 追踪      │
│ 配置错误      │ 启动失败/行为异常   │ 配置对比          │
│ 依赖故障      │ 超时/错误           │ 下游健康检查      │
└──────────────┴────────────────────┴──────────────────┘
```

### 2. 如何排查 OOM（Out Of Memory）？

**答：**

```bash
# 1. 确认 OOM 事件
dmesg | grep -i "oom\|kill"
# Out of memory: Kill process 12345 (java)
# Killed process 12345 total-vm:4096000kB

# K8s 中确认
kubectl describe pod <pod> | grep -A5 "Last State"
#   Reason: OOMKilled
#   Exit Code: 137

# 2. 分析内存使用
# Go 应用
curl http://localhost:6060/debug/pprof/heap > heap.prof
go tool pprof heap.prof
# top 20 -- 查看内存分配热点

# Java 应用
jmap -histo:live <PID> | head -20
jmap -dump:live,format=b,file=heap.hprof <PID>

# 3. 常见 OOM 原因
# - 内存泄漏（未关闭连接/缓存无上限）
# - Goroutine 泄漏（Go）
# - 堆外内存（Java DirectByteBuffer）
# - 大对象缓存
# - K8s limits 设置过小
```

```
OOM 排查决策树：

OOM Killed
├── 内存持续增长（泄漏）
│   ├── 连接未关闭 → defer conn.Close()
│   ├── Goroutine 泄漏 → pprof goroutine
│   ├── 缓存无上限 → 加 LRU 淘汰
│   └── 大对象未释放 → heap profile 分析
└── 瞬间内存飙升
    ├── 大查询结果集 → 分页/流式处理
    ├── 大文件读取 → streaming IO
    └── 并发请求过多 → 限流/背压
```

### 3. 如何排查 Goroutine/线程泄漏？

**答：**

```go
// Go Goroutine 泄漏排查

// 1. 暴露 pprof 端点
import _ "net/http/pprof"
go http.ListenAndServe(":6060", nil)

// 2. 查看 goroutine 数量
// curl http://localhost:6060/debug/pprof/goroutine?debug=1
```

```bash
# 3. 监控 goroutine 数量变化
# PromQL
go_goroutines{job="myapp"}

# 如果持续增长 → 有泄漏

# 4. 获取 goroutine dump
curl http://localhost:6060/debug/pprof/goroutine?debug=2 > goroutine.txt

# 5. 常见泄漏模式
# - channel 没有 consumer
# - 没有 context 超时控制
# - HTTP response body 没有关闭
# - ticker 没有 Stop()
```

```
常见泄漏代码模式：

✗ 泄漏：
  ch := make(chan int)
  go func() {
      ch <- result  // 没人读，goroutine 永远阻塞
  }()

✓ 修复：
  ctx, cancel := context.WithTimeout(ctx, 5*time.Second)
  defer cancel()
  go func() {
      select {
      case ch <- result:
      case <-ctx.Done():
          return
      }
  }()
```

## 二、JVM 与运行时

### 4. Java 应用性能问题如何排查？

**答：**

```bash
# 1. GC 问题排查
# 查看 GC 日志
-XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:gc.log

# GC 频率和耗时
jstat -gcutil <PID> 1000

#  S0   S1     E     O     M   CCS  YGC  YGCT  FGC  FGCT   GCT
#  0.00 99.00 50.00 85.00 95.00 90.00 100 2.5   5    8.0   10.5
# FGC=5, FGCT=8s → Full GC 太频繁太慢

# 2. 线程问题
jstack <PID> > thread_dump.txt
# 查找 BLOCKED 状态线程
grep -c "BLOCKED" thread_dump.txt
# 查找死锁
grep -A 20 "deadlock" thread_dump.txt

# 3. 常见 Java 性能问题
# - Full GC 频繁: 堆太小/内存泄漏
# - 线程池满: 下游慢导致线程阻塞
# - 类加载慢: 动态代理生成过多类
# - 连接池耗尽: 连接未归还
```

### 5. 如何排查应用连接池问题？

**答：**

```
连接池故障表现：
  - 请求超时: "get connection timeout"
  - 连接数打满: active connections = max pool size
  - 连接泄漏: idle connections 越来越少

排查步骤：

1. 确认连接池状态
   # 查看连接数
   ss -tn dst <db_ip>:<db_port> | wc -l
   
   # 应用指标
   db_pool_active_connections
   db_pool_idle_connections
   db_pool_wait_count

2. 分析连接去向
   # 数据库端
   SELECT * FROM pg_stat_activity 
   WHERE client_addr = '<app_ip>';
   
   # 查是否有长事务
   SELECT pid, now() - xact_start AS duration, query
   FROM pg_stat_activity
   WHERE state = 'active'
   ORDER BY duration DESC;

3. 常见根因
   ┌────────────────────┬──────────────────────┐
   │ 根因                │ 解决方案              │
   ├────────────────────┼──────────────────────┤
   │ 慢查询占用连接      │ 优化SQL + 设查询超时  │
   │ 连接未归还(泄漏)    │ 连接池自动回收        │
   │ 池大小太小          │ 调大 max pool size   │
   │ 下游数据库过载      │ 读写分离 / 扩容       │
   │ 死锁占用连接        │ 死锁检测 + 超时       │
   └────────────────────┴──────────────────────┘
```

## 三、面试

### 6. 面试题：服务频繁重启（CrashLoopBackOff）怎么排查？

**答：**

```
CrashLoopBackOff 排查：

1. 查看 Pod 事件
   kubectl describe pod <pod>
   → Events 显示退出原因
   → Exit Code: 
     1   = 应用错误
     137 = OOM Killed (128+9)
     143 = SIGTERM (128+15)

2. 查看日志
   kubectl logs <pod> --previous  # 上一次的日志
   → 找 panic / fatal / error

3. 根据退出码分析
   Exit 1: 应用启动失败
     → 配置错误？
     → 依赖服务不可达？
     → 端口冲突？
   
   Exit 137: OOM
     → limits 太小？
     → 内存泄漏？
   
   Exit 143: 被终止
     → liveness probe 失败？
     → preStop 超时？

4. 检查探针
   livenessProbe 配置是否合理？
   → initialDelaySeconds 够不够？
   → 超时时间够不够？
   → 检查路径是否正确？
```

### 7. 面试题：如何排查微服务间的级联故障？

**答：**

```
级联故障排查：

表现：
  Service A → B → C
  C 变慢 → B 线程池满 → A 超时

排查步骤：

1. 画调用链
   分布式追踪找到完整链路
   标记每个环节耗时

2. 找根源
   沿调用链路从下游往上找
   第一个异常的服务 = 根源

3. 分析为什么级联
   → 没有超时控制？
   → 没有熔断？
   → 没有限流？
   → 重试放大了流量？

4. 快速恢复
   → 上游：开启熔断/降级
   → 根源：扩容/回滚/重启
   → 全链路：限流保护

5. 长期修复
   → 所有调用设合理超时
   → 关键路径加熔断器
   → 非关键依赖可降级
   → 限制重试次数和频率
```
