---

## 一、网络排查基础

### 1. 网络故障排查的分层思路？

**答：**

```
按 OSI 模型自底向上排查：

Layer 1 物理层: 网线/光纤是否连通
  → ethtool eth0 (查看链路状态)
  → ip link show

Layer 2 数据链路: MAC 地址/ARP
  → arp -n (ARP 表)
  → ip neigh show

Layer 3 网络层: IP 路由/连通性
  → ping (ICMP 连通性)
  → traceroute / mtr (路由追踪)
  → ip route show

Layer 4 传输层: TCP/UDP 连接
  → ss -tnlp (监听端口)
  → ss -tn state established
  → tcpdump (抓包分析)

Layer 7 应用层: HTTP/DNS/应用协议
  → curl (HTTP 测试)
  → dig / nslookup (DNS 解析)
  → openssl s_client (TLS 测试)

排查原则：
  先排除低层问题再查高层
  但生产环境通常从 L7 开始（更快定位）
```

### 2. 常见网络排查命令有哪些？

**答：**

```bash
# 1. 连通性测试
ping -c 5 target-host
traceroute target-host
mtr --report target-host  # 综合 ping + traceroute

# 2. DNS 排查
dig example.com +trace     # DNS 解析全链路
dig @8.8.8.8 example.com  # 指定 DNS 服务器
nslookup example.com      # 简单查询

# 3. 端口和连接
ss -tnlp                  # 查看监听端口
ss -tn state time-wait | wc -l  # TIME_WAIT 数量
ss -tn dst 10.0.0.5:3306  # 到特定地址的连接

# 4. 抓包分析
tcpdump -i eth0 port 80 -w capture.pcap
tcpdump -i eth0 host 10.0.0.5 and port 3306
tcpdump -i eth0 'tcp[tcpflags] & (tcp-syn|tcp-rst) != 0'

# 5. HTTP 调试
curl -v https://api.example.com/health
curl -w "@curl-format.txt" https://api.example.com
#  time_namelookup: DNS 解析时间
#  time_connect:    TCP 连接时间
#  time_appconnect: TLS 握手时间
#  time_starttransfer: 首字节时间
#  time_total:      总时间

# 6. 网络统计
nstat -sz                  # 网络统计计数器
netstat -s | grep -i retrans  # 重传统计
cat /proc/net/snmp         # 协议统计
```

### 3. TCP 相关的常见故障如何排查？

**答：**

```
TCP 常见问题：

1. 连接超时
   症状: connect timeout
   排查:
     → telnet target 8080 (能否建立 TCP)
     → 防火墙规则检查
     → 安全组/NetworkPolicy 检查
   常见原因: 防火墙/安全组阻断

2. 连接重置 (RST)
   症状: connection reset by peer
   排查:
     → tcpdump 抓包看 RST 包来源
     → 服务端日志
   常见原因: 服务崩溃、负载均衡超时、防火墙

3. TIME_WAIT 过多
   症状: 无法建立新连接
   排查:
     → ss -s (查看连接统计)
     → sysctl net.ipv4.tcp_tw_reuse
   解决: 开启连接复用、使用长连接

4. 重传率高
   症状: 延迟波动大
   排查:
     → nstat -sz | grep Retrans
     → 网络丢包检测
   常见原因: 网络质量差、MTU 问题

5. 半连接队列满
   症状: 新连接建立缓慢
   排查:
     → dmesg | grep "SYN flood"
     → sysctl net.ipv4.tcp_max_syn_backlog
   解决: 增大 backlog、开启 syncookies
```

## 二、DNS 与网络策略

### 4. DNS 故障排查三板斧？

**答：**

```
DNS 排查流程：

第一步：确认是不是 DNS 问题
  # 用 IP 直连是否正常？
  curl http://10.0.0.5:8080/health   → 正常
  curl http://service.ns.svc:8080/health → 超时
  → 确认是 DNS 问题

第二步：检查 DNS 解析
  # K8s 内
  kubectl exec -it debug-pod -- nslookup service.ns.svc.cluster.local
  kubectl exec -it debug-pod -- cat /etc/resolv.conf
  
  # 宿主机
  dig service.ns.svc.cluster.local @10.96.0.10

第三步：检查 DNS 服务
  # CoreDNS Pod 状态
  kubectl -n kube-system get pods -l k8s-app=kube-dns
  # CoreDNS 日志
  kubectl -n kube-system logs -l k8s-app=kube-dns
  # CoreDNS 指标
  # coredns_dns_request_duration_seconds
  # coredns_dns_responses_total{rcode="SERVFAIL"}

常见 DNS 故障：
  1. CoreDNS Pod OOM → 增大资源限制
  2. ndots 配置导致多次查询 → 使用 FQDN
  3. 上游 DNS 不可达 → 检查网络策略
  4. DNS 缓存 TTL 过长 → 老 IP 仍被解析
```

### 5. K8s 网络策略排查？

**答：**

```bash
# 排查 Pod 网络不通

# 1. 检查 Pod 网络状态
kubectl get pod -o wide  # 确认 Pod IP
kubectl exec pod -- ip addr
kubectl exec pod -- ip route

# 2. Pod 间连通性
kubectl exec pod-a -- curl -v pod-b-ip:8080

# 3. 检查 NetworkPolicy
kubectl get networkpolicy -A
kubectl describe networkpolicy <name>

# 4. 检查 Service
kubectl get svc
kubectl get endpoints <svc-name>  # endpoint 是否有 Pod IP

# 5. 检查 kube-proxy
# iptables 模式
iptables -t nat -L KUBE-SERVICES | grep <svc-name>
# ipvs 模式
ipvsadm -ln | grep <svc-cluster-ip>

# 常见问题：
# - Pod 没有 Ready → endpoint 列表为空
# - NetworkPolicy 阻断 → 检查 ingress/egress 规则
# - Service selector 不匹配 → label 不对
# - CNI 插件故障 → 检查 calico/cilium Pod
```

## 三、面试

### 6. 面试题：用户反馈服务偶尔超时，如何排查？

**答：**

```
偶发超时排查思路：

"偶尔" = 不是 100% 失败 → 排除连通性问题

1. 确认超时模式
   → 固定时间周期？随机？
   → 特定用户？特定地域？

2. 检查超时位置
   Client → LB → Service → DB
   → 用分布式追踪看哪一跳超时

3. 常见偶发超时原因：

   DNS 查询偶尔慢
   → 抓包看 DNS 响应时间
   → ndots 导致多次查询

   TCP 连接复用问题
   → 服务端先关闭连接
   → 客户端用了已关闭的连接
   → 抓包看 RST

   负载均衡到不健康节点
   → 健康检查间隔太长
   → 节点 GC 暂停但未摘除

   网络抖动
   → mtr 长时间测试
   → 查看 retransmit 统计

   资源竞争（GC、锁）
   → 检查 GC 日志
   → 检查线程 dump
```

### 7. 面试题：如何诊断丢包问题？

**答：**

```
丢包诊断逐层排查：

1. 确认丢包
   ping -c 1000 -i 0.01 target  # 发 1000 个包
   → 查看丢失百分比

2. 定位丢包位置
   mtr --report -c 100 target
   → 查看每一跳的丢包率
   → 丢包从哪一跳开始

3. 系统层面
   # 查看网卡丢包
   ip -s link show eth0  # RX dropped/errors
   ethtool -S eth0 | grep -i drop
   
   # 内核丢包统计
   nstat -sz | grep -i drop
   cat /proc/net/softnet_stat

4. 常见丢包原因：
   网卡队列满    → 增大 ring buffer
   内核 backlog  → 增大 net.core.netdev_max_backlog
   iptables DROP → 检查防火墙规则
   TCP 缓冲区满  → 增大 tcp_rmem/wmem
   MTU 不匹配    → ping -M do -s 1472 target
   网络拥塞      → 联系网络团队
```
