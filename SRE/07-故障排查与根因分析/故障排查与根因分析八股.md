## 一、故障排查方法论

### 1. SRE 的系统化故障排查方法是什么？

**答：** Google SRE 推荐的排查框架：

```
1. 分诊（Triage）     → 评估影响范围，决定优先级
2. 检查（Examine）    → 收集症状和指标数据
3. 诊断（Diagnose）   → 提出假设并验证
4. 治疗（Treat/Fix）  → 执行修复
5. 治愈（Cure）       → 确认恢复，防止再发
```

**关键原则：**
- **先止血再排查**：优先恢复服务，而不是钻研根因
- **从已知到未知**：先检查最近的变更、已知问题
- **二分法排查**：通过排除法缩小问题范围
- **告警与指标关联**：从监控数据入手

### 2. Linux 性能排查有哪些常用工具？

**答：** Brendan Gregg 的 Linux Performance Observability Tools：

| 层次 | 工具 | 用途 |
|------|------|------|
| **CPU** | top, htop, mpstat, perf | CPU 使用率、热点函数 |
| **内存** | free, vmstat, slabtop | 内存使用、交换、缓存 |
| **磁盘** | iostat, iotop, blktrace | 磁盘 IO 吞吐、延迟 |
| **网络** | ss, netstat, tcpdump, iftop | 连接状态、流量、抓包 |
| **进程** | ps, strace, lsof | 进程状态、系统调用、文件描述符 |
| **综合** | sar, dstat, nmon | 系统综合性能数据 |
| **高级** | bpftrace, bcc, perf | eBPF 动态追踪 |

### 3. 排查网络问题的思路是什么？

**答：** 按网络层次自底向上排查：

1. **物理/链路层**：`ip link show`、`ethtool`（链路状态、丢包统计）
2. **网络层**：`ping`、`traceroute`、`mtr`（连通性、路由、延迟）
3. **传输层**：`ss -tnp`、`netstat`（连接状态、TIME_WAIT 堆积）
4. **应用层**：`curl -v`、`dig`、`nslookup`（HTTP、DNS 问题）
5. **抓包分析**：`tcpdump`、Wireshark（深入分析协议交互）

## 二、根因分析

### 4. 什么是 RCA（Root Cause Analysis）？常用的 RCA 方法有哪些？

**答：**

| 方法 | 说明 | 适用场景 |
|------|------|----------|
| **5 Why** | 连续追问"为什么"直到找到根因 | 简单、线性因果链 |
| **鱼骨图** | 从人、机、料、法、环分类原因 | 多因素复杂问题 |
| **故障树分析** | 从顶部事件向下展开逻辑树 | 安全关键系统 |
| **时间线分析** | 按时间顺序梳理事件经过 | 事件复盘 |
| **变更回溯** | 追溯最近的变更找到触发因素 | 变更引起的故障 |

### 5. 5 Why 分析法的示例？

**答：**

问题：网站宕机了

1. **为什么宕机？** → 数据库连接池耗尽
2. **为什么连接池耗尽？** → 慢查询太多，连接无法及时释放
3. **为什么有大量慢查询？** → 一个新上线的功能没有加索引
4. **为什么没有加索引？** → 代码审查没有检查 SQL 性能
5. **为什么没有检查？** → 缺少数据库变更的审查清单

**根因**：缺少数据库变更审查机制
**改进项**：建立数据库变更 Checklist，加入 CI 流水线的 SQL 审查
