## 一、分布式故障类型

### 1. 分布式系统特有的故障类型有哪些？

**答：**

```
分布式系统故障分类：

┌─────────────────┬────────────────────────────────────┐
│ 故障类型         │ 说明                                │
├─────────────────┼────────────────────────────────────┤
│ 网络分区         │ 节点间无法通信，脑裂                 │
│ 级联故障         │ 一个服务故障导致上游连锁崩溃          │
│ 惊群效应         │ 大量请求同时冲击恢复中的服务          │
│ 数据不一致       │ 副本间数据不同步                     │
│ 脑裂             │ 多个节点同时认为自己是主节点          │
│ 雪崩             │ 重试风暴导致全链路过载                │
│ 热点问题         │ 流量集中到少数节点                   │
│ 时钟偏移         │ 节点间时间不一致导致逻辑错误          │
└─────────────────┴────────────────────────────────────┘

VS 单体应用故障：
  单体: 要么全好要么全坏
  分布式: 部分失败(Partial Failure)
  → 更难排查，因为系统处于"半工作"状态
```

### 2. 级联故障如何排查和恢复？

**答：**

```
级联故障典型场景：

  ServiceA ──→ ServiceB ──→ ServiceC ──→ DB
  
  DB 慢 → C 线程池满 → B 超时堆积 → A 全部超时

排查步骤：

1. 从告警定位入口
   → 多个服务同时告警
   → 找最早的告警 = 根源附近

2. 画调用链路图
   ServiceA
     ├── ServiceB (timeout)
     │     ├── ServiceC (timeout)
     │     │     └── DB (slow query)  ← 根源
     │     └── Cache (ok)
     └── ServiceD (ok)

3. 沿调用链自下而上
   → DB 的 slow query log
   → 发现缺少索引的大查询

恢复策略（关键：打破级联链路）：
  1. 上游开熔断 → 快速失败，不继续请求
  2. 中间层降级 → 返回缓存/默认值
  3. 根源修复 → 杀掉慢查询/添加索引
  4. 逐层恢复 → 从下游到上游逐步恢复

注意：不要同时恢复所有服务
  → 全部同时恢复 = 惊群效应
  → 逐步放量恢复
```

### 3. 重试风暴如何排查和防止？

**答：**

```
重试风暴场景：

  正常: Client → Server (100 QPS)
  
  Server 故障:
    Client 重试3次 → 300 QPS
    中间层也重试3次 → 300 × 3 = 900 QPS
    → 9倍流量放大！

排查方式：
  1. 监控实际 QPS vs 正常 QPS
     如果故障时 QPS 反而增大 → 重试风暴

  2. 查看请求头
     x-retry-count 标记重试次数
     大量 retry-count > 0 的请求

防止重试风暴：

1. 指数退避 + 抖动
   delay = min(base * 2^attempt + random_jitter, max_delay)

2. 限制重试次数
   最多重试 2-3 次

3. 重试预算
   同一时间窗口内重试比例 < 10%
   if retry_ratio > 0.1:
       stop_retrying()

4. 只重试可重试的错误
   5xx → 可重试
   4xx → 不重试（客户端错误）
   timeout → 看情况（幂等才重试）

5. 熔断器
   连续失败 N 次 → 直接快速失败
   不再发送请求
```

## 二、排查工具

### 4. 如何利用服务网格排查分布式问题？

**答：**

```
Service Mesh (Istio) 排查能力：

1. 流量监控（不改代码）
   Istio 自动采集：
   - 请求量/错误率/延迟
   - 服务间调用关系
   - 请求成功率

2. Kiali 服务拓扑
   → 可视化服务调用关系
   → 标红异常链路
   → 显示每条边的 QPS/延迟/错误率

3. 分布式追踪
   Istio sidecar 自动注入追踪头
   → Jaeger/Zipkin 查看链路
   → 不需要应用改代码

4. 流量镜像（Shadow Traffic）
   排查线上问题时：
   → 镜像流量到调试实例
   → 不影响线上用户
```

```yaml
# Istio 流量镜像用于调试
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: payment-api
spec:
  hosts:
    - payment-api
  http:
    - route:
        - destination:
            host: payment-api
            subset: v1
      mirror:
        host: payment-api-debug
      mirrorPercentage:
        value: 10  # 镜像 10% 流量
```

### 5. 分布式系统的可观测性排查矩阵？

**答：**

```
排查矩阵（按问题类型选工具）：

问题类型      │ 首选工具        │ 次选
─────────────┼────────────────┼──────────
延迟增大      │ 分布式追踪      │ 日志
错误率升高    │ 日志 + 指标     │ 追踪
服务不可达    │ 网络抓包        │ 指标
数据不一致    │ 日志比对        │ 追踪
资源耗尽      │ Prometheus     │ 系统命令
流量异常      │ 指标           │ 网络监控

排查黄金路径：
  1. 看 Dashboard（全局视角）
     → 哪个服务/节点异常
  
  2. 看 Trace（单请求视角）
     → 异常请求的详细链路
  
  3. 看 Log（细节视角）
     → TraceID 关联的所有日志
  
  4. 看 Metric（趋势视角）
     → 指标变化的时间线
```

## 三、面试

### 6. 面试题：如何排查微服务间的数据不一致？

**答：**

```
数据不一致排查：

场景：订单创建成功，但库存没扣减

1. 确认时间线
   → 订单服务：已创建 order-123 at 10:00:00
   → 库存服务：没收到扣减请求

2. 检查消息/事件
   → 消息队列（Kafka）是否有消息
   → 消费者是否有消费记录
   → 是否有消费失败/重试

3. 检查分布式事务
   → Saga 编排器状态
   → 补偿事务是否执行

4. 常见不一致原因
   a. 消息丢失: producer 没有 ack
   b. 消费失败: consumer 处理异常
   c. 超时误判: 实际成功但调用方认为失败
   d. 并发冲突: 乐观锁冲突丢失更新
   e. 重复处理: 缺少幂等控制

5. 修复
   → 跑对账任务比对数据
   → 补偿不一致的记录
   → 修复根因防止再次发生
```

### 7. 面试题：如何排查惊群效应（Thundering Herd）？

**答：**

```
惊群效应场景：
  缓存失效 → 大量请求打到数据库
  服务重启 → 大量连接同时建立
  限流解除 → 积压请求同时涌入

排查信号：
  → 某个时间点 QPS 突然飙升（尖峰）
  → 下游瞬间过载
  → 通常伴随服务重启/缓存失效

排查步骤：
  1. 确认尖峰时间点
  2. 查同一时间的事件
     → 缓存 Key 过期？
     → 服务重启？
     → 限流/熔断恢复？

防止措施：
  1. 缓存击穿防护
     → singleflight: 相同 Key 只查一次 DB
     → 热点缓存永不过期 + 异步更新

  2. 服务启动保护
     → 预热（warmup）：启动后逐步接入流量
     → Readiness Probe 把控就绪状态

  3. 熔断恢复策略
     → 半开状态只放少量流量
     → 慢慢恢复，不一次全开

  4. 随机化
     → 缓存 TTL 加随机偏移
     → 定时任务加随机延迟
```
