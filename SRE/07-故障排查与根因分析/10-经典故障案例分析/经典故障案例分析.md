## 一、互联网公司案例

### 1. 案例一：GitHub 2018 数据库事件

**答：**

```
事件概述：
  2018年10月，GitHub 服务中断 24 小时

时间线：
  Day 1 22:52  网络设备维护导致主从短暂断开
  Day 1 22:54  从库被提升为主库（自动故障切换）
  Day 1 22:58  发现两个数据中心都有"主库"
  Day 2        数据一致性修复花了约 24 小时

根因分析：
  直接原因: 自动故障切换在网络抖动时误判
  根本原因: 
    1. 编排工具对跨数据中心的故障场景考虑不足
    2. 回切流程缺乏自动化
    3. 数据一致性校验工具不够高效

关键教训：
  1. 自动故障切换需要更严格的仲裁机制
  2. 跨数据中心的一致性恢复需要演练
  3. 宁可多停一会也不要带着不一致恢复
  4. 数据完整性 > 可用性

SRE 改进：
  ✓ 改进故障切换的仲裁逻辑
  ✓ 建立跨 DC 的数据一致性验证
  ✓ 定期演练故障切换
```

### 2. 案例二：Cloudflare 2019 全球宕机

**答：**

```
事件概述：
  2019年7月，Cloudflare 全球服务中断 27 分钟

时间线：
  13:42  部署了含正则表达式的 WAF 规则
  13:42  CPU 全球飙升到 100%
  14:02  通过全球 rollback 恢复

根因分析：
  直接原因: 一个正则表达式导致灾难性回溯
  
  问题正则（简化）:
    (?:(?:\"|'|\]|\}|\\|\d|(?:nan|infinity|true|false|
    null|undefined|symbol|math)|\`|\-|\+)+[)]*;?((?:\s|
    -|~|!|{}|\|\||\+)*.*(?:.*=.*)))

  为什么影响全球:
    1. WAF 规则全球同步部署（无灰度）
    2. 正则引擎使用回溯型（NFA）
    3. CPU 100% 导致所有请求被丢弃

关键教训：
  1. 任何全局变更必须灰度发布
  2. 正则表达式需要性能测试
  3. 安全规则也是代码，需要 CI/CD
  4. 需要熔断机制：CPU 异常时禁用新规则

SRE 视角改进：
  ✓ WAF 规则灰度部署
  ✓ 正则性能自动测试
  ✓ CPU 过载自动回退机制
  ✓ 分区域部署（爆炸半径控制）
```

### 3. 案例三：AWS S3 2017 大规模故障

**答：**

```
事件概述：
  2017年2月，AWS S3 us-east-1 中断 4 小时
  大量依赖 S3 的服务跟着挂

时间线：
  09:37  运维人员执行脚本删除少量 S3 子系统服务器
  09:37  输入参数错误，删除了大量关键服务器
  12:30  手动恢复所有子系统
  13:xx  S3 完全恢复

根因分析：
  直接原因: 运维人员输入了错误的参数
  根本原因:
    1. 操作工具缺少安全检查（可删数量无上限）
    2. 系统从未做过大规模重启测试
    3. 恢复流程文档不全

关键教训：
  1. 危险操作必须有安全限制
     → "你确定要删除 xxx 个服务器？"
     → 单次操作上限
  2. 控制面和数据面分离
  3. 定期演练大规模恢复
  4. 不依赖单一云区域

SRE 设计原则：
  ✓ 操作确认和速率限制
  ✓ 爆炸半径控制
  ✓ 依赖方要有降级方案
  ✓ 多区域容灾
```

## 二、案例分析与面试

### 4. 案例四：缓存雪崩导致服务崩溃

**答：**

```
场景：
  电商大促期间，Redis 集群因内存不足触发淘汰
  大量热点缓存同时失效
  数据库被击穿，级联导致全站不可用

时间线：
  20:00  大促开始，流量达到预期的 3 倍
  20:15  Redis 内存使用达 95%，开始淘汰
  20:18  热门商品缓存被淘汰
  20:18  DB QPS 从 500 飙到 50000
  20:19  数据库连接耗尽
  20:20  全站 5xx
  20:35  手动限流 + DB 扩容
  21:00  服务逐步恢复

5Why:
  1. 为什么全站不可用？→ 数据库不可用
  2. 为什么数据库不可用？→ QPS 超过承载100x
  3. 为什么 QPS 突增？→ 缓存大面积失效
  4. 为什么缓存失效？→ Redis 内存淘汰
  5. 为什么内存不足？→ 容量评估未考虑大促

Action Items：
  P0: Redis 集群扩容至 2 倍冗余
  P0: 添加缓存击穿保护（singleflight）
  P1: 热点 key 永不过期 + 异步刷新
  P1: 限流保护数据库（max QPS cap）
  P2: 大促前强制压测
  P2: Redis 内存告警 70% Warning
```

### 5. 案例五：配置变更引发灰度失效

**答：**

```
场景：
  Feature Flag 配置变更导致灰度用户比例
  从 1% 变成 100%，新功能对全量用户开放

根因分析：
  直接原因: 配置值从 0.01 改成了 1（百分比格式不一致）
  有人以为是百分比（1%），实际是小数（1=100%）

  系统原因:
  1. 配置格式没有类型验证
  2. Flag 变更没有灰度生效
  3. 没有变更前后的影响预览

改进：
  1. Flag 值添加类型校验
     percentage 类型: 0.0 - 1.0
     变更时显示: "将从 1% 变为 100%，确认？"
  
  2. 灰度生效
     Flag 变更先在 1% 流量生效
     观察 5 分钟后再全量
  
  3. 变更审计
     谁改了什么 + 新旧值对比 + 影响评估
```

## 三、面试

### 6. 面试题：请分析一个你见过的最有价值的故障案例？

**答：**

```
回答框架（STAR + RCA）：

S - 场景描述
  何时、何服务、什么现象、影响多大

T - 你的角色
  On-Call / IC / 参与者

A - 排查和恢复过程
  时间线 + 关键决策点
  为什么做了某个决策
  恢复的具体步骤

R - RCA 和改进
  根因是什么（5Why）
  Action Items
  系统性改进

加分项：
  → 说出"什么做得好"
  → 说出"下次会怎么改进"
  → 说出"系统性预防"而不只是"修这个Bug"
```

### 7. 面试题：从故障案例中学到的核心 SRE 原则？

**答：**

```
故障案例总结的 SRE 核心原则：

1. 爆炸半径控制
   所有变更必须灰度，没有例外
   → Cloudflare 全球 WAF 事件

2. 防御纵深
   不依赖单一防线
   缓存失效 → 限流 → 熔断 → 降级
   → 缓存雪崩案例

3. 安全操作设计
   危险操作 = 确认 + 限速 + 撤销
   → AWS S3 事件

4. 数据优先
   可用性可以暂时牺牲，数据不能丢
   → GitHub 数据库事件

5. 演练验证
   文档上的恢复方案不等于能用
   必须定期演练
   → 所有案例共同教训

6. 自动化 ≠ 无人值守
   自动化需要监督和限制
   → 自动故障切换误判

每个原则背后都有血淋淋的案例
```
