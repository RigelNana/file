# SRE组织模型八股文

---

## 一、SRE 组织模型

### 1. SRE 团队常见的组织模型有哪些？各自有什么优缺点？

**答：**

| 模型 | 描述 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| **集中式 SRE** | 独立 SRE 部门服务多个产品 | 标准化强、资源池化 | 业务理解不深 | 大型企业、基础设施统一 |
| **嵌入式 SRE** | SRE 嵌入产品团队 | 深入理解业务 | 容易变成救火队 | 业务复杂度高的团队 |
| **咨询式 SRE** | SRE 作为内部顾问 | 影响面广、杠杆效应大 | 执行力依赖产品团队 | SRE 资源有限时 |
| **平台 SRE** | 构建可靠性平台和工具 | 规模化赋能 | 与业务脱节风险 | 大规模工程组织 |
| **混合模式** | 以上模型的组合 | 灵活适配 | 管理复杂 | 中大型企业 |

**Google 模型**：
```
              VP of Engineering
                    │
        ┌───────────┼───────────┐
        │           │           │
    产品团队A    产品团队B    SRE 团队
    (Dev)       (Dev)       (集中式)
                               │
                    ┌──────────┼──────────┐
                    │          │          │
                SRE-搜索   SRE-广告   SRE-基础设施
                (嵌入式)   (嵌入式)   (平台式)
```

### 2. 集中式 SRE 模型的详细特点？

**答：**

**组织结构**：
- 独立的 SRE 部门/团队
- 有自己的管理链（不隶属于任何产品团队）
- 为多个产品线提供 SRE 服务

**工作方式**：
- SRE 团队与产品团队签订**服务协议**
- 通过 SLO 定义服务质量目标
- SRE 有权**退回**不满足要求的服务

**适用条件**：
- 公司规模较大，有多个产品线
- 基础设施相对统一
- 需要标准化的可靠性实践

**常见问题与应对**：

| 问题 | 应对方案 |
|------|----------|
| 业务理解不足 | 定期参加产品 Sprint 会议 |
| 响应速度慢 | 为每个产品指定 SRE 对接人 |
| 资源分配争议 | 基于 SLO 和服务等级分配 |
| 缺乏归属感 | SRE 轮岗到不同产品线 |

### 3. 嵌入式 SRE 模型的详细特点？

**答：**

**组织结构**：
```
产品团队 A
├── 产品经理
├── 后端工程师 × 5
├── 前端工程师 × 3
├── QA × 1
└── SRE × 1-2  ← 嵌入式 SRE
```

**优势**：
- 深入理解业务上下文
- 快速响应产品需求
- 与开发团队紧密协作
- 能在设计阶段就参与可靠性评审

**风险与缓解**：

| 风险 | 缓解措施 |
|------|----------|
| 被当作"运维" | 明确 SRE 角色定义、50% 工程时间 |
| 孤岛效应 | SRE 社区（Guild）定期交流 |
| 标准不统一 | 共享 SRE 平台和最佳实践 |
| Toil 增长 | 定期度量并上报 |

### 4. 什么是 SRE 社区/行会（Guild）模式？

**答：** 当 SRE 分散在各产品团队时，通过 Guild 保持 SRE 之间的横向联系。

**Guild 运作方式**：
- **定期聚会**：每两周一次 SRE Guild 会议
- **知识分享**：分享各团队的 Postmortem 和最佳实践
- **标准制定**：共同制定 SRE 标准和规范
- **工具共建**：避免重复造轮子
- **互助支持**：On-Call 互备、故障排查支援

## 二、SRE 团队建设

### 5. SRE 团队需要多少人？如何确定团队规模？

**答：**

**经验法则**：

| 服务规模 | 建议 SRE 人数 | 说明 |
|----------|-------------|------|
| < 5 个微服务 | 0（开发自运维） | 未达到 SRE 团队门槛 |
| 5-20 个微服务 | 2-3 | 最小 On-Call 团队 |
| 20-50 个微服务 | 4-8 | 可以分专题方向 |
| 50-200 个微服务 | 8-15 | 分小组（平台/嵌入） |
| >200 个微服务 | 15+ | 需要分层 SRE 组织 |

**最小 On-Call 团队原则**：
- 至少 **8 人** 才能组建健康的 On-Call 轮值
- 低于 8 人时轮值间隔太短，容易疲劳
- 可以跨团队组建联合 On-Call

### 6. SRE 团队的服务接入（Onboarding）标准是什么？

**答：** 不是所有服务都应由 SRE 负责，需要满足一定标准：

**接入前置条件**：

| 条件 | 说明 |
|------|------|
| SLI/SLO 已定义 | 必须有可衡量的可靠性目标 |
| 监控和告警完善 | 基本的可观测性已建立 |
| 有 Runbook | 常见问题的处理文档 |
| 架构文档完整 | SRE 能理解系统架构 |
| 开发团队配合 | 签署错误预算策略 |
| 有自动化部署 | CI/CD 流水线已就绪 |

**服务退回机制**：
- 如果服务持续消耗过多 SRE 资源
- 开发团队不配合可靠性改进
- Toil 超标且开发团队不投入修复
- SRE 有权将服务退回给开发团队自运维

### 7. SRE 如何与开发团队协作（PRR - Production Readiness Review）？

**答：** PRR 是服务上线前的可靠性审查。

**PRR 检查清单**：

```markdown
## 架构审查
- [ ] 无单点故障
- [ ] 支持优雅降级
- [ ] 依赖关系明确

## 可观测性
- [ ] SLI/SLO 已定义
- [ ] 关键指标都有监控
- [ ] 告警规则已配置
- [ ] Dashboard 已创建

## 运维就绪
- [ ] Runbook 已编写
- [ ] 自动化部署已就绪
- [ ] 回滚方案已验证
- [ ] On-Call 流程已建立

## 容量规划
- [ ] 压测已完成
- [ ] 自动扩缩容已配置
- [ ] 资源 Request/Limit 已设置

## 安全
- [ ] Secret 管理规范
- [ ] 访问控制已配置
- [ ] 安全扫描已通过
```

## 三、组织演进

### 8. 从传统运维到 SRE 的转型路径是什么？

**答：**

**转型阶段**：

```
阶段1: 传统运维
  └─ 手动操作、工单驱动、被动响应

阶段2: 自动化运维
  └─ 脚本化、CI/CD、IaC

阶段3: 初级 SRE
  └─ 定义 SLO、建立 On-Call、写 Postmortem

阶段4: 成熟 SRE
  └─ 错误预算驱动、系统化消除 Toil

阶段5: 高级 SRE
  └─ 混沌工程、自愈系统、平台化赋能
```

**各阶段关键行动**：

| 阶段 | 关键行动 | 交付物 |
|------|----------|--------|
| 1→2 | 自动化部署和配置 | CI/CD 流水线、IaC 代码 |
| 2→3 | 定义 SLI/SLO、建立事件管理 | SLO 文档、Postmortem 模板 |
| 3→4 | 错误预算策略、Toil 度量与消除 | 错误预算策略、自动化工具 |
| 4→5 | 混沌工程、自服务平台 | 混沌实验、平台化产品 |

### 9. SRE 在小公司/初创企业中如何实践？

**答：** 小公司不一定需要专职 SRE 团队，但可以应用 SRE **原则和实践**。

**轻量级 SRE 实践**：

| 实践 | 小公司版本 | 成本 |
|------|-----------|------|
| SLI/SLO | 定义 2-3 个核心 SLO | 低 |
| 错误预算 | 简单的仪表盘跟踪 | 低 |
| On-Call | 开发团队轮值 | 低 |
| Postmortem | 每次较大故障后复盘 | 低 |
| 消除 Toil | 季度识别和消除 Top 3 Toil | 中 |
| 监控 | Prometheus + Grafana 开源方案 | 中 |

**关键原则**：
- 不需要 SRE 头衔，需要 SRE 思维
- 从最关键的服务开始
- 渐进式引入，不要一步到位
