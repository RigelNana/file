# SRE团队协作实践八股文

---

## 一、SRE 与开发团队协作

### 1. SRE 和开发团队之间的关系应该是怎样的？

**答：** SRE 与开发（Dev）团队是**合作伙伴关系**，而非对立关系。

**关系演进**：
```
传统模式：Dev 扔过墙 → Ops 接住  （对立）
DevOps：  Dev + Ops 协作          （协作）
SRE：     Dev + SRE 共同负责      （共责）
```

**共责模型**：

| 职责 | 开发团队 | SRE 团队 | 共同 |
|------|----------|----------|------|
| 功能开发 | ✅ | | |
| 架构设计 | ✅ | ✅ 评审 | |
| 代码审查 | ✅ | ✅ 可靠性相关 | |
| SLO 定义 | | | ✅ |
| On-Call | 二线 | 一线 | |
| Postmortem | | | ✅ |
| 容量规划 | | ✅ | |
| 发布管理 | ✅ 触发 | ✅ 保障 | |

### 2. SRE 与开发团队常见的摩擦有哪些？如何解决？

**答：**

| 摩擦点 | Dev 视角 | SRE 视角 | 解决方案 |
|--------|----------|----------|----------|
| 发布节奏 | "发布太慢了" | "风险太高了" | 错误预算：量化风险容忍度 |
| 告警处理 | "不是我的问题" | "是你的代码导致的" | SLO 共责：Dev 参与 On-Call |
| 技术债务 | "没时间修" | "影响可靠性" | 错误预算策略驱动修复 |
| 评审时间 | "评审太慢了" | "需要仔细检查" | 明确 SLA 和自动化检查 |

### 3. Production Readiness Review (PRR) 的流程是什么？

**答：**

**PRR 流程**：

```
开发团队提交 PRR 请求
         │
         ▼
SRE 初审（1-2天）
         │
    ┌────┴────┐
    │ 通过    │ 不通过
    │         │
    ▼         ▼
生产就绪   反馈改进项
              │
              ▼
         开发团队修复
              │
              ▼
         重新提交 PRR
```

## 二、跨团队协作模式

### 4. SRE 如何与产品团队协作？

**答：**

**协作节点**：

| 阶段 | 协作方式 | SRE 角色 |
|------|----------|----------|
| 需求阶段 | 评估可靠性影响 | 顾问 |
| 设计阶段 | 架构评审（PRR） | 审查者 |
| 开发阶段 | SLI 埋点指导 | 技术指导 |
| 测试阶段 | 负载/混沌测试 | 执行者 |
| 发布阶段 | 金丝雀发布保障 | 保障者 |
| 运行阶段 | On-Call + 监控 | 守护者 |

### 5. 什么是 Embedded SRE 的协作模式？

**答：**

**嵌入式协作流程**：

- **参加 Sprint Planning**：评估每个 Story 的可靠性影响
- **参加 Daily Standup**：同步运维状态和风险
- **代码审查**：重点审查可靠性相关代码
- **Tech Debt 管理**：推动可靠性相关技术债务修复
- **Postmortem 跟进**：确保改进项 Action Item 落地

**嵌入周期**：
```
一般建议嵌入 6-12 个月：
- 前 2 个月：学习业务和系统
- 中间 4-6 个月：深入优化、建立 SRE 实践
- 后 2-4 个月：知识转移、建立 Dev 自运维能力
```

### 6. SRE 如何进行知识传递和培训？

**答：**

| 方式 | 形式 | 频率 | 目标 |
|------|------|------|------|
| **Postmortem 分享** | 全公司邮件/会议 | 每次事故后 | 经验教训共享 |
| **SRE 工作坊** | 动手实验 | 季度 | 技能培训 |
| **Wheel of Misfortune** | 模拟故障演练 | 月度 | On-Call 训练 |
| **Design Review** | 架构评审 | 每个新项目 | 可靠性设计 |
| **文档** | Runbook/最佳实践 | 持续更新 | 知识沉淀 |
| **Pair On-Call** | 新人和老人一起值班 | 入职期间 | 新人培养 |

### 7. 什么是 Wheel of Misfortune？

**答：** 一种模拟故障演练的培训方式，类似 On-Call 的"沙盘推演"。

**演练流程**：
1. 准备故障场景卡片
2. 随机抽取场景
3. 新人扮演 On-Call 角色
4. 老人扮演"命运之轮"提供信息
5. 新人按照流程进行排查
6. 演练结束后回顾和反馈

**典型场景**：
- "用户报告支付失败，你收到了 P1 告警"
- "数据库主从延迟突然增大到 30 秒"
- "某服务 Pod 频繁 OOMKilled"

## 三、沟通与汇报

### 8. SRE 如何向管理层汇报工作？

**答：**

**关键汇报指标**：

| 指标 | 含义 | 目标 |
|------|------|------|
| SLO 达标率 | 各服务 SLO 完成情况 | ≥ SLO 目标 |
| 错误预算剩余 | 各服务错误预算消耗 | > 25% |
| MTTR | 平均故障恢复时间 | 持续下降 |
| Toil 占比 | Toil 时间/总工作时间 | < 50% |
| 事件数 | P0/P1/P2 事件数量 | 持续下降 |
| 自动化率 | 自动化操作/总操作 | 持续上升 |

**月度报告模板**：

```markdown
## SRE 月度报告 - 2024年X月

### 核心指标
- SLO 达标率：98.5%（3个服务违约）
- 平均错误预算剩余：62%
- P1 事件：2 次（上月 3 次）
- MTTR：23 分钟（上月 35 分钟）

### 重大事件回顾
1. [P1] 支付服务宕机 45 分钟 - 已完成 Postmortem

### 工程项目进展
1. 自动化回滚工具 - 完成 80%
2. SLO 仪表盘重构 - 已交付

### 下月计划
1. 完成自动化回滚工具上线
2. 开展 GameDay 演练
```

### 9. 面试题：你如何推动开发团队重视可靠性？

**答：**

1. **数据驱动**：用 SLO 数据和故障影响数据说话
2. **利益绑定**：让 Dev 参与 On-Call，亲身感受可靠性问题
3. **错误预算策略**：预算耗尽 → Dev 必须投入修复
4. **降低门槛**：提供易用的 SRE 工具和平台
5. **表彰激励**：表彰可靠性做得好的团队
6. **高层支持**：争取管理层将可靠性纳入 KPI
