# 错误预算与风险管理八股文

---

## 一、错误预算基础

### 1. 什么是错误预算（Error Budget）？它的核心思想是什么？

**答：** 错误预算是 SRE 中最重要的概念之一。其核心公式：

$$\text{错误预算} = 1 - \text{SLO}$$

**核心思想**：系统不需要 100% 可靠，SLO 与 100% 之间的差值就是我们可以"花费"的预算。

**通俗理解**：
- SLO = 99.9% 意味着系统**允许 0.1% 的不可靠**
- 这 0.1% 就是错误预算
- 可以用来"支付"新功能发布、系统变更等带来的风险

```
        100%
         │
    ┌────┴────┐
    │ 0.1%    │  ← 错误预算（可以用于创新和变更）
    ├─────────┤
    │         │
    │ 99.9%   │  ← SLO（必须保障的可靠性）
    │         │
    └─────────┘
         0%
```

### 2. 错误预算具体怎么计算？

**答：**

**按时间计算（可用性 SLO）：**

| SLO | 每月错误预算 | 每季度错误预算 | 每年错误预算 |
|-----|-------------|---------------|-------------|
| 99% | 7 小时 12 分 | 21 小时 36 分 | 3 天 15 小时 |
| 99.5% | 3 小时 36 分 | 10 小时 48 分 | 1 天 19 小时 |
| 99.9% | 43 分 12 秒 | 2 小时 10 分 | 8 小时 46 分 |
| 99.95% | 21 分 36 秒 | 1 小时 5 分 | 4 小时 23 分 |
| 99.99% | 4 分 19 秒 | 12 分 58 秒 | 52 分 34 秒 |
| 99.999% | 26 秒 | 1 分 18 秒 | 5 分 15 秒 |

**按请求计算（成功率 SLO）：**

```
SLO = 99.9% 成功率
月总请求 = 10,000,000
错误预算 = 10,000,000 × 0.1% = 10,000 次失败请求
```

**按事件计算（延迟 SLO）：**

```
SLO = 99% 的请求在 200ms 内完成
月总请求 = 1,000,000
错误预算 = 1,000,000 × 1% = 10,000 次超时请求
```

### 3. 错误预算的滚动窗口和日历窗口有什么区别？

**答：**

| 维度 | 日历窗口 | 滚动窗口 |
|------|----------|----------|
| **定义** | 固定时间段（如每月 1-30 号） | 过去 N 天的连续窗口 |
| **重置** | 每月/季初自动重置 | 不会重置，持续滑动 |
| **优点** | 简单易理解，与业务周期对齐 | 避免月初"挥霍"预算 |
| **缺点** | 月初预算充足导致冒进 | 持续消耗无恢复期 |
| **推荐** | 管理和汇报 | 告警和工程决策 |

**日历窗口的"月初效应"问题**：
```
月初：预算 100% ─→ 团队激进发布 ─→ 大量消耗
月中：预算 20%  ─→ 开始紧张
月末：预算 0%   ─→ 冻结变更

下月初：预算重置 100% ─→ 又开始激进...
```

**最佳实践**：同时使用两种窗口：
- 30 天滚动窗口用于**告警和日常决策**
- 日历季度用于**汇报和回顾**

## 二、错误预算策略

### 4. 什么是错误预算策略（Error Budget Policy）？为什么需要它？

**答：** 错误预算策略是一份**预先达成共识的书面协议**，定义了在不同错误预算水平下团队应采取的行动。

**为什么需要**：
- 避免事后争论"该不该冻结发布"
- 让所有利益相关者提前达成一致
- 将主观判断变为**客观规则驱动**的决策
- 减少 SRE 与开发团队的摩擦

**错误预算策略模板**：

```markdown
## 错误预算策略 v1.0

### 签署方
- 产品 Owner：XXX
- 开发 Tech Lead：XXX  
- SRE Tech Lead：XXX

### 策略级别

#### 绿色（预算 > 50%）
- 正常发布节奏
- 标准变更审查流程
- 可以进行实验性变更

#### 黄色（预算 25% ~ 50%）
- 减少非关键功能发布
- 加强变更审查
- 暂停实验性变更

#### 红色（预算 < 25%）  
- 冻结所有非紧急功能变更
- 仅允许可靠性修复和安全补丁
- 开发团队投入 bug 修复

#### 黑色（预算耗尽）
- 冻结所有代码变更
- 开发团队全力投入可靠性改进
- 直到错误预算恢复到 25% 以上
```

### 5. 错误预算消耗的常见原因有哪些？

**答：**

| 类别 | 具体原因 | 应对措施 |
|------|----------|----------|
| **代码变更** | Bug、性能退化、兼容性问题 | 加强测试、金丝雀发布 |
| **配置变更** | 错误配置、Feature Flag 问题 | 配置审查、渐进式生效 |
| **基础设施** | 云平台故障、硬件故障 | 多可用区部署、冗余设计 |
| **依赖故障** | 第三方服务宕机、超时 | 熔断降级、缓存兜底 |
| **流量异常** | 突增流量、DDoS 攻击 | 自动扩缩容、限流 |
| **运维操作** | 维护窗口、数据迁移 | 灰度操作、回滚预案 |

### 6. 错误预算消耗后如何恢复？

**答：**

**短期止血**：
1. 回滚导致问题的变更
2. 启动降级/熔断保护
3. 临时增加冗余资源

**中期修复**：
1. 修复根因问题
2. 补充缺失的监控和告警
3. 更新 Runbook

**长期预防**：
1. 架构层面增强可靠性
2. 改进发布流程（金丝雀、Feature Flag）
3. 增强测试覆盖（混沌测试、负载测试）

```
错误预算恢复路径：

耗尽 ──→ 冻结变更 ──→ 修复问题 ──→ 预算逐步恢复
  │                                      │
  │        恢复到 25%                     │
  │←─────── 解除红色 ←──────────────────┘
  │
  │        恢复到 50%
  │←─────── 解除黄色，恢复正常发布节奏
```

## 三、风险管理

### 7. SRE 如何看待风险？"拥抱风险"是什么意思？

**答：** Google SRE 认为风险是**不可避免**且**可以被管理**的，关键是找到**可靠性与创新速度的平衡点**。

**"拥抱风险"的含义**：
- ❌ 不是鲁莽行事、忽视风险
- ✅ 是承认系统不可能 100% 可靠
- ✅ 是用错误预算**量化**可接受的风险水平
- ✅ 是在预算允许的范围内鼓励创新和变更

**传统运维 vs SRE 的风险观**：

| 维度 | 传统运维 | SRE |
|------|----------|-----|
| 对风险态度 | 规避：变更越少越好 | 管理：在预算内拥抱变更 |
| 衡量标准 | 可用性越高越好 | 刚好满足 SLO 即可 |
| 变更策略 | 大批量低频率 | 小批量高频率 |
| 故障处理 | 追责 | 学习（Blameless） |

### 8. 如何进行服务风险评估？

**答：**

**风险评估矩阵**：

```
          低影响          高影响
高概率 │  中风险   │   高风险    │
       │ 需要计划  │  立即处理   │
       ├──────────┼────────────┤
低概率 │  低风险   │   中风险    │
       │ 接受/监控 │  制定预案   │
```

**风险评估维度**：

| 维度 | 评估内容 | 示例 |
|------|----------|------|
| **TTD** | 检测时间 | 告警延迟、监控覆盖 |
| **TTR** | 恢复时间 | 回滚速度、Runbook 成熟度 |
| **影响面** | 受影响用户比例 | 全站 vs 单个功能 |
| **频率** | 故障发生概率 | 历史故障频率统计 |

**风险量化公式**：

$$\text{风险} = \text{概率} \times \text{影响} = \text{频率} \times \text{用户影响} \times \text{持续时间}$$

### 9. 什么是风险登记册（Risk Register）？

**答：** 风险登记册是记录和跟踪所有已识别风险的文档。

| 风险 ID | 描述 | 概率 | 影响 | 风险评分 | 缓解计划 | Owner | 状态 |
|---------|------|------|------|----------|----------|-------|------|

## 四、错误预算的组织落地

### 10. 如何推动错误预算策略在组织中落地？

**答：**

**落地步骤**：

1. **教育与对齐** → 向管理层和开发团队介绍错误预算概念
2. **选择试点服务** → 选择一个核心但非最关键的服务
3. **定义 SLI/SLO** → 基于用户体验选择 SLI，设定合理 SLO
4. **建立度量体系** → 实时计算 SLI 达标率和错误预算剩余
5. **制定策略文档** → 各方签署错误预算策略
6. **执行与迭代** → 按策略执行，定期回顾和调整
7. **推广到更多服务** → 成功经验推广到其他团队

**常见阻力与应对**：

| 阻力 | 来源 | 应对方式 |
|------|------|----------|
| "我们不能冻结发布" | 产品经理 | 展示故障对用户的影响数据 |
| "SLO 定太高了" | 开发团队 | 从宽松 SLO 开始，逐步收紧 |
| "没时间搞这个" | 管理层 | 用故障成本数据说话 |

### 11. 多个 SLO 的错误预算如何管理？

**答：** 一个服务通常有多个 SLO（可用性、延迟、错误率），每个 SLO 有独立的错误预算。

```
服务 A 的 SLO：
├── 可用性 SLO: 99.9%    → 错误预算: 43分钟/月 → 剩余 78%
├── 延迟 SLO: P99<200ms  → 错误预算: 1%超标   → 剩余 45%
└── 错误率 SLO: <0.1%    → 错误预算: 0.1%     → 剩余 92%
```

**决策规则**：
- **任何一个**SLO 的错误预算耗尽 → 触发对应策略
- 策略级别取所有 SLO 中**最严格**的那个

### 12. 面试题：如果开发团队不认同冻结发布的决定，你怎么办？

**答：**

1. **拿数据说话**：展示错误预算消耗趋势和用户影响数据
2. **回归策略文档**：这是各方**提前签署的策略**，按规则执行
3. **提供替代方案**：只允许低风险变更、更安全的发布方式
4. **升级决策**：无法达成一致时升级到双方管理层
