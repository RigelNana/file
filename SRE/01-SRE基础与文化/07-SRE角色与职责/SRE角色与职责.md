# SRE角色与职责八股文

## 一、SRE 角色定义

### 1. SRE 工程师的日常职责包括哪些？

**答：**

| 职责类别 | 具体工作 | 时间占比 |
|----------|----------|----------|
| **工程项目** | 构建自动化工具、改进系统 | ≥50% |
| **On-Call** | 值班响应告警、处理事件 | 15-25% |
| **Toil** | 手动运维操作 | <25% |
| **协作** | 与开发团队评审、PRR | 10-15% |
| **文档** | Runbook、Postmortem、架构文档 | 5-10% |

**典型一周**：

```
周一：Postmortem 跟进 + 工程项目开发
周二：工程项目开发（自动化工具）
周三：On-Call 交接 + 容量审查会议
周四：On-Call 值班 + 告警处理
周五：On-Call 值班 + Toil 消除计划更新
周末：On-Call（被动值班，仅响应紧急告警）
```

### 3. SRE 与相关岗位的区别？

**答：**

| 维度 | SRE | DevOps 工程师 | 平台工程师 | 系统管理员 |
|------|-----|-------------|-----------|-----------|
| 核心目标 | 可靠性 | 交付效率 | 开发者体验 | 系统运行 |
| 编码比例 | ≥50% | 30-50% | 70-80% | <20% |
| 关注指标 | SLI/SLO | 部署频率 | 平台使用率 | 可用性 |
| On-Call | 核心职责 | 可能有 | 通常没有 | 通常有 |
| 对象 | 生产系统 | 交付流程 | 开发平台 | 基础设施 |

## 二、SRE 职责深入

### 4. SRE 如何参与系统设计评审？

**答：** SRE 在系统设计阶段提供可靠性视角。

**设计评审关注点**：

| 关注点 | 评审问题 |
|--------|----------|
| **单点故障** | 系统是否存在 SPOF？ |
| **故障域** | 故障影响范围有多大？ |
| **降级策略** | 依赖不可用时如何降级？ |
| **可观测性** | 如何发现和诊断问题？ |
| **容量** | 能承受多大流量？扩展方式？ |
| **恢复** | 灾难恢复方案是什么？RTO/RPO？ |
| **安全** | 敏感数据如何保护？ |

**评审输出模板**：
```markdown
## 可靠性设计评审 - [服务名]

### 风险识别
1. 数据库无多可用区部署 → 风险等级: 高
2. 未定义降级策略 → 风险等级: 中

### SLO 建议
- 可用性: 99.95%
- P99 延迟: <300ms

### 改进建议
1. 数据库增加 Read Replica
2. 添加缓存层减少 DB 压力
3. 实现熔断器模式
```

### 5. SRE 的 On-Call 职责是什么？

**答：**

**On-Call 期间职责**：

1. **主要职责**：
   - 响应告警并进行初步诊断
   - 执行应急操作（回滚、扩容、降级）
   - 判断是否需要升级事件严重程度
   - 协调跨团队资源

2. **次要职责**：
   - 记录事件时间线
   - 更新状态页面
   - 非紧急告警的分析和工单创建

3. **On-Call 结束后**：
   - 编写 Postmortem
   - 更新 Runbook
   - 交接未完成的告警和工单

### 6. SRE 团队中的不同角色分工？

**答：**

| 角色 | 职责 | 关注领域 |
|------|------|----------|
| **SRE Manager** | 团队管理、资源分配、与管理层沟通 | 团队建设、策略 |
| **Tech Lead** | 技术方向、架构决策、代码审查 | 技术长板 |
| **On-Call SRE** | 值班响应、事件处理 | 日常运维 |
| **工具链 SRE** | 构建和维护 SRE 工具与平台 | 自动化 |
| **嵌入式 SRE** | 在特定产品团队提供 SRE 支持 | 业务可靠性 |

## 三、SRE 面试视角

### 7. 面试题：你认为好的 SRE 应该具备什么素质？

**答：**

**技术素质**：
- 扎实的系统知识（OS、网络、分布式）
- 编程能力（能写生产级别的工具）
- 故障排查的方法论和直觉

**非技术素质**：
- **冷静**：紧急事件中保持冷静思考
- **沟通**：能清晰表达技术问题和解决方案
- **好奇心**：对系统为什么这样运行感兴趣
- **同理心**：理解用户的需求和开发的压力
- **责任感**：对系统可靠性有 Ownership

### 8. 面试题：描述你做过的最有价值的 SRE 项目。

**答：** 用 STAR 方法回答：

**模板**：
- **Situation**：系统当时面临什么问题
- **Task**：你的职责是什么
- **Action**：你做了什么（具体技术方案）
- **Result**：结果如何（量化数据）

**示例**：
> S: 我们的微服务每月有 3-5 次 P1 事故，错误预算经常耗尽
> T: 作为 SRE，我需要将可靠性提升到 99.95%
> A: 我做了三件事：1) 建立了完整的 SLO 体系 2) 引入金丝雀发布 3) 构建了自动化回滚系统
> R: P1 事故减少到每季度 1 次，可用性从 99.8% 提升到 99.96%
