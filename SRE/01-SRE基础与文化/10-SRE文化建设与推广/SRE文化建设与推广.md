# SRE 文化建设与推广八股文

---

## 一、SRE 文化核心要素

### 1. SRE 文化的核心要素有哪些？

**答：**

| 文化要素 | 描述 | 反模式 |
|----------|------|--------|
| **无责文化** | 事故复盘不追责个人 | 找"替罪羊" |
| **数据驱动** | 用 SLO、指标做决策 | 凭直觉判断 |
| **工程思维** | 用代码解决运维问题 | 手动操作一切 |
| **持续改进** | 从错误中学习优化 | 同样的问题反复出现 |
| **透明共享** | 信息公开、知识共享 | 信息垄断 |
| **风险量化** | 用错误预算量化风险 | "绝对安全"或"无所谓" |

### 2. 什么是无责文化（Blameless Culture）？如何建立？

**答：** 无责文化指在事故发生后，关注**系统性原因**而非个人过错。

**核心理念**：
```
传统文化：              无责文化：
"谁干的？"       →     "什么环节出了问题？"
"这个人犯了错"   →     "系统允许这个错误发生"
"惩罚他"         →     "改进流程和防护"
"隐瞒错误"       →     "鼓励报告和分享"
```

**建立无责文化的手段**：

1. **Postmortem 规范**：
   - 只描述事实和时间线
   - 用"系统"而非"人"作为主语
   - 关注 Action Item 而非追责

2. **领导层以身作则**：
   - 高管主动分享自己的失误
   - 在全员会议上表彰事故报告者

3. **制度保障**：
   - 明文规定 Postmortem 无责原则
   - 将学习（而非惩罚）作为目标

### 3. 如何理解"无责"并不等于"无责任"？

**答：**

| 无责 | 有责任 |
|------|--------|
| 不因犯错惩罚个人 | 每个人有义务遵守流程 |
| 不追究"谁按了错误的按钮" | 但要追究"为什么按错按钮不会被阻止" |
| 鼓励报告和透明 | 隐瞒或篡改信息是不可接受的 |
| 关注系统原因 | 故意的疏忽或违规另行处理 |

## 二、SRE 文化推广策略

### 4. 如何在传统运维团队推广 SRE 文化？

**答：**

**推广路线图**：

```
第1步：定义目标          （为什么要做 SRE）
    │
    ▼
第2步：选择试点服务      （低风险高可见度）
    │
    ▼
第3步：建立 SLO          （量化可靠性目标）
    │
    ▼
第4步：引入错误预算      （量化风险容忍度）
    │
    ▼
第5步：实施 Postmortem    （建立无责文化）
    │
    ▼
第6步：度量 Toil          （推动自动化）
    │
    ▼
第7步：扩展到更多服务    （逐步推广）
```

### 5. 推广 SRE 文化时常见的阻力有哪些？如何应对？

**答：**

| 阻力 | 来源 | 应对策略 |
|------|------|----------|
| "我们一直这样做的" | 一线运维 | 展示 SRE 实践的量化收益 |
| "没有时间做自动化" | 团队 | 用数据证明 Toil 占比和浪费 |
| "SRE 是大厂才需要的" | 管理层 | 从小规模试点开始证明价值 |
| "会不会导致裁员" | 运维人员 | 强调"自动化是为了做更有价值的事" |
| "开发不配合" | 跨团队 | 错误预算策略建立利益绑定 |

### 6. 如何衡量 SRE 文化建设的效果？

**答：**

| 指标 | 衡量方式 | 目标趋势 |
|------|----------|----------|
| Postmortem 数量 | 事故/Postmortem 比值 | 趋近 1:1 |
| Postmortem 质量 | 评审打分 | 持续提升 |
| Action Item 完成率 | 已完成/总数 | > 80% |
| 重复事故率 | 同类事故复发次数 | 持续下降 |
| Toil 占比 | 定期调查 | < 50% |
| SLO 覆盖率 | 已定义 SLO 的服务比例 | > 90% |
| 工程师满意度 | NPS 调查 | 持续提升 |

## 三、SRE 实践推广方式

### 7. 如何建立 SRE 社区推动知识共享？

**答：**

**社区活动**：

| 活动 | 频率 | 内容 |
|------|------|------|
| SRE 周会 | 每周 | 分享当周事件、新工具、经验 |
| 技术分享 | 双周 | 深入技术话题的 Lightning Talk |
| 读书会 | 月度 | 共读 SRE 经典书籍 |
| 内部 SREcon | 季度/年度 | 全公司 SRE 经验分享 |
| Mob Postmortem | 按需 | 跨团队共同分析复杂事故 |
| Chaos Day | 季度 | 联合混沌工程演练 |

### 8. 如何建立公司级的 SRE 最佳实践库？

**答：**

**知识体系结构**：
```
SRE 知识库
├── Runbook/           运维手册
│   ├── 服务A/
│   └── 服务B/
├── Postmortem/        事后复盘
│   ├── 2024/
│   └── 模板/
├── Best-Practices/    最佳实践
│   ├── SLO/
│   ├── On-Call/
│   └── 自动化/
├── Tools/             工具指南
│   ├── 监控/
│   └── 部署/
└── Training/          培训材料
    ├── 新人入门/
    └── 高级进阶/
```

**维护机制**：
- 每个文档指定 Owner
- Postmortem 自动归档
- 周期性审查和更新（至少季度）
- 新人入职时的必读清单

### 9. 如何建立可靠性评审委员会？

**答：**

**委员会组成**：
- SRE 负责人（主席）
- 各业务线 SRE 代表
- 架构师代表
- 管理层代表

**职责**：
1. 审批新服务的 SLO 定义
2. 评审重大架构变更
3. 审查月度可靠性报告
4. 决策错误预算耗尽后的策略
5. 推动跨团队可靠性改进

## 四、面试高频题

### 10. 面试题：如何说服管理层投入 SRE？

**答：** 用**业务语言**讲 SRE 的价值。

**ROI 展示**：

| 投入 | 产出 |
|------|------|
| 1 名 SRE | 减少 50% 重复运维工作 |
| 建立 SLO | 减少无效的"绝对安全"投入 |
| Postmortem | 重复事故减少 60%+ |
| 自动化工具 | 发布效率提升 3-5 倍 |
| 混沌工程 | 提前发现 30%+ 的潜在故障 |

**案例讲述框架**：
```
Before SRE:
- 每月 10+ 次人工变更失败
- MTTR 2 小时
- 同类事故反复发生

After SRE:
- 自动化变更，失败率降低 80%
- MTTR 降至 15 分钟
- 通过 Postmortem 消除了重复事故
```

### 11. 面试题：你如何从零开始在公司建立 SRE 文化？

**答：**

```
阶段一（第 1-3 个月）：播种
├── 选择 1 个核心服务作为试点
├── 定义 SLI/SLO
├── 建立基础监控和告警
└── 开始做 Postmortem

阶段二（第 3-6 个月）：扎根
├── 引入错误预算机制
├── 建立 On-Call 制度
├── 度量并消除 Toil
└── 定期分享 SRE 成果

阶段三（第 6-12 个月）：成长
├── 扩展到 3-5 个服务
├── 建立 SRE 社区
├── 开发自动化工具平台
└── 培训更多工程师

阶段四（第 12+ 个月）：开花
├── 全公司 SRE 实践覆盖
├── 建立可靠性评审机制
├── 引入混沌工程
└── 持续优化和创新
```

**核心建议**：
- **先做出成绩再推广**——用数据说话
- **从痛点入手**——解决团队最头疼的问题
- **不要全盘照搬**——因地制宜
- **持续沟通**——定期向各层级汇报 SRE 的价值
