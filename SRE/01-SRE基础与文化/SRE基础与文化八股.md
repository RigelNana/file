# SRE 基础与文化八股文

---

## 一、SRE 核心理念

### 1. 什么是 SRE？SRE 的起源是什么？

**答：** SRE（Site Reliability Engineering，站点可靠性工程）是 Google 于 2003 年由 Ben Treynor Sloss 创立的一套工程实践方法论。其核心思想是 **用软件工程的方法解决运维问题**。

SRE 的核心定义：
- **"SRE is what happens when you ask a software engineer to design an operations team"** —— Ben Treynor
- SRE 团队由软件工程师组成，负责运行大规模生产系统
- 目标是构建和维护**高可靠、可扩展**的系统

SRE 的三大支柱：
1. **拥抱风险**：100% 可靠性不是目标，要在可靠性和创新速度之间寻找平衡
2. **消除劳作**：通过自动化减少重复性、手工性的运维工作
3. **数据驱动**：基于 SLI/SLO 做决策，而非凭直觉

### 2. SRE 和传统运维（Ops）有什么区别？

**答：**

| 维度 | 传统运维 | SRE |
|------|----------|-----|
| 背景 | 系统管理员 | 软件工程师 |
| 工作方式 | 手动操作、工单驱动 | 自动化、代码驱动 |
| 核心指标 | 系统可用性（uptime） | SLI/SLO/错误预算 |
| 对变更的态度 | 抵触变更（变更=风险） | 拥抱变更（在错误预算内） |
| 扩展方式 | 线性增长（加人） | 通过自动化亚线性增长 |
| 故障处理 | 灭火式响应 | 系统化预防+事后复盘 |
| 编码时间 | 很少写代码 | 至少 50% 时间写代码 |

### 3. SRE 和 DevOps 的关系是什么？

**答：** Google 的说法是：**"class SRE implements interface DevOps"**。

| 维度 | DevOps | SRE |
|------|--------|-----|
| 本质 | 文化与运动 | 具体的岗位与实践 |
| 关注点 | 打破开发和运维的壁垒 | 用工程方法保障可靠性 |
| 核心指标 | 部署频率、变更前置时间 | SLI/SLO、错误预算 |
| 方法论 | CI/CD、IaC、自动化 | 错误预算、Toil消除、On-Call |
| 工具侧重 | CI/CD 流水线、容器编排 | 监控告警、容量规划、事件管理 |
| 组织形式 | 文化变革，人人负责 | 专职 SRE 团队 |

两者互补关系：
- DevOps 提供了**理念和原则**（协作、自动化、持续改进）
- SRE 提供了**具体的实践和方法**（错误预算、SLO、Toil 消除）

## 二、错误预算

### 4. 什么是错误预算（Error Budget）？

**答：** 错误预算 = **1 - SLO 目标值**。它量化了系统在一定时间窗口内"允许不可靠"的额度。

**示例：**
- SLO = 99.9% 可用性
- 错误预算 = 0.1%
- 一个月（30天）的错误预算 = 30 × 24 × 60 × 0.1% = **43.2 分钟**

**错误预算的意义：**

1. **平衡可靠性与创新速度**：
   - 错误预算充足 → 可以加速发布新功能
   - 错误预算耗尽 → 冻结非关键变更，专注修复可靠性

2. **消除开发与运维的对立**：
   - 开发团队希望快速发布、运维团队希望系统稳定
   - 错误预算提供了一个**双方都认可的量化标准**

3. **驱动正确行为**：
   - 让开发团队为系统可靠性负责
   - 让 SRE 团队不过度追求稳定性

### 5. 错误预算耗尽后应该怎么做？

**答：** 典型的错误预算策略（Error Budget Policy）：

1. **冻结功能发布**：暂停非紧急的功能变更
2. **聚焦可靠性**：开发团队投入精力修复可靠性问题
3. **加强代码审查**：提高变更的审查标准
4. **增加测试覆盖**：补充缺失的测试用例
5. **改善监控**：修复告警盲区
6. **复盘分析**：对消耗错误预算的事件进行深入分析

## 三、消除劳作（Toil）

### 6. 什么是 Toil？如何识别和消除 Toil？

**答：** Toil（劳作）是指与运行生产服务相关的**手动的、重复的、可自动化的、战术性的、没有持久价值的**工作。

**Toil 的六大特征：**

| 特征 | 说明 | 示例 |
|------|------|------|
| 手动的 | 需要人工执行 | 手动重启服务 |
| 重复的 | 反复执行相同操作 | 每天手动清理磁盘 |
| 可自动化的 | 机器可以完成 | 手动扩容实例 |
| 战术性的 | 响应式而非战略性的 | 处理工单请求 |
| 无持久价值 | 不能永久改善服务 | 手动处理告警 |
| 随规模线性增长 | 服务越大工作量越多 | 逐台机器打补丁 |

**Google 的原则**：SRE 团队的 Toil 占比不应超过 **50%**，至少有一半时间应用于工程项目。

**消除 Toil 的策略：**
1. 度量 Toil（记录每周花在 Toil 上的时间）
2. 优先自动化高频、高耗时的 Toil
3. 构建自服务平台（Self-Service）
4. 消除 Toil 的根因而不是症状

## 四、SRE 组织模型

### 7. SRE 团队常见的组织模型有哪些？

**答：**

| 模型 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **集中式 SRE** | 独立的 SRE 部门服务多个产品 | 标准化强、资源池化 | 业务理解不深 |
| **嵌入式 SRE** | SRE 嵌入到各产品团队 | 深入理解业务 | 容易变成救火队员 |
| **咨询式 SRE** | SRE 作为内部顾问提供指导 | 影响面广、杠杆效应大 | 执行力依赖产品团队 |
| **平台 SRE** | 构建可靠性平台和工具 | 规模化赋能 | 与业务脱节风险 |
| **混合模式** | 以上模型的组合 | 灵活适配 | 管理复杂 |

### 8. 什么是 SRE 的 50% 原则？

**答：** Google SRE 规定团队成员应将不超过 50% 的时间用于运维工作（Toil/On-Call），至少 50% 的时间应投入到**工程项目**中。

工程项目包括：
- 构建自动化工具
- 改进监控和告警系统
- 优化系统架构
- 编写文档和 Runbook
- 容量规划和性能优化

如果 Toil 超过 50%：
- 应向管理层上报
- 重新优先级排序，投入资源自动化
- 必要时可以将服务退回给开发团队
