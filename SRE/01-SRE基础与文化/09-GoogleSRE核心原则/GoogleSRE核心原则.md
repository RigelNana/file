# Google SRE 核心原则八股文

---

## 一、Google SRE 起源与背景

### 1. Google SRE 的起源是什么？

**答：** Google SRE 由 Ben Treynor Sloss 于 2003 年创立。当时 Google 面临飞速增长的基础设施管理挑战，传统运维模式无法扩展。

**核心理念**：
```
"SRE is what happens when you ask a software engineer
 to design an operations team."
                              — Ben Treynor Sloss
```

**创立背景**：

| 时间 | 事件 |
|------|------|
| 2003 | Ben Treynor 组建第一个 SRE 团队 |
| 2014 | SREcon 首届会议 |
| 2016 | 《Site Reliability Engineering》出版 |
| 2018 | 《The Site Reliability Workbook》出版 |
| 2020+ | SRE 成为行业标准实践 |

### 2. Google SRE 的七大核心原则是什么？

**答：**

| 原则 | 英文 | 含义 |
|------|------|------|
| **拥抱风险** | Embracing Risk | 100% 可靠不现实也不必要，量化可接受的风险 |
| **服务等级目标** | Service Level Objectives | 用 SLI/SLO/SLA 量化可靠性目标 |
| **消除劳作** | Eliminating Toil | 用自动化取代重复性手动操作 |
| **监控分布式系统** | Monitoring Distributed Systems | 可观测性是基础设施的核心 |
| **发布工程** | Release Engineering | 可重复、可靠的发布流程 |
| **简单性** | Simplicity | 系统设计追求简单，避免过度工程 |
| **软件工程方法** | Software Engineering | 用工程方法解决运维问题 |

## 二、拥抱风险（Embracing Risk）

### 3. 为什么 Google 认为 100% 可靠是错误的目标？

**答：**

**原因分析**：
1. **成本指数增长**：从 99.9% → 99.99% 的成本远大于从 99% → 99.9%
2. **用户感知上限**：用户访问路径上其他组件的可靠性限制了最终体验
3. **创新受限**：过高的可靠性目标会阻碍功能迭代

**成本与可靠性关系**：
```
可靠性    投入成本   用户感知提升
99%       $         较大
99.9%     $$        明显
99.99%    $$$$      微小
99.999%   $$$$$$$$  几乎无法感知
```

### 4. Google 如何利用错误预算管理风险？

**答：**

```
SLO = 99.9%
错误预算 = 1 - 0.999 = 0.1%
每月允许宕机时间 ≈ 43.2 分钟

                     错误预算消耗
月初 ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓ 100%
     │
第1周 ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓░░░░  80%  ← 一次小故障
     │
第2周 ▓▓▓▓▓▓▓▓▓▓▓▓░░░░░░░░  60%  ← 发布导致
     │
第3周 ▓▓▓▓▓▓▓▓░░░░░░░░░░░░  40%  ← 日常消耗
     │
决策点：停止高风险变更，投入可靠性工作
```

## 三、消除劳作（Eliminating Toil）

### 5. Google SRE 对 Toil 的定义有哪些特征？

**答：**

| 特征 | 描述 | 示例 |
|------|------|------|
| **手动的** | 需要人工操作 | 手动扩容 |
| **重复的** | 反复执行相同操作 | 每周重启服务 |
| **可自动化的** | 理论上可以用程序替代 | 手动合并配置 |
| **战术性的** | 响应式而非战略性的 | 处理工单 |
| **无持久价值** | 不会带来持久改进 | 每次只是临时修复 |
| **随规模线性增长** | 工作量随服务规模增长 | 手动添加机器 |

**Google 的 50% 规则**：
```
SRE 工程师的时间分配（目标）：
┌──────────────────────────────────────┐
│ ≤ 50% Toil（运维性工作）             │
│  - On-Call                           │
│  - 工单处理                          │
│  - 手动操作                          │
├──────────────────────────────────────┤
│ ≥ 50% 工程项目                       │
│  - 自动化开发                        │
│  - 工具建设                          │
│  - 架构改进                          │
└──────────────────────────────────────┘

如果 Toil > 50%，说明团队需要更多工程化投入
```

### 6. Google 有哪些经典的内部自动化工具？

**答：**

| 工具/系统 | 功能 | SRE 价值 |
|-----------|------|----------|
| **Borg** | 容器编排（K8s 前身） | 自动化部署和资源管理 |
| **Monarch** | 监控系统 | 大规模指标采集和告警 |
| **Spanner** | 全球分布式数据库 | 强一致性 + 高可用 |
| **Borgmon** | 监控报警（Prometheus 灵感来源） | 黑盒 + 白盒监控 |
| **Rapid** | 发布系统 | 自动化发布流程 |
| **Prodtest** | 生产测试 | 持续验证生产环境 |

## 四、监控与可观测性原则

### 7. Google SRE 关于告警的"三个问题"是什么？

**答：** 每条告警都应该能回答以下三个问题：

```
1. 这条告警对应的规则触发了吗？   → 检查告警有效性
2. 需要人工立即介入吗？          → 判断紧急性
3. 我能采取什么行动？            → 确认可操作性
```

**告警分类**：

| 分类 | 描述 | 处理方式 |
|------|------|----------|
| **紧急告警（Page）** | 需要人工立即处理 | 触发 On-Call 响应 |
| **工单告警（Ticket）** | 需要人工处理但不紧急 | 创建工单在工作时间处理 |
| **日志告警（Log）** | 仅供事后分析 | 记录日志，按需查看 |

### 8. Google 的监控四大黄金信号是什么？

**答：**

| 黄金信号 | 英文 | 含义 | 示例 |
|----------|------|------|------|
| **延迟** | Latency | 请求处理时间 | P50/P99 响应时间 |
| **流量** | Traffic | 系统负载量 | QPS、每秒请求数 |
| **错误率** | Errors | 失败请求比例 | HTTP 5xx 比例 |
| **饱和度** | Saturation | 资源使用程度 | CPU/内存接近极限 |

## 五、发布工程与简单性原则

### 9. Google 的发布工程原则有哪些？

**答：**

**四大原则**：
1. **自助式（Self-Service）**：开发团队自行发布
2. **频繁发布**：小批量频繁发布降低风险
3. **密封构建（Hermetic Builds）**：构建结果可复现
4. **策略执行**：构建和发布策略自动化执行

**发布流水线**：
```
代码提交
    │
    ▼
自动构建（密封构建）
    │
    ▼
自动测试（单元 + 集成）
    │
    ▼
金丝雀发布（1%）
    │
    ▼
逐步灰度（10% → 50% → 100%）
    │
    ▼
全量发布
    │
    ▼
生产验证
```

### 10. Google SRE 为什么强调简单性？

**答：**

**核心理念**：
- 简单系统更**可靠**——组件少、故障模式少
- 简单系统更**可理解**——排查更快
- 简单系统更**可维护**——修改更安全

**实践建议**：

| 实践 | 描述 |
|------|------|
| **最小 API** | 只暴露必要的接口 |
| **推拖删** | Push back → Delay → Delete 不必要的功能 |
| **避免过度工程** | 不为假设性需求设计 |
| **定期清理** | 删除未使用的代码和配置 |
| **代码审查** | 关注复杂性引入 |

## 六、面试高频题

### 11. Google SRE 的 50% 规则在实际中如何执行？

**答：**

**度量方式**：
- 定期统计工程师时间分配
- 分类：Toil / 工程项目 / 学习培训
- 如果某服务 Toil 超过 50%，启动改进

**超标处理**：
```
Toil > 50%
    │
    ├── 短期：从开发团队借调人手
    │
    ├── 中期：优先开发自动化工具
    │
    └── 长期：退回服务给开发团队
           ↑
    Google独有做法：如果服务太不可靠，
    SRE有权"退回"服务给Dev，不再提供SRE支持
```

### 12. Google SRE 如何做到服务退回（Hand-back）？

**答：**

**退回条件**（通常满足以下条件时）：
1. Toil 持续超过 50% 且无改善趋势
2. 开发团队不配合优化
3. 错误预算持续耗尽
4. 架构设计不合理且拒绝重构

**退回流程**：
1. SRE 团队提出退回建议
2. 与开发团队和管理层三方会议
3. 明确退回条件和改进目标
4. 设定 3-6 个月改进期
5. 如果改进达标，继续由 SRE 支持
6. 如果未达标，执行退回：On-Call 转给 Dev

### 13. 面试题：如何在非 Google 的企业落地 Google SRE 原则？

**答：**

**分阶段落地**：

| 阶段 | 行动 | 周期 |
|------|------|------|
| **阶段一** | 定义 SLI/SLO，建立监控 | 1-3 个月 |
| **阶段二** | 引入错误预算，建立 On-Call | 3-6 个月 |
| **阶段三** | 度量并消除 Toil | 6-9 个月 |
| **阶段四** | Postmortem 和持续改进 | 9-12 个月 |
| **阶段五** | 混沌工程和高级实践 | 12+ 个月 |

**适配要点**：
- **不必照搬**：根据团队规模和成熟度调整
- **从一个服务开始**：选择核心服务先试点
- **高层支持**：没有管理层支持很难推行
- **文化先行**：先培养无责文化和工程文化
