# 事件严重级别定义八股文

---

## 一、严重级别体系

### 1. 事件严重级别如何定义？

**答：**

| 级别 | 名称 | 影响范围 | 响应时间 | 典型场景 |
|------|------|----------|----------|----------|
| **P0** | Critical | 全部或大部分用户受影响 | 5 分钟内 | 核心服务全面宕机、数据泄露 |
| **P1** | Major | 显著部分用户受影响 | 15 分钟内 | 核心功能降级、部分区域不可用 |
| **P2** | Minor | 少量用户受影响 | 30 分钟内 | 非核心功能故障、性能下降 |
| **P3** | Low | 几乎无用户感知 | 4 小时内 | 内部工具故障、监控盲区 |

### 2. 怎么判断一个事件应该定为什么级别？

**答：**

```
判断决策树：

                     是否影响用户？
                    /             \
                  是               否
                /                   \
         影响核心功能？          影响内部系统？
        /           \           /          \
      是             否       是            否
      |              |        |             |
  影响范围？     影响范围？   P3          记录即可
  /      \       /      \
>10%    <10%   >5%    <5%
  |       |      |      |
  P0     P1     P1     P2

关键维度：
┌─────────────────────────────────────────┐
│ 1. 受影响用户比例                        │
│ 2. 受影响功能的业务优先级                │
│ 3. 是否有数据安全风险                    │
│ 4. 是否有营收损失                        │
│ 5. 是否有 SLO 违规风险                   │
└─────────────────────────────────────────┘
```

### 3. 各级别的响应要求分别是什么？

**答：**

```
P0 - Critical：
  ├── 告警方式：电话 + 短信 + App 推送
  ├── 响应时间：5 分钟内 ACK
  ├── 升级时间：15 分钟内未缓解则升级
  ├── 沟通频率：每 15 分钟更新
  ├── 组织形式：立即开启 War Room
  ├── 涉及角色：On-Call SRE + Dev TL + Manager
  └── Postmortem：必须，48 小时内完成

P1 - Major：
  ├── 告警方式：App 推送 + 短信
  ├── 响应时间：15 分钟内 ACK
  ├── 升级时间：30 分钟内未缓解则升级
  ├── 沟通频率：每 30 分钟更新
  ├── 组织形式：创建事件频道
  ├── 涉及角色：On-Call SRE + 相关开发
  └── Postmortem：必须，1 周内完成

P2 - Minor：
  ├── 告警方式：App 推送
  ├── 响应时间：30 分钟内 ACK
  ├── 升级时间：2 小时内未解决则升级
  ├── 沟通频率：每小时更新
  ├── 组织形式：工单跟踪
  ├── 涉及角色：On-Call SRE
  └── Postmortem：可选

P3 - Low：
  ├── 告警方式：Slack/邮件
  ├── 响应时间：4 小时内 ACK
  ├── 升级时间：次日工作时间
  ├── 沟通频率：按需
  ├── 组织形式：工单跟踪
  ├── 涉及角色：On-Call SRE
  └── Postmortem：不需要
```

## 二、分级标准设计

### 4. 如何为业务定制事件分级标准？

**答：**

```
电商平台事件分级示例：

P0:
  - 订单无法创建                   (核心交易链路)
  - 支付服务完全不可用              (直接营收影响)
  - 用户数据大规模泄露              (安全事件)
  - 全站无法访问                    (100%影响)

P1:
  - 搜索服务不可用                  (影响流量转化)
  - 支付成功率下降 >5%              (部分营收影响)
  - 单个可用区故障导致部分用户受影响  (区域影响)
  - 用户登录异常 >10%              (用户体验)

P2:
  - 推荐系统降级                    (非核心功能)
  - 邮件/短信通知延迟               (非实时业务)
  - 管理后台响应慢                  (内部影响)
  - 单个非核心微服务不可用           (有限影响)

P3:
  - 监控缺失或不完整                (工程债务)
  - 开发环境故障                    (不影响生产)
  - 文档系统不可用                  (内部工具)
  - 非生产证书即将过期              (预防性)
```

### 5. 事件级别在什么情况下需要升级或降级？

**答：**

```
升级条件：
  P2 → P1:
    - 影响范围扩大（更多用户受影响）
    - 持续时间超出预期（>2 小时）
    - 出现连锁故障迹象

  P1 → P0:
    - 核心服务完全不可用
    - SLO 错误预算即将耗尽
    - 出现数据安全问题
    - 需要高管决策

降级条件：
  P0 → P1:
    - 已缓解主要影响，但未完全恢复
    - 只有小部分用户仍受影响

  P1 → P2:
    - 有临时 workaround
    - 影响范围缩小到少量用户

原则：
  ┌──────────────────────────────────────┐
  │ " 宁可过度升级，也不要低估严重性 "    │
  │ " 升级不是失败，是负责任的做法 "      │
  └──────────────────────────────────────┘
```

## 三、SLO 与事件分级联动

### 6. 如何将 SLO 与事件级别关联？

**答：**

| 情况 | 错误预算消耗 | 自动触发级别 |
|------|-------------|-------------|
| 30 天预算消耗 > 50% 在 1 小时 | 极速消耗 | P0 |
| 30 天预算消耗 > 20% 在 6 小时 | 快速消耗 | P1 |
| 30 天预算消耗 > 10% 在 1 天 | 加速消耗 | P2 |
| 30 天预算剩余 < 30% | 持续消耗 | P3 |

```yaml
# Prometheus 告警规则示例
groups:
  - name: slo-incident-severity
    rules:
      - alert: SLOBurnRateCritical
        expr: |
          (
            rate(http_requests_total{code=~"5.."}[5m])
            / rate(http_requests_total[5m])
          ) > 14.4 * 0.001  # 14.4x 燃烧率
        for: 2m
        labels:
          severity: P0
        annotations:
          summary: "错误预算极速消耗，预计 1 小时耗尽"
          
      - alert: SLOBurnRateHigh
        expr: |
          (
            rate(http_requests_total{code=~"5.."}[30m])
            / rate(http_requests_total[30m])
          ) > 6 * 0.001  # 6x 燃烧率
        for: 5m
        labels:
          severity: P1
```

## 四、面试高频题

### 7. 面试题：如何保证事件分级的一致性？

**答：**

```
保证分级一致性的方法：

1. 文档化分级标准
   - 每个级别有明确的判断条件
   - 用具体数字而非模糊描述
   - 提供丰富的示例

2. 自动化分级
   - 告警规则中预设 severity 标签
   - SLO 燃烧率自动映射级别
   - 减少人为主观判断

3. 定期 Calibration
   - 月度 Review 事件分级是否合理
   - 有争议的案例集体讨论
   - 更新分级标准文档

4. 培训
   - 新人 On-Call 培训包含分级演练
   - "Wheel of Misfortune" 故障模拟
```

### 8. 面试题：P0 和 P1 的界限如何界定？

**答：**

```
核心区别：

P0 强调 "不可用"：
  - 核心功能完全不可用
  - 大多数用户受影响（>10%）
  - 有重大安全或合规风险
  → 需要即时组织全力响应

P1 强调 "降级"：
  - 核心功能可用但降级
  - 部分用户受影响（<10%）
  - 有 workaround 或替代方案
  → 需要快速响应但不必全员

灰色地带的处理原则：
  "When in doubt, escalate."
  不确定时，先按高级别处理，
  再根据实际情况降级。
```
