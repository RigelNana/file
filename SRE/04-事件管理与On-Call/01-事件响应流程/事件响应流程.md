# 事件响应流程八股文

---

## 一、事件管理基础

### 1. 什么是事件（Incident）？

**答：** 事件是指导致或可能导致服务降级或中断的**非计划事件**。

| 术语 | 英文 | 定义 |
|------|------|------|
| **事件** | Incident | 影响用户的服务异常 |
| **告警** | Alert | 系统发出的异常通知 |
| **问题** | Problem | 一个或多个事件的根因 |
| **变更** | Change | 对服务的有计划修改 |

### 2. 事件响应的标准流程是什么？

**答：**

```
事件生命周期：

检测(Detect) → 响应(Respond) → 缓解(Mitigate) → 修复(Resolve) → 复盘(Review)

详细流程：
┌─ 检测 ─────────────────────────────────────┐
│ 告警触发 / 用户报告 / 合成监控发现           │
└───────────┬───────────────────────────────┘
            ▼
┌─ 分级 ─────────────────────────────────────┐
│ 判断严重程度：P0/P1/P2/P3                   │
│ 分配负责人和沟通渠道                         │
└───────────┬───────────────────────────────┘
            ▼
┌─ 响应 ─────────────────────────────────────┐
│ On-Call 确认告警（ACK）                      │
│ 初步诊断（5-15分钟内）                       │
│ 决定是否升级                                 │
└───────────┬───────────────────────────────┘
            ▼
┌─ 缓解 ─────────────────────────────────────┐
│ 优先恢复服务（回滚/切流/扩容/降级）           │
│ 不要在故障中执着于找根因                      │
└───────────┬───────────────────────────────┘
            ▼
┌─ 修复 ─────────────────────────────────────┐
│ 确认服务完全恢复                             │
│ 关闭事件                                    │
└───────────┬───────────────────────────────┘
            ▼
┌─ 复盘 ─────────────────────────────────────┐
│ 撰写 Postmortem（48小时内）                  │
│ 制定 Action Items                           │
│ 分享和学习                                   │
└─────────────────────────────────────────────┘
```

### 3. "先恢复后查因"原则怎么理解？

**答：**

```
错误做法：
  故障发生 → 花 2 小时排查根因 → 然后修复
  结果：用户影响持续 2 小时

正确做法：
  故障发生 → 5 分钟内执行恢复操作 → 服务恢复
  → 事后再仔细排查根因

恢复手段优先级：
  1. 回滚到上一个已知好的版本     （最快）
  2. 切流到其他可用区/机房         （秒级）
  3. 扩容稀释故障影响              （分钟级）
  4. 降级关闭非核心功能            （分钟级）
  5. 热修复代码                    （最慢，最后选择）
```

## 二、事件沟通

### 4. 事件中的沟通规范是什么？

**答：**

| 阶段 | 沟通内容 | 频率 | 渠道 |
|------|----------|------|------|
| 事件确认 | "已确认 P1 事件，正在排查" | 立即 | Slack/企微频道 |
| 进展更新 | "已定位原因为 X，正在执行 Y" | 每 15-30 分钟 | 事件频道 |
| 恢复确认 | "服务已恢复，监控确认中" | 恢复时 | 事件频道 |
| 事件关闭 | "事件关闭，Postmortem 将于 48h 内完成" | 关闭时 | 事件频道 + 邮件 |

**状态更新模板**：
```
📢 事件更新 #INC-2024-042
时间：2024-01-15 14:30 UTC
状态：🔴 进行中
影响：支付服务 30% 请求超时
当前操作：正在回滚 v3.2 到 v3.1
预计恢复：15 分钟内
负责人：@张三
下次更新：14:45 UTC
```

### 5. P0 事件的升级链是怎样的？

**答：**

```
T+0m    告警触发 → Primary On-Call 收到
T+5m    无 ACK → Secondary On-Call 收到
T+10m   确认 P0 → 通知 SRE Manager + 开发 TL
T+15m   启动 War Room → 通知 VP
T+30m   未缓解 → 通知 CTO/CEO（视公司规模）
T+60m   未缓解 → 全面升级，调动跨团队资源
```

## 四、面试高频题

### 8. 面试题：描述你处理过的一次重大事件

**答：** 使用 STAR 方法：

```
Situation：
  凌晨 3 点收到 P0 告警
  订单服务 50% 请求返回 500

Task：
  快速恢复服务，减少用户影响

Action：
  T+0m: 收到告警，确认事件
  T+3m: 查看仪表盘，发现数据库连接超时
  T+5m: 决定先回滚最近的发布（v2.1→v2.0）
  T+8m: 回滚完成，错误率下降但未恢复
  T+10m: 发现数据库主节点 CPU 100%
  T+12m: 发现一条慢查询（全表扫描）
  T+15m: Kill 慢查询 + 临时扩容只读副本
  T+18m: 服务完全恢复

Result：
  总影响时间 18 分钟
  次日 Postmortem：添加索引 + 慢查询监控
  MTTR 低于团队 30 分钟目标
```

### 9. 面试题：如何衡量事件管理的效果？

**答：**

| 指标 | 定义 | 目标 |
|------|------|------|
| **MTTD** | 平均检测时间 | < 5 分钟 |
| **MTTA** | 平均确认时间 | < 5 分钟 |
| **MTTM** | 平均缓解时间 | < 30 分钟 |
| **MTTR** | 平均恢复时间 | < 1 小时 |
| **事件数** | P0/P1 事件数量 | 趋势下降 |
| **重复事件率** | 同类事件复发 | < 10% |
| **Postmortem 完成率** | 完成 / 总事件 | 100% |
