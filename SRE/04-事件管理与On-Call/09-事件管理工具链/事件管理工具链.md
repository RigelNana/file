### 2. PagerDuty 的核心功能和最佳实践？

**答：**

```
PagerDuty 核心概念：
  ├── Service: 对应一个微服务/系统
  ├── Escalation Policy: 升级策略
  ├── Schedule: On-Call 排班
  ├── Integration: 告警源接入
  └── Incident: 事件记录

Escalation Policy 配置示例：
  Level 1 (immediately): Primary On-Call
  Level 2 (after 5 min): Secondary On-Call
  Level 3 (after 15 min): Engineering Manager
  Level 4 (after 30 min): VP of Engineering

最佳实践：
  ┌────────────────────────────────────────┐
  │ 1. 每个 Service 都有 Escalation Policy │
  │ 2. 设置告警分组减少噪音              │
  │ 3. 启用 Event Intelligence 去重       │
  │ 4. 配置合理的 Urgency（High/Low）     │
  │ 5. 利用 Analytics 追踪 On-Call 负载   │
  └────────────────────────────────────────┘
```

```yaml
# PagerDuty 通过 Prometheus Alertmanager 接入
receivers:
  - name: pagerduty-critical
    pagerduty_configs:
      - routing_key: "<service-integration-key>"
        severity: "critical"
        description: "{{ .CommonAnnotations.summary }}"
        details:
          firing: "{{ .Alerts.Firing | len }}"
          dashboard: "{{ .CommonAnnotations.dashboard }}"
          runbook: "{{ .CommonAnnotations.runbook }}"
```

### 3. Incident.io 和 Rootly 这类新一代工具有什么优势？

**答：**

| 特点 | 传统工具 (PagerDuty) | 新一代 (Incident.io/Rootly) |
|------|----------------------|---------------------------|
| 事件创建 | 手动或告警触发 | Slack 内一键创建 |
| 沟通 | 切换到 PagerDuty 界面 | 完全在 Slack 内 |
| 角色分配 | 手动 | 自动分配 IC/Comms |
| 状态更新 | 手动切换 | Slack 命令或按钮 |
| Postmortem | 手动写 | 自动生成模板+时间线 |
| 数据分析 | 基础报告 | 深度分析（MTTR趋势等） |

```
Incident.io Slack 工作流示例：

/incident create
  → Bot 创建事件频道 #inc-042-payment-issue
  → Bot 自动邀请 On-Call
  → Bot 发布初始摘要

/incident update
  → Bot 提供结构化表单填写
  → Bot 更新所有相关频道

/incident resolve
  → Bot 关闭事件
  → Bot 自动生成 Postmortem 草稿
  → Bot 创建 Jira Follow-up 工单
```

## 二、工具集成

### 4. 如何设计事件管理工具的集成架构？

**答：**

```
典型集成架构：

                    ┌─────────────────┐
                    │   Incident.io   │
                    │  (事件编排中心)  │
                    └────────┬────────┘
           ┌─────────┬──────┴──────┬──────────┐
           ▼         ▼             ▼          ▼
      ┌────────┐ ┌────────┐ ┌──────────┐ ┌────────┐
      │ Slack  │ │PagerDuty│ │Statuspage│ │ Jira   │
      │告警频道│ │ 呼叫   │ │ 状态页   │ │ 工单   │
      └────────┘ └────────┘ └──────────┘ └────────┘
           ▲
           │
    ┌──────┴──────┐
    │ Alertmanager│
    └──────┬──────┘
           │
    ┌──────┴──────┐
    │ Prometheus  │
    └─────────────┘

数据流：
  1. Prometheus 触发告警 → Alertmanager
  2. Alertmanager → PagerDuty（呼叫 On-Call）
  3. Alertmanager → Slack（告警频道通知）
  4. On-Call 创建 Incident → Incident.io
  5. Incident.io → Slack（创建事件频道）
  6. Incident.io → Statuspage（更新状态页）
  7. 事件关闭 → Incident.io → Jira（创建 AI）
```

### 5. 状态页面工具如何选择？

**答：**

| 工具 | 特点 | 适用场景 |
|------|------|----------|
| **Atlassian Statuspage** | 行业标准，功能全面 | 大中型公司 |
| **Cachet** | 开源自托管 | 预算有限 |
| **Better Uptime** | 简洁易用，监控+状态页 | 小团队 |
| **Instatus** | 快速，使用简单 | 创业公司 |
| **自建** | 完全定制化 | 特殊需求 |

```
状态页面最佳实践：
  1. 按服务/产品分组展示
  2. 订阅功能（邮件/Webhook/RSS）
  3. 历史事件可查
  4. 维护窗口提前通知
  5. 自动化集成（告警自动更新状态）
  6. 多语言支持（国际化服务）
```

## 三、工具评估与选型

### 6. 事件管理工具选型应考虑哪些因素？

**答：**

```
选型维度：

1. 集成能力
   ├── 与现有监控工具集成（Prometheus/Datadog）
   ├── 与沟通工具集成（Slack/Teams）
   ├── 与工单系统集成（Jira/Linear）
   └── API 和 Webhook 支持

2. 工作流自动化
   ├── 自动创建事件频道
   ├── 自动分配角色
   ├── 自动生成 Postmortem
   └── 自动创建 Follow-up 工单

3. 分析和报告
   ├── MTTR/MTTD 趋势分析
   ├── On-Call 负载报告
   ├── 事件分类统计
   └── 团队绩效对比

4. 成本
   ├── 按用户/按事件计费
   ├── 免费层功能是否够用
   └── 规模增长后的费用

典型组合推荐：

小团队（<20人）:
  PagerDuty + Slack + Google Docs

中型团队（20-100人）:
  PagerDuty + Incident.io + Statuspage + Jira

大型团队（100+人）:
  PagerDuty + 自研事件管理 + Statuspage + Jira
```

### 7. 面试题：如果没有预算买工具，如何搭建事件管理流程？

**答：**

```
零成本/低成本方案：

告警：
  Prometheus + Alertmanager（开源免费）

On-Call 排班：
  Google Calendar + Alertmanager webhook

事件管理：
  Slack Free Plan + Google Docs 模板

状态页面：
  Cachet（开源）或 GitHub Pages

Postmortem：
  Google Docs + 标准模板

工单跟踪：
  GitHub Issues / GitLab Issues

自动化：
  Slack Bot（自建）+ Alertmanager Webhook

关键不是工具，而是流程：
  ┌─────────────────────────────────────┐
  │ 1. 有明确的事件响应流程              │
  │ 2. 有 On-Call 排班和升级链           │
  │ 3. 有 Postmortem 机制               │
  │ 4. 有 Action Item 跟踪             │
  │ 5. 有持续改进的循环                  │
  └─────────────────────────────────────┘
```

### 8. 面试题：如何评估事件管理工具的 ROI？

**答：**

| 指标 | 计算方式 | 说明 |
|------|----------|------|
| **MTTR 改善** | (旧MTTR - 新MTTR) × 事件数 × 每分钟损失 | 直接收益 |
| **On-Call 效率** | 节省小时数 × 工程师时薪 | 人力节省 |
| **Postmortem 质量** | AI完成率提升 × 预防的事件价值 | 间接收益 |
| **员工满意度** | On-Call 满意度调查提升 | 留人价值 |
