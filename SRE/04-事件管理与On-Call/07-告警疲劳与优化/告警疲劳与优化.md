## 一、告警疲劳问题

### 1. 什么是告警疲劳（Alert Fatigue）？

**答：**

```
告警疲劳定义：
  当 On-Call 收到大量无需操作或低价值告警时，
  会对告警产生麻木心理，导致真正重要的告警被忽略。

告警疲劳的危害：
  ┌─────────────────────────────────────────┐
  │ 1. 真正的事件被淹没在噪音中              │
  │ 2. On-Call 响应时间变长                  │
  │ 3. On-Call 人员 Burnout                  │
  │ 4. 对整个监控系统失去信心                │
  │ 5. 人员流失率增加                         │
  └─────────────────────────────────────────┘

健康的告警指标：
  ┌──────────────────────────────────────┐
  │ 可操作告警率 > 80%                    │
  │ 每 12h shift 告警 ≤ 2 个             │
  │ 每月 On-Call 被叫不超过 20 次         │
  │ 夜间（23:00-07:00）告警趋于零         │
  └──────────────────────────────────────┘
```

### 2. 告警疲劳的常见原因有哪些？

**答：**

| 原因 | 说明 | 例子 |
|------|------|------|
| **阈值不合理** | 太灵敏 | CPU > 50% 就告警 |
| **缺少去重** | 同一问题重复告警 | Pod 重启每次都告 |
| **信息过载** | 一个故障触发几十条告警 | 级联告警风暴 |
| **非可操作告警** | 收到但啥也做不了 | "磁盘 70%"（还早呢） |
| **环境噪音** | 非生产环境告警 | Dev 环境 Pod CrashLoop |
| **过时告警** | 已废弃的服务仍在告警 | 下线服务的监控未清理 |

## 二、告警优化策略

### 3. 告警优化的系统性方法是什么？

**答：**

```
优化四步法：

Step 1: 盘点（Inventory）
  ├── 收集过去 30 天所有告警
  ├── 分类统计
  └── 生成告警 Top 10 报告

Step 2: 分类（Classify）
  每条告警分为：
  ├── Actionable（需要人介入处理）→ 保留
  ├── Auto-fixable（可以自动修复）→ 自动化
  ├── Informational（仅信息通知）→ 降级为仪表盘
  └── Noise（无任何价值）→ 删除

Step 3: 优化（Optimize）
  ├── 调整阈值和持续时间
  ├── 添加告警抑制和去重
  ├── 合并相关告警
  └── 自动化修复

Step 4: 持续治理（Govern）
  ├── 新增告警需要 Review
  ├── 每月告警健康度报告
  └── 每季度告警大扫除
```

### 4. 告警抑制（Inhibit）和分组（Group）怎么配置？

**答：**

```yaml
# Alertmanager 配置示例

# 告警抑制：高级别告警自动抑制低级别
inhibit_rules:
  # 如果已有 P0 告警，抑制同服务的 P1/P2
  - source_matchers:
      - severity = "P0"
    target_matchers:
      - severity =~ "P1|P2"
    equal: ['service']
  
  # 如果节点 Down，抑制该节点上所有 Pod 告警
  - source_matchers:
      - alertname = "NodeDown"
    target_matchers:
      - alertname =~ "PodCrash|PodRestart"
    equal: ['node']

# 告警分组：避免告警轰炸
route:
  group_by: ['service', 'alertname']
  group_wait: 30s       # 等 30 秒收集同组告警
  group_interval: 5m    # 同组告警最少间隔 5 分钟
  repeat_interval: 4h   # 同一告警 4 小时才重复
  
  routes:
    - matchers:
        - severity = "P0"
      receiver: critical-pager
      group_wait: 10s   # P0 快速通知
      
    - matchers:
        - severity = "P1"
      receiver: oncall-pager
      
    - matchers:
        - severity = "P2"
      receiver: slack-alerts
```

### 5. 如何设计合理的告警阈值？

**答：**

```
阈值设计原则：

1. 基于 SLO 而非固定数字
   ✗ error_rate > 1%（固定阈值）
   ✓ 错误预算燃烧率 > 6x（基于 SLO）

2. 基于异常而非绝对值
   ✗ latency > 500ms（固定阈值）
   ✓ latency > avg_7d * 3（动态基线）

3. 加 for 持续时间避免毛刺
   ✗ cpu > 80%               → 一个毛刺就触发
   ✓ cpu > 80% for 5m        → 持续 5 分钟才触发

4. 分级触发
   80% → Warning（Slack 通知）
   90% → Critical（PagerDuty 呼叫）
   95% → Emergency（电话升级）

常见服务告警阈值参考：
┌───────────────────────────────────────┐
│ 错误率: > 0.1% for 5m → P1          │
│         > 1%   for 2m → P0          │
│ 延迟 P99: > 2x baseline for 5m → P1 │
│ 可用性: < 99.9% 窗口内 → P1         │
│ 磁盘: > 85% for 30m → P2           │
│        > 95% for 5m → P1            │
└───────────────────────────────────────┘
```

## 三、告警即代码

### 6. 什么是"告警即代码"（Alert as Code）？

**答：**

```
告警即代码核心思想：
  告警规则和路由配置和应用代码一样，
  纳入版本控制、Code Review、CI/CD 流程。

好处：
  ├── 变更有记录（Git 历史）
  ├── 变更有审核（PR Review）
  ├── 可以自动化测试（Unit Test for alerts）
  └── 容易回滚（Git revert）

目录结构示例：
  alerts/
  ├── payment-service/
  │   ├── alerts.yaml        # 告警规则
  │   ├── alerts_test.yaml   # 单元测试
  │   └── runbook.md         # 关联的 Runbook
  ├── user-service/
  │   ├── alerts.yaml
  │   └── ...
  └── infrastructure/
      ├── node-alerts.yaml
      └── ...

CI Pipeline:
  1. promtool check rules alerts/*.yaml
  2. promtool test rules alerts/*_test.yaml
  3. alertmanager check-config alertmanager.yaml
  4. Deploy to Prometheus/Alertmanager
```

```yaml
# 告警测试示例 (promtool test)
rule_files:
  - alerts.yaml

evaluation_interval: 1m

tests:
  - interval: 1m
    input_series:
      - series: 'http_requests_total{code="500"}'
        values: '0 0 10 20 30 40 50'
      - series: 'http_requests_total{code="200"}'
        values: '1000 1000 1000 1000 1000 1000 1000'
    alert_rule_test:
      - eval_time: 5m
        alertname: HighErrorRate
        exp_alerts:
          - exp_labels:
              severity: P1
```

### 7. 新增告警的审核标准是什么？

**答：**

```
告警 PR Review Checklist：

□ 告警是否可操作？（收到后知道做什么）
□ 有对应的 Runbook 吗？
□ 阈值是否合理？（不会产生过多误报）
□ 有 for 持续时间吗？（避免毛刺触发）
□ severity 标签是否正确？
□ 描述信息是否清晰？
□ 有单元测试吗？
□ 会影响其他告警的抑制/分组吗？

好的告警 annotation 示例：
  annotations:
    summary: "支付服务错误率过高"
    description: |
      当前错误率: {{ $value | humanizePercentage }}
      影响: 用户支付可能失败
      Runbook: https://runbook.internal/payment-errors
      仪表盘: https://grafana.internal/d/payment
```

## 四、面试高频题

### 8. 面试题：On-Call 每天收到 50+ 告警你怎么办？

**答：**

```
系统性优化计划：

Week 1: 量化问题
  - 导出 30 天告警数据
  - 分析告警分布（Top 10 占比通过多少）
  - 计算可操作率

Week 2-3: 快速止血
  - 静默已知噪音告警
  - 合并重复告警
  - 调整明显不合理的阈值
  目标：减少 50% 噪音

Week 4-6: 工程改进
  - Top 5 告警的根因修复
  - 自动修复可自动化的问题
  - 更新 Runbook
  目标：可操作率 > 80%

Week 7+: 持续治理
  - 告警 Review 周会
  - 新增告警审批流程
  - 月度告警健康度报告
  目标：每 12h shift ≤ 2 告警
```

### 9. 面试题：告警风暴（Alert Storm）如何处理？

**答：**

```
告警风暴 = 短时间内大量告警同时触发

应急处理：
  1. 不要被大量告警淹没
  2. 找到"根"告警（Root Alert）
  3. 其他都是衍生告警

预防措施：
  ┌─────────────────────────────────────────┐
  │ 1. 告警分组：按服务/集群聚合             │
  │ 2. 告警抑制：根因告警抑制衍生告警        │
  │ 3. 级联延迟：依赖服务告警延迟 5 分钟     │
  │ 4. 告警限流：同组告警最多 N 条/时间窗口  │
  │ 5. 拓扑感知：基于依赖关系做关联分析      │
  └─────────────────────────────────────────┘
```
