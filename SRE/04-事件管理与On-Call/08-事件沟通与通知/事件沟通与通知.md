## 一、事件沟通原则

### 1. 事件中的沟通为什么重要？

**答：**

```
沟通不好的后果：
  ├── 关联团队不知道发生了什么 → 重复排查
  ├── 用户/客户没有信息 → 密集咨询客服
  ├── 管理层不了解进展 → 过度介入干扰
  └── 事后没有记录 → Postmortem 无法准确重建

沟通原则：
  ┌─────────────────────────────────────────┐
  │ 1. 主动沟通 > 被催着沟通                 │
  │ 2. 定时更新 > 有变化才更新               │
  │ 3. 事实 > 猜测                           │
  │ 4. 简洁明了 > 技术细节                   │
  │ 5. 统一出口 > 多口径信息                 │
  └─────────────────────────────────────────┘
```

### 2. 不同受群的沟通策略有什么不同？

**答：**

| 受众 | 关注点 | 渠道 | 语言风格 |
|------|--------|------|----------|
| **内部技术团队** | 根因、排查进展 | Slack 事件频道 | 技术术语 |
| **管理层** | 影响范围、恢复时间 | 邮件/IM 私信 | 业务语言 |
| **客户** | 是否影响我、何时恢复 | 状态页面/邮件 | 简单通俗 |
| **客服团队** | 如何回复用户 | 内部知识库 | 话术模板 |
| **合规/法务** | 是否有数据泄露 | 专线通知 | 合规术语 |

### 4. 对管理层的沟通要注意什么？

**答：**

```
管理层关心的 5 个问题：
  1. 多少用户受影响？
  2. 有没有营收损失？
  3. 什么时候能恢复？
  4. 根因是什么？
  5. 需要我做什么？

管理层通知模板：
─────────────────────────────────
  主题：[P1] 支付服务异常 - 第 2 次更新

  影响：约 15% 用户支付受影响
  持续时间：已持续 15 分钟
  原因：新版本引入的数据库问题（正在回滚）
  预计恢复：10 分钟内
  是否需要决策：否
─────────────────────────────────

注意事项：
  ✓ 用业务语言（"15% 用户"不是"500 QPS"）
  ✓ 给出预估恢复时间（即使不确定）
  ✓ 明确是否需要他们做什么
  ✗ 不要过多技术细节
  ✗ 不要频繁更新小进展
```

## 三、对外沟通

### 5. 状态页面（Status Page）怎么管理？

**答：**

```
状态页面级别定义：

🟢 Operational      所有系统正常
🟡 Degraded         性能下降但服务可用
🟠 Partial Outage   部分功能不可用
🔴 Major Outage     核心功能不可用
⚫ Maintenance      计划维护中

状态页面更新模板：

事件标题: 支付服务性能下降
状态: 🟠 Partial Outage

更新 1 (14:30 UTC):
  我们检测到支付服务出现性能下降，
  部分用户可能遇到支付超时。
  我们的团队正在积极排查。

更新 2 (14:45 UTC):
  已识别原因并正在实施修复。
  大部分用户应该能正常使用。

更新 3 (15:00 UTC):
  修复已完成，服务已恢复正常。
  我们将继续监控以确保稳定。

事后说明 (15:30 UTC):
  此次事件由系统升级引起的数据库连接问题导致。
  影响持续约 30 分钟。
  我们已采取措施防止类似问题再次发生。
  感谢您的耐心和理解。
```

### 6. 对外通知的常见渠道和触发条件？

**答：**

| 渠道 | 触发条件 | 受众 | 时效 |
|------|----------|------|------|
| **状态页面** | 所有用户可感知事件 | 所有用户 | 实时 |
| **应用内横幅** | 核心功能受影响 | 活跃用户 | 实时 |
| **邮件** | P0 事件恢复后 | 受影响客户 | 事后 |
| **Twitter/社交媒体** | P0 广泛关注时 | 公众 | 按需 |
| **客户经理** | 大客户受影响 | VIP 客户 | 立即 |

```
对外沟通黄金原则：
  ┌──────────────────────────────────────┐
  │ 1. 快速：认识到问题后立即通知         │
  │ 2. 诚实：不隐瞒，不过度乐观           │
  │ 3. 人性化：态度诚恳，共情用户         │
  │ 4. 具体：说影响范围，不说空话         │
  │ 5. 持续：定期更新直到解决             │
  └──────────────────────────────────────┘

  好的措辞：
    "我们检测到部分用户在使用支付功能时
     可能遇到延迟，我们的团队正在紧急处理。"

  差的措辞：
    "由于后端数据库连接池耗尽导致 payment-api 
     返回 503，正在回滚 Deployment。"
```

## 四、通知自动化

### 7. 如何实现事件通知自动化？

**答：**

```yaml
# 事件通知自动化工作流（Incident.io / Rootly 风格）

workflow:
  trigger: incident_created
  
  steps:
    # Step 1: 创建 Slack 频道
    - action: create_slack_channel
      name: "inc-{{ incident.id }}-{{ incident.slug }}"
    
    # Step 2: 邀请相关人员
    - action: invite_to_channel
      people:
        - "{{ incident.assignee }}"
        - "{{ incident.service.oncall }}"
      
    # Step 3: 发送初始通知
    - action: post_message
      channel: "#incidents"
      message: |
        🚨 新事件: {{ incident.title }}
        严重程度: {{ incident.severity }}
        负责人: {{ incident.assignee }}
        频道: #{{ incident.channel }}
    
    # Step 4: 根据严重程度升级
    - action: page_oncall
      condition: "incident.severity in ['P0', 'P1']"
      service: pagerduty
      
    # Step 5: 更新状态页面
    - action: update_status_page
      condition: "incident.severity == 'P0'"
      status: "investigating"
    
    # Step 6: 通知管理层
    - action: send_notification
      condition: "incident.severity == 'P0'"
      to: "engineering-leadership"
```

### 8. 面试题：如何设计一个事件通知系统？

**答：**

```
事件通知系统架构：

告警源 ──→ 事件管理平台 ──→ 通知引擎 ──→ 多渠道分发

                                     ┌── PagerDuty (电话/短信)
                                     ├── Slack (即时消息)
  Prometheus ──→ Alertmanager ──→    ├── Email (邮件)
  Custom API ──→ Incident.io  ──→    ├── Statuspage (状态页)
  Manual     ──→              ──→    ├── Webhook (自定义)
                                     └── SMS (短信网关)

关键设计要素：
  1. 多通道冗余
     Primary: PagerDuty → 如果 5 分钟无 ACK
     Fallback: 直接短信 → 如果 5 分钟无 ACK
     Fallback2: 电话呼叫

  2. 升级策略
     T+0:  Primary On-Call
     T+5:  Secondary On-Call
     T+15: Team Lead
     T+30: Engineering Manager

  3. 可靠性要求
     通知系统自身的 SLO: 99.99%
     端到端延迟: < 30 秒
     ACK 确认机制
```
