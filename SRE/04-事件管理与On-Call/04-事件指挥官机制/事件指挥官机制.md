## 一、事件指挥体系

### 1. 什么是事件指挥官（Incident Commander, IC）？

**答：** IC 是重大事件中的**总协调者**，负责统筹各方资源、协调沟通、推动事件到解决。

```
IC 的核心职责：
┌─────────────────────────────────────────────┐
│ 1. 协调：组织响应团队，分配任务               │
│ 2. 沟通：对内外统一信息口径                   │
│ 3. 决策：在信息不完整时做出快速判断            │
│ 4. 升级：判断何时需要更多资源或高管介入         │
│ 5. 记录：确保事件时间线被记录                  │
└─────────────────────────────────────────────┘

IC 不做的事：
  ✗ 亲自排查技术问题
  ✗ 直接在生产环境执行操作
  ✗ 独自决定所有事情
```

### 3. IC 如何开启 War Room？

**答：**

```
War Room 启动流程（P0 事件）：

T+0m: 告警触发
T+2m: On-Call 确认，判定 P0
T+3m: On-Call 触发 IC 流程
T+5m: IC 就位，创建沟通渠道

IC 开场白模板：
"大家好，我是本次事件的 IC [姓名]。
 当前情况：[概述影响]
 Ops Lead：[姓名]，请汇报排查进展
 Comms Lead：[姓名]，请准备第一次外部通知
 所有操作和发现请在频道中同步
 我们每 15 分钟做一次状态同步"

War Room 规则：
  ┌────────────────────────────────────┐
  │ 1. IC 是唯一的协调者              │
  │ 2. 所有操作前先沟通               │
  │ 3. 使用结构化更新（Who/What/When） │
  │ 4. 非授权人员静默（Listen Only）    │
  │ 5. IC 说暂停就必须暂停            │
  └────────────────────────────────────┘
```

## 二、IC 实践技巧

### 4. IC 有哪些关键决策点？

**答：**

```
决策点一：缓解策略选择
  问题：回滚 vs 前向修复 vs 降级？
  IC 做法：
  - 询问 Ops Lead 每种方案的预计时间和风险
  - 选择恢复最快、风险最低的方案
  - 犹豫时选回滚（最安全）

决策点二：升级判断
  问题：当前团队是否能处理？
  IC 做法：
  - 设定时间盒（15 分钟无进展就升级）
  - 不等到无路可走才升级
  - 升级不是失败

决策点三：外部通知时机
  问题：何时通知用户/客户？
  IC 做法：
  - 确认影响后立即通知
  - 不等到完全了解后才通知
  - "我们已检测到问题并正在处理" > 沉默

决策点四：恢复确认
  问题：何时宣布事件结束？
  IC 做法：
  - 指标回到正常范围
  - 等待至少 15 分钟观察期
  - 与 Ops Lead 确认
```

### 5. IC 如何管理混乱的事件现场？

**答：**

```
场景：多人同时在频道中讨论不同问题

IC 策略：

1. 结构化沟通
   IC: "@所有人 暂停自由讨论，我来做一次同步"
   IC: "Ops Lead，当前排查到哪里了？"
   IC: "数据库团队，你们那边什么状况？"
   IC: "好的，下一步我们分两条线：
        线索1: @Alice 查数据库
        线索2: @Bob 查网络"

2. 隔离干扰
   "请不在事件处理中的同事转为 Listen Only"
   "非紧急讨论请移到 #inc-xxx-parking-lot 频道"

3. 决策记录
   IC: "决策记录：我们选择回滚 v3.2 到 v3.1"
   IC: "原因：这是最近的变更，回滚风险最低"
   IC: "@Alice 请执行回滚，完成后在频道确认"

4. 时间盒管理
   IC: "我设定 15 分钟 Timer"
   IC: "如果回滚无效，我们切换到方案 B"
```

### 6. 如何培养 IC 能力？

**答：**

| 方法 | 说明 | 频率 |
|------|------|------|
| **Wheel of Misfortune** | 模拟故障场景演练 | 月度 |
| **IC Shadow** | 跟随有经验的 IC 学习 | 每次事件 |
| **桌面推演** | 纸上演练事件决策 | 季度 |
| **事后复盘旁听** | 学习其他 IC 的决策过程 | 每次 |
| **IC 培训课** | 系统化培训（ICS 框架） | 每年 |

```
IC 能力矩阵：

初级 IC（Junior IC）:
  ✓ 能处理 P2/P3 事件
  ✓ 熟悉流程和工具
  ✓ 需要 Senior IC 支持

中级 IC（IC）:
  ✓ 能独立处理 P1 事件
  ✓ 有效协调多团队
  ✓ 能做出快速决策

高级 IC（Senior IC）:
  ✓ 能处理 P0 / 复杂多故障事件
  ✓ 在混乱中保持冷静
  ✓ 能指导和培训新 IC
```

## 三、面试高频题

### 7. 面试题：IC 和 On-Call 有什么区别？

**答：**

| 维度 | On-Call | IC |
|------|--------|----|
| **触发** | 任何告警 | 重大事件（P0/P1） |
| **角色** | 一线响应者 | 总协调者 |
| **工作** | 排查和修复 | 协调和决策 |
| **技术** | 深入技术细节 | 全局视角 |
| **沟通** | 团队内 | 全组织 |
| **流转** | On-Call 发现 P0 → 呼叫 IC |

### 8. 面试题：描述你作为 IC 的一次经历

**答：**

```
使用 STAR 方法：

Situation:
  线上支付服务 P0 事件
  多个微服务同时告警，现场混乱

Task:
  作为 IC 协调 3 个团队 10+ 人

Action:
  1. 开场宣布角色分工
  2. 让所有人 30 秒汇报各自状态
  3. 判断根因可能在数据库层
  4. 分配：DB 团队查数据库，后端查连接池
  5. 设定 15 分钟 Timer
  6. 10 分钟后 DB 团队定位到磁盘满
  7. 决策：扩容磁盘 + 临时清理日志
  8. 通知 Comms Lead 更新状态页

Result:
  20 分钟恢复服务
  无数据丢失
  Postmortem 产出 3 个 Action Items
  被评价为"沟通清晰，决策迅速"
```
