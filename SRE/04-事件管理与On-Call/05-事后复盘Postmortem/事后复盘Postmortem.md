# 事后复盘 Postmortem 八股文

---

## 一、Postmortem 基础

### 1. 什么是 Postmortem？为什么重要？

**答：** Postmortem（事后复盘/事后分析）是事件结束后的**结构化回顾**，目的是从事件中学习并防止类似事件再次发生。

```
Postmortem 的核心价值：
┌─────────────────────────────────────────┐
│ 1. 学习：理解系统实际失败模式            │
│ 2. 改进：产出具体可执行的 Action Items   │
│ 3. 预防：减少同类事件再次发生            │
│ 4. 共享：组织级的知识沉淀               │
│ 5. 文化：建立无责、开放的工程文化        │
└─────────────────────────────────────────┘

何时需要写 Postmortem：
  ✓ 所有 P0、P1 事件（必须）
  ✓ 有用户可感知影响的事件
  ✓ 差点酿成事故的近失事件（Near Miss）
  ✓ 消耗大量错误预算的事件
  ✗ P3 及以下通常不需要
```

### 2. Postmortem 的标准模板是什么？

**答：**

```markdown
# Postmortem: [事件标题]

## 基本信息
- 事件编号: INC-2024-042
- 日期: 2024-01-15
- 作者: [On-Call]
- 审核人: [Manager/Peer]
- 严重程度: P1
- 影响时长: 35 分钟

## 概述
一句话描述：什么坏了，影响了什么，持续了多久。

## 影响
- 受影响用户: 约 15% 的支付用户
- 错误预算消耗: 本月 30 天预算消耗 8%
- 营收影响: 估计损失约 ¥XX 万

## 根因
支付服务 v3.2 引入了一个 N+1 查询，在高并发下
导致数据库连接池耗尽。

## 时间线
- 14:00 UTC - v3.2 灰度发布至 10% 流量
- 14:15 UTC - 灰度扩大到 50%
- 14:22 UTC - 告警触发：支付错误率 > 1%
- 14:24 UTC - On-Call ACK，开始排查
- 14:30 UTC - 判断与新版本相关，决定回滚
- 14:35 UTC - 回滚至 v3.1
- 14:40 UTC - 数据库连接恢复
- 14:45 UTC - 错误率恢复正常
- 14:57 UTC - 事件关闭

## 做得好的
- 告警迅速触发（< 5 分钟）
- 快速决策回滚
- 沟通及时

## 可以改进的
- 灰度从 10% 直接跳到 50%，步进过大
- 缺少数据库连接池的监控告警
- 代码 Review 未发现 N+1 查询

## 行动项 (Action Items)
| # | 项目 | 负责人 | 优先级 | 截止日期 | 状态 |
|---|------|--------|--------|----------|------|
| 1 | 添加数据库连接池监控 | Alice | P1 | 01/22 | TODO |
| 2 | 灰度策略改为 5/10/25/50/100 | Bob | P1 | 01/22 | TODO |
| 3 | CI 添加 N+1 查询检测 | Carol | P2 | 02/01 | TODO |
```

## 二、高质量 Postmortem

### 3. 如何写出高质量的根因分析？

**答：**

```
坏的根因分析（停留在表面）：
  "服务崩溃是因为数据库连接池满了"
  → 为什么满了？没有回答

好的根因分析（层层深入）：
  使用 5-Why 方法：
  
  Why1: 支付服务返回 500
    → 因为数据库连接池耗尽

  Why2: 为什么连接池耗尽？
    → 因为查询执行时间从 5ms 增加到 500ms

  Why3: 为什么查询变慢？
    → v3.2 的新代码引入了 N+1 查询

  Why4: 为什么代码 Review 没有发现？
    → Review checklist 中没有性能相关检查

  Why5: 为什么灰度没有拦截？
    → 灰度从 10% 直接跳到 50%，未设置渐进步骤

根因 = Why4 + Why5（流程和防护层的缺失）
```

### 4. Postmortem 的时间线该怎么写？

**答：**

```
好的时间线要素：
  1. 使用 UTC 时间
  2. 包含检测、响应、缓解、恢复的关键节点
  3. 标注决策和原因
  4. 包含失败的尝试（重要学习点）

示例：
  14:22 [DETECT] 告警触发：payment_error_rate > 1%
  14:24 [RESPOND] On-Call(Alice) ACK
  14:25 [DIAGNOSE] 查看仪表盘，5xx 集中在 payment-api
  14:27 [DIAGNOSE] 检查最近变更，发现 v3.2 刚部署
  14:28 [ATTEMPT] 尝试重启 Pod（无效）
  14:30 [DECISION] 决定回滚 v3.2 → v3.1
  14:30 [ESCALATE] 通知 Secondary On-Call(Bob) 协助
  14:35 [MITIGATE] 回滚完成
  14:40 [VERIFY] 数据库连接数恢复正常
  14:45 [VERIFY] 错误率恢复到 < 0.1%
  14:57 [RESOLVE] 事件关闭

关键指标：
  MTTD (检测): 7 min  (14:15 → 14:22)
  MTTA (确认): 2 min  (14:22 → 14:24)
  MTTM (缓解): 13 min (14:22 → 14:35)
  MTTR (恢复): 23 min (14:22 → 14:45)
```

### 5. Action Items 怎么才能被有效执行？

**答：**

```
常见问题：Action Items 写完就忘了

解决方案：

1. SMART 原则
   ✗ "改善监控"
   ✓ "为 payment-api 添加 DB 连接池使用率告警，
      阈值 80%，@Alice，P1，1/22 前完成"

2. 分级和排期
   ├── P0: 立即修复（24 小时内）
   ├── P1: 本周完成
   ├── P2: 本 Sprint 完成
   └── P3: 下 Sprint 排进

3. 跟踪机制
   ├── 每个 AI 创建 Jira 工单
   ├── 每周 Postmortem Review 会检查进度
   ├── 未完成的 AI 在 Review 会上讨论阻塞
   └── AI 完成率作为 SRE 团队 KPI

4. 分类
   ├── 检测改进（更好的监控/告警）
   ├── 缓解改进（更快的恢复手段）
   ├── 预防改进（根因修复）
   └── 流程改进（变更管理、测试等）
```

## 三、Postmortem 文化

### 6. 如何推行无责 Postmortem 文化？

**答：**

```
有责 vs 无责（Blameless）：

有责文化：
  "这个 Bug 是小王写的，应该处分"
  → 结果：大家隐瞒问题，不敢上报

无责文化：
  "系统允许了这个 Bug 上线，我们如何改进系统？"
  → 结果：开放讨论，持续改进

无责 Postmortem 准则：
  ┌────────────────────────────────────────────┐
  │ 1. 关注系统和流程，而非个人                  │
  │ 2. 假设每个人都基于当时信息做了最好的决策     │
  │ 3. 用 "我们" 而非 "他/她"                   │
  │ 4. 批评行为/决策可以，但不标签化个人          │
  │ 5. IC 和 Manager 带头示范                   │
  └────────────────────────────────────────────┘

语言对比：
  ✗ "小王部署了有 Bug 的代码"
  ✓ "CI 流程未能检测到这个回归"

  ✗ "DBA 没有设连接池告警"
  ✓ "监控覆盖存在盲区，需要补充连接池告警"
```

### 7. Postmortem Review 会议如何组织？

**答：**

```
Postmortem Review 会议流程：

参会人：
  - 事件相关人员（IC / On-Call / SME）
  - SRE 和开发 Manager
  - 感兴趣的工程师（开放旁听）

会议流程（45-60 分钟）：
  1. 作者 5 分钟概述事件        (5 min)
  2. Walk-through 时间线        (10 min)
  3. 讨论根因分析是否充分       (10 min)
  4. Review Action Items        (10 min)
  5. 讨论"做得好"和"可改进" (10 min)
  6. 确认最终版本               (5 min)

主持人原则：
  - 如果讨论偏向指责 → 引导回系统改进
  - 确保所有 Action Items 有 Owner
  - 鼓励提问："如果 X 也失败了会怎样？"
```

## 四、面试高频题

### 8. 面试题：你写过的最好的 Postmortem 是什么？

**答：**

```
一个好的 Postmortem 包含：

1. 有深度的根因分析
   - 不停留在直接原因
   - 揭示了系统性问题

2. 有改变的 Action Items
   - 不是写完就完了
   - 实际推动了系统改进

3. 组织级学习
   - 发现了跨团队的共性问题
   - 推动了流程/工具改进

示例回答：
  "我写过一个关于级联故障的 Postmortem，
   根因分析发现了 3 层深的问题：
   直接原因是超时配置，
   根本原因是缺少断路器，
   系统原因是依赖管理不清晰。
   Action Items 推动团队引入了 Service Mesh，
   后续同类事件减少了 80%。"
```

### 9. 面试题：Postmortem 中的 Near Miss 是什么？

**答：**

```
Near Miss（近失事件）：
  差一点就导致严重事故的事件

例子：
  - 磁盘使用率到了 95%，刚好被发现清理了
  - 生产环境的危险操作被审批流程拦截了
  - 证书差 1 天过期，被巡检发现了

为什么要写 Near Miss Postmortem：
  ├── 比等事故发生再学习更好
  ├── 暴露系统中的隐患
  ├── 成本低（没有实际影响）
  └── 建立预防文化

Near Miss 审查标准：
  "如果这个问题没有被及时发现，
   会造成 P0/P1 事件吗？"
  如果答案是 Yes → 写 Postmortem
```
