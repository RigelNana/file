## 一、On-Call 最佳实践总结

### 1. On-Call 体系建设的完整框架是什么？

**答：**

```
On-Call 体系五大支柱：

     ┌─────── On-Call 体系 ───────┐
     │                             │
  ┌──┴──┐ ┌──┴──┐ ┌──┴──┐ ┌──┴──┐ ┌──┴──┐
  │排班  │ │工具 │ │流程 │ │培训 │ │治理 │
  │制度  │ │支撑 │ │规范 │ │体系 │ │改进 │
  └──────┘ └─────┘ └─────┘ └─────┘ └─────┘

排班制度：
  - 公平的轮值分配
  - Primary + Secondary 双层
  - Follow-the-Sun 跨时区
  - 合理的补偿机制

工具支撑：
  - 告警管理（PagerDuty/OpsGenie）
  - 事件管理（Incident.io/Rootly）
  - 运行手册（Runbook）
  - 仪表盘（Grafana）

流程规范：
  - 事件响应 SOP
  - 升级策略
  - 沟通模板
  - Postmortem 流程

培训体系：
  - Shadow 计划
  - Wheel of Misfortune
  - 持续学习机制

治理改进：
  - 告警优化
  - On-Call Review
  - 负载追踪
  - 持续工程改进
```

### 2. 成熟的 On-Call 体系有哪些衡量维度？

**答：**

| 维度 | 初级 | 成熟 | 卓越 |
|------|------|------|------|
| **告警质量** | 大量无效告警 | >80% 可操作 | >95% 可操作 |
| **MTTA** | >30 分钟 | <10 分钟 | <5 分钟 |
| **MTTR** | >2 小时 | <1 小时 | <30 分钟 |
| **Runbook** | 很少或没有 | 核心服务有 | 全覆盖+定期更新 |
| **Postmortem** | 偶尔做 | P0/P1 必做 | 包含 Near Miss |
| **AI 完成率** | <50% | >80% | >95% |
| **On-Call 负载** | 无追踪 | 有报告 | 主动优化 |
| **培训** | 无体系 | 有 Shadow | 完整培训路径 |
| **自动化** | 几乎没有 | 部分自动修复 | 大部分自动化 |

## 二、面试高频题

### 3. 面试题：你在 On-Call 中遇到过最难的事件是什么？如何解决的？

**答：**

```
回答框架（STAR）：

Situation（情境）:
  - 具体说明时间、服务、影响范围
  - 说明为什么"难"（多系统、复杂根因等）

Task（任务）:
  - 你的角色（On-Call/IC/配合）
  - 需要达成的目标

Action（行动）:
  - 按时间线描述你做了什么
  - 重点讲：
    ├── 诊断思路
    ├── 关键决策点
    ├── 与团队协作
    └── 恢复手段

Result（结果）:
  - 恢复时间
  - 后续改进措施
  - 量化效果

示例：
  "S: 核心 API 服务间歇性 502，无明显模式
   T: 作为 On-Call，需要快速定位并恢复
   A: 排查发现是上游 LB 健康检查配置不当，
      部分 Pod 被误判为不健康。
      短期：调整健康检查超时
      长期：优化健康检查端点
   R: 15 分钟恢复，后续同类事件为零"
```

### 4. 面试题：如何处理跨团队的事件？

**答：**

```
跨团队事件挑战：
  - 责任不清（"不是我们的问题"）
  - 沟通低效（各说各话）
  - 工具和权限不互通

解决方案：

1. IC 机制
   - 指定 IC 负责跨团队协调
   - IC 不属于任何技术团队
   - IC 有权调动任何团队资源

2. 联合 On-Call
   - 建立跨团队的 Bridge Call
   - 每个相关团队派一个代表
   - 统一的事件频道

3. 服务依赖图
   - 维护服务拓扑（谁依赖谁）
   - 故障时快速定位影响链
   - 自动通知相关团队

4. 共享工具和权限
   - 统一的日志/监控平台
   - 跨团队只读权限
   - 标准化的 Runbook 格式
```

### 5. 面试题：如何避免 On-Call Burnout？

**答：**

```
On-Call Burnout 的信号：
  ├── On-Call 同学经常抱怨
  ├── 夜间频繁被打扰
  ├── On-Call 期间工程工作为零
  ├── 人员开始拒绝 On-Call 排班
  └── 离职谈话提及 On-Call 压力

预防和应对：

组织层面：
  ├── 保证轮值池 ≥ 8 人
  ├── 每人每月 On-Call ≤ 1 次
  ├── 合理补偿（经济 + 调休）
  └── 管理层重视 On-Call 质量

工程层面：
  ├── 投资告警治理（减少噪音）
  ├── 投资自动化修复
  ├── 持续修复 Top-N 告警根因
  └── 改善 Runbook 降低排查难度

文化层面：
  ├── On-Call 困难可以求助（不丢人）
  ├── 夜间被叫后可以晚到
  ├── On-Call 反馈被认真对待
  └── 定期调查 On-Call 满意度
```

### 6. 面试题：如何从零搭建 On-Call 体系？

**答：**

```
Phase 1: 基础搭建（1-2 周）
  ├── 选择告警工具（PagerDuty 免费层）
  ├── 建立基本排班（2-3 人轮值）
  ├── 定义 P0-P3 严重级别
  ├── 写 5 个核心服务的 Runbook
  └── 确定升级链

Phase 2: 流程规范（2-4 周）
  ├── 制定事件响应 SOP
  ├── 建立 Postmortem 模板和流程
  ├── 配置告警路由和升级策略
  ├── 建立 Slack 事件频道规范
  └── 第一次 Wheel of Misfortune

Phase 3: 优化提升（1-3 月）
  ├── 告警治理（减少 50% 噪音）
  ├── 扩大 On-Call 轮值池
  ├── 建立 On-Call Review 周会
  ├── 自动化常见修复操作
  └── 建立 On-Call 补偿制度

Phase 4: 持续成熟（3-6 月）
  ├── 告警即代码
  ├── 事件管理自动化
  ├── 跨团队 On-Call 协作
  ├── On-Call 负载分析和优化
  └── 完善培训体系
```

### 8. 面试题：Google SRE 的 On-Call 有什么特点？

**答：**

```
Google SRE On-Call 关键特点：

1. 50% 法则
   On-Call 及相关运维工作 ≤ 50% 时间
   如果超过 → 团队需要扩充人手

2. 双层 On-Call
   Primary: 直接处理
   Secondary: 升级备份

3. 每 12 小时轮值
   不是一周 7×24 小时一个人

4. 事件上限
   每 12h shift 最多 2 个需处理的事件
   超过 → 说明系统需要工程改进

5. Postmortem 必须
   每个事件都有后续行动
   强制无责文化

6. 可以 Hand Back
   如果 On-Call 负载过重
   SRE 可以将服务交还给开发团队

关键启示：
  "On-Call 不是无限制的值班，
   而是有明确边界和保障的制度。
   如果 On-Call 太痛苦，
   问题在系统而不是人。"
```
