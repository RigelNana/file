# 容量规划与性能工程八股文

---

## 一、容量规划

### 1. 什么是容量规划？为什么 SRE 需要做容量规划？

**答：** 容量规划是**预测未来资源需求**并确保系统有足够容量满足预期负载的过程。

**重要性：**
- 容量不足 → 服务降级/宕机 → SLO 违约
- 容量过剩 → 资源浪费 → 成本上升
- SRE 需要在**可靠性和成本**之间找到平衡

**容量规划的核心步骤：**
1. **需求预测**：基于历史数据和业务增长预测未来流量
2. **资源建模**：建立流量到资源消耗的映射关系
3. **瓶颈分析**：识别系统中最先耗尽的资源
4. **规划执行**：采购/扩容/优化
5. **持续验证**：通过负载测试验证容量是否充足

### 2. 负载测试、压力测试和混沌测试有什么区别？

**答：**

| 测试类型 | 目的 | 方法 | 关注点 |
|----------|------|------|--------|
| **负载测试** | 验证预期负载下的表现 | 模拟正常/峰值流量 | 是否满足 SLO |
| **压力测试** | 找到系统极限 | 持续增加负载直到崩溃 | 系统的上限在哪 |
| **浸泡测试** | 检测长时间运行的问题 | 持续运行数小时/天 | 内存泄漏、资源耗尽 |
| **峰值测试** | 验证突发流量应对能力 | 瞬间注入大量流量 | 突增流量的处理 |
| **混沌测试** | 验证故障恢复能力 | 注入故障（停机、延迟） | 韧性和容错 |

### 3. 什么是自动扩缩容（Auto Scaling）？

**答：**

| 扩缩容类型 | 说明 | 适用场景 |
|-----------|------|----------|
| **水平扩展（Scale Out）** | 增加实例数量 | 无状态服务 |
| **垂直扩展（Scale Up）** | 增加单实例资源 | 数据库等有状态服务 |
| **预测性扩展** | 基于历史模式提前扩容 | 有规律的流量波动 |
| **反应性扩展** | 基于实时指标触发扩容 | 不可预测的流量变化 |

## 二、FinOps

### 4. 什么是 FinOps？SRE 如何参与成本优化？

**答：** FinOps（Cloud Financial Operations）是云财务管理实践，SRE 在其中的角色：

1. **可见性**：建立资源使用和成本的可观测性
2. **优化**：识别过度配置的资源、推荐合适的实例类型
3. **效率指标**：定义每请求成本、每用户成本等效率指标
4. **容量与成本权衡**：在冗余（可靠性）和成本之间决策
