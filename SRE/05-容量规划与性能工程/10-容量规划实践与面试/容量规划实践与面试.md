## 一、容量规划实践案例

### 1. 完整的容量规划流程案例？

**答：**

```
案例：电商平台 618 大促容量规划

背景：
  日常峰值: 20000 QPS
  去年 618 峰值: 80000 QPS（4x）
  今年业务增长: 40%

Step 1: 需求评估
  预测峰值 = 20000 × 4 × 1.4 = 112000 QPS
  安全系数 1.5 → 目标容量 168000 QPS

Step 2: 全链路梳理
  ┌────────────┬──────────┬──────────┬──────────┐
  │ 服务       │ 当前容量 │ 目标容量 │ 差距     │
  ├────────────┼──────────┼──────────┼──────────┤
  │ API 网关   │ 50K QPS  │ 168K     │ +118K    │
  │ 商品服务   │ 30K QPS  │ 168K     │ +138K    │
  │ 订单服务   │ 5K TPS   │ 20K TPS  │ +15K     │
  │ MySQL 主库 │ 10K QPS  │ 30K QPS  │ +20K     │
  │ Redis 集群 │ 100K QPS │ 300K QPS │ +200K    │
  └────────────┴──────────┴──────────┴──────────┘

Step 3: 扩容方案
  T-30 天: 采购云资源
  T-14 天: 完成扩容部署
  T-7 天: 全链路压测验证
  T-3 天: 压测报告确认
  T-1 天: 最终检查
  T 日: 值班 + 弹性扩缩就位

Step 4: 压测验证
  阶梯式压测: 50K → 100K → 150K → 168K
  验证 SLO: P99 < 200ms, 错误率 < 0.1%
  
Step 5: 大促后复盘
  实际峰值: 130K QPS
  预测偏差: +29%（偏保守）
  → 下次调整安全系数为 1.3
```

### 2. 如何建立容量规划的周期性机制？

**答：**

| 频率 | 活动 | 参与人 | 输出 |
|------|------|--------|------|
| **周** | 容量水位检查 | SRE | 水位报告 |
| **月** | 容量评审会 | SRE + Dev | 扩缩容计划 |
| **季** | 业务 + 容量对齐 | SRE + PM + Dev | 季度资源预算 |
| **年** | 年度容量规划 | SRE + Infra + Finance | 年度预算 |

```
月度容量评审议程（1 小时）：

1. 上月总结（10 min）
   ├── 资源使用率趋势
   ├── 成本变化
   └── 上月 Action Items 进度

2. 水位预警（15 min）
   ├── 接近阈值的服务
   ├── 增长最快的服务
   └── 即将到来的业务变化

3. 扩缩决策（20 min）
   ├── 需要扩容的服务和方案
   ├── 可以缩容的资源
   └── 技术优化替代扩容

4. Action Items（15 min）
   ├── 分配负责人和截止日期
   └── 下次评审确认
```

## 二、面试高频题

### 3. 面试题：一个新服务即将上线，如何做容量规划？

**答：**

```
新服务容量规划方案：

Phase 1: 估算（无生产数据）
  输入：
  ├── 产品预期用户量和增长
  ├── 参考类似服务的资源模型
  └── API 设计中的预期请求量

  估算公式：
  预期 QPS = DAU × 次/用户/天 / 86400 × 峰值系数
  
  例：10万 DAU × 20 次/天 / 86400 × 5 = 115 QPS 峰值
  
  资源估算（参考同类服务）：
  每 100 QPS → 2C4G × 1 Pod
  → 初始: 2C4G × 3 Pod（保证高可用）

Phase 2: 灰度验证
  灰度 5% 流量 → 收集真实数据
  → 校准资源模型
  → 推算全量资源需求

Phase 3: 全量上线
  按校准后的模型 + 30% Headroom 部署
  开启 HPA 自动扩缩
  设置容量告警

Phase 4: 持续调优
  运行 2 周后做 Right-sizing
  建立资源仪表盘
  纳入月度容量评审
```

### 4. 面试题：如何在面试中展示容量规划经验？

**答：**

```
STAR 方法展示容量规划：

场景类型一：大促容量规划
  S: 面对 xx 大促，预计流量 N 倍增长
  T: 需要确保系统承受峰值流量
  A: 做了什么（预测/扩容/压测/预案）
  R: 大促期间零故障，成本控制在预算内

场景类型二：成本优化
  S: 云成本月度 XX 万，CEO 要求优化
  T: 需要降低 30% 成本而不影响可靠性
  A: 做了什么（Right-sizing/RI/Spot/清理）
  R: 3 个月内成本下降 35%，SLO 无违规

场景类型三：紧急扩容
  S: 流量突增 5 倍（病毒式传播）
  T: 需要快速扩容保证服务可用
  A: 做了什么（限流/弹性扩容/紧急采购）
  R: 30 分钟内扩容完成，用户无感知

通用展示要点：
  ├── 用数据说话（QPS、成本、延迟）
  ├── 展示系统性思维（全链路而非单点）
  ├── 展示与业务的协作
  └── 展示持续改进
```

### 5. 面试题：容量规划有哪些自动化的可能？

**答：**

```
容量规划自动化层次：

Level 1: 数据自动化
  ├── 资源使用率自动采集（Prometheus）
  ├── 成本数据自动聚合（Cloud Cost API）
  └── 自动生成容量报告

Level 2: 预测自动化
  ├── 基于 Prophet/ARIMA 自动预测
  ├── 自动检测异常增长
  └── 自动容量告警

Level 3: 决策辅助
  ├── 自动推荐 Right-sizing
  ├── 自动推荐 RI 购买
  └── 自动标记闲置资源

Level 4: 执行自动化
  ├── HPA/VPA 自动扩缩
  ├── Cluster Autoscaler 自动加减节点
  ├── 自动存储分层
  └── 自动环境清理

工具推荐：
  ├── Kubecost: K8s 成本分析
  ├── CAST AI: 自动优化 K8s 成本
  ├── Spot.io: 智能 Spot 管理
  └── Goldilocks: VPA 推荐
```

### 6. 面试题：容量规划中如何处理不确定性？

**答：**

```
应对不确定性的策略：

1. 范围估算代替点估算
   不是"需要 100 台服务器"
   而是"90%概率在 80-120 台之间"

2. 分场景规划
   乐观场景: 增长 20% → 需要 X 资源
   基准场景: 增长 50% → 需要 Y 资源
   悲观场景: 增长 100% → 需要 Z 资源
   → 按基准场景规划，为悲观场景准备预案

3. 弹性设计
   基础容量用 RI 锁定（确定性强的部分）
   波动部分用 On-Demand + HPA（应对不确定）
   极端情况用 Spot + 限流（最后防线）

4. 快速反馈回路
   每月对比预测 vs 实际
   及时调整模型和安全系数
   积累经验提升准确率

核心心态：
  "容量规划永远不可能 100% 准确，
   重要的是有应对偏差的能力。"
```
