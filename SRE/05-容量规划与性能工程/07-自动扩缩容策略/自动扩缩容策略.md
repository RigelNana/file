# 自动扩缩容策略八股文

---

## 一、自动扩缩容基础

### 1. 自动扩缩容有哪些类型？

**答：**

| 类型 | 维度 | 工具 | 说明 |
|------|------|------|------|
| **HPA** | 水平 Pod | K8s HPA | 增减 Pod 副本数 |
| **VPA** | 垂直 Pod | K8s VPA | 调整 Pod 的 Request/Limit |
| **Cluster Autoscaler** | 节点 | CA | 增减集群节点数 |
| **KEDA** | 事件驱动 | KEDA | 基于外部指标自动扩缩 |

```
自动扩缩容全景：

请求增加
  │
  ▼
HPA 扩 Pod ──→ 节点资源不够 ──→ CA 加节点
  │                                   │
  ▼                                   ▼
Pod 运行        新节点加入集群
  │
  ▼
请求减少 → HPA 缩 Pod → 节点空闲 → CA 移除节点
```

### 2. HPA 的工作原理和配置？

**答：**

```yaml
# HPA 基于 CPU 的自动扩缩
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: payment-api
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: payment-api
  minReplicas: 3        # 最小副本数
  maxReplicas: 20       # 最大副本数
  metrics:
    # CPU 利用率
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70    # 目标 CPU 70%
    # 自定义指标（QPS）
    - type: Pods
      pods:
        metric:
          name: http_requests_per_second
        target:
          type: AverageValue
          averageValue: 1000        # 目标每 Pod 1000 QPS
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60   # 60 秒稳定窗口
      policies:
        - type: Percent
          value: 50                     # 每次最多扩 50%
          periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300  # 缩容等待 5 分钟
      policies:
        - type: Percent
          value: 10                     # 每次最多缩 10%
          periodSeconds: 60
```

```
HPA 扩缩算法：

期望副本数 = ceil(当前副本数 × (当前指标值 / 目标指标值))

例：
  当前 3 Pod，CPU 利用率 90%，目标 70%
  期望 = ceil(3 × 90/70) = ceil(3.86) = 4

扩缩行为关键参数：
  ┌──────────────────────────────────────┐
  │ scaleUp 要快: 60s 稳定窗口          │
  │ scaleDown 要慢: 300s 稳定窗口       │
  │ 原因: 缩容太快会导致抖动            │
  └──────────────────────────────────────┘
```

### 3. KEDA 事件驱动扩缩有什么优势？

**答：**

```yaml
# KEDA: 基于 Kafka 消息积压自动扩缩
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: order-consumer
spec:
  scaleTargetRef:
    name: order-consumer
  minReplicaCount: 1
  maxReplicaCount: 30
  triggers:
    - type: kafka
      metadata:
        bootstrapServers: kafka:9092
        consumerGroup: order-group
        topic: orders
        lagThreshold: "100"    # 积压 100 条就扩容
```

```
KEDA vs HPA 对比：

HPA: 
  基于 CPU/内存 → 响应慢（需要 CPU 先升高）
  
KEDA:
  基于业务指标 → 响应快（直接根据业务量）
  
KEDA 支持的触发源：
  ├── Kafka（消息积压）
  ├── RabbitMQ（队列长度）
  ├── Redis（List 长度）
  ├── Prometheus（自定义指标）
  ├── Cron（定时扩缩）
  ├── AWS SQS/CloudWatch
  └── 30+ 触发器
```

## 二、扩缩策略设计

### 4. 如何设计合理的扩缩策略？

**答：**

```
扩缩策略设计原则：

1. 扩容要快，缩容要慢
   扩容：检测到压力 → 1 分钟内开始扩
   缩容：压力下降 → 等 5-10 分钟再缩
   
2. 设置合理的 min/max
   min: 能处理最低流量 + 保证高可用（≥3）
   max: 预算上限或基础设施极限

3. 多指标组合
   CPU + 自定义业务指标同时判断
   任一指标触发就扩容

4. 预判式扩容（Predictive）
   已知高峰时段提前扩容
   例：每天 9:00 之前扩到 10 Pod
```

```yaml
# 定时预扩容（CronHPA / KEDA Cron）
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: api-service
spec:
  scaleTargetRef:
    name: api-service
  minReplicaCount: 3
  maxReplicaCount: 50
  triggers:
    # 工作时间预扩容
    - type: cron
      metadata:
        timezone: Asia/Shanghai
        start: "0 8 * * 1-5"      # 周一到周五 8:00
        end: "0 20 * * 1-5"       # 周一到周五 20:00
        desiredReplicas: "15"      # 工作时间保持 15 Pod
    # 非工作时间
    - type: cron
      metadata:
        timezone: Asia/Shanghai
        start: "0 20 * * *"
        end: "0 8 * * *"
        desiredReplicas: "5"
```

### 5. 缩容时如何避免服务中断？

**答：**

```
安全缩容三要素：

1. 优雅终止
   spec:
     terminationGracePeriodSeconds: 60
   
   Pod 收到 SIGTERM → 完成当前请求 → 退出
   不是直接 Kill

2. Pod Disruption Budget (PDB)
   apiVersion: policy/v1
   kind: PodDisruptionBudget
   metadata:
     name: payment-api-pdb
   spec:
     minAvailable: 2    # 至少保持 2 个 Pod 可用
     selector:
       matchLabels:
         app: payment-api

3. 缓慢缩容
   scaleDown:
     stabilizationWindowSeconds: 300
     policies:
       - type: Pods
         value: 1           # 每次只减 1 个
         periodSeconds: 120  # 每 2 分钟减 1 个
```

## 三、面试高频题

### 6. 面试题：HPA 扩容不及时怎么办？

**答：**

```
HPA 扩容慢的原因及解决：

原因1：Pod 启动慢（镜像大、初始化久）
  → 优化镜像（多阶段构建、预拉取）
  → 使用 readinessProbe 缩短就绪检测

原因2：节点资源不足（需要等 CA 加节点）
  → 预留 Buffer 节点
  → 使用 OverProvisioning（空 Pod 占位）

原因3：HPA 检测周期慢（默认 15 秒）
  → 使用 KEDA + Prometheus 更快反应
  
原因4：突发流量（来不及扩）
  → 预判式扩容（Cron 提前扩）
  → 限流保护（扩容完成前限流兜底）

组合策略：
  ┌─────────────────────────────────────┐
  │ Cron 预扩容（已知高峰）             │
  │ + HPA 自适应（未知波动）            │
  │ + 限流兜底（极端情况）              │
  └─────────────────────────────────────┘
```

### 7. 面试题：如何选择 HPA vs VPA？

**答：**

| 维度 | HPA | VPA |
|------|-----|-----|
| **方式** | 增减 Pod 数量 | 调整 Pod 资源 |
| **适用** | 无状态服务 | 有状态服务/单例 |
| **速度** | 快（秒级创建 Pod） | 慢（需要重启 Pod） |
| **风险** | 低 | Pod 重启有风险 |
| **推荐** | 首选 | 辅助优化 Request |

```
最佳实践组合：
  1. VPA 用于推荐合理的 Request/Limit（UpdateMode: Off）
  2. HPA 用于实际的扩缩容
  3. 不要同时让 VPA 和 HPA 管理同一资源
```
