# 成本优化与 FinOps 八股文

---

## 一、FinOps 基础

### 1. 什么是 FinOps？SRE 为什么要关注成本？

**答：**

```
FinOps 定义：
  Financial Operations（云财务运营）
  通过跨团队协作实现云成本的可见、可控、可优化。

SRE 与 FinOps：
  ┌──────────────────────────────────────┐
  │ SRE 关注可靠性 + FinOps 关注成本     │
  │ 两者统一目标：用最合理的成本          │
  │ 提供最可靠的服务                      │
  └──────────────────────────────────────┘

FinOps 三阶段：
  Inform（知晓）→ Optimize（优化）→ Operate（运营）
  
  Inform:
    - 成本可视化（谁花了多少？）
    - 成本归属（哪个团队/服务？）
    - 预算和告警

  Optimize:
    - 资源 Right-sizing
    - 预留实例/Spot 实例
    - 清理浪费

  Operate:
    - 成本做为指标持续追踪
    - 工程团队参与成本优化
    - 自动化成本管理
```

### 2. 云成本的主要构成是什么？

**答：**

| 类别 | 占比 | 优化空间 |
|------|------|----------|
| **计算 (EC2/VM)** | 40-60% | Right-sizing, Spot, RI |
| **存储 (S3/EBS)** | 15-25% | 冷热分层, 清理过期数据 |
| **数据库 (RDS)** | 10-20% | 实例规格优化, 缓存分流 |
| **网络 (流量)** | 5-15% | CDN, 减少跨 AZ 流量 |
| **其他** | 5-10% | SaaS 工具, 日志存储等 |

```
常见浪费来源：

  ┌── 35% 闲置资源 ────── 购买了但没用
  ├── 25% 过度配置 ────── 规格太大
  ├── 20% 定价不优 ────── 没用 RI/Spot/SP
  ├── 10% 无主资源 ────── 测试遗留未清理
  └── 10% 架构问题 ────── 数据传输费过高
```

## 二、成本优化策略

### 3. 计算资源的成本优化有哪些方法？

**答：**

```
计算成本优化汇总：

1. Right-sizing（调整规格）
   实际 CPU 使用 20% → 降级 50%
   节省: 20-40%

2. Spot/Preemptible 实例
   无状态工作负载使用 Spot
   节省: 60-90%
   风险: 可能被回收（需要容错设计）

3. Reserved Instance / Savings Plan
   稳定负载购买 1-3 年预留
   节省: 30-60%

4. 自动扩缩容
   低谷期自动缩容（夜间/周末）
   节省: 20-40%

5. ARM 架构
   Graviton/ARM 实例更便宜
   节省: 20-30%

混合策略示例：
  ┌──────────────────────────────────┐
  │ 基础负载: RI/SP (60% 资源)      │
  │ 弹性负载: On-Demand (20%)       │
  │ 无状态批处理: Spot (20%)        │
  │ 综合节省: 40-50%                │
  └──────────────────────────────────┘
```

### 4. 数据存储的成本优化？

**答：**

```
存储分层策略：

  热数据（频繁访问）
  ├── S3 Standard / EBS gp3
  ├── 最近 30 天数据
  └── 成本: $$$$

  温数据（偶尔访问）
  ├── S3 IA / Glacier Instant
  ├── 30 天 - 6 个月
  └── 成本: $$

  冷数据（几乎不访问）
  ├── S3 Glacier / Deep Archive
  ├── 6 个月以上
  └── 成本: $

日志成本优化：
  ├── 采样: 只存 10% 的 Debug 日志
  ├── 压缩: gzip 压缩存储
  ├── 保留策略: 30 天热 → 90 天冷 → 删除
  ├── 日志级别: 生产环境 Info 及以上
  └── 预期节省: 50-80%
```

### 5. 如何建立成本归属机制？

**答：**

```
成本归属层级：

  公司总成本
  ├── 事业部 A: 40%
  │   ├── 团队 A1: 60%
  │   │   ├── 服务 payment-api: 30%
  │   │   ├── 服务 order-api: 25%
  │   │   └── ...
  │   └── 团队 A2: 40%
  └── 事业部 B: 60%

实现方式：
  1. 标签体系（Tag）
     每个资源必须有标签：
     - team: payment
     - service: payment-api
     - env: production
     - cost-center: CC-001

  2. 命名空间（K8s Namespace）
     每个团队/服务独立 Namespace
     通过 Namespace 统计资源消耗

  3. 成本分摊
     共享资源按使用比例分摊
     例：Kafka 集群按各团队消息量分摊

成本仪表盘指标：
  - 每个团队月度成本及趋势
  - 每个服务的单请求成本（Cost per Request）
  - 资源利用率 vs 成本效率
```

## 三、面试高频题

### 6. 面试题：如何降低 30% 的云成本？

**答：**

```
30% 成本优化计划：

Phase 1: 快速收益（1-2 周, 预计节省 10-15%）
  ├── 识别和释放闲置资源
  ├── 删除未挂载的磁盘/快照
  ├── 停止非工作时间的测试环境
  └── 调整明显过大的实例

Phase 2: 中期优化（1-2 月, 预计节省 10-15%）
  ├── Right-sizing（全面调整规格）
  ├── 购买 Reserved Instance
  ├── 无状态服务使用 Spot 实例
  ├── 存储分层（冷数据降级）
  └── 日志保留策略优化

Phase 3: 持续优化（3-6 月, 预计节省 5-10%）
  ├── 架构优化减少跨 AZ 流量
  ├── 代码优化提升资源效率
  ├── 缓存优化减少数据库负载
  └── 建立 FinOps 持续运营机制
```

### 7. 面试题：成本优化和可靠性如何平衡？

**答：**

```
平衡原则：

不能为了省钱牺牲的：
  ✗ 去掉冗余（单点就是定时炸弹）
  ✗ 去掉监控（失去可见性）
  ✗ 降低 Headroom 到 <10%

可以优化而不影响可靠性的：
  ✓ Right-sizing（降规格不降数量）
  ✓ Spot + 容错设计
  ✓ 日志采样（不影响排查能力）
  ✓ 存储分层（冷数据降级）
  ✓ 自动扩缩（低谷期缩容）

评判标准：
  "每一块钱的成本节省都不应该
   增加任何新的 SLO 违规风险。"

决策矩阵：
  省钱方案 → 影响 SLO？→ 是 → 不做
  省钱方案 → 影响 SLO？→ 否 → 做
  省钱方案 → 可能影响？→ 先小范围试 → 观察
```
