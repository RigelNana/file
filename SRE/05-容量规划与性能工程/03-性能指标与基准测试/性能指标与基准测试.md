## 一、性能指标体系

### 1. 衡量系统性能的核心指标有哪些？

**答：**

| 指标 | 英文 | 定义 | 典型值 |
|------|------|------|--------|
| **吞吐量** | Throughput | 单位时间处理请求数 | QPS/RPS/TPS |
| **延迟** | Latency | 请求到响应的时间 | P50/P95/P99 |
| **错误率** | Error Rate | 失败请求占比 | <0.1% |
| **并发数** | Concurrency | 同时处理的请求数 | 依业务 |
| **饱和度** | Saturation | 资源使用程度 | <70% |

```
性能指标金字塔：

        用户感知层
       ┌─────────┐
       │ Apdex   │  用户满意度综合得分
       │ 首字节  │  Time to First Byte
       └────┬────┘
       服务指标层
    ┌───────┴────────┐
    │ 延迟 P50/P99   │
    │ 吞吐 QPS       │
    │ 错误率         │
    │ 可用性         │
    └───────┬────────┘
       资源指标层
  ┌─────────┴──────────┐
  │ CPU / Memory / IO  │
  │ Network / 连接数    │
  │ GC / Thread Count  │
  └────────────────────┘
```

### 2. 为什么要关注 P99 而不仅仅是平均值？

**答：**

```
示例数据（100 个请求的延迟）：
  98 个请求: 10ms
  2 个请求: 5000ms

  平均值: (98×10 + 2×5000) / 100 = 109.8ms
  P50: 10ms
  P99: 5000ms

平均值说"一切正常（110ms）"
P99 说"1% 用户体验极差（5 秒）"

百分位数的业务意义：
  ┌──────────────────────────────────────┐
  │ 如果有 100 万日活用户               │
  │ P99 = 5s 意味着                     │
  │ 每天有 1 万用户等待 5 秒以上        │
  │ 这些可能是你最重要的用户（高频用户） │
  └──────────────────────────────────────┘

关注原则：
  P50  → 大多数用户的体验
  P95  → 长尾用户的体验
  P99  → SLO 目标值
  P999 → 极端情况
```

### 3. Apdex 评分是什么？怎么计算？

**答：**

```
Apdex (Application Performance Index)

公式：
  Apdex = (Satisfied + Tolerated/2) / Total

定义（假设 T = 200ms）：
  Satisfied:  响应时间 ≤ T (≤200ms)
  Tolerated:  T < 响应时间 ≤ 4T (200ms-800ms)
  Frustrated: 响应时间 > 4T (>800ms)

示例计算：
  1000 个请求：
  - 700 个 ≤ 200ms (Satisfied)
  - 200 个 200-800ms (Tolerated)
  - 100 个 > 800ms (Frustrated)

  Apdex = (700 + 200/2) / 1000 = 0.80

评分对照：
  ┌──────────────────────┐
  │ 0.94-1.00  优秀      │
  │ 0.85-0.93  良好      │
  │ 0.70-0.84  一般      │
  │ 0.50-0.69  较差      │
  │ <0.50      不可接受   │
  └──────────────────────┘
```

## 二、基准测试

### 4. 基准测试（Benchmark）的最佳实践是什么？

**答：**

```
基准测试原则：

1. 隔离环境
   ├── 独立的测试环境，无其他负载
   ├── 与生产环境配置一致
   └── 固定硬件规格

2. 预热阶段
   ├── JVM/Runtime 需要 JIT 预热
   ├── 缓存需要预热
   └── 连接池需要建立
   → 预热 2-5 分钟后再记录数据

3. 多次运行
   ├── 至少运行 3 次取中位数
   ├── 排除异常值
   └── 计算标准差确保稳定

4. 控制变量
   ├── 每次只改变一个变量
   ├── 固定数据集和请求模式
   └── 记录完整的环境信息

5. 报告格式
   ├── 包含环境信息（OS、CPU、内存、版本）
   ├── 包含完整的百分位数据
   ├── 包含资源使用数据
   └── 可复现的测试步骤
```

### 6. Go 语言的 Benchmark 怎么写？

**答：**

```go
// handler_test.go
package handler

import (
    "net/http"
    "net/http/httptest"
    "testing"
)

func BenchmarkGetProducts(b *testing.B) {
    handler := NewProductHandler()
    req := httptest.NewRequest(http.MethodGet, "/products", nil)

    b.ResetTimer()
    b.RunParallel(func(pb *testing.PB) {
        for pb.Next() {
            w := httptest.NewRecorder()
            handler.ServeHTTP(w, req)
        }
    })
}

// 运行: go test -bench=. -benchmem -count=3
// 输出:
// BenchmarkGetProducts-8  125000  9520 ns/op  4096 B/op  12 allocs/op
// 含义: 8核, 125000次, 每次9.5μs, 每次分配4KB, 12次分配
```

## 三、面试高频题

### 7. 面试题：如何建立性能基线？

**答：**

```
性能基线建设步骤：

1. 选择关键 API（Top 10 流量 API）
2. 在稳定环境做 Benchmark
3. 记录各指标的基线值

基线文档示例：
  API: GET /api/products
  日期: 2024-01-15
  环境: 4C8G Pod × 3
  
  指标基线:
  ├── P50 延迟: 8ms
  ├── P99 延迟: 45ms
  ├── 最大 QPS: 5000
  ├── CPU@1000QPS: 25%
  └── Memory@1000QPS: 512MB

持续跟踪：
  - CI 中自动运行 Benchmark
  - 与基线对比，性能下降 >10% 则报警
  - 每季度更新基线
```

### 8. 面试题：延迟和吞吐量是什么关系？

**答：**

```
延迟与吞吐量的关系（USL 模型）：

低负载时：
  吞吐量 ↑  延迟 ≈ 恒定
  （线性扩展区）

中负载时：
  吞吐量 ↑↑  延迟 ↑
  （排队效应开始显现）

高负载时：
  吞吐量 → 平坦  延迟 ↑↑↑
  （接近系统极限）

超载时：
  吞吐量 ↓↓  延迟 → ∞
  （系统崩溃/级联故障）

图示：
  吞吐 │         ┌──── 拐点
       │      ╱──╲
       │    ╱     ╲ 
       │  ╱        ╲
       │╱           ╲
       └─────────────── 并发数

核心原理：Little's Law
  L = λ × W
  并发 = 到达率 × 等待时间
```
