## 一、瓶颈分析方法论

### 1. 如何系统性地定位性能瓶颈？

**答：**

```
性能分析方法论：

USE 方法（Brendan Gregg）:
  对每个资源检查：
  U - Utilization（利用率）: 资源忙碌时间比例
  S - Saturation（饱和度）: 额外排队的工作量
  E - Errors（错误）: 错误事件数

  资源列表：
  ┌────────┬──────────┬──────────┬──────────┐
  │ 资源   │ 利用率   │ 饱和度   │ 错误     │
  ├────────┼──────────┼──────────┼──────────┤
  │ CPU    │ %busy    │ run queue│ 硬件错误 │
  │ Memory │ %used    │ swap/OOM │ 分配失败 │
  │ Disk   │ %busy    │ IO queue │ IO 错误  │
  │ Network│ %bw      │ TCP backlog│ 重传   │
  └────────┴──────────┴──────────┴──────────┘

RED 方法（Tom Wilkie）:
  对每个服务检查：
  R - Rate（请求率）
  E - Errors（错误率）
  D - Duration（延迟分布）
```

### 2. 性能瓶颈排查的标准路径是什么？

**答：**

```
自顶向下排查路径：

Layer 1: 用户层
  └── 用户感知慢 → 测量端到端延迟
      └── CDN 慢？DNS 慢？服务端慢？

Layer 2: 网络层
  └── 网络延迟高 → traceroute/ping/mtr
      └── 丢包？高延迟？带宽满？

Layer 3: 应用层
  └── 服务慢 → 分布式追踪（Jaeger/Zipkin）
      └── 哪个 Span 慢？哪个调用链慢？

Layer 4: 代码层
  └── 具体函数慢 → Profiling（pprof/async-profiler）
      └── CPU 热点？内存分配？锁竞争？

Layer 5: 依赖层
  └── 外部依赖慢 → 数据库？缓存？第三方 API？
      └── 慢查询？连接池耗尽？超时？

Layer 6: 基础设施层
  └── 资源不足 → USE 方法检查
      └── CPU 满？内存不足？IO 瓶颈？
```

### 3. 常见的性能瓶颈有哪些？

**答：**

| 瓶颈类型 | 症状 | 定位方法 | 解决方案 |
|----------|------|----------|----------|
| **CPU 瓶颈** | CPU 利用率高 | top/perf/pprof | 算法优化/扩容 |
| **内存瓶颈** | OOM/频繁 GC | heapprof/GC日志 | 减少分配/增加内存 |
| **IO 瓶颈** | IO Wait 高 | iostat/iotop | SSD/缓存/批量IO |
| **网络瓶颈** | 延迟高/丢包 | ss/tcpdump | 带宽扩容/资源就近 |
| **锁竞争** | CPU 利用率低但慢 | goroutine dump | 减少锁/无锁设计 |
| **数据库** | 慢查询/连接满 | slow query log | 索引/读写分离/缓存 |
| **GC 停顿** | 间歇性延迟高峰 | GC 日志 | 减少堆对象/调 GC |

## 二、Profiling 工具

### 4. Go 服务如何做性能分析？

**答：**

```go
// 在 main.go 中注册 pprof
import _ "net/http/pprof"

func main() {
    // 启动 pprof 服务
    go func() {
        http.ListenAndServe(":6060", nil)
    }()
    // ...启动业务服务
}
```

```bash
# CPU Profiling
go tool pprof http://localhost:6060/debug/pprof/profile?seconds=30

# 内存 Profiling
go tool pprof http://localhost:6060/debug/pprof/heap

# Goroutine 分析
go tool pprof http://localhost:6060/debug/pprof/goroutine

# 火焰图
go tool pprof -http=:8080 profile.pb.gz

# 常用 pprof 命令
(pprof) top 10        # 前 10 个热点函数
(pprof) list funcName # 查看函数逐行开销
(pprof) web           # 生成调用图
```

### 5. 如何分析火焰图（Flame Graph）？

**答：**

```
火焰图解读：

┌─────────────────────────────┐
│         main.handleRequest  │ ← 入口函数
├─────────┬───────────────────┤
│ db.Query│  processData      │ ← 两个子调用
│ (60%)   │  (40%)            │
├────┬────┼──────┬────────────┤
│slow│idx │json  │ validate   │
│qry │seek│Marshal│           │
│40% │20% │25%   │ 15%       │
└────┴────┴──────┴────────────┘

解读规则：
  1. 宽度 = 占比时间（越宽 = 越热）
  2. Y 轴 = 调用深度
  3. 颜色 = 区分不同类型（无特殊含义）

分析步骤：
  1. 找到最宽的"平台"（热点函数）
  2. 向下看是哪些子调用占比大
  3. 找可优化的部分
  
  上图发现：db.slowQuery 占 40%
  → 优化数据库查询是最有效的改进

生成火焰图：
  # Linux perf
  perf record -g -p <pid> -- sleep 30
  perf script | stackcollapse-perf.pl | flamegraph.pl > flame.svg
  
  # Go pprof 内置火焰图
  go tool pprof -http=:8080 cpu.prof
```

### 6. 数据库性能瓶颈如何定位？

**答：**

```
数据库瓶颈排查路径：

1. 确认是数据库瓶颈
   应用延迟高 → 分布式追踪显示 DB Span 慢

2. 慢查询分析
   MySQL:
     SHOW PROCESSLIST;
     SELECT * FROM mysql.slow_log ORDER BY query_time DESC LIMIT 10;
   
   PostgreSQL:
     SELECT * FROM pg_stat_activity WHERE state = 'active';
     SELECT * FROM pg_stat_statements ORDER BY total_time DESC LIMIT 10;

3. 连接池分析
   # 当前连接数 / 最大连接数
   SHOW STATUS LIKE 'Threads_connected';
   SHOW VARIABLES LIKE 'max_connections';

4. 锁分析
   SHOW ENGINE INNODB STATUS;
   SELECT * FROM information_schema.INNODB_LOCK_WAITS;

5. 索引分析
   EXPLAIN ANALYZE SELECT ...;
   → 关注：type=ALL（全表扫描）、rows 过大

常见优化：
  ┌──────────────────────────────────┐
  │ 加索引 → 最常见也最有效          │
  │ 优化 SQL → 减少不必要的 Join    │
  │ 读写分离 → 读多写少场景          │
  │ 连接池优化 → 减少连接建立开销    │
  │ 缓存 → 减少数据库查询            │
  └──────────────────────────────────┘
```

## 三、面试高频题

### 7. 面试题：线上服务延迟突然升高，如何排查？

**答：**

```
系统排查流程（5 分钟内）：

T+0: 确认范围
  - 全部 API 还是特定 API？
  - 全量用户还是部分？
  - 最近有没有变更？

T+1: 查看仪表盘
  - 服务指标：QPS/Error/Latency
  - 资源指标：CPU/Memory/IO/Network
  - 依赖指标：DB/Redis/外部 API

T+2: 根据症状判断
  CPU 高 → 代码问题/流量增大
  CPU 低但延迟高 → IO/网络/锁/GC
  错误率也高 → 依赖故障/代码 Bug
  QPS 增加 → 流量突增

T+3: 深入排查
  分布式追踪 → 找到慢的 Span
  Profiling → 找到热点代码
  慢查询日志 → 数据库问题

T+5: 恢复
  回滚 / 扩容 / 降级 / 限流
```

### 8. 面试题：CPU 利用率不高但延迟很高可能是什么原因？

**答：**

```
CPU 低 + 延迟高的可能原因：

1. IO 等待
   ├── 磁盘 IO 慢（HDD/网络存储）
   ├── 数据库查询慢
   └── 诊断: iostat -x 看 await 和 %util

2. 网络等待
   ├── 外部 API 超时
   ├── DNS 解析慢
   └── 诊断: 分布式追踪看网络调用

3. 锁竞争
   ├── 数据库行锁/表锁
   ├── 应用层互斥锁（Mutex）
   └── 诊断: goroutine dump / thread dump

4. GC 停顿
   ├── 堆内存过大导致长 GC
   └── 诊断: GC 日志、GC pause 指标

5. 连接池耗尽
   ├── 所有连接都在等待
   └── 诊断: 连接池指标、活跃连接数

6. 线程/协程泄漏
   ├── 大量 goroutine 阻塞
   └── 诊断: goroutine 数量监控
```
