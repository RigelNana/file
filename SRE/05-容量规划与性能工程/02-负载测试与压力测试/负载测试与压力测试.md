## 一、测试类型区分

### 1. 负载测试、压力测试、基准测试有什么区别？

**答：**

| 类型 | 目的 | 负载 | 关注点 |
|------|------|------|--------|
| **基准测试 (Benchmark)** | 测量基准性能 | 固定低负载 | 单次请求延迟/吞吐 |
| **负载测试 (Load Test)** | 验证正常负载下性能 | 预期峰值负载 | 是否满足 SLO |
| **压力测试 (Stress Test)** | 找到系统极限 | 超过预期负载 | 什么时候崩溃 |
| **浸泡测试 (Soak Test)** | 验证长时间稳定性 | 持续中等负载 | 内存泄漏/连接泄漏 |
| **尖峰测试 (Spike Test)** | 验证突发流量应对 | 瞬间高峰 | 弹性伸缩/恢复能力 |

```
负载曲线对比：

负载测试:          压力测试:          钉子(Spike)测试:
  ┌──────┐           ╱               │
  │      │          ╱                │
  │      │         ╱                 │
──┘      └──    ──╱──────          ──┘└──
```

### 2. 负载测试应该如何设计？

**答：**

```
负载测试设计流程：

Step 1: 确定测试目标
  ├── 验证 SLO：P99 延迟 < 200ms @ 5000 QPS
  ├── 找到容量上限
  └── 验证扩容后性能

Step 2: 准备测试场景
  ├── 模拟真实流量特征
  │   ├── 读写比例（80:20）
  │   ├── API 调用分布
  │   ├── 用户行为模式
  │   └── 数据分布特征
  └── 准备测试数据

Step 3: 设计负载模型
  ├── 阶梯式（推荐）：
  │   100 → 500 → 1000 → 2000 → 5000 QPS
  │   每级持续 5 分钟
  ├── 恒定负载：
  │   固定 5000 QPS 持续 30 分钟
  └── 波动负载：
  │   模拟一天中的流量波动

Step 4: 执行和监控
  ├── 监控应用指标（延迟/错误率/吞吐）
  ├── 监控资源指标（CPU/Memory/IO/网络）
  ├── 监控依赖服务（DB/Cache/MQ）
  └── 记录关键数据点

Step 5: 分析和报告
  ├── 确认是否满足 SLO
  ├── 识别瓶颈和拐点
  └── 输出容量规划建议
```

### 4. k6 如何编写负载测试脚本？

**答：**

```javascript
// k6 负载测试脚本示例
import http from "k6/http";
import { check, sleep } from "k6";
import { Rate, Trend } from "k6/metrics";

// 自定义指标
const errorRate = new Rate("errors");
const latencyP99 = new Trend("latency_p99");

// 负载配置：阶梯式
export const options = {
  stages: [
    { duration: "2m", target: 100 },   // 热身
    { duration: "5m", target: 500 },   // 中等负载
    { duration: "5m", target: 1000 },  // 高负载
    { duration: "5m", target: 2000 },  // 峰值负载
    { duration: "2m", target: 0 },     // 冷却
  ],
  thresholds: {
    http_req_duration: ["p(99)<200"],   // P99 < 200ms
    errors: ["rate<0.01"],              // 错误率 < 1%
  },
};

export default function () {
  // 模拟用户行为
  const actions = [
    { weight: 50, fn: browseProducts },
    { weight: 30, fn: searchProducts },
    { weight: 15, fn: addToCart },
    { weight: 5, fn: checkout },
  ];

  // 按权重随机选择动作
  const rand = Math.random() * 100;
  let cumWeight = 0;
  for (const action of actions) {
    cumWeight += action.weight;
    if (rand < cumWeight) {
      action.fn();
      break;
    }
  }
  sleep(1);
}

function browseProducts() {
  const res = http.get("https://api.example.com/products");
  check(res, { "status 200": (r) => r.status === 200 });
  errorRate.add(res.status !== 200);
}

function searchProducts() {
  const res = http.get("https://api.example.com/search?q=phone");
  check(res, { "status 200": (r) => r.status === 200 });
  errorRate.add(res.status !== 200);
}
```

### 5. 如何用 Locust 做分布式压测？

**答：**

```python
# locustfile.py
from locust import HttpUser, task, between

class WebsiteUser(HttpUser):
    wait_time = between(1, 3)

    @task(5)
    def browse(self):
        self.client.get("/products")

    @task(3)
    def search(self):
        self.client.get("/search?q=phone")

    @task(1)
    def checkout(self):
        self.client.post("/orders", json={
            "product_id": 1,
            "quantity": 1
        })
```

```bash
# 分布式压测部署
# Master 节点
locust --master -f locustfile.py

# Worker 节点（可部署多个）
locust --worker --master-host=master-ip -f locustfile.py
```

## 三、结果分析

### 6. 如何分析负载测试结果？

**答：**

```
关键分析维度：

1. 吞吐量 vs 延迟曲线
   
   延迟
   (ms)
   500 │              ╱  ← 拐点：系统极限
   200 │         ·──·╱
   100 │    ·──·
    50 │·──·
       └─────────────────
       0  500 1000 2000  QPS

   拐点前：线性扩展区（安全工作区间）
   拐点后：性能急剧下降（需要扩容/优化）

2. 资源利用率关联
   QPS   CPU   Memory  P99
   500   25%   40%     50ms   ✅ 健康
   1000  50%   55%     80ms   ✅ 健康
   2000  80%   70%     150ms  ⚠️ 接近极限
   3000  95%   85%     500ms  ❌ 超载

3. 错误分析
   ├── 4xx 错误：客户端问题（限流？）
   ├── 5xx 错误：服务端问题
   ├── 超时：容量不足或依赖瓶颈
   └── 连接拒绝：连接数耗尽

4. 瓶颈定位
   CPU 高 → 计算密集型瓶颈
   Memory 高 → 内存泄漏或缓存不足
   IO Wait 高 → 磁盘/数据库瓶颈
   网络高 → 带宽或连接数瓶颈
```

### 7. 面试题：压测时发现 P99 延迟异常，如何排查？

**答：**

```
排查路径：

1. 确认问题范围
   ├── 所有请求还是特定 API？
   ├── 从什么 QPS 开始异常？
   └── P50 和 P99 差距多大？

2. 应用层排查
   ├── 是否有 GC 停顿？（Java/Go GC 日志）
   ├── 是否有锁竞争？（goroutine/thread dump）
   ├── 是否有热点代码？（Profiling）
   └── 连接池是否够用？

3. 依赖层排查
   ├── 数据库延迟是否增加？
   ├── Redis 是否有慢查询？
   ├── 外部 API 是否超时？
   └── 网络是否有丢包？

4. 基础设施排查
   ├── CPU throttling？（K8s limit）
   ├── 同节点争抢？（noisy neighbor）
   ├── 网络带宽饱和？
   └── 磁盘 IO 瓶颈？
```
