## 一、流量预测方法

### 1. 流量预测的主要方法有哪些？

**答：**

| 方法 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| **趋势外推** | 稳定增长 | 简单快速 | 不能预测突变 |
| **业务驱动** | 已知业务计划 | 准确性高 | 依赖业务输入 |
| **时序分析** | 有明显周期性 | 自动捕获周期 | 需要足够历史数据 |
| **组合预测** | 复杂场景 | 多视角综合 | 复杂度高 |

```
三种预测方法的组合：

        ┌── 趋势外推 ──→ 基础增长预测
        │
流量  ──┼── 业务驱动 ──→ 事件性增长预测
预测    │
        └── 时序分析 ──→ 周期性波动预测

最终预测 = 基础趋势 × 季节系数 + 事件增量
```

### 2. 如何将业务增长转化为技术容量需求？

**答：**

```
转化链路：

业务指标 → 用户行为 → 技术指标 → 资源需求

示例（电商场景）：

业务计划：
  月活用户从 100 万 → 150 万（+50%）
  客单价不变，下单转化率提升 2%

→ 用户行为预测：
  日活: 30万 → 45万
  浏览 PV: 300万 → 450万/天
  下单: 3万 → 5.4万/天（+80%）
  支付: 2.4万 → 4.3万/天

→ 技术指标预测：
  API QPS（高峰）: 500 → 750
  订单 TPS（高峰）: 10 → 18
  DB 查询量: 5000 → 7500 QPS

→ 资源需求：
  API 服务: 8C16G × 3 → 8C16G × 5
  订单服务: 4C8G × 2 → 4C8G × 3
  数据库: 主从 → 主从+只读副本
  Redis: 8G → 16G

成本影响：
  月增加约 ¥3000 云资源费用
```

### 3. 如何预测大促/活动的流量峰值？

**答：**

```
大促流量预测方法：

1. 历史类比法
   上次大促峰值 / 上次日常峰值 = 倍率
   本次预测 = 当前日常峰值 × 倍率 × 增长系数

   例：
   去年 618 峰值: 80000 QPS
   去年 618 前日常: 20000 QPS
   倍率: 4x
   今年日常: 30000 QPS
   今年增长预估: 1.2x
   预测: 30000 × 4 × 1.2 = 144000 QPS

2. 流量瀑布法
   营销流量 → 会场浏览 → 商品详情 → 加购 → 下单

   各环节转化率：
   推送 1000万 → 打开 300万 → 浏览 200万
   → 加购 50万 → 下单 20万 → 支付 15万

   换算为 QPS（假设集中在 10 分钟内）：
   浏览 QPS: 200万 / 600s = 3333
   下单 TPS: 20万 / 600s = 333

3. 安全系数
   最终容量 = 预测峰值 × 1.5（安全系数）
   极端场景 = 预测峰值 × 2（备灾容量）
```

## 二、增长应对

### 4. 如何应对超预期的流量增长？

**答：**

```
分级应对策略：

Level 1: 自动弹性（+30% 以内）
  ├── HPA 自动扩 Pod
  ├── Cluster Autoscaler 自动加节点
  └── 无需人工介入

Level 2: 手动扩容（+30-100%）
  ├── 增加云实例
  ├── 扩展数据库只读副本
  ├── 增加 Redis 节点
  └── 需要 SRE 介入

Level 3: 限流降级（>100%或来不及扩容）
  ├── 限流保护核心服务
  ├── 降级非核心功能
  ├── 开启 CDN 静态化
  └── 紧急措施

限流策略配置：
  核心链路（支付/下单）: 最后限流
  非核心链路（推荐/评论）: 优先降级
  静态页面: CDN 托底
```

### 5. 流量增长导致的常见瓶颈？

**答：**

| 增长倍数 | 常见瓶颈 | 应对方案 |
|----------|----------|----------|
| 2x | 应用层 CPU/内存 | 水平扩容 |
| 5x | 数据库连接/QPS | 读写分离、缓存 |
| 10x | 单点瓶颈暴露 | 架构优化 |
| 50x | 全链路瓶颈 | 重新设计架构 |
| 100x+ | 基础设施极限 | 分布式系统 |

```
增长路上的典型架构演进：

单体应用（<1000 QPS）
  ↓ 数据库成为瓶颈
读写分离（<5000 QPS）
  ↓ 单机 App 成为瓶颈
微服务 + LB（<50000 QPS）
  ↓ 数据量增长
分库分表 + 缓存（<200000 QPS）
  ↓ 单地域极限
多地域/多活架构（>200000 QPS）
```

## 三、面试高频题

### 6. 面试题：如何保证容量规划与业务增长同步？

**答：**

```
容量规划与业务同步机制：

1. 定期同步会议
   ├── 月度：SRE + 产品 + 开发
   ├── 内容：未来 3 个月业务计划
   └── 输出：容量需求变更

2. 预案前置
   ├── 新功能上线前 2 周评估容量
   ├── 大促前 1 个月做全链路压测
   └── 高增长服务设置容量告警

3. 自动化反馈
   ├── 增长率仪表盘
   ├── 容量水位自动预警
   └── 自动扩容策略

沟通模板：
  "基于 Q2 业务计划（用户增长 30%），
   预计 API 流量将从 5000 QPS 增长到 6500 QPS。
   当前容量已用 65%，需要在 3 月底前完成扩容。
   预计增加 2 台 8C16G 实例，月成本增加 ¥2000。"
```

### 7. 面试题：公司流量翻了 10 倍你会怎么处理？

**答：**

```
10 倍增长应急方案：

紧急（0-24 小时）：
  1. 限流保护核心服务
  2. 降级非核心功能
  3. CDN 最大化缓存
  4. 快速扩容（云平台弹性）
  → 目标：不崩溃

短期（1-2 周）：
  1. 完成水平扩容
  2. 数据库只读副本
  3. 加缓存层
  4. 压测验证新容量
  → 目标：稳定支撑

中期（1-3 月）：
  1. 架构优化（异步化/分库分表）
  2. 优化热点代码
  3. 重新评估资源模型
  4. 建立弹性扩缩机制
  → 目标：高效支撑

长期（3-6 月）：
  1. 高可用架构升级
  2. 多地域部署
  3. 成本优化
  4. 容量规划自动化
  → 目标：可持续增长
```
