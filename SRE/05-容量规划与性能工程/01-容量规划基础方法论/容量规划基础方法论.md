## 一、容量规划基础

### 1. 什么是容量规划？为什么 SRE 需要关注？

**答：**

```
容量规划定义：
  根据业务增长预测和性能需求，
  提前规划和准备足够的计算资源，
  确保系统在负载增长时仍然可靠。

为什么重要：
  ┌──────────────────────────────────────┐
  │ 容量不足 → 服务降级/宕机 → 用户流失 │
  │ 容量过剩 → 资源浪费 → 成本过高       │
  │ 目标：在可靠性和成本之间找到平衡      │
  └──────────────────────────────────────┘

容量规划的时间维度：
  ├── 短期（1-4 周）：应对即将到来的流量高峰
  ├── 中期（1-3 月）：季度资源规划和采购
  └── 长期（6-12 月）：年度预算和架构规划
```

### 2. 容量规划的标准流程是什么？

**答：**

```
容量规划五步法：

Step 1: 需求收集
  ├── 当前资源使用情况
  ├── 业务增长预测（产品/销售提供）
  ├── 已知的流量高峰（大促/节假日）
  └── 新功能上线计划

Step 2: 资源建模
  ├── 建立 "请求 → 资源" 映射
  │   例：1 QPS → 0.5 CPU Core + 256MB RAM
  ├── 识别瓶颈资源
  └── 考虑依赖服务容量

Step 3: 需求预测
  ├── 基于历史趋势外推
  ├── 加入业务增长因子
  ├── 加入安全余量（Headroom）
  └── 考虑异常峰值（Peak:Average 比率）

Step 4: 差距分析
  ├── 预测需求 vs 当前容量
  ├── 识别缺口和风险
  └── 按优先级排序

Step 5: 执行计划
  ├── 扩容方案（垂直/水平）
  ├── 时间表和里程碑
  ├── 预算审批
  └── 验证和回顾
```

### 3. 容量规划的核心指标有哪些？

**答：**

| 指标 | 说明 | 警戒线 |
|------|------|--------|
| **CPU 利用率** | 计算能力使用比例 | >70% 需关注 |
| **内存利用率** | 内存使用比例 | >80% 需关注 |
| **磁盘 IOPS** | 磁盘读写操作数 | >80% 设备能力 |
| **网络带宽** | 网络吞吐使用率 | >60% 需关注 |
| **连接数** | 数据库/缓存连接数 | >70% 最大值 |
| **队列深度** | 消息积压量 | 持续增长 |
| **Headroom** | 离极限的余量 | >30% |

```promql
# 容量规划关键 PromQL

# CPU Headroom（剩余可用 CPU 百分比）
(1 - avg(rate(container_cpu_usage_seconds_total[5m]))
/ avg(kube_pod_container_resource_limits{resource="cpu"})) * 100

# 内存 Headroom
(1 - avg(container_memory_working_set_bytes)
/ avg(kube_pod_container_resource_limits{resource="memory"})) * 100

# QPS 增长趋势（过去 30 天）
predict_linear(
  rate(http_requests_total[1h])[30d:1h], 86400 * 30
)
```

## 二、资源建模

### 4. 如何建立资源消耗模型？

**答：**

```
资源建模方法：

方法一：经验法则（Quick & Dirty）
  每 1000 QPS 需要：
  ├── 4 CPU Cores
  ├── 4 GB RAM
  ├── 100 Mbps 网络
  └── 根据历史数据调整系数

方法二：基准测试（推荐）
  1. 在压测环境做负载测试
  2. 记录不同 QPS 下的资源消耗
  3. 画出 QPS vs CPU/Memory 曲线
  4. 找到线性区间和拐点

  QPS    CPU    Memory    P99 Latency
  100    0.5c   128MB     10ms
  500    2.5c   512MB     15ms
  1000   5.0c   1GB       20ms
  2000   12c    2.5GB     50ms   ← 开始非线性
  3000   25c    5GB       200ms  ← 性能拐点

方法三：Little's Law
  并发连接数 = 到达率(QPS) × 平均响应时间(s)
  
  例：1000 QPS × 0.05s = 50 并发
  每并发 10MB 内存 → 需要 500MB + overhead
```

### 5. 什么是 Headroom？如何设定？

**答：**

```
Headroom = 系统极限 - 日常负载峰值

Headroom 的作用：
  ├── 应对突发流量                  (流量 spike)
  ├── 容忍节点故障                  (N-1 冗余)
  ├── 留出部署空间                  (rolling update)
  └── 应对不确定性                  (预测偏差)

Headroom 建议值：
  ┌──────────────────────────────────┐
  │ 一般服务:     30% Headroom      │
  │ 核心服务:     50% Headroom      │
  │ 大促前:       100% Headroom     │
  │ 多可用区:     每 AZ 能承受 N-1  │
  └──────────────────────────────────┘

计算示例：
  日常峰值: 5000 QPS
  需要的极限能力: 5000 / (1 - 0.3) = 7143 QPS
  实际资源: 按 7143 QPS 规划

N-1 冗余：
  3 个可用区，每个 AZ 需要能处理 50% 总流量
  （任一 AZ 故障，剩余 2 个 AZ 承接全部流量）
```

## 三、面试高频题

### 6. 面试题：如何为一个新服务做容量规划？

**答：**

```
新服务容量规划路径：

1. 估算阶段（无数据时）
   ├── 参考类似服务的资源模型
   ├── 产品提供预期用户量和增长
   ├── 按 3x 安全系数预留
   └── 使用云平台弹性伸缩

2. 验证阶段（灰度期）
   ├── 灰度期收集实际资源消耗数据
   ├── 建立 QPS → 资源映射
   ├── 做负载测试验证极限
   └── 调整初始估算

3. 稳定阶段（全量后）
   ├── 建立资源仪表盘
   ├── 设定容量告警
   ├── monthly 容量 Review
   └── 根据增长趋势调整
```

### 7. 面试题：容量规划中最容易犯的错误是什么？

**答：**

| 错误 | 原因 | 避免方法 |
|------|------|----------|
| 只看平均值 | 峰值是平均值的 3-10x | 关注 P99 和峰值 |
| 忽略依赖 | 服务扩容了但 DB 没有 | 全链路容量评估 |
| 线性外推 | 瓶颈导致非线性增长 | 做负载测试找拐点 |
| 忘记冗余 | 没有考虑故障场景 | 按 N-1 规划 |
| 太晚规划 | 扩容需要时间 | 提前 2-3 个月 |
| 只加不减 | 资源只增不减 | 定期缩容评估 |
