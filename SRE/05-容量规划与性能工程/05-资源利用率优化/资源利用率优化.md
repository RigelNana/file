# 资源利用率优化八股文

---

## 一、利用率分析

### 1. 如何评估资源利用率是否合理？

**答：**

| 资源 | 低利用率 | 合理范围 | 高风险 |
|------|----------|----------|--------|
| CPU | <20% 浪费 | 30-70% | >80% |
| 内存 | <30% 浪费 | 40-80% | >85% |
| 磁盘 IO | <10% 浪费 | 20-70% | >80% |
| 网络 | <10% 浪费 | 20-60% | >70% |
| K8s Pod | request/limit < 20% | 50-80% | >90% |

```
资源利用率四象限分析：

  利用率高 ┤
           │  过度分配       │  合理使用
           │  (需要扩容)     │  (理想状态)
           ├─────────────────┼──────────────
           │  空闲浪费       │  需要优化
           │  (应该缩容/释放)│  (效率偏低)
  利用率低 ┤
           └─────────────────┴──────────────
                成本低              成本高
```

### 2. K8s Request/Limit 如何优化？

**答：**

```
常见问题：Request 设得太大

例：一个 Pod 请求 4C8G，实际只用 0.5C1G
  → 节点调度效率 12.5%（严重浪费）

优化方法：

1. 数据驱动设置
   # 查看过去 7 天的实际使用
   # P99 使用量作为 Request
   # P99 × 1.5 作为 Limit

2. VPA (Vertical Pod Autoscaler)
   自动推荐 Request/Limit 值
   
   推荐配置：
   ├── Request = P95 实际使用量
   ├── Limit = Request × 2（CPU）
   ├── Limit = Request × 1.5（内存）
   └── 定期 Review VPA 建议

3. 监控 Request vs 实际使用
```

```promql
# CPU 利用效率（Request vs 实际使用）
sum(rate(container_cpu_usage_seconds_total[5m])) by (namespace, pod)
/
sum(kube_pod_container_resource_requests{resource="cpu"}) by (namespace, pod)

# 内存利用效率
sum(container_memory_working_set_bytes) by (namespace, pod)
/
sum(kube_pod_container_resource_requests{resource="memory"}) by (namespace, pod)

# 集群整体利用率
sum(rate(container_cpu_usage_seconds_total[5m]))
/
sum(kube_node_status_allocatable{resource="cpu"})
```

### 3. 如何识别和处理资源浪费？

**答：**

```
资源浪费识别清单：

1. 闲置资源
   ├── 未使用的云实例/VM
   ├── 空闲的 K8s 节点
   ├── 未挂载的磁盘/存储卷
   └── 分配了但未使用的弹性 IP

2. 过度配置
   ├── Pod Request 远大于实际
   ├── 数据库实例规格过大
   ├── Redis 集群节点过多
   └── 预留实例超过需求

3. 环境浪费
   ├── Dev/Staging 环境 24h 运行
   ├── 非工作时间不需要的服务
   └── 测试集群遗忘清理

优化操作：
  ┌──────────────────────────────────┐
  │ 短期：释放明显闲置资源           │
  │ 中期：Right-sizing 调整规格      │
  │ 长期：自动化资源生命周期管理     │
  └──────────────────────────────────┘
```

## 二、优化策略

### 4. 服务端性能优化的常见手段有哪些？

**答：**

| 层次 | 优化手段 | 效果 |
|------|----------|------|
| **代码层** | 算法优化、减少内存分配、连接复用 | 高 |
| **缓存层** | 本地缓存、Redis 缓存、CDN | 极高 |
| **数据库** | 索引优化、SQL 优化、读写分离 | 高 |
| **架构层** | 异步化、消息队列、服务拆分 | 高 |
| **基础设施** | 更好的实例类型、SSD、网络优化 | 中 |
| **容器** | Request/Limit 优化、调度优化 | 中 |

```
优化优先级漏斗：

高收益/低成本 ──────────────────── 优先做
  │ 缓存（命中率每提升 10% = 大幅减少后端压力）
  │ 慢查询优化（一条 SQL 可能影响全局）
  │ 连接池调优
  ├─────────────────────────────────
  │ 代码热点优化（Profiling 驱动）
  │ 异步化（高延迟操作改异步）
  │ 批量处理（N 次网络调用 → 1 次）
  ├─────────────────────────────────
  │ 架构演进（微服务拆分/合并）
  │ 实例类型优化
  │ 网络拓扑优化
低收益/高成本 ──────────────────── 最后做
```

### 5. 缓存优化策略有哪些？

**答：**

```
多级缓存架构：

请求 → 本地缓存 → Redis → 数据库
       (L1)      (L2)    (L3)

各级缓存特点：
  L1 本地缓存:
    延迟: ~0.1ms
    容量: 有限（进程内存）
    一致性: 最终一致
    适用: 热点数据、配置信息

  L2 Redis 缓存:
    延迟: ~1ms
    容量: 大（集群）
    一致性: 较好
    适用: 会话、排行榜、频繁查询

  L3 数据库:
    延迟: ~5-50ms
    容量: 最大
    一致性: 强一致
    适用: 所有数据

缓存命中率优化：
  目标: > 95%
  
  命中率低的原因：
  ├── TTL 设置不合理（太短）
  ├── 缓存容量不足（淘汰频繁）
  ├── 缓存 Key 设计不当（粒度不对）
  └── 冷启动问题（重启后缓存空）
```

## 三、面试高频题

### 6. 面试题：集群利用率只有 20%，如何优化？

**答：**

```
优化方案：

1. 分析原因
   ├── 是按峰值配置但峰值少？→ 用弹性伸缩
   ├── 是过度配置了 Request？→ Right-sizing
   ├── 是有大量闲置服务？→ 清理和合并
   └── 是多环境浪费？→ 非生产环境缩容

2. 短期改善（1-2 周）
   ├── Right-sizing: 用 VPA 推荐调整 Request
   ├── 释放闲置: 删除未使用资源
   ├── 节点缩容: 减少 K8s 节点数量
   └── 目标: 利用率提升到 40%

3. 中期改善（1-3 月）
   ├── HPA 自动伸缩
   ├── 分时调度（非工作时间缩容）
   ├── Spot/Preemptible 实例混合
   └── 目标: 利用率提升到 50-60%

4. 长期改善
   ├── FinOps 机制（成本归属+优化）
   ├── 自动化资源生命周期管理
   ├── 混部（在线+离线混合调度）
   └── 目标: 利用率 60-70%
```

### 7. 面试题：如何在不影响性能的前提下降低成本？

**答：**

| 策略 | 做法 | 预期节省 |
|------|------|----------|
| **Right-sizing** | 调整实例/Pod 到合理规格 | 20-40% |
| **Spot 实例** | 无状态服务使用 Spot | 60-90% |
| **预留实例** | 稳定负载使用 RI/Savings Plan | 30-50% |
| **弹性伸缩** | 低谷期自动缩容 | 20-30% |
| **存储优化** | 冷数据迁移到低成本存储 | 30-50% |
| **缓存优化** | 减少数据库查询 | 间接节省 |
