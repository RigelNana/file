## 一、AIOps 基础概念

### 1. 什么是 AIOps？

**答：**

```
AIOps 定义（Gartner 2017 提出）：
  Artificial Intelligence for IT Operations
  利用机器学习和大数据分析技术，
  增强 IT 运维的自动化和智能化。

AIOps 核心能力：
  ┌─────────────────────────────────────┐
  │            AIOps 平台               │
  ├──────────┬──────────┬───────────────┤
  │ 异常检测  │ 根因定位 │ 智能预测      │
  ├──────────┼──────────┼───────────────┤
  │ 告警降噪  │ 关联分析 │ 容量预测      │
  │ 基线学习  │ 拓扑分析 │ 故障预测      │
  │ 多维分析  │ 日志聚类 │ 趋势分析      │
  ├──────────┴──────────┴───────────────┤
  │    数据：指标 + 日志 + 链路 + 事件   │
  └─────────────────────────────────────┘

AIOps 与传统监控的区别：
  传统监控：固定阈值 → 触发告警 → 人来排查
  AIOps：  动态基线 → 异常检测 → 自动关联 → 根因推荐

Gartner AIOps 成熟度：
  Level 0: 基于规则的监控
  Level 1: 数据聚合和可视化
  Level 2: 自动化模式发现
  Level 3: 预测性分析
  Level 4: 自治运维
```

### 2. AIOps 有哪些核心应用场景？

**答：**

| 场景 | 问题 | AI 方案 | 效果 |
|------|------|---------|------|
| 异常检测 | 固定阈值误报漏报 | 动态基线学习 | 减少 80% 误报 |

| 故障预测 | 事后才知道有问题 | 异常趋势预测 | 提前预警 |

## 二、核心技术

### 3. 异常检测算法有哪些？

**答：**

```
异常检测算法分类：

1. 统计方法
   - 3-Sigma 法则
     均值 ± 3σ 范围外视为异常
     简单但不适合非正态分布

   - 移动平均 + 标准差
     window_size = 60  # 60分钟窗口
     if value > mean(window) + 3 * std(window):
         alert("异常")

2. 机器学习方法
   - Isolation Forest
     通过随机划分隔离异常点
     适合多维数据

   - DBSCAN
     基于密度的聚类
     离群点即异常

   - Autoencoder（自编码器）
     学习正常模式的压缩表示
     重构误差大 = 异常

3. 时序分析方法
   - Prophet（Facebook）
     分解趋势+季节性+节假日
     适合有周期性的指标

   - ARIMA
     自回归移动平均模型
     经典时序预测

4. 深度学习方法
   - LSTM
     长短期记忆网络
     捕获长期依赖

常见选择：
  ┌────────────────┬────────────────────┐
  │ 场景           │ 推荐算法            │
  ├────────────────┼────────────────────┤
  │ CPU/内存指标   │ 3-Sigma / Prophet  │
  │ QPS/延迟      │ 移动平均 + EWMA    │
  │ 多维异常      │ Isolation Forest   │
  │ 日志异常      │ Autoencoder        │
  │ 容量预测      │ Prophet / ARIMA    │
  └────────────────┴────────────────────┘
```

### 4. 告警降噪怎么做？

**答：**

```
告警降噪策略：

1. 告警聚合（Aggregation）
   同一服务/同一类型的告警合并
   Example：10 个 Pod OOM → 1 条聚合告警

2. 告警去重（Deduplication）
   相同告警只通知一次
   Alertmanager group_by + group_wait

3. 告警抑制（Suppression）
   高优先级告警抑制低优先级
   Example：节点宕机 → 抑制该节点所有 Pod 告警

4. 告警关联（Correlation）
   基于拓扑关系关联告警
   数据库慢 → API 超时 → 前端报错
   合并为一个根因事件

5. 告警分级（Prioritization）
   基于影响范围和 SLO 动态调整优先级

告警降噪效果：
  原始告警 → 聚合 → 去重 → 抑制 → 关联
  1000 条  → 200  → 100  →  50  →  10

Alertmanager 降噪配置：
```

```yaml
# Alertmanager 聚合配置
route:
  group_by: ['alertname', 'service']
  group_wait: 30s        # 等30s聚合
  group_interval: 5m     # 5分钟发一次
  repeat_interval: 4h    # 重复间隔
  
inhibit_rules:
  - source_match:
      severity: 'critical'
    target_match:
      severity: 'warning'
    equal: ['service']    # 同服务，critical 抑制 warning
```

## 三、AIOps 实践

### 5. 如何用 LLM 辅助故障排查？

**答：**

```
LLM 在运维中的应用：

1. 日志分析
   输入：大量错误日志
   LLM：提取关键信息，生成排查建议
   
2. 根因分析辅助
   输入：告警信息 + 拓扑图 + 变更记录
   LLM：关联分析，推荐最可能的根因

3. Runbook 生成
   输入：故障描述
   LLM：生成排查步骤和修复建议

4. 自然语言查询
   用户："最近 1 小时支付服务的错误率"
   LLM → 生成 PromQL → 执行查询 → 返回结果

注意事项：
  ┌──────────────────────────────────┐
  │ ⚠️ LLM 在运维中的注意事项        │
  ├──────────────────────────────────┤
  │ 1. LLM 可能产生幻觉（错误信息） │
  │ 2. 不能让 LLM 直接执行命令       │
  │ 3. 需要人工验证 LLM 的建议       │
  │ 4. 敏感数据不要传给外部 LLM      │
  │ 5. LLM 是辅助工具，不是替代品    │
  └──────────────────────────────────┘
```

### 6. 容量预测怎么做？

**答：**

```python
# 使用 Prophet 进行容量预测示例
from prophet import Prophet
import pandas as pd

# 准备数据（过去90天的CPU使用率）
df = pd.DataFrame({
    'ds': date_range,      # 时间列
    'y': cpu_utilization    # CPU使用率
})

# 建立模型
model = Prophet(
    changepoint_prior_scale=0.05,
    seasonality_mode='multiplicative',
    daily_seasonality=True,
    weekly_seasonality=True,
)

model.fit(df)

# 预测未来30天
future = model.make_future_dataframe(periods=30)
forecast = model.predict(future)

# 找到容量耗尽时间
capacity_limit = 80  # 80% 阈值
breach_date = forecast[
    forecast['yhat_upper'] > capacity_limit
]['ds'].iloc[0]

print(f"预计 {breach_date} 达到容量上限")
```

```
容量预测实践要点：
  1. 数据质量：至少 3 个月历史数据
  2. 周期性：考虑日/周/月/年周期
  3. 异常剔除：剔除故障期间数据
  4. 多指标：CPU + 内存 + 存储 + 带宽
  5. 安全水位：预留 20-30% Buffer
  6. 定期回溯：验证预测准确性
```

## 四、面试高频题

### 7. 面试题：AIOps 的挑战？

**答：**

```
核心挑战：

1. 数据质量
   - 指标不完整、日志格式不统一
   - 缺乏标注数据（什么是"正常"）
   - 数据孤岛（监控/日志/CMDB 不互通）

2. 模型准确性
   - 误报率高影响信任度
   - 冷启动问题（新服务无历史数据）
   - 概念漂移（服务变更后模型失效）

3. 可解释性
   - "AI 说有异常" 不够  
   - 需要告诉为什么异常、影响什么
   - 运维人员需要理解和信任结果

4. 落地难度
   - 过度承诺的期望管理
   - 与现有工具链集成
   - 团队 AI 技能不足

务实建议：
  ┌──────────────────────────────────┐
  │ 1. 从小处开始：先做告警降噪      │
  │ 2. 规则+AI 结合：不要完全依赖 AI │
  │ 3. 持续迭代：反馈循环改进模型    │
  │ 4. 关注数据：数据质量 > 算法      │
  │ 5. 人在回路：AI 建议，人来决策    │
  └──────────────────────────────────┘
```

### 8. 面试题：如何评估 AIOps 平台？

**答：**

| 评估维度 | 评估要点 | 参考标准 |
|----------|----------|----------|
| 数据接入 | 支持的数据源种类 | 指标/日志/链路/事件 |
| 异常检测 | 误报率/漏报率 | 误报 <10%, 漏报 <5% |
| 告警降噪 | 降噪比例 | >80% 告警量减少 |
| 根因分析 | 定位准确率 | Top-3 命中率 >70% |
| 集成能力 | 与现有工具对接 | Prometheus/ELK/PagerDuty |
| 可解释性 | 结果是否可理解 | 提供原因说明和证据 |
| 成本 | TCO（总拥有成本） | 与收益匹配 |
| 部署模式 | SaaS/私有化 | 符合安全合规要求 |
