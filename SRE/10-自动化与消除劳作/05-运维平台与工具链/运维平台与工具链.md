## 一、运维平台设计

### 1. SRE 运维平台的核心功能有哪些？

**答：**

```
运维平台功能全景：

┌──────────────────────────────────────────────┐
│                运维门户（Portal）              │
├────────┬────────┬────────┬────────┬──────────┤
│ CMDB   │ 发布   │ 监控   │ 工单   │ 知识库   │
│ 资产   │ 部署   │ 告警   │ 审批   │ Runbook  │
├────────┴────────┴────────┴────────┴──────────┤
│            编排引擎 / 自动化引擎              │
├────────┬────────┬────────┬────────┬──────────┤
│ 配置   │ 容器   │ 日志   │ 链路   │ 安全     │
│ 管理   │ 平台   │ 平台   │ 追踪   │ 审计     │
├────────┴────────┴────────┴────────┴──────────┤
│           IaaS / 云平台 / K8s                 │
└──────────────────────────────────────────────┘

核心模块：
  1. CMDB（配置管理数据库）
     - 资产管理、服务目录、依赖拓扑
  2. 发布部署系统
     - CI/CD、灰度发布、回滚
  3. 监控告警平台
     - 指标/日志/链路 统一视图
  4. 工单系统
     - 事件管理、变更审批、问题跟踪
  5. 自动化引擎
     - 任务编排、定时任务、Runbook 执行
```

### 2. 如何设计运维平台的技术架构？

**答：**

```
分层架构设计：

展示层：
  - 统一门户 (React/Vue)
  - 仪表盘 (Grafana 嵌入)
  - CLI 工具

API 层：
  - API Gateway (Kong/APISIX)
  - 认证授权 (OAuth2 + RBAC)
  - 审计日志

服务层：
  ┌─────────┬─────────┬──────────┐
  │ CMDB    │ Deploy  │ Monitor  │
  │ Service │ Service │ Service  │
  ├─────────┼─────────┼──────────┤
  │ Ticket  │ Auto    │ Config   │
  │ Service │ Service │ Service  │
  └─────────┴─────────┴──────────┘

数据层：
  - MySQL/PostgreSQL (业务数据)
  - Redis (缓存/分布式锁)
  - Elasticsearch (日志/搜索)
  - Kafka (事件总线)

关键设计原则：
  1. 统一身份认证 (SSO)
  2. 细粒度 RBAC 权限控制
  3. 所有操作有审计日志
  4. API First 设计
  5. 插件化/可扩展
```

## 二、工具链生态

### 3. SRE 常用工具链有哪些？

**答：**

| 领域 | 开源工具 | 商业方案 |
|------|----------|----------|
| 监控 | Prometheus + Grafana | Datadog, New Relic |
| 日志 | ELK/EFK, Loki | Splunk, Sumo Logic |
| 链路追踪 | Jaeger, Zipkin | Lightstep, Honeycomb |
| 告警 | Alertmanager | PagerDuty, OpsGenie |
| CI/CD | Jenkins, GitLab CI, Argo CD | CircleCI, GitHub Actions |
| IaC | Terraform, Pulumi | Env0, Spacelift |
| 配置管理 | Ansible, SaltStack | Puppet Enterprise |
| 容器平台 | Kubernetes | EKS/GKE/AKS |
| 服务网格 | Istio, Linkerd | Consul Connect |
| 混沌工程 | LitmusChaos, Chaos Mesh | Gremlin |
| 事件管理 | — | PagerDuty, Rootly |
| 状态页 | Cachet | Statuspage.io |

### 4. 如何选择工具链？选型原则是什么？

**答：**

```
工具选型原则：

1. 问题驱动而非工具驱动
   ✗ "我们该用什么监控？"
   ✓ "我们的监控需求是什么？"

2. 评估维度
   ┌────────────┬──────────────────────┐
   │ 维度       │ 评估要点              │
   ├────────────┼──────────────────────┤
   │ 功能匹配   │ 核心需求覆盖度        │
   │ 可扩展性   │ 数据量增长后能否支撑   │
   │ 社区活跃度 │ GitHub stars/issues   │
   │ 集成能力   │ 与现有工具链兼容       │
   │ 运维成本   │ 部署/维护/人力成本     │
   │ 团队技能   │ 团队是否有经验         │
   │ 供应商锁定 │ 迁移成本              │
   └────────────┴──────────────────────┘

3. Build vs Buy 决策
   Build 适合：核心竞争力、特殊需求
   Buy 适合：通用功能、团队规模小

4. 统一 > 最优
   10 个各自最优的工具 < 5 个互相集成的工具
```

## 三、内部开发者平台（IDP）

### 5. 什么是内部开发者平台（Internal Developer Platform）？

**答：**

```
IDP 定义：
  为开发者提供自助服务能力的统一平台，
  让开发者专注业务逻辑，
  无需关心基础设施细节。

IDP 核心理念：
  Platform as a Product（平台即产品）

IDP 功能示例：
  ┌──────────────────────────────────┐
  │ 开发者自助门户                    │
  ├──────────┬───────────┬───────────┤
  │ 创建服务  │ 部署上线  │ 查看监控  │
  │ 申请资源  │ 查看日志  │ 管理配置  │
  ├──────────┴───────────┴───────────┤
  │       模板引擎 / 脚手架           │
  ├──────────────────────────────────┤
  │   K8s  │  CI/CD  │  IaC  │ RBAC │
  └──────────────────────────────────┘

知名实现：
  - Backstage (Spotify 开源)
  - Port
  - Humanitec
  - Kratix

Backstage 核心功能：
  1. Software Catalog (服务目录)
  2. Software Templates (脚手架)
  3. TechDocs (技术文档)
  4. Plugins (可扩展插件体系)
```

### 6. IDP 与 DevOps 工具链的关系？

**答：**

```
IDP 是 DevOps 工具链的统一抽象层：

传统方式（开发者直接面对工具链）：
  开发者 → Jenkins → Terraform → K8s → Grafana
          → GitLab → Ansible → ELK → PagerDuty
  问题：工具太多，认知负担重

IDP 方式（开发者面对统一平台）：
  开发者 → IDP 自助门户
              │
              ├── 创建服务 → 自动申请 K8s namespace
              ├── 部署 → 自动触发 CI/CD
              ├── 监控 → 自动配置 Prometheus
              └── 日志 → 自动接入 ELK

团队职责划分：
  平台团队：建设和维护 IDP
  SRE 团队：定义可靠性标准，融入平台
  开发团队：使用 IDP 自助完成工作
```

## 四、面试高频题

### 7. 面试题：如何建设可观测性工具链？

**答：**

```
三大支柱整合方案：

指标（Metrics）：
  采集：Prometheus + Node Exporter + kube-state-metrics
  存储：Thanos / Cortex / VictoriaMetrics（长期存储）
  展示：Grafana

日志（Logs）：
  采集：Fluent Bit（轻量级采集器）
  传输：Kafka（缓冲层）
  存储：Elasticsearch / Loki
  展示：Kibana / Grafana

链路追踪（Traces）：
  SDK：OpenTelemetry
  采集：OTel Collector
  存储：Jaeger / Tempo
  展示：Grafana + Jaeger UI

统一整合关键：
  1. 标准化：OpenTelemetry 统一数据格式
  2. 关联：TraceID 关联日志和指标
  3. 单一面板：Grafana 统一展示
  4. 告警：Alertmanager 统一告警入口
```

### 8. 面试题：运维平台的 RBAC 怎么设计？

**答：**

```
RBAC 设计模型：

角色层级：
  超级管理员 → 全部权限
  平台管理员 → 平台配置管理
  SRE       → 发布/回滚/扩缩容/监控
  开发者     → 查看日志/监控/部署测试环境
  只读用户   → 仅查看

权限粒度：
  ┌──────────┬────────────────────┐
  │ 资源类型  │ 操作                │
  ├──────────┼────────────────────┤
  │ 服务     │ 查看/部署/回滚/删除  │
  │ 监控     │ 查看/配置/静默       │
  │ 工单     │ 创建/审批/关闭       │
  │ 配置     │ 查看/修改/发布       │
  │ 资源     │ 申请/批准/回收       │
  └──────────┴────────────────────┘

数据隔离：
  - 按项目/团队/环境隔离
  - 开发者只能看到自己团队的服务
  - 生产环境操作需要额外审批
```
