### 2. 基础设施测试怎么做？

**答：**

```go
// Terratest 示例：验证 VPC 创建正确
package test

import (
    "testing"
    "github.com/gruntwork-io/terratest/modules/terraform"
    "github.com/stretchr/testify/assert"
)

func TestVPC(t *testing.T) {
    t.Parallel()
    
    opts := &terraform.Options{
        TerraformDir: "../modules/vpc",
        Vars: map[string]interface{}{
            "vpc_cidr": "10.0.0.0/16",
            "name":     "test-vpc",
        },
    }
    
    // 测试结束后自动销毁资源
    defer terraform.Destroy(t, opts)
    
    // 创建基础设施
    terraform.InitAndApply(t, opts)
    
    // 验证输出
    vpcID := terraform.Output(t, opts, "vpc_id")
    assert.NotEmpty(t, vpcID)
    
    subnetCount := terraform.Output(t, opts, "subnet_count")
    assert.Equal(t, "3", subnetCount)
    
    // 验证 VPC 真实存在于 AWS
    vpc := aws.GetVpcById(t, vpcID, "us-east-1")
    assert.Equal(t, "10.0.0.0/16", vpc.CidrBlock)
}
```

## 二、部署验证

### 3. 部署后如何自动化验证？

**答：**

```
部署验证策略（Post-Deployment Verification）：

阶段一：Smoke 测试（秒级）
  - 健康检查端点返回 200
  - 关键 API 可调通
  - 数据库连接正常

阶段二：功能验证（分钟级）
  - 核心业务流程跑通
  - 端到端测试通过
  - 依赖服务连通性

阶段三：性能验证（分钟级）
  - P99 延迟在阈值内
  - 错误率低于基线
  - QPS 达到预期

阶段四：金丝雀观察（小时级）
  - 对比新旧版本指标
  - 日志错误模式分析
  - 用户反馈监控

自动化验证流程：
  deploy → smoke → functional → canary observe
    │        │          │            │
    │      <30s       <5min       <1hr
    │        │          │            │
    │      fail→       fail→       fail→
    │      rollback   rollback   rollback
    │
    └──→ success at all stages → full rollout
```

```yaml
# Argo Rollouts 自动化验证
apiVersion: argoproj.io/v1alpha1
kind: Rollout
spec:
  strategy:
    canary:
      steps:
        - setWeight: 5
        - analysis:
            templates:
              - templateName: success-rate
            args:
              - name: service-name
                value: payment-api
        - setWeight: 25
        - pause: { duration: 10m }
        - setWeight: 50
        - analysis:
            templates:
              - templateName: latency-check
        - setWeight: 100
---
apiVersion: argoproj.io/v1alpha1
kind: AnalysisTemplate
metadata:
  name: success-rate
spec:
  metrics:
    - name: success-rate
      interval: 60s
      count: 5
      successCondition: result[0] >= 0.99
      provider:
        prometheus:
          address: http://prometheus:9090
          query: |
            sum(rate(http_requests_total{
              service="{{args.service-name}}",
              status=~"2.."
            }[5m])) /
            sum(rate(http_requests_total{
              service="{{args.service-name}}"
            }[5m]))
```

### 4. Kubernetes 配置验证有哪些工具？

**答：**

| 工具 | 用途 | 特点 |
|------|------|------|
| kubeval | YAML schema 验证 | 快速验证格式 |
| kubeconform | YAML schema 验证 | kubeval 替代品，更快 |
| kube-score | 最佳实践检查 | 检查资源限制/探针等 |
| OPA/Gatekeeper | 策略验证 | 自定义策略引擎 |
| Kyverno | 策略验证 | K8s 原生策略 |
| Datree | 安全+最佳实践 | CI 集成友好 |
| Pluto | API 弃用检测 | 升级前检查 |

```bash
# 配置验证流水线示例
```

## 三、性能测试

### 5. SRE 视角的性能测试如何做？

**答：**

```
性能测试分类：

1. 基准测试（Benchmark）
   - 确定正常性能基线
   - 每次发布后对比基线

2. 负载测试（Load Test）
   - 预期负载下的表现
   - 验证 SLO 能否达成

3. 压力测试（Stress Test）
   - 超过预期负载的表现
   - 找到系统瓶颈点

4. 容量测试（Capacity Test）
   - 系统最大处理能力
   - 指导容量规划

常用工具：
  ┌──────────┬────────────────────────┐
  │ 工具     │ 特点                    │
  ├──────────┼────────────────────────┤
  │ k6       │ JavaScript 脚本，现代   │
  │ Locust   │ Python 脚本，分布式     │
  │ wrk      │ 轻量高性能 HTTP 基准    │
  │ Vegeta   │ Go 实现，固定 QPS       │
  │ Gatling  │ Scala，报告美观         │
  └──────────┴────────────────────────┘
```

```javascript
// k6 性能测试脚本示例
import http from 'k6/http';
import { check, sleep } from 'k6';
import { Rate } from 'k6/metrics';

const errorRate = new Rate('errors');

export const options = {
  stages: [
    { duration: '2m', target: 100 },  // 2分钟升到100 VU
    { duration: '5m', target: 100 },  // 保持5分钟
    { duration: '2m', target: 200 },  // 升到200 VU
    { duration: '5m', target: 200 },  // 保持5分钟
    { duration: '2m', target: 0 },    // 降到0
  ],
  thresholds: {
    http_req_duration: ['p(99)<500'],  // P99 < 500ms
    errors: ['rate<0.01'],              // 错误率 < 1%
  },
};

export default function () {
  const res = http.get('https://api.example.com/orders');
  check(res, {
    'status is 200': (r) => r.status === 200,
    'latency < 500ms': (r) => r.timings.duration < 500,
  });
  errorRate.add(res.status !== 200);
  sleep(1);
}
```

## 四、面试高频题

### 6. 面试题：如何在 CI/CD 中集成自动化测试？

**答：**

```
CI/CD 测试集成方案：

PR 阶段（快速反馈）：
  ├── YAML lint + schema 验证 (~10s)
  ├── 单元测试 (~1min)
  ├── 配置策略检查 OPA (~10s)
  └── 安全扫描 Trivy (~30s)

Merge 后（部署前验证）：
  ├── 集成测试 (~5min)
  ├── Terratest 基础设施验证 (~10min)
  └── 构建镜像 + 漏洞扫描

部署后（运行时验证）：
  ├── Smoke 测试 (~30s)
  ├── 功能验证 (~3min)
  ├── 金丝雀分析 (~30min)
  └── 性能基准对比

流水线示例：
  git push → lint → test → build → deploy staging
    → smoke → functional → deploy canary
    → analyze → promote → full rollout
    → performance baseline
```

### 7. 面试题：如何保证测试环境的可靠性？

**答：**

```
测试环境管理策略：

1. 环境即代码
   - Terraform + Ansible 管理测试环境
   - 可随时销毁重建

2. 数据管理
   - 测试数据工厂（生成脱敏数据）
   - 数据库快照恢复
   - 避免使用生产数据

3. 环境隔离
   ┌──────────────────────────────┐
   │ 方案           适用场景      │
   ├──────────────────────────────┤
   │ Namespace 隔离  同集群不同NS │
   │ 独立集群       完全隔离      │
   │ Ephemeral 环境  PR 级别环境  │
   │ Feature Flag   共享环境+开关 │
   └──────────────────────────────┘

4. 环境监控
   - 测试环境也要有基本监控
   - 资源告警（避免资源泄漏）
   - 环境健康巡检

5. 成本控制
   - 非工作时间自动关闭
   - TTL 机制自动回收
   - 按需创建临时环境
```
