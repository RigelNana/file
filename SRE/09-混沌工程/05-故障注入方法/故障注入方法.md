## 一、故障注入类型

### 1. 故障注入有哪些层级和方法？

**答：**

```
故障注入层级模型：

┌─────────────────────────────────────┐
│          应用层故障注入              │
│  代码级异常、配置错误、依赖超时      │
├─────────────────────────────────────┤
│          运行时故障注入              │
│  进程 Kill、CPU/内存压力、GC 注入   │
├─────────────────────────────────────┤
│          网络层故障注入              │
│  延迟、丢包、分区、DNS 故障         │
├─────────────────────────────────────┤
│          基础设施故障注入            │
│  实例终止、磁盘故障、AZ 故障        │
└─────────────────────────────────────┘
```

| 层级 | 方法 | 工具 |
|------|------|------|
| 网络 | tc netem | Linux 内核 |
| 网络 | iptables 规则 | Linux 防火墙 |
| 网络 | Toxiproxy | 代理层注入 |
| 进程 | kill -9 / kill -15 | Linux 信号 |
| 资源 | stress-ng | 压力工具 |
| 应用 | 代码故障标志 | Feature Flag |
| K8s | ChaosEngine | LitmusChaos |
| 云 | AWS FIS | 云厂商服务 |

### 2. 如何使用 tc netem 注入网络故障？

**答：**

```bash
# tc netem 网络故障注入（Linux Traffic Control）

# 添加 200ms 延迟
tc qdisc add dev eth0 root netem delay 200ms

# 添加 200ms ± 50ms 的随机延迟
tc qdisc add dev eth0 root netem delay 200ms 50ms

# 添加 10% 丢包
tc qdisc add dev eth0 root netem loss 10%

# 添加 1% 包损坏
tc qdisc add dev eth0 root netem corrupt 1%

# 添加 5% 包重排
tc qdisc add dev eth0 root netem reorder 5%

# 组合：200ms 延迟 + 5% 丢包
tc qdisc add dev eth0 root netem delay 200ms loss 5%

# 清除所有规则
tc qdisc del dev eth0 root

# 只对特定目标 IP 注入延迟
tc qdisc add dev eth0 root handle 1: prio
tc qdisc add dev eth0 parent 1:3 handle 30: netem delay 200ms
tc filter add dev eth0 parent 1:0 protocol ip u32 \
  match ip dst 10.0.0.5/32 flowid 1:3
```

### 3. Toxiproxy 如何做应用级故障注入？

**答：**

```
Toxiproxy 工作原理：

应用 ──→ Toxiproxy ──→ 目标服务
          │
          ├── 注入延迟
          ├── 注入超时
          ├── 注入连接断开
          └── 限制带宽
```

```bash
# 创建代理
toxiproxy-cli create redis-proxy \
  -l localhost:26379 \
  -u redis-server:6379

# 注入延迟
toxiproxy-cli toxic add redis-proxy \
  -t latency -a latency=200 -a jitter=50

# 注入超时（下游不响应）
toxiproxy-cli toxic add redis-proxy \
  -t timeout -a timeout=5000

# 注入连接断开
toxiproxy-cli toxic add redis-proxy \
  -t reset_peer -a timeout=500

# 限制带宽
toxiproxy-cli toxic add redis-proxy \
  -t bandwidth -a rate=10  # 10 KB/s

# 移除所有故障
toxiproxy-cli toxic remove redis-proxy -n latency_downstream
```

## 二、高级故障注入

### 4. 如何使用 stress-ng 做资源压力注入？

**答：**

```bash
# CPU 压力：使用 4 个 CPU 核心满载
stress-ng --cpu 4 --timeout 60s

# 内存压力：分配 2GB 内存
stress-ng --vm 2 --vm-bytes 1G --timeout 60s

# 磁盘 IO 压力
stress-ng --io 4 --hdd 2 --timeout 60s

# 网络压力
stress-ng --sock 100 --timeout 60s

# 组合压力
stress-ng --cpu 2 --vm 1 --vm-bytes 512M \
  --io 2 --timeout 120s
```

### 5. AWS Fault Injection Simulator (FIS) 怎么使用？

**答：**

```json
{
  "description": "模拟 EC2 实例故障",
  "targets": {
    "myInstances": {
      "resourceType": "aws:ec2:instance",
      "resourceTags": {
        "Environment": "production",
        "Service": "payment"
      },
      "selectionMode": "COUNT(1)"
    }
  },
  "actions": {
    "stopInstances": {
      "actionId": "aws:ec2:stop-instances",
      "parameters": {
        "startInstancesAfterDuration": "PT5M"
      },
      "targets": {
        "Instances": "myInstances"
      }
    }
  },
  "stopConditions": [
    {
      "source": "aws:cloudwatch:alarm",
      "value": "arn:aws:cloudwatch:...:alarm:HighErrorRate"
    }
  ]
}
```

```
AWS FIS 支持的故障类型：
  ├── EC2: Stop/Terminate/Reboot
  ├── ECS: Drain/Stop Task
  ├── EKS: Node Terminate
  ├── RDS: Failover/Reboot
  ├── Network: Disrupt Connectivity
  └── SSM: Run Script (自定义)

安全特性：
  ├── Stop Conditions（自动停止条件）
  ├── IAM 权限控制
  ├── CloudTrail 审计日志
  └── 实验时间窗口限制
```

## 三、面试高频题

### 6. 面试题：如何安全地在生产环境做故障注入？

**答：**

```
安全故障注入清单：

准备阶段：
  □ 确认监控和告警正常工作
  □ 确认回滚/恢复机制可用
  □ 通知所有相关团队
  □ 确认不在变更冻结期
  □ 获得审批

执行阶段：
  □ 从最小范围开始（1% 流量 / 1 个实例）
  □ 有 Kill Switch 随时终止
  □ 有自动停止条件（错误率阈值）
  □ 有人实时监控仪表盘
  □ 工作时间内进行

恢复阶段：
  □ 确认故障已完全清除
  □ 确认服务恢复正常
  □ 记录实验结果
  □ 创建发现的 Action Items
```

### 7. 面试题：故障注入与混沌工程的区别？

**答：**

```
故障注入是手段，混沌工程是方法论

故障注入：
  "我要 Kill 一个 Pod，看看会怎样"
  → 工具层面

混沌工程：
  "我假设当 Pod 被 Kill 时，
   系统应在 30 秒内自愈，
   用户不会感知到中断。
   现在让我们验证这个假说。"
  → 科学实验方法

  故障注入 ⊂ 混沌工程
  混沌工程 = 假说 + 故障注入 + 观测 + 学习
```
