# 混沌实验设计原则八股文

---

## 一、实验设计核心

### 1. 混沌实验的设计原则是什么？

**答：**

```
六大设计原则：

1. 最小化爆炸半径
   从小规模开始 → 逐步扩大
   1% 流量 → 10% → 50% → 100%

2. 可观测性优先
   实验前确保监控完备
   没有监控 = 盲目破坏

3. 自动停止机制 (Kill Switch)
   设定安全阈值，超过自动终止
   例：错误率 > 5% → 立即停止

4. 渐进式推进
   Staging → 生产金丝雀 → 生产全量

5. 假说驱动
   不是随机破坏，是验证假说
   "当 X 发生时，我们预期 Y"

6. 记录一切
   实验过程、观察结果、意外发现
   为 Postmortem 和改进计划提供素材
```

### 2. 稳态假说（Steady State Hypothesis）如何定义？

**答：**

```
稳态 = 系统正常工作时的可观测行为

好的稳态假说：
  ┌─────────────────────────────────────────┐
  │ "在正常条件下，                          │
  │  订单创建成功率 > 99.9%，                │
  │  P99 延迟 < 200ms，                     │
  │  错误率 < 0.1%"                         │
  └─────────────────────────────────────────┘

然后混沌实验的假说是：
  "当 [注入故障 X] 时，
   系统的稳态指标仍然保持在可接受范围：
   成功率 > 99.5%，P99 < 500ms"

稳态选择原则：
  ✓ 用户可感知的业务指标（成功率）
  ✓ 可量化的性能指标（延迟）
  ✗ 内部实现指标（CPU 使用率）
  ✗ 模糊的描述（"系统正常工作"）
```

### 3. 实验的安全防护如何设计？

**答：**

| 防护层 | 机制 | 说明 |
|--------|------|------|
| **范围限制** | 流量比例控制 | 只影响 N% 的流量 |
| **自动终止** | Kill Switch | 指标超阈值自动停止 |
| **时间限制** | 超时机制 | 最长实验时间 |
| **人工确认** | 审批流程 | 生产实验需审批 |
| **回滚能力** | 一键恢复 | 可立即撤销故障注入 |
| **通知机制** | 实验通知 | 相关团队收到通知 |

```yaml
# 混沌实验安全配置示例
experiment:
  name: "redis-primary-failure"
  
  # 范围限制
  scope:
    environment: production
    traffic_percentage: 5    # 只影响5%流量
    target_pods: 1           # 只kill 1个Pod
  
  # 安全防护
  safety:
    kill_switch: true
    abort_conditions:
      - metric: "error_rate"
        threshold: 5         # 错误率>5%自动停止
        window: "1m"
      - metric: "p99_latency_ms"
        threshold: 2000      # P99>2s自动停止
        window: "1m"
    max_duration: "10m"      # 最长10分钟
    
  # 通知
  notifications:
    slack_channel: "#chaos-experiments"
    notify_before: true
    notify_on_abort: true
```

## 二、实验分级

### 4. 混沌实验的成熟度分级是怎样的？

**答：**

```
混沌工程成熟度模型：

Level 0: 无混沌实践
  └── 故障来了才发现问题

Level 1: 手动 GameDay
  ├── 偶尔做一次故障演练
  ├── 手动注入，手动观察
  └── 仅在非生产环境

Level 2: 自动化实验
  ├── 工具化的故障注入
  ├── 自动化的安全防护
  ├── 定期在 Staging 运行
  └── 有实验记录和分析

Level 3: 生产混沌
  ├── 在生产环境运行
  ├── 有完善的安全机制
  ├── 持续发现新弱点
  └── 与 CI/CD 集成

Level 4: 持续混沌
  ├── 24/7 随机运行
  ├── 自动化发现和验证
  ├── 混沌实验即代码
  └── Netflix Chaos Monkey 模式
```

### 5. 不同类型的故障注入实验如何设计？

**答：**

```
常见实验清单：

基础设施故障：
  ├── Kill 实例：随机终止一个 Pod/VM
  ├── CPU 压力：注入 CPU 满载
  ├── 内存压力：注入内存消耗
  └── 磁盘填满：模拟磁盘空间耗尽

网络故障：
  ├── 延迟注入：增加 200ms 网络延迟
  ├── 丢包注入：10% 网络丢包
  ├── DNS 故障：模拟 DNS 解析失败
  └── 网络分区：模拟可用区之间断网

依赖故障：
  ├── 数据库不可用：模拟主库故障转移
  ├── 缓存失效：清空 Redis 缓存
  ├── 消息队列延迟：MQ 消费延迟
  └── 第三方 API 超时：下游服务超时

应用故障：
  ├── 进程崩溃：随机杀死应用进程
  ├── 线程阻塞：模拟死锁
  ├── 内存泄漏：注入缓慢内存增长
  └── 证书过期：模拟 TLS 证书失效
```

## 三、面试高频题

### 6. 面试题：如何制定混沌实验计划？

**答：**

```
实验计划制定步骤：

1. 风险评估
   ├── 列出系统关键依赖
   ├── 评估每个依赖的故障影响
   └── 按风险排优先级

2. 实验路线图
   Month 1: 基础实验
     ├── Pod Kill（验证 K8s 自愈）
     ├── 节点 Drain（验证调度）
     └── 服务重启（验证优雅关闭）

   Month 2: 网络实验
     ├── 延迟注入
     ├── DNS 故障
     └── 跨 AZ 通信中断

   Month 3: 依赖故障
     ├── 数据库故障转移
     ├── 缓存雪崩
     └── 第三方 API 超时

3. 频率规划
   ├── Staging：每周自动运行
   ├── Production：每月 GameDay
   └── CI/CD：每次部署前基础验证
```

### 7. 面试题：混沌实验失败了（系统崩了）怎么办？

**答：**

```
实验"失败"其实是"成功发现问题"：

1. 立即恢复
   ├── 触发 Kill Switch 停止实验
   ├── 按 Runbook 恢复服务
   └── 确认稳态恢复

2. 记录发现
   ├── 写 Postmortem
   ├── 记录故障传播路径
   └── 分析为什么防护没生效

3. 修复弱点
   ├── 创建 Action Items
   ├── 优先级排序修复
   └── 修复后重新实验验证

关键心态：
  "实验发现了弱点 = 好事"
  "比用户发现强 100 倍"
```
