# 混沌工程八股文

---

## 一、混沌工程基础

### 1. 什么是混沌工程？为什么需要混沌工程？

**答：** 混沌工程是在分布式系统上进行**受控实验**，以发现系统在面对故障时的弱点，从而**在故障发生之前主动发现和修复问题**。

**核心理念**：
> "发现问题的最佳时机是在它造成影响之前" —— Netflix

**混沌工程的五大原则（Principles of Chaos Engineering）：**

1. **建立稳态假说**：定义什么是系统正常行为
2. **模拟真实世界事件**：注入的故障应反映真实场景
3. **在生产环境运行**：只有在生产环境才能发现真实问题
4. **自动化持续运行**：将混沌实验纳入 CI/CD
5. **最小化爆炸半径**：从小规模开始，逐步扩大

### 2. 混沌工程与故障测试有什么区别？

**答：**

| 维度 | 故障测试 | 混沌工程 |
|------|----------|----------|
| 目标 | 验证已知故障的处理 | 发现未知的系统弱点 |
| 假设 | 知道预期结果 | 验证稳态假说 |
| 环境 | 通常在测试环境 | 尽量在生产环境 |
| 范围 | 特定组件/场景 | 系统级别的探索 |
| 频率 | 发布前执行 | 持续运行 |

### 3. 混沌工程的常见故障注入类型有哪些？

**答：**

| 类型 | 示例 | 目标 |
|------|------|------|
| **实例故障** | 随机终止 Pod/VM | 验证冗余和自动恢复 |
| **网络故障** | 延迟注入、丢包、分区 | 验证超时和重试机制 |
| **依赖故障** | 模拟下游服务不可用 | 验证熔断和降级 |
| **资源耗尽** | CPU/内存/磁盘打满 | 验证资源限制和告警 |
| **时钟偏移** | 系统时间异常 | 验证时间敏感的逻辑 |
| **DNS 故障** | DNS 解析失败/延迟 | 验证 DNS 缓存和容错 |

## 二、混沌工程工具

### 4. 主流混沌工程工具对比？

**答：**

| 工具 | 开发者 | 平台 | 特点 |
|------|--------|------|------|
| **Chaos Monkey** | Netflix | 云环境 | 随机终止实例，开山鼻祖 |
| **LitmusChaos** | CNCF | Kubernetes | 云原生、ChaosHub 实验库 |
| **Chaos Mesh** | PingCAP | Kubernetes | 丰富的故障类型、Dashboard |
| **Gremlin** | Gremlin Inc | 跨平台 | 商业产品、安全可控 |
| **AWS FIS** | AWS | AWS | AWS 原生、与 CloudWatch 集成 |
| **Pumba** | 开源 | Docker | 容器级故障注入 |

### 5. 什么是 GameDay？如何组织一次 GameDay？

**答：** GameDay 是一种有计划的**灾难演练**，团队在受控条件下模拟真实故障场景。

**GameDay 流程：**
1. **规划**：选择演练场景、定义成功标准、准备回滚方案
2. **准备**：通知相关团队、准备监控仪表盘
3. **执行**：注入故障、观察系统反应、记录时间线
4. **恢复**：清除故障注入、验证系统完全恢复
5. **复盘**：总结发现的问题、制定改进计划
