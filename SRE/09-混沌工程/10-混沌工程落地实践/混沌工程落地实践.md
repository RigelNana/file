## 一、落地路径

### 1. 混沌工程从零到一的落地路径是什么？

**答：**

```
混沌工程落地四阶段：

Phase 1: 奠基期（1-2 月）
  ├── 确保基础可观测性完备
  │   └── 没有监控就不要做混沌工程
  ├── 选择工具并搭建环境
  ├── 培训核心团队
  ├── 在 Dev/Staging 做第一次实验
  └── 产出第一份实验报告

Phase 2: 探索期（2-4 月）
  ├── Staging 定期自动化实验
  ├── 建立实验库（10+ 实验场景）
  ├── 组织第一次 GameDay
  ├── 开始在生产做简单实验（Pod Kill）
  └── 修复发现的弱点

Phase 3: 成熟期（4-8 月）
  ├── 生产常态化混沌实验
  ├── 与 CI/CD 集成
  ├── 实验覆盖核心链路
  ├── GameDay 季度化
  └── 产出量化的可靠性改进数据

Phase 4: 卓越期（8+ 月）
  ├── 全面自动化（类似 Netflix ChAP）
  ├── 混沌实验覆盖所有关键服务
  ├── 新服务上线必须通过混沌验证
  ├── 混沌工程文化深入团队
  └── 持续优化和创新
```

### 2. 混沌工程落地的常见挑战和解决方案？

**答：**

| 挑战 | 原因 | 解决方案 |
|------|------|----------|
| 管理层不支持 | 怕出事 | 先在 Staging 展示价值 |
| 团队抵触 | 觉得是找茬 | 强调是"找到弱点"不是"制造问题" |
| 监控不足 | 看不到实验效果 | 先补齐可观测性 |
| 不知道测什么 | 缺乏场景 | 从 Postmortem 中提取场景 |
| 工具复杂 | 学习成本高 | 从简单工具开始 |
| 实验失控 | 影响了用户 | 完善安全防护机制 |

### 3. 从 Postmortem 到混沌实验的转化怎么做？

**答：**

```
Postmortem 驱动的混沌实验：

历史 Postmortem 分析：
  INC-001: Redis 主节点故障，切换慢
  → 实验：模拟 Redis 主节点 Kill
  → 验证：Sentinel 自动切换是否 < 10s

  INC-012: 上游超时导致线程池耗尽
  → 实验：注入上游服务 5s 延迟
  → 验证：熔断器是否正常触发

  INC-025: 磁盘满导致写入失败
  → 实验：填充磁盘到 95%
  → 验证：告警和自动清理是否生效

  INC-030: DNS 故障导致跨服务调用失败
  → 实验：模拟 DNS 不可用
  → 验证：DNS 缓存和降级机制

转化流程：
  Postmortem → 提取故障模式 → 设计实验
  → 验证修复效果 → 防止同类问题
```

## 二、度量与持续改进

### 4. 如何度量混沌工程的效果？

**答：**

```
混沌工程度量体系：

输入指标（投入）：
  ├── 实验覆盖率：核心服务被实验覆盖的比例
  ├── 实验频率：每月/季度运行的实验数
  ├── 场景丰富度：实验场景类型数量
  └── 参与团队数：参与混沌工程的团队数

输出指标（效果）：
  ├── 弱点发现数：每季度发现的弱点数
  ├── 修复率：发现弱点的修复完成率
  ├── 重复事件率降低：同类事件减少比例
  ├── MTTR 改善：事件恢复时间趋势
  └── 信心指数：团队对系统韧性的信心评分

示例 Dashboard 指标：
  ├── 本季度实验次数: 45
  ├── 发现弱点: 12
  ├── 已修复: 10 (83%)
  ├── 验证通过: 8
  ├── 核心链路覆盖率: 75%
  └── P0 事件同比减少: 40%
```

### 5. 混沌工程如何与 CI/CD 集成？

**答：**

```
CI/CD 集成模式：

模式一：部署后验证
  Deploy → 等待稳定(5min) → 混沌实验 → 通过/回滚
  
  # GitLab CI 示例
  chaos-validation:
    stage: post-deploy
    script:
      - sleep 300  # 等待稳定
      - litmus run --experiment pod-delete \
          --target payment-api \
          --namespace canary
      - litmus check --result pass
    only:
      - main
    when: on_success

模式二：定期调度
  每天凌晨在 Staging 环境自动运行
  
  # CronJob
  apiVersion: batch/v1
  kind: CronJob
  metadata:
    name: daily-chaos
  spec:
    schedule: "0 2 * * 1-5"  # 工作日凌晨2点
    jobTemplate:
      spec:
        template:
          spec:
            containers:
              - name: chaos-runner
                image: litmuschaos/runner
                args: ["--suite", "daily-basic"]

模式三：PR 验证
  PR 合并前运行基础混沌测试
  确保新代码不会降低韧性
```

## 三、面试高频题

### 6. 面试题：描述你推动混沌工程落地的经历

**答：**

```
STAR 方法回答：

Situation:
  团队有 20 个微服务，每季度 2-3 次 P0 事件
  大部分故障都是"没想到会这样坏"

Task:
  引入混沌工程，主动发现弱点

Action:
  1. 调研并选择了 LitmusChaos
  2. 在 Staging 做了第一次 Pod Kill 实验
     发现：Pod 被 Kill 后服务中断 2 分钟
     原因：健康检查配置不合理
  3. 修复后重新验证：恢复时间 < 10 秒
  4. 逐步扩展到生产环境
  5. 建立了月度 GameDay 机制
  6. 与 CI/CD 集成自动化

Result:
  - 半年内发现并修复 20+ 个弱点
  - P0 事件从每季度 3 次降到 1 次
  - MTTR 从 45 分钟降到 15 分钟
  - 团队对系统韧性信心显著提升
```

```

   └── 验证安全防护措施

5. 全栈化
   ├── 不只是基础设施
   ├── 应用层 / 业务层 混沌
   └── 人员和流程层面的混沌
```
