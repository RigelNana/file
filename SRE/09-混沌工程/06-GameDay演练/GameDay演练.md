## 一、GameDay 基础

### 1. 什么是 GameDay？和混沌实验有什么区别？

**答：**

```
GameDay 定义：
  有组织、有计划的大规模故障演练活动，
  模拟真实事件场景，练习团队的事件响应能力。

GameDay vs 日常混沌实验：

|              | 日常混沌实验        | GameDay         |
|-------------|-------------------|-----------------|
| 频率         | 每天/每周          | 每月/每季度      |
| 规模         | 小范围单系统        | 大范围跨系统     |
| 参与人       | 自动化执行         | 多团队参与       |
| 准备         | 脚本化             | 需要详细策划     |
| 目的         | 验证单点韧性        | 验证全组织响应    |
| 持续时间      | 分钟级             | 2-8 小时        |
```

### 2. GameDay 的完整流程是什么？

**答：**

```
GameDay 三阶段：

═══ Phase 1: 准备（2-4 周前）═══

  1. 确定演练目标
     "验证支付链路在 DB 故障转移时的可用性"

  2. 设计场景
     ├── 故障类型：DB 主节点故障
     ├── 注入方式：Kill 主节点进程
     ├── 影响范围：支付链路全部服务
     └── 预期行为：自动切换到从节点

  3. 确定参与者
     ├── Red Team（攻击方）：注入故障
     ├── Blue Team（防守方）：发现和恢复
     ├── 观察者：记录过程
     └── 仲裁者：控制节奏

  4. 准备安全措施
     ├── Kill Switch
     ├── 回滚方案
     └── 升级联系人

═══ Phase 2: 执行（当天）═══

  09:00  开场：说明目标和规则
  09:15  注入故障（Red Team）
  09:20  告警触发（验证检测能力）
  09:25  On-Call 响应（Blue Team）
  09:45  缓解/恢复
  10:00  确认恢复
  10:15  注入第二个故障（如有）
  ...
  12:00  演练结束

═══ Phase 3: 复盘（当天或次日）═══

  1. 回顾时间线
  2. 讨论做得好的和待改进的
  3. 列出 Action Items
  4. 分享学习
```

### 3. 如何设计好的 GameDay 场景？

**答：**

| 场景类型 | 示例 | 验证目标 |
|----------|------|----------|
| **单点故障** | Kill 一个数据库主节点 | 自动故障转移 |
| **级联故障** | 核心缓存不可用 | 降级和熔断 |
| **网络故障** | AZ 间网络中断 | 跨区容灾 |
| **容量不足** | 流量突增 3 倍 | 弹性伸缩 |
| **安全事件** | 模拟数据泄露告警 | 安全事件响应 |
| **全站故障** | 模拟区域级故障 | 灾难恢复 |

## 二、组织与执行

### 4. GameDay 的角色分工如何安排？

**答：**

```
角色定义：

GameDay Lead（总负责人）
  ├── 规划整体方案
  ├── 协调各方资源
  ├── 控制演练节奏
  └── 决定是否继续或终止

Red Team（故障注入方）
  ├── 按计划注入故障
  ├── 观察系统行为
  ├── 记录注入时间点
  └── 控制故障持续时间

Blue Team（事件响应方）
  ├── 正常 On-Call 流程响应
  ├── 不知道具体故障（盲测模式）
  ├── 按 SOP 排查和恢复
  └── 暴露真实的响应能力

Observer（观察记录者）
  ├── 记录完整时间线
  ├── 标注关键决策点
  ├── 记录意外发现
  └── 为复盘提供素材

Safety Controller（安全控制员）
  ├── 监控安全指标
  ├── 有最高优先级的终止权
  ├── 实时评估影响
  └── 确保不超出预设范围
```

### 5. GameDay 的沟通模板是什么？

**答：**

```
演练前通知（1 天前）：

📋 GameDay 通知
  日期：2024-01-20 09:00-12:00 UTC
  范围：支付链路故障演练
  参与团队：SRE, 支付, 数据库, 基础设施
  可能影响：支付服务短暂中断（< 5 分钟）
  安全措施：有 Kill Switch，超过阈值自动终止
  联系人：GameDay Lead @张三

演练中通知：

🎮 GameDay 进行中
  ⚠️ 当前看到的告警可能是演练的一部分
  如有疑问联系 @张三
  如果非演练的真实事件 → 呼叫 @Safety

演练后通知：

✅ GameDay 已结束
  所有系统恢复正常
  复盘会议：今天 14:00
  感谢所有参与者
```

## 三、面试高频题

### 6. 面试题：如何衡量 GameDay 的效果？

**答：**

| 指标 | 衡量方法 | 目标 |
|------|----------|------|
| **检测时间** | 从故障注入到告警触发 | < 5 分钟 |
| **响应时间** | 从告警到有人 ACK | < 5 分钟 |
| **诊断准确性** | 是否准确定位根因 | > 80% |
| **恢复时间** | 从故障到服务恢复 | < 30 分钟 |
| **SOP 遵循度** | 是否按流程操作 | > 90% |
| **发现问题数** | 暴露的弱点数量 | 越多越好 |

### 7. 面试题：如何从零开始组织第一次 GameDay？

**答：**

```
第一次 GameDay 建议：

选择简单场景：
  ├── Kill 一个非关键服务 Pod
  ├── 或者在 Staging 环境做
  └── 避免第一次就做生产大规模故障

控制规模：
  ├── 2-3 个团队参与
  ├── 1-2 个故障场景
  └── 总时长 2 小时内

重点是学习：
  ├── 流程是否顺畅
  ├── 工具是否好用
  ├── 沟通是否高效
  └── 为后续 GameDay 积累经验

复盘重于演练本身：
  ├── 发现了什么问题？
  ├── 那些做得好？
  ├── 下次 GameDay 如何改进？
  └── 产出 Action Items
```
