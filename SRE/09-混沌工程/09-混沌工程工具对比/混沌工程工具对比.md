## 一、工具全景

**答：**

```

  ├── Gremlin         商业平台，功能最全面

  ├── LitmusChaos     CNCF 孵化，端到端平台

  ├── AWS FIS         AWS 故障注入服务

网络/应用层：
  ├── Toxiproxy       网络代理级注入
  ├── tc netem        Linux 网络层
  └── Pumba           Docker 容器级

压力/资源型：
  ├── stress-ng       系统资源压力
  └── ChaosBlade      阿里开源多层级工具
```

## 二、工具深入对比

### 4. Chaos Mesh vs LitmusChaos 详细对比？

**答：**

```
Chaos Mesh:
  ├── 核心技术：基于 eBPF + Sidecar
  ├── 网络控制：非常精细（基于 eBPF）
  ├── 时间故障：支持 TimeChaos（模拟时钟偏移）
  ├── IO 故障：支持 IOChaos（文件系统级）
  ├── Dashboard：自带可视化界面
  └── Workflow：支持实验编排

LitmusChaos:
  ├── 核心技术：Operator + Runner Pod
  ├── 实验库：ChaosHub 中有丰富预定义实验
  ├── 云支持：AWS/GCP/Azure/VMware
  ├── Probe：内置探针验证机制
  ├── Portal：中心化管理多集群
  └── GitOps：实验即代码

选哪个：
  精细网络/IO 控制       → Chaos Mesh
  多云/混合环境          → LitmusChaos
  纯 K8s + 简洁         → Chaos Mesh
  需要集中管理多集群     → LitmusChaos
```

### 5. ChaosBlade 适合什么场景？

**答：**

```
ChaosBlade（阿里巴巴开源）：

特点：
  ├── 支持多平台：K8s / Docker / 物理机 / JVM
  ├── 命令行操作：blade create/destroy
  ├── JVM 深度支持：方法级故障注入
  ├── 中文文档友好
  └── 双十一实战检验

使用示例：
# CPU 满载
blade create cpu fullload --cpu-percent 80

# 网络延迟
blade create network delay --interface eth0 \
  --time 200 --offset 50

# JVM 方法延迟
blade create jvm delay --time 3000 \
  --classname com.example.PaymentService \
  --methodname processPayment

# K8s Pod Kill
blade create k8s pod-pod delete \
  --namespace production \
  --labels app=payment

# 销毁（恢复）
blade destroy <experiment-uid>

适用场景：
  ├── 有 Java/JVM 服务（JVM 注入能力强）
  ├── 混合环境（不只是 K8s）
  ├── 国内团队（中文文档和社区）
  └── 阿里云用户（生态集成）
```

```
  ├── 月度 GameDay（低成本高价值）
  ├── Wheel of Misfortune（零成本培训）

关键：工具是次要的，方法论是核心
```
