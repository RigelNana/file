# 稳态假说与实验验证八股文

---

## 一、稳态假说

### 1. 什么是稳态假说？如何科学地定义？

**答：**

```
稳态假说 = 对系统正常行为的量化描述

公式：
  "在 [正常条件] 下，
   [度量指标] 保持在 [可接受范围]"

好的稳态假说示例：
  ✓ "正常情况下，订单 API 的：
     - 成功率 > 99.9%
     - P50 延迟 < 50ms
     - P99 延迟 < 200ms"

差的稳态假说：
  ✗ "系统正常运行"（不可量化）
  ✗ "CPU < 80%"（内部指标，非用户感知）

稳态选择原则：
  ┌──────────────────────────────────────┐
  │ 1. 用户可感知：成功率、延迟、吞吐     │
  │ 2. 可量化：有明确的数值范围            │
  │ 3. 可观测：有配套的监控能力            │
  │ 4. 业务相关：与 SLO 对齐             │
  └──────────────────────────────────────┘
```

### 2. 混沌假说和稳态假说的关系是什么？

**答：**

```
两者关系：

稳态假说：定义"正常"
  "订单成功率 > 99.9%"

混沌假说：定义"故障下的预期"
  "当 Redis 主节点故障时，
   订单成功率仍 > 99.5%
   且在 30 秒内恢复到 99.9%"

实验验证逻辑：
  1. 观察稳态：确认当前满足稳态假说 ✓
  2. 注入故障：执行故障注入
  3. 验证混沌假说：
     ├── 成立 → 系统有足够韧性 ✓
     └── 不成立 → 发现弱点，需要修复 ✗
  4. 恢复后验证：稳态是否恢复 ✓

结果矩阵：
  ┌─────────────┬──────────────────────────┐
  │ 假说成立     │ 好事！系统韧性良好       │
  │ 假说不成立   │ 好事！发现弱点可以修复    │
  │ 实验出了意外 │ 也好！发现了盲区         │
  └─────────────┴──────────────────────────┘
```

## 二、实验验证流程

### 3. 混沌实验的科学验证流程是什么？

**答：**

```
科学方法在混沌工程中的应用：

  科学方法              混沌工程
  ─────────            ──────────
  观察现象       →     分析系统架构和依赖
  提出假说       →     定义稳态和混沌假说
  设计实验       →     设计故障注入方案
  控制变量       →     限定实验范围和条件
  执行实验       →     注入故障并观察
  收集数据       →     收集 SLI/SLO 数据
  分析结果       →     对比稳态前后变化
  得出结论       →     确认假说是否成立
  复现验证       →     修复后重新实验

对照实验设计：
  ┌──────────────────────────────────┐
  │ 对照组：                         │
  │ 正常流量路径，无故障注入          │
  │                                  │
  │ 实验组：                         │
  │ 相同流量，注入故障                │
  │                                  │
  │ 对比：两组的 SLI 差异            │
  └──────────────────────────────────┘
```

### 4. 如何设计有效的实验探针（Probe）？

**答：**

```yaml
# LitmusChaos Probe 配置示例

# HTTP 探针：验证服务可用性
- name: "check-payment-available"
  type: httpProbe
  httpProbe/inputs:
    url: "http://payment-api:8080/healthz"
    method:
      get:
        criteria: "=="
        responseCode: "200"
  mode: Continuous       # 持续检测
  runProperties:
    probeTimeout: 3s
    interval: 5s
    retry: 2

# PromQL 探针：验证 SLO
- name: "check-error-rate"
  type: promProbe
  promProbe/inputs:
    endpoint: "http://prometheus:9090"
    query: |
      sum(rate(http_requests_total{code=~"5.."}[1m]))
      / sum(rate(http_requests_total[1m])) * 100
    comparator:
      criteria: "<="
      value: "1"           # 错误率 <= 1%
  mode: Continuous

# 命令探针：自定义验证
- name: "check-data-consistency"
  type: cmdProbe
  cmdProbe/inputs:
    command: "./verify-data-consistency.sh"
    comparator:
      type: "int"
      criteria: "=="
      value: "0"
  mode: OnChaos           # 故障注入期间执行
```

### 5. 实验结果如何分析和报告？

**答：**

```
实验报告模板：

# 混沌实验报告
## 实验信息
  实验名称: Redis 主节点故障容忍
  日期: 2024-01-20
  执行人: SRE Team

## 稳态假说
  订单成功率 > 99.9%，P99 < 200ms

## 混沌假说
  Redis 主节点故障后：
  - 成功率 > 99.5%（允许短暂降级）
  - 30 秒内恢复到 99.9%
  - 无数据丢失

## 实验结果
  ├── 故障注入: T+0s Kill Redis Primary
  ├── 告警触发: T+5s ✓（预期 <10s）
  ├── 自动切换: T+8s ✓（预期 <15s）
  ├── 成功率低谷: 98.2% ✗（预期 >99.5%）
  ├── 恢复到稳态: T+45s ✗（预期 <30s）
  └── 数据丢失: 0 ✓

## 结论
  混沌假说部分成立。
  发现 2 个弱点：
  1. 切换期间成功率低于预期
  2. 恢复时间超出预期

## Action Items
  1. 优化连接池重连逻辑
  2. 添加本地缓存作为降级
  3. 修复后重新验证
```

## 三、面试高频题

### 6. 面试题：混沌实验的结果都意味着什么？

**答：**

```
四种可能的结果：

结果 1: 假说成立 ✓
  系统表现符合预期
  → 记录为通过，增加信心
  → 下次可以尝试更大范围

结果 2: 假说不成立 ✗
  系统表现不符合预期
  → 最有价值的结果！
  → 发现了具体弱点
  → 创建修复 Action Item

结果 3: 意外发现 ⚡
  发现了假说之外的问题
  → 比如监控没有触发
  → 或者连锁影响了其他系统
  → 记录并作为新的实验目标

结果 4: 实验本身失败 ❌
  故障未能成功注入
  → 检查实验配置
  → 可能权限/网络问题
  → 不算有效实验结果
```

### 7. 面试题：如何确保混沌实验的可复现性？

**答：**

```
可复现性保障：

1. 实验即代码
   所有实验定义为 YAML/代码
   存储在 Git 仓库中
   版本控制

2. 环境一致性
   记录实验时的环境状态
   ├── 服务版本
   ├── 配置版本
   ├── 副本数量
   └── 流量水平

3. 自动化执行
   用 CI/CD Pipeline 触发
   减少人为操作差异

4. 完整记录
   ├── 实验参数
   ├── 环境上下文
   ├── 所有指标快照
   └── 时间线和截图
```
