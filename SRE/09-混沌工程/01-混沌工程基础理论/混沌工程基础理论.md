## 一、混沌工程概述

### 1. 什么是混沌工程？

**答：**

```
混沌工程定义（Netflix）：
  "在分布式系统上进行实验的学科，
   目的是建立对系统承受生产环境中
   湍流条件能力的信心。"

核心思想：
  ┌─────────────────────────────────────────┐
  │ 不是"制造故障"，而是"发现弱点"          │
  │ 不是"测试"，而是"实验"                  │
  │ 不是"破坏系统"，而是"提高韧性"          │
  └─────────────────────────────────────────┘

为什么需要混沌工程：
  1. 分布式系统复杂性 → 无法预见所有故障模式
  2. "在生产中发现"比"用户发现"好
  3. 验证监控/告警/恢复机制是否有效
  4. 建立团队对事件响应的信心
```

### 2. 混沌工程的核心原则是什么？

**答：**

```
混沌工程五大原则（Principles of Chaos Engineering）：

1. 建立稳态假说 (Steady State Hypothesis)
   定义系统正常行为的度量
   例："正常情况下，订单成功率 > 99.9%"

2. 引入真实世界事件 (Real-world Events)
   模拟实际可能发生的故障
   例：服务器崩溃、网络延迟、磁盘满

3. 在生产环境运行实验 (Run in Production)
   非生产环境无法反映真实行为
   但需要做好安全防护

4. 自动化持续运行 (Automate Continuously)
   不是一次性的，而是持续进行
   集成到 CI/CD 流程中

5. 最小化爆炸半径 (Minimize Blast Radius)
   从小范围开始，逐步扩大
   有安全开关（Kill Switch）
```

### 3. 混沌工程与故障注入测试有什么区别？

**答：**

| 维度 | 故障注入测试 | 混沌工程 |
|------|-------------|----------|
| 目的 | 验证已知故障场景 | 发现未知弱点 |
| 方法 | 注入特定故障 | 基于假说的实验 |
| 环境 | 通常在测试环境 | 最终在生产环境 |
| 结果 | 通过/失败 | 学习和发现 |
| 频率 | 发布前做 | 持续进行 |
| 范围 | 单点故障 | 系统级行为 |

## 二、混沌实验设计

### 4. 混沌实验的完整流程是什么？

**答：**

```
混沌实验流程：

Step 1: 定义稳态
  "用户可见的正常行为指标"
  ├── 订单成功率 > 99.9%
  ├── P99 延迟 < 200ms
  └── 错误率 < 0.1%

Step 2: 建立假说
  "当 X 发生时，系统应该保持 Y"
  例："当 Redis 主节点故障时，
      订单成功率仍 > 99.5%（允许短暂降级）"

Step 3: 设计实验
  ├── 故障类型：Kill Redis 主节点
  ├── 影响范围：仅 Staging / 生产 5% 流量
  ├── 持续时间：5 分钟
  ├── 回滚条件：错误率 > 5% 自动停止
  └── 监控指标：成功率、延迟、缓存命中率

Step 4: 执行实验
  ├── 通知相关团队
  ├── 启动实验
  ├── 实时监控指标
  └── 记录观察结果

Step 5: 分析结果
  ├── 稳态假说是否成立？
  ├── 发现了什么弱点？
  └── 需要什么改进？

Step 6: 修复和重验
  ├── 修复发现的弱点
  └── 重新运行实验验证
```

### 5. 常见的混沌实验类型有哪些？

**答：**

| 类别 | 实验类型 | 工具 |
|------|----------|------|
| **基础设施** | Kill 实例/容器 | Chaos Monkey |
| | 磁盘填满 | stress-ng |
| | CPU/Memory 压力 | stress-ng |
| **网络** | 网络延迟注入 | tc netem |
| | 网络丢包 | tc netem |
| | DNS 故障 | CoreDNS 故障注入 |
| | 网络分区 | iptables |
| **应用** | 服务依赖超时 | Toxiproxy |
| | 异常注入 | 代码级故障注入 |
| **平台** | AZ 故障模拟 | AWS FIS |
| | K8s 节点驱逐 | Litmus |

## 三、面试高频题

### 6. 面试题：你如何说服管理层支持混沌工程？

**答：**

```
说服方法：

1. 用数据说话
   "去年 P0 事件中，70% 是我们没有预见的故障模式"
   "混沌工程可以主动发现这些弱点"

2. 从小规模开始
   "先在 Staging 环境做"
   "第一次在生产只影响 1% 流量"
   "有自动停止机制"

3. 展示 ROI
   ├── 预防 1 次 P0 事件 = 节省 ¥XX 万
   ├── 减少 MTTR
   ├── 提升团队事件响应能力
   └── 增加用户信任

4. 引用行业标杆
   "Netflix/Google/Amazon 都在做"
   "这是行业最佳实践"
```

### 7. 面试题：混沌工程和测试的关系？

**答：**

```
互补关系：

测试金字塔 + 混沌工程：

  ┌─────────┐
  │ 混沌工程 │ ← 发现未知弱点
  ├─────────┤
  │ E2E 测试 │ ← 验证用户场景
  ├─────────┤
  │ 集成测试  │ ← 验证组件交互
  ├─────────┤
  │ 单元测试  │ ← 验证代码逻辑
  └─────────┘

  测试回答："代码是否正确？"
  混沌工程回答："系统是否有韧性？"
```
