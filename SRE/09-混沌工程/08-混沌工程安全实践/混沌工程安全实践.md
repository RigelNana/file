## 一、安全框架

### 1. 混沌工程的安全框架包含哪些方面？

**答：**

```
混沌工程安全矩阵：

          准备阶段    执行阶段    恢复阶段
         ─────────  ─────────  ─────────
范围控制   ✓ 限定目标   ✓ 验证范围   ✓ 清理痕迹
权限控制   ✓ 审批流程   ✓ 最小权限   ✓ 权限回收
监控保障   ✓ 确认可用   ✓ 实时观测   ✓ 恢复验证
止损机制   ✓ 预设阈值   ✓ 自动终止   ✓ 回滚验证
沟通机制   ✓ 提前通知   ✓ 实时更新   ✓ 结果分享

安全三道防线：
  第一道：审批流程（人工把关）
  第二道：自动安全防护（系统把关）
  第三道：紧急终止能力（兜底保障）
```

### 2. 混沌实验的审批流程如何设计？

**答：**

```
审批矩阵：

环境 × 规模 → 审批级别

              小规模           中规模           大规模
            (1个Pod/实例)  (多Pod/单服务)   (多服务/跨AZ)
  ─────── ──────────── ──────────── ────────────
  Dev      自动执行       团队负责人        无需
  Staging  团队负责人     SRE Manager      SRE Director
  Prod     SRE Manager   SRE Director     VP + GameDay

审批清单：
  □ 实验目标和假说明确
  □ 安全控制措施完备
  □ Kill Switch 可用
  □ 监控和告警正常
  □ 回滚方案就绪
  □ 相关团队已通知
  □ 不在变更冻结/大促期间
  □ 有经验的人监督执行
```

### 3. 自动安全防护如何实现？

**答：**

```yaml
# 混沌实验安全控制器示例

apiVersion: chaos.example.com/v1
kind: SafetyPolicy
metadata:
  name: production-safety
spec:
  # 全局安全规则
  globalRules:
    # 禁止时段
    blackoutWindows:
      - cron: "0 0 * * 0"     # 周日不实验
      - name: "change-freeze"
        start: "2024-12-20"
        end: "2025-01-05"
    
    # 最大并行实验数
    maxConcurrentExperiments: 1
    
    # 全局停止条件
    globalAbortConditions:
      - type: prometheus
        query: |
          sum(rate(http_requests_total{code=~"5.."}[1m]))
          / sum(rate(http_requests_total[1m]))
        threshold: 0.05      # 全局错误率 > 5%
      - type: pagerduty
        condition: "any_p0_incident"  # 有 P0 就停

  # 服务级安全规则
  serviceRules:
    - service: "payment-api"
      maxPodsAffected: 1     # 最多影响 1 个 Pod
      minAvailable: 3        # 至少保留 3 个 Pod
      allowedExperiments:
        - "pod-delete"
        - "pod-network-latency"
      blockedExperiments:
        - "node-drain"       # 不允许节点驱逐
```

## 二、风险管理

### 4. 混沌实验的风险等级如何分类？

**答：**

| 风险等级 | 描述 | 例子 | 安全要求 |
|----------|------|------|----------|
| **低** | 无用户感知，可自动恢复 | 非生产 Pod Kill | 自动审批 |
| **中** | 可能短暂影响少量用户 | 生产网络延迟注入(1%) | 团队审批+Kill Switch |
| **高** | 可能影响较多用户 | 生产数据库故障转移 | Manager审批+安全员 |
| **极高** | 可能导致服务中断 | AZ级故障模拟 | VP审批+GameDay模式 |

### 5. 混沌实验出事了怎么办？

**答：**

```
紧急应对流程：

Level 1: 自动触发（秒级）
  阈值超限 → 自动终止实验
  └── Kill Switch 生效
  └── 自动恢复注入前状态

Level 2: 人工介入（分钟级）
  自动恢复未生效 → 人工操作
  ├── 手动执行回滚脚本
  ├── 手动清理故障注入
  └── 必要时重启服务

Level 3: 升级处理（5-15分钟）
  手动恢复困难 → 按事件流程升级
  ├── 宣布为真实事件
  ├── 启动事件响应流程
  └── IC 接管协调

事后处理：
  ├── 写 Postmortem（包含混沌实验失败的分析）
  ├── 改进安全防护措施
  ├── 更新安全策略
  └── 团队分享经验教训
```

## 三、生产安全实践

### 6. 在生产环境做混沌实验的安全准则是什么？

**答：**

```
生产混沌安全十条准则：

1.  永远不影响所有流量
    从 1% → 5% → 10% 逐步递增

2.  有明确的回退机制
    每种故障注入都有对应的清除命令

3.  有自动停止条件
    不依赖人来判断是否该停止

4.  在工作时间内进行
    确保有人可以响应

5.  不在变更冻结期/大促前做
    避免火上浇油

6.  不同时做多个实验
    一次只注入一种故障

7.  提前通知所有相关方
    避免误判为真实事件

8.  新实验先在 Staging 验证
    确认工具和流程可用

9.  有经验的 SRE 监督
    新手不独立做生产实验

10. 每次实验都有记录
    为持续改进提供数据
```

### 7. 面试题：如何说服团队在生产做混沌工程？

**答：**

```
阶梯式推进策略：

Step 1: 先用数据说服
  "去年 P0 事件中 X 个可以被混沌工程提前发现"

Step 2: 从安全的地方开始
  "我们先在 Staging 做，零风险"

Step 3: 展示 Staging 的成果
  "Staging 发现了 5 个弱点，已全部修复"

Step 4: 渐进式进入生产
  "生产实验只影响 1% 流量，有自动停止"

Step 5: 用成功案例推广
  "上次生产实验帮我们避免了一次 P0"

如果仍有顾虑：
  ├── "生产中已经有'混沌'了——只是不受控的"
  ├── "受控的实验比意外的故障安全得多"
  └── "每次都是可逆的，有 Kill Switch"
```
